# 架构决策记录(ADR)

## 🎯 什么是架构决策记录

> **架构决策记录（Architecture Decision Record, ADR）** 是一种轻量级文档形式，用于记录项目中重要的架构决策及其背景、理由和影响。
>
> 每条ADR回答以下问题：
> - **背景**：为什么需要做这个决策？
> - **问题**：具体要解决什么问题？
> - **决策**：最终选择了什么方案？
> - **理由**：为什么选择这个方案而不是其他备选？
> - **影响**：这个决策会带来哪些后果和影响范围？

### ADR生命周期状态

| 状态 | 含义 | 说明 |
|------|------|------|
| 📝 `proposed` | 提议中 | 决策方案正在讨论，尚未最终确定 |
| ✅ `accepted` | 已接受 | 决策已通过，开始实施 |
| 🔍 `reviewed` | 已审核 | 决策已实施并经过验证/复盘 |
| ❌ `deprecated` | 已废弃 | 决策被后续新决策取代 |
| 🔄 `superseded` | 已替代 | 决策已被新的ADR明确替代 |

## 📖 如何阅读ADR

### 推荐阅读顺序

1. **Step 1**: 先阅读「问题/场景」章节，理解决策要解决的具体问题
2. **Step 2**: 阅读「背景」章节，了解决策的上下文和约束条件
3. **Step 3**: 重点看「决策」和「决策理由」表格，理解为什么选这个方案
4. **Step 4**: 阅读「影响范围」，评估该决策对你的工作可能产生的影响
5. **Step 5**: 如有需要，查看「参考」链接获取更多外部资料

### 阅读ADR时需要关注的关键点

| 关注点 | 说明 |
|--------|------|
| 🔍 **备选方案对比** | 不要只看最终选择，还要理解为什么不选其他方案 |
| ⚖️ **权衡取舍** | 每个决策都有 trade-off，理解选择方案的代价 |
| 📅 **决策日期** | 技术决策有时效性，早期决策可能因技术演进而需要重新评估 |
| 🎯 **影响范围** | 确认该决策是否影响你正在工作的模块 |
