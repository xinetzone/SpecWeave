# 局限性与对比

> 本章客观介绍 Open Code Review 的局限性、适用边界、已知问题，避免读者踩坑。任何工具都有其适用场景，理解边界才能用对地方。

---

## 不适用场景

| 场景 | 为什么不适用 | 建议替代方案 |
|------|-------------|-------------|
| 安全审计、漏洞挖掘 | OCR 召回率（最高 20.00%）低于 Claude Code（最高 28.90%），可能遗漏安全风险 | 使用 Claude Code 的 `/code-review`，或两者配合使用 |
| 需要"宁可错杀不可放过"的全面扫描 | OCR 设计哲学偏向高准确率、低噪声，会主动丢弃低置信度意见 | 使用 Claude Code 等通用 Agent 方案，最大化召回 |
| 纯本地化离线部署 | 必须配置外部 LLM 端点，无内置模型 | 使用 Ollama 等本地推理框架提供兼容 OpenAI API 的端点 |
| 非主流语言/小众文件类型评审 | 系统默认 13 套规则覆盖主流语言，小众语言无内置规则 | 自定义规则文件（`--rule`），或贡献规则到社区 |
| 需要深度跨仓库依赖分析 | 当前工具集聚焦单仓库内评审，`code_search` 基于 git grep | 配合静态分析工具（如 SonarQube）使用 |
| 实时编码阶段即时反馈 | CLI 形态面向 PR/commit 评审，非 IDE 内实时提示 | 等待官方 IDE 插件发布，或使用 Cursor 等 AI IDE |

## 已知问题与陷阱

### 1. 召回率不及 Claude Code

Open Code Review 的核心优势是准确率，但召回率明显低于 Claude Code：

- **OCR 最优召回率**：20.00%（Claude-4.6-Opus）
- **Claude Code 最优召回率**：28.90%（Claude-4.6-Opus）
- **差距**：CC 比 OCR 多发现约 134 个真实问题，增幅约 **45%**

**原因**：OCR 的分治策略和上下文隔离降低了跨文件关联缺陷的发现概率；同时反思模型会主动拦截疑似误报，可能牺牲部分真实问题。

**应对建议**：对于安全审计、上线前 Checklist 等"宁可错杀不可放过"的场景，应优先选择 Claude Code，或将 OCR 与 Claude Code 配合使用——OCR 负责工程级稳定评审，Claude Code 负责最大化覆盖。

---

### 2. 规则存在边际效益递减现象

AI 代码评审中，规则写得越多，指令跟随越差。这是 Open Code Review 明确指出的一个客观规律：

- 规则数量增加 → 单条规则的注意力权重被稀释 → 模型对规则的执行度下降
- 不同业务 DNA 不同（支付系统的安全规则对内容管理系统可能是噪音）
- 同一个客观问题在不同团队中的重要性也不同

**应对建议**：使用四层规则穿透机制，针对不同文件类型精准匹配规则，避免"一套规则打天下"。优先使用 `path` glob 模式将规则限定在特定文件范围内。

---

### 3. 需要配置 LLM 端点

Open Code Review 本身不内置大模型，必须配置外部 LLM 端点才能工作：

```bash
ocr config provider          # 选择内置供应商或添加自定义供应商
ocr config model             # 为当前供应商选择模型
```

**门槛体现**：
- 需要获取 LLM API Key（支持 OpenAI、Anthropic、Qwen 等）
- 对于纯本地化部署场景，需自行搭建兼容 OpenAI API 的推理服务
- CI/CD 集成时需要通过 Secrets 注入 `OCR_LLM_URL`、`OCR_LLM_AUTH_TOKEN`、`OCR_LLM_MODEL`

**应对建议**：参考官方 `examples/` 目录下的 GitHub Actions 和 GitLab CI 示例，开箱可用。

---

### 4. 内部版特性未完全开源

Open Code Review 的前身是阿里集团内部 AI 代码评审助手，对外开源版本与内部版存在差异：

- **文件打包策略**：内部版支持智能文件打包（如将 `message_en.properties` 与 `message_zh.properties` 打包评审），对外版本**暂时采用固定的分治策略**
- **反思模型**：内部版使用专项训练的 Qwen3-30B-A3B 反思模型，误报拦截率从 30.09% 提升到 52.63%，对外版本依赖基模能力
- **定位模型**：内部版使用专项训练的 Qwen3-8B 定位模型，成功率从 37.35% 提升到 85.65%，对外版本依赖三层递进式定位策略的工程兜底

**应对建议**：关注项目 Roadmap，官方正在逐步将内部版特性开源到社区。

---

### 5. 系统默认规则偏向阿里内部场景

系统默认内置的 13 套语言/文件类型专属规则文档，是根据阿里内部数万用户的使用数据迭代而来：

- Java 规则关注 NPE、线程安全、循环内 N+1 查询（阿里 Java 开发规约视角）
- 部分规则可能不适用于非阿里技术栈的团队
- 规则的优先级排序反映的是阿里内部问题分布，外部团队可能不同

**应对建议**：外部团队应通过项目级规则（`.opencodereview/rule.json`）或用户级规则（`~/.opencodereview/rule.json`）覆盖默认规则，定制符合自身业务 DNA 的评审规则。

---

### 6. 专用模型未开源

Open Code Review 内部版依赖的两个专用模型目前未开源：

| 模型 | 用途 | 性能提升 | 开源状态 |
|------|------|---------|---------|
| Qwen3-30B-A3B 反思模型 | 误报拦截过滤器 | 误报拦截率 30.09% → 52.63%，耗时 5s → 500ms | ❌ 未开源 |
| Qwen3-8B 定位模型 | 评论位置重定位 | 成功率 37.35% → 85.65%，耗时 3s → 1s | ❌ 未开源 |

**影响**：对外版本在误报拦截和位置准确性上依赖通用基模能力，效果可能不及内部版。但工程层面的三层递进式定位策略仍能保证 97%+ 的位置准确率。

**应对建议**：参考反思模型论文（arxiv.org/pdf/2602.20166v1）了解训练方法，有能力的团队可基于自有反馈数据训练专属模型。

## 与 Claude Code / Codex 的对比

基于 AACR-Bench 评测集（50 个热门开源仓库、200 个真实 PR、10 种编程语言、80+ 位资深工程师交叉标注）的客观对比：

| 方案 | 准确率 | 召回率 | F1 | 资源开销 | 适用场景 |
|------|--------|--------|-----|---------|---------|
| **Open Code Review** | 25-38% | 11.70-20.00% | 25.10%（最优） | 352K-743K token / 1-6 分钟 | 工程级稳定评审、CI/CD 集成 |
| **Claude Code** | 7-16% | 12.70-28.90% | 14.13%（最优） | 2,062K-5,664K token / 5-14 分钟 | 安全审计、最大化问题覆盖 |
| **Codex** | 27.82% | 4.92% | — | 525K token / 约 3 分钟 | 轻量级快速扫描 |

### 关键差异解读

1. **准确率维度**：OCR 以 25-38% 显著领先，意味着评审结果噪声更低，工程师处理效率更高。Claude Code 准确率仅 7-16%，但通过海量评论（如 Claude-4.6-Opus 产出 5980 条）弥补单条精度不足。

2. **召回率维度**：Claude Code 以 28.90% 位居榜首，比 OCR 最优组合多发现 134 个真实问题（增幅 45%）。对于"宁可多查、不可遗漏"的安全审计场景，召回率价值不可替代。

3. **F1 综合维度**：OCR 最优 F1（25.10%）约为 Claude Code 最优 F1（14.13%）的 1.78 倍，在准确率和召回率之间取得更均衡的表现。

4. **资源开销维度**：OCR 的 Token 消耗（352K-743K）仅为 Claude Code（2,062K-5,664K）的 1/6 到 1/8，耗时也短一半以上，大规模 CI/CD 集成时成本优势明显。

5. **模型代际现象**：Claude-4.8-Opus 比 Claude-4.6-Opus 表现出"更精确但更保守"——准确率最高（OCR 上 37.80%、CC 上 15.93%），但召回率明显下降（OCR 上 11.70% vs 20.00%、CC 上 12.70% vs 28.90%）。模型代际升级不必然带来评审效果全面提升。

## 适用场景说明

### 推荐使用 Open Code Review 的场景

- **工程级稳定评审**：追求高准确率、低噪声的团队代码评审
- **CI/CD 流水线集成**：Token 消耗可控，适合大规模自动化评审
- **大型仓库评审**：分治策略 + 上下文隔离，变更规模翻倍时 Token 仅线性增长
- **多语言混合项目**：13 套语言专属规则开箱即用
- **需要可观测性**：CLI 形态 + OpenTelemetry 支持，执行轨迹可复现、可调试

### 推荐配合 Claude Code 使用的场景

- **安全审计**：最大化召回率，避免遗漏安全漏洞
- **深度代码审查**：需要跨文件深度推理的复杂缺陷检测
- **上线前全面扫描**：对遗漏风险容忍度低的场景

### 推荐使用 Codex 的场景

- **轻量级快速扫描**：3 分钟内完成评审，Token 消耗适中
- **初步代码质量摸底**：准确率 27.82% 尚可，但召回率仅 4.92%，不适合深度评审
