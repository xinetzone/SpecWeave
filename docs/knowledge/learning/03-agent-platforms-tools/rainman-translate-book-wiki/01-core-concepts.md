# 核心功能详解

Rainman Translate Book 将整书翻译做成了一条完整的流水线。用户扔进去一本 PDF、DOCX 或 EPUB，Claude Code 先调用 Calibre 把书转成 HTMLZ，再拆成 Markdown 小块（每块约六千字符），然后同时启动 8 个 Claude 子代理并行翻译。翻译完成后验证哈希确保没有丢块或错位，接着 Pandoc 合并，Calibre 再输出成各种格式。

本章详细解析五大核心功能的技术原理与实际价值。

---

## 1. 并行子代理翻译

### 技术原理

Rainman Translate Book 默认同时启动 **8 个 Claude 子代理（Sub-agent）**，每个子代理独享一个独立的上下文窗口。一本书被拆成 N 个块（每块约 6000 字符），8 个子代理同时工作，每个子代理只负责翻译一块内容——读取、翻译、写输出，互不干扰。

### 为什么独立上下文窗口是好事

传统思路是把整本书塞进一个上下文窗口，但大语言模型的上下文窗口终究有限，且窗口越长，模型越容易出现"注意力稀释"——翻译到后面时，容易忘记前面的术语约定，导致越翻译越混乱。

Rainman Translate Book 反其道而行之：每个子代理只带一个独立的小上下文窗口，翻译第 42 块的时候，它不会记得第 1 块说了什么。这**反倒成了好事**——上下文不会污染，不会出现越翻译越混乱的情况。

### 实际价值

- **速度提升**：8 个代理并行翻译，比单会话翻译快得多。一本拆成 100 块的书，理论速度可提升接近 8 倍
- **质量可控**：每个小块独立翻译，错误不会在上下文窗口内累积和扩散
- **资源灵活**：可根据需要调整子代理数量，平衡速度与成本

## 2. 术语表锁定（Glossary）

### 问题场景

一本书里同一个专有名词出现一百次，8 个子代理各自翻译，如果不加约束，可能翻出八个版本。比如 "Manhattan" 可能被翻译成"曼哈顿""曼哈坦""曼赫顿"——这会导致全书术语不一致，阅读体验极差。

### 技术原理

Rainman Translate Book 的术语表锁定机制分三步走：

**第一步：采样提取。** 从全书中均匀采样 5 个块（开头 1 块、结尾 1 块、中间均匀分布 3 块），让 Claude 提取其中的专有名词和领域术语。

**第二步：生成 glossary.json。** 基于采样结果，生成一个 `glossary.json` 文件，记录每个术语的：
- 原文（original）
- 别名（aliases）
- 译文（translation）
- 类别（category，如人名、地名、技术术语等）
- 置信度（confidence）

**第三步：硬约束注入。** 每个子代理翻译之前，系统检查当前块中出现了哪些 glossary 里的术语，把对应的术语表作为硬约束注入到子代理的翻译指令中。例如 "Manhattan" 在 glossary 里锁定为"曼哈顿"，那所有子代理翻译到这个词的时候都必须写"曼哈顿"，没有商量余地。

### 增量重译

术语表锁定后的另一个关键能力是**增量重译**。如果翻译完成后发现某个术语翻错了，只需手动修改 `glossary.json` 中对应的条目，重新运行翻译命令。系统会自动比对哪些块的翻译中使用了被修改的术语，**只重新翻译受影响的块**，不会全书重跑。这大大降低了修正成本。

### 实际价值

- **术语一致性**：全书统一的术语翻译标准，媲美人工翻译的术语管理
- **低修正成本**：术语错误只需修改一处，增量重译而非全量重跑
- **可迭代优化**：glossary.json 可反复修正，每次只重译受影响块

## 3. 相邻上下文

### 问题场景

子代理只翻译当前块，但人名和代词经常跨段落指代。例如第 42 块结尾提到了"他"，第 43 块开头继续说"他做了什么"，如果第 43 块的子代理完全不知道第 42 块结尾的内容，这个"他"可能被错误翻译成"它"或在找不到指代对象时产生歧义。

### 技术原理

Rainman Translate Book 的解决方案是：每个子代理翻译时，除了当前块内容，还会收到：

- **前一块结尾约 300 字符的只读摘要**（了解上文）
- **下一块开头约 300 字符的只读摘要**（了解下文）

这些相邻上下文是**只读**的——子代理可以阅读它们以理解代词指代和前后文关系，但不能修改相邻块的内容。这保证了：
- 子代理能正确理解跨块的人名指代、代词回指
- 相邻块之间不会产生内容冲突
- 不会因为上下文信息过多而影响翻译质量

### 实际价值

- **跨段落连贯性**：解决了分块翻译最大的痛点——上下文断裂
- **代词准确翻译**：大幅减少人称代词翻译错误
- **轻量设计**：仅 300 字符的上下文窗口，不会显著增加 token 消耗

## 4. 断点续传

### 问题场景

翻译一本厚书可能需要较长时间，过程中可能因为网络中断、API 限流、手动停止等原因导致翻译中断。如果每次中断都要从头开始，时间和费用成本都会大幅增加。

### 技术原理

Rainman Translate Book 通过两个文件实现断点续传：

- **`manifest.json`**：记录每块原文的 **SHA-256 哈希值**
- **`run_state.json`**：记录每个块的翻译状态（已完成/未完成/已变更）

当翻译中断后重新运行同一个命令时，系统会：

1. 重新计算每块原文的 SHA-256 哈希
2. 与 `manifest.json` 中的记录比对
3. 识别出以下几种情况的块：
   - **缺失的块**：原文存在但输出文件不存在 → 需要翻译
   - **发生变化的块**：原文哈希与记录不符 → 需要重新翻译
   - **已完成的块**：原文哈希与输出文件都匹配 → 跳过
4. 只翻译缺失或发生变化的块，从断点继续

### 实际价值

- **节省时间和费用**：避免重复翻译已完成的内容
- **容错性强**：中断后无需人工干预，重新运行即可自动恢复
- **支持增量更新**：修改原文后，只重新翻译变更部分

## 5. 多格式输出

### 技术原理

翻译完成后，当前目录下会生成一个 `{book_name}_temp` 文件夹，包含以下输出：

| 输出文件 | 格式 | 说明 |
|---|---|---|
| `output.md` | Markdown | 合并后的完整 Markdown 源文件 |
| `book.html` | HTML | 带浮动目录的网页版，适合在线阅读 |
| `book.docx` | DOCX | Word 文档，适合编辑和排版 |
| `book.epub` | EPUB | 电子书格式，适合 Kindle 等阅读器 |
| `book.pdf` | PDF | 打印版 PDF，适合纸质阅读 |

### 技术实现

格式转换流程如下：

1. 所有翻译完成的 Markdown 块通过 **Pandoc** 合并为完整的 `output.md`
2. 从 `output.md` 出发，Pandoc 输出 HTML 和 DOCX
3. Calibre 的 `ebook-convert` 命令将 HTML 输出为 EPUB 和 PDF

### 实际价值

- **一站式输出**：一次翻译，五种格式，覆盖所有主流阅读场景
- **可再编辑**：DOCX 格式支持后续人工校对和排版
- **阅读友好**：HTML 版带浮动目录，EPUB 支持移动端阅读，PDF 适合打印

---

## 核心架构总结

Rainman Translate Book 的流水线架构可以概括为以下六个阶段：

```
输入 → 转换 → 分块 → 并行翻译 → 验证 → 合并 → 输出
PDF      Calibre   6000字符   8个子代理    SHA-256    Pandoc    HTML/DOCX
DOCX     转HTMLZ   每块        +术语表      哈希比对   合并      EPUB/PDF
EPUB                        +相邻上下文
```

这个架构的核心思想是：**把大问题拆成小问题，并行处理，再用全局约束保证一致性。** 这一设计模式不仅在翻译领域有效，在任何需要处理"大规模 + 一致性要求高"的 AI 任务中都有参考价值。
