# 第四章 - 常见问题解答（FAQ）

本章汇总MonkeyCode部署、使用、配置过程中的常见问题与解决方案，分为五大类，方便快速定位和解决问题。

---

## 一、部署相关问题

### Q1: MonkeyCode的最低系统要求是什么？配置不够会有什么问题？

**问题描述**：不确定服务器配置是否满足要求，担心部署后运行不流畅。

**原因分析**：MonkeyCode采用控制台与开发环境分离架构，不同组件对资源的要求不同。配置不足会导致容器启动失败、响应缓慢、任务执行超时等问题。

**解决方案**：

| 组件 | 最低配置 | 推荐配置 | 配置不足的表现 |
|------|---------|---------|--------------|
| 控制台 | 2核CPU / 4GB内存 / 40GB磁盘 | 4核CPU / 8GB内存 / 80GB磁盘 | 页面加载慢、用户管理卡顿 |
| 开发环境 | 8核CPU / 16GB内存 / 100GB磁盘 | 16核CPU / 32GB内存 / 200GB磁盘 | 代码生成慢、预览无法启动、频繁OOM |

> **提示**：如果要在开发环境本地运行7B模型，建议额外配置8GB+显存的GPU；运行70B模型需要80GB+显存。如果没有GPU，建议使用云端模型API。

### Q2: Docker安装失败或Docker Compose命令找不到怎么办？

**问题描述**：执行安装脚本时提示Docker未安装或Docker Compose版本过低。

**原因分析**：
1. 系统未安装Docker或Docker版本低于20.10
2. Docker Compose未安装或版本低于2.0
3. Docker服务未启动
4. 当前用户没有Docker执行权限

**解决方案**：

1. **重新安装Docker（推荐使用官方镜像脚本）**：
```bash
# Ubuntu/Debian 系统
curl -fsSL https://get.docker.com | bash -s docker --mirror Aliyun
systemctl enable --now docker
```

2. **验证安装**：
```bash
docker --version          # 应显示 ≥ 20.10
docker compose version    # 应显示 ≥ 2.0
docker info               # 应显示Docker正常运行
```

3. **添加用户到docker组（避免每次sudo）**：
```bash
usermod -aG docker $USER
newgrp docker  # 立即生效
```

### Q3: 启动时提示端口被占用怎么办？

**问题描述**：执行安装后容器无法启动，查看日志提示"port is already allocated"或"address already in use"。

**原因分析**：默认的80端口（或配置的其他端口）已被其他服务（如Nginx、Apache、其他Web应用）占用。

**解决方案**：

1. **查找占用端口的进程**：
```bash
# 查看80端口占用情况
netstat -tlnp | grep :80
# 或
lsof -i :80
```

2. **方案一：停止占用端口的服务**：
```bash
# 例如停止Nginx
systemctl stop nginx
systemctl disable nginx
```

3. **方案二：修改MonkeyCode使用的端口**（推荐）：
   - 安装时选择自定义端口，或
   - 修改安装目录下的 `.env` 文件，调整 `PORT` 配置项
   - 重启容器：`docker compose down && docker compose up -d`

4. **验证端口是否可用**：
```bash
# 重启后检查
docker ps  # 确认容器端口映射正确
curl http://localhost:<新端口>  # 应能访问
```

### Q4: 安装过程中提示磁盘空间不足怎么办？

**问题描述**：拉取镜像或启动容器时提示"no space left on device"。

**原因分析**：
1. Docker镜像存储目录所在分区空间不足
2. 未清理旧的Docker镜像和容器
3. 系统盘预留空间不足

**解决方案**：

1. **清理Docker无用资源**：
```bash
# 清理停止的容器、未使用的网络、悬空镜像
docker system prune -f

# 更彻底的清理（包括未使用的镜像，谨慎使用）
docker system prune -a -f
```

2. **查看Docker磁盘使用情况**：
```bash
docker system df
```

3. **将Docker数据目录迁移到大容量磁盘**（如果系统盘空间紧张）：
```bash
# 停止Docker服务
systemctl stop docker

# 迁移数据目录（假设目标目录为 /data/docker）
rsync -avz /var/lib/docker/ /data/docker/

# 修改Docker配置
echo '{"data-root": "/data/docker"}' > /etc/docker/daemon.json

# 重启Docker
systemctl start docker
```

## 二、使用相关问题

### Q5: 创建项目时开发环境初始化一直卡在"启动中"怎么办？

**问题描述**：点击创建项目后，等待很久开发环境都无法就绪，一直显示启动中。

**原因分析**：
1. 开发环境镜像拉取失败（网络问题）
2. 服务器资源不足，无法分配足够的CPU/内存
3. 存储驱动问题导致容器无法正常启动
4. Docker网络配置问题

**解决方案**：

1. **检查开发环境容器状态**：
```bash
# 查看所有容器状态
docker ps -a

# 查看异常退出容器的日志
docker logs <workspace容器ID>
```

2. **检查服务器资源使用情况**：
```bash
# 查看CPU、内存、磁盘使用
top
df -h
free -h
```

3. **如果是内存不足**：
   - 关闭其他占用内存的服务
   - 增加服务器内存，或
   - 调整开发环境的内存限制（在系统设置中）

4. **重试创建项目**：
   - 先删除卡住的项目
   - 确认资源充足后重新创建
   - 如果多次失败，尝试重启Docker服务：`systemctl restart docker`

### Q6: AI任务拆分不合理怎么办？拆分得太粗或太细？

**问题描述**：提交需求后，AI自动拆分的任务要么太大（一个任务包含太多功能），要么太碎（拆分出很多不必要的小任务）。

**原因分析**：
1. 需求描述不够清晰具体，AI理解有偏差
2. 需求过于复杂，AI难以把握合适的拆分粒度
3. 没有提供参考的任务拆分方式

**解决方案**：

1. **优化需求描述**：
   - 明确列出要实现的功能点，用序号分点
   - 说明功能之间的依赖关系
   - 给出参考项目或示例（如果有）

2. **手动调整任务拆分**：
   - **拆分过粗**：点击任务进入详情，使用"拆分任务"功能进一步细化
   - **拆分过细**：选中多个小任务，使用"合并任务"功能整合
   - **遗漏任务**：点击"添加任务"手动补充
   - **多余任务**：直接删除不需要的任务

3. **调整任务优先级和依赖**：
   - 拖拽任务调整执行顺序
   - 设置任务依赖关系，确保按正确顺序执行

> **最佳实践**：先给AI一个粗略的拆分指示，如"请按用户管理、内容管理、数据统计三个模块拆分任务"，AI会按照你的框架来拆分。

### Q7: AI生成的代码质量不高，有错误或不符合预期怎么办？

**问题描述**：AI生成的代码存在bug、逻辑错误、风格不统一，或者没有按照需求实现。

**原因分析**：
1. 模型能力限制（小模型代码能力较弱）
2. 需求描述不够具体，缺少细节约束
3. 没有给AI提供足够的上下文（如现有代码结构、编码规范）
4. 没有及时反馈，AI沿着错误方向继续生成

**解决方案**：

1. **选择更合适的模型**：
   - 代码生成推荐使用DeepSeek-Coder、Claude 3 Opus/Sonnet、GPT-4等代码能力强的模型
   - 本地7B模型适合简单任务，复杂任务建议用云端大模型

2. **优化提示方式**：
   - 明确说明技术栈："使用React 18 + TypeScript + Tailwind CSS"
   - 指定编码规范："遵循ESLint标准，使用函数组件和Hooks"
   - 给出示例：参考现有代码的风格，告诉AI"按照xxx文件的风格来写"
   - 分步骤实现：复杂功能先让AI写设计方案，确认后再写代码

3. **及时反馈和纠正**：
   - 发现问题立即在对话框指出，不要等全部生成完
   - 具体说明问题："这个函数的参数校验有问题，缺少对空值的处理"
   - 提供修正方向："应该在第xx行添加xxx判断"

4. **使用代码审查功能**：
   - 完成一个模块后，让AI自己审查一遍代码
   - 提示词："请审查刚才生成的代码，检查是否有bug、安全漏洞、性能问题，列出问题清单并修正"

### Q8: 预览功能无法打开或页面显示异常怎么办？

**问题描述**：点击预览按钮后，预览窗口打不开、显示404、或页面加载不完整样式错乱。

**原因分析**：
1. 项目服务未正常启动（端口未监听、启动报错）
2. 预览端口映射配置问题
3. 前端项目未正确构建
4. 跨域或防火墙问题

**解决方案**：

1. **检查终端日志**：
   - 打开项目的终端面板
   - 查看服务启动日志，是否有报错信息
   - 确认服务监听的端口号（如3000、5173、8080等）

2. **常见启动问题处理**：
   - **依赖未安装**：在终端执行 `npm install` 或 `pip install -r requirements.txt`
   - **端口被占用**：修改项目配置的端口，或杀掉占用进程
   - **启动命令错误**：检查package.json中的scripts，手动执行正确的启动命令

3. **手动配置预览端口**：
   - 在预览设置中手动填写服务实际监听的端口
   - 确认端口映射已正确配置

4. **检查构建产物**：
   - 如果是需要构建的项目，先执行 `npm run build`
   - 确认dist/build目录生成正确
   - 使用生产模式启动预览

## 三、模型配置相关问题

### Q9: 配置云端模型时测试连接失败怎么办？

**问题描述**：填写API Key后点击"测试连接"，提示连接失败或认证错误。

**原因分析**：
1. API Key填写错误或已过期
2. API Endpoint地址不正确
3. 服务器无法访问模型服务商API（网络问题、防火墙限制）
4. 模型服务商账号欠费或被限制
5. 未配置代理（内网环境）

**解决方案**：

1. **检查API Key**：
   - 确认复制的Key完整，没有多余空格
   - 登录模型服务商控制台，确认Key状态正常
   - 重新生成一个新的Key试试

2. **检查API Endpoint**：
   - 不要随意修改默认的Endpoint地址
   - 确认服务商的API地址是否有更新（查阅官方文档）
   - 确认地址格式正确（以http/https开头，不要有多余路径）

3. **测试网络连通性**：
```bash
# 测试能否访问API地址（以Kimi为例）
curl -I https://api.moonshot.cn
# 如果超时，说明网络不通
```

4. **内网环境配置代理**：
   - 在系统设置→网络配置中配置HTTP/HTTPS代理
   - 或配置Docker服务的代理
   - 联系网管确认API地址已加入白名单

5. **验证模型权限**：
   - 确认账号已开通对应模型的访问权限
   - 检查账号余额是否充足
   - 查看是否有IP白名单限制

### Q10: 本地模型（Ollama/vLLM）无法连接或调用失败怎么办？

**问题描述**：配置本地模型后测试连接失败，或调用时提示超时、连接被拒绝。

**原因分析**：
1. Ollama/vLLM服务未启动
2. 服务监听在127.0.0.1，MonkeyCode容器无法访问
3. 防火墙阻止了端口访问
4. 模型名称填写错误
5. 模型未正确拉取/加载

**解决方案**：

1. **确认本地模型服务正常运行**：
```bash
# 检查Ollama状态
systemctl status ollama

# 本地测试API（在运行Ollama的机器上执行）
curl http://localhost:11434/api/tags
# 应返回已拉取的模型列表
```

2. **配置Ollama允许外部访问**（关键！）：
```bash
# 编辑Ollama服务配置
systemctl edit ollama.service

# 添加以下内容：
[Service]
Environment="OLLAMA_HOST=0.0.0.0"

# 重启Ollama服务
systemctl daemon-reload
systemctl restart ollama

# 验证监听地址
netstat -tlnp | grep 11434
# 应显示 0.0.0.0:11434 而不是 127.0.0.1:11434
```

3. **检查防火墙配置**：
```bash
# 开放11434端口（如果使用firewalld）
firewall-cmd --add-port=11434/tcp --permanent
firewall-cmd --reload

# 如果是云服务器，记得在安全组中放行端口
```

4. **正确配置MonkeyCode模型参数**：
   - API Endpoint：`http://<Ollama服务器IP>:11434/v1`（注意要加/v1后缀）
   - API Key：填写任意字符串即可（如"ollama"）
   - 模型标识：必须和 `ollama list` 显示的名称完全一致（如 `qwen2:7b`，不是"Qwen2 7B"）

5. **确认模型已成功拉取**：
```bash
ollama list  # 查看已下载的模型
ollama pull qwen2:7b  # 如果没拉取，执行此命令
```

### Q11: 如何在不同项目中切换模型？默认模型怎么改？

**问题描述**：想要不同项目使用不同模型，或者想更换系统默认模型。

**原因分析**：MonkeyCode支持多模型配置，模型可以在系统级别和项目级别分别设置。

**解决方案**：

1. **修改系统默认模型**：
   - 进入「系统设置」→「模型管理」
   - 在模型列表中，点击想要设为默认的模型的「设为默认」按钮
   - 新建项目时会默认选择这个模型

2. **为已有项目切换模型**：
   - 进入项目页面
   - 点击右上角「项目设置」
   - 在「AI模型」下拉菜单中选择想要切换的模型
   - 点击「保存」，新任务会使用新模型

3. **任务级别临时切换模型**：
   - 在执行具体任务时，可以在对话框顶部选择当前对话使用的模型
   - 不会影响项目的默认模型设置

> **提示**：建议同时配置多个模型，日常编码用快速便宜的模型，复杂架构设计和疑难问题用能力强的大模型，灵活切换平衡成本和效果。

## 四、安全相关问题

### Q12: 使用MonkeyCode会导致代码泄露吗？数据隐私如何保障？

**问题描述**：担心代码上传到AI服务导致商业机密泄露，不确定私有化部署的数据安全情况。

**原因分析**：这是企业用户最关心的问题。数据安全取决于部署方式和模型选择。

**解决方案**：

MonkeyCode提供多层级的数据安全保障，你可以根据安全需求选择：

1. **私有化部署（最安全）**：
   - 整个系统部署在企业内网，数据不流出企业网络
   - 代码、需求、对话记录全部存储在企业自有服务器
   - 配合本地模型（Ollama/vLLM），完全离线运行，数据零流出

2. **云端模型的安全措施**：
   - 选择支持"不训练"承诺的模型服务商（如OpenAI、Anthropic、DeepSeek等都支持API数据不用于训练）
   - 在模型服务商控制台开启数据不用于训练的选项
   - 配置API Key的IP白名单，限制只有MonkeyCode服务器可调用
   - 敏感项目建议使用本地模型

3. **系统层面的安全保障**：
   - 所有用户密码bcrypt加密存储
   - API Key加密存储，不明文展示
   - 支持HTTPS加密传输
   - 操作日志完整审计
   - 支持对接企业SSO统一认证

> **安全建议**：核心业务代码和敏感项目，务必使用私有化部署+本地模型；非敏感项目可以使用云端模型提高效率。

### Q13: 如何配置细粒度的团队成员权限？

**问题描述**：团队中有不同角色（开发、测试、产品、外包），需要控制他们能访问和操作的范围。

**原因分析**：MonkeyCode提供基于角色的访问控制（RBAC），可以灵活配置权限。

**解决方案**：

1. **内置角色**：
   - **系统管理员**：拥有所有权限，可管理系统配置、用户、所有项目
   - **项目管理员**：可创建项目、管理自己负责的项目成员和设置
   - **开发者**：可参与项目、使用AI编码、提交代码，但不能修改系统设置
   - **观察者**：只能查看项目进度和内容，不能编辑或执行任务

2. **自定义角色配置**：
   - 进入「系统设置」→「角色管理」
   - 点击「新建角色」，按需勾选权限点：
     - 项目权限：查看、创建、编辑、删除、导出
     - 成员管理：添加、移除、修改角色
     - 代码权限：查看、编辑、提交、推送Git
     - AI功能：对话、任务执行、模型选择
   - 将自定义角色分配给成员

3. **项目级别权限控制**：
   - 项目默认是私有，只有项目成员可见
   - 在项目成员管理中，可单独设置每个成员在该项目中的角色
   - 支持设置项目为公开（所有登录用户可见）

### Q14: 审计日志在哪里查看？可以记录哪些操作？

**问题描述**：需要追溯用户操作记录，满足安全合规要求。

**原因分析**：MonkeyCode内置完善的审计日志功能，记录关键操作。

**解决方案**：

1. **查看审计日志**：
   - 管理员进入「系统设置」→「审计日志」
   - 可按时间范围、用户、操作类型筛选
   - 支持导出日志为CSV/JSON格式

2. **记录的操作类型包括**：
   - 用户操作：登录、登出、密码修改、权限变更
   - 项目操作：创建、删除、修改设置、成员变更
   - 代码操作：文件创建、修改、删除、Git推送
   - AI操作：模型调用、任务执行、代码生成
   - 系统操作：配置修改、模型增删、备份恢复

3. **日志留存策略**：
   - 默认留存180天
   - 可在系统设置中配置留存时长（30天/90天/180天/365天/永久）
   - 支持配置日志外发到企业ELK/SIEM系统

## 五、故障排查问题

### Q15: 重启服务器后MonkeyCode服务无法启动怎么办？

**问题描述**：服务器重启后，访问MonkeyCode页面无法打开，容器没有自动启动。

**原因分析**：
1. Docker服务没有设置开机自启
2. 容器没有设置restart策略
3. 数据磁盘挂载失败
4. 端口被其他启动的服务占用

**解决方案**：

1. **检查Docker服务状态**：
```bash
systemctl status docker
# 如果未启动，执行：
systemctl enable --now docker
```

2. **启动MonkeyCode容器**：
```bash
# 进入MonkeyCode安装目录
cd /opt/monkeycode  # 默认安装目录，如果你自定义了位置请对应修改

# 启动所有服务
docker compose up -d

# 检查容器状态
docker ps
# 所有容器状态应为Up
```

3. **设置容器开机自启**（确保下次重启自动恢复）：
```bash
# 检查compose文件中是否配置了restart
# 如果没有，在docker-compose.yml中为每个服务添加：
# restart: always

# 然后重新应用配置
docker compose up -d
```

4. **排查启动失败原因**：
   - 如果容器不断重启，查看日志：`docker logs <容器ID> --tail 100`
   - 检查磁盘是否挂载正常：`df -h`
   - 检查端口占用：`netstat -tlnp`

### Q16: 使用过程中系统很卡、响应很慢怎么办？

**问题描述**：页面加载慢、AI响应时间长、代码生成等待很久，整体感觉卡顿。

**原因分析**：
1. 服务器资源不足（CPU/内存占用过高）
2. 同时运行的任务太多，资源争抢
3. 网络延迟高（特别是使用云端模型时）
4. 磁盘IO性能差
5. 日志或临时文件占用大量空间

**解决方案**：

1. **检查资源使用情况**：
```bash
# 实时查看资源占用
top
# 按M排序看内存占用，按P排序看CPU占用

# 查看Docker容器资源使用
docker stats
```

2. **资源瓶颈处理**：
   - **CPU/内存不足**：关闭不必要的项目和开发环境，停止空闲的工作区
   - **磁盘IO高**：检查是否有大量日志写入，考虑使用SSD
   - **内存泄漏**：如果某个容器内存持续增长，尝试重启该容器

3. **优化任务并发**：
   - 进入「系统设置」→「任务配置」
   - 调整同时执行的任务数量（默认2-3个，配置低可设为1）
   - 空闲的开发环境会自动休眠（可配置休眠时间，默认30分钟）

4. **网络优化**：
   - 云端模型选择离你近的API节点
   - 配置合适的网络代理
   - 大文件上传/导出避开网络高峰

5. **定期清理维护**：
```bash
# 定期清理Docker无用资源
docker system prune -f

# 清理系统日志
journalctl --vacuum-size=500M
```

### Q17: 手机端无法访问或页面显示不正常怎么办？

**问题描述**：电脑可以正常访问，但手机浏览器打不开或布局错乱。

**原因分析**：
1. 手机和服务器不在同一网络（内网需要VPN）
2. 服务器防火墙或安全组没有放行手机所在网络的访问
3. 使用了IP地址访问但证书不匹配（HTTPS场景）
4. 手机浏览器兼容性问题

**解决方案**：

1. **检查网络连通性**：
   - 确保手机连接的WiFi可以访问服务器IP
   - 如果是外网访问，确认服务器有公网IP，且端口已映射
   - 内网环境需要先连接企业VPN
   - 在手机浏览器直接访问IP:端口，测试是否能通

2. **访问地址配置**：
   - 不要使用`localhost`或`127.0.0.1`访问（手机上这是手机自己）
   - 使用服务器的实际内网IP或域名
   - 在系统设置中正确配置"系统访问地址"为手机可访问的地址

3. **浏览器兼容性**：
   - 推荐使用Chrome、Safari、Edge等现代浏览器
   - 不要使用微信内置浏览器（可能有兼容性限制）
   - 清除浏览器缓存重试

4. **移动端使用建议**：
   - 移动端主要用于查看项目进度、补充需求、发送指令
   - 复杂代码编辑建议在电脑端操作
   - 将MonkeyCode地址添加到手机桌面书签，方便快速访问

## 快速排查清单

遇到问题时，按以下顺序快速排查：

1. **第一步：看容器状态**
   ```bash
   docker ps -a  # 所有容器是否都是Up状态
   ```

2. **第二步：看服务日志**
   ```bash
   docker logs <容器名> --tail 50  # 查看最近的错误日志
   ```

3. **第三步：检查资源**
   ```bash
   top  # CPU/内存是否满了
   df -h  # 磁盘是否满了
   ```

4. **第四步：检查网络**
   ```bash
   ping <API地址>  # 模型API能否访问
   netstat -tlnp  # 端口是否正常监听
   ```

如果以上步骤都无法解决问题，请联系技术支持，并提供：
- 问题截图
- 相关容器的日志
- 服务器配置信息（CPU/内存/操作系统版本）
- 问题复现步骤
