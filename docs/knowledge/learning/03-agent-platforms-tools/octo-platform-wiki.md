## 一、项目背景与定位

### 1.1 行业背景：Agent 数字劳动力爆发

Agent 等数字劳动力正迎来爆发式增长。当企业开始拥有成百上千个 Agent 时，传统的"单点对话式"协作模式已经无法承载复杂的工作场景。明略科技看到了这一趋势，试图将 Octo 平台打造成为 Private AI 时代的组织基础设施，构建人和 AI 协作的新范式。

> "当企业开始拥有成百上千 Agent 时，Octo 可以像管理互联网节点一样实现它们之间、它们与创建者之间以及创建者与创建者之间的高效连接、通信与协作。"

在这一趋势下，**每个 Agent 既各司其职，又相互协作**，这种工作模式在大多数场景中优于单一巨型模型。常见的工作场景被打包成现成的 Bot 模板，不用自己从头配，「领养」就能直接拉进群里干活。

### 1.2 Octo 平台定位：Private AI 时代组织基础设施

Octo 的定位不是做一个更聪明的聊天工具，而是**重写协作本身**。它从 IM 形态切入，但 IM 只是入口，不是核心。

| 定位维度 | 说明 |
|---------|------|
| **平台名称** | Octo（明略科技出品） |
| **核心定位** | Private AI 时代的组织基础设施 |
| **解决对象** | 多 Agent 协作的连接、通信与编排 |
| **设计哲学** | 人与 Agent 同等身份的消息主体 |
| **部署方式** | 私有化部署，支持开源开放 |
| **入口形态** | IM 交互（入口） + 背后协作结构（核心） |

### 1.3 传统 IM 工具的三大局限

现有的 Bot 助手大多挂在 Discord、Telegram、飞书、钉钉这些 IM 里，通过消息接收指令、执行任务。这种模式在复杂协作场景下暴露出三大局限：

**局限一：信息天然被滚动消息淹没**
- 今天讨论一个方案，明天又有新的消息刷屏
- 一周后想追溯只能在聊天记录里"大海捞针"
- 关键决策上下文难以沉淀、容易丢失

**局限二：协作过程难以追溯**
- 任务缘起、过程时间线、关键产出无法结构化保留
- 修改、打回、补充、确认等带判断的反馈散落各处
- 组织决策资产无法形成可复用的记忆

**局限三：协作拓扑失控**
- 群聊里所有人都能看到所有消息，缺乏信息隔离
- 复杂任务需要"该互见时互见，该互盲时互盲"，群聊做不到
- 多 Agent 间信息流转路径不清晰，容易产生从众或信息污染

### 1.4 Octo 解决的三个层次问题

Octo 将多 Agent 协作问题拆解为三个递进层次：

**第一层：连接（Connection）**
- 把人、Bot、Runtime Agent 和工具这些原本分散的节点连接起来
- 在底层通信协议里，人和 Agent 从一开始就被设计为同等身份的消息主体
- Bot 之间可以直接对话：一只搜集信息，一只分析，一只纠错，最后交人来品鉴

**第二层：干活（Execution）**
- 任务由人发起，由 Bot 调用 Runtime Agent 完成执行
- 执行过程不断被反馈，其他 Bot 接力
- 人在关键节点做判断和取舍
- 复杂任务可以整包交出去，Bot 负责拆解、调度、推进，并实时反馈进度

**第三层：沉淀（Accumulation）**
- 把每个任务沉淀成可追溯的"决策卡"（Matter）
- 人的判断信号被提炼成可复用的偏好（Taste）
- 组织决策资产在端侧可控环境下持续累积

> "人和 Agent 虽然在同一个 IM 界面沟通、下任务、接收结果，但真正发生变化的是背后的连接结构。"

## 二、O.C.T.O. 四维度框架

O.C.T.O. 是 Octo 平台能力的总纲，四个字母分别对应四个核心维度。这四个维度连在一起，构成了 Octo 所提供的完整能力。

### 2.1 四维度对比表

| 维度 | 全称 | 核心含义 | 关键能力 | 价值落点 |
|------|------|---------|---------|---------|
| **O** | Open | 开放生态 | 不同 Runtime 的 Agent 以 Bot 身份接入，获得统一身份 | 避免厂商锁定，生态可扩展 |
| **C** | Context | 共享上下文 | IM 讨论收敛为结构化知识，项目上下文跨 Agent 共享 | 任务过程持续可追溯 |
| **T** | Taste | 偏好进化 | 实战反馈沉淀为偏好，主人品味被结构化留存与调用 | Agent 越用越懂团队 |
| **O** | Orchestration | 多 Agent 编排 | 六种协作模式对应六种信息拓扑，不同 Bot 带不同偏好协作 | 复杂任务合力完成 |

### 2.2 Open：开放生态

Open 代表**开放生态**。不同 Runtime 的 Agent，包括：

- **OpenClaw**
- **Codex**
- **Claude Code**
- **Cursor**
- 其他兼容 Runtime

都能够以 **Bot 身份**接入 Octo，获得统一身份。这意味着企业不需要绑定到某一个特定的 Agent 框架，可以将多种异构 Agent 统一纳入协作体系。

### 2.3 Context：共享上下文

Context 代表**共享上下文**。在 Octo 中：

- IM 中的讨论会收敛为**结构化知识**
- 项目上下文在不同 Agent 之间**共享**
- 任务过程可以被**持续追溯**

这一层的价值在于打破"信息孤岛"——不同 Bot、不同任务、不同时间点的上下文不再散落各处，而是被组织成可调用、可追溯的结构化资产。

### 2.4 Taste：偏好进化

Taste 代表**偏好进化**。这是 Octo 最具特色的设计之一：

- 实战反馈沉淀为偏好
- 每个 Agent 背后主人的品味和判断方式被结构化留存与调用
- 偏好在 Matter 中生长，下次遇到类似任务自动进入上下文

> "Octo 所追求的 Taste，也正是在这个位置生长出来。"

### 2.5 Orchestration：多 Agent 编排

Orchestration 代表**多 Agent 编排**：

- 六种协作模式对应六种信息拓扑
- 不同 Bot 带着不同偏好参与同一任务
- 合力完成复杂工作

Orchestration 不是简单地把多个 Bot 拉到同一个地方，而是规定**信息沿正确路径流动**。

### 2.6 Matter：四维度的共同基座

承载起 Context、Taste、Orchestration 的共同基座正是 **Matter**：

```
        Open（开放生态）
            │
            ▼
    ┌───────────────┐
    │    Matter     │  ← 共同基座
    │  （事项容器）  │
    └───────────────┘
       │   │   │
       ▼   ▼   ▼
   Context Taste Orchestration
```

- 没有 Matter，**Context** 会散落在聊天记录里
- 没有 Matter，**Taste** 会缺少来自真实任务的反馈来源
- 没有 Matter，**多 Agent 编排**也很难留下可追溯的过程和结果

Matter 是复杂任务得以被理解、反馈、校准和编排的核心容器。

## 三、Matter（事项）：复杂任务的承载单元

### 3.1 普通 IM 的信息淹没问题

复杂、长程任务需要回答三个核心问题：

1. **事情怎么干成？**（执行路径）
2. **事情怎么干对？**（判断校准）
3. **事情怎么留得下？**（组织记忆）

在普通 IM 里，这三个问题都难以解决：

> "在普通 IM 里，信息天然会被滚动消息淹没。今天讨论一个方案，明天又有新的消息刷屏，一周后想追溯只能在聊天记录里大海捞针。"

### 3.2 Matter "决策卡"设计

Matter 把每个任务沉淀成一张**可追溯的「决策卡」**。它不只记录最终结果，还包括：

- 任务缘起（Brief）
- 过程时间线（Timeline）
- 关键产出
- 人的反馈
- 验收结论

一个事项从 **Brief 开始**，沿着 **Timeline 展开**，中间有产出、有打回、有补充、有确认，最后形成**可以回看的组织记忆**。

### 3.3 Matter 五要素

| 要素 | 含义 | 价值 |
|------|------|------|
| **Brief** | 任务缘起，描述为什么要做这件事 | 保留任务初衷与上下文 |
| **Timeline** | 过程时间线，记录关键节点 | 还原执行路径，便于复盘 |
| **产出** | 关键交付物（文档、代码、报告等） | 沉淀工作成果 |
| **反馈** | 人的修改、打回、圈注、确认 | 携带人的判断，是 Taste 的原料 |
| **验收** | 最终结论与归档 | 形成组织记忆，可回看可复用 |

### 3.4 组织决策资产价值

Matter 的真正价值在于：很多价值并不单单存在于最终文档里。

> "为什么选择一个方案，哪些判断来自业务负责人，哪些修改来自法务、销售或技术同学，这些信息共同构成了组织的决策资产。"

复杂任务里的每一次修改、打回和验收，本身都带着人的判断。一旦这类反馈进入到 Matter，它们便：

- 从一次性的沟通记录，**转变成了 Agent 学习组织偏好的原材料**
- 成为可被检索、可被复用的组织记忆
- 让"为什么这么决策"这件事变得可追溯

## 四、Taste（品味）：实战沉淀的偏好进化

### 4.1 当前 Agent 自我成长有限的现状

今天的很多 Agent 都有自己的：

- 配置文件
- 工具说明
- 角色设定

但它们的**自我成长仍然有限**。一个团队喜欢什么样的风格，什么样的结论才算有洞察，这些偏好很难靠一次系统提示写清楚。

> "很多时候，人类的判断本来就是隐性的——负责人说'这个感觉不对'，客户说'这个角度不准'，这些反馈背后的经验、品味和行业语境不一定马上就能写成一套规则。"

### 4.2 "偏好对齐必须在实战中完成"设计思路

基于上述现状，Octo 采取了明确的设计思路：

> "偏好对齐必须在实战中完成"

这意味着 Taste 不是预配置出来的，而是在真实任务中**通过反馈循环**逐步建立起来的：

1. **人发起任务** → Bot 调用 Runtime Agent 执行
2. **人给出反馈** → 打回、圈一笔、修改、确认
3. **反馈沉淀到 Matter** → 携带判断信号的原始素材
4. **提炼成偏好** → 下次可复用
5. **偏好自动进入上下文** → 遇到类似任务时自动应用

### 4.3 隐性判断向可复用偏好的沉淀过程

人的反馈背后往往藏着隐性判断：

| 反馈信号 | 可能的隐性判断 |
|---------|--------------|
| 一次方案退回 | 逻辑不够收束 |
| 一次报告重写 | 结论缺少业务视角 |
| "这个感觉不对" | 风格不符合团队调性 |
| "这个角度不准" | 缺少行业语境 |

这些信号在沉淀到 Matter 之后，就有机会被**提炼成下一次可复用的偏好**。这是一个把"隐性判断"逐步转化为"显性偏好"的过程。

### 4.4 Taste 让 Agent 越用越懂团队

Taste 机制的核心效果是**让 Agent 越用越懂团队**：

- 下次遇到类似任务，相关偏好会**自动进入上下文**
- Bot 会在一次次实战中**更接近团队的做事方式**
- 每个 Agent 背后主人的品味和判断方式被**结构化留存与调用**

Taste 让 Agent 从"通用工具"逐步进化为"懂团队的协作伙伴"。

## 五、六种协作模式：六种信息拓扑

多个 Agent 一起协作，并不等于"多叫几个 Bot 进群"。更细化的问题决定了执行效果：

- 信息怎么传？
- 谁负责生成，谁负责验证？
- 哪些任务需要独立视角，哪些任务需要公开讨论？
- 哪些步骤必须按顺序进行，哪些任务可以分头推进？

Octo 将复杂协作拆成了**六种模式**，对应六种信息拓扑。

### 5.1 六种协作模式对比表

| 模式 | 中文名 | 适用场景 | 信息流转 | 参与者可见性 | 典型用例 |
|------|--------|---------|---------|-------------|---------|
| **Solo** | 单干 | 简单明确的任务 | 领队独自完成，无信息流转 | 仅领队可见 | 简单查询、单点执行 |
| **Roundtable** | 圆桌讨论 | 需要形成共识、碰撞观点 | 领队主持，多 Agent 围绕同一议题公开讨论 | 参与者互相可见 | 方案讨论、观点碰撞、结论收束 |
| **Critic** | 生成—验证 | 需要独立审查 | 生成方产出 → 验证方审核（有否决权） | 生成方与验证方必须不同 | 代码检查、事实核查、方案质检 |
| **Pipeline** | 流水线 | 存在明确顺序依赖 | A → B → C 严格串行，每步产出作为下步输入 | 按顺序传递 | 调研→分析→写作→校对 |
| **Split** | 分头干 | 大任务分治 | 领队拆分子块 → 多 Agent 各自处理 → 领队合并 | 子块之间互不可见 | 行业报告拆成政策/市场/技术/案例 |
| **Swarm** | 撒网竞选 | 需要多解并行、避免从众 | 同一任务交多 Agent 独立完成 → 领队择优 | 参与者彼此互盲 | 标题创意、方案创意、产品命名、不同分析路径 |

### 5.2 六种模式详解

#### Solo（单干模式）
- **特点**：由领队独自完成
- **适用**：简单明确的任务
- **信息拓扑**：无协作，单点执行

#### Roundtable（圆桌讨论）
- **特点**：在领队主持下，多个 Agent 围绕同一议题展开公开讨论
- **可见性**：参与者互相可见
- **适用**：需要形成共识、碰撞观点、收束结论的任务

#### Critic（生成—验证模式）
- **特点**：一个 Agent 负责生成，另一个 Agent 负责审核
- **关键约束**：生成方和验证方**必须不同**
- **权限**：验证方有**否决权**，发现问题可以打回重做
- **适用**：需要独立审查的场景，比如代码检查、事实核查、方案质检

#### Pipeline（流水线模式）
- **特点**：从 A 到 B 到 C 严格串行
- **信息流转**：每一步的产出作为下一步的输入
- **适用**：存在明确顺序依赖的任务，比如先调研，再分析，再写作，再校对

#### Split（分头干模式）
- **特点**：领队把任务拆成**互不可见**的子块
- **信息流转**：多 Agent 各自处理 → 最后由领队合并
- **适用**：大任务分治，比如一个行业报告拆成政策、市场、技术、案例几个部分

#### Swarm（撒网竞选模式）
- **特点**：同一个任务交给多个 Agent **独立完成**
- **可见性**：参与者**彼此互盲**
- **决策**：最后由领队**择优**
- **适用**：需要多解并行、避免从众的场景，比如标题、方案创意、产品命名、不同分析路径

### 5.3 与飞书/Slack 群聊的对比

Octo 多协作模型不仅是把 Bot 拉到同一个地方，还规定了**信息流转的方式**：不同任务匹配不同拓扑，系统保证信息沿正确路径流动。

| 对比项 | 飞书/Slack 群聊 | Octo 多协作模式 |
|--------|---------------|---------------|
| 信息可见性 | 所有人看到所有消息 | 按拓扑规定可见性 |
| 信息隔离 | 难以做到精细隔离 | 支持互见/互盲的精细控制 |
| 协作结构 | 扁平群聊 | 六种结构化拓扑 |
| 任务匹配 | 一种结构应对所有任务 | 不同任务匹配不同拓扑 |
| 从众风险 | 容易互相影响 | Swarm/Split 避免从众 |

### 5.4 "该互见时互见，该互盲时互盲"的隔离设计

> "群聊擅长'都看见'，却很难做到'该互见时互见，该互盲时互盲'。"

这是 Octo 协作模式设计的核心洞察：

- **互见场景**：Roundtable 需要共识，参与者互相可见以碰撞观点
- **互盲场景**：Swarm 需要多解并行，参与者互盲以避免从众；Split 子块互不可见以保证独立性
- **隔离场景**：Critic 生成方与验证方必须不同，保证审查独立性

这种精细的信息隔离能力，是 Octo 区别于普通 IM 群聊的关键所在。

## 六、产品四层结构

从产品形态看，Octo 正让 Agent 像组织成员一样进入工作流：用 IM 承载交互，用空间、分组、频道、子区搭建协作结构，用语音提升输入效率，用浏览器插件接入外部工具，再用 group.md 约束协作方式。

### 6.1 结构层：Space / Category / Channel / Thread

| 结构 | 含义 | 作用 |
|------|------|------|
| **Space（空间）** | 顶层协作空间 | 划分项目/团队/业务域 |
| **Category（分类）** | 空间内的分组 | 组织频道归类 |
| **Channel（频道）** | 协作通道 | 承载具体任务流 |
| **Thread（话题）** | 频道内的子区 | 聚焦具体议题 |

协作关系被组织得很清楚：

> "一项任务通常是在某个空间里被提出，带着明确上下文：属于哪个空间，在哪个频道，话题是什么。新消息不会很快被冲掉，自然地落在某个频道或话题里。"

### 6.2 入口层：私聊 + 语音输入

**私聊**：可以让人与 Agent 在同一个上下文中沟通、分工、反馈。

**语音输入**：当协作变复杂，问题可能会出现在输入这一端——很多时候不是 Agent 做不出来，反而是**人来不及把需求讲清楚**。引入语音之后，信息可以更快进入系统。

Octo 内置的语音输入不只是将声音转成文字，它同样也是一个**持续进化和学习的系统**：

- 结合当前对话的上下文，对转写的内容进行**修正和梳理**
- 对于团队中的人名、公司名或者行业专有名词，**出现频率越高它认得越准**
- 你可以语音 @ 他人、修改已有内容甚至删掉前面的输入

### 6.3 环境接入层：浏览器插件 Cmd+K

通过内置的浏览器插件，用户可以通过 **「Cmd + K」** 把外部工具无缝接入进来：

- 在网页、文档还是代码平台上，只要选中一段内容
- 当前页面的链接、标题和选中文本会**自动被带入上下文**
- **不需要把你从现有工具流中「拉走」**，在旁参与协作即可

### 6.4 行为约束层：GROUP.md

GROUP.md 相当于一份专门设给 Bot 看的**「行为准则」**：

- 明确一个群聊的**定位**
- 明确**协作模式**
- 明确**行为边界**

> "每一次对话、每一条任务指令，所有参与进来的 Bot 都会在遵守 GROUP.md 规则的前提下执行。当切换到另一套 GROUP.md 时，同一只 Bot 马上调整工作模式，「进什么庙，念什么经」，绝不逾矩。"

GROUP.md 的关键特性是**上下文敏感**：同一只 Bot 在不同的群聊里，会根据 GROUP.md 调整工作模式，做到"进什么庙，念什么经"。

### 6.5 多端补全

Web、移动端、浏览器插件、CLI 共同构成入口。尤其是 **CLI**，它连接端侧环境和私有化部署叙事，让本地模型、本地文件、本地运行环境进入协作体系。

| 端 | 定位 | 关键价值 |
|----|------|---------|
| **Web** | 主入口 | 完整协作体验 |
| **移动端** | 移动办公 | 随时随地参与 |
| **浏览器插件** | 外部工具接入 | Cmd+K 无缝带入上下文 |
| **CLI** | 端侧/私有化 | 连接本地模型、本地文件、本地运行环境 |

## 七、Private AI 与 Trustworthy AI：产品哲学

### 7.1 Private AI 三大主权

当 Agent 真正进入组织运转，数据主权问题变得无法回避：这些上下文、判断信号与执行记录，究竟归谁所有、留在哪里、由谁控制？

Octo 给出的答案很直接：**走私有化路径，通过开源开放支持本地部署**。

对企业而言，Private AI 不只是本地化部署，更是**三大主权的真正回归**：

| 主权 | 含义 | 对应资产 |
|------|------|---------|
| **数据主权** | 数据归企业所有，留在企业环境 | 聊天数据、协作产出、Bot 记忆 |
| **知识主权** | 组织知识自主可控 | Context、Matter 沉淀的决策资产 |
| **协作主权** | 协作过程自主可控 | 协作拓扑、偏好、工作流 |

### 7.2 Octo 通过 CLI 接入端侧模型的实现方式

在实现上，Octo 以 **CLI 接入**的方式将端侧模型与本地环境接入进来：

- 工作流中产生的**上下文、决策过程与执行结果同样留在端侧**
- 沉淀为**组织可以掌控的资产**
- 包括聊天数据、协作产出、Bot 记忆在内，**每条对话、每行代码都保留在你的环境中**
- **完全跑在自己的服务器上**

### 7.3 Trustworthy AI 三大特征

支撑这一切的底座是 **Trustworthy AI**：

| 特征 | 含义 | 体现 |
|------|------|------|
| **开源** | 代码开放可审查 | 通过开源开放支持本地部署 |
| **白盒** | 内部机制可理解 | 能力来源、运行过程透明 |
| **可审计** | 行为可追溯可验证 | 协作边界、决策路径可追溯 |

> "只有当 AI 的能力来源、运行过程和协作边界足够透明，人才放心将「思」交给它们，把「品」留在自己手里。"

### 7.4 "能力可以流动，但数据不外流"设计原则

Octo 的设计原则可以概括为：

- **能力可以流动**：不同 Runtime 的 Agent 都能接入（Open）
- **数据不外流**：上下文、决策、执行记录留在端侧（Private）

这一原则平衡了**生态开放性**与**数据主权**两个看似矛盾的需求。

### 7.5 企业长期竞争力：Context / Taste / Skill

当模型能力快速趋同，长期竞争力更多来自企业自己的：

| 竞争力要素 | 来源 | 特性 |
|-----------|------|------|
| **Context** | 项目上下文、组织记忆 | 无法被复制 |
| **Taste** | 主人品味、判断方式 | 不应该流失 |
| **Skill** | 组织技能、工作流 | 难以快速复制 |

> "这些东西无法被复制，也不应该流失，它们才是组织在 AI 协作中真正的「护城河」。"

### 7.6 个人价值：Taste——"我品故我在"

对个人而言，真正被放大的价值是 **Taste**：

> "我品故我在：当 AI 逐步接管了思考，人的判断力、鉴赏力与创造力不会被取代，反而成为存在的意义本身。"

这一表述揭示了 Octo 的深层哲学：

- **「思」交给 AI**：思考、推理、执行由 Agent 完成
- **「品」留在人手里**：判断、鉴赏、取舍由人主导
- 人的价值从"做"转向"品"，从"思考"转向"判断"

> "当 Agent 更深入地嵌入分工体系，真正决定效率的是那些无法被标准化、也不应该被外流的东西——组织自己的上下文和人自己的判断。"

## 九、常见问题解答（FAQ）

**Q1: Octo 适合什么场景使用？什么规模的企业适合？**

A: Octo 适合**需要多 Agent 协作**的复杂、长程任务场景，典型包括：
- 需要多种 Runtime Agent 协同的复杂任务（如代码生成 + 审查 + 文档撰写）
- 需要沉淀组织决策资产和上下文的长期项目
- 对数据主权、私有化部署有强诉求的场景

企业规模上，Octo 的核心价值在"成百上千 Agent"的规模下最为凸显——当企业开始拥有大量 Agent 时，Octo 才能发挥像管理互联网节点一样的连接、通信与协作能力。小规模单 Agent 场景下，传统 IM 工具的内置 AI 可能更轻量合适。

---

**Q2: Octo 与飞书/Slack 内置 AI 工具的本质差异是什么？**

A: 本质差异在于**协作结构与信息拓扑**：

| 差异维度 | 飞书/Slack 内置 AI | Octo |
|---------|-------------------|------|
| 定位 | 更聪明的聊天工具 | 重写协作本身 |
| 信息可见性 | 都看见 | 该互见时互见，该互盲时互盲 |
| 协作结构 | 扁平群聊 | 六种结构化拓扑 |
| 上下文沉淀 | 散落在聊天记录 | Matter 结构化决策卡 |
| 偏好进化 | 依赖系统提示 | Taste 实战沉淀 |
| 部署方式 | SaaS 为主 | 私有化部署 |

简言之：飞书/Slack 的 AI 是"在群聊里加个 AI 助手"，Octo 是"用 IM 做入口，重写背后的协作结构"。

---

**Q3: Octo 如何实现私有化部署？需要什么基础设施？**

A: Octo 通过以下方式实现私有化部署：

- **CLI 接入**：以 CLI 接入的方式将端侧模型与本地环境接入
- **本地运行环境**：本地模型、本地文件、本地运行环境进入协作体系
- **数据留存**：工作流中产生的上下文、决策过程与执行结果留在端侧
- **开源开放**：通过开源开放支持本地部署

基础设施方面：
- 需要可运行 CLI 的服务器环境
- 需要本地模型运行环境（如支持的 Runtime Agent）
- 所有数据（聊天数据、协作产出、Bot 记忆）完全跑在自己的服务器上
- 具体部署细节可参考 GitHub 开源组织（https://github.com/Mininglamp-OSS）的相关文档

---

**Q4: Octo 支持哪些 Runtime Agent？如何接入自定义 Agent？**

A: Octo 支持的 Runtime Agent 包括：
- **OpenClaw**
- **Codex**
- **Claude Code**
- **Cursor**
- 其他兼容 Runtime

接入方式的核心设计是**统一身份**：不同 Runtime 的 Agent 都能够以 **Bot 身份**接入 Octo，获得统一身份。这意味着接入自定义 Agent 时，需要让它符合 Bot 的身份协议，即可纳入协作体系。具体接入协议和 SDK 可参考 GitHub 仓库文档。

---

**Q5: 六种协作模式如何选择？什么任务用什么模式？**

A: 模式选择遵循"任务结构决定信息拓扑"的原则：

| 任务特征 | 推荐模式 | 理由 |
|---------|---------|------|
| 简单明确，无需协作 | **Solo** | 单点执行最高效 |
| 需要共识、碰撞观点 | **Roundtable** | 公开讨论收束结论 |
| 需要独立审查 | **Critic** | 生成—验证分离，有否决权 |
| 有明确顺序依赖 | **Pipeline** | 串行流转，每步产出喂下步 |
| 大任务可分治 | **Split** | 子块互不可见，最后合并 |
| 需要多解并行、避免从众 | **Swarm** | 参与者互盲，领队择优 |

判断要点：
- 信息需要"互见"还是"互盲"？
- 是串行依赖还是并行分治？
- 需要独立审查还是公开讨论？

---

**Q6: Taste 偏好沉淀机制如何避免学到错误偏好？**

A: Taste 的设计思路是"偏好对齐必须在实战中完成"，避免错误偏好主要依靠以下机制：

1. **基于真实任务反馈**：偏好来源于真实任务的打回、修改、确认，而非假设性输入
2. **Matter 结构化沉淀**：反馈信号沉淀到 Matter 中，携带完整上下文（Brief、Timeline、产出），便于审计
3. **人在关键节点判断**：人退到关键节点做判断和取舍，避免错误反馈被无差别吸收
4. **可追溯可审计**：Trustworthy AI 的"白盒、可审计"特征，让偏好来源可追溯
5. **持续校准**：下次遇到类似任务，相关偏好自动进入上下文，人可以再次判断和修正

需要强调的是，Taste 不是"一次性配置"，而是"持续进化"——错误的偏好会在后续实战中被新的反馈覆盖和修正。

---

**Q7: GROUP.md 如何编写？有哪些必备字段？**

A: GROUP.md 相当于一份专门设给 Bot 看的「行为准则」，编写时需要明确：

**必备内容**：
- **群聊定位**：这个群是做什么的，解决什么问题
- **协作模式**：采用六种模式中的哪一种（Solo/Roundtable/Critic/Pipeline/Split/Swarm）
- **行为边界**：Bot 可以做什么、不可以做什么
- **协作规则**：信息如何流转、谁负责什么、何时需要人介入

**关键特性**：
- **上下文敏感**：同一只 Bot 在不同群聊里，根据 GROUP.md 调整工作模式
- **可切换**：当切换到另一套 GROUP.md 时，同一只 Bot 马上调整工作模式
- **强约束**：每一次对话、每一条任务指令，所有 Bot 都在遵守 GROUP.md 规则的前提下执行

**编写原则**：「进什么庙，念什么经」——GROUP.md 应该让 Bot 在不同场景下表现出符合场景预期的工作模式，绝不逾矩。
