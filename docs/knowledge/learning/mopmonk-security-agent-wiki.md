## 一、教程概述与学习目标

### MopMonk（扫地僧）背景介绍

2026年6月底，一个名为 **MopMonk**（中文代号"扫地僧"）的神秘AI Agent突然出现在全球AI安全领域最权威的评测榜单——CyberGym上，以 **73.1%** 的漏洞复现成功率一举夺得 **全球第七名、中国第一名** 的成绩，紧咬OpenAI（85.6%）、Anthropic（83.1%）、微软（88.4%）等国际巨头。

这个代号取自金庸《天龙八部》中"扫地僧"的AI系统充满了东方武侠式的传奇色彩：没有大张旗鼓的发布会，没有官方网站，没有团队背景介绍，没有融资消息，甚至GitHub仓库也是刚创建不久，唯一的联系方式是一个加密邮箱。它就像少林藏经阁里那个默默无闻扫了几十年地的老和尚，一出手便镇住了全场高手。

在MopMonk之前，中国团队在CyberGym这类硬核AI安全基准上从未取得过如此高位的成绩。它的出现不仅刷新了中国团队在该榜单上的历史最高分，更重要的是——它证明了在AI安全这个垂直领域，**不一定要靠最大的模型、最多的算力，靠精巧的Agent架构设计同样能与国际顶尖玩家同台竞技**。

### 学习目标

通过本教程，你将能够：

1. 理解CyberGym基准测试的意义和难度，明白为什么MopMonk的73.1%成功率是一个里程碑式的成绩
2. 掌握Harness协调层在AI Agent系统中的核心作用，理解"模型是大脑，Harness是手脚和神经系统"这一关键比喻
3. 理解PoC（概念验证）差分验证的原理，知道为什么"能触发漏洞"和"修复后不触发"同样重要
4. 了解MiniMax M3作为国产开源基座模型的能力特点，以及它为何被MopMonk选中
5. 深入理解MopMonk三大核心技术：结构化漏洞记忆、记忆驱动挖掘、共享记忆多Agent并行
6. 建立对AI安全Agent架构的完整认知，理解"模型能力只是入场券，Agent执行力才决定胜负"这一行业新共识

本教程适合以下读者：
- 对AI Agent、大语言模型应用有基本了解的技术人员

- 希望了解AI前沿应用落地案例的产品经理或技术管理者
- 计算机相关专业的学生

学习本教程前，建议具备以下基础知识：
- 了解大语言模型（LLM）的基本概念和能力边界
- 对软件漏洞、PoC（Proof of Concept）等安全术语有初步认知
- 了解Agent系统的基本组成（模型、工具、记忆、规划等）

## 三、核心概念解析

### 3.1 CyberGym基准：AI安全领域的"奥运会"

**CyberGym** 是由UC Berkeley（加州大学伯克利分校）安全研究团队打造的AI网络安全能力评估基准，其核心论文已被ICLR 2026顶会接收。在安全圈，CyberGym被直接封为"**AI安全领域的奥运会**"——因为它足够大、足够真、足够难，能真正区分出不同模型和Agent框架之间的真实能力差距。

#### CyberGym的核心数据：

| 指标 | 数值 | 说明 |
|---|---|---|
| 漏洞实例数量 | **1507个** | 全部来自真实历史漏洞 |
| 覆盖开源项目 | **188个** | 都是实际在使用的大型项目 |
| 数据来源 | Google OSS-Fuzz | 持续模糊测试沉淀的真实漏洞 |
| 相对于NYU CTF | **7.5倍** | NYU CTF约200题，是此前最大公开基准 |
| 评测超时 | 4小时/任务 | 给Agent足够长的时间进行深度推理 |

#### CyberGym为什么这么难？

CyberGym不做选择题，不考"知不知道"，它考的是"**做不做得到**"。传统的漏洞分析任务只需要判断"这段代码有没有问题"，但CyberGym要求Agent完成完整的工程闭环：

1. **理解漏洞线索**：拿到一段漏洞描述和一个未经修复的代码库
2. **深度代码分析**：在数千个文件、数百万行代码中定位漏洞位置
3. **构造PoC输入**：生成一个能真正触发漏洞的输入（不是说说而已）
4. **差分验证通过**：这个PoC必须在漏洞版本上触发crash（崩溃），在修复后的版本上不触发
5. **执行验证**：所有结果都要在封闭环境中实际运行验证，没有"部分正确"的安慰分

这还只是CyberGym Level 1（PoC生成）的要求。更难的CyberGym-E2E基准覆盖920个漏洞、139个项目，要求Agent完成从漏洞发现→PoC生成→补丁编写→功能回归测试的完整安全工程师工作流。

截至榜单发布时，AI Agent在CyberGym上已经发现了34个零日漏洞（0-day）和18个不完整的历史补丁——这不是实验室里的玩具测试，这是真刀真枪的安全实战。也正因如此，全球头部玩家几乎全员到场：微软、OpenAI、Anthropic、谷歌、Meta、智谱……都在这个擂台上比拼硬实力。

### 3.2 Harness协调层：大脑与手脚的神经系统

在理解MopMonk为什么能赢之前，我们需要先理解一个关键概念——**Harness**（协调层/执行框架）。

很多人第一次看到MopMonk的成绩时，第一反应是："它肯定用了什么特别强大的模型。"但答案让人意外：MopMonk用的是MiniMax M3——虽然能力很强，但和GPT-5.5-Cyber、Claude Mythos这些顶级闭源模型比起来，模型本身并不占优。

MopMonk真正的胜负手，在Harness。

#### Harness是什么？

你可以做一个形象的比喻：
- **大语言模型 = 大脑**：它负责思考、推理、判断——漏洞可能在哪里？下一步该怎么试？
- **Harness = 手脚 + 神经系统**：它负责把大脑的想法变成实际的行动，并把行动的结果反馈给大脑。

具体来说，Harness负责：
1. **工具编排**：什么时候调用grep搜索代码？什么时候运行编译？什么时候执行PoC？
2. **上下文状态管理**：如何管理海量的代码信息和多轮尝试的结果，不丢失关键信息？
3. **执行反馈回收**：程序运行结果是什么？crash了吗？错误信息是什么？把这些反馈整理后再交给模型
4. **错误收敛**：让每一次失败都变成"下一次更像样的尝试"，而不是原地打转或反复走同一条死路

#### 为什么Harness如此重要？

CyberGym是一场需要跑很多轮、要在百万行代码里反复试验的长程任务（最长4小时）。在这样的任务中，一个常见的"翻车组合"是：
- 模型很聪明，很会"想"
- 但Harness设计得不好，不会"协同"
- 结果就是：**想得出来，做不到**，PoC永远跑不成

而真正能出成绩的组合是：
- 模型能力够硬（拿到"入场券"）
- Harness专门为漏洞挖掘任务调过、磨过
- 让试错变成一个**可积累、可收敛、可验证**的过程

用一个通俗的例子：模型就像一个刚毕业的安全天才，理论知识很丰富，但没怎么实际挖过漏洞；Harness就像一个经验丰富的安全专家带教，它会告诉这个天才："这条路我们刚才走过了，走不通""这个输入格式不对，换一个""crash栈指向这里，我们往这个方向查""上次失败是因为边界条件没考虑，这次要避开"。

在MopMonk的案例中，Harness的价值被体现得淋漓尽致——它证明了在需要长时间推理、工具调用、多次试错的复杂任务中，**模型智商只起到入场券的作用，而Agent的执行框架才是决定胜负的关键**。

### 3.3 PoC概念验证：差分判定的黄金标准

**PoC** 是 **Proof of Concept**（概念验证）的缩写。在网络安全领域，PoC指的是一段可以证明漏洞真实存在的代码或输入——简单说，就是"能触发这个漏洞的样例"。

但在CyberGym的评测标准中，PoC的要求比普通安全研究更严格——它必须通过**差分验证**（Differential Verification）。

#### 什么是差分验证？

一个合格的PoC必须同时满足两个条件：

| 版本 | PoC行为 | 结果 |
|---|---|---|
| **漏洞版本**（未修复） | 运行PoC | ✅ 必须触发crash/漏洞行为 |
| **修复版本**（已打补丁） | 运行同一个PoC | ❌ 必须不触发crash/漏洞行为 |

这两个条件缺一不可。

#### 为什么要这么设计？

因为"能让程序崩溃"不一定等于"触发了目标漏洞"。程序崩溃的原因有很多：可能是你输入格式不对，可能是环境配置有问题，可能是触发了另一个无关的bug——只有当"漏洞版本触发，修复版本不触发"时，才能确凿地证明：你确实找到了并触发了那个特定的漏洞。

这个要求看似简单，实际上非常刁钻：
- 漏洞的触发条件往往零散地藏在代码路径、解析逻辑、构建环境、测试Harness和输入格式之间，需要一点点拼出来
- 哪怕PoC在本地把程序跑崩了，也未必算数——只要不能通过差分验证，就是失败
- 没有"差不多对了"这种说法，差一丁点都不算成功

也正因为PoC差分验证如此严格，73.1%的成功率才显得含金量十足——这意味着MopMonk在超过七成的真实漏洞案例中，都完成了从代码理解到输入构造再到执行验证的完整闭环。

### 3.4 MiniMax M3基座：国产开源的六边形战士

MopMonk选择的基座模型是**MiniMax M3**，这是来自上海AI公司MiniMax（稀宇科技）发布的开源大模型。M3被称为"六边形战士"，因为它同时集齐了三大前沿能力，而这是当时第一个也是唯一一个同时具备这三项能力的开源权重模型：

| 能力 | 说明 | 对漏洞挖掘的价值 |
|---|---|---|
| **前沿编程能力** | 代码理解、生成、调试能力达到开源模型第一梯队 | 能读懂数百万行代码，理解复杂逻辑，构造正确的PoC输入 |
| **1M超长上下文** | 基于MSA（MiniMax Sparse Attention）稀疏注意力架构，支持100万token上下文窗口 | 能直接"看"完整个大型代码库，不会出现"翻到后面忘记前面"的情况 |
| **原生多模态** | 从Step 0就进行多模态混合训练，支持图像、视频输入和桌面操作 | 在长周期Agent任务中保持高稳定性，处理复杂的执行反馈 |

#### MiniMax M3的关键基准成绩（官方发布）：

| 基准测试 | M3成绩 | 测试内容 |
|---|---|---|
| **SWE-Bench Pro** | **59.0%** | 软件工程任务——真实GitHub issue解决 |
| **Terminal-Bench 2.1** | **66.0%** | 命令行环境中的复杂真实任务 |
| **MCP Atlas** | **74.2%** | 工具调用和MCP协议能力 |
| SWE-fficiency | 34.8% | 代码修复效率 |
| KernelBench Hard | 28.8% | 内核级编程任务 |

值得注意的是，M3在SWE-Bench Pro上的59.0%甚至超过了GPT-5.5的58.6%——这也是它能支撑起MopMonk漏洞挖掘任务的基础。M3的MSA稀疏注意力架构是一个关键创新：它通过更精确的KV分块和算子级优化，在100万上下文长度下，每token计算量仅为上一代模型的1/20，prefill阶段提速9倍以上，解码阶段提速15倍以上。这意味着长上下文不仅是"能用"，而且是"高效能用"。

MopMonk团队在技术报告中明确说明，选择M3是因为它结合了：长上下文能力、MoE架构的有效容量、高效稀疏注意力、强大的代码能力，以及在长周期Agent任务中的稳定执行。

### 3.5 三大核心技术：记忆驱动的安全Agent范式

MopMonk的核心技术路线被称为"**memory-centric Multi-Agent design**"（以记忆为中心的多Agent设计）。它不是靠堆模型参数，也不是靠堆Agent数量，而是靠结构化的记忆系统，把漏洞挖掘从"反复读代码、反复试错、反复撞同一堵墙"的盲猜过程，变成了"**可积累、可约束、可验证**"的证据收敛过程。

MopMonk的三大核心技术是：
1. 结构化漏洞记忆（七类对象）
2. 记忆驱动的漏洞挖掘
3. 共享记忆下的多Agent并行探索

---

#### 3.5.1 结构化漏洞记忆：七类核心对象

传统Agent要么堆积聊天记录，要么把所有上下文一股脑塞进去，导致信息混乱、成本高昂、关键信息容易丢失。MopMonk没有这么做——它围绕漏洞挖掘过程中的关键对象，设计了**七类结构化记忆**，把每一次代码阅读、每一次执行反馈、每一次失败尝试都转化为结构化的"证据"存起来。

这七类记忆对象分别是：

| 记忆类型 | 存储内容 | 作用 |
|---|---|---|
| **1. 漏洞目标记忆**<br>(Vulnerability-goal memory) | 目标漏洞是什么、成功条件是什么、可接受的验证标准、当前任务最重要的约束 | 让所有探索都不偏离目标，不会"挖着挖着忘了要找什么" |
| **2. 代码路径记忆**<br>(Code-path memory) | 已确认的入口点、测试Harness、解析链、可疑函数、关键数据流 | 后来的尝试不需要重新发现这些路径，避免重复劳动 |
| **3. 输入格式记忆**<br>(Input-format memory) | 文件格式、字段关系、长度约束、边界条件、已知有效/无效输入模式 | 构造PoC时不需要反复猜格式，直接基于已知约束生成 |
| **4. 候选PoC记忆**<br>(Candidate-PoC memory) | 候选输入、生成理由、触发行为、变异方向、仍需验证的假设 | 保留"接近成功"的尝试，在此基础上继续迭代 |
| **5. 负面证据记忆**<br>(Negative-evidence memory) | 未触发的尝试、不可达路径、构建失败、格式错误等负面结果 | **最关键的一类记忆**——明确记录"哪些路走不通"，避免反复踩同一个坑 |
| **6. 验证状态记忆**<br>(Verification-state memory) | 候选PoC是否触发crash、如果没触发是因为什么原因失败 | 清晰追踪每个尝试的状态和结果 |
| **7. 下一步约束记忆**<br>(Next-constraint memory) | 下一次尝试必须满足的约束——比如要到达哪个分支、要调整哪个字段、要避开什么失败原因 | 不产出抽象的"计划"，而是产出具体的、可执行的约束 |

你可以把这七类记忆想象成一个安全专家在挖漏洞时随身带的"破案笔记本"：
- 目标是什么？
- 已经查过哪些地方？
- 输入长什么样？
- 有哪些可能的突破口？
- 哪些方法试过了没用？
- 现在进展到哪一步了？
- 下一步具体该怎么做？

没有这个笔记本，每次尝试都像"第一天上班"；有了这个笔记本，每一次尝试都站在前人（前一次自己）的肩膀上。

---

#### 3.5.2 记忆驱动的漏洞挖掘

有了结构化记忆之后，MopMonk的漏洞挖掘过程就不再是"随机试错"，而是一个**记忆驱动的证据收敛过程**。

#### 工作流程：

1. **初始化阶段**：系统首先扫描代码库，用候选触发路径和目录信息初始化漏洞记忆，作为规划的起点
2. **探索阶段**：每一轮探索都执行相同的循环：
   - 读取当前记忆：现在知道了什么？哪些路走过了？有什么约束？
   - 验证一个具体假设：基于当前证据，选择一个最有可能的方向进行尝试
   - 写入结果回记忆：这次尝试成功了还是失败了？发现了什么新证据？得到了什么新约束？
3. **收敛阶段**：随着记忆不断积累，搜索空间不断缩小，最终收敛到能触发漏洞的PoC

#### 这个机制带来的直接效果：

| 效果 | 说明 |
|---|---|
| **更少的重复探索** | 失败路径被明确记录，后来的尝试不会一直沿着同一条源码路径或输入变异方向走下去 |
| **负面证据被保留** | 没触发的尝试、格式错误的输入、走不通的路径——这些都变成了约束，而不是被丢弃 |
| **更快的PoC收敛** | 每一次PoC变异都可以继承之前发现的代码路径、输入格式约束和验证反馈 |
| **更低的长上下文负担** | 模型不需要每次都从头理解整个任务，只需要从结构化记忆中读取最相关的证据 |
| **预算内更高的有效实验密度** | 在4小时的时间限制内，花更多时间在新的、有用的假设上，而不是重复验证已知错误 |

这就是MopMonk和传统Agent最本质的区别：传统Agent是"上下文驱动"——每次都把所有对话历史塞进去，让模型自己从中找线索；MopMonk是"记忆驱动"——先把信息结构化整理好，模型只需要看结构化后的证据，然后做下一步决策。

---

#### 3.5.3 共享记忆下的多Agent并行探索

MopMonk的第三个核心设计是：**多个Agent并行探索，但共享同一套漏洞记忆**。

这不是简单的"堆Agent数量"——如果多个Agent各干各的，互相不通信，那只是把"一个人瞎试"变成了"一群人瞎试"，本质上还是在浪费资源。MopMonk的聪明之处在于：**所有Agent都读写同一个记忆池**。

#### 工作方式：

- 针对同一个漏洞描述，可以同时启动多个探索尝试
- 每个Agent从自己的角度切入：一个从crash栈回溯推导，一个从边界条件构造最小输入，一个分析数据流中的可疑函数
- 每个Agent在独立的沙盒里运行，互不干扰
- **但所有发现——失败的路径、可疑的代码段、候选PoC、验证结果——都写回到同一个共享记忆池**
- Agent A走过的死路，Agent B不需要再走一遍；Agent B发现的可疑线索，Agent C可以直接跟进

#### 共享记忆的价值：

你可以把这想象成一个侦探团队：多个侦探同时调查同一个案子，有人去问证人，有人去看监控，有人去查物证——但所有线索都汇总到同一个白板上，所有人都能看到最新的进展。这样的团队效率，当然比几个侦探各自闭门造车要高得多。

在4小时的超时限制下，这套共享记忆机制让每一轮探索都踩在前一轮的肩上。并行不是目的，**让每一次尝试都有信息增量**才是目的。

---

#### 三大技术的关系总结：

| 技术 | 角色 | 比喻 |
|---|---|---|
| 结构化漏洞记忆 | 基础 | 破案笔记本 |
| 记忆驱动挖掘 | 流程 | 基于证据的破案过程 |
| 共享记忆多Agent并行 | 组织方式 | 共享白板的侦探团队 |

三者结合起来，就把漏洞挖掘从"一个人瞎摸"变成了"一个有组织、有记录、有协作的团队破案"。

### Q2：MopMonk和传统的漏洞挖掘方法（比如Fuzzing、静态分析）有什么区别？

**A**：这是一个很好的问题。传统漏洞挖掘方法主要有三类：
1. **静态分析**：不运行程序，通过分析代码结构找漏洞——速度快但误报率高
2. **Fuzzing（模糊测试）**：随机或半随机生成大量输入喂给程序，看能不能崩——有效但覆盖率依赖种子质量和变异策略
3. **人工审计**：安全专家逐行审查代码——质量高但速度慢、成本高、依赖专家经验

MopMonk代表的是**AI Agent驱动的自动化漏洞挖掘**新范式，它和传统方法的核心区别是：
- **从"随机/规则"到"推理"**：Fuzzing主要靠随机变异，MopMonk是基于对代码的理解和记忆中的证据进行**推理式探索**
- **端到端闭环**：从理解漏洞线索→定位代码→构造输入→运行验证→根据反馈调整，整个闭环自动完成，不需要人工介入
- **可积累的经验**：传统Fuzzing每次运行（大部分情况下）是从头开始，MopMonk通过结构化记忆让每一次尝试都有积累
- **处理复杂逻辑漏洞**：对于需要跨多个文件、理解复杂业务逻辑才能触发的漏洞，AI Agent比传统方法更有优势

当然，这不是说AI Agent会完全取代传统方法——更可能的情况是两者结合，比如用Fuzzing做大规模初筛，用AI Agent做深度漏洞复现和利用。

---

### Q3：为什么Harness/框架这么重要？直接用最强的模型不行吗？

**A**：这个问题触及了当前AI行业一个正在发生的认知转向。

在单轮问答、简单代码生成这类"短平快"任务中，模型能力确实是决定性的——模型越强，结果越好。但在漏洞挖掘这种**长程、复杂、需要多轮工具调用的任务**中，情况完全不同：

1. **上下文窗口再大也会乱**：哪怕有1M上下文，如果你把几百轮的原始交互记录全都塞进去，模型也会迷失在信息海洋里，找不到重点
2. **失败需要被系统性学习**：人做事情会"吃一堑长一智"，但如果没有结构化的记忆机制，模型下一轮还是会犯同样的错误
3. **工具调用需要编排**：什么时候编译？什么时候运行？什么时候看crash信息？这些需要精确的流程控制，不是模型自己每次都能判断对的
4. **任务需要被分解和收敛**：挖漏洞是一个"搜索空间不断缩小"的过程，需要框架来引导这个收敛过程，而不是让模型漫无目的地"探索"

打个比方：
- 模型就像一个智商很高的学生
- Harness/框架就像好的学习方法、错题本、学习计划
- 没有好的学习方法，再聪明的学生也可能考不好；有了好的学习方法，普通智商的学生也能发挥出最大潜力

MopMonk就是一个典型案例：它用的M3模型虽然很强，但并不是榜单上最强的模型（前面还有Claude Opus 4.6、GPT-5.5-Cyber等），但靠优秀的框架设计，它做到了和顶级闭源模型同场竞技。

### Q4：如果我想学习AI安全Agent开发，推荐的学习路径是什么？

4. **动手实践**：用LangChain、AutoGPT或类似框架做一个简单的工具调用Agent

#### 第二阶段：深入Agent架构（2-3个月）

### Q5：MopMonk这件事对整个AI行业有什么意义？

**A**：MopMonk的出现不是一个孤立事件，它代表了AI行业几个重要的发展趋势：

1. **从"模型竞赛"到"系统竞赛"**：过去几年行业的焦点是"谁的模型更大、参数更多、跑分更高"，但MopMonk证明了——在真实复杂任务中，**系统架构设计的重要性不亚于模型本身**。一个架构优秀但模型稍弱的系统，完全可以击败架构一般但模型更强的系统。

2. **开源模型的崛起**：MopMonk用开源的MiniMax M3取得了和顶级闭源模型同场竞技的成绩，这说明开源模型和闭源模型之间的差距正在缩小。开源生态的力量正在显现——全球开发者都可以站在开源基座的肩膀上，专注于上层架构创新。

3. **垂直领域Agent的价值**：通用大模型是"通才"，但在特定垂直领域（比如安全、医疗、法律），经过专门设计的垂直Agent可以发挥出比通用模型更大的价值。MopMonk证明了"垂直深耕"路线的可行性。

4. **中国AI的差异化竞争力**：在基础大模型上，中国团队和国际最顶尖水平还有一定差距；但在应用层、Agent架构层、垂直领域落地上，中国团队完全有机会实现"弯道超车"。MopMonk就是一个很好的例子。

5. **Agent范式的成熟**：从MopMonk的设计可以看出，Agent架构正在从"靠prompt技巧"走向"靠系统化设计"——结构化记忆、多Agent协作、工具编排、验证反馈……这些工程化的设计正在沉淀为可复用的方法论。

### Q6：MiniMax M3在MopMonk系统中到底起到什么作用？能不能换成其他模型？

**A**：MiniMax M3是MopMonk的"动力引擎"，它提供了几个对漏洞挖掘至关重要的基础能力：

| M3的能力 | 在MopMonk中的作用 |
|---|---|
| **强代码能力**（SWE-Bench Pro 59.0%） | 能读懂数百万行真实项目代码，理解复杂逻辑关系，推理漏洞位置 |
| **1M长上下文**（MSA稀疏注意力） | 能一次性加载和理解整个代码库，不会因为代码太长而"遗忘"前面的内容 |
| **长周期任务稳定性** | 在长达4小时、数百轮交互的任务中保持输出格式稳定、逻辑连贯，不会"崩溃"或"失忆" |
| **工具调用能力**（MCP Atlas 74.2%） | 能正确理解工具返回结果（编译错误、crash栈、运行输出等），并据此调整下一步行动 |
| **指令遵循能力** | 能严格按照记忆系统给出的约束进行PoC构造，而不是"自由发挥" |

理论上MopMonk的架构可以换成其他基座模型——GitHub README里也说了，M3是这个submission（这次提交）使用的模型，并不代表架构只能用M3。但要取得好成绩，替换的模型必须满足几个条件：
1. 代码能力足够强（SWE-Bench Pro至少50%以上水平）
2. 上下文窗口足够大（至少能处理几十万token，最好能到1M）
3. 长周期任务稳定性好（连续跑几小时不会出现格式错乱或逻辑混乱）
4. 工具调用和反馈吸收能力强

如果换成一个代码能力一般的小模型，哪怕架构再好，可能也很难达到73.1%的成绩——模型提供"基础能力下限"，架构决定"能力发挥上限"。
