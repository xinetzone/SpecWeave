# 核心概念与设计理念

## 通用 Agent + Skills 方案的三大问题

如果你深度用过 Claude Code 等通用 Agent + Skills 方案做代码评审，可能会对以下三个问题深有同感。这些问题正是 Open Code Review 试图从架构层面解决的核心痛点。

### 问题一：覆盖不全

当变更规模较大时，Agent 倾向于"偷懒"，选择性地评审部分文件，导致部分改动被遗漏。AI 每天生成的代码量已经远超人工评审的上限——以前每天 review 几百行，现在动辄几千、几万行。在这样规模的变更下，通用 Agent 没有强约束机制来保证"每一个重要改动都被评审到"，遗漏几乎是必然的。

### 问题二：位置漂移

报告的问题与实际代码位置常常对不上，出现行号或文件偏移。位置不准确不仅增加理解成本，还会为后续的自动修复引入噪声。这一问题的根源在于 LLM 对数字不敏感——让模型直接输出行号，改动越复杂，位置偏移越明显。

### 问题三：效果不稳定

基于自然语言驱动的 Skills 难以调试，评审质量因提示词的细微差异而大幅波动。同样是代码评审任务，换了几个措辞、调整一下指令顺序，结果可能完全不同。这种不可预测性使得 Skills 方案难以在生产环境中稳定落地。

### 根源分析

这三个问题的共同根源在于：**纯语言驱动的架构缺乏对评审流程的强约束**。语言模型本身是概率性的、自由度极高的系统，而代码评审场景中存在大量"不能出错"的环节——哪些文件要评审、问题对应哪一行代码、规则如何匹配——这些环节如果交给语言模型自由发挥，就必然出现覆盖不全、位置漂移、效果波动等问题。

## 确定性工程 × Agent 混合驱动

Open Code Review 的核心设计理念是将确定性工程与 Agent 结合，各司其职：

- **确定性工程**负责"不能出错"的强约束环节，由工程逻辑而非语言模型来保证
- **Agent** 负责它真正擅长的动态决策环节，包括动态召回上下文、动态推理等

这种混合驱动的关键在于：把不需要"理解"的步骤全部交给工程代码（零 token 消耗、行为可预测），而把真正需要语义理解和推理的环节交给 LLM（把最贵的资源用在最需要的地方）。

## 确定性工程负责的强约束环节

对代码评审场景中"不能出错"的环节，Open Code Review 由工程逻辑而非语言模型来保证。具体包含四个核心环节。

### 1. 精准的文件筛选

明确哪些文件需要评审、哪些应当过滤，确保真正重要的改动一个不漏。文件筛选、路径规则匹配等不需要"理解"的步骤全部由工程代码完成，零 token 消耗，行为完全可预测。这一环节从源头规避了通用 Agent "偷懒"导致的覆盖不全问题。

### 2. 智能的文件打包

将关联文件归并为同一评审单元。例如 `message_en.properties` 与 `message_zh.properties` 会被打包在一起，在同一上下文中评审，从而解决跨文件关联缺陷的检测问题。

每个包会作为独立的 subagent 进行任务，它们之间的上下文是隔离的。这一分治策略在超大变更场景下表现更为稳定——变更规模翻倍，token 消耗也仅线性增长，而不会因上下文膨胀导致成本失控。同时，这种分治设计天然支持并发评审，多个评审包可以并行处理（由于某些原因，对外版本暂时采用了固定的分治策略）。

### 3. 精细化规则匹配

针对不同文件的特征，匹配对应的评审规则，确保模型的注意力足够聚焦，从源头规避信息噪声的干扰。

Open Code Review 通过 glob pattern 将规则精准匹配到特定文件类型（如 Java 文件用 Java 规则、XML mapper 用 SQL 规则），避免模型"发散"到不相关的问题上。相比纯自然语言驱动的规则引导，**基于模板引擎的规则匹配行为更稳定、结果更可预期**。这些规则是根据当下架构的能力边界以及阿里内部数万用户的使用数据迭代而来。

### 4. 外挂的定位与反思组件

独立的评论定位模块与评论反思模块，系统性地提升 AI 反馈的位置准确性与内容准确性。

- **评论定位模块**：采用三层递进式定位策略（Hunk-based 文本匹配 → 全文件内容扫描 → LLM 重定位），从架构上回避了"让 LLM 数行号"的固有缺陷
- **评论反思模块**：作为过滤器拦截误报，在阿里内部实践中将误报拦截率从 30.09% 提升到 52.63%

这两个组件以"外挂"形式存在，与主评审流程解耦，既可以独立优化，又不会干扰主流程的稳定性。

## Agent 负责的动态决策环节

将 Agent 的优势集中发挥在它真正擅长的地方——动态决策、动态召回上下文等。Open Code Review 在以下两个环节充分发挥 Agent 的能力。

### 1. 场景化提示词调优

针对代码评审场景深度优化提示词模版，在提升效果的同时有效降低 Token 消耗。

通用 Agent 的提示词需要兼顾各种场景，而 Open Code Review 的提示词专门为代码评审这一垂直场景设计，能够更精准地引导模型关注代码缺陷模式，避免在无关方向上浪费 token。这种场景化调优带来的效果提升是显著的——在开源评测集中，Open Code Review 的准确率（25%–38%）远高于 Claude Code 的 7%–16%。

### 2. 场景化工具集沉淀

基于大规模生产数据中的 tool-call traces 分析，对通用 Agent 工具集进行取舍与拆分，最终沉淀出一套在代码评审场景下效果更稳定、行为更可预期的专属工具集。

分析维度包括：

- **调用频率分布**：哪些工具被高频使用，哪些几乎不被调用
- **单一工具的重复调用率**：是否存在无效的重复调用
- **新增工具对整体调用链路的影响**：新增工具是否带来了正向价值，还是增加了噪声

通过对这些数据的深入分析，Open Code Review 对通用 Agent 工具集进行了取舍与拆分，最终形成的专属工具集（如 `file_read`、`code_search`、`file_read_diff`、`file_find` 等）比通用 Agent 工具包更稳定、更可预测，每个工具都有明确的输出边界（如 `file_read` 单次最多返回 500 行，`code_search` 最多 100 条匹配）。

## 确定性工程与 Agent 的职责分工

下表对比了确定性工程与 Agent 在 Open Code Review 架构中的职责分工：

| 维度 | 确定性工程 | Agent |
|---|---|---|
| **核心职责** | 负责强约束环节（不能出错） | 负责动态决策环节（需要推理） |
| **文件筛选** | 精准筛选需要评审的文件，过滤无关文件 | — |
| **文件打包** | 将关联文件归并为评审单元，支持并发 | — |
| **规则匹配** | 基于 glob pattern 精准匹配评审规则 | — |
| **评论定位** | 三层递进式定位策略（Hunk 匹配 → 全文件扫描 → LLM 重定位） | LLM 重定位作为兜底 |
| **评论反思** | 反思模型作为过滤器拦截误报 | — |
| **提示词调优** | — | 场景化提示词模版，降低 token 消耗 |
| **工具集** | — | 场景化专属工具集，行为可预期 |
| **上下文召回** | — | Agent 化动态召回（最多 20 轮 tool-use） |
| **Token 消耗** | 零 token（工程代码完成） | 仅在需要语义理解的环节介入 |
| **行为可预测性** | 高（工程逻辑保证） | 中（场景化调优后显著提升） |
| **适用场景** | 高确定性任务（筛选、匹配、定位） | 需要语义理解和动态推理的任务 |

这种"确定性工程 × Agent 混合驱动"的架构设计，使得 Open Code Review 既具备了工程系统的稳定性和可预测性，又保留了 Agent 的灵活性和推理能力。在阿里集团内部的大规模生产实践中，这一架构支撑了月活 2 万开发者、累计 370 万次真实评审任务，评论位置准确率超过 97%，有效 AI 评论占比近 80%。
