## 二、核心架构与设计理念

### 2.1 Headroom中间层定位

Headroom是一个夹在**AI Agent**和**LLM**之间的**透明中间层**，可以类比为HTTP代理或者压缩网关。它不改变模型本身，也不改变Agent的工作逻辑，只是在数据传输的路径上做了一层智能压缩处理。

这种架构设计有几个关键优势：

- **透明接入**：不需要修改Agent的核心逻辑，也不需要更换模型
- **即插即用**：多种接入方式适配不同使用场景
- **本地部署**：所有数据处理都在本地完成，隐私安全有保障

### 2.2 拦截的内容类型

Headroom会拦截所有送入LLM的内容，覆盖以下6大类：

| 内容类型 | 典型场景 | 噪声特点 |
|----------|----------|----------|
| 工具输出 | API调用返回、工具执行结果 | 常有大量冗余字段、空值、重复结构 |
| 命令行结果 | grep/find/ls等命令输出 | 大量无关行、格式噪声、重复信息 |
| 代码搜索结果 | 代码检索返回的片段列表 | 重复匹配、不相关代码片段 |
| RAG检索片段 | 向量检索返回的文档块 | 冗余上下文、重复信息、不相关片段 |
| 文件内容 | 读取的代码文件、配置文件 | 注释、空行、未使用的import、样板代码 |
| 对话历史 | 多轮对话的历史消息 | 重复确认、冗余问候、已解决问题的残留上下文 |

可以看出，Headroom不是只针对某一类内容做压缩，而是**覆盖了Agent场景下所有可能进入上下文的内容类型**，这是它与很多单点压缩工具最大的区别之一。

### 2.3 工作原理

Headroom的工作流程可以分为以下几个步骤：

1. **内容进入**：AI Agent准备向LLM发送消息（包含工具结果、文件内容等），消息先经过Headroom
2. **内容类型识别**：Headroom先分析内容，判断是JSON、代码、日志、还是自然语言（内容路由）
3. **路由到对应压缩算法**：根据内容类型选择最合适的压缩算法进行智能压缩
4. **压缩版送入LLM**：精简后的压缩内容被送入LLM，Token消耗大幅降低
5. **原始数据存入本地缓存**：原始完整数据被保存在本地存储中，永不删除
6. **LLM需要时可Retrieve**：如果模型发现压缩后信息不够用，可以调用`headroom_retrieve`工具按需取回原文

整个过程对Agent和LLM双方都是透明的，不需要它们做任何特殊适配。

### 2.4 设计理念

Headroom的设计围绕三个核心理念展开：

**1. 不搞一刀切**

不同类型的内容有不同的结构和信息密度，用同一种压缩方法处理所有内容必然效果不佳。Headroom采用"对症下药"的思路，先做内容路由，再选对应算法，这是它能实现高压缩率同时保持质量的关键。

**2. 可逆优先**

市面上绝大多数压缩方案都是"压完就没了"——截断、摘要之后，信息就永久丢失了，模型如果发现关键信息缺失，无能为力。Headroom坚持"可逆优先"，原始数据永远保留在本地，模型可以按需取回，这从根本上解决了"压缩导致信息丢失"的焦虑。

**3. 本地优先**

所有压缩处理和数据存储都在本地完成，数据不会上传到任何云端。这在隐私敏感的企业场景和代码项目场景下尤为重要——你的代码、日志、对话数据，永远留在你自己的机器上。

### 2.5 四种接入方式总览

Headroom提供了四种灵活的接入方式，适配不同的使用场景：

| 接入方式 | 适用场景 | 代码改动量 | 上手难度 |
|----------|----------|------------|----------|
| Library（库） | 自己开发Python/TypeScript应用，想在代码中直接集成压缩能力 | 几行代码 | 低 |
| Proxy（代理） | 已有OpenAI兼容的客户端，不想改任何代码，想直接套一层压缩 | 零代码改动 | 极低 |
| Agent Wrap（包装） | 使用Claude Code、Codex、Cursor、Aider、Copilot等主流编程Agent，想一条命令启用压缩 | 零代码改动 | 极低 |
| MCP Server | 使用支持MCP（Model Context Protocol）的客户端，想通过工具调用方式灵活控制压缩 | 注册配置即可 | 中 |

可以说，从开发者到普通用户，从自定义开发到开箱即用，Headroom的接入方式覆盖了绝大多数使用场景。下一章我们会详细讲解每种接入方式的具体用法。
