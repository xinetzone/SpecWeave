### 3.5 三大核心技术：记忆驱动的安全Agent范式

MopMonk的核心技术路线被称为"**memory-centric Multi-Agent design**"（以记忆为中心的多Agent设计）。它不是靠堆模型参数，也不是靠堆Agent数量，而是靠结构化的记忆系统，把漏洞挖掘从"反复读代码、反复试错、反复撞同一堵墙"的盲猜过程，变成了"**可积累、可约束、可验证**"的证据收敛过程。

MopMonk的三大核心技术是：
1. 结构化漏洞记忆（七类对象）
2. 记忆驱动的漏洞挖掘
3. 共享记忆下的多Agent并行探索

---

#### 3.5.1 结构化漏洞记忆：七类核心对象

传统Agent要么堆积聊天记录，要么把所有上下文一股脑塞进去，导致信息混乱、成本高昂、关键信息容易丢失。MopMonk没有这么做——它围绕漏洞挖掘过程中的关键对象，设计了**七类结构化记忆**，把每一次代码阅读、每一次执行反馈、每一次失败尝试都转化为结构化的"证据"存起来。

这七类记忆对象分别是：

| 记忆类型 | 存储内容 | 作用 |
|---|---|---|
| **1. 漏洞目标记忆**<br>(Vulnerability-goal memory) | 目标漏洞是什么、成功条件是什么、可接受的验证标准、当前任务最重要的约束 | 让所有探索都不偏离目标，不会"挖着挖着忘了要找什么" |
| **2. 代码路径记忆**<br>(Code-path memory) | 已确认的入口点、测试Harness、解析链、可疑函数、关键数据流 | 后来的尝试不需要重新发现这些路径，避免重复劳动 |
| **3. 输入格式记忆**<br>(Input-format memory) | 文件格式、字段关系、长度约束、边界条件、已知有效/无效输入模式 | 构造PoC时不需要反复猜格式，直接基于已知约束生成 |
| **4. 候选PoC记忆**<br>(Candidate-PoC memory) | 候选输入、生成理由、触发行为、变异方向、仍需验证的假设 | 保留"接近成功"的尝试，在此基础上继续迭代 |
| **5. 负面证据记忆**<br>(Negative-evidence memory) | 未触发的尝试、不可达路径、构建失败、格式错误等负面结果 | **最关键的一类记忆**——明确记录"哪些路走不通"，避免反复踩同一个坑 |
| **6. 验证状态记忆**<br>(Verification-state memory) | 候选PoC是否触发crash、如果没触发是因为什么原因失败 | 清晰追踪每个尝试的状态和结果 |
| **7. 下一步约束记忆**<br>(Next-constraint memory) | 下一次尝试必须满足的约束——比如要到达哪个分支、要调整哪个字段、要避开什么失败原因 | 不产出抽象的"计划"，而是产出具体的、可执行的约束 |

你可以把这七类记忆想象成一个安全专家在挖漏洞时随身带的"破案笔记本"：
- 目标是什么？
- 已经查过哪些地方？
- 输入长什么样？
- 有哪些可能的突破口？
- 哪些方法试过了没用？
- 现在进展到哪一步了？
- 下一步具体该怎么做？

没有这个笔记本，每次尝试都像"第一天上班"；有了这个笔记本，每一次尝试都站在前人（前一次自己）的肩膀上。

#### 3.5.2 记忆驱动的漏洞挖掘

有了结构化记忆之后，MopMonk的漏洞挖掘过程就不再是"随机试错"，而是一个**记忆驱动的证据收敛过程**。

#### 工作流程：

1. **初始化阶段**：系统首先扫描代码库，用候选触发路径和目录信息初始化漏洞记忆，作为规划的起点
2. **探索阶段**：每一轮探索都执行相同的循环：
   - 读取当前记忆：现在知道了什么？哪些路走过了？有什么约束？
   - 验证一个具体假设：基于当前证据，选择一个最有可能的方向进行尝试
   - 写入结果回记忆：这次尝试成功了还是失败了？发现了什么新证据？得到了什么新约束？
3. **收敛阶段**：随着记忆不断积累，搜索空间不断缩小，最终收敛到能触发漏洞的PoC

#### 这个机制带来的直接效果：

| 效果 | 说明 |
|---|---|
| **更少的重复探索** | 失败路径被明确记录，后来的尝试不会一直沿着同一条源码路径或输入变异方向走下去 |
| **负面证据被保留** | 没触发的尝试、格式错误的输入、走不通的路径——这些都变成了约束，而不是被丢弃 |
| **更快的PoC收敛** | 每一次PoC变异都可以继承之前发现的代码路径、输入格式约束和验证反馈 |
| **更低的长上下文负担** | 模型不需要每次都从头理解整个任务，只需要从结构化记忆中读取最相关的证据 |
| **预算内更高的有效实验密度** | 在4小时的时间限制内，花更多时间在新的、有用的假设上，而不是重复验证已知错误 |

这就是MopMonk和传统Agent最本质的区别：传统Agent是"上下文驱动"——每次都把所有对话历史塞进去，让模型自己从中找线索；MopMonk是"记忆驱动"——先把信息结构化整理好，模型只需要看结构化后的证据，然后做下一步决策。

---

#### 3.5.3 共享记忆下的多Agent并行探索

MopMonk的第三个核心设计是：**多个Agent并行探索，但共享同一套漏洞记忆**。

这不是简单的"堆Agent数量"——如果多个Agent各干各的，互相不通信，那只是把"一个人瞎试"变成了"一群人瞎试"，本质上还是在浪费资源。MopMonk的聪明之处在于：**所有Agent都读写同一个记忆池**。

#### 工作方式：

- 针对同一个漏洞描述，可以同时启动多个探索尝试
- 每个Agent从自己的角度切入：一个从crash栈回溯推导，一个从边界条件构造最小输入，一个分析数据流中的可疑函数
- 每个Agent在独立的沙盒里运行，互不干扰
- **但所有发现——失败的路径、可疑的代码段、候选PoC、验证结果——都写回到同一个共享记忆池**
- Agent A走过的死路，Agent B不需要再走一遍；Agent B发现的可疑线索，Agent C可以直接跟进

#### 共享记忆的价值：

你可以把这想象成一个侦探团队：多个侦探同时调查同一个案子，有人去问证人，有人去看监控，有人去查物证——但所有线索都汇总到同一个白板上，所有人都能看到最新的进展。这样的团队效率，当然比几个侦探各自闭门造车要高得多。

在4小时的超时限制下，这套共享记忆机制让每一轮探索都踩在前一轮的肩上。并行不是目的，**让每一次尝试都有信息增量**才是目的。

---

#### 三大技术的关系总结：

| 技术 | 角色 | 比喻 |
|---|---|---|
| 结构化漏洞记忆 | 基础 | 破案笔记本 |
| 记忆驱动挖掘 | 流程 | 基于证据的破案过程 |
| 共享记忆多Agent并行 | 组织方式 | 共享白板的侦探团队 |

三者结合起来，就把漏洞挖掘从"一个人瞎摸"变成了"一个有组织、有记录、有协作的团队破案"。
