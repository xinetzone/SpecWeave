## 二、核心观点摘要

1. **场景优先于榜单**：大模型进化到现在，早已不是看榜单的阶段。榜单综合排名有参考价值，但对日常使用远远不够。模型差异特别具体，应按使用场景选择——什么任务优先用谁，什么场景踩过坑，什么组合最省心。

2. **四款模型各有所长**：DeepSeek 开源便宜适合批量任务；Kimi K2.7 Code 稳定适合当编程替补；MiniMax M3 原生多模态强、1M 长上下文，适合处理杂乱资料；GLM 5.2 编程能力最强，但不好买到。

3. **价格梯度清晰**：价格从低到高为 DeepSeek → MiniMax M3 → Kimi K2.7 → GLM 5.2。Coding plan 中档套餐约 400 多元/月，足够三五个小项目日常维护。外部调用 DeepSeek 处理 500 份长文档约 1 元，M3 约 1 元处理 200 份大文件。

4. **信任是核心瓶颈**：阻碍信任国产模型的，现在不是能力差距，而是用户信任。国产模型能力正在一点点补齐短板，但真正还在慢慢补回来的是用户信任。**"能力是入场券，信任才是留下来的理由。"**

5. **信任余额积累**：Claude 和 OpenAI 陪用户走过太多真实工作流，即使不完美也帮用户扛住了活，这种东西会变成隐形的信任余额。国产模型需要更多的时间、更多的场景来积累信任。

## 五、四款模型详细评价

### 5.1 DeepSeek：开源、便宜、脑子活

**核心评价**：DeepSeek 是天然想站它的模型，开源、便宜、脑子活。

**优势**：
- 有时候像 Grok，会给出意料之外的灵光一闪，能提供新思路
- 日常跑批量任务比较多，尤其是大量文本处理、数据清洗、格式转换——又快又省钱
- 在便宜耐用这件事上，是国产模型里最让人放心的选择

**顾虑**：
- 不太敢把长任务交给它。短任务里是好搭子，但长任务里会不会变成"演员"？还没攒够信心去验证

**适用场景**：批量任务、文本处理、数据清洗、格式转换

### 5.2 Kimi K2.7 Code：相对稳定一些

**核心评价**：稳定性是其主要优势，模型能力相比 GLM 5.2 差一些。

**使用体验**：
- Coding plan 体感不错，没有限购、没有变化规则那么多幺蛾子
- 之前的 OpenClaw 小瑶机器人接的是 Kimi 2.6，那时候模型能力还不太足，有点笨，不太会用工具，执行力一般，有幻觉，胜在价格和稳定性
- 2.7 Code 出来后情况改观了一些，现在接在 Claude Code 里当编程替补，特别稳定，别家崩的时候可以拿来稳稳补位

**局限**：
- 仅限于此，再更复杂的长任务对它还是很谨慎
- 社区风评褒贬不一，有不少人说它是小号 Gemini

**适用场景**：编程替补、别家服务崩时的稳定补位

### 5.3 MiniMax M3：被低估的多面手

**核心评价**：四个里面最让人意外的模型，原生多模态能力真的不错。

**优势**：
- 原生多模态能力——图、PDF、视频都能处理
- 1M 长上下文，用起来很方便
- 永久半价后价格也划算
- 复杂前端任务（根据一大堆各种格式适配风格）交给 M3，比让 Claude 在后台写脚本逐个解析方便很多
- 速度现在明显快了不少（刚出时很慢，丢长文档进去要想很久才回）

**适用场景**：经常处理截图、表格、扫描件、音视频等多模态资料的工作场景

### 5.4 GLM 5.2：编程能力最强，但不太好买到

**核心评价**：四个里面编程能力最强的模型，社区好评多。

**优势**：
- coding 能力强，长上下文扎实
- 实测效果很不错，已经信任到敢把它拿过来协助删库了
- 很适合平替主 Agent 的主模型

**局限**：
- 幻觉率不低
- 没有多模态
- 不好买到：Coding plan 要在每天早上十点整排队抢购，相当难买，供不应求
- 编程套餐给的并发量一般，大规模多线程用起来可能有点慢，高峰期速度也慢
- API 价格不太适合拿来做批量任务

**适用场景**：主力 coding、平替主 Agent 主模型

## 七、专业术语表

| 术语 | 释义 |
|------|------|
| **Token plan** | 模型厂商提供的按月订阅套餐，用户支付固定费用后在一定额度内使用模型服务，适合日常固定频率使用场景 |
| **Coding plan** | 专为编程场景设计的 token plan 套餐，针对代码生成、代码补全等 coding 任务优化，通常有独立的并发限制和定价 |
| **多模态（Multi-modal）** | 模型能够同时处理多种类型输入（如文本、图像、PDF、视频、音频等）的能力，原生多模态指模型架构层面直接支持多模态输入 |
| **长上下文（Long Context）** | 模型能够一次性处理和记忆的输入长度上限，如 MiniMax M3 的 1M 长上下文表示可处理约 100 万 token 的输入，适合处理超长文档 |
| **幻觉率（Hallucination Rate）** | 模型生成看似合理但实际不正确或虚构内容的概率，是衡量模型可靠性的重要指标 |
| **高并发（High Concurrency）** | 同时发起大量请求的处理能力，批量任务需要高并发支持，token plan 通常并发有限，需用 API 方式实现 |
| **API（Application Programming Interface）** | 应用编程接口，此处指模型厂商提供的可通过程序调用的接口，按使用量计费，适合自动化和批量任务 |
| **原生多模态** | 模型在架构层面直接支持多模态输入处理，而非通过外部脚本或工具转换后输入，处理效率和准确性更高 |

## 八、信任问题洞察

### 8.1 核心金句

> **"能力是入场券，信任才是留下来的理由。"**

### 8.2 心理对比：国产模型 vs Claude/OpenAI

作者坦诚地剖析了一个普遍存在的心理偏见：

- **Claude/OpenAI 翻车时**：作者会自动给它们找补——会觉得是不是 prompt 没写清楚，或者上下文给得太乱
- **国产模型翻车时**：脑子里第一反应经常是——"果然"

这种差异不是不想支持国产模型，也不是没看到它们这两年的进步，而是有过几次不好的体验之后，就很难再轻易托付。

### 8.3 信任余额概念

作者提出了"信任余额"这一概念：

> 一路走来，Claude 和 OpenAI 陪作者走过太多真实工作流了。很多时候，知道它们不完美，但它们确实帮作者扛住了活。这种东西会变成一种很隐形的信任余额。

国产模型更像一个早就认识的一般朋友，一直没有留下特别突出的印象，即使现在已经比以前强很多，但以前有过一次不好的印象，再加上周围总有人传它的绯闻，作者还是会心存芥蒂。

### 8.4 测试场景偏见

更麻烦的是，当真正拿国产测试时，往往不是从一个干净的小任务开始，而是直接把它塞进 Claude 和 OpenAI 已经推进很远的复杂长任务里。在这种场景里，它一旦出错，依旧会直接归类成不靠谱。

### 8.5 作者结论

- 阻碍信任国产模型的，现在看来**不是能力差距**了
- 能力上的短板，国产模型正在一点点地补
- 前端能力普遍都很强，做出来视觉效果非常好看
- 真正还在慢慢补回来的是**用户信任**
- 国产模型需要更多的时间、更多的场景

## 九、信息价值评估

### 9.1 实用性：★★★★★（高）

文章最大价值在于提供了**场景化的模型选择方案**，而非泛泛的榜单排名。四类人群的划分清晰，推荐方案具体可操作，包含 token plan 还是 API、首选还是替补等细节决策。价格数据具体（400+ 元/月、1 元/500 份文档等），可直接用于成本估算。

### 9.2 时效性：★★★★☆（较高）

文章基于 2026 年 7 月最新模型版本（GLM 5.2、Kimi K2.7 Code、DeepSeek V4、MiniMax M3），时效性强。但 AI 模型迭代速度快，价格和能力评价可能在 3-6 个月后需要更新。GLM 5.2 的"抢购难"问题可能随产能扩展而缓解。

### 9.3 主观性：★★★☆☆（中等）

文章明确标注是"作为 AI 博主"的个人使用体验，评价基于作者实际工作流而非标准化测试。优点是真实可信，缺点是样本量为单人经验，可能存在个体差异。作者也坦诚承认对国产模型存在信任偏见，这份自省增加了可信度。

### 9.4 局限性：★★★☆☆（中等）

- **样本单一**：仅基于一位 AI 博主的使用经验，未覆盖更多行业场景
- **缺少量化数据**：编程能力、多模态能力等评价缺少 benchmark 数据支撑
- **GLM 5.2 评价受购买难度影响**：因抢购难导致的稀缺感可能放大了对其好用的评价
- **信任问题洞察偏感性**：虽有洞察力，但缺少可操作的信任建立方法论

## 十、与项目实践的关联建议

本项目（SpecWeave）是一个多智能体协作的 AI 开发项目，涉及文档生成、代码审查、自动化脚本等多种任务场景。文章的场景化模型选择思路与本项目的多角色智能体配置高度契合，具体建议如下：

### 10.1 按智能体角色匹配模型

| 智能体角色 | 对应文章场景 | 推荐模型 | 理由 |
|-----------|------------|---------|------|
| **代码生成/审查智能体** | 写代码人群 | GLM 5.2（首选）/ Kimi K2.7 Code（替补） | 编程能力最强，适合主力 coding；Kimi 稳定可作替补 |
| **文档撰写智能体** | 不写代码-文案类 | DeepSeek V4 Pro | 文案能力强，开源便宜，适合文档生成 |
| **多模态资料处理智能体** | 不写代码-多模态资料 | MiniMax M3 | 原生多模态，1M 长上下文，适合处理截图/PDF/扫描件 |
| **批量脚本执行智能体** | 高并发批量任务 | DeepSeek V4 Flash + MiniMax M3 API | API 方式高并发，价格合适 |

### 10.2 具体配置建议

1. **主力 coding 智能体**：若项目预算允许，优先采购 GLM 5.2 的 Coding plan（中档套餐约 400+ 元/月），用于 [AGENTS.md](../../../../AGENTS.md) 中定义的代码生成与审查角色。若抢购不到，退而使用 Kimi K2.7 Code 作为稳定替补。

2. **文档生成流水线**：本项目 `../../../docs/` 下有大量文档生成需求（知识库、复盘报告、学习笔记等），可使用 DeepSeek V4 Pro 网页版或 API 处理，成本低廉。

3. **多模态资料处理**：本项目 [微信公众号文章内容提取操作指南](../../operations/wechat-mp-content-extraction.md) 等场景涉及 HTML/截图处理，可引入 MiniMax M3 处理混杂格式资料，替代 Claude 后台写脚本逐个解析的方案。

4. **批量自动化任务**：`../../../.agents/scripts/` 下的验证脚本、CI 检查等批量任务，若需调用大模型，应使用 DeepSeek V4 Flash API，避免使用 token plan（并发不够）。

### 10.3 信任建设策略

文章的"信任余额"概念对本项目智能体配置有重要启示：

1. **从小任务开始**：新引入的国产模型智能体，应先从干净的小任务开始测试，而非直接塞进已推进很远的复杂长任务里（避免文章所述的"测试场景偏见"）
2. **建立信任日志**：记录国产模型智能体的每次执行结果，逐步积累信任余额
3. **多模型组合**：采用"主力 + 替补"模式（如 GLM 5.2 主力 + Kimi K2.7 Code 替补），既享受主力模型的能力，又有稳定兜底
4. **场景隔离**：将信任度不同的模型用于不同风险等级的任务，高信任度模型用于关键任务，低信任度模型用于实验性任务
