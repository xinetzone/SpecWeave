## 一、核心论点：瓶颈外移规律

AI 圈造新词的速度令人应接不暇：Prompt Engineering 刚整明白，Context Engineering 火了一年；2026 年开春 Harness Engineering 被推上风口；到了六月，Loop Engineering 又冒了出来。表面看是四个互相换皮的赛道，实质上是**同一条线上的四个路标**。

这条线有一个特别清晰、也特别反直觉的规律：

> **每当模型变强一截，整个系统的瓶颈，就被迫往外移一层。**

外移的四层路径清晰可辨：

1. **怎么说**（Prompt）—— 卡在你如何向模型表达意图
2. **给什么**（Context）—— 卡在你向模型供给什么信息
3. **干活环境**（Harness）—— 卡在模型之外的那一整套世界
4. **你自己**（Loop）—— 卡在还需要人肉驱动按回车的人身上

从你说的那句话，到你喂的那堆料，到它干活的环境，最后——移到了你本人身上。这条线上最关键的一跃，是 **harness**：它第一次把工程的重心，从"调教模型"彻底搬到了"设计模型外面的世界"。

搞懂了这一跃，不仅能看穿这四个词的关系，还能想明白一个让很多人困惑的现象：**为什么同样用 GPT、用 Claude，有的团队做出来的 Agent 又稳又能打，到别人手里却一跑就崩。**

## 二、第一站：Prompt Engineering——瓶颈卡在"你怎么说"

### 1. 模型的本质：预测，而非思考

把大模型那层外壳扒掉，里面其实就一个东西：一个巨大的参数文件，干的活只有一件——**根据你输入的内容，预测下一个字最该是什么**。

注意是"预测"，不是"思考"。它本质上是在顺着你给的方向往下猜：

- 你给的输入越模糊，它猜的范围越散
- 你给的输入越具体，它猜得越准

### 2. 瓶颈的起点：模型不是不会，是你没说明白

最早，瓶颈卡在一个最朴素的地方：**模型不是不会，是你没把话说明白**。

举例对照：

| 表达方式 | 输入 | 模型输出 |
|---------|------|---------|
| 模糊问法 | "加个排序" | 一段孤零零的排序片段，不知道放哪、要不要动别的函数 |
| 具体问法 | "这是我的完整代码，请给 users 列表按年龄从大到小排序，保留原有所有逻辑，输出完整代码" | 靠谱的完整代码 |

同一件事，换个问法，效果差十倍。

### 3. 提示词配方六要素

于是大家摸出一套"提示词配方"，让模型稳定吐出你要的东西：

1. **角色设定** —— 让模型知道自己是谁
2. **背景** —— 提供任务上下文
3. **参考资料** —— 给出可依据的信息
4. **明确任务** —— 清晰描述要做什么
5. **约束** —— 划定边界与禁忌
6. **输出格式** —— 规定产出的结构

把这些组装好，让模型稳定吐出你要的东西——这就是 **Prompt Engineering**。它解决的，是"怎么说"。

### 4. 适用边界与瓶颈外移的触发条件

在聊天、翻译、写文案这种"一问一答"的短链路任务里，这一招够用了。但很快，人们想干的事变复杂了，这一招就撑不住了。

**触发条件**：你让它"分析我们公司去年的财报"，它再会说话，没见过你的财报，能分析啥？提示词擅长的是把任务表达清楚，但它变不出模型不知道的事实。

瓶颈，被顶到了下一层。

## 三、第二站：Context Engineering——瓶颈移到"你给什么"

### 1. Agent 火起后：从"回答问题"到"干活"

Agent 火起来之后，模型不再只是"回答问题"，而是真要进到环境里"干活"了：

- 多轮对话
- 调工具
- 查数据库
- 在多个步骤间传递中间结果

这时候系统面对的，已经不是"这一次答得对不对"，而是"**整条链路能不能跑通**"。

### 2. 上下文（Context）的完整定义

要跑通，模型得拿到一大堆东西：

- 当前文档
- 历史记录
- 相关规范
- 任务目标
- 之前的中间结论

这些加起来，才叫一个完整的"任务环境"。**每一次发给模型的全部信息加在一起，就叫上下文（Context）。提示词，只是其中很小的一块。**

于是瓶颈从"怎么说"移到了"给什么"：怎么在对的时机，把对的信息送进去。这就是 **Context Engineering**。

### 3. 上下文窗口：短期记忆容量

这里有个要命的现实：模型一次能装的上下文是有上限的，这个上限叫**"上下文窗口"**，你可以理解成它的短期记忆容量。

更麻烦的是，就算窗口够大，你塞太满，它的注意力反而会涣散——Anthropic 给这个现象起了个名字，叫 **context rot（上下文腐化）**：

- 东西一多，模型开始记不住前面
- 开始前后矛盾
- 开始忽略你最初定的规则

像一个被信息淹没的人，你给他越多要看的，他越抓不住重点。

### 4. 核心要义：给得准，而非给得多

所以 Context Engineering 的核心，不是"给得多"，而是"**给得准**"。

Anthropic 那套 Agent Skills 的"渐进式披露"就是这个理念的极致：

> 平时只给模型看一份能力目录，等它判断真要用某个能力了，才把详细说明动态加载进来。

**上下文优化的本质从来不是给得更多，是该给的时候给，不该给的时候藏起来。**

但你把上下文管到极致之后，又撞上了一堵新墙。

## 四、第三站：Harness Engineering——瓶颈移到"它干活的环境"（重点章节）

> 本章是全文的核心。Harness 是这条演进线上"最关键的一跃"——它第一次把工程重心从"调教模型"搬到了"设计模型外面的世界"。

### 1. 长链路任务的跑偏现象

到了 2026 年初，模型已经能连续干好几个小时的活了。这时候你会发现一件扎心的事：**提示词没毛病、上下文也没毛病，可它在长链路任务里，还是会跑偏**。

典型表现：

- 计划做得好好的，执行着突然偏了
- 工具调对了，结果理解错了
- 干着干着，忘了自己最初要干啥

### 2. 问题性质的转变：从输入侧到执行侧

你品品这个问题的性质：

- **提示词优化的是"意图的表达"**
- **上下文优化的是"信息的供给"**

这俩其实都还停留在**输入侧**，都还在"伺候模型把这一步想好"。但当模型开始连续行动，一个全新的问题冒出来了：

- 谁来监督它？
- 谁在它跑偏时把它拉回来？
- 谁在它摔倒后扶它起来？

这就是 harness 要解决的。

### 3. Harness 词源：马具、缰绳

Harness 直译是"**马具、缰绳**"。

马本身有力量，但没有缰绳，这股力量就是失控的——可能冲下悬崖，可能甩你下来。马具的作用，是**让这股力量为你所用**。

这个词不是营销造的。最早把它点响的，是 Terraform 的作者 Mitchell Hashimoto。

### 4. Mitchell Hashimoto 的定义原话

Mitchell Hashimoto 在一篇讲自己怎么从"不信 AI"到"每天让 Agent 干活"的博客里，给出了一个朴素到极致的定义：

> "每次你发现 agent 犯了一个错，就花时间去工程化一个解决方案，让它再也不可能犯同样的错。"
>
> —— Mitchell Hashimoto（Terraform 作者）

你品这个思路的狠劲。绝大多数人遇到 Agent 犯错，是骂两句、手动改掉、祈祷它下次别再犯。而 Hashimoto 是每次都停下来问：**我能不能把这个错永久地修进环境里，让它在结构上就不可能再犯？**

可能是给规则文件加一条，可能是补一个测试，可能是挂一个钩子。关键是——**这个修补必须沉淀到环境里，而不是留在你脑子里**。

### 5. 复利效应机制

这套做法的威力在于它是"**复利**"的：

```
每犯一次错
   ↓
环境就强一点
   ↓
环境强一点，下次就更少犯错
   ↓
犯得少，你改进得就更快
   ↓
时间一长，harness 越来越坚固
   ↓
Agent 在你这个项目里越跑越稳
```

这是 harness 区别于前两站的根本特征：**它是积累性的、是复利的**。Prompt 和 Context 的优化多是即时的、一次性的；而 harness 的改进会沉淀下来，持续产生收益。

### 6. 关键一跃：从"伺候模型"到"设计模型外面的世界"

看到这你应该回过味来了：**这就是那一跃**。

- 前两站（prompt、context）再怎么折腾，重心都在"怎么把模型这一步伺候好"
- harness 第一次把整个工程的重心，搬到了"**模型之外的那一整套世界**"

这是范式级别的转变。它意味着工程师的精力分配发生了根本变化：从琢磨"怎么跟模型说话"，转向"怎么搭建一个让模型不容易出错的环境"。

### 7. Agent = 模型 + Harness 公式

圈子里那个流传很广的公式，把这件事说得最干脆：

> **Agent = 模型 + Harness**

翻译成人话：**一个 Agent 系统里，除了模型本身，几乎所有决定它能不能稳定交付的东西，都属于 harness。**

- 模型决定上限
- harness 决定它能不能真正落地

### 8. 解释开头那个困惑：同模型差几倍

这也就解释了开头那个困惑——同样的模型，效果差几倍，**差的从来不是模型，是外面这层 harness 搭得到不到位**。

为什么同样用 GPT、用 Claude，有的团队做出来的 Agent 又稳又能打，到别人手里却一跑就崩？答案不在模型选型，而在 harness 工程：

- 强团队的 harness 经过了无数次复利式修补
- 弱团队的 harness 可能只是零散的几条 prompt

差距是累积出来的，不是一次选型决定的。

### 9. 层层包含关系：不是取代，是装下

而且要强调一句：**这三站不是谁取代谁，是层层包含**。

```
Harness（最外层）
   └── Context（中间层）
         └── Prompt（最内层）
```

- **Prompt 是 Context 的一部分** —— 提示词只是上下文中的一小块
- **Context 是 Harness 的一部分** —— 上下文管理是 harness 工程的一个环节
- 你做 harness 的时候，里面一定包着上下文工程，上下文工程里又一定包着提示词工程

**Harness 不是来取代前两个的，它是站在更大的系统视角，把它们都装了进去。**

## 五、第四站：Loop Engineering——瓶颈移到"你自己身上"

### 1. 最后一环瓶颈：人肉驱动

现在，假设你把 harness 修炼到家了：

- 模型的环境被驯得服服帖帖
- 工具趁手
- 状态不丢
- 错了能兜底

那这条流水线上，最后还剩一个瓶颈是谁？**是你**。

- 模型在等你布置任务
- harness 在等你启动
- 材料在等你投喂

整条线上唯一还需要人肉驱动的环节，就是你坐在屏幕前，敲下那条 prompt、按下那个回车。**你睡觉，它就停工。**

Loop Engineering 瞄准的，就是这最后一环。

### 2. 松手理念：设计循环而非手动 prompt

这个词几乎是同一周被三个人同时点响的：

> "你不该再给 agent 打 prompt 了，你该去设计那个给 agent 打 prompt 的循环。"
>
> —— Peter Steinberger（OpenClaw 作者）

> （Addy Osmani 给它写了正式定义）
>
> —— Addy Osmani（Google AI 总监）

> "我已经不 prompt Claude 了，是 loop 在运行着 prompt Claude，我的工作是写 loop。"
>
> —— Boris Cherny（Claude Code 创始人）

说人话：过去你跟 Agent 是回合制的——你写一条、读一条、再写一条，全程握着它，一回合都不能松手。

Loop 说的是，**松手吧**——你把"发现任务、布置任务、检查结果、决定下一步"这套流程，设计成一个能自己转的循环，然后让循环去握着 Agent。

### 3. Loop 是 Harness 逻辑的自然延伸

你发现没有？**loop 根本不是 harness 的替代品，它是 harness 逻辑的自然延伸**。

- harness 把"模型的世界"设计好了
- loop 再往外走一步，连"按回车的你"也设计进了这个世界

瓶颈的外移，到这里走到了尽头。

### 4. 瓶颈移到人身上的真相：不是躺平，是更深的卷入

但当瓶颈移到你身上，你以为可以躺平了？**真相恰恰相反**。

两个关键事实：

**事实一：loop 改变了你的工作，但没有把你从工作里删除。**

你不再手动按回车，但你需要设计循环、监督循环、判断循环的产出。工作性质变了，但你依然是不可或缺的环节。

**事实二：harness 编码的是"模型此刻还做不到什么"的假设；模型一变强，这些假设就会过期。**

所以驾驭 AI 这件事，从来不是一劳永逸地"搭好一套"，而是一边盯着模型的边界变化、一边持续地修你那套环境。

写代码、敲提示词这些活，会越来越多地交给 Agent。但**设计一个能让 Agent 高质量产出的环境、并对它的产出保持判断**，这件事在很长一段时间里，都得是人来做。

这，可能就是接下来很多年里，**一个工程师真正的主战场**。

> **模型决定了 Agent 的上限。而你在它外面搭的那套东西、以及你有没有守住自己的判断，决定了这个上限有多少能真正落到实处。**

## 六、四者关系总结

### 1. 层层包含关系

四者不是平行关系，不是取代关系，而是**层层包含**：

| 关系 | 内涵 |
|------|------|
| Prompt ⊂ Context | 提示词是上下文的一小块 |
| Context ⊂ Harness | 上下文管理是 harness 工程的一个环节 |
| Harness ⊂ Loop（延伸） | Loop 是 harness 逻辑的自然延伸，把"按回车的你"也设计进来 |

### 2. 瓶颈外移的终点

外移路径：

```
怎么说（Prompt） → 给什么（Context） → 干活环境（Harness） → 你自己（Loop）
   人的表达        人的供给            模型外的世界           人肉驱动
```

**瓶颈外移，到这里走到了尽头——回到了你自己身上。**

### 3. 核心结论

- **模型决定上限**
- **harness 决定它能不能真正落地**
- **loop 决定它能不能持续运转**
- **人的判断决定这个上限有多少能真正落到实处**

四者之中，**Harness 是最关键的一跃**：它把工程重心从"调教模型"搬到了"设计模型外面的世界"。理解了这一跃，就理解了为什么 Agent 工程的核心不在模型选型，而在环境搭建。

**背景**：Boris Cherny 用一句更狠的话总结了 loop 工程的实践状态——他作为 Claude Code 的创始人，已经把日常工作从"手动 prompt"完全转向了"编写循环"。这句话最具冲击力的地方在于，连 Claude Code 的创始人都不再手动 prompt 了，而是让 loop 去运行 prompt。

## 八、实践启示（基于原文的延伸思考）

> **声明**：本章为基于原文的延伸思考，旨在将原文观点萃取为可执行启示。以下启示均源自原文论述，但具体的可执行表述为笔者基于原文的提炼与延伸。

### 启示一：复利式修补——把每个错误都变成环境的加固

原文引用的 Mitchell Hashimoto 定义揭示了一个核心方法论：**每次 Agent 犯错，都是一次永久改进环境的机会**。

可执行做法：

- 遇到 Agent 犯错时，不要只是手动改掉
- 停下来问：我能不能把这个错永久地修进环境里？
- 修补方式可以是：给规则文件加一条、补一个测试、挂一个钩子
- 关键是：**修补必须沉淀到环境里，而不是留在脑子里**

这是复利的：每犯一次错，环境就强一点；环境强一点，下次就更少犯错。长期下来，harness 会越来越坚固。

### 启示二：环境沉淀而非脑内记忆——把经验外化为结构

原文强调 harness 的修补"必须沉淀到环境里，而不是留在你脑子里"。这条原则可延伸为更普遍的工程纪律：

- 凡是只存在于你脑子里的经验，都是不可复现的
- 凡是不可复现的经验，都无法积累
- 把经验外化为：规则文件、测试用例、自动化钩子、文档
- 这样即便你不在，Agent 也能在加固好的环境里稳定运行

脑内记忆是会衰减的，环境沉淀是会复利的。

### 启示三：持续修环境 + 盯模型边界——驾驭 AI 是动态过程

原文明确指出：**harness 编码的，是"模型此刻还做不到什么"的假设；模型一变强，这些假设就会过期**。

这意味着：

- 驾驭 AI 不是一劳永逸地"搭好一套"
- 而是一边盯着模型的边界变化、一边持续地修你那套环境
- 模型升级后，要重新审视 harness 中哪些假设已经过期
- 过期的假设要么删除（模型已经能做了），要么升级（边界变了）

这是一项持续性的工作，而不是一次性的工程。

### 启示四：人的判断仍是主战场——守住不可替代的那一环

原文最后强调：**设计一个能让 Agent 高质量产出的环境、并对它的产出保持判断，这件事在很长一段时间里，都得是人来做**。

这意味着工程师的角色正在发生转变：

| 过去 | 现在/未来 |
|------|----------|
| 写代码 | 设计让 Agent 高质量产出代码的环境 |
| 敲提示词 | 设计给 Agent 打 prompt 的循环 |
| 手动检查每一步 | 守住对最终产出的判断 |
| 回合制操作 | 设计并监督自动循环 |

**这，可能就是接下来很多年里，一个工程师真正的主战场。**

## 九、常见问题解答（FAQ）

### Q1：这四个 Engineering 是平行的四个赛道吗？

**不是**。它们是同一条线上四个路标，沿着"瓶颈外移"的脉络串联：Prompt（怎么说）→ Context（给什么）→ Harness（干活环境）→ Loop（你自己）。四者之间是**层层包含**关系：Prompt ⊂ Context ⊂ Harness，Loop 是 Harness 逻辑的自然延伸。

### Q2：这四个是谁取代谁的关系吗？

**不是取代，是包含**。原文明确强调：

- Prompt 是 Context 的一部分
- Context 是 Harness 的一部分
- 你做 harness 的时候，里面一定包着上下文工程，上下文工程里又一定包着提示词工程

Harness 不是来取代前两个的，它是站在更大的系统视角，把它们都装了进去。

### Q3：为什么同样用 GPT/Claude，有的团队 Agent 又稳又能打，有的却一跑就崩？

**差的从来不是模型，是外面这层 harness 搭得到不到位**。原文给出的公式是：

> Agent = 模型 + Harness

模型决定上限，harness 决定它能不能真正落地。强团队的 harness 经过了无数次复利式修补，弱团队的 harness 可能只是零散的几条 prompt。差距是累积出来的。

### Q4：Harness Engineering 怎么落地？具体步骤是什么？

基于 Mitchell Hashimoto 的定义，落地步骤是：

1. 让 Agent 干活
2. 发现它犯了一个错
3. 停下来，不要只是手动改掉
4. 问：我能不能把这个错永久地修进环境里？
5. 选择修补方式：给规则文件加一条 / 补一个测试 / 挂一个钩子
6. 确保修补沉淀到环境里，而不是留在脑子里
7. 长期重复，享受复利效应

### Q5：Loop Engineering 和传统的自动化有什么区别？

Loop Engineering 不是简单的"自动化脚本"。它的核心区别在于：

- **传统自动化**：人写脚本，脚本执行固定流程
- **Loop Engineering**：人设计"发现任务、布置任务、检查结果、决定下一步"的循环，让循环去握着 Agent

Boris Cherny（Claude Code 创始人）的原话最能说明区别：

> "我已经不 prompt Claude 了，是 loop 在运行着 prompt Claude，我的工作是写 loop。"

人不再回合制地操作 Agent，而是设计让 Agent 自己转起来的循环。

### Q6：瓶颈外移到"你自己身上"后，人是不是可以躺平了？

**恰恰相反**。原文明确指出两个真相：

1. loop 改变了你的工作，但没有把你从工作里删除——你需要设计循环、监督循环、判断产出
2. harness 编码的是"模型此刻还做不到什么"的假设，模型一变强这些假设就会过期——你需要持续盯边界、修环境

驾驭 AI 是动态过程，不是一劳永逸。**设计环境并守住判断，是工程师接下来很多年的主战场。**

### Q7：Prompt Engineering 是不是过时了？

**没有过时**。它依然是 Context Engineering 的一部分，而 Context Engineering 又是 Harness Engineering 的一部分。你做 harness 时，里面一定包着上下文工程，上下文工程里又一定包着提示词工程。只是它的地位从"全部"降为了"基础一环"。
