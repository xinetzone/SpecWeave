## 六、最佳实践

### 6.1 从真实专业知识出发

**常见陷阱**：让 LLM 在没有领域特定上下文的情况下生成技能——仅依赖 LLM 的通用训练知识。结果是模糊的、通用的过程（"适当处理错误"、"遵循身份验证最佳实践"），而不是使技能有价值的特定 API 模式、边界情况和项目约定。

**有效方法**：

#### 方法一：从实际任务提取
在与智能体的对话中完成一个真实任务，沿途提供上下文、纠正和偏好。然后将可复用模式提取为技能。注意：
- **有效的步骤**：导致成功的操作序列
- **你做出的纠正**：你引导智能体方法的地方
- **输入/输出格式**：数据进出的样子
- **你提供的上下文**：智能体不知道的项目特定事实、约定或约束

#### 方法二：从现有项目产物综合
当你有大量现有知识时，可以将其输入 LLM 并要求它综合成一个技能。好的源材料包括：
- 内部文档、Runbook、风格指南
- API 规范、Schema、配置文件
- 代码审查评论和问题追踪器（捕获反复出现的问题和审查者期望）
- 版本控制历史，特别是补丁和修复（通过实际更改的内容揭示模式）
- 真实失败案例及其解决方案

### 6.2 通过实际执行精炼

技能的第一稿通常需要精炼。针对真实任务运行技能，然后将结果——所有结果，而不仅仅是失败——反馈到创建过程中。问：什么触发了误报？遗漏了什么？什么可以砍掉？

即使一次执行-然后-修订也能显著提高质量，复杂领域通常受益于多次迭代。

**阅读智能体执行轨迹**，而不仅仅是最终输出。如果智能体在非生产性步骤上浪费时间，常见原因包括：指令太模糊（智能体在找到有效方法之前尝试多种方法）、指令不适用于当前任务（智能体无论如何都遵循它们），或者呈现了太多选项而没有明确的默认值。

### 6.3 明智使用上下文

技能激活后，其完整的 `SKILL.md` 正文会与对话历史、系统上下文和其他活动技能一起加载到智能体的上下文窗口中。技能中的每个 token 都与该窗口中的其他所有内容竞争智能体的注意力。

#### 补充智能体缺乏的，省略已知的
专注于没有你的技能智能体*不会*知道的内容：项目特定约定、领域特定过程、非显而易见的边界情况，以及要使用的特定工具或 API。你不需要解释什么是 PDF、HTTP 如何工作，或者数据库迁移是做什么的。

**反模式**：
```markdown
## 提取 PDF 文本
<!-- 太冗长——智能体已经知道 PDF 是什么 -->
PDF（便携式文档格式）文件是一种常见的文件格式，包含文本、图像和其他内容。
要从 PDF 中提取文本，你需要使用一个库。推荐使用 pdfplumber，因为它能很好地处理大多数情况。
```

**推荐模式**：
```markdown
## 提取 PDF 文本
使用 pdfplumber 进行文本提取。对于扫描文档，回退到 pdf2image + pytesseract。

```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
```

对于每一段内容，问自己："没有这条指令，智能体会不会做错？"如果答案是否定的，就砍掉它。

#### 设计内聚单元
决定技能应该覆盖什么就像决定函数应该做什么：你希望它封装一个内聚的工作单元，与其他技能良好组合。范围过窄的技能迫使单个任务加载多个技能，增加开销和指令冲突风险。范围过宽的技能变得难以精确激活。

#### 适度细节
过于全面的技能可能弊大于利——智能体难以提取相关内容，并可能被不适用于当前任务的指令触发非生产性路径。简洁的分步指南加上可用示例往往优于详尽文档。当你发现自己覆盖每个边界情况时，考虑大多数是否最好由智能体自己的判断来处理。

#### 大技能使用渐进式披露
当技能确实需要更多内容时，将详细参考资料移至 `references/` 或类似目录中的单独文件。关键是告诉智能体*何时*加载每个文件："如果 API 返回非 200 状态码，读取 `references/api-errors.md`"比通用的"详见 references/"更有用。

### 6.4 校准控制粒度

并非技能的每个部分都需要相同级别的规范性。将指令的特异性与任务的脆弱性匹配。

#### 匹配特异性与脆弱性

**给智能体自由**：当多种方法有效且任务容忍变化时。对于灵活的指令，解释*为什么*可能比刚性指令更有效——理解指令背后目的的智能体会做出更好的上下文相关决策。

```markdown
## 代码审查流程
1. 检查所有数据库查询是否存在 SQL 注入（使用参数化查询）
2. 验证每个端点的身份验证检查
3. 查找并发代码路径中的竞态条件
4. 确认错误消息不泄露内部细节
```

**要规范性**：当操作脆弱、一致性很重要，或者必须遵循特定序列时：

````markdown
## 数据库迁移
严格按此顺序运行：
```bash
python scripts/migrate.py --verify --backup
```
不要修改命令或添加额外标志。
````

#### 提供默认值，而非菜单
当多种工具或方法可能有效时，选择一个默认值并简要提及替代方案，而不是将它们呈现为平等选项。

**反模式**：
```markdown
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...
```

**推荐模式**：
```markdown
使用 pdfplumber 进行文本提取：
```python
import pdfplumber
```
对于需要 OCR 的扫描 PDF，改用 pdf2image + pytesseract。
```

#### 偏好过程而非声明
技能应该教智能体*如何处理*一类问题，而不是*为特定实例生产什么*。

**特定答案**（仅对这个确切任务有用）：
```
将 `orders` 表连接到 `customers` 表的 `customer_id`，过滤 `region = 'EMEA'`，并对 `amount` 列求和。
```

**可复用方法**（适用于任何分析查询）：
```markdown
1. 从 `references/schema.yaml` 读取 schema 以找到相关表
2. 使用 `_id` 外键约定连接表
3. 将用户请求中的任何过滤器应用为 WHERE 子句
4. 根据需要聚合数字列并格式化为 markdown 表格
```

### 6.5 有效指令模式

这些是构建技能内容的可复用技术。并非每个技能都需要所有这些——使用适合你任务的模式。

#### Gotchas 章节
许多技能中最高价值的内容是 gotchas 列表——违反合理假设的环境特定事实。这些不是通用建议（"适当处理错误"），而是对智能体在没有被告知的情况下会犯的错误的具体纠正：

```markdown
## Gotchas
- `users` 表使用软删除。查询必须包含 `WHERE deleted_at IS NULL`，否则结果将包含已停用账户。
- 用户 ID 在数据库中是 `user_id`，在认证服务中是 `uid`，在计费 API 中是 `accountId`。三者指的是同一个值。
- `/health` 端点只要 Web 服务器在运行就返回 200，即使数据库连接已关闭。使用 `/ready` 检查完整服务健康。
```

将 gotchas 保留在 `SKILL.md` 中，智能体在遇到情况之前会阅读它们。单独的参考文件可以工作，但对于非显而易见的问题，智能体可能无法识别触发条件。

当智能体犯了你必须纠正的错误时，将纠正添加到 gotchas 章节。这是迭代改进技能最直接的方法之一。

#### 输出格式模板
当你需要智能体以特定格式产生输出时，提供一个模板。这比用散文描述格式更可靠，因为智能体对具体结构的模式匹配很好。短模板可以直接放在 `SKILL.md` 中；对于较长的模板或仅在特定情况下需要的模板，存储在 `assets/` 中并从 `SKILL.md` 引用。

````markdown
## 报告结构
使用此模板，根据特定分析调整章节：
```markdown
# [分析标题]
## 执行摘要
[关键发现的一段概述]
## 关键发现
- 发现 1 及支持数据
- 发现 2 及支持数据
## 建议
1. 具体可操作建议
2. 具体可操作建议
```
````

#### 多步工作流检查清单
显式检查清单帮助智能体跟踪进度并避免跳过步骤，特别是当步骤有依赖关系或验证关卡时。

```markdown
## 表单处理工作流
进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填写表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：验证输出（运行 `scripts/verify_output.py`）
```

#### 验证循环
指示智能体在继续之前验证自己的工作。模式是：做工作，运行验证器（脚本、参考检查清单或自检），修复任何问题，重复直到验证通过。

```markdown
## 编辑工作流
1. 进行编辑
2. 运行验证：`python scripts/validate.py output/`
3. 如果验证失败：
   - 查看错误消息
   - 修复问题
   - 再次运行验证
4. 仅在验证通过后继续
```

参考文档也可以作为"验证器"——指示智能体在最终确定之前对照参考检查其工作。

#### Plan-Validate-Execute
对于批量或破坏性操作，让智能体以结构化格式创建中间计划，对照真实源进行验证，然后才执行。

```markdown
## PDF 表单填写
1. 提取表单字段：`python scripts/analyze_form.py input.pdf` → `form_fields.json`
   （列出每个字段名称、类型以及是否必填）
2. 创建 `field_values.json`，将每个字段名称映射到其预期值
3. 验证：`python scripts/validate_fields.py form_fields.json field_values.json`
   （检查每个字段名称是否存在于表单中，类型是否兼容，必填字段是否不缺失）
4. 如果验证失败，修改 `field_values.json` 并重新验证
5. 填写表单：`python scripts/fill_form.py input.pdf field_values.json output.pdf`
```

关键要素是步骤 3：一个验证脚本，对照真实源（`form_fields.json`）检查计划（`field_values.json`）。

#### 捆绑可复用脚本
在迭代技能时，跨测试用例比较智能体的执行轨迹。如果你注意到智能体每次运行都独立地重新发明相同的逻辑——构建图表、解析特定格式、验证输出——这就是一个信号：编写一个经过测试的脚本一次并捆绑在 `scripts/` 中。
