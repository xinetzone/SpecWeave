# 对抗性审查：核心术语表

---

## 1. 说明

本术语表系统梳理对抗性审查方法论涉及的核心概念，覆盖核心概念、来源可信度、认知偏差、AI协作/代码审查、标准工具、方法论六大类。

**使用方法**：
- 术语表按主题分类组织，而非严格按拼音排序
- 每个术语标注可信度评级：🟢A级（多来源交叉验证）、🔵B级（单一权威来源）
- 相关概念使用链接指向其他文档对应章节
- 易混淆点在"备注"栏明确标注

## 2. 核心术语表

### 2.1 核心概念类

| 术语（中文） | 术语（英文） | 定义 | 备注/可信度 |
|-------------|-------------|------|------------|
| 对抗性审查 | Adversarial Review | 主动站在攻击者、破坏者、质疑者的视角，通过构造极端场景、寻找反例、识别逻辑漏洞、模拟恶意输入等方式，系统性地发现被审查对象中隐藏的缺陷、边界问题和认知偏差的质量验证方法论，核心导向是"证明对象有问题"直到无法证伪 | 🟢A级；本知识库核心概念，区别于传统审查"证明正确"的思维导向 |
| 证伪主义 | Falsificationism | 卡尔·波普尔提出的科学哲学理论，认为科学理论不能被证实，只能被证伪；一个理论的科学性在于它是可证伪的，即存在可以被经验反驳的可能性 | 🟢A级；对抗性审查的哲学基础，源自[02-philosophy-origins.md](02-philosophy-origins.md) |
| 红队测试 | Red Teaming | 通过模拟对手/敌方视角对系统、组织或流程进行全链路攻击测试，以检测防御体系有效性的安全评估方法；传统上聚焦安全领域 | 🟢A级；对抗性审查在安全领域的specialized应用分支，覆盖范围更广 |
| 攻击者视角 | Attacker Perspective | 不是"检查有没有错"，而是"怎么才能搞破坏"的思维模式；主动站在恶意用户、破坏者的角度思考问题，专门寻找正常路径走不到的极端场景 | 🟢A级；对抗性审查四大核心原则之一 |
| 质量内建 | Quality Built-in | 将质量保证活动内建到开发/研究过程中，而非等到上线前或出问题后再检查；小步快跑，小模块完成后立刻做小范围对抗审查 | 🟢A级；对抗性审查八大核心原则之一，源自精益制造思想 |
| 自举验证 | Bootstrap Verification | 用对抗性审查方法自身来构建对抗性审查知识库；所有内容均经过五维验证流程审查，主动寻找反证而非堆砌支持性证据 | 🟢A级；本知识库最大特色，[00-overview.md](00-overview.md)明确说明 |

### 2.2 来源与可信度类

| 术语（中文） | 术语（英文） | 定义 | 备注/可信度 |
|-------------|-------------|------|------------|
| 一级来源 | Primary Source | Gold Standard：同行评审的学术论文、权威标准组织官方文档、官方发布的法规文本、一手实验数据、原始档案 | 🟢A级；可信度最高，可作为核心论据 |
| 二级来源 | Secondary Source | Silver Standard：专业机构发布的报告、深度调查报道、权威厂商的技术文档、领域专家的系统性综述 | 🔵B级；可作为辅助论据，鼓励交叉验证 |
| 三级来源 | Tertiary Source | Bronze Standard：个人博客、自媒体文章、论坛讨论、未经验证的传言；仅可作为线索，不可作为独立论据 | 🟡C级；仅作参考，必须标注"待验证" |
| 可信度分级 | Credibility Rating | 四级可信度评级体系：🟢A级（多源交叉验证）、🔵B级（单一权威）、🟡C级（待验证）、🔴D级（存疑/排除） | 🟢A级；详见[00-overview.md#3-可信度评级说明](00-overview.md#3-可信度评级说明) |
| 交叉验证 | Cross-validation | 关键事实必须经至少2个独立来源验证，单一来源的论断即使来自权威也不直接作为核心论据 | 🟢A级；五维验证流程核心环节 |
| 三角验证 | Triangulation | 通过多种不同方法、多个不同来源、多个不同视角对同一事实进行验证，当多维度证据指向同一结论时可信度大幅提升 | 🟢A级；比简单交叉验证更严格的验证方法 |
| ⚠️待验证 | Pending Verification | 内容尚未完成多来源交叉验证，需进一步追溯确认的标记 | 🟢A级；异常标记之一 |
| ❓存疑 | Questionable | 内容与其他权威来源存在矛盾，或逻辑上存在疑问的标记 | 🟢A级；异常标记之二 |
| ⚖️争议 | Disputed | 学术/工程领域内存在明确学派争议，各方均有论据的标记 | 🟢A级；异常标记之三 |
| 🔍利益冲突 | Conflict of Interest | 来源/作者与结论存在商业、职业或立场关联，可能影响客观性的标记 | 🟢A级；异常标记之四 |

### 2.3 认知偏差类

| 术语（中文） | 术语（英文） | 定义 | 备注/可信度 |
|-------------|-------------|------|------------|
| 确认偏差 | Confirmation Bias | 人们倾向于寻找、解释和记忆那些能够证实自己已有信念的信息，而忽视或贬低与自己信念矛盾的证据 | 🟢A级；最普遍、危害最大的认知偏差，对抗性审查首要防御对象 |
| 权威偏差 | Authority Bias | 过度相信权威人士/机构的观点，仅仅因为对方的身份地位就接受其论断，而不追问证据和逻辑 | 🟢A级；又称"权威崇拜偏差"，常见于知识研究场景 |
| 锚定效应 | Anchoring Effect | 人们在做判断时过度依赖最先获得的信息（锚点），后续判断会被第一个信息严重影响 | 🟢A级；常见于代码审查中第一印象主导后续判断 |
| 群体思维 | Groupthink | 在群体决策中，为了维护群体和谐或共识，成员会自我审查并压制不同意见，导致不合理的决策结果 | 🟢A级；团队审查中需警惕的偏差 |
| 幸存者偏差 | Survivorship Bias | 只看到经过某种筛选而产生的结果（"幸存者"），而没有意识到筛选过程，忽略了被筛选掉的关键信息（"未幸存者"） | 🟢A级；经典案例：二战飞机装甲研究 |
| 事后归因偏差 | Hindsight Bias | 事后诸葛亮：事情发生后觉得自己"早就知道会这样"，高估自己事前的预测能力，也容易简单归因 | 🟢A级；又称"我早就知道"效应 |
| 晕轮效应 | Halo Effect | 对一个人/事物的某一方面有好印象，就倾向于认为其他方面也都好；反之亦然（恶魔效应） | 🟢A级；代码审查中容易因为某段代码写得好就忽略其他问题 |
| 可得性启发 | Availability Heuristic | 人们判断事件发生概率时，依赖容易想到的例子而非真实统计数据；越容易回忆起来的事情被认为越常见 | 🟢A级；容易导致对近期发生的BUG过度反应 |
| 框架效应 | Framing Effect | 同一信息用不同方式表述会导致不同的决策判断；人们对"损失"比"收益"更敏感 | 🟢A级；审查中问题描述方式会影响严重程度判断 |
| 乐观偏差 | Optimism Bias | 人们倾向于认为自己比别人更不可能遭遇负面事件，"这不会发生在我身上"的错觉 | 🟢A级；上线前容易低估风险 |
| 知识诅咒 | Curse of Knowledge | 一旦你知道了某件事，就很难想象不知道它的人是什么感受；专家难以站在新手角度思考 | 🟢A级；文档审查和API设计中常见问题 |

### 2.4 AI协作/代码审查类

| 术语（中文） | 术语（英文） | 定义 | 备注/可信度 |
|-------------|-------------|------|------------|
| 安全攻击者 | Security Attacker | 四大攻击者角色之一，🔴标识；攻击目标为数据安全、权限控制、敏感信息泄露；典型手法包括SQL注入、XSS、越权访问 | 🟢A级；详见[01-core-concepts.md#322-四大攻击者角色定义](01-core-concepts.md#322-四大攻击者角色定义) |
| 性能攻击者 | Performance Attacker | 四大攻击者角色之二，🟡标识；攻击目标为系统稳定性、资源消耗；典型手法包括大文件上传、并发请求、递归死循环、正则回溯 | 🟢A级；典型发现：OOM死循环、性能炸弹、DoS漏洞 |
| 边界攻击者 | Boundary Attacker | 四大攻击者角色之三，🟢标识；攻击目标为输入校验、异常处理；典型手法包括空值、超长字符串、特殊字符、未来/过去时间、格式错误 | 🟢A级；典型发现：未来时间污染、空指针、类型错误 |
| 时序攻击者 | Timing Attacker | 四大攻击者角色之四，🔵标识；攻击目标为并发安全、重试机制；典型手法包括竞态条件、重复提交、超时重试、部分失败 | 🟢A级；典型发现：缓存穿透、数据不一致、重试风暴 |
| 攻击成功率 | Attack Success Rate / ASR | 红队测试中成功突破防御的攻击次数占总攻击次数的比例；是衡量AI系统安全性的核心量化指标 | 🔵B级；建议目标：ASR < 20%（基线），< 10%（成熟），高危0% |
| 提示注入 | Prompt Injection | 通过构造特殊的输入提示，让LLM忽略原有系统指令、执行攻击者指令的攻击方式；是LLM应用最普遍的安全风险 | 🟢A级；OWASP LLM Top 10 LLM01 |
| 多轮攻击 | Multi-turn Attack | 通过多轮对话逐步引导、渐进式突破LLM防御的攻击方式；单轮攻击失败不代表安全，多轮迭代后成功率大幅提升 | 🟢A级；200次迭代后ASR可达78.6%（BeyondScale 2026） |
| Crescendo攻击 | Crescendo Attack | 一种多轮攻击策略：从无害话题开始，逐步升级到敏感话题，像音乐渐强（crescendo）一样逐步突破防线 | 🔵B级；PyRIT框架内置支持的高级攻击算法 |
| 越狱 | Jailbreaking | 通过各种手段绕过LLM的安全护栏，让模型输出原本被拒绝的有害、违规内容的行为统称 | 🟢A级；GPT-4平均越狱时间 < 17分钟（arXiv:2505.04806） |
| OOM死循环 | Out-of-Memory Death Loop | 大输入触发内存爆掉→进程被杀→自动重试→再次大输入→再次爆掉→无限循环的典型性能BUG | 🟢A级；AI生成代码高频BUG，正常路径无法发现 |
| 未来时间污染 | Future Time Pollution | 时区错误、时间计算bug导致生成未来时间数据，进而污染整条数据链路、导致业务逻辑异常的边界BUG | 🟢A级；AIHOT项目实战发现的典型问题 |
| 性能炸弹 | Performance Bomb | 异常输入触发指数级计算或无限递归，导致CPU/内存资源耗尽的DoS类BUG | 🟢A级；正则回溯、递归无边界是常见诱因 |
| 重试风暴 | Retry Storm | 服务异常时大量客户端同时重试，导致服务完全无法恢复甚至雪崩的时序类问题 | 🟢A级；需要退避算法+幂等性设计防御 |

### 2.6 方法论类

| 术语（中文） | 术语（英文） | 定义 | 备注/可信度 |
|-------------|-------------|------|------------|
| 五维验证流程 | Five-dimensional Verification | 知识研究场景的核心验证流程：1.来源资质核查→2.交叉验证→3.时效性评估→4.逻辑一致性审查→5.偏差识别 | 🟢A级；详见[01-core-concepts.md#312-五维验证流程](01-core-concepts.md#312-五维验证流程) |
| 阶段0跨领域扫描 | Phase 0 Cross-domain Scan | 知识研究的起始步骤：在深入研究前先进行跨领域快速扫描，了解全局图景、识别相关领域、避免过早陷入局部 | 🔵B级；对抗性审查知识研究方法特色步骤 |
| 语义漂移 | Semantic Drift | 同一个术语在跨领域迁移时，含义发生微妙或显著变化，导致误解和错误推理的现象 | 🔵B级；跨领域知识整合中需要特别警惕 |
| 防御深度 | Defense in Depth | 不依赖单一防御机制，而是在多个层级部署不同的防御措施，即使一层被突破还有其他层保护 | 🟢A级；安全设计核心原则，适用于对抗性审查 |
| CI/CD集成 | CI/CD Integration | 将对抗性审查/红队测试工具集成到持续集成/持续部署流水线中，实现自动化安全门禁：代码提交→快速扫描→阻断高危→夜间深度测试 | 🟢A级；推荐Pipeline分层策略，详见[07-open-source-tools.md#5-cicd集成建议](07-open-source-tools.md#5-cicd集成建议) |
| 多Agent并发 | Multi-Agent Concurrency | 启动多个独立Agent，每个扮演专门的攻击者角色，并发地从不同角度"攻击"代码/方案，比单Agent"多角度思考"效果好得多 | 🟢A级；实战验证：40个Agent并发能发现单Agent完全遗漏的关键BUG |
| 证伪通过 | Falsification Pass | 对抗性审查的通过标准：不是"找到证据证明正确"，而是"无法构造出场景让它失败"；直到无法证伪时才暂时接受其可靠性 | 🟢A级；比"有证据支持"强得多的质量标准 |
| 回归验证 | Regression Verification | 修复BUG后必须重新执行完整的对抗式审查，因为修复一个BUG可能引入新BUG；直到P0/P1问题全部修复才能上线 | 🟢A级；对抗性审查八大核心原则之一 |
| 红队即代码 | Red Teaming as Code | 将红队测试用例、攻击流程、断言都以代码形式管理，像单元测试一样可版本化、可重复执行、可CI集成 | 🔵B级；DeepTeam框架核心理念 |
| PAIR算法 | Prompt Automatic Iterative Refinement | 一种自动迭代优化攻击prompt的算法，通过AI自动生成和改进攻击payload，10次迭代对Gemini成功率达73% | 🔵B级；PyRIT内置支持 |
| TAP算法 | Tree of Attacks With Pruning | 带剪枝的攻击树算法，通过搜索树方式探索多轮攻击路径，自动剪枝无效分支提升效率 | 🔵B级；高级多轮攻击算法 |
