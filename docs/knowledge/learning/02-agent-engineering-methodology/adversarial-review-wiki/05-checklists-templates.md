# 05、检查清单与工具模板

## 1. 概述

检查清单是对抗性审查的核心工具——认知科学反复证明："我会注意的"不可靠，清单可靠。偏差是System 1的自动反应，意志力无法对抗，只有**显性的、可执行的、强制的**检查清单才能把System 2（慢思考）调动起来。

本文件所有工具均可**直接复制使用**，无需修改。分为两大类：
- 知识研究场景工具：用于文献调研、信息收集、事实核查
- AI协作/代码审查场景工具：用于多Agent对抗、代码评审、上线前检查

## 2. 知识研究场景工具

### 2.1 五维验证检查清单

对任何重要事实/主张/数据，逐项勾选以下检查。任何一项无法打勾，都需要进一步验证。

| 验证维度 | 检查项 | 完成状态 | 备注 |
|---------|-------|---------|------|
| **来源资质核查** | □ 我是否确认了来源的作者身份和资质？ | | |
| | □ 作者在该领域是否有相关专业背景/研究经验？ | | |
| | □ 来源是否有同行评审/编辑审核（如学术期刊、权威媒体）？ | | |
| | □ 我是否检查了来源是否存在利益冲突（如商业赞助、立场倾向）？ | | |
| **交叉验证** | □ 我是否找到了至少2个独立来源支持这一主张？ | | |
| | □ 这些来源是否独立（不是互相引用/转载）？ | | |
| | □ 如果是数据，我是否找到了原始数据源而不是二手引用？ | | |
| | □ 是否存在矛盾来源？如果有，我是否记录了矛盾点？ | | |
| **时效性评估** | □ 我是否确认了信息的发布/更新时间？ | | |
| | □ 该领域是否发展较快，信息可能已经过时？（如技术、医学、政策） | | |
| | □ 如果是旧信息，我是否确认它仍然有效？ | | |
| | □ 是否有更新的研究/数据取代了这个结论？ | | |
| **逻辑一致性审查** | □ 主张的论证过程是否逻辑自洽，没有跳步？ | | |
| | □ 证据是否真的能推导出结论，还是存在逻辑断层？ | | |
| | □ 有没有选择性引用证据（只说支持的不说反对的）？ | | |
| | □ 统计数据是否正确解读（区分相关性/因果性、样本代表性）？ | | |
| **偏差识别** | □ 我是否意识到自己对这个结论有预先倾向？ | | |
| | □ 我是否主动寻找过反证？ | | |
| | □ 我是否因为来源权威就直接接受了结论？ | | |
| | □ 这个主张是否"太好了以至于不可能是真的"？ | | |

**可信度快速判定规则**：
- 全部打勾 → 🟢A级（高可信）
- 缺1-2项非核心项 → 🔵B级（中可信，可引用但需标注）
- 缺核心项（如交叉验证）或有矛盾 → 🟡C级（低可信，仅供参考，需标注待验证）
- 来源不明、有明显利益冲突、逻辑漏洞 → 🔴D级（不可信，不应引用）

### 2.2 来源可信度评分表

快速判定一个来源的级别和可信度。

| 来源级别 | 类型 | 可信度 | 典型例子 | 使用规则 |
|---------|------|-------|---------|---------|
| **一级来源** | 原始研究、原始数据、官方一手资料 | 🟢A级 | 学术期刊论文（有DOI）、官方统计数据、政府公开文件、法律法规、作者本人发布、实验原始记录、源代码 | 优先使用，作为事实依据的基石 |
| **二级来源** | 对一级来源的整理、综述、报道 | 🔵B级 | 权威媒体深度报道、教科书、综述论文、行业分析报告、知名专家解读 | 可用但需要追溯到一级来源验证，不能作为唯一依据 |
| **三级来源** | 二次传播、观点评论、非专业创作 | 🟡C级 | 博客文章、社交媒体、自媒体、论坛讨论、PPT分享、未经验证的翻译 | 仅可作为线索使用，必须找到一/二级来源验证才能采信 |
| **不可信来源** | 谣言、广告、阴谋论、有明显利益冲突 | 🔴D级 | 标题党、匿名爆料、明显带货软文、阴谋论内容、无法追溯来源 | 不引用，不传播 |

**快速10秒判断法**：
1. 看域名：.gov/.edu/.ac.cn → 大概率可信；.com/.net看运营方；奇怪域名 → 警惕
2. 看作者：有真实姓名+机构 → 可信度+；匿名/网名 → 可信度-
3. 看引用：是否列出了参考来源？有引用 → 可信度+；无来源的"据说" → 可信度-
4. 看语气：客观陈述 → 可信度+；情绪化/煽动性/绝对化表述 → 可信度-
5. 看利益：是否在卖东西/拉投资/推立场？有明显利益相关 → 可信度-

### 2.2.1 独立来源判定标准（补充细则）

交叉验证要求"至少2个独立来源"。以下是判定两个来源是否"独立"的标准：

**✅ 判定为独立来源（满足任一即可）**：
| 场景 | 判定 | 说明 |
|------|------|------|
| 不同机构/团队独立开展研究得出相同结论 | 独立 | 如OWASP报告和NIST框架分别指出同一风险 |
| 同一机构不同作者团队，使用不同数据集/方法 | 大概率独立 | 需确认是否共享同一研究团队或数据源 |
| 学术论文中独立复现（不同实验室、不同方法） | 独立 | 正式论文的独立复现是最高等级的独立验证 |
| 原始数据/一手资料 + 独立分析/解读 | 独立 | 一手资料（原始数据）+ 第三方独立分析 = 两个独立维度 |
| 厂商A的官方声明 + 独立第三方测试验证 | 独立 | 厂商声明说"性能好"，第三方测试验证 → 独立确认 |

**❌ 判定为非独立来源（满足任一即可）**：
| 场景 | 判定 | 说明 |
|------|------|------|
| A网站/文章直接转载/引用B网站的内容 | 非独立 | 即使换了措辞，信息源是同一个 |
| 同一研究团队/同一作者的多篇文章说同一件事 | 非独立 | 同一团队的自我重复不等于独立验证 |
| 母公司/子公司/关联公司发布的相关内容 | 非独立 | 同一商业实体的不同发声渠道 |
| 基于同一数据集/同一调查报告得出的结论 | 非独立 | 如果都引用同一份IDC数据，即使分析角度不同也不算独立 |
| 新闻通稿/PR稿件被多家媒体转载 | 非独立 | 同一篇通稿发100家媒体 = 1个来源 |
| 引用同一篇论文/同一份报告作为证据 | 非独立 | A说"根据X报告"，B也说"根据X报告" → 实际只有X一个来源 |

**⚠️ 需人工判断的灰色地带**：
| 场景 | 判断方法 |
|------|---------|
| 同一机构不同时期的报告 | 看是否有新数据/新方法：纯重述旧结论→非独立；有新证据→可视为半独立 |
| 预印本→正式发表 | 同一研究的不同阶段，不算新独立来源 |
| 综述论文（Review）引用原始研究 | 综述是二手来源，原始研究才是一手；综述+原始研究不算两个独立来源（如果综述只转述原始研究） |
| 行业联盟/标准化组织的报告 | 看参与方：多公司多机构参与→独立性高；单一公司主导→独立性低 |

**独立来源计数规则**：
- 真正的独立是指**信息获取渠道独立、分析判断独立**
- N个媒体转载同一篇通稿 = 1个来源
- N个团队基于同一数据集 = 1个来源
- 最佳交叉验证：**不同类型**来源的独立确认（如：学术论文+官方数据+行业报告三者一致）

### 2.2.2 可信度动态调整规则（升级/降级触发条件）

可信度不是静态标签，应根据新证据动态调整：

**🟢 升级触发条件（从低→高）**：
| 当前等级 | 升级到 | 触发条件（需同时满足） |
|---------|--------|---------------------|
| 🔵B级 | 🟢A级 | ①已有≥2个符合2.2.1标准的独立来源支持；②独立来源中至少有1个是一级来源；③无与核心结论矛盾的同级或更高可信度反证；④发表/发布后经过≥3个月时间检验未被推翻 |
| 🟡C级 | 🔵B级 | ①找到了至少1个一级来源或2个独立二级来源支持；②无明显反证；③利益冲突已排除或已明确标注不影响核心事实 |
| 🟡C级 | 🟢A级 | 极特殊情况：C级一般不能直接升A级，需先升B级稳定一段时间后再评估 |
| arXiv预印本→🟢A级 | ①论文已在同行评审顶会/顶刊正式发表；②无后续勘误/撤稿；③有至少1项独立研究复现其核心结论 |

**🔴 降级触发条件（从高→低）**：
| 当前等级 | 降级到 | 触发条件（满足任一即可） |
|---------|--------|---------------------|
| 🟢A级 | 🔵B级 | ①发现关键证据有缺陷或无法复现；②主要来源被发现有利益冲突未披露；③新的独立研究提出了有力反证但尚未完全推翻；④依赖的标准/数据已过时（如2023年的LLM安全报告在2026年已不适用） |
| 🟢A级 | 🟡C级 | ①核心结论被后续研究直接反驳；②发现数据造假/学术不端；③原始来源被撤销/撤回 |
| 🔵B级 | 🟡C级 | ①发现引用的来源实际是错误引用/断章取义；②主要支持证据被推翻但尚无反证；③信息明显过时但仍有参考价值 |
| 任意级 | 🔴D级 | ①确认是谣言/造假/欺诈；②作者明确撤稿并声明内容无效；③多个高质量独立来源一致反驳 |

**⏸️ 保持观察的情况**：
- 新证据不足以升级或降级，但增加了不确定性 → 加⚠️标记
- 存在学术争议但双方都有合理证据 → 加⚖️标记，不调整等级但记录争议
- 来源可疑但无法确认 → 保持当前等级，加❓标记

**定期复审建议**：
- 🟢A级内容：每年复审一次，或相关领域有重大进展时复审
- 🔵B级内容：每半年复审，关注是否有新独立证据可升级
- 🟡C级内容：每季度复审，优先寻找验证或排除
- arXiv预印本/快速发展领域（如AI）：每月关注是否有正式发表或新复现

### 2.3 异常标记模板

在研究/审查过程中发现问题时，直接复制以下标记粘贴到对应位置：

```
⚠️ 待验证：[内容] — 需验证原因：[原因]，计划验证方式：[方式]
```
*用途：单个来源提出但尚未交叉验证的信息*

```
❓ 存疑：[内容] — 矛盾来源：[来源A] vs [来源B]，争议点：[争议点描述]
```
*用途：不同来源说法不一致，需要进一步核查*

```
⚖️ 争议观点：[观点A]（支持方论据：[论据]）vs [观点B]（反对方论据：[论据]）
```
*用途：学术/行业内尚无定论的争议性问题，保留双方观点*

```
🔍 利益冲突提示：[来源名称]与[结论/产品]可能存在[商业/职业/立场]关联
```
*用途：来源可能因利益关系存在偏见，需要降低可信度评级*

**使用示例**：
```
⚠️ 待验证：2025年中国大模型市场规模达5000亿元 — 需验证原因：仅有一家咨询公司报告，无官方数据，计划验证方式：查找IDC/信通院官方数据交叉验证

❓ 存疑：某框架性能提升10倍 — 矛盾来源：官方博客说10倍 vs 第三方测试说2.3倍，争议点：测试基准不同，官方用理想场景，第三方用真实场景
```

### 2.4 验证日志模板

对每个重要事实/主张，建立验证记录，直接复制使用：

```
## 验证记录：[待验证的事实/主张简述]

- 验证日期：YYYY-MM-DD
- 验证人：[姓名/ID]
- 初始可信度评估：🟢A/🔵B/🟡C/🔴D
- 来源1：[一级/二级/三级] [出处/链接] [可信度评级] — 支持/反对/中立
  关键证据摘要：
- 来源2：[一级/二级/三级] [出处/链接] [可信度评级] — 支持/反对/中立
  关键证据摘要：
- 来源3（如需要）：[一级/二级/三级] [出处/链接] [可信度评级] — 支持/反对/中立
  关键证据摘要：
- 交叉验证结论：一致/矛盾/部分一致（说明不一致之处）
- 最终可信度评级：🟢A/🔵B/🟡C/🔴D
- 采信方式：直接引用/引用但标注待验证/不引用
- 备注：[其他需要记录的信息]
```

**使用示例**：
```
## 验证记录："Falcon 1前三次发射全部失败"

- 验证日期：2026-07-10
- 验证人：researcher
- 初始可信度评估：🔵B级（常识，但需要原始来源确认）
- 来源1：一级来源 SpaceX官方历史记录 https://www.spacex.com/history — 🟢A级 — 支持
  关键证据摘要：明确列出Falcon 1发射记录：2006年3月24日失败、2007年3月21日失败、2008年8月2日失败、2008年9月28日成功
- 来源2：二级来源 《Elon Musk》传记（Ashlee Vance） — 🟢A级 — 支持
  关键证据摘要：详细描述了三次失败的原因和过程
- 交叉验证结论：一致
- 最终可信度评级：🟢A级
- 采信方式：直接引用
- 备注：第四次成功也是关键背景，不能只说失败不说成功
```

## 3. AI协作/代码审查场景工具

### 3.1 四大攻击者角色检查清单

分角色打印或放在手边，审查时逐项勾选。

#### 🔴 安全攻击者检查清单

目标：找出所有可能导致数据泄露、权限绕过、注入攻击的漏洞。

| 检查项 | 完成状态 | 备注 |
|-------|---------|------|
| □ 所有用户输入是否都做了校验和转义？ | | SQL/NoSQL注入、XSS、命令注入 |
| □ 是否存在SQL注入风险（字符串拼接SQL）？ | | 即使用ORM也要检查原生SQL |
| □ 是否存在XSS风险（用户输入直接渲染到HTML）？ | | 富文本场景特别注意 |
| □ 认证和授权检查是否在服务端做了？ | | 不能只在前端检查 |
| □ 是否存在越权访问漏洞（水平/垂直越权）？ | | 用户A能否访问用户B的数据？普通用户能否用管理员接口？ |
| □ 敏感数据（密码、密钥、Token）是否明文存储/日志打印？ | | 检查日志！很多安全漏洞是日志泄露 |
| □ 依赖包是否有已知安全漏洞（CVE）？ | | 跑npm audit/safety等工具 |
| □ 文件上传功能是否校验了文件类型/大小/内容？ | | 能否上传可执行文件？能否路径遍历？ |
| □ 是否存在SSRF漏洞（服务端请求伪造）？ | | 用户输入URL后服务端去请求，能否访问内网？ |
| □ 错误信息是否泄露了系统内部细节（栈跟踪、路径、SQL语句）？ | | 生产环境要返回通用错误 |

#### 🟡 性能攻击者检查清单

目标：找出所有可能导致OOM、CPU打满、响应超时的问题。

| 检查项 | 完成状态 | 备注 |
|-------|---------|------|
| □ 是否存在N+1查询问题（循环查数据库）？ | | ORM关联查询特别注意 |
| □ 大输入/大文件场景是否做了限制？ | | 上传1GB文件会怎样？ |
| □ 是否存在无限循环/递归风险？ | | 检查循环终止条件，递归深度 |
| □ 正则表达式是否存在回溯陷阱（ReDoS）？ | | 复杂正则匹配恶意输入可能卡死 |
| □ 是否加载了全部数据到内存而没有分页？ | | 查100万条记录到内存会OOM |
| □ 是否存在锁竞争/死锁风险？ | | 并发场景检查锁顺序 |
| □ 缓存策略是否合理？是否存在缓存击穿/穿透/雪崩？ | | 热点key、空值缓存、过期时间 |
| □ 外部调用是否设置了超时时间？ | | 不设超时可能无限等待拖死线程 |
| □ 是否有重试风暴风险（重试没设退避和上限）？ | | 失败后无限重试只会让问题更糟 |
| □ 算法时间复杂度是否合理？大数据量下是否会超时？ | | O(n²)在n=10000时是1亿次操作 |

#### 🟢 边界攻击者检查清单

目标：找出所有极端输入、异常值、空值导致的问题。

| 检查项 | 完成状态 | 备注 |
|-------|---------|------|
| □ 空值处理：null/undefined/空字符串/空数组是否都处理了？ | | 最常见的空指针异常来源 |
| □ 极值处理：0、负数、最大值、最小值、NaN、Infinity | | 金额为0？数量为-1？ |
| □ 字符串边界：空串、超长字符串、特殊字符、emoji、不同编码 | | 用户名能输入10000个字符吗？ |
| □ 时间边界：未来时间、过去很久的时间、时区问题、闰年、跨天/月/年 | | 时间戳是毫秒还是秒？时区对吗？ |

| □ 依赖边界：依赖服务宕机/超时/返回异常、网络分区 | | 熔断/降级/降级逻辑有吗？ |

### 不要机械化走流程——清单是辅助思考，不是替代思考

⚠️ **重要提醒**：检查清单是思考的起点，不是终点。
- 不要打勾打勾打勾就完事了——每个检查项都要停下来真正思考
- 清单是为了防止你漏想，不是为了让你不用想
- 如果清单里没有覆盖你直觉觉得有问题的地方，相信你的直觉——清单永远不可能覆盖所有情况
- 每次漏检后，复盘为什么没检查到，把新的检查项加到清单里——清单是活的，要持续演进

### 根据项目特点裁剪清单

- **小型项目/内部工具**：可以只用3.3快速清单，不必每次都跑全套四大角色
- **对外服务/支付/金融相关**：必须用全套清单，安全攻击者和时序攻击者部分要加倍仔细
- **性能敏感项目（高并发/大数据量）**：性能攻击者清单是重点，建议做压测验证
- **研究性/文档类工作**：重点用2.x知识研究工具
