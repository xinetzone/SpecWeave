# 08、实战案例集

## 1. 概述

实战案例是方法论落地的最佳参考。本章节收录经过可信度分级的对抗式审查真实案例，来源包括：
- 本项目复盘第一手案例
- 行业公开安全事件
- 厂商官方实践披露

所有案例按统一格式呈现：**问题背景→对抗方法→发现的问题→经验教训**，便于读者直接复用。

**可信度评级说明**：
| 评级 | 说明 |
|------|------|
| 🟢 A级 | 第一手实战案例或厂商官方公开，可复现验证 |
| 🔵 B级 | 行业公开案例，有可信来源但非第一手 |
| 🟡 C级 | 参考资料，需进一步验证 |

## 2. 本项目AIHOT项目实战案例 🟢A级

> **来源**：卡兹克AIHOT项目40个Agent并发对抗审查第一手复盘（2026年6月），经本项目学习分析验证。

### 2.1 案例背景

| 项目 | 详情 |
|------|------|
| **项目名称** | AIHOT（AI热点内容聚合系统） |
| **项目规模** | 周请求量超千万，Skill调用量为网页端10倍以上 |
| **审查方式** | Claude Code Ultracode动态工作流，近40个Agent并发执行 |
| **攻击者分组** | 安全攻击者/性能攻击者/边界攻击者/时序攻击者四大角色独立审查 |
| **审查对象** | 核心处理逻辑、HTML清洗模块、任务调度系统、时间处理模块、翻译模块、部署探活机制 |
| **审查时长** | 多Agent并发运行较长时间 |

### 2.2 发现的关键问题

#### 🔴 问题1：OOM死循环BUG（性能攻击者发现）

- **问题现象**：后台worker处理超大HTML任务时内存占用超限→进程被系统杀掉→任务调度器自动重试→重试又拿到同一个大任务→再次OOM被杀→无限循环耗尽系统资源
- **攻击路径**：性能攻击者从"恶意用户提交50MB HTML搞崩worker"角度出发，沿"入口→任务队列→worker处理→内存分配→进程退出→重试"全链路审查
- **为什么自审没发现**：开发测试时只走了正常流程（典型HTML大小10KB-1MB），完全没考虑"超大输入+自动重试机制"这个组合场景
- **后续验证**：审查完成后，生产环境真的遇到了**100MB的HTML文件**，证实了这个BUG的真实性——如果没有提前发现并修复，会导致worker集群雪崩

#### 🔴 问题2：未来时间污染BUG（边界攻击者发现）

- **问题现象**：时区处理错误导致部分信源文章发布时间显示为"未来时间"（如明天），这些文章因时间戳最新被排到精选信息流最前面
- **污染链路**：精选信息流最前→飞书群PUSH推送→RSS订阅输出→日报排序最前→**一篇来自未来的文章污染整条内容分发链路**
- **攻击路径**：边界攻击者从"构造异常时间数据搞乱排序"角度出发，追问"如果发布时间是未来怎么办？"
- **为什么自审没发现**：开发测试时使用的是正确配置的时区，只测试了正常时间范围，完全没考虑跨时区、夏令时、系统时区配置差异、信源时间格式错误等边界情况

#### 🔴 问题3：HTML清洗性能炸弹（性能+安全攻击者联合发现）

- **问题现象**：HTML清洗模块使用的正则表达式在恶意构造输入下产生**灾难性回溯**（ReDoS），导致单请求CPU占用100%持续数秒到数分钟
- **攻击场景**：攻击者可构造特定HTML内容提交，触发正则回溯，少量并发即可占满CPU，造成拒绝服务（DoS）
- **攻击路径**：安全攻击者构造注入Payload，性能攻击者测试极端输入，二者联合发现该组合风险
- **为什么自审没发现**：只测试了正常格式的HTML，完全没考虑攻击者会精心构造专门触发正则回溯的输入

#### 其他发现的隐患

| 隐患类型 | 发现角色 | 风险说明 |
|---------|---------|---------|
| 翻译模块同类性能隐患 | 性能攻击者 | 与HTML清洗类似的正则/解析逻辑存在相同边界风险 |
| 部署探活缓存穿透假阳性 | 时序攻击者 | 探活机制被异常数据干扰，产生假阳性误报，影响自动扩缩容决策 |

### 2.3 对抗方法

1. **四大角色并发独立审查**：安全/性能/边界/时序四类攻击者Agent同时启动，互不干扰，各自专注自己的攻击面
2. **无偏见攻击者视角**：明确要求每个Agent"站在恶意用户立场"，不考虑"代码已经写好了应该没问题"这种先入为主的假设
3. **强制产出要求**：强制要求每个Agent至少找到3个问题，没找到不能结束审查——避免Agent偷懒说"没问题"
4. **全链路穿透测试**：不只是检查单段代码，而是从攻击入口出发，走完"输入→处理→存储→分发→输出"整条链路，找出污染扩散路径

### 2.4 经验教训

1. **正常路径走不出真BUG**：走Happy Path只能证明代码"在理想情况下能工作"，组合风险、边界场景、污染链路只有站在攻击者视角才能发现
2. **多Agent并发抵消确认偏差**：单Agent自审时倾向于"证明代码正确"，多角色并发时每个Agent从不同角度"证明代码有问题"，确认偏差被多角度交叉验证抵消
3. **"这种边界情况不会发生"是最危险的假设**：OOM死循环案例中，100MB HTML真的出现了；未来时间案例中，时区配置差异真的存在——你认为不会发生的边界，恰恰是生产环境最常出问题的地方
4. **组合风险 > 单一风险**：单一"大输入"不致命，单一"自动重试"不致命，但两者组合就是致命的死循环——对抗式审查的价值恰恰在于发现这种组合效应

## 3. 金融公司LLM泄露案例 🔵B级

> **来源**：CSET 2025（乔治城大学安全与新兴技术中心）行业安全报告。

### 3.1 问题背景

某大型金融公司部署内部问答LLM系统，用于员工查询内部FAQ、客户数据结构、合规政策等信息。系统上线前仅做了基础功能测试和简单的内容过滤，未进行结构化对抗红队测试。

### 3.2 对抗方法（被攻击路径）

外部攻击者（或内部恶意人员）通过以下链式攻击突破防护：
1. **提示注入绕过初始过滤**：使用编码、角色扮演、多轮上下文等技术绕过基础内容安全过滤
2. **多轮对话逐步诱导**：不直接问敏感信息，而是通过看似无害的问题建立上下文，逐步让模型输出敏感片段
3. **信息拼接重组**：将多个看似不敏感的输出片段拼接，还原出完整的内部FAQ、系统提示词、甚至部分客户数据结构

### 3.3 发现的问题

- 系统提示词完全泄露
- 内部知识库结构和敏感FAQ内容泄露
- 部分客户数据字段定义和样例数据泄露
- 攻击持续数周未被发现

### 3.4 经验教训

- **上线前必须进行多轮对抗测试**：不能只做功能测试和单轮安全测试，必须模拟真实攻击者的多轮链式攻击
- **ASR（攻击成功率）必须降到可接受阈值**：建议上线前ASR<20%，且持续监控
- **LLM系统的攻击面是对话上下文**：不是单次输入输出，而是多轮对话的累积效应——单轮无害的对话组合起来可能造成严重泄露

## 4. OpenAI Atlas浏览器代理对抗实践 🟢A级

> **来源**：OpenAI官方博客《Hardening Atlas against prompt injection》（2025年）。

### 4.1 问题背景

ChatGPT的Atlas浏览器代理功能允许AI访问网页并根据网页内容回答问题。研究人员发现风险：**恶意网页可以在页面中嵌入隐藏的提示注入内容**（如白色文字、注释、meta标签），当Atlas访问该页面时，隐藏指令会劫持AI的行为，让AI执行非用户预期的操作（如泄露对话历史、点击恶意链接、发送数据等）。

### 4.2 对抗方法

OpenAI采用了**AI驱动的规模化红队**方法：
1. **基于强化学习训练自动化攻击者Agent**：专门训练一个红队AI，目标就是不断尝试攻击Atlas浏览器代理
2. **自我对抗循环**：让AI攻击者持续攻击AI防御者，形成"发现漏洞→修补→验证→再攻击"的快速迭代循环
3. **规模化覆盖**：自动化红队可以24/7运行，覆盖人工测试无法企及的攻击场景数量

### 4.3 发现的问题

- 多个浏览器代理提示注入漏洞被主动发现
- 所有漏洞在公开披露前完成修复
- 建立了持续红队的常态化机制

### 4.4 经验教训

- **红队不是一次性活动，而是持续过程**：不是上线前做一次就完事，新的攻击技术不断出现，必须持续对抗
- **AI可以帮助攻击AI，实现规模化红队**：人工红队成本高、覆盖有限，AI驱动的自动化红队可以将测试规模提升几个数量级
- **浏览器代理/工具调用类Agent风险极高**：任何能访问外部内容的Agent都面临间接提示注入风险——所有外部数据都可能是攻击向量

## 5. 卡兹克对抗式全局审查方法论 🟢A级

> **来源**：卡兹克"Vibe Coding两大神级Prompt"实践，经AIHOT项目和本项目双重验证。

### 5.1 问题背景

常规对抗式审查多针对某个PR、某个功能模块或某次上线，但很多架构级问题、技术债、认知偏差是全局存在的，局部审查无法发现。

### 5.2 对抗方法

每2-3周执行一次**"从第一性原理出发的对抗式全局审查"**：

1. **忽略既有代码**：不看当前实现，假设系统从零开始
2. **从第一性原理重新推导**：这个系统本质上应该是什么样？核心问题是什么？最优解应该是什么结构？
3. **对比现状找差异**：将"应该是什么样"与"现在实际是什么样"对比，差异就是问题
4. **全局对抗问题**：
   - 如果我是竞争对手，我会怎么搞垮这个系统？
   - 如果这个核心数据是错的，会发生什么？整条链路会如何被污染？
   - 如果依赖的第三方服务挂了/变了，系统会怎样？
   - 哪些假设是"现在成立但未来可能不成立"的？
5. **多Agent并发覆盖**：同时从架构、依赖、代码质量、文档一致性等多个维度审查

### 5.3 发现的问题

- 架构层面的技术债和不合理依赖
- 代码层面长期积累的隐患
- 文档与实现不一致的问题
- **类比错误等认知层面问题**（本项目2026-07-09验证：通过全局审查发现了类比推理导致的路径格式错误）
- 新模型能力边界（每次全局审查也可同时测试新模型的审查能力）

### 5.4 经验教训

- **对抗式审查不止于代码，更要审查架构和认知**：局部BUG影响小，架构错误和认知偏差影响整个系统
- **定期全局审查是技术债管理的有效手段**：每2-3周一次的频率既能及时发现问题，又不会过于频繁干扰开发
- **第一性原理+对抗式审查组合效果最佳**：第一性原理保证"应该是什么"的基准正确，对抗式审查保证"实际是什么"与基准的差异被发现

## 6. EchoLeak零交互泄露CVE案例 🔵B级

> **来源**：CVE-2025-32711，Microsoft 365 Copilot安全漏洞，红队测试发现。

### 6.1 问题背景

Microsoft 365 Copilot存在一个严重漏洞：攻击者可以在Office文档的**元数据**（如文档属性、自定义XML、注释、修订记录等用户不可见的位置）中注入隐藏的恶意提示文本。

当用户在Copilot中打开或引用该文档时，**不需要用户做任何额外交互**（如点击、复制、提问），隐藏的提示注入就会自动执行，诱导Copilot泄露用户的邮箱数据、文档内容、聊天记录等敏感信息。

### 6.2 对抗方法

该漏洞由微软内部红队在对抗测试中发现，采用的是**全数据入口模糊测试**方法：
- 不只测试对话输入框
- 对所有可能被AI读取的数据入口（文档元数据、邮件头、表格隐藏列、PPT备注、日历邀请描述等）进行注入测试

### 6.3 发现的问题

- 零交互即可触发泄露
- 影响全系统所有Copilot用户
- 攻击向量极其隐蔽，用户完全无法感知
- CVSS评分高危

### 6.4 经验教训

- **AI系统的攻击面不只是对话输入框**：所有数据入口——文档内容、元数据、邮件、附件、日历、联系人、共享链接——都是潜在的攻击面
- **"用户看不见"不等于"AI看不见"**：人类UI中隐藏的内容（白色文字、注释、元数据、隐藏行列）AI处理时是可见的，这是间接提示注入的核心风险
- **零交互漏洞危害最大**：不需要用户任何操作就能触发的漏洞，传播速度和影响范围远大于需要交互的漏洞

## 8. 案例复用指南

### 8.1 如何将案例应用到你的项目

1. **对照检查**：对照上述案例中的问题类型，检查你的系统是否存在同类风险：
   - 有没有自动重试+大输入的组合？
   - 有没有时区/时间处理？
   - 有没有正则解析用户输入？
   - 有没有多轮对话上下文？
   - 有没有读取外部文档/元数据？
   - 有没有单一数据源污染下游链路的风险？
2. **复用攻击视角**：不需要40个Agent，从四大攻击者角色（安全/性能/边界/时序）出发，每个角色花10分钟思考"我怎么搞垮这个系统"
3. **建立定期审查节奏**：从每2-3周一次的全局对抗审查开始，形成习惯

### 8.2 最小可行对抗审查（1小时快速版）

没有条件跑40个Agent并发时，可以做最小版本：
1. 写下系统的3-5个核心流程
2. 对每个流程问4个问题：
   - 安全：恶意用户能注入什么？偷什么？
   - 性能：最大/最长/最多的输入会怎样？
   - 边界：空值/错值/极端值会怎样？
   - 时序：重试/并发/部分失败会怎样？
3. 对每个答案追问"然后呢？"——找到污染扩散链路
4. 修复P0/P1问题
