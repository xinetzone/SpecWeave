# R - 复盘 (Retrospective)

---

## 从一个生活例子说起

想象你第一次学做番茄炒蛋：
- 第一次：鸡蛋炒糊了，番茄太酸，盐放多了——你说"下次注意"
- 第二次：鸡蛋没糊，但番茄还是酸，盐放少了——你说"哎呀又没做好"
- 第三次：还是不对，你说"我怎么这么笨，就是学不会"

问题出在哪？你每次都"觉得自己知道错在哪了"，但从来没有真正把过程记录下来：第一次到底放了多少盐？炒了几分钟？番茄是哪种番茄？你只是凭感觉说"盐多了"，但多了多少？你只是说"炒糊了"，但火开多大？炒了多久？

这就是大多数人"总结经验"的方式——只有模糊的感受，没有客观的事实；只有结论，没有过程。

复盘R就是要解决这个问题。它不是让你"写个总结"说"下次注意"，而是让你像侦探查案一样，先把现场所有证据原原本本摆出来，不带任何主观判断，然后再推演"如果当时换个做法会怎么样"，最后才得出真正的因果结论。

---

## 一句话公理

**复盘是对已发生事件的结构化反事实推理，将时序经验转化为因果知识。**

这句话有三个关键词：
- **结构化**：不是随便写写，而是按固定的步骤和要素来
- **反事实推理**：不是只说"发生了什么"，还要问"如果当时不这么做会发生什么"
- **时序→因果**：时间上先后发生的事情不等于有因果关系，复盘要帮你区分相关性和因果性

---

## 四个基础要素

复盘包含四个缺一不可的基础要素：

### 1. 事实采集
这是第一步，也是最容易做错的一步。**只记录可验证的客观事实，绝对不能加任何主观判断、因果推测或评价。**

什么是合格的事实？
- "14:32 我执行了 `git commit -m "update"` 命令"——这是事实
- "我不小心提交了错误的代码"——这不是事实，"不小心"是主观判断
- "14:35 CI报错，错误信息是 TypeError: Cannot read property 'x' of undefined，发生在 src/auth.js:42"——这是事实
- "因为我写的代码有bug导致CI挂了"——这不是事实，"因为"是因果判断

判断标准很简单：这句话里有没有"因为/所以/导致/不小心/马虎/聪明/正确/错误"这类词？如果有，那就不是纯事实，混进了你的判断。事实采集阶段，这些词一个都不能有。

### 2. 时序结构化
把采集到的事实按时间顺序排列，理清楚事件发生的先后链路。

不是所有事实都同等重要，你需要找到关键时间节点：什么时候开始的？第一个异常出现在什么时候？谁做了什么动作？之后发生了什么？最后结果是什么？

就像看监控录像一样，你要能从时间线里清晰地看到整个过程：A发生了，然后B发生了，然后C做了一个动作，然后D发生了。时序结构化能帮你排除"我记得好像是先..."的记忆偏差。

### 3. 反事实推演
这是复盘最核心、也是和普通"总结"区别最大的一步。你要问自己一系列"如果...会怎么样"的问题：
- 如果当时我在提交前先跑一下本地测试，还会出现这个问题吗？
- 如果当时我选的是另一个方案，结果会有什么不同？
- 如果当时有个人提醒我一下，这个错误能避免吗？

反事实推演不是让你"后悔"，而是帮你找到真正的因果关系。如果"做X"和"不做X"结果都一样，那X就不是原因；如果"做A"结果是坏的，"做B"结果是好的，那A和B之间的差别就是关键。

### 4. 因果转化
最后一步，把反事实推演的结果转化成明确的因果知识，也就是"在什么条件下，因为什么机制，做什么动作会导致什么结果"——这个格式是不是很熟悉？对，这就是后面要讲的**洞察I**的四元组格式。

因果转化不是写"下次小心"这种正确的废话，而是要写出具体可执行的结论：
- 错误："下次提交前注意检查"——没用，你上次也是这么说的
- 正确："提交前必须先执行 npm run lint，提交信息必须包含变更模块名"——这是可以检查、可以执行的

## 在五层层级模型中的位置

复盘R跨越了三个层级：
- **感知层**：事实采集——像摄像头一样客观记录发生了什么，不带判断
- **认知层**：时序结构化 + 反事实推演——用逻辑思维分析过程，推导因果
- **沉淀层**：因果转化——把结论变成可以复用的知识，为下一步洞察I做准备

复盘是整个七概念体系的**入口**——没有复盘得到的事实和因果，后面的洞察I、萃取E都无从谈起。如果复盘做不好，后面的所有东西都是建立在沙子上的。

---

## 核心价值

为什么复盘这么重要？不做复盘会怎么样？

**第一，避免"经验贬值"。**
你做过的项目、踩过的坑，如果不复盘，这些经历就只是"经历"，不会变成"经验"。过三个月你再遇到同样的问题，还是会踩同样的坑。时间花了，精力花了，但什么也没留下。复盘把一次性的经历变成可以重复使用的知识，让你的每一份付出都有积累。

**第二，区分"运气"和"能力"。**
很多时候你做成了一件事，可能只是运气好，不是你方法对；很多时候你搞砸了一件事，可能只是运气差，不是你能力差。复盘通过反事实推演帮你区分这两者——如果换个条件你还能做成吗？如果再来一次你还会搞砸吗？想清楚这个，你才知道哪些是可以复制的成功，哪些是需要避免的错误。

**第三，为后续所有概念提供原料。**
没有高质量的复盘，就没有高质量的洞察I；没有高质量的洞察，就没有可复用的模式E；没有可复用的模式，你的知识库就是空的。R→I→E这条链路的起点就是复盘，第一步错了，后面全错。

## 常见误区

### 误区1：复盘变成"批斗会"或"表扬会"
要么全是"某某某这里做错了，他应该怎样怎样"，要么全是"大家都很努力，这次做得很好"。这两种都不是复盘。复盘对事不对人，既不追究责任，也不评功摆好，只关注"发生了什么、为什么、下次怎么改"。

### 误区2：事实阶段就下结论
这是最常见的错误，也是危害最大的错误。一开始就写"因为我当时粗心大意导致bug"，后面所有分析都会被这个结论带偏——你不会再去想是不是流程有问题，是不是工具不够好，是不是文档没写清楚，只会把一切归因为"粗心"。记住：事实阶段禁因果词，"因为/所以/导致"这些词一个都不能有。

### 误区3：只有反思没有行动
复盘到最后写一堆"下次注意""加强学习""提高认识"，这些都是正确的废话。复盘的结论必须是具体可执行、可验证的——要么是一个流程上的改变，要么是一个检查项，要么是一个可以加入检查清单的规则。如果一个结论没办法检查"有没有做到"，那它就不是一个合格的复盘结论。

---

## 小结

复盘R不是"写总结"，它是一套严谨的结构化流程：先客观采集事实（不带判断），然后按时序整理过程，接着通过反事实推演找到真正的因果，最后转化成可执行的知识。它是整个七概念体系的入口，也是经验沉淀的第一步。

做好了复盘，你就有了高质量的原料，下一步我们就可以从这些原料中提炼出跨情境可迁移的规律——也就是**洞察I**。
