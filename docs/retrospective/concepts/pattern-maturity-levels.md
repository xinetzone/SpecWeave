> **来源**：从复盘实践需求中提炼——在面对日益增多的萃取模式时，需要一套客观标准来评估模式的可靠性与适用范围

# 模式成熟度分级体系（Pattern Maturity Levels）

## 一、定义

模式成熟度分级是对已萃取的模式进行成熟度评级的一套标准体系。其核心目标是为每个模式标注其经过的实践验证轮次，帮助使用者在采纳模式时判断其可靠性与适用场景。

当一个模式被首次萃取时，它仅代表单次实践中的成功经验，其通用性尚未被检验。随着该模式在不同项目中独立复用并获得正面验证，其成熟度逐步提升。最终，当一个模式的结构足够稳定且可被形式化，它可以沉淀为模板文件或自动化脚本，实现零修改复用。

成熟度分级体系解决的核心问题是：**面对复盘文档中不断积累的模式，如何区分"值得直接复用"的模式与"仅供参考"的模式**。它为资产清单中的每个条目提供了可信度锚点，也为模式的升级、降级与淘汰提供了明确的决策规则。

## 二、三级成熟度定义

| 级别 | 名称 | 标准 | 示例 |
|---|---|---|---|
| L1 | 实验性 | 基于单次实践萃取，尚未经第二个独立项目验证。模式结构可能不稳定，边界条件未充分探索。 | 目录创建"三件套"模式——来源于 apps/ 目录创建工作流，尚未在其他上下文中独立复用 |
| L2 | 已验证 | 已在 2 个或以上独立项目中经过独立验证，模式结构稳定，核心要素明确，可被他人按文档实施。 | 双区开发模型——在 apps/ 工作空间创建与 .agents/protocols/ 协议体系中分别验证，高熵区/低熵区/质量门禁三要素结构稳定 |
| L3 | 标准化 | 已被模板化或工具化，形成可零修改直接复用的自动化或半自动化规范。使用者无需理解全部细节即可使用。 | Spec-driven 开发流程——已沉淀为 spec-template.md、tasks-template.md、checklist-template.md 三套模板，可零修改直接用于新项目初始化 |

### 各级别判定细则

**L1 实验性的特征**：
- 模式文档中包含"来源"字段，但只引用了一个复盘报告或项目
- 模式的适用场景边界尚未充分定义
- 模式结构中可能包含与原始上下文耦合的细节
- 读者需要自行判断是否适用于自身场景

**L2 已验证的特征**：
- 模式文档的"来源"字段引用了两个或以上独立的复盘报告
- 模式在不同场景下的应用方式已被记录（含差异说明）
- 模式的输入、输出、前置条件、边界条件已明确
- 读者可以按文档描述直接实施，无需猜测

**L3 标准化的特征**：
- 模式已被转化为可执行文件（scripts/）或可直接填充的模板文件（templates/）
- 存在配套的验证脚本或自检清单
- 使用者在绝大多数场景下无需修改即可复用
- 模式文档中明确标注了对应的模板/脚本路径

## 三、升级规则

### 3.1 升级路径

```
L1（实验性）──→ L2（已验证）──→ L3（标准化）
     单次验证          多项目验证          模板化/工具化
```

**L1 → L2 升级条件**：
在另一个独立项目中复用时，该模式无须重大修改即可适用。所谓"重大修改"是指：需要改变模式的核心结构、调整核心要素的语义、或重新定义输入输出契约。以下情况不视为重大修改：
- 填充模式参数（如替换具体项目名称、目录路径）
- 在模式框架内增加非核心的可选步骤
- 调整非关键的默认值

**L2 → L3 升级条件**：
被沉淀为模板文件（存放于 `templates/` 目录）或自动化脚本（存放于 `scripts/` 目录），可被零修改复用。具体标准如下：
- 模板文件：定义了可填充的占位符，使用者仅需填入项目特定信息
- 自动化脚本：可独立执行的脚本，接收参数化输入，无需修改源码
- 配套文档：模板或脚本附有使用说明，说明参数含义与填充方式

### 3.2 降级条件

当模式在两次或以上后续复用中需要重大修改时，该模式从当前级别降回 **L1**，并在模式文档中标注降级原因。

降级的典型场景包括：
- 模式的核心假设在新的项目类型中不成立
- 模式的流程步骤在新场景下需要重新排列或大幅增减
- 模式的输出格式无法满足新场景的需求

降级后须执行的操作：
1. 在模式文档中添加"降级记录"章节，标注原级别、降级时间、触发条件
2. 与降级模式关联的模板或脚本须同步标注兼容性警告
3. 若该模式已从 L3 降级，其配套的模板/脚本应标注为"实验性适配"而非"标准化复用"

## 五、与资产清单的关系

成熟度级别是 `asset-inventory.md` 中每个模式条目的核心元数据。两者关系如下：

1. **数据流向**：成熟度级别由此文档（pattern-maturity-levels.md）定义与维护，`asset-inventory.md` 引用此文档的结论。资产清单中每个模式条目的"成熟度"列应与此文档保持一致。

2. **同步要求**：当此文档中任何模式的成熟度级别发生变更（升级、降级）时，`asset-inventory.md` 中对应条目的"成熟度"列必须同步更新。

3. **建议扩展**：当前的 `asset-inventory.md`（见 `d:\AI\docs\retrospective\assets\asset-inventory.md`）按资产类型分为"可直接复用的文件"、"需实例化后复用的模式"、"需按场景适配的决策框架"三类表格。建议在每类表格中新增一列"成熟度"，以 L1/L2/L3 标注，便于读者快速判断资产的可靠性。

4. **权威来源**：当 `asset-inventory.md` 与本文档中的成熟度评级出现不一致时，以本文档为准——因为本文档定义了成熟的判定标准与升级/降级规则，而资产清单仅做引用展示。

## 六、维护规则

### 6.1 触发时机

成熟度评级应在以下时机进行检查与更新：

1. **每次项目复盘后**：复盘报告完成后，检查本次复盘是否涉及已有模式的复用。若复用的模式满足升级条件，执行升级；若发现原有模式需要重大修改，执行降级。

2. **资产清单更新时**：每次更新 `asset-inventory.md` 时，同步检查本文档中对应资产的成熟度是否需要调整，确保两处信息一致。

3. **新模板/脚本创建时**：当某个已记录的模式被转化为模板文件或自动化脚本时，检查其是否满足 L2→L3 升级条件，满足则立即升级。

### 6.2 维护流程

每次维护应执行以下步骤：

1. **扫描触发条件**：确认本次维护的触发原因（复盘完成 / 资产清单更新 / 模板化完成）
2. **评估受影响模式**：列出本次涉及的模式，逐条对照升级/降级条件进行评估
3. **更新本文档**：在"当前资产成熟度快照"章节中更新对应条目的级别与说明
4. **同步资产清单**：在 `asset-inventory.md` 中更新对应条目的成熟度列
5. **记录变更**：在模式文档本身（如 patterns/ 下的 .md 文件）中添加或更新来源与验证记录，确保溯源链完整

### 6.3 年度审查

建议每季度对全部模式的成熟度进行一次全面审查，重点检查：
- L1 模式是否已积累了足够的验证数据支持升级
- L2 模式是否已有条件转化为模板或脚本
- L3 模式的模板/脚本是否仍处于活跃使用状态（若连续两个季度未被使用，考虑在模式文档中标注"休眠"状态）
