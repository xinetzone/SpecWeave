# 洞察 C：滚动复盘八步是文档一致性的低成本保障机制

**发现**：十四轮复盘使用的标准化八步流程，以极低的成本（每轮约15-30分钟）维持了21个文件、~8600行文档的一致性。

## 八步流程精解

| 步骤 | 操作 | 解决的问题 | 竹简悟道中的实际效果 |
|------|------|----------|-------------------|
| 1. 文件统计 | 统计全部文件行数与状态 | 头部声明过时 | 发现 insights-31-52（R14已重命名为31-65）偏差33%，推动文件名修正 |
| 2. 标记搜索 | 搜索全项目 ⚠️ 标记 | 技术债累积 | 14处⚠️跟踪至12处解决 |
| 3. 头部校验 | 声明值 vs 实测值 | 统计数据漂移 | 修正多文件头部行数声明 |
| 4. 三向对照 | Spec ↔ 洞察 ↔ 原型 | 设计与实现脱节 | 发现场景标签双轨→合并为一行 |
| 5. 引用审计 | 跨文件 file:// 链接 | 断链 | 确保9条交叉引用全部有效 |
| 6. 不一致识别 | 发现新产生的矛盾 | 迭代引入冲突 | 发现 P2-2 实际已完成但未标记 |
| 7. 优先级清单 | P0-P3 分级排序 | 改进无重点 | P0/P1全部清零 |
| 8. 新洞察判断 | 是否需要新洞察 | 经验未沉淀 | 发现洞察55/56/57等元洞察 |

## 四级优先级定义（可直接复用）

- **P0（架构阻塞）**：不解决就无法继续开发或交付——必须立即解决
- **P1（支撑缺失）**：缺少关键洞察或设计，影响决策质量——本迭代解决
- **P2（完备性）**：有比没有好，但不阻塞——择机解决
- **P3（规范性）**：数字、格式、命名不一致——积累一批后统一处理

## 频率建议

每完成2-3次实质性变更（新增洞察、修改Spec、更新原型）执行一轮，而非等项目结束。

## 可迁移性

**极高**。这是通用的文档质量保障流程，与具体项目主题无关。
