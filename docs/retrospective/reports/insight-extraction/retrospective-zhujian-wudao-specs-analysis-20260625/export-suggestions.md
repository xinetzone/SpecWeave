## 三、复用操作指南

### 3.1 新项目快速启动（最小可行骨架）

如果要在新项目中应用竹简悟道的方法论，按以下顺序操作（1小时内可完成）：

1. **创建 .agents/ 骨架**（5分钟）
   ```
   项目根/
   ├── AGENTS.md          ← 复制 transferable-patterns.md §1.3 模板，替换{项目名}
   └── .agents/
       ├── project.md     ← 用 transferable-patterns.md §八 模板，写2-3个核心概念
       ├── constraints.md ← 写3-5条铁律（从核心哲学推导）
       └── workflows.md   ← 写1个SOP："如何写新洞察"
   ```

2. **写前5-10条产品层洞察**（20分钟）
   - 使用基础档结构（来源+核心内容，1-3段）
   - 回答：用户是谁？解决什么痛点？和竞品差异是什么？核心承诺是什么？

3. **写Spec九节结构**（30分钟）
   - 用九节checklist确保无遗漏
   - 不需要每节都很长，但九个问题都要回答

4. **初始化滚动复盘机制**（5分钟）
   - 跑第一轮全项目审计（八步流程简化版：统计文件→检查⚠️→校验头部→对照Spec→列优先级）

### 3.2 已有项目迁移指南

如果是已有项目想引入这套方法论，顺序相反：

1. 先做一轮全面盘点（用滚动复盘八步）
2. 补写最核心的5条洞察（定位/差异化/核心承诺/架构原则/边界）
3. 建立constraints.md（3-5条铁律）
4. 之后每2-3次变更跑一轮轻量复盘

### 3.3 何时不需要全套体系

根据 transferable-patterns.md §9.2，以下情况这套体系是过度设计：
- 项目规模 < 3个文件
- 单人一次性脚本，无迭代计划
- 不需要与AI协作（纯手写代码）

最小可行版本只需保留3个文件：project.md（概念）+ constraints.md（铁律）+ workflows.md（1个SOP）。

## 五、关键经验总结

竹简悟道 specs 文档体系验证了以下关键经验，可供后续项目参考：

1. **单日高产的秘密不是速度快，而是决策前置**——54条洞察先于代码，每个设计决策都有依据，减少了HTML开发阶段的犹豫和返工

2. **文档一致性靠"高频轻量复盘"而非"终局大检查"**——每2-3次变更就审计一次，每次15-30分钟，成本远低于项目结束后再修复不一致

3. **可迁移资产要在项目进行中萃取，不要等到结束**——transferable-patterns 和 transferable-methods 是在迭代过程中逐步形成的，不是项目做完后再回头总结

4. **AI和人类需要不同形态的知识**——给AI模板（可执行结构+占位符），给人类方法论（原理+适用场景+迁移方式），不要试图一份文档服务两类读者

5. **接受"不完美的持续迭代"优于"追求完美的停滞"**——头部声明过时是正常的（熵增），用第三策（移除变量+免责声明）零成本应对，不必追求100%准确
