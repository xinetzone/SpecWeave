# 执行过程复盘

## 一、任务背景

本轮任务不是单点“修个权限”问题，而是两类问题同时暴露后的系统化治理：

- **当前风险**：容器启动期若对绑定挂载目录执行 `chmod` / `chown`，会把修改直接传播回宿主机，破坏“宿主机权限零漂移”
- **历史污染**：早期 root 流程遗留的 `npu_tvm/build` 等目录可能已被错误属主污染，导致当前非 root 用户无法清理、增量构建或复用缓存
- **交付要求**：既要保证后续启动流程不再制造新污染，也要提供可控工具修复旧污染，并把方法沉淀为可复用文档

因此，本轮工作的核心不是“把权限都改对”，而是把“自动策略不该做什么”和“人工工具负责什么”拆清楚。

## 二、事实与时间线

### 2.1 阶段一：识别问题边界

1. 从 `CMake`/构建清理相关报错回溯到挂载目录权限问题，确认 `npu_tvm/build` 中存在历史遗留的属主异常
2. 进一步检查 `Dockerfile`、`entrypoint.sh`、`run.py` 后，确认旧流程若在启动时对绑定挂载路径做 `chmod` / `chown`，会把修改写回宿主机
3. 由此形成第一条边界判断：
   - 当前启动流程要解决的是“以后不再漂移”
   - 历史脏目录修复要通过显式命令完成，不能偷偷塞进启动脚本

### 2.2 阶段二：建立零漂移权限模型

针对“启动流程会不会改写挂载目录”这一风险，改造点集中在容器入口与目录初始化策略：

1. 保留 `HOST_UID` / `HOST_GID` 映射，确保容器默认工作用户与宿主机身份对齐
2. 将入口脚本的权限初始化限定在镜像内自建目录：
   - `/workspace`
   - `/workspace/libs`
   - `/workspace/.container-runtime`
   - 开发用户家目录
3. 明确**不**对以下绑定挂载目录及其既有文件执行 `chmod` / `chown`：
   - `/workspace/libs/npu_tvm`
   - `/workspace/libs/npuusertools`
4. 保持 SSH、sudo、默认 shell、环境变量注入等能力不退化

这一阶段的关键，是把“用户身份映射”和“挂载目录权限修正”拆成两件事处理，只保留前者的自动化。

### 2.3 阶段三：补齐验证链，证明不是“看起来没问题”

仅靠静态代码阅读无法证明零漂移模型成立，因此在 `run.py` 中补强了运行后验证：

1. **宿主机快照比对**：记录挂载目录根与代表性文件在容器启动前后的 `uid/gid/mode`
2. **容器视图核对**：比对容器内 `/workspace/libs/npu_tvm`、`/workspace/libs/npuusertools` 与宿主机是否完全一致
3. **非 root 访问探针**：
   - 验证映射后的默认用户可以读取挂载内容
   - 在宿主机既有授权下执行一次真实写入探针
4. **工作目录所有权检查**：确认 `/workspace` 与 `/workspace/libs` 的属主与宿主机 UID/GID 对齐，避免容器内新建文件再次漂移

通过这条验证链，结论不再停留在“脚本里已经删掉了某些 `chown`”，而变成“宿主机前后属性零变化 + 容器中视图一致 + 非 root 业务可用”。

### 2.4 阶段四：补齐历史污染修复工具

完成零漂移改造后，历史问题仍然存在：旧的 `build` 目录可能已经被 root 污染，启动策略不会自动帮用户修复。

于是新增了宿主机侧修复工具，最初聚焦 `build` 目录场景：

1. 增加 `fix_build_permissions.py`
2. 仅允许显式指定 `build` / `build_*` 目录
3. 支持 `--dry-run` 预览，避免误改
4. 对真实目录执行 dry-run 验证，确认在当前环境下具备幂等性

这一阶段解决的是“历史问题有出口”，同时保持“当前启动流程不越权”。

### 2.5 阶段五：将修复能力泛化并收尾命名

后续又发现，历史污染并不必然只出现在 `build` 类目录，因此工具继续演进：

1. 将主脚本能力泛化为 `fix_mount_permissions.py`
2. 对默认安全路径继续放行：
   - `build`
   - `build_*`
3. 对非 `build` 目录增加显式安全门：
   - 需要逐路径传入 `--confirm-non-build PATH`
   - 解析后必须与目标路径完全一致
4. 保留 `--force` 作为兼容旧用法，但不推荐替代逐路径确认
5. 将 `fix_build_permissions.py` 降级为兼容包装器，避免旧命令失效

最终完成了“语义更准确的主入口 + 历史命令不失效”的平滑迁移。

## 四、关键决策

### 决策 1：启动期不再自动修正挂载目录权限

**原因**：

- 自动修权限会把副作用传播到宿主机
- 绑定挂载目录属于用户数据，不应在容器启动时被隐式篡改
- 当前任务首先要保证“以后不再制造新问题”

**结果**：

- 容器启动流程实现零漂移
- 权限修复从“隐式副作用”转为“显式操作”

### 决策 2：历史污染通过宿主机侧独立工具治理

**原因**：

- 历史脏目录是真实存在的问题，不处理会继续阻塞构建
- 但把治理逻辑塞回入口脚本，会重新破坏零漂移模型

**结果**：

- 工具职责边界清晰
- 用户可在真正需要时，通过 `--dry-run` 先预览再修复

### 决策 3：工具默认保守，扩展能力必须显式确认

**原因**：

- `build` 目录风险边界清晰，适合作为默认安全目标
- 非 `build` 目录可能包含源码、三方依赖或其他高价值内容，误修复成本高

**结果**：

- 获得了可扩展的通用工具
- 又不会因为“泛化”而失去安全护栏

### 决策 4：主入口改名，但保留旧命令兼容

**原因**：

- `fix_build_permissions.py` 已不足以表达通用挂载目录修复能力
- 直接删掉旧入口会打断已有使用路径

**结果**：

- 新命名更贴近能力语义
- 老命令仍然可用，迁移成本低

## 五、问题与处理

### 问题 1：如果完全不自动修权限，历史脏目录依然会阻塞工作

- **现象**：启动期零漂移做对了，但旧的 `build` 目录仍可能让当前用户无法清理
- **处理**：新增独立修复脚本，且默认只面向 `build` 类目录
- **收获**：当前策略与历史修复应拆分治理，不能互相污染职责

### 问题 2：工具泛化后，误操作风险随能力一起扩大

- **现象**：从“只修 `build`”扩展到“可修任意挂载子目录”后，误改源码目录的风险上升
- **处理**：对非 `build` 路径引入 `--confirm-non-build` 的逐路径确认
- **收获**：扩展能力时，默认保守 + 显式确认比单纯加一个大开关更稳健

### 问题 3：脚本命名升级可能破坏历史命令

- **现象**：主脚本重命名后，旧命令、旧文档、旧习惯可能失效
- **处理**：保留 `fix_build_permissions.py` 作为薄包装器，内部转发到新主入口
- **收获**：重命名不是只改文件名，还要设计兼容层

## 六、结果评估

本轮任务完成了一个完整的权限治理闭环：

1. **当前风险收敛**：容器启动流程不再改写宿主机绑定挂载目录权限
2. **使用体验保留**：SSH、非 root 开发、工作目录创建与环境变量注入保持可用
3. **验证证据完整**：宿主机快照、容器视图、非 root 读写三类证据互相佐证
4. **历史问题可修复**：提供了宿主机侧显式修复工具，而非回退到“启动时偷偷改权限”
5. **能力可演进**：从 `build` 专用脚本平滑升级到通用挂载目录修复主入口，并保留兼容层

更重要的是，这次任务沉淀出了一条可复用的工程边界：

- **自动化流程只负责维持正确状态，不负责替用户修改历史脏数据**
- **历史污染修复必须是显式、可预览、可审计、可回退的人工动作**
