# 洞察萃取

## 一、核心洞察

### 洞察 1：当前自动策略与历史污染修复必须分轨处理

**发现**：

“容器启动不改挂载权限”和“修复历史脏目录”看似都属于权限治理，实则是两种性质完全不同的动作。前者是**防新增污染**，后者是**清历史存量**，如果混在同一条自动化链路里，职责就会失真。

**证据**：

- `entrypoint.sh` 仍负责 UID/GID 映射与镜像内目录初始化，但不再触碰挂载目录
- `run.py` 负责验证零漂移是否成立，而不是替用户修复历史属主异常
- `fix_mount_permissions.py` 被设计成宿主机侧显式工具，而不是容器启动钩子

**可复用结论**：

- 防止新问题的逻辑，应进入默认自动化路径
- 修复历史脏数据的逻辑，应留在显式人工操作路径
- 这两类逻辑绝不能因为“都和权限有关”就合并

### 洞察 2：对绑定挂载目录，零漂移优先级高于“启动时帮你修好”

**发现**：

对宿主机绑定挂载目录来说，启动期自动 `chmod` / `chown` 看似“贴心”，实际上是最危险的副作用来源之一。真正稳健的模型，不是启动时把一切都改成想要的样子，而是启动时证明“我没有改坏任何东西”。

**证据**：

- README 已明确声明入口脚本只初始化镜像内自建目录
- `run.py` 会验证宿主机挂载目录根与代表性文件在启动前后 `uid/gid/mode` 零变化
- 容器内挂载视图与宿主机属性保持一致，说明没有发生隐式漂移

**可复用结论**：

- 对挂载目录，默认策略应是“保持原样”
- 如果业务可用性与零漂移冲突，优先保住零漂移，再提供显式治理工具
- “没有副作用”本身就是一项需要被验证的能力，而不只是实现细节

### 洞察 3：工具泛化时，默认保守 + 显式确认，是比“大开关”更好的安全模型

**发现**：

`fix_build_permissions.py` 只处理 `build` 类目录，风险相对可控；当能力泛化为 `fix_mount_permissions.py` 后，如果仍用同样宽松的默认行为，工具就会从“修复器”变成“误操作放大器”。

**证据**：

- `build` / `build_*` 被定义为默认安全目标
- 非 `build` 路径必须追加 `--confirm-non-build PATH`
- 确认参数解析后必须与目标路径完全一致，避免“确认了 A，却误修了 B”
- `--dry-run` 作为默认推荐流程，先预览再执行

**可复用结论**：

- 泛化工具时，不要简单用 `--force` 作为唯一护栏
- 最安全的做法是保留默认安全集合，再让高风险目标逐项确认
- “能力越强，默认越保守”应成为工程工具设计原则

### 洞察 4：验证链必须覆盖“宿主机事实 + 容器事实 + 业务事实”

**发现**：

权限治理如果只看某一侧，结论很容易失真：

- 只看脚本实现，不知道运行后是否真会漂移
- 只看容器内视图，不知道宿主机是否已被改坏
- 只看属性一致，不知道非 root 用户是否还能正常工作

因此，本轮任务真正可靠的地方在于构建了三层证据链。

**证据**：

- 宿主机快照：启动前后比对挂载根目录与代表性文件
- 容器视图：核对 `/workspace/libs/...` 的实际 `uid/gid/mode`
- 业务探针：映射后的非 root 用户完成一次真实读取与写入

**可复用结论**：

- 权限问题必须至少验证三件事：有没有漂移、视图是否一致、业务是否可用
- 任何一层缺失，都会留下盲区
- 证据链设计应优先于“我感觉这个脚本没问题”

### 洞察 5：语义升级的重命名，应同时设计兼容层

**发现**：

当工具能力从“修 build”升级为“修挂载目录”时，主脚本改名是正确动作；但如果只改名不保留兼容层，就会把命名优化的收益转化成使用成本。

**证据**：

- 主入口采用 `fix_mount_permissions.py`，更准确表达通用挂载目录修复能力
- `fix_build_permissions.py` 继续存在，但只作为转发入口
- README 统一口径说明：主入口是 `fix_mount_permissions.py`，旧入口仅用于兼容历史命令

**可复用结论**：

- 名称升级要服务语义清晰
- 兼容层要服务已有用户路径
- 好的重命名不是“删旧立新”，而是“新旧并行、逐步收口”

## 二、可复用方法论

### 方法论 1：挂载权限治理的双轨模型

| 轨道 | 目标 | 典型动作 | 禁止事项 |
|------|------|----------|----------|
| 自动策略轨 | 防止产生新污染 | UID/GID 映射、镜像内目录初始化、零漂移验证 | 启动时修改绑定挂载目录及既有文件 |
| 人工修复轨 | 清理历史存量污染 | 宿主机执行修复工具、`--dry-run` 预览、逐路径确认 | 偷偷放回入口脚本或默认自动执行 |

### 方法论 2：修复工具的安全分层设计

1. 先定义默认安全目标集合，例如 `build` / `build_*`
2. 对高风险目标要求逐路径确认，而不是给一个笼统的大开关
3. 默认推荐 `--dry-run`
4. 输出明确的修复计划，确保执行前可审阅
5. 对历史命令保留兼容包装，平滑迁移使用习惯

### 方法论 3：权限治理验证三联证据法

1. **宿主机事实**：启动前后 `uid/gid/mode` 是否零变化
2. **容器事实**：容器视角看到的挂载目录属性是否与宿主机一致
3. **业务事实**：非 root 默认用户是否具备预期读写能力

只有三者同时成立，才能说权限治理真正闭环。

## 三、边界与决策原则

### 原则 1：自动化只维护正确状态，不隐式修复历史问题

- 自动化负责“保持不变”和“尽早发现”
- 人工工具负责“显式修复”和“可审计执行”

### 原则 2：对宿主机数据的操作，必须默认保守

- 绑定挂载目录本质上是用户资产
- 任何会修改宿主机文件属主/权限位的动作，都必须是用户明确触发

### 原则 3：命名、文档、工具入口要三者同步

- 新入口负责表达真实能力
- 兼容入口负责减少迁移摩擦
- README 负责统一对外口径，避免“代码改了，文档还在说旧名字”

## 四、后续可迁移场景

- 其他通过 Docker/Podman 挂载宿主机源码目录的开发环境
- 任何“默认以非 root 工作，但历史上曾以 root 运行过”的构建场景
- 需要把专项修复脚本逐步升级为通用治理工具的项目
- 需要在“安全边界”与“使用便利”之间寻找平衡的内部运维脚本

## 五、建议沉淀

- 将“零漂移优先于启动期自动修权限”推广为挂载式开发环境的默认规范
- 将“默认安全集合 + 显式确认扩展”推广到其他会修改宿主机状态的工具
- 将“三联证据法”推广到后续权限、挂载、缓存、工作目录类问题的验证模板
