# 批量子代理委派的中间检查点缺失风险洞察

## 1. 事实数据采集

**事件经过**：
- 任务：火山引擎AI云原生沙箱产品深度分析（Spec Mode）
- 背景：tasks.md拆解为11个任务，Task1（网页内容提取）单独执行后，将Task2-Task10共9个关联分析任务合并委派给单个general_purpose_task子代理一次性执行
- 结果：子代理产出967行高质量分析报告，所有任务一次完成，质量达标
- 风险识别：本次成功是"运气"而非"流程保障"——若子代理在中间某个任务（如Task3核心优势分析）偏离预期，后续Task4-Task10（场景分析、架构分析、对比分析等）将全部基于错误基础执行，导致整体返工
- 工具浪费成本估算：9个任务合并委派若失败，将浪费约8000-12000 tokens的子代理输出，返工需要重新委派

**涉及的错误假设**：
- 假设子代理在多任务批量执行中能自始至终保持方向正确 → 无保障
- 假设任务间的依赖关系（后一个任务基于前一个产出）不会被批量执行破坏 → 无验证
- 假设"一次委派完成所有任务"是最优效率策略 → 忽视了失败时的返工成本

## 2. 根因分析

### 直接原因

将9个存在顺序依赖关系的任务一次性合并委派，缺乏中间验证点。子代理是无状态执行，不会在中途主动停下来请求验证，一旦早期任务产生偏差，偏差会沿依赖链放大。

### 深层原因（5 Whys）

1. **为什么批量委派9个任务？** → 为了减少交互轮次，提升执行效率
2. **为什么没有设置中间检查点？** → 当时任务拆解清晰，主观认为子代理能一次性正确执行，低估了偏差传导风险
3. **为什么偏差会沿依赖链放大？** → 分析类任务的后续步骤高度依赖前序产出（架构分析依赖核心优势判断，对比分析依赖场景分析结论），一个环节出错会导致连锁错误
4. **为什么本次没有暴露问题？** → 子代理能力强+任务拆解质量高+领域（云原生沙箱）与已有知识重叠度高，属于"高概率成功但非流程保障"
5. **为什么这是一个系统性风险而非偶发问题？** → 批量委派的诱惑始终存在（效率高、交互少），但失败代价随任务数量和依赖深度非线性增长，且缺乏制度化的检查点机制

**核心根因**：**批量子代理委派存在"效率-风险"不对称性——成功时节省N轮交互，失败时浪费全部产出并需从头返工。当前缺乏明确的中间检查点决策准则。**

## 3. 关键洞察

### 洞察1：批量委派的效率-风险不对称原则

批量委派N个关联任务的收益和风险不对称：
- **成功收益**：减少(N-1)轮主代理-子代理交互，节省协调时间
- **失败代价**：N个任务的全部token产出浪费 + 重新执行的全部时间 + 偏差诊断成本
- **风险临界点**：当N>3且任务间存在强依赖关系时，失败代价开始超过成功收益

**规律认知**：关联任务批量委派不是"越多越好"。对于独立任务（无依赖），批量规模可以较大；对于强依赖任务（链式依赖），批量规模应控制在2-3个以内。

### 洞察2：中间检查点的设置原则

需要设置中间检查点的信号：
- 任务数量 ≥ 4个
- 任务间存在链式依赖（后任务基于前任务产出）
- 首次执行某类任务，产出质量不确定性高
- 任务总token量大（预计子代理输出 > 3000 tokens）
- 任务领域是新领域，子代理可能产生方向性偏差

不需要检查点的场景：
- 任务间完全独立（并行任务）
- 任务数量 ≤ 3个且无强依赖
- 同类任务已有多次成功验证记录（L2+成熟度）
- 低风险、可容忍偏差的简单任务

### 洞察3：检查点的两种实现策略

**策略A：批次拆分法**
- 将关联任务拆分为2-3个批次
- 每批完成后由主代理验证产出质量
- 验证通过后继续下一批
- 适用：任务可以自然分段（如"提取→分析→撰写"三阶段）

**策略B：里程碑验证法**
- 保持单批次委派，但在prompt中要求子代理在关键节点产出后暂停并输出中间结果
- 主代理验证中间结果后指示继续
- 适用：任务不可拆分，但有明确的中间里程碑（如"完成核心框架后先输出大纲"）

### 洞察4：任务粒度与子代理能力的匹配

子代理的"可靠性半径"是有限的——在其熟悉的领域和清晰的任务指引下，可以正确执行较多任务；但在新领域或模糊指引下，可靠性半径急剧缩小。

**规律认知**：委派任务数量应与以下因素成反比：
- 任务领域新颖度（越新→批量越小）
- 任务间依赖强度（越强→批量越小）
- 产出质量要求（越高→批量越小）
- 与子代理过往协作经验（越少→批量越小）

## 4. 可行动建议

| 建议 | 优先级 | 验收标准 | 状态 |
|---|---|---|---|
| 建立批量委派检查点决策准则 | 🟡 中 | 子原子任务模板中包含"批量规模决策矩阵" | ⏳ 待积累2-3个案例后沉淀为模式 |
| 关联任务批量规模控制在3个以内 | 🔴 高 | 强依赖任务不超过3个/批次 | ⏳ 下次执行时实践验证 |
| 高不确定性任务先委派1-2个试水 | 🟡 中 | 新领域任务先小规模验证再扩大批量 | ⏳ 下次执行时实践验证 |
| 批量委派prompt中包含中间里程碑要求 | 🟢 低 | prompt模板包含"关键节点输出中间结果"指令 | ⏳ 待模式沉淀后统一更新模板 |

### 批量委派决策速查表

```
任务依赖关系    任务数量    建议策略
─────────────────────────────────────────
无依赖（并行）  任意        可批量，建议≤6个
弱依赖          ≤3          可一次性批量
弱依赖          4-6         拆2批，每批验证
强依赖（链式）  ≤2          可一次性批量
强依赖（链式）  3-4         拆2批+中间验证
强依赖（链式）  ≥5          必须分批+里程碑验证
新领域+强依赖   任意        先1-2个试水，验证后扩展
```
