## 3. 根因分析（5-Whys）

### 3.1 为什么Spec的v1.0决策"暂不创建"被推翻？

**Why 1**：为什么v1.0决策被推翻？
→ 因为第一性原理分析揭示了"知识库→指令集"的必然性——知识库降低"知道"的成本，指令集降低"做到"的成本。

**Why 2**：为什么第一性原理分析能揭示这个必然性？
→ 因为第一性原理强制剥离了经验性假设（如"Spec已经决策了"、"使用频率未知"），从公理层面推导。

**Why 3**：为什么经验性假设会阻碍正确决策？
→ 因为确认偏差——人们倾向于寻找支持已有决策的证据，而不是重新评估决策条件是否变化。

**Why 4**：为什么确认偏差在决策评估中如此普遍？
→ 因为它是System 1的自动反应——大脑默认走"最省力"的路径，而重新评估需要消耗认知资源（System 2）。

**Why 5**：为什么System 2的调动需要显性的方法论？
→ 因为意志力不可靠——"我会注意的"无法对抗自动化的认知偏差，只有显性的、可执行的检查流程（如第一性原理6步分析）才能强制调动System 2。

**根因**：认知偏差（确认偏差）是阻碍决策质量的根本原因，显性的方法论（第一性原理分析）是防御偏差的有效手段。

### 3.2 为什么"指令集vs Skill"的判断容易出错？

**Why 1**：为什么"指令集vs Skill"的判断容易出错？
→ 因为两者的边界模糊——都有frontmatter、都有步骤、都引用知识库。

**Why 2**：为什么边界模糊导致判断错误？
→ 因为缺乏明确的判断标准，决策依赖直觉（"这个重要，应该用Skill"）而非抽象层次分析。

**Why 3**：为什么缺乏判断标准？
→ 因为现有规范中，"指令集"和"Skill"的定义偏重形式（文件位置、格式）而非本质（核心操作类型）。

**Why 4**：为什么本质定义比形式定义更重要？
→ 因为形式定义只能决定"放在哪个目录"，不能决定"应该设计成什么结构"。一个认知方法放在Skill目录下仍然是认知方法，但不会获得Skill的脚本执行优势。

**根因**：指令集与Skill的区分标准应从"形式"（文件位置）升级为"本质"（核心操作类型：认知引导 vs 脚本自动化）。

## 5. 建议生成

### 5.1 高优先级建议

**建议H1**：建立"指令集vs Skill"判断标准

| 维度 | 内容 |
|------|------|
| **问题** | 当前缺乏明确的判断标准，容易将认知方法误判为Skill |
| **建议** | 在skill-development.md中增加"何时创建指令集而非Skill"的判断流程 |
| **判断标准** | 核心操作是"引导AI进行结构化思考"→指令集；核心操作是"执行脚本/自动化流程"→Skill |
| **预期收益** | 减少错误分类，提高方法论资产的可用性 |
| **验收标准** | 新增1个方法论类资产时，能正确选择指令集vs Skill |

### 5.2 中优先级建议

**建议M1**：为所有方法论类指令集增加元审查

| 维度 | 内容 |
|------|------|
| **问题** | 当前仅对抗性审查指令集包含元审查，其他方法论类指令集缺少自检机制 |
| **建议** | 复盘、洞察、第一性原理指令集在下次修改时增加"执行质量自检"部分 |
| **预期收益** | 防止"表演式执行"，提高指令集执行质量 |
| **验收标准** | 至少2个方法论类指令集（含对抗性审查）包含元审查机制 |

### 5.3 低优先级建议

**建议L1**：对抗性审查指令集增加场景速查矩阵
