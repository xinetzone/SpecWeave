# 洞察9:公理系统——一致性保证可组合性的第一性原理

- **分类**:第一性原理类
- **成熟度**:L2 已验证(validation_count=2)
- **可复用性**:高 - 适用于所有文档、代码、系统设计的一致性决策

## 洞察内容

从Spec路径一致性、Markdown相对路径两个经验中，可以提炼出**公理系统**这一底层第一性原理：**文档（或代码）系统本质上是一个公理系统——路径引用、链接格式、命名规范都是这个系统的"公理"。公理的一致性是定理（派生产物、索引、交叉引用）正确性的前提。一旦公理出现矛盾，整个系统的可理解性就会崩塌**。

相对路径的本质不是"格式偏好"，而是保证文档作为一个公理系统在不同环境下的自洽性和可移植性。

## 5-Whys根因追问链

| Why层级 | 问题 | 答案 |
|---------|------|------|
| Why1 | 为什么路径声明要与实际归档一致？ | 否则后续查阅会困惑 |
| Why2 | 为什么查阅会困惑？ | 因为人们按照Spec声明去查找文件却找不到 |
| Why3 | 为什么找不到是个问题？ | 因为这破坏了"Spec→交付物"的映射关系 |
| Why4 | 为什么映射关系重要？ | 因为只有映射一致，多个文档才能组合成可理解的系统 |
| Why5 | **底层本质** | **可组合系统必须建立在一致性公理之上——引用不一致则系统无法被可靠推理** |

## 证据支撑

- **经验E4**:Spec路径与实际归档一致——保证文档可追溯
- **经验E6**:链接使用相对路径——保证文档可移植
- **本次事件**:链接格式错误类比（file:/// vs 相对路径）导致系统可移植性被破坏
- **跨领域验证**:数学公理系统一致性、编程接口一致性、数据库外键一致性

## 一致性校验五查（可操作化）

在创建/修改引用关系时，检查5点：
1. 路径声明与实际一致吗？
2. 链接格式符合规范吗？
3. 命名遵循约定吗？
4. frontmatter元数据完整吗？
5. 跨文件引用能通过自动化检查吗？
