## 二、关键决策分析

### D1：同步策略——以 AGENTS.md 为权威源

**决策**：仅修改 README.md，不修改 AGENTS.md。

**背景**：AGENTS.md 已是 7 角色 + 5 协议的正确版本，README.md 滞后。

**选择理由**：
- AGENTS.md 作为"全局契约"，是项目规范的权威定义文件
- README.md 面向人类读者，属于派生文档
- 派生文档对齐权威源是单向同步的正确策略

### D2：品牌词否决 "Framework" → 转向 "SpecWeave"

**决策**：放弃通用行业词，转向自创合成词。

**否决原因**：用户明确要求"像 DeepSeek 一样有特色"——Framework 缺乏品牌辨识度。

**选择 SpecWeave 的理由**：

| 维度 | SpecWeave 优势 |
|------|---------------|
| 语义覆盖 | 同时命中"规范"（Spec）和"编织/体系"（Weave）两个核心维度 |
| 品牌辨识度 | 复合词结构类似 DeepSeek、DreamWeaver，朗朗上口 |
| 隐喻延展性 | 编织隐喻可衍生丰富的品牌叙事（经纬、针法、图样） |
| 排他性 | Google/GitHub 无同名项目，搜索引擎唯一指向 |

### D3：标题与标识行的分工设计

**决策**：标题承担"品牌名 + 品类描述"，标识行承担"Slogan + 隐喻阐释"。

```
标题：  SpecWeave — AI 智能体开发规范体系    （品牌 + 品类）
标识行：规范之网：将角色、协议、工作流编织为...   （Slogan + 隐喻）
引言：  一套面向多智能体协作开发的开放规范体系...  （详细说明）
```

三层递进阅读结构：品牌识别 → 概念理解 → 详细认知。

## 三、问题诊断与解决

### 问题 1：文档数字不一致（根因）

| 维度 | 详情 |
|------|------|
| 现象 | README.md 显示"5 角色、4 协议"，AGENTS.md 实际"7 角色、5 协议" |
| 根因 | 角色和协议体系在演进中新增了条目（co-founder、team-admin、应用开发生命周期），README.md 的派生数据未同步更新 |
| 影响范围 | 6 处：核心优势表×2、文档导航表×2、折叠索引区×2 |
| 解决 | 以 AGENTS.md 为权威源，全量修正 README.md 对应数字和列表 |

### 问题 2：首次提案被否决后的迭代策略

| 维度 | 详情 |
|------|------|
| 现象 | "Framework" 提案被用户以"不够有特色"否决 |
| 策略调整 | 从"选现有词"转为"造合成词"，一次性提供 4 候选 + 对比矩阵 |
| 效果 | 用户一次批准，避免逐轮试错 |

### 问题 3：标题与标识行的协调

| 维度 | 详情 |
|------|------|
| 现象 | 先添加了标识行，用户提醒标题也需更新 |
| 解决 | 设计三层递进的品牌表达结构，标题和标识行各司其职 |
