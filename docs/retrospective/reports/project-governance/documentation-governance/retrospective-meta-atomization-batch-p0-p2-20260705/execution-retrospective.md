## 一、背景与目标

### 1.1 项目背景

在 [SpecWeave 13天全生命周期复盘](../../comprehensive-reviews/retrospective-specweave-full-lifecycle-20260705/) 的原子化重构过程中，提炼出**元原子化"二分+概览"模式**（详见 [meta-atomization-bisect-overview.md](../../../../patterns/methodology-patterns/document-architecture/meta-atomization-bisect-overview.md)）。该模式包含两个核心子策略：

| 子策略 | 适用场景 | 核心操作 |
|--------|---------|---------|
| **时间二分模式** | 时间线性叙事的大章节 | 按时间/阶段/版本中点拆分为两个原子文件，主文件保留概览表 |
| **概览+详情分离模式** | 列举/比较/清单类大章节 | 将完整描述拆至独立 `*-details.md`，主文件保留概览表+链接 |

该模式在全生命周期复盘的原子化中得到单点验证，但需要在更多类型的文档上验证其普适性。

### 1.2 原始目标

1. **模式验证**：将"二分+概览"模式推广至全项目范围内的复盘报告大文件，验证跨场景适用性
2. **批量原子化**：处理P0（IoT生态）+ P1（高优项目复盘）+ P2（评估跳过项）共8个目录
3. **效果量化**：量化主文件缩减率、链接有效性、模式选择决策树准确率
4. **模式成熟度评估**：基于批量应用结果评估模式是否达到L2（已验证）级别

### 1.3 约束条件

- **时间约束**：单日执行（2026-07-05）
- **质量约束**：零断链（所有本地引用必须有效）
- **范围约束**：仅处理 `docs/retrospective/reports/` 下的复盘报告文件

## 三、关键决策记录

### D1：采用"先评估后处理"而非"全部拆分"策略

- **时间**：2026-07-05（任务初期）
- **决策**：不盲目对所有>300行文件拆分，而是先评估现有原子化程度，已原子化的仅验证确认，低于阈值的跳过
- **依据**：tuyaopen-analysis和home-assistant-integration已有深度原子化结构（patterns/phases/risks子目录），强行再次拆分会制造冗余
- **结果**：✅ 有效 — 避免了2个目录的冗余拆分，验证了"先查现有结构"前置步骤的重要性（见 [insight-extraction.md I-2](insight-extraction.md#i-2先查现有原子化结构是必要前置步骤)）
- **经验**：原子化不是"文件越小越好"，而是消除"同一主题内联过多"的问题

### D2：MDI项目采用"有限拆分"而非"全拆"策略

- **时间**：2026-07-05（P1处理阶段）
- **决策**：MDI insight-extraction.md（419行）仅拆出"核心洞察"一个同质章节（15个洞察→core-insights-details.md），保留剩余342行异质内容
- **依据**：剩余内容涵盖过程分析、模式速查、行动计划、结论等异质主题，强行按单一维度拆分会破坏上下文连贯性
- **结果**：✅ 有效 — 拆分后342行仍然>300行，但每章节<100行，符合原子化三标准（单一职责/独立可阅/命名聚合）
- **经验**：原子化标准是"章节级单一职责"而非"文件级行数限制"（见 [insight-extraction.md I-4](insight-extraction.md#i-4拆分后剩余文件行数300行-的合理性边界)）

### D3：P2（contest-advantage-analysis）标记为跳过

- **时间**：2026-07-05（评估阶段）
- **决策**：contest-advantage-analysis虽然主文件较大（execution 407行, insight 475行），但已部分原子化（insights/子目录含16个独立洞察文件），内联内容精炼，不满足"大章节内联"拆分条件
- **依据**：拆分条件是"存在连续>100行的同质内联章节且无对应原子文件"，该报告不满足
- **结果**：✅ 正确决策，纳入P2后续评估
- **经验**：需要自动化工具来识别"待拆分文件"，减少人工评估成本
