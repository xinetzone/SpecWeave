## 二、断链根因分析

### 2.1 断链分布

14 个断链按原因分类：

| 根因类型 | 数量 | 占比 | 示例 |
|---------|------|------|------|
| 相对路径缺少 `../`（目录变深） | 10 | 71% | spec 文件引用 docs 少一层 |
| 引用已归档/不存在的文件 | 2 | 14% | xinet 归档后旧路径失效 |
| 目录引用尾部缺斜杠 | 1 | 7% | `.agents/roles` 应为 `.agents/roles/README.md` |
| 其他路径错误 | 1 | 7% | 旧产物路径残留 |

### 2.2 为什么会发生"../层数错误"？

**五问法根因分析**：

1. **为什么链接断了？** → 因为相对路径中的 `../` 数量不够，指向了不存在的位置
2. **为什么 `../` 数量不对？** → 因为目标文件所在目录深度比原来多了一层
3. **为什么目录深度会变？** → 因为之前进行了原子化拆分（一个大文件拆成目录+多文件）
4. **为什么原子化后链接没有同步更新？** → 原子化操作只移动了文件，没有反向更新所有引用方
5. **为什么没有反向更新？** → **根本原因：缺少自动化工具检测和修复此类路径错误**

### 2.3 目录原子化对路径的影响

原子化操作示例：

```
原子化前：.agents/roles/README.md （源文件）
原子化后：.agents/roles/
           ├── README.md
           ├── orchestrator.md
           ├── architect.md
           └── ...

此时源文件本身深度不变，但从其他更深目录引用它的相对路径需要调整。
例如从 .trae/specs/xxx/spec.md 引用：
  原子化前：../../.agents/roles/README.md  ❌（现在需要更多层级）
  原子化后：../../../.agents/roles/README.md  ✅
```

## 三、算法设计与实现

### 3.1 核心算法：相对路径层级校正

**设计思路**：

当一个相对路径断链时，最常见的情况是：
- 路径后半段（目标文件名 + 部分中间目录）是正确的
- 只是开头的 `../` 数量多了或少了 1~3 个
- 目录引用（末尾为 `/`）应自动尝试下的 `README.md`

**算法步骤**（实现于 `.agents/scripts/lib/link_fixer.py` 中的 `try_adjust_relative_depth()` 函数，第 209-300 行）：

1. 解析现有路径，统计 `../` 段的数量 N
2. 分离出非 `../` 的剩余路径段 R
3. **优先尝试增加深度**（原子化场景最常见）：N+1, N+2, N+3
4. 再尝试减少深度：N-1, N-2, N-3（直到 N=0）
5. 对每个候选，检查目标文件是否存在
6. 同时处理目录引用：如果候选路径是目录，自动追加 `/README.md`
7. 返回第一个找到的有效目标路径

### 3.2 修复优先级链

在 `fix_link_url()` 中，按以下优先级尝试修复：

```
1. 直接存在 → 无需修复
2. 路径层级校正（新增，精确匹配）→ 优先使用
3. 顶级目录根路径尝试（docs/xxx、.trae/xxx、.agents/xxx）
4. 文件名映射（已知重命名）
5. 文件名模糊搜索（兜底，基于 basename 查找）
```

这样确保最精确的修复优先，避免模糊搜索产生误匹配。

### 3.3 安全机制

- **dry-run 模式**：默认支持 `--fix --dry-run`，只预览不写入
- **零误报设计**：只有目标文件真实存在时才建议修复
- **非破坏性**：修复前不会删除原内容，精确替换 URL 部分
- **增量修复**：只修复检测到的断链，正确链接保持不动

5. **优先级设计**：精确修复（层级校正）优先于模糊搜索（basename 匹配），减少误修概率
