## 一、任务概述

### 1.1 任务背景

在flexloop/chaos并发模块的代码审查与冲突解决机制开发过程中，团队总结出了并发代码审查的"六维检查法"（超时、幂等、边界、防御、配置、国际化）。此前六维检查法依赖人工审查，效率低且容易遗漏。用户要求将这套方法论转化为可自动化执行的静态分析工具，并集成到Git pre-commit钩子中，实现提交前自动扫描。

### 1.2 任务目标

1. **目标一**：生成针对并发模块的自动化测试脚本，验证六维检查法
2. **目标二**：将六维检查规则集成到Git pre-commit钩子中，实现提交前自动扫描

### 2.2 关键节点分析

| 节点 | 问题 | 解决方案 | 经验 |
|------|------|---------|------|
| Thread.join()误判 | `str.join()`被识别为线程join | 实现`_is_thread_join()`方法，通过变量名模式（thread/worker）和类型构造判断 | **AST静态分析的核心挑战是消歧义**——同名方法在不同上下文中语义完全不同 |
| 集合变量误判 | `_pending_set`被识别为列表append | 检查变量名后缀（`_set`/`_dict`/`_map`）排除集合类型 | **命名约定是静态分析的重要信号**，但也意味着依赖代码规范 |
| 循环深度遗漏 | for循环未跟踪嵌套深度 | 添加`visit_For`方法与`visit_While`统一管理`loop_depth`计数器 | AST访问器必须覆盖所有相关节点类型，遗漏一个节点类型就会导致一类问题漏报 |
| JSON输出失败 | FileReport缺少passes属性 | 让FileReport继承ResultGroupMixin | **测试必须覆盖CLI所有输出格式**，而非仅核心逻辑 |
| 钩子import路径 | 脚本直接运行时sys.path[0]是hooks目录而非scripts目录 | 在main()入口统一添加scripts_dir到sys.path | **Git钩子运行环境与直接运行脚本的sys.path不同**，必须在设计时考虑 |

### 2.4 成功经验

1. **遵循现有架构模式**：检查器的模块组织（lib/ + cli入口 + tests/）完全参照了现有`lib/checks/sensitive_info.py`的结构，降低了集成成本
2. **TDD驱动开发**：先写33个测试用例定义期望行为，再实现逻辑，核心visitor逻辑一次通过大部分测试
3. **在真实代码上验证**：使用conflict_resolution.py作为验证基准（已修复代码得100分，故意有缺陷的代码得9分），确保检查器不是"温室花朵"
4. **链式钩子架构**：将pre_commit.py重构为`_run_sensitive_check()` → `run_concurrent_check()`链式调用，既保持了向后兼容，又清晰分离了关注点
5. **完整的环境变量控制**：参考敏感信息检查的SKIP/WARN_ONLY模式，提供了一致的用户体验

### 2.5 存在问题

1. **静态分析的固有限制**：基于AST的静态分析无法追踪运行时类型（如一个变量实际是threading.Lock还是自定义类），只能通过命名约定启发式判断，存在误报/漏报可能
2. **中文比较检测范围有限**：当前I18N维度仅检测`==`/`!=`直接比较中文字符串的场景，未覆盖`in`操作符中文匹配、状态机中文流转等复杂场景
3. **边界维度（BOUNDARY）的列表查找检测**：依赖变量名模式（`*_list`）判断线性查找，对不遵循命名规范的代码会漏报
4. **钩子只扫描暂存文件**：如果开发者分批提交，可能只提交了部分文件，导致跨文件的并发问题未被检测到
5. **缺少自动修复能力**：与敏感信息检查的`--fix`不同，并发安全问题无法自动修复，只能人工处理

## 三、洞察环节

### 3.1 关键发现

**发现1：方法论→工具的转化关键在于"信号识别"**

六维检查法本质上是一组启发式规则（heuristics），而非形式化验证。将方法论转化为自动化工具的核心挑战不是"实现检查逻辑"，而是"找到可自动化的信号"。每条审查规则都需要从"人工语言"翻译为"机器语言"——即AST上可匹配的代码模式：

| 审查维度（人工语言） | 可检测信号（机器语言） | 信号强度 |
|---------------------|---------------------|---------|
| 超时设置不合理 | `acquire()`/`join()`/`wait()`调用是否有timeout参数 | 强（AST直接可见） |
| 幂等性缺失 | `list.append()`前是否有`not in`守卫 | 中（需上下文分析） |
| 边界条件遗漏 | 循环内是否有`x in list`线性查找模式 | 中（依赖命名约定） |
| 防御性不足 | 可变默认参数、返回内部可变对象引用 | 强（AST直接可见） |
| 配置硬编码 | `time.sleep()`是否使用字面量而非配置变量 | 强（AST直接可见） |
| 国际化问题 | 字符串比较中是否包含中文字面量 | 强（AST直接可见） |

**信号转化四步法**（可复用于任何人工Checklist→自动化工具转化）：
1. 将每条审查规则翻译成"在代码中看到什么模式时触发"
2. 评估信号强度（强信号=AST直接可见，弱信号=需要上下文推断）
3. 对弱信号设计启发式消歧策略（命名约定、作用域分析等）
4. 接受边界：宁可漏报不可误报，对不确定的情况保守处理

每条规则都有"能自动检测"和"不能自动检测"的边界。好的静态分析工具不是100%覆盖，而是在"检测率"和"误报率"之间找到平衡。

**发现2：链式pre-commit架构优于多独立钩子——新增检查应遵循链式扩展**

在研究现有pre-commit架构时发现，项目采用的是"一个pre-commit钩子Python文件，内部串联多个检查"的模式，而不是"每个检查一个独立钩子文件"（如pre-commit框架的多钩子模式）。

| 维度 | 多独立钩子方案 | **单入口链式方案** |
|------|--------------|-------------------|
| 跨平台维护 | 需维护多套shell/cmd包装器 | **只需维护一套shell包装器** |
| 检查顺序 | 按文件名字母序，不可控 | **显式链式调用，顺序可控** |
| 输出格式 | 各钩子格式不一，体验碎片化 | **统一格式，一致的用户体验** |
| 早期阻断 | 无法保证快速检查先执行 | **快速检查优先，失败即终止** |
| 新增检查成本 | 新增shell包装器+配置 | **新增Python模块+主入口注册一行** |
| Windows兼容 | 每个钩子都需要.cmd包装器 | **共享一套包装器** |

链式架构的关键优势：敏感信息检测（<0.5秒）排在第1位优先阻断，避免并发安全检查（~2秒）在敏感信息已泄露的情况下仍然执行浪费时间。

**发现3：AST静态分析开发的5类误判模式与消歧策略**

在开发visitor.py（465行）过程中反复遇到的误判模式，每类都有对应的消歧策略：

| 误判类型 | 示例 | 根因 | 消歧策略 |
|---------|------|------|---------|
| **同名不同义** | `str.join()` vs `Thread.join()` | AST只看方法名，不看接收者类型 | 变量名模式（thread/worker/lock前缀）+ 构造函数类型追踪 |
| **类型推断缺失** | `items = []`是list还是deque？ | Python动态类型，AST无类型信息 | 变量名后缀约定（`_list`/`_set`/`_dict`）+ 赋值语句追踪 |
| **上下文遗漏** | for循环内的查找 vs 顶层查找 | AST节点遍历遗漏相关节点类型 | 完整覆盖AST节点（For/While/AsyncFor/If/With/Try等） |
| **作用域穿透** | 函数参数是list还是外部传入的set？ | 无法追踪跨函数数据流 | 仅在函数/方法内部做局部分析，不做跨过程分析 |
| **测试代码污染** | 测试函数中的故意错误被报告 | 检查器不区分测试代码和生产代码 | 按函数名前缀（`test_`）和类名（`Test*`）自动跳过 |

消歧不确定时的铁律：**降级为MEDIUM警告而非HIGH报错**。

**发现4：TDD对静态分析工具开发特别有效**

并发安全检查器采用TDD模式开发：先写33个测试用例定义期望行为（包含正向案例和反向案例），再实现visitor.py核心逻辑，核心逻辑一次通过大部分测试，剩余5个误判问题逐一修复。

TDD对静态分析工具特别有效的三个原因：
1. **正向+反向测试双重验证**：每个规则需要"应该报错的代码"和"不应该报错的代码"，TDD天然要求定义这两种情况
2. **误判回归防护**：修复一个误判后添加对应的回归测试，避免后续修改重新引入同类误报
3. **规则可执行规格**：测试用例就是规则的精确规格文档，比文字描述更准确、无歧义

静态分析工具的测试五件套：阳性测试（召回率）、阴性测试（精确率）、边界测试、CLI测试（所有输出格式）、集成测试（真实代码验证）。

**发现5：Git钩子三层信任模型的分层原则**

pre-commit钩子不是万能的，必须根据时间预算将检查分层放置：

| 层级 | 触发时机 | 时间预算 | 检查类型 | 阻断方式 |
|------|---------|---------|---------|---------|
| L1 pre-commit | 本地commit | **<5秒** | 单文件、高价值、强信号检测 | 阻断commit |
| L2 pre-push | 本地push | **<30秒** | 相关模块测试、增量全量扫描 | 阻断push |
| L3 CI/CD | 合并请求 | **<10分钟** | 全量扫描、集成测试、跨文件分析 | 阻断merge |

为什么pre-commit不能做全量扫描？
- 开发者提交频率高（一天10+次commit），每次全量扫描>10秒会严重影响体验
- 结果：开发者使用`--no-verify`跳过所有检查 → 防线全面失效
- 正确做法：pre-commit只扫暂存文件增量（`git diff --cached`），CI做全量兜底

### 3.2 规律认知

**规律1：AST静态分析工具开发的标准流程**

```mermaid
flowchart LR
    A[定义规则信号] --> B[AST节点模式匹配]
    B --> C{有歧义?}
    C -->|是| D[启发式消歧\n命名约定/上下文/作用域/测试跳过]
    C -->|否| E[直接报告问题]
    D --> F{消歧成功?}
    F -->|是| E
    F -->|否| G[保守:不报/降为MEDIUM]
    E --> H[TDD:正向+反向+边界测试]
    G --> H
    H --> I[真实代码验证\n调优规则]
```

核心原则：**宁可漏报（false negative），不可误报（false positive）**——误报会让开发者不信任工具，最终`--no-verify`跳过检查；漏报可以通过逐步完善规则和CI层兜底来弥补。

**规律2：链式pre-commit扩展标准流程**

新增pre-commit检查的标准四步（不修改Shell包装器）：
1. 在`hooks/`下创建`xxx_check.py`，导出`run_xxx_check(project_root, staged_files) -> int`
2. 在`pre_commit.py`的`main()`中添加一行调用链（按速度和优先级排序）
3. 如需环境变量控制，遵循`XXX_CHECK_SKIP`/`XXX_CHECK_WARN_ONLY`命名约定
4. 编写单元测试（阳性+阴性）

**规律3：静态分析工具测试的双重覆盖原则**

每个检查规则必须至少有1个正向测试用例（应报）和1个反向测试用例（不应报）。反向测试（防误报）比正向测试更重要——误报是导致开发者抛弃工具的首要原因。

### 3.3 交叉洞察（与敏感信息检测钩子复盘共同验证）

两次连续任务（敏感信息检测+并发安全检查）独立验证了5条模式，证明这些模式具有可复用性：

| 共同模式 | 敏感信息检测验证 | 并发安全检查验证 |
|---------|----------------|----------------|
| 链式钩子架构 | ✅ 第一个检查链，确立模式 | ✅ 第二个检查链，验证可扩展性 |
| 增量扫描（只扫暂存文件） | ✅ `git diff --cached` | ✅ 同样模式，保证pre-commit速度 |
| 三级绕过机制 | ✅ SKIP/WARN_ONLY/--no-verify | ✅ 一致的命名约定和用户体验 |
| 宁可漏报不可误报 | ✅ 正则匹配精确率优先 | ✅ AST消歧五法降低误报 |
| pre-commit+CI双层防御 | ✅ pre-commit快速+CI全量 | ✅ 同样架构，CI门禁待实施 |

这5条模式已沉淀为团队可复用模式，未来新增pre-commit检查或静态分析工具时应直接遵循。

### 3.4 潜在机会

1. **规则可配置化**：当前六维规则硬编码在visitor.py中，未来可支持YAML配置文件自定义规则阈值和排除模式
2. **IDE实时诊断**：核心visitor逻辑可被Language Server Protocol(LSP)复用，提供编码时实时提示，而非仅在提交时检查
3. **增量扫描优化**：当前钩子扫描所有暂存Python文件，可通过git diff只扫描变更行相关的代码上下文
4. **更多并发模式检测**：当前六维是基础检查，可扩展检测：
   - 死锁：多锁获取顺序不一致
   - 竞态条件：共享变量读写无锁保护
   - 线程池泄漏：未正确shutdown
   - 异步陷阱：async函数内调用阻塞IO
5. **CI流水线集成**：在pre-commit基础上增加full scan模式，作为MR/PR门禁，扫描全量代码而非仅变更文件
