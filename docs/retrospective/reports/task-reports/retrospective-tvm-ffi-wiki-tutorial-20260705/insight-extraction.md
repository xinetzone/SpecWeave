# TVM FFI Wiki 教程创建 — 洞察萃取

## 洞察概览

从本次任务中萃取了 3 个可复用洞察，其中 2 个为 P0 级高价值模式，1 个为 P1 级改进项。

---

## 洞察 1：Vendor 仓库高层文档优先研究法（P0）

### 5-Whys 根因分析

- **问题**：逐文件读取数十个 C++ 头文件来理解 TVM FFI 架构效率极低，且在 IDE timeout 环境下根本无法完成
- **Why1**：为什么逐文件读取效率低？→ 因为从头文件中需要"逆向工程"出架构概念，相当于从实现反推设计
- **Why2**：为什么需要反推？→ 因为没有先阅读项目自带的架构概述文档
- **Why3**：为什么没读概述文档？→ 传统认知中"源码才是最准确的"，忽略了项目为 AI Agent 准备的 AGENTS.md
- **Why4**：为什么 AGENTS.md 更高效？→ AGENTS.md 是项目维护者为 AI Agent 编写的"使用说明书"，包含目录结构、构建命令、代码规范、核心概念、CI 流程等全部高层信息
- **Why5**：为什么这是可复用模式？→ 越来越多的开源项目开始添加 AGENTS.md/CLAUDE.md/.cursorrules 等 AI 友好文档，这些文档是"自顶向下"研究的最佳起点

### 模式描述

**名称**：Vendor 仓库高层文档优先研究法
**触发条件**：需要研究/学习任何外部仓库或 vendor 子模块时
**核心动作**：
1. 第一步：查找并读取根目录下的 AI 友好文档（AGENTS.md > CLAUDE.md > .cursorrules > README.md）
2. 第二步：基于高层文档建立全局认知框架（目录结构、核心概念、构建方式）
3. 第三步：按需深入具体源码文件验证细节（而非逐文件通读）
4. 第四步：高层文档中已明确的信息（如代码规范、构建命令）直接使用，不需要从源码中重新推导

**预期收益**：研究效率提升 5-10 倍，避免在基础设施不稳定时卡死

**与现有模式关系**：补充了 vendor 协同框架中的"研究方法论"维度，此前 [VENDOR-INTEGRATION.md](../../../../docs/knowledge/VENDOR-INTEGRATION.md) 定义了协同规范但未涉及研究流程

## 洞察 2：基础设施故障降级策略（P1）

### 5-Whys 根因分析

- **问题**：Shell 管道耗尽 + WebFetch 超时 + Read timeout 同时发生时，任务似乎无法继续
- **Why1**：为什么任务没有卡死？→ 因为切换到了 sub-agent 执行和利用工具附带信息的降级策略
- **Why2**：为什么降级策略有效？→ sub-agent 运行在独立进程空间，不受主会话管道耗尽影响；Read 工具返回的规则内容（AGENTS.md）虽然是"副作用"，但包含了关键信息
- **Why3**：为什么一开始没想到？→ 本能反应是反复重试失败的工具，而非立即切换策略
- **Why4**：根本原因→ 缺少预定义的工具故障降级预案

### 模式描述

**名称**：工具故障三级降级策略
**触发条件**：遇到连续 2 次同类工具失败时
**降级优先级**：
1. **Level 1（最优）**：委托 sub-agent 执行——sub-agent 有独立进程空间和工具环境
2. **Level 2（次优）**：利用工具返回的附带信息——如 Read 工具在规则匹配时返回 AGENTS.md 内容，Glob 失败时用 LS 替代
3. **Level 3（保底）**：基于已有知识推进——前序会话摘要 + 通用领域知识，标注需要后续验证的部分

**反模式**：连续重试同一失败工具超过 2 次

## 洞察 3：大规模文档并行分组写作模式（P0）

### 5-Whys 根因分析

- **问题**：17 个章节文档串行编写需要 16+ 轮工具调用，耗时极长
- **Why1**：为什么可以并行？→ 各章节的写作是独立的，没有顺序依赖
- **Why2**：为什么要分组？→ 单个 sub-agent 写 17 个文件上下文过长、质量下降；分成 4 组每组 3-5 个文件，上下文聚焦
- **Why3**：分组依据是什么？→ 主题相关性：概述导航、核心API、高级功能、实战指南
- **Why4**：为什么需要在 prompt 中提供完整上下文？→ sub-agent 无法访问主会话的完整上下文，必须在 prompt 中传递所有关键信息（API 事实、格式要求、代码规范、导航格式）
- **Why5**：这为什么是可复用的？→ 任何大规模原子化文档创作（wiki 教程、API 文档、规范文档）都适用

### 模式描述

**名称**：主题分组并行写作模式
**触发条件**：需要创建 5 个以上原子化文档且各文档相对独立时
**操作步骤**：
1. 按主题相关性将文档分为 N 组（每组 3-5 个文件），每组主题内聚
2. 为每组创建一个 general_purpose_task sub-agent
3. 在 prompt 中传递：完整关键事实（API/概念/规范）、YAML frontmatter 格式、代码示例要求、导航链接格式、相对路径规则
4. 所有 sub-agent 并行执行，单轮完成全部内容
5. 执行后抽样验证质量

**分组原则**：
- 每组不超过 5 个文件（避免上下文过载）
- 组内文件主题连贯（如同属"核心API"或"高级功能"）
- 导航/索引文件与前两章同组（确保整体框架一致）

**预期收益**：写作效率提升 3-5 倍（取决于并行组数）

**与现有模式关系**：此前 IDL Wiki 教程（10个文件，2个并行sub-agent）已验证此模式的有效性，本次 TVM FFI（17个文件，4个并行sub-agent）进一步验证并优化了分组策略（3-5文件/组 vs 5-6文件/组）
