# 第一性原理思维训练题库创建任务复盘报告

## 执行摘要

本次任务（ACT-012）为第一性原理知识库创建思维训练题库，基于已有的六步方法论框架（问题定义→假设列举→拆解要素→质疑验证→重新构建→验证迭代），设计了43道分层级练习题和3个综合案例，总计2108行内容。任务经历完整的spec→实施→验证流程，质量检查全部通过，已原子提交（commit 5df6de5d）。

## 2. 过程分析

### 2.1 成功因素

1. **方法论框架结构清晰**：已有08-methodology-framework.md提供了明确的题目分类维度，每个Step的核心概念清晰
2. **三级难度设计合理**：入门题贴近日常生活（通勤、手机电量、保温杯）降低门槛；进阶级涉及工作场景（会议效率、技术选型）；挑战级触碰认知盲区（确认偏差、逆向思维）
3. **批判性视角一致性**：延续了知识库的审慎态度——
   - 所有答案标注"参考答案要点，非唯一正确解"
   - 包含Theranos等反例警示误用风险
   - 强调"现实优先于推理"、"第一性原理给方向不给答案"
   - 商业案例包含偏差提示（幸存者偏差等）
4. **details折叠答案**：避免读者直接看答案，鼓励独立思考；答案展示思考框架而非固定结论
5. **质量验证完整**：文件名规范检查+150个本地链接检查全部通过

### 2.2 遇到的问题

| 问题 | 影响 | 处理方式 |
|------|------|---------|
| 单线程内容创作效率较低 | 2108行文档全程单线程编写，耗时较长 | 本次接受，后续改进 |
| spec中"审慎态度"未量化为检查项 | human-judgment AC缺乏可操作审查标准 | 人工把控质量 |
| git-commit-utf8.py在目录add时误报 | 首次自动提交失败 | 手动git add + UTF-8临时文件提交 |
| 三级题目编号维护成本高 | 题X.Y.Z格式需要持续维护一致性 | 编写时注意编号对齐 |

## 3. 核心洞察

### 3.1 可复用模式

#### 模式A：练习题库三级设计模式（practice-question-bank-three-tier-design）

**适用场景**：为方法论/思维框架/技能创建训练题库

**核心结构**：
1. **专项练习**：按方法论步骤分解，每个专项内分三级难度
   - 🌱入门：日常生活场景，降低门槛
   - 📚进阶：工作/学习场景，贴近实际应用
   - 🔥挑战：认知偏差/反直觉问题，深度训练
2. **误区识别**：用反例训练识别误用（必须包含真实反面案例）
3. **综合案例**：跨步骤完整流程演练（覆盖不同领域展示通用性）
4. **答案设计**：details标签折叠 + 展示思考框架而非标准答案 + 永远标注"非唯一正确解"

**关键要素**：
- 入门题必须足够"接地气"（一支笔、一张纸、一杯咖啡）
- 挑战题必须触碰认知盲区（确认偏差、幸存者偏差、逆向思维）
- 误区识别不能只讲抽象道理，必须有真实反例

**成熟度**：L2（已在本项目验证成功）

#### 模式B：知识型产出物质量双维度模型（knowledge-artifact-quality-dual-dimension）

**适用场景**：创建知识文档/教程/训练材料等非代码类产出物

**核心洞察**：知识型产出物质量有两个维度：

| 维度 | 可自动化？ | 具体内容 |
|------|-----------|---------|
| 格式维度 | ✅ 是 | 文件名规范、链接有效性、格式一致性、frontmatter完整性、题数统计 |
| 价值维度 | ❌ 否 | 审慎态度（不做万能宣称、承认边界、有失败案例）、批判性视角（有反例、有偏差提示）、认知诚实（标注非唯一解、区分事实/判断）、教学有效性（难度梯度、思考过程展示） |

**实践启示**：
- 代码开发的质量门（lint/test/CI）只能覆盖第一维度
- 知识创作必须在spec阶段将第二维度明确为**具体的yes/no人工检查项**
- 不能因为"链接检查通过"就认为"文档质量合格"

**成熟度**：L1（本次首次萃取，待验证）

### 3.2 系统性问题：内容创作型任务的并行策略缺失

**5-Whys根因分析**：
1. 为什么执行慢？→ 单线程编写2108行内容
2. 为什么单线程？→ spec任务分解了8个任务，但未明确要求使用subagent
3. 为什么没要求subagent？→ 内容创作被当作"简单文档编写"，低估了工作量
4. 为什么低估工作量？→ spec估算题数（30-40题）时，未估算每题的场景设计+任务描述+提示引导+参考答案的总工作量
5. **根本原因**：内容创作型任务缺乏"工作量估算→并行策略"的标准流程，默认单线程执行

**改进方向**：预估超过500行的内容创作任务，spec阶段应规划subagent分章节并行，最后由主线程统一收口（风格一致性、交叉引用、编号统一）。

## 4. 经验教训

### 4.1 做对了什么

1. **在已有框架上构建**：严格对齐08-methodology-framework.md，保证内容一致性
2. **三级难度梯度设计**：不同水平读者都能找到适合题目，避免两极分化
3. **答案展示思考框架**：教"怎么想"比告诉"答案是什么"重要得多
4. **批判性视角贯穿**：不把第一性原理包装成万能方法，包含反例和偏差提示
5. **完整质量验证**：文件名规范、链接检查、内容完整性都执行到位

### 4.2 下次改进

1. **大文档并行创作**：2000+行文档拆成多个subagent任务，分章节并行+统一收口
2. **spec细化human-judgment检查项**：将"审慎态度"转化为具体yes/no检查清单
3. **简化题目编号**：连续编号+Step标签代替三级编号，降低维护成本
4. **增加题目标签系统**：除难度外增加"认知偏差""假设挖掘""物理极限"等标签

### 4.3 可迁移经验

1. **教学材料核心是展示思考过程**：好的教程永远展示"我是怎么想的"而非"正确答案是什么"
2. **反例比正例更有教学价值**：告诉人们"什么是错的""方法容易被怎样误用"比只展示正确用法更能建立深度理解
3. **入门题必须足够接地气**：用日常生活中最常见的事物做入门训练，极大降低心理门槛
4. **知识创作也要原子化**：大文档拆分为独立章节，每个章节可独立验证，最后统一集成
