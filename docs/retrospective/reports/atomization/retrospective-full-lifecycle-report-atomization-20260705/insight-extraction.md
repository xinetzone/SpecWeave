# 洞察萃取

## 一、核心发现

### 洞察1：元原子化的"二分+概览"模式是复盘报告拆分的标准范式

**事实**：本次原子化与20260623首次综合复盘原子化采用了完全一致的拆分策略——execution部分按"前半/后半"二分为s1-s3/s4-s7，大段详细对比明细拆为独立detail文件，主文件保留概览表+链接。

**本质**：复盘报告的原子化存在一个可复用的标准范式：
1. **时间线性内容**（阶段叙述/执行过程）→ 按时间二分或三分，每个文件覆盖连续阶段
2. **详细对比/列表内容**（模板对比/数据明细/代码示例）→ 独立为details文件
3. **主文件保留**：概览表+导航链接→指向原子文件，自身控制在<200行

**复用价值**：未来复盘报告原子化可直接套用此模式，无需每次重新设计拆分策略。

### 洞察2：上下文压缩恢复需要"文件状态快照"而非仅"计划摘要"

**事实**：接续压缩会话时，摘要提供了完整的拆分计划（什么文件拆成什么），但缺少"哪些文件已创建/哪些Edit已完成"的精确状态。执行过程中需要先Read文件确认当前内容，避免基于过期假设操作。

**本质**：当会话被压缩并延续时，仅有TODO列表和计划是不够的，因为：
- 计划描述了"要做什么"，但不描述"做到哪一步了"
- 文件可能在压缩前被部分修改，但摘要无法精确反映修改边界
- 不同Edit操作的原子性不同（创建新文件vs修改现有文件风险不同）

**改进方向**：压缩前的摘要应包含每个TODO项的精确完成状态（已完成/进行中/未开始）和关键文件的当前行数/结构快照。

### 洞察3：复盘报告原子化比代码原子化更简单——因为内容天然具备叙事边界

**事实**：本次原子化全程没有遇到"内容边界模糊"的困难——七阶段之间有明确的时间边界和`---`分隔符，6个模板对比之间也有清晰的编号和`---`分隔符。相比之下，Python代码原子化经常遇到"函数A依赖函数B的内部状态"这类耦合问题。

**本质**：文档（尤其是复盘报告）的天然结构使其比代码更容易原子化：
- 章节之间通过编号和标题形成天然边界
- 叙事体内容（阶段叙述）按时间线组织，耦合度低
- 列表/对比内容（模板对比）按条目组织，条目间独立
- Markdown的`---`分隔符提供了显式的切割标记

**推论**：文档原子化的主要难点不在"怎么拆"，而在"拆完之后怎么维护导航一致性"——即所有引用旧章节位置的地方都需要更新。

## 二、可复用模式

### 模式：报告原子化"概览表+详情文件"分离模式

| 属性 | 值 |
|------|-----|
| 模式ID | report-overview-detail-split |
| 触发条件 | 报告中某章节包含大量并列详细对比/列表，占文件>40%篇幅 |
| 核心做法 | 将详细内容拆为独立details文件，主文件替换为概览汇总表+链接 |
| 概览表要素 | 条目名称、关键量化指标、链接指向 |
| 主文件保留 | 总论/背景/量化分析/结论/建议 |
| details文件 | 完整保留原详细内容，frontmatter标注source |
| 成熟度 | **L2（已验证）**：本次在l3-pattern-application-report上验证成功，与20260623复盘模式一致 |

### 模式：执行复盘"时间二分"拆分模式

| 属性 | 值 |
|------|-----|
| 模式ID | execution-retrospective-time-bisect |
| 触发条件 | 执行复盘包含≥5个连续阶段叙述，单文件>300行 |
| 核心做法 | 按"奠基期/成熟期"或"前半/后半"二分为两个文件 |
| 分界选择 | 选择叙事天然转折点（如"误认为结项"→后续为自举期） |
| 文件命名 | execution-phases-s{N}-s{N}.md 格式 |
| 互链要求 | 两个文件互相引用，s4-s7开头链接回s1-s3 |
| 成熟度 | **L3（标准化）**：已在20260623和20260705两次复盘上验证，结构一致 |

## 三、反模式与教训

### 教训1：不要凭记忆调用脚本CLI参数

**发生**：check-links.py首次调用使用了位置参数（直接传路径），但脚本实际需要`--path` flag。

**根因**：没有先Read脚本的--help或查阅L2文档，凭记忆使用。

**预防**：调用不常用的脚本时，先用`python script.py --help`或查L2文档确认参数格式。

### 教训2：PowerShell行数统计不可靠时用Read验证

**发生**：Get-Content | Measure-Object -Line报告s1-s3为77行，但文件实际有100行内容（含frontmatter后空行处理差异）。

**根因**：PowerShell的Measure-Object对UTF-8 BOM文件和末尾空行的处理可能不一致。

**预防**：行数统计有疑虑时，直接Read文件确认内容完整性，不依赖PowerShell统计。
