# 洞察萃取：火山引擎ACEP云手机学习任务

[CMD-LOG] | level=INFO | cmd=insight | step=S3 | event=ROOT_CAUSE_FOUND | session=insgt-20260707-acep-insights | msg=根因分析完成：识别3个核心洞察

## 一、关键发现（Key Findings）

### 洞察1：动态网页内容采集需要"工具分层降级"策略

**发现描述**：
本次任务中WebFetch作为首选工具失败（返回重复且不全的内容），切换到浏览器工具后才成功获取完整页面。这不是偶发问题——现代云厂商产品页普遍采用React/Vue等SPA框架+动态渲染，静态HTTP抓取（WebFetch）的成功率正在系统性下降。

**支撑事实**：
- 火山引擎ACEP产品页采用动态渲染，WebFetch返回的内容存在明显重复和缺失
- 浏览器工具（browser_navigate + browser_evaluate）通过真实浏览器环境执行JavaScript，能获取完整渲染后的DOM内容
- 近期多个火山引擎产品学习任务（HiAgent、Ark CLI、Sandbox等）都遇到了类似的WebFetch内容不全问题

**5-Whys根因分析**：
```
为什么WebFetch获取内容不全？
→ 因为产品页采用SPA动态渲染，初始HTML只包含壳内容，实际内容通过JavaScript异步加载

为什么静态抓取无法获取异步加载内容？
→ 因为WebFetch只发起单次HTTP请求获取初始HTML，不执行JavaScript、不等待异步数据加载

为什么不直接默认使用浏览器工具？
→ 因为浏览器工具启动成本更高（需要启动浏览器实例），且之前没有建立明确的工具选择决策树

为什么没有建立工具选择决策树？
→ 因为外部产品学习任务的工具策略还停留在"先用简单工具，失败再切换"的经验式应对，没有系统化沉淀
```

**深层含义**：
内容采集工具的选择不应基于"工具简单性"，而应基于"目标站点技术特征"。需要建立基于站点类型的工具选择策略，而不是失败后重试。

### 洞察2：B端技术产品页面遵循"认知递进"七段式信息架构

**发现描述**：
ACEP云手机产品页的信息组织不是随意排列，而是严格遵循用户购买决策路径的"认知递进"逻辑：Hero区建立认知→能力模块建立信任→优势模块强化价值→场景模块匹配需求→架构模块展示技术实力→案例模块提供社会证明→CTA模块促成转化。这七段式结构在火山引擎多个产品页中重复出现，是高度成熟的B端产品展示模式。

**支撑事实**：
- ACEP页面结构：Hero（首屏宣传）→产品规格/管理/音视频/监控（四大能力）→功能/弹性/安全/流畅（四大优势）→云游戏/测试/直播/审核/办公（五大场景）→技术架构→四个客户案例→文档/控制台/体验入口
- 同目录其他火山引擎产品分析（HiAgent等）也呈现类似的递进结构
- 每一段的目标都是回答用户决策路径上的一个关键问题："这是什么？"→"能做什么？"→"为什么选你？"→"我能用在哪里？"→"技术靠谱吗？"→"别人用得怎么样？"→"怎么开始？"

**5-Whys根因分析**：
```
为什么B端产品页普遍采用这种七段式结构？
→ 因为B端购买决策周期长、参与角色多，需要按决策阶段逐步建立信任，而不是像C端产品那样冲动转化

为什么是这七个特定段落？
→ 因为这七个段落正好对应B端技术决策者从认知到转化的七个关键心理阶段，每个阶段解决一个核心疑虑

为什么这种结构能跨产品复用？
→ 因为B端技术采购的决策逻辑是通用的（ regardless of具体是云手机、AI Agent还是其他产品），决策者的疑虑类型和建立信任的路径高度相似
```

**深层含义**：
B端技术产品展示不是"功能列表堆砌"，而是"用户决策路径设计"。理解并复用这个七段式结构，不仅能更好地分析竞品页面，也能指导我们自己的产品展示设计。

### 洞察3："产品功能+UX设计"双轨分析模式能产生超额价值

**发现描述**：
本次任务用户明确要求不仅提取产品信息，还要分析"内容组织方式、信息呈现逻辑及用户体验设计"。最终产出的12章笔记中有3章是UX专项分析（信息架构、UX特点、设计模式总结），这3章内容的可复用价值远超纯产品信息提取——产品信息会随产品迭代而过时，但设计模式和UX经验是跨产品、跨时间可迁移的。

**支撑事实**：
- 纯产品信息（如"延时<70ms"）是ACEP特定的，其他产品不适用
- UX设计模式（如"量化数据前置"、"客户logo墙社会证明"、"场景-价值对应表格"）是通用的，可用于分析任何B端产品页，甚至指导我们自己的页面设计
- 第9章总结的7个B端产品展示设计模式，是本次任务中半衰期最长的知识资产

**5-Whys根因分析**：
```
为什么UX分析的可复用价值更高？
→ 因为UX模式是关于"如何有效展示信息"的元知识，而产品信息是关于"这个产品有什么"的对象知识——元知识的迁移性远高于对象知识

为什么之前的产品学习任务有时会忽略UX分析？
→ 因为任务要求通常聚焦于"理解这个产品"，而UX分析需要额外的视角切换和抽象思考，容易被当作"额外工作"

为什么用户这次明确要求UX分析？
→ 因为用户意识到，对于AI智能体的学习任务来说，提取可迁移的方法论模式比记住单个产品的功能列表更有长期价值
```

**深层含义**：
外部产品/竞品分析的价值不应仅停留在"了解对方有什么功能"（这是情报收集），更应上升到"提炼可复用的设计/产品/方法论模式"（这是知识萃取）。双轨分析模式应成为竞品学习类任务的标准配置。

## 二、可复用模式萃取

### 模式A：动态站点内容采集的"预判-降级"双轨流程

> **入库状态**：现有L2模式升级验证（非新模式）

**模式描述**：
对于外部网页内容采集任务，不采用"先用简单工具，失败再切换"的被动重试策略，而是采用"预判-降级"双轨流程：
1. **预判层**：先识别目标站点类型（云厂商产品页/文档站/博客/技术wiki等），基于站点类型预判动态渲染程度
2. **执行层**：对预判为高动态的站点（如火山引擎/阿里云等云厂商产品页），直接使用浏览器工具；对静态站点优先使用WebFetch
3. **降级层**：即使预判为静态站点，如果WebFetch结果出现内容重复、缺失、结构异常，立即降级到浏览器工具，不做无效重试

**适用场景**：
- 所有外部网页内容采集任务
- 竞品分析、产品学习、技术调研类任务
- 需要完整提取动态渲染页面内容的场景

**触发条件**：
- 目标URL属于云厂商产品页（volcengine.com/aliyun.com/aws等）
- 目标站点是已知的SPA应用（React/Vue/Angular构建）
- WebFetch返回结果出现明显的内容重复或关键区块缺失

**验证状态**：✅ 现有模式第8次验证（L2-verified）
- 对应模式文件：[external-website-analysis-fallback-strategy.md](../../../patterns/methodology-patterns/research-knowledge/external-website-analysis-fallback-strategy.md)
- validation_count：7 → 8（ACEP为第8次验证案例）
- 成熟度：保持L2-verified

### 模式C：外部产品学习"功能信息+UX模式"双轨分析框架

> **入库状态**：⏳ L1候选模式（待后续验证）

**模式描述**：
外部产品/竞品学习任务不应仅提取"这个产品有什么功能"的对象级信息，应同时进行两条分析轨道：
1. **产品信息轨道**：产品定位、核心能力、功能规格、性能指标、应用场景、技术架构、客户案例——这是"了解对方"的情报收集
2. **UX/方法论轨道**：信息架构设计、内容组织逻辑、转化路径设计、展示模式提炼、可复用经验——这是"萃取方法"的知识沉淀

两条轨道的产出比例建议为 7:3（70%产品信息，30%UX/方法论分析），确保既深入理解产品，又不遗漏可迁移的高价值知识。

**适用场景**：
- 竞品分析与研究
- 外部产品系统性学习
- 行业标杆案例拆解
- 设计/产品方法论积累

**触发条件**：
- 用户要求"系统性学习"、"深度洞察"、"分析设计/UX"
- 任务目标包含"总结可借鉴经验"、"提炼模式"
- 高价值标杆产品/页面的分析任务

**验证状态**：⏳ L1-experimental（待后续3+任务验证后正式入库）
- 本次任务验证有效，需在后续产品学习任务中验证框架完整性和产出价值稳定性
- 待验证要点：UX分析占比合理性、7:3产出比例适用性、双轨切换成本

## 三、改进建议（基于洞察）

### 建议1：建立外部网页内容采集工具选择决策树

**问题**：WebFetch对动态站点成功率低，目前依赖失败后手动切换浏览器工具，效率低
**建议措施**：
1. 在`.agents/tools/`或相关规范文档中增加"网页内容采集工具选择决策树"
2. 明确标注：火山引擎/阿里云/AWS等云厂商产品页→直接使用浏览器工具，跳过WebFetch
3. 增加WebFetch结果质量快速校验规则（内容重复度、关键区块完整性检查），发现异常立即降级
**优先级**：高
**预期效果**：内容采集环节重试时间减少50%+，避免无效的WebFetch尝试
**验收标准**：后续3个云厂商产品学习任务无需WebFetch重试，直接使用浏览器工具一次成功
**执行状态**：✅ 已完成——现有模式[external-website-analysis-fallback-strategy.md](../../../patterns/methodology-patterns/research-knowledge/external-website-analysis-fallback-strategy.md)已完整覆盖此内容（含云厂商SPA预判、工具选择优先级、浏览器MCP四步SOP），本次补充ACEP作为第8次验证案例
