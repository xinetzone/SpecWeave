### 4.0.3 交叉叙事线（两个作品互相增强的唯一路径）

```
竹简悟道 Demo 帖的 §4「TRAE 实践过程」中：

  「在开发过程中，我同时维护了一套 AGENTS.md 规范体系（SpecWeave）——
   它让 TRAE 在 142 次对话中始终理解我的架构意图，从未偏离帛书哲学链的核心逻辑。」

  ↓ 评审看到的不只是「我用 TRAE 做了一个 App」
```

**SpecWeave 的参赛价值不是「多一个作品多一次机会」——规则已明确同账号只取最高分 1 个作品晋级。它的价值是「让竹简悟道的 TRAE 应用深度维度打出满分」**。

## 4.2 SpecWeave（第二作品）参赛策略

> SpecWeave 是一个「方法论基础设施」——它不是一个面向终端用户的 App，而是一个让 TRAE 本身更强大的元工具。它的参赛价值不在独立的四维得分，而在于**增强竹简悟道的叙事深度**。

### 4.2.2 轻量级 Demo 帖策略

SpecWeave 的 Demo 帖**投入控制在 1 小时以内**——核心目的不是独立晋级，而是：

| 目标 | 实现方式 |
|------|---------|
| 证明方法论体系的存在 | 展示 .agents/ 四层架构目录截图 + 34 个方法论模式列表 |
| 证明 TRAE 深度 | 142 次对话的时间线 + 23 个验证脚本的 CI 截图 |

| 留下品类独占证据 | 30+ 官方灵感示例矩阵图——证明无任何同类作品 |

## 4.3 共享策略（两作品通用）


### 4.3.1 全人工评审叙事策略

大赛采用**全人工评审**（初赛/复赛/决赛），这意味着叙事说服力与量化证据同等重要。两作品在全人工评审环境下的叙事策略：

| 叙事要素 | 竹简悟道 | SpecWeave | 评审感知 |
|---------|---------|-----------|---------|
| 第一印象 | 竹简卷轴 UI——视觉差异化远超同赛道 C 端应用 | .agents/ 四层架构目录——文档即体验 | 两作品在不同维度上都产生「和之前看过的都不一样」的认知切换 |
| 差异化故事 | 「所有 AI 追求更快，我刻意追求更慢、更深、更静」 | 「所有人都在做应用，我在研究怎么做的方法论」 | 两作品都具备品类独占性，在不同维度产生记忆锚点 |
| 证据链 | ≥3 张竹简 UI 截图 + 哲学链数据结构 Session ID | 142 次对话时间线 + 34 个模式列表 + 23 个验证脚本 | 量化证据密度远超平均值 |

> **一句话**：人工评审看的不只是产品，而是「这个人用 TRAE 做了什么别人没做过的事」。你有一个故事，你有两个维度的证据。

### 4.3.2 Demo 提交格式策略

官方推荐的三种 Demo 提交格式中，**交互式 HTML ZIP 打包上传社区**是两作品的共同选择。但关键差异化在于：

| 作品 | HTML 核心内容 | 差异化要点 |
|------|------------|-----------|
| 竹简悟道 | 竹简卷轴 UI（帛书 81 章展开交互） + 四路径 AI 反思对话 | 视觉冲击力——展开竹简的仪式感是其他所有 HTML 中不存在的体验 |
| SpecWeave | 四层架构 Mermaid 可视化 + 量化仪表盘 + 对比实验数据 | 信息密度——不再像 TRAE Work 自动生成的「AI 风格」页面 |

**关键操作警告**：学习资料提供了 3 套标准 Prompt 模板——大量参赛者的 HTML 文件将由 TRAE Work 根据创意 Doc **自动生成**，导致页面结构、视觉风格和信息密度高度一致。两作品的 HTML 均**不应**通过 TRAE Work 的「创意 Doc → HTML」路径自动生成——应手动构建或精细控制 HTML 结构和视觉设计，在同质化的 HTML 海洋中制造结构性差异。
