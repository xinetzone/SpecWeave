# 四、参赛策略与行动建议（晋级公示对齐 v9）

## 4.1 参赛策略总纲

### 4.1.1 赛道选择（官网确认版）

| 赛道 | 参与方式 | 赛道标签 | 策略 |
|------|---------|---------|------|
| **学习工作 / 造个新解法** | 主报名 | `#学习工作` 或 `#造个新解法` | 核心战场，全部资源投入 |
| 社会公益（附加） | 附加报名 | 对应子赛题标签 | 轻量参与，报名帖增补公益段落 |

**选择理由**：官网学习工作赛道描述为「聚焦新一代学习与工作方式，探索更高效、协同、智能的学习、办公与职业成长体验」——SpecWeave 在 6 个赛道关键词上均获得 ⭐⭐⭐⭐+ 匹配度。

### 4.1.2 目标设定（奖项锚定版）

基于官网详细奖项分布，建议采用「三级目标」策略：

```
最高目标：全场冠军 ¥300,000（不排除，但不依赖）
核心目标：赛道大奖 ¥50,000  ← 资源聚焦点
安全底线：Builder 创造者奖 ¥10,000（前 13 名）
锦上添花：社会公益特别奖 ¥50,000（附加赛道，额外机会）
```

**为什么赛道大奖比全场冠军更可行**：
- 全场冠军需全赛道 TOP 1——生活娱乐赛道中天然有更高感官冲击力的作品（游戏/互动体验）
- 赛道大奖仅需学习工作赛道 TOP 1——赛道内品类独特，无同类竞争者
- 赛道大奖 ¥50,000 × 4 + Builder 奖 ¥10,000 × 13 = 比仅设一个全场冠军更分散竞争

**奖金池确认**：赛事细则确认总奖金池为 **113 万元**，主奖项和专项奖可叠加，一个作品最高可得 35 万元现金大奖。阶段激励（报名/初赛/复赛各阶段达成要求后的奖品）与决赛大奖相互独立。

### 4.1.2a 🆕 单作品聚焦策略（赛事细则确认）

赛事细则 §3.1 明确：**同一账号下只取得分/人气分最高的 1 个作品晋级**。这是一个强制的 Best Shot 机制——多作品策略在规则层面即无效。

对 SpecWeave 的策略意义：

| 此前可能的考量 | 规则后的定论 |
|--------------|------------|
| "要不要再做一个小工具作为备选作品？" | **不需要**——只取最高分的一个，做备选等于分散资源 |
| "方法论体系 vs 小巧 Demo 哪个更讨评审喜欢？" | **方法论体系**——你唯一的作品必须是你的最强表达 |
| "要不要分几个模块分别投稿？" | **不需要**——AGENTS.md 入口统一路由天然是一个完整作品 |

**结论**：100% 资源灌注 SpecWeave 核心体系，不做任何分散。

### 4.1.2b 🆕 四维评审策略分配（初赛指南确认权重 ⭐⭐⭐⭐⭐）

初赛专业评审四个维度的权重已确认。以下从"得分策略"而非"均匀覆盖"的角度制定执行计划：

| 维度 | 权重 | 策略定位 | 具体行动 | 预计时间 |
|------|------|---------|---------|---------|
| **产品创新性** | 30% | **冲击满分** | Demo 帖开头用一句话重新定义评判框架："这不是一个传统应用——它是一个用 TRAE 发现并系统化的 AI 开发方法论。" | 0.5h |
| **用户体验与产品完成度** | 30% | **定向补强（最高优先级）** | 制作交互式 HTML 导航页（四层架构可视化 + 量化数据仪表盘）+ 3-5 分钟对比实验录屏 | 3h |
| **TRAE 应用深度** | 20% | **冲击满分** | 在 Demo 帖中展示 142 次对话的时间线 + ≥3 个关键 Session ID + 23 个验证脚本 + CI 检查截图 | 0.5h |
| **社会/商业价值** | 20% | **叙事突出** | Demo 帖中强调 Apache 2.0 开源（数字包容性公益）+ AGENTS.md 标准生态贡献 + 任何 AI 开发团队可复用 | 0.25h |

**资源分配逻辑**：60% 精力在叙事铺排（创新性+TRAE 深度=50%权重），40% 精力在用户体验补强（完成度=30%权重），社会价值仅需在帖子文字中自然呈现（20%权重）。

策略核心——**不是均匀覆盖四维，而是用两维的满分冲击四维的总分**。

### 4.1.2f 全人工评审策略适配

赛事细则 §5.3 明确：**初赛、复赛和决赛全部采用人工评审**。与 AI 评审相比，人工评审需要不同的呈现策略：

| 策略维度 | AI 评审导向 | 人工评审导向（实际） | SpecWeave 行动 |
|----------|-----------|-------------------|---------------|
| 作品呈现 | 关键词密度、量化指标 | **叙事清晰度、差异化记忆点** | 报名帖/初赛帖都用"故事"而非"规格说明书"的风格 |
| 证据链 | 数据点即可 | **完整证据链的主观可信度** | 展示 142 次迭代的时间线 + Session ID 列表（不是"我有"，而是"你看"） |
| 首印象 | 信息提取效率 | **情感共鸣 + 认知冲击** | 报名帖开头用一句话制造"这是我第一次看到有人这么做"的感觉 |
| 技术细节 | 越多越好 | **够用即止——评审不是你的用户** | 技术实现细节放在仓库 README，帖子中只用类比和可视化 |

**核心原则**：人工评审不是"评分算法"——他们是在快节奏下做主观判断的人。给他们一个**3 秒钟能记住的标签**（"那个研究 Vibe Coding 方法论的小伙"），比给他们一堆参数更有效。

### 4.1.2c 🆕 Demo 提交格式确认（保姆级教程明确）

教程确认了 Demo 提交的三种官方认可格式：

| 格式 | 说明 | 适用场景 | SpecWeave 选择 |
|------|------|---------|---------------|
| 可公开访问的体验链接 | 部署到服务器的 URL | 需要后端的 Web 应用 | ❌（无后端需求） |
| **交互式 HTML ZIP 打包上传** | 打包为 ZIP 直接上传社区 | 纯前端交互式 Demo | ✅ **官方推荐路径** |
| Bilibili 演示视频 | 硬件交互赛道专用 | 不便提供在线体验的场景 | ❌（非硬件赛道） |

**关键认知修正**：教程原文——**"可运行 ≠ 部署上线"**。评审的标准是能正常体验核心功能、理解 Demo 的核心价值即可达标，无需做成正式上线产品。

**对 SpecWeave 的策略影响**：

| 影响维度 | 说明 |
|---------|------|
| 部署焦虑消除 | 无需租服务器、配域名、搭 CI/CD——一个 HTML ZIP 即是一等提交格式 |
| 文档即产品 | 交互式导航页（四层架构可视化 + 量化数据展示）打包为 ZIP = 合法的 Demo 提交物 |
| TRAE Work Auto 模式 | 报名帖 HTML 按官方推荐的 Auto 模式生成——操作路径与官方教程完全一致 |
| 竞争力增强 | 大多数 0 技术门槛参赛者的 HTML ZIP 是"3-5 分钟生成的创意提案"；SpecWeave 的 HTML ZIP 是"142 次迭代后的知识体系可视化"——同一格式，天壤之别的内容密度 |

### 4.1.2d 🆕 报名 HTML 差异化策略（创意文档学习资料揭示的同质化风险）

学习资料提供了 3 套标准 Prompt 模板——大量参赛者的 HTML 文件将由 TRAE Work 根据创意 Doc **自动生成**，导致页面结构、视觉风格和信息密度高度一致。这暴露了一个关键的差异化窗口：

| 层面 | 标准化路径（其他参赛者） | 差异化路径（SpecWeave） |
|------|----------------------|---------------------|
| 生成方式 | TRAE Work 自动生成 HTML | **手动构建**——四层架构 Mermaid 可视化 + 量化仪表盘 + 对比实验数据 |
| 视觉风格 | "AI 生成感"——标准布局、通用配色 | 定制化设计——信息密度远超自动生成的上限 |
| 信息维度 | 创意描述（文字为主） | 方法论可视化（Mermaid 图）+ 数据证明（142 次对话量化）+ 效率对比（有无规范实验） |

**具体行动**：报名帖 HTML 不应通过 TRAE Work 的"创意 Doc → HTML"路径生成，而应手动构建为一个独立的交互式页面，包含：
- 四层架构的 Mermaid 可视化（规范层/工程层/治理层/知识层）
- 量化仪表盘（70+ 交付物、142 次迭代、23 个验证脚本）
- 对比实验数据展示（有 AGENTS.md vs 无规范）

> 这不是"做一个更好看的 HTML"——这是"创建一个评审从未见过的品类"。

| 子项 | 策略化内容 |

| 大概是什么产品 | 一套基于 AGENTS.md 开放标准的 AI 智能体开发规范体系（开源仓库），包含 7 个角色定义、5 项协作协议、23 个验证脚本和 34 个方法论模式。 |

#### 第 2 部分：目标用户及痛点（~120 字）

| 子项 | 策略化内容 |
|------|----------|
| 面向哪些用户 | AI 开发团队（5-20 人规模）和个人 Vibe Coder |
| 在什么场景下使用 | 团队在 TRAE 中启动新项目时，需要统一的 AI 协作规范；个人开发者在每次 TRAE 对话前，需要一份"好的对话长什么样"的指南 |
| 当前痛点 | 每次对话从零开始，上下文丢失后需要反复重复项目背景；AI 角色边界不清晰导致产出质量不稳定；好的协作经验停留在单次对话中，无法跨项目复用 |

#### 第 3 部分：价值与意义（~120 字）

**从效率提升角度**：142 次 TRAE 协作实践验证，有 AGENTS.md 规范的对话完成同一任务的效率提升显著——不需要每次向 AI 重复解释项目背景和规则。

**从社会价值角度**：Apache 2.0 开源，免费服务于全球 AI 开发者。降低 AI 协作的门槛，让更多非技术背景的人也能高效使用 AI 进行开发——推动数字包容。

#### 第 4 部分：HTML 产物说明

用 TRAE Work（Auto 模式）生成一个交互式 HTML 文件，内容包括：
- SpecWeave 四层架构可视化（感知→认知→执行→治理）
- 关键量化数据展示（70+ 交付物 / 34 个模式 / 23 个脚本 / 16+ 份复盘）
- 一句话价值主张：「让 AI 协作不再是随机对话，而是有序工程」
- 文件大小控制在 5MB 以内（社区上传限制 20MB）

**TRAE Work 生成策略**：
1. 使用 Auto 模式
2. 输入一句话：「生成一个展示 AI 智能体开发规范体系的单页 HTML 页面，包含四层架构流程图、核心数据展示和一句话价值主张。使用现代简约设计，配色以深蓝和白色为主。」
3. 在生成结果基础上迭代优化（补充具体数据、调整布局）

### 4.3.3 抖音传播策略（抖音表单对齐版，⚠️ 此前话题格式有误已修正）

#### 发布规范（硬性要求）

每条抖音视频发布时**必须同时满足**以下条件才能申请流量扶持：

| 要求 | 精确格式 | 备注 |
|------|---------|------|
| 话题 1 | `#vibecoding 大赏` | ⚠️ 此前错误写为 `#vibe coding 大赏`（含空格），已修正 |
| 话题 2 | `#TRAEAI 创造力大赛` | ⚠️ 此前错误写为 `#TRAE AI 创造力大赛`（含空格），已修正 |
| 提及 1 | `@TRAE` | 硬性要求，缺一不可 |
| 提及 2 | `@抖音科技` | 硬性要求，缺一不可 |

#### 流量申请流程（关键路径）

```
发布抖音视频（含话题+@）→ 填写飞书表单 → 2 工作日人工审核 → 获得 50,000 曝光/条
│                      │
│                      └── 需要提供：抖音用户名、Demo 帖链接、抖音帖链接、社区个人主页截图
│
└── 前置条件：必须先通过报名审核，且已在初赛区发布 Demo 帖
```

> ⚠️ **关键认知修正**：官方抖音流量不是发布即得——发布抖音只是"产生内容"，填写飞书表单才是"申请流量"。漏填表单 = 无流量扶持。

#### 内容方向矩阵

| 优先级 | 内容方向 | 视频形式 | 话题（精确） | 目标互动 | 预计制作时间 |
|--------|---------|---------|------------|---------|------------|
| P0 | Vibe Coding 深度对比 | 对比实验：有规范 5 分钟完成 vs 无规范 20 分钟 | 两个话题 + 两个 @ | 开发者共鸣评论 | 2 小时 |
| P1 | 知识可视化 | Mermaid 四层架构动画 + 34 个模式思维导图 | 两个话题 + 两个 @ | 收藏 + 提问 | 1 小时 |
| P1 | 成果大赏 | 142 次提交 + 23 个脚本 + 10 个概念，数字滚动展示 | 两个话题 + 两个 @ | 互动（数字冲击） | 0.5 小时 |
| P2 | 方法论故事 | 「我是如何从混乱到有序的」第一人称叙事 | 两个话题 + 两个 @ | 情感共鸣 | 1 小时 |

#### 发布节奏建议

```
Week 1（初赛提交后）：对比实验（首发，冲击力最强）
Week 2：知识可视化（承接流量，解答"这到底是什么"）
Week 3：成果大赏（数据证明，加深印象）
Week 4：方法论故事（情感升华，完整叙事）
```

每条视频发布后**立即填写飞书表单**申请流量扶持。每条符合条件的帖子获得 50,000 曝光——按 4 条计算，理论总曝光可达 200,000。

> **核心策略**：抖音人气分 = 点赞 + 评论×2 + 收藏 + 转发，评论权重最高——设计"引发讨论"的内容（如"你认为 AI 协作需要规范吗？"）。

### 4.3.4 🆕 复赛产品定义（7 月 21 日-8 月 9 日，如晋级）

复赛要求「交付一个具备完整用户路径和核心能力的正式产品」。对 SpecWeave 而言：

| 产品要素 | 当前状态 | 复赛增强方向 |
|----------|---------|-------------|
| 核心能力 | .agents/ 规范体系 + 23 个验证脚本 | CLI 工具（specweave init/check/generate） |
| 用户路径 | 阅读文档 → 理解规范 → 手动应用 | 交互式 CLI 引导（specweave wizard） |
| 产品外壳 | GitCode 仓库（被动） | 独立产品站（specweave.dev）+ 交互式 Playground |
| 用户留存 | 克隆即走 | 定期复盘方法论更新 + 社区贡献机制 |

**复赛资源申请**：如晋级复赛，可获得 TRAE 导师一对一指导（产品/技术）——优先选择产品导师帮助打磨「方法论 → 产品」的转化路径。

### 4.3.5 🆕 决赛路演策略（8 月 21-22 日，如晋级）

决赛为「作品展示 + 项目讲解 + 现场答辩」三位一体线下路演。建议叙事结构：

```
开场（30 秒）：我在 TRAE 中工作了 142 次对话，发现了一个问题——
            Vibe Coding 很火，但所有人都在"做"，没人在"研究怎么做"

产品展示（2 分钟）：SpecWeave 四层闭环体系演示
            → AGENTS.md 入口 → .agents/ 规范 → 验证脚本 → 复盘方法论

量化证据（1 分钟）：70+ 交付物 / 34 个模式 / 23 个脚本 / 10 个概念
            —— 所有产出都在 TRAE 中完成

价值陈述（1 分钟）：
            对 TRAE 生态：Vibe Coding 的方法论支撑
            对开发者：可克隆、可迁移、可持续演进
            对行业：AGENTS.md 标准的中国实践

答辩准备：预判评审可能的 5 个问题
            ① "和其他 AI 开发工具兼容吗？"（AGENTS.md 是开放标准）
            ② "有真实用户吗？"（AgentForge 落地案例）
            ③ "赛后怎么持续？"（Apache 2.0 开源 + 发展规划）
            ④ "和 TRAE 的竞品对比？"（TRAE 生态共建者，非竞品）
            ⑤ "最大的难点是什么？"（从单个实践萃取为方法论——需要足够多次的复盘迭代）
```
