# 导出建议通用模式目录（Export Suggestions Pattern Catalog）

## 模式类型
方法论模式

## 目标

将单次项目的“导出建议清单”抽象为可迁移的通用模式集合，用于任何同类任务的导出建议梳理、落地、验证与收口。

## 一、原文件建议要素梳理（需求 / 场景 / 问题 / 解决方案）

本节把源文件中的内容拆成可复用的“要素单元”，用于后续模式提取与覆盖验证。

### 1.1 需求（What）

1. 交付物可追溯：列出**已交付资产清单**，便于复用与审计
2. 行动可执行：按优先级分层输出“下一步行动”，并能追踪状态（✅/☐）
3. 运维可持续：提供日志归档、镜像清理、备份等长期维护建议
4. 使用可复制：提供可直接复制执行的命令片段（构建切换、SSH 修复、备份/清理）
5. 关联可导航：明确与其他资产（memory/知识库/脚本/文档）之间的链接关系

### 1.2 场景（When）

1. 环境交付：产出一个可复用的开发/调试环境（容器、脚本、文档）
2. 构建修复：解决“某构建路径失败”并给出稳定的替代路径
3. 重命名重构：镜像/目录/脚本/文档存在命名漂移，需要统一收口
4. 远程接入：提供 SSH / Remote-SSH 入口并处理指纹变更
5. 长期治理：为“可维护性/可复用性/可审计性”补齐制度与工具

### 1.3 问题（Why）

1. 证据丢失：日志散落在源码目录导致仓库膨胀、证据难找
2. 路径不一致：文档、镜像、入口脚本的默认值不一致导致“写的是 A，跑的是 B”
3. 工具链漂移：编译器/依赖版本变化造成非预期失败，需要策略化决策
4. 连接不稳定：容器重建后 SSH host key 变化导致无法连接
5. 旧资产残留：旧镜像/旧容器残留造成空间浪费、误用风险

### 1.4 解决方案（How）

1. 用“导出建议模板”组织输出：资产清单 → 行动计划 → 命令片段 → 备份/清理 → 关联资源
2. 用“状态标记 + 分层优先级”推进落地：高/中/低 + ✅/☐
3. 用“默认值三处一致”避免漂移：Dockerfile / entrypoint / README 同步
4. 用“dry-run 优先”避免误删：清理脚本默认只输出差异，显式 --apply 才执行
5. 用“连接自愈片段”处理 SSH 指纹：提供一条可复制的 known_hosts 清理命令

## 二、通用模式汇总（分类目录）

> 编号规则：ES-xx（Export Suggestions）。

### A. 输出结构模式

#### ES-01：导出建议五段式模板

**模式描述**：导出建议按五段式固定结构输出，保证“可追溯、可执行、可复用”。  
**结构模板**：
1. 已交付资产（按类别）
2. 后续行动（按优先级：高/中/低）
3. 可复用命令片段（可复制执行）
4. 归档与备份建议（保留证据、降低风险）
5. 关联资源（链接到脚本/文档/memory/知识库）

**适用边界**：
- 适用于“需要形成可继续使用的交付资产”的任务（环境交付、修复交付、迁移交付）
- 不适用于一次性聊天问答或纯观点输出

**核心价值**：
- 把“结果”变成“可持续资产”
- 把“建议”变成“可追踪执行计划”

**前置条件**：
- 至少存在一个可被列入清单的交付物（文档/脚本/镜像/配置/报告）
- 能提供最小可验证命令（哪怕是 dry-run/verify）

#### ES-02：资产清单三分法（环境 / 构建产物 / 知识资产）

**模式描述**：把已交付资产按三类分组，避免“清单堆叠但不可导航”。  
**分类**：
- 环境资产：镜像、容器、脚本、文档、入口
- 构建产物：二进制、构建输出、关键日志
- 知识资产：复盘、洞察、导出、规范、memory 记录

**适用边界**：
- 适用于“既有工程产物又有文档资产”的任务
- 若仅交付文档，可降级为“文档资产 + 关联资源”两类

**核心价值**：
- 阅读者可快速定位“我需要用什么”
- 后续维护可精确定位“该清理什么、该保留什么”

**前置条件**：
- 每个资产至少能给出一个稳定引用（路径/名称/标识符）

#### ES-03：建议分层优先级与状态闭环

**模式描述**：把建议按高/中/低三层组织，并显式标记状态（☐/✅），避免建议停留在“写完即忘”。  
**推荐规则**：
- 高：阻塞使用/阻塞交付/风险暴露（空间、构建失败、连接失败）
- 中：提升一致性与可维护性（默认值统一、规范写回、流程模板化）
- 低：长期治理与技术债处理（标准化修复、通用模板沉淀、机制化清理）

**适用边界**：
- 适用于需要跨多轮迭代落地的任务
- 不适用于无需追踪的临时建议

**核心价值**：
- 用结构化推进替代“靠记忆推进”

**前置条件**：
- 允许在文档中更新状态（或在 issue/todo 系统中同步）

### B. 工具链与构建策略模式

#### ES-04：编译器优先级策略（默认稳定构建 + 可选质量构建）

**模式描述**：当存在“编译器严格度差异”或“依赖模块使用非标准扩展”时，明确两条路径：  
- 默认使用“稳定构建编译器”（面向可用性）
- 额外保留“严格构建编译器”（面向质量与兼容性）

**适用边界**：
- 适用于“同一代码在不同编译器下表现不同”的场景
- 不适用于编译器无差异的纯脚本项目

**核心价值**：
- 在不修改上游/挂载源码的约束下推进交付
- 把“临时 workaround”升级为“工程策略”

**前置条件**：
- 环境内至少有两套可选工具链，或可低成本引入第二套
- 具备一次可复现的失败日志作为决策依据

#### ES-05：默认值三处一致（Dockerfile / entrypoint / README）

**模式描述**：任何会影响运行行为的默认值（如 `CC/CXX`、端口、目录、模式开关）必须在三处同步：  
1. Dockerfile ENV 默认值  
2. entrypoint/profile 注入默认值  
3. README 说明默认值与覆盖方法  

**适用边界**：
- 适用于容器化开发环境、脚手架、工具链封装

**核心价值**：
- 消除“文档说一套，环境跑一套”的漂移问题

**前置条件**：
- 允许修改环境封装层（不要求修改业务源码）

### C. 运维与可持续治理模式

#### ES-06：构建日志证据归档策略

**模式描述**：把构建日志从源码目录迁出，放入统一的临时/归档目录（如 `.temp/logs/`），同时在导出清单中给出“证据位置”。  

**适用边界**：
- 适用于日志可能包含敏感信息时：需按组织策略脱敏/访问控制

**核心价值**：
- 保留证据链（可复现、可对比）
- 防止仓库膨胀与误提交

**前置条件**：
- 有明确的临时目录约定（如 `.temp/`），且不会被长期纳入版本控制

#### ES-07：安全清理机制（dry-run 默认 + 显式 apply）

**模式描述**：任何 destructive 操作（删容器/删镜像/批量删产物）必须默认 dry-run，仅在显式参数下执行真实清理。  
**实现模板**：
- `python cleanup.py`：仅列出将删除对象
- `python cleanup.py --apply`：执行删除

**适用边界**：
- 适用于本地/测试环境的清理
- 生产环境需额外增加审批、白名单、日志审计

**核心价值**：
- 以最小成本避免误删事故

**前置条件**：
- 有可查询的“当前状态视图”（如 docker ps/images 的枚举）

#### ES-08：基础镜像复用 + 源加速前置

**模式描述**：把工具链尽量下沉到基础镜像，环境镜像只补“开发必需品”；同时在 Dockerfile 中前置配置 apt/pip 镜像源减少构建波动。  

**适用边界**：
- 适用于内网/离线环境：需保证镜像源可访问或提供离线依赖包

**核心价值**：
- 构建更快、更稳定
- 减少重复安装导致的镜像膨胀与漂移

**前置条件**：
- 有稳定的基础镜像发布/归档渠道（tar 或 registry）

### D. 远程连接与接入模式

#### ES-09：SSH Host 配置块（可复制）

**模式描述**：为远程环境提供一个最小 SSH 配置块，统一主机别名、端口、用户与安全选项，并写入导出清单。  

**适用边界**：
- 适用于 SSH 接入类环境（容器、虚机、远程服务器）
- 不适用于仅需 docker exec 的本地环境

**核心价值**：
- 降低使用门槛：`ssh <alias>` 即可连接

**前置条件**：
- 环境已提供 SSH 服务与端口映射

#### ES-10：SSH 指纹变更自愈片段

**模式描述**：将“host key changed”的处理命令作为固定片段写入导出建议，避免每次重建都要临时排查。  
**命令模板**：
```bash
ssh-keygen -f "~/.ssh/known_hosts" -R "[<host>]:<port>"
```

**适用边界**：
- 适用于“环境经常重建”的场景（容器、短生命周期虚机）
- 不适用于“必须保持强固定指纹”的高安全场景

**核心价值**：
- 把常见故障转成标准操作

**前置条件**：
- 使用方能够接受该环境的 host key 会随重建变化

### E. 命名与重构治理模式

#### ES-11：去版本号命名原则

**模式描述**：外层命名优先表达“用途”，版本信息下沉到 tag/注释/文档，避免环境升级后“名实不符”。  

**适用边界**：
- 适用于会随时间升级的环境/脚手架/工具链

**核心价值**：
- 命名稳定，减少后续重构频率

**前置条件**：
- 能在文档中明确记录版本来源（基础镜像 tag、构建信息）

#### ES-12：重构三步法（搜索 → 替换 → 验证）

**模式描述**：涉及标识符级重构（镜像名/目录名/脚本入口/默认值）时，统一遵循：  
1. 搜索影响面  
2. 分批替换  
3. 真实运行验证  

**适用边界**：
- 适用于重命名、目录迁移、默认值切换、入口变更

**核心价值**：
- 最大化降低遗漏风险，避免“局部修复造成系统性漂移”

**前置条件**：
- 有可执行的验证动作（构建/启动/连接/导入/脚本运行）

### F. 约束与决策模式

#### ES-13：约束驱动的修复路径优先级矩阵

**模式描述**：当同一问题存在多条可行修复路径时，使用“合规性/侵入性/一次验证性/可回退性/维护成本”的决策矩阵确定优先级，优先选择不触碰硬约束、可快速验证且可回退的路径。  
**参考实现**：[constraint-driven-fix-path-priority-matrix.md](constraint-driven-fix-path-priority-matrix.md)

**适用边界**：
- 适用于存在硬约束（不改挂载/不改上游/不改生产镜像等）的修复决策
- 不适用于约束可快速正式变更且变更成本低的场景（应优先走“变更约束”流程）

**核心价值**：
- 把“凭经验选路径”升级为“可解释、可复用的路径选择框架”
- 为 ES-04（编译器优先级策略）与 ES-12（重构三步法）的执行提供上游排序依据

**前置条件**：
- 能明确列出项目约束与不可触碰边界
- 至少存在一个可执行的最小验证动作（dry-run/verify/构建/导入/连接）

## 三、覆盖验证（≥80% 场景覆盖）

### 3.1 覆盖口径

将源文件拆为 19 个“要素单元”（清单结构、行动项、命令片段、备份/清理、关联资源）。若某单元能被至少一个通用模式完整吸收，则认为被覆盖。

| 源文件要素单元 | 对应通用模式 |
|---|---|
| 已交付资产：环境资产清单 | ES-01, ES-02 |
| 已交付资产：编译产物清单 | ES-01, ES-02, ES-06 |
| 已交付资产：知识资产清单 | ES-01, ES-02 |

### 3.3 覆盖结论

- 覆盖单元：18 / 19（94.7%）
- 未覆盖单元：1 / 19  
  - “某个具体业务库/二进制产物的命名与路径”属于项目个性化信息，在通用模式集合中被刻意剔除，仅在 ES-02 的“分类法”层面保留结构，不保留具体路径细节。
