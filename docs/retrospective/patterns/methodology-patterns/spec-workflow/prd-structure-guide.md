# 通用PRD正文结构指南

## 概述

本指南定义通用PRD Spec正文结构标准，基于高质量Spec的第一性原理解构分析提炼，覆盖10个核心章节，确保需求文档完整、可验证、可追溯。

---

## 章节全景

```
YAML Frontmatter → 1.Overview → 2.Goals → 3.Non-Goals → 4.Background
→ 5.FR → 6.NFR → 7.Constraints → 8.Assumptions → 9.AC → 10.Open Questions
→ 11.版本历史（必填收尾）
[可选] Requirements Traceability Matrix
[可选] Risks & Mitigation
[可选] Timeline & Milestones
```

---

## 核心章节详解

### 1. Overview

**核心目的**：3秒理解项目，降低认知负荷。

**必填要素**：Summary（做什么）、Purpose（为什么做）、Target Users（为谁做）。

**写作要求**：Summary≤50字；Purpose说明价值而非实现；Target Users具体不泛化。

**正例**：
```markdown
## Overview
**Summary**: 为知识库建立统一引用验证机制，确保外部来源可追溯可验证。
**Purpose**: 解决引用格式混乱、死链无检测、可信度无标准问题，提升知识档案权威性。
**Target Users**: 知识库维护者、内容贡献者、需追溯来源的读者。
```

**反例**：`本项目优化系统，做一些改进提升用户体验。`（模糊无物）

**检查要点**：[ ] 3秒能说清做什么 [ ] 三要素齐全 [ ] 无技术细节 [ ] 用户明确

---

### 2. Goals

**核心目的**：将"成功"转化为可衡量目标，正向定义范围边界。

**必填要素**：编号列表（G1,G2...），每个Goal可衡量，每个Goal映射至少一个AC。

**写作要求**：动词+宾语+可衡量结果；避免"优化/提升"等模糊词；3-8个Goals；包含质量目标。

**正例**：
```markdown
## Goals
- G1: 建立100%外部引用自动验证机制，死链检测率100%
- G2: 实现A/B/C/D四级可信度评级，评级一致性≥90%
- G3: 提供引用格式统一工具，支持3种风格自动转换
- G4: 验证逻辑可脚本化，单次检查<30秒
- G5: 建立对抗性审查，关键论断≥2个独立来源交叉验证
```

**反例**：`- 优化用户体验 - 提升性能 - 做好引用管理`（不可衡量）

**检查要点**：[ ] 有可衡量指标 [ ] 编号清晰 [ ] 无模糊词 [ ] 3-8个 [ ] 含质量目标

---

### 3. Non-Goals

**核心目的**：明确"不做什么"，防止范围蔓延——这是spec最有价值的章节之一。

**必填要素**：编号列表（NG1,NG2...），每个说明"不做什么"+"为什么"，推迟事项说明未来入口。

**写作要求**：采用"明确推迟而非删除"策略；解释"为什么不做"更重要；敢于列显而易见的边界。

**正例**：
```markdown
## Non-Goals
- NG1: 不实现自动事实核查（仅验证来源存在性）——需领域知识，v2.0独立评估
- NG2: 不支持书籍/视频等非网页引用——v1.0聚焦网页/论文，待需求验证后纳入
- NG3: 中国古代哲学文献对比暂不纳入——需同等深度系统性研究，浅尝辄止比不做更差，列为v2.0候选
```

**反例**：`- 不做没用的功能 - 暂时不考虑其他需求`（无理由无边界）

**检查要点**：[ ] 每个说明"不做什么"+"为什么" [ ] 推迟项说明条件 [ ] 无"以后再说" [ ] 覆盖争议点

---

### 4. Background & Context

**核心目的**：回答"为什么是现在"，提供决策的时空坐标。

**必填要素**：历史脉络、问题现状、关联项目。子章节为推荐结构，可根据项目特点灵活调整。

**写作要求**：提供前因后果让新人快速上手；引用具体证据（文档/commit）；说明时机理由。

**正例**：
```markdown
## Background & Context
### 历史脉络
- 2026-Q2: vibe-coding-prompts发现30%外链1个月后失效
- 2026-07-02: first-principles项目验证了对抗性审查价值
- 2026-07-08: 复盘发现跨项目引用规范不统一
### 问题现状
1. 同一来源3种标注方式 2. 已发现27个死链无检测 3. 可信度无评判标准
### 关联项目
- SpecWeave: 作为其质量控制组件
```

**反例**：`因为领导要求所以做。`（无背景无分析）

**检查要点**：[ ] 说明"为什么是现在" [ ] 有历史脉络 [ ] 描述清问题 [ ] 提及关联项目

---

### 5. Functional Requirements (FR)

**核心目的**：将Goals分解为可执行功能单元，描述"系统应该做什么"（而非"怎么做"）。

**必填要素**：编号列表（FR-1,FR-2...），每个FR是独立功能，主需求+子项，标记追溯Goal。

**写作要求**：用"系统应该能够..."句式；只描述"做什么"不描述"怎么做"；颗粒度1-3天工作量；标记`[→Gx]`追溯。

**正例**：
```markdown
## Functional Requirements
- FR-1 [→G1]: 系统应该能够自动检测Markdown文件中所有外部链接
  - FR-1.1: 支持HTTP/HTTPS  FR-1.2: 支持标准格式和裸链接  FR-1.3: 支持递归扫描
- FR-2 [→G1]: 系统应该能够验证链接可访问性
- FR-3 [→G2]: 系统应该能够根据域名自动给来源可信度评级
- FR-4 [→G3]: 系统应该能够统一格式化引用为指定风格
- FR-5 [→G5]: 系统应该支持交叉验证，标记需多来源确认的论断
```

**反例**：`用Python写爬虫，requests库，存SQLite`（描述了"怎么做"）

**检查要点**：[ ] 用"系统应该能够..." [ ] 只描述做什么 [ ] 编号清晰 [ ] 颗粒度合适 [ ] 标记追溯 [ ] 无技术实现

---

### 6. Non-Functional Requirements (NFR)

**核心目的**：定义"做得怎么样"——质量属性。没有NFR会产出"能用但不好用"的东西。

**必填要素**：编号列表（NFR-1,NFR-2...全局连续编号），覆盖性能/可靠性/可维护性等，每个NFR可量化，含工程化要求。按需增加分类（安全性/可用性/兼容性/可扩展性/可测试性等）。

**写作要求**：每个NFR必须有量化指标；分类组织；包含文件规范/代码规范等工程化要求。

**正例**：
```markdown
## Non-Functional Requirements
### 性能
- NFR-1 [→G4]: 单文件(<1000行)检查<5秒  - NFR-2 [→G4]: 100文件全量<30秒
### 可靠性
- NFR-4: 检测准确率>99%（误报<1%）  - NFR-5: 网络错误重试3次不崩溃
### 可维护性
- NFR-7: pep8规范，类型注解100%  - NFR-9: kebab-case命名，单文件<500行
### 质量透明
- NFR-10 [→G5]: 评级有明确依据，记录日志  - NFR-12: 主动识别认知偏差
```

**反例**：`- 系统要快 - 代码要好 - 要稳定`（不可衡量）

**检查要点**：[ ] 有量化指标 [ ] 编号清晰 [ ] 覆盖关键维度 [ ] 含工程规范 [ ] 无模糊表述

---

### 7. Constraints

**核心目的**：承认现实边界——技术/预算/依赖限制。约束是创新的催化剂。

**必填要素**：Technical Constraints、Business Constraints、Dependencies，每个说明"是什么"+"为什么"。

**写作要求**：现实不理想化；区分硬约束/软约束；依赖标注状态（已就绪/待完成）；不遗漏法律合规。

**正例**：
```markdown
## Constraints
### Technical
- TC1: 不能访问付费学术数据库——无机构订阅，预算限制
- TC3: 仅支持Python 3.10+——利用类型注解新特性
### Business
- BC1: 不购买付费内容/API——零预算
- BC3: 不存储受版权保护全文——仅存元数据和链接
### Dependencies
- D1 [已就绪]: first-principles项目提供对抗性审查经验
- D3 [待完成]: 可信度评级域名名单需人工整理初始版本
```

**反例**：`没有什么约束，我们什么都能做。`（不承认现实）

**检查要点**：[ ] 分类清晰 [ ] 说明"是什么"+"为什么" [ ] 依赖标注状态 [ ] 含法律/预算约束

---

### 8. Assumptions

**核心目的**：把隐含假设显式化——这是风险管理核心。不写假设≠没有假设，只是把风险藏起来。

**必填要素**：编号列表（A1,A2...），每个说明"假设什么是真的"+"不成立的影响"。

**写作要求**：敢于列"显而易见"的假设；定期回顾验证；3-8个假设。

**正例**：
```markdown
## Assumptions
- A1: 公开网络可获取足够高质量资料——若不成立，调整评级标准或申请数据库权限
- A2: 大多数网站支持HEAD/正常响应GET——若不成立，为特定站实现自定义检查
- A3: 域名与可信度相关性足够强可作初评依据——若误判>10%，引入人工审核
- A5: 执行环境有稳定网络——若离线，提供缓存/离线模式
```

**反例**：`- 用户会喜欢 - 一切顺利`（美好愿望而非可验证假设）

**检查要点**：[ ] 是可验证命题 [ ] 编号清晰 [ ] 说明不成立影响 [ ] 列显而易见假设 [ ] 3-8个 [ ] 无美好愿望

---

### 9. Acceptance Criteria (AC)

**核心目的**：提供可证伪的完成定义——Given/When/Then让验收客观化。没有AC的Goal只是愿望。

**必填要素**：编号列表（AC-1,AC-2...），Given/When/Then分小节格式，标注Programmatic/Human-Judgment，标记完整追溯链。

**写作要求**：Given(前置)→When(操作)→Then(结果)分小节书写更清晰；区分自动化验证和人工判断；Programmatic指明检查脚本；每个FR对应AC。

**正例**：
```markdown
## Acceptance Criteria
- AC-1 [→FR-1→G1] [Programmatic]:
  - Given: 含10个链接的测试文件
  - When: 运行检测脚本
  - Then: 10个链接全识别无遗漏
  - Verification: `python scripts/check-links.py --test` 通过率100%
- AC-3 [→FR-3→G2] [Human-Judgment]:
  - Given: 10个不同域名来源
  - When: 运行自动评级
  - Then: 与专家判断一致性≥90%
  - Verification: 2名维护者独立审核记录准确率
```

**反例**：`- 功能正常 - 用户满意 - 没有bug`（不可证伪）

**检查要点**：[ ] 有Given/When/Then分节 [ ] 标注验证方式 [ ] 编号清晰 [ ] Programmatic指明脚本 [ ] Human-Judgment有标准 [ ] 每个FR有对应AC [ ] 可明确通过/不通过

---

### 10. Open Questions

**核心目的**：记录决策过程而非只记结论，让spec从"静态合同"变成"活的决策日志"。

**必填要素**：统一字段结构（问题/分析/决策&理由/后续计划）；已解答[x]未解答[ ]；未解答标注阻塞点。

**写作要求**：记录完整思考过程，包括被否决选项；已解答和未解答使用统一字段结构。

**正例**：
```markdown
## Open Questions
- [x] Q1: 是否支持书籍/视频等非网页引用？
  - 分析: 验证机制不同复杂度高
  - 决策&理由: v1.0仅支持网页/论文——当前90%引用是此类，先解决主要问题
  - 后续计划: 非网页引用>20%时重评估
- [ ] Q3: 是否提供IDE插件实时检查？
  - 分析: 需调研使用场景——CI检查还是编写时检查？
  - 阻塞点: 使用场景不明确
  - 后续计划: v1.0先做CLI，收集1个月数据再决策
```

**反例**：`还有些问题没想清楚，以后再说。`（无具体内容）

**检查要点**：[ ] 已解答含分析/决策/理由 [ ] 未解答说明阻塞点 [ ] 记录被否决选项 [ ] 推迟项说明重评估条件

---

### 11. 版本历史（必填收尾）

**核心目的**：记录Spec的演化轨迹，让变更可追溯，避免"改了什么没人知道"。

**必填要素**：版本号、日期、变更说明表格。

**写作要求**：初始版本0.1，正式立项1.0，重大变更升级主版本号，增量更新升级次版本号；每次变更简要说明改了什么、为什么改。

**正例**：
```markdown
## 版本历史
| 版本 | 日期 | 变更说明 |
|------|------|----------|
| 0.1 | 2026-07-09 | 初始Spec框架 |
| 1.0 | 2026-07-09 | Spec评审通过，正式立项 |
| 1.1 | 2026-07-09 | Dogfooding验证后优化AC格式和追溯标记 |
```

**检查要点**：[ ] 有版本号/日期/变更说明三列 [ ] 初始版本为0.1 [ ] 正式版本为1.0 [ ] 重大变更有记录

## 需求追溯关系

### 追溯逻辑链
```
Goals (G) → 分解 → FR [→G] → 附着质量属性 → NFR → 验证 → AC [→FR→G]
```

### 追溯标记规范
- FR: `FR-1 [→G1]`
- NFR: `NFR-1 [→G4,→FR-2]`（可选）
- AC: `AC-1 [→FR-1→G1]`

### 完整性原则
1. 每个Goal至少一个FR
2. 每个FR至少一个AC
3. 每个AC必须可证伪
4. 追溯关系双向可查

## 可选增强章节

### Requirements Traceability Matrix（推荐）

**适用**：FR/AC>10个、多人协作、需严格需求管理。

**模板**：
```markdown
## Requirements Traceability Matrix
| Goal | Functional Requirements | Acceptance Criteria |
|------|-------------------------|---------------------|
| G1 | FR-1, FR-2 | AC-1, AC-2 |
| G2 | FR-3 | AC-3 |
| G5 | FR-5, NFR-10-12 | AC-7, AC-8 |
```

**检查**：[ ] 所有Goal出现 [ ] 所有FR映射Goal [ ] 所有AC映射FR [ ] 与正文标记一致

---

### Risks & Mitigation（推荐）

**适用**：中大型/高风险/创新项目。

**模板**：
```markdown
## Risks & Mitigation
| 风险来源 | 风险描述 | 可能性 | 影响 | 缓解策略 | 触发条件 |
|----------|----------|--------|------|----------|----------|
| A1 | 公开网高质量资料不足 | 中 | 高 | 1.预筛5个开放平台 2.调整范围 | 前2周一 级来源<50% |
| TC1 | 付费库限制部分来源无法验证 | 高 | 中 | 1.优先arXiv等开放平台 2.无法验证标"待验证" | 单来源连续3次失败 |
```

**检查**：[ ] 关联Assumption/Constraint [ ] 缓解策略具体 [ ] 有明确触发条件

---

### Timeline & Milestones（多人/长周期项目推荐）

**适用**：多人协作、跨周/跨月项目、需要协调里程碑的项目。单执行者/短周期项目（<1周）可省略。

**模板**：
```markdown
## Timeline & Milestones
| 里程碑 | 预计日期 | 交付物 | 验收标准 |
|--------|----------|--------|----------|
| M1: Spec完成 | YYYY-MM-DD | spec.md v1.0 | 所有AC定义完毕，评审通过 |
| M2: 核心功能 | YYYY-MM-DD | 核心模块 | AC-1至AC-5通过 |
| M3: 项目完成 | YYYY-MM-DD | 全部交付物 | 所有AC验证通过 |
```

**检查**：[ ] 里程碑可验证 [ ] 交付物明确 [ ] 与AC关联

---

## 写作顺序建议

1. YAML Frontmatter → 2. Overview → 3. Goals + Non-Goals（最关键）→ 4. Background
→ 5. FR + NFR → 6. Constraints + Assumptions → 7. AC（最花时间最有价值）
→ 8. Open Questions → 9. 版本历史（必填收尾）→ 10. Traceability Matrix（可选）
→ 11. Risks & Mitigation（可选）→ 12. Timeline & Milestones（可选）

---

## Spec质量检查清单

### 完整性
- [ ] 11个核心章节齐全（10个正文+版本历史）
- [ ] YAML必填字段填写
- [ ] 每个FR有对应AC，每个Goal有对应FR
- [ ] Constraints/Assumptions无遗漏明显边界

### 可验证性
- [ ] Goals无模糊词，可衡量
- [ ] NFR有量化指标
- [ ] AC有Given/When/Then，可证伪
- [ ] Programmatic AC指明验证方法

### 边界清晰
- [ ] Non-Goals说明"不做什么"+"为什么"
- [ ] 推迟项说明未来条件
- [ ] 无"以后再说"模糊表述

### 追溯性
- [ ] FR标记追溯Goal
- [ ] AC标记完整追溯链
- [ ] 无孤立需求项

## 常见错误速查

| 错误 | 表现 | 修正方法 |
|------|------|----------|
| 范围模糊 | Goals用"优化/提升" | 问"优化到什么程度？怎么衡量？" |
| 越俎代庖 | FR描述技术实现 | 改写为"系统应该能够..." |
| 乌托邦 | Constraints写"无" | 诚实面对现实限制 |
| 愿望清单 | AC写"用户满意/没bug" | 改写Given/When/Then确保可证伪 |
| 范围蔓延 | Non-Goals缺失 | 强迫列至少3条"不做什么" |
| 空架子 | Background一句话 | 补充前因后果历史脉络 |
| 无追溯 | FR/AC无标记 | 加[→Gx]追溯标记 |
| 决策黑箱 | Open Questions只有问题 | 补充分析/决策/理由 |
