# 细粒度最小权限模式（Fine-grained Least Privilege）

## 一、核心定义

细粒度最小权限（Fine-grained Least Privilege）是对经典最小权限原则（Principle of Least Privilege, PoLP）在代理操作系统（远程控制、AI Agent、RPA）中的操作化落地：**将代理权限拆解为独立可开关的细粒度功能项，按授权生命周期（默认最小→按需申请→用完收回）进行管理，而非"连接上就能做任何事"的粗放模式**。

**核心公式**：

```
安全授权 = 权限粒度拆解 × 授权生命周期管理
```

**与用户主权默认的区别**：
- **用户主权默认**（user-sovereignty-default）回答"谁有最终控制权"——被代理方始终拥有否决权
- **细粒度最小权限**回答"权限如何拆解和授予"——代理方获得的是完成任务所需的最小粒度权限集

两者互补：用户主权是底层控制权原则，细粒度最小权限是权限结构设计方法。

## 二、两个核心层面

### 层面一：权限粒度拆解（Fine-grained Decomposition）

将代理权限从"全有或全无"的二元开关，拆解为按操作风险分级的多级权限项：

| 权限等级 | 操作类型 | 默认策略 | 示例（AI Agent） |
|---------|---------|---------|----------------|
| L0 只读 | 信息查询、文件读取、搜索 | 默认允许 | 搜索代码、读取文件内容、列目录 |
| L1 常规写入 | 内容创建与修改（可追溯可回滚） | 允许但记录 | 创建文件、编辑代码、写入配置 |
| L2 敏感操作 | 有副作用但影响可控 | 需人类确认 | 删除文件、执行shell命令、发送消息、访问网络 |
| L3 高危操作 | 不可逆/影响范围大/涉及资金安全 | 必须明确授权+二次确认 | rm -rf、转账、删除数据库、推送生产环境 |

**拆解原则**：
1. 权限项之间应**正交独立**——禁用文件传输不影响屏幕查看，禁用鼠标不影响剪贴板
2. 颗粒度应**适中**——过粗则无法精确控制（如"文件操作"包含读和写），过细则管理成本过高
3. 按**操作风险**而非**功能类别**分级——同样是文件操作，读取（L0）和删除（L2）风险不同

### 层面二：授权生命周期管理（Authorization Lifecycle）

权限不是一次性授予，而是贯穿任务生命周期的动态管理：

```
默认最小 → 按需申请 → 明确说明用途 → 人类授权 → 执行监控 → 任务完成 → 自动收回
```

1. **默认最小**：初始只授予完成当前任务所必需的最小权限集（通常L0只读+部分L1）
2. **按需申请**：需要额外权限时，Agent必须明确说明"要做什么+为什么需要+影响范围"
3. **明确授权**：人类对L2/L3操作逐项授权，而非一次性放开所有权限
4. **用完收回**：任务完成后，临时申请的权限自动收回，不跨任务/跨会话保留
5. **独立边界**：不同Agent、不同任务、不同会话有独立的权限边界，不跨任务共享

## 三、验证案例

### 案例1：向日葵远程控制（来源：2026-07-04安全产品Wiki学习）

向日葵在个人场景和企业场景分别实践了细粒度最小权限：

**个人场景（接受远程协助）**：
- 连接建立后，被控端可以**单独禁止**：鼠标操作、文件传输——不是全给或全不给
- 即使保持连接，被控端依然掌握细粒度的权限开关，可以随时收回某项权限
- 设计理念："让对方帮我解决问题，但不能乱动我的东西"——可用但不可乱为

**企业场景（精细化角色权限）**：
- 管理员根据员工岗位和工作需要，精确配置每个人的：
  - 可访问的设备范围
  - 可使用的功能（文件传输、剪贴板共享、远程CMD等可按需开关）
- 可全局禁用或按角色配置文件传输、剪贴板共享等数据外泄通道
- 设计理念：每个用户只拥有完成其工作所必需的最小权限集

两个场景的共同特点：**打破"连接即全权"的粗放模式，将权限拆解为独立开关，按需配置，随时可收回**。

## 四、适用场景

- **AI Agent工具授权**：Agent工具调用的权限分级与生命周期管理（核心适用场景）
- **远程访问/控制工具**：远控会话的细粒度权限开关
- **企业级权限管理**：RBAC角色权限的精细化配置
- **API访问控制**：OAuth Scope的细粒度权限设计
- **RPA自动化平台**：机器人操作权限的分级管控
- **SaaS多租户系统**：租户/用户/角色三级权限边界设计

## 五、与其他模式的关系

| 模式 | 关系 | 区别 |
|------|------|------|
| user-sovereignty-default | 互补 | 主权默认回答"谁有否决权"（被控方），细粒度最小权限回答"权限怎么给"（拆解+生命周期） |
| scenario-based-security-matrix | 协同 | 场景矩阵回答"什么场景配什么特性组合"，细粒度最小权限提供权限项的分级标准和生命周期管理，是场景矩阵的权限维度输入 |
| non-intrusive-security-ux | 协同 | 安全不打扰回答"何时打断用户确认"（风险评分→响应级别），细粒度最小权限提供权限分级（L0-L3）作为风险评分的操作类型输入 |
| full-process-defense-depth | 互补 | 纵深防御提供事中守护层（实时监控+人类确认），细粒度最小权限是事前预防层的核心——通过权限拆解降低攻击面 |

## 六、反模式

❌ **二元权限模型**：只有"有权限/没权限"两种状态，无法做精细化控制

❌ **默认全权模式**：代理一连接/启动就拥有全部权限，需要时再限制——违背最小权限原则

❌ **权限粒度过粗**：将多个风险等级不同的操作捆绑为一个权限项（如"文件权限"包含读、写、删、传输），无法精确控制

❌ **权限粒度过细**：拆分为数十个权限开关，导致管理成本过高，用户不知道该开什么关什么

❌ **一次授权永久有效**：权限一旦授予就不收回，跨任务/跨会话累积，权限边界不断膨胀

❌ **"为了方便"预授权**：为了减少确认弹窗，预先授予所有权限，失去了最小权限的保护意义

## 七、实践要点

1. **四级分级适用于大多数Agent场景**：L0只读/L1常规写入/L2敏感/L3高危，颗粒度适中
2. **权限正交设计**：每个权限项独立可开关，禁用A不影响B
3. **默认L0+部分L1起步**：Agent初始权限最小化，需要更多时主动申请
4. **权限申请必须附带说明**：Agent申请L2/L3权限时必须说明"做什么+为什么+影响范围"，帮助人类决策
5. **临时权限自动过期**：为单次任务申请的权限，任务完成后自动收回，设置会话级过期时间
6. **权限变更审计**：所有权限申请、授权、收回操作记录日志，支持事后追溯
7. **与场景矩阵结合**：不同场景（本地开发/生产运维/数据查询）下的默认权限集可以不同，通过scenario-based-security-matrix配置

## 八、AI Agent工具权限速查表

基于四级分级模型，常见Agent工具的建议权限等级：

| 工具/操作 | 建议等级 | 说明 |
|----------|---------|------|
| 文件搜索/读取/列目录 | L0 | 只读操作，默认允许 |
| 代码编辑/文件创建 | L1 | 可追溯可回滚，允许但记录 |
| 运行测试/构建命令 | L1 | 常规开发操作 |
| 文件删除 | L2 | 有副作用需确认 |
| 执行任意shell命令 | L2 | 敏感操作需确认 |
| 网络请求（外部API） | L2 | 数据外发风险 |
| 发送消息/邮件 | L2 | 外部影响需确认 |
| git push/生产部署 | L3 | 高危不可逆，必须明确授权 |
| 数据库删除/修改 | L3 | 数据安全风险 |
| 转账/支付操作 | L3 | 资金安全最高风险 |
| rm -rf/格式化 | L3 | 毁灭性操作，二次确认 |
