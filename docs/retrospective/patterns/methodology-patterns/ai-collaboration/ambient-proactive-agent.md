# 主动介入 Agent 模式（Ambient Proactive Agent）

## 模式类型

方法论模式（AI协作/主动介入）

## 成熟度

L2 验证级（2+ 案例验证：Claude Tag Ambient Mode 主动提醒 + WPS Comate 定时任务主动推送新闻，两个独立产品验证了"AI 主动介入"的通用性）

## 适用场景

- 长期项目的 AI 协作（持续监测、定期汇总、紧急标记）
- 多角色协作的异常检测与告警
- 阶段守卫与流程合规监控
- 任何需要 AI 主动暴露问题的场景
- 异步任务执行与主动汇报
- 团队决策事项的主动跟进

## 问题背景

传统 AI 助手是**被动响应式**的——用户提问，AI 回答。这导致四个核心问题：

1. **问题被忽视**：重要讨论被新消息淹没，长时间未被发现，错失处理时机
2. **决策被遗忘**：需要决策的事项无人跟进，决策悬而未决，项目停滞
3. **异常被掩盖**：流程异常、规范违反隐藏在日志深处，无人主动检查
4. **同步阻塞**：用户布置任务后必须等待 AI 完成才能继续，无法离开，效率低下

典型症状：
- 重要讨论在 Slack 频道中被新消息淹没，3 天后才被发现
- 需要决策的事项无人跟进，2 周后项目停滞
- 阶段守卫日志中有异常但无人检查，直到问题爆发
- 用户布置任务后必须盯着 AI 执行，无法离开做其他事

## 核心规则

### 规则 1：主动优于被动

不等待用户提问，AI 主动暴露异常与遗漏。主动提醒被忽视的重要讨论、跟进长时间未解决的问题、标记需要决策的事项。

> **为什么？** 被动响应意味着问题只有在用户主动检查时才会被发现。主动介入能在问题扩大前预警，降低修复成本。

### 规则 2：异步推进

任务执行不阻塞用户。用户布置任务后可离开，AI 自行安排执行计划、持续推进项目，完成后主动 @ 用户汇报结果，类似一个真正的 Agent。

> **为什么？** 同步阻塞是效率的最大杀手。异步推进让用户可以并行处理多个任务，AI 完成后主动汇报，实现真正的协作者而非工具。

### 规则 3：异常优先

主动介入应聚焦真正重要的事项——重要讨论被忽视、问题长期未解决、流程异常、规范违反。不应主动介入琐碎事务，避免告警疲劳。

> **为什么？** 如果 AI 对所有事项都主动提醒，用户会被告警淹没，反而忽略真正重要的信号。异常优先是信噪比的保障。

### 规则 4：汇报可追溯

主动介入与异步执行的结果必须有日志记录，可回溯审计。每次主动介入应记录触发原因、介入内容、处理结果，便于事后复盘与责任追溯。

> **为什么？** 不可追溯的主动介入是黑箱操作，用户无法判断 AI 的介入是否合理。可追溯是建立信任的基础。

## 实施检查清单

设计主动介入 Agent 时自问：
- [ ] AI 是否在用户提问前就提醒了重要问题？
- [ ] 用户离开后任务是否仍在推进，而非停滞等待？
- [ ] 主动介入是否聚焦真正重要的事项，而非琐碎事务？
- [ ] 每次主动介入是否有日志记录，可回溯审计？
- [ ] 异步执行完成后是否主动通知用户，而非等待用户查询？
- [ ] 主动介入是否有权限边界，不会越权操作？

## 反例警示

| 错误做法 | 后果 |
|---------|------|
| AI 被动等待用户提问 | 问题被忽视，错失处理时机 |
| 用户必须盯着 AI 执行任务 | 同步阻塞，效率低下 |
| AI 对所有事项都主动提醒 | 告警疲劳，用户忽略真正重要的信号 |
| 主动介入无日志记录 | 黑箱操作，无法审计与复盘 |
| AI 主动越权操作 | 权限失控，引发安全问题 |
| 异步执行完成后不通知用户 | 用户需主动查询，仍属被动模式 |

## 正例

**Claude Tag 的 Ambient Mode 实践**：
- 开启后 Claude 主动提醒被忽视的重要讨论
- 跟进长时间未解决的问题，定期 @ 相关成员
- 标记需要决策的事项，推动团队做出决策
- 发现相关信息后主动通知团队
- 用户布置任务后可离开 Slack，Claude 自行安排执行计划
- 完成后主动 @ 用户汇报结果，类似真正的 Agent

**WPS Comate 的实践**（第二个独立案例验证）：
- 支持定时任务功能，Agent 可设定时间主动执行任务（如定时推送行业新闻）
- Agent 不等待用户提问，按预设计划主动产出并推送结果
- 验证了"异步推进"和"主动汇报"模式在不同产品形态中的通用性
- 体现了从"用户拉取信息"到"Agent 推送信息"的交互范式转变

**SpecWeave 的对应实践**：
- 阶段守卫（stage-guardrails）在阶段边界主动拦截违规跳转，而非等待用户检查
- SG-LOG 日志记录每次拦截的触发原因、拦截内容、处理结果
- 自我洞察（self-insight）模块主动分析执行数据，识别异常与优化机会
- 自我迭代（self-iteration）模块在用户布置后持续推进，而非同步阻塞

## 设计启示

主动介入 Agent 模式的核心价值在于**从被动工具到主动协作者的跃迁**。设计 AI 协作体系时，应突破"用户提问→AI 回答"的传统交互范式，探索"AI 主动监测→主动介入→主动汇报"的新型协作模式。关键挑战在于平衡主动性与信噪比——过度主动会引发告警疲劳，过度被动则失去协作者价值。

## 未来演进方向

SpecWeave 可在自我演进模块中实践此模式：
1. **阶段守卫升级**：从"被动拦截违规跳转"升级为"主动扫描潜在风险并预警"
2. **自我洞察升级**：从"用户触发分析"升级为"定期主动分析并推送异常报告"
3. **自我迭代升级**：从"用户布置后执行"升级为"自行安排计划并主动汇报进展"
4. **SG-LOG 升级**：从"记录拦截事件"升级为"主动聚合异常模式并生成趋势报告"

成熟度升级路径：L1（当前，概念映射）→ L2（SpecWeave 内独立实践验证）→ L3（被其他任务复用）→ L4（标准化产品形态）
