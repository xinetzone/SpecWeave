# 子代理原子任务描述模板：七要素精确委托法

## 模式概述

使用general_purpose_task委托子代理执行任务时，任务描述必须精确无歧义。对于文档创建类任务，使用**七要素模板**（精确绝对路径 + 完整frontmatter模板 + 结构化章节大纲 + 导航一致性强约束 + 硬约束清单 + Mermaid安全规则 + 主代理侧产出强制验证）；对于分析/对比/研究类任务，使用**四要素扩展模板**（明确目标与交付物 + 结构化分析框架 + 完整性强制要求 + 输出格式规范）；所有任务通用：**主代理侧产出存在性强制验证SOP**（要素7），子代理返回后必须用LS+Read验证文件真实存在。模糊的任务描述（如"帮我分析一下X"、"完成任务3-5"）会导致子代理返回结果几乎必然是摘要/截断/需要重写——因为子代理是无状态的，没有项目上下文，所有约束必须显式传递。

## 问题现象

子代理任务描述模糊的常见后果：

### 文档创建类任务问题
1. **文件路径错误**：子代理把文件写到根目录或错误子目录，需要移动文件并修正所有相对路径
2. **frontmatter字段缺失**：子代理省略必需字段（category/tags/status/author/summary），导致索引生成降级
3. **章节结构不一致**：每个子代理返回的章节结构不同，无法横向对比，需要统一重构
4. **导航文件名错误**：子代理猜测前后章文件名（如05-practice而非规划的05-comparison），导致链接断裂
5. **文件行数失控**：没有行数约束，子代理可能写出500+行单文件，违反原子化原则
6. **重复劳动**：主代理需要逐一修正上述问题，修正时间甚至超过自己写
7. **链式错误**：一个子代理的路径错误导致依赖它的其他子代理导航也出错
8. **Mermaid图表渲染失败**：子代理在Mermaid代码块中使用 `\n` 换行、引号嵌套、click事件、危险HTML标签等，导致图表在严格渲染环境（如飞书）中渲染失败，需事后逐一排查修复

### 分析/研究类任务特有问题
9. **输出截断/摘要化**：任务粒度过大（如合并多个分析任务），子代理返回摘要性总结而非完整详细分析
10. **提前宣布完成（Agentic laziness）**：子代理在输出量达到一定程度后，用"综上所述"、"以上是..."等语句提前收尾，即使任务未完成
11. **部分覆盖**：任务要求分析A、B、C三个维度，子代理只详细分析A，对B和C只简单带过
12. **重试无效**：第一次返回摘要后，即使重新委派强调完整性，第二次仍可能返回摘要——这是单窗口Agent的固有行为模式，不是靠简单重复就能解决

这些问题的共同根因是：主代理假设子代理"应该知道"项目规范、文件结构、命名约定、输出详细程度要求——但子代理是无状态的，没有项目上下文，所有约束必须显式传递。

## 解决方案

### 六要素任务描述模板

使用general_purpose_task时，query参数必须包含以下六个要素：

#### 要素1：精确绝对路径（必须）

使用Windows绝对路径（从盘符开始），不要让子代理猜路径：

```

#### 要素2：完整frontmatter模板（必须）

提供完整YAML frontmatter，所有字段预填好值，子代理只需写入文件：

```yaml

- tags：标签数组
- date：创建日期
- status：stable

#### 要素3：结构化章节大纲（必须）

提供到二级标题的大纲，明确每个章节要写什么内容、包含什么元素：

✅ 正确：列出每个二级标题，说明每个标题下要包含的具体内容和元素

#### 要素4：导航一致性强约束（必须）

**特别强调**导航链接中的文件名必须与规划一致，这是最易出错的点：

```

```

❌ 错误："加上前后章导航"（子代理会猜文件名）
✅ 正确：列出每个导航链接的准确文件名，特别是容易猜错的对比章/总结章名称

#### 要素5：硬约束清单（必须）

列出不可违反的硬性规则：

```

3. 代码示例必须使用正确的markdown代码块语法（```language）

5. 不要添加注释（<!-- -->形式的HTML注释也不要加）
6. 所有本地链接使用相对路径，不要用绝对路径
7. Mermaid代码块使用```mermaid标记（如需要）
```

❌ 错误："遵循项目规范"（子代理不知道项目规范）
✅ 正确：逐条列出不可违反的规则，特别是行数限制和禁止事项

#### 要素6：Mermaid安全规则（必须，含Mermaid图表时）

当文档包含 Mermaid 图表时，必须在任务描述中显式传递安全编码规则，否则子代理会使用 `\n` 换行、引号嵌套、click 事件等导致渲染失败的写法：

```
Mermaid安全编码规则（严格遵守）：
1. 代码块内禁止空行（空行会被部分渲染器解析为代码块结束）
2. 含中文/特殊字符/空格的节点文本必须用双引号包裹：A["中文节点"]
3. 节点换行使用 <br/>，禁止使用 \n
4. Subgraph 使用 subgraph EN_ID ["中文标题"] 格式，ID为纯英文
5. 边标签使用 -->|"标签"| 格式，含中文/特殊字符必须加引号
6. 禁止使用 click 事件、<script>、<img>、<iframe> 等危险HTML标签
7. 禁止使用 "end" 作为节点ID（与Mermaid保留字冲突）
8. 节点文本避免 Markdown 列表触发模式（"数字. "、"- "、"* "）
9. 引号内文本不要再嵌套英文双引号，改用中文引号或去除内层引号
```

**关键提醒**：Mermaid 图表中引号嵌套是最隐蔽的陷阱。当节点文本本身包含引号（如 `O("N+M")`），自动引号补全会导致 `["O("N+M")"]` 嵌套错误。解决方法是去除内层引号或改用中文引号。

❌ 错误："Mermaid按项目规范写"（子代理不知道Mermaid安全规则）
✅ 正确：逐条列出安全编码规则，特别是 `\n` 禁用和引号嵌套陷阱

### 分析/研究类任务四要素扩展模板

对于非文档创建的任务（如技术分析、对比研究、机制梳理、问题诊断），使用四要素扩展模板。这类任务没有明确的文件路径/frontmatter/导航约束，但**完成边界更模糊**，子代理更容易偷懒返回摘要，因此需要更明确的框架和完整性约束。

#### 要素1：明确目标与交付物（必须）

```
任务目标：对X进行系统性的Y分析
交付物要求：
- 结构化的分析结果，使用Markdown格式
- 包含[列出具体要包含的部分，如：对比矩阵、根因分析、最佳实践]
- 输出应该是完整可直接使用的内容，不是大纲或摘要
- 预计详细程度：[如：每个机制至少分析5个维度，每个维度3-5句话说明]
```

❌ 错误："分析一下Task 3-5的内容"
✅ 正确：明确说明分析目标、交付物格式、详细程度要求

#### 要素2：结构化分析框架（必须）

提供明确的分析维度和结构，消除"要分析什么"的歧义：

```
分析框架（严格遵循，每个维度都要详细展开）：
1. [维度A名称]：[说明这个维度要分析什么，如：从加载时机、token成本、可见性三个子维度分析]
2. [维度B名称]：[说明内容，如：对比6种编排模式的工作原理、适用场景、解决的问题]
3. [维度C名称]：[说明内容，如：总结5个常见误区，每个误区包含错误做法、为什么错、正确做法]
```

**关键**：不仅说"分析A、B、C"，还要说明每个维度内部要包含什么子项，就像文档创建任务中给出二级标题大纲一样。

❌ 错误："完成对比矩阵和Dynamic Workflows分析"
✅ 正确："对比矩阵需要覆盖8种机制×7个维度（加载时机、token成本、可见性、确定性、适用内容、场景、压缩后行为）；Dynamic Workflows需要分析三大问题根因+三个核心函数+6种编排模式"

#### 要素3：完整性强制要求（P0级，必须）

**必须**在query中附加以下内容（参考subagent-output-quality-checklist.md）：

```
⚠️ 输出完整性强制要求（极其重要）：
1. 必须返回完整、详细的内容，绝对禁止摘要性、概括性的总结
2. 不要在内容末尾写"以上是...的总结"、"综上所述..."等提前收尾语句
3. 上面列出的每个分析维度/每个部分都必须详细展开，不能只完成一部分就宣布完成
4. 如果包含表格，表格的每个单元格都要有实质性内容，不能只填"是/否"或留空
5. 我需要的是可直接使用的完整分析结果，不是提纲、摘要或概括
6. 一次只完成这一个分析任务，不要合并其他任务
```

#### 要素4：输出格式规范（必须）

明确输出格式，避免子代理"自由发挥"：

```
输出格式要求：
1. 使用标准Markdown格式
2. 标题层级：# 一级标题（分析主题），## 二级标题对应每个分析维度，### 三级标题用于子维度
3. 对比类内容使用Markdown表格
4. 列表项使用-或1.格式
5. 关键洞察使用**加粗**标记
```

#### 要素7：产出存在性强制验证（主代理侧P0级执行，非子代理责任）

⚠️ **本要素不是传递给子代理的约束，而是主代理在收到子代理返回后必须强制执行的验证流程。**

**问题背景**：子代理可能"幻觉报告"文件已创建成功（返回了文件路径和行数），但实际文件并未写入磁盘。如果主代理不验证直接进入下一步，会导致后续任务基于虚假产物执行，造成连锁返工。

**强制验证SOP（子代理返回后、进入下一步前必须执行）**：

```
子代理返回"已完成/已创建"
    │
    ▼
┌──────────────────────────────────────┐
│ Step 0（P0，10秒）：文件存在性验证     │
│ 1. 使用 LS 工具列出目标目录           │
│ 2. 确认报告的每个文件确实存在于磁盘    │
│ 3. 若文件不存在 → 要求子代理重新创建   │
│    （不要自行修复，让子代理重试）       │
│    验证失败不标记任务为completed       │
└──────────────────────────────────────┘
    │
    ├── 文件不存在 → 要求子代理重试（计入第一次失败）
    │
    ▼ 文件存在
┌──────────────────────────────────────┐
│ Step 1（P0，10秒）：内容非空验证       │
│ 1. 使用 Read 工具读取文件前10-20行    │
│ 2. 确认文件包含预期的YAML frontmatter │
│ 3. 确认内容不是空文件/占位文件        │
│ 4. 若内容异常 → 要求子代理修正        │
└──────────────────────────────────────┘
    │
    ├── 内容异常 → 要求子代理修正
    │
    ▼ 内容正常
┌──────────────────────────────────────┐
│ Step 2：进入内容质量验证（原有流程）   │
│ 文档类：frontmatter/行数/链接检查     │
│ 分析类：完整性/覆盖度/无摘要化检查     │
└──────────────────────────────────────┘
    │
    ▼
验证全部通过 → 标记任务completed，继续下一步
```

**铁律**：
1. **禁止信任子代理的文件创建报告**：无论子代理描述多么详细（路径、行数、内容概要），都必须用LS+Read验证
2. **验证失败不前进**：Step 0/1未通过时，绝不进入下一个任务，绝不标记Todo为completed
3. **LS优先于Read**：先LS确认文件存在，再Read确认内容。不要直接Read一个不存在的文件然后报错
4. **批量创建逐个验证**：子代理一次创建多个文件时，必须逐个LS验证每个文件，不能只验证第一个

**反模式警示**：
- ❌ 子代理说"已创建3个文件"→ 主代理直接标记任务完成（本次方舟复盘即因此问题导致复盘目录文件实际未写入）
- ❌ 用子代理的返回信息更新TODO状态，而非实际磁盘状态
- ❌ 只Read第一个文件就认为全部OK

#### 任务粒度铁律（所有任务通用）

无论文档创建还是分析研究，都必须遵守：

1. **原子化原则**：一次`general_purpose_task`调用只委派一个原子任务（对应tasks.md中的单个Task）
2. **禁止合并**：绝对禁止将多个不相关的子任务合并委派给同一个子代理（如"完成Task 3、4、5"）
3. **并行替代**：如果有多个独立任务，使用多个`general_purpose_task`并行调用
4. **事不过二**：子代理连续两次返回不合格结果后，必须切换兜底策略（主会话整合或拆分更小任务），不要重试第三次

### 内容获取类任务扩展模板

当委派子代理执行内容获取类任务（如网页抓取、API 调用、数据采集）时，在标准模板基础上增加以下内容。内容获取类任务的核心风险是子代理重复尝试主智能体已验证失败的方法，浪费时间和上下文窗口。

#### 已尝试方法清单

主智能体在委派前已尝试的方法（避免子智能体重复尝试）：

| 策略名称 | 命令 | 结果 | 失败原因 |
|---|---|---|---|
| defuddle 直连 | `defuddle parse <url> --md` | 失败 | 403 Forbidden |
| curl + UA | `curl -s -A "..." <url>` | 失败 | JS challenge |
| agent-browser 默认 | `agent-browser open <url>` | 失败 | 40362 错误码 |

**关键**：每一行都必须包含具体命令（子代理可复现）、结果（成功/失败）、失败原因（帮助子代理判断是否值得换策略重试）。失败原因标注"40362 错误码"比标注"访问失败"更有诊断价值。

#### 已知约束

- 沙箱环境限制：[列出不可达的服务，如 archive.org]
- 可用工具：[列出可用的工具，如 agent-browser, curl, defuddle]
- 认证状态：[是否已登录目标站点]

**关键**：沙箱限制必须显式列出，否则子代理会尝试不可达的服务并超时等待。认证状态必须明确（"未登录知乎"比"可能需要登录"更可执行）。

#### 成功标准

- 最低标准：获取至少 N 条回答正文（如 N=10）
- 理想标准：获取全部回答 + 关键评论
- 元数据要求：保留作者标识、赞同数、发布时间

**关键**：最低标准是"可接受的最小产出"，理想标准是"完整产出"。子代理达到最低标准即可返回，不必为追求理想标准而耗尽时间盒。元数据要求必须显式列出，否则子代理可能只抓正文而丢弃作者/赞同数等结构化信息。

#### 产出验证流程

1. 检查样本覆盖率（已获取数量 / 总数量）
2. 检查内容完整性（每个样本是否含正文 + 元数据）
3. 检查内容质量（是否含导航/广告等噪声）
4. 记录获取方式及可靠性等级（A 直连 / B 登录态 / C 缓存 / D API 拼接）

**可靠性等级说明**：
- **A 直连**：无需特殊处理即可获取，最可靠，可重复
- **B 登录态**：依赖登录 Cookie，中等可靠，Cookie 过期后失效
- **C 缓存**：来自第三方缓存（如 archive.org），低可靠，内容可能滞后
- **D API 拼接**：通过多个 API 端点拼接而成，最低可靠，结构可能不完整

## 适用场景

- ✅ 使用general_purpose_task委托子代理创建原子markdown文档
- ✅ 使用general_purpose_task委托子代理执行技术分析、对比研究、机制梳理
- ✅ 一个子代理负责一个文件/一个分析任务的场景
- ✅ 多文件/多分析任务并行创建（多个独立子代理同时工作）
- ✅ 文档有统一的frontmatter规范和导航约定（文档类）
- ✅ 分析任务有明确的分析框架和维度要求（分析类）
- ✅ 对文件大小/输出详细程度有明确约束
- ❌ 子代理负责多步操作（如"创建目录+写三个文件+运行验证"）——应拆成多个单任务
- ❌ 简单的单句查询（不需要要素模板，直接问即可）

## 实际案例

### 案例1：Interface/API/ABI/Protocol技术Wiki子代理委托

主进程创建00-overview.md、05-comparison.md、06-resources.md（核心和总览章节由主进程掌控），同时委托3个子代理并行创建01-interface.md、02-api.md、03-abi.md、04-protocol.md。

**子代理任务五要素**（当时尚无要素6）：
- 要素1 路径：如`d:\spaces\SpecWeave\docs\knowledge\learning\interface-api-abi-protocol-wiki\03-abi.md`
- 要素2 frontmatter：完整7字段模板直接给出
- 要素3 大纲：定义→核心特征→API vs ABI对比→代码案例→导航，二级标题明确
- 要素4 导航约束：明确指出前一章02-api.md、后一章04-protocol.md、对比章05-comparison.md
- 要素5 硬约束：<300行、必须有ctypes代码示例、禁止额外章节

**结果**：4个子代理返回的文件一次性通过frontmatter检查和链接检查，仅发现1处导航文件名偏差（05-practice vs 05-comparison），主进程在即时验证阶段10秒修正。无大规模重写。

**对比模糊委托的反事实估计**：如果只说"写ABI章节"，预计会出现：路径错误（需移动）、frontmatter缺3+字段（需补全）、缺少代码示例（需添加）、导航文件名错误（需修正），每个文件修正时间约3-5分钟，4个文件合计12-20分钟。

### 案例2：Agent通信协议Wiki子代理委托（要素6引入背景）

委托子代理创建包含大量Mermaid图表的技术教程（MCP/ACP/A2A/ANP协议解析，30+个Mermaid代码块）。初始仅使用五要素，未传递Mermaid安全规则。

**问题暴露**：子代理创建的章节中Mermaid图表出现三类问题：
1. 节点文本使用 `\n` 换行，在飞书渲染器中显示为字面文本（38个代码块需修复）
2. 自动引号补全导致引号嵌套错误（如 `["O("N+M")"]`），4处渲染失败
3. stateDiagram边标签含 `<br/>` 但未加双引号，8处渲染警告

**修复成本**：开发 `check-mermaid.py --fix` 脚本自动修复 `\n`→`<br/>`，手动修复引号嵌套和边标签引号，合计耗时约30分钟。

**要素6引入**：将Mermaid安全编码规则提炼为第六要素，在后续子代理委托中显式传递。同时在 `check-mermaid.py` 中补充安全检测（click事件、危险HTML标签、end节点ID、javascript: URL等），从源头消除安全违规。

**验证结果**：引入要素6后，新创建的Mermaid图表未再出现 `\n` 换行和引号嵌套问题，`check-mermaid.py` 安全检测全部通过。

### 案例3：Claude Code上下文注入分析——合并任务导致截断（反面案例→模式升级）

委托子代理执行Task 3-5（对比矩阵+Dynamic Workflows分析+误区总结，三个任务合并委派），未使用分析类四要素模板，也未强调输出完整性。

**问题暴露**：
1. 第一次返回：内容被截断，只有对比矩阵的部分内容，没有Dynamic Workflows和误区总结
2. 重新委派并强调"返回完整内容"后：返回了概括性总结而非完整详细分析，相当于高级摘要
3. 两次失败后切换兜底策略：主会话基于Task 1-2已返回的详尽内容直接整合完成，效率高于继续重试

**根因分析**：
- 违反任务粒度铁律：将3个不相关的分析任务合并委派给一个子代理
- 缺少完整性强制约束：query中没有明确要求"返回完整详细内容，禁止摘要"
- 缺少结构化分析框架：只说"完成Task 3-5"，没有给出每个任务内部的分析维度要求

**模式升级**：
- 基于本次案例，将六要素模板扩展为覆盖文档+分析两类任务
- 新增分析类任务四要素扩展模板
- 新增任务粒度铁律和"事不过二"兜底策略
- 子代理输出验证流程更新为包含完整性检查

## 反模式

### 反模式1：一句话任务描述

```
query: "帮我写02-api.md，关于API的章节"
```

子代理需要猜路径、猜frontmatter字段、猜章节结构、猜导航文件名——几乎必然猜错至少2-3项。

**正确做法**：使用六要素模板，每个要素都明确给出。

### 反模式2：给参考文件但不提取规范

```
query: "参考01-interface.md的格式写02-api.md"
```

子代理可能参考了格式但遗漏了关键字段，或复制了01的标签但没改成API相关的标签，或导航链接保留了01的文件名而没更新。

**正确做法**：不要让子代理"参考"其他文件推断规范——直接把规范（frontmatter、大纲、导航、硬约束）显式写在任务描述里。

### 反模式3：一个子代理写多个文件/多个任务

```
query: "帮我写01-interface.md、02-api.md、03-abi.md三个文件"
query: "完成Task 3、4、5"
```

任务复杂度指数上升，子代理上下文过长容易出错，而且更容易提前"宣布完成"返回摘要。出错后排查也更困难。

**正确做法**：一个general_purpose_task只委托一个原子任务（一个文件或一个分析维度），多个任务用多个并行子代理。

### 反模式4：省略导航约束（最常见错误）

主代理觉得"文件名不是很明显吗"，但子代理没有上下文，对比章可能叫05-comparison、05-compare、05-practice、05-summary、comparison.md等等——子代理猜对的概率很低。

**正确做法**：总是显式列出所有导航链接的准确文件名，特别标记容易猜错的名称。

### 反模式5：子代理返回后不验证

觉得"子代理应该做对了"，直接继续下一个任务。等所有文件写完才检查，发现4个文件有3个的frontmatter缺字段、2个的导航文件名错了，或者分析类任务返回的是摘要，需要逐个修正/重写。

**正确做法**：每个子代理返回后立即做30-60秒快速验证（内容纯净性+完整性+覆盖度），在上下文新鲜时修正或切换兜底策略。

### 反模式6：分析类任务不要求完整输出

```
query: "分析一下这8种机制的区别"
```

分析类任务的完成边界比文档类更模糊，不明确要求"返回完整详细内容，禁止摘要"的话，子代理几乎必然返回概括性总结。

**正确做法**：使用分析类四要素模板，必须包含完整性强制约束。

### 反模式7：第二次失败后继续重试

子代理第一次返回摘要→重新委派强调完整性→第二次仍然返回摘要→继续第三次重试。

**实证数据**：第二次失败后第三次成功的概率<30%，继续重试的ROI为负。

**正确做法**：遵循"事不过二"原则，第二次失败后立即切换兜底策略（主会话整合或拆分为更小任务）。

## 边界与选型

**何时使用本模式**：
- 委托子代理创建单个markdown文档文件 → 使用六要素模板
- 委托子代理执行技术分析/对比研究/机制梳理 → 使用四要素扩展模板
- 项目有明确的规范/框架/交付物要求
- 需要精确控制输出格式、内容范围和详细程度

**何时不需要完整要素模板**：
- 委托简单的搜索/查询任务 → 直接描述问题即可
- 委托代码编写（非文档） → 提供代码规范和接口定义即可
- 主进程自己执行任务 → 不需要委托，直接写

**任务复杂度升级策略**：
- 简单任务（搜索、查询）：一句话描述即可
- 单文件创建（无Mermaid）：使用文档类五要素模板（要素1-5）
- 单文件创建（含Mermaid）：使用文档类六要素模板（要素1-6），要素6为Mermaid安全规则
- 单分析/研究任务：使用分析类四要素扩展模板（必须包含完整性强制约束）
- 多文件/多任务：每个任务一个子代理，并行执行
- 多步复杂操作（创建+验证+提交）：拆分为多个顺序任务，每个任务仍遵循相应模板

**失败重试策略**：
- 第一次失败：可快速修正的问题（标签污染、小格式错误）→ 直接修正；内容不全/摘要化 → 重新委派强调完整性
- 第二次失败：⚠️ 立即切换兜底策略，不要重试第三次
  - 推荐：主会话基于已有内容直接整合完成
  - 备选：将任务拆分为更小粒度重新委派
