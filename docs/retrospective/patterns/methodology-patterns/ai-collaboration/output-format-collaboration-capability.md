# 输出格式-协作能力映射（Output Format – Collaboration Capability Mapping）

## 模式类型

方法论模式（AI协作/产品设计）

## 成熟度

L2 验证级（2+ 案例验证：WPS Comate 云文档交付 + Claude Tag 云文档共享）

## 适用场景

- AI 产品输出格式设计：在纯文本、Markdown、云文档、真文件格式之间做出战略选择
- 协作工具评估：评估 AI 工具的"协作友好度"
- 企业级 Agent 关键能力定义：将输出格式纳入企业级 Agent 的必备能力清单
- AI Agent 产品架构评审：审查产品输出格式是否支持团队协作与接力

## 问题背景

传统 Agent 的输出格式以纯文本为主，用户获取结果后只能通过复制粘贴分享给他人。这导致三个核心问题：

1. **协作断裂**：Agent 的输出是"仅供个人消费的内容"，无法直接融入团队协作流，分享需要手动操作，协作链路被切断
2. **接力成本高**：成员 A 拿到 Agent 输出后，成员 B 需要修改或继续编辑时，必须重新输入或重新生成，Agent 产出无法作为协作资产流转
3. **格式降级**：Agent 产出的是"模拟结果"而非"真实产出"——例如生成 PPT 内容却只能输出文本描述，用户需要手动重建，丧失格式保真度

典型症状：
- Agent 生成了一份报告，用户只能复制文本通过微信发送，接收方无法直接编辑
- 团队讨论 Agent 产出时，每个人都看到不同的版本，因为没有共享的"源文件"
- 产品演示中 Agent 生成了"PPT 内容"，但用户仍需打开 PowerPoint 手动制作

## 核心规则

### 规则 1：格式决定协作

输出格式的协作友好度遵循明确的排序：云文档链接 > 真文件格式（PPT/DOCX/XLSX）> Markdown > 纯文本。云文档链接天然支持分享、协作、接力编辑；真文件格式保留格式保真度但需要文件传输；Markdown 可版本控制但协作门槛高；纯文本仅支持复制粘贴，协作能力最弱。

> **为什么？** 输出格式不是技术细节，而是产品哲学——Agent 输出的是"仅供个人消费的内容"还是"可供团队协作的资产"，由输出格式决定。

### 规则 2：默认共享而非私享

Agent 输出结果应默认共享至团队空间（云文档链接、团队文件夹），而非仅返回给个人聊天窗口。默认共享意味着团队任何成员都可以直接访问、编辑、接力 Agent 的产出，无需发起者手动转发。

> **为什么？** 默认私享意味着协作需要额外操作，而额外操作在繁忙的工作流中往往被省略。默认共享将协作从"需要主动做"变为"默认已就绪"。

### 规则 3：格式即承诺

输出格式决定了 Agent 对协作的支持程度，必须在产品设计阶段明确选择。选择纯文本输出意味着产品承诺"仅服务个人"，选择云文档输出意味着产品承诺"支持团队协作"。这一选择不可在后期简单追加——它影响产品架构、用户心智和竞争定位。

> **为什么？** 输出格式的升级不是功能叠加，而是架构变更。从纯文本升级到云文档输出需要底层文档引擎、权限系统、协作基础设施的支撑，不是简单的"加一个功能"。

## 实施检查清单

设计 AI 产品输出格式时自问：
- [ ] 输出格式是否支持团队协作，而非仅个人消费？
- [ ] 输出结果是否默认共享至团队空间，而非仅返回给个人？
- [ ] 输出格式是否保留格式保真度（如真 PPT 文件而非图片）？
- [ ] 团队成员能否直接编辑 Agent 产出，而非重新生成？
- [ ] 输出格式的选择是否与产品的协作定位一致？
- [ ] 是否在输出格式上建立了竞争壁垒（如原生格式支持）？

## 反例警示

| 错误做法 | 后果 |
|---------|------|
| Agent 输出纯文本，用户手动复制粘贴分享 | 协作断裂，产出无法作为团队资产流转 |
| Agent 输出结果仅返回给个人，不共享至团队空间 | 其他成员需要重复提问，浪费上下文和计算资源 |
| Agent 生成 PPT 内容但仅输出文本描述 | 用户需手动重建，格式保真度丧失，效率降低 |
| 产品设计阶段忽略输出格式的战略意义 | 后期从纯文本升级到云文档需要重构底层架构 |
| 将输出格式视为"技术细节"而非"产品承诺" | 产品定位模糊，用户无法判断是否适合团队使用 |

## 正例

**WPS Comate 的实践**：
- Agent 执行搜索和编辑任务后，以云文档链接形式发送结果，而非纯文本
- 团队成员可直接点击链接打开云文档，进行接力编辑和协作
- 生成 PPT 是真文件格式（.pptx），保存在本地电脑，保留完整格式保真度
- 输出格式默认支持多人协作，发起者无需手动转发

**Claude Tag 的实践**：
- Claude Tag 在 Slack 频道中产出结果后，支持以云文档链接形式共享
- 频道内所有成员均可直接访问 Claude 的产出，无需发起者转发
- 输出结果作为团队资产沉淀，而非仅存在于个人对话历史中

**SpecWeave 的对应实践**：
- Markdown 作为 Agent 间协作的标准输出格式，支持版本控制和 diff 比较
- 通过 atomic-commit 规范确保产出可追溯、可审计
- 阶段守卫日志（SG-LOG）作为共享的决策记录，团队可共同查阅

## 与 SpecWeave 的映射

| 输出格式实践 | SpecWeave 对应机制 | 映射关系 |
|------------|-------------------|---------|
| 云文档链接共享 | SG-LOG 日志共享查阅 | 共享产出 |
| 真文件格式保真 | Markdown 源码版本控制 | 格式保真 |
| 默认团队空间共享 | `.agents/` 规范体系全团队可见 | 默认共享 |
| 协作接力编辑 | handoff 协议支持角色间接力 | 协作接力 |

## 与现有模式的关系

- `team-shared-ai-colleague.md`：本模式是团队共享 AI 同事模式的输出层支撑——共享上下文是协作的基础，云文档输出是协作的载体
- `ambient-proactive-agent.md`：本模式的"默认共享"与主动介入 Agent 的"异步汇报"互补——前者确保产出可被团队发现，后者确保产出被主动推送给团队
- `markdown-as-interface.md`：本模式提供了 Markdown 之外更丰富的输出格式阶梯，Markdown 是协作友好度的中间档位，云文档和真文件格式是更高阶选择

## 设计启示

输出格式-协作能力映射模式的核心价值在于**将输出格式从技术决策升级为战略决策**。设计 AI 产品时，不应将输出格式视为"实现细节"，而应将其视为产品对协作支持程度的明确承诺。输出格式的选择决定了产品在"个人工具→团队工具→组织基础设施"光谱上的位置，且这一选择具有不可逆性——从纯文本向云文档的升级成本远高于从云文档向纯文本的降级。
