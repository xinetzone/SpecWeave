# 场景化命名模式（用户语言优先于技术语言）

## 模式概述

ToC/ToSMB智能硬件和软件产品的功能命名与营销文案，不使用技术术语，而是直接用用户场景语言——从"我们的技术能做什么"转换为"用户能用它解决什么问题"。用户不需要知道底层技术原理，只需要知道"点击这个按钮能帮我做什么"。

## 核心逻辑

```
功能命名 = 动词+用户场景（用户听得懂，能联想到自己的需求）
        ≠ 技术术语（工程师听得懂，用户一脸懵）
        ≠ 抽象概念（高大上但不知道有什么用）
```

**核心洞察**：用户购买的不是技术，而是问题的解决方案。功能命名是产品与用户沟通的第一触点——用用户的语言说话，能瞬间降低认知门槛，让用户3秒内理解"这个功能对我有什么用"，从而提高转化率。

## 命名转换对照表

| 技术术语（工程师语言） | 场景化命名（用户语言） | 为什么后者更好 |
|----------------------|----------------------|--------------|
| Wake-on-LAN网络唤醒 | 远程开机 | 用户不需要知道WOL是什么，只关心"人不在也能开机" |
| RTC定时唤醒 | 定时开机 | 用户关心"我要电脑8点自动开"，不关心RTC实时时钟 |
| 多设备并发唤醒指令 | 批量开机 | 用户关心"一键开100台机器"，不关心并发协议 |
| IPKVM硬件旁路远控 | 无网远控 | 用户关心"没网也能控制"，不关心KVM over IP技术 |
| USB-HID仿真输入 | 即插即用远控 | 用户关心"插上去就能用"，不关心HID设备类协议 |
| PDU电源分配单元 | 智能排插 | 用户关心"能远程控制每个插座"，不关心配电单元概念 |
| 端侧AI推理 | 本地AI/离线AI | 用户关心"没网也能用AI"，不关心推理框架 |

## 正反例

### 正例：向日葵开机盒子功能命名

| 功能 | 产品页命名 | 技术真相 | 用户感知 |
|------|-----------|---------|---------|
| 开机功能 | 远程开机 | WOL魔术包网络唤醒 | "人不在公司也能开服务器" |
| 定时功能 | 定时开机 | RTC定时器定时发送唤醒包 | "设置好时间，电脑自动开" |
| 多设备管理 | 批量开机 | 循环/并发发送多个唤醒包 | "机房几百台机器一键开" |
| 跨网能力 | 异地开机 | P2P打洞/中继转发穿越NAT | "在家也能开公司电脑" |

### 反模式

| 反模式 | 表现 | 问题 |
|--------|------|------|
| **技术参数堆砌** | 产品页首页放"采用XX架构/支持XX协议/搭载XX芯片" | 用户不关心技术，关心能解决什么问题 |
| **自造概念词** | 创造"智慧生态/全域互联/赋能引擎"等大词 | 用户听不懂，没有感知，觉得在忽悠 |
| **功能清单罗列** | 只列"支持WOL/支持RTC/支持VPN" | 用户不知道这些功能能帮他做什么 |
| **工程师视角文案** | "本设备通过发送6字节0xFF前缀加16次MAC地址重复的魔术包实现开机" | 普通用户完全看不懂，直接流失 |

## 适用边界

### 适用场景

- ✅ ToC/ToSMB智能硬件产品页和营销文案
- ✅ 大众软件产品的功能命名和按钮文案
- ✅ 面向非技术用户的产品说明书和帮助文档
- ✅ 应用商店/电商平台的产品描述

### 不适用场景

- ❌ 面向专业开发者/工程师的技术产品（API文档/开发板）
- ❌ 技术规格书/白皮书/学术论文
- ❌ 企业级IT采购（技术决策者需要了解技术细节）
- ❌ 极客/发烧友产品（用户以懂技术为荣）

## 实施步骤

1. **功能场景化翻译**：列出所有功能点，对每个功能问"用户在什么场景下用这个功能？解决他什么问题？"
2. **用户访谈验证**：找几个目标用户，问"你觉得这个功能是做什么的？"如果答不出来，名字就起错了
3. **动词开头原则**：功能名尽量用动词开头（远程XX、定时XX、一键XX、批量XX），让用户一眼知道能做什么
4. **技术参数后置**：技术参数、协议细节、芯片型号放在产品页后面的规格表，不要放首页
5. **3秒测试**：用户看到功能名，3秒内能说出来"这能帮我做XX"，就是好名字
