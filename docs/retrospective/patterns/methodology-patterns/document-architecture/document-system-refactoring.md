# 文档体系原子化重构方法论

## 来源
`docs/retrospective/` 目录系统性重构（2026-06-23），将 3 个大型文件拆分为 24 个原子化模块文件。

## 适用场景
- 单文件体积过大（>300 行），内容涵盖多个不同主题
- 不同维度的内容混杂在一起，缺乏清晰的模块边界
- 缺乏统一的目录索引，新成员难以理解整体结构
- 后续维护时难以进行增量更新，修改一处可能影响整个文件

## 流程

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 1. 内容审计    │ → │ 2. 原子化拆分   │ → │ 3. 模块化归类   │ → │ 4. 索引生成    │
│ 分析现有结构   │    │ 按主题拆分为   │    │ 建立功能子目录   │    │ 生成README.md │
│ 识别主题边界   │    │ 独立最小单元   │    │ 制定命名规范   │    │ 标注引用关系   │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

## 五个核心步骤

### 第一步：内容审计

对现有文档进行逐章节分析，识别主题边界。每个章节应满足以下任一条件即可独立为一个模块：

- 围绕一个独立概念展开（如"元文档"、"上下文感知"）
- 描述一个可独立复用的模式（如"三段式检查工具"）
- 承载一个独立的决策框架（如"目录命名决策矩阵"）
- 提供一个可独立使用的模板（如"spec.md 模板"）

**判断标准**：如果一个章节的内容可以被独立引用、独立复用、独立更新，它就是合格的原子化候选。

### 第二步：原子化拆分

将每个主题拆分为独立的 Markdown 文件，每个文件仅包含一个主题的完整内容。拆分原则：

- **一文件一主题**：每个文件只聚焦一个概念、一个模式或一个框架
- **内容自包含**：每个文件应包含该主题的完整信息，无需跳转其他文件即可理解
- **保留代码完整性**：代码片段必须完整可用，不得截断
- **保留上下文标注**：每个文件标注来源与关联模块

**反模式**：
- 将多个弱相关主题合并到一个文件中（如"所有代码模式"放在一个文件）
- 将一个主题拆分得过细（如将"定义"和"特征"拆成两个文件）

### 第三步：模块化归类

按功能或主题建立子目录，创建清晰的模块边界。归类原则：

- **同类相聚**：相同类型的内容放在同一目录下（如所有模板放在 `templates/`）
- **层级适度**：目录深度不超过 3 层，避免过度嵌套
- **边界清晰**：每个目录的职责单一明确，不出现"杂项"目录

**推荐目录结构**：

```
{target}/
├── README.md              ← 目录索引与模块说明
├── templates/             ← 可复用模板（如：文档模板、报告模板）
├── patterns/              ← 可复用模式
│   ├── {category-1}/      ← 模式子类（如：代码模式、架构模式、方法论）
│   └── {category-2}/
├── frameworks/            ← 决策框架（如：决策矩阵、选择原则）
├── concepts/              ← 知识概念（如：术语定义、概念解释）
├── reports/               ← 项目报告（如：复盘报告、审计报告）
└── assets/                ← 资产清单（如：复用指南、资源索引）
```

### 第四步：命名规范

制定统一的命名规范，确保风格一致、易于检索。

| 规范项 | 约定 | 示例 |
|-------|------|------|
| 文件名 | `kebab-case`（小写字母 + 连字符） | `three-tier-check-tool.md` |
| 目录名 | `kebab-case`（小写字母 + 连字符） | `code-patterns/` |
| 禁止 | 空格、中文、下划线、驼峰命名 | `CodePatterns.md` ✗ |

### 第五步：引用追溯

在每个模块文件中标注来源与关联模块，确保引用关系可追溯。

**来源标注**（文件开头）：

```markdown
> **来源**：从 `{原文件路径}` {原章节} 拆分
```

**关联模块标注**（文件末尾）：

```markdown
> **关联模块**：
> - `{相关文件路径 1}`
> - `{相关文件路径 2}`
```

### 第六步：索引生成

在目标目录根目录下生成 `README.md`，作为完整的目录索引。索引文档应包含：

```markdown
# {目录名称}

> {一句话描述目录用途}

## 目录树
```
{完整的目录树结构，展示所有子目录和文件}
```

## 模块说明
### [{子目录名}]({子目录路径})
{1-2 句话描述该子目录的用途}

- [{文件名}]({文件路径}) — {1 句话描述文件内容}

## 命名规范
{命名规范说明}

## 来源说明
{内容来源与重构说明}
```

## 关键原则

1. **增量可维护**：每个模块可独立更新，无需修改其他文件
2. **引用可追溯**：每个模块标注来源，任意内容可追溯到原始出处
3. **导航可自解释**：通过 README.md 即可理解整个目录体系的组织逻辑
4. **命名可预测**：通过 kebab-case 统一命名，无需查看内容即可推测文件主题
5. **层级可扩展**：新增模块只需在对应子目录下添加文件，更新 README.md 即可

## 复用场景

任何需要将大型文档拆分为模块化体系的场景，包括但不限于：

- 项目知识库的体系化重构
- 团队文档规范从"大而全"到"小而精"的迁移
- 开源项目文档的模块化重组
- 企业内部知识管理系统的结构化建设
