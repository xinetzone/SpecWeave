# 模块化接口设计四步法（Modularization Interface Design）

## 模式类型
方法论模式

## 成熟度
L1 实验性（1 次成功案例：methodology-analysis-report.md 综合方法论分析）

## 适用场景
将原子单元组合为更大粒度的可复用模块，重点解决模块边界、接口耦合、版本兼容性等问题。

## 问题背景

原子化解决了"每个单元的内部质量"问题，但单元之间的组织方式决定了系统的可维护性和可扩展性。模块化最常见的失败是"模块边界模糊"——多个模块相互纠缠、修改一处牵动多处。

Parnas 在 1972 年提出的**信息隐藏原则**指出：模块化的目的不是让系统更容易理解，而是让系统在面对变化时成本更低。模块化接口设计四步法是该原则的可操作化版本。

## 四步框架

```mermaid
flowchart LR
    S1["第一步<br/>确定模块边界"] --> S2["第二步<br/>定义对外接口"]
    S2 --> S3["第三步<br/>控制耦合度"]
    S3 --> S4["第四步<br/>建立版本管理"]
    S4 -.->|"接口变更时回退"| S2
```

## 四步详解

### 第一步：确定模块边界

**目标**：基于原子单元的主题聚类，确定每个模块的职责范围
**关键工具**：**一句话原则**——如果无法用一句话清晰描述一个模块的职责，说明边界还需要调整

**判定方法**：
- 用一句话描述模块职责
- 检查描述中是否出现"和""与""及"等连接词
- 出现 → 边界还需要调整

**正例**：
- "智能体角色定义模块：定义 7 个角色的职责、权限与协作接口"
- "知识库检索模块：提供基于标签和分类的全文检索能力"

**反例**：
- "智能体管理模块：定义角色、分配任务、记录日志、处理错误"——多职责混合
- "数据工具模块：读写数据库、调用 API、生成报告"——多领域混合

### 第二步：定义对外接口

**目标**：每个模块对外暴露最小化的接口
**关键原则**：**接口只暴露"做什么"，不暴露"怎么做"**——这正是 Parnas 信息隐藏原则的核心

**接口设计要素**：

| 要素 | 说明 | 反模式 |
|------|------|--------|
| 接口签名 | 方法/函数的输入输出 | 暴露内部数据结构 |
| 错误处理 | 失败时的返回值或异常 | 暴露内部异常堆栈 |
| 副作用声明 | 是否修改全局状态 | 隐瞒副作用 |
| 性能约束 | 时间/空间复杂度上限 | 隐含但未声明的复杂度 |

**本项目实践**：
- 角色定义文件通过统一 TOML frontmatter（id/domain/layer）实现接口标准化
- AGENTS.md 路由表作为"接口描述"——通过路由即可理解模块功能和依赖
- 系统提示词可独立更新而不影响其他角色（接口稳定性的实证）

### 第三步：控制耦合度

**目标**：模块之间的依赖最小化
**耦合度七等级**（从"理想"到"灾难"）：

| 等级 | 类型 | 描述 | 推荐度 |
|------|------|------|--------|
| 1 | 无耦合 | 模块间无任何依赖 | 最佳（不常见） |
| 2 | 数据耦合 | 仅通过参数传递简单数据 | 推荐 |
| 3 | 标记耦合 | 共享数据结构的一部分 | 谨慎 |
| 4 | 控制耦合 | 通过 flag 参数影响行为 | 避免 |
| 5 | 外部耦合 | 共享外部数据格式 | 避免 |
| 6 | 公共耦合 | 共享全局变量 | 禁止 |
| 7 | 内容耦合 | 直接访问内部实现 | 禁止 |

**知识管理领域的对应**：
- 数据耦合：模块间仅通过"参数化引用"（如引用 URL）交互
- 内容耦合：模块 A 直接引用模块 B 的内部章节号（违反信息隐藏）

**实操建议**：
- 模块之间仅通过数据耦合（参数传递简单数据）
- 避免控制耦合（flag 参数）、公共耦合（共享全局变量）、内容耦合（直接访问内部实现）

### 第四步：建立版本管理

**目标**：对于被多个模块依赖的接口，建立版本号机制
**核心原则**：**接口变更应遵循"只增不减"原则**

**版本管理规则**：

| 操作 | 是否为破坏性变更 | 处理方式 |
|------|----------------|---------|
| 新增接口方法 | 否 | 可直接合并 |
| 删除已有方法 | 是 | 需要迁移计划与废弃期 |
| 修改方法签名（参数/返回值） | 是 | 需要迁移计划与废弃期 |
| 修改方法语义（不变输入下输出含义变化） | 是 | 需要迁移计划与废弃期 |

**典型工作流**：
1. 标记旧方法为 `@deprecated`，附带替代方案
2. 在文档中给出迁移指南
3. 设置废弃期（典型 3-6 个月）
4. 废弃期结束后删除旧方法

**本项目实践**：TOML frontmatter 字段的演进遵循"只增不减"——新增字段不破坏已有 frontmatter 解析。

## 模块化产出物清单

| 产出物 | 位置 | 内容 |
|--------|------|------|
| 模块规格说明 | 模块根目录 README | 一句话职责 + 接口列表 + 依赖关系 |
| 接口定义文件 | 模块对外 API 文档 | 输入/输出/错误/约束 |
| 耦合度评估表 | 模块评审记录 | 跨模块依赖清单与等级判定 |
| 版本变更日志 | 模块根目录 CHANGELOG | 接口变更历史与迁移指南 |

## 实施决策准则

**30 秒准则**：如果无法在 30 秒内向一个新成员解释清楚模块间的依赖关系，说明当前的模块化设计需要重新审视——要么是原子化不充分导致模块边界模糊，要么是接口设计不够清晰。

## 反模式警示

| 错误做法 | 后果 |
|---------|------|
| 在原子化未完成时强行模块化 | 模块化在混乱之上加盖一层"看起来整齐"的表层结构 |
| 接口暴露内部实现细节 | 任何内部重构都会破坏所有调用方 |
| 多个模块共享全局变量 | 任何模块的修改都可能影响其他模块的运行时行为 |
| 接口变更无版本管理 | 调用方无法预期升级影响，升级恐惧蔓延 |
| 过早模块化（仅 2-3 个原子单元就建模块） | 模块化收益 < 模块化成本 |
