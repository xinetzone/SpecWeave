# 向日葵硬件系列Wiki标准结构

## 模式概述

经过智能插座（C1Pro/C2/C4）、智能插线板（P4/P1Pro）、智能PDU（P8一代/二代）、智能远控鼠标（MM110/BM110）、开机棒系列、智能摄像头SU1、无网远控硬件（控控2/Q1/Q2Pro/Q0.5/Q5Pro）七次向日葵硬件学习Wiki任务的迭代验证，沉淀出一套稳定高效的硬件产品学习Wiki文档结构。该结构包含两种结构变体（单文件结构和原子化结构），覆盖从1款到5款以上产品的不同复杂度场景，从产品概述到核心概念、逐产品详解、对比分析、原理解析、操作指南、应用场景、市场策略、设计洞察、行业趋势、FAQ、资源链接，形成从"信息"到"理解"到"场景"到"商业"到"前瞻"的完整认知阶梯。

## 核心逻辑

```
Wiki结构 = 产品认知五层金字塔的文档映射
信息层（概述/概念/规格/FAQ/资源）→ 理解层（对比/原理/操作）→ 场景层（应用场景）→ 商业层（市场定位/产品策略/UX洞察）→ 前瞻层（行业趋势）
```

**核心洞察**：硬件产品学习Wiki不应只是参数表和功能列表的堆砌，而应遵循读者的认知路径组织——先了解"是什么"（概述/概念），再逐个深入"每个产品怎么样"（产品详解），然后理解"产品间差异和为什么"（对比分析/工作原理），接着知道"怎么用/在哪用"（操作指南/应用场景），最后上升到"为什么这么设计/行业往哪走"（设计洞察/市场策略/行业趋势）。这种结构让读者从被动接收信息到主动形成判断。

## 标准章节结构

| 序号 | 章节名称 | 必要性 | 内容要点 | 对应认知层级 |
|------|---------|--------|---------|------------|
| 1 | 产品概述 | 必须 | 产品线概览、品牌背景、学习目标、产品矩阵一览 | 信息层 |
| 2 | 核心概念解析 | 必须 | 解释产品涉及的专业术语和技术概念（如IPKVM、BLE低功耗、HID仿真等），为后续理解打基础 | 信息层→理解层 |
| 3~N | 各产品详解 | 必须（每款一章） | 每款产品的定位、工业设计、技术规格表、核心功能特点、适用人群分析 | 信息层 |
| N+1 | 产品对比分析 | 多产品时必须 | 核心参数横向对比表、相同点总结、关键差异点深度分析、分场景选型指南 | 理解层 |
| N+2 | 核心功能与工作原理 | 必须 | 技术原理深度解析（不只是罗列功能，要讲清楚"怎么工作的""为什么能实现"）、通信链路、关键技术 | 理解层 |
| N+3 | 使用流程与操作指南 | 推荐 | 分步骤操作指南、首次设置流程、特殊配置说明、常见问题排障 | 理解层→场景层 |
| N+4 | 典型应用场景 | 必须 | 每个场景配"用户痛点→产品方案→操作流程→价值收益"四要素 | 场景层 |
| N+5 | 市场定位与产品策略 | 洞察类必须 | 用户画像、定价策略、产品矩阵分析、商业模式解读（三层漏斗/双产品矩阵等） | 商业层 |
| N+6 | 产品设计与UX洞察 | 推荐 | 设计哲学分析、技术取舍解读、UX亮点总结、可优化方向讨论 | 商业层→前瞻层 |
| N+7 | 行业趋势 | 可选 | 结合产品分析行业发展方向、技术演进路线、竞争格局变化 | 前瞻层 |
| N+8 | FAQ | 必须 | 官方FAQ + 用户常见问题 + 购买决策FAQ，覆盖购买前/使用中/故障时的疑问 | 信息层 |
| N+9 | 相关资源链接 | 必须 | 官方产品页、App下载链接、固件升级页、帮助中心、社区论坛等 | 信息层 |

### 章节数量说明与结构变体选择

根据产品数量和文档复杂度，有两种结构变体：

**变体A：单文件结构**（适用1-2款产品）
- 所有章节在一个MD文件中
- 章节总数约11-13章
- 适合：产品≤2款、预计总行数<800行的Wiki

**变体B：原子化结构**（适用≥3款产品，推荐≥5款使用）
- 1个索引文件（含frontmatter+产品矩阵表+目录导航）+ 多个独立章节文件
- 编号前缀保证排序（00-overview.md, 01-core-concepts.md, 02-xxx.md...）
- 每款产品单独一个文件，对比分析单独成章
- 每个MD文件对应一个TOML元数据文件
- 适合：产品≥3款、预计总行数>800行、需要多人协作或增量更新的Wiki

**结构决策树**：
```
产品数量=1? → 单文件结构
产品数量=2? → 单文件结构（注意控制长度）
产品数量=3? → 评估预计行数：>800行用原子化，≤800行用单文件
产品数量≥5? → 必须用原子化结构
```

原子化结构的文件命名约定：
| 文件名 | 内容 | 必要性 |
|--------|------|--------|
| `00-overview.md` | 产品概述、产品矩阵一览、目录导航 | 必须 |
| `01-core-concepts.md` | 核心概念解析 | 必须 |
| `02-{product1}.md` ~ `0N-{productN}.md` | 各产品详解（每款一章） | 必须 |
| `{N+1}-comparison.md` | 产品横向对比分析 | 多产品时必须 |
| `{N+2}-technology.md` | 核心功能与工作原理 | 必须 |
| `{N+3}-usage-guide.md` | 使用流程与操作指南 | 推荐 |
| `{N+4}-scenarios.md` | 典型应用场景 | 必须 |
| `{N+5}-market-strategy.md` | 市场定位与产品策略 | 洞察类必须 |
| `{N+6}-design-insights.md` | 产品设计与UX洞察 | 推荐 |
| `{N+7}-industry-trends.md` | 行业趋势 | 可选 |
| `{N+8}-faq.md` | FAQ | 必须 |
| `{N+9}-resources.md` | 相关资源链接 | 必须 |

## 七次验证案例

| 验证次序 | 产品 | 结构变体 | 文件数 | 验证重点 |
|---------|------|---------|-------|---------|
| 第1次 | 智能插座C1Pro/C2/C4（3款） | 单文件 | 1 | 基础结构验证，多产品差异处理 |
| 第2次 | 智能插线板P4/P1Pro（2款） | 单文件 | 1 | 对比分析章节优化，场景章节标准化 |
| 第3次 | 智能PDU P8一代/二代（2款） | 单文件 | 1 | 技术原理深度验证，工业级vs消费级差异处理 |
| 第4次 | 智能远控鼠标MM110/BM110（2款） | 单文件 | 1 | 双产品矩阵策略分析验证，40倍功耗差异量化方法沉淀 |
| 第5次 | 开机棒系列 | 单文件→原子化 | 1→10 | 首次原子化拆分验证，source-document-downgrade模式配合 |
| 第6次 | 智能摄像头SU1（1款） | 单文件 | 1 | 单产品场景验证，简化结构适配 |
| 第7次 | 无网远控硬件控控2/Q1/Q2Pro/Q0.5/Q5Pro（5款） | **原子化结构** | 12 | 多产品原子化Wiki结构首次大规模验证（1索引+11章节），确立≥3款产品的结构决策标准 |

七次验证结论：两种结构变体在单产品/双产品/多产品（最多5款）、消费级/工业级/专业级、不同品类（插座/插线板/PDU/鼠标/开机棒/摄像头/远控硬件）的硬件Wiki中均能高效组织内容，结构稳定。原子化结构在产品≥3款时优势明显：增量更新、并行协作、导航清晰、文件大小可控。

## 章节编写质量要点

### 产品详解章（3~N）

每款产品的详解章应包含统一结构：
1. **产品定位**：一句话说明这是什么、面向谁、解决什么问题
2. **外观设计**：形态、材质、颜色、接口、指示灯、按键
3. **技术规格**：表格形式列出核心参数
4. **核心功能**：3-5个核心功能点，每个功能点讲"是什么+怎么用+价值是什么"
5. **适用人群**：明确什么样的用户适合这款产品

### 产品对比分析章（N+1）

- 必须包含横向参数对比表（所有产品的核心参数放在一张表里）
- 不能只列参数，必须有"相同点"和"差异点"的文字总结
- 差异点分析要回答"为什么有这个差异"（对应场景驱动参数取舍模式）
- 必须提供选型指南："如果你是X场景用户，选A；如果是Y场景，选B"

### 核心功能与工作原理章（N+2）

- 这是Wiki区别于官网产品页的核心价值所在
- 不讲功能列表，讲"黑盒里发生了什么"
- 用流程图/Mermaid图展示通信链路和工作流程
- 解释关键技术的原理（如BLE低功耗为什么省电、IPKVM怎么实现无网远控）

### 典型应用场景章（N+4）

每个场景必须包含四要素：
1. **用户痛点**：用户遇到什么具体问题？
2. **产品方案**：本产品如何解决这个问题？
3. **操作流程**：具体怎么设置/使用？
4. **价值收益**：解决后带来什么可量化的收益？

### 市场定位与产品策略章（N+5）

这是洞察类Wiki的高价值章节，必须回答：
- 谁会买这个产品？（用户画像）
- 为什么这么定价？（定价策略分析）
- 和其他产品怎么形成矩阵？（产品组合策略）
- 软件公司为什么做这个硬件？（商业模式分析）

## 适用边界

### 适用场景

- ✅ 消费电子/智能硬件产品的学习Wiki/教程编写
- ✅ SaaS公司生态硬件（配合App使用的专用外设）的产品分析
- ✅ 同一品类下多产品（2-5款）的对比分析文档
- ✅ 需要兼顾"技术深度"和"商业洞察"的产品分析

### 不适用场景

- ❌ 纯软件产品/服务的介绍文档（缺少硬件规格/原理/设计等要素）
- ❌ 单一SKU无对比的极简产品（对比章节无内容可写）
- ❌ 纯评测/开箱文（只讲体验不讲原理和策略）
- ❌ 官方产品页/电商详情页（面向营销而非学习）
