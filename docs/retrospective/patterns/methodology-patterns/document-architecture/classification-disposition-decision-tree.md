# 文档原子化分类处置决策树模式

## 模式概述

当面对**N≥3个文档/目录**需要进行原子化拆分或结构治理时，避免"一刀切全部拆分"的反模式，先对所有待处理对象进行分类，针对不同类别采取不同处置策略（新建拆分/补全导航/跳过保留/归档不碰），从源头避免过度拆分、重复拆分和无效操作。

本模式是[document-atomization-u-curve.md](document-atomization-u-curve.md)（原子化U型曲线）的前置决策框架——在决定"怎么拆"之前，先决定"要不要拆"。也是[meta-atomization-bisect-overview.md](meta-atomization-bisect-overview.md)批量推广场景的前置依赖。

## 问题现象

大规模文档治理/原子化时常见的失败模式：

1. **一刀切全部拆分**：不管文件大小、结构、状态，所有文件都按同一标准拆分，导致小文件过度碎片化
2. **重复创建冗余文件**：不检查就新建details文件，结果目录下已有phases/子目录做了同样的拆分
3. **过度拆分耦合内容**：把逻辑高度关联的内容强行拆分，导致读者必须在5个文件间跳转才能理解一个完整主题
4. **扰动已归档文档**：对已完成/已归档的稳定文档做结构改动，引入不必要的变更风险
5. **遗漏已有导航**：已有原子化结构的目录，没有补全README导航就继续处理下一个，导致导航断裂

这些模式的共同问题是：在没有判断"是否需要处理"和"如何处理"之前就开始动手，产生大量无用功和新的质量问题。

## 解决方案

### 四类处置决策树

```
待处理对象扫描完成
│
├─ Q1: 已有原子化结构？
│   ├─ 是（已有phases/、details/、*-details.md、*-sX-sY.md等）
│   │   └─ ✅ 处置方式：补全导航
│   │        检查父README导航完整性 → 补全缺失条目 → 修复断链
│   │        不重新拆分
│   │
│   └─ 否 → Q2: 文件<100行 或 内容高度关联？
│            ├─ 是（单段<30行/连续推理链/拆分后无法独立阅读）
│            │   └─ ⏭️ 处置方式：保留原状
│            │        不拆分 → 适当添加changelog和导航标记即可
│            │
│            └─ 否 → Q3: 文件已归档/已完成？
│                     ├─ 是（archive状态、已结项、不再活跃维护）
│                     │   └─ ⏭️ 处置方式：保持原状
│                     │        仅修复断链 → 不做结构改动
│                     │
│                     └─ 否 → Q4: 模板/格式轻量升级场景？
│                              ├─ 是（仅更新frontmatter/导航表/新增标准文件，不拆分结构）
│                              │   └─ ✂️ 处置方式：轻量升级（按SOP拆分的子类）
│                              │        补齐缺失文件/字段 → 更新导航表 → 集中格式校验
│                              │
│                              └─ 否（需要结构拆分重组）
│                                   └─ ✂️ 处置方式：按SOP拆分
│                                        使用元原子化二分+概览模式或大文档原子化法
```

### 五类处置方式详解

| 处置方式 | 标记 | 适用场景 | 操作内容 | 工作量 |
|---------|------|---------|---------|--------|
| **轻量升级** | ✂️ | 模板/格式标准化，仅补文件/字段/导航，不拆结构 | 补齐缺失标准文件→更新frontmatter字段→更新导航表→P1.5集中格式校验→抽查链接 | 低-中 |
| **按SOP拆分** | ✂️ | 250-500行、结构清晰、章节占比>50% | 按元原子化6步法执行：识别→策略→提取→概览→更新README→验证 | 中 |
| **补全导航** | ✅ | 已有原子化结构但导航不完整 | 检查并补全父README导航表、双向链接、changelog | 低 |
| **保留原状** | ⏭️ | 文件过小、内容高度耦合、不适合拆分 | 添加必要的导航标记和changelog，记录"不拆分"决策原因 | 极低 |
| **保持原状** | ⏭️ | 已归档/已完成文档 | 仅修复断链，不做任何结构变更 | 极低 |

### 判断准则补充

#### Q1判断：如何识别"已有原子化结构"？

"已有原子化结构"不仅包括显式的拆分文件，还包括：
- 文件形式：`*-details.md`、`*-s1-s3.md`、`*-part1.md`等拆分产物
- 目录形式：`phases/`、`chapters/`、`details/`、`sections/`等子目录包含拆分后的文件
- 隐式形式：主文件中已有"详见xxx"指向独立章节文件的链接

#### Q2判断：什么是"内容高度关联/独立阅读价值低"？

符合以下任一条件即可判定为不适合拆分：
1. **推理链条连续**：段落构成连续的因果/推理链条，拆分后断裂
2. **单段过短**：所有子段落平均<30行，拆分后每个文件内容过少
3. **交叉引用密集**：段落间互相引用超过3处，拆分后需要大量回链
4. **整体不可分**：内容是一个完整的整体（如一篇完整的分析文章），拆分后失去意义

**保守原则**：当拿不准是否拆分时，先不拆分。如果后续使用中确实发现导航困难，再拆分也不迟——拆分容易，合并难。

#### Q3判断：如何识别"已归档/已完成"状态？

- frontmatter中标记`status: archived`或`completed`
- 文档changelog显示最近6个月无更新
- 目录名包含`archive`、`completed`、`final`等标记
- 项目/任务已明确结项，不再有后续迭代

#### Q4判断：什么场景属于"轻量升级"（vs 按SOP拆分）？

轻量升级是"按SOP拆分"的子类，适用于**模板/标准版本更新**场景，核心判定条件（全部满足）：

| 判定条件 | 说明 |
|---------|------|
| **不涉及结构拆分** | 目标对象已有合理的文件结构，本次操作不需要拆分任何文件 |
| **仅补全/更新格式要素** | 操作内容仅限于：补齐缺失的标准文件（如backlog）、更新frontmatter字段（scenario/source/version）、更新导航表/交付物清单 |
| **日期≥分界点** | 对象创建日期在新版本方法论/模板稳定发布日期之后（分界点之前的历史项目追溯修改ROI低，归入"保持原状"） |
| **非嵌套迭代** | 不是v*-iteration/、retrospective-meta-*/等迭代或元复盘嵌套目录 |
| **是标准正式复盘项目** | 包含≥4个标准核心文件（README+至少2个核心文档），非迷你复盘 |

**轻量升级vs按SOP拆分的区别**：
- 轻量升级：结构不变，只补文件/字段/导航，是"格式标准化"
- 按SOP拆分：结构需要重组拆分，是"架构治理"
- 轻量升级执行效率更高（单项目<5分钟），适合批量处理；按SOP拆分单项目需要15-30分钟

### 批量处置流程

使用本模式进行批量文档治理的标准流程：

1. **扫描盘点**：列出所有待处理对象，记录路径、行数、当前状态
2. **分类标记**：对每个对象运行决策树Q1-Q4，标记处置方式（五类：轻量升级/按SOP拆分/补全导航/保留原状/保持原状）
3. **生成处置清单**：形成表格（对象+处置方式+预计工作量+备注）
4. **分批执行**：按[phased-rollout-validation.md](../governance-strategy/phased-rollout-validation.md)四阶段推广
   - P0验证批：3-5个低风险对象（轻量升级选结构最一致的，按SOP拆分选最简单的）
   - P1推广批：大部分"轻量升级"/"按SOP拆分"对象 + 部分"补全导航"对象
   - P1.5集中格式校验：P1完成后全量扫描frontmatter字段/导航表完整性/文件计数，批量修正共性格式问题（N≥20时必选）
   - P2收尾批："补全导航"+"保留/保持原状"的特殊对象
5. **全量验证**：所有对象处理完成后运行链接检查

## 适用场景

- ✅ 批量文档原子化治理（≥5个对象）
- ✅ 历史遗留文档结构清理
- ✅ 新方法论/模板推广前的资产盘点
- ✅ 知识库重构、目录重组
- ✅ 复盘报告批量标准化
- ❌ 单文档拆分（直接使用元原子化模式即可）
- ❌ 纯机械操作（如批量重命名、批量frontmatter修复）
- ❌ 内容创作（新写文档不涉及分类处置）

## 实际案例

### 案例1：元原子化模式批量推广（2026-07-05）

**背景**：16个复盘报告目录需要应用元原子化模式。

**如果用反模式（一刀切拆分）**：
- 5个已有原子化目录的项目会被重复创建冗余details文件
- 3个单文件大但内容高度关联的项目会被过度拆分
- 2个已完成项目会被不必要地扰动
- 预计无效工作量：约40%，并产生大量冗余文件和断链

**使用分类处置决策树**：
1. **扫描分类**：16个对象分为4类：
   - ✂️ 按SOP拆分：6个（真正需要原子化的项目）
   - ✅ 补全导航：5个（已有phases/目录但README导航不完整）
   - ⏭️ 保留原状：3个（内容高度关联不适合拆分，如tuya-official 326行大文件）
   - ⏭️ 保持原状：2个（已完成的归档项目）
2. **分批执行**：P0验证批3个拆分对象→P1推广批6个→P2收尾批7个（补全+保留）
3. **结果**：
   - 避免了5个项目的重复拆分
   - 避免了3个项目的过度拆分
   - 仅6个项目真正执行拆分，工作量减少62.5%
   - 最终33个链接0断链

### 案例2：TuyaOpen项目已有结构识别

**背景**：TuyaOpen项目目录下已有`phases/phase-1~6.md`的深度拆分。

**分类判断**：Q1=是（已有phases/子目录）→ 处置方式=补全导航。

**操作**：删除最初错误创建的`execution-phase-details.md`冗余文件，更新主README的导航表指向已有phases/目录，添加双向导航链接。

**结果**：避免了冗余文件，复用了已有拆分成果。

### 案例3：复盘模板v1.2批量标准化升级（2026-07-06）

**背景**：comprehensive-retrospective-template升级至v1.2.0后，需要将新模板要素（insight-action-backlog.md、scenario标识、导航更新）推广到所有6月29日之后创建的复盘项目。

**如果用反模式（一刀切全量升级）**：
- 27个嵌套子目录（v11/v12迭代、retrospective-meta等）会被错误创建冗余backlog
- 27个6月28日及之前的历史复盘会被不必要扰动
- 预计无效工作量：约45%，并产生大量无意义变更

**使用分类处置决策树**：
1. **扫描分类**：119个缺少backlog的项目分为4类：
   - ✂️ 轻量升级：61个（6月29日及之后、非嵌套、标准结构项目）→ P0验证5个→P1推广56个
   - ✅ 补全导航：4个（已有子目录结构的项目，检查导航完整性）
   - ⏭️ 保留原状：27个（嵌套子目录、迷你复盘）
   - ⏭️ 保持原状：27个（6月28日及之前历史复盘，追溯修改ROI低）
2. **分批执行**：按phased-rollout-validation三阶段执行
3. **结果**：
   - 避免了54个项目的无效升级（27保留+27历史）
   - 61个目标项目完成轻量升级
   - P1批完成后修正了11个scenario字段格式问题，抽查链接零断链
   - 工作量相比一刀切方案减少约45%

## 反模式

### ❌ 反模式1：一刀切拆分

"所有超过200行的文件都拆了"——问题：忽略了内容耦合度、已有结构、归档状态等因素，导致过度拆分和冗余文件。原子化U型曲线告诉我们：拆分不足不好，拆分过度更糟。

### ❌ 反模式2：不检查现有结构

上来就创建details.md——问题：可能已有phases/目录做了同样的拆分，产生冗余。步骤2的前置检查"先ls目标目录"必须执行。

### ❌ 反模式3：扰动归档文档

归档文档反正都在这，一起改了吧——问题：归档文档是稳定的历史记录，结构改动会引入不必要的风险和diff噪音。归档文档只修复断链，不改结构。

### ❌ 反模式4：不记录"不拆分"决策

这个文件没拆，可能是忘了——问题：后续维护者看到没拆分的文件可能以为是遗漏，重复尝试拆分。对"保留/保持原状"的决策应在changelog中简要记录原因。

## 边界与选型

**什么时候不应该用这个模式？**
- 单文档处理：直接判断三标准即可，不需要完整决策树
- 全新文档创作：新写文档不涉及存量分类
- 机械批量操作：如批量添加frontmatter、批量修复链接格式，所有对象处理方式相同

**什么时候必须用？**
- 批量原子化/文档治理≥5个对象
- 不确定哪些文档需要拆分、哪些不需要
- 接手遗留项目，需要先盘点再动手
- 新方法论推广前的资产梳理

## 检查清单

使用本模式时，确认以下事项：

- [ ] 已扫描所有待处理对象并形成清单（含路径、行数、状态）
- [ ] 每个对象都通过Q1-Q4决策树分类（五类：轻量升级/按SOP拆分/补全导航/保留/保持）
- [ ] "已有原子化结构"检查了子目录和文件两种形式
- [ ] "内容高度关联"按4条判断准则评估
- [ ] "已归档"检查了frontmatter标记和changelog历史
- [ ] "轻量升级"按Q4的5项判定条件验证（不拆结构+仅补格式+日期分界+非嵌套+标准项目）
- [ ] 拿不准的对象遵循"保守原则"暂不拆分/暂不升级
- [ ] 处置清单已生成，包含每个对象的处置方式
- [ ] "保留/保持原状"的决策在changelog中记录了原因
- [ ] 分类后按phased-rollout-validation四阶段分批执行（N≥20时含P1.5集中校验）
