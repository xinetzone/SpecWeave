# 技术规格与叙述报告分离原则

## 模式类型
文档架构模式

## 成熟度
**L2 已验证**（并发安全检查器复盘报告拆分验证：264行→187行/-29%，独立spec可引用）

## 一、核心原理

文档中存在两种性质不同的内容，它们的阅读模式、更新频率和引用需求截然不同，不应混合在同一文件中：

| 维度 | 叙述内容（Narrative） | 参考规格（Spec/Reference） |
|------|---------------------|--------------------------|
| **阅读模式** | 流式：从头到尾顺序阅读，理解上下文脉络 | 查阅式：按需跳读特定条目，不需要上下文 |
| **更新频率** | 归档后基本不变（记录历史事实） | 可能独立更新（规则调整、参数变化） |
| **引用需求** | 通常只在报告上下文中有意义 | 可能被多个文档单独引用 |
| **内容形态** | 时间线、过程分析、经验总结、改进建议 | 规则表格、API清单、配置项枚举、参数定义 |
| **认知负荷** | 高（需要理解因果关系和决策逻辑） | 低（查表获取信息） |

**核心思想**：当文档中包含大段参考规格类内容时，应将其独立为spec文件，叙述性文档保留摘要和引用链接。

## 二、识别标准（何时需要分离）

满足以下**任意2项**即应考虑分离：

| 判断标准 | 阈值 | 说明 |
|---------|------|------|
| 内容规模 | 表格/清单≥30行 | 大段参考资料打断叙述流 |
| 独立引用 | 可能被≥2个其他文档引用 | 其他文档需要引用规则定义而非整个复盘 |
| 独立演进 | 内容可能在报告归档后更新 | 规则/参数/配置随代码迭代更新 |
| 形态差异 | 主要是表格/枚举/清单而非段落 | 结构上就是参考资料而非叙述 |
| 阅读打断 | 读者需要"跳过"这部分才能继续 | 叙述流被参考资料阻断 |

### 反模式识别信号

- ❌ 报告中出现"详细规则见下表，后文继续讨论..."之类的过渡语
- ❌ 读者反馈"报告太长，找不到核心结论"
- ❌ 其他文档需要引用规则定义时被迫链接整个复盘报告
- ❌ 规则更新需要修改已归档的复盘报告

## 三、实现步骤

### Step 1：内容识别与标记

通读文档，标记所有参考规格类内容块：

```
识别关键词：
- - "XX规则如下"、"XX清单"、"XX定义"、"XX参数表"
- 大段Markdown表格（≥10行）
- 枚举/列表形式的配置项、检查项、API参数
- 代码块中的常量定义/配置模板
```

### Step 2：创建Spec文件

1. 在同一目录下创建独立spec文件，命名规范：`{topic}-spec.md`
2. 包含完整frontmatter（id/title/source/tags）
3. 将标记的内容块迁移到spec文件，补充必要的上下文说明
4. spec文件内部按逻辑分节，保持独立可读性

### Step 3：重构原文档

1. 在原文档中移除迁移的内容块
2. 替换为1-2段摘要 + 指向spec文件的链接
3. 如果原文档有章节编号，调整编号保持连续性
4. 在"文件索引"或"相关文档"部分添加spec文件条目

### Step 4：双向引用验证

| 检查项 | 方法 |
|--------|------|
| 原文档→spec文件 | 链接有效，摘要准确反映spec内容 |
| spec文件→来源 | spec的source字段指向原复盘报告 |
| cross_refs | frontmatter中互相引用（如适用） |

## 四、分离模板

### 原文档中的替换格式

```markdown
### X.X {章节名}

{topic}的详细定义、规则和参数见独立规格文档：[{topic}-spec.md]({topic}-spec.md)。

**核心要点摘要**：
- 要点1：一句话概括
- 要点2：一句话概括
- 要点3：一句话概括
```

### Spec文件标准结构

```markdown
---
id: "{topic}-spec"
title: "{Topic}技术规格"
source: "external: 不存在-docs/.../retrospective-report.md#X.X"
maturity: "L1/L2/L3"
tags: ["specification", ...]
---

# {Topic}技术规格

> 本文档从{来源报告}中独立出来，提供{主题}的完整参考规格。

## 一、概述
{简要说明本规格的用途和适用范围}

## 二、详细规则/定义
{表格、清单、枚举等核心参考内容}

## 三、使用注意事项
{使用时的约束、边界条件、常见误区}

## 四、已知局限
{当前实现的已知限制和未来改进方向}
```

## 五、适用场景

| 场景 | 示例 | 是否分离 |
|------|------|---------|
| 复盘报告中的规则详表 | 八维检查法规则表、测试五件套清单 | ✅ 分离 |
| 架构文档中的API清单 | 接口列表、参数定义 | ✅ 分离 |
| 开发规范中的检查项清单 | Code Review Checklist | ✅ 分离 |
| 报告中的2-3行小表格 | 关键数据对比表 | ❌ 不分离 |
| 嵌入叙述中的短代码片段 | 解释概念用的代码示例 | ❌ 不分离 |
| 完全独立的参考手册 | CLI命令参考、配置参考 | ✅ 独立成文档 |

## 六、潜在限制

1. **过度碎片化风险**：如果每个小表格都分离，会导致文件数量过多，维护成本上升。使用"任意2项"判断标准避免过度拆分。
2. **上下文丢失**：分离后的spec文件缺少复盘报告中的决策背景，需要在spec的"概述"部分补充必要上下文。
3. **双向链接维护**：分离后需要维护原文档→spec和spec→原文档的双向链接，增加了链接维护成本。
4. **不适合时间序列内容**：时间线、过程回顾等强上下文依赖的叙述内容不适合分离。

## 七、使用注意事项

1. **分离后验证可读性**：分离后独立阅读原文档，确认叙述流连贯，不因移除内容而产生逻辑断裂
2. **spec文件可独立阅读**：确保spec文件本身是自包含的，读者不需要回到原报告就能理解内容
3. **控制文件粒度**：单个spec文件建议30-150行，过小考虑合并，过大考虑进一步拆分
4. **命名一致性**：spec文件命名使用`{descriptive-name}-spec.md`格式，与目录中其他文件风格一致
5. **原子提交**：分离操作单独一次提交，不要混入内容修改，便于review和回滚

## 八、验证案例

### 案例：并发安全检查器复盘报告拆分

| 指标 | 拆分前 | 拆分后 | 变化 |
|------|--------|--------|------|
| retrospective-report.md | 264行 | 187行 | -29% |
| 八维规则详表 | 内嵌§1.4（55行） | eight-dimensions-concurrent-safety-spec.md（70行） | 独立可引用，后迁移至best-practices |
| 报告结构 | 四章节混合 | 标准五段式 | 对齐模板 |
| 可引用性 | 需链接整个报告 | 直接链接spec | 提升 |
| 链接验证 | — | 28个链接全部有效 | ✅ |
