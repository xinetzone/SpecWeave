# 知识档案四层架构（Knowledge Archive 4-Layer Architecture）

## 模式类型

方法论模式（研究知识/信息架构/内容组织）

## 成熟度

L2 已验证（1次完整端到端项目验证，第一性原理知识档案项目成功构建12个核心文件的结构化档案，覆盖哲学/物理/商业/方法论四大领域）

| 验证指标 | 数值 | 说明 |
|---|---|---|
| 知识档案文件数 | 12 | 不含Spec管理文档 |
| 内容总行数 | 4869 | 知识档案内容行数 |
| 覆盖领域数 | 4 | 哲学/物理/商业/方法论 |
| 本地链接数 | 85 | check-links验证全部有效 |
| 内容模块耦合度 | 低 | 领域内容可独立增量更新 |
| 跨领域术语统一 | 12个核心概念 | 术语表明确定义 |

## 适用场景

| 场景 | 是否适用 | 说明 |
|---|---|---|
| 系统性专题知识档案 | ✅ 核心场景 | 本模式原生设计目标 |
| 学习笔记体系构建 | ✅ 核心场景 | 四层结构清晰，便于增量添加 |
| 多领域研究资料库 | ✅ 核心场景 | 跨领域整合层解决一致性问题 |
| 开源项目文档/知识库 | ✅ 适用 | 规则统一，内容可分模块维护 |
| 企业内部知识管理 | ✅ 适用 | 可审计性和可维护性良好 |
| 单篇文章/单主题笔记 | ❌ 过度设计 | 内容量小不需要四层结构 |
| 临时信息收集/草稿 | ❌ 不适用 | 结构化成本过高 |
| 个人日记/零散记录 | ❌ 不适用 | 不需要规则层和跨领域整合 |

## 核心问题

为什么知识档案需要分层架构？

1. **规则与内容混在一起**：质量标准、审查流程散落在各个文件中，新增内容时不知道遵循什么标准
2. **领域内容耦合**：不同领域的内容相互交织，新增一个领域需要修改多个现有文件
3. **跨领域语义漂移**：同一个术语在不同领域含义不同，没有统一术语层导致概念混乱
4. **索引过早固化**：一开始就做README和导航，内容还没稳定导致导航频繁大幅修改
5. **缺乏过程可追溯**：没有独立的方法/规则/验证记录层，知识档案的构建过程不可审计

## 解决方案：四层分离架构

```
┌─────────────────────────────────────────────────────────┐
│  Layer 4: 索引层（Index Layer）                         │
│  ─────────────────────────────────────────────────      │
│  - README.md（档案首页+导航入口）                        │
│  - 阅读路径推荐                                          │
│  - 内容快速索引                                          │
│  ⚠️  【最后做】内容稳定后再构建索引，避免频繁重构         │
├─────────────────────────────────────────────────────────┤
│  Layer 3: 跨领域整合层（Integration Layer）             │
│  ─────────────────────────────────────────────────      │
│  - 术语表（Glossary）：统一跨领域概念定义                │
│  - 时间线（Timeline）：按时间组织发展脉络                │
│  - 方法论框架（Methodology）：提炼通用方法                │
│  - 延伸阅读（Further Reading）：分级推荐                  │
│  🔧 【解决问题】跨领域语义漂移、概念不一致               │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 领域内容层（Domain Content Layer）            │
│  ─────────────────────────────────────────────────      │
│  - 按领域/主题分文件（01-philosophy, 02-physics...）     │
│  - 每个文件聚焦单一领域                                  │
│  - 独立成模块，便于增量扩充                              │
│  - 模块间通过术语层保持一致性                            │
│  📦 【设计原则】高内聚低耦合，可独立维护                 │
├─────────────────────────────────────────────────────────┤
│  Layer 1: 规则层（Protocol Layer）                      │
│  ─────────────────────────────────────────────────      │
│  - 00-adversarial-review-protocol.md                    │
│  - 来源分级标准、可信度评分规则                          │
│  - 审查流程、偏差检查清单                                │
│  - 验证日志模板                                          │
│  ⚠️  【最先做】先定义规则，再开始搜集内容               │
└─────────────────────────────────────────────────────────┘
```

## 逐层设计规范

### Layer 1: 规则层（00文件，最先做）

**文件命名**: 必须以 `00-` 开头，明确标识为规则文件

**核心内容**:
1. 来源准入标准（什么级别的来源可以进入）
2. 质量验证流程（需要通过哪些检查）
3. 可信度评分规则（每个等级的标准）
4. 写作/编辑规范（格式、引用、标注方式）
5. 偏差防御清单（需要检查的认知偏差）
6. 异常处理规则（存疑内容如何标记）
7. 验证日志模板

**设计原则**:
- ✅ 规则必须在内容搜集开始前定义完成
- ✅ 规则应该具体可执行（"检查确认偏误：主动搜索反方观点"而非"注意客观性"）
- ✅ 规则层只包含"怎么做"，不包含"是什么"（内容）
- ❌ 不要在规则层混入具体领域内容
- ❌ 不要边做内容边改规则（除非发现规则有根本性缺陷）

**为什么规则层必须最先做？**
> 这是"质量内建"原则的体现——如果先做内容再定规则，会出现"用错误标准做了很多工作再推翻"的大规模返工。规则前置看似"慢了一步"，实际避免了后续数十倍的返工成本。

### Layer 2: 领域内容层（01-N，分领域填充）

**文件命名**: `NN-{domain-name}.md`，NN从01开始顺序编号

**核心内容**:
- 每个文件聚焦单一领域/主题
- 内容组织遵循该领域的内在逻辑（时间线/主题分类/重要性排序）
- 按规则层标准进行来源标记和可信度评分
- 文件内部结构统一（便于读者预期）

**设计原则**:
- ✅ 单一职责：一个文件只讲一个领域
- ✅ 高内聚：同一领域的内容集中在一个文件中
- ✅ 低耦合：领域文件之间尽量减少直接引用，通过术语层和索引层关联
- ✅ 可独立扩充：新增领域只需新增文件，不修改已有内容文件
- ✅ 遵守规则层标准：不自行降低来源要求
- ❌ 不要在领域文件中定义跨领域概念（放到术语层）
- ❌ 不要跨领域直接引用具体内容（容易产生断链和不一致）

**领域划分原则**:
- 按学科/主题划分（哲学/物理/商业）而非按来源类型
- 每个领域文件大小控制在300-500行（超过则考虑拆分）
- 领域数量建议3-7个（过多会增加整合层工作量）

### Layer 3: 跨领域整合层（N1-N2，内容基本完成后做）

**典型文件**:
- 术语表（Glossary）：统一跨领域概念的定义
- 时间线（Timeline）：按时间顺序串联跨领域事件
- 方法论框架（Methodology）：提炼跨领域通用方法
- 延伸阅读（Further Reading）：分级阅读路径
- 验证日志（Validation Log）：来源验证过程记录

**核心内容**:
- 识别并解决跨领域语义漂移（同一个词在不同领域含义不同）
- 建立跨领域的关联（A领域的X概念如何影响B领域的Y发展）
- 提炼跨领域的通用模式和方法论
- 提供统一的检索和导航辅助
- 记录质量验证过程（与规则层对应）

**设计原则**:
- ✅ 必须等领域内容基本完成后再开始整合层
- ✅ 每个跨领域概念在术语表中有且仅有一个统一定义
- ✅ 整合层是"视图"，不新增原始领域知识（原始知识在领域层）
- ❌ 不要在内容搜集早期就做时间线和方法论——内容不全时整合会遗漏关键节点
- ❌ 不要在术语表中遗漏任何跨领域使用的概念

**为什么整合层是隐性难点？**
> 同一个术语在不同领域可能有完全不同的含义，这种"语义漂移"是隐性的——在你尝试整合之前，你甚至不会意识到它存在。第一性原理项目中，"第一性原理"在哲学（第一因）、物理（基本粒子）、商业（反类比）三个领域含义差异显著，术语表花费了约15%的总工作量，而这个工作在初期任务分解中被低估了。

### Layer 4: 索引层（README，最后做）

**核心文件**: `README.md`（档案首页）

**核心内容**:
- 档案定位说明（这是什么、覆盖什么范围）
- 阅读路径推荐（不同背景读者从哪里开始）
- 内容快速索引（链接到各层文件）
- 可信度说明（如何使用可信度标记）
- 贡献/维护指南（如何新增内容）
- 版本/更新记录

**设计原则**:
- ✅ 必须等其他三层内容都稳定后再做索引
- ✅ 索引只做导航，不重复内容（内容在各层文件中）
- ✅ 提供多种阅读路径（初学者/进阶/研究者）
- ❌ 不要一开始就写README——内容还没定型，README会频繁大幅修改
- ❌ 不要在README中堆砌内容，保持简洁作为入口

**为什么索引层要最后做？**
> 索引是内容的"地图"，如果领土（内容）还在变化，地图不可能准确。早期做的README会随着内容扩充频繁推翻重写，而且会反过来约束内容结构（"因为README里是这么写的，所以内容要这么组织"），导致内容为了适配索引而扭曲。让内容自然生长，稳定后再绘制地图，才是正确的顺序。

## 文件命名约定

四层架构推荐的文件命名方案：

```
knowledge-archive/
├── 00-{protocol-name}.md      # Layer 1: 规则层
├── 01-{domain-1}.md           # Layer 2: 领域内容
├── 02-{domain-2}.md
├── ...
├── 0N-{domain-N}.md
├── 0{N+1}-{glossary}.md       # Layer 3: 跨领域整合
├── 0{N+2}-{timeline}.md
├── 0{N+3}-{methodology}.md
├── 0{N+4}-{further-reading}.md
├── 0{N+5}-{validation-log}.md
└── README.md                  # Layer 4: 索引层
```

**编号规则**:
- 规则层固定为 `00-` 开头
- 领域内容层从 `01-` 开始顺序编号
- 整合层编号紧接领域内容之后
- README没有编号（作为入口文件特殊处理）

## 实施顺序（关键！）

```
Phase 1: 规则先行
  ↓ 创建 00-{protocol}.md
  ↓ 定义来源标准、可信度规则、审查流程
  ↓ ✅ 规则层评审通过
  ↓
Phase 2: 分领域填充
  ↓ 按领域逐个创建 01..0N 内容文件
  ↓ 搜集内容时即时标记来源可信度
  ↓ 遵守规则层标准，不随意降低要求
  ↓ ✅ 所有领域内容基本完成
  ↓
Phase 3: 跨领域整合
  ↓ 创建术语表，统一跨领域概念
  ↓ 创建时间线，梳理发展脉络
  ↓ 提炼方法论框架
  ↓ 生成分级阅读推荐
  ↓ 完成验证日志记录审查过程
  ↓ ✅ 跨领域一致性检查通过
  ↓
Phase 4: 索引导航
  ↓ 创建README.md档案首页
  ↓ 设计阅读路径
  ↓ 添加快速索引
  ↓ ✅ 全量链接检查通过
  ↓
🎉 完成！
```

**实施反模式（必须避免）**:
- ❌ 先写README，再填充内容（索引过早固化）
- ❌ 边做内容边改规则（规则不稳定导致返工）
- ❌ 没有术语表直接整合（语义漂移导致概念混乱）
- ❌ 领域文件之间大量互相引用（耦合度高难以维护）

## 质量验证检查清单

| 检查项 | 通过标准 |
|---|---|
| 规则层完成度 | 00文件包含来源标准、评分规则、审查流程、偏差清单 |
| 领域单一职责 | 每个领域文件只覆盖一个主题 |
| 术语表覆盖度 | 所有跨领域使用的术语都有统一定义 |
| 链接有效性 | 所有本地链接通过check-links验证 |
| 可信度标记 | 关键信息都有来源级别和可信度标记 |
| 验证日志完整性 | 关键事实的验证过程都有记录 |
| 索引准确性 | README中的链接全部有效，描述与内容一致 |
| 模块独立性 | 新增领域内容不需要修改已有内容文件 |

## 第一性原理对应

本架构本身的设计也遵循第一性原理思维，回归到知识档案的基本要求：

1. **知识质量取决于规则执行** → 规则层独立且先行（质量内建）
2. **不同领域的知识有内在边界** → 领域层高内聚低耦合
3. **跨领域沟通需要统一语言** → 整合层解决语义漂移
4. **地图必须匹配领土** → 索引层最后构建，避免约束内容
