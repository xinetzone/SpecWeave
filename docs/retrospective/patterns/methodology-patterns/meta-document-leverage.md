# 元文档杠杆效应（Meta-Document Leverage Effect）

## 模式类型
方法论模式

## 成熟度
L1 实验性（1 次成功案例：本项目的 README.md + 导航体系）

## 适用场景
项目文档体系设计中，需要决定资源分配——是优先完善功能文档还是优先建设元文档（导航、索引、概览）。

## 问题背景

文档建设的常见误区是按"功能重要性"分配写作资源——核心模块写详细文档，辅助模块写简要文档。但这种策略忽视了文档的**读者路径**：读者首次接触项目时，最先阅读的永远是导航文档（README.md → 目录索引 → 快速开始），而非某个模块的详细功能说明。

好的元文档 = 项目的地图 + 使用说明书 + 推广册。它的战略价值远超功能文档，因为它决定了读者"是否留下来"——读者在判断一个项目是否值得深入之前，首先接触的就是元文档。

## 核心定义

```
元文档 = 不直接描述项目功能，而是描述"如何理解和使用项目"的文档

杠杆效应 = 元文档的质量 × 读者基数 >> 功能文档的质量 × 读者基数
```

| 文档类型 | 读者接触率 | 战略作用 | 示例 |
|---------|----------|---------|------|
| **元文档** | 100%（第一接触点） | 决定读者是否留下 | README.md、目录索引、导航表 |
| **概览文档** | 60-80% | 建立全局理解 | 项目架构概览、知识库索引 |
| **功能文档** | 10-30% | 解决具体问题 | API 文档、模块使用说明 |
| **参考文档** | 5-10% | 深度查阅 | 协议定义、规则细则 |

## 元文档的核心要素

| 要素 | 作用 | 示例（本项目） |
|------|------|---------------|
| **快速开始** | 降低首次体验门槛 | README.md 的安装与使用章节 |
| **导航地图** | 帮读者找到目标信息 | NAV_TABLE 导航表、上下文路由表 |
| **全景视图** | 呈现项目整体结构 | 目录结构图、模块依赖图 |
| **复用入口** | 展示项目可产出价值 | 可复用模式库索引、资产清单 |
| **演进路径** | 呈现项目发展方向 | 模式成熟度等级、后续路线图 |

## 本案例验证

**支撑事实**：README.md 重建后新增的三个章节（可复用模式体系、提示词萃取系统、泛化与资产复用）本质上都是"元文档"——它们不直接描述项目功能，而是描述"如何理解和使用项目"。

**杠杆效果**：项目积累的 70+ 交付物中，读者最先接触的就是 README.md → docs/retrospective/README.md → docs/knowledge/README.md 这一元文档链。这些元文档的质量直接决定了读者"留下来"的意愿。

**量化**：本项目通过元文档呈现了 3 个维度的杠杆：
- **知识杠杆**：29 个方法论模式通过 1 个入口索引页全部可发现
- **资产杠杆**：70+ 交付物通过 asset-inventory.md 可检索
- **路径杠杆**：AGENTS.md 上下文路由表将 30+ 个 .agents/ 模块按任务类型映射

## 建设原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **读者路径优先** | 按读者首次接触的路径分配文档资源 | 核心模块写 5000 字但 README 只有 3 行 |
| **索引优于正文** | 先建立可导航的索引结构，再逐步填充正文 | 正文详细但没有索引导航 |
| **入口层层递进** | README → 目录索引 → 领域索引 → 具体文档 | 所有文档平铺在同一层级 |
| **元信息先行** | 每个文档开头说明"这是什么/谁需要读/读完去哪" | 直接进入正文无导航 |

## 实施检查清单

- [ ] 是否存在 README.md，且能在 3 分钟内让新读者理解项目全貌？
- [ ] 是否存在从 README 到各级目录索引的递进式导航链？
- [ ] 每个主要模块是否有概览页（README.md），说明"谁需要读这个模块"？
- [ ] 是否存在全局资产清单，让读者能发现项目的全部产出？
- [ ] 是否存在"下一步阅读"的引导链接？
