# 精度优先于召回率（Precision Over Recall）—— 破坏性工具的零误报原则

## 模式类型
方法论模式（设计哲学）

## 成熟度
L1 实验性（1次案例：check-links --fix 零误报验证）

## 适用场景
设计任何会**修改用户文件/数据**的自动化工具（自动修复、批量替换、代码重构、数据库迁移、配置更新等）。

## 问题背景

自动修复/修改类工具面临经典权衡：
- **高召回率**：尽可能多地修复问题 → 风险：可能误改正确内容
- **高精度**：只修复非常确定的问题 → 风险：可能漏掉一些可修复问题

很多工具设计者倾向于"尽量多修"，结果导致误改用户数据，严重损害用户信任。

## 核心原则

> **对于会修改用户文件的自动化工具，"做错"比"不做"代价大10倍。用户可以接受"有些问题需要手动修"，但绝不能接受"工具把我正确的东西改坏了"。这是信任的基础。**

即**精度优先到极致**，追求零误报率。

## 三层安全保障

| 层级 | 机制 | 对应模式 |
|------|------|---------|
| **原则层（本模式）** | 精度优先设计哲学：宁可不修，不可错修 | precision-over-recall |
| **流程层** | dry-run预览→用户确认→执行写入→立即验证 | dry-run-first |
| **代码层** | 修复策略按精确度排序，高优先级零误报 | fix-priority-chain（代码模式） |

## 关键设计决策（以check-links --fix为例）

| 设计决策 | 选择 | 理由 |
|---------|------|------|
| 修复策略排序 | 精确路径校正→根路径匹配→文件名映射→模糊搜索 | 精确度从高到低排列 |
| 深度调整范围 | ±3级，不是±5或±10 | 范围过大会导致误匹配 |
| 验证条件 | 必须目标文件真实存在才建议修复 | 不做猜测 |
| 目录引用处理 | 自动尝试README.md但要求文件存在 | 不假设目录结构 |
| 失败处理 | 明确报告"无法自动修复"，交给人工 | 不强猜 |
| 执行前 | 必须dry-run预览，用户确认后才写入 | dry-run-first |

## 零误报验证方法

在所有内容都正确的基准状态下运行工具，验证输出"无需修改"：
```bash
# 先确保零断链
python check-links.py
# 输出：XXX links OK, 0 broken

# 再运行--fix --dry-run，验证零误修改
python check-links.py --fix --dry-run
# 预期输出：未发现需要修复的断链（而不是"修复"了正确的链接）
```

## 精度-召回权衡矩阵

| 场景 | 推荐策略 | 理由 |
|------|---------|------|
| 修改用户文件/数据 | 精度优先（零误报） | 误改代价极高，破坏信任 |
| 只读检测/报告 | 可平衡偏召回 | 检测多一些不造成损害，用户可筛选 |
| 代码补全/建议 | 召回优先 | 建议不自动执行，用户有最终决定权 |
| 数据查询/搜索 | 召回优先 | 漏检比多检更影响体验 |
| CI门禁阻断 | 精度优先 | 误阻断会阻塞开发流程 |

## 检查清单

- [ ] 工具是否会修改用户文件？如果是，必须采用精度优先设计
- [ ] 高优先级修复策略的误报率是否接近零？
- [ ] 不确定时是否选择"不修复+报告"而非"猜测修复"？
- [ ] 是否有dry-run模式让用户在执行前预览所有变更？
- [ ] 是否在零问题基准状态下验证过零误报？
- [ ] 模糊/低精确度策略是否明确标记"需人工确认"？

## 反例警示

| 陷阱 | 后果 |
|------|------|
| 追求"修复率"数字好看 | 误改正确内容，用户信任崩塌 |
| 模糊匹配默认自动执行 | 正确链接被"修复"成错误路径 |
| 没有dry-run直接写入 | 用户无法预览变更，错误直接落盘 |
| 不验证目标存在就替换 | 产生新的断链/错误 |
| 失败时静默跳过 | 用户不知道哪些没修，以为全修好了 |

## 与现有模式的关系

- `dry-run-first.md`：流程层安全保障——预览→确认→执行→验证
- `fix-priority-chain.md`：代码层实现——策略按精确度从高到低排序
- `three-level-problem-solving.md`：精度优先是L3系统免疫的基础——建立信任
