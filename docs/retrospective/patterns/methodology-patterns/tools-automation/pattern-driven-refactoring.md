# 模式驱动重构（Pattern-Driven Refactoring）

## 模式类型

方法论模式（工具自动化/工程实践）

## 成熟度

L1 首次提炼（MDI项目单日完成14个大文件拆分、132个文件变更、零回归验证）

## 适用场景

大规模代码重构、批量文件拆分、跨模块结构优化；当需要在短时间内完成多个相似模块的结构调整时。

## 问题背景

大规模重构的常见陷阱：
- 每个文件从零开始设计拆分方案，效率低下
- 不同开发者的拆分风格不一致，增加后续维护成本
- 后期拆分的文件可能出现与前期不一致的结构问题，需要返工
- "边拆边想"模式容易引入Bug，因为缺乏统一的设计参考

## 核心思想

**投入20%时间在1-2个试点文件上摸索可复用模式，再以80%时间批量推广——将重构从"创造性工作"转化为"流水线执行"。**

## MDI项目量化验证

| 阶段 | 文件数 | 平均耗时 | 说明 |
|------|--------|---------|------|
| 试点摸索阶段 | 前3个文件 | ~30分钟/文件 | 摸索目录结构、依赖方向、垫片策略、测试验证流程 |
| 批量推广阶段 | 后续11个文件 | ~10分钟/文件 | 复制统一模式，仅调整具体业务逻辑，效率提升3倍 |
| 总计 | 14个文件 | ~200分钟 | 单日完成，159+测试全部通过，零回归Bug |

## 核心规则

### 规则1：试点先行，模式沉淀

正式开始批量重构前，选择1-2个有代表性的文件作为试点：
- 完整走通拆分→测试→验证全流程
- 记录遇到的问题和解决方案
- 提炼出可复用的"拆分模板"：目录结构、文件命名、依赖方向、垫片写法

### 规则2：统一模式五要素

每个可复用的重构模式必须明确五个要素：
1. **目录结构模板**：新包的子目录/文件组织方式
2. **依赖方向规则**：模块间的单向依赖链（如 cli → checks → models）
3. **薄入口垫片**：原文件保留为<30行re-export层，确保import路径不变
4. **验证检查清单**：拆分后必须通过的测试/校验步骤
5. **常见问题处理**：试点中遇到的坑及解决方案

### 规则3：风险递进顺序

批量执行时按风险从低到高排序：
- 先拆分独立工具模块（无外部依赖），验证流程
- 再拆分核心业务模块（有内部依赖）
- 最后拆分入口/CLI模块（依赖最多，影响最大）
- 每完成一个文件立即运行测试，发现问题立即回滚

### 规则4：模式随实践演进

- 批量执行中发现更好的做法时，更新模板
- 遇到新模式问题时，先做小型试点再推广
- 完成后将最终模式沉淀为可复用文档

- [ ] 最终模式是否沉淀为可复用文档/模板？

## 反例警示

| 错误做法 | 后果 |
|---------|------|
| 直接开干，不做试点 | 第一个文件踩的坑在所有文件重复踩，返工成本高 |
| 每个文件自定义拆分方式 | 风格不一致，后期维护成本高，容易引入Bug |
| 从最复杂的核心模块开始 | 初期问题多打击信心，且核心模块出问题影响范围大 |
| 拆分完不立即验证 | 问题累积到最后才发现，定位困难 |

## 正例：MDI项目验证

MDI项目单日原子化拆分战役：
- 试点文件：versioning.py（872行）和 validator.py（639行）
- 提炼模式：CLI→lib/{name}/包结构、薄入口垫片、cli→checks→models单向依赖、测试四步验证
- 批量推广：11个文件按统一模式拆分，平均耗时从30分钟降至10分钟
- 结果：14个文件拆分、132个文件变更、🟠高风险区14→0清零、159+测试零回归
