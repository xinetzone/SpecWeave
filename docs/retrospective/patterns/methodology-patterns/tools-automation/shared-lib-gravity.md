# 共享库引力定律：覆盖面越大复用率越高的正反馈循环

## 模式概述

多脚本项目中的共享库（lib/）存在"引力效应"：共享库覆盖的概念域越多，开发者"先查共享库"的预期收益越高，复用率随之提升，进而推动更多功能被提取到共享库，形成自我强化的正反馈循环。反之，共享库覆盖面小时，"自己写一个"的边际成本更低，重复代码持续积累（负反馈）。

本模式描述这一规律，提供共享库临界点识别方法和推广策略，指导多脚本项目有效推进代码复用。

## 问题现象

项目初期脚本数量少（<3个），每个脚本独立实现工具函数（路径解析、YAML读取、文件遍历等），此时：
1. **看不出重复**：代码量小，重复逻辑不明显
2. **复用收益低**：花时间提取共享函数不如直接写
3. **认知门槛**：新开发者不知道有共享库，直接复制粘贴

随着脚本数量增加（3→5→10个），问题逐渐显现：
- 相同功能（如`resolve_project_root()`）在5个脚本中重复实现
- 修复一个bug需要修改5个地方
- 新增脚本时复制旧脚本的工具函数，技术债务持续累积
- 每个脚本都有细微差异，难以统一维护

## 核心规律

### 引力效应公式

```
共享库覆盖域 ↑ → "先查"预期收益 ↑ → 复用率 ↑ → 新函数提取↑ → 覆盖域进一步↑
```

这是一个正反馈循环——共享库越大、越完善，开发者越倾向于复用而非重复实现。

### 临界点：5个概念域

当共享库覆盖**≥5个独立概念域**时，引力效应开始显著：

| 覆盖域数量 | 开发者行为 | 复用率 |
|-----------|-----------|--------|
| 1-2个 | "功能太少，不如自己写" | <20% |
| 3-4个 | "偶尔查一下，大部分还是自己写" | 20-50% |
| **≥5个** | **"先查lib有没有，没有再自己写"** | **>60%** |

本项目的临界点验证：
- `lib/`从5个模块扩展到6个模块时（新增`rules.py`），引力效应显著增强
- 后续新增`fix-x-toml-ref.py`、`check-frontmatter.py`时，开发者主动查阅lib复用了`cli.py`、`project.py`、`frontmatter.py`等约10个函数
- 新增代码约600行，复用共享库函数显著降低了重复代码

### 反引力（负反馈）机制

共享库覆盖面小时，存在反向循环：
```
共享库覆盖域小 → "自己写"成本更低 → 重复实现 → 无人提取新函数 → 覆盖域维持小
```

打破负循环需要**主动投入**提取共享函数，直到覆盖域达到临界点。

## 解决方案

### 三阶段推广策略

```mermaid

    P1["阶段一：种子期<br/>(1-2个模块)<br/>强制提取基础功能"]
    P2["阶段二：临界点<br/>(3-4个模块)<br/>主动推广+文档化"]
    P3["阶段三：自增长<br/>(≥5个模块)<br/>引力效应自发推动"]
    P1 -->|"强制提取project.py/cli.py等基础设施"| P2
    P2 -->|"API文档+\"先查lib\"约定+PR检查"| P3
    P3 -->|"开发者主动复用，正反馈循环"| P3
    style P1 fill:#f8d7da,stroke:#dc3545
```

### 阶段一：种子期（强制提取）

**目标**：建立共享库基础，覆盖最核心的概念域。

**必选概念域**（先提取这5个）：
1. **project.py**：项目根目录解析、路径常量
2. **cli.py**：命令行参数解析、输出格式化、错误处理
3. **frontmatter.py**：YAML/TOML frontmatter解析、字段提取
4. **files.py**：Markdown文件发现、遍历、读写工具
5. **links.py**：链接解析、相对路径计算

**策略**：
- 不要等"需要的时候再提取"——写第二个脚本时就开始提取共享函数
- 每个新脚本开发时，发现重复立即提取，不要拖延
- 强制代码评审检查：新增脚本中如有10行以上与现有脚本重复，必须提取

### 阶段二：临界点（主动推广）

**目标**：推动覆盖域达到5个，触发引力效应。

**关键动作**：
1. **API文档**：为共享库生成API文档（如`lib/README.md`），列出所有可用函数及用法示例，降低发现成本
2. **约定写进规范**：在开发规范中明确写"新增脚本前先查lib/"，将建议升级为约定
3. **"先查"检查清单**：新脚本开发Checklist中加入"是否复用了lib/中的已有函数"
4. **定期重构**：每完成1-2个脚本后，花时间提取新发现的共享功能
5. **PR检查**：代码评审时主动询问"这个功能lib/里有没有？"

### 阶段三：自增长（引力效应）

**目标**：正反馈循环自发运转，无需强制推动。

**特征**：
- 新开发者加入项目时，会主动查阅lib/再开始写代码
- 发现lib/中缺少的功能时，会主动提交PR补充，而不是在自己脚本里写
- 共享库函数持续增加，覆盖面越来越广
- 重复代码率维持在低位

**维护要点**：
- 持续更新API文档
- 防止共享库变成"垃圾场"——定期重构，移除不再使用的函数
- 对新增共享函数做代码评审，保证质量
- 概念域分离清晰，避免单个文件过大（参考concept-domain-separation模式）

### lib/目录结构参考

```
.agents/scripts/lib/
├── README.md          # API文档，列出所有可用函数
├── __init__.py
├── project.py         # 项目根解析、路径常量（概念域1）
├── cli.py             # 命令行工具、输出格式化（概念域2）
├── frontmatter.py     # YAML/TOML解析（概念域3）
├── files.py           # 文件遍历、读写（概念域4）
├── links.py           # 链接解析、路径计算（概念域5）
├── rules.py           # 规则引擎、误报过滤（概念域6+）
└── checks/            # 复杂检查逻辑子目录
    ├── __init__.py
    └── ...
```

## 实际案例

### 案例1：SpecWeave脚本共享库演进

**时间线**：
1. **种子期**（5个模块）：project.py、cli.py、frontmatter.py、files.py、links.py——覆盖5个基础概念域
2. **临界点触发**：scripts-shared-lib-extraction复盘后，新增`rules.py`和`checks/`目录，达到6个模块
3. **引力效应验证**：
   - `docgen.py`重构后使用新的`parse_toml_frontmatter_as_dict`便捷函数，从3行简化为1行
   - `check-links.py`、`check-source-traceability.py`改用共享的`find_markdown_files`
   - 后续新增`check-duplication.py`时开发者主动查阅lib/复用了cli.py和project.py
   - Frontmatter迁移中新增的`fix-x-toml-ref.py`、`check-frontmatter.py`复用共享库约10个函数，新增600行代码重复率极低

### 案例2：反例——共享库过小导致的重复

某项目只有2个共享模块（project.py和utils.py），每个新脚本都自己实现frontmatter解析和文件遍历逻辑：
- 8个脚本中有4个独立实现了YAML解析
- 修复frontmatter解析bug时需要修改4个地方
- 开发者从来不会"先查lib"——因为里面确实没什么可用的

**修正**：提取frontmatter.py、files.py、cli.py三个模块，覆盖域达到5个后，新脚本开始主动复用。

## 反模式

### 反模式1：等"足够多重复"再提取

```
等有3个脚本用了相同功能，我再提取到共享库。
```

**问题**：
- 到3个脚本重复时，已经有3份代码需要修改
- 开发者已经形成了"复制粘贴"的习惯
- 技术债务累积，重构成本越来越高

**正确做法**：第二个脚本用到相同功能时就提取。

### 反模式2：共享库没有文档

lib/目录下有很多.py文件，但没有README说明每个函数干什么、怎么用。

**问题**：
- 开发者不知道有哪些函数可用，"先查lib"的成本高于自己写
- 即使有函数也不敢用——不知道参数和返回值是什么
- 共享库形同虚设

**正确做法**：维护lib/README.md，为每个公共函数写简要说明和示例。

### 反模式3：把所有东西都塞进一个utils.py

```python
# utils.py - 1000行，什么都有
def resolve_project_root(): ...
def parse_yaml(): ...
def print_success(): ...
def calculate_hash(): ...
def send_email(): ...  # 什么鬼？
```

**问题**：
- 概念域不分离，文件越来越大，难以维护
- 函数之间没有逻辑关联，难以发现和复用
- 违反单一职责，修改一个功能可能影响不相关的功能

**正确做法**：按概念域拆分文件（project.py、cli.py、frontmatter.py...）。

## 与其他模式的关系

| 相关模式 | 关系 | 说明 |
|---------|------|------|
| large-scale-duplication-elimination | 前置依赖 | 重复消除五步法是提取共享库的系统方法 |
| concept-domain-separation | 设计指导 | 共享库模块必须按概念域清晰分离 |
| tool-self-validation | 配套 | 共享函数必须有自生验证能力，保证质量 |
| gitignore-validation | 经验同源 | 新脚本引入产出物要同步.gitignore，新脚本也要"先查共享库" |
| diff-driven-refactoring | 触发时机 | 代码diff是发现可提取共享功能的最佳时机 |

## 检查清单

开发新脚本前：
- [ ] 是否先浏览了`lib/README.md`了解可用函数？
- [ ] 需要的功能是否已经在lib/中实现？
- [ ] 如果没有，是否应该在开发完后提取到lib/？

共享库维护：
- [ ] 覆盖域是否达到了5个概念域？
- [ ] lib/README.md是否最新？
- [ ] 模块是否按概念域分离，没有大杂烩utils.py？
- [ ] 新提取的函数是否有文档和示例？
- [ ] 代码评审是否检查了复用情况？
