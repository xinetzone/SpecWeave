# 能力清单/功能矩阵（Capability Matrix）

## 模式类型
方法论模式

## 成熟度
L1 实验性（1次案例：check-links.py 修复类型清单，但尚未在多个工具文档中验证）

## 适用场景
开发工具、脚本、CLI 程序时，为工具编写用户文档或设计功能边界声明。

## 问题背景

工具的能力边界如果不显式声明，会导致三类问题：
1. **用户视角**：不知道工具能做什么、不能做什么、精确度如何，只能试错
2. **维护者视角**：新增功能时没有清晰的检查清单，容易遗漏能力声明
3. **规划视角**：看不出哪些能力缺失、哪些精确度可以提升，backlog不清晰

## 核心设计

为工具维护一张**功能矩阵表**，显式声明每类操作的能力边界和精确度：

| 能力类型 | 精确度 | 说明 |
|---------|--------|------|
| 能力A | 高 | 精确描述A能做什么，准确率多少 |
| 能力B | 高 | ... |
| 能力C | 中 | 描述C的模糊匹配边界，需要人工确认 |
| 能力D | 不支持 | 显式声明不支持，避免用户误用 |

## 三重价值

1. **对用户**：清晰知道工具能做什么、不能做什么、精确度如何——看到"中"精确度就知道需要人工确认，不用试错
2. **对维护者**：新增能力时必须加一行，强迫思考类型、精确度、优先级位置——是集成检查清单
3. **对未来扩展**：这张表本身就是缺口分析工具——一眼能看出哪些没覆盖、哪些精确度可以提升，是天然的backlog

## 精确度分级标准

| 精确度 | 含义 | 用户需要做什么 |
|--------|------|--------------|
| **高** | 工具自动处理，零误报，结果可直接使用 | 无需人工确认 |
| **中** | 工具能识别但可能有歧义，需要人工检查 | 预览结果后确认 |
| **低** | 工具提供候选建议，最终决策由人工完成 | 人工选择或修改 |
| **不支持** | 工具明确不处理该场景 | 需手动操作或等待功能开发 |

## 应用实例：check-links.py 修复能力矩阵

| 修复类型 | 精确度 | 说明 |
|---------|--------|------|
| `file:///` 绝对路径转相对路径 | 高 | 基于文件位置精确计算相对路径 |
| 相对路径层级校正（../深度） | 高 | 基于目录深度差精确调整 |
| 目录链接尾部斜杠补全 | 高 | 目录链接必须以/结尾的规则 |
| 顶级目录根路径识别 | 高 | docs/.agents/等顶级目录精确识别 |
| 文件名重命名映射 | 高 | 基于--rename参数精确替换 |
| 文件名模糊搜索 | 中 | 按文件名相似度搜索，需人工确认 |

## 检查清单

- [ ] 是否列出了工具支持的所有能力类型？
- [ ] 每种能力是否标注了精确度（高/中/低/不支持）？
- [ ] 是否显式声明了"不支持"的能力，避免用户误用？
- [ ] 中/低精确度的能力是否说明了用户需要做什么（人工确认/选择）？
- [ ] 新增能力时是否同步更新矩阵？

## 反例警示

| 问题 | 后果 |
|------|------|
| 不声明能力边界 | 用户期望过高，遇到"不支持"场景时认为是bug |
| 不标注精确度 | 用户盲目信任自动修复结果，错误修复被直接提交 |
| 只有"支持"没有"不支持" | 用户在不支持的场景反复尝试，浪费时间 |
| 矩阵不随功能更新 | 文档与实际能力脱节，用户按旧文档操作出错 |

## 与现有模式的关系

- `actionable-suggestion-five-elements.md`：能力矩阵是工具文档"可执行性"的基础——用户知道工具能做什么才能正确使用
- `tool-automation-decision-model.md`：决策模型判断是否值得自动化，本模式描述自动化后的能力声明方式
- `prove-usefulness-check.md`：新功能证明有用后，应同步更新能力矩阵
