# 模块大小-Bug密度非线性相关模式（Module Size-Bug Density Correlation）

## 模式类型

方法论模式（治理策略）

## 成熟度

L1 首次提炼（MDI项目8970行代码/10个Bug量化验证）

## 适用场景

代码审查、架构重构决策、模块拆分规划、技术债务识别；当你在犹豫"这个文件是不是太大了要不要拆"时作为量化决策依据。

## 问题背景

开发过程中容易对"单文件多大算大"产生主观判断：
- "这个文件虽然大，但逻辑是内聚的，没必要拆"
- "拆文件还要改import，增加复杂度，反而不好"
- "我能hold住几千行的文件，没问题"

这些判断的问题在于：**只考虑了"拆分的显性成本"，没考虑"大文件的隐性Bug成本"——而Bug密度与模块大小呈非线性正相关。**

## 核心数据：MDI项目量化验证

| 模块行数区间 | 模块数 | 总Bug数 | 平均Bug数/模块 | Bug密度（Bug/千行） |
|-------------|--------|---------|---------------|-------------------|
| 800-1500行 | 2个（parser.py 1465行/versioning.py 872行） | 6个 | 3.0个/模块 | 2.6个/千行 |
| 500-700行 | 4个 | 1个 | 0.25个/模块 | 0.4个/千行 |
| <500行 | 22个 | 3个 | 0.14个/模块 | 0.7个/千行 |

**关键发现**：
1. >800行的模块Bug密度是<500行模块的**3.7-6.5倍**
2. 2个最大文件（占总代码26%）贡献了**60%的Bug**
3. 超过1000行后，Bug密度呈指数级上升而非线性

## 核心规则

### 规则1：设置三级文件大小警戒线

| 警戒线 | 行数阈值 | 应对策略 |
|-------|---------|---------|
| 🟢 绿色安全区 | <500行 | 正常开发，无需担心 |
| 🟡 黄色预警区 | 500-800行 | 规划拆分方案，下次重构时优先处理 |
| 🟠 橙色高风险区 | 800-1200行 | 立即制定拆分计划，新增功能优先放到新文件 |
| 🔴 红色警报区 | >1200行 | "上帝类/上帝文件"反模式，停止在此文件新增代码，优先拆分 |

### 规则2：按职责拆分而非按行数拆分

拆分不是机械地按行数切分，而是按单一职责原则：
- Parser类大文件 → tokenizer（词法分析）/ section-builder（语法树构建）/ directive-parser（特定语法解析）
- 大Service类 → 按业务域/按功能职责拆分
- 大工具类 → 按输入/处理/输出阶段拆分
- 大配置类 → 按环境/按模块拆分

### 规则3：Bug密度比绝对行数更重要

如果一个文件虽然只有600行，但：
- 修改时频繁引入新Bug
- 代码审查时很难一次看完
- 新成员接手需要很长时间理解
- 单元测试很难写（需要太多mock）

即使没到800行，也应该考虑拆分——**行数是proxy指标，认知复杂度才是本质**。

## 决策速查表

当你在犹豫"要不要拆分这个文件"时：

| 检查项 | 如果是 | 如果否 |
|-------|-------|-------|
| 文件行数>800行？ | 🔴 立即拆分 | 继续看其他项 |
| 过去3个月这个文件有多少个Bug？ | ≥3个 → 拆分优先级高 | ≤1个 → 可暂缓 |
| 新增功能时经常需要改这个文件？ | 是 → 拆分减少冲突 | 否 → 可暂缓 |
| 代码审查时需要分多次才能看完？ | 是 → 超过认知负荷，拆分 | 否 → 可以接受 |
| 单元测试覆盖率低于项目平均？ | 是 → 可测试性差，拆分提升可测试性 | 否 → 可暂缓 |

## 实施检查清单

- [ ] 是否设置了项目级别的文件大小警戒线（推荐500/800/1200三级）？
- [ ] 是否在CI/代码审查中检查文件大小超限？
- [ ] 拆分时是否遵循单一职责原则，而非机械切分？
- [ ] 拆分后是否同步更新了import和单元测试？
- [ ] 是否优先拆分Bug密度高的"红色警报"文件？

## 反例警示

| 错误想法 | 后果 |
|---------|------|
| "这个文件逻辑内聚，大一点没关系" | 逻辑内聚≠认知内聚，超过1000行后人脑无法一次性hold住所有上下文 |
| "拆分文件会增加import复杂度" | 一次性的导入调整成本 vs 长期的高Bug率成本，ROI是1:10以上 |
| "我写代码很小心，不会出Bug" | 这是过度自信偏差——大文件的Bug率高是结构性问题，与个人细心程度无关 |
| "等以后有时间再重构" | "以后"永远不会来，而且越晚拆成本越高（代码继续往里面加） |

## 正例：MDI项目验证

MDI项目实际数据：
- `parser.py` 1465行，3个Bug（Bug#1/#9/#10）—— 红色警报区，建议拆分为tokenizer.py/section_builder.py/directive_parser.py
- `versioning.py` 872行，3个Bug（Bug#5/#6/#7）—— 橙色高风险区，建议拆分为diff_engine.py/semver_rules.py/impact_analyzer.py
- 其余24个文件平均275行，4个Bug —— 绿色安全区，质量良好

**反事实推理**：如果在开发到700行时就拆分parser.py，虽然会花约30分钟做拆分，但可能避免后续2-3个Bug（每个Bug平均排查+修复15-30分钟），净收益为正。

## 与现有模式的关系

- `nonlinear-correction-cost.md`：本模式是非线性纠偏成本在代码结构领域的具体体现——大文件的维护/Bug修复成本是非线性增长的
- `bottleneck-first-refactoring.md`：识别出大文件（尤其是>1000行的红色警报区）就是找到重构瓶颈点
- `root-cause-diagnosis.md`：当某个模块Bug频繁出现时，"文件太大认知负荷过载"是常见根因之一，需要用5-Whys追问到这一层
