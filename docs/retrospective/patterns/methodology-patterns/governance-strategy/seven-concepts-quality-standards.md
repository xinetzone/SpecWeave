# 七概念方法论质量标准与检查清单

## 一、质量检查清单（33项，可判定yes/no）

### A. 单概念使用质量（19项）

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| R-1 | 复盘事实无判断 | 事实章节"因为/所以/导致/原因是"出现次数=0 |
| R-2 | 事实可验证 | 可验证事件≥3条（P2/P3小Bug修复≥2条），每条含timestamp+actor+action |
| R-3 | 反事实推演完整 | "如果X则Y"格式推演≥2条（小任务≥1条） |
| I-1 | 洞察四元组完整 | 每条含「条件C→机制M→行动A→结果B」 |
| I-2 | 洞察可证伪 | 有明确证伪条件或反例场景 |
| I-3 | 洞察可迁移 | 每条附≥1个跨情境迁移场景 |
| E-1 | 萃取层级正确 | 标注L1洞察/L2模式/L3原则，与内容匹配 |
| E-2 | 模式有案例支撑 | L2+模式有≥2个独立案例来源 |
| E-3 | 模式配反模式 | L2+模式配套≥1个反模式案例 |
| C-1 | 提交职责单一 | 无跨模块不相关变更混在同一提交 |
| C-2 | 提交可独立回滚 | revert该提交不破坏其他功能 |
| C-3 | 提交信息规范 | 符合Conventional Commits（type(scope): subject） |
| A-1 | 原子化后粒度合适 | 拆分后单文件≤500行 |
| A-2 | 链接完整性 | 拆分/合并后无断链，无file:///绝对路径 |
| F-1 | 第一性原理假设显式 | 所有隐含假设已列出并标记hidden=true |
| F-2 | 公理自洽 | 公理体系无逻辑矛盾，可经V验证 |
| V-1 | 对抗审查证伪导向 | 以构造反例为主，而非证明方案正确；规则/匹配/推荐类功能必须覆盖"不该匹配的别乱匹配"场景 |
| V-2 | 反例数量充分 | 高影响场景≥3个具体反例（覆盖无关/边界/混合三类）；中影响场景≥2个；简单重构/纯函数≥1个反例或等价性验证；每个反例必须有明确预期结果；修复后重跑正向测试防回归 |
| V-3 | 审查可追溯 | 验证过程记录完整，可复现可审计；反例类型参考[adversarial-review-prompt-pattern.md](../ai-collaboration/adversarial-review-prompt-pattern.md)四类标准模板 |

### B. 概念组合质量（4项）

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| B-1 | 组合选择正确 | 按触发决策树选择概念，无过度/不足使用 |
| B-2 | 执行顺序合规 | 符合DAG依赖，无反向越级调用 |
| B-3 | 接口契约满足 | 每步前置条件检查通过后才执行 |
| B-4 | 质量门全通过 | 流程中3个质量门100%通过 |

### C. 产出物质量（5项）

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| C-O1 | frontmatter完整 | id/title/source/x-toml-ref四字段齐全 |
| C-O2 | 路径引用正确 | 相对路径层级正确，无断链 |
| C-O3 | 元数据完整 | 知识资产含maturity/validation_count/reuse_count |
| C-O4 | DRY原则 | 洞察/模式通过链接引用，不重复嵌入正文 |
| C-O5 | 职责单一 | 单文件≤500行，单一主题 |

### D. 流程执行质量（5项）

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| D-1 | 前置条件验证 | 每步Pre检查通过后才进入下一步 |
| D-2 | RACI明确 | 角色职责清晰，无越权操作 |
| D-3 | 回退机制执行 | 验证失败时回退上一步，而非跳过 |
| D-4 | 冲突仲裁合规 | 按CR1-CR5规则解决概念间冲突 |
| D-5 | 调用路径合法 | 无7×7矩阵中的红格（禁止直连）调用 |

## 二、反模式库（8种，来自项目实际教训）

| ID | 反模式名称 | 表现形式 | 危害 | 正确做法 |
|----|-----------|---------|------|---------|
| AP1 | 超大批量提交 | 单次提交含>100文件/跨模块无关变更（如vendor引入1693文件混提交） | review困难、无法精确回滚、问题定位慢 | 按单一职责拆分，原子化粒度与revert意愿对齐 |
| AP2 | 事实阶段下结论 | 复盘S1直接出现因果判断，选择性忽略矛盾证据 | 确认偏误未被防御，因果关系失真 | S1强制"无判断"约束，禁用因果词，只列可验证事实 |
| AP3 | 原子化拆分过细 | 每条洞察/每个小模块独立文件，深层目录嵌套 | 导航成本激增，认知负荷未降反升 | U型曲线寻优，在认知负荷与导航成本间取平衡 |
| AP4 | 缺乏对抗验证 | 知识图谱/方案交付前未做证伪检查，大量孤立节点 | 确认偏误盲区未暴露，交付后才发现缺陷 | V作为横切层，对认知层所有输出做对抗审查 |
| AP5 | 隐含假设未验证 | 直接假设"标准库正确""API行为符合预期"（如tomllib中文Bug） | 边界场景触发隐性Bug | F显式列出所有假设（含hidden），V逐个构造反例验证 |
| AP6 | 经验未抽象萃取 | 复盘只有"学了什么/做了什么"，无"为何重要/如何复用" | 经验无法迁移，重复踩坑 | 走四层漏斗：事件→洞察三元组→模式→原则 |
| AP7 | 报告冗余违反DRY | 洞察/模式重复嵌入复盘正文，而非链接归档 | 信息冗余、维护困难、更新不一致 | 洞察萃取后独立归档为L1模式，复盘只保留链接引用 |
| AP8 | 洞察金句化 | 洞察只有感悟/金句，无三元组结构、无证伪条件 | 看似深刻实则无法复用、无法验证 | 洞察必须形式化为「C→M→A→B」四元组，附迁移场景 |
| AP9 | 正向测试通过就上线 | 规则/匹配/推荐类功能只测"该匹配的都匹配"，不测"不该匹配的别乱匹配" | 边界case在用户现场触发，泛化过度导致误判，工具可信度下降 | 规则类功能必须做V反例测试，构造≥3个"应该不匹配"的反例，修复后重跑正向测试防回归 |

## 三、L1-L4方法论成熟度评估标准

### L1 初始级（Initial）——偶尔使用、无流程

可观测特征：
1. 凭个人经验随机使用部分概念，无统一触发标准
2. 单概念使用要素不全（如复盘无反事实推演、洞察无三元组）
3. 无质量检查机制，产出质量高度依赖执行人
4. AP1-AP8反模式重复出现，无识别和防御意识

### L2 可重复级（Repeatable）——有固定流程、单概念使用正确

可观测特征：
1. 有明确的触发决策树，能按场景选择正确概念组合
2. 单概念4要素基本齐全，C/A/R等执行层概念常规化
3. 有基础格式检查（frontmatter、链接、提交信息）
4. 反模式库建立，常见AP1-AP3可被识别和规避
5. 原子提交覆盖率>80%（按提交数统计）

### L3 已定义级（Defined）——多概念协同、有质量门

可观测特征：
1. 五种核心流程（里程碑/问题解决/重构/知识沉淀/创新）常规化执行
2. 概念组合正确，7×7交互矩阵无红格违规调用
3. 流程质量门作为CI门禁强制执行，不通过不提交
4. 知识资产入库有审核，maturity/validation_count元数据完整
5. 对抗性审查（V）应用于所有L2+模式入库前检查
6. AP4-AP6反模式可被主动检测和防御

### L4 优化级（Optimizing）——自举验证、持续改进、对抗审查常规化

可观测特征：
1. 元复盘常态化（每5个复盘1次元复盘），方法论自身持续演进
2. 对抗性审查嵌入每一步（元审查），V作为默认环节而非可选
3. 自举验证：方法论体系用七概念自身验证和改进
4. 成熟度量化驱动：validation_count/reuse_count决定模式升级，CI自动检测孤立节点、断链、超大提交
5. AP7-AP8反模式基本绝迹，DRY原则和洞察形式化成为肌肉记忆
6. 新问题优先检索现有模式库，模式复用率>60%
