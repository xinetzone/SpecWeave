# 两阶段开发模式（Two-Phase Development: Validate First, Optimize Later）

## 模式类型

方法论模式（治理策略/项目管理）

## 成熟度

L1 首次提炼（MDI项目阶段一4天功能验证+阶段二1天结构优化验证）

## 适用场景

创新型项目、探索性项目、需求不明确的项目、AI辅助快速原型开发；当你不确定"做的东西对不对"时。

## 问题背景

项目启动阶段的两难选择：
- **选项A：初始即追求完美架构**——在需求验证前投入大量时间做架构设计，可能设计出不符合实际需求的"完美架构"
- **选项B：快速堆代码跑通再说**——MVP跑通后代码结构混乱，技术债务高到无法重构，最终原型变屎山

两种选择都有问题：前者可能浪费时间在错误方向上做过度设计，后者可能做出无法维护的原型最终推倒重来。

## 核心思想

**分两阶段走：第一阶段接受"不完美"快速验证核心价值，第二阶段在功能验证完成、测试保障充分、模式指导明确的前提下完成结构优化。比"初始即追求完美架构"更快验证价值、重构质量更高、心理负担更小。**

| 阶段 | 目标 | 可接受的"不完美" | 门禁条件 |
|------|------|-----------------|---------|
| **阶段一：价值验证** | 快速跑通核心功能，验证"这件事值得做" | 单文件较大、部分代码重复、测试覆盖可略低、功能不完整 | 核心价值被验证（用户认可/案例跑通/关键假设成立） |
| **阶段二：结构优化** | 重构到高质量水平，为长期迭代奠基 | 不接受结构性技术债务 | 测试充分、重构模式明确、模块边界清晰 |

## MDI项目量化验证

| 指标 | 如果初始即追求完美架构（预估） | 两阶段模式（实际） |
|------|---------------------------|------------------|
| 产出可用原型时间 | 2周+ | 4天（阶段一） |
| 架构设计时间 | 3-5天 upfront | 0天 upfront，1天重构（阶段二） |
| 总工时 | 2-3周 | 5天（4+1） |
| 架构符合实际需求 | 低（upfront设计必然偏差） | 高（基于真实代码优化） |
| 重构后代码质量 | - | 🟠高风险区清零，所有模块<500行，159+测试 |

**关键数据**：阶段二重构仅用1天时间，远少于"一开始就做对架构"所需的upfront设计时间，且最终架构质量更高——因为它是基于真实代码演化出来的，而非白板上推测出来的。

## 核心规则

### 规则1：阶段一纪律——明确什么是"暂时可以接受的"

阶段一不是"无底线乱堆代码"，而是有明确边界的快速验证：

✅ **可以接受的**：
- 单文件略大（但建议不超过1000行，避免后期根本拆不动）
- 部分模块重复代码（后期可提取公共函数）
- 测试覆盖核心路径而非全路径
- 非核心功能标记为TODO/STUB

❌ **绝对不能接受的**：
- 没有任何测试（阶段二重构没有安全网）
- 核心逻辑不可测试（硬编码依赖/全局状态）
- 不做任何提交记录（无法回溯）
- 不标记哪些是技术债务（后期遗忘）

### 规则2：阶段一门禁——三个必要条件

只有同时满足三个条件才能进入阶段二：
1. **价值验证通过**：核心功能可用，关键假设被验证（有用户/有案例/有数据）
2. **测试安全网就绪**：核心路径有测试覆盖，重构可以验证正确性
3. **重构模式明确**：从阶段一代码中识别出了清晰的模块边界和拆分模式

### 规则3：阶段二纪律——一次性还清结构性债务

阶段二不是"慢慢优化"，而是集中时间一次性解决结构性问题：
- 优先拆分大文件（参考module-size-bug-correlation模式）
- 应用在阶段一识别出的重构模式（参考pattern-driven-refactoring模式）
- 使用薄入口垫片保障向后兼容（参考thin-entry-shim模式）
- 所有测试必须通过，不允许引入功能退化

### 规则4：显式标记技术债务

阶段一产生的"未验证代码""实验性功能"必须显式标记STATUS：
- `STATUS: VALIDATED` — 已验证可生产使用
- `STATUS: EXPERIMENTAL` — 实验性质，API可能变更
- `STATUS: UNVALIDATED` — 已实现但未充分测试，存在风险（参考mvp-unvalidated-code-debt模式）
- `STATUS: STUB` — 占位实现，待完善

## 实施检查清单

**阶段一启动前**：
- [ ] 是否明确了阶段一要验证的核心假设（1-3个）？
- [ ] 是否定义了阶段一完成的验收标准（什么叫"价值验证通过"）？
- [ ] 是否约定了阶段一代码质量的"红线"（测试/提交/标记规范）？

**阶段一完成时（进入阶段二前）**：
- [ ] 核心假设是否被验证？
- [ ] 核心路径是否有测试覆盖？
- [ ] 是否识别出了清晰的模块边界和重构模式？
- [ ] 所有技术债务是否显式标记了STATUS？

**阶段二完成时**：
- [ ] 所有大文件是否拆分为<500行模块？
- [ ] 测试是否全部通过？
- [ ] 薄入口垫片是否正确设置？
- [ ] 是否所有模块都有明确的单一职责？

## 反例警示

| 错误做法 | 后果 |
|---------|------|
| 阶段一写代码完全不考虑后期重构 | 代码耦合严重到无法拆分，只能推倒重来 |
| 阶段一不写任何测试 | 阶段二重构没有安全网，不敢动代码 |
| 价值还没验证就开始大规模重构 | 在错误方向上优化，浪费时间 |
| 阶段二拖成"慢慢改" | 技术债务长期存在，越拖越难改 |
| 不标记技术债务 | "未验证代码"被误以为是成熟代码，产生线上Bug |

## 正例：MDI项目验证

**阶段一（4天，价值验证）**：
- 快速实现核心Parser→Validator→Generator三层架构
- 产出8970行代码、259个测试、3个Profile、9种Generator
- parser.py达到1465行（红色警报区），versioning.py 872行（橙色高风险区）
- 验证了核心假设：Markdown IDL在AI场景下确实比OpenAPI更高效

**阶段二（1天，结构优化）**：
- 测试充分（259个测试），重构有安全网
- 从versioning.py/validator.py拆分试点中识别出统一模式
- 单日完成14个大文件拆分，132个文件变更
- 🟠橙色高风险区14→0清零，159+测试零回归
- 最终架构质量高于"upfront设计"——因为它是从真实代码中长出来的

## 与现有模式的关系

- `mvp-unvalidated-code-debt.md`：阶段一产生的未验证代码是此模式的具体场景
- `module-size-bug-correlation.md`：阶段二拆分优先级的量化决策依据
- `pattern-driven-refactoring.md`：阶段二批量重构的执行模式
- `thin-entry-shim.md`：阶段二重构中保障向后兼容的关键技术
- `learn-validate-adopt.md`：两阶段模式是"学习-验证-采用"循环在项目级别的应用
