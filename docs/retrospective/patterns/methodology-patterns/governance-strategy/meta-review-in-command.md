# 指令集元审查设计模式（Meta-Review-in-Command）

## 模式类型
方法论模式（质量保障/治理策略）

## 成熟度
L1 实验级（1次验证：对抗性审查指令集的自举验证设计）

## 适用场景

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 方法论类指令集（对抗性审查/第一性原理/复盘/洞察） | 核心场景 | 方法论自洽性要求——审查方法必须审查自身 |
| 质量门禁类指令集 | 核心场景 | 门禁本身需要质量验证 |
| 自动化工具类Skill | 低 | Skill有脚本执行结果可验证，元审查价值有限 |

## 核心设计

### 四维元审查框架

```
质量验收
  └── 自举验证（元审查）
        ├── 维度1: 审查完整性检查
        │   └── 是否覆盖了所有应审查的维度？是否有遗漏的攻击面？
        ├── 维度2: 偏差自检
        │   └── 审查者自身是否存在确认偏差、权威偏差？
        ├── 维度3: 方法论合规检查
        │   └── 是否严格按照流程执行？是否存在跳过步骤的情况？
        └── 维度4: 反模式检测
            └── 是否存在表演式审查、角色混同、过度攻击、一次性审查等反模式？
```

### 元审查触发时机

| 时机 | 说明 |
|------|------|
| 每次审查完成后 | 标准元审查，检查执行质量 |
| 方法论更新后 | 验证更新后的方法论是否仍然自洽 |
| 审查结果异常少时 | 触发深度元审查，排查表演式执行 |

### 元审查检查项模板

```markdown
## 自举验证（元审查）

在完成每次审查后，执行以下元审查：

- **审查完整性检查**：是否覆盖了所有应审查的维度？是否有遗漏的攻击面？
- **偏差自检**：审查者自身是否存在确认偏差（倾向于找支持性证据）、权威偏差（对权威来源降低审查标准）？
- **方法论合规检查**：是否严格按照[五维验证/四大攻击者]流程执行？是否存在跳过步骤的情况？
- **反模式检测**：是否存在[列出该指令集特有的反模式]？
```

## 验证来源

- **对抗性审查指令集**（2026-07-10）：在"质量验收"部分包含4个维度的自举验证（元审查），与知识库中7种反模式对齐

## 与相关模式的关系

| 模式 | 关系 |
|------|------|
| 对抗性审查协议 | 元审查是协议的自指应用——用协议审查协议自身的执行 |
| 双质量门禁 | 元审查是第二道门禁——第一道审查产出物，第二道审查审查过程 |
| 知识库→指令集转化管道 | 元审查是转化管道中"质量验收"步骤的扩展 |

## 注意事项

1. **元审查不能替代质量验收**：元审查是额外的自检层，不替代原有的质量验收标准
2. **元审查检查项必须具体**：空泛的"是否完整"无效，应为"是否每个攻击者输出了≥1个具体问题"
3. **元审查有成本**：每次审查后执行元审查增加约10-20%的时间成本，高频场景需要权衡
