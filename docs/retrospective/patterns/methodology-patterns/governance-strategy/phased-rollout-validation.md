# 方法论推广渐进式验证模式

## 模式概述

当需要将新沉淀的方法论、流程、模板或工具**批量推广到N≥5个对象**（文档、项目、模块等）时，避免"一次性全量推广"的反模式，采用"验证批→推广批→收尾批"的三阶段渐进式推广策略，每批完成后验证并修复问题再推进，将大规模错误风险控制在最小范围。

本模式是[learn-validate-adopt.md](learn-validate-adopt.md)思想在方法论推广场景的扩展——不仅是"学习-验证-采用"的线性流程，而是将验证环节嵌入推广过程本身，形成"小范围验证→问题修复→中范围推广→问题修复→特殊收尾"的多轮闭环。

## 问题现象

新方法论/流程批量推广时常见的失败模式：

1. **一次性全量推广**：16个项目同时应用新模板，结果发现3类路径错误，需要逐个回退修复
2. **跳过验证直接推广**：凭直觉认为方法论没问题，结果在第8个项目发现根本性设计缺陷，前7个都需要返工
3. **验证批范围过大**：选了10个项目做"验证"，结果验证批本身就引入了大量问题
4. **问题不转化为检查项**：验证批发现的问题只在当前批修复，推广批继续犯同样的错误
5. **高风险对象先处理**：先处理结构最复杂的特殊项目，卡住后影响整体进度

这些模式的共同问题是：在没有验证方法论适用性的情况下就大规模推广，导致错误被批量复制，修复成本呈指数级增长。

## 解决方案

### 三阶段推广模型

```
P0 验证批（3-5个低风险对象）
  ↓ 运行所有检查，记录问题
  ↓ 将问题转化为检查项
P1 推广批（中等风险对象，占总量60%）
  ↓ 应用新增检查项预防已知问题
  ↓ 批量执行（可分子批）
  ↓ ✅ P1完成后：集中格式校验（批量修正共性问题）
P2 收尾批（高风险/特殊对象，占总量20%）
  ↓ 已有充足经验应对特殊情况
  ↓ 全量最终验证
```

### P0：验证批（Validation Batch）

**目标**：在最小范围内验证方法论的适用性，发现并记录所有问题。

**选择标准**：
- 风险最低：结构最简单、最典型的3-5个对象
- 代表性：覆盖最常见的场景，但避开特殊/边缘情况
- 快速反馈：可以快速完成并运行检查

**执行要点**：
1. 严格按方法论执行，不做"聪明的简化"
2. 每完成一个对象就运行检查脚本
3. 详细记录遇到的所有问题（即使是"看起来很小"的问题）
4. 完成后对问题进行分类：哪些是流程问题、哪些是工具问题、哪些是判断标准问题
5. **关键动作**：将问题转化为可执行的检查项，添加到检查清单中

### P1：推广批（Rollout Batch）

**目标**：将验证后的方法论推广到大部分对象，用新增检查项预防已知问题。

**选择标准**：
- 中等风险：结构比较典型，没有极端特殊情况
- 占总量约60%
- 可以分2-3个小批执行，每批后快速验证

**执行要点**：
1. 使用P0阶段更新后的检查清单作为门禁
2. 重点关注P0发现问题的场景，确认不再重复出现
3. 记录P0检查项未能捕获的新问题
4. 如果发现新的根本性问题，暂停推广，回到P0重新验证设计

### P1.5：集中格式校验（Centralized Format Validation）—— 标准必选环节

**目标**：P1批量执行完成后，对所有已处理对象进行一次集中式格式扫描与共性问题批量修正，解决"子代理/批量执行时的系统性格式偏差"问题。

> **为什么需要这个环节？** 案例3验证发现：当使用子代理或脚本批量执行P1时（N≥20），即使有检查清单，仍可能出现10%-20%的对象存在共性格式问题（字段值错误、缺省值未填、导航表漏项等）。这些问题不影响核心流程，但积累起来影响一致性，且分散逐个检查效率极低。集中校验可以一次性扫描发现所有共性偏差，批量脚本修正，效率提升80%以上。

**执行要点**：
1. **全量扫描**：对P1批所有对象执行统一格式校验，扫描字段值、导航表、frontmatter完整性、链接有效性
2. **共性归类**：将发现的问题归类为"可批量修正"和"需逐个处理"两类
   - 可批量修正：字段值统一错误（如scenario填成路径）、冗余字段清理、固定格式缺失——用脚本批量替换
   - 需逐个处理：导航表缺项、链接路径错误、内容缺失——逐个修正
3. **验证修正结果**：批量修正后重新运行检查，确认0错误
4. **沉淀新检查项**：如果发现新的共性问题类型，补充到检查清单，供后续批量执行使用

**典型问题类型（来自案例3验证）**：
- frontmatter字段值填充错误（如scenario字段填了目录路径而非标准枚举值）：11/61 ≈ 18%
- source字段错误指向其他项目：2/61 ≈ 3%
- 导航表/交付物清单漏项：1/61 ≈ 2%
- 合计问题率约23%，全部在集中校验环节一次性发现并修正

### P2：收尾批（Closure Batch）

**目标**：处理剩余的高风险/特殊对象，完成全量验证。

**选择标准**：
- 高风险：结构复杂、历史遗留、特殊定制的对象
- 占总量约20%
- 在前两批经验充足后最后处理

**执行要点**：
1. 基于前两批积累的经验处理特殊情况
2. 特殊情况可以例外处理，但必须记录例外原因
3. 所有对象处理完成后，运行**全量最终验证**
4. 将特殊情况的处理经验反馈到方法论的"适用边界"章节

### 关键配套机制

| 机制 | 作用 | 实现方式 |
|------|------|---------|
| **分类前置** | 推广前先分类处置，避免无效操作 | 使用[classification-disposition-decision-tree.md](../document-architecture/classification-disposition-decision-tree.md)对所有对象分类 |
| **问题→检查项转化** | 防止同类问题重复出现 | P0发现的每个问题必须对应至少一个检查项 |
| **批次门禁** | 不把问题带入下一批 | 每批完成后所有检查0错误才能进入下一批 |
| **低风险优先** | 减少早期卡住的概率 | 按风险等级排序，最简单的先做 |
| **P1.5集中格式校验** | 解决批量执行的系统性格式偏差 | P1完成后全量扫描+共性问题批量修正+补充新检查项（N≥20时必选） |

## 适用场景

- ✅ 新模板/新检查清单推广到≥5个文档/项目
- ✅ 新流程/新SOP在团队范围内推广
- ✅ 大规模重构/批量文档治理
- ✅ 工具链升级影响多个模块
- ❌ 一次性变更（<3个对象，直接执行即可）
- ❌ 紧急修复（时间不允许分批验证）
- ❌ 全新项目从0开始（无历史资产需要迁移）

## 实际案例

### 案例1：元原子化模式批量推广（2026-07-05）

**背景**：需要将"元原子化二分+概览"模式从单点验证推广到16个复盘报告目录。

**如果用反模式（一次性全量推广）**：
- 可能在所有16个项目中都出现相对路径层级错误
- README计数未更新的问题会在所有项目中重复
- 发现3个不适合拆分的项目时已经做了无用功
- 预计修复时间：>4小时，且容易遗漏

**使用渐进式验证模式**：
1. **P0验证批（3个项目）**：tuya-homeassistant-integration、tuya-mqtt、tuyaopen-analysis
   - 发现问题：相对路径层级错误（3次）、README计数未更新（1次）、误创建冗余文件（1次）
   - 转化检查项：子目录路径验证、README导航同步、已有结构前置检查
2. **P1推广批（6个项目）**：6个中等规模IoT项目
   - 应用新检查项，P0发现的路径错误0复现
   - 仅发现1个新问题（PowerShell行统计不准确），记录为P2改进
3. **P2收尾批（5个项目+2个跳过+3个补全）**：
   - 正确识别5个已有原子化目录的项目，仅补全导航
   - 正确保留3个不适合拆分的大文件
   - 全量链接验证33个链接0断链

**结果**：总计处理16个对象，新增38个文件，问题6个全部在P0阶段捕获并修复，P1/P2无同类问题重复，总耗时约2小时（比反模式预计节省50%时间）。

### 案例2：L3模式模板升级（2026-07-05，自举验证）

**背景**：将5个L2模式升级为L3并嵌入6个核心开发模板。

**三阶段执行**：
1. **P0**：先升级1个模板（wiki-spec-template），运行检查发现锚点格式问题
2. **P1**：升级其余4个模板，应用锚点检查项
3. **P2**：验证模板在后续项目中的实际应用

**结果**：模板检查项从~24项增加到~35项，四层质量防御体系成功建立。

### 案例3：复盘模板v1.2批量标准化升级（2026-07-06）

**背景**：将comprehensive-retrospective-template v1.2.0（新增scenario标识、insight-action-backlog分离、导航增强）推广到61个6月29日后的复盘项目。

**三阶段执行**：
1. **P0验证批（5个项目）**：选择competitive-analysis目录下结构最一致的5个项目
   - 验证SOP三步法（创建backlog→更新frontmatter→更新导航表）可行
   - 发现问题：相对路径层级正确、已闭环项目行动项迁移无丢失
   - SOP补充：部分完成项目需在backlog中清晰区分状态
2. **P1推广批（56个项目）**：子代理并行处理所有剩余目标项目，覆盖四大目录
   - P0验证的SOP适用，backlog创建顺利
   - 执行P1.5集中格式校验：发现14个共性格式问题（11个scenario字段错误、2个错误source字段、1个导航表补全），问题率约23%，全部一次性批量修正
   - 抽查4个代表项目共约311个本地引用，零断链
3. **P2收尾批（4个补全导航+验证+模式更新）**：已完成
   - 3个历史补全导航项目验证确认无需变更，1个已在P1中完成升级
   - 模式maturity更新（validation_count 2→3）
   - 最终项目总结报告完成，原子提交归档

**结果**：P0验证SOP→P1批量执行→P1.5集中格式校验→P2收尾验证的四步流程闭环，共61个项目完成升级，抽查零断链，共性格式问题在集中校验环节100%发现并修正。验证了"轻量升级而非方法论落地"场景下三阶段模型依然有效，沉淀了"子代理批量执行后必须进行集中格式校验"的标准实践（N≥20时必选）。

## 反模式

### ❌ 反模式1：一次性全量推广

"反正都是一样的操作，批量执行更快"——问题：错误被批量复制，修复时需要逐个回退，总成本比分批验证高得多。非线性返工成本规律：验证阶段发现问题修复成本是1，推广后发现是10，全量完成后发现是100。

### ❌ 反模式2：验证批选择不当

选最复杂的项目做验证——问题：验证批本身就卡住了，无法获得有效反馈；或者选太简单的项目——问题：复杂场景的问题没有被发现。正确做法：选典型的、中等复杂度的3-5个。

### ❌ 反模式3：问题不转化为检查项

P0发现路径错误，手动修完就继续推广——问题：P1批继续犯同样的路径错误。每个问题必须对应至少一个可自动化或可检查的门禁项。

### ❌ 反模式4：高风险先处理

"最难的先搞定，后面就轻松了"——问题：高风险项目遇到意外问题会卡住整个进度，打击信心。正确做法：低风险→中风险→高风险，积累经验后再处理难题。

### ❌ 反模式5：跳过P1.5集中格式校验（N≥20时）

"P1每个项目都检查过了，不需要再集中扫一遍"——问题：当使用子代理/脚本批量执行（N≥20）时，即使每个项目都经过检查，仍会有10%-25%的共性格式偏差（字段值错误、导航漏项等），分散在各个项目中难以发现。跳过集中校验意味着这些问题会流入P2甚至最终交付，后续修复需要逐个定位，成本提升5-10倍。正确做法：N≥20的批量执行后必须执行P1.5集中格式校验，一次性扫描、分类、批量修正。

## 边界与选型

**什么时候不应该用这个模式？**
- 处理对象<3个：直接执行即可，分批的 overhead 大于收益
- 紧急安全修复：时间不允许分批，先修复再说，事后复盘
- 完全标准化的机械操作：如批量重命名、批量格式转换，已有成熟脚本且幂等

**什么时候必须用？**
- 首次推广新方法论/新模板
- 批量处理≥5个非标准化对象（结构有差异）
- 历史遗留系统改造
- 涉及质量门禁的规则变更

## 检查清单

使用本模式时，确认以下事项：

- [ ] 已完成所有待处理对象的扫描盘点
- [ ] 已使用分类决策树标记每个对象的处置方式
- [ ] P0验证批选择了3-5个典型低风险对象
- [ ] P0批每个问题都已转化为检查项
- [ ] P1批使用了更新后的检查清单
- [ ] **N≥20时，P1完成后执行了P1.5集中格式校验**（全量扫描+共性问题批量修正）
- [ ] 集中校验发现的新共性问题已补充到检查清单
- [ ] 每批完成后运行了全量检查
- [ ] P2批最后处理高风险/特殊对象
- [ ] 全量完成后运行了最终验证
- [ ] 特殊情况的处理已记录反馈到方法论
