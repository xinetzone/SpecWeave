# 公理系统一致性第一性原理（Axiom System: Consistency Enables Composability）

## 模式类型

方法论模式（治理策略/第一性原理/系统设计）

## 成熟度

L2 已验证（4次验证来源：Spec路径一致性、Markdown相对路径规范、file:///格式误用反面案例、数学公理系统/编程接口/数据库外键跨领域验证）

## 适用场景

所有涉及引用关系、格式规范、命名约定的系统设计决策。适用于：

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 文档系统路径/链接规范 | ✅✅✅ 核心场景 | Markdown文档的相对路径、链接格式、引用规范 |
| 代码系统接口设计 | ✅✅✅ 核心场景 | API接口、函数签名、模块依赖的一致性 |
| 知识库/元数据规范 | ✅✅✅ 核心场景 | Frontmatter字段、TOML元数据、标签分类的一致性 |
| 规范/协议制定 | ✅✅✅ 核心场景 | 编码规范、协作流程、设计原则的自洽性 |
| 数据库Schema设计 | ✅✅ 强烈推荐 | 外键约束、字段命名、数据类型的一致性 |
| 配置文件管理 | ✅✅ 推荐 | 多环境配置、配置项命名、默认值的一致性 |
| 创意写作/艺术表达 | ❌ 不适用 | 创意领域需要打破一致性以产生新颖性 |

## 问题背景

在文档和代码系统中，一致性问题常被低估为"格式偏好"或"个人风格"：
- "路径用绝对路径还是相对路径无所谓，能打开就行"
- "命名规则不统一也没关系，大家看得懂"
- "frontmatter字段可有可无，不影响内容"

这种认知是危险的。路径格式错误（如把AI对话中的`file:///`格式套用到Markdown文档中）、命名不一致、元数据缺失，看似是小问题，但随着系统规模扩大，会导致：
- 文档无法移植（换个环境链接全断）
- 索引无法自动生成（脚本依赖一致性规则）
- 跨文件引用失效（一个地方改名，所有引用方全断）
- 新人理解成本指数级上升（每个文件都有自己的"风格"）

这些问题的根源不是"不认真"，而是没有认识到**文档/代码系统本质上是公理系统**。

## 核心原则：公理系统一致性

### 第一性原理表述

> **文档（或代码）系统本质上是一个公理系统——路径引用、链接格式、命名规范都是这个系统的"公理"。公理的一致性是定理（派生产物、索引、交叉引用）正确性的前提。一旦公理出现矛盾，整个系统的可理解性就会崩塌。**

类比数学公理系统：
- **公理**：系统中无需证明的基础规则（如"两点确定一条直线"、"Markdown链接使用相对路径"）
- **定理**：基于公理推导出的结论/产物（如欧几里得几何定理、自动生成的索引、交叉引用网络）
- **矛盾**：如果公理不一致，可以推导出任何结论（同时证明P和非P），系统失去意义

| 数学公理系统 | 文档/代码系统 | 一致性要求 |
|------------|-------------|----------|
| 公理（基础假设） | 路径格式、命名规范、frontmatter字段、链接规则 | 全局统一，无矛盾 |
| 定理（推导结论） | 自动索引、交叉引用、派生产物、工具脚本 | 基于公理正确推导 |
| 矛盾（公理冲突） | 有的文件用相对路径有的用绝对路径、frontmatter字段时有时无 | 系统不可靠，工具失效，理解成本飙升 |
| 可组合性 | 多个文档/模块可以组合成更大系统而不冲突 | 公理一致才能组合 |
| 可移植性 | 系统可以在不同环境下正常运行 | 不依赖特定环境的公理（如绝对路径依赖本地文件系统） |

### 为什么相对路径不是"格式偏好"？

相对路径的本质不是"看起来好看"或"个人偏好"，而是保证文档作为公理系统在不同环境下的自洽性和可移植性：
- `file:///d:/AI/...` 是本地环境的绝对路径，换台电脑、换个操作系统、换个目录位置就失效
- `../relative/path.md` 是基于文档间关系的相对路径，只要文档间的相对位置不变，在任何环境下都有效
- 一致性公理要求：所有文档都使用同一种路径规则，不能混用

### 一致性五查（可操作化检查清单）

在创建/修改引用关系或格式规范时，检查5点：

```mermaid
flowchart LR
    C1["① 路径声明<br/>与实际一致？"] --> C2["② 链接格式<br/>符合规范？"]
    C2 --> C3["③ 命名遵循<br/>约定？"]
    C3 --> C4["④ frontmatter<br/>元数据完整？"]
    C4 --> C5["⑤ 跨文件引用<br/>能自动检查？"]
    
    style C1 fill:#FFEBEE
    style C2 fill:#FFF3E0
    style C3 fill:#FFFDE7
    style C4 fill:#E8F5E9
    style C5 fill:#E3F2FD
```

| 检查项 | 具体问题 | 验证方式 |
|-------|---------|---------|
| **1. 路径一致性** | Spec/文档中声明的路径与实际归档位置一致吗？ | 对照实际目录结构，用check-links.py验证 |
| **2. 格式一致性** | 链接格式符合全局规范吗？（相对路径而非绝对路径） | Grep搜索`file:///`等违规格式 |
| **3. 命名一致性** | 文件/目录/变量命名遵循项目约定吗？（kebab-case、英文等） | 运行check-filename-convention.py |
| **4. 元数据一致性** | Frontmatter字段完整吗？必填字段都有吗？ | 运行frontmatter检查脚本 |
| **5. 可验证性** | 跨文件引用能通过自动化工具检查吗？还是只能靠人眼？ | 确保所有引用都能被工具覆盖 |

## 反模式

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| "格式是个人偏好，怎么方便怎么来" | 个人偏好的叠加导致系统公理不一致，工具失效、理解成本上升 | 全局统一规范，个人偏好服从系统一致性 |
| "这个是特殊情况，破例一次" | 一次破例就是公理系统的一个矛盾，破例多了系统就没有公理可言 | 特殊情况要么修改公理适配它，要么把它隔离成独立子系统 |
| "先这么写，以后再统一" | 熵增原理：不一致只会累积不会自动消失，以后统一的成本是现在的N倍 | 第一次就遵守规范，技术债的利息远高于本金 |
| 绝对路径比相对路径"更准确" | 绝对路径依赖本地环境，换环境就断；相对路径依赖文档间关系，可移植 | 所有内部链接使用相对路径，外部链接才用URL |
| "内容才重要，格式不重要" | 格式不一致导致工具无法自动化处理，内容再好在大规模系统中也无法被发现和复用 | 内容和格式是一体两面：格式是公理，内容是定理，公理错了定理也没意义 |
| 工具能处理各种格式，不用严格统一 | 工具的容错是额外复杂度，容错越多性能越差、维护成本越高；"Postel定律"（宽进严出）是边界层策略，不是核心系统策略 | 核心系统内部严格一致，边界层（输入/输出）才做容错适配 |

## 检验标准

做完之后怎么知道做对了？

1. **五查全部通过**：创建/修改后执行一致性五查，没有违规项
2. **工具验证通过**：自动化检查脚本（check-links、check-filename、frontmatter检查）全部通过
3. **可移植性验证**：把文档目录复制到另一个位置/另一台电脑，链接仍然有效
4. **可组合性验证**：新文档可以和现有文档任意交叉引用，不产生冲突
5. **新人可理解**：不了解项目历史的人看一个文件就能推断出全局规范，不需要问"为什么这里不一样"

## 跨场景迁移示例

| 领域 | 公理是什么？ | 一致性破坏的后果 | 一致性五查对应 |
|------|------------|----------------|--------------|
| **Markdown文档** | 相对路径、kebab-case命名、YAML frontmatter | 链接断裂、索引失效、移植失败 | 路径/格式/命名/元数据/可验证 |
| **编程接口** | 函数签名、参数类型、返回值格式、错误码 | 调用方崩溃、集成失败、版本地狱 | API一致性/Schema/命名/文档/类型检查 |
| **数据库设计** | 外键约束、字段类型、命名约定、范式 | 数据不一致、查询错误、孤儿记录 | 关系/类型/命名/约束/外键检查 |
| **数学公理系统** | 欧几里得公理、皮亚诺公理 | 可以推导出矛盾命题，系统崩塌 | 公理独立/无矛盾/完备/可判定 |
| **法律体系** | 宪法、上位法、下位法的效力层级 | 法律冲突、判决矛盾、无法可依 | 合宪性/一致性/明确性/公开性/可救济 |
| **度量衡系统** | 米、千克、秒的定义 | 单位混乱、交易失败、工程事故 | 标准统一/溯源/换算准确/强制使用 |

## 5-Whys根因分析示例

以"为什么路径声明要与实际归档一致？"为例：

| Why层级 | 问题 | 答案 |
|---------|------|------|
| Why1 | 为什么路径声明要与实际一致？ | 否则后续查阅会困惑 |
| Why2 | 为什么查阅会困惑？ | 因为人们按照Spec声明去查找文件却找不到 |
| Why3 | 为什么找不到是个问题？ | 因为这破坏了"Spec→交付物"的映射关系 |
| Why4 | 为什么映射关系重要？ | 因为只有映射一致，多个文档才能组合成可理解的系统 |
| Why5 | **底层本质** | **可组合系统必须建立在一致性公理之上——引用不一致则系统无法被可靠推理** |

同样的5-Whys链条适用于：
- 为什么命名要统一？→不一致则无法全局搜索→无法组合→公理系统崩塌
- 为什么frontmatter字段要完整？→元数据不一致则索引无法生成→无法组合→公理系统崩塌
- 为什么链接要用相对路径？→绝对路径不可移植→跨环境引用断裂→公理系统崩塌
