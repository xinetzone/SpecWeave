# 评分标准：七概念方法论里程碑复盘演练

> **配套练习**：[实战演练01：智能文档系统v2.0里程碑复盘](seven-concepts-practice-01.md)
> **总分**：100分 | **及格线**：60分 | **优秀线**：85分
> **适用**：讲师评分、小组互评、自我评估

---

## 一、等级划分

| 分数区间 | 等级 | 说明 | 团队培训建议 |
|---------|------|------|-------------|
| 85-100 | 🏆 优秀 | 七概念核心方法熟练掌握，洞察有深度，模式可复用，行动项可落地 | 可作为团队复盘方法论种子讲师，辅导其他小组 |
| 70-84 | ✅ 良好 | 方法基本掌握，流程完整执行，产出合格，但洞察深度或模式抽象度有提升空间 | 可独立带领小范围复盘，建议多练习2-3个真实案例 |
| 60-69 | ⚠️ 及格 | 理解了流程，但执行有明显瑕疵，洞察偏表面，行动项不够原子化 | 重新精读七概念核心文档，再做一次本演练 |
| <60 | ❌ 需重做 | 流程有重大缺失，或出现一票否决项 | 从零开始重新做演练，建议有讲师全程引导 |

## 二、评分维度详解

### 维度A：事实采集质量（20分）—— 对应R·复盘感知层

考察是否掌握了"纯事实无判断"的复盘第一原则。

#### A1. 事实数量（4分）

| 分值 | 评分标准 |
|------|---------|
| 4分 | ≥25条事实，覆盖时间线所有关键节点 |
| 3分 | 20-24条事实，基本覆盖关键事件 |
| 2分 | 15-19条事实，有重要节点遗漏 |
| 1分 | 10-14条事实，遗漏较多 |
| 0分 | <10条事实 |

**扣分细则**：
- 每少1条（以20条为基准）扣0.5分

#### A2. 事实客观性（10分）—— 核心考察项

| 分值 | 评分标准 |
|------|---------|
| 9-10分 | 零因果词、零责任归因、零价值判断，所有事实都可被对立角色验证认可 |
| 7-8分 | 有1-2处轻微判断词（如"可惜""幸好"）但无明确责任归因 |
| 5-6分 | 有3-5处因果推断或责任归属，但不影响整体事实基础 |
| 3-4分 | 有6-10处判断混入，事实和观点混杂 |
| 0-2分 | 大量个人判断、责任归因，事实清单变成了"批判大会" |

**高风险判断词清单**（每出现1个扣1分）：
| 类型 | 典型词汇 |
|------|---------|
| 因果连接 | 因为、所以、导致、由于、使得、造成 |
| 责任归因 | 某某的错、某某失误、某某没做好、能力差、不配合 |
| 价值判断 | 错误地、不应该、幸好、可惜、荒谬、愚蠢 |
| 程度副词（非量化） | 严重、非常、很、太（修饰问题时） |

**例外**：在引述角色观点时可以说"张工认为是李工的问题"，但不能直接写"是李工的问题"——这是事实（张工说了这话）vs 判断（确实是李工的问题）的区别。

#### A3. 事实结构化（3分）

| 分值 | 评分标准 |
|------|---------|
| 3分 | 事实编号清晰，按时间或维度分类，关键数据量化 |
| 2分 | 有编号但较零散，部分量化 |
| 1分 | 没有编号或结构混乱 |

#### A4. 小组汇总质量（3分）

| 分值 | 评分标准 |
|------|---------|
| 3分 | 小组成员个人事实有汇总去重，有集体审查记录 |
| 2分 | 有汇总但审查不明显 |
| 1分 | 直接用某一个人的事实清单作为小组产出 |

---

### 维度B：本质洞察深度（25分）—— 对应F·第一性原理 + I·洞察认知层

考察是否能穿透表面现象找到系统性根因，而不是简单归因于人。

#### B1. 5Why根因分析完整性（8分）

| 分值 | 评分标准 |
|------|---------|
| 7-8分 | 3个核心问题都完整追问到第5层以上，根因是系统性的而非个人的 |
| 5-6分 | 3个问题都做了追问，但有1-2个停在第3-4层没问到底 |
| 3-4分 | 只追问了1-2个问题，或追问停在表面原因（"因为沟通不好"） |
| 0-2分 | 没做5Why，直接跳到结论 |

**扣分细则**：
- 根因如果是"某某人能力差""某某人不负责"这类个人归因 → 本项0分
- 根因是"大家不够重视""沟通不到位"这类正确的废话 → 扣4分

#### B2. 洞察四元组完整性（8分，每条洞察约2.7分）

| 分值 | 评分标准 |
|------|---------|
| 8分 | 3条洞察都有完整四元组（陈述/证据/反常识/行动），证据引用事实编号 |
| 6分 | 3条洞察都有四元组，但有1-2处缺项或证据不具体 |
| 4分 | 有2条洞察有完整四元组，第3条有明显缺失 |
| 2分 | 只有1条洞察结构完整，其余只有陈述没有其他部分 |
| 0分 | 洞察只有一句话，没有任何支撑 |

#### B3. 洞察反常识性（5分）—— 核心质量项

| 分值 | 评分标准 |
|------|---------|
| 5分 | 洞察有明显反直觉点，不是大家一看就知道的常识，让人有"原来如此"的感觉 |
| 4分 | 洞察有一定深度，不是表面原因，但反常识性不突出 |
| 3分 | 洞察是对的，但属于"正确的常识"，没有惊喜 |
| 1-2分 | 洞察停留在表面（如"沟通很重要""排期要留缓冲"） |
| 0分 | 洞察是个人归因或错误结论 |

**优秀洞察示例**（可打5分）：
> "延期不是因为意外太多，而是因为我们的进度汇报机制天然压制坏消息——大家默认'自己能搞定'就不说，等搞不定时已经没有缓冲了。"

**差的洞察示例**（0-1分）：
> "张工被抽调是关键转折点""AI算法预估不足是主要原因"（这两个都是表面原因，不是根因）

#### B4. 洞察行动导向（4分）

| 分值 | 评分标准 |
|------|---------|
| 4分 | 每条洞察的下次行动都是具体可做的，不是空话 |
| 3分 | 行动基本具体，但有1条较笼统 |
| 2分 | 行动都是"加强沟通""提高重视"这类无法执行的话 |
| 0分 | 没有行动部分 |

---

### 维度C：模式萃取质量（20分）—— 对应E·萃取沉淀层

考察是否能从具体问题中提炼出可复用、可迁移的模式。

#### C1. 模式结构完整性（8分）

| 分值 | 评分标准 |
|------|---------|
| 8分 | 模式包含名称/触发场景/核心做法/反模式/检验标准/迁移验证6个部分，结构完整 |
| 6分 | 模式包含4-5个部分，缺1-2个非关键部分 |
| 4分 | 模式只有核心做法，其他部分缺失较多 |
| 2分 | 模式只是一个想法，没有结构化 |
| 0分 | 没有萃取模式，只有问题罗列 |

#### C2. 模式可迁移性（6分）—— 核心质量项

| 分值 | 评分标准 |
|------|---------|
| 6分 | 模式可以清晰迁移到至少1个非项目管理场景（如个人生活、家庭、学习等），抽象层次恰到好处 |
| 4-5分 | 模式可以迁移到其他类型的项目中，但跨领域迁移不明显 |
| 2-3分 | 模式只适用于本项目或非常类似的项目，抽象不足 |
| 1分 | 模式就是本次问题的解决方案，没有任何抽象 |
| 0分 | 没有模式 |

**优秀迁移示例**（6分）：
> "红灯上报机制不仅适用于项目管理：
> - 个人任务管理：一个任务卡了超过2天，必须在日记里写下来寻求帮助，不许自己闷头扛
> - 家庭出行规划：如果有人迟到超过15分钟，必须给所有人打电话更新，不能让大家等"

#### C3. 模式命名与传播性（3分）

| 分值 | 评分标准 |
|------|---------|
| 3分 | 名字好记、朗朗上口，一句话核心概括清晰，别人听了就想了解 |
| 2分 | 名字准确但不够好记，有概括 |
| 1分 | 名字很学术/很长，记不住 |

#### C4. 反模式明确性（3分）

| 分值 | 评分标准 |
|------|---------|
| 3分 | 明确列出3个以上反模式，告诉大家"不要怎么做"，反模式来自本次教训 |
| 2分 | 有1-2个反模式 |
| 1分 | 没有反模式 |

---

### 维度D：对抗性审查质量（15分）—— 对应V·验证层（选做，不做时分数分配到其他维度）

> **注意**：如果团队<4人跳过此环节，将D维度15分按比例分配到A/B/C/E维度（各加3-4分）。

#### D1. 审查意见数量与具体性（5分）

| 分值 | 评分标准 |
|------|---------|
| 5分 | ≥7条审查意见，每条都具体指出哪里有问题，引用对方材料中的具体内容 |
| 4分 | 5-6条意见，基本具体 |
| 3分 | 3-4条意见，或意见较笼统 |
| 1-2分 | 1-2条意见，或意见都是"写得很好"这类客套话 |
| 0分 | 没有实质审查意见 |

**扣分细则**：
- "你们分析得很深入""洞察挺到位的"这类正面评价不算数，必须是**问题/质疑/改进建议**
- 每条意见必须具体到对方的某条洞察/事实，不能泛泛而谈

#### D2. 四视角覆盖度（4分）

| 分值 | 评分标准 |
|------|---------|
| 4分 | 四个视角（魔鬼/新人/老板/未来）都有覆盖，视角多样 |
| 3分 | 覆盖3个视角 |
| 2分 | 覆盖2个视角 |
| 1分 | 只从1个视角攻击 |

#### D3. 意见采纳与修正（6分）

| 分值 | 评分标准 |
|------|---------|
| 6分 | 被审查方认真阅读了意见，采纳≥3条并明确记录了修正内容，对不采纳的意见也有回应说明 |
| 4-5分 | 采纳2条意见，有修正记录 |
| 2-3分 | 采纳1条意见，或修正不明显 |
| 1分 | 拿到意见但没修正，或全盘反驳不接受任何意见 |
| 0分 | 根本没看对方意见 |

---

### 维度E：原子行动项质量（15分）—— 对应A·原子化 + C·原子提交执行层

考察复盘是否最终落地为可执行的行动。

#### E1. 原子性符合度（9分，每项3分）

| 分值 | 评分标准 |
|------|---------|
| 9分 | 3个以上行动项都符合全部5个原子标准（单一职责/可验证/有负责人/有时间盒/可独立交付） |
| 7-8分 | 3个行动项基本原子化，有1项存在小问题（如时间不明确） |
| 5-6分 | 有2个行动项原子化较好，其余较笼统 |
| 3-4分 | 行动项都是"优化XX""加强XX""完善XX"这类模糊词 |
| 0-2分 | 没有原子行动项，只有复盘总结文字 |

**反模式示例**（每项扣2分）：
- ❌ "加强项目风险管理"（不具体、不可验证、无负责人）
- ❌ "提高代码质量"（什么叫"提高"？做完怎么验证？）
- ❌ "以后注意多沟通"（这是口号不是行动项）

**好的示例**：
- ✅ "刘经理在7月20日前修改项目管理模板，加入延期1天上报规则（验证：下一个项目启动会上宣讲）"

#### E2. 与洞察对应性（3分）

| 分值 | 评分标准 |
|------|---------|
| 3分 | 每个行动项都能追溯到前面的某条洞察/模式，不是凭空来的 |
| 2分 | 大部分行动项有对应，少数脱节 |
| 1分 | 行动项和前面的洞察/模式没什么关系 |

#### E3. 可落地性（3分）

| 分值 | 评分标准 |
|------|---------|
| 3分 | 行动项做完后确实能解决问题，不会引入新的大问题，成本可接受 |
| 2分 | 行动项基本可行，但需要进一步细化 |
| 1分 | 行动项听起来很好但落地成本极高或不切实际 |

---

### 维度F：团队参与与个人反思（5分）

#### F1. 全员参与度（2分）

| 分值 | 评分标准 |
|------|---------|
| 2分 | 每个角色都有贡献，讨论充分，没有"划水"成员 |
| 1分 | 主要是1-2个人在说，其他人参与较少 |
| 0分 | 只有一个人完成了所有内容 |

#### F2. 个人反思质量（3分）

| 分值 | 评分标准 |
|------|---------|
| 3分 | 每个人的3句话反思真诚具体，能看出确实有学习收获，不是套话 |
| 2分 | 反思基本认真，但有套话 |
| 1分 | 反思很敷衍，全是"学到了很多""非常有价值"这类空话 |

## 四、一票否决项

出现以下任何一种情况，直接判定为❌需重做（<60分），无论其他维度得分如何：

1. **事实清单变批判大会**：超过一半的"事实"是责任归因和价值判断，完全没有掌握"纯事实"原则
2. **根因全部归因于人**：所有问题的根因都是"某某人能力差/不负责/失误"，完全没有系统性思考
3. **没有原子行动项**：复盘完没有任何具体行动项，只有空泛总结
4. **跳过关键环节**：三个核心环节（事实→洞察→行动）缺了任意一个
5. **对抗审查变吵架**：组间对抗变成互相攻击，没有实质性质疑和学习

## 五、常见扣分点速查（讲师快速评分用）

| 问题 | 扣分幅度 | 出现频率 |
|------|---------|---------|
| 事实中出现"因为/所以"因果词 | 每个扣1分 | ⭐⭐⭐⭐⭐ 极高 |
| 根因停留在"沟通不好/重视不够" | 扣4-5分 | ⭐⭐⭐⭐ 高 |
| 洞察是个人归因（怪某个人） | 洞察项0分 | ⭐⭐⭐⭐ 高 |
| 行动项是"加强XX/优化XX"空话 | 每个行动项扣2分 | ⭐⭐⭐⭐⭐ 极高 |
| 模式只能用在这个项目上，不能迁移 | C2扣3-4分 | ⭐⭐⭐ 中 |
| 对抗审查意见都是客套话 | D1扣3-4分 | ⭐⭐⭐ 中 |
| 5Why只问了2-3层就停了 | B1扣3-4分 | ⭐⭐⭐⭐ 高 |
| 洞察没有引用事实编号作为证据 | B2每条扣1分 | ⭐⭐⭐ 中 |

## 六、讲师评分引导问题

讲师评分时，不要只看产出物，还要问小组以下问题来验证真实理解：

**关于事实采集**：
- "你写的这条'F07：技术负责人被抽调是重大失误'——张工本人会同意这是'事实'吗？"
- "这条事实里的'幸好'是什么意思？能不能去掉？去掉后信息损失了吗？"

**关于洞察深度**：
- "你说根因是'乐观偏差'，那为什么大家会乐观偏差？这是人性的问题，还是我们机制的问题？"（继续往下追问一层）
- "如果下次张工不被抽调、第三方API不限流，这个项目就不会延期了吗？"

**关于模式迁移**：
- "你这个模式除了管软件项目，还能用在什么地方？比如用在你家装修上怎么用？"

**关于原子行动项**：
- "这个行动项'加强风险管理'——具体谁做？做什么？什么时候做完？做完怎么知道成了？"

## 七、自评引导问题

做完演练后，小组先用以下问题自我评估：

1. **关于事实**：
   - 我们写的这些"事实"，如果项目组7个角色每个人都看一遍，会有人拍桌子说"这不是事实！"吗？如果有，那条事实就有问题。
   - 读我们的事实清单，能看出我们"想怪谁"吗？如果能，说明混入了立场。

2. **关于洞察**：
   - 我们的洞察，说给一个不在这个项目里的人听，他会觉得"原来如此"还是"这不是废话吗"？
   - 如果我们的洞察是对的，那为什么之前没人这么做？是什么阻碍了大家看到这个根因？

3. **关于模式**：
   - 我们萃取的模式，能写进公司的过程资产库吗？下一个项目组看了能用吗？
   - 如果半年后回头看，这个模式还成立吗？什么情况下会失效？

4. **关于行动项**：
   - 我们写的行动项，下周一回到公司真的会有人去做吗？如果不会，说明行动项不够原子化，或者没有触及真问题。

**A. 事实采集：16/20**

- 事实结构化清晰，有编号
- 小组讨论了但审查不仔细

**B. 洞察深度：18/25**
- 3个问题都做了5Why，但第2个问题（P0漏测）问到"测试不充分"就停了，没继续问为什么测试不充分
- 3条洞察都有四元组，但第2条的反常识性不够（"测试要留足够时间"）

**C. 模式萃取：14/20**
- 模式结构完整（6部分都有）
- 模式"风险缓冲池"能迁移到其他项目，但跨领域迁移有点勉强
- 名字好记，反模式有3条

**D. 对抗审查：10/15**
- 提了5条意见，有2条是客套话
- 覆盖了3个视角（缺未来视角）

- 全员参与，反思真诚

1. 事实采集要更严格地区分事实和判断
