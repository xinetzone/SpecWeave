# 不重构清单：明确划定不改动边界防止范围蔓延

## 模式概述

重构和标准化时，开发者往往倾向于"全面改造"——凡是觉得不够优雅的代码都想改。这种"清理强迫症"导致范围无限蔓延、破坏稳定系统、精力分散、回归风险增加。不重构清单模式通过在项目启动前就明确划定"哪些模块即使有瑕疵也不重构"，反向控制改造范围。

## 问题现象

- 范围无限蔓延：从修复一个问题扩展到重写整个模块
- 破坏稳定系统：改动运行良好的代码引入新Bug
- 精力分散：在非瓶颈区域投入过多时间，核心问题反而没解决
- 回归风险增加：不必要的改动带来不必要的测试负担

典型场景：标准化脚本时发现 `check-spec-consistency.py` 也有类似函数，"顺手"重构了它，结果引入Bug，花费时间调试，原本计划的核心任务延期。

## 解决方案

维护一份**明确的不重构清单（No-Touch List）**，在项目启动前就划定边界。清单应该包含四类模块：

1. **功能已稳定且无Bug的模块**：如果它运行良好且无变更需求，不要为了"代码更优雅"而去动它
2. **当前迭代不涉及的模块**：本次重构聚焦特定瓶颈，其他模块即使有问题也留给未来处理
3. **复杂度高且收益低的模块**：改造成本远大于收益的"硬骨头"，除非有明确需求否则不碰
4. **依赖链复杂的基础设施**：牵一发而动全身的底层模块，除非有充分的测试覆盖和回滚方案，否则不轻易改动

不重构清单不是"永远不改"，而是"当前阶段不改"。当未来的需求或架构演进真正触碰到这些模块时，再做系统性评估。

### 决策三问

判断一个模块是否应该列入不重构清单，问三个问题：
1. 它是否在当前瓶颈路径上？（不是 → 不重构）
2. 改动它的收益是否显著大于风险？（不是 → 不重构）
3. 是否有充分的测试覆盖保障改动安全？（没有 → 谨慎）

任何一个答案为"否"，都应该考虑列入清单。

### 与瓶颈优先重构法的关系

两个模式是互补的关系：
- **瓶颈优先重构法**回答"应该重构什么"（正向聚焦）
- **不重构清单**回答"不应该重构什么"（反向边界）
- 两者结合，形成"有所为有所不为"的治理框架

```
┌─────────────────────────────────────────┐
│            代码库全景                     │
│  ┌─────────────────────────────────┐    │
│  │   瓶颈区域（优先重构）            │    │
│  └─────────────────────────────────┘    │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │   不重构清单（保持现状）          │◄───┼── 本模式
│  │   稳定/低收益/高风险/非焦点       │    │
│  └─────────────────────────────────┘    │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │   观察区（技术债务登记）          │    │
│  │   未来可能重构但当前不处理        │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

## 实际案例

### 案例：SpecWeave 脚本标准化边界

在 `.agents/scripts/` 脚本标准化改造中，应用不重构清单：

**✓ 列入重构范围（瓶颈路径上）**：
- `.agents/scripts/lib/frontmatter.py` — 被所有脚本依赖，YAML解析Bug影响全局
- `.agents/scripts/lib/cli.py` — 输出层统一，所有命令行脚本共用
- `.agents/scripts/lib/markdown.py` — Markdown处理核心，docgen/link-fixer共用

**✗ 列入不重构清单（不在当前瓶颈路径上）**：
- `.agents/scripts/check-spec-consistency.py` — 独立运行，与本次改造无关
- `.agents/scripts/generate-tests.py` — 代码生成器，运行频率低
- `.agents/scripts/agents.py` — 项目初始化脚本，极少使用
- `.prompt_extraction/` — 独立子项目，有自己的架构

发现其他脚本有类似问题时，记录到技术债务清单（但不在本次处理），专注完成当前瓶颈改造，未来有需求时再处理已记录的债务。

## 效果

- **范围控制**：防止重构/优化时的"顺手"改码，确保每次改造都有明确焦点
- **风险隔离**：不随意改动稳定模块，降低回归Bug概率
- **精力集中**：把有限的精力投入到真正的瓶颈上，产出最大价值
- **技术债务可视化**：把"暂时不改"的问题登记在册，而不是遗忘，为未来决策提供依据

## 适用场景

- 大规模重构或标准化改造前的边界划定
- 多个模块存在问题但资源有限时的取舍
- 代码审查时识别"顺手改码"的反模式
- 技术债务管理：区分"立即处理"和"登记观察"的债务

## 反模式

以下行为违背不重构清单原则：

1. **清理强迫症**：看到不优雅的代码就想改，不管它是否在当前任务范围内
2. **顺手改码**：修复A问题时"顺便"重构B模块，导致变更范围不可控
3. **过早优化**：对当前无性能问题的模块做预优化，浪费时间且引入风险
4. **没有清单**：完全靠自觉控制范围，边界模糊时容易越界

## 与其他模式的关系

- **与bottleneck-first-refactoring互补**：一个正向聚焦，一个反向边界
- **被meta-capability-inversion引用**：元能力依赖倒置决定构建顺序，不重构清单决定改造边界
- **配合governance-tier-priority**：治理分层优先级决定哪些层级优先处理
