# 不可变约束清单模式：踩坑经验的工程化沉淀

## 模式概述

将项目中"屡踩屡犯"的坑点整理为**不可变约束清单**，每条约束都包含：约束内容、历史踩坑原因、代码位置。新人可读、改动有据、经验可传承。

## 问题现象

项目演进过程中常见的经验流失问题：

1. **新人重复踩坑**：老员工知道的暗坑，新人不知道，一次次重蹈覆辙
2. **凭印象改回去**：有人忘了为什么加某段代码，觉得没用就删掉，结果问题复现
3. **调试无记忆**：每次踩坑都从零开始排查，上次怎么修的全忘了
4. **知识个人化**：经验只在某个人脑子里，人走了知识也走了
5. **约束散落各处**：代码注释、PR 描述、群聊记录里都有，就是找不到

## 解决方案

创建专门的约束文档（如 `constraints.md`），以表格形式列出所有不可变约束：

```markdown
# 不可变约束（违反会立即失败）

每一条都对应过真实的失败现场，**禁止凭印象撤销**。

| # | 约束 | 原因 / 历史踩坑 | 代码位置 |
|---|------|-----------------|---------|
| 1 | `TVM_FFI=ctypes` 运行时设置 | 排除 Cython 模块，避免 Nuitka 与 Cython 双重编译冲突 | `xmnn/tvm/_tvm_nuitka_init.py:22` |
| 2 | Nuitka 必须 `--clang` | conda gcc 版本格式不被识别 | `nuitka_*_args` 传入 |
| 3 | 所有编译必须在容器内执行 | 宿主机环境不一致，结果不可复现 | `tasks.py → utils/build.py` |
| ... | ... | ... | ... |

## 改动这些约束的纪律

若一定要改：

1. 在 `postmortems/task-summary-YYYYMMDD.md` 中记录（现象 / 根因 / 修复）
2. 同步更新本文件对应行
3. 如属设计决策级，同步更新 `architecture.md`
```

## 约束三要素

每条约束必须包含三个要素：

| 要素 | 作用 | 示例 |
|------|------|------|
| **约束内容** | 明确什么能做、什么不能做 | "Nuitka 必须使用 `--clang`" |
| **原因 / 历史踩坑** | 为什么有这个约束，背后踩过什么坑 | "conda gcc 版本格式不被识别" |
| **代码位置** | 约束的具体实现位置，方便核查 | `nuitka_*_args` 参数传入处 |

**关键是第二要素**：没有原因的约束就是教条，有原因的约束才是智慧。

## 维护纪律

1. **每条约束对应真实失败现场** — 不凭想象加约束，每条都踩过坑
2. **禁止凭印象撤销** — 想改可以，先证明为什么原来的约束不成立
3. **改动必须留痕** — 改约束要写复盘，记录"现象/根因/修复"
4. **设计级变更同步架构文档** — 重大约束变更更新 architecture.md

## 约束来源

约束通常来自以下场景：

| 来源 | 典型约束 |
|------|---------|
| **工具版本兼容** | "LLVM 必须 ≥16 且 <18"、"Python 必须 3.13" |
| **第三方限制** | "Nuitka 必须 --module 不能 --standalone" |
| **性能陷阱** | "bind mount 不能放 Nuitka 缓存，太慢" |
| **历史 Bug** | "PYTHONPATH 必须包含 src_dir，否则静默用旧版" |
| **安全要求** | "禁止在容器内挂载宿主机 /tmp" |
| **跨平台兼容** | "Windows 下输出必须走 stdout.buffer" |

## 适用场景

- 踩坑多、约束多的项目（编译器、构建工具、底层系统）
- 人员流动大、知识传承难的团队
- 依赖复杂、暗坑多的第三方集成
- 跨平台（Windows/Linux/macOS）工具项目
- 有 AI 智能体参与开发的项目（约束前置，避免 AI 瞎试）

## 正反例

### 正例

```markdown
| 15 | xmnn Nuitka 编译 PYTHONPATH 必须包含 src_dir |
     缺少时 Python 从 site-packages 导入旧版 xmnn .so，
     源文件修改全部被忽略，表现为编译通过但 .so 内容不变 |
     utils/nuitka_scripts.py:107 |
```

→ **三要素齐全**：知道做什么、为什么、在哪看代码

### 反例

```markdown
# 注意事项
- 注意 PYTHONPATH
- 记得用 clang
- 小心编码问题
```

→ **信息密度为零**：全是正确的废话，出问题时找不到具体依据

## 模式价值

1. **新人保护**：读一遍约束清单，避免 80% 的低级踩坑
2. **决策依据**：争议时查约束清单，用历史事实说话
3. **知识固化**：个人经验 → 团队资产，人走知识留
4. **AI 友好**：AI 智能体读了约束清单，不瞎试不走弯路
5. **调试加速**：出问题先查清单，看是不是已知坑

## 与其他模式的关系

- **根因诊断（root-cause-diagnosis）**：诊断模式是找原因，约束清单是存结果
- **复盘知识管理（retrospective-knowledge）**：复盘是过程，约束清单是产出物之一
- **三级规范防御（three-layer-spec-constraint）**：约束清单是最具体的一层
