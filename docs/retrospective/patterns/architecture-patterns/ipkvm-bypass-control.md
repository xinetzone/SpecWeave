# IPKVM硬件旁路远控模式（IPKVM Bypass Control Pattern）

## 模式类型

架构模式（硬件技术架构）

## 成熟度

L2 已验证（5款向日葵无网远控硬件产品验证：控控2/Q1/Q2Pro/Q0.5/Q5Pro）

## 适用场景

需要在以下场景实现对计算机的远程控制：
- 无网络/网络隔离环境下的设备运维
- BIOS/UEFI级别控制（操作系统启动前）
- 高安全等级场景（涉密网络、医疗设备、工业控制），要求物理隔离不安装被控端软件
- 无人值守设备的远程管理（户外机柜、工业设备、服务器）

## 问题背景

纯软件远控方案（如TeamViewer、ToDesk、RDP）存在三个根本局限：
1. **操作系统依赖**：必须在被控机安装软件，操作系统崩溃或未启动时无法控制
2. **网络依赖**：被控机必须联网，且控制链路与被控机网络不隔离，存在网络攻击风险
3. **权限限制**：无法进入BIOS设置、无法在启动过程中干预、无法处理蓝屏死机

传统KVM over IP方案（如Raritan、ATEN）价格昂贵（数千元到数万元），配置复杂，主要面向数据中心。

## 核心规则

IPKVM硬件旁路远控通过"HDMI视频采集+USB-HID仿真+独立网络链路"三大核心技术，实现不依赖被控机操作系统和网络的BIOS级远控：

**核心公式**：
```
IPKVM旁路远控 = HDMI视频采集（看见画面）⊕ USB-HID仿真（发送控制）⊕ 独立网络链路（不侵入被控网络）⊕ 旁路部署（串接不侵入）
```

### 五大核心要素

| 要素 | 技术实现 | 解决的问题 |
|------|---------|-----------|
| HDMI视频采集 | 通过HDMI输入接口直接采集被控机视频输出信号，内置视频采集芯片（如MS2109/MS2130） | 不依赖显卡驱动或操作系统，BIOS/蓝屏/启动画面均可获取 |
| USB-HID仿真 | 模拟标准USB键盘/鼠标复合设备，被控机识别为通用HID输入设备 | 无需专用驱动，支持所有支持USB的系统（BIOS/Windows/macOS/Linux） |
| 旁路部署 | 串接在HDMI线和USB线上（HDMI IN→设备→HDMI OUT接显示器，USB IN→设备→USB OUT接被控机），不接入被控机内部 | 不安装任何软件，不侵入被控机网络，物理隔离 |
| BIOS级控制 | 基于HID仿真在任何USB支持阶段发送键盘鼠标输入 | 可在开机自检、BIOS设置、OS启动前、蓝屏等任何阶段控制 |
| 物理隔离 | 控制走独立网络（有线/WiFi/4G/5G），与被控机网络完全分离 | 杜绝通过远控链路攻击被控机网络的风险，满足高安全等级要求 |

### 部署架构

```
┌─────────────┐     HDMI OUT      ┌──────────┐     HDMI IN     ┌─────────────┐
│  被控计算机  │──────────────────→│ IPKVM设备 │────────────────→│   显示器     │
│             │     USB OUT        │ (旁路串接) │    USB IN       │             │
│             │←──────────────────│          │←────────────────│  键盘/鼠标   │
│             │                    │  独立网口  │                  │  (本地)      │
└─────────────┘                    │ WiFi/4G  │                  └─────────────┘
       ↑ 被控机不安装任              │  蓝牙    │                        ↑
       │ 何软件,完全隔离             └────┬─────┘                        │
       │                                 │ 独立网络链路                   │ 本地直连
       │                                 ↓                              │ (可选)
       │                         ┌──────────────┐                       │
       └─────────────────────────│ 远控端App/Web │←──────────────────────┘
           物理隔离无             │ (手机/电脑)  │
           网络攻击风险            └──────────────┘
```

## 与纯软件远控的边界条件对比

| 对比维度 | 纯软件远控 | IPKVM硬件旁路远控 |
|---------|-----------|-----------------|
| 被控端软件 | 必须安装 | 不需要 |
| BIOS级控制 | ❌ 不支持 | ✅ 支持 |
| 操作系统崩溃时控制 | ❌ 不支持 | ✅ 支持 |
| 网络隔离环境 | ❌ 需要联网 | ✅ 独立网络/近场蓝牙/USB直连 |
| 物理安全隔离 | ❌ 控制链路走被控网络 | ✅ 控制网络与被控网络完全隔离 |
| 部署复杂度 | 安装软件即可 | 需要硬件串接HDMI+USB |
| 成本 | 软件免费/订阅 | 硬件购买成本（158~1599元+） |
| 延迟 | 依赖网络质量 | 本地视频采集延迟低（通常<100ms） |
| 适用场景 | 日常远控/办公 | 运维/安全隔离/BIOS级/无人值守 |

## 反模式

- **反模式：在被控机安装客户端**：违背了IPKVM旁路"无侵入"的核心原则，失去了BIOS级控制和物理隔离的优势
- **反模式：只做视频采集不做HID仿真**：只能看不能控，不构成完整的KVM方案
- **反模式：接入被控机网络**：破坏物理隔离安全模型，引入网络攻击面
- **反模式：需要专用驱动**：违背USB-HID即插即用原则，降低兼容性
