# 验证规则元自查清单

> **何时使用**：当你创建、更新、审查任何**验证规则**时——包括检查清单、模式文档中的验证章节、SOP验证步骤、CI门禁规则、pre-commit钩子、自动化检查脚本。
>
> **核心递归教训**：对验证覆盖面的验证本身也是一次验证，也存在被遗漏的风险。创建验证规则时，你的思维模式是"构建安全网"而非"突破安全网"——这导致你天然不会想到"安全网自己有没有漏洞"。
>
> **一句话**：写完检查清单后，**用这份清单检查你的检查清单**。

---

## 一、覆盖完整性检查（最容易遗漏）

> 历史教训：初版Step 5只覆盖"模式/模板/脚本"3类，遗漏了检查清单、资产清单、规范文件、导航路由4类，导致这4类资产中的数据漂移无法被发现。

- [ ] **类型穷举**：我列出了所有需要验证的**对象类型/文档类型**吗？用"还有吗？"问自己三遍
- [ ] **路径枚举**：我给出了具体的文件路径/glob模式吗？（如 `patterns/**/*.md` 而非"模式文件"）
- [ ] **配角文件**：钩子、配置、脚本、工具类等"次要文件"是否被包含？（最易漏统计）
- [ ] **索引/清单类**：README、索引文件、资产清单是否被包含？（它们包含统计数字）
- [ ] **规则/规范类**：.agents/下的规范、AGENTS.md等入口路由是否被包含？（它们包含量化阈值）
- [ ] **反向测试**：把已知的历史错误案例（如9处数据漂移）逐个对照，看是否都能被本规则捕获
  - 案例1：{错误类型} → □ 能捕获 □ 不能捕获，需补充：____
  - 案例2：{错误类型} → □ 能捕获 □ 不能捕获，需补充：____
  - 案例3：{错误类型} → □ 能捕获 □ 不能捕获，需补充：____
- [ ] **边界场景**：当对象"不确定是否属于范围"时，默认值是什么？（建议：不确定时按"属于"处理）

## 二、自反性检查（规则是否适用于自身）

> 历史教训：创建"防止数据漂移"的模式文件时，模式文件自身也复制了数据漂移。灭火者自带火种。

- [ ] **规则适用于自身**：这份验证规则（本清单/本模式/本SOP）本身是否在自己定义的验证范围内？
- [ ] **示例数据验证**：规则中引用的所有示例数字（如"发现9处漂移"）是否已追溯到原始来源核对？
- [ ] **计算重验证**：规则中的合计/总计/百分比是否已重新独立计算？
- [ ] **自我扫描**：用本规则扫描它自己的文档，是否能通过所有检查项？
- [ ] **元层声明**：我是否在文档中明确声明"本文件自身也需要通过本验证"？

## 三、可操作性检查（空话vs可执行）

- [ ] **每个检查项可打勾**：每个`- [ ]`项目是否是一个可以明确回答"完成/未完成"的动作？（而非"注意质量"这类空话）
- [ ] **方法明确**：每个检查项是否说明了"怎么做"？（如"用`Get-ChildItem | Measure-Object -Line`统计行数"而非"检查行数"）
- [ ] **工具指定**：如果需要工具/脚本，是否给出了具体命令或脚本名称？
- [ ] **结果可判定**：检查完成后，是否有明确的"通过/不通过"标准？（而非"大概差不多"）
- [ ] **失败处理**：发现问题后怎么做？（修正？标记？上报？）是否有明确指引？

## 四、数据与示例检查

- [ ] **所有数字有源**：规则中出现的每个数字（行数、数量、百分比、版本号）都有来源标注吗？
- [ ] **日期标注**：示例数据/统计数字是否标注了获取日期？（代码会演进，数字会过时）
- [ ] **路径正确**：规则中引用的其他文件/模式路径是否经过链接验证？
- [ ] **无手算结果**：合计/总计是否通过计算而非记忆/估计写入？
- [ ] **无"合理即正确"**：我没有因为数字"看起来合理"就跳过验证吗？

## 五、一致性检查

- [ ] **跨文档一致**：如果同一规则/案例在多个文档中描述，内容是否一致？（如模式A和检查清单B对同一概念的描述）
- [ ] **术语统一**：使用的术语（如"数据漂移""放大效应""可复用资产"）在整个文档中含义一致吗？
- [ ] **与上级规范一致**：本规则是否与项目更高级别的规范（AGENTS.md、global-core-rules.md）一致？
- [ ] **无矛盾**：规则内部是否有自相矛盾的检查项？

## 六、成本与适用性检查

- [ ] **成本分级**：是否区分了"必须执行"和"可选/简化"场景？（避免验证疲劳）
- [ ] **触发条件明确**：什么场景必须全量执行？什么场景可以简化？条件是否清晰？
- [ ] **快速路径**：是否有针对简单任务的快速验证路径？（5分钟能完成的小编辑不需要跑30分钟验证）
- [ ] **无过度验证**：是否要求对所有内容都做深度验证？（应按风险等级匹配验证深度）

## 七、演进与维护检查

- [ ] **升级路径**：发现新的漏洞类型后，如何更新本规则？是否说明了更新流程？
- [ ] **经验反馈**：是否有位置记录"本规则在哪些场景下应用过，发现了什么新问题"？
- [ ] **成熟度标注**：规则的成熟度（L1/L2/L3/L4）是否标注？是否基于实际应用次数？
- [ ] **版本/日期**：规则是否标注了创建/更新日期？

---

## 八、终极自检（递归终止）

> 这是最后一步，也是最关键的一步。

- [ ] **我刚刚用这份清单检查了我的规则，但这份清单本身是否有遗漏？**
- [ ] **如果有人在未来发现这份清单漏了某类检查，那会是什么？**（把你想到的写下来）
  - 可能的遗漏：________________________
- [ ] **我是否在文档中留下了"欢迎补充遗漏"的开放说明？**

> **注**：递归不会无限终止——当你已经穷举了你能想到的所有漏洞类型，并留下开放的补充入口时，就可以停止了。追求"绝对完美"是过度验证；追求"已知漏洞全部覆盖，未知漏洞有反馈通道"是务实的做法。
