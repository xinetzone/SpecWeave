# 测试计划文档库

> **本目录**存放 SpecWeave 项目中各类脚本、工具、功能模块的测试运行计划，包含测试用例矩阵、环境准备、执行顺序、结果记录模板与通过标准，确保测试过程可复现、可追溯、可验证。

## 🎯 测试计划概述

> **什么是测试计划？**
>
> 测试计划（Test Plan）是在执行测试前制定的结构化文档，明确测试对象、环境要求、用例设计、执行策略、通过标准与风险预案，避免"边测边想"导致的测试遗漏或覆盖不全。

### 核心设计原则

| 原则 | 说明 |
|------|------|
| 📋 **用例矩阵化** | 所有测试用例以表格形式呈现，包含ID、场景、命令/操作、预期结果、优先级 |
| 🎚️ **优先级分级** | P0（阻塞级）→ P1（核心功能）→ P2（边界/鲁棒性），指导测试执行顺序 |
| 🔄 **阶段递进** | 测试分阶段执行（基础→核心→辅助→异常→安全），前置阶段不通过不进入后续 |
| 🧪 **dry-run优先** | 写操作测试先dry-run验证，确认无误后再执行实际操作，避免污染生产/测试数据 |
| 📊 **结果可记录** | 提供测试结果记录模板，便于事后追溯问题与回归验证 |
| ⚠️ **风险预案** | 明确已知风险与应对方案，测试中遇到问题可快速定位处理 |

### 三级决策模型定位

测试计划遵循"三级决策模型"，针对不同测试场景选择合适方案：

| 层级 | 场景 | 方案 | 测试计划覆盖 |
|------|------|------|-------------|
| Level 1 | IDE内临时探索/调试 | integrated_browser MCP | 不需要正式测试计划 |
| **Level 2** | **本地独立脚本/批量操作/CI** | **Playwright Python 等自动化脚本** | **✅ 本目录测试计划主要覆盖** |
| Level 3 | 长期服务/系统集成 | REST API / MCP Server | 需要独立的集成测试计划 |

## 📋 测试计划模板说明

以 `forum-bot-playwright-test-plan.md` 为参考模板，标准测试计划包含以下章节：

### 一、三级决策模型定位说明

| 章节 | 内容 | 必要性 |
|------|------|--------|
| **层级定位表** | 明确本计划覆盖Level 1/2/3中的哪一级 | ✅ 必需 |
| **适用场景** | 说明本计划适用的具体测试场景 | ✅ 必需 |
| **测试对象** | 明确被测脚本/模块/功能的路径 | ✅ 必需 |
| **测试环境** | 操作系统、依赖、运行环境要求 | ✅ 必需 |

### 二、测试环境准备

| 章节 | 内容 | 必要性 |
|------|------|--------|
| **前置依赖检查** | 表格列出依赖项、检查命令、预期结果 | ✅ 必需 |
| **环境初始化命令** | 可直接复制执行的环境准备脚本（PowerShell/bash） | ✅ 必需 |
| **测试账号与数据准备** | 测试所需账号、测试文件、前置状态说明 | ⚪ 按需 |

> 💡 **提示**：环境初始化命令应可一键执行，包含目录创建、依赖安装、前置验证等步骤。

### 三、测试用例矩阵（核心）

测试用例按功能域/测试阶段分组，每个用例包含：

| 字段 | 说明 |
|------|------|
| **用例ID** | 统一编号（T01、T02...），便于失败记录与追溯 |
| **测试场景** | 一句话描述测试的具体场景 |
| **命令/操作** | 可直接执行的命令，或具体操作步骤 |
| **预期结果** | 明确的、可验证的预期输出/行为 |
| **优先级** | P0（阻塞级，必须通过）/ P1（核心功能）/ P2（边界/鲁棒性） |

#### 用例阶段划分参考

| 阶段 | 典型内容 | 优先级分布 |
|------|---------|-----------|
| 第一阶段 | 基础环境与帮助信息 | 全部P0 |
| 第二阶段 | 登录/认证流程 | 全部P0 |
| 第三阶段 | 读操作测试（核心查询） | P0为主，含少量P1边界 |
| 第四阶段 | 写操作dry-run测试 | 全部P0，先dry-run |
| 第五阶段 | 写操作实际执行（可选） | P1为主，使用测试数据 |
| 第六阶段 | 辅助功能测试 | P1为主 |
| 第七阶段 | 日志/可观测性验证 | P1为主 |
| 第八阶段 | 异常处理与边界测试 | P1/P2 |
| 第九阶段 | 持久化/状态管理测试 | P0/P1 |
| 第十阶段 | 幂等性与安全测试 | P0/P1/P2 |

> ⚠️ **重要原则**：写操作（编辑/删除/发布等）必须先做 **dry-run 测试**，确认dry-run全部通过后再考虑实际执行。

### 四、测试执行顺序

使用 Mermaid flowchart 展示测试阶段依赖关系与执行路径，并给出执行建议：
- P0用例必须全部通过才能进行后续测试
- dry-run测试全部通过后，再考虑实际写操作测试
- 实际写操作建议使用专用测试数据/测试帖

### 五、测试结果记录模板

提供结构化的结果记录表格：
- **测试摘要表**：按阶段统计用例数、通过、失败、阻塞数
- **失败用例记录表**：记录失败用例ID、现象、错误日志、复现步骤、严重程度、修复状态

### 六、快速验证命令集（冒烟测试）

提供一组可一键执行的冒烟测试命令（全部为只读或dry-run操作，安全无副作用），用于快速验证基本功能是否正常。

**示例结构**：
```powershell
# 1. 显示帮助（验证脚本可执行）
# 2. 读取数据（验证登录状态+读功能）
# 3. 编辑试运行（验证编辑流程，不提交）
# 4. 回复试运行（验证回复流程，不提交）
# ...
Write-Host "`n=== 冒烟测试完成 ===" -ForegroundColor Green
```

### 七、已知风险与预案

表格列出：
- 风险描述
- 影响范围
- 发生概率
- 应对预案

### 八、测试通过标准

明确分级通过标准：
- **冒烟测试通过标准**：核心P0用例通过，无未捕获异常，无FAIL级别错误
- **全量测试通过标准**：P0 100%通过，P1 ≥95%通过，P2 ≥80%通过，无严重未处理异常

### 九、测试后清理

测试完成后的清理步骤：
- 删除临时测试数据/文件
- 清理测试产生的草稿/临时资源
- 查看测试日志等

## 🧭 测试计划使用指南

### 执行者指南

| 步骤 | 操作 | 注意事项 |
|------|------|---------|
| 1 | 阅读"三级决策模型定位"确认适用Level 2 | Level 1用MCP直接探索，不需要完整测试计划 |
| 2 | 按"二、测试环境准备"完成环境检查与初始化 | 所有依赖检查通过后再开始测试 |
| 3 | 先执行"六、快速验证命令集"做冒烟测试 | 冒烟不通过则无需继续，先排障 |
| 4 | 按"四、测试执行顺序"分阶段执行测试 | 严格按P0→P1→P2顺序，前置不通过不继续 |
| 5 | 写操作先做dry-run，确认无误再实际执行 | 实际写操作必须使用测试数据 |
| 6 | 在"五、测试结果记录模板"中记录结果 | 失败用例必须记录错误日志与复现步骤 |
| 7 | 测试完成后执行"九、测试后清理" | 避免测试数据残留污染环境 |

### 编写者指南

创建新测试计划时：

1. **复制模板结构**：参考 `forum-bot-playwright-test-plan.md` 的章节结构
2. **TOML frontmatter**：包含 `id`、`date`、`type`、`target`、`maturity` 字段
3. **用例ID不重复**：从T01开始顺序编号
4. **命令可复制**：测试命令应可直接复制粘贴执行，不需要手动修改路径
5. **优先级明确**：每个用例必须标注P0/P1/P2
6. **dry-run隔离**：所有写操作必须有对应的dry-run用例，且在实际执行用例之前
7. **包含Mermaid流程图**：执行顺序用Mermaid可视化
8. **提供冒烟脚本**：一键可执行的冒烟测试命令集
9. **风险预案完善**：提前预判可能的问题与应对方案
10. **通过标准量化**：避免"基本正常"等模糊表述，用百分比或明确条件
