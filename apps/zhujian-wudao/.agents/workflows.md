# .agents/workflows.md — 标准工作流

## 工作流一：撰写新洞察

### 触发条件
- 用户说"洞察"或"撰写洞察"
- spec 中有设计点未在洞察库中独立展开
- 报名帖中的概念在洞察库中无对应条目
- 复盘报告标记了 P0/P1 缺口

### 执行流程

1. **定位缺口**：搜索现有洞察，确认该概念是否已被覆盖
2. **确定编号**：取当前最高编号 + 1
3. **确定归属层级**：产品层（1-15）→ 架构层（16-30）→ 哲学层（31+）
4. **确定结构**：
   - 体道四法相关 → 七节完整操作手册
   - 其他哲学概念 → 「来源 + 核心内容」基本结构
5. **写入文件**：使用 `SearchReplace` 在对应洞察库文件末尾（统计行之前）插入
6. **更新统计**：修改文件头部和末尾的统计声明（洞察数 + 1，行数更新）
7. **添加交叉引用**：在相关洞察中反向链接到新洞察
8. **更新复盘报告**：将新洞察添加到 §二 洞察列表

### 校验清单
- [ ] 编号正确递增
- [ ] 归属层级正确
- [ ] 头部统计已更新
- [ ] 末尾统计已更新
- [ ] 交叉引用已添加
- [ ] 复盘报告洞察列表已更新

## 工作流二：全面复盘

### 触发条件
- 用户说"复盘"或"复盘+洞察"
- 多轮密集编辑后
- 文件结构发生变化后（拆分、重命名等）

### 执行流程

1. **统计全部文件**：行数、洞察数、状态标记
2. **搜索 ⚠️ 标记**：找出所有未完成项
3. **Spec vs 洞察对照**：检查 spec 每个设计点是否有对应洞察
4. **报名帖 vs 洞察对照**：检查报名帖每个概念是否有独立展开
5. **HTML vs Spec 对照**：检查实现是否与设计一致
6. **文件引用检查**：验证所有跨文件链接的文件名和行号是否正确
7. **头部声明校验**：检查统计数字是否与实际一致
8. **生成优先级清单**：按 P0（架构阻塞）→ P1（支撑缺失）→ P2（完备性）→ P3（规范性）分级

### 输出
- 复盘报告（`-review.md`）或对话中的结构化复盘
- 至少包含：洞察列表、缺口清单、优先级行动清单

## 工作流三：文件更新后的同步

### 修改洞察库后必须同步更新

| 修改内容 | 需同步的位置 |
|----------|-------------|
| 增/删洞察 | 头部统计、末尾统计、复盘报告洞察列表 |
| 修改洞察标题 | 检查是否与复盘报告一致 |
| 添加交叉引用 | 确认目标文件路径和行号正确 |
| 修改文件头部描述 | — |

### 修改 spec 后必须同步检查

| 修改内容 | 需检查 |
|----------|--------|
| 新增设计点 | 是否需要新增洞察 |
| 修改概念定义 | 洞察中是否有引用，是否需要同步修正 |
| 修改节号 | 所有引用该节的洞察链接是否需更新 |

### 修改 HTML 后必须同步检查

| 修改内容 | 需检查 |
|----------|--------|
| 新增场景标签 | Spec 是否需要同步，复盘报告是否需要标记 |
| 修改每日一问题目 | 洞察 53 第三节的覆盖分析是否需更新 |
| 修改 CSS 变量 | 无直接影响 |
