# 竹简悟道 · 可迁移方法论全集

> 萃取日期：2026-06-17 | 源项目：竹简悟道（帛书《老子》AI 反思引导工具）
> 萃取标准：所有与产品具体主题无关、可直接或经适配后迁移到其他项目的模式、模板与原则

---

## 一、项目流程模式

### 1.1 洞察驱动开发

**做法**：在动手实现任何功能之前，先将该功能背后的设计原理写为一条独立的"洞察"。洞察按层级分段（产品层 / 架构层 / 领域层），开发过程中每做一个决策都可以追溯到对应的洞察编号。

**两档结构**：

| 档位 | 适用场景 | 结构 |
|------|---------|------|
| 基础档 | 记录设计决策 | 来源 + 核心内容（1-3 段文字） |
| 完整档 | 需要系统化操作手册 | 来源 + 核心内容 + 七节子结构（一至七） |

**为什么有效**：确保代码不是"凭感觉做出来的"，而是在可追溯的、结构化的认知体系基础上构建的。每个洞察同时是决策记录（当时为什么这样决定）和未来依据（将来要改的时候知道原委）。

**迁移方式**：将"产品层 / 架构层 / 哲学层"替换为目标领域的分层体系。

---

### 1.2 滚动复盘机制

**做法**：每完成 2-3 次变更立即执行一次全项目审计，而非等项目结束后做一次性总结。

**标准八步流程**：

```
1. 统计全部文件 → 行数、条目数、状态标记
2. 搜索 ⚠️ 标记 → 找出所有未完成项
3. 核心文档 A vs 核心文档 B 对照
4. 对外文档 vs 对内核文档对照
5. 实现层 vs 设计层对照
6. 跨文件引用校验
7. 头部声明 / 统计数据校验
8. 生成优先级清单 → P0（架构阻塞）→ P1（支撑缺失）→ P2（完备性）→ P3（规范性）
```

**四级优先级体系**：

| 级别 | 定义 | 响应策略 |
|------|------|---------|
| P0 | 架构级阻塞，影响产品正常运行 | 立即解决 |
| P1 | 支撑性缺失，影响关键决策 | 本迭代解决 |
| P2 | 完备性提升，锦上添花 | 择机解决 |
| P3 | 规范性修正，不影响功能 | 积累一批后统一处理 |

**迁移方式**：将对照对（如"Spec vs 洞察"、"HTML vs Spec"）替换为目标项目的核心文档类型。

---

### 1.3 文档 → 原型 → 对外声明（三层递进流水线）

**做法**：

```
第一步：文档先行      → 撰写完整规格文档，定义设计原则和边界
第二步：原型验证      → 根据规格文档构建可交互原型，反向修正设计
第三步：对外包装      → 从规格和原型中提炼对外声明（不做知识增量，只做认知压缩）
```

**核心原则**：三步之间有严格先后顺序——先定义原则，再验证假设，最后对外传播。颠倒顺序会导致"先做功能再想理由"。

**迁移方式**：适用于任何面向评审 / 路演 / 竞赛的产品项目，直接套用。

---

### 1.4 模块化开发 + 单文件交付

**做法**：

| 阶段 | 策略 | 目的 |
|------|------|------|
| 开发期 | 按职责拆分为独立文件（结构 / 样式 / 数据 / 逻辑） | 可维护性 |
| 交付期 | 合并为单个自包含文件（所有 CSS/JS 内联） | 独立性（零外部依赖） |

**迁移方式**：任何需要同时满足开发可维护和交付可独立运行的 Web 项目都适用。

---

### 1.5 原子化提交规范

**做法**：采用约定式提交（Conventional Commits），定义项目级作用域枚举（如 `spec` / `insights` / `html` / `agents` / `review`），每次提交仅包含一个逻辑变更。

**迁移方式**：任何采用 Git 管理的项目都适用，核心是将 Conventional Commits 的标准类型映射到项目特定的作用域集。

## 二、文档模板

### 2.1 产品规格文档 · 九节结构模板

设计逻辑：从内向外逐层展开——先定义"我们是什么"，再描述"我们做什么"，然后界定"边界在哪里"，最后阐述"如何生存"。

```markdown
# §一、产品定位        （1.1 定义 / 1.2 核心价值主张 / 1.3 核心承诺）
# §二、核心功能        （功能模块 + 设计原则）
# §三、交互与体验设计   （视觉语言 / 信息架构 / 动效）
# §四、核心内容/数据体系 （支撑产品的"原材料"）
# §五、用户生命周期设计  （留存 / 流失预警 / 推送）
# §六、合规与边界       （版权 / 数据 / 使用限制）
# §七、商业模式        （收入模型 / 竞争格局 / 市场风险）
# §八、技术架构        （前后端 / 部署 / 安全）
# §九、社会价值/项目意义 （为什么它值得存在）
```

**辅助说明**：
- 每节内部采用三级编号（如 1.1、1.2、1.3），使用 `§` 符号做节号前缀（`§一` 至 `§九`），便于跨文件引用时精确锚定
- 第九节可按项目性质替换为"项目意义""技术创新""教育价值"等

---

### 2.2 洞察条目 · 标准结构模板

**基础档**（适用于记录设计决策）：

```markdown
## 洞察{N}：{标题}

**来源**：[§{章节}或 {上下文说明}]

**核心内容**：

{1-3 段文字说明该决策是什么、为什么、边界在哪里}
```

**完整档**（适用于需要系统化展开的深度概念）：

```markdown
## 洞察{N}：{标题}

**来源**：[§{章节}]

**核心内容**：

{简短摘要}

---

### 一、{概念定位}
### 二、{哲学/理论根基}
### 三、{当前实现分析}
### 四、{核心机制}
### 五、{边界与约束}
### 六、{演进方向}
### 七、{战略位置/协同关系}
```

**切换标准**：需要提供完整操作手册 / 方法论体系的概念走完整档，仅需记录设计决策的概念走基础档。

**迁移方式**：将"来源"映射为设计上下文，将"核心内容"映射为设计决策与理由，将七节展开映射为深度论证。

---

### 2.3 对外声明（报名帖）· 四段结构模板

**设计哲学**：面向评委 / 合作方时，不是先介绍功能，而是先锚定需求、再交代凭什么是你、最后展示差异化。

```markdown
## 1. 创意名称 + 创意介绍
   - 想解决什么问题     → 锚定被忽视的真实需求
   - 为什么会想到做这个  → 交代凭什么是你（而非别人）
   - 大概是什么产品      → 一句话说清形态

## 2. 目标用户及痛点
   - 面向哪些用户        → 具体到人群画像
   - 在什么场景下使用    → 具体的"when"
   - 当前痛点（表格对比） → 现有方案 vs 核心痛点

## 3. 价值与意义
   - 社会/商业价值       → 为什么重要
   - 差异化亮点（列表）   → 每一条同时回答"是什么"和"为什么重要"
   
## 4. 交付物声明
   - 简洁收尾，不重复前文
```

**核心原则**：对外字数控制在 3,000 字以内，用表格替代大段文字，每条差异化亮点背后有一条可追溯的内部洞察。

**迁移方式**：任何面向评审 / 路演 / 竞赛的报名材料直接套用。

---

### 2.4 复盘报告 · 十节结构模板

**设计哲学**：用对照表发现问题，而非靠印象判断——Spec vs 洞察对照、对外声明 vs 洞察对照、实现 vs 设计对照。

```markdown
## 一、项目文件总览        （表格：文件/行数/用途/状态）
## 二、条目完整列表        （按层级分段）
## 三、未完成项与显式标记   （搜索 ⚠️ 等标记）
## 四、对外声明中未独立展开项（对外文档 vs 内核文档对照）
## 五、规格文档中未覆盖的设计点
## 六、实现层与设计层对照
## 七、文件引用与规范性问题
## 八、核心概念覆盖缺口
## 九、优先级行动清单        （P0-P3 四级）
## 十、总结
```

**迁移方式**：任何多文件项目都可复用此结构，只需替换具体的对照对。

### 2.5 AGENTS.md · 路由索引模板

**设计哲学**：不重复任何子模块的内容，仅提供"我该去哪里找什么"的索引。

```markdown
# AGENTS.md — {项目名} AI 协作指南

## 项目身份
{一句话定义 + 当前阶段}

## 文件地图
{树状目录 + 每文件一行注释说明其角色}
```

**迁移方式**：替换"项目身份"和"文件地图"，其余结构直接复用。

## 三、Agent 协作基础设施

### 3.1 目录结构

```
项目根目录/
├── AGENTS.md                      ← 路由索引（唯一入口）
└── .agents/                       ← 全部规范与文档
    ├── project.md                 ← 项目概述、核心概念词典
    ├── conventions.md             ← 命名规范、代码风格、文档格式
    ├── workflows.md               ← 标准工作流（触发条件 → 步骤 → 校验清单）
    ├── constraints.md             ← 编号禁令清单（含出处）
    ├── git.md                     ← Git 提交规范（可选）
    ├── docs/                      ← 设计文档归档
    │   └── specs/
    └── html/                      ← 开发资源（可选）
```

### 3.2 四模块职责边界

| 模块 | 角色 | 回答的问题 | 不做什么 |
|------|------|-----------|---------|
| `project.md` | 知识库 | 这是什么？核心概念是什么？ | 不写规范，不写工作流 |
| `conventions.md` | 规范手册 | 怎么命名？怎么写代码？ | 不解释为什么（那是 project 的事） |
| `workflows.md` | 操作手册 | 遇到 X 该走哪几步？ | 不重复概念定义 |
| `constraints.md` | 禁令清单 | 什么绝对不能做？ | 只写"不"，不写"要" |

### 3.3 工作流模板：触发条件 → 执行流程 → 校验清单

```markdown
## 工作流{编号}：{名称}

### 触发条件
- 用户说"X"
- 检测到 Y 状态

### 执行流程
1. {步骤一}：{具体操作}
2. {步骤二}：{具体操作}
...

### 校验清单
- [ ] 检查项 1
- [ ] 检查项 2
...
```

**迁移方式**：四模块结构直接复用。将 `project.md` 中的概念词典替换为目标项目的术语体系。workflows 按项目需要增删。

## 四、命名与文件组织规范

### 4.1 文件命名规则

| 用途 | 格式 | 示例 |
|------|------|------|
| 设计文档 | `{YYYY-MM-DD}-{project-slug}-{type}-{range}.md` | `2026-06-17-project-spec.md` |
| 对外交付物 | `{中文名}.html` 或 `{中文名}.md` | `产品展示.html` |
| Agent 入口 | `AGENTS.md`（全大写，根目录） | — |
| Agent 模块 | `.agents/{module}.md`（隐藏目录，小写） | `.agents/project.md` |

**设计原则**：日期前缀放在最前（确保自然时间排序），类型字段用固定枚举值（便于 Glob 搜索），对外文件用中文名（降低评委 / 用户的认知成本），内部文件用英文小写。

### 4.2 交叉引用格式

所有跨文件引用使用相对路径 + 行号定位：

```markdown
[可读名称]({相对路径}#L{起始行}-L{结束行})
```

- 同目录引用示例：`[spec §5.2](2026-06-17-zhujian-wudao-spec.md#L245-L263)`
- 上级目录引用示例：`[HTML](../../../../竹简悟道_完整版.html#L710-L751)`
- 同文件自引用：可省略文件名，直接写 `[洞察53](#L1364-L1487)`

**使用相对路径**——路径相对于当前文件所在目录，同一文档内自引用可省略文件名直接使用锚点。

### 4.3 编号管理原则

- 全局递增，不能跳号、插号
- 按层级分段（如 1-15 产品层、16-30 架构层、31+ 领域层）
- 标题不能同名（保证可搜索性）
- 内容不能高度重复（保证信息密度）

## 五、禁止事项清单方法论

### 推导链条

```
产品核心哲学（1-3 条）
  → 交互层禁令（从哲学推导出来的行为约束）
    → 产品功能禁令
    → 文档编写禁令
    → 代码/技术禁令
```

### 每一条禁令的格式

```markdown
| 编号 | 禁止事项 | 出处 |
|------|---------|------|
| C-01 | 不给予答案 | spec §3.2 |
```

**关键约束**：每条禁令必须有"出处"字段，确保它不是"我觉得不好"，而是"因为产品哲学决定了 X，所以不能做 Y"。当需要增删禁令时，判断标准是"它是否与产品哲学一致"。

### 迁移执行步骤

1. 定义你的产品的 1-3 条核心哲学（如"不做建议者""不制造焦虑"）
2. 从核心哲学推导出交互层面的禁令
3. 从交互禁令推导出产品功能禁令
4. 从产品禁令推导出文档编写禁令
5. 从文档禁令推导出代码 / 技术禁令
6. 每条标注出处，使用 C-{编号} 格式，确保可追溯

## 六、通用设计方法论

### 6.1 概念自繁殖定律

**定律**：任何一个概念被引入系统后，它会自动繁殖出它的对立面，两者又各自繁殖出新的概念对，无穷无尽。

```
概念 A → 对立面 ¬A → 新概念 B → 对立面 ¬B → ...
```

**设计启示**：好的产品设计不是"不断添加概念来覆盖所有情况"，而是"让概念在需要的时候自然浮现，不需要的时候不占据用户心智"。在添加任何一个新概念之前，先问：

> 这个概念会产生什么对立面？我是否需要为这个对立面再添加新概念？

### 6.2 概念解缚框架

**定义**：帮用户看清自己正在被哪个概念支配——不是推给他一个新概念替代旧概念，而是让他看到自己与概念之间的关系。

**通用步骤**：

```
1. 识别用户当前处于哪个概念框架下   → "我必须成功"
2. 帮助用户看清这个概念的来源       → 这个"必须"是谁说的？
3. 禁止提供替代概念                 → "你应该放下"只是换了一个框
4. 让用户在看清之后自己做选择       → 不替用户决定
```

**适用领域**：任何涉及"改变用户认知"的产品——教育、心理辅导、决策辅助、习惯养成。

### 6.3 "并列即冲突，共现即互补"命名法则

**法则**：当两套命名体系描述同一事物时：

| 呈现方式 | 效果 |
|---------|------|
| **并列**（让用户在 A/B 间做分类选择） | 制造冲突——"我到底属于哪一类？" |
| **共现**（A 为主名，B 为视觉注释） | 形成互补——"哦，你们说的这个就是我想说的那个" |

**UX 实践规则**：当系统内部使用命名体系 A，而用户更熟悉命名体系 B 时，不应让用户在二者之间做分类选择，而应将 B 降级为 A 的视觉注释（例如 A 为大字主标签，B 为灰色辅助小字）。

**适用领域**：任何存在"系统命名"和"用户语言"双轨的 UI 场景——标签系统、导航分类、筛选器、搜索提示。

## 七、前台-后台分离的文档撰写原则

### 7.1 公式化表达

| 层次 | 受众 | 目标 | 原则 |
|------|------|------|------|
| 后台文档 | 内部团队 | 完整、冗余、可执行 | 假设读者有时间深入理解 |
| 前台文档 | 外部受众 | 压缩、提炼、可感知 | 假设读者只有 3 分钟 |

### 7.2 两者关系

前后台之间不是"简化"，而是"认知压缩"——后台的每一条洞见到前台后变成一句感知层级的表述，但这一句背后有完整的逻辑链支撑。

### 7.3 必须建立的映射

前后台之间必须维护一个映射表，确保对外文案不是凭空编造：

```
前台亮点 1  → 后台洞察 {a,b,c}
前台亮点 2  → 后台洞察 {d,e}
...
```

### 迁移方式

适用场景：任何需要同时维护对内设计文档和对外声明的项目。

## 八、洞察库元分析方法

### 8.1 洞察分类学（按生成方式）

**做法**：当洞察库达到一定规模（建议30条以上）后，按**生成方式**而非**内容领域**对洞察进行分类，可揭示洞察库的真实结构。

**十类型分类框架**：

| 类型 | 定义 | 识别信号 |
|------|------|---------|
| 定位型 | 回答"产品是什么/不是什么" | 出现在项目早期，标题含"定位""差异化""承诺" |
| 哲学锚点型 | 提出核心概念作为后续洞察地基 | 被多条后续洞察引用 |
| 架构设计型 | 描述产品如何组织、如何交互 | 标题含"架构""结构""流程" |
| 命名型 | 为概念或功能命名 | 标题含"命名""称谓" |
| 方法应用型 | 将哲学概念映射到具体场景 | 标题含"在XX中的运用""XX实践" |
| 系统化操作手册型 | 七节完整操作体系 | 标题含"操作手册""系统化" |
| 矩阵组合型 | 两个维度的交叉组合 | 标题含"矩阵""×" |
| 内容差异型 | 对比两个版本/方案的差异 | 标题含"vs""差异" |
| 元洞察型 | 反思洞察过程本身 | 标题含"元""洞察的洞察" |
| 数据洞察型 | 基于数据/审计的发现 | 标题含"统计""审计" |

**迁移方式**：适用于任何需要建立洞察库的项目，帮助判断"洞察库处于哪个阶段"。

---

### 8.2 洞察库演化三阶段

**做法**：观察洞察编号与信息密度的关系，识别洞察库的演化阶段。

**三阶段模型**：

| 阶段 | 编号范围 | 特征 | 信息密度 |
|------|---------|------|---------|
| 描述期 | 1-15 | 回答"是什么" | 高→极高 |
| 概念展开期 | 16-48 | 每个概念独立展开 | 高→中→低（U形谷底） |
| 系统期+元期 | 49-N | 整合已有概念、自我反思 | 回升 |

**关键发现**：信息密度下降是正常的，不要在下降期停止写作——系统化手册需要足够的素材积累。

**概念完备线**：洞察库会在某一条达到"概念完备"——之后不再引入新概念，只组织和反思已有概念。竹简悟道的完备线是第46条。

**迁移方式**：用于判断"洞察库何时可以停止写作"——标准不是"写够N条"，而是"不再有新概念需要展开 + 所有核心概念都有操作手册 + 至少有1-2条元洞察"。

---

### 8.3 核心锚点识别法

**做法**：找出洞察库中"移除后会导致结构性受损"的锚点洞察。

**识别标准**：
- 被多条后续洞察引用
- 定义了核心概念或框架
- 位于关键分支点（如从描述期转向展开期）

**竹简悟道的5个锚点**：洞察1（概念解缚）、洞察8/35（玄同）、洞察15（体道链）、洞察26（四法×八景）、洞察46（概念自繁殖）。

**迁移方式**：用于快速理解一个陌生项目的核心逻辑——先读锚点洞察，再读展开洞察。

## 九、文档声明的熵增定律

### 9.1 问题定义

**现象**：文档头部的统计字段（行数、条目数、日期）会随着编辑自然过时，且过时速度与文档活跃度正相关。

**根本原因**：
1. 头部声明是人工维护的"副本"
2. 文件内容是自动增长的"原文"
3. 副本与原文之间存在"同步摩擦"

**熵增定律**：任何需要人工同步的字段，过时是必然，准确是偶然。

---

### 9.2 三策对抗方案

| 策略 | 做法 | 适用场景 | 效果 |
|------|------|---------|------|
| **第一策：勤更新** | 每次编辑后手动同步头部 | 小型项目、低频更新 | 治标不治本 |
| **第二策：自动化** | 用脚本在提交前自动更新 | 有 CI/CD 流水线 | 根治，但需要技术投入 |
| **第三策：移除变量** | 头部只保留定性信息，行数等变量移除或标注"仅供参考" | 任何项目 | 零维护成本 |

**竹简悟道的选择**：采用第三策——在 `project.md` 核心文件表中新增免责声明："行数为编辑时实测值，仅供量级参考。每次编辑后行数会自然增长，无需逐次同步。"

**迁移方式**：适用于任何需要维护统计数据的项目文档。

## 十、解缚决策法（通用决策方法论）

> 萃取来源：洞察63（定位解缚）+ 洞察65（解缚决策法）

### 10.1 决策僵局的通用结构

任何"原则 vs 需求"的僵局都有相同结构：

```
原则（"我应该……"）  vs  需求（"我需要……"）
     ↓                        ↓
  不可变通的铁律           当下的现实压力
     ↓                        ↓
  "违背原则就是背叛"      "坚持原则就是死板"
```

**僵局本质**：不是原则和需求之间的矛盾，而是**把手段当成了目的**——原则原本服务于某个目的，但时间久了，原则本身变成了不可质疑的目的。

### 10.2 四步流程

| 步骤 | 名称 | 核心操作 | 关键问题 |
|------|------|---------|---------|
| 一 | 目的考古 | 追问原则的原始目的 | "这个原则最初为了实现什么？" |
| 二 | 概念X光 | 拆解为本质属性 + 历史包袱 | "哪些是不可去的，哪些是历史惯性？" |
| 三 | 递进设计 | 把"A or B"改为"A→B"递进 | "能否从A的方向进入，在尽头转弯到B？" |
| 四 | 可逆验证 | 先设计退回机制再执行 | "如果错了，能退回吗？代价多大？" |

**三档可逆性**：

| 可逆性 | 特征 | 决策建议 |
|--------|------|---------|
| 高可逆 | 实验失败可无损退回 | 放手去做 |
| 中可逆 | 退回有一定成本但不致命 | 谨慎实验，控制范围 |
| 低可逆 | 退回代价极大或不可退 | 需要更强共识才执行 |

### 10.3 三个检验标准

1. **目的检验**：这个决策服务于什么目的？当前路径是唯一的实现方式吗？
2. **递进检验**：这个决策提供了递进路径吗？还是只做了非此即彼的选择？
3. **可逆检验**：如果这个决策是错的，能退回吗？退回代价是什么？

### 10.4 vs 传统决策框架

| 维度 | 传统决策 | 解缚决策法 |
|------|---------|-----------|
| 起点 | 列出选项，评估利弊 | 挖掘被遗忘的目的 |
| 矛盾处理 | 权衡取舍（trade-off） | 设计递进路径（不取舍） |
| 核心动作 | 在选项中选最优 | 质疑选项本身是否是虚假矛盾 |

### 10.5 适用边界

- **适用**：原则与需求矛盾、"保守 vs 激进"僵局、长期原则面临新场景
- **不适用**：纯事实判断、紧急危机响应、价值观根本冲突

**迁移方式**：将"原则"替换为目标场景中的任何长期坚持的规则或信念，将"需求"替换为当下的现实压力。四步流程和三检验标准不变。

## 十一、反效率产品设计框架

> 萃取来源：洞察64（反效率工具——产品本质定位）

### 11.1 效率陷阱诊断

效率工具的承诺是"帮你省时间"，但实际效果是"答案越多选择越困难"。效率工具把"想不清"当成了"效率不够"来治——方向错了。

**诊断公式**：效率工具解决"怎么做更快"，用户真正困境是"想不清为什么做"。

### 11.2 反效率 ≠ 低效率

| | 低效率 | 反效率 | 效率工具 |
|---|---|---|---|
| 定义 | 做事慢且差 | 刻意选择"慢"作为策略 | 追求"快"作为目标 |
| 意图 | 无意识的慢 | 有意识地拒绝"快=好" | 有意识地追求"快=好" |

**关键区分**：反效率是"我知道可以更快，但选择慢下来，因为慢下来才能看到快的时候看不到的东西"。

### 11.3 三层结构

| 层级 | 核心 | 产品体现 | 用户感受 |
|------|------|---------|---------|
| 慢 | 拒绝速度崇拜 | 刻意延迟响应、每日限量推送 | "它不急着给我答案，这让我也不急了" |
| 深 | 拒绝表面解答 | 不给表面答案，追问问题背后的问题 | "它没告诉我怎么做，但我看到了真正的问题" |
| 静 | 拒绝信息填充 | 留白本身就是产品，不填充注意力 | "打开它，我就从信息洪流里上岸了" |

### 11.4 五条产品设计原则

| 原则 | 内容 | 与效率工具的对比 |
|------|------|----------------|
| 克制原则 | 每次交互只给一个内容单元，不堆砌 | 效率工具追求"一屏多任务" |
| 延迟原则 | 刻意的等待时间，不追求即时 | 效率工具追求"0延迟" |
| 不可搜索原则 | 按顺序浏览，让用户"遇见"而非"检索" | 效率工具追求"秒搜" |
| 无进度原则 | 不显示进度条，不做可量化进度 | 效率工具追求"可量化的进度" |
| 可遗忘原则 | 不要求用户记住任何东西 | 效率工具追求"可记忆/可复习" |

### 11.5 反度量体系

| 效率指标（不适用） | 反效率指标（适用） | 含义 |
|------------------|------------------|------|
| DAU（日活） | 回访间隔分布 | 不追求每天来，追求"需要时自然回来" |
| 会话时长 | 单次对话深度 | 不追求停留久，追求"走了之后想到了什么" |
| 转化率 | "主动离开"率 | 用户想清楚了主动结束——这是成功不是流失 |
| NPS（推荐值） | "不推荐但记得"率 | 用户可能不推荐给所有人，但会记得某个问题 |

**核心反度量**："想清楚率"——用户在对话结束后，是否从"想不清"变成"想清楚了"。

### 11.6 "以快入慢"漏斗

```
用户带着"快"的需求来
    ↓
解惑（满足效率需求）—— 策略性让步
    ↓
过渡标记（减速带）—— 在效率的终点放一个缝隙
    ↓
问道（进入反效率）—— 产品的灵魂
```

**设计要求**：解惑是入口，问道是深化——不是并列，而是递进。两者之间有明确的过渡标记，用户知道"现在从解惑进入问道"。

### 11.7 商业生存三层策略

1. **不争之争**：不与效率工具竞争"快"，占据"慢思考"无人竞争的生态位
2. **效率世界的反效率需求**：效率越高，人对"反效率"的需求越强（"反者道之动"的市场版）
3. **付费意愿的反逻辑**：不为"省时间"付费，为"值得花的时间"付费

**迁移方式**：将"反效率"替换为目标产品的差异化方向（如"反消费""反社交""反即时"等），五条设计原则和反度量体系可适配后直接使用。

## 十二、反效率留存策略——"完成型产品"的留存悖论与解法

> 萃取来源：洞察67（定期回顾与流失预警）

### 12.1 留存悖论

当产品的成功标准包含"用户想清楚了主动离开"时（如反效率工具、心理咨询、教育产品），传统的"留住用户"逻辑失效——"不来了"可能意味着成功而非流失。

### 12.2 三种"不来了"的区分框架

| 类型 | 内在状态 | 判断依据 | 产品回应 |
|------|---------|---------|---------|
| **完成了不来了** | 目标达成，自然停止 | 深度交互后自然结束 | 不打扰 |
| **放弃了不来了** | 目标未达，觉得没用 | 浅层交互后离开 | 温和召回（用问题而非催促） |
| **遗忘了不来了** | 无明确判断 | 超长周期未登录 | 极弱提醒（仅一次，不重复） |

**迁移方式**：将"完成"替换为目标产品的成功状态（如"学会""治愈""想通"），三种类型的判断逻辑不变。

### 12.3 路径回顾而非内容回顾

定期回顾展示用户的**行为路径**（去了哪里、做了什么），不展示**行为内容**（说了什么、写了什么）。

| 展示内容 | 效果 | 适用场景 |
|---------|------|---------|
| 展示内容 | 用户评判过去的自己 | 需要详细复盘的专业场景 |
| 只展示路径 | 用户看见自己的模式 | 反思型/成长型产品 |

**三层回顾设计**：路径回顾（看见模式）→ 模式辨认（看见自己）→ 简短反思（一个开放式问题，不要求回答）。

### 12.4 "不争之争"召回策略

召回不是"争用户回来"，而是在用户需要时出现、不需要时沉默。

| 预警档位 | 触发条件 | 推送原则 | 度量标准 |
|---------|---------|---------|---------|
| 浅层活跃 | 只浏览不深入 | 给入口，不追问原因 | 不是点击率，是后续深度 |
| 沉默期 | 短期未登录 | 用"问题"召回，不用"提醒" | 不是召回率，是召回后深度 |
| 流失风险 | 长期未登录 | 只表达"在"，不解释不诱导 | 不重复推送 |

**核心反度量**：召回的成功标准不是"召回率"（效率指标），而是"召回后的交互深度"（反效率指标）。

**迁移方式**：适用于任何"完成型产品"——教育、心理咨询、反思工具、冥想APP等。将"深度交互"替换为目标产品的核心价值行为。

## 十三、时间哲学场景化——自然节律到产品模式的映射

> 萃取来源：洞察68（睡前静心）

### 13.1 核心模式：自然节律 → 产品模式

不是为产品功能设定人为时间（如"每天打卡""定时提醒"），而是从自然节律中找到产品模式的对应——让产品节奏顺应人的自然节奏，而非强制人适应产品节奏。

| 自然节律 | 产品模式 | 设计原则 |
|---------|---------|---------|
| 早晨（苏醒/开始） | "吸"——一个问题/目标带入一天 | 轻量入口，不压迫 |
| 夜晚（休息/结束） | "呼"——一个反思归回根本 | 留白收束，不填充 |
| 周末（停顿/回顾） | 路径回顾，不内容回顾 | 减法，不加法 |
| 季节/周期 | 模式辨认，不进度量化 | 看见模式，不评判进度 |

### 13.2 时段模式切换

产品功能不通过用户手动切换，而是根据时间自动切换模式。

| 设计要素 | 说明 |
|---------|------|
| 触发条件 | 特定时段（如22:00-02:00）自动切换 |
| 切换信号 | AI的第一句话从"选择场景"变为"今天有什么事还在脑子里转？" |
| 用户感知 | 不需要知道"切换了"，只感觉"这个时间打开它，它问的问题不一样了" |
| 可逆性 | 用户可以主动退出该模式，回到常规模式 |

**迁移方式**：适用于任何有"时段属性"的产品——早晨的健身APP、夜晚的阅读工具、周末的复盘应用等。关键是找到"自然节律→产品模式"的映射，而非"产品功能→人为定时"的强制。

### 13.3 "留白即产品"的极致形态

在场景的收束层（如睡前场景的第三层），产品不做任何信息输出——只展示一句与用户状态相关的内容，然后沉默，屏幕渐暗。

| 层级 | 产品行为 | 用户感受 |
|------|---------|---------|
| 入口层 | 提问引导 | "被看见了" |
| 深入层 | 追问根本 | "想通了" |
| 收束层 | 一句话 + 沉默 + 渐暗 | "可以了" |

**收束层的三个"不"**：不追问、不总结、不填充。沉默本身就是产品功能。

**迁移方式**：适用于任何"有明确收束点"的产品场景——冥想的结束、咨询的尾声、学习的总结。关键是：收束不是"做完了一件事"，而是"放下了一件事"。
