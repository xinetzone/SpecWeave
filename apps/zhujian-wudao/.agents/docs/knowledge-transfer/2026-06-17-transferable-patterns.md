## 一、.agents/ 模块化项目骨架

### 1.1 核心思想

传统做法是写一个 `README.md` 放所有说明，或用 Notion/Wiki 维护项目知识——但 AI 协作环境下，问题变了：AI 需要在**上下文窗口有限**的情况下快速定位到正确的规范文件。一个 300 行的 AGENTS.md 不如一个 40 行的路由索引 + 按需加载的 6 个子模块。

### 1.2 目录结构模板

```
项目根/
├── AGENTS.md                        ← 唯一入口，AI 首先读它
├── .agents/                         ← 全部规范（隐藏目录，不干扰业务文件）
│   ├── project.md                   ← 项目概述、概念词典、当前阶段
│   ├── conventions.md               ← 命名、代码风格、文档格式规范
│   ├── workflows.md                 ← 标准操作流程（SOP）
│   ├── constraints.md               ← 禁止事项清单（反模式）
│   ├── git.md                       ← Git 约定式提交规范（可选）
│   └── docs/                        ← 深度文档（规格书、洞察库、复盘报告）
│       └── {子目录}/
│           ├── ...-spec.md
│           ├── ...-insights-N-M.md
│           └── ...-review.md
```

### 1.3 AGENTS.md 模板

```markdown
# AGENTS.md — {项目名} AI 协作入口

> 本文档是路由索引，不包含具体规范。根据需求选择性阅读子模块。

## 项目身份

**{项目名}**：{一句话定位}。{核心理念}。

当前阶段：{阶段描述}。

## 文件地图

```
项目根/
├── AGENTS.md                      ← 本文件（路由索引）
├── .agents/                       ← 全部项目规范与文档
│   ├── project.md                 # 项目概述、核心概念词典
│   ├── conventions.md             # 文件命名、代码风格、文档规范
│   ├── workflows.md               # 标准工作流
│   ├── constraints.md             # 禁止事项与约束清单
│   ├── git.md                     # Git 约定式提交规范
│   └── docs/.../
├── {入口文件}.html                 # 主程序入口
└── {其他业务文件}
```

## 路由索引

| 需要了解 | 阅读文件 |
|---------|---------|
| 项目整体定位 | [project.md](file:///{绝对路径}/.agents/project.md) |
| 代码规范 | [conventions.md](file:///{绝对路径}/.agents/conventions.md) |
| 操作流程 | [workflows.md](file:///{绝对路径}/.agents/workflows.md) |
| 什么不能做 | [constraints.md](file:///{绝对路径}/.agents/constraints.md) |
| Git 规范 | [git.md](file:///{绝对路径}/.agents/git.md) |
```

**关键原则**：
- 路由表用 `file://` 绝对路径，AI 可一键跳转
- AGENTS.md 控制在 60 行以内——越长越不会被读完
- 子模块按需加载，不一次性塞进上下文

## 二、洞察驱动开发方法论

### 2.1 核心思想

传统开发：需求 → 设计 → 代码 → 测试（代码先行）
洞察驱动：**洞察 → Spec → 原型 → 交付物的三层递进**

洞察的定位不是"事后总结"而是"事前决策依据"。每条洞察回答一个问题："为什么这个设计决策是对的？"

### 2.2 洞察库三层结构

```
产品层（Product Layer）    · 回答"用户为什么要用"
架构层（Architecture Layer）· 回答"系统怎么支撑产品承诺"
哲学层（Philosophy Layer）  · 回答"设计背后的底层信念从哪来"
```

| 层级 | 编号段 | 典型问题 | 示例 |
|------|--------|---------|------|
| 产品层 | 1-N | 用户痛点、差异化、竞品、留存 | "为什么不做知识付费赛道" |
| 架构层 | N+1-M | 对话引擎、数据模型、模块划分 | "为什么前台极简后台极深" |
| 哲学层 | M+1+ | 底层信念、伦理立场、本体论前提 | "概念解缚"的帛书渊源 |

### 2.3 洞察标准结构

```markdown
## 洞察{N}：{标题}

**来源**：[{出处}]

**核心内容**：{一段话概括}

---

### 一、{第一节}
### 二、{第二节}
...
### 七、{第七节}
```

**规则**：
- 全局递增编号，不跳号不插号
- 每文件头部声明洞察数、行数——每次编辑后必须更新
- 体道四法类系统化手册必须七节完整结构；非系统化洞察可简化为"来源+核心内容"

### 2.4 何时写洞察 vs 何时不写

| 写 | 不写 |
|----|------|
| 产品方向性决策（"为什么不做X"） | 执行层操作（"打包了文件"） |
| 可复用的设计模式或法则 | 措辞精修（"深夜→慢慢"） |
| 两个已有洞察之间发现的新连接 | 环境清理（"删了 npm"） |
| 复盘报告中标记的 P0/P1 缺口 | 过程记录（"生成了复盘文档"） |

## 三、滚动复盘流程模板

### 3.1 核心思想

不是"做完再检查"——而是在每次阶段性变更后立即做**全项目快照审计**。频率：每 2-3 次变更一轮。

### 3.2 标准化 8 步流程

```
1. 统计全部文件行数与状态
2. 搜索全项目 ⚠️ 标记
3. 校验所有头部声明 vs 实测数据
4. Spec ↔ 洞察库 ↔ 原型 三向对照
5. 跨文件 file:// 引用审计
6. 识别新产生的不一致
7. 生成优先级清单（P0-P3）
8. 判断是否需要新洞察
```

### 3.3 复盘报告模板

```markdown
# {项目名}全面复盘报告

> 复盘日期：{日期}（第{N}轮） | 覆盖范围：{M}个文件，共{X}行 | 洞察总数：{Y}条

## 一、文件全貌

| 文件 | 行数 | 用途 | 状态 |
|------|------|------|------|
| ... | ... | ... | ... |

## 二、⚠️ 标记分析

| 编号 | 位置 | 内容 | 状态 |
|------|------|------|------|
| ... | ... | ... | ... |

## 三、头部声明校验

| 文件 | 声明值 | 实测值 | 偏差 |
|------|--------|--------|------|

## 四、三向对照（Spec ↔ 洞察 ↔ 原型）

## 五、跨文件引用审计

## 六、优先级行动清单

| 编号 | 行动项 | 严重度 | 状态 |
|------|--------|--------|------|
| P0-1 | ... | P0 | 未解决 |

> P0 = 架构级阻塞（不解决就无法继续）
> P1 = 支撑性缺失（缺少关键洞察或设计）
> P2 = 完备性提升（有比没有好）
> P3 = 规范性修正（数字/格式不一致）
```

### 3.4 ⚠️ 标记管理

⚠️ 是项目内部的"技术债标记"。管理原则：
- 只在 Markdown 文件中使用，不在代码中使用
- 每次复盘逐条过一遍，已解决的立即移除或标记为 ✅
- 最终目标是全项目 0 处 ⚠️

## 四、文件同步规则

### 4.1 修改洞察库后必须同步

| 操作 | 需更新的位置 |
|------|-------------|
| 新增洞察 | 头部统计(洞察数+1/行数更新) + 末尾统计 + 复盘报告洞察列表 |
| 删除洞察 | 同上 + 检查所有交叉引用 |
| 修改标题 | 检查复盘报告是否一致 |
| 添加交叉引用 | 确认目标路径有效 |

### 4.2 修改 Spec 后必须检查

| 操作 | 需检查 |
|------|--------|
| 新增设计点 | 是否需新洞察 |
| 修改概念定义 | 洞察中引用是否需同步 |
| 修改节号 | 所有引用链接是否失效 |

### 4.3 修改原型后必须检查

| 操作 | 需检查 |
|------|--------|
| 新增功能 | Spec 是否需同步 |
| 修改数据结构 | 洞察覆盖分析是否需更新 |

## 五、比赛/提交材料标准化流程

### 5.1 三层递进模型

```
Spec / 洞察库         ← 第 1 层：设计原则（对内）
    ↓
HTML 原型 + 完整版     ← 第 2 层：可运行验证（对内+对外）
    ↓
报名帖 + 展示页        ← 第 3 层：对外传播（评委/用户）
```

### 5.2 标准交付清单模板

| 序号 | 交付物 | 格式 | 定位 |
|------|--------|------|------|
| D1 | 报名帖 | Markdown | 大赛报名正文 |
| D2 | 展示页 | HTML (自包含) | 落地页叙事 |
| D3 | 完整版原型 | HTML (自包含) | 可运行 Demo |
| D4 | Spec | Markdown | 设计权威文档 |
| D5 | 洞察库 | Markdown (×2) | 决策依据库 |

### 5.3 报名帖四段式结构

```markdown
# 【赛道A + 赛道B】项目名 —— 一句话口号

## 1. 创意名称 + 创意介绍
- 想解决什么问题
- 为什么会想到做这个
- 大概是什么产品
- 初赛 Demo 范围

## 2. 目标用户及痛点
- 面向哪些用户
- 在什么场景下使用
- 当前痛点（表格：现有方案 vs 痛点）

## 3. 价值与意义
- 社会价值
- 商业化可能
- 差异化亮点

## 4. 创意产物
- HTML 展示说明
```

### 5.4 展示页叙事弧模板

```
Hero → 痛点 → 解决方案 → 核心功能 → 效果预览 → 用户画像 → 对比 → 价值 → CTA
```

### 5.5 提交前检查清单

| 检查项 | 
|--------|
| 报名帖赛道标签正确 |
| HTML 原型可独立运行 |
| 展示页视觉完整 |
| 所有文件版本一致性（术语、slogan、标签名统一） |
| 文件可正常打开 |

## 六、单文件打包 vs 模块化开发策略

### 6.1 核心思想

**开发阶段模块化**（可维护）→ **交付阶段自包含**（可分发）。两者之间通过一个生成脚本或手动合并切换。

### 6.2 模块化结构（开发时）

```
{入口}.html           ← HTML 骨架（~80行，只含结构 + 外部引用）
{assets}/
├── styles.css        ← 全部 CSS
├── data.js           ← 全部数据（数组/对象）
└── app.js            ← 全部逻辑（IIFE）
```

### 6.3 单文件结构（交付时）

```
{入口}_完整版.html    ← 所有 CSS/JS 内联（~1000行，零依赖）
```

转换规则：
1. `<link rel="stylesheet" href="...">` → `<style>` 标签嵌入
2. `<script src="data.js">` → `<script>` 标签嵌入数据
3. `<script src="app.js">` → `<script>` 标签嵌入逻辑

### 6.4 适用场景

| 场景 | 用什么 |
|------|--------|
| 日常开发迭代 | 模块化（4 文件） |
| 比赛提交 | 单文件（1 文件） |
| 部署到 CDN | 单文件 |
| 给非技术人员测试 | 单文件 |

## 七、约束清单反模式模板

### 7.1 核心思想

不是"告诉我应该怎么做"——而是"告诉我绝对不能做什么"。约束清单的价值在于**防止退化**：当项目迭代中有人想加一个看起来无害的功能时，约束清单就是最后一道闸门。

### 7.2 约束清单分类模板

```markdown
## {产品层 | 文档层 | 代码层 | 哲学立场}禁令

| 编号 | 禁止事项 | 说明 / 出处 |
|------|---------|------------|
| C-01 | ...     | ...        |
```

### 7.3 标准约束类别

| 层级 | 典型约束 |
|------|---------|
| 产品层 | 不做什么功能、不对用户说什么话、不引入什么机制 |
| 文档编写 | 编号规则、文件命名一致性、引用格式 |
| 代码/技术 | 不暴露的复杂性、不依赖的库、兼容性底线 |
| 哲学立场 | 不采取的价值判断、不预设的立场 |

## 九、本文件使用指南

### 9.1 迁移到新项目时的操作顺序

1. **创建 .agents/ 骨架**：复制 §一 的目录结构，创建 AGENTS.md + 4 个核心 .md
2. **填充 project.md**：用 §八 的模板编写项目概念词典
3. **建立洞察库**：用 §二 的结构开始写前 5-10 条产品层洞察
4. **设置约束清单**：用 §七 的模板定义"绝对不能做什么"
5. **初始化复盘机制**：用 §三 的流程跑第一轮全项目审计
6. **规范命名与文档格式**：用 §四 的同步规则建立一致性保障

### 9.2 何时不需要这套体系

在以下情况下，这套体系是过度设计：
- 项目规模 < 3 个文件
- 单人一次性脚本，无迭代计划
- 不需要与 AI 协作（纯手写代码）

### 9.3 最小可行骨架

如果觉得全套太重，至少保留这三个：

```
.agents/
├── project.md        ← 概念词典（2-3 个核心概念）
├── constraints.md    ← 3-5 条铁律（"绝对不能加打卡功能"）
└── workflows.md      ← 1 个 SOP（"怎么写新洞察"）
```
