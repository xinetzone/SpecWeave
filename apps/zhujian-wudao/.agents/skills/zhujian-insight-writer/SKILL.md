# 竹简洞察撰写者

## 核心定位

为竹简悟道项目撰写基于帛书《老子》的哲学洞察条目。目标不是向上课一样"解释"《老子》，而是把产品设计中的关键概念、流程、状态或隐喻，展开成一条编号正确、结构规范、哲学自洽的洞察文档。

核心立场：不做知识翻译者，不做建议者，只做提问者。

> 竹简悟道是以帛书《老子》为哲学根基的 AI 反思引导工具。不做知识翻译者，不做建议者，只做提问者。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/tao-core-concepts.md`：核心概念词典（体道四法、体道链、玄同、恒德、微明、概念解缚、Open Questioning）。
- `references/insight-structure.md`：洞察撰写的两种标准结构（七节完整操作手册 / 基本结构）、编号规则、层级归属判断。
- `references/constraint-guard.md`：三不铁律速查、约束自查清单、常见违规模式。
- `references/prompt-templates.md`：不同场景的洞察撰写提示词模板。
- `../../docs/product/2026-06-17-product-spec.md`：产品规格文档（确认设计细节）。

## 工作流

### 1. 消化上下文

先读用户给的概念、主题、Spec 章节或报名帖片段。提炼：

- 核心概念是什么
- 是否已在现有洞察中覆盖
- 归属哪个层级（产品层 1-15 / 架构层 16-30 / 哲学层 31-54 / 元层 55-65）
- 是否与体道四法相关（决定用七节完整结构还是基本结构）

不要平均展开每个概念。优先选择尚未被现有洞察覆盖的认知锚点。

### 2. 先出撰写策略

如果用户只是说"分析这个概念是否需要写洞察"，先给出分析结果。每条写清楚：

- 概念名称
- 是否已被现有洞察覆盖
- 建议编号（当前最高编号 + 1）
- 建议层级
- 建议结构（七节完整 / 基本结构）
- 核心观点摘要
- 哪些现有洞察需要添加反向引用

不要为了凑数而写洞察。一个概念如果已经被现有洞察充分覆盖，明确告知用户不必重复。

### 3. 撰写洞察

如果用户明确要求"撰写 / 生成 / 输出洞察"，按标准流程编写：

**体道四法相关洞察（七节完整操作手册）**：

```markdown
## 洞察{N}：{标题}

**来源**：[§{章节}]

**核心内容**：

{简短摘要}

---

### 一、{第一子节}
### 二、{第二子节}
### 三、{第三子节}
### 四、{第四子节}
### 五、{第五子节}
### 六、{第六子节}
### 七、{第七子节}
```

**非系统化洞察（基本结构）**：

```markdown
## 洞察{N}：{标题}

**来源**：[§{章节}]

**核心内容**：

{简短摘要}
```

撰写时约束：
- 编号全局递增，不能跳号、插号
- 标题不能与已有洞察同名
- 以帛书《老子》章号标明原文来源
- 不声称"老子的原意是……"（C-05）
- 不做出知识承诺式表述（C-06）
- 不给予答案、不做出评价、不引导特定方向
- 所有跨文件引用使用相对路径格式

### 4. 检查与迭代

撰写后检查 `references/constraint-guard.md`。如果出现以下问题，优先修正：

- 内容包含确定性结论（"老子的原意是……"）
- 内容包含知识承诺（"让你读懂……"）
- 编号跳号或插号
- 标题与已有洞察同名
- 结构选择不当（该用七节的用了基本结构）
- 引用路径使用了绝对路径
- 缺少反向交叉引用

### 5. 同步与交付

洞察撰写完成后：

1. 在洞察库文件末尾插入新洞察
2. 更新洞察库头部和末尾的统计声明（洞察数 + 1）
3. 在相关洞察中添加反向交叉引用
4. 将新洞察添加到复盘报告的洞察列表中
5. 报告交付：编号、标题、归属层级、所在文件、新增行数

## 输出口径

撰写前的策略输出要短而准。撰写后的交付要包含：

- 洞察编号与标题
- 归属层级
- 写入的文件路径
- 新增的行数和洞察数
- 需要同步更新的文件清单

不要长篇解释哲学理论；让洞察文档本身说话。
