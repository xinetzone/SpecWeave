# ACT-004 发布摘要真实示例与平台语义映射轻量抽象设计文档

## 1. 背景

目前已完成两轮与 `.xlsx` 测试报告智能体化相关的能力增强：

- ACT-002：引入基于规则词典的风险语义聚类，生成 `risk_clusters` 与 `risk_cluster_details`
- ACT-003：新增发布判断摘要模板，并进一步补充“平台语义映射”与“平台影响面”

现阶段脚本已经能输出带有 `平台影响面` 的一页式发布摘要，但还缺两个关键闭环：

1. 缺少基于真实样本 `d:\AI\.temp\【20260327】单目1M插值3M232测试报告.xlsx` 的正式摘要产物，无法直接验证模板在真实数据下的呈现效果
2. 平台语义映射仍以内联常量形式维护，虽然当前可用，但可读性和后续扩展性一般，不利于未来按设备类型或风险标签做增量复用

因此，本轮目标不是继续扩张功能，而是在现有能力之上完成一个“先验证真实产物，再做轻量结构整理”的最小闭环。

## 2. 目标与非目标

### 2.1 目标

- 基于真实 `xlsx` 样本生成正式 `summary.md` 产物，验证摘要模板中 `平台影响面` 的实际可读性
- 为真实摘要产物建立最小可回归验证，避免后续模板或脚本修改破坏输出结构
- 对现有平台语义映射做轻量抽象，提升配置可读性与后续扩展性
- 保持当前发布判断、风险聚类和模板结构行为兼容

### 2.2 非目标

- 不修改 `DI`、严重问题数等发布门槛判定口径
- 不新增新的风险类别或新的文本解析链路
- 不把脚本升级为独立 Skill
- 不引入设备类型多态、动态配置加载或外部配置文件
- 不在本轮扩展到除 `camera` 之外的新设备品类

## 3. 方案比较

### 3.1 方案 A：先示例，后抽象（推荐）

做法：

- 先用真实 `xlsx` 生成正式 `summary.md`
- 验证摘要中 `平台影响面` 的信息密度与可读性
- 再对平台语义映射做轻量抽象，保持行为不变

优点：

- 先验证真实价值，再做结构整理，返工风险最低
- 更符合“最小闭环”原则
- 如果真实产物暴露信息过载或字段不足，可以在抽象前及时修正

缺点：

- 需要分两次小改动，而不是一次性完成

### 3.2 方案 B：先抽象，后示例

做法：

- 先整理平台语义映射结构
- 再用真实样本生成摘要产物

优点：

- 代码结构会更早变干净

缺点：

- 若真实摘要展示效果不佳，可能需要回头调整抽象设计
- 先优化结构、后验证价值，闭环顺序不够自然

### 3.3 方案 C：一次性同时完成

做法：

- 同一轮同时改示例产物、测试、抽象结构

优点：

- 表面上“一步到位”

缺点：

- 变更点更多，验证面更大
- 更难区分是“展示问题”还是“结构问题”导致输出不理想

### 3.4 推荐理由

采用方案 A。当前最重要的是先用真实样本确认摘要产物是否真正适合作为决策与联调输入，再对映射层做轻量抽象。这样既不耽误验证价值，也避免过早设计。

## 5. 分阶段实施设计

### 5.1 阶段一：生成真实示例摘要

基于：

- `d:\AI\.temp\【20260327】单目1M插值3M232测试报告.xlsx`

生成正式 `summary.md`，验证以下内容：

- `结论摘要` 是否足够支撑快速决策
- `Top 风险` 与 `阻塞项` 是否与现有复盘结论一致
- `平台影响面` 是否能在一页摘要中提供恰到好处的观察面提示
- `复测建议` 是否可直接给人工测试/联调复用

#### 阶段一输出要求

- 真实摘要文件必须是正式 Markdown 产物，而不是临时终端输出
- 产物内容应包含现有模板的全部固定章节
- 若真实数据下 `平台影响面` 过长或过空，应以最小修正方式调整，而不是重做摘要结构

### 5.2 阶段二：平台语义映射轻量抽象

当前平台语义映射已经存在，但主要以固定常量形式内嵌在脚本中。本轮只做轻量整理，不改变行为。

#### 抽象目标

- 让风险标签到平台观察面的关系更容易读懂
- 让后续新增风险标签或设备类型时，能在同一结构中按约定补充
- 避免逻辑和配置混在一起，降低后续修改成本

#### 抽象边界

- 仍保留内置配置，不拆成外部 YAML / JSON
- 不引入新的 CLI 参数
- 不新增运行时动态装载能力
- 不扩展 `camera` 之外的新实体范围

#### 推荐结构

可将当前平台语义映射整理为统一字段结构，例如：

```python
{
    "弱网": {
        "ha_domain": "tuya",
        "entity_scope": "camera",
        "integration_traits": ("hub", "cloud_push"),
        "observation_surfaces": ("status", "function", "diagnostics"),
        "diagnostic_focus": "..."
    }
}
```

重点不是引入新字段，而是统一字段定义、减少分散常量和后续阅读障碍。

## 6. 数据与渲染行为约束

### 6.1 行为兼容

- `extract_report_context()` 输出结构继续保持兼容
- `platform_semantic_mapping` 字段继续存在
- `render_release_summary()` 的摘要章节顺序不变

### 6.2 一页式摘要的信息密度控制

`平台影响面` 在摘要里只保留最小必要信息：

- 风险标签
- 域/实体范围
- 集成特征
- 关键观察面

不在摘要中直接展开完整诊断说明段落，避免一页摘要信息过载。

### 6.3 真实产物优先原则

若抽象结构与真实摘要展示效果冲突，以“真实摘要可读性”优先，必要时允许保持局部重复，而不是为了结构纯度牺牲产物质量。

## 7. 测试策略

### 7.1 测试重点

本轮测试只覆盖两个核心问题：

1. 真实摘要产物是否能稳定生成，并包含关键章节
2. 平台语义映射抽象后，摘要输出内容是否保持不变或仅有预期内变化

### 7.2 建议测试项

- 真实样本生成摘要文件成功
- 摘要文件中包含：
  - `结论摘要`
  - `Top 风险`
  - `阻塞项`
  - `平台影响面`
  - `复测建议`
- `平台影响面` 中包含至少一个真实风险标签与对应观察面
- `.agents/scripts/tests/test_analyze_xlsx_test_report.py` 全部通过

### 7.3 回归重点

- 不影响全量学习报告生成
- 不影响 `--summary-output` / `--summary-only` 现有行为
- 不影响 `risk_clusters` 与 `platform_semantic_mapping` 的既有消费方式

## 8. 验收标准

满足以下全部条件即可验收：

1. 基于 `【20260327】单目1M插值3M232测试报告.xlsx` 成功生成正式 `summary.md`
2. 真实摘要中包含 `平台影响面`，且内容可读、不明显过载
3. 平台语义映射完成轻量抽象，但摘要行为保持兼容
4. `.agents/scripts/tests/test_analyze_xlsx_test_report.py` 全部通过
5. 相关文件无新增诊断错误

## 9. 风险与应对

### 9.1 风险：真实摘要信息过载

应对：

- 仅压缩 `平台影响面` 的展示层，不扩大或改写底层上下文字段

### 9.2 风险：抽象过度

应对：

- 只做结构整理，不做能力扩张
- 若抽象会带来额外分支和配置复杂度，则优先保持当前实现

### 9.3 风险：真实样本路径依赖过强

应对：

- 把真实样本产物作为示例和验证物，不把该路径硬编码为脚本默认输入

## 10. 兼容性与边界

- 本轮本质上是“产物验证 + 配置整理”，不是新功能扩张
- 不改变现有项目对 Tuya IPC 发布判断的既有标准
- 不把平台语义映射升级成完整领域模型
- 若后续需要支持更多设备类型，应另开独立设计文档，不在本轮顺带完成
