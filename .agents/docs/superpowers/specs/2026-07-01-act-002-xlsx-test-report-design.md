# ACT-002 Excel 测试报告解析与 Markdown 导出设计文档

## 1. 背景

在上一轮任务中，已经完成过一次对 `d:\AI\.temp\【20260327】单目1M插值3M232测试报告.xlsx` 的人工学习与 Markdown 导出。该任务验证了一个明确需求：面对 `.xlsx` 格式的测试报告，当前仓库缺少可复用的解析骨架与固定输出模板，导致每次都需要临时编写解析命令、手工组织报告结构，重复劳动较多。

ACT-002 的目标不是做一个“通用 Excel 平台”，而是为“测试报告学习 + 导出 Markdown”这个高频场景，落地一个最小可复用闭环。

## 2. 目标与非目标

### 2.1 目标

本设计要交付一套轻量、可复用、可验证的最小闭环，满足以下目标：

1. 输入一个 `.xlsx` 测试报告路径，能稳定运行
2. 自动提取工作簿结构、总表指标、状态分布、失败与风险证据
3. 生成一份结构化 Markdown 报告
4. 固定报告章节结构，减少后续手工编排
5. 在“总表不标准”时提供降级输出，而不是整体失败

### 2.2 非目标

本次明确不做以下内容：

- 不做 GUI 界面
- 不做面向任意 Excel 业务场景的通用解析器
- 不做图表渲染
- 不做 Jira/缺陷平台同步
- 不把 JSON 作为正式交付物
- 不扩展到多模板体系

## 3. 方案选择

### 3.1 备选方案

#### 方案 A：解析脚本 + Markdown 模板（推荐）

产出：

- 一个 `.xlsx` 解析脚本骨架
- 一份固定 Markdown 报告模板

优点：

- 最贴合当前需求
- 轻量、低摩擦、容易验证
- 既沉淀脚本，又沉淀输出结构

缺点：

- 暂不输出结构化 JSON

#### 方案 B：解析脚本 + 中间数据模型 + Markdown 模板

优点：

- 为未来 JSON、仪表盘、更多渲染方式留出更强扩展性

缺点：

- 本轮实现偏重
- 会引入暂时不需要的抽象

#### 方案 C：模板优先，脚本最小化

优点：

- 上手最快

缺点：

- 对不同工作簿结构的适配能力弱
- 复用价值主要停留在写作层，而不是解析层

### 3.2 推荐理由

采用方案 A。当前目标是先形成“能跑、能复用、能稳定输出”的最小闭环，而不是提前建设更重的抽象层。等后续出现第二个、第三个同类脚本后，再判断是否需要把公共逻辑提升到共享库。

## 4. 文件结构与职责拆分

本次采用“脚本层 + 模板层”的双文件最小拆分。

### 4.1 新增文件

- `.agents/scripts/analyze-xlsx-test-report.py`
  - 负责读取工作簿、提取关键信息、执行降级策略、渲染 Markdown
- `docs/retrospective/templates/xlsx-test-report-template.md`
  - 负责定义固定报告章节与输出顺序

### 4.2 职责边界

#### `.agents/scripts/analyze-xlsx-test-report.py`

职责：

- 读取 `.xlsx`
- 识别工作表结构
- 提取总表核心指标
- 提取状态分布与重点失败项
- 识别高风险专题页
- 按模板渲染 Markdown 并写出文件

不负责：

- 图表绘制
- 缺陷系统同步
- 通用表格分析平台能力

#### `xlsx-test-report-template.md`

职责：

- 定义 Markdown 报告章节骨架
- 约束章节顺序与字段口径
- 作为后续同类报告的统一输出规范

不负责：

- 解析 Excel
- 执行统计或推断逻辑

### 4.3 为什么本轮不拆 `lib/`

当前先不把逻辑拆进 `.agents/scripts/lib/`，原因如下：

- 本轮目标是验证场景闭环，而不是建设脚本框架
- 拆分过早会增加实现复杂度
- 等出现多份同类脚本时，再根据重复情况决定是否抽共享库，更符合 YAGNI

## 5. 处理流程设计

脚本处理流程固定为 6 步：

```mermaid
flowchart TD
    A["加载工作簿"] --> B["识别总表"]
    B --> C["抽取工作簿结构"]
    C --> D["提取失败与风险证据"]
    D --> E["生成结论摘要"]
    E --> F["渲染 Markdown 报告"]
```

### 5.1 加载工作簿

- 使用 `openpyxl` 读取 `.xlsx`
- 收集工作表名称、行列规模、非空行数

### 5.2 识别总表

- 优先识别类似 `测试报告` 的总览页
- 提取核心指标：总用例、Pass、Fail、NoTest、Block、DI、严重问题数
- 若未识别标准总表，则降级为“按所有 sheet 汇总状态词统计”

### 5.3 抽取工作簿结构

- 输出每个 sheet 的名称与角色理解
- 区分总表、功能主表、专项页、弱网页、升级页等

### 5.4 提取失败与风险证据

- 抽取 `FAIL`、`NT`、`Block` 相关行
- 按工作表聚类
- 优先保留失败最多或明显影响发布判断的页面

### 5.5 生成结论摘要

自动生成：

- 是否建议发布
- 风险最高的 3-5 条主线
- 是否存在系统性问题，而非单点异常

### 5.6 渲染 Markdown 报告

- 按固定模板输出
- 写入 frontmatter 与正文结构

## 6. 输出字段设计

本轮先固定以下字段，不继续扩张：

### 6.1 `basic_info`

- 项目
- 设备型号
- 固件版本
- APP 版本
- 测试时间
- 测试人员

### 6.2 `workbook_summary`

- 工作表数量
- 工作表名称
- 每个工作表的角色理解

### 6.3 `overall_metrics`

- 总用例
- Pass
- Fail
- NoTest
- Block
- DI
- 严重问题数

### 6.4 `release_judgment`

- 是否建议发布
- 发布门槛
- 当前差距

### 6.5 `module_findings`

- 各重点 sheet 的学习结论

### 6.6 `risk_clusters`

- 风险聚类，例如音频、预览传输、存储回放、弱网

### 6.7 `final_conclusion`

- 面向读者的一段最终判断

## 7. 降级策略

为了避免脚本只适配一个特定 Excel，本轮加入两个降级规则：

### 7.1 未识别标准总表

- 不视为失败
- 改用全工作簿状态词统计
- 在输出中明确标注“总表未识别，结论基于降级统计”

### 7.2 部分工作表结构异常

- 不视为整体失败
- 跳过无法稳定解析的字段
- 保留工作表名称和“结构不规则”提示

## 8. 错误处理

脚本只处理 4 类核心错误：

### 8.1 文件不存在 / 路径错误

- 直接报错退出
- 提示检查 `.xlsx` 输入路径

### 8.2 工作簿无法读取

- 例如文件损坏或不是合法 `.xlsx`
- 直接报错退出

### 8.3 未识别标准总表

- 不报错退出
- 启动降级策略

### 8.4 部分工作表结构异常

- 不报错退出
- 跳过异常字段并输出提示

## 9. 验收标准

ACT-002 实施完成后，应满足以下验收标准：

1. 能输入一个 `.xlsx` 文件路径并成功运行
2. 能输出一份 Markdown 报告文件
3. 报告至少包含以下章节：
   - 结论摘要
   - 基本信息
   - 工作簿结构
   - 总体结果
   - 风险聚类
   - 最终判断
4. 对“标准总表存在”的测试报告，能正确提取核心指标
5. 对“总表不标准”的测试报告，能触发降级策略并输出可读报告
