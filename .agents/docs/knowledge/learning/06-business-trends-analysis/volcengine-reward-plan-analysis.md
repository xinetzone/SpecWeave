# 火山方舟协作奖励计划深度分析报告

---

## 1. 摘要与核心发现

火山方舟协作奖励计划是火山引擎推出的一项创新性"数据换免费Tokens"用户增长与数据采集计划。该计划通过清晰的价值交换机制——用户授权采集推理调用数据（经匿名化加密处理），平台返还等量免费资源包——构建了一个可持续的数据飞轮闭环。第二期合作期已延长至2026年7月31日，当前支持Doubao-Seed-Evolving模型，每日每模型最高采集500万Tokens并于次日返还等量奖励包。

从产品设计角度看，该计划体现了大模型平台对高质量训练数据的核心需求。冷启动资源包即时发放降低参与门槛，每日T+1返还机制形成持续行为强化，个人/企业分级权益引导用户完成更高等级认证，"无授权不采集、随时可撤回"的设计构建了用户信任基础。特别值得关注的是撤回授权机制的设计——虽然用户可随时一键停止新数据采集，但已被用于模型训练的历史数据在技术上具有不可逆性，这一透明告知体现了平台的合规审慎态度。

本报告基于火山引擎官方公开文档（2026.07.01更新）进行系统性分析。由于控制台页面需登录认证，部分交互细节基于产品逻辑推断（已明确标注）。报告重点解析"撤回授权/取消链接"操作的完整链路与设计考量，并深入洞察该计划背后的数据飞轮商业模式与增长策略。

核心洞察包括：(1) "数据换额度"是大模型行业解决训练数据来源可持续性问题的有效模式；(2) 即时奖励+持续反馈的双轨激励设计符合行为心理学原理；(3) 透明规则+用户控制权+匿名化承诺构成信任三角；(4) 分级权益体系是引导用户从个人向企业升级的有效增长杠杆。

## 2. 产品概述与商业模式

### 2.1 产品简介：协作奖励计划是什么

协作奖励计划是火山引擎方舟（Volcengine Ark）大模型服务平台推出的一项用户激励与数据采集计划。用户通过授权火山引擎采集其在指定模型推理接入点（Endpoint）的调用数据（包括用户输入和模型输出的全部内容），即可获得免费Tokens资源包作为奖励。该计划第二期合作期为2024年12月1日至2026年7月31日（已延长），是火山方舟获取高质量真实场景训练数据的重要渠道。

### 2.2 核心定位：用户增长活动+数据采集入口+开发者激励计划

协作奖励计划具有三重核心定位：

1. **用户增长活动**：通过免费额度吸引新用户开通模型服务，降低大模型API使用门槛，扩大用户基数
2. **数据采集入口**：以合法合规、用户知情同意的方式获取真实场景下的推理调用数据，用于模型优化训练
3. **开发者激励计划**：为AI应用开发者提供成本补贴，鼓励开发者基于火山方舟构建应用，形成生态粘性

三重定位相互支撑：增长活动带来用户，用户使用产生数据，数据优化模型提升服务质量，更好的服务又吸引更多用户。

### 2.3 商业模式："数据换免费Tokens"的价值交换

该计划的核心商业模式是清晰的双向价值交换：

- **用户付出**：授权火山引擎采集其在奖励额度内的推理调用数据（输入+输出）
- **用户获得**：冷启动免费资源包 + 每日等量免费Tokens奖励包
- **平台付出**：免费Tokens资源包（边际成本随规模扩大而降低）
- **平台获得**：高质量、真实场景的用户交互数据，用于模型训练与算法优化

这是一种典型的"数据即货币"模式，在AI时代具有重要意义——数据是大模型最重要的生产资料，而免费Tokens是开发者最直接的痛点需求。

### 2.4 双向价值逻辑：用户视角（降本）vs平台视角（获取高质量训练数据）

**用户视角的价值逻辑**：
- 直接收益：降低API调用成本，尤其是对于高频调用的开发者，每日500万Tokens（个人/企业额度不同）的免费额度可显著减少支出
- 低参与成本：一键授权，自动采集，自动发放奖励，无需额外操作
- 灵活退出：随时可撤回授权，未使用完的资源包仍可继续使用
- 数据安全感知：匿名化、加密存储的承诺降低隐私顾虑

**平台视角的价值逻辑**：
- 获取高质量真实数据：真实应用场景下的用户输入和模型输出比人工标注数据更有价值，更能反映实际使用模式
- 数据飞轮闭环：更多数据→更好模型→更多用户→更多数据，形成正向循环
- 用户增长与留存：免费额度是强获客手段，每日奖励机制提升用户活跃度和留存率
- 企业认证转化：通过分级权益引导个人用户升级企业认证，提升用户价值
- 合规风险可控：明确的授权协议、随时撤回机制、匿名化处理降低法律合规风险

### 2.5 在方舟产品体系中的位置

协作奖励计划位于火山方舟控制台的"开通管理"模块，是连接用户开通流程与模型使用的关键节点：

- 向上承接：用户开通模型服务后的首次体验引导
- 横向连接：模型广场中各支持模型的详情页入口（场景化触发）
- 向下支撑：为模型迭代提供数据燃料，为运营增长提供用户激励手段

从产品架构看，它既是一个运营活动模块，也是数据采集管道的入口，同时还是用户权益体系的组成部分。

## 3. 参与条件与入口路径

### 3.1 开通前提三条件

参与协作奖励计划需满足三个前提条件：

1. **已开通模型服务**：用户需先开通火山方舟的模型服务，这是基础前提
2. **关闭安心体验模式**：安心体验模式可能是一种数据隔离或体验受限模式，关闭后才能参与数据采集计划
3. **完成实名认证**：未完成实名认证的用户需先完成认证。个人实名认证可参与享受个人权益，企业认证可享受更高的企业权益

三个条件层层递进：开通服务是基础，关闭安心体验是数据采集的技术前提，实名认证是合规要求和权益分级的基础。

### 3.2 两个入口路径详解

协作奖励计划设计了两个互补的入口路径，覆盖不同场景下的用户：

**入口一：控制台-开通管理页 → 「立即参与」按钮**
- 用户特征：主动寻找权益、有明确降本需求的用户
- 场景：用户在控制台管理服务开通时，主动发现奖励计划
- 优势：用户意图明确，参与转化率高
- 定位：主动型入口，承接有明确需求的用户

**入口二：模型广场 → 模型详情页 → 「火山方舟模型奖励计划」入口**
- 用户特征：正在浏览/选择模型、对特定模型感兴趣的用户
- 场景：用户在模型广场查看Doubao-Seed-Evolving等支持模型详情时，场景化触发
- 当前支持模型：Doubao-Seed-Evolving
- 优势：在用户决策模型使用的关键时刻呈现激励，降低决策门槛
- 定位：场景型入口，在用户使用路径上自然触发

双入口设计体现了精细的用户路径规划：既覆盖主动寻找权益的用户，也在用户使用模型的关键场景进行触发。

### 3.3 当前支持模型

根据官方文档，当前协作奖励计划支持的模型为：
- **Doubao-Seed-Evolving**（豆包种子进化版模型）

文档明确说明"支持采集的模型以产品活动介绍页公示为准"，这意味着支持的模型列表是动态调整的，未来可能扩展到更多模型。

根据活动规则和操作流程，协作奖励计划详情页【基于逻辑推断】应包含以下功能区域：

## 4. 授权操作流程

### 4.1 前置条件检查

在进行授权操作前，系统会自动检查用户是否满足参与条件：
- 是否已开通模型服务
- 是否已关闭安心体验模式
- 是否已完成实名认证（个人/企业）
- 是否存在同一用户的其他账号已参与（反作弊检查）

如不满足条件，会显示相应的引导提示（如"请先完成实名认证"）。

### 4.2 进入活动页面

用户通过两个入口之一（开通管理页或模型详情页）进入协作奖励计划详情页。首次进入时页面可能展示活动介绍弹窗或新手引导，简要说明活动规则和权益。

### 4.3 选择模型与推理接入点

由于"每款模型按推理接入点独立申请授权并提供奖励"，用户需要：
1. 选择要参与的模型（当前可选Doubao-Seed-Evolving）
2. 选择该模型下要授权的具体推理接入点（Endpoint）
3. 可同时选择多个接入点，每个接入点独立授权、独立计算奖励

### 4.4 阅读并同意《数据授权使用协议》

点击授权前，用户需要：
- 查看协议摘要（核心要点快速浏览）
- 可点击查看《火山引擎数据授权使用协议》完整内容
- 勾选"我已阅读并同意《数据授权使用协议》"复选框
- 【基于逻辑推断】可能需要单独勾选确认理解"历史数据不可撤回"条款

这是合规要求的必要环节，确保用户在充分知情的情况下做出授权决策。

### 4.5 点击"授权"按钮完成操作

确认协议后，用户点击「授权」按钮提交授权请求。系统执行以下操作：
- 验证用户资格和接入点状态
- 记录授权关系和授权时间
- 触发冷启动资源包发放流程
- 更新页面状态为"已授权"

### 4.6 【基于逻辑推断】授权后页面状态变化

> ⚠️ 以下内容为基于产品交互逻辑的推断。

授权完成后，页面【基于逻辑推断】会发生以下状态变化：

1. **按钮状态变化**：
   - 原「授权」主按钮变为「撤回授权」按钮（样式变为警告色，如红色描边）
   - 按钮旁可能增加"已授权"状态标签（绿色/蓝色）

2. **状态标签变更**：
   - 接入点状态从"未授权"变为"已授权"
   - 显示授权时间
   - 显示数据采集中状态指示

3. **冷启动包到账提示**：
   - 弹出Toast/消息提示："冷启动资源包已发放至您的账户"
   - 显示获得的资源包额度（具体额度以页面为准）
   - 可能提供"查看资源包"快捷链接

4. **数据采集状态展示**：
   - 显示"今日已采集Tokens"计数器（初始为0）
   - 显示"明日预计发放奖励"预览
   - 显示采集上限进度条

### 4.7 冷启动资源包即时发放

授权成功后，冷启动资源包立即发放至用户账户，这是关键的"即时满足"设计：
- 无需等待，授权即得
- 用于抵扣按tokens后付费用量
- 降低用户的决策犹豫，即时正反馈
- 具体奖励额度以产品活动介绍页公示为准（文档未公开具体数值）

冷启动资源包是降低参与门槛的关键设计——用户在做出授权决策的瞬间就能获得 tangible 的回报，符合行为心理学中的"即时强化"原则。

## 5. 撤回授权/取消链接操作详解（重点章节）⭐

> 🎯 **本章重点**：撤回授权是协作奖励计划中保障用户数据控制权的核心功能，也是用户最关心的操作之一。本章节将从操作定位、触发方式、确认流程、即时效果、后续影响、设计理念等多个维度进行深度解析。

### 5.1 操作定位：用户的数据控制权保障

撤回授权（即用户提到的"取消链接"）操作在整个产品设计中处于核心地位——它是用户行使数据控制权的直接体现。在"数据换免费额度"的价值交换中，用户信任的建立不仅取决于奖励的吸引力，更取决于用户是否相信自己能够控制数据的使用边界。

撤回授权功能的存在，本质上是对用户数据所有权的尊重：用户有权决定何时开始、何时停止数据授权。这种"可进可退"的设计，而非"一选定终身"的捆绑，反而能降低用户的心理防备，提高授权意愿。从合规角度看，这也是《个人信息保护法》等法律法规对用户"撤回同意"权利要求的具体落实。

### 5.2 触发方式：页面「撤回授权」按钮

撤回授权的触发方式非常直接：
- 在已授权的模型/接入点对应的操作区域，点击「撤回授权」按钮
- 该按钮与「授权」按钮位于同一位置，状态随授权状态切换
- 不需要跳转页面，不需要联系客服，不需要复杂流程
- 这是一个用户自助式操作，体现"授权自主"原则

与某些平台故意隐藏退出选项、设置繁琐退出流程的做法不同，火山方舟将撤回授权按钮与授权按钮放在同等显眼的位置，这一设计选择本身就传递了平台对数据合规的信心和对用户权利的尊重。

### 5.3 【基于逻辑推断】确认流程设计

> ⚠️ 以下内容为基于安全交互设计原则和B端产品惯例的推断，实际流程以登录控制台后为准。

由于撤回授权是一个具有重要后果的操作（会停止奖励发放、可能影响服务使用），【基于逻辑推断】应有二次确认流程设计，符合"不可逆/高风险操作需二次确认"的安全交互原则：

1. **第一步：点击「撤回授权」按钮**
   - 按钮点击后不立即执行撤回操作
   - 弹出确认弹窗（Modal）

2. **第二步：二次确认弹窗展示**
   弹窗【基于逻辑推断】应包含以下要素：
   - **标题**："确认撤回授权？"或"是否确认取消数据授权？"
   - **后果告知区**（清晰列出撤回后的影响，使用图标+文字的方式可视化）：
     * ⚠️ 后续调用数据将立即停止采集
     * ❌ 次日起不再发放每日奖励包
     * ✅ 已发放未使用的资源包仍可在有效期内继续使用
     * ⚠️ 历史已采集并用于模型训练的数据无法撤回
     * ⚠️ 可能影响相应模型服务的继续使用
   - **协议确认**：【基于逻辑推断】可能需要勾选"我已了解撤回授权的后果"复选框
   - **操作按钮**：
     * 主按钮：「确认撤回」（警告色，如红色）
     * 次按钮：「取消，继续授权」（取消关闭弹窗）
   - **快捷链接**：可能提供"查看常见问题"或"联系客服"链接

3. **第三步【可选】：输入验证或额外确认**
   - 【基于逻辑推断】对于高风险操作，可能需要输入手机验证码或进行其他身份验证
   - 也可能要求用户输入"撤回"或特定文字进行确认（类似删除账户的设计）
   - 这一步是为了防止误操作，确保用户是经过深思熟虑做出的决定

4. **第四步：系统执行撤回操作**
   - 用户确认后，系统立即执行撤回
   - 显示加载状态（"正在处理..."）
   - 操作完成后显示成功提示

### 5.4 即时生效效果

根据官方文档，撤回授权后以下效果**立即生效**：

1. **后续调用数据立即停止采集**
   - 撤回操作完成后，该接入点新产生的调用数据立即停止采集
   -    - "无授权不采集"原则严格执行：从撤回这一刻起，不再有新数据进入采集管道
   - 这是对用户停止授权意愿的即时响应

2. **奖励计划自动终止**
   - 该模型/接入点的协作奖励计划状态立即变为"已终止"
   - 不再计入每日采集量统计

3. **页面状态恢复为未授权状态**
   - 【基于逻辑推断】按钮从「撤回授权」变回「授权」
   - 状态标签从"已授权"变回"未授权"
   - 采集进度、每日统计等数据展示模块重置或隐藏
   - 【基于逻辑推断】可能显示"授权已撤回"的状态提示

即时生效是"授权自主"原则的技术体现——用户的停止意愿应该得到立即尊重，而不是设置冷却期或等待人工审核。

### 5.5 后续影响详细说明

撤回授权后，除了即时生效的效果外，还会产生以下后续影响，需要用户特别关注：

✅ **未使用完毕的奖励资源包：仍可在有效期内继续使用（30天有效期不变）**
- 这是一个非常重要的用户友好设计：撤回授权不没收已获得的奖励
- 已发放到账户的资源包，只要在30天有效期内，仍可正常使用
- 可用于抵扣在线推理的按Token后付费费用
- 这一设计避免了用户"担心撤回就损失已得奖励"的顾虑，让撤回决策更无负担

❌ **历史已采集数据：已被用于模型训练的数据在技术上无法撤回（机器学习模型训练的不可逆性）**
- 这是最需要用户理解的关键条款，文档中做了明确透明的告知
- **技术原因分析**：
  * 机器学习模型训练是一个将数据中的模式和知识"编码"到模型参数中的过程
  * 一旦数据参与了训练，其影响已经分散到模型的数十亿甚至数千亿参数中
  * 无法像删除数据库记录那样，从训练好的模型中"删除"某条特定数据的影响
  * 即使能够做到，成本也极高，且可能影响模型整体性能
- **平台承诺**：火山引擎仅能停止使用新输入的数据，但无法撤回已使用的历史数据
- **透明告知的意义**：平台没有隐瞒这一技术现实，而是在授权前和撤回前都明确告知，让用户在充分知情的情况下做出决策，这是合规和信任建设的重要体现

⚠️ **模型服务影响：撤回授权可能影响相应模型服务的继续使用**
- 文档原文："撤回授权可能影响相应模型服务的继续使用"
- 这是一个需要注意的提示，但具体影响程度文档未详细说明
- 【基于逻辑推断】可能的影响包括：
  * 如果该接入点是专门为奖励计划创建的，可能需要确认服务是否继续
  * 某些活动专属接入点可能在撤回后无法继续使用
  * 但正常的付费推理服务应该不受影响（这是合理推测，需以实际为准）
- 这一提示体现了平台的审慎态度，不做过度承诺

📊 **新产生调用：不再产生采集，不再获得次日奖励包**
- 撤回授权后新产生的调用：
  * 数据不会被采集
  * 不会计入采集量统计
  * 次日不会获得相应的每日奖励包
  * 调用费用正常按后付费方式结算（或使用已有的其他资源包）

### 5.6 "授权自主"设计理念分析：用户控制权原则

"授权自主"作为数据安全三大保障之一，其设计理念根植于用户控制权原则：

1. **用户是数据的主人**：数据的使用权应该由用户决定，而非平台单方面决定
2. **进入自由，退出自由**：参与计划是自愿的，退出计划也应该是自由的，不应设置不合理障碍
3. **控制权可视化**：通过显眼的「撤回授权」按钮，让用户随时能看到自己的控制权
4. **即时响应**：用户行使控制权时，系统应立即响应，而非设置等待期或审核流程
5. **后果透明**：撤回前明确告知所有后果，让用户在知情前提下做决策
6. **不惩罚退出**：已获得的奖励不收回，不因为用户退出而惩罚用户

这一设计理念不仅是合规要求，更是建立长期用户信任的基础。当用户知道自己"随时可以走"，反而更愿意"留下来试一试"。

### 5.7 授权vs撤回的对称性与不对称性分析

授权和撤回作为一对互逆操作，在设计上既有对称性，也有因技术现实导致的不对称性：

**对称性设计**：
- **同一页面操作**：都在协作奖励计划详情页完成，无需跳转
- **一键完成核心操作**：核心操作都是点击一次按钮（授权/撤回）完成
- **状态即时切换**：操作完成后页面状态立即更新
- **无需人工审核**：都是系统自动处理，自助完成
- **都有明确反馈**：操作完成后都有明确的成功提示和状态变化

**不对称性设计（核心差异）**：

| 维度 | 授权操作 | 撤回操作 |
|------|---------|---------|
| **时间效应** | 授权是即时全量生效：授权后新数据立即开始采集 | 撤回只停止未来数据：对历史已采集数据无影响 |
| **数据流向** | 数据开始流入采集管道 | 停止新数据流入，但已流入数据无法"倒回" |
| **确认流程** | 需同意协议，一次确认即可 | 【基于逻辑推断】需二次确认，明确告知后果 |
| **奖励影响** | 立即获得冷启动包，开始累计每日奖励 | 停止获得新奖励，但已得奖励保留 |
| **可逆性** | 授权后可以撤回（可逆） | 历史数据部分不可逆（技术限制） |
| **按钮样式** | 主按钮（吸引点击） | 警告按钮（谨慎点击） |
| **心理门槛** | 低门槛（获得奖励） | 高门槛（可能担心损失，需理解后果） |

最核心的不对称性在于：**授权是面向未来的、可逆的；而撤回虽然能停止未来，但无法改变过去**。这不是平台设计的问题，而是机器学习技术本身的特性决定的。平台选择透明告知这一不对称性，而非回避或隐瞒，恰恰是负责任的表现。

### 5.8 【基于逻辑推断】交互设计考量

> ⚠️ 以下内容为基于交互设计最佳实践的分析推断。

从产品交互设计角度，撤回授权功能【基于逻辑推断】有以下设计考量：

1. **明确的后果告知**
   - 不在细则里藏后果，而是在确认弹窗中用图标+要点清晰列出
   - 区分"什么是立即生效的"、"什么是保留的"、"什么是无法撤回的"
   - 避免用户因为误解后果而产生投诉或纠纷

2. **二次确认防误操作**
   - 高风险操作必须有二次确认，防止用户点错
   - 【基于逻辑推断】可能通过颜色对比（红色确认按钮）、复选框确认、甚至文字输入确认等方式，确保用户是有意识地操作
   -    - "取消"按钮应该比"确认"更容易点击（符合安全设计原则）

3. **撤回后可重新参与的可能性**
   - 【基于逻辑推断】撤回授权后，用户应该可以随时重新授权参与
   - 不应设置"撤回后X天内不能再次参与"的冷却期（除非有反作弊考虑）
   - 重新授权后，冷启动包是否再次发放？文档未说明，【基于逻辑推断】可能每个模型/接入点只能获得一次冷启动包
   - 这种"可反复进出"的设计进一步降低了参与顾虑

4. **状态反馈的清晰性**
   - 撤回成功后应有明确的成功提示
   - 页面状态清晰切换回未授权状态
   - 如有未使用完的资源包，应提示用户资源包仍可使用及有效期

5. **情感化设计考量**
   - 撤回确认文案不应有责备语气（如"您确定要放弃免费奖励吗？"）
   - 应保持中性、尊重的语气，告知事实，尊重用户选择
   - 可以表达"欢迎随时回来参与"的友好态度

## 6. 权益分级体系

### 6.1 个人实名认证权益

完成个人实名认证的用户可以参与协作奖励计划，享有个人权益：
- 可参与计划，获得冷启动资源包
- 每日采集量有对应上限（文档未公开具体数值，以页面公示为准）
- 每日奖励包额度与个人采集上限对应
- 可升级为企业认证享受更高权益

个人权益是参与计划的基础门槛，主要面向个人开发者、小团队、学习者等用户群体。

### 6.2 企业认证权益（更高采集上限与资源包额度）

完成企业认证的用户享有企业权益：
- 采集量上限提升至企业权益对应水平（高于个人用户，文档未公开具体数值）
- 每日免费资源包上限也相应提升
- 享受企业级服务和支持
- 适合有一定规模的AI应用和企业用户

分级体系的核心差异在于**采集量和奖励包上限**——企业用户用量更大，能贡献更多数据，因此获得更高的免费额度，这是一种按贡献量级的差异化激励。

### 6.3 个人→企业升级路径

个人用户升级企业认证的路径设计非常顺畅：
- **入口**：直接在活动页面即可完成企业认证（无需跳转到其他页面）
- **操作**：完成企业认证流程后，需要在活动页面**手动确认升级**享受企业权益
- **效果**：确认升级后，采集量和资源包上限立即提升至企业水平

这一设计将企业认证转化嵌入到奖励计划场景中，当个人用户看到"更高额度"的权益对比时，升级动机最强，此时提供便捷的升级入口能最大化转化率。手动确认升级的步骤也给了用户选择权，避免自动升级带来的困惑。

### 6.4 分级设计商业逻辑分析

权益分级体系的设计背后有清晰的商业逻辑：

1. **引导企业认证**：企业用户价值更高（付费能力强、使用量大、合规性好），通过更高额度激励个人用户升级企业认证
2. **提升数据质量与数量**：企业用户的使用场景更接近真实生产环境，数据质量更高；更高的采集上限也能带来更多数据
3. **用户分层运营**：不同层级用户提供不同激励，实现精细化运营
4. **降低入门门槛**：个人用户零门槛即可参与，扩大用户基数；企业用户获得更高额度，提升高价值用户留存
5. **合规分层**：企业认证需要更严格的资质审核，数据合规风险更低

从增长角度看，这是一个经典的"Freemium"（免费增值）模式变体——免费参与吸引大众用户，更高权益引导用户向高价值层级转化。

## 7. 奖励机制详解

### 7.1 冷启动资源包：首次授权即时发放，降低参与门槛

冷启动资源包是用户首次成功授权某模型推理接入点后立即获得的免费额度资源包：
- **发放时机**：授权成功后立即发放（即时满足）
- **发放条件**：首次授权某模型的推理接入点
- **用途**：抵扣按tokens后付费用量
- **额度**：以产品活动介绍页公示为准（文档未公开具体数值）

冷启动资源包的设计目的：
- **降低决策门槛**：用户在授权瞬间就能获得回报，不需要等待
- **体验奖励价值**：让用户立即体验到免费Tokens的价值，形成使用习惯
- **"先给后取"**：平台先付出价值，再获取数据，建立互惠感知

### 7.2 每日奖励包：T+1自动发放机制，采集多少返还多少

每日奖励包是授权后持续激励的核心机制：
- **采集周期**：每日0:00-24:00为一个采集周期
- **采集规则**：在每日额度内自动采集推理数据
- **发放时间**：次日（T+1）发放
- **返还规则**：根据前一日采集量**返还等量**免费资源包（采集多少，返还多少）
- **公平性**：未获授权的接入点数据不予采集，该部分使用量次日不返还资源包

"采集多少返还多少"的对等设计是关键：
- 用户感知到"用数据换Tokens"是公平交易
- 用量越大的用户获得越多奖励，激励深度使用
- 没有复杂的计算规则，简单易懂，降低认知负担

### 7.3 采集上限：每模型每日不超过500万Tokens

每日采集量设有上限：
- **上限标准**：每个模型每日不超过500万tokens（这是文档提到的数值，可能是企业用户上限，个人用户以页面为准）
- **超过上限部分**：超过采集上限后的数据不再采集，也不会额外返还奖励包
- **上限的意义**：
  * 控制平台成本：避免极端高用量用户带来过高的资源包支出
  * 公平性：让更多用户能获得奖励，而非少数大用户独占
  * 数据多样性：一定程度上鼓励更多不同用户使用，而非单一用户刷量

### 7.4 有效期机制：发放后30天，到期清零，促进持续使用

所有奖励包（冷启动+每日奖励）都有30天有效期：
- **有效期时长**：发放后30个自然日内有效
- **到期处理**：到期未使用的部分自动清零
- **适用范围**：所有奖励资源包统一执行30天有效期规则

30天有效期的设计考量：
- **促进持续使用**：避免用户囤积资源包后不再活跃，激励用户持续使用平台服务
- **成本可控**：平台的负债（未使用资源包）有明确的到期日，便于财务测算
- **行为强化**：每日获得的奖励需要及时使用，形成每日使用的习惯循环
- **合理期限**：30天既不会太短让用户觉得没用，也不会太长失去激励效果

### 7.5 多模型独立计算：每个模型/接入点独立授权独立计算

多模型/接入点参与规则：
- 可同时使用任意一款或几款模型服务
- 每款模型按推理接入点独立申请授权
- 每个接入点的采集量、奖励包独立计算
- 多接入点的奖励包可叠加使用（推测）

独立计算的设计：
- 灵活性：用户可以选择只授权部分接入点，而非全部
- 清晰性：每个接入点的授权状态和奖励清晰可查
- 独立性：一个接入点撤回不影响其他接入点

### 7.6 发放延迟说明：参与人数多时可能延迟

文档特别说明："近期参与人数较多，资源包下发可能存在延迟"。这一说明体现了平台的透明度：
- 提前告知用户可能存在延迟，降低用户焦虑和投诉
- 暗示活动热度高，参与人数多（社会证明）
- T+1发放不是绝对保证，合理预期管理

### 7.7 激励设计心理学分析：即时奖励（冷启动）+持续反馈（每日返还）的双轨激励

奖励机制的设计符合经典的行为心理学原理：

1. **即时奖励（冷启动包）—— 即时强化**
   - 授权瞬间获得奖励，建立"授权=获得奖励"的神经联结
   - 降低决策时的犹豫，因为"现在就能拿到"
   - 符合 operant conditioning（操作性条件反射）中的正强化原则

2. **每日奖励包—— 可变间隔强化+固定比率强化**
   - 每日固定发放（固定间隔），但每日额度随用量变化（可变比率）
   - 每日查看奖励到账形成习惯循环（提示→行为→奖励）
   - 类似"每日签到"的心理效应，提升用户留存和日活
   -    - "采集多少返还多少"让用户感知到努力与回报直接挂钩

3. **双轨激励协同**
   - 冷启动包负责"拉新"（让用户开始）
   - 每日奖励负责"留存"（让用户持续）
   - 30天有效期负责"促活"（让用户经常来用）

4. **损失厌恶的利用**
   - 30天到期清零利用了人们的"损失厌恶"心理——不使用就"损失"了，比"获得"同样额度更有驱动力
   - 每日不用就少拿一天的奖励，也是一种损失框架

### 8.2 采集数据内容范围

协作奖励数据的采集范围包括：
- **输入内容**：调用模型期间用户输入的所有内容，涵盖多种模态：
  * 文本
  * 声音
  * 图形
  * 图片
  * 图像
  * 视频
  * 以及其他形式的输入
- **模型输出内容**：模型针对用户输入生成的全部返回内容
- **范围边界**：仅采集"奖励包额度内调用指定模型期间"的输入输出，不在此范围内的数据不采集

简单说：授权接入点在额度内产生的"一问一答"全部内容都会被采集。

### 8.3 不支持的调用方式：无奖励说明

以下调用方式不在采集范围内，不产生奖励：
- **批量推理**：批量推理API调用不采集、不奖励
- **按模型单元付费**：采用预付费模型单元方式的调用不参与
- **TPM保障包**：使用TPM保障包的调用量不参与
- **推理缓存存储**：缓存存储相关计费项不在抵扣范围内，也不参与采集
- 其他非支持API的调用方式

文档明确列出不支持的场景，避免用户误解"只要用了就能拿奖励"。

### 8.4 支持模型范围：以页面公示为准，动态调整

支持采集的模型以产品活动介绍页公示为准：
- 当前明确支持：Doubao-Seed-Evolving
- 支持列表是动态调整的，未来可能增加新模型
- 不同模型可能有不同的参与规则或奖励额度（以页面为准）

这种动态调整机制给平台留了灵活性，可以根据模型迭代需要和运营策略调整支持范围。

### 8.5 不采集承诺：范围外数据不采集、不存储

平台明确承诺："不在采集范围内的数据不会被采集和存储"。这是"无授权不采集"原则的延伸：
- 未授权的接入点：不采集
- 授权但超出每日额度的部分：不采集
- 不支持的API调用：不采集
- 非指定模型的调用：不采集
- 其他非计划范围内的任何数据：不采集、不存储

这一承诺对于建立用户信任至关重要——用户需要确信平台不会"顺手牵羊"采集额外数据。

## 10. 数据安全与合规设计

> ⚠️ **免责声明**：本章节为产品机制分析，不构成法律意见。数据合规相关问题请咨询专业法律人士。

### 10.1 三大保障机制详解

协作奖励计划建立了三大数据安全保障机制，构成用户信任的基础：

**1. 隐私无忧：加密传输+匿名化处理**
- **严格加密**：授权数据在传输和存储过程中都经过严格加密处理
  * 传输加密：使用TLS/SSL等加密协议传输，防止中间人窃取
  * 存储加密：数据落盘时加密存储，防止数据泄露时直接暴露原文
- **匿名化处理**：采集的数据会经过匿名化处理
  * 去除可直接识别个人身份的信息
  * 分离数据与用户身份标识
  * 使数据无法（或难以）回溯到特定个人
- **匿名化 vs 去标识化**：文档用"匿名化"表述，意味着处理后的数据原则上不能识别特定个人（区别于"去标识化"——去标识化仍可能通过额外信息重新识别）

**2. 无授权不采集：默认不采集，授权才采集，撤回即停止**
这是最核心的原则，包含三层含义：
- **默认状态不采集**：用户没有明确授权前，系统默认不采集任何数据
- **授权后才采集**：只有用户主动完成授权操作后，才开始采集指定范围内的数据
- **撤回立即停止**：用户撤回授权后，立即停止采集新数据
- **计划终止不采集**：如果协作奖励计划本身终止，也自动停止采集
- 这一原则从根本上避免了"默认 opt-out"（默认采集，用户需手动退出）的争议模式，采用更尊重用户的"opt-in"（用户主动选择加入）模式

**3. 授权自主：随时可撤回，无隐藏门槛**
- 用户可随时在活动页面撤回授权
- 不需要联系客服
- 不需要说明理由
- 没有等待期或冷却期（【基于逻辑推断】）
- 撤回后已得奖励不收回
- 这一机制让用户始终掌握控制权

三大机制层层递进：匿名化加密保护数据本身安全，无授权不采集从源头控制数据收集范围，授权自主让用户掌握最终控制权。

### 10.2 匿名化处理机制说明

根据官方文档和数据处理的通用实践，匿名化处理【基于逻辑推断】可能包括：
- 移除直接标识符（如用户ID、手机号、账号名等）
- 对可能间接识别身份的信息进行泛化处理（如精确时间→时间段、精确位置→区域等）
- 数据聚合或扰动（在不影响模型训练效果的前提下添加噪声）
- 分离身份信息与数据内容存储
- 访问控制：只有经过授权的算法工程师才能在严格权限控制下访问处理后的数据

匿名化的目的是在保留数据训练价值的同时，切断数据与特定个人的关联，保护用户隐私。

### 10.3 数据存储安全：加密存储

采集的数据经过匿名化处理后加密存储：
- 用于火山引擎及其关联公司的模型算法和服务优化
- 存储过程加密
- 有严格的访问控制机制
- 文档未明确说明数据存储期限（协议中授权期限为永久）

### 10.4 授权期限设计分析：默认永久但可随时终止新数据

授权期限的设计：
- **默认永久**：授权后数据授权持续有效，除非用户主动撤回
- **随时可撤回**：用户可随时撤回，终止新数据的授权
- **撤回效果**：撤回后后续调用数据不再提供，奖励计划终止
- **已发奖励**：未用完的奖励包仍可在有效期内使用

这种"默认长期，但随时可终止新数据"的设计平衡了平台和用户双方需求：
- 平台获得稳定的数据来源预期，便于规划模型训练
- 用户获得灵活的退出机制，不会被"一锁定终身"

### 10.5 "历史数据不可撤回"的技术原因与透明告知

文档中最具争议也最体现透明性的条款是："已授权并已被使用的数据在技术上无法撤回，火山引擎仅能停止使用新输入的数据"。

**技术原因深度解析**：
1. **模型训练的本质**：大模型训练是一个将海量数据中的统计模式、语言规律、知识关联"编码"到数十亿到数千亿参数中的过程
2. **数据影响的弥散性**：单条数据的影响不是存在模型的某个"位置"，而是弥散在整个模型参数网络中，与其他数据共同影响模型
3. **无法"反向工程"删除**：训练完成后，无法精确计算并移除某条特定数据对参数的影响，这在数学上是极其困难甚至不可能的
4. **灾难性遗忘问题**：即使可以通过某些技术"遗忘"特定数据，也可能导致模型整体性能下降（即"灾难性遗忘"问题）
5. **已生成模型的传播**：如果模型已经训练完成并部署、甚至分发给其他用户，更无法收回

**平台的透明告知策略**：
- 不在小字或隐藏条款中说明，而是在公开文档和（【基于逻辑推断】）授权流程中明确告知
- 使用平实语言解释"技术上无法撤回"，不回避问题
- 明确告知用户平台"仅能停止使用新输入的数据"，不做不切实际的承诺
- 这种透明告知虽然可能"劝退"一些用户，但长期来看建立了信任——用户知道平台说的是真话

### 10.6 用户承诺与平台义务的对等性分析

> ⚠️ 本部分为协议条款的对等性分析，不构成法律意见。

从协议条款看，用户与平台的权利义务存在一定的对等性设计：

**用户承诺**：
- 合法持有数据
- 个人信息处理合规
- 不提供保密数据/商业秘密
- 特殊数据需提前审批
- 不用于违法违规用途

**平台义务**：
- 匿名化处理数据
- 加密存储
- 按约定范围使用（模型优化、学术研究等）
- 无授权不采集
- 允许随时撤回
- 按约定发放奖励

当然，实际协议中平台的权利范围更广泛（如非独家、可分许可给关联方等），这是此类数据授权协议的常见模式，但至少在形式上设计了双方的基本权利义务。

## 12. 法律协议要点解析

> ⚠️ **免责声明**：本章节为协议内容梳理和客观要点总结，不构成法律意见。如有法律合规相关问题，请咨询专业律师。

### 12.1 授权性质（非独家/不可转让/免费/可分许可关联方）

《火山引擎数据授权使用协议》中授权性质条款：
- **非独家**：用户授予平台的授权不是排他性的，用户可以同时授权给其他方，也可以自己使用数据
- **不可转让**：平台不能将授权整体转让给第三方（但可分许可，见下条）
- **不可分许可（例外）**：原则上不能再许可他人，但有例外——**可分许可给关联方和第三方外包服务商**
  * 关联方：火山引擎的关联公司（如实控人控制的其他企业）
  * 第三方外包服务商：为火山引擎提供数据处理、模型训练等服务的外包厂商
- **免费**：授权是免费的，平台不需要为获得授权向用户支付费用（奖励包是平台自愿提供的激励，不是授权对价——这是法律上的重要区分）

### 12.2 授权权利范围（传输/存储/使用/复制/修改等）

用户授予平台广泛的使用权利，包括：
- **传输**：将数据在服务器间、不同系统间传输
- **存储**：存储数据（含加密存储）
- **使用**：使用数据进行模型训练、算法优化等
- **复制**：复制数据（用于备份、分发到训练集群等）
- **下载**：下载数据到处理环境
- **修改**：对数据进行修改、预处理、匿名化等
- **处理**：对数据进行任何形式的处理操作

这些权利覆盖了数据处理和模型训练所需的全部操作。

### 12.3 授权目的（模型优化/学术研究）

授权数据的使用目的限定为：
- 机器学习/人工智能技术和模型的优化
- 模型的开发、使用
- 学术研究及发表

这意味着数据不能被用于与AI技术研发无关的目的（如直接向用户推送广告、出售给第三方做营销等）。允许学术研究及发表是一个值得注意的条款，意味着基于这些数据的研究成果可能以论文形式公开发表（经过匿名化处理的前提下）。

### 12.4 用户数据合规承诺

用户需要做出以下承诺：
- **合法持有数据**：用户保证自己是数据的合法持有者，有权进行授权
- **个人信息合规**：如果数据中包含个人信息，用户保证已合规取得个人同意，符合个人信息保护相关法律要求
- **不提供保密数据/商业秘密**：用户不得将包含保密信息、他人商业秘密的数据输入模型（这既是用户承诺，也是对用户的保护提示——不要输入敏感信息）
- **特殊数据需提前审批**：涉及特殊类型数据（可能指重要数据、敏感个人信息等）需要提前经过平台审批

这些承诺将数据合规的基础责任放在用户侧——用户需要确保自己输入的数据是合法合规、不侵权、不涉密的。

### 12.5 被禁用途与被禁行为

协议明确禁止某些用途和行为：

**被禁用途**：
- 诽谤、侮辱他人

## 13. 数据飞轮与增长策略洞察

### 13.1 数据飞轮闭环：用户使用→数据采集→模型优化→更好服务→更多用户

协作奖励计划的核心是构建并加速"数据飞轮"（Data Flywheel）正向循环：

```
    ┌─────────────┐     更多数据      ┌─────────────┐
    │   更多用户   │ ───────────────→ │ 模型优化训练  │
    │  更多使用    │                   │  能力提升    │
    └─────────────┘                   └─────────────┘
           ↑                                │
           │                            更好的模型
           │                            更好的服务
           │                                ↓
    ┌─────────────┐     免费奖励      ┌─────────────┐
    │  更低成本    │ ←─────────────── │  用户体验提升  │
    │  更高粘性    │                   │  更强竞争力   │
    └─────────────┘                   └─────────────┘
```

飞轮运转逻辑：
1. 用户通过授权参与计划，获得免费Tokens
2. 用户使用模型服务产生真实场景交互数据
3. 平台用这些数据训练和优化模型
4. 模型能力提升后，服务质量更好、更智能
5. 更好的服务吸引更多用户使用
6. 更多用户使用产生更多数据，飞轮加速旋转

协作奖励计划在这个飞轮中起到"润滑剂"和"加速器"的作用——通过激励降低用户参与门槛，让飞轮更快启动和旋转。

### 13.2 激励机制设计：即时满足（冷启动包）+可变强化（每日返还）

从行为设计学（Behavioral Design）角度，激励机制设计非常精妙：

| 激励类型 | 设计方式 | 心理学原理 | 行为目标 |
|---------|---------|-----------|---------|
| 冷启动包 | 授权即得，即时发放 | 即时满足、正强化、互惠原理 | 降低决策门槛，完成首次授权 |
| 每日奖励 | T+1发放，用量挂钩 | 变动比率强化、习惯循环 | 促进每日持续使用，提升留存 |
| 30天有效期 | 到期清零 | 损失厌恶 | 提升活跃度，防止用户沉睡 |
| 企业升级 | 更高额度激励 | 目标梯度效应 | 引导个人用户升级企业认证 |

这种"即时奖励建立行为、持续奖励固化习惯、有效期促进活跃、分级引导升级"的组合拳，是非常成熟的增长激励设计。

### 13.3 信任构建三角：透明规则+随时撤回+匿名化承诺

在涉及数据授权这类敏感操作时，用户信任是参与的前提。协作奖励计划通过三角支撑构建信任：

1. **透明规则**：
   - 清晰说明采集什么、不采集什么
   - 明确告知奖励如何计算、何时发放
   - 坦诚说明"历史数据无法撤回"等局限性
   - 规则公开可查，不隐藏在小字条款

2. **用户控制权（随时撤回）**：
   - 一键撤回，无门槛
   - 撤回即时生效
   - 已得奖励不收回
   -    - "我可以随时退出"的感知大幅降低焦虑

3. **数据安全承诺（匿名化+加密）**：
   - 加密传输存储
   - 匿名化处理
   - 无授权不采集
   - 不采集范围外数据

三角缺一不可：没有透明规则，用户不知道自己同意了什么；没有控制权，用户会担心"上船容易下船难"；没有安全承诺，用户担心数据泄露滥用。三者共同作用，才能让用户放心点击"授权"。

### 13.4 低摩擦设计：一键授权、自动采集、自动发放，无需额外操作

好的激励设计应该让用户"忘记自己在参与活动"：

- **授权操作简单**：选择接入点→同意协议→一键授权，三步完成
- **无额外负担**：授权后不需要用户做任何额外操作——继续正常使用API即可
- **自动采集**：后台自动采集额度内的数据，用户无感知
- **自动发放奖励**：次日自动到账，不需要申请、不需要签到、不需要兑换
- **自动抵扣**：资源包自动用于抵扣费用，不需要手动选择使用

零额外操作成本意味着参与计划的边际成本几乎为零——用户只需要做一次授权决策，之后所有过程自动完成。这种"设置后即遗忘"（set it and forget it）的设计是提升参与率的关键。

### 13.5 分级增长路径：个人用户→企业认证的升级引导

权益分级不仅是差异化激励，更是精心设计的增长转化路径：

```
个人实名认证用户
     │
     │ 看到更高额度对比
     │ 场景化引导（活动页直接升级）
     ↓
企业认证用户（更高价值）
```

增长转化逻辑：
1. **低门槛进入**：个人实名认证即可参与，用户基数大
2. **权益对比刺激**：在页面上清晰展示个人vs企业权益差异，让个人用户看到"升级能获得更多"
3. **场景化升级入口**：在奖励计划页面直接提供企业认证入口，不需要跳转寻找
4. **手动确认升级**：完成认证后手动确认，让用户有掌控感
5. **升级即享更高额度**：即时奖励，形成正向反馈

这是经典的AARRR漏斗模型中"Revenue"（变现/价值提升）环节的典型设计——先通过免费服务获取大量用户，再通过权益差异引导高价值用户转化。

### 13.6 入口设计双路径：开通管理（主动寻找）+模型详情（场景触发）

两个入口的设计覆盖了用户旅程中的不同关键节点：

| 入口 | 用户场景 | 用户意图 | 设计特点 |
|-----|---------|---------|---------|
| 开通管理页「立即参与」 | 用户开通/管理服务时 | 主动寻找优惠、降本 | 直接、明确，承接高意图用户 |
| 模型详情页奖励计划入口 | 用户浏览/选择模型时 | 考虑是否使用该模型 | 场景化触发，降低模型使用决策门槛 |

双路径设计的精妙之处在于：
- 主动找优惠的用户能找到（开通管理入口）
- 还在犹豫用不用模型的用户被推一把（模型详情入口）
- 两个入口都指向同一页面，运营维护成本低
- 符合"在用户需要的时刻出现"的场景化运营理念

### 13.7 该模式对大模型行业的普适意义：解决训练数据来源的可持续性问题

"数据换免费额度"模式对整个大模型行业具有普适意义：

1. **解决训练数据瓶颈**：大模型训练需要海量高质量数据，单纯依靠公开数据集或人工标注成本高、规模有限。用户真实交互数据是最有价值的增量数据来源。

2. **可持续的数据获取机制**：相比一次性的数据采购，这种"用户持续使用→持续产生数据→持续获得奖励"的模式形成了可持续的数据供给管道。

3. **合规的数据获取方式**：通过明确的授权协议、用户知情同意、随时撤回机制，以合规方式获取数据，规避数据爬取等灰色方式带来的法律风险。

4. **双赢的价值分配**：用户获得实际的成本降低，平台获得数据价值，而非零和博弈。

5. **可复制的模式**：不仅火山引擎，几乎所有大模型平台都可以采用类似模式（事实上行业内已有类似实践）。

可以预见，"数据换服务/额度"将成为大模型平台获取训练数据的标准模式之一。

### 13.8 潜在挑战与平衡：隐私顾虑vs激励力度、数据质量vs采集数量

该模式在运营中需要持续平衡几个关键矛盾：

1. **隐私顾虑 vs 激励力度**
   - 激励力度太小：用户觉得"不值得拿隐私换这点钱"
   - 激励力度太大：平台成本过高，可能吸引来只是为了刷奖励的"羊毛党"
   - 需要在数据价值和激励成本间找到平衡点

2. **数据质量 vs 采集数量**
   - 放宽采集范围（更多API、更少限制）：能获得更多数据，但数据质量可能参差不齐
   - 严格采集限制（只采特定API、高质量接入点）：数据质量高，但数量可能不足
   - 当前500万Tokens的日上限、限定支持API，是在质量和数量间的平衡

3. **用户体验 vs 数据需求**
   - 过于复杂的授权流程：用户放弃参与
   - 过于简单的授权流程：用户可能没充分理解授权范围，事后产生纠纷
   - 当前一键授权+协议链接+二次确认（撤回）的设计，试图在体验和合规间平衡

4. **透明性 vs 商业机密**
   - 过度透明（公开模型训练细节、数据用量等）：可能泄露商业机密
   - 透明度不足：用户不信任，不愿参与
   - 当前在关键条款（采集范围、撤回影响）上保持透明，但在具体额度数值上保留弹性（"以页面公示为准"）

这些平衡没有标准答案，需要平台根据运营数据持续调整优化。

## 15. 专业术语表

| 术语 | 解释 |
|-----|------|
| **协作奖励计划** | 火山方舟推出的"数据换免费Tokens"用户激励计划，用户授权采集推理数据获得免费资源包 |
| **冷启动资源包** | 用户首次授权某模型推理接入点后立即发放的一次性免费资源包，用于降低参与门槛 |
| **每日奖励包** | T+1发放的持续性奖励，根据前一日采集Tokens数量返还等量免费资源包 |
| **推理接入点 (Endpoint)** | 火山方舟中模型服务的访问入口，用户创建接入点后通过API调用模型，协作奖励按接入点独立授权 |
| **匿名化** | 对采集的数据进行处理，去除或泛化可识别个人身份的信息，使数据无法回溯到特定个人 |
| **数据飞轮 (Data Flywheel)** | 指"更多用户→更多数据→更好模型→更多用户"的正向循环，是大模型平台的核心增长机制 |
| **实名认证** | 用户向平台提供身份信息完成验证，分为个人实名认证和企业认证，是参与活动的前提条件 |
| **企业认证** | 面向企业用户的认证类型，完成后享有更高的采集上限和资源包额度，权益高于个人认证 |
| **后付费** | 一种计费模式，先使用服务，根据实际使用量（Tokens数量）在结算周期结束后付费 |
| **Chat API** | 对话补全API，是大模型最常用的API接口，用于多轮对话交互场景 |
| **TPM保障包** | Tokens Per Minute（每分钟Tokens数）吞吐量保障包，是一种预付费的性能保障产品，不在奖励包抵扣范围内 |
| **T+1发放** | 指当日产生的采集量，次日（下一个自然日）发放对应奖励的机制 |
| **数据授权** | 用户同意平台按照约定范围和方式采集、使用其数据的行为，是协作奖励计划的核心法律行为 |
| **推理缓存命中** | 使用推理缓存功能时，重复请求直接返回缓存结果而无需重新推理，缓存命中的Tokens费用可由奖励包抵扣 |
| **推理缓存存储** | 推理缓存产生的存储费用，属于不可抵扣项（注意：缓存命中的Tokens费可抵扣） |
| **模型精调 (Fine-tuning)** | 在基础模型之上，使用特定数据进行进一步训练，使模型适配特定场景或任务的过程。协作奖励采集的数据可能用于此用途 |
| **安心体验模式** | 火山方舟的一种模式设置，参与协作奖励计划需要关闭此模式（推测为数据隔离或受限体验模式） |
| **批量推理** | 离线批量提交推理任务的服务方式，不属于在线推理，不在奖励计划采集和抵扣范围内 |
| **按模型单元付费** | 预付费购买模型计算单元的计费方式，类似云服务器的包年包月，不在奖励包抵扣范围内 |
| **Opt-in（选择加入）** | 用户主动选择参与（即主动授权）才开始采集数据的模式，与Opt-out（默认参与需手动退出）相对，更尊重用户选择权 |
