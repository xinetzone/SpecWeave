# 常见问题解答

本章汇总Fable 5成本优化相关的常见问题与解答，分为基础问题和进阶问题两部分。

## 一、基础问题

### Q1: 这些方案现在（7月）还能用吗？

**A:** 分阶段回答：

- **7月12日前（北京时间7月13日15:00前）**：Fable 5仍在订阅套餐内，可以放心使用。建议趁此期间用技能蒸馏法沉淀知识，把项目的核心经验、调试手册、架构约定等提炼成skills，传承给更便宜的模型。
- **7月12日后**：订阅通道关闭，需要开启按量计费，所有方案仍然适用，且更有价值。此时缓存续命、批量接口、pxpipe、包工头模式等成本优化手段会直接影响你的账单金额。

### Q2: pxpipe对Opus 4.8或其他模型有效吗？

**A:** pxpipe是为Fable 5的视觉能力量身定做的。根据原文测试数据：
- Fable 5读这种渲染页：100/100准确率
- Opus 4.8误读约7%

迁移到其他模型需要自担风险。对于视觉能力弱的模型不建议使用，因为识别准确率下降可能导致输出质量降低，反而得不偿失。

### Q3: 技能蒸馏生成的skills质量如何保证？

**A:** 方案设计了三步质量保障机制：

1. **生成阶段**：Fable 5像新来的首席工程师那样通读整个项目（README、构建脚本、测试、CI配置、git提交历史全部过一遍），然后并行开出十几个agent，一个agent负责一份skill。
2. **评审阶段**：由三个评审agent分别审查：
   - 事实是否属实
   - 前后是否矛盾
   - 新手能否照做
3. **修复阶段**：最后由一个修复agent统一修改后才算完工。

这种多轮评审机制确保了生成的skills质量可靠，不是随便凑数的内容。

### Q4: 包工头模式需要复杂配置吗？

**A:** 不需要复杂配置，项目设计得非常友好：

- 项目本身自带**安装器脚本**，帮你把配置文件复制到正确位置
- 是往现有`CLAUDE.md`**追加**配置，安装hooks和skill模板，**不会替换原有设置**
- 安装过程是非侵入式的，不会破坏现有工作流，可以随时启用或停用

### Q5: 缓存续命具体怎么操作？

**A:** 提供三种实操策略，按优先级排列：

**最简单（推荐）：派完子任务后不要idle**
- 等待子agent返回的间隙，让Fable顺手做点别的小事
- 比如：整理文档、写注释、梳理决策记录、做格式检查等
- 只要保持请求间隔小于5分钟，缓存就会自动续命

**应急方案：空请求保活**
- 估算子agent完成时间，还差1分钟左右交活时
- 发一个简单请求（如"继续"、"等待中"、"等一下子任务返回"）
- 这个请求成本极低（几个token），但能避免几千甚至几万token的冷启动重付

**判断标准：**
- 如果等待时间 > 缓存TTL（5分钟），要么保持活动，要么接受冷启动重付费
- 预计跑半小时以上的大活，索性让缓存死掉，连续续命花的钱可能比过期后重新计费还贵

## 二、进阶问题

### Q6: 批量接口和缓存可以叠加使用吗？

**A:** 是的，原文明确说可以叠加，而且叠加效果非常可观：

- 缓存命中：0.1倍原价
- 批量接口：0.5倍原价
- **两者叠加：0.1 × 0.5 = 0.05倍原价**

输入价低至**$0.5/百万token**，相当于打了**0.5折**。同一个模型，会用机制的人和不会用的人，账单能差好几倍甚至20倍。

### Q7: 一个通宵跑任务大概要花多少钱？

**A:** 根据原文警示：一两千万token的话，按输入$10+输出$50计算，睁眼就是几百美元账单。

**风险控制建议：**
1. **设置预算上限**：在Anthropic控制台设置消费告警和预算上限，避免超支
2. **重要任务开启前预估token用量**：先小范围测试，估算单任务token消耗，再计算总成本
3. **使用包工头模式拆分**：让Fable做判断，便宜模型做执行，大幅降低高价值token消耗
4. **非紧急任务走Batch API**：夜间任务尽量走批量通道，输入输出直接半价
5. **长上下文用pxpipe**：需要喂入大量上下文时，开启pxpipe可省59%~70%

### Q8: Fable 5还会回到订阅套餐吗？

**A:** 基于原文信息：Anthropic工程师Thariq在X上表示，一旦容量允许会努力让Fable 5回到订阅套餐。这次赶在断供前几小时宣布延期5天（从7月7日延至7月12日），也算是把这句话兑现了一小截。

但请注意：目前**没有明确的回归时间表**，不要抱太大期望，应该做好长期按量计费的准备，掌握成本优化方法才是正道。

### Q9: 开源方案之间可以组合使用吗？

**A:** 可以，而且推荐组合使用。详见[04-selection-guide.md](04-selection-guide.md)的"方案组合建议"章节，里面提供了针对不同场景的推荐组合策略：

- **日常开发场景**：缓存续命 + 包工头模式
- **大型项目分析场景**：pxpipe + 批量接口
- **项目启动期场景**：技能蒸馏 + 缓存
- **夜间/周末批量处理**：包工头 + 批量接口 + 缓存叠加
- **紧急故障排查**：纯Fable + 谨慎保活，其他都别加

### Q10: 没有Claude Code订阅，直接按量计费，应该优先上哪个方案？

**A:** 建议按以下优先级逐步实施：

1. **先理解并用好缓存（零成本，立刻见效）**：这是ROI最高的优化手段，不需要任何额外工具，只要理解机制并调整使用习惯，输入就能省90%。任务间隙控制在5分钟内，没事发个空请求续命。

2. **非紧急任务走批量（零配置，官方原生）**：不需要实时返回的任务，攒起来走Batch API，输入输出直接半价，相当于用Fable 5的能力付Opus的价格。

3. **然后根据场景选开源方案**：
   - 上下文特别长 → pxpipe（省59%~70%）
   - 重复任务多 → 自己手动沉淀skills（虽然没有订阅期的免费额度，但一次性沉淀长期复用仍然划算）
   - 团队使用、任务混合 → 包工头模式

详见[04-selection-guide.md](04-selection-guide.md)的决策流程图，可以一步步引导到最合适的方案组合。

### Q11: pxpipe的有损压缩会影响日常开发吗？

**A:** 日常任务问题都不大，但需要注意适用边界：

- **可以放心用的场景**：代码理解、架构分析、日志排查、文档总结、代码审查等探索性任务，对精度容错较高
- **需要谨慎的场景**：哈希值、ID、密钥、精确字符串匹配等需要逐字节准确的内容，建议仍然用文本输入，不要为了省钱冒出错的风险

**最佳实践**：关键信息（报错信息、日志ID、密钥、配置参数）用文本输入，大段上下文（整个代码库、日志文件、文档）用pxpipe，两者结合使用。

### Q12: 缓存首次写入有25%溢价，会不会反而多花钱？

**A:** 不会，只要后续能命中超过2次就开始净赚。

计算一下：
- 首次写入：1.25倍价格（多付25%）
- 第1次命中：0.1倍价格（省90%）→ 累计已经赚了
- 第2次及以后命中：都是纯赚

在多轮对话场景下，缓存的ROI非常高。只有在单轮对话、后续不会再复用上下文的场景下，缓存才没有收益。

### Q13: oh-my-fable和技能蒸馏是什么关系？

**A:** oh-my-fable是技能蒸馏思路的工程化扩展项目：

- **fable-5-train-opus-skills-after-it-retires**：核心是一段提示词，让Fable 5在退休前蒸馏出具体的skills文件
- **oh-my-fable**：更进一步，把Fable 5干长任务的方法论本身（先规划、自我纠错、不丢线索）抽象成一个不挑模型的执行框架

oh-my-fable的核心特性是**每一步自动存档**，程序崩溃也没关系，可以原地续跑。按量计费之后，"不用从头跑"这五个字本身就值钱——崩了不用从头重跑，避免重复付费。
