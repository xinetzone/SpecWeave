# 官方成本优化机制

除了社区开源方案外，Anthropic官方在计费文档中也提供了明确的成本优化机制。本章详细介绍两个官方优化技巧：Prompt Cache（缓存经济学）和Batch API（批量接口）。这两个机制是官方原生支持的，无需额外工具即可使用，其中缓存机制是文章重点强调、其他人很少提但特别重要的内容。

## 一、技巧一：缓存经济学（Prompt Cache）

Prompt Cache是Anthropic提供的一项重要成本优化机制，其核心价值在于：**对于每轮对话重复喂给模型的内容（系统提示词、工具定义、整个代码库上下文），命中缓存后输入价格直接砍90%**。

### 1.1 缓存价值

在多轮对话场景中，有大量内容是重复发送的：
- **系统提示词**：每轮对话都需要携带
- **工具定义**：function calling的tools schema在会话期间基本不变
- **代码库上下文**：大型项目的代码索引、文档内容在短时间内不会变化
- **对话历史前缀**：历史对话的前半部分在后续轮次中是固定的

这些重复内容如果每次都按全价计费，会造成巨大的浪费。缓存机制正是为了解决这个问题而设计的。

| 场景 | 未缓存价格 | 缓存命中价格 | 节省比例 |
|------|-----------|-------------|---------|
| 输入token（重复部分） | $10 / 百万token | **$1 / 百万token** | **省90%** |

### 1.2 缓存价格机制（准确记录）

官方缓存的计费规则非常明确，需要准确理解：

| 操作 | 价格倍数 | 实际价格（输入基准$10/百万token） | 说明 |
|------|---------|----------------------------------|------|
| **写缓存** | 1.25倍输入价 | **$12.5 / 百万token** | 首次写入缓存需要额外付费，有25%的写入溢价 |
| **读缓存** | 0.1倍输入价 | **$1 / 百万token** | 命中缓存后读取，节省90%成本 |
| **TTL（存活时间）** | - | **默认5分钟** | 缓存写入后5分钟内有效 |
| **命中刷新** | - | 自动刷新 | 每次命中缓存会自动重置TTL，重新计时5分钟 |

> **关键理解**：首次写入缓存需要多付25%，但只要后续能命中超过2次，就开始净赚。在多轮对话场景下，这个ROI非常高。

### 1.3 冷启动成本陷阱

理解了缓存机制后，最需要警惕的是**缓存过期导致的反复冷启动**：

- 如果Fable派完任务等**6-8分钟**才继续下一轮，缓存已经过期
- 缓存过期后，下一轮对话需要重新写入完整上下文
- 这意味着你会**反复支付冷启动的完整上下文费用**（1.25倍写入价）
- 在长任务场景下，如果任务间隙安排不当，冷启动成本可能会累积成一笔可观的开销

**典型反例**：Fable派子agent去执行一个需要7分钟才能完成的任务，主流程在那里干等，子agent回来时缓存已死，所有上下文需要重新发送。

### 1.4 缓存续命策略（实操建议）

围绕缓存的TTL机制，有三种可落地的实操策略：

#### 策略A：保活——派完任务别闲着

Fable派完任务后，不要让主会话空等。让它顺手处理别的正事，只要一直在输出，缓存就会自动续命。

**实操场景**：
- 等待子agent返回的间隙，可以让Fable整理一下刚才的决策记录
- 并行处理一些不依赖子agent结果的辅助任务（如文档梳理、格式检查）
- 只要保持请求间隔小于5分钟，缓存就永远不会过期

#### 策略B：空请求保活——精准续命

如果确实没有别的正事可做，可以采用空请求保活的方式：

**操作方法**：
- 估算子agent完成时间，还差1分钟左右交活时
- 发一个几乎空的请求（比如"继续"、"等待中"等）
- 这个请求的唯一目的是**命中缓存并重新计时**
- 成本极低：一个空请求只有几个token，即使按全价也花不了1美分，但能避免几千甚至几万token的冷启动重付

> **注意**：不要太频繁地发空请求，建议间隔4分钟左右发一次即可，过于频繁反而会增加不必要的token消耗。

#### 策略C：放弃缓存——长任务场景

如果是确定跑半小时以上的大活，**索性让缓存死掉**，不要强行续命。

**决策逻辑**：
- 连续续命半小时，需要发送约6次保活请求（每5分钟一次）
- 保活本身也需要消耗token，而且需要保持上下文连接
- 如果续命的累计成本超过了过期后重新计费的成本，不如放弃
- 对于单次超长时间的批量任务，缓存机制收益不大，直接走普通计费或Batch API更划算

**决策参考表**：

| 任务预计时长 | 推荐策略 |
|-------------|---------|
| < 10分钟 | 策略A或B，积极保活，缓存收益大 |
| 10-30分钟 | 视任务间隙而定，间隙超过6分钟考虑放弃 |
| > 30分钟 | 策略C，放弃缓存，考虑Batch API |

### 1.5 多模型协作节奏排期

深刻理解缓存机制后，可以制定**多模型协作的最优节奏**——围着缓存的死活来排期任务。

**排期原则**：
1. **批量处理短任务**：把多个短任务集中在缓存存活期内连续处理，共享同一份缓存上下文
2. **长任务异步化**：预计超过缓存TTL的长任务，派给子agent后不要空等，主会话转去处理别的短任务
3. **任务间隙控制**：主会话的任务间隙严格控制在5分钟以内，避免缓存过期
4. **保活信号预排**：预先排好保活空请求的发送时机，不要等到缓存快过期了才想起

这种围绕缓存设计的协作节奏，能让整体成本下降一个数量级。

## 二、技巧二：批量接口（Batch API）

对于不要求实时返回结果的任务，官方提供了Batch API批量处理通道，价格直接半价。

### 2.1 适用场景

Batch API适合**非实时、异步处理**的场景：

- 批量代码生成/重构任务
- 大规模文档处理、翻译、摘要
- 离线数据分析、日志批量审查
- 测试用例批量生成
- 任何不需要立即拿到结果的后台任务

不适用场景：
- 需要即时交互的对话
- 实时编程助手场景
- 需要快速迭代调试的任务

### 2.2 价格优势

Batch API的价格非常有吸引力，**输入输出全部半价**：

| 类型 | 原价（实时API） | Batch API价格 | 折扣 |
|------|----------------|---------------|------|
| 输入token | $10 / 百万token | **$5 / 百万token** | 5折 |
| 输出token | $50 / 百万token | **$25 / 百万token** | 5折 |

这个价格是什么概念？——**等于把Fable 5打回Opus 4.8的价位**。对于可异步化的任务，这相当于用Fable 5的能力，付Opus级别的价格。

### 2.3 使用方式要点

- 将多个请求攒起来，一次性提交批量任务
- 任务完成后通过API批量获取结果
- 处理延迟通常在几小时级别（官方不保证具体SLA）
- 适合夜间、周末等非工作时间批量处理

## 三、叠加优惠（重要）

两个官方优化技巧**可以叠加使用**，产生极其可观的折扣效果：

### 3.1 折扣叠加计算

| 组合 | 价格倍数 | 输入实际价格 | 折扣力度 |
|------|---------|-------------|---------|
| 原价 | 1.0x | $10 / 百万token | 无折扣 |
| 仅缓存命中 | 0.1x | $1 / 百万token | 1折 |
| 仅Batch API | 0.5x | $5 / 百万token | 5折 |
| **缓存 + Batch叠加** | **0.1 × 0.5 = 0.05x** | **$0.5 / 百万token** | **0.5折** |

**结论**：缓存命中（0.1倍）+ 批量接口（0.5倍）= **0.05倍原价**，输入价能压到每百万0.5美元左右，相当于打了**0.5折**。

### 3.2 成本差异总结

同一个模型，会用官方机制的人和不会用的人，账单能差**好几倍甚至20倍**：

- 不懂机制的人：纯实时API，所有token按全价算，$10/$50
- 会用缓存的人：重复部分1折，整体成本降50-80%
- 会用Batch的人：异步任务5折，适合批量处理
- **两个都会用的人**：叠加0.5折，把Fable 5用出比小模型还便宜的价格

## 四、官方优化机制使用建议

| 场景 | 推荐机制 |
|------|---------|
| 实时多轮对话、编程助手 | **Prompt Cache**，积极保活 |
| 夜间批量处理、离线任务 | **Batch API**，攒起来处理 |
| 有大量重复前缀的批量离线任务 | **缓存 + Batch叠加**，最大化折扣 |
| 超长时间单任务（>30分钟） | 放弃缓存，视实时性选实时API或Batch |
