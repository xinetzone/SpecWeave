# 实践练习

---

光看不练假把式。七概念是实践方法论，必须通过动手练习才能真正掌握。这里设计了两个练习，从轻到重，帮你把理论转化为能力。

---

## 练习1：七概念快速自检（30分钟，个人练习）

### 🎯 练习目标
检查自己对七概念基本要素的理解，建立"概念→设计"的映射直觉。

### ⏱️ 时间分配
- 步骤a：10分钟（默写）
- 步骤b：5分钟（对照检查）
- 步骤c：10分钟（案例匹配）
- 步骤d：5分钟（自己的思考）

### 📋 练习步骤

#### 步骤a：默写公理和要素（10分钟，**不要翻资料**）

拿出一张纸或打开一个空白文档，不看理论章节，为七个概念分别写下：
1. 一句话公理（用你自己的话，不用一字不差）
2. 4个基础要素（写出要素名称即可）

格式参考：
```
R 复盘：
- 公理：______
- 要素1：______
- 要素2：______
- 要素3：______
- 要素4：______

I 洞察：
...（以此类推七个概念）
```

> ⚠️ **重要**：默写不出来就空着，不要猜。空着说明你需要回去复习，猜错了会形成错误记忆。

#### 步骤b：对照检查（5分钟）

翻到[理论篇公理速查表](../01-seven-concepts-theory/README.md#七概念公理速查表)，对照检查你的默写：
- 用✅标记正确的
- 用❓标记部分正确但不准确的
- 用❌标记错误或遗漏的
- 计算正确率：正确的要素数 / 总要素数（35个：7个概念×5项=公理+4要素）

#### 步骤c：DeepTutor案例匹配（10分钟）

在DeepTutor案例中，为每个概念找一个你觉得最贴切的设计（可以和教程分析不同，只要你能说清理由）。

格式参考：
```
R 复盘：我觉得______（某个模块/设计）最能体现R，因为______
I 洞察：我觉得______最能体现I，因为______
...（以此类推）
```

> 💡 提示：如果你找的和教程分析不一样，没关系——只要你能说清楚这个设计对应了概念的哪个要素，就是对的。同一个设计可能对应多个概念，同一个概念也可能在多个设计中体现。

#### 步骤d：你的设计思考（5分钟）

写下你的答案：如果让你设计一个学习工具（或者你熟悉的任何工具），七概念中你最先应用哪三个？为什么？

不需要长篇大论，3-5句话说清楚就行。重点是**为什么**这三个优先，而不是其他。

### ✅ 验收标准

| 检查项 | 通过标准 | 自检 |
|--------|---------|------|
| 公理和要素正确率 | ≥80%（28/35正确） | □ |
| 案例匹配 | 为7个概念都找到了至少一个对应设计，且能说出理由 | □ |
| 设计思考 | 选择了3个优先概念，说明了理由（不是随便选的） | □ |

### 📊 结果解读

- **≥90%（32+/35）**：理论掌握很好，可以直接做练习2
- **80-89%（28-31）**：不错，把错的地方复习一下，继续做练习2
- **60-79%（21-27）**：建议回去复习理论章节中错得比较多的概念，再做练习2
- **<60%（<21）**：理论还没掌握，重新读一遍理论篇再来

## 练习2：用七概念分析一个你熟悉的开源项目/工具（60-90分钟）

### 🎯 练习目标
掌握用七概念框架分析真实系统的方法，能从R/F/A/C/V/E/I七个维度系统地拆解一个系统。

### 🛠️ 选择分析对象

选一个你日常使用、比较熟悉的开源工具/项目/框架。推荐（但不限于）：
- 开发工具：VS Code、Git、Vim、Docker
- 前端框架：React、Vue、Svelte
- 笔记工具：Obsidian、Logseq、Notion（如果熟悉）
- 其他：你自己参与的开源项目（推荐！收获最大）

**选择标准**：你至少用了1个月以上，知道它的核心功能和主要交互。

### ⏱️ 时间分配

严格按时间盒来，不要在某一步卡太久：
- R事实采集：10分钟
- F第一性原理分析：15分钟
- A原子化分析：10分钟
- C原子提交分析：10分钟
- V对抗审查：10分钟
- R→I→E知识沉淀：10分钟
- 总结：15分钟

### 📋 练习步骤（严格按顺序）

#### 步骤a：R事实采集（10分钟）

**要求**：客观描述这个工具的核心功能、架构特点，**不带评价，纯事实**。

写至少10条客观事实，每条满足：
- ✅ 可验证：任何人看了都能同意这是真的
- ✅ 无因果：不要写"因为XX所以它设计成这样"
- ✅ 无评价：不要写"这个设计很好/很差/很巧妙"
- ✅ 尽量具体：有数字写数字，有名字写名字

**事实清单格式**：
```
F01: 这个工具叫______，主要功能是______
F02: 它的核心界面/交互入口是______
F03: 它把功能分成了______个主要模块/菜单/面板
...（继续写，至少10条）
```

> ⚠️ **R质量门**：检查你的事实中有没有出现这些词——"因为/所以/导致/好/坏/巧妙/愚蠢/合理/不合理"。出现一个就说明你混入了判断，删掉重写。

#### 步骤b：F第一性原理分析（15分钟）

思考并回答：
1. 这个工具解决什么**本质问题**？（不是表面功能，而是用户最根本的需求）
2. 它的作者可能是从什么**公理/假设**出发设计的？找到1-2个最核心的设计公理
3. 有没有**隐含假设**？作者默认了什么前提是成立的？这些假设在什么情况下可能不成立？

**格式**：
```
本质问题：这个工具要解决的根本问题是______
核心公理1：作者假设______，所以设计了______
核心公理2：作者假设______，所以设计了______
隐含假设1：作者默认了______，但这个假设在______情况下可能不成立
隐含假设2：（如果能找到第二个）
```

> 💡 提示：好的F分析应该能让你有"哦，原来如此！"的感觉——你突然理解了为什么它这么设计，而不是那样设计。

#### 步骤c：A原子化分析（10分钟）

分析它的模块拆分：
1. 它的功能模块是怎么拆分的？粒度如何？
2. 认知负荷和导航成本平衡得怎么样？
   - 模块会不会太大（一个模块里塞太多东西，认知负荷高）？
   - 模块会不会太碎（太多小模块，导航成本高，找不到东西）？
3. 模块之间的边界清晰吗？有没有职责重叠？

**格式**：
```
模块拆分方式：______
粒度评价：太粗/合适/太碎，因为______
认知vs导航平衡：______
模块边界：清晰/有重叠，问题在______
```

#### 步骤d：C原子提交分析（10分钟）

分析它的操作/交互的原子性：
1. 用户的主要操作/变更是否支持"原子性"？
2. 能不能独立撤销/回滚单个操作？
3. 有没有出现"做了A必须连带做B，不能单独做A"的情况？
4. 变更历史是线性的还是分支的？

**格式**：
```
核心操作的原子性：好/一般/差，因为______
撤销能力：支持/部分支持/不支持，例如______
非原子操作举例：______（如果有的话）
```

#### 步骤e：V对抗审查（10分钟）

分析它的验证/证伪机制：
1. 这个工具在哪些地方体现了证伪/验证思想？（例如：错误提示、实时预览、确认弹窗、测试功能等）
2. 有哪些地方V做得不足？哪里容易出错但没有防呆/验证？
3. 至少找出1个优点和1个不足。

**格式**：
```
V做得好的地方：
1. ______（具体设计），它如何帮助验证：______
2. ______（如果有第二个）

V做得不足的地方：
1. ______（具体场景），问题是：______
2. ______（如果有第二个）
```

#### 步骤f：R→I→E知识沉淀机制分析（10分钟）

分析它的记忆/复盘/知识沉淀：
1. 这个工具有没有记忆/复盘/知识沉淀机制？
2. 用户的使用数据/历史/偏好是如何保存和积累的？
3. 知识能不能从一次会话迁移到下一次？能不能从一个用户迁移到另一个用户？

**格式**：
```
记忆机制：有/没有，体现在______
知识积累方式：______
知识迁移能力：强/一般/弱，因为______
```

#### 步骤g：总结（15分钟）

写下你的分析结论：
1. 这个工具在哪些概念上做得特别好？（选2-3个）
2. 哪些概念有明显的改进空间？（选1-2个）
3. 如果让你给这个工具提3个改进建议，用七概念的语言来说，你会提什么？

**格式**：
```
做得好的概念：
- ______：因为______
- ______：因为______

可以改进的概念：
- ______：问题在于______，可以改进的方向是______

三个改进建议：
1. 用______概念的思路，改进______：______
2. ...
3. ...
```

### 📊 评分Rubric（简化版，100分）

| 维度 | 分值 | 评分标准 | 自评分 |
|------|------|---------|--------|
| R事实采集 | 20分 | - 18-20分：≥10条事实，完全没有因果词和评价词<br>- 14-17分：≥8条事实，偶尔有评价词但不影响理解<br>- 10-13分：5-7条事实，有较多判断混入<br>- <10分：<5条事实，大部分是判断 | ____ |
| F第一性原理 | 20分 | - 18-20分：找到了真正的本质问题和核心公理，隐含假设合理<br>- 14-17分：找到了核心公理，但本质问题停留在表面<br>- 10-13分：只描述了功能，没分析到本质<br>- <10分：完全没做F分析 | ____ |
| A+C粒度与原子性 | 20分 | - 18-20分：A和C都分析到位，粒度判断有理有据，原子性问题找得准<br>- 14-17分：两个概念都分析了，但深度一般<br>- 10-13分：只分析了一个，或者两个都很表面<br>- <10分：这部分没做 | ____ |
| V对抗审查 | 20分 | - 18-20分：至少1个优点1个不足，都很具体，不是泛泛而谈<br>- 14-17分：有优点有不足，但不够具体<br>- 10-13分：只说了优点或只说了不足<br>- <10分：V分析缺失 | ____ |
| 总结与结论 | 20分 | - 18-20分：结论具体不空泛，改进建议用七概念语言，可落地<br>- 14-17分：有结论有建议，但比较笼统<br>- 10-13分：结论很简单，建议不可行<br>- <10分：没有总结 | ____ |
| **总分** | **100分** | | ____ |

### 📈 结果解读

- **≥90分**：🏆 优秀！你已经可以熟练用七概念分析系统了
- **80-89分**：✅ 良好！基本掌握了，看看哪项扣了分，针对性练习
- **70-79分**：⚠️ 及格！还需要多练，建议再分析一个简单的工具
- **<70分**：❌ 建议回去复习理论和融合分析章节，再做一次

## 💡 练习小贴士

1. **严格守时**：用计时器，到点就进入下一步，不要追求完美。七概念里也有"完成比完美重要"的思想。
2. **先独立完成**：做的过程中不要翻前面的章节，做完再对照。
3. **找个朋友一起做**：两个人分析同一个工具，然后对比结果，互相用V攻击对方的分析，收获会加倍。
4. **从简单工具开始**：第一次分析不要选太复杂的（比如不要一上来就分析Kubernetes），从Git、VS Code这种你天天用的开始。
5. **多练几个**：分析完第一个，再分析第二个你会发现顺很多，练3个基本就形成肌肉记忆了。
