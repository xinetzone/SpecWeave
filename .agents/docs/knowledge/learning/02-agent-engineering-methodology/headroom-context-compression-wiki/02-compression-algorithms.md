## 三、六种压缩算法详解

### 3.1 内容路由机制：先诊断，再开方

Headroom不靠一把锤子敲所有钉子。在压缩之前，它会先做**内容路由（Content Routing）**：判断输入内容是JSON、代码、日志、还是自然语言，然后挑选最合适的算法去压。

这是一个非常重要的设计思想——"对症下药"而不是"一刀切"。不同类型的内容有不同的结构特点和信息分布规律：

- JSON是结构化数据，有字段名、数组、嵌套对象
- 代码有语法结构、import、函数签名、类型信息
- 日志有时间戳、级别、重复模式
- 自然语言有语义连贯性和冗余表达

如果用同一种方法（比如简单截断或者小模型摘要）处理所有内容，要么压缩率上不去，要么关键信息丢失。Headroom的内容路由机制是它能够同时实现高压缩率和高质量的基础。

### 3.2 SmartCrusher：JSON专用压缩

**SmartCrusher**是专门针对JSON数据设计的统计式压缩算法，主要处理工具返回的数组、嵌套对象、混合类型数据。

**压缩效果**：70-90%的Token节省率。

**适用场景**：
- API调用返回的JSON响应
- 工具执行结果（如数据库查询结果）
- 结构化配置数据
- 包含大量重复字段、空值、默认值的JSON

**工作原理**：SmartCrusher通过统计分析JSON结构，识别并移除：
- 重复出现的冗余字段
- 空值、null、默认值
- 过长的数组中与当前查询无关的元素
- 嵌套对象中的冗余层级

但它会智能保留关键字段和数据结构，确保模型理解返回结果的核心含义。

### 3.3 CodeCompressor：代码专用压缩（基于AST）

**CodeCompressor**是基于语法树（AST，Abstract Syntax Tree）进行代码压缩的算法，适用于Python、JavaScript、Go、Rust、Java、C++六种主流编程语言。

**核心特点**：
- 基于AST语法分析，不是简单的文本截断
- 保留`import`语句、函数签名、类型信息
- 移除函数体内部的实现细节（如样板代码、注释、空行）
- 保留代码结构信息，模型读完压缩后的代码仍能正确理解项目架构

**为什么保留这些关键信息？**

模型理解代码结构主要靠这些"骨架"信息：
- `import`告诉模型这个文件依赖哪些模块
- 函数签名（函数名+参数+返回类型）告诉模型有哪些能力可用
- 类型信息帮助模型理解数据结构

至于函数体内部的具体实现，大部分时候模型不需要看——它知道这个函数"做什么"就够了，真需要看细节的时候可以通过CCR机制取回原文。

### 3.4 Kompress-v2-base：自然语言专用压缩

**Kompress-v2-base**是作者自己训练的自然语言压缩模型，专门针对Agent场景优化。

**训练特点**：
- 用大量**agentic trace**（Agent执行轨迹）训练
- 模型"知道"在Agent场景下哪些话可以丢掉，哪些必须保留
- 不是通用文本摘要，而是面向Agent交互的智能压缩

**与通用摘要模型的区别**：
通用摘要模型可能会把对话中的关键指令、错误信息、工具调用参数也"摘要"掉，但Kompress-v2-base理解Agent工作流，知道哪些信息对后续决策至关重要。

### 3.5 其他压缩算法

除了上述三种重点算法，Headroom还包含另外3种压缩方案，针对日志等特殊场景：

- **日志专用压缩算法**：识别日志的时间戳、级别、重复模式，移除重复INFO日志，保留ERROR/WARN等关键信息
- **通用结构化数据压缩**：处理CSV、表格等其他结构化格式
- **混合内容自适应压缩**：处理包含多种类型的混合内容

### 3.6 与简单方案的对比优势

我们可以把Headroom的多算法路由方案，与常见的简单压缩方案做个对比：

| 方案 | 压缩率 | 信息保留 | 适用场景 | 缺点 |
|------|--------|----------|----------|------|
| 简单截断 | 可控 | 差（容易丢尾部关键信息） | 无 | 暴力、不可靠、关键信息经常丢失 |
| 统一小模型摘要 | 中等 | 一般（容易丢结构化信息） | 纯文本 | 对代码/JSON/日志效果差，容易"幻觉" |
| 规则-based过滤 | 低-中 | 一般（规则写死不灵活） | 特定已知格式 | 泛化能力差，新场景需要写新规则 |
| **Headroom多算法路由** | **高（50-90%）** | **好（保留结构+可逆）** | **全场景覆盖** | **实现复杂度高** |

### 3.7 设计洞察：内容感知路由模式

Headroom的压缩算法设计体现了一个非常有价值的可复用模式——**内容感知路由（Content-Aware Routing）**：

> 不要试图用一种方法处理所有类型的输入。先识别输入的类型和结构，再路由到专门的处理策略。

这个模式不只适用于上下文压缩，还可以推广到很多场景：
- 日志处理系统：先识别日志类型（访问日志/错误日志/应用日志），再用不同策略解析
- 数据清洗管道：先判断数据格式（JSON/CSV/XML），再用对应cleaner
- API响应处理：先识别响应类型（成功/失败/分页/错误），再做不同处理

这是"分类讨论"思想在工程设计中的体现，也是Headroom能做到既省Token又保质量的核心原因。
