# 对抗性审查方法论框架

## 1. 概述

对抗性审查（Adversarial Review）是一套基于证伪主义和偏差防御的质量保障方法论，其核心哲学是：**与其试图"证明正确"，不如主动"寻找问题"**。

该方法论目前有两大成熟应用分支：

| 分支 | 应用场景 | 核心目标 | 成熟度 |
|------|---------|---------|--------|
| **知识研究场景** | 知识档案构建、学术研究、资料整合 | 质量内建——保证知识来源可靠、论证严谨、偏差受控 | 🟢 A级 |
| **AI协作/代码场景** | 代码审查、安全审计、方案评审 | 攻击者视角审查——发现正常路径遗漏的边界问题、安全漏洞和崩溃路径 | 🟢 A级 |

两大分支共享核心哲学基础（证伪主义、认知偏差防御、多重验证），但在具体操作流程、角色分工和检查维度上各有侧重。本框架将分别阐述两大场景的方法论，最后说明适用边界和与其他方法论的关系。

## 2. 场景一：知识研究的对抗性审查协议

本协议源自第一性原理综合研究项目的实践验证，是一套用于知识构建场景的七模块质量内建方法论。🟢A级

### 2.1 阶段0：跨领域概念扫描（语义漂移防御）

**适用条件**：项目涉及2个及以上领域，或单领域内存在学派分歧时必须执行。

**操作步骤**：
1. **列出核心术语**：识别项目涉及的5-15个核心术语
2. **跨领域定义核查**：逐一核查每个术语在不同领域的定义差异
3. **标记歧义术语**：标记存在跨领域语义漂移的术语
4. **约定项目定义**：在Spec中明确本项目采用哪个领域的定义或综合性定义
5. **规划术语表**：预留10-20%总工作量用于术语对齐和术语表创建
6. **歧义标注规范**：歧义术语首次出现时标注领域语境（如"在物理学意义上"）

> 📚 **歧义术语示例**："第一性原理"在哲学中是第一因、在物理中是基本单元、在商业中是反类比推理。各领域搜集时内部一致，整合时才暴露矛盾。

**跳过条件**：若仅涉及单一领域且无学派分歧，可跳过。

参考模式：[cross-domain-semantic-drift.md](../../../../retrospective/patterns/methodology-patterns/research-knowledge/cross-domain-semantic-drift.md)

### 2.2 模块1：来源三级分类体系

所有资料必须先进行来源分级，不同级别来源的使用权限不同。🟢A级

| 级别 | 名称 | 判定标准 | 典型示例 | 使用权限 |
|------|------|---------|---------|---------|
| **一级来源** | Gold Standard | 经过严格同行评审或专家审核；发布机构具有公认学术权威性；内容可追溯至原始研究或第一手记录；无明显商业利益驱动 | 同行评审学术期刊（Nature/Science/PRL）；权威出版社学术专著；知名大学官方课程讲义/研究报告；政府/国际组织官方文件；当事人一手演讲/访谈/自传/原始著作 | 可作为核心论据直接引用 |
| **二级来源** | Silver Standard | 由专业机构或资深从业者发布；具有事实核查机制但非严格学术同行评审；内容基于一级来源的整理或解读；立场相对中立，有一定编辑审核 | 权威媒体深度报道（经济学人/纽约时报/财新）；知名大学公开课（MIT OCW/Coursera顶尖课程）；知名咨询公司公开报告（麦肯锡/BCG/Gartner）；权威科普机构内容（Scientific American/果壳网专业内容） | 可作为辅助论据，标注来源，鼓励交叉验证 |
| **三级来源** | Bronze Standard | 发布主体资质不明确；无正式审核流程；内容主观性较强 | 个人博客/Medium文章；自媒体内容/微信公众号文章；论坛讨论（Reddit/知乎/Stack Exchange）；社交媒体帖子 | 仅作线索发现，**必须追溯至一级或二级来源验证后方可引用** |

### 2.3 模块2：可信度四级评分体系

在来源分级基础上，对每条具体信息进行可信度评分。🟢A级

| 评分 | 判定标准 | 使用建议 | 颜色标记 |
|------|---------|---------|---------|
| **A级** | 多权威来源交叉验证（≥2个独立一级来源）；无利益冲突；可追溯至原始出处；方法论明确可检验 | 核心论据，可直接引用，作为推理基石 | 🟢 绿色 |
| **B级** | 单一权威一级来源；逻辑自洽；无明显偏差；领域内共识度高 | 可作为辅助论据，标注来源，鼓励交叉验证 | 🔵 蓝色 |
| **C级** | 二级来源；需进一步验证；存在潜在偏差；观点非普遍共识 | 仅作参考，必须标注"待验证"，不得作为独立论据 | 🟡 黄色 |
| **D级** | 存疑；无法验证；存在明显矛盾或利益冲突；来源不可追溯 | 不纳入核心档案；仅作为反面案例或问题线索记录 | 🔴 红色 |

### 2.4 模块3：五维验证流程

所有一级/二级来源必须通过五维验证。🟢A级

| 验证维度 | 核心检查项 | 操作要点 |
|---------|----------|---------|
| **1. 来源资质核查** | • 发布机构背景与同行认可度<br>• 作者学术背景与相关领域研究经历<br>• 发表渠道是否有正规审核机制<br>• 该资料被其他权威来源引用情况<br>• 作者/机构是否存在利益冲突 | 核查信誉记录；区分研究结果与个人观点 |
| **2. 交叉验证** | • 提取关键事实主张形成待验证清单<br>• 每个关键事实寻找≥2个独立来源确认<br>• 优先使用不同类型/不同立场来源三角验证<br>• 记录一致点与分歧点<br>• 分歧内容标注"争议观点"并追溯根源 | **关键事实必须交叉验证**：核心概念定义、核心论断、关键数据、重要时间/人物/事件 |
| **3. 时效性评估** | • 经典理论类（哲学/数学/基础物理）：重点考察经典版本，关注最新修正<br>• 实证研究类（科学实验/社会调查）：近10年内研究，经典实验确认是否被重复验证<br>• 前沿领域类（AI/量子计算/生物科技）：近3-5年最新研究，关注共识形成状态<br>• 历史研究类：原始史料越早越好，现代考证用最新学术成果 | 标注内容"证据保质期"，定期复审时效性敏感内容 |
| **4. 逻辑一致性审查** | • 内部逻辑：是否存在自相矛盾、循环论证、偷换概念<br>• 论据支持：结论是否有充分论据支撑，论据是否与结论直接相关<br>• 数据合理性：数据来源是否标注，统计方法是否科学，解读是否准确<br>• 论证结构：识别演绎/归纳/类比论证，检查推理有效性<br>• 反例审查：主动寻找与结论相反的证据或案例 | 主动寻找反证而非只找支持证据 |
| **5. 偏差识别** | • 识别作者/机构的立场倾向与价值预设<br>• 检查是否存在选择性使用证据<br>• 评估语言情绪化程度与客观性<br>• 识别常见认知偏差在论证中的体现<br>• 对明显偏差内容标注偏差类型与影响程度 | 参见2.5节认知偏差防御清单 |

### 2.5 模块4：认知偏差防御

认知偏差是知识构建的隐性敌人，必须系统性识别和防控。🟢A级

以下是知识研究场景中10种高频偏差：

| 偏差名称 | 审查场景典型表现 | 防御措施 |
|---------|----------------|---------|
| **确认偏差** | 只引用支持某一理论的研究，忽略反对证据；对反例进行特殊辩解 | 强制搜索反证；要求列出与结论矛盾的证据并逐一回应 |
| **幸存者偏差** | 用成功企业家案例推导成功法则，忽略失败企业；只引用被引次数高的论文 | 系统性收集正反案例；使用全样本数据分析；区分相关性与因果性 |
| **权威崇拜偏差** | "某某大师说..."直接作为论据；引用诺贝尔奖得主在非专业领域的观点 | 对权威观点同样进行五维验证；区分权威的研究结果与个人观点 |
| **事后归因偏差** | 用历史案例推导因果关系，忽略其他变量；"因为某人做了X，所以成功了" | 寻找对照案例；考虑替代解释；评估因果机制合理性 |
| **锚定效应** | 阅读顺序影响对理论的评价；第一个接触的解释框架成为后续思考锚点 | 采用不同顺序阅读对立观点；刻意从多个起点重新分析问题 |
| **框架效应** | 对同一理论使用褒义/贬义词汇影响接受度；选择性强调某些方面 | 使用中性语言描述对立观点；要求正反方都使用最强论证形式呈现 |
| **可得性启发** | 用容易想到的生动案例替代统计数据；因媒体报道频繁而高估某些事件概率 | 优先使用系统性统计数据；明确区分个案证据与统计证据 |
| **立场偏向** | 某些流派学者只引用本流派文献；商业机构发布有利于自身行业的研究 | 多立场来源对比；识别预设前提；标注作者立场可能带来的影响 |
| **过度简化偏差** | 将复杂理论简化为口号式总结；忽略重要边界条件；无信息量表述 | 保留理论必要复杂性；明确标注适用范围与边界条件 |
| **跨领域语义漂移** ⭐ | 同一术语在不同领域含义完全不同，默认按本领域含义理解 | 阶段0概念扫描；建立术语表；歧义术语首次出现标注领域语境 |

📚 详细认知科学基础参见：[13-cognitive-science-foundations.md](../../first-principles/13-cognitive-science-foundations.md)

### 2.6 模块5：异常标记模板

审查过程中发现的问题必须使用标准化标记，便于后续追溯和处理。🟢A级

| 标记 | 使用场景 | 格式模板 |
|------|---------|---------|
| **⚠️ 待验证** | 来自二级来源尚未找到一级来源支撑；关键事实仅单一来源提及未完成交叉验证；数据来源标注不清晰 | `> **⚠️ 待验证**：此处关于[具体内容]的信息来自[来源简述]，尚未完成多来源交叉验证。需追溯至[建议验证方向]后确认。` |
| **❓ 存疑** | 内容与其他权威来源存在明显矛盾；逻辑存在明显漏洞或无法自洽；数据看起来不合理；来源资质存在疑问 | `> **❓ 存疑**：此处[具体内容]与[矛盾来源]的记载存在冲突/逻辑上存在以下疑问：[具体疑点描述]。需进一步考证。` |
| **⚖️ 争议观点** | 学术领域内存在明确学派争议；不同权威来源持对立观点且均有论据；尚无定论的前沿问题 | `> **⚖️ 争议观点**：关于[具体问题]，学术界存在不同立场：<br>> - 观点A（代表人物/来源：xxx）：[核心论点]<br>> - 观点B（代表人物/来源：yyy）：[核心论点]<br>> - 争议焦点：[核心分歧点]<br>> 本档案采用[观点A/B/折中处理]，原因：[说明]。` |
| **🔍 利益冲突提示** | 研究由有利益关联的机构资助；作者与结论存在商业/职业/立场关联；发布平台可能存在系统性偏向 | `> **🔍 利益冲突提示**：本资料[来源/作者]与[相关利益方]存在[资金/雇佣/立场]关联，可能影响结论客观性。阅读时请注意甄别。` |
| **📚 歧义术语** | 术语存在跨领域语义漂移；同一术语在不同章节/来源中含义不同 | `> **📚 歧义术语**：本处"[术语名称]"在[领域]意义上使用，定义为[简要定义]。该术语在其他领域（如[其他领域]）有不同含义，详见[术语表链接]。` |

### 2.7 模块6：独立验证日志

所有验证过程必须可审计、可追溯。🟢A级

**日志要求**：
- 文件命名：`source-validation-log.md`
- 每条记录必须完整填写以下字段
- 验证过程记录必须具体可追溯，不能只写"已验证"

| 字段名 | 说明 |
|--------|------|
| 资料ID | 唯一标识符，格式：SRC-YYYYMMDD-XXX |
| 资料标题 | 资料的完整标题 |
| 来源类型 | 一级/二级/三级 |
| 可信度评分 | A/B/C/D |
| 来源URL/出版信息 | URL或出版社、出版年份、ISBN等 |
| 验证过程记录 | 五维验证的简要过程与发现 |
| 交叉验证来源列表 | 用于交叉验证的其他资料ID |
| 偏差识别结果 | 识别到的偏差类型与说明 |
| 审查结论 | 采纳/待验证/排除 |
| 审查人 | 审查者标识 |
| 审查时间 | YYYY-MM-DD |

**可审计性原则**：任何第三方根据验证日志应能复现审查过程和结论。

## 3. 场景二：AI协作/代码审查的多Agent对抗模式

本模式源自卡兹克AIHOT项目40个Agent并发审查实战验证，是一套用于AI协作和代码审查场景的攻击者视角方法论。🟢A级

### 3.1 核心原理：为什么"攻击者视角"比"证明正确"更有效

AI写代码（或做方案）的核心矛盾：
- **第一性原理**能帮AI找到好方案、定位根因，但**无法保证代码稳健上线**
- AI"自审"有根本性盲区——自己写的代码自己审，和人类一样有**确认偏差**（confirmation bias）
- 单Agent审查时，Agent倾向于"证明代码正确"而非"找出代码问题"
- 边界条件、异常输入、极端场景是AI自审最容易遗漏的地方

**典型AI自审遗漏的BUG特征**：
> 正常路径走不出来，只有站在"搞破坏"的角度才能发现。

| 典型BUG类型 | 说明 | 攻击者视角如何发现 |
|------------|------|------------------|
| OOM死循环 | 大输入触发内存爆掉→被杀→重试→再爆→无限循环 | 恶意用户提交50MB HTML搞崩worker，从入口到崩溃全链路审查 |
| 未来时间污染 | 时区错误导致"未来时间"数据排到信息流最前面，污染整条链路 | 攻击者会问"如果发布时间是未来怎么办？" |
| 性能炸弹 | 异常输入触发指数级计算/正则回溯 | 构造畸形输入触发最坏情况复杂度 |
| 缓存穿透 | 异常数据导致探活机制假阳性 | 构造异常数据干扰缓存逻辑 |
| 安全漏洞 | 未校验输入导致注入攻击 | 从恶意用户角度构造注入payload |

---

### 3.2 四大攻击者角色详解

必须启动**多个独立Agent**分别扮演不同攻击者角色，并发审查，互不干扰。🟢A级

#### 🔴 安全攻击者

| 维度 | 说明 |
|------|------|
| **攻击目标** | 数据安全、权限控制、系统完整性 |
| **典型BUG类型** | SQL注入、XSS跨站脚本、越权访问、敏感数据泄露、权限绕过、认证绕过、命令注入、路径遍历 |
| **关键检查项** | • 所有用户输入是否经过严格校验和转义<br>• 权限检查是否在每个敏感操作前执行<br>• 敏感数据（密码/Token/密钥）是否明文存储或传输<br>• 是否存在直接拼接SQL/命令的地方<br>• 文件上传是否校验类型和内容，是否存在路径遍历<br>• 认证Token是否安全存储和传输 |
| **攻击思路** | "我是一个恶意用户，我能输入什么来偷数据/删数据/拿权限？"；构造边界payload（单引号、尖括号、../、超长字符串、特殊编码）；尝试垂直越权（访问其他用户数据）和水平越权（访问管理员功能） |

---

#### 🟡 性能攻击者

| 维度 | 说明 |
|------|------|
| **攻击目标** | 系统稳定性、资源消耗、可用性 |
| **典型BUG类型** | OOM内存溢出、死循环、递归深度溢出、正则表达式回溯、性能炸弹、DoS拒绝服务、资源耗尽（文件句柄/连接池）、大文件上传内存泄漏 |
| **关键检查项** | • 是否存在无界循环/递归，终止条件是否可靠<br>• 正则表达式是否可能触发灾难性回溯<br>• 大文件/大输入是否流式处理而非全量加载到内存<br>• 并发请求是否有合理限流和资源隔离<br>• 是否存在N+1查询等性能反模式<br>• 重试机制是否有退避和最大次数限制<br>• 缓存是否有穿透/击穿/雪崩防护 |
| **攻击思路** | "我能输入什么让这个服务变慢/崩溃/重启？"；构造超大输入（MB级文本/ deeply nested JSON）；触发最坏情况算法复杂度；并发发起大量请求；构造触发无限重试的异常输入 |

---

#### 🟢 边界攻击者

| 维度 | 说明 |
|------|------|
| **攻击目标** | 输入校验、异常处理、类型安全 |
| **典型BUG类型** | 空指针异常（NPE）、类型转换错误、数组越界、格式错误数据未处理、极端值（0/负数/最大值/未来时间/过去时间）、编码问题、必填字段缺失、枚举值越界 |
| **关键检查项** | • 所有可能为null/None/undefined的地方是否判空<br>• 数值计算是否考虑0、负数、最大值、溢出<br>• 时间处理是否考虑时区、未来时间、公元前、闰年<br>• 字符串处理是否考虑空串、超长、特殊字符、多字节编码<br>• 数组/列表访问是否考虑空数组、越界索引<br>• 外部API返回是否考虑异常格式、缺失字段<br>• 类型转换是否有合法性校验 |
| **攻击思路** | "什么输入是开发者没想到的？"；空值、0、负数、""、[]、{}；极端值（Integer.MAX_VALUE、Long.MIN_VALUE）；畸形格式（错误JSON、非法编码）；边界时间（1970-01-01、2099-12-31） |

---

#### 🔵 时序攻击者

| 维度 | 说明 |
|------|------|
| **攻击目标** | 并发安全、重试机制、分布式一致性 |
| **典型BUG类型** | 竞态条件（Race Condition）、重复提交导致的数据重复/不一致、重试风暴、缓存穿透/击穿、幂等性缺失、死锁、分布式锁失效、部分失败导致数据不一致、消息重复消费 |
| **关键检查项** | • 并发写共享资源是否加锁，锁粒度是否合理<br>• 接口是否实现幂等性（重复调用结果一致）<br>• 重试是否有指数退避和最大次数，是否可能引发重试风暴<br>• 缓存与数据库双写是否有一致性保证<br>• 事务边界是否正确，是否存在部分提交<br>• 消息队列消费是否考虑重复消费、乱序、丢失<br>• 超时设置是否合理，是否可能过早超时导致级联失败 |
| **攻击思路** | "如果两个请求同时来会怎样？如果请求超时重试会怎样？如果服务处理到一半崩溃会怎样？"；并发发起相同请求；模拟超时和重试；模拟部分失败（数据库写入成功但缓存更新失败）；打乱消息顺序重复消费 |

---

### 3.3 多Agent对抗执行流程

🟢A级

| 步骤 | 操作 | 要点 |
|------|------|------|
| **步骤1：明确审查目标和范围** | • 明确审查目标（特定功能/模块/整个项目）<br>• 明确审查维度（安全/性能/边界/时序，或全部）<br>• 提供代码上下文（仓库路径、相关文件、架构说明） | 范围越明确，审查越聚焦。如果是全面审查，建议分模块进行。 |
| **步骤2：并发启动四大攻击者Agent** | • 每个角色由独立Agent扮演<br>• 要求每个Agent独立思考、互不干扰<br>• 明确告知Agent："你的目标是找出问题，不是证明代码正确"<br>• 提供四大角色的检查清单作为思考框架 | **关键**：不能让一个Agent同时扮演多个角色——角色混同会降低攻击性，回到"证明正确"的思维定式。 |
| **步骤3：攻击者输出问题清单+复现路径** | 每个攻击者按严重程度输出问题：<br>• P0：崩溃/安全漏洞/数据丢失——必须修复<br>• P1：严重问题/性能瓶颈——上线前修复<br>• P2：一般问题/边界case——评估后修复<br>• P3：优化建议/代码质量——可选修复<br>每个问题必须附带：攻击路径、复现条件、影响范围、根因分析 | 要求Agent给出**具体的复现步骤和输入示例**，而不是泛泛的"可能存在XX问题"。 |
| **步骤4：汇总交叉验证** | • 合并四个攻击者的问题清单，去重<br>• 重点关注**不同攻击者发现的关联问题**（例如：边界攻击者发现的空值可能被性能攻击者触发OOM）<br>• 按优先级排序，制定修复计划 | 交叉验证往往能发现更深层的系统性问题。 |
| **步骤5：修复后回归对抗** | • 修复发现的P0/P1问题<br>• **必须对修复后的代码重新执行对抗式审查**<br>• 验证两点：(1)原问题是否真正修复；(2)修复是否引入新问题<br>• 直到所有P0/P1问题修复，P2问题评估后决定是否阻塞上线 | 修复往往会引入新BUG——这是常态，不是例外。回归对抗是必须的。 |

**定期全局审查**：每2-3周对整个项目做一次"从第一性原理出发的对抗式审查"，覆盖架构、依赖关系、代码质量、文档对应，每次都能挑出之前没注意到的技术债。

### 3.4 单Agent自审 vs 多Agent对抗对比

🟢A级

| 维度 | 单Agent自审 | 多Agent对抗式审查 |
|------|-----------|-----------------|
| **审查视角** | "证明代码正确" | "证明代码有问题" |
| **思维模式** | 防御性（检查有没有明显错误） | 进攻性（构造攻击来击穿系统） |
| **角色分工** | 一个角色，思维定式 | 四个独立攻击者角色，多角度并发 |
| **边界覆盖** | 常规边界、正常路径 | 极端边界+恶意边界+异常路径 |
| **BUG发现率** | 低（确认偏差、自证预言） | 高（攻击者无偏见，以搞破坏为目标） |
| **时间成本** | 低 | 中高（多Agent并发，但发现问题价值高） |
| **适用阶段** | 开发中快速自检 | 功能完成后、上线前、重大重构后、定期全局审查 |
| **典型遗漏** | 边界条件、极端输入、并发问题、安全漏洞 | （系统性覆盖四大类问题） |

---

### 3.5 与第一性原理的闭环关系

🟢A级

对抗式审查不是第一性原理的替代品，而是其互补闭环：

```
┌─────────────┐     找到好方案      ┌─────────────┐
│  🔵 第一性原理  │ ───────────────→ │   开发实现    │
│  （生成层）    │                   │             │
└─────────────┘                   └──────┬──────┘
     ↑                                   │
     │ 发现问题，回到根因                   │ 实现完成
     │                                   ↓
     │                             ┌─────────────┐
     └─────────────────────────────│  🔴 对抗式审查  │
                                   │  （验证层）    │
                                   └──────┬──────┘
                                          │
                        ┌─────────────────┴─────────────────┐
                        │ 通过                              │ 不通过
                        ↓                                   ↓
                  ┌─────────────┐                     回到第一性原理
                  │ ✅ 稳定上线   │                     定位根因修复
                  └─────────────┘
```

| 方法论 | 核心职责 | 思维方向 | 目标 |
|--------|---------|---------|------|
| **第一性原理** | 生成层 | 正向构建：从基本原理出发找最优方案 | 保证**做对的事**——方向正确、根因被真正定位 |
| **对抗式审查** | 验证层 | 反向证伪：从攻击者视角找所有可能问题 | 保证**把事做对**——实现稳健、边界被覆盖、能稳定上线 |

二者构成完整的"生成-验证"闭环，缺一不可：
- 只有第一性原理没有对抗审查：可能"正确地做错事"——方向对但实现漏洞百出，上线就崩
- 只有对抗审查没有第一性原理：可能"正确地做错事"——把一个根本错误的方案做得滴水不漏

## 4. 适用边界与反模式

### 4.1 什么时候该用对抗性审查

🟢A级，以下场景强烈推荐使用：

| 场景 | 适用分支 | 原因 |
|------|---------|------|
| **核心/关键模块** | 两个分支都适用 | 关键模块故障影响大，值得投入审查成本 |
| **安全敏感功能** | AI协作/代码场景 | 安全漏洞可能导致数据泄露、资金损失 |
| **上线前质量门禁** | AI协作/代码场景 | 上线后修复成本是上线前的10-100倍 |
| **知识档案构建** | 知识研究场景 | 知识档案一旦形成会被反复引用，错误会传播放大 |
| **重大重构后** | AI协作/代码场景 | 重构影响范围大，容易引入回归问题 |
| **学术研究/资料整合** | 知识研究场景 | 保证研究结论建立在可靠证据基础上 |
| **新模型/新工具上线后** | AI协作/代码场景 | 测试新模型能力，同时发现技术债 |

---

### 4.2 什么时候不需要用对抗性审查

🔵B级，以下场景投入产出比低，不必使用：

| 场景 | 原因 |
|------|------|
| **快速原型/PoC验证** | 原型本来就是用来快速试错的，健壮性不是首要目标 |
| **草稿/个人笔记** | 非正式内容，不对外发布，错误影响范围小 |
| **一次性脚本/临时工具** | 用完即弃，不需要长期维护 |
| **创意写作/灵感发散阶段** | 对抗思维会抑制创造力，发散阶段需要的是数量不是质量 |
| **时间极度紧迫的应急处理** | 先止血再复盘，应急阶段首要目标是快速恢复 |

---

### 4.3 常见反模式

🔴D级，以下做法会让对抗性审查流于形式甚至产生反效果：

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| **表演式审查：只走流程不真找问题** | 为了"有审查记录"而审查，Agent说一些"整体不错，建议加些注释"之类的空话 | 明确要求每个攻击者必须输出至少N个具体问题（哪怕最后验证不是真BUG）；要求给出复现路径 |
| **角色混同：让一个Agent同时扮演多个角色** | 单Agent无法真正并发多角度攻击，角色之间会相互干扰，回到"证明正确"的思维定式 | 必须启动独立Agent分别扮演四个角色，要求它们互不干扰独立思考 |
| **过度攻击导致审查瘫痪** | 吹毛求疵，把代码风格、命名偏好等非关键问题当P0阻塞上线，忽略优先级 | 按P0/P1/P2/P3分级；P0/P1必须修复，P2评估后决定，P3只是建议；记住：完美是好的敌人 |
| **一次性审查：只做一次不做回归对抗** | 修复往往引入新问题；系统在演进，新代码会引入新BUG | 修复后必须回归对抗；每2-3周做一次定期全局审查 |
| **只用单Agent做"对抗式审查"** | 单Agent有确认偏差，无法真正跳出"证明正确"的思维框架 | 必须多Agent并发，每个角色独立思考 |
| **对抗审查替代第一性原理** | 对抗审查找问题但不保证方向对，可能把一个根本错误的方案做得"没有BUG" | 第一性原理+对抗审查形成闭环，缺一不可 |
| **只看代码不看架构** | 架构级问题比代码级问题影响更大，代码层面没问题不代表系统层面稳健 | 定期全局审查必须覆盖架构、依赖、设计层面 |

### 5.2 与质量内建：对抗性审查是质量内建的核心手段

🟢A级

质量内建（Quality Built-In）的核心思想是：质量不是测试出来的，而是构建过程中内建进去的。对抗性审查是质量内建在知识构建和AI协作场景的具体实践：
- **知识研究场景**：通过七模块协议，在知识采集、验证、整合每个环节都植入对抗验证
- **AI协作/代码场景**：通过多Agent攻击者模式，在开发完成后第一时间植入对抗审查

---

### 5.3 与代码审查：对抗性审查是代码审查的"攻击模式升级"

🔵B级

| 维度 | 传统代码审查 | 对抗式审查 |
|------|-----------|-----------|
| 视角 | 同事视角："有没有明显问题" | 攻击者视角："我怎么搞破坏" |
| 思维 | 防御性、建设性 | 进攻性、破坏性 |
| 角色 | 审查者（人或单Agent） | 四大独立攻击者并发 |
| 覆盖 | 常规问题、代码风格 | 安全/性能/边界/时序四大类极端问题 |
| 产出 | 评论建议 | 问题清单+复现路径+攻击payload |

对抗式审查不替代传统代码审查，而是其强有力的补充——传统审查适合发现逻辑错误、设计问题、代码质量问题，对抗式审查适合发现边界、安全、性能、并发等"正常路径想不出来"的问题。
