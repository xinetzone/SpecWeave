# 常见问题（FAQ）

## Q1: Harness Engineering和Context Engineering有什么区别？

**A**: Harness Engineering是Context Engineering的**工业化升级**，包含但远超Context Engineering：

- **Context Engineering**专注于：如何给模型喂对信息、信息筛选、上下文装配、分段结构化
- **Harness Engineering**除了Context，还包含：
  - 工具设计与管理（工具签名、工具隔离）
  - 记忆与状态持久化（Workspace文件系统）
  - 反馈回路（Linter/Test/CI自动闭环）
  - 架构边界（Sub-Agent隔离、双阶段架构）
  - 熵管理（文档园丁、定期清理）
  - 硬护栏（白名单、Linter拦截、Agent审稿三层防护）

简单说：Context Engineering是"给马看地图"，Harness Engineering是"给马造整条高速公路"。

## Q2: "上下文越少越好"是否意味着应该用小上下文窗口？

**A**: 不是。

- 大窗口是**安全垫**，不是**仓库**
- 大窗口是为了让你有空间**选择**该放什么，而不是让你把所有东西都倒进去
- 原则是"**精挑细选**"，不是"**硬塞**"
- 类比：你的办公桌很大（大窗口），但你工作时只会把当前需要的文件放在桌上（精简Context），不会把整个档案柜都搬上来

如果你的Context策略是"反正窗口大，全塞进去"，那大窗口反而害了你——信息越多，模型越容易"逛超市"、被干扰、产生幻觉。

## Q3: "Agent数量不超过3个"是硬性规定吗？

**A**: 这是来自血泪教训的**经验法则**，不是物理定律。

- 实测经验：当Agent数量堆到6个时，编排层（Orchestrator）开始频繁选错Agent——"该派给谁"本身成了新的决策难题，错误率飙升
- 建议思考顺序：
  1. 这个需求能不能用Skill解决？（首选）
  2. 如果不行，能不能在现有Agent里加个工具？（次选）
  3. 确实需要独立Context/独立工具集/独立Prompt时，再加新Agent
- 复杂系统可以在3个Agent内按职责清晰划分，而非无限增加
- 如果你的系统确实需要更多Agent，要确保Orchestrator的路由逻辑足够简单清晰（比如按任务类型硬编码路由，而不是让模型自己选）

## Q4: Skill和Sub-Agent应该如何区分？

**A**: 判断标准很简单：

| 维度 | Skill（技能） | Sub-Agent（子智能体） |
|------|--------------|---------------------|
| 成本 | 廉价 | 昂贵 |
| Context | 无独立Context，复用调用者的Context | 有独立Context Window |
| 工具集 | 没有自己的工具，就是一个函数 | 有自己独立的工具集 |
| Prompt | 没有自己的系统Prompt，就是函数签名+docstring | 有自己独立的系统Prompt |
| 适用场景 | 原子操作、确定性任务、不需要复杂推理 | 需要多步推理、需要多工具协同、需要独立决策 |
| 例子 | `parse_resume`、`score_match`、`query_calendar` | 人岗匹配Agent（要解析简历+调用RPA+打分+写入结果） |

**决策原则**：能用Skill就别加Agent。如果一个函数能搞定，就不要起一个新Agent。只有当确实需要"独立思考空间"时，才用Sub-Agent。

## Q5: 已经有大量AGENTS.md了还需要写Linter吗？

**A**: 需要。AGENTS.md是宪法，但宪法需要执法机构。

- **文档的问题**：
  - Agent可能没读到（Prompt太长被截断）
  - Agent可能"创造性解读"（"我理解的规则是这样的"）
  - Agent可能"忘了"（读到了但执行时忽略了）
  - 规则更新了，之前的对话历史里还有旧规则
- **Linter的优势**：
  - 机器执行，100%覆盖，不会偷懒、不会误解、不会忘记
  - 每次运行都检查，规则更新立即生效
  - 不通过就是不通过，没有"创造性解读"的空间

**最佳实践**：
- 每条AGENTS.md规则背后都对应一个真实失败案例（Mitchell Hashimoto原则）
- 然后问：这个规则能不能写成代码检查？
- 能机器化的优先机器化，写进Linter/CI
- 确实无法自动化的原则性指导，才留在AGENTS.md里

## Q6: 双阶段架构中plan.md应包含什么？

**A**: plan.md应该包含以下要素：

1. **任务目标**：用户到底要什么，验收标准是什么
2. **分步执行计划**：拆分成可执行的步骤，每步有明确的输入输出
3. **Workspace文件约定**：这个任务会读哪些文件、写哪些文件、临时文件放哪里
4. **已知约束**：不能做什么、有哪些硬规则必须遵守
5. **异常处理指引**：遇到什么情况应该停下来请示，什么情况可以自己处理

**关键原则**：
- Initializer写完plan.md就退出，**不执行任何实际操作**
- Executor启动时是全新Context，**只读取plan.md**，不共享Init阶段的思考过程
- plan.md要足够具体，让一个"新接手的人"（Executor）能照着做，不需要问为什么
- 但也不要太啰嗦——plan.md本身也会占Context，要精炼

## Q7: "智能体审智能体"会不会增加成本延迟？

**A**: 会，但这是**必要的投资**，尤其是对对外说话/动用户数据的场景。

**成本分析**：
- 第一层（白名单）：几乎零成本，就是配置检查
- 第二层（Linter）：机器执行，毫秒级，成本可忽略
- 第三层（Agent审稿）：确实会增加一次模型调用，增加延迟和token成本

**收益分析**：
- 一次对外发错消息的事故，代价可能是几百上千倍的护栏成本
- 前两层护栏会过滤掉80%以上的问题，第三层只需要处理剩下的疑难案例
- 长期来看，减少事故、减少回滚、减少用户投诉的收益远大于护栏成本

**适用场景**：
- ✅ 必须做：对外发消息、操作用户数据、涉及金钱交易
- ⚠️ 可选做：内部工具、非关键路径、错误代价低的场景
- ❌ 不需要：纯只读、纯分析、不产生外部影响的任务

## Q8: 小团队如何落地"文档园丁"？

**A**: 不需要一开始就做复杂的后台Agent，**从小处开始，持续迭代**：

1. **第一步：从简单定时脚本开始**
   - 每周一早上跑一次脚本，扫描：死链、30天没更新的文档、TODO/FIXME标记
   - 输出一份报告发到群里，谁的孩子谁抱走
   - 不需要自动提交PR，先人工处理
2. **第二步：自动化小修复**
   - 死链自动检测+自动修复（能修的直接修，不能修的报出来）
   - 格式自动修复（如Markdown lint）
3. **第三步：做成定时任务**
   - 配置成CI定时任务，每天凌晨跑
   - 自动提交PR，人review后合并
4. **关键原则：持续小额还债**
   - 不要攒到"我们来一次大重构吧"
   - 每天修一点，每周清一点，技术债不会爆雷
   - 就像打扫卫生——每天擦桌子比一年扫一次灰容易多了

## Q9: Harness Engineering只适用于编程Agent吗？

**A**: 不是，适用于**所有需要Agent可靠工作的场景**。

凡是Agent需要以下能力的场景，Harness Engineering都适用：
- 多步执行（不是一问一答，是要做多件事）
- 调用工具（要查数据库、调API、操作软件）
- 对外交互（要发消息、写邮件、接触用户）
- 持久状态（任务要跨会话、需要断点续传）

**其他场景示例**：
- **招聘**：本文悟空案例，不是编程但适用
- **客服**：多轮对话、查用户数据、发消息、需要护栏防止说错话
- **数据分析**：多步分析、写文件（报表）、调用查询工具
- **运维**：操作服务器、执行命令、需要护栏防止误删库
- **内容创作**：查资料、写草稿、需要审稿Agent审核

Harness Engineering本质上是"如何构建可靠的Agent工作环境"，和具体场景无关。

## Q10: 如何判断团队现在处于哪一代范式？

**A**: 对照以下特征：

| 代际 | 特征 |
|------|------|
| **第一代（Prompt Engineering）** | - 还在卷话术："怎么说才能让模型听我的"<br>- 主要工作是调提示词、试措辞、加"你是一个专家..."<br>- 一个Prompt从50行调到500行<br>- 出问题了第一反应是"我Prompt写得不好，再改改" |
| **第二代（Context Engineering）** | - 开始重视Context：给模型什么信息、怎么筛选、怎么排序<br>- 做RAG、做上下文装配、做信息分段<br>- 知道"垃圾进垃圾出"，开始关注输入质量<br>- 但主要关注点还是在"给模型看什么" |
| **第三代（Harness Engineering）** | - 开始**系统设计**：工具边界怎么划、状态存在哪、反馈回路怎么建<br>- 有硬护栏：Linter/CI不通过就是不行，不是靠模型"自觉遵守"<br>- 状态写文件不塞Context，Workspace是唯一真相<br>- 专才Agent分工，不是一个万能Agent包打天下<br>- 有Agent审稿，不是自己说自己对<br>- 有熵管理机制，持续清理技术债<br>- 出问题了第一反应是"我们的Harness哪里有缺陷，怎么从系统层面防止再发"，而不是"再改改Prompt" |

简单自测：如果你们团队遇到Agent问题，第一反应是"改Prompt"，那还在第一代；如果第一反应是"加个检查机制"，那已经在第三代了。
