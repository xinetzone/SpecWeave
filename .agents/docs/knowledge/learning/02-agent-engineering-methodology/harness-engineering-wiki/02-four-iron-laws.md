# 四条反直觉铁律

## 铁律对比总表

| 铁律 | 人的本能反应 | Harness真相 |
|------|-------------|-------------|
| 上下文越少越好 | 窗口越大越好，能塞多少塞多少 | 上下文是稀缺资源，会被污染/干扰/让模型"逛超市" |
| 专才Agent永远赢过通才Agent | 一个万能Agent搞定所有事 | 通才在工具列表里逛超市，拆成Sub-Agent各司其职 |
| 状态要写文件，不要塞上下文 | 状态放上下文里最方便 | Workspace是真相，Context Window只是当前工位 |
| 能写成Linter的约束，别写成文档 | 写好文档大家就会遵守 | 文档是建议，Linter/CI是强制；自然语言可被创造性解读，代码不能 |

---

## 铁律一：上下文越少越好

### 核心观点
上下文是**稀缺资源**，不是无限仓库。不要把"模型支持200K"当成"可以塞200K"。

### 为什么少即是多
1. **污染效应**：无关信息会干扰模型判断，就像考试时桌子上堆满无关书籍反而影响发挥
2. **干扰效应**：信息过多会分散注意力，模型容易被不重要的细节带偏
3. **"逛超市"现象**：工具和信息太多时，模型会在选项间徘徊，难以做出决策
4. **长上下文阈值效应**：超过一定阈值后，模型对中间内容的关注度显著下降

### 实践原则
- 精挑细选上下文，如同Code Review时只看必要的diff
- 窗口大是安全垫，不是仓库——大窗口是为了让你有空间选择该放什么，不是让你把所有东西都倒进去
- 每次上下文构造都要问：这个信息是完成当前任务必须的吗？

## 铁律二：专才Agent永远赢过通才Agent

### 核心观点
通才Agent在长工具列表里"逛超市"，工具选错率高；把任务拆给Sub-Agent，每个只装需要的工具，效果显著更好。

### 为什么专才赢
- **通才问题**：10+个工具摆在面前，模型需要花精力"决定用哪个"，决策成本高，选错概率大
- **专才解法**：每个Sub-Agent只看到自己需要的3-5个工具，决策空间小，准确率高
- **成本公式**：Agent是昂贵的（独立Context、独立工具集、独立Prompt），Skill是廉价的（一个函数+明确签名）

### 实践原则
- Agent数量控制在≤3个，Skill可以无限加
- 能用Skill解决的，就不要加Agent
- 每个Agent只负责一类职责，只拥有完成该职责所需的最小工具集

## 铁律三：状态要写文件，不要塞上下文

### 核心观点
Workspace是真相之源，Context Window只是当前工位。把Workspace当成Agent的Git仓库。

### 类比说明
| 概念 | 类比 |
|------|------|
| Context Window | 办公桌上的当前工位——只放正在处理的东西 |
| Workspace | 公司档案库/Git仓库——所有状态的永久记录 |

### 为什么写文件
1. **可回放**：任何时候都能回到之前的状态，会话间不丢失记忆
2. **可审计**：状态变更有迹可循，谁在什么时候改了什么一目了然
3. **可断点续传**：任务中断后，从文件状态恢复，不用从头开始
4. **跨Context接力**：Initializer和Executor不共享Context，只通过Workspace文件通信
5. **可diff**：状态变更可以做版本对比，便于调试

### 实践原则
- 所有持久状态写入Workspace文件：state.json、chat_history.log、jd.md等
- Context只加载当前步骤需要的状态
- 用文件锁（如rpa_lock/{batch_id}.json）处理并发和事务边界

## 铁律四：能写成Linter的约束，别写成文档

### 核心观点
文档是建议，Linter/CI是强制。自然语言可以被创造性解读，代码不能。

### 为什么Linter更可靠
- **文档问题**：
  - 人会偷懒，不看文档
  - 人会误解，"我以为是这个意思"
  - 人会忘记，昨天记得今天忘了
  - 文档可能过时，和实际代码不一致
- **Linter优势**：
  - 机器执行，不会偷懒
  - 规则明确，没有歧义
  - 每次运行都检查，不会忘记
  - 错误信息直接告诉你哪里错了、怎么改

### Mitchell Hashimoto在Ghostty的实践[1]
- AGENTS.md作为"宪法"，每条规则背后都对应一个真实的失败案例
- 但能机器化的规则，绝不留在自然语言层面
- 规则从失败中来，到Linter中去

### 实践原则
- 每条规则先问：能不能写成代码检查？
- Linter错误信息本身就是上下文工程——不仅说"错了"，还要解释"为什么错"+"正确做法是什么"
- 文档只放那些确实无法机器化的原则性指导
