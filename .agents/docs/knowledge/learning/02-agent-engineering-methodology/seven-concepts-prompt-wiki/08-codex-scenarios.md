# Codex/Agent开发基础：安全原则与标准结构

---

## 1. 章节引言

上一章我们学习了Work场景的五段式结构，用来生成专业文档和长内容。但Work场景的输出是"静态"的——模型写一份文档给你，改不改、怎么改是你自己的事。

Codex/Agent开发场景完全不同：Agent会**真的动手操作**——读写你的文件、修改代码、执行命令、安装依赖、跑测试……它不是在"写一份代码给你看"，而是直接在你的项目里改代码。

这是效率最高的场景，也是**风险最高**的场景：
- 改坏你的代码，导致项目跑不起来
- 误删重要文件
- 自己装一堆你不想要的依赖
- 修改核心配置文件导致整个项目挂掉
- 执行危险命令（比如rm -rf）

所以Codex场景的第一原则不是"怎么写Prompt让它写出好代码"，而是**"怎么防止它搞破坏"**。这一章我们学习Codex场景的安全原则、标准结构和安全Prompt模板；8个高频实战场景模板见配套文档[08b-codex-examples.md](08b-codex-examples.md)。

> 🟢 **来源标注**：本章内容基于OpenAI Codex官方最佳实践和GPT-5.6 Agent开发场景的安全实践整理。

## 2. Codex场景适用范围

Codex/Agent适合需要**读写文件、执行命令、真实操作代码**的开发任务：

| ✅ Codex场景典型任务 | ❌ 不适合Codex的任务 |
|---------------------|---------------------|
| 添加新页面、新组件、新功能 | 代码片段解释、概念学习 |
| 修复Bug、排查问题 | 架构设计讨论（先在Chat里聊清楚） |
| 代码重构、优化性能 | 简单代码片段生成（Chat就能做） |
| 编写单元测试、集成测试 | 技术方案选型讨论 |
| 添加API接口、后端逻辑 | 长文档、报告、翻译（用Work场景） |
| 更新文档、注释、README | 头脑风暴、创意发散（Chat场景） |
| 配置修改、CI/CD脚本 |  |
| 依赖升级、版本兼容处理 |  |

> 🔵 **B级经验**：判断要不要用Codex，问自己一个问题："这个任务需要读/写我项目里的多个文件吗？"如果需要，就用Codex；如果只是写个独立的代码片段，Chat就够了。

## 3. 🔴 Codex场景安全第一原则

在Codex场景，安全永远是第一位的，比"写出完美代码"重要100倍。记住这四条铁律：

### 3.1 原则一：最小权限原则

只给它完成任务必需的最小权限：
- 只允许修改特定目录/文件，不要说"整个项目随便改"
- 能读不能写的目录要明确说
- 禁止执行危险命令（rm、sudo、curl | bash等）
- 是否允许安装新包要明确说，默认不允许

### 3.2 原则二：明确可修改范围，列白名单而不是黑名单

❌ 错误："不要改配置文件"——它可能还是改了其他你没想到的核心文件
✅ 正确："只允许修改：src/components/xxx/、src/app/api/xxx/，其他所有文件一律不要碰"

**白名单思维**：说清楚哪些可以改，其他的都不行；不要说哪些不能改（你永远列不完）。

### 3.3 原则三：设置明确的停止条件（Checkpoint）

遇到什么情况必须停下来问你，不能自己"发挥"：
- 需要修改核心文件（layout.tsx、middleware、package.json等）
- 需要安装新包
- 工具连续两次失败
- 发现需要大改架构或Schema
- 不确定怎么做的时候，必须问

### 3.4 原则四：动手前先列计划，确认后再改

在让它真的改文件之前，必须先让它列出：
- 它打算修改哪些文件
- 每个文件改什么
- 大概的步骤

你确认没问题了，再让它开始动手。这是防止大规模返工的最后一道防线。

> 🟢 **A级原则**：记住Codex场景的优先级：不破坏 > 可运行 > 代码质量。首先保证别搞坏你的项目，其次保证代码能跑，最后才是写得好不好看。

## 4. Codex Prompt核心要素

Codex场景的Prompt比Work场景多两个关键部分：明确的文件白名单和工具白名单。完整结构是6段：

### 4.1 # Context：项目上下文

和Work场景类似，但要更具体到代码层面：
- 项目是什么技术栈（Next.js 14 + TypeScript、Python FastAPI + React等）
- 关键目录位置（组件在哪、API在哪、类型定义在哪）
- 已经安装了哪些依赖，用的什么状态管理、UI库、ORM
- 代码风格约定（TypeScript strict、要不要类型注解、用不用any等）
- 现有代码的模式（比如表单都用react-hook-form+zod）

### 4.2 # Request：具体功能需求

明确要做什么功能，**完成标准必须可验证**：
- 不要说"做一个用户管理功能"，要拆成具体的功能点
- 每个功能点说清楚交互逻辑
- 完成标准必须是可以客观检查的：
  - ✅ "所有代码有完整TypeScript类型，不要用any"
  - ✅ "表单验证用react-hook-form+zod，和现有风格一致"
  - ✅ "所有异步操作有loading状态和错误提示"
  - ❌ "代码质量高、用户体验好、优雅"

### 4.3 # Constraints：文件边界与硬约束

这是**最最重要**的部分，分三部分：

1. **✅ 允许修改的文件/目录（白名单）**：明确列出来哪些可以改
2. **❌ 绝对禁止修改的文件/目录**：核心文件、配置文件等
3. **其他硬约束**：能不能装新包、能不能用新库、数据库操作规则等

### 4.4 # Checkpoint：必须停下来的情况

列清楚遇到什么情况必须停，不能自己继续：
- 需要修改不在白名单里的文件
- 需要安装新包
- 工具调用（读/写文件、执行命令）连续失败两次
- 发现Schema/架构和预期不符
- 需要密钥、密码等敏感信息
- 有多个方案不确定选哪个

### 4.5 # Tools：允许使用的工具白名单

明确允许它用哪些工具：
- 读文件：✅ 允许
- 写文件：✅ 允许（仅白名单目录）
- 执行命令：✅ 允许（但禁止哪些命令）
- 安装包：❌ 默认不允许，需要确认
- 执行测试：✅ 允许
- 删除文件：❌ 绝对禁止（除了它自己新建的临时文件）

### 4.6 最后：先列计划再动手

最后一句话必须是：**"开始之前，先列出你打算修改的所有文件和每个文件的修改内容，我确认后你再开始动手改。"**

## 5. Codex完整安全Prompt模板（可复制）

直接复制这个模板，填空，这是经过验证的安全模板。8个具体场景的实战模板见[08b-codex-examples.md](08b-codex-examples.md)。

```
# Context
当前目录是一个[技术栈，如：Next.js 14 + TypeScript + App Router]项目。
- UI组件库：[如：shadcn/ui，已有Button/Card/Input]
- 状态管理：[如：Zustand]
- 表单：[如：react-hook-form + zod]
- API路由：[如：src/app/api/]
- 组件目录：[如：src/components/]
- 类型定义：[如：src/types/]
- ORM/数据库：[如：Prisma，schema在prisma/schema.prisma]
- 已有依赖：[说清楚已经装了什么，不要加新包]
- 代码风格：[如：TypeScript strict模式，不要用any，函数组件]

# Request
请帮我实现[功能名称]，具体要求：
1. [功能点1，如：列表页展示所有数据，支持分页和搜索]
2. [功能点2，如：新增/编辑表单，弹窗形式]
3. [功能点3，如：删除前有确认提示]
4. [功能点4，如：权限控制，只有管理员能操作]

完成标准（必须全部满足）：
- [ ] [标准1，如：所有代码有完整TypeScript类型，不要用any]
- [ ] [标准2，如：表单验证用react-hook-form+zod，和现有代码风格一致]
- [ ] [标准3，如：所有异步操作有loading状态和错误提示]
- [ ] [标准4，如：API错误有统一处理，不要silent fail]
- [ ] [标准5，如：代码符合项目现有风格，不要自己发明新模式]

# Constraints（红线，严格遵守）
✅ 允许修改/新建的文件/目录（只允许动这些）：
- [如：src/app/[功能名]/ 下的页面文件，可以新建]
- [如：src/app/api/[功能名]/ 下的API路由，可以新建]
- [如：src/components/[功能名]/ 下的组件，可以新建]
- [如：src/types/[功能名].ts，可以新增类型定义]

❌ 绝对不要修改以下内容，碰都不要碰：
- 不要修改src/app/layout.tsx
- 不要修改src/middleware.ts
- 不要修改prisma/schema.prisma（schema已经定死了）
- 不要修改package.json，不要添加任何新的npm包
- 不要修改现有的其他路由和页面
- 不要修改配置文件（next.config.js、tsconfig.json等）

其他规则：
- 所有数据库操作走现有的prisma client实例，不要新建连接
- 敏感信息（API密钥、数据库密码）不要硬编码，用环境变量
- 不要写任何console.log，用完的debug代码删掉
- 如果需要用工具函数，先看src/lib/下有没有现成的，不要重复写

# Checkpoint
遇到以下任何一种情况，**立即停止**，告诉我问题是什么，等我回复，不要自己尝试解决或继续：
1. 需要修改/新建的文件不在上面的白名单里
2. 需要安装任何新的npm包
3. 需要修改prisma schema或数据库结构
4. 需要修改layout、middleware、配置文件等核心文件
5. 工具调用（读文件、写文件、执行命令）连续两次失败
6. 发现现有代码结构/依赖版本和你预期的不一样，可能有兼容问题
7. 需要我提供API密钥、数据库连接串等敏感信息
8. 有多个实现方案不确定选哪个
9. 你觉得这个需求需要超过[如：300行]代码，或者需要大改架构

# Tools
允许使用的工具：
- ✅ 读文件：可以读项目里的任意文件，帮助理解现有代码
- ✅ 写文件：仅限上面白名单里的目录和文件
- ✅ 执行命令：可以运行npm run dev、npm run test、npm run typecheck等开发/测试命令
- ❌ 禁止执行：rm、del、sudo、curl | bash、npm install -g、git push等危险命令
- ❌ 禁止删除任何不是你这次新建的文件
- ❌ 禁止安装新包（npm install、yarn add、pnpm add都不行）

⚠️  重要：开始动手改文件之前，先做这件事：
1. 先读一下相关的现有文件，理解现有代码模式
2. 然后列出你打算修改/新建的所有文件清单，以及每个文件大概要做什么
3. 等我确认说"可以开始"之后，你再动手写代码

现在开始第一步，先列文件清单给我确认。
```

## 6. Codex常见事故与预防

| 事故类型 | 怎么发生的 | 怎么预防 |
|---------|-----------|---------|
| **文件被误删** | Agent清理临时文件时删多了，或者重构时删错了 | Constraints明确禁止删除非本次新建的文件；重要文件用git commit先提交 |
| **核心配置被改** | Agent"好心帮你优化"配置，或者为了解决问题改了配置 | 白名单+黑名单：layout、middleware、package.json、config文件明确列在禁止修改列表 |
| **偷偷装了新包** | 你没说能不能装包，它遇到问题就自己npm install了 | Constraints明确"不要加任何新包"，需要装包必须Checkpoint确认 |
| **大重构/改了不相关的代码** | 修Bug时"顺手"重构了整个模块，导致一堆回归问题 | 明确"只改解决这个问题必需的代码，不要重构"；强调白名单 |
| **执行了危险命令** | 为了"解决问题"执行了rm -rf或者其他危险命令 | Tools里明确禁止危险命令；执行命令前如果是写/删操作必须确认 |

> 🟡 **C级提醒**：在让Codex开始工作之前，**先git commit一次**，把当前干净的代码状态保存下来。就算它搞坏了，你也可以轻松回滚。这是最简单也最有效的防护措施。

## 7. Steer vs Queue在Codex中的使用

在Codex/Agent运行过程中，你有两个选择：

### 7.1 Steer（按Enter）：当前轮直接调整方向

当你发现Agent做错了、方向跑偏了、或者想补充要求的时候，**立即打断，直接说**，不要等它做完：
- "不对，不要用Dialog，用Drawer组件"
- "停一下，这个API的参数我记错了，应该是xxx"
- "等一下，不要删那个文件，那个还要用"

**什么时候用Steer**：
- 方向错了，继续错下去浪费时间
- 它违反了约束（在改不该改的文件）
- 你发现你漏说了重要信息
- 它在做无用功

### 7.2 Queue（按Tab）：排队等下一轮

如果它现在做的事情没问题，但是你想到了下一个要调整的点或者要加的功能，不要打断它，**等它做完当前这一轮**，下一轮再说：
- （它在写表单页面）你想到"写完之后顺便加个表单验证"→ 等它写完再说，不要打断
- （它在列文件清单）你想到"对了，记得加loading状态"→ 等它列完确认开始写的时候再说

**什么时候用Queue**：
- 当前方向是对的，只是有补充要求
- 你想到的是下一步的事情，不影响当前操作
- 打断反而会让它上下文混乱

> 🔵 **B级经验**：如果是"它在做不该做的事"→ 立即停（Steer）；如果是"做得对，还要加XX"→ 等一下（Queue）。
