# 6组Before/After实战对照

---

## 1. 章节引言

上一章我们学习了新范式的核心规则——做减法而非做加法，删除5类冗余内容，按任务复杂度控制长度。但规则说起来容易，实际应用时可能还是不知道具体怎么改。

这一章我们提供**6组覆盖最常见任务场景的Before/After实战对照**，从内容分析到代码生成，从翻译到Agent任务，从复杂研究到日常简短任务——每组例子都是完整可直接复制的Prompt，你可以拿来就用，或者根据自己的场景稍作修改。

每个例子我们都会分析：旧写法问题出在哪、新写法好在哪里、核心改写思路是什么。看完这6组例子，你就能掌握新范式Prompt的改写技巧。

> 🟢 **来源标注**：本章示例基于GPT-5.6新写法指南整理，覆盖日常工作最高频的场景。

## 2. 样例1：内容分析类

### ❌ 旧写法

```
请深入、全面、专业地分析以下文章，从多个角度进行解读，给出深刻洞察。一定要认真分析，不要泛泛而谈，要挖掘文章背后的深层含义，给出有价值的观点。
```

### ✅ 新写法

```
请分析以下文章，回答4个问题：

1. 核心论点：作者的主要观点是什么？用1-2句话概括
2. 证据支撑：作者用了哪些论据/数据/案例来支持论点？列出3-5个
3. 论证薄弱环节：哪些地方论证不充分、有逻辑漏洞或证据不足？指出2-3个
4. 反驳切入点：如果要反对作者的观点，可以从哪几个角度入手？给出2个角度

输出要求：
- 严格区分"文章中的事实"和"你的推论"，事实部分引用原文，推论部分标注[推论]
- 每个问题回答简洁，不要展开无关内容
- 总字数控制在800字以内

[文章内容粘贴在这里]
```

### 要点分析

旧写法的问题：
- "深入、全面、专业、深刻洞察"都是无法验证的形容词，模型不知道你具体要什么
- "从多个角度"是哪几个角度？没说，模型只能自己猜
- "不要泛泛而谈"还是主观要求，没有判断标准
- Token花了不少，但有效信息几乎为零

新写法的优势：
- 没有一个形容词，直接列出4个明确的回答要点，模型一眼就知道要做什么
- 每个要点有数量要求（1-2句话、3-5个、2-3个），可验证
- 明确区分事实和推论，这是具体的质量标准，不是空话
- 有字数控制，避免模型无限展开

**核心改写思路**：把"深入分析"这种模糊要求，拆解成几个具体的、可验证的回答要点。

## 3. 样例2：代码生成类

### ❌ 旧写法

```
你是一位资深Python工程师，拥有十年以上开发经验，精通数据处理和pandas库，代码风格优雅，注重性能和可维护性。

请你仔细思考，先设计整体架构，再编写代码，完成后一定要仔细检查一遍，确保没有bug，代码质量要高。

帮我写一个数据清洗函数。
```

### ✅ 新写法

```
请写一个Python数据清洗函数，具体要求：

输入：CSV文件路径
输出：(清洗后的pandas DataFrame, 清洗报告字典)

必须处理：
1. 缺失值：数值列用中位数填充，分类列用众数填充，缺失率>30%的列直接删除
2. 异常值：用IQR方法识别，超过1.5倍IQR的用边界值截断
3. 日期列：自动识别并统一转成YYYY-MM-DD格式
4. 清洗报告：包含删除了哪些列、填充了多少缺失值、处理了多少异常值

约束：
- 只用pandas，不要用其他第三方库
- 缺失率阈值30%做成可配置参数
- 不要修改原CSV文件，只读取，结果在DataFrame里
- 函数要有类型注解和docstring
```

### 要点分析

旧写法的问题：
- "资深Python工程师、十年经验"是没用的角色设定，模型不会因为你给它贴这个标签就写出更好的代码
- "先设计架构、再写代码、再检查"是不必要的流程规定，模型自己知道怎么写代码
- "没有bug、代码质量高"是主观要求，没有具体标准
- 最致命的是：连输入输出是什么、要清洗什么都没说清楚

新写法的优势：
- 直接明确输入输出是什么，这是代码函数最基本的要求
- 4个具体处理要求，每个说清楚怎么处理（缺失值怎么填、异常值用什么方法）
- 3个硬约束（只用pandas、参数可配置、不改原文件），防止模型做出你不想要的选择
- 要求类型注解和docstring，这是可验证的质量标准，不是"代码质量高"这种空话

**核心改写思路**：写代码类Prompt，三个要素缺一不可：明确输入输出、具体功能要求、硬约束（能用什么不能用什么、不能改什么）。

## 4. 样例3：翻译任务

### ❌ 旧写法

```
请将以下英文翻译成中文，注意信达雅，翻译要流畅自然，符合中文表达习惯，不要直译，要专业准确。
```

### ✅ 新写法

```
请将以下英文翻译成中文：

受众：我司产品经理，需要理解海外竞品动态
要求：
1. 所有数据、百分比、专有名词保留英文原文，不用翻译
2. 行业术语第一次出现时附英文原文括号标注，如"客户成功（Customer Success）"
3. 语气：专业但不僵硬，不要用翻译腔
4. 长度控制在原文的1.2倍以内，不要太长

[英文内容粘贴在这里]
```

### 要点分析

旧写法的问题：
- "信达雅"是翻译的最高境界，但也是最模糊的标准——什么叫"雅"？不同人理解完全不同
- "流畅自然、符合中文习惯、不要直译"还是主观感受，没有具体判断标准
- 谁看？用来干嘛？这些影响翻译风格的关键信息完全没有

新写法的优势：
- 先说清楚**受众是谁**（产品经理看竞品动态），这直接决定了翻译风格——不用太文艺，准确专业就行
- 具体要求一条一条列出来：数据保留原文、术语第一次附英文、语气是什么、长度限制
- 没有"信达雅"这种空话，每条要求都是可验证的——你可以检查是不是保留了数据、是不是附了术语、长度是不是超标

**核心改写思路**：翻译任务不要说"信达雅"，要说清楚：给谁看、用来干嘛、哪些东西保留原文、术语怎么处理、语气和长度要求。

## 5. 样例4：研究分析类（完整五段结构模板）

复杂研究任务需要用完整的五段结构（Context/Request/Output Format/Constraints/Checkpoint），这是可直接复用的模板。

### ✅ 新写法（完整可直接用）

```
# Context
我是一家B轮SaaS公司的产品负责人，正在做2027年的产品路线图规划。这份研究报告是给管理层决策用的，他们需要快速判断要不要投入资源做AI Copilot功能。资料截止到2026年Q2，太老的数据不要用。

# Request
请研究SaaS产品中AI Copilot功能的现状和趋势，回答以下4个问题：
1. 当前主流SaaS产品的AI Copilot都在做什么功能？列出TOP5共性功能
2. 用户付费意愿如何？有没有公开的付费转化率或ARPU提升数据？
3. 技术门槛有多高？自研vs调用第三方API的成本对比大概是多少？
4. 我们这个体量的公司（B轮，100人左右）现在切入，机会和风险分别是什么？

完成标准：
- 每个观点要有来源支撑，不要自己编
- 区分客观数据和主观判断
- 涉及调研数据要注明样本量
- 最后给出明确的建议：现在做/观望6个月/不做

# Output Format
- 简体中文Markdown格式
- 结构：先给1页纸结论摘要，再展开详细论证
- 竞品对比和成本对比用Markdown表格
- 总字数控制在2000字以内，不要太长
- 关键数据用加粗标出

# Constraints
- 优先来源：同行评审论文 > 权威咨询公司报告（Gartner/IDC等）> 公司财报 > 行业媒体
- 不要用知乎、微信公众号等未经验证的自媒体帖子作为主要依据
- 如果不同来源数据有冲突，注明各方数据并说明差异
- 不要说"AI Copilot是未来趋势"这种绝对化的空话，要具体
- 不要推荐任何具体的第三方服务商，避免广告嫌疑

# Checkpoint
遇到以下情况立即停下来问我，不要自己猜或继续：
- 不同来源的核心数据（如市场规模）差异超过30%
- 找到的调研数据样本量小于200
- 需要付费才能获取的核心报告
- 发现2026年Q2之后有重大行业变化
```

### 要点分析

为什么复杂研究必须用完整五段结构：
1. **Context**：告诉模型你的身份、报告给谁看、用来做什么决策、资料时间范围——这些信息直接决定了研究的深度、侧重点、数据新鲜度要求。没有Context，模型可能给你一份学术味很重但对商业决策没用的报告。
2. **Request**：不是"帮我研究一下AI Copilot"，而是列出4个具体要回答的问题，每个问题说明要什么信息，还有明确的完成标准——模型知道回答完这4个问题就算做完了，不会无限展开。
3. **Output Format**：给管理层看的报告，结构很重要——先结论再论证，用表格对比，关键数据加粗，控制字数。不说清楚格式，模型可能给你一份不分段的长文。
4. **Constraints**：明确信源优先级，告诉你不要用什么、不要说什么话——避免模型拿自媒体帖子当证据，避免说空话套话。
5. **Checkpoint**：明确什么情况下应该停下来问你——数据冲突、样本太小、要付费、有重大变化——而不是自己随便选一个数据继续做，或者默默忽略冲突。

**核心改写思路**：复杂任务不要嫌麻烦，五段结构每一段都有它的作用——Context给背景防偏差，Request给问题防跑偏，Output给格式防乱套，Constraints给红线防踩坑，Checkpoint给停止条件防瞎猜。

## 6. 样例5：Agent工具调用类（Codex编码完整模板）

Agent任务（比如让Codex CLI帮你写代码、改bug、重构）比研究任务要求更严格——因为Agent会真的动手改你的文件，没有明确的边界和停止条件，可能会搞砸你的项目。这是Codex编码的完整模板。

### ✅ 新写法（完整可直接用）

```
# Context
当前目录是一个Next.js 14 + TypeScript项目，使用App Router，UI组件库用的是shadcn/ui，已经安装了所有依赖。
- 现有组件在src/components/下，已经有Button、Card、Input、Dialog
- API路由在src/app/api/下
- 类型定义在src/types/下
- 目前package.json里已经有zod、react-hook-form，不要加新包
- 代码风格：用TypeScript，有类型注解，函数组件，不要用any

# Request
请帮我实现一个团队邀请功能，具体要求：
1. 邀请表单：输入邮箱+选择角色（管理员/成员/只读），点击发送邀请
2. 邀请列表页：展示已发送的邀请，状态（待接受/已接受/已过期），可以撤销邀请
3. 接受邀请页面：通过链接进来的用户点击接受后加入团队
4. 权限控制：只有管理员能发送和撤销邀请

完成标准：
- 所有代码有完整TypeScript类型，不要用any
- 表单验证用react-hook-form + zod，和项目现有风格一致
- 所有异步操作有loading状态和错误提示
- API失败时重试不超过3次

# Constraints（严格遵守，这是红线）
- ✅ 允许修改：
  - src/app/team/invite/ 下的页面文件
  - src/app/api/team/invitations/ 下的API路由
  - src/components/team/ 下新建组件
  - src/types/team.ts 可以加邀请相关类型
- ❌ 绝对不要修改：
  - 不要修改src/app/layout.tsx
  - 不要修改src/middleware.ts
  - 不要修改现有的prisma schema（schema已经定好了）
  - 不要添加任何新的npm包
  - 不要修改路由配置以外的现有路由
- 所有数据库操作走现有的prisma client实例，不要新建数据库连接
- 密码、token等敏感信息不要硬编码，用环境变量

# Checkpoint
遇到以下任何一种情况，立即停止，告诉我问题是什么，不要自己尝试解决或继续：
1. 需要的UI组件shadcn/ui里没有，需要自己从头写复杂组件
2. 发现需要修改middleware或layout才能正常工作
3. 需要的字段在现有prisma schema里没有
4. 工具调用（如读文件、写文件）连续两次失败
5. 发现项目依赖版本和你预期的不一样，可能导致兼容性问题
6. 需要我提供API密钥、数据库连接串等敏感信息

开始前，先告诉我你打算改哪些文件，确认后再动手。
```

### 要点分析

为什么Agent任务必须有这么详细的Constraints和Checkpoint：
- **文件边界是生命线**：Agent会真的写文件改文件，你必须明确说清楚哪些目录可以改、哪些文件绝对不能碰——否则它可能"好心帮你重构"了整个项目，或者改错了核心配置文件。
- **依赖红线要明确**："不要加新包"这种话一定要说，否则Agent可能自己npm install一堆你不想要的东西。
- **完成标准要可验证**：TypeScript类型、用现有库、有loading状态、重试次数——这些都是可以客观检查的，不是"代码质量高"这种空话。
- **Checkpoint必须具体**：不是"有问题问我"，而是明确列出来什么情况必须停——需要新组件、需要改核心文件、Schema不对、工具失败、版本不对、要敏感信息。Agent遇到这些情况必须停下来，不能自己"发挥"。
- **让它先列计划**：Agent动手改文件之前，先让它说要改哪些文件，你确认后再开始——这是防止大规模返工的最后一道防线。

**核心改写思路**：让Agent写代码，最重要的不是"要做什么"，而是"不能做什么"和"什么时候必须停"。明确的文件边界、依赖红线、停止条件，比详细描述功能还重要。

## 7. 样例6：简短的日常任务

不是所有任务都需要五段结构——日常简短任务，把关键信息说全就行，不用搞复杂结构。

### ❌ 旧写法

```
帮我写一封邮件，要专业一点。
```

### ✅ 新写法

```
帮我写一封商务邮件：
- 收件人：我们的客户X公司的项目负责人李总
- 目的：通知他们项目进度比计划提前3天，下周三可以交付验收
- 语气：专业、积极、不卑不亢
- 需要对方回复：请他们下周三前安排验收人员，提供验收环境
- 顺便约个时间：如果他们方便，下周二下午开个15分钟短会过一下验收清单
- 长度：200-300字，不要太长
```

### 要点分析

旧写法的问题：
- "专业一点"是最模糊的要求——什么叫专业？给老板的专业和给客户的专业不一样，道歉的专业和报喜的专业也不一样
- 收件人是谁？写邮件干嘛？需要对方做什么？这些最关键的信息全没说
- 模型只能猜，写出来的邮件大概率不符合你的场景

新写法的优势：
- 5个关键要素说清楚：谁（收件人）、为什么（目的）、什么语气、要对方做什么（需要回复）、额外动作（约会议）
- 有长度控制
- 不用Context/Request这种结构标签，直接列点就行，因为任务很简单
- 模型拿到这些信息，可以直接写出符合场景的邮件，不需要猜

**核心改写思路**：短任务不需要五段结构，但关键信息不能缺——记住5要素：收件人是谁、目的是什么、语气是啥、需要对方做什么、有什么额外信息或要求。

## 8. 本章小结

### 8.1 6组样例核心模式提炼

| 场景 | 核心写法要点 | 最容易犯的错 |
|------|-------------|-------------|
| **内容分析** | 拆成3-5个具体要回答的问题，区分事实和推论 | 用"深入全面专业"等形容词，不说具体要分析什么 |
| **代码生成** | 明确输入输出、具体功能要求、硬约束（能用什么不能改什么） | 写一大段角色设定，不说清楚输入输出和具体要求 |
| **翻译任务** | 说清受众、保留内容、术语处理方式、语气长度 | 说"信达雅"，不说谁看、用来干嘛、哪些保留原文 |
| **复杂研究** | 用完整五段结构：Context+Request+Output+Constraints+Checkpoint | 只说"帮我研究一下XX"，不给背景、不设边界、不说停止条件 |
| **Agent编码** | 文件边界、依赖红线、停止条件是生命线，让它先列计划再动手 | 只说"帮我做个XX功能"，不说哪些文件不能改、什么情况要停 |
| **日常短任务** | 5要素：收件人、目的、语气、需要对方做什么、额外信息 | 只说"帮我写个XX，专业一点"，关键信息全靠猜 |

### 8.2 通用改写公式（适用于所有场景）

拿到一个旧Prompt，按以下四步改写：

1. **删**：先删掉所有形容词、角色设定、流程规定、重复规则——把水分挤干
2. **补**：补上缺失的关键信息——要什么交付物、什么格式、给谁看、不能做什么
3. **具体**：把模糊要求换成可验证的具体标准——不要"高质量"，要"有类型注解+错误处理+测试"；不要"简洁"，要"300字以内"
4. **加边界**：复杂任务和Agent任务，一定要加Constraints（红线）和Checkpoint（停止条件）

### 8.3 下一步
