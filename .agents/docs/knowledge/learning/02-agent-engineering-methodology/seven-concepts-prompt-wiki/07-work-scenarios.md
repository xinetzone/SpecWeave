# Work场景实战指南

---

## 1. 章节引言

上一章我们学习了Chat场景的技巧——轻量、快速、多轮迭代，适合日常问答和简单任务。但工作中有一类任务，Chat场景就不够用了：写行业研究报告、做数据分析、写产品需求文档、翻译专业技术文档、写商业计划书……

这类任务我们称之为Work场景，特点是：**长文档生成、专业内容生产、有明确格式要求、需要事实准确性**。输出不是几百字的回答，而是几千字甚至上万字的结构化文档；不能靠多轮对话慢慢磨，需要一开始就把要求说清楚，按标准流程产出。

Work场景必须使用**五段式标准结构**，不能像Chat那样随便写。这一章我们详解Work场景的标准结构、模板和实战示例。

> 🟢 **来源标注**：本章内容基于OpenAI官方Prompting指南和GPT-5.6长文档生成实践整理。

---

## 2. Work场景适用范围

Work场景适合需要**结构化产出、专业质量、事实准确**的任务：

| ✅ Work场景典型任务 | ❌ 不适合Work场景的任务 |
|-------------------|-----------------------|
| 行业研究报告、竞品分析 | 日常闲聊、简单问答 |
| 数据分析报告、数据解读 | 快速头脑风暴、创意发散 |
| 产品需求文档（PRD）、技术方案 | 代码片段解释、小bug排查 |
| 专业技术翻译、文档翻译 | 简单邮件、短文案润色 |
| 商业计划书、项目提案 | 学习概念、入门指导 |
| 用户研究报告、会议纪要整理 | 需要实时交互的多轮探索 |

> 🔵 **B级经验**：判断一个任务是否要用Work场景，看三个指标：①输出超过2000字；②有明确的格式/结构要求；③需要事实准确不能瞎编。满足两个以上，就用五段式结构。

---

## 3. Work场景Prompt标准结构（五段式）

Work场景必须使用完整的五段结构，每一段都有明确的作用，缺一不可。

### 3.1 # Context：背景信息

这是最容易被忽略但最重要的一段。告诉模型：
- 你是谁，在什么公司/团队
- 这份文档给谁看（读者是谁）
- 用来做什么决策/解决什么问题
- 时间范围（资料截止到什么时候，太老的不要）
- 已有资料、前提条件、特殊背景

> 🟡 **C级提醒**：没有Context，模型可能给你一份学术味很重但对商业决策没用的报告，或者用了3年前的过时数据。Context决定了内容的侧重点、深度和数据新鲜度。

### 3.2 # Request：核心任务

明确说清楚要做什么：
- 核心任务是什么
- 需要回答哪几个具体问题（列出来，不要泛泛说"研究一下"）
- 质量标准/完成标准（必须可验证）

**错误写法**："帮我研究一下AI Agent市场"——这是让模型自己猜要研究什么。
**正确写法**："请研究AI Agent市场，回答4个问题：1）市场规模和增速；2）TOP5玩家和他们的产品；3）主流商业模式；4）我们进入的机会和风险。完成标准：每个观点要有来源，区分事实和判断，最后给明确建议。"

### 3.3 # Output Format：输出格式

明确要求输出的结构和形式：
- 用什么格式（Markdown/Word/PPT大纲/纯文本）
- 整体结构（先给摘要再展开/按章节/分模块）
- 哪些内容要用表格/列表/加粗
- 长度要求（控制在多少字以内）
- 风格要求（正式/轻松/专业/口语化）

不说清楚格式，模型可能给你一份不分段的长文，或者结构完全不是你想要的。

### 3.4 # Constraints：约束条件

这是质量控制的关键：
- 信源优先级（优先用什么来源，不要用什么来源）
- 禁止内容（不要说什么话、不要推荐什么、不要做什么假设）
- 特殊注意事项（数据冲突怎么处理、哪些内容要标注）
- 硬约束（不能超过多少字、不能加新的依赖等）

### 3.5 # Checkpoint：停止确认点

明确什么情况下模型必须停下来问你，不能自己猜或继续：
- 数据冲突超过一定比例
- 需要付费才能获取的核心资料
- 发现超出你预期的重大情况
- 样本量太小、数据不可靠
- 需要你提供敏感信息或额外资料

> 🟢 **A级原则**：复杂任务不要嫌麻烦，五段结构每一段都有它的作用——Context给背景防偏差，Request给问题防跑偏，Output给格式防乱套，Constraints给红线防踩坑，Checkpoint给停止条件防瞎猜。

## 4. Work场景完整Prompt模板（可复制）

直接复制这个模板，填空就行：

```
# Context
[你的身份/公司/团队背景]
这份文档是给[读者，如：管理层/投资人/客户/技术团队]看的，
用来[做什么决策/解决什么问题]。
资料截止到[时间，如：2026年Q2]，[其他前提/已有资料说明]。

# Request
请[核心任务，如：研究XX市场/分析XX数据/撰写XX文档]，具体完成以下内容：
1. [第一个要回答的问题/要做的模块]
2. [第二个问题/模块]
3. [第三个问题/模块]
4. [第四个问题/模块]

完成标准：
- [标准1，如：每个观点要有来源支撑，不要自己编]
- [标准2，如：区分客观数据和主观判断]
- [标准3，如：涉及调研数据要注明样本量]
- [标准4，如：最后给出明确的结论/建议]

# Output Format
- 简体中文Markdown格式
- 结构：[说明整体结构，如：先给1页纸结论摘要，再展开详细论证]
- [特殊格式要求，如：对比用Markdown表格、关键数据加粗、配图用占位符说明]
- 总字数控制在[XX]字以内
- [风格要求，如：专业严谨，不要空话套话]

# Constraints
- 信源优先级：[如：权威咨询公司报告 > 公司财报 > 行业媒体，不要用自媒体]
- 禁止内容：[如：不要说"XX是未来趋势"这种空话、不要推荐具体服务商、不要做无依据的预测]
- 数据要求：[如：如果不同来源数据有冲突，注明各方数据并说明差异]
- [其他约束]

# Checkpoint
遇到以下情况立即停下来问我，不要自己猜或继续：
1. [如：不同来源核心数据差异超过30%]
2. [如：需要付费才能获取的核心报告]
3. [如：发现[特定重大情况]]
4. [如：需要我提供额外的敏感信息]

现在开始。
```

## 5. 5个Work场景实战示例

下面是5个最高频的Work场景，每个都是可以直接复制使用的完整Prompt。

### 5.1 示例一：行业研究报告

```
# Context
我是一家B轮SaaS公司（100人左右）的产品负责人，正在做2027年产品路线图规划。
这份报告是给公司管理层（CEO/CTO/COO）决策用的，他们需要判断要不要
投入资源做AI Agent产品。资料截止到2026年Q2，2025年之前的数据如果
太老就不用了。我们现有产品是做企业内部知识库的。

# Request
请研究"企业级AI Agent产品"的现状和趋势，回答以下5个问题：
1. 当前市场上主流的企业级AI Agent产品都在做什么场景？列出TOP5场景，每个场景举2个代表产品
2. 客户付费意愿如何？有没有公开的客单价、续费率、ROI数据？
3. 技术实现路径：自研vs基于开源框架vs调用大模型Agent API，成本和门槛对比
4. 我们这个体量的公司现在切入，最大的3个机会和3个风险分别是什么？
5. 如果要做，建议第一个版本做什么场景？为什么？

完成标准：
- 每个观点要有来源支撑，注明是哪家机构/公司的数据
- 区分客观事实和主观判断，判断部分标注[判断]
- 涉及调研数据要注明样本量和时间
- 最后给出明确建议：现在做/观望6个月/不做，理由是什么

# Output Format
- 简体中文Markdown
- 结构：先给500字以内的结论摘要（TL;DR），再按5个问题展开
- 竞品对比、成本对比用Markdown表格
- 关键数据用**加粗**标出
- 总字数控制在3000字以内，不要太长
- 语气：专业、务实，不要用太夸张的词

# Constraints
- 信源优先级：Gartner/IDC/Forrester等咨询公司报告 > 上市公司财报 > 36氪/虎嗅等严肃科技媒体 > 公司官方博客
- 不要用知乎、微信公众号、个人博客作为主要依据
- 如果不同来源数据有冲突，列出各方数据并说明差异原因
- 不要说"AI Agent是下一代互联网"这种绝对化的空话，要具体
- 不要推荐任何具体的第三方服务商或技术供应商，避免广告嫌疑
- 不要做超过3年的预测，太远的预测没有意义

# Checkpoint
遇到以下情况立即停下来问我，不要继续：
1. 不同来源的市场规模数据差异超过40%
2. 找到的付费意愿/ROI相关调研样本量小于100
3. 需要付费才能下载的核心报告（不要自己编数据）
4. 发现2026年Q2之后有重大的行业并购或产品发布
5. 需要我提供我们公司的具体数据（如预算、团队配置）
```

**要点分析**：Context里说清楚"我们是做企业知识库的B轮SaaS"，模型会针对性分析，而不是给你一份通用的AI Agent报告；Request列5个具体问题，每个问题说清楚要什么；Checkpoint明确数据冲突、样本太小、要付费等情况必须停。

### 5.2 示例二：数据分析报告

```
# Context
我是电商公司的运营负责人，手上有过去6个月（2026年1-6月）的用户订单数据。
这份分析是给运营团队看的，用来找到问题、优化下个月的运营策略。
数据我会粘贴在最后，是CSV格式导出的。

# Request
请对这份订单数据做分析，回答以下问题：
1. 整体情况：总销售额、订单量、客单价、复购率这几个核心指标的月度趋势，是增长还是下降？
2. 用户分层：按消费金额把用户分成高/中/低三档，每档用户的占比、贡献销售额占比、复购率分别是多少？
3. 问题诊断：哪个月/哪个用户群的指标下滑最明显？可能的原因是什么（从数据里找线索）？
4.  actionable建议：基于数据，给出3-5条下个月可以直接执行的运营优化建议，每条建议说清楚预期能提升什么指标。

完成标准：
- 所有结论必须有数据支撑，不要说"可能是因为XX"这种没有数据依据的猜测
- 计算过程简单说明一下，我可以验证
- 如果数据有异常值（比如某个订单金额特别大），先指出来，问要不要排除

# Output Format
- 简体中文Markdown
- 结构：核心结论先行 → 详细分析（按上面4个问题） → 建议清单
- 趋势用文字描述+关键数字，不用画图（我自己会做图）
- 用户分层用表格展示
- 建议用编号列表，每条建议说清楚：做什么、针对谁、预期效果
- 总字数控制在2000字以内

# Constraints
- 只基于我给的数据做分析，不要引入外部数据或行业benchmark（除非我问）
- 不要做复杂的统计模型，用描述性统计就行（占比、趋势、对比）
- 如果有数据缺失或字段看不懂，立即问我，不要猜字段意思
- 建议要具体可执行，不要说"提升用户体验"这种空话

# Checkpoint
遇到以下情况立即停下来问我：
1. 发现数据里有明显的异常值（如单条订单超过平均10倍）
2. 有字段你看不懂是什么意思
3. 数据时间不连续（比如缺了某个月）
4. 需要我提供更多维度的数据（如渠道、商品品类）

数据：
[粘贴CSV数据在这里]
```

**要点分析**：数据分析最容易犯的错是模型瞎猜原因、乱建模型——Constraints里明确说"只基于我给的数据、用描述性统计、不要瞎猜原因"；Checkpoint里说遇到异常值、看不懂字段必须问，不能自己猜。

### 5.3 示例三：产品需求文档（PRD）

```
# Context
我是SaaS产品的产品经理，要写一个"团队成员权限管理"功能的PRD，给开发、测试、设计团队看。
我们现有产品已经有基础的团队功能，现在要做细粒度权限控制。技术栈是React+Node.js，
预计2周开发时间，不要搞太复杂。

# Request
请帮我写这个功能的PRD，包含以下部分：
1. 需求背景和目标：为什么要做这个功能，解决什么用户痛点，成功指标是什么
2. 用户故事：3-5个典型用户场景，谁在什么情况下要做什么
3. 功能需求：详细列每个功能点，包括：
   - 角色管理：有哪些默认角色，能不能自定义角色
   - 权限点列表：有哪些权限可以配置（页面访问、操作权限、数据权限）
   - 成员管理：怎么给成员分配角色、怎么修改、怎么移除
4. 非功能需求：性能、安全性、兼容性要求
5. 验收标准：每个功能点怎么算做完了，列可验证的标准
6.  out of scope：明确这次不做什么，防止范围蔓延

完成标准：
- 功能点要具体到开发能看懂、能估时，不要太笼统
- 验收标准必须是可验证的（如"点击保存后2秒内提示成功"，不是"体验要好"）
- 明确说哪些不做，比说做什么还重要

# Output Format
- 简体中文Markdown
- 标准PRD结构，每个部分用二级/三级标题分清楚
- 权限点对比、角色权限矩阵用Markdown表格
- 用户故事用"作为[角色]，我想要[功能]，以便[价值]"的标准格式
- 总字数控制在2500字以内

# Constraints
- 不要搞太复杂的自定义权限，第一版只做预设角色+少量可配置项，2周能做完
- 数据权限先做"只能看自己的/能看全团队的"两档就行，不要做更细的
- 不要涉及SSO、LDAP这些企业级功能，那是下个版本的事
- 所有交互要符合我们现有产品的设计风格，不要提出需要大改交互的方案
- 不要加新的第三方服务依赖

# Checkpoint
遇到以下情况立即停下来问我：
1. 你觉得某个功能需要超过3人天开发，可能影响工期
2. 需要和现有其他模块（如计费、审计日志）做复杂集成
3. 你不确定某个功能要不要做，或者有多个方案选哪个
4. 发现需要修改数据库核心表结构
```

**要点分析**：PRD最重要的是"明确不做什么"——Constraints里反复强调不要搞复杂、第一版先做什么；验收标准要求"可验证"，不能是主观描述；Checkpoint里说可能影响工期、需要大改结构时必须停。

### 5.4 示例四：专业技术翻译

```
# Context
我是前端团队的技术负责人，要把React官方文档的最新Server Components章节翻译成中文，
给团队内部学习用，也可能发在团队技术博客上。读者是有1-3年React经验的前端开发。
原文是React官方文档的英文，我会粘贴在后面。

# Request
请把这份英文技术文档翻译成中文，要求：
1. 准确：技术术语翻译准确，不要错译
2. 通顺：符合中文技术文档的表达习惯，不要翻译腔
3. 统一：术语翻译前后一致，第一次出现时附原文括号标注

# Output Format
- 简体中文Markdown，保留原文的标题层级、代码块、列表格式
- 代码块里的代码不要翻译，保留原样
- 专有名词第一次出现时：中文翻译（英文原文），如"服务器组件（Server Components）"
- 后面再出现可以直接用中文翻译
- 长度和原文差不多，不要扩充也不要大幅删减
- 翻译完之后，附一个"术语对照表"，列出本次翻译中用到的所有术语和对应的翻译

# Constraints
- 以下内容不要翻译，保留英文原样：
  - 代码、API名称、变量名、组件名（如useState、<App />）
  - 已经在中文社区有通用译法但大家更习惯用英文的（如Hook不要翻成"钩子"，就用Hook）
  - 版本号、命令行指令
- 语气：和React官方文档一致，平实、清晰、教程风格，不要太正式也不要太口语
- 遇到你不确定怎么翻的术语，直接保留英文，加个[?]标记，我来定
- 不要添加你自己的解释或点评，忠实原文

# Checkpoint
遇到以下情况立即停下来问我：
1. 有术语你不确定行业通用翻译是什么
2. 原文有明显的错误或前后矛盾
3. 有句子太长或太复杂，你不确定意思

原文：
[粘贴英文原文在这里]
```

**要点分析**：技术翻译不要说"信达雅"，要具体：哪些保留原文、术语怎么标注、语气是什么风格、要术语表；Checkpoint里明确不确定的术语不要瞎翻，标记出来问。

### 5.5 示例五：商业计划书（执行摘要章节）

```
# Context
我是AI代码工具创业公司的创始人，正在写A轮融资的商业计划书，现在需要写"执行摘要"（Executive Summary）这一章。
读者是VC投资人，他们每天看10份BP，这一章必须在1分钟内抓住注意力。
我们的产品是面向中小企业的AI代码审查工具，目前有500家付费客户，月ARR 80万，月增速20%。

# Request
请帮我写商业计划书的执行摘要章节，包含以下内容，按顺序：
1. 一句话介绍公司：我们是谁，做什么，给什么人解决什么问题
2. 问题：现在开发者做代码审查有什么痛点，这个市场有多大
3. 解决方案：我们的产品是什么，怎么解决这个问题，核心差异化是什么
4.  traction：我们现在做到什么程度了（客户数、ARR、增速、典型客户）
5. 为什么是现在：为什么这个问题现在能解决，时机到了
6. 融资需求：我们要融多少钱，用来做什么，预期达到什么目标
7. 团队：核心团队一句话背景（我会填具体名字，你留占位符就行）

完成标准：
- 每一句话都要有信息量，不要空话套话
- 数据要具体，不要"很多""快速增长"这种模糊词
- 看完这一页，投资人能知道我们做什么、做得怎么样、为什么值得投

# Output Format
- 简体中文Markdown
- 结构清晰，每个部分用小标题，短段落，不要大段文字
- 关键数据加粗
- 总长度控制在800字以内！这是执行摘要，不能太长，投资人没耐心
- 语气：自信、务实、不卑不亢，不要用"颠覆""革命"这种夸张的词

# Constraints
- 绝对不要空话：不要说"市场前景广阔""团队经验丰富"这种废话，要说具体的
- 不要贬低竞争对手，只说我们好在哪里，不说别人差
- 数据就用我上面给的，不要自己编额外的数据
- 不要用太技术化的术语，投资人不一定懂技术，要让非技术背景的人也能看懂
- 我提到的具体数据（500家客户、80万ARR、20%增速）必须用上

# Checkpoint
遇到以下情况立即停下来问我：
1. 你觉得需要更多团队背景、市场规模数据才能写好
2. 你有多个写法/角度不确定选哪个
3. 需要我提供竞争对手信息
```

**要点分析**：BP执行摘要最关键的是长度控制（800字以内），以及不能有空话——Constraints里明确禁止"市场前景广阔"这种废话；要求关键数据必须用上；语气要求务实不要夸张。

## 6. 长文档处理技巧

超过3000字的长文档，不要让模型一次写完，用以下方法：

### 6.1 大纲先行，分步生成
1. 第一步：先用上面的五段式Prompt，只让它写**大纲**，不要写全文
2. 第二步：你审一遍大纲，调整结构、增删章节
3. 第三步：确认大纲后，让它一章一章写，每次写1-2章
4. 第四步：最后让它通读全文，统一风格和术语

> 🔵 **B级经验**：一次生成超过5000字，模型很容易"失忆"——前后不一致、重复、遗漏要点。分步生成质量高得多。

### 6.2 事实核验清单
长文档生成后，自己核验这几点：
- [ ] 所有关键数据有没有来源？
- [ ] 有没有前后矛盾的地方？
- [ ] 有没有模型自己编的"看起来很真"的事实？
- [ ] 数字计算对不对？（模型经常算错简单的数）
- [ ] 引用的公司/产品/人名是不是真实存在的？

> 🟡 **C级提醒**：模型会"幻觉"——编造看起来很真实的数据、案例、引用。Work场景的文档，事实部分你一定要自己核验，不要直接用。

## 7. Work场景质量控制：输出后自检清单

拿到输出后，按这个清单快速检查：

| 检查项 | 通过标准 |
|--------|---------|
| **背景匹配** | 内容是不是针对我的背景、读者、用途写的？有没有答非所问？ |
| **完整性** | 我要求回答的所有问题都回答了吗？有没有遗漏？ |
| **结构格式** | 是不是按我要求的结构写的？表格、列表、加粗用对了吗？ |
| **事实准确性** | 关键数据、引用、案例是真实的吗？有没有明显的幻觉？ |
| **约束遵守** | 有没有说我不让说的话？有没有用我不让用的信源？ |
| **长度控制** | 是不是在我要求的字数范围内？有没有太长？ |
| **可操作性** | 建议/结论是不是具体？有没有空话套话？ |

如果有几项不通过，不用重写，直接告诉它哪里要改：
- "第三部分漏了XX问题，补上"
- "这个数据不对，应该是XX，改一下"
- "结论太虚了，给我更具体的3条建议"
- "太长了，压缩到2000字以内，保留核心观点"
