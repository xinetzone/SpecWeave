# 新范式核心规则：做减法而非做加法

---

## 1. 章节引言：新范式的核心原则

理解了GCOB四要素框架之后，你可能已经发现：新范式下写Prompt的核心不是"加什么"，而是"删什么"。

**新范式的核心原则只有一句话：少教模型怎么思考，多告诉它你最终要什么、怎样才算完成、以及哪些事情不能擅自做。**

GPT-5.6时代的强模型就像一个能力极强但不了解你具体情况的资深专家——你不需要手把手教他"第一步做什么、第二步做什么、每一步怎么思考"，他自己知道怎么高效完成工作。你需要做的是：
- 清晰告诉他最终交付物长什么样
- 明确告诉他质量验收标准是什么
- 划定不能碰的红线和边界
- 提供他不知道的上下文信息
- 告诉他什么时候应该停下来问你，而不是自己猜

很多人写Prompt时习惯"做加法"——觉得写得越长越详细、加的修饰词越多、角色设定越丰满、思考流程越复杂，效果就越好。但实际上恰恰相反：**冗余信息会稀释关键信号，增加模型的理解成本，甚至干扰模型选择最优解题路径**。OpenAI内部测试数据显示，删除冗余内容后，任务完成分数提升10-15%，Token消耗减少41-66%。

这一章我们将学习具体应该删什么、留什么，以及不同复杂度任务的Prompt长度指南。

> 🟢 **来源标注**：本章核心原则来自OpenAI官方Prompting指南，辅以GPT-5.6新写法实践总结。

## 2. 应删除的5类冗余内容

写Prompt时，以下5类内容不仅没有帮助，反而有害，应该果断删除。

### 2.1 第一类：无法验证的形容词

#### 说明
"请深入、全面、专业、严谨、细致、高质量地分析"——这类形容词是最常见的冗余。问题在于："深入"到底多深算深？"专业"具体指什么？"高质量"的标准是什么？

#### 为什么有害
模型没有读心术，它只能猜你心里的"专业"是什么意思。不同的人对"专业"的理解完全不同——有人认为专业是用数据说话，有人认为专业是术语堆砌，有人认为专业是结构清晰。你用无法验证的形容词，模型只能按照它的理解来做，结果大概率不符合你的预期。

更糟的是，这类形容词会占据宝贵的Token位置，稀释真正重要的具体要求。

#### Before/After示例

❌ **Before（有冗余形容词）**：
```
请你深入、全面、专业、严谨地分析一下2026年中国新能源汽车市场，给出高质量的洞察和深刻的见解。一定要认真分析，不要泛泛而谈。
```

✅ **After（删除形容词，给出具体标准）**：
```
请分析2026年中国新能源汽车市场，回答以下4个问题：
1. 预计销量和同比增速是多少？驱动力是什么？
2. TOP5厂商市场份额预计是多少？谁在上升谁在下降？
3. 最值得关注的3个技术方向是什么？给出判断依据
4. 有哪些政策风险？影响程度如何评估？

输出要求：
- 每个回答要有数据支撑，标注数据来源年份
- 避免"可能""大概"等模糊表述
- 总字数1500字以内
```

#### 改写要点
- 删除所有主观形容词
- 把形容词转化为可验证的具体要求（"深入"→"回答4个具体问题+数据支撑"）
- 用数量、格式、质量标准代替模糊描述

---

### 2.2 第二类：重复出现的规则

#### 说明
同一条规则写两遍、三遍，甚至更多——比如先写"不要猜测缺失信息"，中间又写"信息不足时请标注"，最后再写一遍"缺数据不要自己编"。

#### 为什么有害
很多人觉得"重要的事情说三遍"，但模型不是人——重复规则并不会让它更重视这条规则，反而会：
1. 浪费Token，增加上下文长度
2. 可能造成规则冲突（如果两次表述略有差异）
3. 让模型困惑："为什么同一件事说两遍？是不是有什么特殊含义？"

#### Before/After示例

❌ **Before（重复规则）**：
```
请整理这份会议记录。

注意：
- 不要猜测任何缺失的信息，信息不足就标出来
- 只使用记录中有的内容，不要自己编造
- 如果有不确定的地方，请标注[待确认]
- 缺失信息不要脑补，一定要明确指出
- 信息不足时不要瞎猜，如实标注
```

✅ **After（规则只说一遍，说清楚）**：
```
请整理这份会议记录。

边界：
- 只用记录中有的内容，缺失或不确定的信息标注[待确认]，不要猜测或编造
```

#### 改写要点
- 同一条规则只说一遍
- 把重复表述合并成一条清晰、准确的规则
- 规则说清楚"要怎么做"和"不要怎么做"即可，不需要反复强调

### 2.3 第三类：不必要的思考流程

#### 说明
"请进行三轮深度反思、模拟五位不同领域专家的观点、先做SWOT分析再做PEST分析、一步一步仔细思考、先理解问题再制定计划然后执行最后检查"——这类人为规定思考流程的内容。

#### 为什么有害
GPT-5.6的强模型已经具备成熟的自主推理能力，它知道根据任务复杂度选择合适的思考路径。你人为规定思考流程会带来三个问题：
1. **限制最优路径**：模型本来有更高效的解题方法，但被你规定的流程限制住了
2. **浪费Token**：模型要花Token解析你规定的流程，而不是直接解决问题
3. **形式主义**：模型可能会机械地按照你说的步骤走一遍，但不是真的在深度思考，只是"表演"思考过程给你看

大部分日常任务根本不需要"三轮反思""五位专家"，简单任务直接回答就好，模型自己知道什么时候需要多想一想。

#### Before/After示例

❌ **Before（规定思考流程）**：
```
你现在是一位战略分析专家，请按照以下步骤帮我分析是否应该进入这个市场：

第一步：仔细阅读我给你的材料，确保完全理解
第二步：进行第一轮反思，列出你初步想到的3个机会和3个风险
第三步：模拟一位乐观的投资人，说出他会怎么看
第四步：模拟一位谨慎的风控专家，说出他会怎么反对
第五步：模拟一位一线销售，说出他的实际顾虑
第六步：进行第二轮反思，整合三方观点
第七步：进行第三轮深度反思，检查是否有遗漏
第八步：给出最终建议

请一步一步来，不要着急，让我们认真思考。
```

✅ **After（不要规定流程，直接说要什么）**：
```
请根据以下材料，分析是否应该进入XX市场，给出：
1. 3个核心机会，每个附判断依据
2. 3个主要风险，每个附发生概率评估（高/中/低）和缓释建议
3. 最终建议：进入/观望/不进入，说明理由

输出要求：
- 乐观视角和风险视角都要覆盖
- 结论明确，不要模棱两可
- 800字以内
```

#### 改写要点
- 除非你真的需要看到思考过程用于教学/审计，否则删除所有"第一步...第二步..."的流程规定
- 不要规定"模拟N位专家""进行N轮反思"——模型需要时自己会做
- 直接告诉它你最终要什么交付物、包含哪些内容

### 2.4 第四类：没有实际作用的专家角色

#### 说明
"你是一位拥有20年经验的资深XX专家、曾在顶级公司工作、获得过XX奖项、擅长XX领域..."——这类长长的角色设定。

#### 为什么有害
这是GPT-3.5时代流传最广的"Prompt技巧"，但在GPT-5.6时代已经基本失效。原因很简单：模型的专业能力来自训练数据，不来自你给它贴的标签。你告诉它"你是20年经验的专家"，它不会突然变得更专业；你不告诉它，它也不会变笨。

角色设定只有在一种情况下有用：**角色会影响判断角度和验收标准**。比如"你是一个挑剔的产品经理，从用户体验角度评审这个设计"——这里的角色不是为了"增加专业性"，而是为了明确评审视角。如果角色不影响输出，那就是纯粹的冗余。

#### Before/After示例

❌ **Before（无用角色设定）**：
```
你是一位拥有20年经验的资深Python后端工程师，毕业于顶尖大学计算机系，曾在Google和Meta工作过，主导过多个千万级用户的系统架构，精通Python、Go、Java等多种编程语言，擅长高并发、分布式系统设计，代码风格简洁优雅，注重可维护性和性能优化，对代码质量有极高的要求，拿过多个编程比赛奖项，发表过技术论文...

请帮我写一个数据清洗函数。
```

✅ **After（删除无用角色，只保留影响判断的视角）**：
```
请写一个Python数据清洗函数。

要求：
- 输入：CSV文件路径
- 输出：清洗后的pandas DataFrame + 清洗报告
- 处理缺失值、异常值、日期格式统一
- 代码要有类型注解和docstring
```

#### 改写要点
- 如果角色不影响输出角度和验收标准，直接删掉
- 如果确实需要特定视角（如"从合规角度审查""站在新用户角度看"），简单说清楚视角即可，不需要编简历
- 不要用角色设定代替具体的质量标准（"资深工程师"≠"有类型注解+错误处理"）

---

### 2.5 第五类：当前任务用不上的工具

#### 说明
一次性给模型列出所有可用工具，不管当前任务用不用得上——比如"你可以用搜索、计算器、文件读取、代码执行、浏览器、数据库查询、API调用等所有工具"。

#### 为什么有害
工具越多，模型的选择空间越大，出错概率也越高：
1. **决策成本增加**：模型需要判断"这个任务要不要用工具？用哪个工具？"，简单任务反而可能想复杂了
2. **上下文占用**：工具描述占据大量Token
3. **误用风险**：不需要工具的任务，模型可能多此一举去调用工具，甚至调用错工具
4. **注意力分散**：太多工具选项会让模型偏离核心任务

#### Before/After示例

❌ **Before（列出所有工具）**：
```
你现在可以使用以下所有工具：
1. web_search：搜索互联网，获取最新信息
2. calculate：计算器，进行数学计算
3. read_file：读取本地文件
4. write_file：写入文件
5. run_code：执行Python代码
6. browse_website：浏览网页
7. query_database：查询数据库
8. call_api：调用外部API
9. send_email：发送邮件
10. create_calendar：创建日历事件

请帮我把这段中文翻译成商务英文。
```

✅ **After（只提需要的工具，或不提）**：
```
请把以下中文翻译成商务英文，面向海外客户，语气专业友好，术语保留英文原文。
```

#### 改写要点
- 简单任务不需要提任何工具，模型需要时会知道
- 只有当任务确实需要特定工具时才明确指出（如"需要最新数据请搜索"）
- 不要一次性罗列所有工具，按需授权即可

## 3. 不同任务复杂度的Prompt长度指南

写多长的Prompt合适？答案是：**看任务复杂度，越简单的任务越短，越复杂的任务越长，但永远只包含必要信息**。

| 任务复杂度 | 典型场景 | Prompt结构 | 参考长度 | 示例 |
|-----------|---------|-----------|---------|------|
| **简单任务** | 翻译、改写、简单问答、格式转换 | Request + Output Format | 1-2句话，30-100 token | "把这段中文改成商务英文，语气专业，长度差不多。" |
| **普通任务** | 写邮件、写文案、简单分析、代码片段 | Context + Request + Output Format | 1段话，100-300 token | 说明背景+说清要什么+说清格式要求 |
| **复杂研究** | 行业研究、竞品分析、深度报告、方案设计 | 完整五段结构（Context/Request/Output Format/Constraints/Checkpoint） | 结构化分段，300-800 token | 给背景+列要回答的问题+说输出格式+说约束+说停止条件 |
| **Agent编码** | 写完整功能、修Bug、重构、多文件操作 | 五段结构 + 工具权限 + 明确文件边界 + Checkpoint | 结构化分段，500-1200 token | 说项目背景+列功能要求+说允许改哪些文件+说不能改什么+说什么时候停下来问 |

### 3.1 简单任务示例

```
把这段中文改成商务英文，面向客户，语气专业友好。
```

### 3.2 普通任务示例

```
背景：我下周要给客户发一封项目进度更新邮件。
请帮我写这封邮件，说明：
- 当前进度：完成了80%，比计划提前2天
- 下周计划：完成剩余20%并进行测试
- 需要客户配合：提供验收环境
语气专业积极，300字以内。
```

### 3.3 复杂研究示例（五段结构）

```
# Context
我是一家B轮SaaS公司的产品经理，正在做2027年产品规划，需要一份AI客服赛道的研究报告给管理层看。资料截止到2026年Q2。

# Request
请研究AI客服赛道，回答以下4个问题：
1. 2026年市场规模和增速是多少？
2. TOP5厂商的核心差异是什么？
3. 客户最在意的3个购买因素是什么？
4. 我们作为新进入者有什么机会？

# Output Format
- 简体中文Markdown
- 先给结论，再给论证
- 竞品对比用表格
- 总字数2000字以内
- 每个数据点标注来源年份

# Constraints
- 优先用同行评审论文和权威机构报告
- 不要用未经验证的自媒体帖子
- 有争议的观点注明双方立场
- 不要说"XX是最佳选择"这种绝对化表述

# Checkpoint
遇到以下情况停下来问我，不要自己猜：
- 不同来源数据冲突超过20%
- 样本量小于100的调研数据
- 需要付费才能获取的核心资料
```

## 4. Steer与Queue机制：对话中的转向与排队

写好第一条Prompt只是开始，实际工作中你几乎总是需要在过程中调整方向、补充信息。Codex CLI等工具提供了两种交互机制：Steer（转向）和Queue（排队），理解它们的区别能让你的协作效率大幅提升。

### 4.1 Steer（转向）：消息加入当前轮，立即改方向

**什么是Steer**：你发的消息会立即打断当前工作，模型会先处理你的新消息，调整方向或补充细节后再继续。在Codex CLI中按Enter发送就是Steer。

**适用场景**：
- 发现模型方向错了，需要立即纠正
- 想到了关键信息，必须马上补充，否则模型继续做下去会白干
- 看到初步结果，需要立即调整要求
- 有紧急的新指令

**Steer示例**：
> "等等，不要用那个API，我们换成本地的mock数据先做。"
> "方向不对，我不是要写报告，我是要做PPT的大纲。"
> "补充一下：目标用户是中小企业，不是大企业。"

### 4.2 Queue（排队）：消息留到下一轮，等当前工作干完再说

**什么是Queue**：你发的消息会排进队列，模型会先把当前正在做的工作做完、交付一个完整的中间结果，然后在下一轮处理你的排队消息。在Codex CLI中按Tab发送就是Queue。

**适用场景**：
- 当前方向是对的，不想打断，让模型先做完
- 想到了后续要调整的点，但现在说会打乱节奏
- 有多个小修改，想等一个完整版本出来后一起说
- 不紧急的补充信息

**Queue示例**：
> "（排队）做完这个之后，顺便帮我加个导出CSV的按钮。"
> "（排队）这个版本没问题，下一轮把颜色换成蓝色系。"
> "（排队）写完后帮我写个简单的README。"

### 4.3 什么时候用Steer，什么时候用Queue？

| 判断标准 | 用Steer（Enter） | 用Queue（Tab） |
|---------|-----------------|---------------|
| 模型是不是在往错的方向走？ | ✅ 是，立即打住 | ❌ 方向是对的 |
| 继续做下去会不会白干/返工？ | ✅ 会，赶紧停 | ❌ 不会，先做完再说 |
| 你的消息是不是紧急？ | ✅ 紧急 | ❌ 不紧急，可以等 |
| 是不是想先看一个完整结果再改？ | ❌ 不是 | ✅ 是，先看完整版 |

💡 **实践建议**：大部分时候用Queue更好——让模型把一个完整的东西做出来，你再基于完整结果提修改意见，比中途反复打断效率高得多。只有当方向明显错了、继续做下去会造成大量返工时，才用Steer打断。

## 5. 写Prompt后检查5个问题

写完Prompt、点击发送之前，快速问自己这5个问题。如果有任何一个答案是"否"，那就回去修改，改完再发。

### 问题1：模型知道我要什么吗？
- 我有没有清晰描述最终交付物？
- 有没有模糊的形容词需要换成具体标准？
- 一个完全不了解背景的人看到这个Prompt，能准确知道要做什么吗？

❌ 反例："帮我分析一下这个市场。"
✅ 正例："帮我分析这个市场，回答4个问题：市场规模、竞争格局、技术趋势、政策风险。"

### 问题2：模型知道怎样才算完成吗？
- 我有没有说清楚输出格式、长度、受众？
- 我有没有列出来交付物必须包含的内容点？
- 拿到输出后，我能明确判断"合格/不合格"吗？还是只能凭感觉？

❌ 反例："给我一个方案。"
✅ 正例："给我一个3页PPT的方案大纲，包含背景、问题、方案、预算、时间表5个部分。"

### 问题3：模型知道哪些信息不能猜吗？
- 我有没有提供足够的上下文？
- 我有没有说清楚"信息缺失时怎么办"？是停下来问，还是标注[待确认]？
- 有没有什么信息是我知道但模型不知道、必须告诉它的？

❌ 反例："分析一下我们的产品。"（模型不知道你们是什么产品）
✅ 正例："分析一下我们的产品（产品介绍见附件），只用附件里的信息，缺失的标注[待确认]。"

### 问题4：模型知道哪些操作不能越界吗？
- 我有没有划定不能碰的红线？
- 对于Agent任务，我有没有说清楚哪些文件不能改、哪些操作不能做？
- 我有没有说清楚"不要做什么"，而不只是"要做什么"？

❌ 反例：（什么边界都没说，模型擅自改了路由配置）
✅ 正例："只允许修改src/components/Login.tsx，不要改路由、不要加新包、不要改Schema。"

### 问题5：模型知道什么时候应该停下来吗？
- 我有没有说清楚遇到什么情况应该先问我，而不是自己继续？
- 我有没有定义"完成"的终点？
- 模型会不会无限展开、做很多我没要的东西？

❌ 反例："帮我调研一下这个方向。"（模型可能写了50页报告还在继续）
✅ 正例："调研这个方向，回答3个问题，2000字以内。如果需要付费资料就停下来问我。"

## 6. 本章小结

### 6.1 核心要点回顾

1. **核心原则：做减法而非做加法**：少教模型怎么思考，多告诉它要什么、怎么验收、什么不能做
2. **果断删除5类冗余**：无法验证的形容词、重复的规则、不必要的思考流程、没用的专家角色、用不上的工具
3. **按任务复杂度定长度**：简单任务1-2句话，复杂研究用五段结构，Agent编码加边界和Checkpoint
4. **善用Steer和Queue**：方向错了用Steer立即打断，方向对了用Queue等完整结果
5. **发送前过五问检查**：知道要什么吗？知道怎么算完成吗？知道不能猜吗？知道不能越界吗？知道什么时候停吗？
