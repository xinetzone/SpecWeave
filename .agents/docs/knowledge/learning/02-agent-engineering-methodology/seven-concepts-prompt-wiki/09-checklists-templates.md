# 检查清单与可复用模板库

---

## 1. 章节引言

前八章我们系统学习了七概念方法论、GCOB框架、三场景（Chat/Work/Codex）的写法原则。理论已经很充分了，但实际写Prompt的时候，你可能还是会忘记某条规则，或者每次都要从零开始想结构。

本章提供**可直接复制使用**的检查清单和Prompt模板——每次写Prompt前花30秒过一遍清单，能避免80%的常见问题；需要写Prompt时直接复制对应模板填空，省去每次构思结构的时间。

这些清单和模板都是从前面章节的原则提炼出来的，经过大量实践验证，你可以打印出来贴在显示器旁边，写Prompt的时候对照检查。

## 2. 📋 Prompt写完必查五问（扩展版）

这是第04章"五问检查"的详细扩展版，每个问题都有具体的检查方法和常见症状，写完Prompt后逐条对照：

### Q1：模型知道我要什么吗？

**验收标准**：任务目标是否具体可衡量？

✅ **检查方法**：
- 把你的Prompt给一个完全不了解背景的同事看，他能准确说出来要做什么吗？
- 有没有具体的交付物描述（而不是形容词）？
- 要回答/完成的内容点是不是列成了清晰的清单？

❌ **常见症状**：
- 用了"深入分析""全面了解""专业评估"等形容词，但没说具体分析什么
- 只说"帮我优化一下"，没说优化什么、优化成什么样
- 交付物模糊，拿到结果后你也说不清"合格/不合格"

❌ **反例**："帮我深入分析一下我们的产品，给出专业建议。"
✅ **正例**："帮我分析我们的产品，回答3个问题：核心用户是谁？最大的3个痛点是什么？和竞品相比有什么优势？"

---

### Q2：模型知道怎样才算完成吗？

**验收标准**：完成标准是否可验证？

✅ **检查方法**：
- 输出格式有没有明确说（Markdown？PPT大纲？代码？表格？）
- 长度/字数有没有限制？
- 受众是谁（老板？客户？开发？新用户？）
- 拿到输出后，你能不能客观判断"完成了/没完成"，而不是凭感觉？

❌ **常见症状**：
- "结构化输出"但没说什么结构
- "不限字数"导致输出冗长无比
- 没有说给谁看，模型不知道该用什么深度和术语
- 交付完你发现少了关键部分，但Prompt里没说必须有

❌ **反例**："给我一个方案。"
✅ **正例**："给我一个3页PPT的方案大纲，包含背景、问题、方案、预算、时间表5个部分，面向非技术背景的管理层，总字数800字以内。"

---

### Q3：模型知道哪些信息不能猜吗？

**验收标准**：缺失信息是否要求询问/标注？

✅ **检查方法**：
- 模型不知道的关键上下文你有没有提供？
- 有没有说清楚"信息不足时怎么办"？是停下来问你，还是标注[待确认]？
- 有没有什么信息是你知道但模型不可能知道的（比如你们公司的内部数据、项目背景）？

❌ **常见症状**：
- "分析一下我们的系统"——模型根本不知道你们是什么系统
- 没说信息缺失怎么办，模型自己脑补了一堆错误信息
- 关键背景没给，模型只能按通用情况回答，结果不符合你的实际

❌ **反例**："分析一下我们的用户留存问题，给出解决方案。"
✅ **正例**："分析我们的用户留存问题（数据见附件CSV），只用附件里的数据，缺失或不确定的信息标注[待确认]，不要自己编造数据。如果需要更多信息就停下来问我。"

---

### Q4：模型知道哪些操作不能越界吗？

**验收标准**：边界约束是否明确？

✅ **检查方法**：
- 有没有划定明确的红线——什么事情绝对不能做？
- 对于Agent/Codex任务，有没有说清楚哪些文件/目录不能改？
- 有没有说清楚"不要做什么"，而不只是"要做什么"？
- 默认权限是不是最小的？

❌ **常见症状**：
- 什么边界都没说，Codex擅自改了路由配置或package.json
- 没说能不能装新包，模型遇到问题就自己npm install了
- 只说"要做XX"，没说"不要做YY"，模型自由发挥过了头

❌ **反例**：（什么约束都没说，只说"帮我加个登录功能"）
✅ **正例**："帮我加登录功能，只允许修改src/pages/Login.tsx和src/api/auth.ts，不要改路由、不要加新包、不要修改现有Schema。"

---

### Q5：模型知道什么时候应该停下来吗？

**验收标准**：Checkpoint停止条件是否设置？

✅ **检查方法**：
- 有没有说清楚遇到什么情况应该先问你，而不是自己继续？
- 有没有定义"完成"的明确终点，防止模型无限展开？
- 对于复杂任务，有没有设置中间检查点？
- 出错/失败时的重试次数有没有限制？

❌ **常见症状**：
- "帮我调研一下这个方向"——模型写了50页报告还在继续
- 工具调用失败了，模型一直在重试，死循环
- 遇到不确定的地方，模型自己选了一个方案继续做，结果方向错了
- 做了很多你没要求的额外工作，浪费Token和时间

❌ **反例**："帮我调研一下AI Agent赛道。"
✅ **正例**："调研AI Agent赛道，回答4个问题，2000字以内。如果需要付费资料、数据冲突超过20%、或者有多个方案不确定选哪个，就停下来问我，不要自己猜。"

## 3. 📋 Chat场景快速检查清单（5项）

Chat场景（简单问答、头脑风暴、翻译、概念解释）快速过这5项：

| # | 检查项 | ✅ 通过标准 |
|---|--------|------------|
| 1 | 有没有给出必要的上下文？ | 模型需要知道的背景信息一句话说清了 |
| 2 | 问题是不是具体？ | 不是"聊聊XX"，而是具体问什么/要什么 |
| 3 | 有没有说想要的回答形式？ | 要列表？要对比？要举例？要多少字？ |
| 4 | 有没有说受众/深度？ | 是给小白解释还是给专家讨论？ |
| 5 | 简单任务有没有过度设计？ | 1-2句话能说清的，不要写五段式结构 |

💡 Chat场景原则：越简单越好，不要过度工程化。

## 4. 📋 Work场景快速检查清单（8项）

Work场景（写报告、文档、PRD、专业内容）过这8项，特别注意事实核验：

| # | 检查项 | ✅ 通过标准 |
|---|--------|------------|
| 1 | Context（背景）是否清晰？ | 你是谁、给谁看、为什么要做这个、资料截止时间 |
| 2 | Request（需求）是否具体？ | 要回答/完成的内容点列成了清单 |
| 3 | Output Format（输出格式）是否明确？ | 格式、结构、长度、是否用表格/图表 |
| 4 | Constraints（约束）是否清楚？ | 什么能做什么不能做、什么信息能用什么不能用 |
| 5 | ⚠️ 事实核验要求有没有说？ | 数据要不要标注来源？要不要区分事实和观点？信息不足怎么办？ |
| 6 | Checkpoint（停止条件）有没有设？ | 什么情况停下来问，不要自己猜 |
| 7 | 有没有删掉冗余形容词？ | 没有"深入、全面、专业、高质量"等无法验证的词 |
| 8 | 有没有教模型思考？ | 没有"第一步...第二步..."、"模拟N位专家"等流程规定 |

> 🔵 **B级经验**：Work场景最容易犯的错是"事实核验不说清楚"——模型自己编数据、用过时信息、把猜测当结论。一定要明确数据来源要求。

## 5. 🔴 Codex场景安全检查清单（10项）

Codex/Agent写代码场景，这10项**必须全部检查**，标⚠️的是强制项，缺一个都可能导致事故：

| # | 检查项 | 强制 | ✅ 通过标准 |
|---|--------|------|------------|
| 1 | ⚠️ 允许修改的文件/目录白名单 | ⚠️ | 明确列出了哪些文件/目录可以改，用白名单思维 |
| 2 | ⚠️ 禁止的操作有没有明确列？ | ⚠️ | 删文件、改路由、装包、改配置、改Schema等红线说清楚了 |
| 3 | ⚠️ Checkpoint停止条件是否设置？ | ⚠️ | 什么情况必须停下来问，不能自己继续 |
| 4 | 技术栈/依赖是否说明？ | | 用什么框架、UI库、状态管理、ORM，现有代码模式 |
| 5 | 有没有要求先列计划再动手？ | | 最后说了"先列文件清单和修改计划，我确认后再开始" |
| 6 | 完成标准是否可验证？ | | 不是"代码质量高"，而是有TypeScript类型、有错误处理等客观标准 |
| 7 | 工具权限是否明确？ | | 什么工具能用、什么不能用（特别是危险命令） |
| 8 | 有没有强调"最小修改"？ | | 修Bug就只修Bug，不要顺手重构 |
| 9 | 开始前你git commit了吗？ | | ⚠️ 重要提醒：开始前先commit一次，出问题能回滚 |
| 10 | 有没有让它先读现有代码？ | | 新功能先看现有代码怎么写的，保持风格一致，不要自己发明模式 |

> 🟢 **A级原则**：Codex场景优先级：不破坏 > 可运行 > 代码质量。这10项检查花1分钟，能帮你节省几小时的回滚时间。

## 6. 📦 通用Prompt模板库（可直接复制填空）

以下所有模板都用```markdown代码块包裹，直接复制，把【待填写】部分换成你的内容即可。

---

### 模板1：Chat快速问答模板

```markdown
【待填写：一句话背景，可选】
请帮我【待填写：具体要做什么，问什么问题】。

要求：
- 【待填写：回答形式，如：分3点说明、举2个例子、用表格对比】
- 【待填写：受众/深度，如：给我这个小白解释、面向技术人员、简洁一点100字以内】
- 如果有信息不确定，直接说不知道，不要编。
```

**适用场景**：简单问答、概念解释、快速翻译、头脑风暴、小改写。

---

### 模板2：Work五段式完整模板

```markdown
# Context
我是【待填写：你的角色】，正在做【待填写：什么事】，需要一份【待填写：交付物类型】给【待填写：受众】看。资料截止到【待填写：时间】。
【待填写：其他必要背景，可选】

# Request
请【待填写：具体任务】，具体需要包含：
1. 【待填写：内容点1】
2. 【待填写：内容点2】
3. 【待填写：内容点3】
4. 【待填写：内容点4，按需增减】

# Output Format
- 【待填写：格式，如：简体中文Markdown】
- 【待填写：结构要求，如：先给结论再给论证、对比用表格】
- 【待填写：长度要求，如：总字数2000字以内】
- 【待填写：其他格式要求】

# Constraints
- 【待填写：事实核验要求，如：数据标注来源年份、优先用权威报告】
- 【待填写：内容约束，如：不要用未经验证的自媒体内容、不要绝对化表述】
- 【待填写：其他约束】

# Checkpoint
遇到以下情况停下来问我，不要自己猜：
- 【待填写：停止条件1，如：不同来源数据冲突超过20%】
- 【待填写：停止条件2，如：需要付费才能获取的资料】
- 【待填写：停止条件3，按需增减】
```

**适用场景**：行业研究、报告写作、PRD、竞品分析、专业文档、长文翻译。

### 模板3：Codex安全编码模板

```markdown
# Context
当前目录是一个【待填写：技术栈，如：Next.js 14 + TypeScript + App Router】项目。
- UI组件库：【待填写：如：shadcn/ui，已有Button/Card/Input】
- 状态管理：【待填写：如：Zustand，没有就写"无"】
- 表单：【待填写：如：react-hook-form + zod】
- API路由：【待填写：如：src/app/api/】
- 组件目录：【待填写：如：src/components/】
- 类型定义：【待填写：如：src/types/】
- ORM/数据库：【待填写：如：Prisma，schema在prisma/schema.prisma】
- 代码风格：【待填写：如：TypeScript strict模式，不要用any，函数组件】

# Request
请帮我实现【待填写：功能名称】，具体要求：
1. 【待填写：功能点1】
2. 【待填写：功能点2】
3. 【待填写：功能点3】

完成标准（必须全部满足）：
- [ ] 【待填写：标准1，如：所有代码有完整TypeScript类型，不要用any】
- [ ] 【待填写：标准2，如：表单验证用react-hook-form+zod，和现有风格一致】
- [ ] 【待填写：标准3，如：所有异步操作有loading状态和错误提示】
- [ ] 【待填写：标准4，按需增减】

# Constraints（红线，严格遵守）
✅ 允许修改/新建的文件/目录（只允许动这些）：
- 【待填写：如：src/app/xxx/ 下的页面文件，可以新建】
- 【待填写：如：src/app/api/xxx/ 下的API路由，可以新建】
- 【待填写：按需增减】

❌ 绝对不要修改以下内容，碰都不要碰：
- 不要修改【待填写：核心文件1，如：src/app/layout.tsx】
- 不要修改【待填写：核心文件2，如：prisma/schema.prisma】
- 不要修改package.json，不要添加任何新的npm包
- 不要修改现有的其他页面和路由
- 【待填写：其他禁止项】

其他规则：
- 【待填写：其他规则，如：所有数据库操作走现有的prisma client】
- 不要写console.log，debug代码用完删掉
- 先看现有代码怎么写的，保持风格一致，不要自己发明新模式

# Checkpoint
遇到以下任何一种情况，立即停止，告诉我问题是什么，等我回复，不要自己尝试解决：
1. 需要修改/新建的文件不在上面的白名单里
2. 需要安装任何新的npm包
3. 需要修改schema或数据库结构
4. 需要修改layout、middleware、配置文件等核心文件
5. 工具调用（读文件、写文件、执行命令）连续两次失败
6. 发现现有代码结构/依赖版本和预期不一样，有兼容问题
7. 有多个实现方案不确定选哪个
8. 你觉得这个需求需要大改架构

# Tools
允许使用的工具：
- ✅ 读文件：可以读项目里的任意文件
- ✅ 写文件：仅限上面白名单里的目录和文件
- ✅ 执行命令：可以运行【待填写：如：npm run dev、npm run test、npm run typecheck】
- ❌ 禁止执行：rm、del、sudo、curl | bash、npm install -g、git push等危险命令
- ❌ 禁止删除任何不是你这次新建的文件
- ❌ 禁止安装新包

⚠️ 重要：开始动手改文件之前，先做这件事：
1. 先读相关的现有文件，理解现有代码模式
2. 然后列出你打算修改/新建的所有文件清单，以及每个文件大概要做什么
3. 等我确认说"可以开始"之后，你再动手写代码

现在开始第一步，先列文件清单给我确认。
```

**适用场景**：添加新功能、写新页面/组件、修Bug、重构、加API接口。

### 模板4：代码Review模板

```markdown
# Context
这是一个【待填写：技术栈】项目，我需要你帮我Review以下代码。
代码文件：【待填写：文件路径或粘贴代码】

# Request
请帮我Review这段代码，从以下几个角度：
1. 有没有Bug/逻辑错误？
2. 有没有安全问题（SQL注入、XSS、敏感信息泄露等）？
3. 性能有没有问题？（N+1查询、不必要的循环等）
4. 可维护性怎么样？（命名、结构、重复代码、注释）
5. 是否符合最佳实践？
6. 有没有边界情况没处理？

# Output Format
- 按严重程度分类：🔴 严重问题（必须修）、🟡 建议改进、🟢 做得好的地方
- 每个问题说明：位置、问题是什么、为什么是问题、怎么改
- 如果代码没问题，直接说"没发现明显问题"
- 不要为了找问题而找问题，真正有问题才说

# Constraints
- 只Review我给的这段代码，不要扯太远
- 批评要具体，不要说"代码质量不高"这种笼统的话
- 给出的修改建议要可操作，不要说"优化一下"
```

**适用场景**：代码审查、PR Review、找Bug、代码质量评估。

### 模板5：文档写作模板

```markdown
# Context
我需要写一份【待填写：文档类型，如：README、API文档、使用指南、贡献指南】。
项目/产品是：【待填写：简单介绍】
目标受众是：【待填写：如：第一次用的新用户、项目贡献者、第三方开发者】

# Request
请帮我写这份文档，需要包含以下部分：
1. 【待填写：部分1，如：简介/项目介绍】
2. 【待填写：部分2，如：快速开始/安装】
3. 【待填写：部分3，如：使用示例】
4. 【待填写：部分4，如：API参考】
5. 【待填写：按需增减】

# Output Format
- Markdown格式
- 语言简洁明了，面向目标受众的技术水平
- 代码示例要完整可运行
- 适当用标题、列表、代码块让文档易读

# Constraints
- 不要写我没提到的功能/特性
- 如果需要参考代码/现有文档，我会提供，不要自己编
- 保持【待填写：风格要求，如：简洁实用、不要营销话术】
```

**适用场景**：写README、API文档、用户指南、开发文档。

# Request
请翻译以下内容：
---
【待填写：粘贴要翻译的内容】
---

要求：
- 【待填写：语气要求，如：专业正式、友好自然、准确严谨】
- 术语处理：【待填写：如：专业术语保留英文、第一次出现加括号注释、按照给定术语表（见附件）】
- 不要逐字直译，要符合目标语言的表达习惯

## 7. 七概念自检表：R-I-E-C-A-F-V对应检查项

写完Prompt后，也可以对照七概念方法论的七个维度快速自检：

| 七概念 | 对应检查项 | 自检问题 |
|--------|-----------|---------|
| **R - Role（角色）** | 视角是否清晰 | 有没有明确需要的判断视角？如果角色不影响输出，是不是删掉了没用的"资深专家"设定？ |
| **I - Intent（意图）** | 目标是否明确 | 最终要什么是不是说清楚了？有没有模糊的形容词？ |
| **E - Execution（执行）** | 约束是否清楚 | 什么能做什么不能做？边界有没有划清？（注意：不要规定执行步骤，让模型自己找路径） |
| **C - Completion（完成）** | 验收标准是否可验证 | 怎么算完成说清楚了吗？输出格式、长度、质量标准明确吗？ |
| **A - Audit（核验）** | 事实核验是否要求 | 数据要不要来源？信息不足怎么办？有没有要求区分事实和观点？ |
| **F - Feedback（反馈）** | Checkpoint是否设置 | 什么时候停下来问？出错怎么办？有没有设置反馈节点？ |
| **V - Value（价值）** | 是否在解决正确的问题 | 这个Prompt真的能帮你拿到想要的结果吗？有没有遗漏关键信息？ |

> 🔵 **B级经验**：七概念自检是"元检查"——五问清单查的是"有没有写对"，七概念自检查的是"有没有漏维度"。两者结合使用效果最好。
