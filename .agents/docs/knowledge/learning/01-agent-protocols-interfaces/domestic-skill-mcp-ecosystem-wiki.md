## 一、引言：Agent 化浪潮来袭

2025 年以来，国内 AI Agent 生态进入加速构建期。从支付到餐饮，从出行到办公协作，越来越多的品牌开始把自己的服务能力封装成 Skill、MCP、CLI 三种形态，让 AI Agent 可以直接调用。

本文基于**卡兹克**与**可达**合著的微信公众号文章《国内 Skill/MCP 生态盘点》系统整理，覆盖 **16 个品牌**的 Agent 化产品形态、核心能力、使用体验和生态趋势，力求为开发者、产品经理、研究者提供一份完整的国内 Skill/MCP 生态地图。

正如原文所述：

> "对于大家来说，其实都是把网址扔过去然后说给我安装其实就行了。"

这份文档的定位是：**系统整理国内 Skill/MCP 生态现状**，帮助读者快速建立全局认知，并根据自己的需求选择合适的接入方式与品牌方案。

## 二、概念入门：Skill / MCP / CLI 三种集成方式

在进入品牌盘点之前，必须先把三个核心概念区分清楚：**Skill**、**MCP**、**CLI**。这三者是当前 Agent 调用外部能力的主流集成方式，但面向人群、技术门槛、能力范围都有明显差异。

### 2.1 Skill：面向 Agent 使用者的能力封装

**Skill** 是面向 Agent 使用者的能力封装，类似"插件"的概念。用户不需要写代码，只需要一句话或者一个网址即可让 Agent 安装使用。它是三种方式中门槛最低的，主要面向普通用户和产品经理。

### 2.2 MCP：面向开发者的标准化上下文协议

**MCP（Model Context Protocol）** 是面向开发者的标准化上下文协议。能力更强，但需要技术接入，需要开发者按协议规范完成服务端实现。MCP 是 Anthropic 在 2024 年提出的开放协议，目的是让 LLM 与外部数据源、工具服务以统一方式交互。

### 2.3 CLI：面向开发者的命令行工具

**CLI** 是命令行工具，面向开发者。它可以被 Claude Code、Cursor 等编程 Agent 调用，让 Agent 在开发过程中直接执行外部服务的命令，例如查询数据、触发操作。CLI 的优势在于与开发工作流天然契合，缺点是普通用户难以使用。

### 2.4 三者对比一览

| 对比维度 | Skill | MCP | CLI |
|---------|-------|-----|-----|
| **面向人群** | Agent 使用者（普通用户/产品经理） | 开发者 | 开发者 |
| **技术门槛** | 低（一句话即可安装） | 中（需按协议开发接入） | 中（需命令行操作） |
| **能力范围** | 受限封装好的能力，开箱即用 | 强，可暴露任意自定义能力 | 强，可执行任意命令 |
| **典型场景** | 让 ChatGPT/扣子/Claude 直接点咖啡、叫车 | 让自研 Agent 接入支付、地图服务 | 让 Claude Code 在编码时调用飞书 API |

原文用一句通俗的话总结了三者的使用方式：

> "对于大家来说，其实都是把网址扔过去然后说给我安装其实就行了。"

也就是说，从用户体验角度，三种方式的安装入口都是"丢链接给 Agent"，区别在于背后的实现机制与适用人群。

## 三、餐饮行业盘点

餐饮是 Agent 化落地最早的行业之一，因为它具备**高频、低决策成本、链路短**的特点，非常适合用 Skill/MCP 一句话完成点单。

### 3.1 瑞幸咖啡 Skill

| 项目 | 信息 |
|------|------|
| **品牌** | 瑞幸咖啡 |
| **集成形态** | Skill + MCP + CLI（三种全支持） |
| **官方平台** | [open.lkcoffee.com](https://open.lkcoffee.com) |
| **核心能力** | 点咖啡、推荐饮品、定位门店、扫码支付 |

**使用体验**：

瑞幸咖啡的 Skill 是国内餐饮行业 Agent 化的标杆案例，支持 Skill、MCP、CLI 三种接入方式，覆盖开发者与普通用户两类人群。

它的核心能力包括：点咖啡、推荐饮品、定位门店、扫码支付。在实际使用中，Skill 能根据用户的健康需求做个性化推荐。例如当用户表达"心脏有点不舒服不喝咖啡因"时，Skill 会推荐无咖啡因的饮品选项。

**杯型选择特点**：

瑞幸的杯型设计很有意思，提供 **大杯、特大杯、超大杯** 三档，唯独没有"中杯"。这与瑞幸线下门店的杯型策略一致，避免了星巴克"中杯/大杯/超大杯"长期存在的用户认知争议。

**局限**：

目前瑞幸 Skill **只支持到店自取，不支持外卖配送**。这意味着 Agent 化只覆盖了下单环节，配送环节仍需用户自己到店取餐。

**支付方式**：

采用**扫码支付**，不打通"最后一公里"。下单完成后 Agent 会生成支付二维码，用户需要自己扫码完成付款，而不是 Agent 直接调用支付接口完成扣款。

### 3.2 麦当劳 MCP

| 项目 | 信息 |
|------|------|
| **品牌** | 麦当劳 |
| **集成形态** | MCP |
| **官方平台** | [https://open.mcd.cn/mcp](https://open.mcd.cn/mcp) |
| **核心能力** | 查活动日历、领券 |

麦当劳选择了 MCP 这一种形态接入，核心能力集中在**查活动日历**和**领券**两件事上。相比瑞幸覆盖点单全链路，麦当劳的 Agent 化能力相对克制，更偏向"营销前置"——让用户在 Agent 中先看到活动、领到优惠券，再回到 App 完成下单。

**支付方式**：最后一步需跳 App 完成支付。

## 四、出行行业盘点

出行行业是 Agent 化覆盖最密集的赛道之一，覆盖了**机票预订、酒店、打车、地图服务**等子领域。

### 4.1 飞猪 Skill

| 项目 | 信息 |
|------|------|
| **品牌** | 飞猪 |
| **集成形态** | Skill（底层接自己的 MCP 服务） |
| **官方平台** | [https://flyai.open.fliggy.com/](https://flyai.open.fliggy.com/) |
| **核心能力** | 机票、酒店、门票、用车咨询规划预订 |

**接入特点**：

飞猪的 Skill 底层是接自己的 MCP 服务，**无需 API Key 即可试用**，这对普通用户非常友好。但要使用完整能力，仍需要申请 API Key。

**使用体验**：

实测中，用 Skill 规划"杭州周末游"，它能列出完整的航班表，包含**航司、起飞时间、飞行时长、价格、首选/备选/性价比**等多维度信息，并配套推荐酒店。

**局限**：

体验版数据有缺失，完整能力需申请 API Key。这意味着普通用户可以"先体验再付费"，但生产环境部署仍需要正式接入。

### 4.2 滴滴 Skill

| 项目 | 信息 |
|------|------|
| **品牌** | 滴滴 |
| **集成形态** | Skill + MCP |
| **官方平台** | [https://mcp.didichuxing.com/](https://mcp.didichuxing.com/) |
| **核心能力** | 实时叫车、预约出行、订单查询、查看司机位置 |

**上线节奏**：

- **2024 年 9 月**：上线 MCP
- **2025 年 4 月**：上线 Skill

**特色设计**：

滴滴 Skill 的特色设计在于**能直接提醒司机状态**，并且可以组合 hook 触发飞书电话。这意味着当司机到达指定地点时，Agent 可以主动通过飞书电话通知用户，避免用户错过接驾。

这种"出行 Agent + 通讯 hook"的组合是出行场景中比较创新的设计，把打车从"用户主动查看"变成了"Agent 主动通知"。

### 4.3 高德地图 Skill

| 项目 | 信息 |
|------|------|
| **品牌** | 高德地图 |
| **集成形态** | Skill + MCP |
| **官方平台** | [https://lbs.amap.com/](https://lbs.amap.com/) |
| **核心能力** | 位置服务、地图开发、Android Agent、iOS Agent、RTOS 地图 |

**上线节奏**：

- **2024 年 7 月**：推出 MCP
- **2025 年 4 月**：上线 Skill 市场

高德的 Agent 化覆盖非常广泛，除了常规的位置服务和地图开发能力，还延伸到了 **Android Agent、iOS Agent、RTOS 地图**，意味着车载系统、移动端、IoT 设备都可以接入。

**使用体验**：

实测搜索"杭州余杭区酒店"，Skill 会生成 **5 公里范围搜索链接**，并按照**名称、评分、地址、图片**四个维度列出结果。这种结构化输出非常适合 Agent 后续做决策。

### 4.4 腾讯地图 Skill

| 项目 | 信息 |
|------|------|
| **品牌** | 腾讯地图 |
| **集成形态** | Skill + MCP |
| **官方平台** | [https://lbs.qq.com](https://lbs.qq.com) |
| **核心能力** | 搜索、规划、天气查询、模型展示 |

**差异点**：

腾讯地图相比高德，多了一个**前端地图开发 Skill**，支持 **3D 地图、Three.js 集成、GLTF 模型**渲染。这意味着前端开发者可以直接通过 Skill 在自己的网页中嵌入 3D 地图，而不用自己封装 Three.js 地图组件。

| 对比维度 | 高德地图 | 腾讯地图 |
|---------|---------|---------|
| Skill + MCP | ✅ | ✅ |
| 位置服务 | ✅ | ✅ |
| Android/iOS Agent | ✅ | — |
| RTOS 地图 | ✅ | — |
| 前端 3D 地图 Skill | — | ✅ |
| Three.js / GLTF 集成 | — | ✅ |

两家各有侧重，高德偏**多端覆盖**，腾讯地图偏**前端可视化**。

## 五、跑腿与办公协作盘点

跑腿与办公协作是 Agent 化的另一个主战场，覆盖了**即时配送**和**企业内部协作**两大场景。

### 5.1 美团跑腿 Skill

| 项目 | 信息 |
|------|------|
| **品牌** | 美团跑腿 |
| **集成形态** | Skill |
| **发布时间** | 2025 年 6 月 |
| **GitHub 地址** | [github.com/meituan/MT-Paotui-For-Client](https://github.com/meituan/MT-Paotui-For-Client) |
| **核心能力** | 跑腿订单提交、地址簿管理 |

**特色设计**：

美团跑腿 Skill 的两个特色设计非常贴合实际场景：

1. **地址簿优先匹配**：不需要每次重新输入地址，直接复用历史地址簿，降低用户输入成本。
2. **订单预览卡片确认后提交**：在提交订单前，会先以卡片形式展示订单详情（取件地址、送件地址、物品信息、预估费用），用户确认后才正式提交。

**支付方式**：

需要打开 App 操作。这意味着 Skill 主要解决"下单"环节，付款仍需要回到美团 App 完成。

### 5.2 飞书 Skill / CLI / MCP

| 项目 | 信息 |
|------|------|
| **品牌** | 飞书 |
| **集成形态** | Skill + CLI + MCP（三种全支持） |
| **官方平台** | [https://open.feishu.cn/?lang=zh-CN](https://open.feishu.cn/?lang=zh-CN) |
| **核心能力** | 办公协作（消息、文档、日程、任务等） |

飞书是办公协作赛道的**主战场产品**，三种形态全部支持。其中 **CLI 在 2025 年 3 月份开源**，这意味着开发者可以直接把飞书 CLI 集成到 Claude Code、Cursor 等编程 Agent 中，在开发过程中调用飞书的 API。

飞书的三种形态覆盖了不同人群：
- **Skill**：让普通用户在 ChatGPT/扣子中操作飞书
- **MCP**：让自研 Agent 接入飞书能力
- **CLI**：让开发者在编程 Agent 中调用飞书

### 5.3 钉钉 Skill / CLI / MCP

| 项目 | 信息 |
|------|------|
| **品牌** | 钉钉 |
| **集成形态** | Skill + CLI + MCP（三种全支持） |
| **官方平台** | [https://open.dingtalk.com/](https://open.dingtalk.com/) |
| **核心能力** | 消息、待办、日程、审批流 |

钉钉同样三种形态全支持，能力覆盖**消息、待办、日程、审批流**四大模块，与飞书功能高度重叠。

**选型建议**：

钉钉和飞书功能高度重叠，**选哪个看公司用哪家**。如果公司已经在用钉钉作为协作平台，那么 Agent 接钉钉的 Skill/CLI/MCP 是顺理成章的选择；反之亦然。技术维度上的差异不大，决策维度更多在组织层面。

### 5.4 企业微信 CLI / Skill / MCP

| 项目 | 信息 |
|------|------|
| **品牌** | 企业微信 |
| **集成形态** | CLI + Skill + MCP |
| **GitHub 地址** | [github.com/WecomTeam/wecom-cli](https://github.com/WecomTeam/wecom-cli) |
| **核心能力** | 消息收发、通讯录管理 |

企业微信的 Agent 化能力相对聚焦，核心是**消息收发**和**通讯录管理**。这对于做企业内部 Agent 的开发者来说是必备能力，比如让 Agent 自动同步企业组织架构变化、批量发送内部通知。

### 5.5 腾讯文档 Skill / MCP

| 项目 | 信息 |
|------|------|
| **品牌** | 腾讯文档 |
| **集成形态** | Skill + MCP |
| **官方平台** | [https://docs.qq.com/open/document/](https://docs.qq.com/open/document/) |
| **核心能力** | 创建编辑在线文档、知识库管理、AI PPT 生成 |

腾讯文档的 Skill/MCP 覆盖**文档创建编辑、知识库管理、AI PPT 生成**。其中 AI PPT 生成是相对特色的能力，意味着 Agent 可以根据一段文字需求自动生成完整的 PPT 文档，而不仅是文字内容。

## 六、支付能力盘点（重点章节）

支付是 Agent 化生态中**最关键也最敏感**的能力。没有支付的 Agent 只能"说"，有支付的 Agent 才能"做"。但支付同时涉及资金安全和信任问题，是整个生态最难啃的骨头。

### 6.1 支付宝 Skill / MCP

| 项目 | 信息 |
|------|------|
| **品牌** | 支付宝 |
| **集成形态** | Skill + MCP |
| **官方平台** | [https://open.alipay.com/](https://open.alipay.com/) |
| **上线节奏** | 2024 年 4 月推出国内首个支付 MCP |

**能力覆盖**：

支付宝 MCP 覆盖五项核心能力：

1. 手机支付
2. 网页支付
3. 订单查询
4. 退款
5. 退款查询

**面向开发者**：

后续推出的**支付集成 Skill** 面向开发者——也就是"收钱的人"用。它可以让 AI 快速接入支付能力，包括：
- 生成收款链接
- 创建订单
- 查询支付状态
- 发起退款

**核心价值**：

支付宝提供了**完整的收款闭环**，并且**个人开发者也可以使用**。这意味着独立开发者不需要企业资质，也可以让自研 Agent 接入支付能力，对个人开发者生态非常友好。

### 6.2 微信支付 Skill / MCP

| 项目 | 信息 |
|------|------|
| **品牌** | 微信支付 |
| **集成形态** | Skill + MCP |
| **GitHub 地址** | [https://github.com/wechatpay-apiv3/wechatpay-skills](https://github.com/wechatpay-apiv3/wechatpay-skills) |
| **上线节奏** | 2024 年开放 MCP（仅腾讯元器可用），2025 年 4 月上线 Skill |

**面向开发者**：

微信支付的 Skill/MCP 面向开发者，核心能力包括：

- 判断支付产品类型
- 给出示例代码
- 检查代码安全

**商品券能力**：

支持完整的商品券生命周期管理：
- 发券
- 核销
- 查询
- 退券

### 6.3 "支付最后一公里"信任难题深度分析

这是整个生态盘点中**最值得深思的问题**。

**现象观察**：

盘点 16 个品牌可以发现一个共性——**几乎所有付钱环节都让用户跳出去自己完成**：

| 品牌 | 支付方式 |
|------|---------|
| 瑞幸 | 扫码支付 |
| 麦当劳 | 跳 App 支付 |
| 美团跑腿 | 打开 App 操作 |

**技术维度**：

技术上**轻轻松松就能做到** Agent 直接付款。支付宝和微信支付都已经开放了完整的 MCP/Skill 接口，从生成订单到扣款的全链路都能 API 化。所以"做不到"显然不是技术原因。

**社会维度**：

真正卡住的是**整个社会的信任**还没到那一步。让 AI 直接动用户的钱，这件事在任何国家、任何文化里都需要极高的信任成本。用户会担心：
- AI 会不会误下单？
- AI 会不会被钓鱼诱导？
- 出了问题谁负责？
- 资金安全怎么保障？

**生态维度**：

信任不是靠一两个产品能建起来，需要**整个生态慢慢磨**。需要支付平台、品牌方、Agent 框架、监管机构、用户教育共同推进。任何一个环节掉链子都可能导致信任崩塌。

引用原文最经典的一句：

> "技术上肯定轻轻松松就能做到，但，整个社会的信任上，还没到那一步。"

这是当前 Agent 化生态最清醒的认知。技术已经到位，但社会接受度还没到位。这种"技术领先于社会"的鸿沟，需要时间去填补。

## 八、第三方集成盘点

第三方集成是 Skill/MCP 生态的另一种形态：不是品牌自己发布 Skill，而是被集成到大型 AI 产品中。

### 8.1 两种集成模式

需要区分两种模式：

| 模式 | 描述 | 典型案例 |
|------|------|---------|
| **独立发布 Skill** | 品牌自己发布 Skill/MCP/CLI，开放给所有 Agent 调用 | 瑞幸、飞书、支付宝 |
| **第三方集成** | 在 AI 产品中集成第三方服务，由 AI 产品方主导 | 千问接淘宝、豆包接曹操出行 |

第一种是开放生态模式，第二种是封闭生态模式。两者各有优劣，开放模式覆盖更广，封闭模式体验更深度。

### 8.2 千问（通义千问）

**接入节奏**：

- **2025 年 1 月**：接入淘宝、支付宝、淘宝闪购、飞猪、高德等阿里生态
- **2025 年 6 月**：开放第三方 Skill，肯德基、蜜雪冰城、东方航空首批接入

千问走的是"**阿里生态优先 + 逐步开放第三方**"的路线。先打通阿里内部生态形成闭环，再向外部品牌开放。首批第三方接入的肯德基、蜜雪冰城、东方航空都是高频消费场景，能快速验证生态价值。

### 8.3 豆包

**接入动态**：

- **2025 年 6 月 22 日**：上线打车服务，接曹操出行，灰测中

豆包选择了**曹操出行**作为打车服务的接入方，而不是滴滴。这可能是为了避免与滴滴自有 Skill 形成直接竞争，也可能是字节与曹操出行的战略合作。

目前还在灰测中，意味着豆包在 Agent 化生态上还在小步试错。

### 8.4 WorkBuddy（腾讯产品）

WorkBuddy 是腾讯推出的产品，特点是**内置大量 Skill 和 MCP**，并且深度集成腾讯系生态：

- 微信支付 AI 专属卡
- QQ 邮箱
- 腾讯文档
- 腾讯 ima
- 腾讯问卷
- 微云

这种"全家桶"模式的优势是开箱即用，劣势是封闭性强。用户在 WorkBuddy 中可以一站式调用腾讯系产品的能力，但跨生态协作相对受限。

## 九、趋势洞察

基于 16 个品牌的盘点，可以提炼出三个核心趋势论点。

### 9.1 核心论点 1：Agent 化时代正在到来

我们正在向一个**逐渐全面 Agent 化的时代过渡**。当 Agent 能点咖啡、叫出租车、查航班、发飞书消息、发邮件、管理文档、搜酒店，最后自循环完成支付——这意味着 Agent 已经覆盖了用户日常生活和工作的大部分场景。

Agent 不再是"陪聊"的工具，而是"代做"的代理。这种从"对话"到"执行"的跃迁，是 2025 年 AI 行业最重要的范式转换。

### 9.2 核心论点 2：2017 年小程序类比

当前阶段非常类似 **2017 年小程序刚出来时**。

- **2017 年**：大家觉得小程序没啥用，三年后很多品牌都在做，特别是与现实交互多的场景。
- **2025 年**：Skill 和 MCP 现在处于窗口期，先做的人在探路，大量品牌在观望。

但**趋势已不可逆**。一旦早期采用者验证了 Agent 化的价值，观望的品牌会快速跟进。这种"先慢后快"的扩散曲线，与小程序的发展轨迹高度一致。

### 9.3 核心论点 3：贾维斯愿景

Agent 不再只是工具，**慢慢变成数字世界里的另一个自己**。

或者说，是我们每个人心中的那个**贾维斯**。

引用原文最经典的一句：

> "它在慢慢变成你在数字世界里的另一个自己。"

这份名单还会**越来越长**。从 16 个品牌起步，未来会扩展到几十、几百、几千个品牌。Agent 化不是某个行业的特权，而是所有联网服务的必然演化方向。

## 十、FAQ 常见问题解答

### Q1: Skill、MCP、CLI 有什么区别？我该用哪个？

**A**: 三者面向不同人群：

- **Skill** 面向普通用户和产品经理，门槛最低，一句话即可安装使用。
- **MCP** 面向开发者，需要按协议规范接入，能力更强。
- **CLI** 面向开发者，可被 Claude Code 等编程 Agent 调用，适合在开发工作流中使用。

**选择建议**：普通用户选 Skill；自研 Agent 接入选 MCP；用 Claude Code 编码的开发者选 CLI。

### Q2: 普通用户可以使用这些 Skill 吗？需要技术背景吗？

**A**: 可以使用，**不需要技术背景**。Skill 的设计目标就是让普通用户也能用。用户只需要把 Skill 的网址发给 Agent，说一句"安装"即可。安装后用自然语言描述需求，例如"给我点一杯瑞幸的拿铁，要大杯"，Agent 就会自动调用 Skill 完成点单。

### Q3: 为什么所有支付环节都要跳出去自己完成？

**A**: 这不是技术问题，而是**社会信任问题**。技术上 Agent 完全可以直接调用支付接口完成扣款，但整个社会对"AI 直接动钱"的信任还没有建立。用户会担心误下单、被钓鱼、责任归属等问题。信任需要支付平台、品牌方、Agent 框架、监管机构、用户教育共同推进，需要时间。

### Q4: 飞书和钉钉的 Skill 功能差不多，该怎么选？

**A**: **看公司用哪家就用哪家**。飞书和钉钉的 Skill/CLI/MCP 三种形态都支持，能力覆盖高度重叠（消息、待办、日程、审批流）。技术维度差异不大，决策维度在组织层面——选择公司已经在用的协作平台，避免双平台切换的成本。

### Q5: 高德地图和腾讯地图的 Skill 有什么差异？

**A**: 两者侧重不同：

- **高德**偏**多端覆盖**，除了位置服务和地图开发，还提供 Android Agent、iOS Agent、RTOS 地图，适合移动端和车载场景。
- **腾讯地图**偏**前端可视化**，提供前端地图开发 Skill，支持 3D 地图、Three.js 集成、GLTF 模型渲染，适合网页端 3D 地图场景。

### Q6: 千问接入的第三方 Skill 和品牌独立发布的 Skill 有什么不同？

**A**: 主要差异在于**开放性**：

- **品牌独立发布 Skill**（如瑞幸）：开放给所有 Agent 调用，任何 Agent 框架都可以接入。
- **千问接第三方 Skill**（如肯德基接入千问）：被集成到千问产品中，只能在千问内调用，跨生态协作受限。

前者是开放生态，后者是封闭生态。前者覆盖更广，后者体验更深度。

### Q7: Agent 化浪潮现在处于什么阶段？

**A**: 处于**窗口期**，类似 2017 年小程序刚出来时。先做的品牌在探路，大量品牌在观望。但趋势已不可逆——一旦早期采用者验证了价值，观望者会快速跟进。目前 16 个品牌已经覆盖餐饮、出行、办公、支付、内容创作五大场景，验证了 Agent 化的可行性。

### Q8: 这些 Skill 都是免费的吗？

**A**: 大部分 Skill 的**安装和使用是免费的**，但具体调用底层服务可能产生费用。例如：

- 瑞幸 Skill 试用免费，但点咖啡需要付款。
- 飞猪 Skill 体验版免费，完整能力需申请 API Key（可能涉及商务合作）。
- 支付宝 MCP 免费接入，但每笔交易按支付宝标准费率收费。
- 美图 Skill 部分高级能力（如 AI 写真）可能需要付费。

具体费用以各品牌官方平台公示为准。

### Q9: 如何让自己的产品也支持 Skill/MCP？

**A**: 通用路径是：

1. **明确接入目标**：是面向普通用户（选 Skill）还是开发者（选 MCP/CLI）？
2. **梳理能力清单**：把产品的核心能力列出来，例如查询、下单、支付。
3. **按协议开发**：Skill 按目标 Agent 框架规范开发；MCP 按 Model Context Protocol 规范开发；CLI 按命令行工具规范开发。
4. **发布到市场**：通过官方平台或 GitHub 发布，提供文档和示例。
5. **持续维护**：根据用户反馈迭代能力，处理边界情况。

参考支付宝、飞书、瑞幸等品牌的开源仓库和官方文档，可以快速建立认知。

### Q10: 国内 Skill/MCP 生态未来会如何发展？

**A**: 基于当前趋势，未来发展方向包括：

1. **品牌数量持续增长**：从 16 个扩展到几十、几百个，覆盖更多垂直场景。
2. **支付最后一公里逐步打通**：随着信任建立，Agent 直接付款会成为可能。
3. **生态标准化**：Skill/MCP/CLI 三种形态会进一步统一规范，降低接入成本。
4. **行业从封闭走向开放**：第三方集成模式会逐步开放，跨生态协作增强。
5. **Agent 跨品牌组合**：用户可以用一句话串联多个品牌 Skill，例如"安排周末去杭州的行程"自动调用飞猪订票、高德查酒店、瑞幸点咖啡。

引用原文愿景：**"它在慢慢变成你在数字世界里的另一个自己。"**
