# Step 2 专项练习——现有方案与假设列举

**Step 2核心目标**：系统性列出通行做法、"常识"和所有隐含假设，让看不见的前提浮出水面。参考[08-methodology-framework.md](../08-methodology-framework.md#78-100)。

### 题3.2.1 【🌱】视频会员自动续费——显式与隐式假设列举
**场景**：各大视频平台都推行"首月低价+自动续费"模式，用户开通后经常忘记取消，持续扣费。你觉得这个模式有问题，想分析它。
**你的任务**：
1. 列出视频会员自动续费模式的主流设计（行业"标准答案"）
2. 区分并列出：
   - 至少3条显式假设（平台明确说出来的前提）
   - 至少5条隐式假设（心照不宣但没明说的前提）
3. 列出这个领域至少3条被认为"不可能"或"一直如此"的事情

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#82-88)中显式/隐式假设、"不可能"清单的操作方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 主流方案：首月6元/9元低价，次月起25元/月自动续费，取消入口深，到期前无提醒或提醒不明显
  2. 显式假设：
     - 用户认为自动续费更方便，不用每月手动开通
     - 连续订阅比单月购买更优惠（平台宣称）
     - 用户有随时取消的自由
  3. 隐式假设：
     - 大部分用户会忘记取消（这是收入的核心来源）
     - 发现被扣费后，嫌麻烦不会申请退款
     - 用户不会因为"被自动扣费"而彻底流失
     - 监管不会严格要求显著提醒或简化取消流程
     - 即使被吐槽，转化率仍然高于其他模式
     - 用户对"省几块钱"的敏感度高于"未来长期支出"
  4. "不可能"清单：
     - 不可能取消自动续费作为默认选项（否则收入会大幅下降）
     - 不可能把取消按钮放得和开通一样显眼
     - 不可能到期前3天主动强提醒
     - 不可能让单月价格和连续包月一样便宜
- 常见陷阱：只列显式假设，漏掉最关键的隐式假设（忘记取消、嫌麻烦不退款这些才是模式成立的核心）；把道德判断混入——这一步是列举不是批判，先列全再质疑
- 关键检查点：隐式假设数量是否是显式的2倍以上？是否列出了"心照不宣"的商业假设？是否识别出了"不可能"清单里最反常识的那条？
</details>

### 题3.2.2 【🌱】在线教育课程定价——"不可能"清单
**场景**：在线教育课程通常定价99元、199元、299元，或者几千元的"训练营"，很少有中间价位。你想分析这个现象。
**你的任务**：
1. 列出在线教育课程定价的3-5种主流模式（行业标准答案）
2. 为每种模式列出其背后的显式+隐式假设
3. 列出这个领域至少5条被认为"一直如此"或"不可能"的事情
4. 用"魔法棒"问题思考：如果消除所有约束，理想的定价方式可能是什么？

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#82-89)中假设列举、"不可能"清单、魔法棒问题的操作方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 主流模式：99元以下低价引流课、199-399元标准录播课、2000-5000元直播训练营、万元以上1对1/私教
  2. 假设示例（以训练营为例）：
     - 显式：高定价对应高价值、有服务监督才会学完、直播比录播效果好
     - 隐式：用户对"贵=好"有心理预期、低价买了用户不珍惜不学、必须有高价产品才能覆盖获客成本、社群督促能提升完课率、用户为"可能改变自己"付费而非为内容付费
  3. "不可能"清单：
     - 不可能按学习效果付费（结果不可控）
     - 不可能把课程定在500-1500元区间（两头不讨好，卖不动）
     - 不可能让用户先学满意再付费（道德风险太高）
     - 不可能只卖内容不卖服务（内容太容易被盗版/比价）
     - 不可能不做营销转化（自然流量不够）
  4. 魔法棒理想方案：可能是按学习进度和效果阶梯付费、个性化定价、学习成果对赌等
- 常见陷阱：只从"消费者觉得坑"的视角列假设，忽略商业侧的隐式假设（获客成本高这个前提）；过早判断哪些假设"合理"——这一步只是列举不是评价
- 关键检查点：是否同时考虑了用户侧和商业侧的假设？"不可能"清单是否足够长（至少5条）？魔法棒方案是否真的消除了约束而不是现有方案的小改进？
</details>

### 题3.2.3 【🌱】健身房年卡模式——假设全扫描
**场景**：健身房普遍推年卡，价格比月卡便宜很多，但统计显示70%以上年卡用户去不到10次就不去了。
**你的任务**：
1. 列出健身房年卡模式的完整设计（价格结构、销售话术、合同条款）
2. 分类列出所有假设：
   - 用户侧显式假设（用户办卡时以为的）
   - 用户侧隐式假设（用户没意识到的自己的行为模式）
   - 健身房侧显式假设
   - 健身房侧隐式假设
3. 列出3条"不可能"清单
4. 思考：为什么健身房不主推次卡/月卡？

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#82-89)的完整假设列举方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 主流设计：年卡2000-3000元（折合每天5-8元），月卡300-500元（折合每天10-16元），销售反复强调"算下来每天只要一杯咖啡钱"，合同约定不能退
  2. 假设分类：
     - 用户显式：我会每周去3次以上、年卡更划算、办了卡我就会有动力健身
     - 用户隐式：我未来一年的健身热情会和现在一样高、我能克服惰性、我不会受伤/搬家/换工作、"划算"是重要决策因素
     - 健身房显式：年卡给用户最大优惠、鼓励用户长期坚持健身
     - 健身房隐式：大部分人坚持不下来（这是盈利核心）、用户高估自己的自律能力、退卡的人比例很低、年卡提前回笼资金可以开新店、设备容量按同时在店人数设计不需要接待所有会员
  3. "不可能"清单：
     - 不可能主推次卡（用户按次付费的话健身房接待能力不够，收入不稳定）
     - 不可能给年卡用户 easy 退款通道
     - 不可能在销售时提醒"80%的人坚持不下来，你确定吗？"
  4. 为什么不主推次卡：因为商业模式建立在"用户付费但不来"的基础上，如果都按次付费，健身房需要大几倍的面积和设备，成本结构完全不同
- 常见陷阱：只骂"健身房套路"，不系统分析双方的假设；把用户"不理性"当偶然，实际上这是模式设计时就考虑到的可预测行为
- 关键检查点：是否从买卖双方都列了假设？是否区分了用户"以为的"和"实际的"行为假设？是否理解了商业模式成立的核心隐式假设？
</details>

### 题3.2.4 【📚】技术选型场景——四类假设分类
**场景**：你们团队要做一个新的Web系统，技术负责人提议"用我们最熟悉的Java技术栈，数据库用MySQL，缓存用Redis，单体架构先快速上线"。这是很多团队的标准选择。
**你的任务**：
1. 列出这个"标准技术选型"方案背后的所有假设，至少10条
2. 按照四类假设分类：
   - 🟢 物理定律类（难质疑，必须接受）
   - 🔵 技术约束类（当前技术条件下成立，未来可能变）
   - 🟡 经济成本类（成本/收益/资源约束）
   - 🔴 惯例文化类（最值得质疑，"一直这么做"的）
3. 对每条假设标注可信度，说明在什么条件下可能不成立

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#L88)中假设四分类的方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径（假设示例）：
  - 🟢 物理类：网络请求延迟不可能为0、数据存储需要持久化介质、单台服务器有性能上限
  - 🔵 技术类：Java生态成熟问题都能找到解决方案、MySQL在大多数场景下性能足够、Redis缓存能提升读性能、单体架构初期开发速度更快
  - 🟡 经济类：团队学习新技术有成本、招聘Java程序员更容易更便宜、服务器成本初期可控、快速上线比技术完美更重要
  - 🔴 惯例类："我们一直用Java"所以新项目也用Java、Web系统就该用关系型数据库、单体是初创项目的标准选择、微服务是后面才考虑的"高级"架构、缓存必须用Redis
- 条件不成立示例：
  - 技术类假设不成立：如果系统需要处理每秒10万+写请求，MySQL可能不是最优
  - 经济类假设不成立：如果团队里有Go高手且系统是高并发场景，学习成本假设不成立
  - 惯例类假设不成立：如果是全新领域且团队学习能力强，"一直用Java"就不是理由
- 常见陷阱：把惯例假设当成技术必然（"做Web就该用MySQL"）；把当前团队的技术栈熟悉度当成"Java本身更好"；只列技术假设忽略组织/经济/惯例假设
- 关键检查点：四类假设是否都覆盖了？惯例类假设数量是否不少于技术类？每条假设是否能说出"不成立的条件"？
</details>

### 题3.2.5 【📚】"996加班"问题——多维度假设
**场景**：互联网行业普遍存在加班文化，很多人认为"互联网公司就该加班"、"不加班做不完"、"加班说明你努力"。你想分析这个现象。
**你的任务**：
1. 列出"996加班"模式下，公司和员工双方的主流做法
2. 分类列出所有假设（至少12条），覆盖：
   - 工作效率假设
   - 管理假设
   - 市场竞争假设
   - 人性激励假设
   - 行业惯例假设
3. 哪些是物理/技术约束？哪些纯粹是惯例？
4. 列出3条最值得质疑的隐式假设

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#88-94)中假设分类、常见陷阱（只列技术忽略商业/组织/人性）。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径（假设示例）：
  - 效率类：工作时长=产出、加班时工作效率和正常上班一样、人可以连续高强度工作不降低效率、更多时间总能做出更多东西
  - 管理类：员工在公司才是在工作、看不见就是没干活、加班的员工更努力忠诚、靠延长时间可以弥补能力/流程不足、项目延期是因为时间不够而非方向错了
  - 竞争类：竞争对手在加班所以我们必须加、市场窗口期极短错过就没机会、速度比质量/效率更重要、人才供大于求不怕没人加班
  - 激励类：年轻人愿意用健康换钱/成长、给加班费/调休就有动力、加班是"奋斗者"的标志、文化灌输能让大家接受加班
  - 惯例类：互联网行业就是这样、创业公司必须加班、不加班的公司没有前途、大家都加我不加显得我不努力
- 约束vs惯例：
  - 接近物理约束：人需要休息（长期加班效率下降是生理学事实）、沟通协调成本随人数非线性增长
  - 纯惯例：9点上班9点下班的时间设定、加班就是努力的文化、项目延期就靠加人加班解决（Brooks定律：加人反而更慢）
- 最值得质疑的隐式假设：
  1. 延长工作时间=增加有效产出（忽略效率递减和疲劳导致的错误）
  2. 加班是必要的（忽略很多加班是流程低效/方向错误/管理混乱导致的）
  3. 员工愿意/能够长期接受这种模式（忽略倦怠期和人才流失）
- 常见陷阱：只从"老板剥削"道德批判，不系统分析假设；把所有加班都归为一类（有些真的是紧急情况，有些是低效导致的常态）；忽略员工侧也有接受加班的隐式假设（赚快钱、学东西、Peer Pressure）
- 关键检查点：是否同时分析了公司侧和员工侧的假设？是否区分了"真的需要"和"文化惯性"的加班？是否识别出了人性/管理层面的假设而非只谈工作本身？
</details>

### 题3.2.6 【🔥】"远程办公效率低"——隐式假设深度挖掘
**场景**：疫情后很多公司要求员工回办公室，理由是"远程办公效率低"、"沟通不方便"、"团队凝聚力差"，这是一个非常普遍的说法。
**你的任务**：
1. 列出"远程办公效率低"这个说法背后，所有能想到的隐式假设——尽可能多，至少15条
2. 对这些假设进行分类：哪些是物理约束？哪些是技术工具问题？哪些是管理能力问题？哪些纯粹是文化惯例？
3. 找出其中**最核心、最隐蔽、最少被提及**的3条隐式假设
4. 思考：这些假设在什么条件下成立？在什么条件下不成立？

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#91-94)中"遗漏隐式假设"的常见陷阱，多维度思考（技术/管理/人性/商业/文化）。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径（隐式假设示例，至少15条）：
  - 效率定义类："效率"=坐在工位的时长、效率=老板能看到你在干活、产出难以量化所以用"在场"代理
  - 沟通类：面对面沟通效率最高、很多沟通必须即时、自发的饮水机聊天很重要、不在场就会信息脱节
  - 管理类：员工在家会偷懒摸鱼、管理者不会用目标/结果管理，只会用"在场"管理、需要随时能看到人叫到人、远程无法做绩效评估
  - 文化类：办公室文化是公司文化的载体、团队凝聚力必须靠面对面、远程员工对公司忠诚度低、新人必须在办公室才能被带出来
  - 工具/环境类：家里环境不适合工作、员工家里没有合适的办公条件、现有工具无法支持远程协作
  - 权力类：管理者的权威部分来自"在现场看着大家"、失去在场感会削弱管理控制力
  - 商业类：写字楼租金付了不用浪费、客户觉得我们在办公室才靠谱
  - 人性类：人天生有惰性需要监督、大部分人不够自律、居家会被家务/家人打扰
- 最核心最隐蔽的3条假设：
  1. **效率的度量标准**：工业时代的"在场=工作"假设不适应知识工作——知识工作的产出是结果不是时长，但大多数管理者还没学会管理结果
  2. **管理能力的边界**：很多管理者的管理方法建立在"现场监督"基础上，远程让这些方法失效，所以他们觉得"效率低"——其实是管理方法失效，不是员工效率低
  3. **可见性偏差**：在办公室时员工摸鱼你看不到（因为在你面前反而不好意思一直摸，或者看起来在忙），远程时你看不到他们工作所以默认他们在摸鱼——这是认知偏差不是事实
- 成立条件：如果工作必须物理在场（如制造业、实验室）、如果团队完全不会结果导向管理、如果员工确实高度自律性差、如果协作工具极差，远程效率可能真的低
- 不成立条件：如果工作是知识创造类、如果有清晰的目标和结果度量、如果管理者懂得异步沟通和授权、如果有好的协作工具，远程效率可能更高（更专注、更少无用会议、更灵活）
- 常见陷阱：把"我不适应远程"等同于"远程效率低"；把特定公司/团队远程做不好的经验推广为"远程就是不行"；只列技术/沟通类表面假设，漏掉管理权力/效率度量/认知偏差这些深层假设
- 关键检查点：假设数量是否足够多（15+）？是否挖到了管理方法论、权力结构、认知偏差层面？是否区分了"远程本身的问题"和"用旧方法管理远程的问题"？是否承认不同工作类型/团队/人适合不同模式？
</details>
