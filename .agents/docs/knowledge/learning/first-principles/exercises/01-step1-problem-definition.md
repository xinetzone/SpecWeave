# Step 1 专项练习——问题定义与边界澄清

**Step 1核心目标**：确定真正要解决的问题，避免在错误问题上浪费精力。参考[08-methodology-framework.md](../08-methodology-framework.md#55-75)。

---

### 题2.1.1 【🌱】通勤时间太长——症状vs问题区分
**场景**：你每天上下班通勤需要1.5小时，你觉得"通勤时间太长"是个问题。
**你的任务**：
1. 列出"通勤时间太长"背后可能的3-5个不同层级的真实问题
2. 区分哪些是表面症状，哪些是真正值得解决的问题
3. 用问题陈述模板（现状/期望/障碍/价值）描述你最终认定的核心问题

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#59-64)中区分症状vs问题、问题陈述模板的操作方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 连续追问：通勤长→为什么长？→住得远+交通堵→为什么住得远？→房租便宜→为什么要省房租？→...
  2. 症状可能包括：睡眠不足、陪伴家人时间少、精力消耗大、错过早会、可支配时间少
  3. 真正问题可能是："如何在住房成本、通勤时间、生活质量三者间找到平衡"，而非单纯"缩短通勤"
  4. 问题陈述示例：现状是每天通勤1.5小时，消耗20%可用精力；期望是通勤控制在45分钟内同时不显著增加住房成本；核心障碍是公司附近房租贵30%；解决价值是每天多出1小时可支配时间
- 常见误区：直接接受"通勤时间太长"就是问题（这是症状），或者直接跳到"应该搬家/换工作"的解决方案
- 关键检查点：是否区分了症状和问题？是否追问了至少3层"为什么"？问题陈述是否完整包含四要素？
</details>

### 题2.1.2 【🌱】手机总是没电——5Why练习
**场景**：你的手机经常在下午3点就没电了，需要到处找充电宝。
**你的任务**：
1. 对"手机总是没电"连续问5个"为什么"，穿透表面症状
2. 在每个"为什么"层级，说明对应的可能问题方向
3. 最终你发现的最根本问题是什么？

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#60-61)中连续问3-5个"为什么"的操作方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径（5Why示例）：
  1. 为什么没电？→亮屏时间长，耗电快
  2. 为什么亮屏时间长？→一直在刷社交App+工作消息多
  3. 为什么刷这么久？→碎片时间无意识打开，App推送频繁
  4. 为什么推送这么频繁还不关？→担心错过重要工作消息，推送权限没精细管理
  5. 为什么不换大电池手机/随身带充电宝？→不想增加负重，觉得"应该"能撑一天
- 真正问题可能不是"电池太小"，而是"手机使用习惯与电池容量不匹配"或"通知管理导致无意识耗电"
- 常见误区：Why问到第二层就停在"电池不行"（症状），直接跳到"换手机"方案；Why链条逻辑跳跃
- 关键检查点：每个Why是否是对上一层回答的直接追问？是否避免了逻辑跳跃？最终问题是否指向可行动的根本原因而非表面现象？
</details>

### 题2.1.3 【🌱】外卖塑料包装太多——问题边界划定
**场景**：你注意到每天点外卖会产生大量塑料包装垃圾，觉得这是个问题。
**你的任务**：
1. 用问题陈述模板描述这个问题
2. 明确划定问题边界：列出3-5项**在范围内**的内容，和3-5项**不在范围内**的内容
3. 从"消费者视角"和"外卖平台视角"两个不同视角重述这个问题

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#63-64)中划定边界、多视角重述的操作方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 问题陈述：现状是单次外卖产生3-5件塑料包装，个人每月产生约60件塑料垃圾；期望是外卖包装减量50%以上同时不影响餐品质量和配送；核心障碍是商家成本、配送防漏要求、平台缺乏激励；解决价值是减少个人环境足迹，推动可持续消费
  2. 边界示例：
     - 在范围内：个人外卖消费决策、可重复使用餐具选择、对环保包装商家的偏好
     - 不在范围内：国家限塑政策制定、塑料材料技术研发、外卖平台整体规则重构（个人无法直接影响）
  3. 视角重述：
     - 消费者视角：如何在不牺牲便利性和餐品体验的前提下，减少个人产生的外卖塑料垃圾？
     - 平台视角：如何在不显著增加商家成本、不影响配送体验的前提下，推动平台上包装减量？
- 常见误区：边界划得太宽（要"解决全球塑料污染"）导致无从下手；边界划得太窄（只考虑"不用一次性餐具"）忽略其他选项；视角单一，只从自己角度看问题
- 关键检查点：边界是否清晰可执行？两个视角的问题陈述是否有本质差异而非换个说法？
</details>

### 题2.1.4 【📚】团队会议效率低——多视角问题重述
**场景**：你所在的团队每周有5-8个会议，大家普遍抱怨"会议太多太长效率低"，但会议数量一直减不下来。
**你的任务**：
1. 分别从以下4个视角重述"会议效率低"这个问题：
   - 参会人视角（普通员工）
   - 会议组织者视角（团队负责人）
   - 项目进度视角（交付目标）
   - 组织成本视角（公司投入）
2. 对比不同视角下问题定义的差异
3. 划定清晰的问题边界，说明哪些会议问题不在本次解决范围内

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#64-69)中多视角重述、边界划定的操作方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 多视角重述示例：
     - 参会人：如何减少不必要的参会时间，让我能有整块时间专注工作？
     - 组织者：如何通过会议高效对齐信息、做出决策，而不浪费团队时间？
     - 项目进度：如何确保关键信息同步和决策节点不因为会议问题延误？
     - 组织成本：如何让占团队20%时间的会议投入产生相应价值？
  2. 视角差异：参会人关注"我的时间被浪费"，组织者关注"会议目的是否达成"，项目关注"决策效率"，组织关注"投入产出比"
  3. 边界示例：范围内是本团队内部会议的效率优化；不在范围内：跨部门大会议、公司全员会、需要大量协作共创的工作坊（这些有不同的规律）
- 常见陷阱：只从"参会人抱怨"单一视角接受问题定义；问题定义过宽试图"改革所有会议"；不加审视就接受"会议太多"的前提——可能不是数量问题而是质量问题
- 关键检查点：四个视角是否真正不同而非同义反复？是否识别出不同视角下的不同痛点？边界是否明确排除了不适合用同一方法解决的会议类型？
</details>

### 题2.1.5 【📚】代码Bug率高——问题分层与边界
**场景**：你们团队最近上线的功能Bug率比其他团队高30%，领导要求"降低Bug率"。
**你的任务**：
1. 区分"Bug率高"是症状还是问题？连续追问3-5层"为什么"
2. 从技术视角、流程视角、人员视角三个角度重述问题
3. 明确划定问题边界：本次分析聚焦哪些环节？哪些不在范围内？
4. 用问题陈述模板完整描述最终认定的核心问题

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#59-69)中症状vs问题区分、多视角重述、边界划定的完整方法。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 5Why示例：Bug率高→为什么？→测试没覆盖到+开发自测不足→为什么？→需求变更频繁+排期紧没时间测→为什么？→需求评审不充分+排期评估时没留测试时间→为什么？→只看功能点交付速度不考核质量→...
  2. 多视角重述：
     - 技术视角：如何提升代码可测试性，让缺陷更早被发现？
     - 流程视角：如何在需求-开发-测试全流程中建立质量门禁，不让带Bug代码上线？
     - 人员视角：如何让开发有动力和时间做自测，而不是只赶进度？
  3. 边界示例：范围内是本团队新功能开发的质量流程优化；不在范围内：历史遗留代码重构、自动化测试框架大改造、公司整体绩效考核体系调整
  4. 问题陈述示例：现状是新功能上线后千行代码Bug数为3.5（团队平均2.2），30%线上Bug本可在开发阶段发现；期望是3个月内将千行代码Bug数降到2.5以下；核心障碍是排期紧张时测试时间被挤压，缺乏质量门禁；解决价值是减少线上故障处理时间，提升团队交付可信度
- 常见陷阱：直接接受"降低Bug率"作为问题陈述（这是期望结果不是问题定义）；只从技术视角看问题（认为"技术差"），忽略流程/激励因素；边界划得太宽试图"全面提升质量"而失焦
- 关键检查点：是否区分了"Bug率高"这个症状和背后的根本问题？三个视角是否覆盖了问题的不同维度？问题陈述是否可衡量、有明确边界？
</details>

### 题2.1.6 【🔥】"我们需要一个更好的项目管理工具"——识别伪问题
**场景**：团队例会上，有人提出"我们需要一个更好的项目管理工具"，很多人附和："现在的工具太难用了"、"信息太分散"、"进度跟踪不透明"，领导让你调研选型。
**你的任务**：
1. 为什么"需要更好的项目管理工具"是一个伪问题（把方案当问题）？
2. 这个"方案"背后真正要解决的问题可能是什么？列出至少3个不同方向的真实问题
3. 从至少2个视角重述这些真实问题
4. 如果真的要选工具，应该在完成哪些问题定义步骤之后？

**提示**：参考[08-methodology-framework.md](../08-methodology-framework.md#66-69)中"把解决方案当问题"的常见陷阱。
<details>
<summary>👉 点击查看参考答案要点（先独立思考再展开！）</summary>

**参考答案要点（非唯一正确解）**：
- 思考路径：
  1. 为什么是伪问题：它已经预设了解决方案——"换工具"，但工具可能不是根本原因。这是典型的"把方案当问题"：就像"我们需要一把更快的锤子"可能真正问题是"钉子钉不直"或者"我们根本不该钉钉子"。
  2. 背后可能的真实问题：
     - 进度不透明：团队成员不知道其他人在做什么，依赖关系不清晰
     - 信息分散：需求、讨论、决策分散在多个工具/群聊中，找不到上下文
     - 责任不清：任务分配不明确，出了问题不知道找谁
     - 流程问题：缺乏明确的状态流转规则，即使换工具也会乱
     - 协作习惯：大家不习惯更新状态，工具再好没人用也没用
  3. 视角重述：
     - 项目负责人视角：如何实时掌握项目真实进度和风险，而不是到截止日才发现延期？
     - 团队成员视角：如何减少在多个工具间切换找信息的时间，清晰知道自己该做什么？
  4. 正确顺序：先定义真正的问题→列举现有方案和假设→拆解问题要素→验证"工具是否是瓶颈"这个假设→如果确认工具是瓶颈之一，再调研选型
- 常见陷阱：不加审视就接受别人提出的问题（尤其是领导/同事提出时容易从众）；直接进入"工具选型对比"的解决方案模式；换了工具发现问题依旧（因为工具不是根本原因）
- 关键检查点：这个陈述里是否包含了具体的解决方案？如果去掉方案，问题还成立吗？换了这个方案，是否有其他完全不同的路径也可能解决问题？是否有人因为"新工具"听起来很有吸引力而忽略了真正问题？
</details>
