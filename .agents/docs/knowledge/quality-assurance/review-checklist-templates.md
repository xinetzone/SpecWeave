# 知识评审清单模板（2份开箱即用）

别自己瞎评审了，对着这个清单勾，勾完比你自己看三遍都管用。

我见过太多评审会了：要么大家坐那说"写得不错"、"挺好的"，散会之后啥问题没发现；要么东一榔头西一棒子，想到啥说啥，重要问题漏了，鸡毛蒜皮的事扯半小时；要么评完了连个记录都没有，作者改没改、改了啥，没人知道。

这两份清单就是解决这个问题的。对着项来，一项一项过，一项一项打分，一项一项写问题，评审完了有记录、有结论、有跟踪。别再凭感觉评审了。

---

## 这模板怎么用？先讲清楚再看表格

### 什么时候用哪个模板？

| 文档类型 | 用哪个模板 |
|---------|-----------|
| API文档、接口文档 | 模板1：通用技术文档 |
| 教程、入门指南、操作手册 | 模板1：通用技术文档 |
| 技术方案、设计文档、架构文档 | 模板1：通用技术文档 |
| SOP、标准操作流程 | 模板2：方法论/模式库 |
| 最佳实践、经验总结、模式沉淀 | 模板2：方法论/模式库 |
| 可复用的流程规范、方法论 | 模板2：方法论/模式库 |

简单记：**教人"怎么做某件具体事"用模板1，教人"怎么做好某类事/什么情况下用什么方法"用模板2。**

### 怎么打分？1-5分是什么意思？

别打人情分，别都打3分和稀泥。每个分值有明确标准，对照着打：

| 分值 | 说明 | 判断标准 |
|-----|------|---------|
| 1分 | 完全没做到 | 问题很严重，这一项基本等于没写，或者全错了 |
| 2分 | 做得很差 | 有这个意思，但大部分内容有问题，缺很多东西 |
| 3分 | 基本做到但有不足 | 大方向对，但有明显瑕疵，需要修改 |
| 4分 | 做得不错 | 大部分地方都很好，只有小问题，微调就行 |
| 5分 | 做得很好 | 这一项挑不出毛病，可以当范本 |

### 严重程度怎么判断？三档很清楚

- 🔴 **必须改**：会误导读者、有事实错误、步骤走不通、看完会踩坑——不改不能发
- 🟡 **建议改**：表达不好、结构乱、例子不够、说不清楚——改了体验会好很多，不改也能用但有遗憾
- 🟢 **可选**：个人偏好、风格问题、锦上添花的建议——作者自己判断要不要改

### 评审完怎么算结论？

算总分+看有没有🔴必须改的项：
- 所有项都≥3分，且没有🔴必须改的问题 → **通过**
- 有个别项<3分或者有🟡建议改的问题，但没有🔴必须改 → **修改后通过**（作者改完不用再开评审会，改完给评审人确认一眼就行）
- 有🔴必须改的问题，或者有一半以上项<3分 → **不通过**（改完需要重新评审）

好了，说清楚了，下面上模板。空白表格直接复制就能用。

| 分值 | 通用标准 |
|-----|---------|

| 2分 | 做得很差，大部分内容有问题 |

| 4分 | 做得不错，只有小瑕疵 |
| 5分 | 做得很好，可以当范本 |

| **一、准确性** | 1. 关键数据/结论有可追溯来源（不是拍脑袋说的） | | | | |

| | 3. API参数、命令、代码示例是正确的，能跑通 | | | | |

| | 21. 版本信息/更新时间有标注，读者知道这文档是什么时候的 | | | | |

### 填写示例：评审《FastAPI快速上手指南》

（假设这是一份给Python后端新手看的入门教程，下面是评审人填的内容）

| 评审维度 | 具体评审项 | 评分(1-5) | 问题位置 | 问题描述/修改建议 | 严重程度 |
|---------|-----------|----------|---------|-----------------|---------|
| **一、准确性** | 1. 关键数据/结论有可追溯来源 | 2 | 第3节"为什么FastAPI比Flask快" | 说"FastAPI性能是Flask的300%"，但没给测试数据、测试环境、对比基准，这个数字不知道哪来的。建议要么附上测试代码和结果，要么改成"FastAPI基于ASGI，性能通常比WSGI框架的Flask更高"这种模糊但准确的表述，别拍脑袋给数字。 | 🔴 必须改 |
| | 3. API参数、命令、代码示例是正确的，能跑通 | 1 | 第5节"第一个接口"代码示例 | 示例代码里写的是`@app.get("/")`，但下面启动命令写的是`uvicorn main:app --reload`，文件名写的是`app.py`不是`main.py`，新手照着敲会直接报错找不到模块。代码里也缺了`from fastapi import FastAPI`和`app = FastAPI()`这两行。建议补全导入和实例化，把文件名统一成`main.py`或者把启动命令改成对应文件名。 | 🔴 必须改 |
| **二、完整性** | 6. 步骤完整没有跳步 | 2 | 第7节"连接数据库" | 上来就写"使用SQLAlchemy连接数据库"，然后直接贴代码，但没说要先安装SQLAlchemy、没说数据库连接字符串怎么配置、没说需要先启动数据库服务。新手照着做肯定卡在第一步。建议补充安装命令`pip install sqlalchemy`，补充前置条件"确保你本地已经安装并启动了PostgreSQL/MySQL"，给一个完整的连接配置示例。 | 🔴 必须改 |
| **三、可读性** | 9. 第一次出现的术语/缩写有解释 | 2 | 第2节"ASGI vs WSGI" | 直接用了ASGI、WSGI、Uvicorn这些词，但完全没解释是什么。新手看完会懵：ASGI是啥？为什么用了它就快？建议用一两句话简单解释："ASGI是Python的异步Web服务器网关接口，WSGI是同步版本；Uvicorn是一个ASGI服务器，用来跑你的FastAPI应用"。 | 🟡 建议改 |
| | 11. 复杂/抽象的地方有例子/类比/截图 | 4 | 全文 | 大部分地方都有代码示例，这点做得不错。但第8节"依赖注入"概念比较抽象，建议再加一个生活中的类比（比如"依赖注入就像你点外卖，你不需要自己做饭，餐厅直接把做好的饭给你送过来"），帮助新手理解。 | 🟢 可选 |
| **五、实用性** | 16. 目标读者明确，内容深度匹配 | 3 | 第9节"中间件原理" | 这是一份"快速上手"教程，但第9节花了很大篇幅讲中间件的底层实现原理、源码分析，对新手来说太深入了，反而会劝退。快速上手里点到为止告诉大家"中间件是什么、怎么用"就行，底层原理可以放进阶文档或者加个"深入了解"的扩展阅读链接。 | 🟡 建议改 |
| **六、规范性** | 21. 版本信息/更新时间有标注 | 1 | 文档开头/结尾 | 整篇文档没写适用于FastAPI哪个版本、文档是什么时候写的。FastAPI更新很快，0.x版本API经常变，不写版本的话，过几个月有人照着老文档做踩坑了都不知道为什么。建议在开头加上："本文基于FastAPI 0.100.0版本，最后更新：2024年1月"。 | 🟡 建议改 |

## 模板2：方法论/模式库评审清单

**适用**：SOP、最佳实践、可复用模式、流程规范

### 评分标准总览（每个评审项都按这个打）

| 分值 | 通用标准 |
|-----|---------|
| 1分 | 完全没做到，问题严重 |
| 2分 | 做得很差，大部分内容有问题 |
| 3分 | 基本做到但有明显不足 |
| 4分 | 做得不错，只有小瑕疵 |
| 5分 | 做得很好，可以当范本 |

### 评审清单（空白可复制）

| 评审维度 | 具体评审项 | 评分(1-5) | 问题位置 | 问题描述/修改建议 | 严重程度 |
|---------|-----------|----------|---------|-----------------|---------|
| **一、可迁移性** | 1. 讲清楚了"这是什么问题"，而不是只讲"在我们这怎么解决"（换个项目/团队/场景，别人能识别出这是同一个问题） | | | | |
| | 2. 抽象到了合适的层级，不是绑定太具体的技术/工具/公司流程（别写"在我们公司用Jira的时候要XXX"，要写"使用任务管理工具时要XXX"） | | | | |
| | 3. 有至少2-3个不同场景的应用示例，证明这个方法不是只在一个地方好用 | | | | |
| | 4. 说了和其他类似方法的区别（什么时候用这个，什么时候用那个，别让读者自己猜） | | | | |
| **二、反模式完整性** | 5. 明确说了"不要怎么做"，反模式和正模式至少一样多（光说要怎么做，不说不要怎么做，读者一定会踩坑） | | | | |
| | 6. 每个反模式都讲了"为什么不要这么做"（有后果/有踩坑案例，不是拍脑袋说不行） | | | | |
| | 7. 说了常见的变种和误区（很多人会"改良"你的方法，但改着改着就改歪了，这些歪路要指出来） | | | | |
| **三、边界清晰度** | 8. 明确说了适用场景（什么情况下这个方法好用） | | | | |
| | 9. 明确说了不适用场景（什么情况下别用这个，用了会更糟）——这一条比适用场景还重要 | | | | |
| | 10. 说了前置条件/必要前提（用这个方法之前，你需要已经具备什么条件/什么基础） | | | | |
| | 11. 说了规模/阶段的边界（1个人能用吗？10个人呢？100个人呢？项目初期能用吗？成熟期呢？） | | | | |
| **四、可操作性** | 12. 有step-by-step的操作步骤，不是只讲原则讲大道理（看完知道第一步做什么、第二步做什么） | | | | |
| | 13. 有检查点/验收标准（做到什么程度算做好了，怎么判断有没有做对） | | | | |
| | 14. 有失败案例/踩坑记录（别人用这个方法的时候踩过什么坑，怎么避过去） | | | | |
| **五、成熟度** | 15. 说明这个方法经过了多少个项目/多长时间的验证（是刚想出来的，还是已经在10个项目里跑过一年了） | | | | |
| | 16. 有已知的局限性（这个方法解决不了什么问题，有什么缺点）——没有局限性的方法一定是假的 | | | | |
| | 17. 有版本/演进记录（这个方法是怎么从v1.0迭代到现在的，之前踩过什么坑所以改了） | | | | |

### 填写示例：评审《第三方依赖变更双校验法》

（假设这是一个沉淀的模式文档，讲的是升级/替换第三方依赖的时候要做的校验流程，下面是评审人填的内容）

| 评审维度 | 具体评审项 | 评分(1-5) | 问题位置 | 问题描述/修改建议 | 严重程度 |
|---------|-----------|----------|---------|-----------------|---------|
| **一、可迁移性** | 2. 抽象到了合适的层级，不绑定具体技术/工具 | 2 | 第3节"具体操作步骤" | 步骤里写的全是"在package.json里改版本号"、"跑npm test"、"看Sentry报错"，这绑定死了Node.js+npm+Sentry技术栈。Java项目用Maven、Python项目用pip的人看完没法直接套用。建议把技术栈相关的内容抽象一下："在依赖配置文件中更新版本号"、"运行项目的自动化测试套件"、"观察线上监控系统的异常指标"，然后加个"不同技术栈示例"小节，给npm/maven/pip的对应写法。 | 🔴 必须改 |
| **二、反模式完整性** | 5. 明确说了"不要怎么做"，反模式和正模式一样多 | 1 | 全文 | 整篇文档只讲了"应该做双校验：自动化校验+灰度流量校验"，完全没讲反模式。比如：不要只改版本号不跑测试直接上线、不要全量切流不做灰度、不要只看接口返回200就认为没问题（业务逻辑可能错了）、不要在业务高峰期做依赖变更——这些常见错误一个都没写。光说要怎么做，不说不要怎么做，等于没写。建议补充至少5个反模式，每个说清楚后果。 | 🔴 必须改 |
| **三、边界清晰度** | 9. 明确说了不适用场景（什么情况下别用） | 1 | 第2节"适用场景" | 只写了"适用于所有第三方依赖升级"，这等于没说边界。不是所有依赖变更都需要这么重的流程：比如升级一个只在构建时用的dev依赖（比如eslint、prettier），完全没必要走灰度双校验，浪费时间。还有紧急修复线上安全漏洞的时候，可能要先快速上线再补校验。这些边界必须说清楚，不然会有人把这个流程套到所有地方，降低效率。建议明确写出："以下情况不需要走完整流程：1. 开发依赖且不影响运行时代码；2. 版本号patch级别的小修复（如1.2.3→1.2.4）且changelog确认无breaking change；3. 紧急安全修复可先上线后补校验"。 | 🔴 必须改 |
| | 11. 说了规模/阶段的边界 | 3 | 第2节 | 没说这个方法对项目规模有没有要求。如果是个人项目、内部小工具，或者还在MVP阶段的产品，用户量很少，走完整的灰度双校验流程太重了。建议补充："项目日活<1000或者还在原型验证阶段时，可以简化流程，只做自动化测试+小流量验证即可，不需要完整的双周灰度"。 | 🟡 建议改 |
| **四、可操作性** | 13. 有检查点/验收标准 | 2 | 第3节"验收标准" | 只写了"校验通过就可以上线"，"校验通过"的标准是什么？错误率低于多少算通过？性能下降超过多少算不通过？业务指标波动在什么范围内算正常？这些都没说。建议给出量化的验收标准，比如："灰度流量≥20%，观察24小时，错误率<0.1%，P99延迟上升不超过5%，核心业务指标（下单量/支付成功率）波动在±2%以内，方可全量上线"。 | 🟡 建议改 |
| **五、成熟度** | 15. 说明经过了多少验证 | 4 | 开头"背景"部分 | 说了"这个方法在我们团队5个项目中用了1年多，成功拦截了3次依赖升级导致的线上故障"，这点做得不错，可信度很高。但可以再补充一下：这3次故障具体是什么情况？如果不用双校验会有多大影响？这样更有说服力。 | 🟢 可选 |
| | 16. 有已知的局限性 | 1 | 全文 | 完全没说这个方法的缺点。任何方法都有局限性：双校验会拉长发布周期（本来一天能上线的，现在要灰度一周）、需要有完善的监控和灰度基础设施（没有监控的小团队用不了）、对于不兼容的API变更可能根本无法灰度（新旧版本不兼容没法双跑）——这些局限性必须说，不然读者用了发现不符合预期会觉得这个方法没用。 | 🔴 必须改 |

## 最后几个小提示，别不当回事

1. **打分别都打3分，要有区分度**。我见过太多评审人，所有项都打3分，然后写一句"整体不错"——这不叫评审，这叫走过场。做得好的项就打4分5分，做得差的就打1分2分，有区分度作者才知道哪块是重点要改的。

2. **意见要具体，别只说"写得不好"**。"这部分写得不好"这不叫意见，这叫废话。要写清楚：哪里不好？为什么不好？建议怎么改？你写得越具体，作者改起来越快，评审效率越高。

3. **问题位置要精确到章节/段落**。别写"文档中间部分有问题"，作者找半天找不到。精确到"第3章第2节"、"第5步那个代码块"、"开头第二段"，一找一个准。

4. **🔴必须改的项，一定要追踪到改完为止**。别评完了打个🔴就不管了，过两周发现根本没改，还是发出去了，那评审就白做了。

5. **模板不是死的，可以加项**。如果你的领域有特别需要关注的点，直接往表格里加行就行。这两份是通用模板，覆盖80%的场景，剩下20%你自己根据情况补。

就这些。下次评审别空着手去，把这个清单打开，一项一项过，评出来的质量绝对比你凭感觉看高得多。
