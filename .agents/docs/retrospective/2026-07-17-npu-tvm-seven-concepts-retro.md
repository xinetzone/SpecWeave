## 一、概述

本报告对 `SpecWeave / external/xmhub/npu_tvm` 的接入、容器化、构建打通、运行验证与 Python 3.14 兼容修复过程进行系统性复盘。

本轮工作的核心成果，不是一次性的“把 TVM 编译出来”，而是将 `npu_tvm` 从一个外部复杂仓，推进为一个具备以下特征的工程资产：

- 有稳定宿主工程承载
- 有标准化容器构建入口
- 有多角色运行面
- 有源码级兼容修复
- 有可验证的运行结果
- 有可沉淀的方法模式

最终结果可概括为：

> 本项目已完成从“外部复杂仓导入”到“可运行、可修复、可沉淀”的跨越，但距离“稳定发布级工程资产”仍差最后一轮版本口径与知识归档收口。

## 四、项目背景

`npu_tvm` 不是一个从零设计的新系统，而是建立在 TVM 主干能力之上的、带有 NPU / VTA / XM 定制痕迹的复杂工程仓。

其基本特征包括：

- 以 TVM 编译器栈为主体
- 同时包含 `src/`、`python/tvm/`、`vta/`、`web/` 等多层结构
- 构建系统以 `CMake` 为核心
- 同时面向 Python 使用、运行时部署、RPC、VTA/NPU 扩展
- 是典型的“上游大型仓 + 本地增强分支”的维护型项目

因此，本轮工作的实际难点并不只是“搭环境”，而是：

- 如何将外部复杂仓稳定接入现有主工程
- 如何让环境、源码、文档、验证形成闭环
- 如何在新版本 Python 约束下继续维持可用
- 如何避免把一次性成功误判为可复用工程资产

## 五、R（Retrospective 复盘）

### 5.1 事实回顾

本轮工作按时间顺序大致经历了以下阶段：

#### 阶段一：宿主工程治理先行

在引入 `npu_tvm` 之前，先对 `SpecWeave` 外层工程进行了文档与目录治理，包括：

- Sphinx 文档站搭建
- 根目录冗余内容迁移
- `.agents` 体系归位
- 历史目录规范化

这一阶段的意义在于：先形成承载面，再接入复杂子系统，避免外部仓导入后加剧宿主工程混乱。

#### 阶段二：外部仓接入

`npu_tvm` 被接入到 `external/xmhub/npu_tvm`，并以既有分支为基线继续工作。

这一步明确了项目边界：

- 宿主工程负责治理与承载
- 外部仓保留其大型多模块特征
- 后续工作基于“接入后工程化”，而非“重写一套”

#### 阶段三：容器化环境成型

围绕 `npu_tvm` 建立了两类入口：

- 单容器入口：`docker/build_conda.sh`、`docker/run_conda.sh`
- 多服务入口：`docker-compose.yml`、`compose.sh`

运行面角色逐渐清晰：

- `tvm-builder`
- `rpc-tracker`
- `rpc-server-cpu`
- `jupyter`

这意味着项目目标已经从“编译成功”升级为“构建、调试、交互、远程执行分工明确”。

#### 阶段四：构建治理增强

`tasks.py` 中逐步加入：

- 构建预设
- cache key
- `CMakeCache.txt` 一致性检查
- 跨环境缓存防冲突逻辑

这一步非常关键，因为它标志着项目从“命令驱动”进入“构建治理驱动”。

#### 阶段五：主链编译打通

在容器环境中完成配置与编译，生成了：

- `libtvm.so`
- `libtvm_runtime.so`

并通过 Python 加载和向量加法测试验证主链可用。

这意味着项目第一次跨过了“能编译”和“能实际调用”的边界。

#### 阶段六：兼容问题收敛

在编译与运行过程中暴露的关键问题包括：

- Python 3.14 AST 兼容问题
- LLVM API 变动
- `libbacktrace` 配置问题
- `TIRToRuntime` 属性映射问题
- named volume 与构建目录清理边界
- CRLF / LF 行尾问题

这些问题并非独立存在，而是共同揭示了本项目的真实复杂度：它是语言、工具链、容器和源码的复合系统。

#### 阶段七：源码级修复固化

最关键的阶段成果，是 AST 兼容问题没有停留在“降级 Python 版本规避”，而是：

1. 先临时降级到可运行组合，打通主链
2. 识别 Python AST 节点变更带来的根因
3. 在源码中修补兼容逻辑
4. 最终重新将 Python 升回 `3.14.6`
5. 以原子提交形式固化修复结果

这是本轮最有价值的工程突破。

## 六、F（First Principles 第一性原理）

### 6.1 本项目的第一性目标

如果剥离所有表面工作，本项目的第一性目标不是：

- 写一个 Dockerfile
- 编译一次成功
- 修一两个 bug
- 让某个开发者本地跑通

而是：

> 建立一条未来可重复、可解释、可迁移、可验证的 `npu_tvm` 工程化路径。

### 6.2 从第一性目标推导出的必要条件

若要实现这一目标，必须满足以下条件：

- 宿主工程必须具备稳定承载能力
- 构建入口必须清晰
- 运行入口必须清晰
- 验证路径必须明确
- 源码兼容问题必须能落到具体文件修复
- 文档中的版本声明必须与真实验证结果一致

### 6.3 第一性误区

本轮若只关注“版本号”，就会陷入伪目标：

- `Python 3.14` 不是成果
- `LLVM 22` 不是成果
- `Docker Compose` 不是成果

只有当这些要素共同构成一个可复用工程闭环时，它们才是成果的一部分。

因此，本轮真正的第一性判断是：

> 版本只是约束，闭环才是成果。

## 七、I（Insight 洞察）

### 7.1 洞察一：真正成功的是“工程闭环”，不是“单点可用”

本轮最重要的成功，不是某条命令不报错，而是将以下要素串成闭环：

- 宿主工程治理
- 外部仓接入
- 容器环境
- 构建治理
- 编译产物
- Python 调用验证
- 源码兼容修复
- 阶段性提交固化

这意味着项目已经从“试验性操作”升级为“可复用实践”。

### 7.2 洞察二：最有价值的技术动作是“先规避，后修复，再回升”

面对 Python 3.14 AST 兼容问题，项目没有停在“版本降级规避”，而是采用了更成熟的路线：

- 先找到一条可运行基线
- 在基线上定位根因
- 用源码修补兼容逻辑
- 最后回到目标 Python 版本

这比死守目标版本更理性，也比长期停留低版本更长期主义。

### 7.3 洞察三：构建治理代码的价值被严重低估

`tasks.py` 中的 cache key、一致性检查、构建预设等逻辑，本质上是工程系统的“防幻觉层”。

它们不直接生成模型结果，不直接生成产物，但会直接影响：

- 是否能稳定复现
- 是否能快速定位问题
- 是否能避免跨环境缓存污染
- 是否能减少“改一处坏三处”的误判

### 7.4 洞察四：多服务入口不是复杂化，而是角色边界显性化

对 `TVM + RPC + Jupyter + VTA/NPU` 这类项目而言，把所有能力塞进一个容器入口看似简单，实际上是在制造隐性复杂性。

本轮通过 `docker-compose.yml` 将角色拆开后，复杂度不是增加了，而是显性化了。

显性化复杂度比隐性耦合更可控。

### 7.5 洞察五：当前最大风险不是技术不可行，而是认知不一致

项目当前已能运行、能编译、能修补，但仍存在一个重大隐患：

- 文档中的版本声明
- 实际验证中采用的版本
- 团队口头共识中的版本

三者尚未完全统一。

这意味着当前最大的工程风险已经不是“做不到”，而是“团队以为自己已经统一了，但其实没有”。

## 八、E（Extraction 萃取）

### 8.1 模式一：外层承载先行模式

**适用场景**

- 外部复杂仓接入主工程
- 主工程本身尚在演进
- 接入后需要沉淀为长期资产

**核心步骤**

1. 先治理宿主工程目录与文档
2. 再引入外部仓
3. 再搭构建环境
4. 再做源码兼容
5. 最后做正式归档与冻结

**反模式**

- 直接 clone 外部仓后立刻 build
- 文档、脚本、知识记录和产物四处散落
- 完成后无法判断哪些是宿主变化、哪些是外部仓变化

### 8.2 模式二：先打通主链，再抬版本模式

**适用场景**

- 目标版本激进
- 依赖链长
- 问题来源不明确

**核心步骤**

1. 找到最短可运行组合
2. 建立主链成功基线
3. 基于基线定位高版本兼容根因
4. 修复源码
5. 回升至目标版本
6. 重新验证

**反模式**

- 一开始就死守理想版本
- 在多层复杂系统中同时调环境、调源码、调依赖
- 最后无法判断哪个改动真正起效

### 8.3 模式三：环境层与源码层分治模式

**适用场景**

- 容器脚本与源码都在频繁变化
- 需要区分基础设施问题和业务内核问题

**核心步骤**

- 环境层只处理：
  - Dockerfile
  - Shell 脚本
  - Compose
  - 依赖安装
  - 文档口径
- 源码层只处理：
  - Python 兼容
  - Target/Codegen 逻辑
  - Runtime 属性映射
  - 编译器内部语义问题

**反模式**

- 环境适配、源码修复、文档口径一次性混合提交
- 后续既无法追责，也难以回滚

### 8.4 模式四：构建治理前置模式

**适用场景**

- 多环境切换
- 宿主机与容器共享构建目录
- 需要重复 configure / build

**核心步骤**

- 引入 cache key
- 做 `CMakeCache.txt` 路径一致性检查
- 做 preset 管理
- 避免通过粗暴删目录来“碰运气解决问题”

**反模式**

- 每次失败都 `rm -rf build`
- 在 named volume 场景下继续沿用物理目录思维
- 不追踪构建状态来源

## 九、V（Adversarial Review 对抗审查）

### 9.1 反方观点一：这轮工作只是“脚本包装”，没有真正解决问题

**审查结果：不成立**

理由：

- AST 兼容问题已进入源码层修复
- 修复已形成正式提交
- Python 版本最终回升到目标版本
- 说明不仅环境变了，源码本身也被修复了

### 9.2 反方观点二：既然版本口径未冻结，就不能说项目成功

**审查结果：部分成立**

理由：

- 从“主链打通”角度看，本轮是成功的
- 从“发布级稳定资产”角度看，本轮仍未最终完成

因此更准确的定义应是：

> 阶段性成功，而非发布态成功。

### 9.3 反方观点三：多服务入口会增加维护负担，属于过度工程

**审查结果：部分不成立**

理由：

- 对简单项目可能成立
- 对 `TVM + RPC + Jupyter + VTA/NPU` 方向不成立

因为这些角色本来就是不同运行面，强行合并只会让复杂度隐形化。

### 9.4 反方观点四：先降级再升版本会浪费时间

**审查结果：不成立**

理由：

- 没有可运行基线，无法判断高版本问题是环境问题还是源码问题
- 基线的价值在于将“系统不可用”拆解为“局部兼容问题”

### 9.5 反方观点五：板端知识资产当前价值有限，可以后面再补

**审查结果：风险很高**

理由：

- 编码损坏说明知识正在流失
- 这类知识一旦依赖单人记忆，后续恢复成本极高
- 越晚重写，越难还原原始上下文
