# 规范自举性（Specification Bootstrapping）

## 定义
规范体系具备"自举能力"——规范本身规定了如何扩展规范。体系内包含"元规范"（定义如何创建新规范的规范），使得体系能够自我演化而不破坏一致性。

## 特征

| 特征 | 说明 | 判定标准 |
|---|---|---|
| 元规范存在 | 体系内存在定义"如何扩展规范"的规范 | 存在文件规定新规范的创建格式与流程 |
| 自我遵循 | 元规范本身遵循规范格式 | 元规范文件的结构与既有规范一致 |
| 可无限扩展 | 新规范遵循元规范创建，不破坏一致性 | 扩展后体系一致性校验通过 |

## 示例

```
.agents/teams/role-auto-creation.md   ← 元规范：定义如何创建新角色文件
.agents/roles/architect.md            ← 既有规范：架构师角色
.agents/roles/developer.md            ← 既有规范：开发者角色
.agents/roles/{new-role}.md           ← 新规范：按元规范创建
```

`role-auto-creation.md` 定义了新角色文件的格式（TOML frontmatter + 三段式正文）、触发条件（4 类）、创建流程（6 步），而它本身就是一个遵循该格式的规范文件。这使得规范体系可在不破坏一致性的前提下无限扩展。

## 与元编程的类比

| 概念 | 编程领域 | 规范领域 |
|---|---|---|
| 自举 | 编译器用自身语言编写 | 规范定义自身的扩展规则 |
| 元编程 | 程序将其他程序作为数据处理 | 元规范将规范创建作为内容描述 |
| 反射 | 程序在运行时检查自身结构 | 规范体系在扩展时引用自身规则 |

## 价值

1. **可持续演化**：体系可通过元规范无限扩展，无需外部干预即可保持一致性
2. **降低扩展成本**：新规范的创建有明确流程可循，无需重新设计结构
3. **一致性保障**：元规范强制新规范遵循既有格式，防止体系碎片化

## 推广建议

任何需要长期维护与扩展的规范体系，都应包含元规范。元规范须定义：
- 新规范的文件格式（frontmatter、正文结构）
- 创建的触发条件（何时需要新规范）
- 创建的执行流程（如何创建新规范）
- 索引同步要求（新规范如何被发现）

## 判定体系是否具备自举性

- [ ] 体系内是否存在"如何扩展规范"的元规范？
- [ ] 元规范本身是否遵循规范格式？
- [ ] 元规范是否定义了触发条件，防止无序扩展？
- [ ] 元规范是否要求索引同步，确保新规范可被发现？
