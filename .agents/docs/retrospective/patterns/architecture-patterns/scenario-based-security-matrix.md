# 场景化安全矩阵（Scenario-based Security Matrix）

> 安全不是一刀切的开关，而是按典型使用场景配置差异化的安全特性组合矩阵。

## 问题场景

设计安全系统时容易犯两种对立的错误：

1. **安全过度**：对所有场景施加统一的最高安全等级（每次操作都双重验证、每步都需人工确认），导致用户体验极差，用户寻找绕过方法反而降低真实安全性
2. **安全不足**：为了用户体验简化安全机制（无验证、无审计、无感知），导致高风险场景防护薄弱

这两种错误的根源都是**将安全视为单一全局设置**，而非根据场景差异化配置。

## 解决方案

构建**场景×安全特性**矩阵：

```
                │ 特性A  │ 特性B  │ 特性C  │ 特性D  │ ...
────────────────┼────────┼────────┼────────┼────────┼────
场景1（高风险）  │  ●强制  │  ●强制  │  ●强制  │  ○可选  │
场景2（中风险）  │  ●强制  │  ○可选  │  ○可选  │  —关闭  │
场景3（低风险）  │  ○可选  │  —关闭  │  —关闭  │  —关闭  │
场景4（企业级）  │  □策略  │  □策略  │  ●强制  │  ●强制  │
```

**符号说明**：●强制开启 / ○可选（用户可配置）/ □策略下发（管理员统一配置）/ —关闭

### 核心步骤

1. **识别典型场景**：列出用户使用产品的2-5个核心场景，按风险等级或用户类型划分
2. **梳理安全特性清单**：枚举所有可用的安全机制（认证、授权、加密、审计、提醒、隐私保护等）
3. **逐场景匹配特性**：为每个场景选择"必须有/可选/关闭/策略统一配置"的特性组合
4. **识别交叉共性**：找出所有场景都需要的基础安全特性（如加密传输）作为基线
5. **设计场景识别机制**：确定系统如何自动判断当前场景（设备绑定/IP位置/操作类型/企业策略等）

### 四象限映射方法

| | 高敏感度操作 | 低敏感度操作 |
|---|---|---|
| **高风险环境**（陌生设备/异地/首次） | 双重验证+人工确认+审计+提醒 | 简化验证+行为监控 |
| **低风险环境**（常用设备/可信网络） | 单次验证+操作日志 | 免验证+异常触发 |

## 验证案例

### 案例1：向日葵远程控制（来源：2026-07-04安全产品Wiki学习）

向日葵的安全设计是典型的场景化安全矩阵：

| 安全特性 | 接受他人远控 | 远控自己设备 | 企业管理 |
|---------|------------|------------|---------|
| 双重验证 | ●强制 | ○可选 | ●强制（SSO） |
| 白名单 | ●强制 | —关闭 | □策略下发 |
| 隐私屏 | ●强制 | —关闭 | □策略下发 |
| 水印 | ○可选 | —关闭 | ●强制 |
| 权限细粒度 | ●强制 | ○简化 | ●强制（分级） |
| 会话审计/录像 | ○可选 | —关闭 | ●强制 |
| 可疑检测 | ●强制 | ●强制 | ●强制 |
| 设备绑定 | —关闭 | ●强制 | □策略下发 |
| 异地登录提醒 | ○可选 | ●强制 | ●强制 |
| 零信任架构 | —关闭 | —关闭 | ●强制 |
| 数据脱敏 | —关闭 | —关闭 | ●强制 |

**关键洞察**：接受远控场景核心是"防他人"（8项特性），远控自己设备核心是"防丢防盗"（7项，侧重设备绑定/异地提醒），企业场景核心是"管内部"（6项，侧重零信任/分级权限/审计追溯）。三个场景特性有重叠但各有侧重，避免了一刀切。

## 适用场景

- **SaaS/云服务**：区分个人/团队/企业客户的安全需求
- **AI Agent系统**：区分只读操作/常规写入/敏感操作/高危操作的安全响应
- **权限管理系统**：区分不同角色、不同环境、不同操作类型的访问控制
- **远程访问/协作工具**：区分不同参与者、不同网络环境的安全策略
- **金融/支付系统**：区分查询/小额交易/大额转账/管理操作的验证级别
- **IoT/智能硬件**：区分本地操作/远程操作/授权访客的访问权限

## 与其他模式的关系

| 模式 | 关系 | 区别 |
|------|------|------|
| full-process-defense-depth | 互补 | 本模式是**场景维度**（横向切面），纵深防御是**时间维度**（事前/事中/事后纵向分层）。两者交叉形成三维防护矩阵：场景×时间×特性 |
| non-intrusive-security-ux | 协同 | 场景化矩阵决定每个场景"有哪些安全特性"，安全不打扰UX决定"这些特性如何以最低打扰的方式呈现"。场景识别是风险评分的核心输入之一 |
| user-sovereignty-default | 协同 | 用户主权是所有场景的底层原则：被控方/被代理方始终拥有否决权，与具体场景配置叠加使用 |
| fine-grained-least-privilege | 互补 | 最小权限是授权原则（默认给最小权限），场景化矩阵是权限分配方法（不同场景给不同权限） |

## 反模式

❌ **统一安全等级**：所有用户所有场景都使用相同的安全策略，要么安全过度要么不足

❌ **二元安全开关**：只有"安全模式开/关"两种状态，无法适配中间场景

❌ **场景识别依赖人工选择**：让用户手动选择"我现在是高风险/低风险场景"，用户不会正确选择，应该自动判断

❌ **场景爆炸**：划分过多细分场景（>5个）导致矩阵维护成本过高、配置复杂、易出错

❌ **企业场景与个人场景混为一谈**：直接将企业级安全特性套用到个人用户，或反之，忽略两类用户本质差异

## 实践要点

1. **场景数量控制在2-5个**：超过5个场景会显著增加复杂度，通常3个（低/中/高风险或个人/团队/企业）足够
2. **场景识别自动化**：基于设备指纹、IP地址、行为模式、登录历史等自动判断场景，不依赖用户手动选择
3. **设置安全基线**：所有场景共享最低安全基线（如TLS加密传输），差异化特性在基线之上叠加
4. **场景可迁移**：用户从低风险环境切换到高风险环境时（如常用设备突然异地登录），自动升级安全等级
5. **企业场景独立设计**：企业级场景通常需要策略统一下发、审计强制、数据脱敏等特性，与个人/团队场景分开设计

## AI Agent应用启示

将场景化安全矩阵应用于AI Agent工具授权决策：

| 安全特性 | 只读查询 | 常规写入 | 敏感操作 | 高危操作 |
|---------|---------|---------|---------|---------|
| 人类确认 | —关闭 | —关闭 | ●强制 | ●二次确认 |
| 操作审计 | ●强制 | ●强制 | ●强制 | ●强制+录像 |
| 沙箱隔离 | —关闭 | ○可选 | ●强制 | ●强制+快照 |
| 权限申请 | —关闭 | ○按需 | ●明确说明用途 | ●逐项授权 |
| 可回滚 | —关闭 | ●强制 | ●强制 | ●强制+人工审批回滚 |
| 速率限制 | —关闭 | ○可选 | ●强制 | ●强制+熔断 |

这与non-intrusive-security-ux的风险评分模型结合：场景矩阵作为评分输入（什么操作+什么环境→风险等级→安全特性组合）。
