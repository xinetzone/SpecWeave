# 环境多样性设计模式（Environment Diversity Design）

## 模式类型

架构模式（设计约束/系统设计）

## 成熟度

L1 首次提炼（xuanspace 项目 5 个瓶颈的共性根因抽象）

## 适用场景

任何面向多用户、多平台、多工具链的软件项目，尤其是开源项目或 CLI 工具项目。

## 问题背景

软件开发中普遍存在"环境假设"陷阱：开发者在自己熟悉的环境中构建和测试，默认所有用户都在相同环境中运行。这种假设在以下维度都会导致问题：

| 维度 | 假设 | 现实 | xuanspace 案例 |
|------|------|------|---------------|
| 编译器 | GCC/Clang 行为一致 | MSVC 编码和警告策略不同 | 模板 MSVC C4819 错误 |
| 包管理器 | 都用 PDM | pip/uv/pdm 三足鼎立 | 多包管理器兼容需求 |
| Git 规则 | 目录语义 = Git 语义 | .gitignore 与目录命名冲突 | 子模块 external/ 被忽略 |
| 元数据格式 | YAML 足够 | 文档数量增长后 YAML 膨胀 | 内容-元数据二分法迁移 |
| 平台 | Linux 开发 = 全平台通过 | Windows/macOS 各有差异 | 跨平台 CI 构建验证 |

xuanspace 项目的 5 个关键瓶颈有一个共同根因：**缺乏"环境多样性"的系统性考虑**。每个问题单独看都是"技术细节"，但合在一起揭示了架构层面的缺陷——环境多样性没有被提升为显式的设计约束。

## 解决方案

### 核心原则

1. **环境多样性是设计约束，不是事后补丁**：在架构设计阶段就引入环境多样性作为显式约束
2. **假设显式化**：每个设计决策都应标注环境假设，并验证假设是否成立
3. **最小多样性矩阵**：为每个子系统定义最低验证矩阵（平台 × 工具链 × 配置）

### 实施步骤

1. **识别环境多样性维度**：
   - 操作系统：Windows / macOS / Linux
   - 编译器：MSVC / GCC / Clang
   - Python 版本：3.13+（最低要求）
   - 包管理器：pdm / uv / pip
   - 字符编码：UTF-8 / CP936 / Latin-1

2. **定义验证矩阵**：
   ```
   | 子系统       | OS矩阵 | 编译器矩阵 | 包管理器矩阵 | 最低验证组合 |
   |-------------|--------|-----------|-------------|-------------|
   | 项目模板     | 3      | 3         | 1           | 3           |
   | CLI 命令     | 3      | -         | 3           | 9           |
   | 文档构建     | 3      | -         | 1           | 3           |
   | CI 流水线    | 3      | 3         | 3           | 27          |
   ```

3. **CI 配置策略**：
   - 全矩阵测试（所有组合）：模板验证、发布前检查
   - 抽样测试（关键组合）：日常 PR 检查
   - 快速路径（最常用组合）：pre-commit hooks

4. **设计检查清单**：
   - [ ] 新功能是否在不同操作系统下有不同行为？如有，是否已文档化？
   - [ ] 新模板是否在 3 个平台/编译器下验证通过？
   - [ ] 路径处理是否使用 `pathlib` 而非字符串拼接？
   - [ ] 编码假设是否显式声明（UTF-8 / 系统默认）？

## 反模式

- **单一环境开发**：只在 WSL/Linux 中开发，假设其他平台"应该也能跑"
- **事后补平台支持**：先发布 Linux-only 版本，再"逐步支持" Windows/macOS
- **过度矩阵**：对所有子系统都要求全矩阵验证，导致 CI 时间爆炸
