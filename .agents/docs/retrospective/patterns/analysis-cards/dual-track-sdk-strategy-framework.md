# 双轨SDK策略识别框架

## 卡片概述

**什么是双轨SDK策略**：API/大模型平台在提供服务时，同时维护两条接入路径——
- **兼容轨**：兼容行业事实标准API（如OpenAI API格式），降低存量用户迁移门槛
- **原生轨**：平台自研原生SDK/API，提供差异化能力，建立用户粘性

**为什么要分析SDK策略**：SDK接入方式是平台生态战略的最直接体现——"用户第一行代码写什么"决定了用户迁移成本、生态开放程度、平台成熟度阶段。这是竞品分析时最容易提取也最诚实的信号之一。

**核心假设**：双轨策略的演进路径是单向的：L1→L2→L3→L4，反映平台从"借生态获客"到"建生态留客"的战略重心转移。

## 4级成熟度模型

| 阶段 | 特征信号 | 战略含义 | 典型表现 |
|------|---------|---------|---------|
| **L1 兼容优先** | 仅提供兼容API端点，无原生SDK或原生SDK隐藏很深；兼容API文档不完善，存在风格差异 | **早期探索阶段**：团队资源有限，优先验证市场需求，借成熟生态快速获客，不投入原生生态建设 | 文档只提"兼容OpenAI API"，没有自己的SDK命名；快速开始示例直接用openai库改base_url；兼容层存在各种不兼容坑 |
| **L2 兼容为主** | 兼容层是默认推荐方式，原生SDK存在但篇幅小、位置靠后；兼容API覆盖完整核心能力，原生SDK仅提供少量扩展 | **快速成长期**：已有一定用户规模，开始建设原生能力，但仍以兼容层获客为主；原生能力用于验证差异化，尚未大规模推广 | SDK列表里OpenAI兼容排在最前面，原生SDK在"更多接入方式"折叠区；快速开始示例用兼容API，原生示例作为补充；原生SDK功能比兼容层少 |
| **L3 双轨并行** | 兼容层与原生SDK同等篇幅、并列展示；兼容层覆盖核心推理能力，原生SDK提供平台独有扩展能力；文档明确说明两套方案的适用场景 | **成熟阶段**：用户规模和差异化能力都已成熟，双轨并行服务不同用户群体——新用户用兼容层快速迁移，老用户用原生能力挖掘平台价值；"借船出海+差异化留客"战略成型 | 方舟当前处于此阶段：5种接入方式（OpenAI兼容+Python/Java/Go/Node.js原生SDK）并列展示；核心API两套都支持，MCP/批处理/缓存等高级能力只有原生SDK有；快速开始同时提供两套代码示例 |
| **L4 原生为主** | 原生SDK是默认推荐，兼容层仅作为"迁移指南"存在；平台建立了自己的生态标准（如工具链、插件市场、认证体系）；兼容层不再更新新功能 | **生态完善阶段**：平台已成为事实标准或有足够强的差异化，用户主动选择原生能力；兼容层仅服务于存量迁移用户，不再作为获客主力；开始建立自有生态壁垒 | 首页快速开始只用原生SDK；兼容API放在"迁移指南"章节，明确说明"新用户建议使用原生SDK"；有自己的插件市场/工具生态，不依赖第三方标准 |

## 6项信号提取清单

分析SDK策略时，按以下6项逐一提取信号：

| 序号 | 信号项 | 提取位置 | 观察要点 |
|------|--------|---------|---------|
| 1 | **SDK章节顺序** | 快速开始页、SDK文档首页导航 | 哪种SDK排在第一个？是OpenAI兼容还是原生SDK？顺序直接反映推荐优先级 |
| 2 | **快速开始示例** | 5分钟快速开始、Hello World示例 | 示例代码默认用的是兼容API还是原生SDK？第一印象决定了大部分新用户的选择 |
| 3 | **功能覆盖对比** | API参考文档、功能列表 | 核心功能（如Chat Completion）是否两套都支持？高级功能（如批处理、缓存、MCP、微调）是否只在原生SDK有？功能覆盖差异越大，双轨战略越明显 |
| 4 | **迁移指南** | 文档目录、帮助中心 | 是否有"从OpenAI迁移"的专门指南？迁移指南是放在显眼位置还是深藏不露？迁移指南的详细程度反映对兼容获客的投入度 |
| 5 | **术语使用** | 所有文档、示例代码、错误信息 | 文档中用"兼容OpenAI"还是"类OpenAI格式"？错误码、参数命名是沿用OpenAI还是有自己的命名体系？术语选择反映心理认同度 |
| 6 | **更新日志** | Changelog、版本更新记录 | 新功能是先上原生SDK还是同时上兼容层？兼容层bug修复响应速度如何？更新优先级反映资源投入重心 |

## 使用流程

标准化使用步骤：

1. **获取文档**：找到目标产品的"快速开始"/"SDK接入"文档（不要看营销页，要看实际给开发者看的接入文档）
2. **提取6项信号**：按上述清单逐一观察并记录，不要先入为主
3. **对照4级模型定位阶段**：根据6项信号综合判断属于L1-L4哪个阶段（注意：可能存在过渡状态，如L2后期向L3过渡）
4. **解读战略含义**：
   - L1/L2：平台重心在获客，差异化能力还在建设中，生态开放度高，锁定风险低
   - L3：平台进入收获期，既有获客入口又有留客能力，是最平衡的阶段
   - L4：平台已建立生态壁垒，锁定风险高，迁移成本大，但功能体验可能最好
5. **辅助决策**：根据战略判断给出选型建议（如：早期平台选型优先选L3以上的，避免L4锁定；快速原型验证可以选L1/L2的兼容层快速上手）

## 应用案例：方舟平台L2-L3阶段判定

### 信号提取

| 信号项 | 观察结果 |
|--------|---------|
| SDK章节顺序 | OpenAI兼容、Python、Java、Go、Node.js五种方式并列展示，无明显先后 |
| 快速开始示例 | 同时提供curl示例（通用）、OpenAI兼容Python示例、原生Python SDK示例，用户可自行选择 |
| 功能覆盖对比 | 核心Chat/Embedding/Responses API两套都支持；批处理、上下文缓存、MCP服务、推理接入点等高级能力仅原生支持 |
| 迁移指南 | 有专门的"OpenAI兼容指南"章节，详细说明接口差异和迁移方法，位置较显眼 |
| 术语使用 | 明确称"OpenAI兼容接口"，参数命名基本与OpenAI一致，部分扩展参数加前缀区分 |
| 更新日志 | 新API（如/v3/responses）同步更新兼容层和原生SDK，两者更新节奏基本一致 |

### 判定结论

方舟平台处于**L2后期向L3过渡阶段**，核心证据：
- 并列展示+快速开始双示例 → 已达到L3的双轨并行标准
- 高级能力仅原生支持 → L3的差异化留客特征明显
- 迁移指南详细且位置显眼 → 仍重视兼容获客，L2特征尚未完全褪去
- 综合判断：整体已进入L3双轨并行阶段，是"借船出海+差异化留客"战略的典型范本

## 升级条件

当前版本为L1（仅1次验证：火山引擎方舟），满足以下条件后升级L2：
1. 完成**5个以上**不同平台的横向验证（包括大模型平台、云API、开发者工具等不同品类）
2. 每个阶段（L1-L4）至少有**1-2个**典型案例支撑
3. 补充反模式案例（如"名义上双轨但兼容层实际不可用"的伪双轨情况）
4. 信号提取清单扩充到8-10项，增加定量指标（如"原生SDK代码示例行数占比"等）
5. 增加不同品类产品的SDK策略差异（如：IaaS平台vs SaaS平台vs大模型平台的双轨策略差异）
