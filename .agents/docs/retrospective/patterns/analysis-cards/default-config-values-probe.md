# 默认配置价值观探针

## 卡片概述

**核心原理**：默认配置是产品价值观的"诚实探针"——
- **功能列表可以堆砌**：产品宣传时可以说"支持XX功能"、"支持YY模式"，把所有能力都列出来
- **但默认配置必须取舍**：用户第一次打开产品、第一次运行示例代码时，产品团队必须选择"默认是什么"
- **取舍就是投票**：每一个默认选择的背后，都是资源投入优先级、能力信心、目标用户假设、trade-off权重的真实投票
- **不会撒谎的信号**：营销话术可以美化，但默认参数不会撒谎——"默认开启深度思考"比100句"我们追求高质量"都更有说服力

**为什么这是跨品类通用框架**：从大模型API到SaaS软件，从操作系统到消费App，从开发者工具到手机App，只要有"默认设置"的产品都适用。这是极少数能跨ToB/ToC/API/软件品类通用的分析框架。

---

## 三层次分析框架

按深度分为三个层次，逐层深入解读：

| 层次 | 分析对象 | 关键问题 | 可解读信息 |
|------|---------|---------|-----------|
| **L1 默认选择** | 默认选中/默认推荐的选项 | 默认选了什么？没选什么？哪些选项被放在最显眼位置？哪些藏在高级设置里？ | 资源投入优先级、能力信心投票、主推产品/功能、商业化优先级 |
| **L2 默认参数** | 具体参数的默认数值（温度、超时、重试、并发等） | 这个数值是偏A方向还是偏B方向？这个数值对哪类用户友好？trade-off是怎么做的？ | 目标用户画像、核心场景假设、产品价值观（质量vs速度vs成本）、风险偏好 |
| **L3 默认关闭** | 默认隐藏/默认关闭/需要手动开启的功能 | 哪些功能默认不开启？为什么要用户手动开？是因为风险高？还是因为不是主流场景？还是因为这是付费增值功能？ | 风险控制策略、用户分层逻辑、产品边界感、增值服务设计 |

## 信号解读示例库

以下是从实际产品中总结的信号-解读对，持续扩充：

| 观察到的默认配置 | 对应价值观/战略解读 | 典型产品案例 |
|-----------------|-------------------|-------------|
| 大模型平台默认模型是自研模型，而非第三方开源模型 | **主推自研**：自研模型是核心竞争力和利润中心，资源投入重点，对自研能力有信心 | 方舟默认用doubao-pro-32k |
| 大模型平台默认模型是开源/第三方模型 | **平台定位**：自研能力弱或非核心，主打模型聚合/平台能力/成本优势 | 部分聚合类API平台 |
| 默认temperature=0.1-0.3（低随机性） | **任务型场景假设**：面向代码生成、结构化输出、事实性问答、企业级应用等确定性要求高的场景；目标用户是企业开发者 | 代码专用模型、企业级API |
| 默认temperature=0.7-1.0（中高随机性） | **通用对话场景假设**：面向创意写作、多轮对话、开放性问题、C端聊天等场景；目标用户是通用开发者/C端用户 | 通用对话模型、C端聊天产品 |
| 深度思考/高级推理/长思考等高级功能**默认开启** | **质量优先价值观**：愿意为结果质量牺牲速度和成本，相信"用户拿到好结果比快速拿到一般结果更重要"；面向对质量敏感的用户 | 方舟默认开启深度思考 |
| 深度思考/高级功能**默认关闭**，需要手动开启 | **成本/速度优先**：默认提供快速廉价的基础能力，高级能力作为可选项；面向成本敏感/速度敏感的用户；或高级能力尚不成熟 | 多数大模型平台默认关闭CoT |
| API默认超时时间设置得很长（如60s+） | **质量优先/复杂任务假设**：预期用户会跑复杂任务、长文本生成，愿意等待；面向重度用户 | 长文本生成API |
| API默认超时时间设置得很短（如10-15s） | **速度优先/实时交互假设**：预期用户做实时对话、简单查询，不能接受等待；面向实时交互场景 | 实时聊天API、C端产品 |
| 默认开启自动重试 | **用户友好/可靠性优先**：替用户处理网络波动等异常，减少用户代码复杂度；相信重试能解决大部分问题 | 多数成熟云SDK |
| 默认关闭自动重试，需要用户手动配置 | **可控性优先**：把重试策略交给用户，避免重复提交等副作用；面向高级开发者/对一致性要求高的场景 | 金融支付类API |
| 内容审核/安全过滤**默认开启**且无法关闭 | **风险厌恶/合规优先**：面向大众用户、C端场景、企业合规要求高的场景；宁愿错杀也不放过风险 | C端聊天产品、公有云API |
| 内容审核/安全过滤**默认关闭**，需要用户手动开启 | **面向开发者/灵活优先**：假设用户是开发者，会自己做内容审核；或审核能力尚不成熟；或面向内部/私有部署场景 | 早期大模型API、私有部署版本 |
| 新用户注册默认送免费额度 | **PLG增长驱动**：降低试用门槛，让用户先用起来再谈付费；产品体验好到能留住用户 | 成熟SaaS、API平台 |
| 新用户注册必须绑定信用卡/充值才能用 | **销售驱动/企业客户**：目标客户是付费企业，不服务免费薅羊毛用户；产品价值高不担心转化 | 传统企业软件、高价API |
| 高级功能/高用量在文档中提示"联系商务" | **销售分层**：自助服务中小客户，大客户走商务洽谈；产品有足够的差异化价值让大客户愿意谈 | 企业级云服务 |
| 所有功能/用量明码标价自助购买 | **产品驱动增长**：相信产品本身能转化，不依赖销售介入；面向开发者和中小客户 | 多数PLG驱动的SaaS |
| 手机操作系统默认安装自家全家桶App | **生态闭环/流量变现优先**：优先推广自家服务，构建生态壁垒；流量在自己手里效率最高 | 多数手机厂商系统 |
| 手机操作系统默认不预装第三方App | **用户体验优先/合规要求**：尊重用户选择权，或因反垄断合规要求；但系统关键位置仍会留自家服务入口 | iOS早期、合规后的Android |
| 代码编辑器默认开启语法检查/自动格式化 | **新手友好/规范化优先**：降低新手门槛，统一代码风格；面向团队协作/教育场景 | VS Code默认配置 |
| 代码编辑器默认关闭所有辅助功能，让用户自己配置 | **高级用户/可控性优先**：面向资深开发者，相信用户知道自己要什么；不做替用户做决定 | Vim、部分高级IDE配置 |

## 使用流程

标准化使用步骤：

1. **找到默认配置**：
   - API/SDK：看快速开始示例代码里的参数、SDK初始化默认值、"推荐配置"说明
   - SaaS/软件：看新账号首次登录的默认设置、默认勾选的选项、初始化向导默认选择
   - 消费产品：看安装后的默认状态、首次打开的默认界面、默认开启的功能
2. **按三层次逐一分析**：
   - L1：列出来所有默认选择的东西，问"为什么选A不选B？"
   - L2：记录所有具体的默认数值，问"这个数值在trade-off曲线上偏向哪一端？"
   - L3：列出来所有默认关闭/隐藏的功能，问"为什么不让用户第一次就看到？"
3. **交叉验证形成判断**：
   - 不要只看单个默认值，要看多个默认值是否指向同一个方向（如：默认自研模型+默认深度思考开启+默认temperature=0.7 → 三个信号都指向"质量优先+主推自研"，判断可信度高）
   - 如果不同默认值指向矛盾方向，分析哪个信号更强（通常：代码里的默认值 > UI上的默认勾选 > 文档里的"推荐"描述）
4. **输出画像**：总结产品的目标用户、核心场景、价值观优先级
5. **验证（可选）**：如果有条件，实际运行一下看真实默认行为是否和文档一致（有时候文档写的和实际代码不一致）

## 应用案例：方舟默认配置分析

### L1 默认选择层信号

| 默认选择 | 解读 |
|---------|------|
| 默认模型：doubao-pro-32k（自研旗舰模型） | 主推自研模型，doubao-pro是当前最核心的旗舰产品，32k上下文是主流场景的甜点配置 |
| 默认接入方式：同时提供OpenAI兼容和原生SDK示例，无强制默认 | 双轨并行战略，不强迫用户选边，给用户选择权（L3阶段的自信表现） |
| 默认API版本：/v3/responses（最新统一接口） | 主推新API标准，引导用户向统一接口迁移，而非停留在旧的Chat Completions |

### L2 默认参数层信号

| 默认参数 | 数值 | 解读 |
|---------|------|------|
| temperature | 0.7 | 通用对话场景假设，平衡创造性和确定性，既不是保守的0.1也不是激进的1.0 |
| 深度思考/推理 | 默认开启 | 质量优先价值观，相信用户更看重结果质量，愿意接受稍长的等待时间和稍高的成本 |
| 流式输出（stream） | 示例代码默认开启 | 注重用户体验，实时流式响应比等待完整结果体验更好，面向对话交互场景 |

### L3 默认关闭层信号

| 默认关闭的功能 | 解读 |
|---------------|------|
| 批处理（Batch） | 批量离线处理是高级功能，不是主流场景，默认隐藏避免新手混淆；属于成本优化进阶能力 |
| 上下文缓存（Context Caching） | 高级成本优化功能，需要用户理解缓存机制才能用好，默认关闭；面向有一定规模的用户 |
| MCP服务接入 | 前沿能力，生态尚在建设中，默认不展示在快速开始，放在进阶文档；面向Agent开发者 |

### 综合价值观画像

三个层次信号交叉验证后，方舟的产品价值观非常清晰：
- **用户**：面向有一定经验的开发者，既照顾从OpenAI迁移的用户（兼容层），也服务深度使用的用户（原生能力）
- **场景**：通用对话+复杂推理为主流场景，质量优先于速度和成本
- **优先级**：自研模型是核心，深度思考是差异化卖点，新用户体验（3步快速开始+流式输出）很重要，高级成本优化能力留给进阶用户
- **阶段**：产品成熟度高，对自研能力有信心，不需要靠"开源模型默认"吸引用户，也不需要靠"默认关闭高级功能"节省成本

## 升级条件

当前版本为L1（仅1次验证：火山引擎方舟），满足以下条件后升级L2：
1. 完成**3-5次跨品类验证**（覆盖：大模型API之外的至少3个品类，如SaaS软件、操作系统、消费App、开发者工具等）
2. 信号解读示例库扩充到**20+条**，每个品类至少3-5条典型信号
3. 补充"信号欺骗"反模式（如：默认推荐的其实是利润率最高的而非体验最好的、默认开启是为了收集数据等）
4. 增加"默认配置演进"分析维度（同一个产品不同版本的默认配置变化反映战略转型）
5. 制作"默认配置价值观雷达图"模板，可视化6个维度（质量/速度/成本/开放/可控/风险）的倾向
