# 一站式操作指南模式（One-Stop Operation Guide）

## 模式类型
方法论模式（document-architecture）

## 成熟度
L2 已验证（在 Mermaid 图表操作指南中完整验证一次）

## 适用场景
当某个高频任务/工具/流程的知识分散在多个位置（模板目录、规则文档、脚本代码、排错指南、开发规范），导致使用者需要多文件跳转才能获得完整信息时。

典型信号：
- 同一主题的规则、模板、工具用法分散在 3 个以上位置
- 新成员/新智能体经常问"这个怎么用"
- 规范存在但执行率低（规范落地"最后一公里"断裂）
- 用户反馈"知道有规范但找不到"

## 问题背景
规范落地不仅需要规则本身正确，还需要**降低规则获取成本**。当知识分散在多个文件中时，即使每个文件都正确，使用者也可能因为查找成本高而放弃查阅，导致"规范存在但未执行"的治理失效。

这是一种**结构性摩擦力**——不是规范写得不好，而是到达规范的路径太长。

## 核心思想
> 为高频任务创建一个一站式入口文档，将规则、模板、工具用法、排错流程和检查清单整合到单一文件中，将知识获取路径从"多文件跳转"压缩为"单文件直达"。

入口文档不是替代详细文档，而是作为**路线图和导航枢纽**——提供快速开始步骤、核心规则摘要、常用命令、排错决策树和检查清单，同时链接到详细参考文档供深入查阅。

## 文档结构六要素

一份合格的一站式操作指南应包含以下六个核心部分：

### 1. 快速开始（3步以内）

用最少的步骤让用户完成第一次成功操作。包含：
- 一句话说明文档用途
- 流程图/步骤图展示整体流程
- 可直接复制的起步代码/模板
- 预期结果说明

### 2. 核心规则（最常违反的 Top N）

不是完整规则列表（那放在详细文档里），而是最常被违反的 5-8 条规则，每条规则包含：
- 一句话总结
- 错误/正确示例对照
- 记忆口诀（如有）

### 3. 工具用法（常用命令）

该任务涉及的自动化工具的核心命令：
- 最常用的 3-5 个命令及参数说明
- 检查/修复/预览等不同模式的用法
- CI 集成方式

### 4. 排错流程（决策树）

遇到问题时的排查顺序，最好用 Mermaid 流程图展示：
- 按出现频率从高到低排列检查项
- 每个分支给出具体修复动作
- 形成闭环（修复→重新检查→仍有问题→下一项）

### 5. 检查清单（完成前必过）

提交/完成前的对照检查清单：
- 10项以内，可快速勾选
- 每项对应一条规则或工具检查
- 最后一项是运行自动化检查

### 6. 参考索引（链接不重复）

链接到更详细的参考文档：
- 不重复详细文档的内容
- 按用途分类（模板、详细规则、陷阱表、工具源码、CI脚本）

## 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 入口文档成为详细文档的完整副本 | 内容重复，维护时容易不一致 | 只放摘要+链接，详细内容保留在原文档 |
| 缺少快速开始章节，直接进入详细规则 | 新用户不知道从何开始 | 3步快速开始必须放在最前面 |
| 没有排错流程 | 遇到问题时用户仍需到处搜索 | 用Mermaid决策树展示排查路径 |
| 只描述"应该怎么做"，不给示例 | 用户无法判断自己写的是否正确 | 每个规则配正反例 |
| 检查清单太长（超过15项） | 用户跳过不看 | 控制在10项以内，覆盖最高频问题 |

## 成功案例

| 文档 | 整合前分布 | 整合后效果 |
|------|----------|---------|
| [mermaid-guide.md](../../../../knowledge/best-practices/mermaid-guide.md) | 模板(8文件)、规则模式(2文件)、检查脚本(1文件)、CI脚本(2文件)、开发规范(1文件)共14个文件 | 单文件含快速开始+六规则+10项检测能力表+6步排查图+提交前清单，7个参考链接 |

## 与其他模式的关系

- **entry-container-separation**：本模式是入口-容器分离原则的正向应用——入口文档精简且导航性强，详细内容留在容器文档中
- **three-layer-rule-enforcement**：一站式入口降低了第一道防线（源头预防）的摩擦力
- **mermaid-safe-coding-rules**：六规则是操作指南中"核心规则"章节的内容来源
- **safe-starter 模板**：是操作指南中"快速开始"章节的可复制素材
