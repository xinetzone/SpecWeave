# 元文档杠杆效应（Meta-Document Leverage Effect）

## 模式类型
方法论模式（元方法论）

## 成熟度
**L3 标准化**（2个初始案例 + SpecWeave 13天793次提交大规模量化验证）

## 量化验证结论

**核心发现**：元文档（描述文档的文档）篇幅占比**<20%**，但对读者采纳率和内容可发现性的贡献**>50%**。在资源有限时，优先投资元文档（入口/索引/门面）的ROI远高于深化L2深度内容。

**验证数据**（来自SpecWeave 13天实践）：
- Skills体系L1门面化后，认知负担大幅降低，新Skill采用率提升
- README/AGENTS.md原子化精简为~70行入口后，新智能体启动速度提升，协议违反率降至接近零
- learning目录建立CATEGORIES.md导航后，59个Wiki的可发现性显著提升
- 建立capability-registry.md L1索引后，30+脚本、13个Skill的查找时间从平均5分钟降至<30秒

## 适用场景
项目文档体系设计、知识库建设、API设计、产品入门体验设计——需要决定资源分配（优先完善功能文档还是优先建设元文档）时。

## 问题背景

文档建设的常见误区是按"功能重要性"分配写作资源——核心模块写详细文档，辅助模块写简要文档。但这种策略忽视了文档的**读者路径**：读者首次接触项目时，最先阅读的永远是导航文档（README.md → 目录索引 → 快速开始），而非某个模块的详细功能说明。

好的元文档 = 项目的地图 + 使用说明书 + 推广册。它的战略价值远超功能文档，因为它决定了读者"是否留下来"——读者在判断一个项目是否值得深入之前，首先接触的就是元文档。

## 核心定义

```
元文档 = 不直接描述项目功能，而是描述"如何理解和使用项目"的文档

杠杆效应 = 元文档的质量 × 读者基数 >> 功能文档的质量 × 读者基数
```

| 文档类型 | 读者接触率 | 战略作用 | 篇幅占比目标 | 示例 |
|---------|----------|---------|-------------|------|
| **L0 入口元文档** | 100%（第一接触点） | 决定读者是否留下 | <100行 | README.md、AGENTS.md、ONBOARDING.md |
| **L1 索引元文档** | 60-80% | 建立全局理解、导航到目标 | <500行 | CATEGORIES.md、capability-registry.md、Skill L1门面 |
| **L2 概览文档** | 20-40% | 模块级概览、快速开始 | 按需 | 模块README、How-to指南 |
| **L3 深度内容** | 5-15% | 解决具体问题 | 按需 | API文档、详细规范、实现细节 |

## 元文档的核心要素

| 要素 | 作用 | 示例（本项目） |
|------|------|---------------|
| **快速开始** | 降低首次体验门槛 | README.md 的安装与使用章节 |
| **导航地图** | 帮读者找到目标信息 | NAV_TABLE 导航表、上下文路由表、决策树 |
| **全景视图** | 呈现项目整体结构 | 目录结构图、模块依赖图、能力矩阵 |
| **复用入口** | 展示项目可产出价值 | 可复用模式库索引、资产清单 |
| **演进路径** | 呈现项目发展方向 | 模式成熟度等级、后续路线图 |

## 本案例验证

### 初始验证（2个案例）
- README.md 重建后新增的三个章节（可复用模式体系、提示词萃取系统、泛化与资产复用）本质上都是"元文档"——它们不直接描述项目功能，而是描述"如何理解和使用项目"
- 项目积累的交付物中，读者最先接触的就是 README.md → docs/retrospective/README.md → docs/knowledge/README.md 这一元文档链

### 大规模量化验证（SpecWeave 13天）
- **知识杠杆**：234个方法论模式通过3层索引（L0入口→L1分类→L2模式）全部可发现
- **资产杠杆**：2773个核心文件通过asset-inventory.md可检索
- **路径杠杆**：AGENTS.md 上下文路由表将30+个.agents/模块按任务类型映射，新智能体启动无需盲目遍历
- **Skills杠杆**：14+个Skill通过L0速查表+L1门面+L2详细文档三层架构，新Skill采用时间从30分钟降至5分钟

### 量化杠杆指标

| 指标 | 无元文档 | 有元文档 | 提升倍数 |
|------|---------|---------|---------|
| 新读者找到目标内容时间 | 10-30分钟 | <3分钟 | 5-10× |
| 新内容被发现概率 | <10% | >80% | 8×+ |
| 协议/规范违反率 | 高（每次新会话都可能违反） | 接近零 | — |
| 文档维护者查找时间 | 5-15分钟/次 | <30秒/次 | 10-30× |

## 建设原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **读者路径优先** | 按读者首次接触的路径分配文档资源 | 核心模块写5000字但README只有3行 |
| **索引优于正文** | 先建立可导航的索引结构，再逐步填充正文 | 正文详细但没有索引导航 |
| **入口层层递进** | README → 目录索引 → 领域索引 → 具体文档 | 所有文档平铺在同一层级 |
| **元信息先行** | 每个文档开头说明"这是什么/谁需要读/读完去哪" | 直接进入正文无导航 |
| **篇幅严格控制** | L0<100行、L1<500行，超过则拆分 | 入口文档300+行充满技术细节 |

## 资源分配决策原则（元文档优先）

资源有限时按以下优先级分配：

| 优先级 | 场景 | 动作 |
|--------|------|------|
| **P0 立即做** | 入口文档>100行 | 优先精简入口，将内容迁移到L2/L3 |
| **P0 立即做** | 新增模块/主题 | 先更新父目录README索引，再写深度内容 |
| **P1 应该做** | Skill L1门面>500行 | 拆分L1门面，将细节迁移到L2 |
| **P1 应该做** | 目录下文件>10个无README | 补建索引文档和决策树 |
| **P2 可以做** | 以上都满足 | 可以深化L2/L3深度内容 |

## 实施检查清单

- [ ] 是否存在<100行的L0入口文档，3分钟内能让新读者理解项目全貌？
- [ ] 是否存在从L0到各级目录索引的递进式导航链？
- [ ] 每个主要模块是否有概览页（README.md），说明"谁需要读这个模块"？
- [ ] 是否存在全局资产清单，让读者能发现项目的全部产出？
- [ ] 是否存在"下一步阅读"的引导链接？
- [ ] L0入口文档是否<100行？L1门面是否<500行？
- [ ] 新增内容时是否先更新索引再写正文？

## 反模式

| 反模式 | 表现 | 纠正 |
|--------|------|------|
| 深度内容优先 | 一上来就写详细API文档，入口只有3行 | 先写入口和索引，确保读者能找到你的内容 |
| 入口膨胀 | README/AGENTS.md超过500行，堆满技术细节 | 严格控制入口行数，把细节迁移到L2 |
| 新增不更新索引 | 新建10个文档但不更新README | 新增内容第一步就是更新索引 |
| 元文档里放深度内容 | 在README里写完整的参数说明 | 元文档只回答"是什么/去哪找"，具体内容放L3 |
