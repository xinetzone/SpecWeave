# 认知闭环划分原则（Cognitive Closure Document Split）

## 模式类型

方法论模式（文档架构/文档工程）

## 成熟度

L2 二次验证（MDI复盘6文件合并验证 + 竹简悟道洞察库四层结构验证）

## 适用场景

技术文档拆分、复盘报告组织、知识库原子化、论文/报告结构设计；当你需要判断"哪些内容应该放在同一个文件"时。

## 问题背景

文档原子化的核心难题：边界在哪里？
- 代码可以按"功能职责"清晰拆分（一个函数做一件事，一个类负一个责）
- 文档的"单一职责"边界非常模糊：按主题分？按阶段分？按读者分？
- 错误的拆分方式（机械按主题/阶段拆分）会导致：跨文件重复内容、读者需要在多个文件间跳转才能理解一个完整主题、维护困难

## 核心思想

**代码按"功能职责"拆分，文档按"读者认知闭环"拆分——"事实→分析→洞察→结论→行动"构成一个完整思考单元，不应强行按阶段/主题拆分。**

### 认知闭环单元

一个完整的认知闭环包含五个环节：

```mermaid
graph LR
    F["事实(Facts)"] --> A["分析(Analysis)"]
    A --> I["洞察(Insights)"]
    I --> C["结论(Conclusions)"]
    C --> AC["行动(Actions)"]
    style F fill:#e1f5fe
    style A fill:#fff9c4
    style I fill:#c8e6c9
    style C fill:#ffccbc
    style AC fill:#d1c4e9
```

当读者阅读一个主题时，自然期望经历完整的思考链条——看完事实想知道分析，看完分析想知道结论，看完结论想知道怎么做。强行把这些环节拆分到不同文件，会破坏阅读流，制造重复（每个文件都需要重新交代上下文）。

### 三层/四层文档结构

遵循认知闭环原则，文档体系自然收敛为分层结构：

**基础三层（MDI复盘验证）**：

| 层级 | 文件类型 | 职责 | 闭环要求 |
|------|---------|------|---------|
| L1 索引层 | README/目录 | 导航、文件清单、阅读顺序建议 | 自身是完整闭环（告诉读者怎么读） |
| L2 事实层 | 事实数据、客观记录、过程日志 | 只陈述客观事实，不做分析判断 | 每个事实文件是独立的事实单元 |
| L3 思考层 | 完整复盘/洞察报告/建议文档 | 整合事实→分析→洞察→结论→行动 | **必须包含完整认知闭环** |

**扩展四层（竹简悟道验证）**：

当知识体系本身有清晰的抽象层级时（如产品→架构→哲学→元认知），三层可扩展为N层：

| 层级 | 竹简悟道实例 | 读者对象 | 变更频率 |
|------|------------|---------|---------|
| 产品层 | insights-01-15 | 产品经理、设计师 | 高（随功能迭代） |
| 架构层 | insights-16-30 | 架构师、工程师 | 中（随架构演进） |
| 哲学层 | insights-31-53 | 方法论研究者 | 低（核心思想稳定） |
| 元层 | insights-50/54-68 | 团队自身 | 中（随复盘更新） |

分层原则：不同层级的读者不同、变更频率不同、抽象层次不同——混在一起会导致"不该改的人改了不该看的内容"。

## 核心规则

### 规则1：L3思考层文件必须包含完整闭环

一个完整的思考类文档必须包含五个要素，缺一不可：

1. **事实/背景**：支撑分析的客观数据和过程描述（可以引用L2事实文件，但要在文内提供足够上下文）
2. **分析**：对事实的解读、趋势识别、异常发现
3. **洞察/模式**：从分析中提炼的可复用经验
4. **结论**：整体判断、核心发现总结
5. **行动**：基于结论的具体可执行建议

如果缺少任何一环，读者就会产生"然后呢？""所以呢？""怎么做？"的疑问。

### 规则2：L2事实文件只陈述事实，不做判断

事实层文件的职责是客观记录：
- 发生了什么
- 量化数据
- 时间线
- 关键事件

不要在事实文件中加入分析、结论或建议——那些属于L3思考层。

### 规则3：识别伪拆分（应该合并的信号）

当出现以下信号时，说明拆分是错误的，应该合并：
- 多个文件开头需要大段重复的背景介绍
- 读者需要同时打开3个以上文件才能理解一个主题
- 同一个洞察/结论在多个文件中重复叙述
- "下一章"链接的内容是理解当前章节的必要前提（不是可选参考）
- 新增内容不知道该放A文件还是B文件，边界模糊

### 规则4：合并时保留导航，消除重复

合并不是简单的文本拼接：
- 消除重复的背景介绍，只保留一份
- 统一术语和表述
- 调整章节顺序，形成线性阅读流
- 更新导航链接，确保跳转到正确位置（可以使用锚点链接到具体章节）

### 规则5：逻辑边界优先于数字编号/机械切分

不要按"每N个章节/洞察一个文件"或"每M行切一刀"机械拆分。

**拆分前先做「主题组块识别」**：通读文件，识别天然的主题簇（通常由以下信号标记）：
- "---"分隔符划分的大段落
- 标题层级的主题转向
- 内容视角的转换（如外部分析→内部方法论、客观描述→主观建议）
- 读者对象的切换

数字编号是线性的，知识结构是网状的——逻辑主题的边界几乎从来不与数字编号对齐。例如竹简悟道insights-59-65.md的正确拆分点在洞察62/63之间（外部视角→内部视角），而非数字中间（61/62或62/63之间随便切）。

### 规则6：跨文件榫卯衔接设计

拆分不是"切一刀就完事"。在文件切换的边界处，需要设计"榫卯结构"保证逻辑不断裂：

**显式衔接（导航层）**：每个文件头部包含统一的关联文件导航块，告诉读者"你从哪来、可以到哪去"。

**隐式衔接（逻辑层）**：当后一个文件是前一个文件的逻辑延续/修正/深化时，在后一个文件的开头或结尾专门设计承接段。典型例子：竹简悟道洞察63（定位解缚）第七节专门承接洞察62（竞争格局）的"不争之争"，指出"如果只做'问道'而排斥'解惑'，就是'避实'过度"——这个承接段让两个文件在切口处形成对话，而非生硬断裂。

**最终检验：顺序阅读测试**：拆分完成后，从第一个文件开始像新读者一样按导航顺序阅读，重点关注文件切换处——是否需要回头看前一个文件才能理解？是否有未解释的术语突然出现？如果有，说明需要补充跨文件衔接或调整拆分点。

## 实施检查清单

**拆分前**：
- [ ] 这个文件是L1索引、L2事实还是L3思考？（或属于哪一抽象层级？）
- [ ] 如果是L3思考层，是否能在一个文件内承载完整闭环？
- [ ] 拆分后每个文件是否有独立存在的价值？
- [ ] 是否已做「主题组块识别」，确定拆分点在逻辑边界而非数字编号/行数中间？
- [ ] 拆分点两侧的读者对象、视角、抽象层次是否一致？（不一致才是正确的拆分边界）

**拆分后**：
- [ ] 是否有文件大段重复其他文件的内容？
- [ ] 读者理解主题A需要同时打开几个文件？（理想：1个L3文件+必要的L2事实引用）
- [ ] 新增内容是否有清晰的归属？
- [ ] 跨文件存在逻辑递进关系时，是否设计了榫卯衔接段（显式导航+隐式承接）？
- [ ] 是否做了「顺序阅读测试」——按导航顺序阅读，文件切换处是否突兀？

**合并时**：
- [ ] 是否消除了重复的背景和上下文？
- [ ] 章节顺序是否符合"事实→分析→洞察→结论→行动"的阅读流？
- [ ] 导航链接是否更新到正确位置？

## 反例警示

| 错误拆分方式 | 为什么错 |
|------------|---------|
| 按项目阶段拆分（01分析/02洞察/03结论） | 人为切断认知闭环，每个文件都需要重复上下文 |
| 按文档长度机械拆分（每300行一个文件） | 不考虑内容逻辑，可能在一个论证中间切断 |
| 按数字编号机械拆分（每N个洞察/章节一个文件） | 数字是线性的，知识是网状的，逻辑边界几乎不与数字编号对齐 |
| 按"主题"拆分但主题边界重叠 | 新增内容无明确归属，产生重复 |
| 不同抽象层级内容混在同一文件 | 读者被无关内容干扰，变更时容易误改不该改的部分 |
| 切分后不做衔接设计 | 文件切换处逻辑断裂，读者需要回头翻找上下文 |
| 索引层包含大量分析内容 | 索引变厚，失去导航作用 |
| 事实层加入主观判断和结论 | 事实与观点混淆，可复用性降低 |
