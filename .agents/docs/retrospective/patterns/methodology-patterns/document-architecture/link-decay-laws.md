# 文档系统链接衰变四规律（Four Laws of Link Decay）

## 模式类型
领域/方法论模式

## 成熟度
L1 实验性（1次案例：14个断链修复观察+多次原子化操作统计）

## 适用场景
Markdown/HTML文档系统的目录重构、原子化拆分、文件移动、链接维护、文档工具链设计。

## 问题背景

文档原子化和目录重构会导致断链，但断链的分布不是随机的——它遵循可预测的规律。理解这些规律可以：
- 预测哪些操作最可能产生断链
- 设计自动修复算法时优先处理高频场景
- 在操作前评估影响面
- 设计工具链时重点防护高风险操作

## 四条核心规律

### 规律一：移动越深，断链越多

文件向目录树深处移动时（如原子化拆分：单文件 → 目录+index+子文件），引用它的链接断链概率与深度差成正比。

**量化**：每增加一级深度，引用该文件的所有跨目录链接都需要增加一个 `../`。

### 规律二：向浅移动影响小

文件向上移动时（如合并：目录+多文件 → 单文件），原有链接可能仍然有效——多出来的 `../` 可能仍然指向正确位置或其父目录，浏览器/Markdown解析器通常能容忍多余的 `../`。

**含义**：自动修复算法应优先尝试增加深度（N+1, N+2...）而非减少深度，因为下移是高频且高风险操作。

### 规律三：跨目录引用最脆弱

从一个顶级目录引用另一个顶级目录的链接（如 `.trae/specs/...` 引用 `docs/...`），路径中包含的 `../` 数量最多，在深度变化时最容易断裂。

**量化**：跨顶级目录引用通常需要5-6个 `../`，每级深度变化影响所有这些层级。

### 规律四：同目录内引用最稳定

同一目录下的相对链接（不含 `../`）几乎不受目录深度变化影响，是最健壮的引用方式。

## 断链风险矩阵

| 操作类型 | 断链风险 | 影响范围 | 自动修复优先级 |
|---------|---------|---------|--------------|
| 文件下移1级（原子化拆分） | **高** | 所有跨目录引用该文件的链接 | 最高优先级 |
| 跨顶级目录移动 | **极高** | 几乎所有外部引用 | 最高优先级 |
| 文件上移1级 | **低** | 通常无断链（多../可容忍） | 低优先级 |
| 同目录内移动 | **极低** | 几乎无影响 | 无需自动修复 |
| 文件重命名 | **中** | 所有引用该文件名的链接 | 中优先级（需映射表） |

## 规律的工具设计应用

| 应用场景 | 规律指导 |
|---------|---------|
| 自动修复算法（relative-depth-adjustment） | 优先尝试增加深度（N+1, N+2...），调整幅度限制±3级 |
| 修复优先级链（fix-priority-chain） | 精确路径校正（深度调整）排第一优先级 |
| 原子化一键收尾（finalize-atomization） | 原子化是下移操作，重点处理跨目录链接 |
| 影响面评估（build-ref-index） | 跨目录引用多的文件移动时需重点检查 |
| 最佳实践配套工具（best-practice-hidden-cost） | "链接税"主要由规律一和规律三产生 |

## 操作前影响评估口诀

移动文件前，用build-ref-index查一下：
- 同目录引用多 → 放心移动，风险低
- 跨目录引用多 → 准备好finalize-atomization收尾
- 跨顶级目录引用多 → 高风险操作，仔细检查
- 引用数为0 → 安全移动（但也要验证）

## 检查清单

- [ ] 执行文件移动前，是否评估了移动方向（上移/下移/跨目录）？
- [ ] 下移操作后是否运行了链接检查？
- [ ] 跨顶级目录移动是否有特别检查？
- [ ] 自动修复算法是否优先处理高频场景（增加深度）？
- [ ] 原子化操作后是否自动运行finalize-atomization？
