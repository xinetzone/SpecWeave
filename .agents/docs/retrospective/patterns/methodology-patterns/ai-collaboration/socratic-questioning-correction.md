# 苏格拉底提问纠错模式（Socratic Questioning Correction）

## 模式类型

方法论模式（AI协作/沟通策略）

## 成熟度

L1 实验性（1次验证来源：2026-07-09用户用提问引导发现链接格式错误）

## 适用场景

当发现对方（人类同事或AI）的决策/产出有问题时，选择用提问而非直接指正的方式引导自我发现。适用于：

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 方法论教学/培训 | ✅✅✅ 核心场景 | 教对方用方法论（如第一性原理）自己发现问题，而非直接给答案 |
| Code Review/质量审查 | ✅✅✅ 核心场景 | 发现代码问题时用提问引导作者自己发现，比直接说"这里错了"更有效 |
| AI协作纠错 | ✅✅✅ 核心场景 | 发现AI产出错误时，用提问启动其第一性原理思考，而非直接给正确答案 |
| 团队沟通/反馈 | ✅✅ 强烈推荐 | 给同事/下属反馈时，避免防御心理，引导自我反思 |
| 复盘/根因分析 | ✅✅ 推荐 | 用连续追问（5-Whys+提问）引导找到真正根因 |
| 紧急线上故障修复 | ⚠️ 谨慎使用 | 紧急情况需要直接指令，不要绕弯子提问 |
| 事实性错误（如错别字、数字错） | ❌ 不适用 | 明显的事实错误直接指出即可，不需要提问绕弯 |

## 问题背景

纠错是协作中的高频场景，但**直接指正往往效果不佳**：

1. **触发防御心理**：直接说"你错了"容易让对方进入辩解模式，反而听不进正确意见
2. **只记住结论，没记住推理过程**：直接给出答案，对方只记住"应该是X"，下次遇到类似问题还是会错
3. **没有启动方法论思考**：直接纠错相当于替对方做了思考，对方没有练习用方法论自己分析
4. **记忆不深刻**：别人指出的错误远不如自己发现的错误印象深刻

苏格拉底提问法通过**不直接给答案，而是引导对方自己推导**来解决这些问题。

### 两种纠错方式对比

| 维度 | 直接指正 | 苏格拉底提问 |
|------|---------|-------------|
| 典型话术 | "你错了，应该用相对路径" | "这个判断哪里来的？符合第一性原理吗？" |
| 心理反应 | 可能触发防御、辩解 | 无防御，主动进入思考 |
| 短期效果 | 立即修正当前错误 | 自我发现并修正 |
| 长期效果 | 容易重复犯错（只记结论） | 深刻记住方法论（自己推导结论） |
| 学习价值 | 低（鱼） | 高（渔） |
| 适用场景 | 紧急修复、明显事实错误 | 方法论教学、能力培养、根因分析 |

## 核心规则

### 规则1：提问的目标是"启动思考"，不是"考倒对方"

提问不是为了证明对方错了、显示你高明，而是为了：
- 打断对方当前的直觉/类比思考路径
- 引导对方回到基本事实和方法论
- 让对方自己发现逻辑漏洞

❌ 错误提问（带攻击性）："你连这个规范都没看？"
✅ 正确提问（引导思考）："我们来核对一下，规范里对链接格式是怎么规定的？"

### 规则2：两层提问结构——事实层+方法论层

有效的提问通常包含两层：

| 层级 | 提问目标 | 示例问题 |
|------|---------|---------|
| **第一层：事实溯源** | 打断直觉，追问判断依据 | "这个判断/格式/决策是从哪里来的？" |
| **第二层：方法论对齐** | 启动第一性原理/规范检查 | "回到基本事实，我们的本质目标是什么？这个选择符合吗？" |

本模式的经典案例（2026-07-09事件）正是完美的两层结构：
1. 第一层："所有链接统一为 file:/// 绝对路径格式？"（质疑判断本身，溯源依据）
2. 第二层："这个判断哪里来的？符合第一性原理吗？"（启动方法论思考）

### 规则3：三不原则

- **不预设立场**：提问不是"我知道正确答案，我来引导你说出我的答案"，而是"我们一起回到事实重新推导"——万一你是错的呢？
- **不攻击个人**：对事不对人，质疑的是"决策/判断"，不是"人/能力"
- **不急于给答案**：如果对方一时答不上来，可以给提示，但不要直接抛出答案。给对方3秒钟思考时间。

### 规则4：对方自我发现后，及时强化方法论

当对方通过你的提问自己发现错误后，不要说"你看，我早就说了你错了"，而是：
1. 肯定对方的自我发现："对，这就是关键区别"
2. 明确点出方法论："这就是第一性原理——打断类比推理，回到规范和基本事实"
3. 总结可复用教训："下次做类似决策前，可以先问自己这两个问题"

这样才能把一次纠错变成一次方法论学习。

## 提问框架模板

### 通用纠错提问框架

```
第一步：确认理解（避免误解）
"我理解一下，你这里的意思是XXX，对吗？"

第二步：事实溯源（打断直觉）
"这个判断的依据是什么？是来自规范文档，还是之前的例子？"

第三步：方法论对齐（启动慢思考）
"如果我们回到第一性原理/规范本身，本质目标是什么？当前选择满足这个目标吗？"

第四步：对比验证（强化发现）
"我们看看规范/现有实例里是怎么做的，和当前选择有什么不一样？"

第五步：总结强化（沉淀方法论）
"很好，你自己发现了问题。我们来总结一下，下次遇到类似情况应该怎么检查？"
```

### 不同场景的提问示例

| 场景 | 直接指正（反模式） | 苏格拉底提问（推荐） |
|------|------------------|-------------------|
| 发现格式错误 | "链接格式错了，应该用相对路径" | "我们来核对一下：开发规范里对Markdown内部链接格式是怎么规定的？你这个判断的依据是什么？" |
| 发现代码逻辑Bug | "这里边界条件没处理，会NPE" | "如果输入是空/极端值，这里会发生什么？我们来顺着异常路径走一遍？" |
| AI类比推理错误 | "你混淆了两个场景，不能直接套" | "这个类比成立吗？两个场景的约束条件完全一样吗？我们回到问题本质重新推导？" |
| 设计方案不合理 | "这个方案性能有问题" | "如果流量涨10倍，这个架构的瓶颈在哪里？我们的核心约束条件是什么？" |

## 反模式

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| 反问句质问："你难道没看规范吗？" | 带攻击性，触发防御心理，对方第一反应是辩解而非思考 | 用中性的提问："我们来查一下规范是怎么写的" |
| 连环追问不给思考时间 | 变成"审问"而非"引导"，对方会紧张而非思考 | 每个问题后停3秒，给对方思考时间 |
| 问了半天对方还没发现，你也不说答案 | 浪费时间，对方会有挫败感 | 给提示："要不要看看XX文件第X段？"，实在发现不了再直接指出 |
| 对方发现错误后说"我早就说了你错了" | 破坏信任，下次对方不会愿意自我反思 | 肯定对方："很好，自己发现问题印象才深刻" |
| 所有错误都用提问法 | 紧急情况需要直接指令，明显事实错误不需要绕弯 | 紧急情况直接给指令，事实错误直接指出，方法论/逻辑错误才用提问 |
| 提问是为了证明自己对 | 变成"抬杠"而非协作，对方会感觉到你的预设立场 | 保持开放心态：万一最后发现是你错了呢？ |

## 与对抗式审查的关系

本模式是[对抗式审查模式](adversarial-review-prompt-pattern.md)的**温和协作版本**：

| 维度 | 对抗式审查 | 苏格拉底提问纠错 |
|------|-----------|----------------|
| 立场 | 站在对立面，模拟攻击者/挑错者 | 站在同一边，作为思考伙伴 |
| 语气 | 尖锐、挑战、攻击性 | 温和、引导、协作性 |
| 适用场景 | 高风险代码/架构审查、安全审计 | 日常协作、教学培训、AI纠错 |
| 目标 | 发现所有可能的漏洞 | 引导对方自己发现问题、学会方法 |
| 防御心理 | 可能触发（但提前约定了对抗角色） | 最小化防御心理 |

两者的核心共通点：**都不直接给答案，而是通过提问/挑战让创作者自己重新思考**。区别在于强度和立场——苏格拉底提问更温和，适合日常协作；对抗式审查更尖锐，适合高风险场景。

## 实际案例

### 案例1：第一性原理类比推理错误纠错（本模式来源）

| 项目 | 内容 |
|------|------|
| 时间 | 2026-07-09 |
| 背景 | AI刚刚学习完"第一性原理Prompt"模式，在做简单格式更新任务时犯了类比推理错误：将13个文件的链接批量改为file:///绝对路径 |
| 用户提问 | 1. "所有链接统一为 file:/// 绝对路径格式？"<br>2. "这个判断哪里来的？符合第一性原理吗？" |
| 效果 | AI没有产生防御心理，立即启动第一性原理思考，自己回到规范核查，发现混淆了"AI对话输出格式"和"Markdown文档内部链接格式"两个场景，18分钟内完成修正 |
| 长期价值 | AI不仅修正了错误，还深刻理解了"第一性原理需要在每个决策点主动启动，简单任务也不能跳过"，沉淀出3个可复用洞察和2个新模式 |
| 关键成功因素 | 第二个问题直接点出"符合第一性原理吗"——这不是普通提问，而是在教AI用刚学的方法论自查，本身就是第一性原理Prompt的正确用法 |
