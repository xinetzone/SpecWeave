# 视觉通用操作模式（Visual-based Universal Operation）

## 一、核心定义

视觉通用操作（Visual Universal Operation）是AI Agent操作异构现实世界系统（遗留软件、闭源应用、无API系统、物理设备）的务实技术路线：**不等待、不依赖目标系统提供API/SDK/插件接口，而是通过"屏幕视觉识别+键鼠/触控输入模拟"的方式，像人类一样"看屏幕、点按钮、敲键盘"完成操作**。这种方式具有真正的通用性——任何人类可以操作的图形界面，AI都可以用同样方式操作，无需目标系统做任何改造。

**技术路线对比**：

```
路线A：API集成路线（理想但不现实）
AI Agent → 调用目标系统开放API → 完成操作
问题：90%以上的软件/系统没有开放API，或API不全，或API权限受限

路线B：视觉通用操作路线（务实通用）
AI Agent → 屏幕截图视觉识别（OCR+CV+UI理解）→ 理解界面元素 → 模拟键鼠/触控操作 → 完成操作
优势：100%兼容所有图形界面系统，目标系统零改造、零配合
```

**向日葵MCP案例**：OrayClaw AI助手不要求被控电脑上的软件开放API，而是直接通过远程桌面的画面进行视觉识别，模拟鼠标点击、键盘输入、滚轮滚动，就能操作任何Windows/Mac应用——无论是旧版Office、定制工业软件、还是任何闭源程序，AI都能像人一样操作。

## 二、为什么这是务实路线而非权宜之计

### API路线的三个根本问题

| 问题 | 说明 |
|------|------|
| **覆盖度问题** | 绝大多数软件没有API，特别是闭源商业软件、工业软件、遗留系统、定制企业应用 |
| **版本碎片化问题** | 即使有API，不同版本API不兼容，软件升级后API可能失效 |
| **权限问题** | API通常需要用户申请权限、配置密钥、管理员授权，普通用户无法使用 |

### 视觉路线的三个根本优势

| 优势 | 说明 |
|------|------|
| **真正的通用性** | 任何有图形界面的系统都可以操作，不需要目标系统做任何工作 |
| **零集成成本** | 不需要与软件厂商谈判、不需要等厂商排期开发API、不需要适配每个版本 |
| **用户零配置** | 用户不需要申请API key、不需要安装插件、不需要改配置，开箱即用 |

**关键洞察**：很多人认为"等所有软件都有API了AI才能真正操作电脑"，这是不切实际的幻想——历史遗留系统、闭源商业软件、工业控制软件永远不会开放完整API。视觉路线不是"过渡方案"，而是"最终方案"之一，就像人类不需要软件有API也能操作电脑一样。

## 三、技术实现四层架构

```
┌─────────────────────────────────────────────────────────┐
│  第四层：任务规划层（大模型）                              │
│  将用户自然语言需求分解为具体操作步骤序列                  │
│  "帮我打开Excel，把A列数据复制到B列" → 步骤序列            │
└─────────────────────────────────────────────────────────┘
                              ▲
                              │ 操作步骤
                              │
┌─────────────────────────────────────────────────────────┐
│  第三层：UI理解层（视觉大模型+OCR+CV）                     │
│  理解屏幕内容：识别按钮、输入框、菜单、文本等UI元素        │
│  将视觉信息转化为结构化UI语义："右上角有个'保存'按钮"       │
└─────────────────────────────────────────────────────────┘
                              ▲
                              │ 截图
                              │
┌─────────────────────────────────────────────────────────┐
│  第二层：输入模拟层（键鼠/触控仿真）                        │
│  执行具体输入：鼠标移动/点击/拖拽、键盘输入、快捷键、滚轮  │
│  支持精确坐标点击和语义元素点击两种模式                    │
└─────────────────────────────────────────────────────────┘
                              ▲
                              │ 输入事件
                              │
┌─────────────────────────────────────────────────────────┐
│  第一层：画面采集层（屏幕采集/远控画面流）                  │
│  实时获取屏幕画面：本地截屏、远程桌面视频流、摄像头画面    │
│  向日葵案例：远控视频流作为视觉输入                        │
└─────────────────────────────────────────────────────────┘
```

## 四、核心设计原则

### 原则1：语义理解优先，坐标操作为辅

- ✅ 优先通过UI语义（"蓝色的确定按钮"、"包含'保存'文字的按钮"）定位元素
- ⚠️ 仅在语义无法定位时使用精确坐标点击
- ❌ 完全依赖固定坐标（窗口移动/分辨率变化/软件换肤后立即失效）

### 原则2：操作后验证闭环

每次执行操作后必须验证结果：
1. 操作前截屏，记录初始状态
2. 执行操作（点击/输入）
3. 操作后截屏，对比变化
4. 确认操作达到预期效果（如对话框消失、新页面出现、文字已输入）
5. 如果验证失败，重试或调整策略，不能盲目执行下一步

**反模式**：不管操作有没有生效，按预设步骤一路点下去，最后出错了也不知道哪一步出了问题。

### 原则3：渐进式复杂度处理

| 场景复杂度 | 处理策略 |
|---------|---------|
| **标准控件**（原生按钮/输入框/菜单） | OCR+控件识别直接定位，准确率>99% |
| **自定义UI**（自绘界面、游戏界面） | 图标匹配+文字识别+位置关系推断 |
| **动态界面**（动画、加载中、弹窗） | 等待界面稳定后再操作，设置超时重试 |
| **错误状态**（报错弹窗、网络错误） | 识别错误信息，判断是否需要用户介入 |

### 原则4：人在回路（Human-in-the-loop）兜底

AI不是100%正确的，必须设计人工介入机制：
- 危险操作（删除文件、发送消息、支付）必须用户确认
- AI识别置信度低时主动请求用户指引
- 操作失败多次后停止并请求用户接管
- 全程可被用户中断和纠正

## 五、向日葵OrayClaw MCP实现验证

向日葵MCP协议+OrayClaw AI助手是视觉通用操作模式的典型落地：

| 架构层 | 向日葵实现 |
|-------|----------|
| 画面采集层 | 远控H.264/H.265视频流作为视觉输入 |
| 输入模拟层 | 向日葵远控协议原生支持鼠标移动/点击/拖拽、键盘输入、滚轮、快捷键 |
| UI理解层 | 视觉大模型对远控画面进行UI元素识别、文字识别、状态判断 |
| 任务规划层 | OrayClaw大模型将自然语言任务分解为远控操作步骤序列 |

**已实现的22个MCP工具覆盖**：
- 基础操作：截屏、鼠标点击、键盘输入、滚轮滚动
- 系统操作：打开应用、文件管理、进程管理
- 办公操作：Office文档编辑、浏览器操作、邮件发送
- 运维操作：命令执行、服务管理、日志查看

所有操作都不需要被控电脑上安装任何插件或开放API——只要人类能通过向日葵远控操作的电脑，AI就能操作。

## 七、优势与局限性

### 优势

1. **通用性极强**：一个AI Agent可以操作所有软件，不需要每个软件单独集成
2. **部署成本为零**：目标系统零改造，不需要厂商配合
3. **用户门槛低**：不需要API key、不需要插件、不需要配置
4. **版本无关**：软件升级、换肤、语言变化不影响核心能力（只要视觉上能识别）

### 局限性与应对

| 局限性 | 应对方案 |
|-------|---------|
| 视觉识别准确率不是100% | 置信度阈值+人在回路确认+操作后验证 |
| 比API调用慢（需要截屏+识别+模拟操作） | 简单标准操作可以提供"快路径"API，API不可用时回退到视觉路线 |
| 游戏/视频等高速动态界面识别困难 | 专门优化动态界面识别模型，降低操作延迟 |
| 无法看到屏幕外的内容（如长文档未滚动部分） | 自动滚动+分屏识别+记忆整合 |
| 安全风险（AI可能误操作） | 危险操作二次确认+权限分级+操作审计+全程可中断 |

## 八、与"API开放"路线的关系

视觉通用操作和API开放不是互斥关系，而是互补关系：

```
最优策略 = API快路径（优先）+ 视觉通用路径（兜底）

1. 目标系统有API → 优先用API（快、准、可靠）
2. 目标系统无API/API不全 → 自动降级到视觉通用操作
3. AI Agent不需要提前知道哪个系统有API，统一用"操作"语义，底层自动选择最优路径
```

向日葵MCP正是这种策略：它既提供原生MCP工具（快路径），也支持视觉通用操作（兜底路径），两者结合实现最大覆盖。

参见关联模式：[vertical-saas-mcp-capability-exposure.md](../product-growth/vertical-saas-mcp-capability-exposure.md)

## 九、适用边界

### 适用场景

- ✅ AI Agent操作闭源/遗留/无API的软件系统
- ✅ 远程控制/远程运维场景（天然有屏幕画面和输入通道）
- ✅ RPA流程自动化（特别是非标准UI的应用）
- ✅ 无障碍辅助（为视障用户提供智能操作辅助）
- ✅ 跨应用通用任务（如"把微信收到的文件保存到网盘并发邮件给同事"）
- ✅ 工业控制/老旧系统（无法升级改造但需要智能化）

### 不适用场景

- ❌ 极高吞吐量操作（每秒上千次操作级别，API更合适）
- ❌ 纯数据处理（没有界面的数据操作直接用API/代码更高效）
- ❌ 3D游戏/高速实时操作（视觉延迟可能影响体验）
- ❌ 强安全场景（纯视觉操作可能绕过安全机制，需要额外权限控制）
