# 技术选型「偏好-惯例-本质」三查法（Preference-Convention-Essence Three Checks）

## 模式类型
方法论模式（治理策略/决策质量门禁）

## 成熟度
L2 验证级（2次验证：配置文件格式决策workspace.yaml、复盘报告中提炼）

## 问题陈述

技术选型决策中存在三类系统性偏差：

1. **偏好盲区**：忽略用户或团队的明确偏好设置，选择了技术上可行但不符合团队习惯的方案（如用户偏好YAML却选了JSON，用户偏好某种框架却选了另一种）
2. **惯例破坏**：不搜索项目现有代码和配置，盲目引入新格式/新依赖/新风格，导致项目内出现多种并行方案，增加认知负担和维护成本
3. **本质误判**：被"XX生态更成熟"、"XX性能更好"等表面因素吸引，忽略数据结构和使用场景的本质特征对技术方案的适配需求

这三类偏差单独或共同作用时，会导致决策返工（如本次workspace.json→workspace.yaml替换38处引用）、项目不一致、维护成本上升等问题。

## 解决方案

在任何技术选型决策前，强制按顺序执行三步检查，每步检查通过后才能进入下一步。

### 三查法流程

```mermaid
flowchart LR
    A["面临技术选型决策"] --> B{"查偏好"}
    B -->|"有明确偏好"| C["优先遵循偏好"]
    B -->|"无明确偏好"| D{"查惯例"}
    C --> D
    D -->|"有现有惯例"| E["遵循现有惯例"]
    D -->|"无现有惯例"| F{"查本质"}
    E --> F
    F --> G["基于本质需求决策"]
    G --> H["记录决策依据"]
```

### 第一查：查偏好（Preference Check）

| 维度 | 检查内容 | 检查方法 | 权重 |
|------|---------|---------|------|
| 用户偏好 | user_profile.md是否有明确偏好设置？ | 读取用户配置文件 | 高 |
| 团队规范 | 是否有团队编码规范/架构决策记录(ADR)？ | 搜索docs/knowledge/decisions/ | 高 |
| 项目规则 | AGENTS.md/开发规范中是否有明确规定？ | 读取AGENTS.md和开发规范文档 | 高 |

**决策规则**：
- 有明确偏好 → 优先遵循偏好，除非有强理由偏离并记录原因
- 无明确偏好 → 进入第二查

**典型案例**：本次配置文件格式决策中，user_profile.md明确标注"prefers YAML over TOML"，这是最高优先级的决策依据。

### 第二查：查惯例（Convention Check）

| 维度 | 检查内容 | 检查方法 | 权重 |
|------|---------|---------|------|
| 同类文件格式 | 项目中同类配置/数据文件使用什么格式？ | Grep搜索现有文件扩展名和格式 | 高 |
| 同类库/框架 | 项目中已使用哪些同类库/框架？ | 查看依赖配置文件（requirements.txt/package.json等） | 高 |
| 目录结构惯例 | 同类资源放置在什么目录？命名约定是什么？ | LS/Glob查看现有目录结构 | 中 |
| 代码风格 | 同类代码采用什么风格/模式？ | 读取现有同类代码示例 | 中 |

**决策规则**：
- 有明确现有惯例 → 遵循惯例，保持一致性（Convention over Configuration）
- 有多种并行惯例 → 选择使用最广泛的，记录统一计划
- 无现有惯例（全新领域）→ 进入第三查

**典型案例**：本次配置文件格式决策中，Grep发现`.agents/teams/data/*.yaml`均使用YAML格式，角色frontmatter虽用TOML引用但主体是Markdown，YAML是配置文件的主流惯例。

### 第三查：查本质（Essence Check）

| 维度 | 检查内容 | 分析方法 | 权重 |
|------|---------|---------|------|
| 数据结构特征 | 嵌套层级？是否需要注释？是否手工编辑？ | 分析目标数据的结构特征 | 高 |
| 使用场景特征 | 谁来读写？机器→机器 / 人→机器 / 人→人？ | 分析使用场景和受众 | 高 |
| 工具链需求 | 是否需要Schema验证？是否需要跨语言？ | 分析工具链需求 | 中 |
| 依赖成本 | 引入新依赖的成本？学习曲线？ | 评估引入成本 | 中 |

**决策规则**：
- 基于数据结构本质特征选择最适配的方案
- 人→人场景优先可读性和表达力
- 机器→机器场景优先解析性能和Schema生态
- 人→机器场景优先可读性+注释支持+工具兼容性

**典型案例**：本次配置文件格式决策中，Workspace manifest的本质特征：
- 多层嵌套结构（roles/skills/a2a/exports/hooks等）→ YAML表达最自然
- 手工编写维护 → 需要注释支持 → TOML/YAML支持，JSON不支持
- 机器验证 → 需要JSON Schema → YAML是JSON超集，可直接用JSON Schema验证
- 结论：YAML完美匹配本质需求

### 决策记录模板

完成三查后，记录决策依据：

```markdown
## 决策记录：[决策主题]
- **查偏好结果**：[有/无明确偏好，偏好内容]
- **查惯例结果**：[有/无现有惯例，惯例内容]
- **查本质结果**：[本质特征分析，适配度评估]
- **最终决策**：[决策结果]
- **偏离说明**：[如偏离偏好/惯例，说明强理由]
```

## 适用场景

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 配置文件格式选择（JSON/YAML/TOML等） | 核心场景 | 本次验证场景，完美匹配 |
| 第三方库/框架选型 | 核心场景 | 查偏好+查惯例+查需求本质 |
| 代码风格/命名约定决策 | 核心场景 | 惯例优先 |
| 目录结构设计 | 核心场景 | 惯例优先+本质适配 |
| 架构模式选择 | 适用 | 与第一性原理决策门禁互补使用 |
| 算法/数据结构选择 | 部分适用 | 本质检查权重更高，偏好/惯例权重较低 |

## 与第一性原理决策门禁的互补关系

| 维度 | 本模式（三查法） | first-principles-decision-quality-gate |
|------|----------------|---------------------------------------|
| 决策类型 | 常规技术选型（格式/框架/风格） | 高影响架构决策/需要重新评估的决策 |
| 核心方法 | 偏好优先→惯例一致→本质适配 | 假设剥离→公理提炼→自下而上重构 |
| 认知负荷 | 低（三步检查清单） | 高（6步深度分析） |
| 适用时机 | 日常决策、批量决策 | 重大决策、争议决策、条件变化时 |
| 防止的偏差 | 偏好盲区、惯例破坏、本质误判 | 确认偏差、锚定效应、直觉跳跃 |

**组合使用建议**：常规技术选型用三查法快速决策；当三查法发现决策存在争议或高影响时，升级为第一性原理决策门禁做深度分析。

## 5-Whys 根因链

```
为什么技术选型容易返工？→ 因为决策时忽略了关键因素（偏好/惯例/本质）
为什么会忽略关键因素？→ 因为没有结构化检查清单，依赖直觉判断
为什么直觉判断容易出错？→ 因为System 1思维倾向于"哪个熟悉选哪个"或"哪个新选哪个"
为什么需要结构化检查？→ 因为显式检查清单能强制调动System 2，覆盖盲区
为什么按偏好→惯例→本质顺序？→ 偏好是最高优先级（人的因素），惯例其次（一致性），本质是最后决策依据（适配性）
```

## 反模式警示

| 错误做法 | 后果 | 正确做法 |
|---------|------|---------|
| 跳过查偏好，直接做技术比较 | 选择了技术可行但不符合团队习惯的方案，用户不满 | 第一步必查偏好，有明确偏好优先遵循 |
| 不查现有代码直接引入新格式/新依赖 | 项目内多种并行方案，维护成本翻倍 | 第二步必查惯例，能复用就不新增 |
| 被"XX更流行"、"XX性能更好"迷惑 | 忽略本质需求，选了不适配的方案（如JSON用于深度嵌套+手工编辑的配置） | 第三步必查本质，从数据结构和使用场景出发 |
| 三查顺序颠倒（如先查本质再查偏好） | 浪费时间在技术比较上，最后发现有明确偏好却已做了大量分析 | 严格按偏好→惯例→本质顺序，前一步有答案就无需做后一步 |
| 决策后不记录依据 | 未来复盘时无法追溯"当初为什么选这个" | 使用决策记录模板记录三查结果 |
