# 量化数据验证四查法

## 模式类型
文档治理/质量保证模式

## 成熟度
**L2 已验证**（并发安全检查器复盘报告数据漂移修正：9处数据漂移全部发现并修正；本次复盘洞察文档更新时二次验证，发现并修正了模式文件自身的元层数据漂移）

## 一、核心原理

文档（尤其是技术复盘报告）中包含大量量化数据：代码行数、测试数量、覆盖率、变更统计、性能指标等。这些数据在文档编写时可能是准确的，但随着代码迭代（TDD红绿循环、bug修复、功能扩展），数字会持续变化，而文档不会自动更新——形成**数据漂移（Data Drift）**。

数据漂移比功能描述漂移更隐蔽：
- 功能错误通常有逻辑矛盾（如"六维"与八维表格不一致）
- 数字错误可以"自洽"（465行、33个测试、1893行总计之间没有矛盾，只是都过时了）

**量化数据验证四查法**提供系统化的数据准确性验证框架，四个维度层层递进：

| 检查维度 | 核心问题 | 发现的问题类型 |
|---------|---------|--------------|
| **一查：数据来源** | 这个数字从哪来？源头是否可靠？ | 凭空估计、记忆错误、来源过时 |
| **二查：计算逻辑** | 数字是怎么算出来的？公式是否正确？ | 统计口径错误、求和/平均错误 |
| **三查：异常值** | 数字是否在合理范围内？ | 数量级错误、不可能值、突变 |
| **四查：关联性** | 相关数字之间是否一致？ | 部分数字更新了但关联数字没更新 |

## 二、四查法详解

### 一查：数据来源（Source Verification）

**核心问题**：文档中的每个数字，是否能追溯到实时查询命令或脚本输出？

**验证方法**：

| 数据类型 | 正确获取方式 | 错误方式 |
|---------|------------|---------|
| 文件行数 | `(Get-Content file).Count` 或 `wc -l file` | 目测估计、使用上次记忆值 |
| 测试数量 | `pytest --collect-only -q` 并计数 | 编写测试时的数量、上次运行记忆 |
| 文件数量 | `Get-ChildItem -Recurse -File | Measure-Object` | 目测目录列表 |
| 代码覆盖率 | `pytest --cov` 实际运行 | 上次覆盖率报告 |
| 提交数 | `git log --oneline \| Measure-Object` | 估计大概数量 |
| 变更行数 | `git diff --stat` 实际查看 | 记忆中的大概数字 |

**检查清单**：
- [ ] 文档中每个量化指标是否都有对应的获取命令？
- [ ] 数字是否是在本次编辑/更新时实时获取的？
- [ ] 如果数字来自历史报告，是否验证过代码自那以后没有变化？

**铁律**：**"数字不信任原则"**——文档中已有的任何数字都必须视为"待验证"状态，必须通过实时查询确认。

### 二查：计算逻辑（Calculation Logic）

**核心问题**：数字的计算方式是否正确？统计口径是否一致？

**常见计算逻辑错误**：

| 错误类型 | 示例 | 正确做法 |
|---------|------|---------|
| 统计口径不一致 | "核心代码行数"只算visitor不算scanner/cli | 明确定义"核心代码"包含哪些文件 |
| 求和错误 | 各文件行数之和≠总代码行数 | 实际加总验证 |
| 遗漏文件 | 交付物清单遗漏新增文件 | 用glob/ls核对清单与实际文件 |
| 重复计数 | 同一文件在两个类别中被计数 | 去重后验证 |
| 单位错误 | KB vs MB，秒vs毫秒 | 标注单位并验证 |

**验证方法**：
1. 明确定义每个统计指标的计算口径（包含哪些文件/哪些范围）
2. 用脚本/命令实际计算，而非心算或目测
3. 交叉验证：多个独立方式计算同一指标，结果应一致
4. 验证"合计"行：各行数字之和是否等于合计数

### 三查：异常值（Anomaly Detection）

**核心问题**：数字是否在合理范围内？是否存在不可能值或突变？

**异常值检测规则**：

| 检测维度 | 判断方法 | 异常信号 |
|---------|---------|---------|
| 数量级合理性 | 同类文件对比 | visitor.py 465行→840行属于增长，但如果是10行→5000行就要警惕 |
| 物理约束 | 常识判断 | 行数不能为负、测试数不能≥代码行数*10、覆盖率不能>100% |
| 时间一致性 | 与历史版本对比 | 行数突然翻倍或减半需要解释 |
| 比例合理性 | 相关指标比例 | 测试代码:生产代码≈1:1到2:1为正常，1:10或10:1需检查 |
| 离散值检测 | 与同类项目对比 | 单个文件远超其他同类文件平均大小 |

**检查清单**：
- [ ] 每个数字是否在物理上可能？（非负、不超限）
- [ ] 与上次报告/同类项目相比，变化是否可解释？
- [ ] 相关指标的比例是否合理？
- [ ] 如果数字异常，是否有注释说明原因？

### 四查：关联性（Cross-Reference Consistency）

**核心问题**：文档中互相引用的数字是否一致？同一数字在不同位置出现时值是否相同？

**常见关联性错误**：

| 错误类型 | 示例 | 检测方法 |
|---------|------|---------|
| 摘要与正文不一致 | 执行摘要说"33个测试"，交付物清单说"48个" | 全文搜索数字，检查所有出现位置 |
| 表格与文字不一致 | 文字说"八维"，表格只列了6行 | 文字描述与表格数据交叉核对 |
| 父指标与子指标不一致 | 合计1893行，但各文件行数之和=2565行 | 汇总数字=各分项之和验证 |
| 前后更新不一致 | 更新了测试数但未更新测试代码行数 | 检查相关联的数字是否同步更新 |
| 跨文档不一致 | README写33个测试，retrospective-report写48个 | 同一目录下文件的相同数字一致性 |

**验证方法**：
1. 提取文档中所有量化数据，建立"数字清单"
2. 对每个数字，全文搜索其出现位置
3. 同一指标在不同位置出现时，值必须一致
4. 父子关系指标：子项之和必须等于父项
5. 关联指标：比例关系必须合理（如测试数↑ 测试代码行数也应↑）

## 三、自动化验证脚本设计

四查法可以通过自动化脚本辅助执行（详见配套脚本 `check-retro-data.py`）：

```
输入：retrospective-report.md 文件路径
处理：
  1. 解析Markdown，提取所有数字（含上下文）
  2. 识别表格中的数字及其行列标签
  3. 检测"合计"行与分项之和是否一致
  4. 检测同一数字在文档中多次出现时值是否一致
  5. 检测行数/测试数等指标与源文件实际值是否匹配
  6. 标记超出合理范围的异常值
输出：验证报告（通过/警告/错误 + 异常数据点 + 可能原因）
```

## 四、实现步骤（手动执行流程）

### Step 1：建立数字清单

通读文档，提取所有量化数据，记录：
- 数字值
- 所在位置（章节/表格/行号）
- 指标名称（如"visitor.py行数"、"单元测试数量"）
- 数字来源（是来自实时查询还是文档继承）

### Step 2：一查——逐个数验证来源

对清单中每个数字：
1. 确定正确的获取命令
2. 执行命令获取实时值
3. 对比文档值与实时值
4. 不一致的标记为"来源错误"

### Step 3：二查——验证计算逻辑

对汇总类数字（合计、总计、比例、增长率）：
1. 明确计算公式
2. 用分项数字重新计算
3. 对比文档值与计算值
4. 不一致的标记为"计算错误"

### Step 4：三查——异常值筛查

对每个数字：
1. 判断是否在物理可能范围内（>0，<上限）
2. 与同类文件/历史版本对比
3. 检查相关指标比例
4. 可疑值标记为"异常值"并附可能原因

### Step 5：四查——关联一致性

1. 按指标名称分组，检查同一指标在不同位置是否一致
2. 验证父子指标关系（合计=分项之和）
3. 验证跨文档一致性（同一目录下README与正文）
4. 不一致的标记为"关联错误"

### Step 6：生成验证报告

汇总所有发现，按严重级别分类：
- 🔴 Error：数据与源代码不一致（必须修正）
- 🟡 Warning：计算逻辑有问题或异常值无法解释（需要复核）
- 🔵 Info：关联不一致可能需要更新（建议修正）

## 五、适用场景

| 场景 | 适用程度 | 说明 |
|------|---------|------|
| 技术复盘报告中的代码统计 | ✅ 强烈推荐 | 行数、测试数、文件数等随代码迭代变化 |
| 性能测试报告 | ✅ 强烈推荐 | 性能指标数据必须实时获取 |
| 项目进度报告 | ✅ 推荐 | 完成百分比、燃尽图数据需验证 |
| API文档中的参数说明 | ⚠️ 选择性适用 | 参数范围/默认值需要验证，但"支持3种认证方式"这类描述变化不频繁 |
| 概念性/架构文档 | ❌ 不适用 | 不含量化数据或数据不随代码变化 |
| 会议纪要/讨论记录 | ❌ 不适用 | 非技术文档，数据漂移风险低 |

## 六、潜在限制

1. **无法自动验证语义正确性**：脚本能检测数字格式错误和来源错误，但无法判断"这个维度是否应该算作八维之一"这类语义问题。
2. **需要配置数据源映射**：自动化脚本需要知道"visitor.py行数"对应哪个文件路径，这需要配置而非自动推断。
3. **合理范围阈值主观**：什么算"异常值"需要根据项目特点设定阈值，不同项目阈值不同。
4. **无法验证外部数据**：如果数字来自外部系统（CI结果、监控数据），脚本无法自动获取，需要手动提供。

## 七、使用注意事项

1. **不要因为"上次是对的"就信任数字**：代码在持续迭代，上次正确的数字现在可能已经过时。
2. **优先使用脚本而非手动计算**：手动求和容易出错，尤其是多行表格。
3. **文档更新时必须全量验证**：即使你只改了一个数字，也要检查相关联的其他数字是否需要同步更新。
4. **记录数字获取时间**：对于会随时间变化的指标（如测试覆盖率），标注"截至YYYY-MM-DD"。
5. **与pre-commit结合**：将自动化数据验证作为pre-commit钩子的可选检查项，在提交复盘报告时自动运行。
6. **"四查"不是"四选一"**：四个维度必须全部执行，缺一不可。

## 八、验证案例

### 案例：并发安全检查器复盘报告数据漂移

应用四查法发现9处数据漂移：

| # | 位置 | 文档值 | 实际值（脚本验证） | 错误类型 | 四查维度 |
|---|------|--------|-------------------|---------|---------|
| D1 | visitor.py行数 | 465行 | 840行 | 来源过时（TDD扩展DEADLOCK+LEAK+I18N后未更新） | 一查 |
| D2 | 单元测试数量 | 33个 | 48个（pytest收集） | 来源过时（新增15个边界/死锁/泄漏/集成测试） | 一查 |
| D3 | constants.py行数 | 50行 | 85行 | 来源过时（新增DEADLOCK/LEAK维度定义） | 一查 |
| D4 | scanner.py行数 | 90行 | 104行 | 来源过时（新增POOL_CLASSES识别+with跟踪） | 一查 |
| D5 | cli.py行数 | 117行 | 138行 | 来源过时（新增DIM环境变量+输出格式优化） | 一查 |
| D6 | models.py行数 | 33行 | 44行 | 来源过时（新增ResultGroupMixin+passes属性） | 一查 |
| D7 | 合计行数 | ~1893行 | ~2334行（核心1226+钩子206+测试902） | 计算错误（分项更新了合计没更新，且原估算值2565仍有误） | 二查+四查 |
| D8 | concurrent_check.py钩子 | 173行 | 206行 | 来源过时（新增DEADLOCK/LEAK提示+环境变量处理） | 一查 |
| D9 | 测试代码行数 | 534行 | 902行 | 来源过时+关联不一致（测试数↑但代码行数未同步） | 一查+四查 |

**元层教训**：在模式化归档过程中再次应用四查法，发现模式文件自身也存在数据漂移（7处→9处、2565→2334），证明四查法需要在每次文档更新（包括模式更新）时全量执行。

**修正后验证**：所有数字通过脚本实时获取验证，合计=分项之和（1226+206+902=2334），跨文档一致（retrospective-report.md、README.md、insight-extraction.md、模式文件四处数据一致）。
