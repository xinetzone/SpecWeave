# 治理策略模式 - 案例材料维护指南

本目录存放治理策略模式的原始案例材料，为模式提供真实项目验证支撑。

## 一、核心概念

### 模式 vs 案例

| 维度 | 模式（Pattern） | 案例（Case） |
|------|----------------|-------------|
| **位置** | 上级目录 `*.md` 文件 | `cases/<pattern-id>/` 子目录 |
| **性质** | 抽象后的通用方法论 | 具体项目中的验证材料 |
| **内容** | 触发场景、核心做法、反模式、检验标准 | 配置文件、架构图、部署脚本、验证数据 |
| **生命周期** | 相对稳定，随新案例持续迭代 | 归档后只读，新案例新增子目录 |
| **成熟度关联** | `maturity` 字段（L1→L2→L3） | `validation_count` 计数（每案例+1） |

### 为什么需要案例归档？

模式萃取遵循「多案例支撑原则」——单案例只能标记为 L1-draft，≥2个独立案例验证才能升级为 L2-formal：

- **单案例风险**：可能是特定环境产物、运气因素、过度拟合
- **多案例价值**：跨项目验证可迁移性，提炼真正不变的核心
- **案例的作用**：作为模式可复用性的证据，为新使用者提供参考起点

## 二、目录结构

```
cases/
├── README.md（本文件：维护指南）

    ├── <脚本/配置文件>              # 可复用的脚本、配置模板等
    └── ...
```

## 三、维护操作指南

### 操作1：为已有模式新增案例

当一个模式在新的项目中得到验证时，按以下步骤归档：

#### 步骤1：创建案例目录

```bash
# 在 cases/ 下创建子目录，目录名 = 模式ID
mkdir cases/<pattern-id>
```

目录命名规则：
- ✅ 与模式 frontmatter 中的 `id` 字段完全一致
- ✅ 使用小写英文+连字符（kebab-case）
- ❌ 不要使用中文目录名
- ❌ 不要使用版本号后缀（多个案例使用 README 中的案例序号区分）

#### 步骤2：编写案例索引 README.md

每个案例子目录必须包含 `README.md`，使用以下模板：

```markdown
# 案例：<项目名称> - <模式应用简述>

## 案例信息

| 项 | 值 |
|---|---|
| 案例ID | case-<pattern-id>-NNN（NNN为3位序号，从001开始） |
| 对应模式 | [<pattern-id>.md](../../<pattern-id>.md) |
| 项目来源 | <项目路径或名称> |
| 验证日期 | YYYY-MM-DD |
| 验证人 | <验证者> |
| 环境 | <操作系统/关键依赖版本> |

## 案例材料清单

| 文件 | 类型 | 说明 |
|---|---|---|
| <文件名> | 配置手册/架构图/部署指南/脚本/其他 | <一句话说明> |

## 验证结果

### 关键指标对比

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| <指标1> | <值> | <值> |

### 验证范围

- ✅ <已验证的模式要点>
- ❌ <未验证/不适用的模式要点>（说明原因）

### 环境规格

- <关键依赖版本>

### 实际效果

1. <可观测的效果1>
2. <可观测的效果2>

## 原始位置

案例原始文件位于项目工作目录：`<原始路径>`（是否保留在原地）

## 模式偏差说明（可选）

如果本案例在应用模式时做了适配调整，在此说明：
- 调整内容：<做了什么改动>
- 调整原因：<为什么需要改动>
- 对模式的启示：<是否应该更新模式来涵盖这种情况>
```

**必填字段检查清单**：
- [ ] 案例ID 格式正确（case-<pattern-id>-NNN）
- [ ] 对应模式链接正确（`../../<pattern-id>.md`，从 cases/<id>/ 向上两级）
- [ ] 验证日期格式为 YYYY-MM-DD
- [ ] 指标对比表包含至少2个量化指标
- [ ] 验证范围明确标注 ✅/❌
- [ ] 原始位置字段已填写

#### 步骤3：复制原始材料到案例目录

将相关的配置文件、文档、图表、脚本复制到案例子目录：

```bash
cp <源文件路径> cases/<pattern-id>/
```

注意事项：
- ✅ 复制后检查文件中的相对链接是否需要修正（路径深度变化）
- ✅ 配置文件优先使用实际可用的版本，而非模板
- ✅ 脚本文件保留可执行权限
- ❌ 不要复制大文件（>10MB）、二进制文件、敏感信息（密钥/token）
- ❌ 不要复制 build/、node_modules/ 等构建产物目录

#### 步骤4：修正复制文件中的相对路径

文件从原位置复制到 `cases/<pattern-id>/` 后，跨目录引用的相对路径会失效，需要统一修正：

| 引用目标 | 从 cases/<pattern-id>/ 出发的相对路径 |
|---------|-------------------------------------|
| 上级目录的其他模式 | `../../<other-pattern>.md` |
| 同案例目录内的文件 | `<filename>`（同级直接引用） |
| 模式主文档 | `../../<pattern-id>.md` |
| 本文件（cases/README.md） | `../README.md` |

修正后可运行链接检查验证：
```bash
python .agents/scripts/check-links.py --path ".agents/docs/retrospective/patterns/methodology-patterns/governance-strategy/cases"
```

#### 步骤5：更新模式文档

在模式主文档中做两处更新：

1. **更新 frontmatter**：`validation_count` 加1
   ```yaml
   validation_count: 2  # 从1增加到2
   ```

2. **添加案例章节**：在「## 实际案例」下新增案例
   ```markdown
   ### 案例N：<项目名称>（<验证日期>）
   
   | 指标 | 优化前 | 优化后 |
   |------|--------|--------|
   | ... | ... | ... |
   
   关键配置：<一句话说明>。完整材料见 [cases/<pattern-id>/](cases/<pattern-id>/README.md)
   ```

#### 步骤6：更新案例索引

在本文件（cases/README.md）的「现有案例索引」表格中新增一行：

```markdown
| [<pattern-id>/](<pattern-id>/README.md) | [<pattern-name>](../<pattern-id>.md) | <项目来源> | <YYYY-MM-DD> |
```

#### 步骤7：更新成熟度评估（如果满足条件）

当 `validation_count ≥ 2` 时，评估是否升级模式成熟度：

| 当前成熟度 | 升级条件 | 升级后 |
|-----------|---------|--------|
| L1-draft | validation_count ≥ 2 + 至少1个跨技术栈案例 | L2-formal |
| L2-formal | validation_count ≥ 5 + 跨≥3种技术栈 + 有反模式验证案例 | L3-validated |

升级时需更新：
- 模式 frontmatter 的 `maturity` 字段
- 模式末尾的「成熟度说明」章节
- TOML 元数据文件（`.meta/toml/.../<pattern-id>.toml`）的 `maturity` 字段

#### 步骤8：最终验证

```bash
# 1. 链接检查
python .agents/scripts/check-links.py --path ".agents/docs/retrospective/patterns/methodology-patterns/governance-strategy"

# 2. Mermaid 检查（如案例包含图）
python .agents/scripts/check-mermaid.py --path ".agents/docs/retrospective/patterns/methodology-patterns/governance-strategy/cases"
```

---

### 操作2：更新已有案例材料

案例归档后原则上**只读**，仅在以下情况更新：

| 更新场景 | 是否允许 | 操作方式 |
|---------|---------|---------|
| 修正断链/路径错误 | ✅ 允许 | 直接修正，无需审批 |
| 补充缺失的验证数据 | ✅ 允许 | 补充后在案例 README 末尾添加更新记录 |
| 修改原始配置内容 | ⚠️ 谨慎 | 案例是"历史快照"，建议新增案例而非修改旧案例 |
| 删除案例材料 | ❌ 禁止 | 案例一经归档不应删除，保留验证历史 |

补充数据时在案例 README 末尾添加：

```markdown
## 更新记录

- YYYY-MM-DD：补充了<什么>，原因：<为什么补充>
```

### 操作3：为新模式创建首个案例

萃取新模式时，同步完成案例归档：

1. 创建新模式文档（参考萃取流程）
2. 按操作1的步骤创建案例目录、归档材料
3. 模式的 `maturity` 设为 `L1-draft`
4. `validation_count` 设为 `1`
5. frontmatter 中必须包含 `source` 和 `case_archive` 字段：
   ```yaml
   source: "<原始文件路径>"
   case_archive: "cases/<pattern-id>/"
   ```

## 四、质量标准

### 案例材料合格标准

每个案例归档后应满足：

| 检查项 | 标准 |
|--------|------|
| 案例ID格式 | case-<pattern-id>-NNN，NNN为3位数字序号 |
| 量化指标 | 至少2个可对比的量化指标（时间/成功率/资源占用等） |
| 验证范围 | 明确标注哪些模式要点已验证、哪些未覆盖 |
| 环境规格 | 列出关键依赖的具体版本号（便于复现） |
| 可复现性 | 他人根据材料可在相似环境中复现类似效果 |
| 链接完整性 | 所有本地引用通过 check-links.py 检查，零断链 |
| 无敏感信息 | 不包含密钥、token、内部地址、个人隐私等 |

### 什么不算有效案例？

- ❌ 只有结论没有数据（"这个模式很好用"）
- ❌ 只描述做法没有效果对比（"我们这样配置了"）
- ❌ 与模式无关的项目（把不相关的项目材料塞进来凑数）
- ❌ 无法复现的环境特定技巧（依赖特定机器/特定人员的隐性知识）
- ❌ 虚拟案例/假设场景（"假如我们用在X项目上应该会有效"）

## 五、案例与模式成熟度的关系

```mermaid
flowchart LR
    A["新模式萃取"] -->| "validation_count=1" | B["L1-draft"]
    B -->| "+1个同技术栈案例" | C{"技术栈是否不同?"}
    C -->| "否" | B
    C -->| "是" | D["L2-formal<br/>validation_count≥2"]
    D -->| "+3个案例" | E{"跨≥3种技术栈?"}
    E -->| "否" | D
    E -->| "是" | F{"有反模式验证案例?"}
    F -->| "否" | D
    F -->| "是" | G["L3-validated<br/>validation_count≥5"]
    
    style B fill:#FFCDD2,stroke:#B71C1C
    style D fill:#FFF9C4,stroke:#F57F17
    style G fill:#C8E6C9,stroke:#2E7D32
```

| 成熟度 | 案例要求 | 可信度 | 适用场景 |
|--------|---------|--------|---------|
| **L1-draft** | 1个案例 | 低（可能是特例） | 建议在相似环境试用，注意验证 |
| **L2-formal** | ≥2个案例，跨≥1种技术栈 | 中（已证明可迁移） | 可在生产环境评估使用 |
| **L3-validated** | ≥5个案例，跨≥3种技术栈，含反模式案例 | 高（经过充分验证） | 可作为标准实践推广 |

## 六、常见问题

**Q: 同一个项目可以作为多个模式的案例吗？**
A: 可以。如果一个项目同时验证了多个模式，在每个模式的案例目录下分别归档该模式相关的材料即可。

**Q: 案例材料需要和原始文件保持同步吗？**
A: 不需要。案例归档的是"验证时的状态"，是历史快照。源文件后续更新不影响已归档案例。如果源文件更新后重新验证了模式，应该新增一个案例而非更新旧案例。

**Q: 案例目录可以放代码文件吗？**
A: 可以放关键的配置文件和脚本片段（如 Dockerfile、daemon.json、setup.sh），但不要放完整代码仓库。脚本应控制在单文件1000行以内，大文件考虑仅保留关键片段或链接到源位置。

**Q: 发现案例材料中有错误怎么办？**
A: 修正错误，在案例 README 的「更新记录」章节说明修正内容和原因。不要直接删除或覆盖原有内容，保持可追溯性。

**Q: 其他模式分类目录（如 code-patterns/、architecture-patterns/）也应该建 cases/ 目录吗？**
A: 是的。本维护指南同样适用于其他模式分类目录，可按需在各自目录下创建 cases/ 子目录，结构和维护流程相同。
