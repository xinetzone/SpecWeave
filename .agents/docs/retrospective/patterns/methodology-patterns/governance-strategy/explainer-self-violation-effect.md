# 讲解自犯效应（Explainer Self-Violation Effect）

## 模式类型

方法论模式（治理策略/元认知/认知科学）

## 成熟度

L2 初步验证（1次强验证来源：2026-07-10 Mermaid陷阱讲解中连续4次递归错误，每次错误精确验证正在讲解的规则）

## 适用场景

写文档、讲教程、做复盘、写规范、教学等"讲解某个陷阱/规则/最佳实践"的场景。在AI协作项目中，这是**最高频的场景之一**——写模式文档、复盘报告、规范指南都是核心工作。

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 写陷阱/反模式文档 | ✅✅✅ 最高危 | 刚解释完"不要做X"，自己的代码示例就做了X |
| 编写最佳实践指南 | ✅✅✅ 最高危 | 讲编码规范时自己的示例代码违反规范 |
| 复盘报告/教训总结 | ✅✅✅ 最高危 | 刚写完"这次因为X出错"，新写的复盘文档里又出现X |
| 代码Review/教学 | ✅✅ 高危 | Review别人代码时指出X问题，自己写的Review意见代码示例里就有X |
| 工具/脚本开发 | ✅✅ 高危 | 改进防错工具时，新代码里出现正在防的那个错误 |
| 日常编码（非讲解场景） | ⚠️ 间接适用 | 日常编码的"知道做不到"属于践行鸿沟模式，本模式专门针对讲解场景 |
| 创造性写作 | ❌ 不适用 | 创意工作不涉及"违反正在讲解的规则"的问题 |

## 问题背景

在修复+复盘Mermaid"Unsupported markdown: list"错误的过程中，出现了一个极其讽刺的现象：

> **在整个修复+复盘+模式归档过程中，连续4次犯了完全相同的Mermaid错误，每一次错误都精确验证了正在讲解的规则。**

| 错误序号 | 发生位置 | 错误内容 | 正在讲解的规则 |
|---------|---------|---------|--------------|
| 递归错误1 | quoting-scope-limits.md演示图 | 加了引号+`1. `+`\"`转义→渲染失败 | 引号不阻止列表触发（引号作用边界定律） |
| 递归错误2 | execution-retrospective.md修复前代码 | 用mermaid围栏包裹错误代码片段→渲染失败 | 错误示例不能用mermaid围栏 |
| 递归错误3 | execution-retrospective.md修复后代码 | 用mermaid围栏包裹不完整片段→渲染失败 | 不完整片段不能用mermaid围栏 |
| 递归错误4 | execution-retrospective.md错误代码示例 | 用mermaid围栏包裹故意写错的示例→渲染失败 | 错误示例不能用mermaid围栏 |

这不是"粗心"，而是有认知科学底层机制的。

## 核心机制：认知资源竞争模型

当你专注于"讲解X陷阱"时，你的认知资源分配出现结构性偏移：

```
写陷阱文档时的认知资源分配：
┌─────────────────────────────────────────┐
│  System 2（慢思考/理性）                │
│  占用90%注意力：组织语言、解释原理、    │
│  举例子、写元数据、想跨领域案例         │
├─────────────────────────────────────────┤
│  System 1（快思考/直觉）                │
│  自动执行写代码动作：                   │
│  "写Mermaid→加个引号吧→1. 这样写"       │
│  （完全没有System 2介入检查）           │
└─────────────────────────────────────────┘
```

**关键机制**：
1. System 2被"解释"任务完全占用，没有剩余资源执行"自我检查"
2. System 1自动执行写代码的动作，而System 1里存储的恰好是那个**错误的直觉**——那个你正在批判的直觉
3. 你正在讲解的规则越复杂、越需要注意力去组织语言，System 2剩余资源越少，犯错概率越高

这解释了为什么：
- 教别人安全编码的老师，自己写代码时可能犯安全错误
- 写测试最佳实践的人，自己的代码可能没测试
- 讲时间管理的人，自己可能拖延
- **讲Mermaid陷阱的人，自己写的Mermaid可能全部渲染失败**（本人亲测）

### 与践行鸿沟的关系

本模式是[践行鸿沟与递归践行定律](practice-gap-recursive-practice.md)在**讲解场景**下的特化：

| 维度 | 践行鸿沟（通用） | 讲解自犯效应（特化） |
|------|----------------|-------------------|
| 触发场景 | 任何简单任务 | **讲解某个陷阱/规则时** |
| 核心机制 | System 1默认接管 | **认知资源竞争**：System 2被解释任务占用，执行完全交给System 1 |
| 错误内容 | 刚学过的规则 | **正在讲解的规则**——你此刻最"知道"的那条规则 |
| 讽刺程度 | 高（刚学完就犯） | **极高**（正在讲就犯，错误本身就是讲解内容的活例证） |
| 防御难度 | 中（检查点可覆盖） | **高**（你高度专注于解释，完全想不到自己需要被检查） |

## 三个衍生推论

### 推论1：验证盲区效应

修复Bug时，你会认真验证"被修复的文件"，但往往**忘记验证你新创建的文件**（修复补丁、模式文档、复盘报告）。

**原因**：你的注意力锚定在"修复对象"上，形成认知隧道——新创建的文档/模式文件在你的注意力范围之外，即使它们包含和修复对象相同类型的代码（如Mermaid图表）。

**对策**：双验证规则——每次修复/修改后，必须同时验证**修复目标**和**新创建的产物**。

### 推论2：元语言陷阱

当你用X语言写"X语言的错误示例"时，解析器不知道你在写示例——它会照常执行/渲染。

**本质**：这是**语言层级混淆**问题——你在用X语言作为**元语言**（描述X语言），但解析器只把它当**对象语言**（执行X语言）。

**实例**：
- 用```mermaid围栏包裹"错误的Mermaid代码"来展示错误→Mermaid解析器尝试渲染→渲染失败
- 用```python包裹"会抛异常的Python代码"来展示常见错误→Python执行器（如果有）会真的抛异常
- 在Markdown中写"这里要写```"但没有正确转义→Markdown解析器会真的认为代码块开始/结束

**对策**：用显式机制区分"展示的代码"和"执行的代码"——不完整片段/错误示例用普通围栏，只有完整可运行的代码才用语言围栏。

### 推论3：警惕性不跨上下文

你在"Bug修复模式"下高度警惕每一个`1. `，但当你切换到"文档写作模式"，警惕性就消失了。

**原因**：认知警惕性是绑定在特定任务上下文中的，不会自动迁移到新的任务上下文。模式切换时，大脑会加载新的任务脚本（schema），旧的检查清单被卸载。

**对策**：模式切换时显式重载检查清单——从"修复模式"切换到"文档模式"时，重新执行一遍相关检查清单，不要假设警惕性会跟随你。

### 推论4：自指盲区（递归讽刺）

当你分析/论述某个**质量保障机制**（V对抗审查、测试、验证、checklist）自身时，最容易犯该机制所防范的完全相同的错误——而且你用来验证自己产出的方法，恰好就是你正在分析的那个有缺陷的方法。

**实例**（2026-07-14 WorkBuddy文章分析）：
- 报告V章节深入论证"Generator和Evaluator共享误解导致错的实现+全部通过的测试"
- 报告自身的验证（checklist 35/35全过）恰好是"Generator和Evaluator共享误解"的实例——组装子代理既做生成又做自检
- 结果：报告深入分析V机制，但V未能检测出报告自身的18处事实错误

**本质**：这是比讲解自犯更高阶的递归——不仅是"正在讲解X时犯X错误"，而是"正在讲解X的失效模式时，恰好以X的失效模式失效"。你的验证逻辑和产出逻辑使用相同的上下文、相同的假设、相同的盲点，形成了认知上的"不可能自证"。

**为什么推论1-3的对策不够**：
- 双验证规则（推论1）：验证的是"新创建文件"，但这里的问题是验证方法本身有缺陷
- 显式围栏（推论2）：适用于代码/语法级错误，不适用于事实性/语义级错误
- 重载检查清单（推论3）：如果检查清单本身就是问题的一部分（只查结构不查事实），重载无效

**对策**：
- **独立异体验证**：涉及质量保障主题的产出，最终验证必须由**未参与产出**的独立Agent执行，不给中间推理，只给原始素材+最终产出+核查指令
- **反向应用测试**：把你正在论述的防御方法，反向应用到你自己的产出上。你写V审查，就用V审查V；你写checklist验证，就用checklist验证checklist本身。明确记录"我声称X能防御Y，我具体是如何用X防御X自身的？"
- **否定测试**：对你的核心论点取反，如果取反后的陈述恰好描述了你产出中的缺陷，你正处于自指盲区中。例如：主张"V防止共享误解"→取反"V未能防止共享误解"→检查是否存在角色不分离→确实存在→必须修复
- 详细防御检查清单见 [.agents/checklists/self-reference-blindspot-defense.md](../../../../../.agents/checklists/self-reference-blindspot-defense.md)

## 四个衍生推论总结

| 推论 | 核心问题 | 一句话对策 |
|------|---------|-----------|
| 推论1：验证盲区 | 忘记验证新创建的文件 | 修复目标+新产物双验证 |
| 推论2：元语言陷阱 | 错误示例被解析器执行 | 片段/错例用普通围栏，完整代码才用语言围栏 |
| 推论3：警惕性不迁移 | 模式切换时警惕性消失 | 切换时显式重载检查清单 |
| 推论4：自指盲区 | 分析验证机制时被该机制击败 | 独立异体验证+反向应用+否定测试 |

## 反模式识别信号

当你出现以下想法时，你正处于高风险状态：

| 危险信号 | 为什么危险 |
|---------|----------|
| "我刚讲完这个问题，我肯定不会犯" | 这是认知偏差——你正在讲，恰恰说明System 2被占用，System 1最可能自动执行错误模式 |
| "这个代码是用来展示错误的，不用检查" | 解析器不知道你在展示错误，它会照常执行/渲染 |
| "我只是写个文档/示例，不会有问题" | 文档中的代码也是代码，和产品代码一样需要检查 |
| "这个是新创建的文件，应该没问题" | 验证盲区——新文件最容易被遗漏验证 |
| "我刚修好那个Bug，不会再犯了" | 递归践行——刚修好时正是最容易在新地方再犯的时候 |
| "我在分析V机制/验证方法，checklist全过了应该没问题" | 自指盲区——你用来验证的方法恰好是你正在分析的方法，共享上下文导致共享误解 |
| "这个checklist验证了所有项都✅，内容肯定正确" | 结构验证≠事实验证，checklist查"有没有"不查"对不对" |

## 核心对策：自动化验证覆盖一切

唯一可靠的解决方案不是"更小心"，而是**让自动化工具检查所有文件，包括你新创建的文档**——人会因为上下文切换、注意力分散、认知资源占用而犯错，但自动化工具不会。

### 四层防御（针对讲解场景特化）

| 防御层 | 机制 | 针对的推论 |
|--------|------|-----------|
| 第一层：显式围栏区分 | 完整可运行代码用语言围栏，片段/错误示例用普通围栏 | 推论2（元语言陷阱） |
| 第二层：产物双验证 | 修复目标+新创建文件同时验证 | 推论1（验证盲区） |
| 第三层：模式切换检查点 | 任务模式切换时显式重载检查清单 | 推论3（警惕性不跨上下文） |
| 第四层：独立异体验证+否定测试 | 质量保障主题产出由未参与的独立Agent验证；核心论点取反检查自指盲区 | 推论4（自指盲区/递归讽刺） |

### 核心原则

**工具验证优于自我提醒**。把"避免X错误"这件事交给工具，不要交给你的记忆力和注意力。当你在讲X陷阱时，你的注意力是最不可靠的。

## 反模式

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| 写完陷阱文档后只看一遍就提交 | System 2已经被解释任务耗尽，自审时仍然在"解释模式"而非"检查模式" | 运行自动化工具（lint/check脚本），不依赖自审 |
| 错误示例用语言围栏（如```mermaid） | 解析器不知道你在展示错误，会尝试渲染 | 错误示例/不完整片段用普通围栏 |
| 修复后只验证被修复的文件 | 新创建的文档/模式文件包含相同类型的代码，同样可能出错 | 双验证：修复目标+新产物都要检查 |
| "我在写文档，不需要运行测试" | 文档中的代码块也是代码，同样会被解析/渲染/执行 | 对文档也运行相应的检查工具 |
| 写完教训后立刻开始下一个任务 | 模式切换时警惕性不迁移，下一个任务中可能再犯 | 提交前运行自动化验证，作为模式切换的强制检查点 |
| 把错误归因为"太粗心了" | "粗心"不是根因，认知资源竞争才是——问题在于流程允许这种错误发生 | 用5-Whys追根因：为什么工具没有覆盖到这个文件？ |
| 分析验证机制时靠自检通过就放心 | 自指盲区——你用来验证的方法可能就是你正在分析的那个有缺陷的方法 | 委派独立Agent验证，并执行否定测试（核心论点取反检查自身缺陷） |
