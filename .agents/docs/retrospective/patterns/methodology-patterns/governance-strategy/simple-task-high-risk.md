# 简单任务高风险定律（Simple Task High-Risk Law）

## 模式类型

方法论模式（治理策略/认知科学）

## 成熟度

L2 已验证（5次验证来源：2026-07-09格式修正批量套用错误、2026-07-10目录链接类比错误、2026-07-10路径层级直觉错误、2026-07-10写洞察时路径错误、spec引用验证错误）

## 适用场景

在执行任何"看起来简单、不用想、机械重复"的任务时，主动防范高错误风险。适用于：

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 格式批量更新 | ✅✅✅ 核心场景 | 链接格式统一、错别字修正、重命名等机械任务 |
| 配置项批量修改 | ✅✅✅ 核心场景 | 批量改路径、批量改参数、批量改配置值 |
| 文档批量迁移 | ✅✅✅ 核心场景 | 文件移动、目录重组、引用路径更新 |
| "改一行代码"类小修复 | ✅✅ 强烈推荐 | 越是觉得"只改一行不用测试"，越容易出问题 |
| 重复性数据录入 | ✅✅ 推荐 | 表格填写、数据迁移、批量录入 |
| 复杂任务（架构设计/新功能） | ❌ 不适用 | 复杂任务本身有高认知投入，自然会验证，本模式主要防范"过度自信跳过验证" |

## 问题背景

简单任务是错误的高发区——不是因为任务本身难，而是因为**大脑会自动降低警惕**：

1. **认知负荷自动降低**：大脑识别到"简单任务"标签后，自动切换到System 1（直觉/类比/快思考）模式，消耗最少认知资源
2. **验证环节被主动跳过**："这么简单的事，查规范/跑测试是浪费时间"——过度自信导致验证流程被省略
3. **类比推理自动激活**：看到"格式统一"任务，大脑自动从记忆中提取"最近看到的格式"，直接套用，不验证是否匹配当前场景
4. **批量复制放大错误**：简单任务往往批量执行（一次性改10+个文件），一个错误被快速复制到所有文件
5. **错误潜伏时间长**：简单任务的错误往往不影响核心功能（链接在本机还能点、参数名看起来对），不容易被发现

### 主观难度反比定律

这是与直觉完全相反的规律：

> **错误率与任务主观感知难度成反比，而非成正比。**

"简单"不是任务的客观属性，而是大脑的主观判断——当大脑判断"这个我熟、不用想"时，就是风险最高的时候。

| 任务类型 | 认知投入 | 主动验证倾向 | 流程保护 | 实际错误风险 |
|---------|---------|-------------|---------|-------------|
| 复杂任务（架构设计、新功能） | 高 | 主动写Spec、跑测试、Review | 强（多层流程保护） | **低** |
| 简单任务（格式更新、改错别字、写引用路径） | 低 | "这么简单不会错"，跳过验证 | 弱（无流程保护，全靠自觉） | **高** |

**反直觉解释**：
- 复杂任务因为"看起来难"，人们会主动启动System 2、走完整流程（spec→TDD→评审→验证），反而不容易出错
- 简单任务因为"看起来简单、不用想"，大脑判断"不需要消耗认知资源"，所有验证环节被主动跳过，恰恰是错误高发区
- 越是心里想"这还用查？"的时候，越必须查——这是最高危的信号

## 核心规则

### 规则1：简单任务 = 高风险任务，必须主动降速

改变认知标签：不要把任务标记为"简单"，要标记为"高风险——需要验证"。

- ❌ 错误心态："这个任务这么简单，5分钟搞定"
- ✅ 正确心态："这个任务看起来简单，简单任务最容易出错，我需要放慢速度、逐个验证"

### 规则2：简单任务必须走"三件套"验证流程

即使是"改一个字"的任务，也要执行最小验证流程：

| 验证项 | 检查内容 | 为什么必须 |
|--------|---------|-----------|
| **查规范** | 这个格式/路径/写法在规范中是怎么规定的？ | 记忆不可靠，特别是"最近看到的"可能来自错误场景 |
| **查实例** | 项目中同类文件实际用的什么写法？（至少看2-3个例子，不要只看一个） | 单个例子可能本身就是错的，多看几个才能发现规律 |
| **小批量验证** | 先改1个文件，验证正确后再批量改 | 不要一上来就批量改10个文件，先做小范围验证 |

### 规则3：批量操作必须设置"检查点"

当批量操作N≥5个对象时，必须设置中间检查点：

1. **P0试点**：先改1个文件，完整验证（跑检查工具、人工核对）
2. **P1小批量**：再改2-3个文件，再次验证
3. **P2推广**：确认无误后再批量修改剩余文件
4. **P3终验**：全部改完后跑一次全量验证（如check-links.py）

### 规则4：警惕"我刚做过类似的"——这是类比推理的信号

当你心里想"这个我刚做过，直接套就行"的时候，**这是最高危的信号**：

- 这说明你的大脑正在走类比推理捷径
- 必须停下来问："两个场景真的完全一样吗？有没有什么细微差别？"
- 最好的方法：直接查规范/查实例，不要凭"刚做过"的记忆

## 四阶段错误放大模型

简单任务的错误不是单点发生的，而是有一个放大链条：

```mermaid
flowchart LR
    A["阶段1：认知标签<br/>'这个任务简单'"] --> B["阶段2：System1接管<br/>类比/直觉/捷径"]
    B --> C["阶段3：跳过验证<br/>'不用查也知道'"]
    C --> D["阶段4：批量复制<br/>一次性改N个文件"]
    D --> E["结果：错误被放大N倍<br/>且潜伏不被发现"]
    style A fill:#FFF3CD
    style B fill:#FFE5B4
    style C fill:#F8D7DA
    style D fill:#F5C6CB
    style E fill:#DC3545,color:white
```

| 阶段 | 心理活动 | 防范措施 |
|------|---------|---------|
| 阶段1：认知标签 | "这个太简单了" | 主动重标记："简单任务=高风险任务" |
| 阶段2：System1接管 | "凭直觉/记忆/类比就行" | 强制启动System2：问自己"这是事实还是类比？" |
| 阶段3：跳过验证 | "查规范是浪费时间" | 强制执行"三件套"：查规范/查实例/小批量验证 |
| 阶段4：批量复制 | "一次性改完效率高" | 设置检查点：P0→P1→P2→P3渐进推广 |

## 反模式

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| "这么简单，不用查规范" | 简单任务正是规范最容易被违反的时候——因为你觉得"我知道规范是什么" | 越是简单任务，越要花30秒查一下规范 |
| "我刚做过类似的，直接套" | 场景可能有细微差别（路径层级不同、上下文不同），类比推理最容易在"看起来一样"时出错 | 每个任务都当第一次做，查2-3个现有实例对比 |
| "一次性改完再验证" | 错误已经被复制到N个文件，修改成本线性上升 | 先改1个验证对了，再批量改 |
| "只改一行不用跑测试" | 一行代码的改动可能破坏整个链路，历史上无数P0故障都是"只改了一行" | 改一行也要跑相关测试，至少跑静态检查 |
| "改完了，应该没问题" | 人类的自审偏差率极高，特别是简单任务 | 用自动化工具验证（check-links.py、lint、类型检查），不要相信"应该没问题" |
| 发现错误后只改当前文件 | 简单任务错误往往是批量的，可能还有其他文件有同样错误 | 发现一个错误后，Grep全局搜索类似模式，检查是否有其他文件被同样感染 |

## 实际案例

### 案例1：链接格式批量套用错误（初始验证）

| 项目 | 内容 |
|------|------|
| 时间 | 2026-07-09 |
| 任务 | "把所有旧链接更新为统一格式" |
| 错误决策 | 看到`file:///`格式（AI对话输出格式），直接套用到13个Markdown文档的内部链接 |
| 错误根源 | 混淆了两个场景：系统提示要求"AI对话输出用file:///"≠"Markdown文档内部用file:///" |
| 为什么简单任务出错 | 1. 觉得"格式统一"是简单任务 → 2. 看到file:///就类比套用 → 3. 没查开发规范 → 4. 一次性批量改13个文件 |
| 后果 | 115个链接格式错误，18分钟后才被用户质疑发现 |
| 修正 | 全部回退为相对路径，跑check-links.py验证 |

### 案例2：目录链接类比套用错误（第二次验证）

| 项目 | 内容 |
|------|------|
| 时间 | 2026-07-10 |
| 任务 | "更新文档间的交叉引用" |
| 错误决策 | 看到README.md链接到目录，就类比套用到新链接中，直接链接到目录而非具体文件 |
| 错误根源 | IDE需要指向具体.md文件才能打开，链接到目录会显示"此文件是目录"无法打开 |
| 为什么简单任务出错 | 觉得"改链接"是小事，凭最近的记忆类比，没查现有链接实例验证 |
| 后果 | 多个目录链接无效，用户点击无法到达预期内容 |
| 修正 | 升级check-links.py增加目录链接检测，自动修复为dir/README.md |

### 案例3：路径层级直觉计算错误（第三次验证）

| 项目 | 内容 |
|------|------|
| 时间 | 2026-07-10 |
| 任务 | "创建新模式文件，添加相对路径引用" |
| 错误决策 | 凭直觉数层级写了`../../`，而不是查现有同类文件的路径写法 |
| 错误根源 | 路径层级计算看起来"不用想"，但直觉数层级很容易数错，特别是跨多层目录时 |
| 为什么简单任务出错 | "数层级"是典型的简单任务，大脑觉得"这还能错？"，直接凭直觉写，不验证 |
| 后果 | 引用路径错误，链接失效 |
| 修正 | 建立"查现有实例而非数层数"原则，强制复制同类文件的路径写法 |

### 案例4：写洞察时递归犯错（第四次验证）

| 项目 | 内容 |
|------|------|
| 时间 | 2026-07-10 |
| 任务 | "正在写'不要凭直觉数路径层级'的关键洞察" |
| 错误决策 | 写洞察的过程中，创建新文件引用又凭直觉写错了路径 |
| 错误根源 | 注意力集中在组织洞察内容（System 2忙于写作），路径计算这种"简单任务"完全交给System 1自动执行 |
| 为什么简单任务出错 | 这是践行鸿沟的递归表现——正在讲解"不要犯X错误"的时候，X错误最容易发生，因为注意力被占用了 |
| 后果 | 刚写完的教训自己立刻违反，形成递归验证 |
| 修正 | 这证明了"简单任务高风险"不是态度问题，是认知架构问题——必须用流程和工具防御，不能靠"记住教训" |

### 案例5：spec引用路径错误（第五次验证）

| 项目 | 内容 |
|------|------|
| 时间 | 2026-07-10 |
| 任务 | "在spec中添加引用路径" |
| 错误决策 | 觉得"写引用是小事"，没有验证引用的文件是否存在 |
| 错误根源 | `self-cognition.md`不存在，但因为写引用"看起来太简单"，没做存在性验证 |
| 为什么简单任务出错 | spec阶段不验证引用，等到实施阶段才发现文件不存在，需要回溯修正 |
| 后果 | 实施阶段发现引用错误，回溯修正spec |
| 修正 | 沉淀spec-reference-validation.md模式，强制spec引用必须做存在性验证 |

### 跨案例对比：复杂任务反而不出错

| 任务类型 | 错误数量 | 原因 |
|---------|---------|------|
| 知识图谱开发（复杂） | 0 | 采用TDD，29个测试保护，走完整流程 |
| v1.7四大新章节创作（复杂） | 0 | 有对抗性审查和checklist保护，走完整流程 |
| 格式更新/路径计算/写引用（简单） | 5次 | 所有验证环节被主动跳过 |

这完美验证了主观难度反比定律：复杂任务有流程保护反而不出错，简单任务因为"看起来简单"跳过所有保护，反而错误高发。
