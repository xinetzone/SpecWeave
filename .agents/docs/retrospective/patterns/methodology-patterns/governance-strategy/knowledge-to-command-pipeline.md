# 知识库→指令集转化管道（Knowledge-to-Command Pipeline）

## 模式类型
方法论模式（治理策略/流程管理）

## 成熟度
L1 实验级（1次验证：对抗性审查指令集创建任务）

## 适用场景

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 知识库建成后评估 | 核心场景 | 知识库已完成，需要评估是否创建指令集 |
| 方法论采纳率低 | 核心场景 | 知识库存在但使用频率低，需要降低使用成本 |
| 知识库→可执行工具转化 | 核心场景 | 将"参考文档"转化为"可执行流程" |

## 核心步骤

```
步骤1: 盘点知识库资产
  ├── 检查清单（可直接复制使用的）
  ├── 流程模板（需要按步骤执行的）
  ├── 评分表/判定标准（需要决策的）
  └── 验证日志模板（需要记录的）

步骤2: 第一性原理分析
  ├── 是否需要指令集？（公理：使用成本↓→采纳率↑）
  ├── 指令集 vs Skill？（认知方法→指令集；自动化工具→Skill）
  └── 设计要点（场景分支、元审查、知识库引用）

步骤3: 参照现有模板
  └── 参照 first-principles.md 的结构（触发条件→RACI→步骤→输出→验收→约束）

步骤4: 设计场景分支
  ├── 单场景：线性步骤
  └── 多场景：独立章节，各含完整步骤

步骤5: 引用知识库
  ├── 引用而非复制（保持单一可信源）
  └── 每个引用附带一句话说明（降低导航成本）

步骤6: 质量验收
  ├── 文件名检查
  ├── RACI合规（A唯一性、R≠A分离、双列设计）
  └── 知识库引用完整性
```

## 判断标准：是否需要创建指令集

| 判断维度 | 需要指令集 | 不需要指令集 |
|----------|-----------|------------|
| 知识库规模 | 10+文件，用户需自行导航 | 1-2个文件，流程显而易见 |
| 执行复杂度 | 多步骤、多场景、需要决策分支 | 单一步骤、无需决策 |
| 使用频率 | 预期高频使用 | 一次性或极低频使用 |
| 偏差风险 | 执行偏差会导致严重后果 | 执行偏差影响可控 |

## 判断标准：指令集 vs Skill

| 核心操作类型 | 选择 | 示例 |
|------------|------|------|
| 引导AI进行结构化思考 | 指令集 | 第一性原理、对抗性审查、复盘 |
| 执行Python脚本/自动化流程 | Skill | forum-posting、link-check、archive-folder |
| 两者都有 | Skill（双方案模式） | 既有脚本又有决策树 |

## 验证来源

- **对抗性审查指令集**（2026-07-10）：知识库（15文件、3456行）→ 指令集（283行），含场景分支、元审查、12个知识库引用
- **第一性原理指令集**（早期）：知识库（11文件）→ 指令集，含6步执行流程、RACI矩阵、6个知识库引用

## 注意事项

1. **不是所有知识库都需要指令集**：小规模、单一步骤的知识库不需要指令集，强制创建会增加维护负担
2. **指令集不应复制知识库内容**：引用而非复制，保持单一可信源
3. **指令集是活的**：知识库更新后，指令集的引用和流程可能需要同步更新
