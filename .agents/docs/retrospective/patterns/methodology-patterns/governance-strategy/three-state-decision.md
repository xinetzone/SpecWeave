# 三态决策模式：做/明确不做/推迟（Three-State Decision Model）

## 模式类型

方法论模式（治理策略/决策质量/Backlog管理）

## 成熟度

L1 实验性（1次端到端验证：在对抗性审查项目复盘中明确决策"不做"pre-commit链接检查，避免了无效投入，待更多场景验证）

| 验证指标 | 对抗性审查知识库复盘 | 说明 |
|---|---|---|
| 待决策项总数 | 5条改进建议 | ACT-001~ACT-005 |
| ✅ 决策"做" | 3条 | ACT-001/002/003已执行 |
| ❌ 决策"明确不做" | 1条 | ACT-004（pre-commit链接检查） |
| ⏸️ 决策"推迟" | 1条 | ACT-005（待后续场景验证） |
| Backlog残留 | 0条 | 所有项都有明确决策，无模糊项 |
| 避免无效投入估计 | 约2-3人天 | 明确不做pre-commit链接检查，避免了错误的优化方向 |

## 适用场景

| 场景 | 是否适用 | 说明 |
|---|---|---|
| 复盘后的改进建议处理 | ✅ 核心场景 | 本模式的起源场景 |
| 需求评审后的待决策项 | ✅ 核心场景 | 每个需求都要有明确去向 |
| 技术方案评审的选项决策 | ✅ 适用 | 明确选哪个、不选哪个、为什么 |
| Backlog梳理/优先级排序 | ✅ 适用 | 避免Backlog变成"永不做"的垃圾场 |
| 技术债务处理决策 | ✅ 适用 | 哪些债要还、哪些明确不还、哪些以后还 |
| 日常快速决策且无长期影响 | ❌ 过度设计 | 简单决策不需要正式三态 |
| 紧急/突发情况处理 | ❌ 不适用 | 紧急情况下先执行，事后再补决策记录 |

## 核心问题

为什么需要三态决策？两态（做/不做）或单态（默认全做）有什么问题？

### 问题1：默认全做→Backlog垃圾场→虚假闭环

最常见的反模式：评审/复盘结束后，把所有建议都放入Backlog，然后：
- Backlog越来越长，90%的条目永远不会被执行
- 没人记得这些条目为什么在那里、当时的决策背景是什么
- 同样的建议下次复盘还会被提出来，因为没有明确说过"不做"
- Backlog从"待办清单"变成"想法垃圾场"

**虚假闭环的典型表现**："这些建议都记录在Jira/Backlog里了"——但实际上永远不会做，这不是闭环，是拖延。

### 问题2：跳过/忽略不做的建议→反复讨论→决策债务

另一个极端：对于不想做的建议，直接跳过/忽略，不明确说"不做"：
- 过段时间换个人又会提出同样的建议
- 同样的讨论反复进行，浪费团队时间
- 建议提出者不知道为什么没被采纳，感到不被尊重
- "为什么不做"的决策理由没有记录，新成员无法了解背景

**决策债务的典型表现**："这个建议上次好像提过，后来怎么样了？"——没人记得，也查不到决策记录。

### 问题3：两态决策（做/不做）的边界问题

简单的"做/不做"二元决策也有问题：
- 有些建议确实有价值，但现在时机不对/优先级不高/需要等前置条件
- 如果勉强决策"不做"，以后条件成熟时可能忘记
- 如果勉强决策"做"，会挤占当前高优先级任务的资源
- "以后再说"是一种状态，但需要明确记录，而不是模糊状态

### 核心洞察：三种状态，三种闭环，缺一不可

> **三态决策的第一性原理思考**：
> 1. 对于**值得现在做**的建议→✅ **做**：明确优先级、执行方案、负责人、时间节点
> 2. 对于**经过分析确认不值得做**的建议→❌ **明确不做**：明确记录理由，永久关闭，以后不再讨论（除非条件发生重大变化）
> 3. 对于**有价值但现在不做**的建议→⏸️ **推迟**：明确推迟条件/触发时机、回顾时间点，放入"待触发"清单而非主Backlog
>
> **关键**：三种状态都是"闭环"，没有第四种状态（"没有明确决策"是不允许的）。

## 状态一：✅ 做（Do）——清晰执行

### 判断标准

满足以下条件的建议应该决策为"做"：
1. 符合第一性原理（从根本原理推导应该做，不是"别人都做"）
2. ROI清晰（投入产出比可衡量，值得投入资源）
3. 当前优先级足够（比其他待做事项更重要/更紧急）
4. 没有无法解决的风险/障碍

### 决策记录要求

决策"做"时必须记录：
- **具体执行内容**：不是"优化一下"这种模糊描述，而是具体做什么
- **优先级**：P0/P1/P2，或者在当前迭代/Backlog中的位置
- **负责人**：谁来做（可以是"待定"但要明确什么时候定）
- **时间节点**：什么时候开始、什么时候完成（或哪个迭代/版本）
- **成功标准**：怎么算做完了、做好了
- **依赖关系**：有没有前置条件需要满足

### 处理方式

- 纳入当前执行计划或近期Backlog
- 按优先级分配资源
- 定期跟踪进度直到完成

## 状态二：❌ 明确不做（Explicitly No）——永久关闭

这是最容易被忽略、但价值最大的状态。

### 判断标准（什么情况下应该明确说"不做"）

经过第一性原理分析，满足以下任一条件的建议应该决策为"明确不做"：

1. **违反分层防御/架构原则**
   - 例子：把慢操作放在快通道（如pre-commit里放全量链接检查）
   - 第一性原理：pre-commit是<5秒的快通道，全量链接检查是慢IO操作，混在一起会破坏用户体验导致绕过

2. **ROI为负或极低**
   - 投入大但收益极小
   - 问题发生概率极低，防御成本远高于损失
   - 例子：为极端边缘情况增加复杂的防御逻辑

3. **已有其他机制覆盖**
   - 现有机制已经能解决这个问题，新增机制是冗余的
   - 例子：CI已经做了全量链接检查，pre-commit再做一遍是重复建设

4. **方向错误/与目标相悖**
   - 建议的方向与项目/产品目标不一致
   - 看起来合理但实际上会把系统带向错误方向

5. **被证伪/已有反例证明无效**
   - 行业实践或已有经验证明这个做法无效
   - 有明确的反面案例

### 决策记录要求（关键！不记录等于没决策）

决策"明确不做"时**必须详细记录**：
- **决策结论**：清晰的"不做XXX"，不要模棱两可
- **核心理由**：为什么不做（从第一性原理/架构原则/ROI等角度说明）
- **分析过程**：当时是怎么分析的、考虑了哪些因素、参考了什么
- **决策背景**：什么时候、在什么场景下做的决策、参与人
- **什么情况下可能重新考虑**：如果条件发生什么重大变化，可以重新讨论（避免绝对化）

**示例（ACT-004）**：
> ❌ **明确不做**：在pre-commit钩子中加入全量链接检查
>
> **理由**：
> 1. 分层防御原则：pre-commit是快通道（<5秒），全量链接检查是慢操作（网络IO+全目录遍历）
> 2. 用户体验：慢的pre-commit会导致开发者使用`--no-verify`绕过，反而破坏整个钩子体系
> 3. 已有覆盖：CI流水线已有全量链接检查，commit时link-check-cmd做即时验证（仅检查本次变更相关文件），双层覆盖足够
>
> **分析过程**：最初想"加一层检查更安全"，但用第一性原理重新审视分层防御架构后发现问题
>
> **重新考虑条件**：除非未来出现"链接错误大量逃逸CI且commit时即时检查无法覆盖"的情况，否则不再讨论

### 处理方式

- **永久关闭**这个建议/选项，不放入Backlog
- 在决策日志/复盘报告中明确记录
- 下次有人提出同样建议时，直接指向决策记录，不需要重新讨论（除非触发了"重新考虑条件"）
- 这不是"永远不做"的绝对承诺，而是"基于当前信息和条件，明确不做，且记录理由"

## 状态三：⏸️ 推迟（Defer）——有条件等待

### 判断标准（什么情况下应该推迟）

满足以下条件的建议应该决策为"推迟"：
1. 建议本身有价值、方向正确
2. 但**当前时机不成熟**（前置条件未满足、优先级不够、需要更多信息）
3. 不是"不做"，而是"以后做"
4. 推迟是有明确条件的，不是无限期等待

### 决策记录要求

决策"推迟"时必须记录：
- **建议内容**：推迟的是什么
- **为什么推迟**：现在不做的原因是什么
- **触发条件**：**什么条件满足时重新激活**（必须具体可验证，不能是"以后有空再说"）
- **回顾时间点**：什么时候回顾一次（如"1个月后回顾"、"下个版本规划时"）
- **当前障碍**：是什么阻止了现在做

**好的触发条件示例**：
- ✅ "当知识库文档数量超过50篇时"
- ✅ "当出现至少2个场景需要这个能力时"
- ✅ "当X功能上线后1个月，收集到用户反馈后"
- ❌ "以后有时间的时候"（无法验证，等于无限期推迟）
- ❌ "优先级高的时候"（循环定义）

### 处理方式

- 不放入主Backlog，放入单独的"待触发清单"（Deferred List）
- 指定回顾时间点，到期自动提醒重新评估
- 触发条件满足时，重新拿出来评估是否进入"做"状态
- 如果触发条件一直不满足、回顾时发现价值已经消失，可以转为"明确不做"

## 决策流程：如何应用三态模型

```
步骤1：列出所有待决策项
   ↓
步骤2：逐项用第一性原理分析
   - 从根本原理推导，不是"别人都做"
   - 考虑架构原则/ROI/现有覆盖
   ↓
步骤3：判断属于哪一态
   ├─ 值得现在做？→ ✅ 做（明确执行细节）
   ├─ 不值得做/方向错误？→ ❌ 明确不做（详细记录理由）
   └─ 有价值但现在不做？→ ⏸️ 推迟（明确触发条件）
   ↓
步骤4：所有项必须有明确状态，不允许模糊项
   ↓
步骤5：记录决策，同步给相关人
   - "做"的进入执行跟踪
   - "不做"的进入决策日志
   - "推迟"的进入待触发清单
   ↓
步骤6：定期回顾
   - "做"的跟踪进度直到完成
   - "推迟"的到期/触发条件满足时重新评估
```

## 反模式（需要避免的做法）

| 反模式 | 后果 | 正确做法 |
|---|---|---|
| 默认全部接受，全放入Backlog | Backlog膨胀成垃圾场，虚假闭环，大部分永远不做 | 逐项决策，"不做"也是闭环 |
| 不想做的就跳过/忽略 | 决策债务，同样问题反复讨论，浪费时间 | 明确说"不做"并记录理由 |
| "以后再说"作为第四种状态 | 无限期推迟，永远不会执行，不如明确"不做" | "推迟"必须有明确触发条件和回顾时间，否则应该"不做" |
| 决策"不做"但不记录理由 | 过段时间没人记得为什么不做，又重新讨论 | 详细记录分析过程、理由、背景、重新考虑条件 |
| 把"推迟"当"不做"的委婉说法 | 虚假推迟，实际永远不会做 | 要么明确"不做"，要么真的设触发条件和回顾时间 |
| 把"不做"当永久否决 | 条件变化时无法重新考虑 | 记录"什么情况下可以重新考虑"，条件重大变化时可以重开 |
| 一个人偷偷决策"不做" | 决策不透明，团队不理解、不买账 | 重要决策需要团队讨论，至少要告知相关人理由 |
| 为了"政治正确"不敢说"不做" | 什么都答应但什么都做不好，团队耗尽 | 诚实说"不做"比虚假承诺更负责任 |

## 效果评估指标

| 指标 | 理想值 | 测量方式 |
|---|---|---|
| 待决策项闭环率 | 100% | 评审/复盘后所有项都有明确三态之一 |
| Backlog中"僵尸条目"比例 | <10%（超过3个月没动的条目） | 定期Backlog梳理统计 |
| 重复建议出现率 | 显著下降 | 同样的建议被重复提出的频率 |
| 决策记录完整率 | 100%"不做"和"推迟"有记录 | 检查决策日志 |
| "推迟"项激活/转化率 | 定期有回顾 | 待触发清单回顾执行情况 |
| 团队对决策的理解度 | 团队成员知道"为什么做/为什么不做" | 团队反馈 |
| 避免无效投入 | 可量化的时间/人力节省 | 记录明确"不做"避免的投入 |

- **prove-usefulness-check.md**："先证明有用再推广"原则是"推迟"判断的重要依据——还没证明有用的应该先推迟或小范围验证
- **exemption-mechanism-legalization.md**：豁免机制合法化是"明确不做"的制度化版本——对于不遵守某些规则的情况，明确豁免并记录
- **three-tier-governance.md**：三层治理模型提供了不同层级决策的权限划分，与本模式结合使用

## 关键洞察总结

1. **"不做"也是闭环**：明确说"不做"并记录理由，比虚假放入Backlog或忽略更负责任
2. **没有第四种状态**："以后再说/再说吧/研究一下"不是决策，是拖延。所有待决策项必须归入三态之一
3. **决策记录是关键**：尤其是"不做"的决策，如果不记录理由，等于没决策，以后还会反复讨论
4. **第一性原理是判断依据**：做不做不应该看"别人有没有做"，而应该从根本原理、架构原则、ROI推导
5. **推迟≠不做，但需要条件**："推迟"必须有具体可验证的触发条件和回顾时间，否则就是变相的"不做"
6. **"不做"不是永久否决**：记录"什么情况下可以重新考虑"，条件发生重大变化时可以重开
7. **Backlog不是垃圾桶**：Backlog里只应该放真正"要做"的事项，"推迟"的放待触发清单，"不做"的进决策日志
8. **说"不"需要勇气，但更需要透明**：诚实说"不做"并解释理由，比什么都答应但什么都做不好更能建立信任
9. **避免无效投入是最大收益**：明确"不做"可以节省大量原本会浪费的时间和资源，这些资源可以投入到真正有价值的事情上
10. **三态都是负责任的决策**："做"、"不做"、"推迟"没有高下之分，都是基于当前信息的负责任判断，关键是明确和透明
