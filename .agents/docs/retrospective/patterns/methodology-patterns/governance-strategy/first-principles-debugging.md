# 第一性原理调试法（First-Principles Debugging）

## 模式类型

方法论模式（治理策略/调试方法论/问题解决）

## 成熟度

L2 初步验证（2次验证来源：2026-07-10 Mermaid列表错误六步推导链实战；第一性原理Prompt模式的调试场景应用验证）

## 适用场景

遇到未知Bug/错误时，从错误信息出发，通过分层追问系统性定位根因并修复。适用于：

| 场景 | 适用度 | 说明 |
|------|--------|------|
| 解析/渲染类错误（Mermaid/Markdown/编译器） | ✅✅✅ 核心场景 | 多层解析系统天然适合分层推导 |
| 不熟悉的技术栈报错 | ✅✅✅ 核心场景 | 没有经验可类比时，第一性原理是唯一可靠方法 |
| 反复出现的同类Bug | ✅✅✅ 核心场景 | 试错法反复失败时需要切换方法论 |
| AI协作中的Bug修复 | ✅✅ 强烈推荐 | 引导AI按推导链思考，避免随机试错 |
| 配置/环境问题排查 | ✅✅ 推荐 | 多层配置加载同样适合分层定位 |
| 熟悉领域的简单错误 | ⚠️ 杀鸡用牛刀 | 确定知道原因时直接修复即可，不必走完整流程 |
| 生产事故应急响应 | ❌ 不适用 | 紧急情况下先恢复服务再根因分析 |

## 问题背景

修复Bug有两种根本不同的范式：

| 范式 | 思维方式 | 速度 | 修复质量 | 可迁移性 |
|------|---------|------|---------|---------|
| **类比试错法** | "这个错误我见过/好像是XX问题→先试试→不行再换" | 碰运气，快则2分钟慢则2小时+ | 往往只修表面，同类问题下次还犯 | 低，换个错误类型就不会了 |
| **第一性原理调试法** | "错误说什么？→谁抛的？→什么机制？→哪个阶段？→规范怎么说？→根因是什么？→系统性修复" | 稳定，通常5-15分钟 | 修根因，一次性修复所有同类问题 | 高，通用方法论适用于所有Bug |

**核心洞察**：当你遇到不熟悉的错误时，试错法是随机游走；第一性原理调试法是确定性的推导链——前者靠运气，后者靠逻辑。

## 核心规则

### 规则1：从错误信息本身出发，不从假设出发

错误信息是你唯一可靠的起点。不要跳过错误信息直接"我猜是XX问题"。

**错误起点三问**：
1. 错误信息**原文**说了什么？（逐字读，不要快速浏览）
2. 错误信息中是否包含**关键术语**？（如"Unsupported markdown: list"中的"list"是关键）
3. 这个错误信息**是谁产生的**？哪个模块/组件/解析器/编译器抛出的？

### 规则2：理解分层架构，定位错误阶段

大多数复杂系统是分层解析/处理的，错误一定发生在某一个具体阶段。

**分层追问链**（六步标准推导链）：

```
错误信息（起点：你看到了什么）
  ↓ 问题1：这个错误信息是谁产生的？哪个模块/组件/解析器？
定位到具体组件
  ↓ 问题2：这个组件的工作原理是什么？它有几个处理阶段？
理解分层架构
  ↓ 问题3：错误发生在哪个阶段？这个阶段的输入输出是什么？
定位错误阶段
  ↓ 问题4：这个阶段有什么已知陷阱/规则/边界条件？查规范/文档
找到根因
  ↓ 问题5：还有哪些地方可能犯同样的错误？一次性全部修复
系统性修复
  ↓ 问题6：用自动化工具验证所有同类问题都解决了吗？
验证闭环
```

### 规则3：查规范，不要查直觉

定位到错误阶段后，**查阅该组件的规范/文档/源代码**，确认这个阶段的规则到底是什么。不要凭直觉猜测"它应该是这样工作的"。

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| "引号应该能阻止解析吧" | "应该"是直觉，不是事实 | 查规范确认引号在哪个阶段生效 |
| "我之前遇到过类似的错误" | 类似≠相同，类比可能错位 | 即使类似也要按推导链确认 |
| "先试一下这个方案" | 试错法没有信息增益 | 先完成推导再动手修复 |

### 规则4：系统性修复，不只修眼前

找到根因后，**搜索所有同类位置**一次性修复，不要只修改报错的那一处。

- 如果错误模式是`数字. `触发列表解析，检查所有节点文本是否包含此模式
- 如果错误原因是缺少空行，检查所有subgraph边界
- 如果错误是配置项缺失，检查所有配置文件

### 规则5：验证闭环必须覆盖新产物

修复完成后，用自动化工具验证。特别注意[讲解自犯效应](explainer-self-violation-effect.md)中提到的**验证盲区**：不要只验证被修复的文件，你在修复过程中新创建的文档/复盘/模式文件同样需要验证。

## 实战案例

### 案例1：Mermaid "Unsupported markdown: list" 错误

| 步骤 | 推导过程 |
|------|---------|
| 错误信息 | "Unsupported markdown: list"——list是什么？ |
| 谁产生的 | Mermaid内置的Markdown渲染器（不是Mermaid语法解析器） |
| 分层架构 | 查[mermaid-safe-coding-rules.md](../../code-patterns/mermaid-safe-coding-rules.md)→两阶段解析模型：①语法解析→②Markdown渲染 |
| 错误阶段 | 第②阶段Markdown渲染层→什么触发list？→`数字+英文句点+空格`模式 |
| 查规范 | Mermaid安全编码规则2：节点文本含列表触发模式时，用中文冒号替代英文句点或加引号 |
| 系统性修复 | 检查所有7个节点→全部将`1. `改为`1：`（中文冒号），同步清理3条冗余连线 |
| 验证闭环 | check-mermaid.py 0错误通过 |

**结果**：全程10分钟完成（3分钟定位+5分钟修复+2分钟验证）。

### 案例2：引号作用边界的反例（反面案例——违反规则3）

在修复完Mermaid错误后写[引号作用边界定律](../tools-automation/quoting-scope-limits.md)文档时，演示图中加了引号+`1. `+`\"`转义→渲染失败。这是违反规则3的典型——**直觉认为"引号能阻止解析"**，但实际上引号只在第①阶段（语法边界解析）生效，不阻止第②阶段（Markdown渲染）的列表触发。

正确做法应该是：先按推导链查规范确认引号的作用层级，而不是凭直觉"加个引号应该就安全了"。

## 与试错法的本质区别

```
试错法的路径（随机游走）：
  看到错误 → 猜是A → 修A → 还是错 → 猜是B → 修B → 又错了 → ...
  （每一步没有信息增益，全靠运气）

第一性原理调试法的路径（确定性推导）：
  看到错误 → 读错误信息 → 定位组件 → 理解架构 → 定位阶段 → 查规范 → 找到根因 → 修复 → 验证
  （每一步都在缩小问题空间，信息增益是确定的）
```

## 反模式

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| 看到错误就Google/Stack Overflow | 别人的答案可能不是你的根因，且跳过了理解过程 | 先自己完成推导链前三步，再查外部资料验证 |
| "我见过这个错误，应该是XX问题" | 类比可能错位——表面相似但根因不同 | 即使"见过"也要按推导链确认 |
| 先改改看，不行再说 | 试错法修改可能引入新问题，且浪费时间 | 先完成推导，再动手修改 |
| 只改报错的那一行 | 同类问题可能潜伏在其他位置 | 搜索所有同类模式，系统性修复 |
| 修完只看一眼就提交 | 人眼验证不可靠，且存在验证盲区 | 运行自动化工具验证所有相关文件 |
| 修完一个错误发现新错误就慌 | 这是[分层错误屏蔽](root-cause-diagnosis.md#layered-error-masking)，表层错误修复后深层错误暴露是正常的 | 继续按推导链逐层修复，直到0错误 |
| 在"讲解模式"下调Bug | [讲解自犯效应](explainer-self-violation-effect.md)：写文档/讲教程时认知资源被解释任务占用，System1自动执行错误模式 | 切换到"调试模式"，显式重载检查清单 |
