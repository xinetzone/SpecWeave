# 规范自举性驱动持续演化

## 模式类型
方法论模式（元方法论）

## 成熟度
L2 已验证（SpecWeave 13天793次提交验证）

## 问题场景
方法论项目何时"完成"？传统软件项目以功能交付为完成标志（功能做完就结项），但规范体系在具备自我演化能力后，不存在传统意义上的"结项"点。用传统项目思维判断方法论项目的完成时间会导致错误决策（如过早停止投入、错过加速增长期）。

## 核心定义

```
自举点（Bootstrap Point）= 规范体系可以用自身的方法论来扩展自身的时刻
```

达到自举点后：
- 归类决策树能自动为新内容找到归属
- 模板和检查清单能指导新内容的创建
- 复盘机制能从每次实践中萃取新模式
- 模式库能自我增长和完善
- 项目进入**持续演化阶段**，里程碑从"功能完成"变为"能力建立"

## 解决方案

### 自举点判断标准

当以下条件全部满足时，认为达到自举点：

| 判断维度 | 标准 |
|---------|------|
| **分类自举** | 存在MECE分类体系和决策树，新内容无需修改核心框架即可自动归位 |
| **模板自举** | 核心产出物有标准模板，新内容按模板创建即可保证格式一致 |
| **检查自举** | 有自动化检查脚本，能验证新内容是否符合规范 |
| **复盘自举** | 复盘→洞察→模式萃取闭环已建立，每次实践都能反哺体系 |
| **导航自举** | 入口文档和索引结构完整，新内容能被自动发现和导航 |

### 项目生命周期重新定义

| 阶段 | 里程碑定义 | 传统思维误区 |
|------|-----------|-------------|
| **阶段一：框架搭建** | 核心框架建立、基础规则就绪 | — |
| **阶段二：内容填充** | 用框架产出第一批内容、验证框架可用性 | — |
| **阶段三：自举点达到** | 分类/模板/检查/复盘/导航全部闭环 | ❌ 误认为"结项" |
| **阶段四：持续演化** | 能力里程碑（新模式数、验证项目数、复用次数） | ❌ 按功能完成判断进度 |

## 支撑证据

**SpecWeave案例验证**：
- 6/26（229次提交）：29/29 Spec完成，按传统思维误认为"结项"
- 6/27-7/5（后续9天）：增长246%（提交229→793，模式46→234，Wiki 0→59）
- 增长加速原因：达到自举点后，新模式通过组合产生，开发速度不降反升（日均提交57→63）

## 复用场景

- 所有方法论项目、框架项目、规范体系项目
- 判断是否应该继续投入：未达自举点前不要停止框架建设
- 规划项目里程碑：将"达到自举点"作为关键里程碑，而非"完成所有功能"

## 反模式

| 反模式 | 表现 | 后果 |
|--------|------|------|
| **过早结项** | 功能完成就认为项目结束，停止投入 | 错过自举后的加速增长期，前期投入无法复利 |
| **永远在搭框架** | 一直重构核心框架，不产出实际内容验证 | 框架无法通过实战验证，成为空中楼阁 |
| **自举后仍按功能迭代** | 用功能列表管理已自举的项目，按"还剩多少功能没做"判断进度 | 无法衡量体系的真实价值（价值在于自增长能力而非功能点） |

## 实施检查清单

- [ ] 是否定义了清晰的自举点判断标准？
- [ ] 分类/模板/检查/复盘/导航五个维度是否全部闭环？
- [ ] 达到自举点后，里程碑是否从"功能完成"切换为"能力建立"？
- [ ] 是否预留了持续演化的资源（而非结项解散）？
