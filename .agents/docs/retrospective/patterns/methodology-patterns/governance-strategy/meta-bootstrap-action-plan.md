# 元方法论自举行动计划

> 将"元方法论自举"洞察应用到下一个项目——用七概念方法论执行一个日常开发任务，用七概念质量门检验每一步，验证方法论在非自身构建场景下的适用性。

## 一、自举目标

| 维度 | 目标 |
|------|------|
| **验证目标** | 证明七概念方法论在日常功能开发任务中可执行、有价值、不过度 |
| **项目交付** | 一个可工作的CLI工具：`seven-concepts-trigger`，输入任务描述，输出概念组合建议 |
| **方法论验证** | 33项质量检查通过率≥80%，5种流程中至少使用3种，无致命反模式 |
| **反馈目标** | 发现≥3个方法论可改进点，为L3升级提供实战数据 |

## 二、候选项目选择

**选定项目**：七概念触发匹配CLI工具（`seven-concepts-trigger.py`）

**选择理由**：
1. ✅ 是日常功能开发（非方法论构建），能验证方法论在"普通任务"中的适用性
2. ✅ 规模适中（预计200-400行Python），能完整走完R→I→E→C或C→V→(R)流程
3. ✅ 与七概念方法论直接相关，执行中能自然触发各概念使用
4. ✅ 有明确验收标准（16种场景匹配准确率），可量化验证
5. ✅ 不涉及紧急止血/PoC等方法论不适用场景

**项目描述**：
- 输入：自然语言任务描述（如"修复线上支付Bug"、"Sprint结束做复盘"、"重构超长文档"）
- 输出：推荐的概念组合、执行顺序、参考流程编号、质量门提醒
- 核心算法：关键词匹配+权重评分+规则引擎，基于16种场景映射表
- 交付：可执行Python脚本 + 单元测试 + README使用说明
- 预期耗时：1-2小时

## 三、自举验证执行流程

### 阶段0：启动前基线测量（C→V前置）

| 测量项 | 基线值 | 采集方式 |
|--------|--------|---------|
| 七概念方法论成熟度 | L2.8（首次自举后L2.9） | 按L1-L4标准自评 |
| 执行者对方法论熟悉度 | 熟悉（刚完成构建） | 主观评分1-10 |
| 预期方法论使用率 | 预估使用C/V/A/R 4个概念 | 事前预测 |
| 任务预估时间 | 90分钟 | 估算 |
| 预期质量门通过率 | 85% | 事前预测 |

**⚠️ 前置检查（P1改进项，2026-07-11二次验证新增）**：
基线测量完成后，**必须先检查目标文件/功能是否已存在**：
1. 列出目标路径下的现有文件
2. 检查是否已有雏形代码/文档
3. 根据现状调整执行计划：
   - ✅ 从零开发：按原计划6个原子提交执行
   - ⚠️ 已有雏形：转为"验证+补全"模式，先运行现有代码→找问题→修复→补文档
4. 重新调整时间预估（已有代码会显著缩短时间，避免过度承诺）

| 现状判定 | 执行策略 | 时间预估调整 |
|---------|---------|-------------|
| 目录为空，从零开始 | 按原计划6个原子提交 | 原预估时间 |
| 已有核心代码，缺测试/文档 | 先测试→找Bug→修复→补文档 | 预估时间×0.4-0.6 |
| 代码文档基本完整 | V验证→边界case测试→小修复 | 预估时间×0.2-0.3 |

### 阶段1：F第一性原理——问题定义与假设剥离

**执行内容**：
1. 明确问题本质：开发者需要"3秒判断当前任务该用哪个概念组合"，而非查阅10页文档
2. 剥离假设：
   - ❓ 假设：开发者愿意用CLI工具（可能更愿意用速查卡）
   - ❓ 假设：关键词匹配足够准确（可能需要语义理解）
   - ❓ 假设：16种场景覆盖80%日常任务（可能遗漏）
3. 提炼公理：
   - 公理A：决策树是if-else规则集，可机械执行
   - 公理B：任务描述中的关键词与场景有统计相关性
   - 公理C：工具输出建议≠强制执行，最终判断在人
4. 重构方案：关键词匹配→规则评分→Top3建议→附带参考链接

**V对抗性审查**（本阶段质量门）：
- 构造反例：输入"做个蛋糕"这种无关描述，工具应该返回"无匹配"而非乱推荐
- 构造反例：输入混合场景（"重构+复盘"），应该返回多概念组合
- 假设验证：CLI vs 速查卡决策——保留速查卡作为人工备选，工具是辅助

**本阶段产出**：F公理清单 + V反例测试用例≥3个

---

### 阶段2：C原子提交驱动开发（主开发流程）

采用**C→V→(R)**流程（新功能开发流程）：

| 提交# | 提交内容（单一职责） | V验证点 |
|-------|---------------------|---------|
| C1 | 项目骨架：CLI参数解析、输入输出框架 | --help正常工作，无报错 |
| C2 | 16种场景关键词库+规则数据结构 | 规则数据结构可序列化，无循环引用 |
| C3 | 关键词匹配评分算法 | 单元测试：16种典型输入匹配正确≥14个 |
| C4 | 输出格式化（建议组合+流程编号+质量门提醒） | 输出可读，引用的流程编号和质量门存在 |
| C5 | 边界处理（无匹配/低置信度/混合场景） | V反例测试通过（无关输入返回无匹配） |
| C6 | README使用文档 + 错误提示优化 | 新用户按README可独立使用 |

**C阶段质量门（每提交检查）**：
- C-1：无跨模块不相关变更混同提交 ✅/❌
- C-2：revert该提交不破坏其他功能 ✅/❌
- C-3：提交信息符合Conventional Commits ✅/❌
- V-1：每个提交对应≥1个可验证测试/检查点 ✅/❌
- V-2：高影响逻辑（评分算法）≥2个反例测试 ✅/❌

---

### 阶段3：A原子化评估（粒度检查）

完成开发后执行粒度寻优：
1. 评估`seven-concepts-trigger.py`文件行数（目标≤500行）
2. 如果>400行，考虑拆分：规则数据→独立JSON/TOML，评分算法→独立模块
3. 检查链接完整性（README中引用的文档路径正确）
4. 评估是否需要独立的速查卡输出模式

**A阶段质量门**：
- A-1：单文件≤500行 ✅/❌
- A-2：链接无断链，无file:///绝对路径 ✅/❌

---

### 阶段4：R里程碑复盘（开发完成后复盘）

执行完整里程碑复盘流程：

**事实采集（无因果词）**：
- 实际提交数、每个提交大小（行数）
- 实际开发时间（对比预估90分钟）
- V阶段发现的Bug数量和类型
- 实际使用了哪几个概念（对比预期4个）
- 16种场景测试准确率
- 33项质量检查清单通过率

**反事实推演（≥2条）**：
- CF1：如果不用F直接开始写代码，会怎样？
- CF2：如果跳过V直接提交，会漏什么问题？
- CF3：如果不做A粒度检查，可维护性如何？

**洞察提炼（≥1条四元组）**：
- 从开发过程中发现的方法论自身不足或强化点

### 阶段5：E萃取与反馈（复盘后萃取）

将自举验证发现沉淀为方法论改进：

| 产出类型 | 内容 |
|---------|------|
| 方法论改进点 | ≥3个具体可执行的改进项 |
| 新反模式（如有） | 如果开发中犯了新错误，补充到反模式库 |
| 质量门调整 | 如果某些质量项不切实际/过于宽松，提出调整建议 |
| 场景补充 | 如果发现决策树遗漏的场景，补充到16种场景表 |
| 成熟度更新 | 根据实战数据更新七概念方法论成熟度评级 |

**E阶段质量门**：
- E-1：≥2个独立"案例"支持（本项目+原七概念构建项目）✅/❌
- E-2：每个改进点配套≥1个反模式/失败案例描述 ✅/❌

## 五、验证通过/失败标准

### ✅ 自举成功标准（全部满足）

1. **项目交付成功**：CLI工具功能完整，16种典型场景匹配准确率≥80%（13/16），5个边界case合理处理
2. **流程可执行**：完整走完F→C→V→A→R→E流程，没有因为"太麻烦"而跳过某个质量门
3. **质量门达标**：33项检查清单通过率≥80%，无致命反模式（AP1-AP3）出现
4. **方法论有价值**：事后评估V阶段至少发现1个真实问题（证明对抗审查有用）
5. **有改进反馈**：产出≥3个具体的方法论改进建议

### ❌ 自举失败/部分失败判定

| 失败信号 | 含义 | 后续动作 |
|---------|------|---------|
| 多个质量门"走过场"没发现问题 | 质量门设计可能过于宽松 | 收紧质量门标准 |
| 强烈感受"某步骤完全没必要" | 方法论存在过度设计 | 删减或合并该步骤 |
| 遇到方法论未覆盖的决策点 >3个 | 场景覆盖不足 | 补充决策树场景 |
| 实际耗时>预期3倍且主要卡在方法论流程 | 方法论开销过大 | 简化流程，区分轻重 |
| 项目本身交付质量差（Bug多/结构混乱） | 方法论未能保障质量 | 深度根因分析 |

## 九、启动信号

当你（用户）说"开始执行元方法论自举"时，立即按本计划从阶段0开始执行。
执行过程中严格按数据采集清单记录，每个阶段完成后标记质量门通过/不通过，不事后补录。

> **核心原则**：这次自举的目的不是完美交付工具，而是真实检验方法论——宁可暴露方法论的缺陷，也不要为了"证明方法论有效"而走过场。发现问题才是自举的最大价值。
