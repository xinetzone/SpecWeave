# 版本涟漪Grep清扫模式：单点更新后的多点引用同步

## 模式类型
治理策略/文档一致性维护模式

## 成熟度
L1 实验性（1次验证：并发安全检查器复盘原子化Step 5从7类9项→8类11项后，Grep发现2处下游失同步）

## 适用场景
更新可复用资产（模式文件、模板、检查清单、规范、SOP）后的一致性维护；量化指标（项数、分类数、步骤数）变更后；跨文档引用同步。

## 问题背景

当你更新一个核心模式/模板/规范中的关键数字或描述时，变更会像涟漪一样向下游传播——但不会自动传播：

**版本涟漪传播路径**：
```
1. 模式文件更新 → 2. 配套模板/清单更新 → 3. 资产清单/索引描述更新 → 4. 其他引用该模式的文档
```

每一跳都可能遗漏。如果只更新"源头"不更新"下游"，版本不一致就会产生——读者在不同文档中看到不同的数字和描述，不知道哪个是最新版本。

**典型现象**：
- 验证步骤从"7类9项"升级为"8类11项"，但资产清单仍写"7类可复用资产"
- 检查清单从"5步"扩展为"7步"，但教程文档仍描述旧流程
- 模式从L1升级为L2，但引用该模式的复盘报告仍标注L1
- 术语重命名（如"三查法"→"四查法"）后，部分文档仍使用旧名称

**根因分析**：
1. 人类和AI都倾向于"改完即止"，缺乏"变更影响分析"意识
2. 文档之间的引用关系没有显式索引（cross_refs只覆盖双向演进链，不覆盖所有提及）
3. 数字变更的影响范围难以通过肉眼判断——你不知道有多少文档引用了"7类9项"

## 核心原则

### 原则1：任何量化变更必须执行Grep清扫

当你的变更涉及以下内容时，必须执行Grep搜索：
- 数字（N类、N项、N步、N维、N个）
- 术语名称（如"三查法"→"四查法"）
- 版本号/成熟度等级
- 标准文件名/路径

### 原则2：搜索关键词要覆盖变体

旧版描述可能有多种表述方式：
- "7类可复用资产" / "7类资产" / "七类资产"
- "9项增强验证" / "9项验证" / "九项"
- 数字写法变化（阿拉伯数字/中文数字）、简称/全称

### 原则3：涟漪传播有固定跳数，逐跳检查

不要假设"改了源头就够了"，按传播路径逐跳检查：
1. 源头文件自身（确保内部一致）
2. 直接配套文件（模板、清单、与源头同目录的文件）
3. 索引/资产清单（README、asset-inventory）
4. 跨目录引用（复盘报告、其他模式中提及本模式的地方）

## 解决方案

### 标准操作流程（Grep清扫四步法）

```mermaid
flowchart LR
    A["完成核心变更"] --> B["提取搜索关键词"]
    B --> C["Grep搜索所有变体"]
    C --> D["逐一评估每个命中"]
    D --> E["同步更新失同步引用"]
    E --> F["再次Grep确认清扫干净"]
```

#### 步骤1：提取搜索关键词

完成核心编辑后，停下来思考：
- 我改了哪些**数字**？（如：7→8、9→11）
- 我改了哪些**术语/名称**？（如：三查→四查）
- 我改了哪些**成熟度/版本**？（如：L1→L2）

列出需要搜索的关键词清单，包括：
- 精确表述（"7类可复用资产"）
- 简称（"7类资产"）
- 中文数字变体（"七类"）

#### 步骤2：执行Grep搜索

对每个关键词执行Grep搜索（或ripgrep）：

```bash
# PowerShell示例（Windows）
Select-String -Path "docs/**/*.md" -Pattern "7类可复用资产|7类资产|七类"

# 或使用项目的搜索功能
# grep -r "7类可复用资产\|7类资产\|七类" docs/
```

搜索范围建议：
- 变更文件所在目录（近场）
- docs/retrospective/整个目录（远场，复盘报告中可能引用）
- .agents/目录（规范和模板中可能引用）

#### 步骤3：逐一评估命中

对每个搜索命中，判断：
- ✅ **需要更新**：该文档引用了旧版数字/术语，且属于应同步的下游
- ❌ **不需要更新**：该文档是历史记录（如旧复盘报告如实记载当时的情况），不应修改历史
- ⚠️ **需要判断**：如果不确定是否该改，标记后人工判断

**关键判断标准**：
- 历史复盘报告中的事实描述（"当时我们发现了9处漂移"）→ 不改，这是历史事实
- 模式/清单/规范中的现行标准描述（"本模式包含7类资产"）→ 必须改
- 资产清单/索引中的摘要描述（"edit-verify-separation包含7类9项验证"）→ 必须改

#### 步骤4：同步更新并验证

对需要更新的引用逐一修正，然后再次执行Grep确认：
- 除了历史记录外，没有其他过时引用
- 新数字/新术语已正确替换

### 关键词提取模板

当你做了以下类型的变更时，用这个模板提取搜索词：

| 变更类型 | 需要搜索的关键词示例 |
|---------|-------------------|
| 数字N→M | "N类"、"N项"、"N步"、"N维"、"N个"，以及中文数字变体 |
| 术语改名A→B | A的全称、简称、常见缩写 |
| 成熟度Lx→Ly | "成熟度.*Lx"、"Lx.*验证" |
| 文件移动/重命名 | 旧文件名、旧路径 |
| 步骤扩展 | "第N步"、"步骤N"、旧步骤名称 |

## 实际案例

### 案例：Step 5验证从7类9项→8类11项

**场景**：edit-verify-separation模式的验证阶段（Step 5）初版为"7类可复用资产、9项验证"，后扩展为"8类可复用资产、12项验证"（新增TOML元数据文件类别、frontmatter完整性检查、TOML同步检查）。

**Grep清扫执行**：

1. **提取关键词**：
   - "7类可复用资产"、"7类资产"、"七类"
   - "9项增强验证"、"9项验证"、"九项"

2. **搜索结果**（2处失同步）：
   - `asset-inventory.md`：描述edit-verify-separation模式时仍写"7类可复用资产...9项增强验证"
   - `data-drift-checklist.md`：D9预防措施仍写"7类资产全覆盖"

3. **处理**：两处均为现行规范描述，需要同步更新为"8类11项"。

4. **验证**：再次Grep确认，除历史复盘记录外无其他过时引用。

## 反模式

### 反模式1：改完即止，不搜索

更新了模式文件后认为"任务完成"，不执行任何下游检查。

**问题**：涟漪效应导致下游文档过时，读者在不同地方看到矛盾的信息。

**修正**：把Grep清扫作为编辑-验证分离模式中验证阶段的必做步骤。

### 反模式2：只搜索精确字符串，不搜索变体

只搜索"7类可复用资产"，遗漏"7类资产"、"七类"等变体表述。

**问题**：部分过时引用未被发现，清扫不彻底。

**修正**：为每个变更点列出所有可能的表述变体，用正则`|`分隔批量搜索。

### 反模式3：修改历史记录

发现旧复盘报告中写"9项验证"就直接改成"11项"。

**问题**：历史复盘报告如实记载当时的状态，修改历史等于篡改记录。读者读历史复盘时看到的应该是"当时"的真实情况。

**修正**：区分"历史记录"和"现行规范"：
- 复盘报告、历史分析、旧版本记录 → 不改（可加演进说明但不改事实）
- 模式文件、现行规范、资产清单、模板 → 必须改

### 反模式4：搜索范围太窄

只在当前目录搜索，不搜索整个docs/目录。

**问题**：其他目录下的复盘报告或模式可能也引用了旧数字。

**修正**：近场（同目录）+ 远场（docs/retrospective/ + .agents/）都要覆盖。

## 检查清单

更新可复用资产后，逐项检查：
- [ ] 我是否列出了本次变更涉及的所有数字/术语/版本变更？
- [ ] 每个变更点是否提取了所有表述变体（全称/简称/中文数字）？
- [ ] 近场（同目录）是否已搜索？
- [ ] 远场（docs/retrospective/ + .agents/）是否已搜索？
- [ ] 每个搜索命中是否判断了"历史记录vs现行规范"？
- [ ] 现行规范中的过时引用是否已全部更新？
- [ ] 历史记录是否保持原样未被修改？
- [ ] 更新后是否再次Grep确认清扫干净？
- [ ] 是否在模式演进记录中记录了本次变更？
