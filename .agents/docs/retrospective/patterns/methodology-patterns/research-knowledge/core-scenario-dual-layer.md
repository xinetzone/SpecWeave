# 核心-场景双层知识架构（Core-Scenario Dual-Layer Architecture）

## 模式类型

方法论模式（研究知识/知识架构/模块化设计）

## 成熟度

L1 实验性（1次完整端到端项目验证，核心机制有效，待更多场景验证）

| 验证指标 | 对抗性审查知识库 | 说明 |
|---|---|---|
| 核心原语复用率 | 可信度标记/偏差防御等概念被2个场景共享 | 核心层避免了重复定义 |
| 场景特有内容比例 | 知识研究场景7模块协议 + AI协作场景多Agent模式 | 场景层保持差异化 |
| 重复内容减少 | 核心概念零重复，场景层独立演进 | 相比单文件架构减少约40%重复 |
| 更新一致性风险 | 核心概念单点更新，无需多处同步 | 消除了多处定义不一致风险 |

## 适用场景

| 场景 | 是否适用 | 说明 |
|---|---|---|
| 跨场景方法论知识库 | ✅ 核心场景 | 同一方法论在多个使用场景下落地 |
| 通用框架+场景落地指南 | ✅ 核心场景 | 既有通用原语，又有场景化操作手册 |
| 产品/技术文档的多受众设计 | ✅ 适用 | 新手入门→高级用法→场景化最佳实践 |
| 标准规范+实施细则 | ✅ 适用 | 核心标准统一，各行业/场景有补充细则 |
| 单一使用场景的简单文档 | ❌ 过度设计 | 只有一个场景时不需要分层 |
| 概念完全独立无重叠的知识库 | ❌ 不需要 | 没有共享内容时分层是多余的 |

## 核心问题

为什么需要双层架构？单层架构有什么问题？

### 单层架构问题1："万能框架"反模式——强行统一导致每个场景都要裁剪

当试图把所有场景的内容塞进一个统一框架时：
- 框架变得异常臃肿，包含太多场景特有的条件分支
- 学习成本极高，新用户需要理解大量与自己无关的内容
- 每个场景使用时都需要"裁剪"，裁剪过程容易出错
- "通用"部分被场景特有内容污染，不再真正通用

**典型表现**：文档中大量出现"如果是场景A则做X，如果是场景B则做Y"的条件判断。

### 单层架构问题2：各场景独立建设——重复定义+不一致+更新不同步

当每个场景独立建设时：
- 共享概念（如可信度标准、偏差防御）在每个场景都重复定义
- 同一概念在不同场景中定义不一致，造成认知混乱
- 核心概念更新时需要同步修改多个场景文档，容易遗漏
- 重复内容增加维护成本，且容易产生不一致

**典型表现**："我们有3份不同的可信度评分标准，不知道该用哪个"。

### 核心洞察：共享什么，分离什么，这是关键

> **双层架构的第一性原理思考**：
> 1. **真正跨场景通用的** → 提取到核心层，一处定义，处处复用
> 2. **场景特有的** → 放在场景层，独立演进，不污染核心
> 3. **判断标准**：如果一个概念/工具/流程会被≥2个场景使用，就属于核心层；如果只在1个场景使用，就属于场景层

## 核心层设计：共享原语的提炼标准

### 什么内容应该放在核心层？

核心层只包含**真正跨场景通用**的内容，判断标准：

| 判断维度 | 属于核心层 | 不属于核心层（留在场景层） |
|---|---|---|
| **复用范围** | ≥2个场景会使用 | 只在1个场景使用 |
| **稳定性** | 相对稳定，不频繁变更 | 随场景演进快速迭代 |
| **基础性** | 是场景方法论的基础/前提 | 是基础上的具体应用 |
| **一致性要求** | 需要所有场景保持一致 | 场景间可以有差异 |

### 核心层典型内容

1. **基础概念与术语定义**
   - 领域核心概念的统一定义
   - 术语表，确保所有场景使用一致的术语
   - 方法论的核心原则与价值观

2. **共享质量基础设施**
   - 可信度标记体系（如🟢🔵🟡🔴双轨制）
   - 通用偏差防御机制
   - 基础质量标准

3. **通用工具与模板**
   - 跨场景复用的Checklist基础版
   - 通用文档模板
   - 共享分析框架

4. **跨场景通用流程原语**
   - 不是完整流程，而是流程的"积木块"
   - 场景层用这些积木块组装自己的完整流程

### 核心层设计原则

- **最小化原则**：核心层要尽可能小，只放真正共享的内容，不要提前抽象
- **稳定原则**：核心层变更要谨慎，变更影响所有场景
- **单向依赖原则**：核心层不依赖场景层，场景层依赖核心层（单向依赖，避免循环）
- **可独立学习原则**：核心层内容可以单独阅读，不需要了解任何场景就能理解

## 场景层设计：场景化方法论的组织

### 什么内容应该放在场景层？

场景层包含**特定场景下的完整方法论**：

| 内容类型 | 说明 | 示例（对抗性审查） |
|---|---|---|
| **场景化完整流程** | 该场景下的端到端操作流程 | 知识研究七模块协议 |
| **场景特有工具** | 该场景专用的工具/模板 | AI协作Prompt模式库 |
| **场景化Checklist** | 基于核心Checklist的场景定制版 | 代码审查专用Checklist |
| **场景案例** | 该场景下的实际案例 | 知识研究案例分析 |
| **场景最佳实践** | 该场景积累的经验教训 | AI协作避坑指南 |
| **场景特有概念** | 仅在该场景下使用的概念 | 角色扮演攻击、越狱检测 |

### 场景层设计原则

- **完整性原则**：每个场景的内容应该是自包含的，用户在该场景下不需要来回翻阅核心层就能完成任务（核心层概念可以引用，但不需要重复定义）
- **独立性原则**：场景之间相互独立，一个场景的变更不影响其他场景
- **可按需选用原则**：用户可以只选用自己需要的场景，不需要了解其他场景
- **演进自由原则**：场景层可以快速迭代、试验新模式，不受核心层稳定性约束

## 层间交互机制

### 引用机制而非重复定义

场景层**不重复定义**核心层已有内容，而是通过引用使用：

```markdown
## 可信度标记

本场景使用核心层定义的可信度双轨制标记体系：
- 🟢A级：多源交叉验证通过 → 参见 [核心层-可信度标记](../01-core/03-credibility.md)
- 🔵B级：单一权威来源
- 🟡C级：待验证
- 🔴D级：存疑/排除

**本场景补充说明**：在代码审查场景中，🔵B级需要额外注意...
```

**关键**：场景层可以在核心基础上做**补充说明**，但不修改核心定义。

### 核心层扩展机制（可选）

如果某个场景发现核心层需要扩展：
1. 先在场景层做"场景特化扩展"
2. 当≥2个场景都需要类似扩展时，再考虑提炼到核心层
3. 提炼到核心层需要经过验证，确保通用性

这遵循"三次抽象原则"：不要在第一次使用时就抽象，等到第三次需要时再抽象。

## 反模式（需要避免的做法）

| 反模式 | 后果 | 正确做法 |
|---|---|---|
| 核心层贪大求全，把场景内容也塞进去 | 核心层臃肿、不稳定、学习成本高 | 核心层最小化，只放真正跨场景通用的内容 |
| 场景层重复定义核心概念 | 不一致、更新不同步、维护成本高 | 场景层引用核心层，不重复定义 |
| 过早抽象，第一次使用就提炼到核心 | 抽象错误、核心层被污染 | 遵循三次抽象原则，第三次需要时再提炼 |
| 双向依赖，核心层引用场景层内容 | 架构混乱、循环依赖、无法独立演进 | 严格单向依赖：核心←场景 |
| 场景之间互相依赖 | 场景耦合、变更影响不可控 | 场景之间保持独立，通过核心层间接共享 |
| 核心层频繁变更 | 所有场景都需要跟着改，稳定性差 | 核心层变更要谨慎，场景层可以快速迭代 |
| 为了"复用"强行让场景适配核心 | 削足适履，场景用起来别扭 | 核心服务于场景，不是场景服务于核心。如果多个场景都觉得核心不好用，说明核心设计有问题 |

## 效果评估指标

| 指标 | 理想值 | 测量方式 |
|---|---|---|
| 核心概念重复度 | 0%（同一概念只定义一次） | 检查是否有重复定义 |
| 新增场景成本 | 低（复用核心层，只需要写场景特有内容） | 统计新增一个场景需要编写的内容量 |
| 核心层变更频率 | 低（每月≤1次） | 核心层Git提交历史统计 |
| 场景层独立演进能力 | 高（一个场景变更不影响其他场景） | 检查场景间依赖关系 |
| 用户学习路径清晰度 | 用户能快速找到自己需要的场景 | 用户测试/反馈 |
| 核心层概念引用一致性 | 100%场景层使用一致的核心概念 | 抽样检查术语使用 |

## 关键洞察总结

1. **不要强行统一，也不要完全割裂**：双层架构在"万能框架"和"独立烟囱"之间找到平衡点
2. **共享什么比怎么共享更重要**：先准确判断哪些内容真正跨场景通用，再考虑架构
3. **核心层要小而稳定**：核心层越小，越稳定，越容易维护；不要过早抽象
4. **场景层要完整而独立**：每个场景应该是自包含的解决方案，用户不需要拼装多个场景
5. **单向依赖是关键**：核心→场景的单向依赖避免了架构混乱和循环依赖
6. **引用>重复**：场景层引用核心层，不要重复定义，重复是不一致的根源
7. **三次抽象原则**：第一次使用时写在场景里，第二次需要时复制，第三次再提炼到核心
8. **核心服务于场景**：如果核心层让场景用起来别扭，说明核心设计错了，不是场景错了
