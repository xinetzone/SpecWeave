# 外部内容分析的渐进式 Spec 规划流程

## 模式类型

方法论模式（Spec 规划/时间盒管理/外部内容分析）

## 成熟度

L1 首次提炼（1 次实战验证，待更多场景验证）

## 适用场景

| 场景 | 是否适用 | 说明 |
|------|---------|------|
| 外部内容获取与分析任务 | ✅ 核心场景 | 网页学习、文章分析、知识萃取等依赖外部信息源的任务 |
| 样本量未知的分析任务 | ✅ 核心场景 | 规划时无法预知能获取多少样本，需基于实际调整 |
| 信息源可达性不确定 | ✅ 核心场景 | 可能遭遇反爬、登录墙、权限限制等访问障碍 |
| 用户反馈规划阶段耗时过长 | ✅ 核心场景 | 已出现"为何一直在打转"的反馈，需要时间盒约束 |
| 内部文档分析（信息源确定） | ⚠️ 部分适用 | 可跳过阶段 2 的试错，直接进入阶段 3 |
| 纯代码开发任务（无外部依赖） | ❌ 不适用 | 标准规划流程即可，无需渐进式调整 |

## 核心原则

**最小启动 + 渐进细化**：不追求一次规划到位，而是通过三阶段渐进式规划快速启动并基于实际情况调整。

传统规划方法假设"规划时信息完备"，但外部内容分析任务的信息在获取前是缺失的——不知道能拿到多少样本、不知道内容质量、不知道是否有访问障碍。强行在信息缺失时做详细规划，必然导致规划与实际脱节，返工成本高于渐进调整成本。

**时间盒是硬约束**：每个阶段的时间盒是上限而非目标，到达上限必须退出当前阶段（即使产出不完美），避免陷入"再想想就能规划到位"的陷阱。

## 三阶段规划流程

### 阶段 1：最小可行 Spec（时间盒：15 分钟）

**目标**：快速定义任务范围和核心产出物

**内容**：
- 1-2 句话描述任务目标
- 3-5 个核心 Task（不细化 SubTask）
- 预期产出物清单

**退出条件**：spec.md + 简版 tasks.md 已创建

**反模式**：不要在此阶段细化 SubTask 或编写详细 checklist——此时样本量未知，详细规划必然基于假设，后续大概率返工

**产出示例**：
```
spec.md:
  任务目标：对知乎问题 637007780 下的回答进行系统性学习与知识萃取
  预期样本：23 条回答（待确认可达性）
  产出物：learning-notes.md + raw-content.md + analysis-report.md

tasks.md（简版）:
  Task 1: 获取知乎回答内容
  Task 2: 执行系统性学习层分析
  Task 3: 执行深度洞察层分析
  Task 4: 执行知识萃取层分析
  Task 5: 生成分析报告
```

### 阶段 2：内容获取试错（时间盒：30 分钟）

**目标**：获取外部内容并评估样本量

**内容**：
- 按反爬策略优先级尝试获取内容
- 记录成功/失败策略
- 评估样本覆盖率

**退出条件**：已获取内容或全部策略失败

**反模式**：不要在此阶段开始分析——分析会消耗大量上下文窗口，导致后续阶段无法基于完整信息调整 Spec

**策略优先级**（参考 [external-website-analysis-fallback-strategy.md](external-website-analysis-fallback-strategy.md)）：
1. 直接访问（defuddle / curl）
2. 工具增强（agent-browser + 反自动化 flag）
3. 官方替代源（API / 移动版 / 缓存）
4. 第三方权威源（archive.org / 第三方转载）

**产出**：raw-content.md（已获取的样本）+ 获取策略记录（成功/失败及原因）

### 阶段 3：基于实际样本调整 Spec（时间盒：10 分钟）

**目标**：根据样本量决定分析深度

**内容**：
- 按小样本分析方法论的降级规则调整 Task
- 细化 SubTask（仅在样本充足时）
- 编写 checklist

**退出条件**：tasks.md 和 checklist.md 已完成

**反模式**：不要在此阶段重新规划任务范围——阶段 1 已锁定范围，此阶段仅调整分析深度和 SubTask 粒度

**降级决策**（参考 [small-sample-analysis-methodology.md](small-sample-analysis-methodology.md) 样本量前置检查）：

| 样本量 | 调整动作 |
|---|---|
| ≥ 10 | 全规格执行，细化全部 SubTask 和 checklist |
| 5-9 | 降级执行，深度洞察层标注"初步"，SubTask 适度简化 |
| 3-4 | 大幅降级，跳过共识/Top N 分析，SubTask 最小化 |
| < 3 | 仅信息结构化，跳过深度分析和萃取，重新评估任务可行性 |

**产出**：更新后的 tasks.md（含 SubTask）+ checklist.md（含降级标注）

## 三阶段时间盒总览

    P2 --> P3["阶段3：基于样本调整<br/>10分钟"]
    P3 --> Execute(["进入执行阶段"])

**总规划时间盒：55 分钟**（15 + 30 + 10）。超过此时间盒仍未完成规划，应立即退出规划阶段，基于当前信息启动执行——执行中的调整比规划中的空转更有价值。

## 案例参考

### 知乎 637007780 分析任务（2026-07-06）

**问题**：Spec 规划阶段耗时过长，用户反馈"为何一直在打转"。执行者在样本量未知时就细化了 41 个 SubTask 和 38 项 checklist，但实际仅获取 3/23 条回答（覆盖率 13%），大量 SubTask 基于错误假设（如"统计 Top 5 高赞回答"在 3 个样本下无意义）。

**根因**：
1. 未在阶段 1 锁定最小范围，试图一次规划到位
2. 未在阶段 2 先获取样本再细化，而是边规划边获取
3. 未在阶段 3 按样本量降级，仍按全规格执行

**改进**：本模式将规划拆为三个时间盒阶段，强制"先获取样本再细化"，避免在信息缺失时做详细规划。

**验证结果**：1 次实战验证（知乎 637007780 分析任务的复盘改进行动），待在后续外部内容分析任务中复用验证。

## 反模式与注意事项

### 绝对禁止的反模式

| 反模式 | 为什么错误 | 正确做法 |
|--------|----------|---------|
| **阶段 1 细化 SubTask** | 样本量未知，详细规划必然基于假设 | 阶段 1 只列 3-5 个核心 Task |
| **阶段 2 开始分析** | 分析消耗上下文窗口，影响后续 Spec 调整 | 阶段 2 只获取内容，不分析 |
| **阶段 3 重新规划范围** | 范围应在阶段 1 锁定，阶段 3 只调整深度 | 阶段 3 只调整 SubTask 粒度和分析深度 |
| **超时间盒继续规划** | "再想想就能规划到位"是陷阱 | 到时间盒上限必须退出，启动执行 |
| **跳过阶段 2 直接规划** | 没有实际样本，阶段 3 的降级规则无法应用 | 必须先获取样本再调整 Spec |

### 注意事项

1. **时间盒是上限不是目标**：每个阶段提前完成应立即进入下一阶段，不要"用满"时间盒
2. **阶段 2 可并行**：如果阶段 1 已定义清楚，阶段 2 的内容获取可委派子代理并行执行（参考 [subagent-atomic-task-template.md](../ai-collaboration/subagent-atomic-task-template.md) 的内容获取类任务扩展模板）
3. **降级不是失败**：阶段 3 决定降级执行是正确的规划调整，不是任务失败——基于实际样本调整分析深度是科学方法论
4. **总时间盒 55 分钟**：超过此时间盒仍规划不完，说明任务复杂度超出预期，应拆分为多个子任务或与用户重新对齐目标

## 模式演进方向

当前版本为 L1（首次提炼，1 次复盘验证），后续可在以下方向迭代：

1. **跨场景验证（L1→L2 路径）**：在更多外部内容分析任务中验证（如竞品调研、行业报告分析、技术文档学习），积累至少 2 次成功验证
2. **时间盒量化校准**：基于实际执行数据校准各阶段时间盒（当前 15/30/10 分钟为首次设定，可能需要调整）
3. **与 Spec Mode 工作流集成**：将三阶段规划流程集成到项目的 Spec Mode 工作流中，作为外部内容分析任务的默认规划流程
4. **自动化时间盒提醒**：开发工具在规划阶段计时并在接近时间盒上限时提醒
