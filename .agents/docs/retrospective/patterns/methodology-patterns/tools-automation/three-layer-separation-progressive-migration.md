# 三层分离·渐进迁移

> **提炼自**：[retrospective-seven-concepts-refactor-20260723/README.md](../../../reports/project-reports/retrospective-seven-concepts-refactor-20260723/README.md) —— seven-concepts-cmd三层分离重构项目复盘

## 一、模式类型

方法论模式（架构重构/工程实践）

## 二、成熟度

L2 已验证（2个案例：1个跨端架构Q&A + 1个工具重构；其中工具重构案例为自引用验证，需独立第三方案例验证后升级L3）

## 三、触发场景

- **当**单文件脚本/单体模块需要拆分为可复用库且保持向后兼容**时**，使用这个模式
- **当**核心逻辑被IO/CLI/格式化代码耦合，无法被其他模块直接import复用**时**，使用这个模式
- **当**测试只能通过subprocess黑盒执行，速度慢且无法精确定位内部Bug**时**，使用这个模式
- **适用于**：单文件→多模块重构、单体→分层架构迁移、CLI脚本→可import库转化
- **不适用于**：全新项目从零开发（无需"迁移"，直接按分层架构设计即可）；纯数据/配置文件拆分（无逻辑层可分离）；<50行的微小脚本（过度工程）

## 四、问题背景

单文件脚本的常见痛点：
- 常量、数据模型、业务逻辑、格式化、CLI入口全部耦合在一个文件中
- 核心逻辑无法被其他脚本/MCP/Skill直接import调用，必须通过subprocess黑盒调用
- 测试只能通过subprocess执行（每次启动Python解释器），速度慢（~3s/次）
- 无法针对单个函数编写精确的单元测试
- 修改一处常量可能影响不相关的格式化逻辑，回归风险高

## 五、核心思想

**将单文件拆分为三层（领域层=纯逻辑/渲染层=纯格式化/平台层=薄CLI壳），按依赖顺序渐进迁移，每步保持向后兼容并用黑盒测试守底线。**

## 六、核心做法

### 步骤1：萃取/选择模式，定义三层边界

- 明确三层职责划分：
  - **领域层**：零IO纯逻辑（常量、数据模型、匹配算法、业务规则），无print/sys/argparse
  - **渲染层**：纯格式化（输入数据结构，输出字符串/dict），不直接调print
  - **平台层**：薄CLI壳（argparse + 调用领域层 + 调用渲染层 + print输出），<100行
- 列出公开API清单（哪些函数/类/常量需要对外导出）

### 步骤2：建立spec约束，锁定行为边界

- 生成spec.md，明确Non-Goals：不修改核心逻辑、不改CLI参数/输出格式（等价重构）
- 生成tasks.md，按依赖顺序分解为原子Task（建目录→迁移领域→迁移渲染→导出API→改薄壳→黑盒验证→白盒补充）
- 生成checklist.md，定义每步验收标准
- **关键**：重构前的黑盒测试用例充当"行为契约"，spec的Non-Goals是对这些测试覆盖行为的显式声明
- **必须包含「测试输入预设表」**（在spec.md或checklist.md中）：

| 测试输入（描述/关键词） | 预期匹配场景 | 预期匹配概念 | 预期置信度 | 覆盖路径 |
|----------------------|------------|------------|-----------|---------|
| "帮我提交代码" | 原子提交 | C(Commit) | high | 单概念匹配 |
| "项目复盘，生成报告" | 复盘报告导出 | R+I+E+V | high | 多概念组合 |
| "提交" | 需澄清 | C vs git命令 | medium | 歧义路径 |
| ... | ... | ... | ... | 边界/异常路径 |

  - 表中必须覆盖：单概念匹配路径、多概念组合路径、歧义/低置信度路径、边界异常路径
  - 多匹配路径（多个概念/场景可能命中）必须有明确的优先级预期
  - 该表作为白盒测试编写时的测试用例来源，避免重构后临时想测试输入

### 步骤3：按依赖顺序渐进迁移（逐Task执行）

严格按以下顺序执行，每个Task完成后立即验证：
1. 创建目录结构 + 常量模块 + 数据模型模块（无依赖，最先做）
2. 迁移核心匹配/业务逻辑到独立模块（依赖步骤1的常量/模型）
3. 迁移渲染/格式化函数到独立模块（依赖步骤1-2的数据结构）
4. 更新 `__init__.py` 导出公开API（依赖步骤1-3的模块）
5. 将原文件改为薄CLI壳（依赖步骤4的公开API，保留原文件名确保向后兼容）
6. 运行黑盒测试验证零回归
7. 新增白盒测试（直接import领域层），验证测试速度提升

### 步骤4：黑盒测试守底线 + 白盒测试提速度

- **黑盒测试**（守底线）：用重构前的subprocess测试验证CLI对外行为100%不变
- **白盒测试**（提速度）：新增直接import领域层函数的单元测试，无subprocess开销
- **分工互补**：黑盒验证"行为不变"，白盒验证"逻辑正确"且"速度快"

### 步骤5：原子提交，每阶段独立可回滚

- spec规划 → 独立提交
- 重构成果 → 独立提交（确认黑盒零回归后）
- 文档清理 → 独立提交
- 每个提交单一职责，若某阶段出问题可独立回滚

## 七、适用条件

| 条件 | 说明 |
|------|------|
| 重构前有黑盒测试 | 零回归的前提是"行为契约"已存在；无测试则先补测试再重构 |
| 核心逻辑与IO可分离 | 领域层能提取为纯函数（无副作用），否则三层分离无意义 |
| 需要被其他代码import复用 | 如果永远只通过CLI调用，拆分收益有限 |
| 原文件行数≥200行 | 太小的文件拆分是过度工程 |

## 八、反模式（不要这么做）

### ❌ 反模式1：三层一起改，不分阶段验证

**错误做法**：同时修改常量、逻辑、CLI，一次性提交全部变更。
**后果**：如果黑盒测试失败，无法定位是哪一层引入的问题；回滚需要撤销全部工作。
**正确做法**：按步骤3的依赖顺序逐Task迁移，每步完成后立即验证。

### ❌ 反模式2：领域层残留IO副作用

**错误做法**：领域层的match_task函数中保留print语句或sys.exit调用。
**后果**：领域层无法被其他代码import复用（import时会触发print），白盒测试会产生副作用输出，违背"纯函数"原则。
**正确做法**：领域层所有函数只接收参数、返回结果，不产生任何IO副作用。print/sys/argparse只出现在平台层（CLI壳）。

### ❌ 反模式3：改薄壳时修改CLI参数或输出格式

**错误做法**：在步骤5改薄壳时，顺手"优化"CLI参数名或调整输出格式。
**后果**：破坏向后兼容，黑盒测试失败，已有调用者（脚本/Skill/CI）断裂。
**正确做法**：等价重构——CLI参数和输出格式100%不变。功能优化留待后续独立提交。

### ❌ 反模式4：只做黑盒测试不补白盒测试

**错误做法**：黑盒测试通过就结束，不新增直接import的白盒测试。
**后果**：测试仍需subprocess，速度未提升；无法精确定位内部函数Bug；重构的"可import复用"价值未被测试覆盖。
**正确做法**：黑盒守底线 + 白盒提速度，两者缺一不可。

## 九、检验标准（单文件脚本重构验收 Checklist）

做完之后怎么知道做对了？逐项确认：

### 9.1 三层分离验收

- [ ] **领域层零IO**：所有领域模块（constants/models/matcher/scenarios等）可在纯Python环境 `import`，执行时无print输出、无sys.exit、无argparse依赖
- [ ] **渲染层纯格式化**：格式化函数只接收数据结构参数，返回字符串/dict，不直接调print
- [ ] **平台层薄壳**：CLI入口脚本行数<100行，仅包含argparse解析+调用领域层+调用渲染层+print输出

### 9.2 API导出验收

- [ ] **公开API清晰导出**：`__init__.py` 显式导出所有公开函数/类/常量，外部可 `from lib.xxx import func` 直接调用
- [ ] **向后兼容**：原CLI脚本文件名保留，命令行参数和输出格式100%不变（等价重构）

### 9.3 测试验收（核心项）

- [ ] **测试输入预设表已在spec中定义**：重构前已明确各路径的测试输入和预期输出，多匹配路径有优先级预期
- [ ] **黑盒测试100%通过**：重构前的subprocess测试用例全部通过，CLI对外行为零回归
- [ ] **✅ 白盒测试同步新增**：新增直接 `import` 领域层函数的单元测试，覆盖核心匹配/业务逻辑、边界情况、格式化函数；测试用例覆盖预设表中的所有路径；测试用例数≥黑盒测试数
- [ ] **白盒测试速度提升≥5倍**：从subprocess黑盒（~3s/次）改为直接import白盒（<0.6s/次），验证测试效率改善
- [ ] **测试代码无副作用**：白盒测试运行时无stdout污染（除非是测试格式化输出），不依赖外部资源

### 9.4 提交验收

- [ ] **原子提交**：spec规划→重构成果→文档清理各阶段独立提交，每个提交单一职责
- [ ] **pre-commit hooks全通过**：关键配置文件位置校验、敏感信息检测、.temp生命周期检查均通过

## 十、迁移验证记录

### 案例1：SPA+SSR跨端架构Q&A（模式萃取来源）

- **场景**：用户询问"SPA+SSR混合渲染架构转手机app容易吗"
- **模式应用**：从Q&A中提炼出"三层分离"（渲染层/UI层/平台层分离）+"渐进迁移"（分阶段从Web→Hybrid→Native）的跨端架构决策模式
- **验证结果**：模式成功从技术Q&A中萃取，证明了跨端架构决策可以结构化为"三层+渐进"模式
- **验证类型**：概念验证（从Q&A萃取模式，非代码重构）
- **贡献**：模式首次定义，触发场景和核心步骤确立

### 案例2：seven-concepts-cmd三层分离重构（模式应用验证）

- **场景**：将377行单文件 `seven-concepts-trigger.py` 重构为 `lib/seven_concepts/` 六模块分层架构
- **模式应用**：完整执行步骤1-5，三层分离为领域层（constants/models/matcher/scenarios）+ 渲染层（formatters）+ 平台层（trigger薄壳）
- **验证结果**：
  - 黑盒测试 19/19 = 100% 零回归
  - 白盒测试 26/26 全通过，速度 3s→0.30s（10倍提升）
  - CLI壳 76行（<100行达标）
  - 11个公开API导出
  - 3个原子提交，pre-commit hooks全通过
- **验证类型**：代码重构验证（完整执行+量化验收）
- **特殊性**：谱系自引用——萃取模式的工具自身是被重构对象。既是强验证（最苛刻条件），也需警惕确认偏误（验证者=被验证者）
- **贡献**：模式从概念验证升级为代码验证，反模式和检验标准从实际执行中补充

## 十二、跨领域迁移示例

- **场景1（文档拆分）**：将一个超长Markdown文档（含正文+模板+脚本）拆分为内容层（纯正文）/模板层（模板定义）/脚本层（自动化脚本），渐进迁移保持链接不断
- **场景2（团队组织）**：将一个全能型角色拆分为领域专家（纯专业判断）/沟通层（报告格式化）/执行层（操作落地），渐进迁移保持职责不空窗
