# 工具自举效应（Tool Bootstrap Effect / Dogfooding）

## 模式类型
方法论模式

## 成熟度
L1 实验性（1次案例：check-links自举发现能力不足→增强→发现更多问题）

## 适用场景
内部工具开发、开发平台建设、工具链演进、质量保障体系建设。

## 问题背景

工具开发容易陷入"预先大设计"陷阱——试图在一开始就想清楚所有需求，结果要么过度设计，要么脱离实际使用场景。另一个极端是"只修眼前问题"，工具永远停留在最低可用状态。

## 核心模型：自举正反馈循环

```mermaid
flowchart LR
    A["使用工具解决实际问题"] --> B["工具发现自身不足<br/>(bug/能力缺失/体验问题)"]
    B --> C["分析不足→增强工具"]
    C --> D["工具能力提升"]
    D --> E["能发现更多/更深层的问题"]
    E --> A
```

**这就是"吃自己的狗粮"（Eating your own dog food）在工具开发中的体现**：
- 工具不是在"用户环境"中发现bug，而是在"检查自己项目"时发现不足
- 工具的使用者就是工具的开发者，反馈环长度 = 0
- 每一次使用工具的过程，都是一次工具审计

## 自举效应的关键特征

1. **反馈环极短**：使用者=开发者，问题发现→修复→验证在同一会话中完成
2. **真实场景驱动**：不是基于想象的需求，而是基于真实使用中的痛点
3. **递增演进**：不需要大设计，每次使用发现一个痛点，解决一个痛点
4. **能力叠加**：每次增强后，工具能发现之前发现不了的问题，形成正反馈
5. **信任建立**：通过dogfooding建立对工具的信心——自己天天用的工具才可靠

## 自举演进路径示例

`.agents/scripts/` 目录从几个脚本增长到体系化工具链的路径：
1. 最初只有 check-links.py（仅检测断链）
2. 用check-links检查自己项目 → 发现相对路径断链模式一致 → 增加--fix自动修复
3. 用--fix修复真实断链 → 发现需要dry-run预览 → 增加dry-run模式
4. dry-run验证后发现需要看板 → 开发generate-dashboard.py
5. 原子化操作后发现手动收尾繁琐 → 开发finalize-atomization.py
6. 移动文件前不知道影响面 → 开发build-ref-index.py
7. CI中运行check-links → 发现需要缓存外部检查结果 → 增加缓存机制

**没有预先的大设计，全部是在使用中发现痛点，在痛点中演进能力。**

## 适用条件

自举效应成立的前提：
- 工具的使用者是开发者自己（内部工具/开发平台）
- 工具使用频率足够高（每天/每次任务都用）
- 反馈机制畅通（发现问题能立即修改）
- 有"零摩擦"修改环境（改完就能跑，不需要复杂部署）

## 与现有模式的关系

- `prove-usefulness-check.md`：证明有用模式是自举的起点——先让工具在真实场景中证明有用，再通过自举持续增强
- `tool-automation-decision-model.md`：自动化决策模型决定什么值得自动化，自举效应驱动自动化的持续演进
- `toolchain-maturity.md`：自举效应是推动工具链从L1向L5演进的核心动力机制
- `review-insight-export-loop.md`：复盘-洞察-导出闭环与工具自举形成"双环驱动"——工具发现问题，复盘萃取模式，模式指导工具增强

## 检查清单

- [ ] 工具是否在自己的项目中使用？（dogfooding）
- [ ] 使用中发现的问题是否有快速修复通道？
- [ ] 是否有意识地在使用中观察工具不足？
- [ ] 每次增强后是否重新运行工具发现新问题？
- [ ] 是否避免了"预先大设计"，让真实使用驱动演进？

## 反例警示

| 陷阱 | 后果 |
|------|------|
| 工具开发者不用自己的工具 | 问题发现滞后，用户成了免费测试员 |
| 追求"一次做对"的大设计 | 过度设计或脱离实际 |
| 反馈环过长（需要部署/审批才能修改） | 自举循环断裂，工具停止演进 |
| 增强后不重新运行 | 错过正反馈机会，能力停滞 |
