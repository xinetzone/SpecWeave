# 经验迁移映射（Experience Transfer Mapping）

## 模式类型
方法论模式（模式萃取质量保障）

## 成熟度
L1 实验性（1次案例：断链修复7项经验的跨领域迁移映射）

## 适用场景
- 复盘/洞察萃取时评估经验的通用价值
- 模式编写时明确适用边界和跨领域示例
- 判断一个洞察是"上下文特定"还是"可复用模式"
- 评审模式质量时检查迁移性论证是否充分

## 问题背景

经验萃取时常见两个极端问题：
1. **过度泛化**：把特定上下文的细节当成通用模式，导致在其他场景中失效
2. **过度保守**：认为"这只是我们项目的特殊情况"，错过真正有价值的可复用经验

反事实推演证明了改进的价值（counterfactual-debt-analysis），但还需要回答："这个经验只是解决了Markdown链接问题，还是对其他领域也有价值？"

## 核心方法：三列迁移映射表

每个萃取的模式/洞察，必须回答"能迁移到哪里"，用三列表格强制思考：

| 本案例经验 | 可迁移到（领域/场景） | 迁移示例（具体案例） |
|-----------|-------------------|-------------------|
| 经验A的核心机制 | 领域X中的同类问题 | X场景中如何应用 |
| 经验B的核心机制 | 领域Y中的同类问题 | Y场景中如何应用 |
| ... | ... | ... |

## 迁移判断标准：识别"核心机制"

不是所有经验细节都能迁移，要区分**核心机制**和**上下文细节**：

| 类型 | 是否迁移 | 举例（断链修复案例） |
|------|---------|-------------------|
| **核心机制**（抽象原理） | ✅ 可迁移 | 精度优先零误报原则、最佳实践需配套工具链、工具自举正反馈 |
| **实现细节**（技术选择） | ❌ 不迁移 | 使用Python标准库、check-links的具体参数、7天缓存TTL |
| **问题表现**（具体症状） | ❌ 不迁移 | Markdown相对路径断链、`../`层数错误 |

## 迁移映射的四重价值

1. **泛化检验**：强制思考"核心机制是什么"，区分模式本质与上下文细节
2. **边界明确**：通过列举迁移场景，自然发现适用边界（能迁移到哪里，不能迁移到哪里）
3. **示例丰富**：迁移示例就是模式文档中的跨领域案例，帮助读者理解
4. **信心建立**：证明经验不是"一次性技巧"，而是有跨领域价值的通用规律

## 模式萃取检查清单（与本模式配套使用）

萃取一个新模式时，必须回答：
- [ ] 核心机制是什么？（不是"做了什么"，而是"为什么有效"）
- [ ] 能迁移到哪3个以上不同领域？
- [ ] 每个迁移领域有什么具体示例？
- [ ] 有没有无法迁移的上下文细节需要明确排除？
- [ ] 迁移后的适用条件是否相同？（如：精度优先原则适用于破坏性工具，但不适用于只读检测工具）

## 反例警示

| 陷阱 | 后果 |
|------|------|
| 不做迁移映射，凭感觉判断"这是通用模式" | 过度泛化，模式在不适用场景中失效 |
| 只列"可迁移到"不给具体示例 | 迁移方向模糊，读者无法理解如何应用 |
| 把实现细节也当成可迁移经验 | 模式文档臃肿，核心原理被细节淹没 |
| 认为"我们的场景太特殊"而放弃萃取 | 错过真正有价值的通用经验 |
| 只有1个迁移场景就认为够了 | 可能只是巧合，≥3个不同领域才能证明通用性 |

## 与现有模式的关系

- `explicit-maturity-tracking.md`：L1实验性模式仅有1次验证，迁移映射表可帮助判断升级到L2时的验证方向
- `pattern-merge-boundary.md`：迁移映射有助于判断两个洞察是否描述同一核心机制（应合并）
- `insight-library-evolution.md`：跨领域迁移是洞察库演进的核心驱动力
- `three-level-problem-solving.md`：L3系统免疫层的经验最具迁移价值（因为它抽象掉了具体问题）
