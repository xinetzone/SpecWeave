# 约束驱动的修复路径优先级矩阵（Constraint-Driven Fix Path Priority Matrix）

## 模式类型
方法论模式

## 成熟度
L1 实验性（至少 1 次成功案例：llvm-dev 环境与构建修复）

## 适用场景
当同一问题存在多条可行修复路径，且项目存在硬约束（例如“不修改挂载目录”“不改上游”“不改生产镜像”“不可中断线上服务”）时，用于确定修复路径优先级，避免优先走到高侵入、高风险、不可回退的方案。

## 问题背景
许多工程故障并非“没有正确解”，而是“有多条解”。在多解并存时，如果缺少路径排序框架，团队往往会倾向于技术上更“标准”的方案（例如改代码），但该方案可能违反约束、引入更大联动风险、导致不可控交付。

## 核心原则
在约束存在且不可随意变更时，优先级排序以“项目安全”优先于“技术正确”为原则：

1. 先满足约束与可交付性
2. 再优化技术正确性与长期质量

## 优先级决策矩阵
按以下维度对候选路径做打分或排序（从上到下优先级递减）：

| 维度 | 问题 | 说明 |
|------|------|------|
| 合规性 | 是否违反硬约束？ | 违反硬约束的方案默认降级（除非约束可被正式变更） |
| 侵入性 | 是否改动业务/上游代码？ | 侵入越大，回滚成本与联动风险越高 |
| 一次验证性 | 能否快速验证成功/失败？ | 可快速验证的方案适合作为首选试探路径 |
| 可回退性 | 失败后是否容易回到原状态？ | 不可回退操作需要更高门槛（审批/备份/隔离） |
| 维护成本 | 后续会不会引入新的复杂度？ | workaround 需要可解释、可复现、可维护 |

## 标准化实施流程模板
1. 明确约束：把硬约束写成可检查条目（例如“不改挂载源码”“不引入新系统依赖”）
2. 列出备选路径：至少 2 条，并对每条说明“改什么/改哪里/影响面”
3. 用矩阵排序：先过滤不合规，再按侵入性、一次验证性、可回退性排序
4. 先做最快验证：对排名第一的路径做一次最小验证（构建/启动/连接/导入）
5. 产出证据并沉淀策略：归档失败日志与成功验证，明确“默认稳定路径/可选质量路径”

## 适用边界
- 不适用于：约束本身可以被快速正式变更且变更成本低的场景（此时应优先走“变更约束”流程）
- 不适用于：不存在备选路径的场景（只有“改代码”一条路时，本模式无法降低侵入性）
- 注意：当 workaround 长期存在时，需要定期复审其维护成本，必要时升级为标准修复

## 前置条件
- 能明确列出项目约束与不可触碰边界
- 至少存在一个可执行的最小验证动作（哪怕是 dry-run/verify）
- 能保留证据（失败日志、成功输出、关键版本信息）
