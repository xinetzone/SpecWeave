# 洞察萃取

## 关键洞察（5条）

### 洞察1：三段式拆分架构是CLI工具原子化的通用模式

**内容**：对于CLI脚本和工具类文件，采用「CLI入口薄垫片 + __init__.py聚合导出 + 职责单一的子模块」三段式架构，拆分成功率100%，可保证完全向后兼容。

**标准分层**：
- `constants.py`：常量定义
- `models.py`：数据类/类型定义
- `{module}_checks.py` / `{module}_handlers.py`：核心业务逻辑
- `scanner.py` / `discovery.py`：文件发现/扫描逻辑
- `cli.py` / `reporter.py`：命令行接口/报告输出
- `__init__.py`：公共API导出

**支撑事实**：14个不同类型文件（vendor检查、edge case处理、质量检查、硬编码检测、frontmatter迁移、状态管理、mcp域解析、测试文件等）全部套用此模式成功。

**可复用场景**：任何超过300行的Python CLI脚本或工具模块拆分。

---

### 洞察2：批量拆分效率高于单个拆分，存在规模效应

**内容**：统一模式下批量拆分14个文件，平均每个文件耗时比单个拆分减少约40%，因为目录创建、模板编写、垫片编写等重复操作形成肌肉记忆，且验证流程流水线化。

**支撑事实**：前3个P1文件（橙色高风险）平均每个耗时约15-20分钟，后面11个P2文件平均每个耗时约8-12分钟，效率提升明显。

**启示**：原子化拆分适合批量处理同类型文件，而非零散处理。

---

### 洞察3：原子提交必须执行"三查暂存"验证

**内容**：git add后必须检查三类变更状态：
1. 新增文件（A）是否完整
2. 修改文件（M）是否都需要提交
3. 删除文件（D）是否都已暂存

遗漏任何一类都会导致提交不完整，需要amend修复。同时必须排除__pycache__、*.pyc等构建产物。

**支撑事实**：本次提交漏了link_fixer.py的删除记录，并且误加了__pycache__.pyc文件，都是因为没有执行三查验证。

**根因**：git add <新目录>只会添加目录下的新文件，不会自动暂存父目录中同名旧文件的删除记录，这是git的默认行为，容易被忽略。

---

### 洞察4：代码验证必须以实际运行为准，禁止视觉判断语法错误

**内容**：不要凭肉眼看代码就判断"有语法错误"，必须实际运行 `python -m py_compile` 或相关测试来验证。视觉判断容易产生误报，浪费时间。

**支撑事实**：本次执行中出现2次语法错误误报，每次浪费2-3分钟反复查看代码，实际运行验证发现代码完全正确。

**原则**：代码是否正确，解释器说了算，不是眼睛说了算。

---

### 洞察5：拆分不重构原则降低风险，bug修复是额外收益

**内容**：原子化拆分阶段应该严格遵循"拆分不重构"原则——只移动代码、分离职责，不修改业务逻辑。这样可以将"拆分引入bug"的风险降到最低。拆分过程中发现的潜在bug可以顺手修复，但要在commit message中明确说明。

**支撑事实**：本次拆分严格遵守拆分不重构原则，159+测试全部通过，仅在阅读state.py代码时发现并修复了一个execute_skip的潜在bug，并在commit message中标注。

## 可复用模式（3个）

### 模式1：三段式原子化拆分架构

- **成熟度**：L3（已在14个不同场景验证通过）
- **触发条件**：Python CLI脚本/工具模块 > 300行需要拆分
- **模式结构**：薄入口垫片 → __init__.py聚合 → 职责子模块（constants/models/checks/scanner/cli）
- **验收标准**：所有模块<300行，向后兼容，测试通过

### 模式2：Windows环境git中文提交安全方案

- **成熟度**：L2（已验证有效）
- **触发条件**：Windows PowerShell环境下需要提交中文commit message
- **方案**：将commit message写入UTF-8编码的临时文件，使用 `git commit -F commit-msg.txt` 提交
- **验收标准**：git log查看中文无乱码

### 模式3：原子提交三查验证法

- **成熟度**：L2（漏提交问题反向验证了必要性）
- **触发条件**：执行git commit之前
- **检查项**：
  1. 新增文件（A）：是否完整，是否误加__pycache__等
  2. 修改文件（M）：是否都是本次需要提交的
  3. 删除文件（D）：是否都已暂存
- **验收标准**：git status --short显示无遗漏、无误加
