# 三、洞察萃取

## 3.1 洞察一：逐层优化自带"下一层可见性"

**事实**：每一轮优化完成后，原本被掩盖的冗余才会暴露。连接器 .md 在一开始看起来是"合理的轻量级入口"，但当 project-overview 被合并后（阶段三），连接器和 README 的功能重叠就变得刺眼——它们都成了"导航表 + 元信息"。

**规律**：

```
阶段一完成后 → 连接器看起来合理（精简到 7 行）
阶段三完成后 → 连接器变得多余（README 也有导航表 + 元信息）
阶段四完成后 → 一目了然（0 中间层）
```

消除一层，下一层的冗余才可见。这不是设计失误，而是**渐进式重构的自然规律**。每一次重构改变"参照系"，之前看起来合理的设计在新的参照系下会暴露缺陷。

**可复用模式**：渐进式重构的"逐层暴露"规律。不要在第一轮就试图找出所有冗余——先做最大的一步，等新结构稳定后再评估。

## 3.2 洞察二：TOML `source` 字段是重构的锚定参照系

**事实**：四轮操作中，`source` 字段始终是正确重定向的目标。

| 阶段 | source 字段操作 |
|------|----------------|
| 主题迁移 | 路径从 `reports/xxx.md` 更新为 `reports/<topic>/xxx.md` |
| 连接器合并（有 TOML） | `source` 从连接器值 → 原始报告值 |
| 连接器合并（无 TOML） | 删除空 source（无更深源头） |

**规律**：`source` 溯源链本身就是一张"重构地图"。连接器的 source 指向原始报告，README 的 source 指向连接器。当删除中间层时，只需要把下游的 source 直接指向更上游即可——**溯源信息不会因删除中间层而丢失**。

**可复用模式**：将 TOML `source` 字段作为重构锚点。任何文件移动/合并/删除操作前，先读取 `source` 字段，将其传递到新的目标位置。

## 3.3 洞察三：atomization 作为试点验证了跨主题推广的可行性

**事实**：atomization 经历了完整的四轮优化，形成了最精简的结构。其余 4 个主题目录结构完全一致，推广路径清晰。

**各主题当前状态**：

| 主题 | 目录数 | 连接器 | project-overview | 待执行阶段 |
|------|--------|--------|------------------|-----------|
| atomization | 9 | 0 | 0 | ✅ 已完成 |
| insight-extraction | 8 | 8 | 8 | 阶段二+三+四 |
| spec-system | 7 | 7 | 7 | 阶段二+三+四 |
| roles-teams | 3 | 3 | 3 | 阶段二+三+四 |
| project-governance | 7+1 | 6 | 7+1 | 阶段二+三+四 |

**推广策略**：选择一个非关键主题（如 roles-teams，仅 3 个目录）作为第二试点，验证推广方案的"一次通过率"，然后批量处理其余主题。
