## 二、过程分析（S2）

### 成功因素

1. **第一性原理需求澄清有效**：在编码前先澄清"对抗式审查"的歧义（系统健壮性vs规避审查），避免了方向性错误
2. **Spec先行模式验证有效**：先完成PRD+任务分解+验证清单三件套，再开始编码，任务边界清晰
3. **冒烟测试驱动Bug发现**：通过实际运行加密/解密往返测试，在提交前发现了5个实现Bug
4. **共享库复用**：frontmatter.py已存在解析逻辑，通过增强而非重写，减少了重复代码
5. **原子提交规范执行到位**：使用git-commit-utf8.py处理中文编码，显式add文件，无无关文件混入

### 问题与挑战

1. **实现阶段缺少TDD**：5个Bug都是在手动冒烟测试中发现的，而非单元测试先行发现，增加了调试成本
2. **默认值设置错误**：security_level默认值设为'internal'而非需求定义的'public'，反映了需求到实现的映射验证不足
3. **IO重复读取**：初版实现中文件被读取两次（raw_content + parse_frontmatter_unified内部），说明函数接口设计时未考虑调用场景
4. **代码复用不彻底**：frontmatter边界检测正则最初在两个文件中重复，直到冒烟测试阶段才发现并提取共享函数
5. **lstrip()副作用**：使用content.lstrip()是常见的"看起来无害"操作，但破坏了有意的前导空白格式

### 瓶颈分析

| 瓶颈 | 影响 | 根因类型 |
|------|------|---------|
| 无单元测试先行 | Bug发现晚，调试成本高 | 流程缺失 |
| 需求-实现映射无验证 | 默认值错误 | 检查遗漏 |
| 函数接口设计未考虑调用上下文 | IO重复读取 | 设计不足 |
| DRY原则执行不严格 | 正则重复 | 代码审查缺失 |

## 三、核心洞察（S3）

### 洞察1：默认值是最容易出错的契约边界

**发现**：5个Bug中有1个是默认值错误（security_level默认'internal'而非'public'）。默认值是隐式契约——当调用方不显式指定时，系统的行为由默认值决定，一旦错误，影响所有未升级的旧文件。

**可复用模式**：
- 默认值必须在Spec中明确记录，实现时与Spec逐条对照
- 默认值选择应遵循"最小权限/最小影响"原则——public是最安全的默认（不加密），不会导致旧文件无法读取
- 旧格式兼容测试必须覆盖"缺失关键字段"场景

### 洞察2：IO操作的重复是性能隐患的早期信号

**发现**：初版`read_knowledge_entry`先读取`raw_content`，再调用`parse_frontmatter_unified`内部重新读取文件，导致重复IO。这在大文件或高并发场景下会成为性能瓶颈。

**可复用模式**：
- 文件读取函数应支持传入已读取的content参数，避免重复IO
- 函数设计时应考虑"我已有的数据如何复用"，而非总是"我自己再读一次"
- 解析类函数的合理接口：接受content字符串，而非自行打开文件

### 洞察3：字符串处理方法的副作用常被低估

**发现**：`content.lstrip()`看似"清理空白"的无害操作，实际破坏了正文前导空白的有意格式。这是典型的"过度清理"反模式。

**可复用模式**：
- 对用户内容使用`strip()`/`lstrip()`/`rstrip()`前必须问："这些空白是用户有意为之的吗？"
- Markdown/代码内容中前导空白有语义意义（缩进代码块、列表层级等），不应盲目strip
- frontmatter与正文之间的分隔应精确控制（两个换行符），而非依赖strip()

### 洞察4：第一性原理澄清在模糊需求场景下价值巨大

**发现**：本项目中"对抗式审查"是高度模糊的术语，可以解释为"规避外部审查"或"代码健壮性对抗测试"。通过第一性原理回归（参考Vibe Coding方法论的两个Prompt），我们选择了正确的方向。

**可复用模式**：
- 收到模糊需求时，第一步不是设计或编码，而是从第一性原理出发："这个术语在本项目上下文中最根本的含义是什么？"
- 参考权威来源（本项目中是vibe-coding-prompts-learning-analysis.md）验证解释
- 歧义必须在编码前解决，方向性错误的代价比实现Bug高10倍以上

### 洞察5：三级加密设计的核心权衡点

**发现**：public/internal/confidential三级设计不是简单的"加密强度递增"，而是每个级别有不同的权衡：
- public：100%兼容Git diff，但无保护
- internal：元数据加密但正文可Git diff，平衡安全与协作
- confidential：全文加密，安全性最高但Git diff失效

**可复用模式**：
- 安全级别设计应考虑"协作可见性"与"安全性"的权衡，而非只看加密强度
- internal级别保留tags/category/title明文是关键设计决策——确保检索和Git diff可用，同时保护敏感元数据
- 加密格式需要自描述标记（如`---ENC---`头），避免误处理

### 洞察6：冒烟测试是最小可行验证，但不应替代单元测试

**发现**：手动冒烟测试（public/internal/confidential三级加密往返测试）发现了所有5个Bug，但效率低且不可重复。这验证了"测试必须存在"的必要性，也暴露了"手动测试不可持续"的问题。

**可复用模式**：
- 新模块的最小验证：写一个快速冒烟测试脚本验证核心路径
- 冒烟测试发现Bug后，应立即将该场景转化为正式单元测试（防止回归）
- 加密模块等安全相关代码必须有自动化测试，手动验证不可接受

### 洞察7：加密模块的"无密钥优雅降级"是关键设计

**发现**：internal级别无密钥时，正文明文仍可读（只是metadata加密字段无法解密）；confidential级别无密钥时抛出明确异常。这种设计确保了不同级别在无密钥环境下有合理行为，而非crash或返回乱码。

**可复用模式**：
- 安全相关API在缺少密钥时不应静默失败或返回乱码，应有明确的分级行为
- public：完全正常
- internal：返回可读部分+加密字段标记为[ENCRYPTED]
- confidential：抛出DecryptionError异常
- 永远不要因为加密失败而导致数据丢失或损坏

### 洞察8：原子提交前的"三查"有效防止了无关文件混入

**发现**：本次原子提交执行三查暂存法时，发现工作区有多个不相关的retrospective文件（删除/修改/新增），显式git add 6个目标文件而非git add .，成功避免了无关文件混入。

**可复用模式**：
- 禁止`git add .`不是教条，而是防止"临时文件/不相关修改/敏感文件"混入提交的有效防线
- 三查法（查新增/查修改/查删除）应在每次提交前执行
- Windows中文提交必须使用UTF-8通道（git-commit-utf8.py）
