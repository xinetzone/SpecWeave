# 洞察分析：seven-concepts-cmd 三层分离重构

> 本文档对 [README.md](README.md) 复盘报告中的关键问题执行 5-Whys 根因分析、异常检测和自引用验证，产出可行动改进建议。

---

## 一、根因分析：为什么这次重构能零回归完成？

> 追问5层Why，从表层现象追溯到底层根因。

**Why-1：为什么零回归？**
因为19个黑盒测试100%通过，CLI对外行为完全不变。

**Why-2：为什么黑盒测试能100%通过？**
因为重构是"等价迁移"——match_task中的关键词、置信度、工作流定义原样迁移到新模块，未修改任何匹配规则。CLI入口脚本保留了原文件名 `seven-concepts-trigger.py`，命令行参数和输出格式完全兼容。

**Why-3：为什么能做到"等价迁移"而不引入偏差？**
因为重构前有明确的spec约束：Non-Goals明确写了"不修改匹配规则本身"、"不改变任何命令行参数的行为或输出格式（等价重构）"。spec的约束在重构执行前就锁定了行为边界，执行者无需在迁移过程中临时判断"这个行为要不要改"。

**Why-4：为什么spec能在重构前就准确锁定行为边界？**
因为重构前已经有19个黑盒测试用例（`test-seven-concepts-trigger.py`），这些测试用例精确描述了19个输入→预期输出的映射关系。spec的Non-Goals实际上是对这19个测试用例覆盖的行为的显式声明——测试用例充当了"行为契约"。

**Why-5：为什么重构前就有19个黑盒测试？**
因为 `seven-concepts-trigger.py` 在开发时就建立了基于subprocess的批量测试（TEST_CASES列表，16正向+3新增）。这是项目"测试先行"文化的体现——工具开发时同步建立测试，而不是重构时才补测试。

**根因结论**：零回归的根因不是重构过程做得多好，而是**重构前就已经存在的行为契约（19个黑盒测试）锁定了等价迁移的边界**。测试用例作为"可执行的行为规格"，比文档描述更精确、更难被误读。重构只是"搬运"已被测试覆盖的逻辑，不触碰未被测试覆盖的行为。

> **可复用洞察**：重构的零回归保障不在重构过程中，而在重构前——如果重构前没有测试覆盖，重构本身就是"盲飞"。先建测试、再重构，是零回归的必要前提。

## 二、异常检测：子代理临时文件残留——为什么会发生？

### 2.1 事件描述

子代理在执行 Task 1-7 过程中创建了2个临时文件：
- `_create_seven_concepts.py`（用于创建模块的辅助脚本）
- `_test_multi_match.py`（用于测试多匹配场景的临时脚本）

这两个文件在子代理任务结束时未被清理，残留在工作区中，需要主会话手动发现并删除。

### 2.2 5-Whys 根因分析

**Why-1：为什么临时文件未被清理？**
因为子代理在任务结束时没有执行清理步骤。

**Why-2：为什么子代理没有执行清理步骤？**
因为子代理的任务定义中未包含"退出前清理临时文件"的约束。任务定义聚焦于"完成功能目标（创建模块/迁移逻辑）"，未覆盖"工作区卫生"这一非功能性目标。

**Why-3：为什么任务定义遗漏了工作区卫生？**
因为子代理委派协议的设计假设是"子代理只产出最终交付物"，但实际上子代理在完成交付物的过程中会创建中间辅助文件（脚本化执行是AI代理的常见工作方式）。

**Why-4：为什么委派协议假设与实际不符？**
因为委派协议基于"人类开发者"的心智模型——人类开发者会自觉清理自己的临时文件。但AI代理缺乏"自觉性"，只执行任务定义中明确列出的步骤。

**Why-5：为什么AI代理缺乏"自觉性"？**
因为AI代理是"指令驱动"的——只做被要求做的事，不做未被要求但"应该"做的事。这是当前AI代理与人类开发者的根本差异，也是 [practice-gap-recursive-practice](../../../patterns/methodology-patterns/governance-strategy/practice-gap-recursive-practice.md) 模式描述的"践行鸿沟"的一种表现。

**根因结论**：临时文件残留的根因是**委派协议基于人类心智模型设计，未适配AI代理"指令驱动"的特性**。解决方向不是期望AI代理"自觉"清理，而是在任务定义中显式增加清理约束。

### 2.3 改进方案

| 方案 | 描述 | 成本 | 收益 |
|------|------|------|------|
| 方案A：任务模板增加清理条款 | 子代理任务模板末尾固定追加"任务完成后必须清理所有 `_*.py` 临时文件" | 低 | 中 |
| 方案B：委派后自动扫描 | 主会话在子代理返回后自动扫描工作区的 `_*.py` 文件并提示清理 | 中 | 高 |
| 方案C：子代理沙箱化 | 子代理在临时目录中工作，完成后只拷贝最终产出物 | 高 | 高 |

**推荐**：方案A（立即实施）+ 方案B（中期增强）。方案C成本过高，暂不推荐。

## 三、自引用验证：seven-concepts-cmd 萃取的模式应用到自身——这是真正的验证还是确认偏误？

### 3.1 自引用的两面性

| 维度 | 正面价值（验证有效性） | 风险警示（确认偏误） |
|------|----------------------|-------------------|
| 通用性验证 | 模式在最苛刻条件下（工具自身架构问题）仍有效，验证了强健性 | 验证者=被验证者，倾向于选择支持模式的证据 |
| 完整性验证 | 模式的7个步骤全部被实际执行，非纸上谈兵 | 可能下意识选择"容易通过"的执行路径 |
| 可操作性验证 | 模式指导了真实的代码变更，产出可测量的结果 | 结果测量标准由验证者自己定义 |

### 3.2 如何区分"真正验证"与"确认偏误"？

**真正验证的标志**：
1. ✅ 模式在执行过程中暴露过问题（而非全程顺畅）——本次Task 5多匹配测试输入选择不当，说明模式执行中遇到了实际困难
2. ✅ 存在客观的、不依赖模式定义的验收标准——19个黑盒测试是重构前就存在的，不是为验证模式而创建的
3. ✅ 有失败的可能——如果三层分离设计错误，黑盒测试会失败

**确认偏误的标志**：
1. ⚠️ 验证者同时定义模式和验收标准——"三层分离"的标准和"渐进迁移"的步骤都是模式自身定义的
2. ⚠️ 缺乏独立第三方视角——没有非seven-concepts-cmd的案例来交叉验证
3. ⚠️ 成功的归因可能过度——零回归可能主要归因于"等价迁移+黑盒测试"，而非"三层分离模式"本身

### 3.3 结论

本次自引用验证**部分有效**：
- **有效性部分**：模式提供了清晰的架构骨架和迁移路径，确实指导了重构的执行顺序和模块划分。黑盒测试的零回归是一个相对客观的验收标准。
- **局限性部分**：零回归的根因是"等价迁移+重构前已有测试"（见第一节5-Whys分析），而非三层分离模式本身。三层分离模式解决的是"如何拆分"的问题，不直接保证"零回归"。

**成熟度判定**：模式当前只能标记为 **L2（本项目验证）**，不可升级L3。必须等待≥1个非自引用的独立案例（非seven-concepts-cmd的重构项目）验证后，才能升级L3。

### 3.4 防偏误机制建议

1. **预设验收标准**：模式应用前，先用黑盒测试定义"什么算成功"，再执行模式。避免事后调整标准。
2. **记录失败案例**：模式执行中遇到的困难（如多匹配测试输入选择不当）必须如实记录，不得隐藏。
3. **等待独立验证**：自引用验证通过后，主动寻找非自引用场景应用同一模式，交叉验证。
4. **对抗审查**：对模式应用结果执行V（对抗审查），从"模式是否过度归功"的角度攻击。
