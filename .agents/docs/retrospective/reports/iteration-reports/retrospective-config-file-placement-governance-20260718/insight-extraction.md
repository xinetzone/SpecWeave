基于复盘报告 S3 识别的 3 个系统性问题，使用 5-Whys 根因法进行深度分析，挖掘根本原因与共性模式，为改进建议提供因果链支撑。

## 5-Whys 根因分析

### 系统性问题 1：实现与 docstring 声明不一致

**现象**：check-temp-lifecycle.py 的 docstring 声明"名称必须包含创建日期（YYYYMMDD）或关联 task-id"两种合规条件，但代码仅实现 YYYYMMDD 日期检测（`DATE_PATTERN = re.compile(r"\d{8}")`），未实现 task-id 模式匹配。

**5-Whys 追问**：

| 层级 | 问题 | 回答 |
|------|------|------|
| Why 1 | 为什么代码没有实现 task-id 检测？ | 实现时聚焦于日期检测（主要场景），忽略了 task-id 作为替代条件的实现 |
| Why 2 | 为什么实现时会忽略 task-id？ | 实现者（子代理）在编写代码时，主要关注日期检测的逻辑实现，对 docstring 中的"或 task-id"部分没有给予足够重视 |
| Why 3 | 为什么实现者会对 docstring 部分内容不够重视？ | 缺乏"docstring 声明 vs 代码实现"的一致性检查机制——实现完成后，没有自动化的检查来验证 docstring 声明的功能是否全部实现 |
| Why 4 | 为什么没有一致性检查机制？ | Spec 模式第七阶段验证 checklist 主要关注"spec 要求 vs 实现成果"的一致性，没有显式包含"docstring 声明 vs 代码实现"的一致性检查 |
| Why 5 | 为什么 checklist 没有覆盖这一维度？ | checklist 模板设计时，假设"docstring 声明的功能会被实现"是隐式保证的，没有将其作为独立检查项 |

**根本原因**：Spec 模式第七阶段验证 checklist 模板缺乏"docstring 声明 vs 代码实现"一致性检查项，导致 docstring 中声明但代码未实现的功能无法被自动发现。当前依赖独立验证者的细致阅读发现，但这种依赖人工的机制覆盖不完整（本次仅因验证者认真阅读 docstring 才发现）。

**因果链**：
```
checklist 模板未覆盖 docstring 一致性
  → 缺乏自动化检查机制
    → 实现者忽略 docstring 部分声明
      → task-id 检测未实现
        → 纯 task-id 命名被误判为不合规
```

### 系统性问题 2：文本模式与 JSON 模式输出完整性不一致

**现象**：JSON 模式对每项输出 date_source 字段，但文本模式仅在过期项输出中显示 `({it.date_source})`，不合规项与合规未过期项未显示。

**5-Whys 追问**：

| 层级 | 问题 | 回答 |
|------|------|------|
| Why 1 | 为什么文本模式没有对每项输出 date_source？ | 实现时聚焦于过期项的诊断信息，忽略了不合规项与合规未过期项的 date_source 标注 |
| Why 2 | 为什么实现者会忽略这些项的 date_source 标注？ | 实现者关注"过期项需要 date_source 来诊断为什么过期"，没有意识到"不合规项也需要 date_source 来诊断为什么不合规" |
| Why 3 | 为什么实现者会有这种认知偏差？ | 缺乏"文本模式 vs JSON 模式"输出完整性一致性检查——实现者假设"JSON 模式是机器读取的，需要完整；文本模式是人类读取的，可以简化" |
| Why 4 | 为什么会有这种假设？ | 没有明确的规范说明"文本模式与 JSON 模式的输出完整性应该一致" |
| Why 5 | 为什么没有这样的规范？ | SpecWeave 脚本输出规范中没有"模式间一致性"的明确要求，依赖实现者的判断 |

**根本原因**：SpecWeave 脚本输出规范缺乏"文本模式与 JSON 模式输出完整性一致性"的明确要求，导致实现者根据主观判断决定哪些信息需要在文本模式中展示。这种"模式间一致性"问题在多输出模式的脚本中具有普遍性。

**因果链**：
```
脚本输出规范未明确模式间一致性要求
  → 实现者主观判断"文本模式可简化"
    → 不合规项与合规未过期项未标注 date_source
      → 文本模式诊断信息不完整
        → 用户无法判断不合规项的基准日期来源
```

### 系统性问题 3：Windows 平台脚本编码安全防御缺失

**现象**：pre_commit.py 在默认 GBK 终端运行时崩溃：`UnicodeEncodeError: 'gbk' codec can't encode character '\U0001f4c1'`（📁 emoji）。

**5-Whys 追问**：

| 层级 | 问题 | 回答 |
|------|------|------|
| Why 1 | 为什么 pre_commit.py 会因 emoji 字符崩溃？ | main() 函数未调用 `setup_safe_output()`，emoji 字符（📁🕒🔒）在 GBK 终端无法编码 |
| Why 2 | 为什么 main() 没有调用 `setup_safe_output()`？ | pre_commit.py 是较早编写的脚本，当时可能还没有 `setup_safe_output()` 函数；后续添加新检查（含 emoji 输出）时没有补加调用 |
| Why 3 | 为什么添加新检查时没有补加调用？ | 缺乏"Python 脚本编码安全防御"的自动化检查——新增/修改 Python 脚本时，没有自动检测是否调用了 `setup_safe_output()` |
| Why 4 | 为什么没有自动化检查？ | check-file-placement.py 的受管文件清单关注的是"文件位置"，不是"脚本编码安全" |
| Why 5 | 为什么受管文件清单不包含脚本编码安全？ | 受管文件清单的设计目标是"防止文件错误放置"，没有扩展到"脚本编码安全"维度 |

**根本原因**：缺乏"Python 脚本编码安全防御"的自动化检查机制，新增/修改 Python 脚本时依赖人工记忆调用 `setup_safe_output()`，容易遗漏。这是 Windows 平台脚本开发的常见陷阱——GBK 是 Windows 中文终端的默认编码，而 emoji 与许多 Unicode 字符无法用 GBK 编码。

**因果链**：
```
缺乏脚本编码安全自动化检查
  → 依赖人工记忆调用 setup_safe_output()
    → pre_commit.py 添加 emoji 输出时未补加调用
      → GBK 终端运行时 UnicodeEncodeError
        → pre-commit 钩子阻塞提交
```

## 共性根因挖掘

3 个系统性问题的根本原因都指向同一个更深层模式：

> **SpecWeave 缺乏"实现一致性自动检查"框架**——无论是 docstring vs 实现、文本模式 vs JSON 模式、还是编码安全防御，都依赖人工记忆或独立验证者发现，没有自动化检查。

### 共性模式：人工依赖陷阱

| 问题 | 当前依赖 | 失败模式 |
|------|---------|---------|
| docstring vs 实现一致性 | 独立验证者细致阅读 | 验证者未读到 docstring 部分内容则遗漏 |
| 文本模式 vs JSON 模式一致性 | 实现者主观判断 | 实现者认为"文本模式可简化"则不完整 |
| 编码安全防御调用 | 人工记忆 | 添加新检查时忘记调用则崩溃 |

### 根因层级

```
表层原因（症状层）：
  - task-id 检测缺失
  - date_source 标注不全
  - setup_safe_output() 未调用

直接原因（实现层）：
  - 实现者聚焦主要场景忽略替代条件
  - 实现者主观判断文本模式可简化
  - 添加新检查时遗忘编码安全防御

根本原因（机制层）：
  - 缺乏"docstring 声明 vs 代码实现"一致性自动检查
  - 缺乏"文本模式 vs JSON 模式"输出完整性一致性规范
  - 缺乏"Python 脚本编码安全防御调用"自动化检查

更深层原因（治理层）：
  - SpecWeave 缺乏统一的"实现一致性自动检查"框架
  - 各类一致性检查分散在人工流程中，没有系统化整合
```

## 异常分类

### 异常 1：PARTIAL 检查点的发现机制

- **类型**：流程有效捕获
- **特征**：2 个 PARTIAL 通过 Spec 模式第七阶段独立验证机制发现
- **评估**：这是流程的成功，不是失败——证明独立验证机制有效
- **改进方向**：将独立验证机制系统化，扩大覆盖范围

### 异常 2：GBK 编码崩溃的发现时机

- **类型**：端到端验证发现
- **特征**：pre_commit.py GBK 崩溃在 Task 7 端到端验证时发现，而非代码审查阶段
- **评估**：发现时机较晚（实现完成后），但通过端到端验证有效捕获
- **改进方向**：将编码安全检查前移至实现阶段（编码安全自动检查）

### 异常 3：工具超时频次

- **类型**：环境噪声
- **特征**：Edit/Write 工具频繁超时（IDE Command timeout），但实际操作已成功
- **评估**：这是 IDE 层面问题，非操作失败——通过 Read 验证可确认
- **改进方向**：建立"超时后用 Read 验证"的标准操作模式

## 关键洞察

### 洞察 1：独立验证机制是质量保障的最后防线

本次 2 个 PARTIAL 均通过独立验证者（未参与实现的子代理）发现，证明"独立验证者 + checklist"机制能有效发现实现与规范的偏差。这一机制的价值在于：

- **视角独立性**：验证者未参与实现，不带"实现者偏见"
- **结构化覆盖**：checklist 提供系统性的检查点覆盖
- **客观证据**：基于代码内容给出证据，非主观判断

但当前机制存在覆盖盲区：checklist 模板未覆盖"docstring 声明 vs 代码实现"一致性，本次发现依赖验证者细致阅读 docstring。

### 洞察 2：多输出模式脚本的"模式间一致性"是隐性陷阱

当一个脚本支持多种输出模式（如文本模式与 JSON 模式）时，容易出现"模式间不一致"问题——JSON 模式输出完整，文本模式输出简化。这种不一致源于：

- **认知偏差**：实现者认为"机器读取需要完整，人类读取可以简化"
- **缺乏规范**：没有明确的"模式间一致性"要求
- **测试盲区**：测试通常分别验证各模式，不比较模式间一致性

这类问题在 SpecWeave 的多输出模式脚本中具有普遍性（check-file-placement.py、check-temp-lifecycle.py、verify-sitecustomize-autoload.py 都支持 --json）。

### 洞察 3：Windows 平台编码安全是跨环境兼容的常见陷阱

Windows 中文终端默认 GBK 编码，而 Python 脚本常输出 emoji 与 Unicode 字符。`setup_safe_output()` 函数提供了系统化的解决方案，但缺乏自动化检查机制确保所有需要的脚本都调用了此函数。

这是跨环境兼容问题的典型代表——在 Linux/Mac 开发环境下不会暴露（默认 UTF-8），但在 Windows 环境下会崩溃。SpecWeave 作为跨平台项目，需要建立"Windows 平台编码安全"的自动化检查机制。

### 洞察 4：表层问题背后有共性治理层根因

3 个系统性问题的表层表现各不相同（docstring 不一致、输出不完整、编码崩溃），但根本原因都指向同一个治理层根因：**缺乏统一的"实现一致性自动检查"框架**。

这提示改进方向不应是"头痛医头"地为每个问题单独建立检查，而应建立统一的"实现一致性自动检查"框架，系统化覆盖：
1. docstring 声明 vs 代码实现一致性
2. 文本模式 vs JSON 模式输出完整性一致性
3. Python 脚本编码安全防御调用检查

## 改进建议

### 建议 1（高优先级）：建立"实现一致性自动检查"框架

**目标**：系统化覆盖 3 类实现一致性问题，减少对人工记忆的依赖

**具体行动**：
1. 在 `.agents/scripts/` 下新增 `check-implementation-consistency.py` 脚本
2. 实现 3 个检查模块：
   - docstring 一致性检查（扫描 docstring 声明的功能，验证代码是否实现）
   - 输出模式一致性检查（对比 JSON 模式与文本模式的输出字段完整性）
   - 编码安全防御检查（扫描输出 emoji/非 ASCII 字符的 Python 脚本，验证是否调用 `setup_safe_output()`）
3. 集成到 CI 质量门禁与 pre-commit 钩子

**预期收益**：将 3 类实现一致性问题从"人工发现"升级为"自动检查"，减少实现偏差

**实施成本**：中等（需新增 1 个脚本 + 3 个检查模块）

### 建议 2（高优先级）：扩展 Spec 模式第七阶段 checklist 模板

**目标**：在 checklist 模板中增加"实现一致性"维度的检查项

**具体行动**：
1. 在 checklist 模板中新增"docstring 声明 vs 代码实现一致性"检查项
2. 新增"文本模式 vs JSON 模式输出完整性一致性"检查项（对多输出模式脚本）
3. 新增"Python 脚本编码安全防御调用"检查项（对输出 emoji/非 ASCII 的脚本）

**预期收益**：将实现一致性检查纳入标准验证流程，避免依赖独立验证者的细致阅读

**实施成本**：低（模板更新）

### 建议 3（中优先级）：建立"脚本输出规范"文档

**目标**：明确多输出模式脚本的输出完整性要求

**具体行动**：
1. 在 `.agents/docs/knowledge/best-practices/` 下新增 `script-output-convention.md`
2. 明确"文本模式与 JSON 模式的输出字段完整性应该一致"（除非有明确理由简化）
3. 明确"每项标注"类要求适用于所有输出模式

**预期收益**：消除"实现者主观判断文本模式可简化"的认知偏差

**实施成本**：低（文档撰写）
