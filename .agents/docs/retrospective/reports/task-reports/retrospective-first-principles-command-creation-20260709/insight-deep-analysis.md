## 三、根因分析（5-Whys）

### 3.1 问题描述

**核心问题**：为什么 spec 阶段未发现 `self-cognition.md` 不存在？

### 3.2 5-Whys 追问链

| 层级 | 问题 | 答案 | 根因类型 |
|------|------|------|---------|
| **Why1** | 为什么 spec 阶段未发现 self-cognition.md 不存在？ | spec 中没有引用验证步骤，创建 spec 时未检查引用的文件是否存在 | 流程缺失 |
| **Why2** | 为什么 spec 中没有引用验证步骤？ | spec 模板没有要求验证引用的文件存在性，模板设计未包含此检查项 | 模板缺陷 |
| **Why3** | 为什么 spec 模板没有要求验证引用？ | 项目没有建立 modules 目录的文件清单，spec 创建者无法快速查询可用文件 | 资源缺失 |
| **Why4** | 为什么没有 modules 目录的文件清单？ | 没有自动化工具检查 spec 中的引用有效性，引用验证依赖人工记忆 | 工具缺失 |
| **Why5** | 为什么引用验证意识不足？ | 习惯性地假设引用的文件存在，缺少"引用必须验证"的流程约束和意识 | 意识缺失 |

### 3.3 根因定位

```
Why1（流程缺失）── 直接原因，可在 spec 流程中增加验证步骤解决
    ↑
Why2（模板缺陷）── 流程原因，需修改 spec 模板
    ↑
Why3（资源缺失）── 资源原因，需创建 modules 文件清单
    ↑
Why4（工具缺失）── 工具原因，需开发自动化验证工具
    ↑
Why5（意识缺失）── 根本原因，需建立"引用必须验证"的流程约束
```

### 3.4 根因结论

**根本原因**：引用验证意识不足，习惯性地假设引用的文件存在，缺少"引用必须验证"的流程约束。

**系统性根因**：spec 模板没有"引用验证"检查项，导致所有 spec 都可能存在引用未验证的问题。

**解决优先级**：
1. **立即解决**：在 spec 模板中增加引用验证检查项（ACT-001）
2. **短期解决**：创建 modules 目录文件清单（ACT-002）
3. **中期解决**：建立指令集关联模块存在性检查清单（ACT-003）

- **影响**：如果验证阶段不严格，可能产出物中存在断链
