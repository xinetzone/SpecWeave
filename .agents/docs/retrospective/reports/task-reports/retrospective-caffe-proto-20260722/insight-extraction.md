## 方法论模式萃取

> 以下3个方法论模式萃取自第9.2节「可复用方法论」，按六步标准萃取法（触发场景→核心做法→反模式→边界条件→检验标准→迁移示例）展开。

### 模式1: 双层架构的项目组织

**一句话描述**：当需要分析、包装或扩展大型第三方开源项目时，通过外层最小化包装 + 内层只读上游的双层架构实现低耦合、高可控的项目组织。

- **触发场景**：你需要基于一个大型第三方开源项目（数万行代码、复杂构建系统）构建工具链或分析平台，但只需要其中一小部分能力（如 protobuf 定义、特定算法模块），且不希望引入上游的完整构建依赖。
- **核心做法**（5步）：
  1. **识别最小依赖子集**：分析上游项目，确定你真正需要的模块（如 caffe.proto 定义），排除无关模块（如 CUDA 内核、数据加载器）
  2. **创建外层最小化包装**：在外层目录仅放置必要的包装代码（Python 工具链、proto 定义），构建系统独立于上游
  3. **内层上游保持原样**：将上游代码作为 `vendor/` 或 `upstream/` 子目录引入，标记为只读，不修改原始源码
  4. **通过 AGENTS.md 明确边界**：在 AGENTS.md 中定义两层边界和协作规则——外层不依赖内层的构建系统，内层作为参考只读
  5. **提供桥接机制**：当外层需要引用内层数据时，通过复制/符号链接/代码生成等方式桥接，而非直接依赖
- **反模式**（来自本项目实际教训）：
  - ❌ **混入上游构建系统**：外层直接 `add_subdirectory(caffex/)` 引入上游 CMake 树，导致构建复杂度爆炸、依赖冲突
  - ❌ **直接修改上游源码**：为了快速修复 Bug 直接修改 `caffex/` 内文件，导致上游更新时合并冲突、diff 不可追溯
  - ❌ **边界模糊**：外层代码同时 import 内层模块和外层模块，阅读者无法区分哪些是自有代码、哪些是上游代码
  - ❌ **过度包装**：将上游 80% 的功能都包装进来，失去"最小化"的初衷，变成另一个 fork
- **边界条件**：
  - 上游项目规模足够大（>100 文件），才有必要引入双层架构；小项目（<20 文件）直接 fork 或依赖管理更高效
  - 你需要的模块与上游其他模块耦合度低，可以独立提取；如果强耦合（如依赖上游内部工具类），双层架构的维护成本会超过收益
  - 上游更新频率低或你需要锁定特定版本；如果上游频繁更新且你需要跟踪最新版，考虑使用 git submodule + patch 机制
- **检验标准**：
  - ✅ 外层构建系统不包含任何上游 CMakeLists.txt / Makefile 引用
  - ✅ 可以删除 `caffex/` 目录后外层代码仍能通过语法检查（虽不能运行），证明无硬依赖
  - ✅ AGENTS.md 中明确标注两层边界，新协作者在 5 分钟内理解项目结构
  - ✅ 从上游提取的代码量 < 上游总代码量的 20%
- **迁移示例**（跨领域）：
  - **Web 前端框架迁移**：将大型 AngularJS 单体应用逐步迁移到 React，外层新建 React 组件，内层保留 AngularJS 作为只读参考，通过 iframe/Web Component 桥接，每次迁移一个页面
  - **数据库迁移**：从 Oracle 迁移到 PostgreSQL，外层新建 PostgreSQL schema + 数据访问层，内层保留 Oracle DDL 作为参考，通过数据同步管道桥接，逐步切换读写流量
  - **机器学习框架切换**：从 TensorFlow 迁移到 PyTorch，外层新建 PyTorch 模型定义 + 训练循环，内层保留 TF 代码作为精度对照参考，逐算子验证数值等价性

### 模式2: Agent 友好的上下文路由

**一句话描述**：当项目结构复杂且需要 AI 协作者（或新开发者）快速理解项目并定位目标代码时，通过 AGENTS.md 入口 + 分层索引 + 任务路由表的三层文档体系，实现"零上下文"的项目导航。

- **触发场景**：项目包含多个模块、多种技术栈、多层目录结构，且你需要频繁与 AI 协作者（如 Trae/Cursor/Copilot）协作，希望 AI 能在无需反复解释架构的情况下自主完成任务。
- **核心做法**（4步）：
  1. **AGENTS.md 作为唯一入口**：在项目根目录放置 AGENTS.md，包含项目一句话描述 + 路由表（任务类型→文件映射），AI 启动时自动读取
  2. **architecture-map.md 提供全局架构索引**：列出所有核心组件（模块/服务/类），每个组件附一句话说明 + 文件路径，形成"项目地图"
  3. **context-routing.md 提供任务→文件映射**：按任务类型（如"修改 protobuf 定义"、"添加 TVM 算子"）列出相关文件清单，AI 按图索骥
  4. **每个文件/目录有"一句话说明"**：在 architecture-map.md 中为每个文件提供一行注释，说明其职责和边界
- **反模式**（来自本项目实际教训）：
  - ❌ **只有 README.md 没有 AGENTS.md**：README 面向人类（大段说明、截图），AI 需要的是结构化路由表；README 的叙事风格不适合 AI 快速解析
  - ❌ **路由表过时**：架构变更后未同步更新 context-routing.md，导致 AI 被引导到错误文件，产生幻觉性修改
  - ❌ **一句话说明过于模糊**：如"utils.py — 工具函数"，AI 无法判断该文件是否与当前任务相关；应写为"utils.py — TVM Relax 算子（Conv2D/ConvTranspose2D/L2Norm），将 Caffe Layer 参数映射为 nn.Module"
  - ❌ **入口文件过多**：同时有 AGENTS.md、CONTRIBUTING.md、ARCHITECTURE.md、DEVELOPMENT.md 等多个入口，AI 不知道该读哪个，或全部读取造成 token 浪费
- **边界条件**：
  - 项目规模：单文件脚本或 <5 个文件的微型项目不需要此模式，直接内联注释即可
  - 协作频率：如果你不常与 AI 协作者一起工作，或项目主要由单人维护，投入产出比不高
  - 团队接受度：如果团队不习惯维护 AGENTS.md，可以在 CI 中加检查脚本，确保架构变更时路由表同步更新
- **检验标准**：
  - ✅ AI 协作者可以通过 AGENTS.md → context-routing.md → 具体文件的路径，在 ≤3 步内定位到任何目标代码
  - ✅ 新开发者在不阅读任何源码的情况下，通过 architecture-map.md 在 10 分钟内画出项目架构图
  - ✅ 每次架构变更（新增/删除/重命名模块）后，context-routing.md 同步更新（可通过 CI 检查）
  - ✅ AGENTS.md 的 token 开销 < 500 tokens（精简原则），让 AI 在预算内完成路由
- **迁移示例**（跨领域）：
  - **微服务架构文档**：每个微服务仓库放置 AGENTS.md，列出该服务的 API 端点、数据库表、消息队列 Topic，AI 可自主完成跨服务调用链追踪
  - **Monorepo 导航**：在根目录 AGENTS.md 中列出所有 packages/ 子项目的一句话说用 + 依赖关系图，AI 可判断修改影响范围
  - **数据管道文档**：为 ETL 管道的每个阶段（Extract→Transform→Load）编写 context-routing.md，AI 知道修改某个 Transform 需要检查哪些上下游

### 模式3: 代码生成器的自验证设计

**一句话描述**：当代码生成工具需要跨版本运行时环境兼容时，通过"预检→生成→验证"三步流程设计，确保生成的代码在目标环境中立即可用，而非"生成后才发现跑不起来"。

- **触发场景**：你正在编写代码生成工具（如 protoc wrapper、OpenAPI 代码生成器、模板引擎），且生成的代码需要在不同版本的运行时环境中运行，版本不匹配会导致生成代码编译失败或运行时崩溃。
- **核心做法**（3步）：
  1. **预检（Pre-check）**：在生成前检查工具链版本与目标运行时版本的兼容性。如 gen_proto.py 的 `version_check()` 检查 protoc 版本与 protobuf Python 运行时版本的对应关系，给出明确的兼容/不兼容判定和修复建议
  2. **生成（Generate）**：执行代码生成，捕获所有编译错误和警告，而非静默忽略
  3. **验证（Verify）**：生成后立即 import 生成的代码，并执行最小化功能测试（如序列化/反序列化往返），确保代码在目标运行时中实际可用
- **反模式**（来自本项目实际教训）：
  - ❌ **跳过验证步骤**：生成代码后直接提交，假设"反正 protoc 没报错就是没问题"，结果在 CI 或用户环境中因 protobuf 版本不匹配而 import 失败
  - ❌ **版本兼容性硬编码**：在代码中写死 `assert protoc_version == "3.21.0"`，导致每次 protoc 升级都要修改代码
  - ❌ **只检查不修复**：预检发现问题后只打印警告继续执行，生成不可用代码后让用户自行排查；正确做法是给出明确的修复建议（如"请安装 grpcio-tools==1.62.0"）
  - ❌ **验证范围过窄**：只验证 import 成功，不验证序列化/反序列化等核心功能，导致运行时错误延迟暴露
- **边界条件**：
  - 如果代码生成器只服务于单一固定版本环境（如 Docker 镜像内固定 protoc 版本），自验证的收益有限，可以简化为预检步骤
  - 如果生成的代码量极小（<10 行）且不依赖运行时版本敏感的特性，验证步骤可以简化为 import 检查
  - 如果代码生成器本身是第三方工具（如直接用 protoc），验证逻辑应放在调用脚本中，而非修改工具本身
- **检验标准**：
  - ✅ 生成代码后 3 秒内完成 import + 序列化往返验证，无需人工介入
  - ✅ 版本兼容性检查矩阵覆盖至少 3 个大版本（如 protoc 25.x/29.x/35.x）
  - ✅ 不兼容时给出明确的错误信息，包含"当前版本→期望版本→修复命令"三段式建议
  - ✅ 验证失败时生成代码不覆盖已有可用代码（防止"好的被坏的替换"）
- **迁移示例**（跨领域）：
  - **OpenAPI → SDK 代码生成**：生成 Python/TypeScript/Go SDK 后，立即 import + 编译 + 调用一个模拟 API 端点，确保 SDK 可用
  - **数据库迁移脚本生成**：生成 SQL 迁移脚本后，在临时数据库实例中执行 `UP` + `DOWN` 往返，验证迁移的可逆性和无语法错误
  - **模板引擎（如 Jinja2 → HTML）**：生成 HTML 后，用 HTML 验证器检查标签闭合、属性合法性，防止模板错误导致页面渲染异常

## 最佳实践

> 以下 4 个最佳实践萃取自第9.3节，按「实践描述→适用场景→反模式→本项目证据」四要素展开。

### 实践1: 显式传入关键参数，不依赖默认值

- **实践描述**：调用框架 API 时，对所有影响语义正确性的关键参数显式传值，不依赖框架的隐式默认值。特别是当"不传参数"和"传默认值"在语义上不同时（如 `nn.Parameter` 的 `init` 参数缺失→零初始化 vs 显式传入→常量初始化），隐式默认值可能导致隐蔽 Bug。
- **适用场景**：
  - 使用深度学习框架（TVM/PyTorch/TensorFlow）的 `Parameter`/`Variable` 创建 API
  - 调用 ORM 框架的字段定义（如 `default` vs `server_default`）
  - 任何框架 API 的默认值在文档中未明确说明或与直觉不一致时
- **反模式**（什么情况下不要这样做）：
  - ❌ 对框架明确标注为"可选"且默认值语义清晰的参数也强制显式传入，导致代码冗长（如 `dtype=tf.float32` 在 TensorFlow 中本身是默认值）
  - ❌ 对所有参数不加区分地显式传入，导致代码行数膨胀 2-3 倍，降低可读性
  - 正确做法：只对"影响语义正确性"的关键参数显式传入，对"不影响语义的配置项"（如日志级别、格式化选项）保留默认值
- **本项目证据**：L2Norm `scale_init` Bug——`nn.Parameter(scale_shape, name="scale")` 因未传入 `init` 参数，scale 被默认零初始化，而非期望的 `scale_init` 值（默认 1.0）。修复：`nn.Parameter(scale_shape, name="scale", init=relax.const(self.scale_init, dtype=None))`。详见 [README.md](README.md#52-问题1l2norm-scale-初始化-bug) 第5.2节。

### 实践2: 可选依赖提供降级机制

- **实践描述**：对于非核心的可选依赖（如 TVM、CUDA、特定云 SDK），使用 `try/except ImportError` + `pytest.skip()` 机制，在依赖不可用时优雅降级而非直接报错。这保证了核心功能在最小依赖环境下仍可运行，同时为有完整环境的用户提供全部能力。
- **适用场景**：
  - 项目依赖大型框架（TVM/PyTorch/TensorFlow）但核心功能不需要它们
  - 跨平台工具需要适配不同操作系统的可选能力（如 Windows 无 `fcntl`，macOS 无 `systemd`）
  - CI 环境中某些依赖安装成本高或不可用，但核心测试仍需运行
- **反模式**（什么情况下不要这样做）：
  - ❌ 对核心依赖也使用降级机制——如果 protobuf 不可用，整个项目都无法运行，此时应该直接报错而非静默降级
  - ❌ 降级时只打印 warning 不记录，导致用户不知道某些功能缺失
  - ❌ 降级后的行为与正常行为差异过大（如降级后返回空数据而非 mock 数据），导致下游逻辑出错
- **本项目证据**：`test_l2norm.py` 中 TVM 测试用例使用 `pytest.skip()` 在 TVM 不可用时自动跳过，而非报错。protobuf 多版本兼容性检查矩阵覆盖 5.x/6.x/7.x 三个大版本。详见 [README.md](README.md#54-问题模式分析) 第5.4节。

### 实践3: 工具函数保持类型无关

- **实践描述**：编写通用工具函数时，不针对特定类型写分支逻辑（如 `if layer_type == "Convolution": ... elif layer_type == "BatchNorm": ...`），而是基于类型的通用属性（如 `layer.top`、`layer.bottom`）编写统一处理逻辑。这遵循开闭原则——新增层类型无需修改工具代码。
- **适用场景**：
  - 处理 protobuf 的 `oneof` 或 `repeated` 消息类型，需要对多种子类型做统一处理
  - 编写插件系统或中间件，需要对不同类型的请求/消息做统一拦截处理
  - 编写数据转换管道，输入格式多样但输出格式统一
- **反模式**（什么情况下不要这样做）：
  - ❌ 当不同类型的处理逻辑差异巨大（如 Conv2D 需要 NCHW→NHWC 布局转换，而 ReLU 完全不需要），强行统一反而导致代码晦涩——此时应使用策略模式，每个类型有独立处理器
  - ❌ 为了"类型无关"而牺牲性能——如果某个类型可以走快速路径（如 `type == "Input"` 直接跳过），应该保留特殊处理但标注清楚原因
- **本项目证据**：`caffe_utils.py`（147 行）的 `unity_struct` 标准化流程中，仅 `Input` 层因语义特殊（无 `bottom` blob）需要特殊处理，其余所有层类型（75+ 种）均通过通用的 `layer.top`/`layer.bottom` 属性统一处理，无需逐个写分支。详见 [README.md](README.md#决策清单) 第4章决策#5。

### 实践4: 生成代码附带验证步骤

- **实践描述**：任何代码生成脚本（protoc、模板引擎、代码脚手架）在生成代码后，立即执行最小化验证步骤——import 生成的模块 + 执行核心功能测试（如序列化/反序列化往返）。这确保生成代码在目标环境中实际可用，而非"生成成功但运行失败"。
- **适用场景**：
  - protobuf/gRPC 代码生成（protoc → `_pb2.py`）
  - OpenAPI/Swagger → SDK 代码生成
  - 模板引擎（Jinja2/EJS/Handlebars）→ 目标代码生成
  - 数据库迁移脚本生成（Alembic/Flyway）
- **反模式**（什么情况下不要这样做）：
  - ❌ 验证步骤本身有副作用（如验证中写入数据库或发送网络请求），应使用 mock/sandbox 隔离
  - ❌ 验证耗时过长（>30 秒），影响开发体验——此时应将验证步骤作为可选 CI 流程，日常开发中跳过
  - ❌ 验证失败时覆盖已有可用代码，导致"好的被坏的替换"——验证应在临时目录进行，通过后再替换
- **本项目证据**：`gen_proto.py` 的 `verify_generated()` 函数在生成 `caffe_pb2.py` 后立即 import + 创建 NetParameter 消息 + 序列化/反序列化验证，确保生成的代码在目标 protobuf 运行时版本中可用。详见 [README.md](README.md#方法论3代码生成器的自验证设计) 第9.2节。

## 问题模式分析

> 以下 2 个问题模式萃取自第5.4节，按「根因→表现→预防措施」三要素展开。

### 环境兼容性问题模式

- **根因**：项目的"桥梁"定位——连接 Caffe、TVM、protobuf 三个独立生态系统，每个系统有独立的版本号体系、安装方式（pip/conda/源码编译）和兼容性约束。protoc 版本（25.x~35.x）与 Python protobuf 运行时版本（4.x~7.x）之间存在非线性的版本对应关系，且不同安装方式（grpcio-tools/protoc/conda libprotobuf）提供的版本组合不同。
- **表现**：
  - 生成的 `_pb2.py` 文件的 gencode 版本与 Python protobuf 运行时版本不兼容，导致 `import caffe_pb2` 失败
  - `caffe_fuse.py` 处理新旧 protobuf 格式时，`layer` 字段（旧版）与 `layers` 字段（新版）名称不一致，代码需同时兼容两种格式
  - TVM 不可用时测试直接报错，而非优雅跳过
- **预防措施**：
  - 在代码生成脚本中建立版本兼容性检查矩阵（如 gen_proto.py 的 `version_check()`），覆盖至少 3 个大版本
  - 优先推荐版本精确匹配的安装方式（如 grpcio-tools 与 protobuf 版本一一对应），降低兼容性风险
  - 对可选依赖（如 TVM）使用 `try/except ImportError` + `pytest.skip()` 降级机制
  - 对新旧格式兼容问题，使用运行时字段检测（如 `hasattr(net, 'layers')`）而非硬编码字段名

### API 默认行为问题模式

- **根因**：框架 API 的隐式默认值与"不传参数"之间的语义差异。对于 TVM Relax 的 `nn.Parameter`，"不传 `init` 参数"意味着零初始化，而"显式传入 `init=relax.const(value)`"意味着常量初始化。开发者凭直觉假设"不传参数 = 使用默认值 = 文档中声明的默认值"，但实际行为是"不传参数 = 零初始化"（框架内部实现细节），文档与实现之间存在语义鸿沟。
- **表现**：
  - `L2Norm.__post_init__` 中创建的 `nn.Parameter(scale_shape, name="scale")` 的 scale 参数被默认初始化为零，而非期望的 `scale_init` 值（默认 1.0）
  - 用户在使用 L2Norm 算子时发现输出全为零，回溯到 scale 参数未正确初始化
  - Bug 隐蔽性高——代码语法正确、无运行时错误，但数值结果完全错误，只有通过数值测试才能发现
- **预防措施**：
  - 对新引入的框架 API 调用，显式传入所有影响语义正确性的关键参数，不依赖隐式默认值
  - 在代码审查中增加"默认值依赖检查"——对于每个框架 API 调用，确认不传参数的行为是否与期望一致
  - 为关键模块（特别是涉及数值计算的算子）编写数值精度测试，验证输入→输出的数值正确性，而非仅验证代码不报错
  - 在项目知识库中记录框架 API 的"陷阱"（如 nn.Parameter 的 init 默认行为），供后续开发者参考

## 设计模式索引

> 以下 6 个设计模式来自附录B「设计模式索引」，总结其在本项目中的应用位置与核心思想。

| # | 模式 | 应用位置 | 核心思想 | 一句话总结 |
|---|------|---------|---------|-----------|
| 1 | **自注册工厂** | `gen_proto.py` → protoc 自动发现 | 自动化代码生成——通过脚本自动发现 protoc 路径、版本匹配，无需手动配置 | 将"找 protoc → 匹配版本 → 生成代码"的重复操作固化为自动化工厂 |
| 2 | **策略模式** | `L2Norm` 的 `across_spatial` / `channel_shared` 参数 | 参数化行为切换——通过 bool 参数在两种 L2Norm 实现（SSD-style / ParseNet-style）之间切换，而非创建两个独立类 | 两种行为模式共享同一骨架，差异仅在 scale 参数的维度广播方式 |
| 3 | **适配器模式** | `utils.py` → TVM Relax `nn.Module` | 将 Caffe Layer 参数（`ConvolutionParameter`、`LRNParameter`）适配为 TVM Relax 的 `nn.Module` 子类，实现异构框架间的类型映射 | 一个 Caffe Layer → 一个 TVM nn.Module，参数结构一一对应 |
| 4 | **管道模式** | `caffe_utils.py` 的 `unity_struct` 三步流程 | 将模型标准化分解为"输入统一→in-place 修复→命名规范"三个独立步骤，每步的输出是下步的输入，形成处理管道 | 每步职责单一，可独立测试、独立替换，符合 Unix 管道哲学 |
| 5 | **降级模式** | `test_l2norm.py` 的 `pytest.skip()` 机制 | 可选依赖（TVM）不可用时优雅降级，跳过相关测试而非报错，保证核心测试在最小依赖环境下仍可运行 | "有条件就用，没条件就跳过"——不给用户添堵 |
| 6 | **模板方法** | `gen_proto.py` 的三步流程（预检→生成→验证） | 固定流程骨架（预检→生成→验证）+ 可替换步骤（不同 protoc 版本用不同的命令行参数），子步骤可重写但流程顺序不可变 | 流程骨架固定，保证每次生成都经过完整验证；步骤可替换，适配不同版本 protoc |
