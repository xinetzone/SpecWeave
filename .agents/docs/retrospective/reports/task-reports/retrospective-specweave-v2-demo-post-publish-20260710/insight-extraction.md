# 洞察萃取：SpecWeave v2 Demo帖发布任务

## 可复用洞察（按价值排序）

### 洞察1：API优先于UI自动化——当前端框架有复杂交互时，直接调用后端API是最低摩擦路径

**场景**：Discourse使用Ember.js框架，前端有复杂的双向绑定、自动保存、模态框、动态组件等交互逻辑，通过UI操作发帖需要处理7+个框架特定问题。

**模式**：
- UI自动化适合"人做起来简单"的操作（点击按钮、填写简单表单）
- 当框架有复杂数据绑定/自动保存/动态加载时，UI操作的试错成本呈指数级增长
- 此时应直接寻找后端API——现代Web应用几乎都有REST/GraphQL API，且API是为机器调用设计的，没有UI层的各种交互陷阱

**验证**：UI方案尝试7+次，耗时约40分钟仍未成功；API方案从设计到执行成功仅用约15分钟，一次通过。

**可复用性**：⭐⭐⭐⭐⭐（任何Web平台自动化任务都适用）

---

### 洞察2：框架感知操作原则——修改UI控件值时必须使用框架的原生setter并触发正确事件

**场景**：Ember.js（以及React/Vue/Angular等现代框架）通过数据绑定监听控件值变化，直接设置DOM元素的`.value`属性不会触发框架的脏检查/响应式更新。

**模式**：
```javascript
// ❌ 错误：直接设置value，框架无感知
textarea.value = content;

// ✅ 正确：使用nativeSetter + 触发input/change事件
const nativeSetter = Object.getOwnPropertyDescriptor(
  window.HTMLTextAreaElement.prototype, 'value'
).set;
nativeSetter.call(textarea, content);
textarea.focus();
textarea.dispatchEvent(new Event('input', {bubbles: true}));
textarea.dispatchEvent(new Event('change', {bubbles: true}));
```

**可复用性**：⭐⭐⭐⭐⭐（所有现代JS框架的UI自动化都适用）

---

### 洞察3：base64编码是解决"长文本跨上下文传递"问题的通用方案

**场景**：需要在浏览器控制台执行JS，而JS需要包含超长Markdown内容（约10KB）。直接在字符串中嵌入会遇到引号转义、换行符、特殊字符、Python三引号嵌套等问题。

**模式**：
1. Python端：将内容base64编码为ASCII字符串
2. JS端：atob()解码为二进制→TextDecoder解码为UTF-8字符串
3. 优势：完全避免转义问题，纯ASCII可安全嵌入任何代码上下文

```python
# Python端编码
b64 = base64.b64encode(content.encode('utf-8')).decode('ascii')
```
```javascript
// JS端解码
const binStr = atob(b64);
const bytes = new Uint8Array(binStr.length);
for (let i = 0; i < binStr.length; i++) bytes[i] = binStr.charCodeAt(i);
const raw = new TextDecoder('utf-8').decode(bytes);
```

**可复用性**：⭐⭐⭐⭐⭐（任何需要跨语言/跨上下文传递长文本的场景）

### 洞察4：四层验证法——API返回200不等于操作成功，必须多层验证

**场景**：Discourse API返回状态200，但action是"enqueued"（进入审核队列），帖子不会立即公开显示。直接访问topic ID返回404，因为未审核的帖子对非作者不可见。

**模式**（Web操作验证四层法）：
1. **API响应层**：检查HTTP状态码+业务状态码（action/error字段）
2. **直接访问层**：尝试直接访问资源URL（可能因权限/审核返回404，需结合场景判断）
3. **列表索引层**：在对应分类/列表页面查找资源是否出现
4. **个人中心层**：在"我的帖子/待处理/草稿"等用户专属页面验证最终状态

**可复用性**：⭐⭐⭐⭐（任何Web平台的发布/提交/创建操作都适用）

### 洞察5：第一性原理复盘能发现"看不见"的关键缺口

**场景**：用户最初提到"提交过初版，提交最新版本Demo"，直觉反应是"编辑旧帖更新内容"。但通过第一性原理复盘（回到"我们到底要做什么？为什么做？"），发现了两个关键问题：
1. 旧帖在"大赛报名专区"而非"大赛初赛专区"（分区错误）
2. 旧帖是v1版本（142次提交），而当前项目是v2（1,256次提交），内容增长近9倍

**模式**：收到"更新X"类需求时，不要直接执行更新，先问：
- X的根本目的是什么？
- X的当前状态是否仍符合目的？（分区/分类/版本/受众）
- 是"修补"还是"重做"更符合根本目的？

**可复用性**：⭐⭐⭐⭐（所有需求分析、任务规划场景）

### 洞察6：临时文件"即用即清"原则——任务执行中产生的临时脚本/文件应在任务结束时立即清理

**场景**：任务过程中为解决问题创建了5个临时Python/JS文件（temp_create_post.py等），如果不清理会污染工作区。

**模式**：
- 临时文件使用统一前缀（temp_/tmp_/test_）便于识别
- 任务结束时（无论成功失败）立即清理所有临时文件
- 可复用的工具逻辑应沉淀到.agents/scripts/而非留在根目录

**可复用性**：⭐⭐⭐（所有开发/运维任务）

## 5-Whys根因分析：为什么UI操作失败了7次才切换到API？

**问题**：前端UI操作发帖尝试7+次，耗时约40分钟仍未成功，最终切换到REST API方案一次通过。

| 层级 | 问题 | 回答 |
|------|------|------|
| Why1 | 为什么UI操作反复失败？ | 因为Ember.js框架有复杂的双向绑定、自动草稿保存、模态框拦截、动态组件加载等交互陷阱，每解决一个问题又遇到下一个 |
| Why2 | 为什么一开始选择UI操作路径，没有考虑API？ | 因为"发帖"在人类直觉中是一个UI操作（人发帖就是点击按钮填写表单），机器自动化也默认模拟人类操作路径 |
| Why3 | 为什么这种"模拟人类"的直觉在机器自动化中是错的？ | 因为UI是为人类设计的——人类有视觉反馈、容错能力、模糊理解；API是为机器设计的——确定性强、输入输出明确、没有交互摩擦 |
| Why4 | 为什么试错7次后才切换，而不是更早？ | 因为每次失败都看起来是"只差最后一步"（标题好了→标签选了→被模态框挡住→草稿保存了），存在"再试一次就能成功"的沉没成本谬误 |
| Why5 | 为什么缺乏"及时切换方案"的判断机制？ | 因为没有沉淀Web自动化任务的方案选择决策树，依赖经验直觉而非系统化判断框架——不知道什么时候应该放弃UI路径 |

**根本原因**：缺乏"Web平台自动化任务"的方案选择决策框架，默认依赖"模拟人类操作"的直觉路径，且没有设定"试错N次后必须切换方案"的止损机制。

**纠正措施**：
1. 沉淀"Web自动化方案选择决策树"（见改进建议P1-5）
2. 设定"3次失败强制切换"止损规则：同一方案连续失败3次后，必须停下来重新评估方案而非继续试错
3. 将"API优先"作为Web自动化的默认选项，UI自动化作为备选而非首选

## 改进建议

### 高优先级（P0）

1. **沉淀零依赖的Discourse发帖工具**：基于本次验证的REST API方案，创建一个零第三方依赖的Python脚本（.agents/scripts/discourse-post.py），支持：
   - 从Markdown文件读取内容
   - 自动获取CSRF token（通过浏览器cookie或直接配置）
   - 支持指定category、tags、title
   - 返回待审核状态和post/topic ID
   - 这样下次类似任务不需要重新写临时脚本

2. **更新forum-automation.md知识库**：将本次验证的"API优先方案"、"框架感知操作"、"base64编码"、"四层验证法"等内容补充到论坛自动化知识库。

### 中优先级（P1）

3. **临时文件自动清理清单**：任务结束前的检查清单增加"清理临时文件"项。

4. **方案选择决策树**：为"Web平台自动化任务"沉淀决策树：
   - 简单操作（点击/输入）→ UI自动化
   - 复杂表单/框架交互 → 优先寻找API
   - API不可用 → 研究框架特定的事件触发方式

### 低优先级（P2）

5. **Playwright环境预装检查**：在依赖Playwright的Skill中增加环境检查，提前告知用户需要安装。
