# 洞察提取：第一性原理思维训练题库创建

## 1. 可复用模式萃取

### 模式1：练习题库三级设计模式

**模式名称**：practice-question-bank-three-tier-design

**触发场景**：为方法论/思维框架/技能创建训练题库时

**核心结构**：
```
1. 按方法论步骤分解专项练习（Step1-StepN）
2. 每专项内分三级难度：
   - 🌱入门：日常生活场景，降低门槛建立信心
   - 📚进阶：工作/学习场景，贴近实际应用
   - 🔥挑战：认知偏差/反直觉问题，深度训练
3. 误区识别专项：用反例训练识别误用
4. 综合案例：跨步骤完整流程演练
5. 所有答案折叠（details标签），提供思考框架而非标准答案
```

**关键要素**：
- 入门题必须足够简单（如"一支中性笔"、"一张A4纸"），让读者建立"我也能做到"的信心
- 进阶题必须贴近读者真实场景（会议效率、技术选型、时间管理）
- 挑战题必须触碰认知盲区（确认偏差、幸存者偏差、逆向思维）
- 误区识别必须包含真实反例（如Theranos），而非仅抽象描述
- 综合案例必须覆盖不同领域，展示方法通用性
- 参考答案永远标注"非唯一正确解"，展示思考路径而非结论

**成熟度**：L2（已在本项目验证成功，待更多场景复用确认）

**相关模式**：待沉淀至docs/retrospective/patterns/

### 模式2：知识型产出物质量双维度模型

**模式名称**：knowledge-artifact-quality-dual-dimension

**触发场景**：创建知识文档/教程/训练材料等非代码类产出物时

**核心洞察**：知识型产出物的质量有两个维度：
1. **可自动化验证维度**：文件名规范、链接有效性、格式一致性、frontmatter完整性、题数统计等
2. **不可自动化验证维度（核心价值）**：
   - 审慎态度：不做万能宣称、承认边界、包含失败案例
   - 批判性视角：有反例、有偏差提示、不盲目推崇方法
   - 认知诚实：标注"非唯一正确解"、区分事实/判断/偏见
   - 教学有效性：难度梯度合理、答案展示思考过程、贴近真实场景

**实践启示**：
- 代码开发的质量门（lint/test/CI）只能覆盖第一维度
- 知识创作必须在spec阶段将第二维度明确为人工审查项
- 不能因为"链接检查通过"就认为"文档质量合格"

**成熟度**：L1（本次首次萃取，待验证）

## 2. 系统性问题识别

### 问题1：内容创作型任务的subagent利用率不足

**现象**：本次2108行的内容创作全程单线程执行，没有利用subagent分章节并行

**根因分析（5-Whys）**：
1. 为什么单线程？→ 因为直接开始写，没有规划并行策略
2. 为什么没有规划并行？→ 因为spec的tasks.md虽然分了8个任务，但任务粒度是"Step1-3"、"Step4-6"这种大章节，且任务描述中未明确要求使用subagent
3. 为什么spec没有规划subagent？→ 因为内容创作被当作"简单的文档编写"，低估了工作量（2108行是相当大的文档）
4. 为什么低估了工作量？→ 因为spec阶段估算题数时（30-40题），没有估算每题需要的场景设计、任务描述、提示引导、参考答案编写的总工作量
5. **根本原因**：内容创作型任务缺乏"工作量估算→并行策略"的标准流程，默认单线程执行

**改进方向**：对于预估超过500行的内容创作任务，spec阶段应规划subagent分章节并行策略，最后由主线程统一收口（风格一致性检查、交叉引用、编号统一）。

### 问题2：spec中human-judgment AC缺乏可操作审查标准

**现象**：spec中"保持审慎态度和批判性视角"是human-judgment类型AC，但没有给出具体的审查checklist，只能凭主观判断

**根因**：知识文档的"审慎态度"确实难以完全量化，但可以给出具体的检查点：
- [ ] 是否包含"方法不是万能的"类边界说明？
- [ ] 是否包含失败案例/反例？
- [ ] 是否标注"非唯一正确解"？
- [ ] 商业案例是否有偏差提示（幸存者偏差等）？
- [ ] 是否区分了"第一性原理能做到什么"和"不能做到什么"？

**改进方向**：对于知识型产出物的human-judgment AC，在checklist.md中转化为具体的yes/no检查项，即使是人工审查也要有明确标准，而非"感觉质量还行"。

### 问题3：git-commit-utf8.py的add一致性检查在目录add时误报

**现象**：使用`git add <directory>/`时，git-commit-utf8.py报错"暂存区文件与指定列表不一致"

**根因**：目录add会递归添加目录下所有文件，脚本的文件列表比对逻辑无法正确处理目录参数展开后的文件列表变化

**改进方向**：脚本应支持目录参数，或者在文档中明确说明"必须逐个文件add，不能用目录"。本次通过手动add+临时文件提交绕过，属于工具使用层面的问题。

## 3. 经验教训总结

### 3.1 做对了什么

1. **在已有方法论框架上构建**：不是凭空设计题库，而是严格对齐08-methodology-framework.md的六步框架和7大误区，保证了内容与现有知识库的一致性
2. **三级难度设计**：入门→进阶→挑战的梯度设计让不同水平的读者都能找到适合的题目，避免了"要么太简单要么太难"的两极分化
3. **答案展示思考框架而非结论**：这是教学类材料最核心的设计决策——教会读者"怎么想"比告诉他们"答案是什么"重要得多
4. **批判性视角贯穿始终**：没有把第一性原理包装成万能方法，而是包含了Theranos反例、确认偏差训练、"现实优先于推理"等警示内容
5. **质量验证完整执行**：文件名规范、链接检查、内容完整性检查都做了，没有为了快跳过验证

### 3.2 下次可以改进什么

1. **大文档并行创作**：2000+行的文档应该拆成多个subagent任务，每个subagent负责一个章节（如Step1-2、Step3-4、Step5-6+误区、综合案例），最后统一收口调整风格和编号
2. **spec阶段细化human-judgment检查项**：将"审慎态度"转化为具体的yes/no检查清单
3. **题目编号可以更简单**：三级编号（题X.Y.Z）虽然结构严谨但维护成本高，可以考虑连续编号（题1-43）+Step标签的方式
4. **可以增加题目标签系统**：除了难度等级，还可以标签"认知偏差""假设挖掘""物理极限""商业场景""日常生活"等维度，方便读者按兴趣选题

### 3.3 可迁移的经验

1. **教学材料的核心是"展示思考过程"**：无论是编程教程、思维方法还是技能训练，好的教学材料永远是展示"我是怎么想的"而非"正确答案是什么"
2. **反例比正例更有教学价值**：告诉人们"什么是错的"、"这个方法容易被怎样误用"，比只展示"正确用法"更能帮助读者建立深度理解
3. **入门题必须足够"接地气"**：用日常生活中最常见的事物（一支笔、一张纸、一杯咖啡）作为入门训练，能极大降低心理门槛
4. **知识创作也要"原子化"**：就像代码提交要原子化一样，大文档创作也要拆分为独立章节，每个章节可独立验证，最后统一集成
