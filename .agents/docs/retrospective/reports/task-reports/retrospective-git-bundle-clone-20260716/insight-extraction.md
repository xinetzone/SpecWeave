### 触发场景

- 当你需要从本地 `.bundle` 文件克隆 Git 仓库时
- 适用于：离线代码交付、网络受限环境、仓库备份恢复、代码审计场景
- 不适用于：正常网络环境下从远程 Git 服务器克隆（直接用 git clone URL 即可）

### 核心做法（五步法）

1. **预检**：列出 bundle 目录，确认有哪些 `.bundle` 文件及配套 `.sha256` 校验文件
2. **校验**：对每个 bundle 执行 SHA256 完整性校验，校验不通过则中止
3. **预览（可选）**：用 `git bundle list-heads <bundle>` 查看包含哪些分支和标签
4. **克隆**：创建目标目录后执行 `git clone <bundle-path> <target-dir>`，多独立仓库可并行
5. **验证**：逐个进入仓库执行 `git status`，确认分支正确、工作区干净、与 origin 同步

### 反模式（不要这么做）

- ❌ **反模式1：跳过校验直接 clone**——bundle 文件可能在传输/存储中损坏，不校验可能得到损坏的仓库
- ❌ **反模式2：不做验证就认为成功**——大仓库克隆可能因磁盘IO/空间不足不完整，但git不报错，必须验证
- ❌ **反模式3：忘记调整 remote URL**——bundle克隆后origin指向本地文件路径，无法pull/fetch，必须按需调整
- ❌ **反模式4：有依赖关系的仓库并行**——如果仓库A依赖仓库B（如submodule），并行可能导致问题

### 检验标准

做完之后怎么知道做对了？
- 标准1：每个仓库 `git status` 显示 "nothing to commit, working tree clean"
- 标准2：每个仓库 `git log --oneline -1` 能正常显示最新提交
- 标准3：如配置了远程URL，`git remote -v` 显示正确的远程地址
- 标准4：目标目录存在且包含预期的文件结构

## 5-Whys 根因分析：为什么需要校验步骤？

**Why1**：为什么本次任务没有执行SHA256校验？
→ 因为只看到了bundle文件，没注意到配套的.sha256文件的存在和用途。

**Why2**：为什么没注意到校验文件？
→ 因为执行流程中没有"完整性校验"这一强制步骤的检查点。

**Why3**：为什么流程中缺少校验检查点？
→ 因为本次是首次执行bundle克隆任务，缺乏标准化流程指导。

**Why4**：为什么缺乏标准化流程？
→ 因为Git bundle操作不是日常高频操作，经验未沉淀为流程。

**Why5（根因）**：为什么经验没沉淀？
→ 因为任务完成后没有立即复盘萃取，经验停留在单次操作层面。

**改进方向**：本次复盘即沉淀候选流程，下次遇到bundle操作时执行并验证，升级为正式模式。
