# 洞察萃取

## 核心洞察

### 洞察1：错误的API能力假设是冗余代码的根源

**现象**：代码注释明确写着"rms_norm normalizes the last axis, so we transpose when needed"，并为此编写了`_normalize_rmsnorm_plan`函数（20行代码计算transpose排列），在计算图中插入了2个条件性transpose操作。

**根因**：在编写Normalize→RMSNorm转换逻辑时，开发者没有查阅TVM `rms_norm`算子的参考实现和API签名，而是基于"RMSNorm通常归一化最后一维"的领域经验做出了假设。这个假设在很多RMSNorm实现中确实成立（如LLaMA的RMSNorm固定在最后一维hidden_size上），但TVM的通用rms_norm算子设计上支持任意axis。

**影响**：
- 计算图中引入了2个不必要的数据重排操作，增加kernel launch开销和内存带宽消耗
- 多出约20行辅助函数+约9行分支条件代码
- 第一轮优化时，虽然识别出了重复调用问题并合并了判定+准备函数，但仍然保留了这个错误假设
- 用户通过提供参考实现文件（test_rms_norm.py）纠正了这个假设

**改进建议**：使用框架算子/第三方API前，强制执行"参考实现验证三步法"：(1)查numpy/python参考实现确认数学语义 (2)查API签名确认参数默认值和语义 (3)查项目中已有调用示例确认真实用法

---

### 洞察2：判定函数与准备函数分离导致重复计算和调用方负担

**现象**：原始代码中`_should_convert_normalize_to_rmsnorm()`内部调用了`_normalize_rmsnorm_params()`来验证参数有效性，返回bool后，调用方`convert_normalize`又再次调用`_normalize_rmsnorm_params()`和`_normalize_rmsnorm_plan()`来获取实际参数。

**根因**：遵循了"查询与命令分离（CQS）"原则的教条式应用——认为判定函数（query）应返回bool且无副作用，准备函数（command）执行实际工作。但在这个场景下，判定过程本身已经完成了大部分参数计算工作，分离反而导致重复计算。

**影响**：
- 参数校验和计算逻辑执行两次
- 调用方必须记忆"判定通过后需要调哪些函数获取参数"，增加认知负担
- 新增校验逻辑需要同步修改两个函数，维护成本高

**改进建议**：当判定过程需要执行大量计算或验证时，采用"TryPrepare"模式——一个函数同时完成判定和参数准备，成功返回参数元组，失败返回(None, None, ...)。这不是CQS的违反，而是对CQS在特定场景下的务实调整：try_prepare是一个"尝试性命令"，成功时产出结果，失败时不产生副作用。

### 洞察3：参考实现的价值具有双重性——不仅验证数学，更指导API使用

**现象**：第一轮优化时，虽然参考了test_rms_norm.py中的数学公式来验证L2→RMSNorm重参数化的正确性，但没有仔细对照其`rmsnorm_tensor`函数的实现方式——它直接在任意axis上调用`np.mean(np.square(x), axis=axis)`，没有任何transpose操作。

**根因**：参考实现被窄化为"数学正确性验证工具"，仅用于验证数值等价性，而忽略了其作为"API使用方式权威参考"的价值。

**影响**：第一轮优化在数学等价性验证上做了充分工作（数值测试），但API使用方式仍然保留了错误模式。直到用户显式指出参考文件中的实现方式，才发现transpose可以移除。

**改进建议**：当用户提供参考实现时，应同时从两个维度学习：(1)数学维度：验证算法正确性和数值精度 (2)工程维度：对照API使用方式、代码结构、是否有不必要的中间操作。

## 根因分析（5-Whys）

1. **为什么有冗余transpose？** 因为代码假设rms_norm只能归一化最后一维。
2. **为什么做这个假设？** 因为没有查阅rms_norm的参考实现和API签名。
3. **为什么没有查阅？** 因为"RMSNorm归一化最后一维"是深度学习领域的常见模式，凭经验认为这是普遍成立的。
4. **为什么经验在此失效？** 因为TVM的rms_norm是通用算子（支持任意axis），而LLaMA等特定实现固定在最后一维。通用框架算子vs特定模型实现的capability不同。
5. **为什么第一轮优化没发现？** 因为参考实现仅被用于数学验证，未被用于API使用方式对照。

**根本原因**：缺乏"使用算子/API前必须验证其能力边界"的编码纪律，过度依赖领域经验假设。
