## 3. 关键技术挑战

### 3.1 挑战一：双围栏语法的兼容与统一

**问题描述：** MyST支持冒号围栏（`:::`）和反引号围栏（```` ``` ````）两种形式，而现有解析器（第377-421行）仅实现反引号围栏处理逻辑。mdit-py-plugins 0.5.0虽提供colon_fence插件，但启用后需解决两种围栏的识别优先级、嵌套规则统一、错误恢复策略等问题。此外，存量文档中反引号代码块使用频繁，若冒号围栏引入不当，可能导致普通代码块被误识别为Directive。

**影响范围：**
- 解析器核心token处理逻辑（约40行代码需重构）
- Directive内容解析方法（第641-672行）
- 错误提示与用户文档
- 所有存量文档的兼容性验证（66份文档）

**复杂度评级：** 中等（P1）

**解决方向：**
1. 采用"内容类型决定围栏"的明确策略：Markdown内容类指令（admonitions/card等）使用`:::`，代码/公式类使用```` ``` ````
2. 启用colon_fence插件但增加二次校验：只有围栏内第一行匹配`{name}`模式才识别为Directive，否则按普通代码块处理
3. 围栏长度规则：要求Directive围栏长度为3（基础）、5、7...（奇数递增），与普通代码块的任意长度形成区分
4. 在解析器中添加围栏类型标记，后续处理逻辑可根据围栏类型选择不同渲染策略

### 3.2 挑战二：选项格式的扩展与向后兼容

**问题描述：** 当前解析器（第113行）仅支持`:key: value`单行选项格式，MyST还提供YAML块（`---`包裹）和内联选项（`.class/#id/key="value"`）两种格式。YAML块支持复杂数据结构（嵌套对象、数组），对复杂接口参数定义很有价值，但引入YAML解析会增加依赖；内联选项对语义提取价值有限但对渲染输出有用。更关键的是，三种选项格式的解析顺序、冲突处理、混合使用规则需要明确定义。

**影响范围：**
- `_parse_directive_content()`方法（第641-672行，需重写）
- 选项数据结构（当前为简单dict，需支持嵌套结构）
- 参数解析方法（第1227-1258行）
- 文档编写规范（需明确何种场景用何种格式）

**复杂度评级：** 中等偏高（P1）

**解决方向：**
1. 分阶段引入：第一阶段保持`:key: value`作为唯一标准格式，确保现有逻辑不变；第二阶段为自定义接口指令（`{interface}`/`{param}`/`{response}`）单独支持YAML块格式，因为这些场景确实需要复杂结构
2. YAML解析轻量化：不引入完整PyYAML依赖，而是针对Directive选项的有限场景实现简化解析（仅支持平铺键值和简单数组），或在TOML加载逻辑（第277-341行）基础上扩展
3. 内联选项延后：作为渲染层优化，解析层可识别但不做深度处理，存为原始字符串供渲染器使用
4. 混合使用规则：不允许同一Directive内混用三种格式，按优先级YAML块 > `:key: value` > 内联选项，检测到混用则给出警告

### 3.3 挑战三：Roles语法的解析缺口

**问题描述：** 现有解析器完全没有Roles解析能力。Roles语法`{name}`content``与代码反引号相似但语义不同，解析时需要：1）区分普通代码反引号与Roles；2）支持反引号转义；3）处理Roles内的嵌套内容；4）建立自定义Role的注册机制。此外，Roles的行内特性意味着它运行在inline解析阶段，与当前Directive的block级解析时机不同，需要在markdown-it-py的inline ruler上注册新规则。

**影响范围：**
- 解析器初始化流程（第171-172行插件配置区域）
- 新增inline rule（约200-300行代码）
- 文本提取方法（第131-155行`_extract_inline_text`需扩展）
- 自定义Role注册机制（当前无此架构）
- 所有行内内容的解析路径（影响范围广）

**复杂度评级：** 高（P0）

**解决方向：**
1. 语法歧义解决：规定Roles使用单个反引号（`` ` ``），而代码使用≥2个反引号或明确标记语言，与markdown-it-py默认规则一致
2. 分阶段支持：第一阶段实现核心Roles（`{abbr}`/`{literal}`/`{strong}`/`{type}`/`{param-ref}`），不支持选项和嵌套；第二阶段添加引用类Roles（`{ref}`）和选项支持
3. 自定义Role注册表：设计简单的注册机制（字典映射），key为Role名称，value为处理函数，类似现有Directive处理模式
4. 降级策略：无法识别的Role不报错，按普通文本处理（保留`{name}``形式），保证文档不崩溃
5. 优先实现语义价值高的自定义Roles（`{type}`/`{param-ref}`），这些是Agent Spec场景的特有需求，MyST标准Roles反而优先级稍低

### 3.4 挑战四：自定义Directive/Role的扩展架构

**问题描述：** 当前解析器对Directive的处理是硬编码的——Admonition类型固定在`_ADMONITION_TYPES`集合（第115行），接口参数/响应解析是专门方法（第1227-1272行）。这种方式对预定义类型高效，但缺乏灵活的扩展机制。引入MyST生态后，若未来需要新增领域特定指令（如`{error-code}`/`{rate-limit}`/`{auth-requirement}`等），硬编码方式会导致维护成本线性上升。需要设计既保持当前性能、又具备可扩展性的架构。

**影响范围：**
- Directive识别逻辑（第377-421行）
- 接口提取逻辑（第1124-1225行）
- 解析器类整体架构（第158行起的MDIParser类）
- 插件/扩展的加载机制（当前无此机制）

**复杂度评级：** 高（P1）

**解决方向：**
1. 采用"注册表+处理器"模式：建立Directive注册表字典，每个条目包含名称、处理器函数、是否接受内容、选项schema
2. 分层处理：核心Admonitions和接口类指令作为内置指令注册，用户自定义指令通过装饰器或配置文件注册
3. 处理器签名标准化：`def handler(directive_name, argument, options, body, context) -> dict`，返回结构化数据
4. 保留快速路径：对高频内置指令（如`{param}`/`{response}`）保留优化处理，避免注册表查询开销
5. 复用现有逻辑：将第1124-1272行的接口提取逻辑重构为可注册的处理器，而非推翻重写

### 3.5 挑战五：存量文档的迁移成本与风险

**问题描述：** 虽然当前Directive使用率为0%，但一旦启用新语法，将面临：1）如何引导用户从表格形式迁移到Directive形式；2）新旧格式文档的长期共存问题；3）文档编写规范的制定与培训成本；4）历史文档是否需要批量迁移的决策。特别是SKILL.md文档表格密度达3.79个/文档，这些表格如果迁移为Directive，工作量较大；但如果不迁移，会出现两种风格长期并存的局面。

**影响范围：**
- 52份spec.md文档
- 14份SKILL.md文档
- 文档生成器（docgen-cmd相关逻辑）
- 模板与脚手架
- CI检查规则
- 团队编写习惯

**复杂度评级：** 中等（P0，业务复杂度）

**解决方向：**
1. 双格式兼容期：解析器同时支持表格形式和Directive形式，过渡期不少于3个月
2. 不强制迁移：历史文档保持原状，新文档鼓励使用Directive形式，提供自动迁移工具供可选使用
3. 优先迁移高价值场景：SKILL.md中的参数表、接口定义等高结构化内容优先提供迁移路径
4. 提供Codemod脚本：编写自动化迁移脚本，能识别标准格式的参数表/响应表并转换为`{param}`/`{response}`指令，人工审核后提交
5. 模板先行：更新所有新建文档模板，默认使用Directive形式，老文档自然淘汰
6. 渐进式规范：先规范自定义指令（`{interface}`/`{param}`/`{response}`）的使用，再逐步推广Admonitions，最后考虑Roles，避免一次性学习成本过高

### 3.6 挑战六：错误恢复与调试体验

**问题描述：** MyST语法比标准Markdown更复杂，尤其是围栏嵌套、选项格式、Roles转义等场景，初学者容易写错。当前解析器（基于markdown-it-py）对标准Markdown的错误恢复能力较好，但Directive/Roles作为自定义扩展，错误处理需要自行实现。常见错误包括：围栏不匹配、选项格式错误、指令名称拼写错误、Roles未闭合、嵌套层级过深等。若错误提示不友好，会显著提升调试成本。

**影响范围：**
- 所有Directive/Roles解析路径
- 用户编写体验
- CI校验反馈质量
- 文档验证工具

**复杂度评级：** 中等（P2）

**解决方向：**
1. 精确位置追踪：解析错误时记录行号、列号、错误上下文（前后各1行）
2. 友好错误信息：不仅说"解析失败"，还要给出"您是否想写{note}？"这类建议
3. 容错模式：非致命错误（如未知指令名）降级为普通代码块并发出警告，而非整体解析失败
4. 开发工具支持：提供lint命令检查Directive/Roles语法正确性，CI集成
5. 错误示例文档：在规范文档中列出常见错误示例与正确写法对比
