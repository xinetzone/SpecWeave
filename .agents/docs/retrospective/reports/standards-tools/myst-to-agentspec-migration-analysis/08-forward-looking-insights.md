## 8. 前瞻性洞察

### 洞察一：结构化标记是从"文档"到"可执行规范"的关键跨越

当前的Markdown Spec文档本质上是"给人读的文档"，机器解析依赖启发式规则和关键词匹配，脆弱且能力有限。引入Directives/Roles系统后，Spec文档向"可执行规范"演进——文档本身携带精确的语义标记，机器可以确定地解析、验证、转换、执行。

这一演进的意义不亚于从汇编到高级语言的跨越：
- 汇编时代：程序员写机器友好的指令，人需要迁就机器
- 高级语言时代：编译器理解人写的结构化代码，机器迁就人
- 文档领域正在发生类似的转变：从"人勉强写机器能猜的格式"到"人写语义清晰的结构化文档，机器精确理解"

Agent开发场景尤其需要这种能力，因为Agent需要基于Spec进行推理、生成代码、构造请求——这些都需要对Spec有精确的理解，而不是模糊的文本匹配。当前0%的Directive使用率意味着项目还处在"汇编时代"，有很大的提升空间。

### 洞察二：表格作为结构化载体是过渡阶段的产物，长期将让位于语义化指令

本次统计发现表格是当前核心结构化元素（spec.md 0.67个/文档，SKILL.md 3.79个/文档）。表格作为一种通用二维结构，在Markdown生态中被广泛用于表达各种结构化信息——参数表、响应表、错误码表、配置表等等。但表格存在固有局限：

1. **语义模糊**：同样的表格结构，可能是参数表，也可能是变更历史，机器无法仅从结构判断
2. **表达能力有限**：无法表达嵌套结构（如对象字段内的子字段）
3. **元数据表达困难**：很难为字段分组、标记版本、添加示例值等附加信息
4. **可扩展性差**：新增一个属性需要新增一列，表格变得过宽时难以阅读

MyST的Directive机制提供了更好的结构化方式：每个指令块携带明确的类型信息，选项机制可以灵活添加元数据，嵌套指令支持层次结构。可以预测，随着Directive的推广，表格将回归其"展示二维数据"的本职，而领域结构化信息将更多通过自定义Directive表达。

当然，这一转变将是渐进的。表格具有简单、易写、渲染器支持好的优点，在过渡期（甚至长期）两种形式会共存。解析器保持双格式兼容是务实的选择。

### 洞察三：自定义Domain扩展比标准特性更重要

MyST标准定义了丰富的Directives和Roles，但在Agent Spec场景中，真正高价值的是领域特定的自定义扩展——`{interface}`/`{param}`/`{response}`/`{error-code}`/`{type}`/`{param-ref}`等。这些扩展承载了Agent开发领域的语义，是连接"文档"与"代码生成/验证/推理"的桥梁。

这给我们的启示是：在引入MyST特性时，优先级判断的标准不应是"这个特性在MyST标准中是否重要"，而应是"这个特性能否帮助我们表达Agent Spec领域的语义"。因此：
- 自定义Domain指令（P0）> 标准Admonitions（P0）> 标准Roles（P1）> UI组件类指令（P2/P3）
- 为Agent Spec场景设计精炼的自定义指令集（5-10个核心指令，5-8个核心Roles），比支持所有标准特性更有价值
- 注册表架构的重要性高于某个具体特性的实现——它决定了未来扩展领域语义的成本

这也解释了为什么"不引入完整myst-parser"在当前阶段是合理的：myst-parser提供了完整的标准特性，但我们最需要的自定义Domain扩展仍然需要自己写。在轻量架构上做好自定义扩展机制，收益高于引入一个重引擎再在上面做定制。

### 洞察四：编写体验和工具链将成为 adoption 的关键瓶颈

技术上实现Directive/Role解析只是万里长征第一步。能否让团队愿意用、喜欢用，取决于：

1. **模板和示例**：新建文档时有开箱即用的模板，比阅读规范文档有效10倍
2. **编辑器支持**：语法高亮、自动补全、片段（snippet）、实时预览能极大降低使用门槛
3. **Lint和校验**：写错时立即得到友好提示，而不是等到解析失败才发现
4. **渐进式推广**：从收益最明显的场景（如参数定义）开始，而不是要求一次性全面切换
5. **迁移辅助工具**：提供自动化转换脚本，减少人工迁移的工作量

历史上许多优秀的技术方案因为工具链和体验不佳而未能推广。MyST本身也面临这个问题——工具链支持尚不及标准Markdown完善。SpecWeave作为特定领域的项目，反而有优势：使用场景有限，不需要支持所有编辑器和工具，可以集中精力为团队常用的环境（VS Code、CI、CLI）提供优质体验。

建议在实施功能开发的同时，同步投入工具链建设：
- VS Code扩展提供语法高亮和snippet
- CLI命令提供lint和自动修复
- 文档模板内置到脚手架
- 错误提示持续优化

### 洞察五：Roles的价值被低估，行内语义将带来惊喜

相较Directives的块级大块结构，Roles的行内特性显得不那么起眼。但本评估认为Roles的长期价值可能更高，原因在于：

1. **使用频率更高**：每个参数引用、每个类型标注、每个术语缩写都是Roles，使用频率远高于块级Directive
2. **网络效应**：当文档中大量使用`{param-ref}``user_id```这样的引用时，就可以构建文档内的引用网络，支持跳转、重构、影响分析、未引用检测等高级功能
3. **更非侵入**：Roles可以逐步添加到现有文本中，不需要像Directive那样重构大块内容，适合渐进式采用
4. **与IDE/AI工具结合紧密**：行内语义标记是AI辅助编写、智能提示、重构操作的基础

当前解析器完全没有Roles支持，这是最大的能力缺口。建议在实施中把Roles放在高优先级，甚至可以像"保守方案"那样，先于Directive增强实现Roles。
