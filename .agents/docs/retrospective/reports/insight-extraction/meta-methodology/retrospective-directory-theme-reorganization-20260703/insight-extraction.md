# 洞察萃取

## 一、核心洞察

### 洞察 A：大规模文件移动前必须先同步远程状态

**内容**：在执行涉及数百个文件的 `git mv` 操作前，未先 `git fetch` 检查远程更新，导致移动到一半时 `git pull` 冲突，需要 stash→pull→pop 额外处理。

**触发场景**：任何涉及大量文件重命名/移动的重构操作前。

**深层原因**：
- 本地工作区与远程的时间差在长时间会话中容易被忽略
- 文件移动操作会产生大量暂存变更，与远程更新的冲突概率远高于普通编辑
- stash 虽然能解决问题，但恢复后的冲突解决需要逐个文件判断"ours vs theirs"，增加出错风险

**可复用模式**：**Pre-Pull 检查点**——开始大规模重构前，先执行 `git fetch && git status`，确认工作区干净且与远程同步后再动手。

### 洞察 B：目录重组的核心不是"搬家"而是"路径一致性维护"

**内容**：本次任务中，实际移动文件的操作（git mv）仅占 ~10% 工作量，而路径更新（x-toml-ref、相对链接、跨文件引用、TOML source 字段）占了 ~60% 的工作量。

**触发场景**：任何目录结构重组任务。

**深层原因**：
- Markdown 生态的链接系统是一个隐式依赖网络——文件 A 引用文件 B 时，路径深度是写死的
- 目录深度变化一级，所有相对路径中的 `../` 计数都要相应增减
- x-toml-ref 等 frontmatter 字段将路径作为元数据存储，不受 IDE 重构工具影响
- 跨文件引用（模式文件 source 字段、README 链接）不在被移动文件本身，容易遗漏

**可复用模式**：**路径依赖图预扫描**——移动文件前，先用 `grep -r "旧路径前缀"` 全局搜索所有引用点，列出需要更新的文件清单，移动后逐一验证。

### 洞察 C：冲突解决的"内容 vs 位置"二分法

**内容**：当 git 冲突是"我们移动了文件位置，他们更新了文件内容"时，正确的策略不是选 ours 或选 theirs，而是：位置采用 ours（新目录结构），内容采用 theirs（远程最新内容）。

**触发场景**：文件移动/重命名与远程内容更新同时发生时的 git 冲突。

**深层原因**：
- `git mv` 本质上是"删除旧位置 + 创建新位置"，git 通过内容相似度识别 rename
- 远程 pull 的内容更新是基于旧位置的，git 在新位置上产生冲突标记
- 简单选 ours 会丢失远程更新，选 theirs 会把文件放回旧位置
- 正确做法是：`git checkout --theirs <新路径>` 拿到远程内容放到新位置，然后 git add

**可复用模式**：**Rename-Update 冲突解决三步骤**：
1. "由我们添加"的文件（远程没有的新文件）：直接 `git add`
2. "双方修改"的文件：`git checkout --theirs <新路径>` 取远程内容 + `git add`
3. 手动合并的文件：需要逐个检查冲突标记，保留新路径 + 最新内容

### 洞察 D：主题分类是文档可维护性的非线性放大器

**内容**：30 个文件平铺时，查找特定主题报告需要通读整个列表；分为 4 个子目录后，每个子目录 3-12 个文件，认知负担大幅降低，且新增报告时有明确归属指引。

**触发场景**：文档/报告目录增长到 15+ 个条目时。

**深层原因**：
- 人类短期记忆容量约为 4±1 个组块（Miller 定律），4 个主题子目录恰好落在最佳认知负荷区间
- 主题分类提供了"先定位主题域，再找具体报告"的二级导航模式
- 分类本身也形成了知识领域的显式映射——看到目录结构就能理解项目关注的领域

**可复用模式**：**4±1 主题分组阈值**——当同目录下条目超过 15 个时，按主题分为 3-6 个子目录，每个子目录 3-12 个条目。子目录数和每个子目录条目数都控制在认知舒适区间内。

### 洞察 E：原子化报告的子目录分组应保持"深度增加一级"原则

**内容**：原子化报告目录本身已有 README.md 等 4 个固定文件，外层增加一级主题子目录后，路径深度增加一级，但报告内部结构不变。这种"深度+1"策略最小化了路径计算的复杂度。

**触发场景**：已有固定内部结构的原子化单元分组时。

**深层原因**：
- 如果主题子目录下再嵌套子子目录，路径深度增加两级，x-toml-ref 和相对链接的 ../ 计数更易出错
- 单层分组（直接在父目录下按主题创建子目录，原子化目录直接在主题子目录下）是最简结构
- 每个主题子目录内的报告是平级关系，不需要进一步细分

**可复用模式**：**单层分组原则**——对原子化单元集合分组时，仅增加一层目录深度，不创建多级嵌套。

## 二、反模式与教训

### 反模式 A：边移动边 pull

**教训**：在执行了大量 `git mv` 但尚未 commit 时，不应执行 `git pull`。正确做法是先 commit 当前移动操作（哪怕是 WIP commit），再 pull，然后 rebase 或 merge。

### 反模式 B：子代理任务未包含远程新增内容

**教训**：在路径更新子代理任务的映射表中，遗漏了远程新增的 `retrospective-skills-article-learning-20260629`，导致该报告的 x-toml-ref 路径和相对链接需要额外修复。下次应在映射表生成后，先列出目录实际内容再对照。

### 反模式 C：stash drop 过早

**教训**：`git stash pop` 成功后，stash 条目自动删除。但如果 pop 后发现问题需要回退，没有 stash 可以恢复。对于大规模操作，应先用 `git stash apply` 验证无误后再 `git stash drop`。
