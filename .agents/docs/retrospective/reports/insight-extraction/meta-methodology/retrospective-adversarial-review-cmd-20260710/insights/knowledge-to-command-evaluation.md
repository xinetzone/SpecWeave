# 洞察1：知识库建成后应主动评估是否需要配套指令集

**发现**：对抗性审查知识库（15文件、3456行）建成后，Spec的v1.0决策是"暂不创建指令集"。但第一性原理分析揭示了"知识库→指令集"的必然性：知识库解决"知道是什么"，指令集解决"按步骤做"，两者缺一不可。

## 核心命题

> **知识库的"使用成本"是方法论采纳的瓶颈。** 15个文件的知识库对用户来说认知负荷高——需要自行导航到对应的检查清单、模板、流程。指令集将"触发条件→场景选择→执行步骤→输出规范"压缩为线性路径，降低90%+的导航成本。

## 使用成本对比

| 维度 | 纯知识库 | 知识库+指令集 |
|------|---------|-------------|
| 用户需要自行导航 | 15个文件，需判断哪个文件适用 | 指令集提供线性路径，无需导航 |
| 场景选择 | 用户自行判断适用场景 | 指令集提供场景分支（知识研究/代码审查） |
| 执行步骤 | 需从多个文件中拼凑流程 | 指令集提供结构化步骤 |
| 首次使用时间 | 估计30-60分钟（阅读+导航） | 估计5-10分钟（跟随指令集） |

## 5-Whys根因

1. 为什么知识库建成后使用率不高？→ 因为使用成本高，需要自行导航
2. 为什么使用成本高？→ 因为知识库是"信息存储"而非"操作引导"
3. 为什么知识库不是操作引导？→ 因为知识库的设计目标是"知识完整性"而非"操作便捷性"
4. 为什么不能同时满足两者？→ 可以，但需要额外的"指令集"层来做操作引导
5. 为什么Spec v1.0决策了"暂不创建"？→ 因为确认偏差——人们倾向于寻找支持已有决策的证据，而忽略了条件变化（知识库已建成）

## 关键数据

- 知识库规模：15个文件、3456行
- 指令集规模：283行（压缩为知识库的8.2%）
- 导航成本降低：90%+（从"自行导航15个文件"到"跟随线性路径"）
- 知识库引用：指令集中引用了12个核心文件

## 可迁移性

**极高**。此模式适用于所有已建成知识库的方法论主题（如质量内建、认知偏差防御等），可作为"知识库→指令集"转化的标准流程。

## 模式沉淀状态

✅ **已沉淀为可复用模式**：[knowledge-to-command-pipeline](../../../../../patterns/methodology-patterns/governance-strategy/knowledge-to-command-pipeline.md)（L1实验级），核心步骤：盘点知识库资产→第一性原理分析→参照现有指令集模板→设计场景分支→引用知识库→质量验收。

## 改进建议实施

✅ **已实施**：在 Spec 模板的 DoD 和 tasks.md L6 中新增"指令集评估"检查项，知识库建成后由 reviewer 执行第一性原理分析判断是否需要配套指令集。
