# 洞察3：指令集需要有"自我验证"的设计

**发现**：对抗性审查指令集的独特之处在于包含了"自举验证（元审查）"——即审查指令集自身是否正确执行。这一设计来自方法论的自洽性要求：如果对抗性审查方法声称能发现偏差和问题，那么它自身也必须接受同样的审查标准。

## 核心命题

> **方法论类指令集（如对抗性审查、第一性原理）应该包含"元审查"机制——即用本方法论审查本方法论的执行质量。** 这不仅是自洽性要求，也是防止"表演式执行"（只走流程不真找问题）的有效手段。

## 元审查的四个维度

对抗性审查指令集的元审查设计包含4个维度：

| 维度 | 检查内容 | 防护目标 |
|------|---------|---------|
| 完整性检查 | 是否覆盖所有应审查维度 | 防止遗漏关键审查领域 |
| 偏差自检 | 审查者自身是否存在认知偏差 | 防止"审查者免于审查"的盲区 |
| 方法论合规 | 是否严格按照流程执行 | 防止"表演式执行"（走流程不走心） |
| 反模式检测 | 是否存在表演式审查、角色混同等 | 防止方法论自身的反模式 |

## 与知识库反模式的对齐

元审查的4个维度与知识库中定义的7种反模式形成对应关系：

| 反模式 | 元审查维度 | 检测方式 |
|--------|-----------|---------|
| 表演式审查 | 方法论合规 | 检查审查输出是否包含实质性发现 |
| 角色混同 | 反模式检测 | 检查RACI矩阵是否被遵守 |
| 确认偏差 | 偏差自检 | 检查审查结论是否过于一致（缺乏对立观点） |
| 锚定效应 | 偏差自检 | 检查审查起点是否受初始信息过度影响 |
| 信息遗漏 | 完整性检查 | 检查是否覆盖所有应审查维度 |
| 群体思维 | 反模式检测 | 检查多Agent审查是否缺乏异议 |
| 过度自信 | 偏差自检 | 检查审查者是否标注了不确定性 |

## 自洽性要求的哲学基础

方法论类指令集的"元审查"设计体现了一个基本哲学立场：**一套方法论如果声称能发现偏差和问题，而自身不接受同样的审查标准，那么这套方法论是不可信的。** 这类似于科学方法中的"可证伪性"要求——好的理论不仅要能解释现象，还要能接受检验。

## 关键数据

- 元审查维度：4个
- 覆盖反模式：7种（与知识库中7种反模式一一对应）
- 已应用：对抗性审查指令集（adversarial-review.md）+ 第一性原理指令集（first-principles.md）

## 可迁移性

**高**。适用于所有方法论类指令集：复盘（retrospective-cmd）、洞察（insight-cmd）、第一性原理（first-principles）等。判断标准：如果指令集定义了"如何审查/分析/评估某事物"，则该指令集应包含元审查机制。
