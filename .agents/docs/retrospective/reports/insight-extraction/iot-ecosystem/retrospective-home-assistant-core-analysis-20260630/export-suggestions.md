# Home Assistant Core 导出建议（行动项与风险预警）

---

## 一、改进建议（面向“如何复用 HA 能力”）

| 问题 | 改进措施 | 优先级 | 预期效果 | 状态 |
|------|---------|--------|---------|------|
| 直接把 HA Core 当库引入成本高 | 将集成目标从“嵌入式 import”调整为“外挂式集成”（HTTP/WebSocket API、Webhook、Sidecar） | 高 | 降低 Python 版本/依赖冲突与升级成本 | 待规划 |
| 装配编排机制难复用 | 抽象“装配层”而不是复用“具体集成”——复用 stage 启动与 Future 去重模式到自研系统 | 高 | 复用其可扩展性关键机制 | 待规划 |
| 启动性能与长尾风险不可控 | 引入分阶段启动 + 超时推进 + 启动阻塞任务告警（参考 HA 机制） | 中 | 启动稳定性提升，问题可诊断 | 待规划 |
| 依赖闭包易死锁 | 若系统存在 after-deps 语义，必须保证“等待的一定已被调度”，并提供 deadlock 诊断输出 | 中 | 减少线上冷启动死锁 | 待规划 |

---

## 二、行动计划（Action Items）

| 优先级 | 改进项 | 具体措施 | 建议时间 | 状态 |
|--------|--------|---------|---------|------|
| 高 | 选型决策：嵌入 vs 外挂 | 明确集成目标：需要的是“设备接入生态”还是“运行时内核能力”；优先走外挂式集成 | 2026-07-01 | 待规划 |
| 高 | 模式落地：分阶段启动 | 在自研系统中实现 stage 启动（基础能力优先、长尾并发），并为每个 stage 定义超时策略 | 2026-07-03 | 待规划 |
| 高 | 模式落地：装配并发去重 | 对 domain/component/setup 做 Future 去重与异常一致传播，减少并发竞态 | 2026-07-03 | 待规划 |
| 中 | 可观测性：启动阻塞告警 | 启动阶段聚合 pending tasks，输出可定位信息与“继续推进”策略 | 2026-07-05 | 待规划 |
| 低 | 研究：HA config entries 机制 | 若需要插件化配置，评估其 config entries + storage 的可迁移点 | 2026-07-10 | 待规划 |

## 三、风险预警（Risks）

| 风险 | 描述 | 影响面 | 缓解策略 |
|------|------|--------|---------|
| Python 版本硬门槛 | HA Core 要求 Python 3.14+，与多数工程当前环境可能不兼容 | 构建/部署 | 不做嵌入式依赖；改用外挂式集成 |
| 依赖 pin 与升级成本 | 依赖严格锁定，升级需要跟随上游节奏 | 依赖治理 | 以进程边界隔离依赖；对外用协议/API |
| 启动链路复杂 | stage 启动 + 并发装配 + 依赖闭包，调试成本高 | 研发效率 | 落地时引入强可观测性（阶段日志、pending 任务快照） |
