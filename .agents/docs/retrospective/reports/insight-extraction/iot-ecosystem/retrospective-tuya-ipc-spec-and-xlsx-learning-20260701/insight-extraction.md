# 洞察萃取

## 一、核心洞察

### Insight 1：知识交付型任务的稳定做法不是“直接写”，而是“先建约束骨架”

任务 A 的成功关键不在于文档写得多快，而在于先通过 `/spec` 把交付范围、章节骨架、验收标准和依赖关系固定下来。这样后续写出的不是一篇泛泛说明，而是一条可以按步骤执行的最小闭环路径。

可复用结论：

- 当用户要的是“可执行知识资产”时，优先用规格把边界和验收标准前置
- 文档型任务同样适合采用“Spec -> Tasks -> Checklist -> 实施”的闭环

### Insight 2：最小闭环方法可以显著降低复杂系统说明文档的失控风险

Tuya IPC 的真实链路很长，容易越写越散。但一旦用“配网、音视频、绑定、控制、事件”五个节点定义闭环出口，文档就自然形成了强约束：所有前置准备、依赖关系和验收标准都服务于这五个节点，而不是无限扩张。

可复用结论：

- 写复杂系统操作指南时，先定义“闭环出口”，再倒推步骤
- 闭环节点一旦明确，文档会自动具备“删繁就简”的能力

### Insight 3：面对二进制源资料时，最重要的不是“强行打开”，而是尽快切换到格式感知解析

任务 B 最大的分水岭，是不再试图把 `.xlsx` 当成文本，而是承认其源格式特性，直接切换到 `openpyxl`。这一步避免了无效调试，也保证了后续结论都能追溯到真实单元格数据。

可复用结论：

- 二进制资料优先匹配原生解析库，不要把格式问题误判为工具问题
- “先识别源格式，再决定工具链”应成为固定前置动作

### Insight 4：大型测试报告的高效学习路径不是“逐条读完”，而是“总表先判定，再专题聚类”

本轮 Excel 学习没有从 489 条用例逐条平铺总结，而是先抓总表的发布门槛、Fail 分布和严重问题，再下钻到功能、音频、TF 卡、WiFi 穿墙等高风险专题页。这使得报告很快聚焦到真正影响发布判断的主线。

可复用结论：

- 面对大型测试报告，先识别“总表 -> 风险主线 -> 证据页”三层结构
- 如果目标是形成结论，不必平权处理所有工作表

### Insight 5：真正可复用的交付不是“答复完成”，而是“留下能被下一轮直接接手的文件资产”

本轮两项任务最终都沉淀成文件：

- 任务 A 变成了知识库文档与规格资产
- 任务 B 变成了可直接复用的 Markdown 学习报告

这意味着下一轮工作可以直接从文件继续，而不是重放一遍对话上下文。

可复用结论：

- 高价值任务应尽量以文件资产结尾，而不是只停留在聊天回复
- Markdown 是跨“知识沉淀”和“临时导出”两类场景的低摩擦统一介质

## 二、模式萃取

### Pattern 1：规格前置知识交付模式

- 触发条件：用户要求输出“完整路径”“可执行文档”“独立验证指南”
- 解法：先创建 `spec.md`，明确边界、交付物、章节结构、验收口径，再进入实施
- 价值：避免文档实施阶段范围漂移

### Pattern 2：最小闭环倒推模式

- 触发条件：主题复杂、步骤很多、容易写成散点知识
- 解法：先定义闭环出口与必经节点，再倒推出前置准备、步骤依赖和验收标准
- 价值：让复杂主题收敛成主干路径

### Pattern 3：二进制源解析回退模式

- 触发条件：原始资料为 `.xlsx`、`.pdf`、压缩包等非纯文本格式
- 解法：先验证文本读取是否可行；若不可行，立即切换对应格式解析库
- 价值：减少错误工具链带来的时间浪费

### Pattern 4：总表先行风险判定模式

- 触发条件：大型测试报告、用例数量多、目标是形成质量结论
- 解法：先抽取总表指标、发布门槛和高风险项，再逐页验证证据
- 价值：快速形成管理层可消费的结论

### Pattern 5：Markdown 统一交付模式

- 触发条件：同轮任务既包含知识沉淀，也包含临时分析导出
- 解法：统一以 Markdown 交付，并把最终文件纳入知识库或归档目录
- 价值：降低格式切换成本，提升后续复用性

## 三、根因诊断

### 为什么任务 A 能形成高质量结果

根因不只是“执行认真”，而是有三层支撑：

1. 用户目标清晰：要 `/spec`，且强调完整闭环、依赖、验收、排查
2. 项目已有规范：存在 `.trae/specs/`、知识库目录和索引规则
3. 实施路径正确：先规格化，再知识库落地，再索引更新和自检

### 为什么任务 B 能在短时间内形成有效结论

根因同样有三层：

1. 没有沉没在 Excel 格式问题里，而是快速切换解析方式
2. 不是逐行读表，而是先抓总表和状态统计
3. 导出目标明确，始终围绕“形成可读结论报告”组织内容

## 四、行动建议

| 优先级 | 建议 | 预期收益 | 验收标准 |
|---|---|---|---|
| 高 | 为“文档型任务”固化一套 `/spec -> 文档 -> 索引 -> 自检` 模板 | 减少知识交付任务的结构漂移 | 后续类似任务可直接套用统一模板 |
| 高 | 为 `.xlsx` 学习类任务沉淀标准解析脚本或流程片段 | 降低重复手工统计成本 | 下次同类任务可复用同一解析骨架 |
| 中 | 为测试报告类任务补一份“总表优先”分析模板 | 加快结论提炼速度 | 新报告能在更短时间内输出发布判断 |
| 中 | 将“格式识别 -> 解析回退”写成通用工作准则 | 避免在错误工具链上反复消耗 | 面对二进制资料时有统一决策路径 |
| 低 | 将本次复盘再抽象为模式条目，沉淀到 `docs/retrospective/patterns/` | 提升方法复用率 | 至少产出 2-3 条稳定模式 |
