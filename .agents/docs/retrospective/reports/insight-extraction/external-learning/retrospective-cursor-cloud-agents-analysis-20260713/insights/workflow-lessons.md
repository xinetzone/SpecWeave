## 经验教训1：格式参考样本选择偏差——为什么"看了同类文件"还是放错位置

### 现象描述

创建first-principles-insight.md时，遵循了"格式一致性优先原则"——先读取同目录现有文件确认格式——但仍然将文件放在了根目录而非insights/子目录，用户指出"部分文件存放位置好像不对"后才修正。

### 5-Whys根因分析

```
Why 1：为什么文件放错位置？
→ 因为我读取的参考样本是gpt56、codex等目录，它们的task文件直接堆在根目录。

Why 2：为什么读取那些目录而不是architecture-priority这样的规范目录？
→ 因为那些目录更"方便"访问（LS时先看到），且我的直觉假设"同类目录结构应该一致"。

Why 3：为什么直觉假设会出错？
→ 因为external-learning目录下存在两类目录：
  (a) 简单归档目录（如gpt56/codex/deep-code）——文件直接堆在根目录，是归档的过渡状态
  (b) 原子化规范目录（如architecture-priority）——有execution/insights/exports子目录，是整理完成状态
  我选择了(a)而非(b)作为参考。

Why 4：为什么选择了不规范的样本？
→ "格式一致性优先原则"只说"读取同目录1-2个现有文件"，没有说"选择什么样的样本"——规则缺少样本选择指导。

Why 5：为什么规则缺少这部分？
→ 因为之前假设"同类目录结构一致"，但实际上归档目录存在整理完成度差异。
```

### 核心洞察

**格式一致性检查不仅要"看样本"，更要"选对样本"**。当存在多个同类参考时，参考样本选择本身就是一个决策点，不能偷懒选最方便的。

### 选择正确参考样本的三规则

| 优先级 | 规则 | 判断方法 |
|-------|------|---------|
| 1 | **结构完整性优先** | 选择子目录组织最完整的（有execution/insights/等子目录）而非文件堆在根目录的 |
| 2 | **版本新鲜度优先** | 选择最近更新/最新版本的目录，目录结构规范也在演进 |
| 3 | **避免过渡状态** | 如果目录中的文件命名/组织看起来"临时"或"未整理"，不要将其作为格式参考 |

### 反模式信号

当你出现以下想法时，你可能选了错误的参考样本：
- "这几个目录都长这样，应该没问题"——确认一下是否有更规范的目录存在
- "这个目录文件少，看起来简单"——简单可能意味着未完成整理
- "之前那个任务也是这么放的"——检查那个任务是否是最新规范

## 经验教训2：已知Git陷阱的认知-践行鸿沟——读过文档不代表不会踩

### 现象描述

原子提交时，我已经读取了atomic-commit-cmd的SKILL.md和L2文档，文档§9 Gotchas中明确记录：
> "git add新目录不会自动暂存旧文件删除记录，必须显式git add删除的旧文件"

但第一次提交（ec3dde7e）后，根目录下5个旧task文件的删除记录未被暂存，需要第二次提交（7cb4c69c）补删。

### 核心机制

这不是"粗心"，而是**认知资源竞争**导致的践行鸿沟：

1. 原子提交是多步骤流程（三查暂存→预提交验证→构建提交信息→执行提交→验证），每一步都消耗认知资源
2. 构建提交信息（写中文commit message正文）占用了大量System 2注意力
3. "显式add删除文件"这个检查点被System 1自动跳过——因为直觉认为"git add新目录"已经处理了所有相关文件
4. 关键矛盾：你"知道"这个陷阱的时刻是读取Skill文档时（System 2活跃），但你需要执行检查的时刻是提交操作时（System 2被占用）

### 洞察：陷阱文档的"知道"与执行时的"做到"之间存在上下文切换损耗

| 阶段 | 认知状态 | System 2占用 | 陷阱防御能力 |
|------|---------|-------------|------------|
| 读取Skill文档 | "我知道这个陷阱，下次注意" | 低（学习模式） | 高（刚学完，记忆清晰） |
| 实际执行提交 | 专注于写commit message、验证文件列表 | 高（执行模式） | **低**（注意力被占用，陷阱记忆未被激活） |

### 对策：将"知道"转化为"强制检查点"

读过陷阱文档≠不会踩陷阱。唯一可靠的防御是：**在三查暂存法中强制执行"查删除"步骤，用工具输出而非记忆来确认**。

原子提交三查暂存法的强制执行流程：
```
git status --short
  ↓
检查每一行的状态标记：
  - ?? 未追踪文件：确认都是需要的新文件
  - M  修改文件：  确认都属于本次提交范围
  - A  新增文件：  确认路径正确（包括新子目录内的文件）
  - D  删除文件：  ✅ 必须显式检查——如果有D状态，必须git add这些路径
```

### 关联模式

- 本案例是 [cognitive-practice-gap-recursive-defense.md](../../../../../patterns/methodology-patterns/governance-strategy/cognitive-practice-gap-recursive-defense.md) 的第8个验证案例，已更新该模式validation_count至8
- 属于"递归践行定律"范畴：即使你知道防错机制，在执行多步骤任务时仍可能跳过检查点

## 经验教训3：多步任务的原子提交拆分原则——新增内容与结构调整分开提交

### 现象描述

本次提交最终拆分为2个原子提交：
1. `ec3dde7e docs(...)`：新增洞察报告+创建子目录+添加索引README+移动文件到子目录+更新根README
2. `7cb4c69c refactor(...)`：删除根目录下的旧位置文件

虽然拆分是事后补救（因为第一次提交遗漏了删除），但事后分析这其实是合理的原子拆分。

### 核心洞察

文件移动/目录重构类操作天然包含两个独立变更：
- **内容层面**：新位置的文件（作为新增出现）
- **清理层面**：旧位置的文件（作为删除出现）

这两个变更虽然"逻辑上是一回事"（移动文件），但Git不会自动识别跨目录重命名（特别是当新位置有多个新文件和修改的README时）。

### 原子提交拆分策略

当执行"创建新文件+目录重构"类任务时，按以下原则拆分提交：

| 提交顺序 | 类型 | 内容 | 原因 |
|---------|------|------|------|
| 提交1 | feat/docs | 新增文件+新子目录+新索引+更新README | 新增内容是正向交付，独立可回滚 |
| 提交2 | refactor | 显式删除旧位置文件 | 清理操作单独提交，便于追溯文件迁移历史 |

### 为什么不合并为一个提交？

1. **可审查性**：审查者可以先看新增内容，再看删除清理，变更逻辑清晰
2. **可回滚性**：如果新目录结构有问题，可以只回滚refactor提交恢复旧结构，不丢失新增内容
3. **Git历史可读性**：git log显示"新增内容"→"清理旧文件"的清晰脉络，而非一个混合提交
