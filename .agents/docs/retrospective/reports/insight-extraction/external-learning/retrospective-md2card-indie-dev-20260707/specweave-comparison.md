# md2card 与 SpecWeave 体系对照分析

本章节从 4 个维度对照分析 md2card 独立开发案例中的方法论与 SpecWeave 现有规范体系，提炼可借鉴点与需警惕点。

---

## 维度一：AI辅助开发工作流对照

**md2card 做法**：采用 Codex Image2Code 三步工作流——Image2生成设计稿→根据设计稿生成HTML→基于HTML生成交互组件，核心特征是**可视化设计先行、渐进式生成、对话式纠偏**。

**SpecWeave 现状**：采用 Skill 体系与规范驱动开发——通过 `.agents/` 规范体系显式传递知识，阶段守卫、CI 检查、代码审查等多层机制保障质量，Skill 复用 + `.agents/scripts/lib/` 共享工具库实现能力复用。

**可借鉴点（3个）**：
1. **原型快速迭代的"设计稿先行"理念**：在前端类 Skill 开发中引入"视觉基准先行"原则——先生成设计稿/Mockup 作为参照，再基于参照生成代码，而非直接生成最终代码。
2. **对话式纠偏与显式规范的互补**：Skill 内部可以封装"渐进式生成"流程，而 Skill 的触发条件、输入输出格式由显式规范定义——两者不是对立关系，而是互补关系。
3. **每步有明确输入输出的渐进式生成优于一次性生成**：这与 SpecWeave 的三阶段递进原则哲学高度一致，可以将这一原则进一步应用到 Skill 的执行流程设计中。

**需警惕点**：完全依赖对话式纠偏无法规模化（不同会话/智能体产出质量不一致、上下文压缩后经验丢失）；"设计稿先行"不等于"无规范先行"（跳过规范读取可能导致非线性返工）；AI 生成代码必须有人工审查机制。

---

## 维度二：快速原型与阶段守卫平衡对照

**md2card 做法**：采用周末 MVP 验证法——核心功能裁剪只做一件事，速度优先于完美，一个周末完成开发发布到 V2EX，但核心体验不能崩（提前解决了 API 存图和服务器扩容问题）。

**SpecWeave 现状**：建立了严格的阶段守卫体系，同时通过 `.temp/` 目录机制提供快速原型空间——`.temp/` 暂存开发允许频繁修改、不要求代码审查，全量迁移到 `apps/` 需满足测试通过、代码审查通过、无P0/P1缺陷、文档完善4项条件，L0探针豁免规则为探索性任务提供快速通道。

**可借鉴点（3个）**：
1. **强化 `.temp/` 的"快速验证"定位**：`.temp/` 阶段应该聚焦单一核心痛点验证，允许"不完美但能用"的代码快速上线获取反馈，但核心体验必须提前预案。
2. **提炼"快速验证但保留质量框架"的平衡点**：`.temp/` 阶段可以跳过代码审查、测试覆盖率等"增量质量"要求，但不能跳过"核心功能可用""无致命缺陷"等"底线质量"要求。
3. **功能裁剪的艺术**：`.temp/` 阶段的新 Skill/新工具只实现最小核心功能，不提前设计"未来可能需要"的抽象层，验证通过后再考虑扩展和规范化。

**需警惕点**：过度规范化会扼杀探索速度（`.temp/` 阶段门禁要比 `apps/` 宽松）；"周末MVP"不适合所有场景（核心规则体系修改不适用）；快速原型不等于无架构设计（核心能力仍要有模块化和复用意识）。

## 维度三：用户验证与规范驱动对照

**md2card 做法**：采用真实用户反馈优先——快速做出MVP发布到V2EX，通过用户实际痛点迭代产品，用户反馈是方向判断的唯一标准，避免闭门造车。

**SpecWeave 现状**：采用规范前置方法论——启动协议强制读取规范，前置文档强制读取协议，查阅知识库规则，歧义主动澄清，通过复盘沉淀可复用模式形成组织记忆。

**可借鉴点（3个）**：
1. **新工具/新功能可采用 MVP→用户反馈→再规范化的路径**：新 Skill 开发先在 `.temp/` 快速实现原型→在实际任务中试用→收集使用反馈→再规范化（编写SKILL.md、添加验证脚本、纳入上下文路由）；新规范制定先通过实际案例验证有效性→再沉淀为正式规范。
2. **建立"内部用户反馈"机制**：Skill 使用效果追踪、规范违反案例收集（SG-LOG已在做）、智能体间的"用户反馈"渠道、定期复盘使用实际数据优化规范。
3. **区分"探索期规范"和"稳定期规范"**：探索期（新工具/新方向）轻规范、重验证、允许快速试错；稳定期（核心流程/高频使用）重规范、强门禁、保证一致性——L0-L3流程分级模板已有这个思路，可以进一步强化。

**需警惕点**："用户反馈优先"不能成为不读规范的借口（规范本身就是大量"用户反馈"的沉淀）；内部工具的"用户反馈"有其特殊性（智能体不会像真实用户那样主动抱怨，需要通过日志和客观指标评估）；规范前置与用户验证不是非此即彼（实践→复盘→规范→再实践的闭环）。

## 维度四：内容优先与文档媒介选择对照

**md2card 做法**：采用内容优先于模板的产品设计哲学——与Canva"先选模板再填内容"相反，用户已有Markdown内容→一键生成卡片，复杂排版/适配/拆分全部自动化，内容是起点而非终点，形式服务于内容。

**SpecWeave 现状**：采用 spec/tasks/checklist 结构化文档体系——有明确模板（任务模板、交接模板、Mermaid模板、复盘模板等），部分自动化（docgen生成导航/看板、链接检查自动修复），明确区分AI面向文档（`.agents/`）和人类面向文档（`docs/`）。

**可借鉴点（3个）**：
1. **工具输出优先保证内容质量，格式/排版自动化处理**：智能体产出文档时先保证内容（分析、决策、代码）质量，格式合规性尽量自动化——进一步自动生成/校验frontmatter、自动检查文档结构、自动格式化Mermaid图表，让规范检查工具承担更多"形式自动化"工作。
2. **提炼"结构化但不束缚创造力"的文档哲学**：模板提供"骨架"但不强制"血肉"——必填项保证最低结构质量，可选项给予灵活空间；进一步明确哪些部分是"必须结构化"的（frontmatter、章节标题、元数据），哪些是"自由发挥"的（分析内容、经验描述）。
3. **"内容即营销"理念在文档体系中的应用**：好的文档本身就是最好的推广——一份结构清晰、内容扎实的spec或复盘报告比任何"如何使用"指南都更有说服力；`.trae/specs/`下的实际案例就是最好的"使用示范"；可以在文档索引中加入"优秀案例"或"推荐阅读"。

**需警惕点**：过度结构化确实可能束缚创造力（模板过于复杂导致花更多精力"符合格式"而非"解决问题"）；"一键自动化"有其适用边界（深度思考和创造性分析无法自动化，工具应该自动化"机械性工作"释放人的精力到"创造性工作"）；"内容优先"不等于"可以忽略形式"（格式规范是为了提升可读性、可维护性、可检索性）。

## 综合启示：12个可借鉴点汇总

| 维度 | 可借鉴点 |
|---|---|
| AI辅助开发 | 1. 前端类Skill引入"视觉基准先行"原则 |
| AI辅助开发 | 2. Skill封装渐进式生成流程，显式规范定义接口 |
| AI辅助开发 | 3. 渐进式生成原则应用到Skill执行流程设计 |
| 快速原型 | 4. 强化.temp/的"快速验证"定位，聚焦单一痛点 |
| 快速原型 | 5. 明确"底线质量"和"增量质量"的区分 |
| 快速原型 | 6. .temp/阶段只做最小核心功能，不提前抽象 |
| 用户验证 | 7. 新Skill/新规范采用"原型→试用→规范化"路径 |
| 用户验证 | 8. 建立基于使用日志的内部反馈机制 |
| 用户验证 | 9. 明确探索期/稳定期的不同规范强度 |
| 文档媒介 | 10. 扩大格式自动化处理范围（frontmatter/结构/图表） |
| 文档媒介 | 11. 模板定义必填结构，给予内容表达自由度 |
| 文档媒介 | 12. 通过高质量实际案例引导规范采纳 |

**对 SpecWeave 最大的启示不是"放松规范"，而是"让规范更聪明"**——在需要探索的地方提供足够的自由度（`.temp/`、L0豁免、渐进式披露），在需要稳定的地方坚守质量底线（阶段守卫、CI检查、测试要求），并通过自动化和数据闭环让规范本身持续进化。好的方法论应该像 md2card 一样——你专注于核心内容（解决问题），它自动处理形式（流程合规）。
