# 洞察萃取

## 洞察1：第一性原理功能分析法——产品功能定义的标准工作流

**洞察陈述**："悬置→拆解→质疑→重构"四步法可复用于任何产品功能的重新定义，其核心威力在于打破"类比思维"陷阱（竞品做什么我们做什么），穿透到用户需求的认知/心理/物理基础层。

**触发场景**：当需要对一个已有功能进行根本性重构、或从零定义一个功能时（如睡眠模式、阅读模式、健身模式、驾驶模式等"模式类"功能），当团队陷入"加功能"迭代而无法突破时。

**为什么重要**：大多数产品功能设计依赖竞品对标和用户反馈，这两种方法都无法发现"现有方案系统性缺失什么"——例如学习模式产品普遍缺失C2认知过渡引导（0-5%实现度），不是因为用户不需要，而是因为竞品也没做、用户也无法描述自己不知道的需求。第一性原理分析能发现这些"未知的未知"。

**如何应用**：
1. 悬置阶段：列出所有已知实现，明确标记"这些是类比对象，不是答案"
2. 拆解阶段：从学科基础理论出发（认知心理学/神经科学/行为经济学等），识别本质机制
3. 质疑阶段：对每个"显而易见"的设计假设进行反事实测试（"如果不做X，会怎样？"）
4. 重构阶段：从必要条件出发反推最小完整要素集，再映射到技术实现

**模式映射**：建议沉淀为新模式"第一性原理功能分析法"（methodology-patterns/research-knowledge/），可直接复用SOP。

## 洞察2："先完整写作→后原子化拆分"优于"直接写原子文件"

**洞察陈述**：对于深度分析/思考类写作任务，先在单文件中完成完整内容（保持思维连续性），再通过原子化命令拆分为单一职责文件，比一开始就写原子文件效率更高、质量更好。

**触发场景**：任何需要深度连贯思考的写作任务——分析报告、理论框架构建、复杂系统设计文档、长篇技术方案等。

**为什么重要**：写作即思考。在深度思考阶段，频繁切换文件、考虑导航结构、维护交叉链接会严重打断思维流。原子化拆分本质是"编辑/排版"工作，不属于"思考/创作"工作，应在思考完成后作为独立步骤执行。SpecWeave的atomization-cmd+finalize-atomization.py工具链已经将拆分成本降到极低（<5分钟），没有理由在思考阶段承受原子化的认知负担。

**如何应用**：
1. 思考/创作阶段：单文件写作，不管文件大小，专注内容质量
2. 完成后：使用atomization-cmd拆分，让子智能体处理文件分割、frontmatter添加
3. 收尾阶段：finalize-atomization.py自动修链接、更新导航、刷新看板

**模式映射**：这验证了现有原子化工作流的设计合理性，建议在原子化相关模式中明确"写作与拆分分离"原则。

## 洞察3：PRD摘要应作为分析类任务的标准产出物

**洞察陈述**：完整分析报告（4-5万字）和精炼PRD摘要（280行）服务于不同场景——报告用于深度理解和溯源，摘要用于决策评审和快速对齐。两者都是必要产出，不应互相替代。

**触发场景**：所有产出长篇分析报告（>2000行）的任务，都应该在完成后提炼一份10%篇幅以内的摘要/PRD。

**为什么重要**：4-5万字的分析报告只有少数人会完整阅读，但决策者和执行者需要快速获取核心结论。本次PRD摘要提炼过程中发现，摘要不是简单的"截断"——它需要重新组织结构（从分析推理链→决策参考结构），重新表达（从论证式→陈述式），并补充执行视角的内容（MVP优先级、成功指标、不做什么）。这是一个独立的思考和蒸馏过程。

**如何应用**：在分析类spec.md的Non-Goals或Goals中，明确将"精炼摘要/PRD"作为标准产出物纳入任务计划（类似本次task11整合报告之后应该有task12提炼摘要、task13原子化、task14提交）。

**模式映射**：建议更新分析类任务模板，将"摘要提炼"作为标准步骤加入工作流。

## 洞察4：子智能体最适合"规则明确的机械性任务"而非"深度跨章节推理"

**洞察陈述**：子智能体在原子化拆分（按行号/标题分割文件、添加frontmatter）等规则明确、边界清晰的机械性任务中表现优秀，准确率高；但在需要跨章节综合推理（如从10个task-output中整合出连贯报告）的任务中，容易遗漏关键连接、丢失推理链深度。

**触发场景**：任务分配决策时——什么任务委托子智能体，什么任务主agent直接执行。

**为什么重要**：错误的任务委托会导致质量下降和返工。本次经验表明：文件操作、格式转换、按模板生成、重复性检查等任务适合委托；跨多文件综合推理、创新概念生成、关键决策判断等任务必须主agent主导。

**如何应用**：
- ✅ 适合委托：文件拆分/合并/重命名、格式转换、按固定模板生成、链接检查、重复文本替换
- ⚠️ 需要指导后委托：报告整合（需要提供详细结构大纲和关键论点）、简单分析
- ❌ 不适合委托：核心分析推理、创新概念生成、关键设计决策、跨领域综合判断

**模式映射**：验证了现有"主agent分析+子agent执行"分工模式的正确性。

## 洞察5：反效果功能的识别需要"证据反向扫描"

**洞察陈述**：在技术-心理学映射中，单纯扫描"什么有效"会遗漏一类关键发现——"什么看似合理但实际有害"。需要专门设置"反效果扫描"环节，主动寻找有科学证据表明反效果的常见设计。

**触发场景**：功能设计评审、技术方案评估、最佳实践清单制作。

**为什么重要**：产品团队容易犯"加了总没坏处"的错误——虚拟奖励、社交排行、严格锁机等功能看似积极，但过度合理化效应、心理抗拒理论、社会比较焦虑等心理学研究明确表明它们长期损害内在动机。这类"好心办坏事"的功能如果不主动扫描识别，会默认进入产品。

**如何应用**：在技术-心理学映射矩阵中增加"反效果"列（强支持/弱支持/无证据/不利影响），强制每个常见设计都经过四档评估，不遗漏"不利影响"选项。

**模式映射**：建议将"证据反向扫描"作为第一性原理功能分析法的标准步骤（质疑阶段的子步骤）。

## 洞察6：上下文恢复后应执行"进度对齐检查"

**洞察陈述**：Session continuation后，仅依赖summary恢复上下文存在细节丢失风险。应在继续执行前，快速扫描关键中间产出物（task-output.md）以对齐分析深度和风格，而非立即推进。

**触发场景**：任何收到session continuation summary的场景。

**为什么重要**：Summary是高度压缩的，会丢失维度细节（如"8维度用户差异框架"具体是哪8个维度）。如果不重新确认就推进，可能导致后续产出与已有内容不一致或深度不匹配。

**如何应用**：收到summary后，按以下顺序对齐：
1. 读取spec.md和tasks.md确认任务范围和依赖
2. 对最近完成的2-3个task-output.md快速扫读关键结论和表格结构
3. 检查checklist.md确认验收标准
4. 然后才开始下一个任务

**模式映射**：建议更新上下文恢复SOP，增加"进度对齐检查"步骤。

## 洞察7：功能边界定义需要"对比框架"而非简单描述

**洞察陈述**：定义一个功能"是什么"不够，必须同时定义它"不是什么"——通过与易混淆功能（勿扰模式/专注模式/家长控制/冥想模式等）的系统对比，才能清晰界定功能的本质边界。

**触发场景**：定义新功能或重构已有功能、撰写PRD、功能评审。

**为什么重要**：如果只说"学习模式是帮助专注学习的系统功能"，这无法指导设计——勿扰模式也能帮助专注、番茄钟也能帮助专注。只有通过"本质目的→作用对象→干预层面→核心机制"四维度对比，才能发现学习模式的独特性在于"认知过程脚手架"而非"行为约束工具"。

**如何应用**：在功能定义中增加"功能边界"章节，强制列出3-6个最易混淆的功能，用多维度对比表清晰区分。使用"如果只有X特征，那它是A不是B"的判定句式。

**模式映射**：已在第10章实践（三层判定标准+决策树），建议沉淀为功能定义文档的标准章节模板。
