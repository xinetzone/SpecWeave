# v12 核心洞察：二手源校验与隐性奖项识别

## 洞察 1：二手源三层可信度校验法——多源分析中不可忽视的"来源纯度"问题

### 现象

第13个数据源《创作规范与参赛指南》（耿家威个人汇总）虽然信息密度高、结构清晰，但与官方一手源存在2处直接矛盾：
- 报名帖修改规则（二手源称"可修改等重审" vs 报名指南称"重发新帖"）
- 评审维度名称（二手源称"表达与呈现" vs 赛事细则称"社会商业价值"）

若不加校验直接采用二手源信息，将导致策略错误（如在旧帖上反复修改等待审核，结果审核只看新发帖子）。

### 规律

在多源信息分析中，**来源的"转述层级"与其可信度并非线性关系**——需要建立显式的三层校验框架：

```
一手源（官方章程/规则文档/产品经理直播）：可信度最高，但可能信息分散
    ↓ 转述
二手源（个人汇总/社区教程/第三方解读）：信息结构化程度高，但可能引入转述误差
    ↓ 转述
三手源（基于二手源的再加工）：可信度最低，仅作线索参考
```

关键规律：**二手源的价值不在于"提供正确答案"，而在于"提供线索"**——它可能揭示一手源中你尚未注意到的关键信息（如社会公益奖），但具体规则细节必须回溯到一手源验证。

### 行动

在信息源分层表中新增"来源层级"列（一手/二手/三手），二手源信息标注"待验证"标签，关键决策点必须回溯一手源确认。

> **一句话**：二手源是侦探的线索，不是法官的判决书——它告诉你去哪里找答案，但答案本身必须去一手源中确认。

## 洞察 2：隐性奖项通道识别——规则文档中"附加项"可能是竞争者最少的主战场

### 现象

社会公益赛题特别奖（4名额×¥50,000）在赛事细则、初赛指南、官网三个最高可信度一手源中均未单独列出奖金数额和名额——它出现在一个二手汇总文档的决赛奖项表中。竹简悟道已报名社会公益附加赛道，但此前策略仅将其作为"标签"而非"独立获奖通道"。

### 规律

赛事/投标/申请场景中的**奖项结构往往是分层的**：

| 奖项层级 | 可见度 | 竞争者数量 | 策略关注度 |
|---------|--------|-----------|-----------|
| 全场冠军/主赛道大奖 | 极高（首页宣传） | 极多 | 所有人关注 |
| 赛道分项奖 | 高（规则文档正文） | 多 | 大部分人关注 |
| **附加赛题特别奖** | **低（附录/表格角落/二手源）** | **少** | **少数人关注** |
| 参与奖/优秀奖 | 高（保底奖励） | 极多 | 多数人知道 |

核心规律：**附加赛题≠次要赛题**。当附加赛题有独立的奖金池和评审标准时，它本质上是一个独立的竞争赛道——而且由于其"附加"标签，竞争者往往远少于主赛道。

识别隐性奖项通道的检查清单：
- [ ] 奖项列表是否逐行扫描（而非只看主赛道）？
- [ ] "附加""特别""推荐"类标签背后是否有独立奖金和名额？
- [ ] 该奖项的竞争者估算是否显著少于主赛道？
- [ ] 自己的作品是否天然符合该奖项的评审标准？

### 行动

将社会公益特别奖从"附加标签"提升为"与学习工作赛道大奖同等主攻"的双通道策略，Demo帖中公益叙事与商业叙事平起平坐。

> **一句话**：最容易被忽略的奖项，往往是获奖概率最高的奖项——前提是你真的读了奖项表的每一行。

## 洞察 3：评审维度预期管理——当不同来源对评审标准表述不一致时，以章程级文档为准并覆盖全维度

### 现象

二手源提出的第四个评审维度"表达与呈现"与赛事细则的"社会商业价值"不一致。此前策略仅按初赛参赛指南的四维权重（创新性30%/体验30%/TRAE深度20%/价值20%）准备，但"表达与呈现"的提法暗示Demo帖本身的排版、截图质量、文字可读性可能影响评审体验。

### 规律

评审维度存在**显性权重**与**隐性印象**两层：

- **显性权重**：章程级文档明确公布的维度和百分比（如四维权重30/30/20/20）
- **隐性印象**：评审在阅读过程中的主观感受（排版美观度、叙事流畅度、截图质量）——这不会出现在官方评分表中，但会通过"体验与完成度"维度的打分间接体现

关键规律：**不要因为某维度未在官方权重表中列出就完全忽略它**——如果二手源或社区讨论反复提及某个评审关注点，即使章程未列，也应将其作为"体验与完成度"维度下的子项来准备。

### 行动

Demo帖准备中增加"表达质量"子检查清单：截图清晰度、排版美观度、段落节奏感、标题吸引力——这些归入"用户体验与产品完成度"30%权重下的"提交物体验"子项。

> **一句话**：官方评分表是骨架，评审的阅读体验是血肉——骨架决定得分上限，血肉决定你能不能摸到那个上限。

## 洞察 4：品牌叙事锚定——从"产品功能定位"到"产品线定位"的叙事升维

### 现象

v11阶段已知TRAE有IDE和Work两个入口，但将其作为背景信息处理。v12深化分析发现：Community Live产品介绍场以正式品牌语言定义了两条独立产品线（IDE「你的专属AI开发工程师」vs Work「全新上线的智能工作助手」），这一区分直接影响评审对作品的预期深度。

### 规律

在平台生态类竞赛中，**品牌的产品线划分是评审预期的隐形框架**：

```
产品线定位 → 目标用户画像 → 使用深度预期 → 评审评判标准
```

以TRAE为例：
- Work产品线目标是"所有人"，预期是"轻量提效"——用Work做一个App是符合预期的
- IDE产品线目标是"需要精细掌控代码的开发者"，预期是"深度使用"——用IDE做一个App是基础分，**用IDE构建方法论体系**才是超出预期

核心规律：**当你把作品定位到正确的产品线上时，你不再是在和所有参赛者竞争——你只在和该产品线目标用户的预期深度竞争**。SpecWeave位于IDE产品线的最高使用深度（四层架构方法论），这比"在所有作品中做到最好"更容易形成差异化。

### 行动

在SpecWeave的叙事中明确锚定TRAE IDE产品线定位，在Demo帖开头即声明"这是在TRAE IDE的Rules特性基础上构建的方法论体系"，而非泛泛地说"用TRAE做的项目"。

> **一句话**：评委评判的不是"你的作品好不好"，而是"你的作品在该产品线的目标深度上做到了什么程度"——选对产品线定位，等于选对了评委的评判坐标系。
