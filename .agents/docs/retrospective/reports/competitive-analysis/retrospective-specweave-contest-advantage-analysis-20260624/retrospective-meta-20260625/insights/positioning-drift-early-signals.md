# 洞察 4：定位漂移的早期识别信号——"如果核心术语不在你的问题域里，漂移已经开始" ⭐⭐⭐⭐

**现象**：Vibe Coding 作为 SpecWeave 的核心定位标签在 v10 被引入，到 v11 被修正为 AI 智能体协作。修正的触发点是用户一句话的提醒，但事后复盘发现，在 v10 使用 Vibe Coding 作为定位标签时，已经出现了三个早期信号：

| 早期信号 | 表现 | 为何当时被忽视 |
|---------|------|--------------|
| 标签来源错位 | Vibe Coding 来自大赛 Twitter 话题 #vibecoding 大赏，非来自 SpecWeave 解决的问题域 | 以为"蹭标签"是策略——实则是标签在替换问题 |
| 定义范围不自洽 | 每次写"Vibe Coding 方法论"时都需要额外解释"不只是写代码"——正常的定位标签不需要"降维声明" | 解释成本高是信号，而非文案问题 |
| 评审视角不对称 | 评委看到 Vibe Coding 会将其归类为"又一个蹭热词的参赛作品"——但分析时未做评审视角预演 | 忽略了"这个标签在别人眼中意味着什么" |

**操作指南**：产品定位引入新术语时，执行三项检查：(1) 这个术语来自问题域还是外部环境？(2) 是否需要额外解释才能覆盖产品全部能力？(3) 目标受众（评委/用户/投资人）看到这个术语的第一反应是什么？三项中任意一项亮红灯 = 定位漂移已开始。

> **一句话**：如果你的核心定位标签需要一句"不只是 XX"来解释，那它就已经框小了你的产品。
