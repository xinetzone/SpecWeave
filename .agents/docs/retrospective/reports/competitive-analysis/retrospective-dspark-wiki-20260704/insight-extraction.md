# 洞察萃取

## 核心洞察

### 洞察1：工具降级策略应成为标准操作

**触发场景**：WebFetch 工具获取微信公众号文章内容失败，立即改用 defuddle 技能成功提取内容。

**核心发现**：
- 关键路径上的单一工具失败不应阻塞任务，应有预案的降级策略
- 本次任务中 WebFetch → defuddle 的降级是依赖经验判断而非标准流程
- 如果 WebFetch 和 defuddle 都失败，任务将无法继续——关键路径的工具冗余不足
- 工具降级不是"失败后的补救"，而是"预先规划的备选方案"

**5-Whys 分析**：
1. **Why 1: WebFetch 失败？** → 微信公众号文章可能有反爬机制或特殊格式
2. **Why 2: 为什么没有立即降级？** → 实际上立即降级了，但这依赖经验判断而非标准流程
3. **Why 3: 为什么不是标准流程？** → 缺乏"工具失败降级矩阵"文档
4. **Why 4: 为什么没有降级矩阵？** → 工具规范中没有针对关键路径工具的失败预案
5. **根本原因**：**缺乏工具降级的标准化决策流程——关键路径工具失败应有三級降级预案**

**可复用模式**：网页内容获取应遵循"WebFetch → defuddle → 浏览器 MCP"三级降级策略，写入工具规范。每个关键路径工具都应有明确的降级路径和触发条件。

**可复用价值**：未来遇到 WebFetch 失败时，团队可直接查阅降级矩阵，减少决策时间。降级策略标准化后，不再依赖个人经验判断，新成员也能正确处理工具失败场景。

### 洞察2：子代理产出需要格式质量门

**触发场景**：Sub-agent 2 创建的 `docs/knowledge/README.md` 索引条目摘要和标签为空，需要手动用 Edit 工具补充。

**核心发现**：
- sub-agent 能完成核心任务（追加索引行），但对格式细节（如表格列完整性、摘要内容）不够敏感
- 子代理不是"黑盒"——委派任务后不能直接信任产出，必须在关键节点设置质量验证门
- 问题根因不在于子代理能力，而于提示中没有明确要求"参考现有条目格式填充所有列"
- 假设子代理会自动参考现有格式是错误的——必须在提示中显式给出格式参照样本

**5-Whys 分析**：
1. **Why 1: 索引条目摘要为空？** → sub-agent 不知道需要填充摘要
2. **Why 2: 为什么不知道？** → 提示中没有明确要求"参考现有条目格式填充所有列"
3. **Why 3: 为什么没有明确要求？** → 假设 sub-agent 会自动参考现有格式
4. **Why 4: 为什么假设会自动参考？** → 子代理提示工程中缺乏"格式参照样本"的标准要求
5. **根本原因**：**子代理提示中缺乏"格式参照样本"和"完整性检查清单"——格式敏感任务必须显式给出示例和必填字段清单**

**可复用模式**：子代理提示必须包含两个要素：（1）"格式参照样本"——给出 1-2 个现有条目示例；（2）"完整性检查清单"——明确列出所有必须填充的字段。委派索引更新、表格追加等格式敏感任务时，这两个要素缺一不可。

**可复用价值**：未来委派索引更新、表格追加等格式敏感任务时，可避免返工。这一模式可与 `retrospective-mopmonk-wiki-20260704` 中沉淀的"子代理 5 点验收检查清单"结合使用，形成"提示预防 + 验收拦截"的双重质量保障。

### 洞察3：验证脚本需要容错机制

**触发场景**：文件名规范检查脚本 `check-filename-convention.py` 因访问 `.chaos/FlowXM/plugins/dist/` 中的无关文件崩溃（WinError 1920）。

**核心发现**：
- 验证脚本扫描整个项目目录，单个文件访问错误导致整个脚本崩溃
- 脚本使用 rglob 遍历整个项目目录，缺乏 try-except 异常处理
- EXCLUDED_DIRS 配置不完整，未包含 `.chaos/`、`vendor/` 等非本项目目录
- 这导致验证步骤无法正常完成，只能基于人工判断跳过——削弱了验证的可信度

**5-Whys 分析**：
1. **Why 1: 脚本崩溃？** → 访问 `.chaos/FlowXM/plugins/dist/` 中的文件失败（WinError 1920）
2. **Why 2: 为什么会访问这个文件？** → 脚本使用 rglob 遍历整个项目目录
3. **Why 3: 为什么没有容错？** → rglob 遍历时没有 try-except 异常处理
4. **Why 4: 为什么没有排除机制？** → EXCLUDED_DIRS 配置不完整
5. **根本原因**：**验证脚本缺乏异常处理和目录排除机制——单文件失败不应阻塞整体扫描**

**可复用模式**：验证脚本应包含两个机制：（1）try-except 容错——单文件失败记录警告但不中断扫描；（2）EXCLUDED_DIRS 机制——排除 `.chaos/`、`vendor/`、`node_modules/` 等非本项目目录。这两个机制是验证脚本的基本质量要求。

**可复用价值**：修复后所有使用该脚本的任务都能避免因无关文件崩溃。这一模式也适用于其他扫描整个项目的验证脚本，是脚本工程质量的基本要求。

### 洞察4：Spec 驱动 + 并行子代理是高效模式

**触发场景**：9 个任务在 2 个 sub-agent 中并行完成，大幅缩短执行时间。

**核心发现**：
- Spec 定义清晰后，无依赖的任务可以并行委派给多个 sub-agent
- 关键策略：将写入同一文件的任务合并到一个 sub-agent，写入不同文件的任务分拆到不同 sub-agent
- 本次任务中 Tasks 1-7 都写入 `dspark-paper-wiki.md`，合并到一个 sub-agent；Task 8 写入 `README.md`，分拆到另一个 sub-agent
- 这种"按文件分組"的并行策略避免了写入冲突，同时最大化并行度

**可复用模式**：Spec → 任务依赖分析 → 按目标文件分组 → 并行子代理委派 → 验证。任务依赖分析的关键是识别"写入同一文件的任务必须串行，写入不同文件的任务可以并行"。

**可复用价值**：未来类似的多任务文档创建可复用此模式。例如创建多文件 Wiki、批量更新多个索引文件等场景，都可以通过"按文件分组"实现并行执行。

### 洞察5：单文件 Wiki 适合强耦合主题

**触发场景**：DSpark 论文的 10 个核心概念递进式讲解，后一个概念引用前一个概念。

**核心发现**：
- 概念间强耦合时（后一个概念引用前一个概念），单文件比原子化目录更利于递进式阅读
- 本次任务选择单文件 Wiki 而非原子化目录结构，因为 10 个概念是递进关系，拆分后会破坏阅读流
- 文档结构选择应基于内容耦合度，而非"一刀切"地选择原子化或单文件
- 与 `retrospective-mopmonk-wiki-20260704` 不同——MopMonk 的各章节相对独立，适合原子化；DSpark 的概念强耦合，适合单文件

**可复用模式**：文档结构选择决策树——内容强耦合（后文引用前文、递进式讲解）→ 单文件 Wiki；内容弱耦合（各章节独立、可随机阅读）→ 原子化目录。判断标准：如果把文件拆分后读者需要频繁跳转才能理解，则不应拆分。

**可复用价值**：未来创建 Wiki 时可据此快速决策文档结构。这一模式补充了 `retrospective-mopmonk-wiki-20260704` 中沉淀的"原子化拆分判断标准"（>300 行/章节独立/未来扩展），形成了完整的文档结构决策框架。

## 模式成熟度评估

**成熟度等级**：L1-experimental（实验性）

**评估依据**：
- 本次复盘为首次提炼这些洞察，`validation_count = 1`（仅本次任务验证）
- `reuse_count = 0`（尚未在其他任务中复用）
- 部分洞察与 `retrospective-mopmonk-wiki-20260704` 中沉淀的洞察有关联（如子代理质量门、文档结构选择），但本次是首次从 DSpark 任务场景中提炼

**升级路径**：
- L1 → L2（验证中）：需要在后续 2-3 个类似任务中验证洞察有效性，特别是"工具降级矩阵"和"子代理格式参照样本"两个高优建议
- L2 → L3（已验证）：需要在验证基础上，将洞察写入工具规范、子代理委派模板等强制执行点
- L3 → L4（标准化）：需要在多个项目中推广使用，形成团队共识

**关联已有模式**：
- 洞察2（子代理格式质量门）关联 `retrospective-mopmonk-wiki-20260704` 洞察3（子代理质量门），是同一模式在不同场景下的验证
- 洞察5（单文件 Wiki 适合强耦合主题）补充了 `retrospective-mopmonk-wiki-20260704` 洞察4（原子化拆分模式），形成完整的文档结构决策框架
