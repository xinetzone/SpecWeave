# 洞察萃取

## 一、核心洞察

### 洞察1：上下文注入机制的"信任-确定性"光谱

**现象**：Claude Code的8种上下文注入机制不是孤立存在的，它们沿着一条清晰的设计光谱排列。

**本质**：从CLAUDE.md到Hooks，存在一条从"完全信任模型判断力"到"完全不信任模型、代码确定性执行"的连续光谱：

```
信任模型 ←————————————————————————————————→ 机制保障
CLAUDE.md → Rules → Skills → Subagents → Output Styles → SysPrompt Append → Hooks
全程在场   按需出现  懒加载   隔离执行   system prompt   临时追加    代码执行
提示词级   提示词级 提示词级 独立窗口   system prompt   system prompt 硬护栏
```

每向右移动一步，对模型自觉性的依赖降低一层，执行确定性提升一层。这不是偶然的设计，而是对大模型行为特性深刻理解后的工程选择——模型在长上下文、压力、prompt injection等场景下，遵循提示词的可靠性会下降，因此安全关键操作必须从提示词层下沉到代码执行层。

**对SpecWeave的启示**：
- 目前阶段守卫（Stage Guardrails）已经在Hooks层级提供硬护栏，方向正确
- 需要审查：是否有"绝对不能做X"这类约束目前只写在AGENTS.md/规范文件中（提示词级），而没有对应的代码级检查？如果有，应该考虑下沉到自动化脚本
- Skills在"流程"场景的使用可以加强——目前有些流程性规范（如代码提交前检查）是写在全局规范中的，可以考虑封装为按需加载的Skill

### 洞察2：单上下文窗口是Agent三大缺陷的共同根因

**现象**：文章指出默认harness存在三个问题——偷懒（Agentic laziness）、自我偏好（Self-preferential bias）、目标漂移（Goal drift）。

**本质**：这三个问题看似独立，实际上有共同的架构根因——**单一共享上下文窗口**：

1. **偷懒**：单窗口中任务列表越长，注意力衰减越严重，模型倾向于提前"宣布完成"。注意力是有限资源，长列表中靠后的项目获得的注意力指数级下降。

2. **自我偏好**：单窗口中模型刚写完代码，思考过程和代码都在上下文中，审查时会产生认知惯性（confirmation bias），倾向于认为自己的输出没问题。独立窗口中的验证agent看不到思考过程，只看到最终输出，才能真正客观。

3. **目标漂移**：单窗口中每次上下文压缩都是有损的，最早进入的边缘需求和约束最先被遗忘，目标在多次压缩后逐渐漂移。

Dynamic Workflows的解决思路不是"让模型更聪明"或"写更好的prompt"，而是**从架构上改变上下文的组织方式**——通过子代理的独立上下文窗口，从结构上消除这三个问题的根因。这是一个深刻的架构洞察：**不要试图用prompt工程解决架构问题**。

**实证**：本次任务执行中子代理两次返回截断的摘要内容而非完整分析，恰好验证了Agentic laziness在我们自己的系统中同样存在——这不是Claude Code特有的问题，而是所有单窗口Agent系统的共性问题。

**对SpecWeave的启示**：
- 对抗验证（Adversarial verification）模式值得引入——特别是对代码修改、规范符合性检查这类需要高质量验证的任务，执行agent和验证agent应该分离
- 任务分解后通过子代理并行执行，可以天然缓解偷懒问题——每个子任务在自己的窗口中有明确的完成边界
- 长会话中应该有机制定期重新加载核心目标，防止目标漂移

### 洞察3："事实-流程-护栏-隔离任务"四象限分类法

**现象**：文章给出了决策口诀"事实放CLAUDE.md，流程放Skill，护栏放Hook，隔离任务给Subagent"。

**本质**：这背后是一个二维分类框架：

|  | 需要模型判断力 | 不需要模型判断力（确定性） |
|---|---|---|
| **全程需要在场** | 事实（CLAUDE.md） | - |
| **按需出现/懒加载** | 流程（Skills） | - |
| **不允许失败** | - | 护栏（Hooks） |
| **产生大量中间结果** | - | 隔离任务（Subagents） |

- **事实类信息**（目录结构、构建命令、编码规范）：不需要"执行"，只需要"知道"，全程在场才能在任何步骤都参考到 → CLAUDE.md/AGENTS.md
- **流程类信息**（部署步骤、review checklist、发布流程）：不是一直需要参考，只有在做特定操作时才需要完整说明 → Skills懒加载
- **护栏类约束**（禁止rm -rf、禁止删生产库、写文件后必须格式化）：这不是"参考"，而是"必须执行/必须阻止"，不能依赖模型记得 → Hooks代码级执行
- **脏活类任务**（搜索30个文件、翻日志、审计依赖）：会产生大量中间内容，不需要看过程只需要结论 → Subagents隔离执行

这个分类法的价值在于它提供了一个**决策标准**，而不是列举——未来出现新的上下文注入需求时，可以用这个框架判断应该放在哪里。

**对SpecWeave的启示**：
- 可以用这个四象限框架审查现有规范体系：是否有流程类内容错误地放在了AGENTS.md（全局入口）？是否有护栏类约束只写在规范中而没有自动化检查？
- Rules（路径限定）在这个框架中位于"事实"和"流程"之间——是路径限定的事实/局部规则
- Output Styles和System Prompt Append是特殊分类：它们改变模型行为模式，属于system prompt层级

### 洞察4：经验复用形成正反馈闭环（复盘的价值）

**现象**：本次任务直接采用了defuddle方案获取微信公众号内容，没有经历WebFetch失败的试错过程，比claude-tag任务（6月29日）节省了约3分钟。

**本质**：这是复盘知识沉淀产生实际价值的一个微小但具体的例证。复盘→经验入库→下次复用→节省时间→更多复盘，形成正反馈循环。这个循环的关键在于：

1. **经验必须结构化**：不是"我记得上次用了什么方法"，而是明确记录"什么场景→用什么方案→为什么→有什么备选"
2. **经验必须可检索**：在做同类任务时能够找到相关经验，而不是靠"记得"
3. **经验必须经过验证**：入库的经验是经过实际验证有效的，而非猜测

本次任务中，从ian-xiaohei先例（6月25日）验证defuddle有效→claude-tag任务建立双路径互补模型（6月29日）→本次直接复用首选方案（7月4日），这是一个三次迭代的经验积累过程。虽然每次节省的时间只有3分钟，但随着经验库增长，复利效应会越来越明显。

**对SpecWeave的启示**：
- 复盘流程不是"额外工作"，而是投资——每次复盘花10分钟，未来同类任务可能节省3-5分钟，多次复用后净收益为正
- 经验沉淀需要标准化格式，确保后续任务能够快速定位和复用
- 应该建立"同类任务先例参考"机制：在任务开始时自动检索相似任务的复盘报告

## 二、可复用模式识别

### 模式候选1：上下文生命周期管理原则

| 项目 | 内容 |
|---|---|
| 模式名称 | 上下文生命周期分层管理（Context Lifecycle Layering） |
| 适用场景 | 设计多智能体系统的上下文注入机制、规范体系架构、AI辅助编程框架 |
| 核心原则 | 不同类型的指令/规范/约束应该有不同的生命周期管理策略——从全程在场到按需加载到懒加载到隔离执行到代码硬护栏 |
| 决策框架 | 事实→全局入口文件；流程→懒加载技能；护栏→代码级Hook；脏活→隔离子代理 |
| 反模式 | 将所有内容都塞进全局入口文件导致膨胀；将安全约束写在提示词中期望模型自觉；让主会话承担所有搜索和阅读任务 |
| 成熟度建议 | L2（在Claude Code中已验证，在SpecWeave中部分实现） |

### 模式候选2：对抗验证工作流

| 项目 | 内容 |
|---|---|
| 模式名称 | 对抗验证（Adversarial Verification） |
| 适用场景 | 代码审查、安全检查、内容验证、任何需要高质量验证的任务 |
| 核心原则 | 执行任务的agent和验证结果的agent必须在独立上下文窗口中运行，验证agent看不到执行agent的思考过程，消除自我偏好偏差 |
| 实施要点 | 执行agent输出结果→验证agent只收到结果和验收标准→验证agent独立判断通过/不通过→不通过则返回具体问题给执行agent修复 |
| 反模式 | 同一个agent写完代码自己审查——单窗口自审必然存在认知盲区 |
| 成熟度建议 | L1（Claude Code提出，SpecWeave尚未采用） |

### 模式候选3：架构问题不用prompt解决

| 项目 | 内容 |
|---|---|
| 模式名称 | 架构优先于提示（Architecture Over Prompting） |
| 适用场景 | Agent系统设计、可靠性工程、安全防护 |
| 核心原则 | 当遇到系统性问题（偷懒、自我偏好、目标漂移、安全漏洞）时，优先从架构层面解决（改变上下文组织方式、增加隔离层、代码级拦截），而不是试图通过写更好的prompt来弥补架构缺陷 |
| 判断标准 | 如果一个问题反复出现，且调整prompt只能缓解不能根除，那它大概率是架构问题 |
| 反模式 | "模型不够聪明所以犯错"→"写更好的系统提示词"→问题在特定场景下再次出现→循环往复 |
| 成熟度建议 | L2（在Claude Code Hooks设计中已验证） |

## 三、规律认知

### 规律1：提示词遵循率的上下文衰减定律

在长上下文中，模型对系统提示词和约束的遵循率随着上下文长度增加和对话轮次增加而衰减。具体表现为：
- 上下文中较早出现的指令在后期被忽略的概率更高
- 对话轮次越多，模型越容易"忘记"初始约束
- 复杂、多步骤的约束比简单约束更容易被遗忘
- 压缩操作会加速这一衰减过程

**推论**：安全关键约束必须放在代码执行层（Hooks），不能依赖提示词在长会话中始终被遵循。

### 规律2：子代理任务粒度的token上限定律

子代理（subagent）单次输出存在有效token上限，超过这个上限的任务会导致输出截断或提前"宣布完成"。具体表现为：
- 合并多个不相关子任务委派给一个子代理，返回结果大概率是摘要而非完整内容
- 子代理倾向于在输出量达到一定程度后总结性收尾，即使任务未完成
- 任务描述中没有明确要求"返回完整内容"时，子代理更容易返回概括性结果

**推论**：子代理任务应该原子化——单个Task一次委派，明确要求完整输出，避免多任务合并委派。

## 四、认知升级点

本次学习带来的几个认知更新：

1. **从"机制列表"到"设计光谱"**：最初看到7种机制以为是功能罗列，深入分析后发现它们构成一条连续的设计光谱，每个机制的位置都是对模型特性深刻理解后的权衡选择。

2. **从"prompt工程万能"到"架构优先"**：过去遇到Agent行为问题第一反应是"怎么改prompt"，现在理解到架构层面的改变（独立窗口、代码级拦截）往往比prompt优化更根本、更可靠。

3. **从"子代理是并行工具"到"子代理是问题解决方案"**：之前认为Subagents主要用于并行加速，现在理解到子代理的独立上下文窗口本身就是解决偷懒、自我偏好、目标漂移三大问题的架构手段。

4. **对"经验复用"价值的具体感知**：3分钟的节省看起来不多，但这是一个具体的、可量化的复盘价值例证——当经验库足够丰富时，试错成本会系统性降低。
