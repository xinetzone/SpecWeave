# 05 - 内容优化建议生成（E阶段·萃取建议）

## 一、E阶段方法论说明

### E（萃取）阶段核心原则
基于R（事实采集）→I（概念清单）→C（概念关系）→F（第一性原理）→V（对抗性审查）→I（结构洞察）阶段的发现，生成**具体可操作、可溯源、可验证**的优化建议，遵循以下原则：

1. **可溯源原则**：每条建议必须明确引用前面评估中发现的具体问题（来自03或04文件），不凭空创造建议
2. **建设性语气**：尊重原作者的写作意图和劳动成果，所有建议以"可以优化为"而非"这里写错了"的方式表达
3. **分级原则**：按优先级P0-P3区分，让作者清楚知道先改什么、后改什么
4. **可操作原则**：每条建议给出具体的优化方向和可直接套用的改写示例，而非泛泛而谈
5. **最小改动原则**：在不改变文章核心风格和定位的前提下进行优化，不重构整篇文章

### 优先级定义
| 优先级 | 标记 | 说明 | 修改必要性 |
|--------|------|------|-----------|
| P0 | 🔴 必须改 | 事实错误/严重误导，会导致读者获得错误信息 | 发布前必须修正 |
| P1 | 🟠 高优 | 关键信息缺失/影响核心理解，会显著降低文章价值 | 强烈建议修改 |
| P2 | 🟡 中优 | 体验优化/补充信息，能提升文章完整性和友好度 | 建议修改 |
| P3 | 🟢 低优 | 锦上添花/结构微调，不影响核心内容 | 可选修改 |

### 建议分类标准
| 分类 | 说明 |
|------|------|
| 事实修正类 | 修正文中的事实错误、前后矛盾、技术不准确 |
| 术语澄清类 | 补充关键术语的解释，消除理解障碍 |
| 结构优化类 | 调整章节顺序、增加过渡、优化信息呈现方式 |
| 信息补全类 | 补充缺失的关键信息、操作闭环、边界条件说明 |
| 表述改进类 | 优化不准确、有歧义、过于绝对的表述 |

## 六、可复用模式萃取：技术科普文写作Checklist

从本次WSL Containers文章评估中，萃取一套**面向开发者的技术科普文/教程通用写作Checklist**，可复用在后续类似技术文章写作中。

### 模式名称：《技术科普文"六有三不"写作Checklist》

#### ✅ 写作前"三明确"
- [ ] **明确目标读者**：是新手入门？还是有经验的开发者迁移？不同读者需要的前置知识准备完全不同
- [ ] **明确读者能带走什么**：读完能做什么？（跑hello-world？能做开发？能评估是否采用？）
- [ ] **明确前置知识假设**：哪些术语是读者必须知道的？哪些是需要解释的？列出来

---

#### ✅ 内容结构"六有"标准

| 检查项 | 说明 | 本文做得如何 |
|--------|------|-------------|
| **1. 有预期管理** | 开头说明：阅读时长、适合谁/不适合谁、读完能获得什么，让读者快速判断"这篇是不是给我看的" | ✅ 做得好（摘要有阅读时长和适合人群） |
| **2. 有术语脚手架** | 第一次出现的专业术语必须给出一句话解释，或在开头提供术语速查表；不假设读者知道所有缩写 | ❌ 缺失（8个术语零解释） |
| **3. 有操作闭环** | 实践教程不能只教"启动"，要覆盖完整生命周期：创建→使用→调试→排错→停止→清理；至少给一个完整工作流示例 | ❌ 半闭环（缺exec/logs/rm/持久化） |
| **4. 有前置检查** | 操作步骤前告诉读者怎么确认环境满足条件，不要让读者做了几步才发现环境不支持 | ❌ 缺失（无虚拟化检查） |

| 检查项 | 说明 | 本文做得如何 |
|--------|------|-------------|
| **1. 不前后矛盾** | 关键信息（如版本要求、支持的功能）在文中多次出现时必须一致；发布前通读一遍检查相同说法是否统一 | ❌ 存在矛盾（Win10支持前后说法不一） |
| **2. 不绝对化表述** | 技术文章慎用"零门槛"、"一模一样"、"照样能用"、"完全替代"这类绝对词；加上"大多数"、"基础场景下"、"目前"这类限定词更严谨 | ⚠️ 有多处绝对化表述（可接受但可优化） |
| **3. 不混淆场景** | 同一个技术特性在不同场景下行为不同时（如CLI vs API、普通用户vs开发者），必须明确说明适用范围，不能以偏概全 | ❌ 存在混淆（隔离模型CLI/API不分） |
