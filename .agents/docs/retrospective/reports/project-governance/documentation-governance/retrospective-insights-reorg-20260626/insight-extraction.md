## INS-01：文件拆分的自然边界识别原则

### 支撑事实

本次重组将 3265 行的 `insights-31-65.md` 拆分为两个文件。拆分点选择在洞察 53/54 之间，依据是竹简悟道 Specs 文档体系的四层结构定义：

| 洞察范围 | 内容层级 | 拆分后文件 | 行数 |
|---------|---------|----------|------|
| 31-53 | 哲学层（认知/方法论哲学） | `insights-31-53.md` | 1503 |
| 54-68 | 元层（洞察本身的结构与演化） | `insights-54-68.md` | 1844 |

行数均衡作为次要验证指标：拆分后行数比为 1:1.23，最大/最小行数比为 2.6:1（重组前为 4.7:1），均衡度显著改善。

若按行数均分（约 1982/1983 行），拆分点将落在洞察 58 附近，割裂「元层」洞察群组的内聚性。

### 深层含义

文件拆分存在两种边界：

| 边界类型 | 判定依据 | 优势 | 风险 |
|---------|---------|------|------|
| **语义边界**（首选） | 内容主题/层级/职责的自然分界 | 保持内容内聚性，便于按主题定位 | 行数可能不均 |
| **机械边界**（次选） | 行数/字数均分点 | 行数均衡，加载性能一致 | 割裂内容内聚性 |

**核心原则**：语义边界优先于机械边界。行数均衡应作为验证指标（防止极端失衡），而非决策依据。当语义边界与机械边界冲突时，优先满足语义内聚性，再通过其他手段（如进一步细分）改善均衡度。

### 可迁移性说明

适用于以下场景：
- 长文档拆分为多文件（如规格文档按章节拆分、复盘报告按模块拆分）
- 大型代码文件拆分为模块（按职责边界而非代码行数）
- 知识库按主题分层组织（按领域边界而非条目数量）
- 数据集分片（按业务域边界而非记录数）

**判断流程**：
1. 识别内容的天然分层结构（主题/层级/职责）
2. 在相邻层的交界处确定候选拆分点
3. 验证拆分后的行数/规模均衡度是否在可接受范围（建议最大/最小比 < 3）
4. 若均衡度严重失调，考虑在语义边界内进一步细分，而非跨越语义边界均分

### 成熟度评估

**L3（标准化）**：原则清晰、判断流程明确，已在本次重组中验证有效，且与既有方法论 `document-system-refactoring.md` 的「原子化拆分」步骤形成互补。可推广至所有文档拆分场景。

## INS-02：交叉引用更新三步法

### 支撑事实

本次重组涉及全项目 11 个文件对旧文件名 `insights-31-65.md` 的交叉引用更新。采用以下三步法完成，最终回归验证旧文件名搜索 0 匹配：

| 步骤 | 操作 | 工具 | 产出 |
|------|------|------|------|
| 1. 全量搜索 | 搜索旧文件名 `insights-31-65` 的所有引用点 | Grep（全项目） | 11 个引用点清单 |
| 2. 分类归并 | 按引用的洞察编号归类到新文件（31-53 或 54-68） | 人工分类 | 每个引用点的新目标映射 |
| 3. 逐文件替换 + 回归验证 | 逐文件精确替换引用；替换后再次全量搜索旧文件名 | Grep + Edit | 0 残留 |

**关键节点**：步骤 3 的回归验证是闭环关键。更新完成后再次执行 `insights-31-65` 全量搜索，确认 0 匹配，证明无遗漏引用。

### 深层含义

交叉引用更新本质上是「引用图的重构」：当一个节点（文件）被拆分为多个节点时，所有指向原节点的边（引用）须重新指向正确的新节点。

三步法的核心设计：

| 设计点 | 原理 |
|--------|------|
| 先搜索后更新 | 避免边更新边发现新引用点导致的遗漏 |
| 分类归并 | 确保每条引用指向语义正确的新文件，而非机械替换 |
| 回归验证 | 闭环确认，将「相信已全部更新」转化为「证明无残留」 |

**与机械全局替换的对比**：若直接全局替换 `insights-31-65` 为 `insights-31-53`，会导致引用 54-68 洞察的文件指向错误的新文件。分类归并步骤避免了这一错误。

### 可迁移性说明

适用于以下场景：
- 文件重命名引发的全局引用更新
- 文件拆分/合并引发的多对多引用重映射
- API 重命名引发的调用点更新
- 文档迁移（路径变更）引发的链接更新
- 符号重命名引发的代码引用更新

**适配建议**：
- 小规模引用（< 10 处）：三步法手工执行
- 中等规模（10-50 处）：三步法 + 脚本辅助分类
- 大规模（> 50 处）：三步法 + 自动化替换脚本 + 回归测试

### 成熟度评估

**L3（标准化）**：流程明确、可重复、有回归验证闭环。与既有方法论 `fact-statement-consistency-loop.md`（事实表述一致性闭环：修正一处→搜索同类→统一修正）同构，可视为其在文件引用场景的具体实例。建议入库 `docs/retrospective/patterns/methodology-patterns/`。

## INS-03：结构债务的渐进式积累模式

### 支撑事实

本次重组清理了 4 类历史结构债务，共计 21+ 处：

| 债务类型 | 数量 | 推测成因 | 残留时长 |
|---------|------|---------|---------|
| 临时统计块残留 | 4 处 | 前期内容审计生成，未标注临时性质，事后未清理 | 长期 |
| 标题层级错乱（`##` 应为 `###`） | 4 处 | 编辑时疏忽，肉眼审查未发现 | 长期 |
| 标准结构缺失（来源/核心内容） | 9 条 | 洞察 54-62 创建时未遵循标准结构 | 创建即缺失 |
| 双分隔符 | 若干处 | 编辑冲突或合并遗留 | 长期 |
| 文件名与内容不符 | 1 处 | 增量追加洞察 66-68 时未同步更新文件名 | 增量期 |

### 深层含义

结构债务的积累遵循「渐进式积累」模式，类似于代码技术债务：

```
债务生成（一次性/增量） → 未被检测 → 沉淀为「正常状态」 → 积累至阈值 → 触发重构
```

**关键转化节点**：债务从「一次性」演变为「永久残留」的转化发生在「未被检测」阶段。以临时统计块为例：

| 阶段 | 状态 | 可识别性 |
|------|------|---------|
| 生成时 | 明确为临时内容 | 高（生成者知道是临时的） |
| 未标注 | 无标记区分正式/临时 | 低（后续维护者无法区分） |
| 沉淀 | 被视为「正常内容」 | 极低（已被接受为现状） |
| 重构时 | 重新识别为债务 | 高（全量审查时暴露） |

**核心机制**：临时内容若不在生成时标注其临时性质，会在维护者更迭后「伪装」为正式内容，从而失去被清理的机会。

### 可迁移性说明

适用于以下场景的债务识别与预防：
- 文档体系长期演进（临时块/TODO/草稿残留）
- 代码库技术债务管理（临时 hack/调试代码残留）
- 配置文件演化（废弃配置项残留）
- 数据库 schema 演进（废弃字段残留）

**预防机制**：
1. **临时内容标注**：临时内容须用约定标记标注（文档用 `<!-- TEMP: 原因 -->`，代码用 `// TODO(temp): 原因`）
2. **生命周期声明**：临时内容生成时声明预期清理时机（如「本次任务收尾时清理」）
3. **定期审计**：将临时内容清理纳入定期审查清单

### 成熟度评估

**L2（已验证）**：模式清晰，已在本次重组中识别出债务积累路径，但预防机制尚未在项目中常态化实施。需在第 2 次文档重构场景中复测预防机制有效性后，可升级为 L3。

## INS-04：标题层级作为文档结构健康度指标

### 支撑事实

本次重组修复了 4 处标题层级错乱（洞察 53/55/56/57 的子节误用 `##` 而非 `###`）：

| 洞察 | 错误层级 | 正确层级 | 影响 |
|------|---------|---------|------|
| 53 子节 | `##`（与洞察主标题同级） | `###` | 子节「伪装」为独立洞察 |
| 55 子节 | `##` | `###` | 同上 |
| 56 子节 | `##` | `###` | 同上 |
| 57 子节 | `##` | `###` | 同上 |

**为何长期未被发现**：
- `##` 与 `###` 在渲染外观上差异较小（字号相近），肉眼审查易忽略
- 目录生成工具会按错误层级生成目录，但读者通常不会逐项核对
- 文档规模增长后，全量人工审查标题层级成本过高

### 深层含义

标题层级是文档结构的「骨架」，其正确性直接影响：

| 影响维度 | 错误层级的后果 |
|---------|--------------|
| 目录生成 | 产出错误层级目录，误导导航 |
| 语义结构 | 子节「伪装」为同级内容，破坏编号体系严谨性 |
| 可读性 | 读者难以区分主内容与子内容 |
| 自动化解析 | 基于标题层级的解析工具（如提取大纲）产出错误结果 |

**作为健康度指标的价值**：标题层级错误是文档结构退化的早期信号。与「临时块残留」「结构字段缺失」相比，标题层级错误更易被自动化检测（仅需检查层级序列的严格递进性）。

**检测规则**：
- 标题层级须严格递进（`#` → `##` → `###`），禁止跳级（`#` → `###`）
- 同级标题下的子标题须统一为下一级（不能部分 `##` 部分非 `##`）
- 文件首个标题须为 `#`（一级标题）

### 可迁移性说明

适用于以下场景：
- Markdown 文档结构健康度监控（本项目所有 `.md` 文件）
- 文档审查清单的自动化补充项
- CI 检查中的文档质量门禁
- 技术文档写作规范的执行验证

**自动化建议**：开发标题层级检测脚本，检查项包括：
1. 层级严格递进（无跳级）
2. 同级标题子标题统一
3. 首标题为一级
4. 无孤立的深级标题（如文件中直接出现 `###` 而无前置 `##`）

### 成熟度评估

**L2（已验证）**：指标定义清晰、检测规则明确，但自动化检测工具尚未实现。待工具开发并纳入 CI 后，可升级为 L3。

## 方法论提炼：四层结构拆分法

基于本次重组，提炼出「四层结构拆分法」方法论：

```mermaid
flowchart LR
    S1["1： 识别内容分层"] --> S2["2： 定位语义边界"]
    S2 --> S3["3： 验证规模均衡"]
    S3 --> S4{"均衡度可接受?"}
    S4 -->|"是"| S5["4： 按边界拆分"]
    S4 -->|"否"| S6["4a. 边界内细分"]
    S6 --> S5
    S5 --> S7["5： 交叉引用更新"]
    S7 --> S8["6： 多维验证"]
```

| 步骤 | 输入 | 输出 | 关键原则 |
|------|------|------|---------|
| 1. 识别内容分层 | 原始文件 | 内容层级划分 | 依据内容主题/职责，非行数 |
| 2. 定位语义边界 | 层级划分 | 候选拆分点 | 优先相邻层交界 |
| 3. 验证规模均衡 | 候选拆分点 + 行数 | 均衡度评估 | 最大/最小比 < 3 |
| 4. 按边界拆分 | 拆分点 | 新文件 | 保持内容完整性 |
| 5. 交叉引用更新 | 旧文件名引用 | 新引用映射 | 三步法（INS-02） |
| 6. 多维验证 | 重组结果 | 验证报告 | 完整性/均衡性/引用/结构 |

### 适用范围

- 知识库按主题分层组织
- 长文档按章节拆分
- 规格文档按模块分解
- 复盘报告按子模块归档
