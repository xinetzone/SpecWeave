# 洞察萃取：Mermaid 治理闭环执行中的模式发现

## 关键洞察

### 洞察1：工具自测效应（Tool Self-Testing Effect）

**现象**：在创建内置 `%%` 注释的安全模板时，check-mermaid 首次运行就发现了工具自身的 bug（注释行中 `\n` 误报）。这个 bug 在之前对全项目653+文件的扫描中从未触发，因为之前没有文件在 `%%` 注释中写过 `\n` 字面量。

**分析**：当你用工具来验证包含该工具反例/说明的文档时，天然创造了一个"对抗性测试"场景——文档必须展示错误写法作为反例，而工具应该只在真实代码中检测错误、不在注释/说明中报警。这种"文档即测试用例"的效应比单纯的单元测试更有效，因为它测试的是工具在真实使用场景下的行为。

**本质**：**使用工具来验证描述该工具规则的文档，是发现工具边界case的高效方法。** 这补充了上一轮复盘萃取的"点修复偏误"——不仅需要对同类问题做系统性扫描，还需要用工具验证工具自身的规则说明文档。

### 洞察2：默认缺失偏误（Default-Absence Bias）

**现象**：在上一轮复盘中，"CI集成check-mermaid"被列为🔴高优先级待办项。但验证时发现 ci-check.ps1/sh 第4步早已包含了 check-mermaid 检查。改进建议制定者（我自己）在没查看 CI 脚本内容的情况下，默认假设"没有CI集成"。

**分析**：在制定改进建议时存在一种认知偏误——发现一个问题后，倾向于假设"什么都没有"，从零开始建设方案，而不是先验证现有基础设施是否已经覆盖了这个需求。这种偏误导致：
- 重复建设风险：可能投入资源做已有功能
- 优先级误判：把"验证现有能力"标记为"新建能力"
- 资源浪费：时间花在规划已完成的事情上

**对策**：制定改进建议前，应执行"已有能力验证"步骤——先搜索现有脚本、配置、文档中是否已有相关能力，再决定是"新建"还是"激活/修复/补全"。

### 洞察3：文档入口聚合效应（Documentation Entry-Point Aggregation）

**现象**：Mermaid 相关知识分散在4个位置（模板目录、模式库、检查脚本、开发规范），用户需要在多个文件间跳转才能获取完整信息。创建 mermaid-guide.md 一站式操作指南后，信息获取路径从"多文件跳转"变为"单文件直达"。

**分析**：规范落地不仅需要规则本身正确，还需要**降低规则获取成本**。如果规则分散在多个文件中，即使每个文件都正确，用户也可能因为查找成本高而放弃查阅。这解释了为什么上一轮复盘中"规范存在但未执行"——不是规范写得不好，而是获取规范的"摩擦力"太大。

**本质**：**知识分散是规范不被遵守的结构性原因之一。** 为高频任务创建一站式入口文档，可以显著降低规范遵守的摩擦力。这与"三道防线"中的"源头预防"互补——模板降低写作摩擦力，入口文档降低查阅摩擦力。

### 洞察4：自指问题（Self-Reference Problem in Documentation）

**现象**：编写 Mermaid 操作指南时遇到了一个独特的约束：在 Mermaid 图中展示 Mermaid 反例（如"不要用 `\n`"）时，不能直接写 `\n`，否则会被 check-mermaid 检测为错误；在 Mermaid 节点内展示代码示例 `["文本"]` 时，不能使用 `\"` 转义（Mermaid 不支持），必须改用中文引号「」。

**分析**：这是一个"用X描述X"的自指问题。当文档主题是某种语法/语言时，在该语言的代码块内展示该语言的反例和语法示例会遇到特殊约束：
- 反例不能触发检测器
- 语法示例中的特殊字符需要替代表示
- 代码示例中的括号/引号可能与外层语法冲突

**对策**：在编写自指文档时，需要专门设计"安全的反例展示策略"：
- 用自然语言描述替代字面量（如"反斜杠+n"而非`\n`）
- 用不同类型的引号（中文「」替代英文""）
- 将反例放在Mermaid代码块之外的Markdown表格中

## 可复用模式萃取

### 模式A：工具自测文档模式（Tool Self-Documentation Testing Pattern）

**描述**：当为一个检查/修复工具编写规则说明文档、模板或教程时，将反例和正确示例都放在文档中的Mermaid代码块里，然后用该工具检查文档本身。如果工具误报了文档中的反例说明，说明工具存在边界case bug。

**适用场景**：
- 为linter/checker类工具编写规则文档
- 创建内置安全提醒的代码模板
- 更新陷阱速查表
- 编写操作指南和最佳实践文档

**步骤**：
1. 编写包含正反例的文档/模板
2. 用目标工具检查该文档
3. 如果工具对注释/说明中的反例误报，修复工具的注释感知能力
4. 再次检查直到0误报
5. 此时文档和工具都达到了更高质量

### 模式B：改进建议前置验证模式（Pre-Recommendation Verification Pattern）

**描述**：在复盘报告中提出改进建议前，必须先验证现有能力是否已经覆盖了该建议。避免"默认缺失偏误"导致的重复建设和优先级误判。

**验证清单**：
- [ ] 搜索现有脚本/工具中是否已有相关功能
- [ ] 检查CI/配置文件中是否已有相关集成
- [ ] 查阅模式库/知识库中是否已有相关模式
- [ ] 确认问题真的是"能力缺失"而非"能力未被使用"

### 模式C：一站式入口模式（One-Stop Entry Point Pattern）

**描述**：对于高频使用的规范/工具/流程，创建一个一站式入口文档，整合规则、模板、工具用法、排错流程和参考链接，降低知识获取的摩擦力。

**入口文档的核心要素**：
1. **快速开始**：3步以内完成核心任务
2. **核心规则**：最常违反的规则（含正反例）
3. **工具用法**：检查/构建/部署命令
4. **排查流程**：出问题时按什么顺序排查（最好用Mermaid流程图）
5. **检查清单**：完成前对照检查
6. **参考索引**：链接到详细文档（不重复内容）

**与详细文档的关系**：入口文档是路线图，详细文档是参考书。入口文档不替代详细文档，而是降低到达详细文档的路径成本。

## 与已有模式的关系

| 新萃取模式 | 关联已有模式 | 关系 |
|-----------|------------|------|
| 工具自测文档模式 | 三道防线模型 | 补充"源头预防"层——模板和文档自带测试效果 |
| 改进建议前置验证模式 | 点修复偏误 | 从"修复遗漏"扩展到"规划遗漏" |
| 一站式入口模式 | 规范落地断裂 | 解决"规范存在但找不到"的摩擦力问题 |
| 自指问题策略 | mermaid-safe-coding-rules | 补充"文档编写"场景的特殊约束 |
