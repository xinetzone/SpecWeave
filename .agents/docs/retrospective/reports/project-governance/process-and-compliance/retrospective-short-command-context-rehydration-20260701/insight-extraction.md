# 洞察萃取 — 短指令跨会话上下文重建与参数澄清

## 一、关键洞察

### 洞察 1：短指令在新会话中的第一职责是“触发上下文恢复”

在连续会话里，`复盘+洞察+萃取+导出` 可以直接映射到既有工作流；但在新会话中，它首先表达的是“我要进入这个阶段”，而不是“你已经拥有全部执行参数”。因此，短指令的正确解释顺序应是：

1. 识别命中已注册短指令；
2. 恢复项目协议与任务路由；
3. 判断上下文是否足够；
4. 仅对缺失槽位发起澄清。

### 洞察 2：协议优先不是形式主义，而是防止三类高频返工

若跳过启动协议，短指令执行最容易出现三类偏差：

- **对象错位**：把当前会话误当成某个已有目录、某份报告或某次故障。
- **结构错位**：把四件套误降级成摘要，或反过来过度生成。
- **归档错位**：写入错误主题目录，后续还要补索引、补引用、补分类。

本次会话先读 `AGENTS.md` 与复盘命令集，再进入澄清，实质上是在前置消除这三类返工。

### 洞察 3：短指令的最小参数集可以标准化为两个槽位

对“复盘类短指令”而言，最少只需补齐两个参数就足以稳定执行：

| 槽位 | 作用 | 本次取值 |
|------|------|---------|
| 处理对象 | 决定事实源、主题分类与标题命名 | `当前会话` |
| 交付形态 | 决定文件数量、结构深度与是否导出附件 | `标准四件套` |

这说明短指令并不必然导致高沟通成本。只要把缺失参数抽象为固定槽位，就能把澄清成本压到 1-2 步。

### 洞察 4：样例先行能把抽象需求快速收敛为仓库兼容产物

本次没有直接从用户短语推导最终文件结构，而是先读取：

- 报告总索引；
- 复盘/洞察/导出命令说明；
- 现有原子化样例；
- 目录级 README。

这种“先对齐现有产物，再开始生成”的方式，比纯靠经验组织文档更稳定，尤其适合文档体系已经成熟的仓库。

### 洞察 5：短指令模式需要补上一条“跨会话澄清守则”

已有 `short-command-patterns.md` 主要强调“首轮建立完整上下文，后续用短指令推进”。本次验证补充了另一半边界条件：

> 当短指令出现在新会话或上下文不完整场景时，智能体应先通过结构化问题补齐对象与交付形态，再继续执行。

这不是对短指令模式的削弱，而是让它从“连续会话有效”扩展为“跨会话也可安全使用”。

## 二、可复用模式映射

| 模式 | 本次验证结论 | 说明 |
|------|--------------|------|
| `short-command-patterns` | 再次验证通过 | 短指令仍然有效，但需补充跨会话澄清守则 |
| `review-insight-export-loop` | 再次验证通过 | 复盘 → 洞察 → 导出 的闭环结构仍然稳定 |
| 启动协议优先 | 再次验证通过 | 协议读取前置能显著降低目录/结构返工风险 |

## 三、可沉淀结论

- **结论 1**：复盘类短指令的执行前置条件，不是“用户写得够长”，而是“智能体能否识别缺失槽位”。
- **结论 2**：跨会话短指令最适合搭配“固定槽位澄清”，而不是开放式追问。
- **结论 3**：成熟仓库里，模板与索引本身就是强约束工具，先读样例比先写正文更重要。
