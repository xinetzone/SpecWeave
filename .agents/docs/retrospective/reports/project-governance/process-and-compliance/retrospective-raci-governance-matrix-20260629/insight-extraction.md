# RACI治理责任矩阵落地 — 洞察萃取

## 一、规律认知（Laws）

### law-01: RACI矩阵A唯一性强制约束

**规律**：RACI矩阵中每项活动必须有且仅有一个A（Accountable），这是不可妥协的硬约束。当出现双A时，本质是两个不同审批层级/场景被合并到了同一行。

**触发条件**：
- 审批活动涉及"常规审批"和"重大/紧急审批"两个层级
- 审批角色根据场景不同而变化
- RACI行中出现两个单元格含A标记

**正确做法**：按审批层级拆分为两行，明确区分常规场景（低层级A）和重大场景（高层级A），避免职责模糊。

**反模式**：在同一个RACI行中给两个角色都标记A，用模糊描述覆盖所有场景。

### law-02: R≠A分离原则——执行与审批必须解耦

**规律**：执行操作类活动（代码实现、文件拆分、提交、漏洞修复等）必须遵循"执行者≠审批者"原则，即R和A必须是不同角色。这是治理体系防止自我审批、保证质量门禁独立性的核心机制。

**触发条件**：
- 活动涉及实际代码/文件修改
- 活动产出物需要质量验收
- 活动执行者有动机绕过检查

**正确做法**：developer执行（R）→ reviewer审批（A），形成执行-审批的制衡关系。

**例外**：只有完全独立的质量门禁类活动（如reviewer自己做安全审计、自己审自己的检查清单）才允许reviewer R/A，但这类活动不涉及修改产出物。

### law-03: 五层审批模型的R/A双列设计优于单列审批角色

**规律**：审批模型表格设计中，单独一列"审批角色"会导致角色边界模糊——无法区分"谁执行"和"谁审批"，容易让执行者同时成为审批者。采用"主要执行者(R)"+"最终审批者(A)"双列设计可以天然避免R/A混淆。

**触发条件**：
- 设计审批层级模型
- 定义角色职责边界
- 建立流程治理框架

**正确做法**：每个层级明确标注R和A，而非笼统说"审批角色"。

### law-04: RACI格式一致性是批量应用质量的前提

**规律**：在多个文档中批量应用RACI矩阵时，格式不一致问题（A未加粗、标记遗漏）会反复出现。根本原因是缺少前置的RACI格式模板和自动化校验工具，导致人工检查有遗漏。

**触发条件**：
- 需要在3个以上文档中新增RACI矩阵
- 多人/多轮次编辑RACI内容
- RACI格式包含特殊标记（如加粗）

**正确做法**：
1. 先建立RACI格式模板（明确加粗规则、R/A/C/I标记约定）
2. 编写自动化校验脚本检查A唯一性、A加粗、行列完整性
3. 每新增/修改一个RACI文档后立即运行校验

### law-05: 治理模型修正必须触发关联文档同步检查

**规律**：治理模型（如五层审批模型）发生变更时，所有引用该模型的RACI文档都需要同步检查和更新。如果依赖人工记忆来完成同步，必然会有遗漏。

**触发条件**：
- 审批模型/角色职责定义发生变更
- 某角色的R/A分配原则被修正
- 层级顺序/流程边界被调整

**正确做法**：建立变更影响分析机制——模型变更时自动扫描所有引用该模型的RACI文档，输出需要同步检查的文件清单。

## 二、关键发现（Findings）

### finding-01: 数据收集/流程协调类活动归orchestrator R/A

在所有5个指令集中，"触发与范围确认""事实数据收集""数据采集""归档与通知"等流程协调类活动统一归为orchestrator R/A。这与五层模型Layer 1（日常流程）定义一致：orchestrator既是执行者也是最终负责人，因为这类活动本质是流程管理而非技术决策或质量验收。

### finding-02: 质量验收是reviewer的专有A域

跨所有治理流程，"质量验收""质量门禁""审批（常规）"类活动的A角色统一是reviewer。这确立了reviewer作为项目质量门禁唯一最终负责人的地位，5个指令集+数据安全RACI共69行RACI中无一例外。

### finding-03: co-founder审批严格限定在重大/敏感场景

co-founder作为A角色仅出现在：重大复盘审批、重大架构调整审批、重大数据出境审批、紧急/--no-verify提交审批、L4核心数据操作审批、安全规则变更审批等6类场景。所有常规审批均不经过co-founder，遵循层级审批原则。

### finding-04: 技术决策类活动由architect作为A

涉及方案设计、源文件/架构分析、根因分析、加密/脱敏方案设计等技术决策类活动，A角色是architect。R角色可能是orchestrator（做初步分析）或architect本人（技术评审）。

## 三、可复用模式（Patterns）

### pattern-raci-apply-workflow: RACI矩阵批量应用工作流

**场景**：需要在多个治理流程文档中批量新增RACI责任矩阵。

**步骤**：
1. **准备阶段**：确定角色列表（遵循角色最小化原则，不新增角色）→ 建立RACI格式模板 → 准备A唯一性校验脚本
2. **单文档应用**：逐文档添加RACI矩阵 → 每完成一个文档立即运行校验脚本 → 修正格式问题
3. **汇总阶段**：创建跨流程RACI总览表 → 对比各文档A角色分配一致性 → 发现模型级矛盾
4. **模型修正**：修正总览中发现的模型问题 → 回溯更新单个文档
5. **关联同步**：扫描所有引用该模型的关联文档 → 同步修正
6. **全量验证**：所有文档运行A唯一性+链接有效性检查

**关键质量门**：
- 每个RACI行必须有且仅有一个 **A**
- A必须加粗标记
- 执行操作类活动R≠A
- 跨文档A角色分配一致（同类活动A角色相同）

### pattern-five-layer-approval: 五层审批模型R/A分配模式

| 层级 | R角色 | A角色 | 活动特征 |
|------|-------|-------|---------|
| L1 日常流程 | orchestrator | orchestrator | 触发、协调、采集、通知、归档 |
| L2 技术决策 | orchestrator/architect | architect | 方案设计、架构分析、根因分析 |
| L3 执行操作 | developer | reviewer | 代码实现、文件修改、提交、漏洞修复 |
| L4 质量门禁 | reviewer | reviewer | 质量验收、安全审计、常规审批 |
| L5 关键决策 | orchestrator | co-founder | 重大审批、架构变更、核心数据操作 |

**注意**：此版本为修正后的五层模型（原L3/L4顺序已调整，审批角色拆分为R/A双列）。
