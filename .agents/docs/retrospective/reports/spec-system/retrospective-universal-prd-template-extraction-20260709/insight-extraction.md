# 洞察与模式萃取

## 可复用方法论模式

### 模式1：第一性原理模板提炼法（L2，validation_count=1）
**场景**：当需要建立标准、模板、规范、流程时
**核心逻辑**：
1. 不要从零发明，找1-2个经过完整生命周期验证的高质量样本（必须是实际用过、迭代过、验证过的，不是凭空写的）
2. 逐要素追问"为什么存在？解决什么根本问题？没有它会导致什么风险？"
3. 严格区分"本质要素"（所有场景通用）与"样本特定要素"（仅适用于该样本所在场景）
4. 剥离所有特定要素，只保留本质要素作为核心骨架
5. 补充本质要素中缺失但逻辑上必须有的部分（通过自验证发现）
**效果**：提炼出的模板不是理论上的"完美模板"，而是经过实战验证、解决真实问题的实用模板，避免"纸上谈兵"

### 模式2：Dogfooding自验证法（L2，validation_count=1）
**场景**：当产出方法论、模板、工具、规范时
**核心逻辑**：
1. 产出物第一版完成后，必须立即在当前项目中使用自己产出的东西
2. 记录使用过程中遇到的所有问题：哪里不好用、哪里缺东西、哪里有歧义、哪里不符合实际工作流
3. 所有发现的问题必须在发布前修复，迭代优化产出物
4. 在复盘中记录自验证过程和发现的问题
**为什么有效**：
- 静态审视永远发现不了实际使用中的问题（如本次发现的Version History缺失、字段填写时机不明确等，都是实际用的时候才会想到）
- 自己用着别扭的东西，给别人用肯定也不好用
**效果**：本次通过Dogfooding发现15个问题，模板从v1.0优化到v1.1，实用性提升一个量级

### 模式3：规范双轨制决策框架（L1，validation_count=1）
**场景**：当面临"是否要统一为单一格式/方法/工具"的决策时
**核心逻辑**：
1. 不要强行追求"一刀切"的统一——不同场景适用不同工具，强行统一只会导致效率降低或形式主义
2. 明确每种格式/方法/工具的适用场景边界：在什么情况下用A，什么情况下用B
3. 提供简单易用的决策树/判断标准，帮助用户快速选对
4. 说明混合使用场景：大框架下如何组合使用不同工具
**反模式**：为了"规范统一"而强行废弃已经验证好用的工具，强制所有人用单一格式

### 模式4：可用规范四要素模型（L1，validation_count=1）
**场景**：当需要编写/发布新的规范、标准、流程时
**核心逻辑**：一个真正能用的规范必须包含四个部分，缺一不可：
1. **模板/骨架**：长什么样，有哪些必填部分（提供可复制的空白模板）
2. **写作/使用指南**：每个部分怎么填、核心目的是什么、什么是好的写法、什么是不好的写法（正反示例）
3. **选择决策框架**：什么时候用这个规范、什么时候用别的规范、如何判断（避免滥用或用错）
4. **最佳实践与质量门禁**：常见陷阱、自检清单、检查标准（确保交付质量）
5. **体系整合**：必须融入现有规范体系，建立双向导航链接，不能成为"孤儿文档"
**反模式**：只发一个模板文件，没有使用说明、没有场景边界、没有质量标准，结果就是每个人写出来的都不一样，规范形同虚设

## 系统性经验总结

1. **Spec质量与项目成功率正相关**：目标清晰、边界明确、验收标准可衡量的Spec，执行过程几乎不会走偏，返工率极低，本次项目9个任务一次全部通过验收，无重大返工
2. **非功能性需求决定交付质量**：本次明确提出了"通用性、实用性、简洁性、兼容性、可演进性、可追溯性"6个NFR，确保了模板不仅"有"，而且"好用"
3. **自描述/自指涉是规范成熟的标志**：规范制定过程本身必须遵循自己制定的规范（本次项目Spec就使用了正在提炼的PRD模板，自己约束自己）
4. **元数据必须区分生命周期阶段**：不是所有字段都是一开始就填的，必须明确"规划时填/执行中更新/完成后补充"的时机，否则元数据要么填不全要么成为负担
5. **显式化隐含假设**：Assumptions章节看似无用，实则是避免后期扯皮的关键——把大家默认但没说出来的假设摆到台面上，提前发现分歧

## 模式成熟度更新

本次项目验证并升级了以下现有模式：
- spec-writing-guide：从仅支持Change Spec升级为支持双格式（PRD+Change），覆盖全场景Spec编写需求
- Spec Mode PRD模板：从内置在系统提示中的隐性模板，升级为显性化、有完整指南、经过Dogfooding验证的显性模板
