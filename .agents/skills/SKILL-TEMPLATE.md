## 4. 方案选择决策

【要素2：Decision Tree】提供清晰的选型决策树，而非简单罗列方案。

```
需要执行<领域>操作？
├─ 在 Trae IDE 内临时操作、环境已就绪？ → 方案<N>（第X节）
├─ 需要独立运行/命令行/CI/dry-run预览？ → 方案<M>（第Y节）
└─ 首次使用未初始化？ → 先执行<初始化步骤>
```

**写操作（创建/编辑/删除）原则**：优先使用支持 dry-run 的方案预览内容，确认无误再正式执行。

> **为什么 dry-run 是最重要的安全防线？** 【要素4：Why-Explanation】写操作一旦执行就可能产生不可逆影响（对外可见、产生历史记录、影响他人）。dry-run 在完全不提交的情况下展示完整结果，是成本最低的防误操作手段。支持dry-run的方案应作为写操作首选；无内置dry-run的方案必须先向用户展示diff并获得明确确认。

## 9. 通用工具函数

【要素3补充：封装复用逻辑】在 browser_evaluate 或脚本中使用这些函数，减少重复代码和出错概率。

```javascript
// 示例：状态检测函数（多信号组合，不依赖单一选择器）
function checkStatus() {
  const signal1 = /* 检测方式1 */;
  const signal2 = /* 检测方式2 */;
  const signal3 = /* 检测方式3 */;
  // 多信号投票判断
  return {isReady: !!(signal1 || signal2 || signal3), /* 其他状态 */};
}

// 示例：幂等操作函数
function idempotentOperation(content) {
  // 先检查是否已存在
  if (/* 已存在条件 */) return {success: true, skipped: true};
  // 执行操作
  return {success: true, skipped: false};
}

// 示例：框架兼容的输入设置（必须触发事件让框架感知变更）
function setInputValue(selector, value) {
  const el = document.querySelector(selector);
  if (!el) return {success: false, error: 'element not found'};
  el.focus();
  el.value = value;
  el.dispatchEvent(new Event('input', {bubbles: true}));
  el.dispatchEvent(new Event('change', {bubbles: true}));
  return {success: true, length: value.length};
}
```

> **为什么需要多信号检测/工具函数封装？** 【要素4：Why-Explanation】页面结构/框架行为可能因版本/主题/插件而异，单一选择器/方式容易失效；重复代码容易出错。封装工具函数+多信号组合大幅提高鲁棒性。
