# 数据脱敏：可逆与严格脱敏技术（FPE/令牌化/删除抑制）

## 加密脱敏（Format-Preserving Encryption, FPE）

**技术原理**：使用分组密码算法在保持原始数据格式（长度、字符类型、结构）不变的前提下进行加密，需要时可使用密钥解密恢复原始值。

**适用场景**：
- 需要可逆脱敏的业务场景
- 生产环境数据脱敏后需回写原系统
- 支付卡行业（PCI DSS）数据保护
- 数据库字段级加密保护
- 数据在脱敏后仍需保持原格式通过校验

**适用数据类型**：信用卡号、社保号、账号、固定格式标识符等需要保持格式且需可逆的场景。

**实施方法**：
- 使用NIST标准的FF1/FF3-1算法
- 密钥管理遵循企业密钥管理规范，密钥与密文分离存储
- 每个数据域使用独立的密钥加密
- 严格控制解密权限，所有解密操作必须审计

**正例**：
```python
# 使用FF1算法进行格式保留加密示例（基于pycryptodome）
from Crypto.Cipher import AES
import ff3

# 信用卡号加密（保持16位数字格式）
key = b'0123456789abcdef0123456789abcdef'  # 32字节密钥
tweak = b'tweak_value'
cipher = ff3.FF3Cipher(key, tweak)

original_pan = "4111111111111111"
encrypted_pan = cipher.encrypt(original_pan)
# 输出示例："4111111111111111" → "8392047165382910"（保持16位数字，可通过Luhn校验）
decrypted_pan = cipher.decrypt(encrypted_pan)
# 解密后恢复原始卡号
```

**反例**：
```
❌ 使用AES-ECB直接加密（输出Base64，格式破坏，无法通过格式校验）
❌ 密钥硬编码在代码中（密钥泄露即所有数据泄露）
❌ 所有字段使用同一密钥加密（一个密钥泄露影响全部数据）
❌ 无审计日志记录解密操作（无法追溯数据泄露来源）
❌ FPE加密后的数据传出信任域（可逆，等同于明文外传）
```

**注意事项**：
- FPE为可逆脱敏，加密后数据密级等同原始数据，出境与第三方API传入规则按原始级别执行
- 必须配套严格的密钥管理体系（密钥生成、存储、轮换、销毁全生命周期管理）
- 解密权限最小化原则，仅授权极少数管理员在审批后可临时解密
- FPE不适用于AI场景下传入第三方API的数据保护（应使用不可逆脱敏）

## 令牌化（Tokenization）

**技术原理**：使用随机生成的无意义令牌（Token）替换原始敏感值，令牌与原始值的映射关系存储在独立的令牌服务（Token Vault）中，仅通过令牌服务可换回原始值。

**适用场景**：
- API密钥、访问令牌保护
- 信用卡号支付处理（PCI DSS合规）
- 跨系统数据交换时的引用标识符
- 微服务架构中敏感字段传递
- 需要最小化敏感数据触达范围的场景

**适用数据类型**：API密钥、信用卡号、支付令牌、会话ID、全局唯一标识符（GUID/UUID）。

**实施方法**：
- 令牌使用密码学安全随机数生成，与原始值无数学关联
- 令牌服务独立部署，与业务系统网络隔离
- 所有令牌化/去令牌化操作经过令牌服务，全程记录审计日志
- 令牌本身不携带任何原始值信息，无法从令牌反推原始值

**正例**：
```
原始API Key: sk-abcdef1234567890abcdef1234567890
令牌: tok_7f3d9a2b8c1e4f6a
（令牌映射关系存储在独立的令牌服务中，业务系统只接触令牌）

原始信用卡号：4111-1111-1111-1111
支付令牌：pm_8a2b9c4d7e1f3a5b
（支付网络可通过令牌服务映射回原始卡号完成扣款，商户系统仅存储令牌）
```

**反例**：
```
❌ 令牌由原始值哈希生成（彩虹表攻击可逆，不具备无关联性）
❌ 令牌映射表与业务数据存储在同一数据库（数据库被攻破即全部泄露）
❌ 令牌包含原始数据的部分特征（如前6位BIN码保留，缩小猜测范围）
❌ 无令牌有效期管理（永久有效令牌一旦泄露影响持久）
```

**注意事项**：
- 令牌与原始值无算法关联，仅通过令牌服务映射，令牌本身泄露无风险
- 令牌服务必须作为最高安全等级组件保护，纳入L4核心数据保护区
- 令牌不应流出信任域，跨域传输时需再次脱敏
- 令牌服务高可用部署，防止单点故障导致业务中断
- 定期轮换令牌，设置合理的令牌有效期

## 删除/抑制（Suppression）

**技术原理**：直接删除敏感字段或用固定标识（如`[REDACTED]`、`***`、空值）完全替换，是最严格的脱敏方式，彻底消除敏感信息。

**适用场景**：
- L4核心数据外传前的强制处理
- 非必要敏感字段的彻底移除
- 日志中敏感信息清除
- 超出使用目的的数据字段处理
- 其他脱敏技术无法满足保护要求的场景

**适用数据类型**：所有类型敏感数据，特别是禁止外传的核心字段。

**实施方法**：
- 整列删除：数据集中不必要的敏感字段直接删除整个列
- 记录删除：包含高度敏感信息的整条记录删除
- 单元格替换：个别敏感单元格用统一标记替换
- 日志过滤：日志输出前过滤掉敏感字段

**正例**：
```
日志输出前过滤密码字段：
原生日志：{"user": "zhangsan", "password": "my_secret_pwd123", "ip": "192.168.1.1"}
脱敏日志：{"user": "zhangsan", "password": "[REDACTED]", "ip": "192.168.1.1"}

L4数据外传前删除核心字段：
外传数据：删除"api_key"、"model_weight"、"source_code"字段，仅保留元数据
```

**反例**：
```
❌ 密码字段未删除仅做掩码（日志中不应出现密码，无论是否掩码）
❌ L4核心字段做掩码后外传（仍可能通过侧信道泄露，应删除）
❌ 删除字段但保留记录中的其他信息可与外部数据关联重识别
❌ 仅替换第一个敏感值出现位置，后续出现位置未处理
```

**注意事项**：
- 删除/抑制是最彻底的脱敏方式，但也是数据可用性损失最大的方式
- 删除前必须确认该字段确实不是业务必需，避免破坏数据完整性
- 日志敏感字段过滤必须在日志生成源头完成，避免中间环节泄露
- L4核心数据原则上通过删除处理，不使用掩码/泛化等部分脱敏方式外传
