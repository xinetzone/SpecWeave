# 数据脱敏：基础脱敏技术（掩码/替换/泛化/扰动）

## 掩码（Masking）

**技术原理**：保留数据的部分前缀和/或后缀，中间部分用固定掩码字符（如`*`、`#`、`X`）替换，使数据无法识别完整值但可辨认部分特征。

**适用场景**：
- 手机号、身份证号、银行卡号展示
- 邮箱地址部分展示
- 姓名、地址部分隐藏
- API密钥、Token前端展示

**适用数据类型**：固定格式的标识符（手机号、身份证、银行卡、账号）、联系方式、密钥类字符串。

**实施方法**：
- 固定长度数据：保留前N位和后M位，中间全部掩码
- 可变长度数据：保留前N位，后续固定长度掩码
- 统一使用`*`作为掩码字符，掩码长度应与原始字符长度一致以保持格式

**正例**：
```
手机号：138****1234（保留前3后4）
身份证：110***********1234（保留前3后4，出生日期全掩码）
银行卡：6222 **** **** 1234（保留前4后4，分组展示）
邮箱：z****@example.com（保留首字符）
姓名：张*（两字名）、张**（三字名）
API Key：sk-****abcd（保留前缀和后4位）
```

**反例**：
```
❌ 手机号：1381234****（保留前7位，可通过号段识别归属地+出生日期组合重识别）
❌ 身份证：11010119********（保留前6位地区码，可精确到区县）
❌ 姓名：张伟（未脱敏）、*伟（保留全名中间字，熟人可识别）
❌ 邮箱：zhangwei@****（域名掩码，仍可通过用户名识别）
```

**注意事项**：
- 掩码后剩余可见字符数不应超过可重识别阈值（手机号≤7位、身份证≤7位可见）
- 同一数据在不同场景应使用一致的掩码规则，避免通过掩码模式关联推断
- 掩码字符数量应与原始字符数量一致，避免泄露原始长度信息

## 替换（Substitution）

**技术原理**：使用虚构但符合数据格式规范、具有合理语义的假数据替换真实敏感数据，保持数据格式与业务逻辑一致性。

**适用场景**：
- 测试环境数据构建
- 数据分析与报表展示
- 产品演示与截图
- 模型训练数据去标识化
- PII字段假名化

**适用数据类型**：姓名、地址、公司名、邮箱、手机号、身份证号等有语义约束的结构化数据。

**实施方法**：
- 使用预构建的假数据集（如中文姓名库、地址库、手机号段库）
- 保持数据一致性：同一原始值在所有关联表中映射到同一替换值
- 替换值应符合业务规则（如手机号段合法、身份证校验位正确、邮箱格式有效）

**正例**：
```
原始姓名：王小明 → 替换为：张伟、李强、赵敏（中文常用姓名库随机选取）
原始手机号：13812345678 → 替换为：13987654321（合法号段随机生成）
原始地址：北京市海淀区中关村大街1号 → 替换为：上海市浦东新区张江高科技园区博云路2号
原始公司：字节跳动 → 替换为：上海云帆科技有限公司（虚构公司名）
原始邮箱：wangxiaoming@bytedance.com → 替换为：zhangwei@yunfantech.com
```

**反例**：
```
❌ 原始"王小明"→"张三"，在订单表中显示"张三"，在用户表中显示"李四"（一致性破坏，关联查询失败）
❌ 原始"13812345678"→"12345678901"（手机号段不存在，格式校验失败）
❌ 原始"北京市海淀区"→"***"（完全无意义替换，数据可用性丧失）
❌ 使用名人姓名替换（如"马云"、"马化腾"），容易造成误解或关联
```

**注意事项**：
- 必须维护原始值与替换值的映射表，映射表按L4核心数据保护
- 生产环境严禁使用可通过映射表可逆的替换方案外传
- 替换后的数据集应通过格式校验、业务规则校验
- 同一数据集内替换值应唯一，避免多个原始值映射到同一替换值造成冲突

## 泛化（Generalization）

**技术原理**：通过降低数据精度、将具体值泛化为区间或类别，在保留整体统计分布特性的同时隐藏具体个体信息。

**适用场景**：
- 年龄、收入等数值型数据分析
- 地理位置信息展示
- 时间戳精度降低
- 职业、职位等分类信息粗化
- k-匿名隐私保护模型实施

**适用数据类型**：数值型数据（年龄、收入、金额）、时间日期、地理位置、分类/层级数据。

**实施方法**：
- 数值型：将精确值映射到预定义区间
- 地理位置：将精确坐标→区县→城市→省份→大区逐级泛化
- 时间：精确时间→日期→月份→季度→年份逐级泛化
- 分类：将细分类别合并为粗粒度类别

**正例**：
```
年龄：28岁 → 25-30岁（5岁区间）
收入：25000元/月 → 20000-30000元/月（万元区间）
位置：北京市海淀区中关村街道（GPS: 39.9847, 116.3054）→ 北京市 → 中国华北
时间：2024-06-15 14:32:18 → 2024年6月 → 2024年Q2
职业：高级算法工程师 → 技术研发人员
消费金额：128.50元 → 100-200元
```

**反例**：
```
❌ 年龄：28岁 → 28-29岁（区间过小，仍可精确识别）
❌ 位置：GPS坐标泛化到街道级（精度过高，结合其他信息可定位到个人）
❌ 收入：精确到元直接展示（精度过高，可用于唯一识别）
❌ 时间：日志时间戳保留到毫秒（结合其他日志可精确追踪用户行为轨迹）
```

**注意事项**：
- 泛化粒度需根据k-匿名要求确定，确保每个泛化组内至少包含k条记录
- 泛化层次应一致，同一字段在同一数据集中使用相同的泛化层级
- 泛化区间划分应考虑数据分布，避免某一区间数据量过少
- 过度泛化会导致数据可用性丧失，需在隐私保护与可用性之间平衡

## 扰动（Perturbation/Noise Addition）

**技术原理**：在原始数值上添加服从特定统计分布的随机噪声，改变精确值但保持数据集的整体统计特性（均值、方差、分布形态）。

**适用场景**：
- 统计数据分析与报表
- 机器学习模型训练数据保护
- 聚合查询结果隐私保护
- 差分隐私实施
- 数值型敏感字段保护

**适用数据类型**：连续数值型数据（收入、金额、年龄、评分、传感器数据、统计指标）。

**实施方法**：
- 拉普拉斯机制：添加拉普拉斯分布噪声，适用于数值查询
- 指数机制：适用于分类数据选择
- 噪声强度应与隐私预算ε成反比（ε越小噪声越大，隐私保护越强）
- 保证添加噪声后的数据仍在合理业务范围内（如年龄不出现负数或>150）

**正例**：
```python
import numpy as np

def add_laplace_noise(value, epsilon, sensitivity=1.0):
    """
    拉普拉斯噪声添加（差分隐私基础实现）
    value: 原始值
    epsilon: 隐私预算，越小隐私保护越强（建议0.1-1.0）
    sensitivity: 查询敏感度
    """
    scale = sensitivity / epsilon
    noise = np.random.laplace(loc=0, scale=scale)
    result = value + noise
    return max(0, min(150, round(result)))

original_age = 28
noised_age = add_laplace_noise(original_age, epsilon=0.5)
# 输出示例：28 → 26 或 31 或 27（在±5范围内波动，统计均值接近28）
```

**反例**：
```
❌ 所有值都添加固定偏移+5（固定偏移不改变分布，可通过统计分析还原）
❌ 噪声幅度过大（年龄28±30，结果出现-2或58等不合理值，数据完全不可用）
❌ 对分类数据（如性别、职业）添加数值噪声（无意义，破坏数据语义）
❌ 同一数据多次查询添加相同噪声（失去随机化保护意义）
```

**注意事项**：
- 扰动仅适用于数值型数据，不适用于标识符、分类数据
- 需评估数据可用性扰动后不应造成统计结论偏差超过5%
- 差分隐私场景下需合理设置隐私预算ε，防止预算耗尽
- 多次查询累加噪声会导致可用性下降，需设置查询次数阈值
