# 消息传递协议

本协议定义了智能体之间进行消息传递时的标准格式、消息类型、优先级规则以及使用约束，确保多智能体协作过程中的沟通高效、有序且可追溯。

## 消息格式

消息传递采用 YAML 格式，所有字段使用字符串类型，便于序列化与解析。

```yaml
message:
  from: "orchestrator"
  to: "developer"
  type: "task_assignment"
  content: "消息内容"
  priority: "high"
  timestamp: "2026-06-23T10:00:00Z"
```

## 消息类型枚举

| 类型 | 说明 | 典型场景 |
|---|---|---|
| task_assignment | 任务分配 | orchestrator 向下属智能体分配新任务 |
| status_update | 状态更新 | 智能体向 orchestrator 汇报任务进度 |
| question | 询问 | 智能体向其他智能体提出问题或澄清需求 |
| review_request | 审查请求 | 开发智能体向审查智能体请求代码审查 |
| handoff | 任务交接 | 智能体将未完成任务交接给其他智能体 |
| conflict_report | 冲突报告 | 智能体报告职责冲突、技术分歧或资源竞争 |

## 优先级枚举

| 优先级 | 说明 | 使用场景 |
|---|---|---|
| high | 高优先级 | 阻塞性问题、紧急任务分配、冲突报告、关键路径任务 |
| medium | 中优先级 | 常规任务沟通、状态更新、审查请求 |
| low | 低优先级 | 非紧急通知、信息同步、可选事项 |

## 使用约束

1. **格式合规**：所有消息必须遵循上述 YAML 格式，包含 `from`、`to`、`type`、`content`、`priority`、`timestamp` 全部字段，缺一不可。
2. **类型合规**：`type` 字段必须使用消息类型枚举中定义的类型之一，禁止使用未定义的自定义类型。
3. **优先级合理**：`high` 优先级仅用于阻塞性问题、紧急任务或冲突报告；常规沟通使用 `medium`；非紧急通知使用 `low`。滥用高优先级将导致告警疲劳。
4. **发送方与接收方明确**：`from` 与 `to` 字段必须为有效的智能体标识，禁止使用通配符或广播地址。
5. **内容结构化**：`content` 字段应使用结构化文本，复杂信息应使用列表或表格格式，避免冗长叙述。
6. **时间戳统一**：`timestamp` 必须使用 ISO 8601 格式（如 `2026-06-23T10:00:00Z`），时区统一为 UTC。
7. **单一主题原则**：每条消息应聚焦单一主题，复合需求应拆分为多条消息分别发送。
8. **消息频率控制**：单次任务中向同一接收方发送消息不应超过 10 条，频繁沟通应考虑合并消息或提升为同步会议。
9. **接收确认**：`high` 优先级消息要求接收方在收到后明确确认；`medium` 与 `low` 优先级消息无需显式确认。
10. **消息留存**：所有消息记录应留存备查，便于协作过程追溯与复盘。
11. **禁止越级沟通**：下属智能体之间的直接沟通应通知 orchestrator，避免信息孤岛；紧急情况下可直接沟通但需事后同步。
12. **使用 send_message 工具**：所有消息发送必须通过 `send_message` 工具（见 communication.md），禁止使用非标准方式传递消息。
