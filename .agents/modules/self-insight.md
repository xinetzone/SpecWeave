# Self-Insight（自我洞察）

## Description
构建数据分析与问题诊断模块，实现系统状态的实时监控与异常预警。属于感知层模块，负责采集系统指标、日志与事件，进行趋势与根因分析，并基于阈值与机器学习识别异常，多渠道通知。

## 所属层级
感知层（Perception Layer）

## 技术架构
数据采集层（系统指标、日志、事件）+ 分析引擎层（趋势分析、根因分析）+ 异常检测层（基于阈值与机器学习）+ 预警通知层（多渠道通知）。

## 关键实现步骤
| 步骤 | 说明 |
|------|------|
| 构建状态监控管道 | 采集系统指标、日志与事件 |
| 实现数据分析引擎 | 支持趋势分析与根因分析 |
| 开发异常检测算法 | 基于阈值与机器学习识别异常 |
| 建立预警通知机制 | 多渠道（邮件、IM、Webhook）通知 |
| 构建可视化仪表盘 | 实时展示系统状态与告警 |

## 资源需求
1 名架构师 + 1 名开发者，2 个月

## 时间节点
M3-M5

## 预期成果指标
| 指标 | 目标 |
|------|------|
| 异常检出率 | ≥ 95% |
| 预警准确率 | ≥ 90% |
| 告警延迟 | ≤ 1min |
| 误报率 | ≤ 5% |

## 交互方式
- 上游：采集自我迭代、自我验证等模块的运行指标与日志
- 下游：向自我进化模块输出优化机会，向自我复盘模块输出异常事件
- 协作：向自我管理模块上报系统状态以支持调度决策

## 能力范围
- 系统指标、日志与事件的采集
- 趋势分析与根因分析
- 基于阈值与机器学习的异常检测
- 多渠道预警通知（邮件、IM、Webhook）
- 系统状态与告警的可视化展示

## 约束条件
- 不负责异常的修复执行（归自我迭代）
- 不负责优化方案的生成（归自我进化）
- 异常检测算法需控制误报率在 5% 以内
- 预警通知不得造成通知风暴，需具备去重与分级机制

## 定位自查子模块

### 触发条件
在以下场景中触发定位术语审查：
- 产品/项目定位中使用了外部来源的核心术语（如赛事标签、平台流行词、行业热词）
- 定位表述经历了从"借用标签"到"问题定义"的漂移
- 对外文档（报名帖、宣传文案、答辩材料）即将发布前

### 审查三步法

| 步骤 | 检查项 | 判定标准 |
|------|--------|---------|
| 1. 来源溯源 | 核心定位术语来自问题本身还是外部？ | 若术语来自赛事/平台/流行词而非问题域，标记为"借用标签" |
| 2. 范畴校验 | 术语的范畴是否覆盖产品的完整能力域？ | 若术语暗示的范畴（如"写代码"）窄于实际能力（如"文档+架构+研究+编码"），存在定位漂移 |
| 3. 生命周期对比 | 术语的生命周期是否长于问题域？ | 若术语为热潮词（生命周期 < 2 年），而问题域长期存在，存在时效性风险 |

### 修正策略

当任一审查步骤失败时，执行以下修正：
1. **主定位回归问题域**：用问题本身定义产品（如"AI 智能体协作的工程化方法"），而非借用外部标签
2. **外部术语降级为佐证**：将原借用标签作为背景切面，表述为"这些标签也在描述同一个问题"
3. **保留少量引用作为入口**：在问题引入段落保留 3-5 处外部术语引用，作为读者认知切口，但全文其他位置统一使用问题域术语

### 输出产物
- 定位术语审查报告（包含来源溯源表、术语范畴对比、修正建议）
- 定位修正后的文档 diff 清单
