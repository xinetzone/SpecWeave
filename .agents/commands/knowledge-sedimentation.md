# 知识沉淀指令集

## 触发条件

- 完成一个知识密集型任务后（如Wiki生成、专题研究、技术调研）
- 同类任务已完成≥2次，需要提炼可复用方法论
- 项目/团队需要建立知识库或最佳实践库
- 用户明确请求"沉淀知识"、"总结方法论"、"生成Wiki"
- 七概念方法论E阶段（萃取）需要结构化产出时

## 输入规范

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| topic | string | 是 | 知识沉淀的主题（如"勾股定理"、"API设计规范"） |
| source_type | string | 是 | 来源类型：`from_scratch`（从零创建）/ `from_existing`（基于已有材料） |
| output_type | string | 否 | 产出类型：`wiki`（百科式Wiki）/ `guide`（指南手册）/ `pattern`（模式文档）/ `checklist`（检查清单），默认`wiki` |
| domain | string | 否 | 知识领域：`technical`/`business`/`mathematics`/`methodology`/`other` |
| depth | string | 否 | 深度：`quick`（快速版，3-5章节）/ `standard`（标准版，7-10章节）/ `deep`（深度版，10+章节），默认`standard` |
| with_validation | boolean | 否 | 是否启用对抗审查V阶段，默认true |
| target_path | string | 否 | 目标输出路径，默认根据内容敏感度自动分配 |

## RACI责任分配矩阵

**RACI模型说明**：
- **R** = 负责执行（Responsible）：实际完成工作的角色
- **A** = 最终审批（Accountable）：对结果负最终责任，拥有最终决策权，每项活动有且仅有一个A
- **C** = 需咨询（Consulted）：决策前需征求意见、提供专业输入的角色，双向沟通
- **I** = 需知会（Informed）：决策后需告知进展与结果的角色，单向沟通

| 知识沉淀核心活动 | orchestrator | architect | reviewer | domain-expert | co-founder |
|:---|:---:|:---:|:---:|:---:|:---:|
| 触发沉淀与主题定义 | **R/A** | C | C | C | I |
| 范围定义与边界设定 | **R/A** | C | C | C | I |
| 事实采集与素材整理（S1） | **R** | C | C | C | I |
| 洞察提炼与本质抽象（S2） | R | **A** | C | C | I |
| 结构化呈现与内容组织（S3） | **R** | C | C | C | I |
| 反模式与边界提炼（S4） | R | C | **A** | C | I |
| 对抗审查与验证（S5） | C | C | **R/A** | C | I |
| 索引入库与交叉引用（S6） | **R/A** | I | C | I | I |
| 知识沉淀质量验收 | C | C | **R/A** | C | I |
| 重大知识库建设审批 | R | C | C | C | **A** |

### 审批权限边界

- **常规知识沉淀（单主题/单模块）**：reviewer审批质量，orchestrator负责执行
- **方法论级沉淀（跨领域通用）**：architect审批抽象合理性，reviewer负责质量验收
- **重大知识库建设（项目级/团队级）**：co-founder最终审批，确认与整体知识体系一致
- **领域知识准确性争议**：domain-expert作为专业裁判，reviewer负责流程合规
- **沉淀成果入库**：orchestrator负责索引更新和交叉引用修复

## 核心原则

### 可迁移性优先原则

知识沉淀的黄金标准是**换一个人、换一个场景还能用**。判断沉淀质量的三个问题：
1. 把这份文档给一个不了解背景的人看，他能看懂吗？
2. 把这份方法用在另一个类似项目中，能直接套用吗？
3. 把这份知识放在另一个领域，核心逻辑还成立吗？

如果三个问题的答案都是"能"，说明沉淀质量高；如果有"不能"，说明抽象层级不够或边界不清晰。

### 结构化表达原则

知识不是信息的堆砌，而是**结构化的信息网络**。好的知识沉淀必须具备：
- **层级清晰**：从概述→原理→方法→应用→资源，层层递进
- **模块独立**：每个章节可独立阅读，也可串联成体系
- **交叉引用**：相关知识点之间有明确的链接关系
- **快速导航**：有目录、索引、速查表等入口点

### 反模式对等原则

只告诉"怎么做"不告诉"什么情况下不该做"是危险的。每一份知识沉淀都必须对等包含：
- **正面做法**：应该怎么做，最佳实践是什么
- **反面警告**：什么情况下不要用，常见错误做法是什么
- **边界条件**：适用范围在哪里，超出范围会怎样

反模式不是"做得差的例子"，而是**模式的适用边界**——明确边界比给出答案更重要。

### 多视角验证原则

单一视角的知识容易有盲区。高质量的知识沉淀必须经过至少三个视角的验证：
1. **准确性视角**：事实是否正确，数据是否可靠
2. **完整性视角**：有没有遗漏重要方面，有没有盲区
3. **可用性视角**：别人能不能看懂、能不能直接用

这也是V阶段（对抗审查）的核心价值——通过不同视角的攻击来补全盲区。

## 执行步骤

### 步骤 0：范围定义与边界设定

在开始沉淀之前，先明确"沉淀什么、不沉淀什么"：

- **核心主题**：用一句话定义本次沉淀的核心主题
- **目标读者**：这份知识是给谁看的？（初学者/进阶者/专家）
- **深度范围**：讲到什么程度为止？（入门概念/完整方法/前沿研究）
- **排除边界**：明确哪些内容不在本次沉淀范围内

**产出**：主题定义 + 范围说明 + 目录大纲

---

### 步骤 1：事实采集与素材整理（S1）

**目标**：收集足够多的高质量原始素材，为后续提炼打基础。

**采集渠道**：
- 权威资料：教科书、学术论文、官方文档
- 历史案例：过往项目中的相关经验和教训
- 多个来源：至少3个独立来源交叉验证
- 正反两面：不仅收集支持性材料，也收集质疑和反例

**整理方法**：
- 按主题分类归档素材
- 标注来源和可信度
- 识别事实与观点的区别
- 排除主观判断，保留客观描述

**质量门G1**：
- [ ] 事实清单中无因果推断词（"因为"、"所以"、"导致"）
- [ ] 每条事实都有可追溯的来源
- [ ] 至少覆盖3个独立信息来源
- [ ] 包含正反两面的材料

---

### 步骤 2：洞察提炼与本质抽象（S2）

**目标**：从零散的事实中提炼出核心规律和本质理解。

**提炼方法**：
- **连续追问**：不断问"这意味着什么？""背后的本质是什么？"
- **共性识别**：从多个案例/事实中找出共同的结构和规律
- **去场景化**：剥离具体项目/技术/人物细节，保留核心逻辑
- **反常识发现**：找出与直觉相反但有事实支撑的洞察

**洞察四元组**（每条洞察必须包含）：
1. **陈述**：用一句话概括洞察
2. **证据**：支撑这个洞察的事实依据（引用S1中的事实编号）
3. **反常识**：这个洞察挑战了什么常见误解？
4. **行动**：基于这个洞察，我们应该怎么做？

**质量门G2**：
- [ ] 产出≥3条核心洞察
- [ ] 每条洞察都包含四元组（陈述/证据/反常识/行动）
- [ ] 洞察有事实支撑，不是主观臆断
- [ ] 至少有1条洞察是反常识的（挑战直觉）

---

### 步骤 3：结构化呈现与内容组织（S3）

**目标**：将洞察和素材组织成结构化、易读、易用的知识体系。

**Wiki标准结构**（可根据主题调整）：
```
README.md（入口索引 + 快速回顾 + 学习路径）
├── 00-overview.md（概述：定义、重要性、适用范围）
├── 02-history.md（历史/背景/演进）
├── 03-core-concepts.md（核心概念/原理）
├── 04-methods.md（方法/实践/步骤）
├── 05-applications.md（应用场景/案例）
├── 06-faqs.md（常见问题/误解）
└── 07-resources.md（参考资料/延伸阅读）
```

**组织原则**：
- **由浅入深**：从概述→原理→方法→应用→进阶，符合学习曲线
- **模块独立**：每章可独立阅读，不依赖前后章节
- **前后呼应**：章节间有交叉引用，形成知识网络
- **入口清晰**：README提供导航、速查表、学习路径建议

**质量门G3**：
- [ ] 结构完整，覆盖从入门到进阶的完整链条
- [ ] 每个章节有明确的主题和读者定位
- [ ] 有清晰的导航和学习路径指引
- [ ] 交叉引用合理，知识网络连通

---

### 步骤 4：反模式与边界提炼（S4）

**目标**：明确知识的适用边界和常见误用，防止被错误使用。

**反模式挖掘方法**：
- **失败案例分析**：从失败案例中找出"为什么用错了"
- **边界条件思考**：在什么条件下这个知识会失效？
- **易混淆点识别**：哪些概念容易被搞混？区别是什么？
- **极端场景测试**：在极端条件下（时间极紧/资源极少）还成立吗？

**反模式标准格式**：
```
❌ 反模式名称：[简短描述]
   场景：在什么情况下容易犯这个错误？
   后果：会导致什么负面结果？
   识别：怎么判断自己正在犯这个错误？
   纠正：正确的做法是什么？
```

**质量门G4**：
- [ ] 至少提炼3个反模式
- [ ] 每个反模式都有明确的场景、后果和纠正方法
- [ ] 明确标注适用边界和前置条件
- [ ] 包含易混淆概念辨析

---

### 步骤 5：对抗审查与验证（S5）

**目标**：通过多视角审查，发现盲区、修正错误、提升质量。

**审查视角**：
1. **准确性视角**（事实核查）
   - 关键事实是否准确？
   - 数据来源是否可靠？
   - 有没有引用错误？

2. **完整性视角**（盲区扫描）
   - 有没有遗漏重要方面？
   - 反对意见和反例有没有呈现？
   - 不同学派/观点有没有平衡？

3. **可用性视角**（用户模拟）
   - 目标读者能看懂吗？
   - 结构清晰吗？好找内容吗？
   - 看完能直接用吗？

**审查流程**：
- 每个视角至少提出3条具体意见
- 对每条意见给出处理：采纳/部分采纳/不采纳（附理由）
- 统计采纳率（建议≥40%采纳率，否则审查流于形式）

**质量门G5**：
- [ ] 经过≥3个视角的审查
- [ ] 每条审查意见都有明确回应
- [ ] 至少采纳2条实质性修改建议
- [ ] 关键事实经过双重来源验证

---

### 步骤 6：索引入库与交叉引用（S6）

**目标**：让沉淀的知识可被发现、可被检索、可被复用。

**入库动作**：
- 存入正确的目录（按知识库分类体系）
- 更新所在目录的README索引
- 添加到全局知识导航
- 建立与相关知识的交叉引用

**可发现性优化**：
- 标题包含核心关键词
- 有明确的标签和分类
- 开头有摘要/快速回顾
- 提供不同深度的阅读路径

**质量门G6**：
- [ ] 存入正确目录，分类合理
- [ ] 所在目录索引已更新
- [ ] 至少添加3个交叉引用
- [ ] 链接检查通过，无断链

## 常见反模式

### ❌ 反模式1：信息堆砌症

- **场景**：收集了大量素材后，直接按时间/来源顺序罗列，没有提炼和结构化
- **后果**：读者找不到重点，看完记不住什么，知识没有转化为可复用的能力
- **识别**：章节标题是"资料1"、"资料2"，或直接复制粘贴原文，没有自己的组织逻辑
- **纠正**：先提炼核心洞察，再按逻辑结构组织素材；素材是论据，不是主体

### ❌ 反模式2：只有结论没有过程

- **场景**：只给出最终结论和方法，不说明为什么这样做、怎么推导出来的
- **后果**：读者只能机械套用，遇到边界情况就不会变通；知其然不知其所以然
- **识别**：全是"应该怎么做"，没有"为什么这么做"、"基于什么原理"
- **纠正**：每个结论都要有推导过程或事实依据；讲清楚"是什么→为什么→怎么做"的完整链条

### ❌ 反模式3：完美主义拖延

- **场景**：总想收集到"所有"信息、做到"完美"才开始写，结果一直拖延
- **后果**：知识永远停留在大脑或收藏夹里，没有沉淀下来；越拖越不想做
- **识别**：口头禅是"等我研究透了再说"、"还不够全面"；行动为0但焦虑感拉满
- **纠正**：先出最小可用版本（MVP），再迭代完善；完成比完美重要，80分的产出远好过0分的完美计划

### ❌ 反模式4：过度抽象空泛化

- **场景**：为了追求"可迁移性"，把知识抽象到正确的废话层面
- **后果**：听起来很有道理，但实际用不上；"要注意质量"、"要有创新思维"这类话谁都会说
- **识别**：全是正确的废话，没有具体步骤、没有判断标准、没有反例
- **纠正**：抽象层级要适配使用场景；原则层面要有具体方法支撑，方法层面要有具体例子说明

### ❌ 反模式5：闭门造车不验证

- **场景**：自己觉得写得很好，但不给别人看，也不实际用一下验证效果
- **后果**：自嗨式沉淀，自己觉得逻辑完美，但别人看不懂、用不了，盲区巨大
- **识别**：沉淀完成后只有自己看过，没有经过任何外部反馈
- **纠正**：必须经过对抗审查或至少找一个目标读者试用；根据反馈修正盲区

## 约束条件

### 禁止事项

- ❌ **禁止为了凑内容而注水**：宁少毋滥，每一条内容都要有价值
- ❌ **禁止只有正面没有反面**：反模式和边界条件是知识的必要组成部分
- ❌ **禁止主观判断冒充事实**：观点和事实必须明确区分
- ❌ **禁止单案例过度泛化**：只有1个案例支撑的只能标记为"待验证"
- ❌ **禁止抄袭不注明来源**：引用必须标注出处，尊重知识产权

### 适用边界

- 本指令集适用于**显性知识**的沉淀（可文字化、结构化的知识）
- 不适用于**隐性知识**的沉淀（技艺、直觉、经验手感等需通过实践习得的）
- 单主题沉淀建议控制在7-15个章节，过大会导致信息过载
- 对抗审查需要独立视角，自己审查自己效果有限，条件允许时请他人审查

## 成熟度评估

| 等级 | 标志 | 验证标准 |
|------|------|---------|
| L1-draft（草稿） | 单案例支撑，完成六步流程 | 1个案例，结构化完成，未经过跨场景验证 |
| L2-validated（已验证） | 多案例支撑，跨场景可用 | ≥2个独立案例验证，反模式完整，边界清晰 |
| L3-mature（成熟） | 跨领域复用，工具辅助 | 已在≥3个不同领域应用，有配套工具/模板 |
| L4-optimized（优化） | 体系化集成，自动化支持 | 纳入整体知识体系，有自动化索引和推荐 |
