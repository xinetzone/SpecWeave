# 对抗性审查指令集

## 触发条件

- 知识档案构建、研究报告撰写、资料整合时，需要确保来源可靠、论据可信
- 代码审查、安全审计、上线前质量门禁时，需要从攻击者视角全面排查问题
- 方案评审、架构评审时，需要验证方案的稳健性和边界覆盖
- 需要防御认知偏差对结论的影响（确认偏差、权威崇拜、幸存者偏差等）
- 多源信息整合需要系统性验证机制，而非仅凭印象判断
- 复盘或洞察过程中识别到"信息质量存疑"或"审查不充分"的问题
- 知识库构建完成后，需要对知识库自身进行自举验证（用对抗性审查方法审查知识库本身）

### 场景速查矩阵

> 不确定使用哪个场景时，对照下表快速决策：

| 常见场景 | 推荐 scenario | 推荐 scope | 推荐 depth | 说明 |
|----------|:---:|:---:|:---:|------|
| 知识档案/研究报告构建 | `knowledge-research` | `full` | `deep` | 启用七模块协议，来源三级分类+可信度四级评分 |
| 单一来源可信度验证 | `knowledge-research` | `full` | `quick` | 仅执行五维验证，不做完整的来源体系构建 |
| 代码审查/PR Review | `code-review` | `security` | `standard` | 部署安全/性能/边界/时序四个攻击者角色 |
| 安全审计 | `code-review` | `security` | `deep` | 重点部署安全攻击者，增加渗透测试视角 |
| 架构方案评审 | `architecture-review` | `boundary` | `standard` | 重点验证边界条件和组件交互的异常路径 |
| 上线前质量门禁 | `code-review` | `full` | `quick` | 快速全维度扫描，识别高风险问题后升级为 deep |
| 复盘/洞察中信息质量存疑 | `knowledge-research` | `full` | `quick` | 快速验证复盘引用的关键数据来源 |
| 知识库自举验证 | `knowledge-research` | `full` | `standard` | 用对抗性审查方法审查知识库自身，含元审查 |

## 输入规范

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| subject | string | 是 | 待审查对象：知识资料/代码/方案/文档 |
| scenario | string | 是 | 审查场景：`knowledge-research`（知识研究）/ `code-review`（代码审查）/ `architecture-review`（架构评审） |
| scope | string | 否 | 审查范围：`full`（全面）/ `security`（安全）/ `performance`（性能）/ `boundary`（边界）/ `timing`（时序） |
| depth | string | 否 | 审查深度：`quick`（10分钟快速检查）/ `standard`（标准流程）/ `deep`（深度审查），默认 `standard` |
| attacker_roles | list | 否 | 参与的攻击者角色列表：`security`/`performance`/`boundary`/`timing`，默认全部四个 |
| time_budget | string | 否 | 时间预算，用于控制审查深度和范围 |

## RACI责任分配矩阵

**RACI模型说明**：
- **R** = 负责执行（Responsible）：实际完成工作的角色
- **A** = 最终审批（Accountable）：对结果负最终责任，拥有最终决策权，每项活动有且仅有一个A
- **C** = 需咨询（Consulted）：决策前需征求意见、提供专业输入的角色，双向沟通
- **I** = 需知会（Informed）：决策后需告知进展与结果的角色，单向沟通

| 对抗性审查核心活动 | orchestrator | architect | developer | reviewer | tester | co-founder |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|
| 触发分析与场景判定 | **R/A** | C | C | C | C | I |
| 审查范围与深度确认 | C | **R/A** | C | C | I | I |
| 来源分级与五维验证（知识研究场景） | I | **A** | R | C | I | I |
| 多Agent攻击者部署（代码审查场景） | I | C | **R/A** | C | R | I |
| 攻击者问题清单汇总与交叉验证 | I | C | R | **R/A** | C | I |
| 可信度评分与异常标记 | I | C | R | **R/A** | I | I |
| 审查报告归档与通知 | **R/A** | I | I | C | I | I |
| 审查质量验收（元审查/自举验证） | C | C | I | **R/A** | I | I |
| 重大安全/架构问题审批 | R | C | I | C | I | **A** |

### 审批权限边界

- **知识研究场景**：reviewer负责可信度评分审批和异常标记核准，architect负责技术领域的事实判断
- **代码审查场景**：developer负责部署攻击者Agent并整合问题清单，reviewer负责问题分级和上线决策
- **重大安全漏洞/架构缺陷**：涉及数据泄露、资金损失、系统崩溃的P0问题，co-founder最终审批
- **审查质量验收**：reviewer对审查本身的质量负责——确保审查不是"表演式审查"，攻击者真正找到了问题
- **自举验证**：对知识库自身的对抗性审查，由reviewer负责执行和质量验收

## 核心原则

### 对抗性审查的定义

对抗性审查是一套基于证伪主义和认知偏差防御的质量保障方法论。其核心哲学是：**与其试图"证明正确"，不如主动"寻找问题"**。通过系统性地从攻击者视角审视对象，发现正常路径难以发现的缺陷、漏洞和偏差。

### 核心特征

- **证伪导向**：不追求"证明正确"，而是主动寻找反证和问题。一个被充分攻击后仍然站得住脚的结论，比一个从未被挑战的结论更可信
- **多角色并发**：不同攻击者角色（安全/性能/边界/时序）独立审查，互不干扰，避免单一视角的盲区
- **质量内建**：审查不是事后质检，而是嵌入工作流每个环节的质量保障机制
- **偏差防御**：系统性识别和防御确认偏差、权威崇拜、幸存者偏差等认知偏差对审查结论的影响
- **可审计性**：所有验证过程必须可追溯，第三方根据验证日志应能复现审查过程和结论

### 与代码审查的区别

传统代码审查是"同事视角"——找明显问题、提改进建议，思维是防御性的。对抗性审查是"攻击者视角"——主动构造攻击来击穿系统，思维是进攻性的。二者互补而非替代：传统审查适合发现逻辑错误和设计问题，对抗性审查适合发现边界、安全、性能、并发等"正常路径想不出来"的问题。

### 与第一性原理的关系

第一性原理负责"生成层"——从基本原理出发找最优方案，保证**做对的事**（方向正确）。对抗性审查负责"验证层"——从攻击者视角找所有可能问题，保证**把事做对**（实现稳健）。二者构成完整的"生成-验证"闭环，缺一不可。

### 价值

- **降低线上故障率**：上线前系统性发现P0/P1问题，避免线上事故
- **提升知识可信度**：通过五维验证和交叉验证，确保知识档案建立在可靠证据基础上
- **防御认知偏差**：显性的检查清单强制调动System 2慢思考，对抗System 1的自动偏差
- **建立质量文化**：从"差不多就行"升级为"经得起攻击"的质量标准

## 执行步骤

### 场景一：知识研究场景（七模块协议）

适用于知识档案构建、文献调研、资料整合、事实核查。

#### 步骤 0：跨领域概念扫描（语义漂移防御）

**适用条件**：项目涉及2个及以上领域，或单领域内存在学派分歧时必须执行。

- 列出核心术语（5-15个）
- 逐一核查每个术语在不同领域的定义差异
- 标记歧义术语，约定项目内使用定义
- 规划术语表（预留10-20%工作量用于术语对齐）

**跳过条件**：仅涉及单一领域且无学派分歧，可跳过。

#### 步骤 1：来源三级分类

对所有资料按来源级别分类，不同级别使用权限不同：

- **一级来源（Gold Standard）**：同行评审学术期刊、权威出版社专著、知名大学官方课程、政府/国际组织官方文件、当事人一手资料 → 可作为核心论据直接引用
- **二级来源（Silver Standard）**：权威媒体深度报道、知名大学公开课、知名咨询公司报告、权威科普机构内容 → 可作为辅助论据，标注来源，鼓励交叉验证
- **三级来源（Bronze Standard）**：个人博客、自媒体、论坛讨论、社交媒体 → 仅作线索发现，必须追溯至一级或二级来源验证后方可引用

#### 步骤 2：可信度四级评分

对每条具体信息进行可信度评分：

- 🟢 **A级**：多权威来源交叉验证（≥2个独立一级来源），无利益冲突，可追溯至原始出处 → 核心论据，可直接引用
- 🔵 **B级**：单一权威一级来源，逻辑自洽，无明显偏差 → 可作为辅助论据，标注来源
- 🟡 **C级**：二级来源，需进一步验证，存在潜在偏差 → 仅作参考，必须标注"待验证"
- 🔴 **D级**：存疑，无法验证，存在明显矛盾或利益冲突 → 不纳入核心档案

#### 步骤 3：五维验证流程

对每个关键事实（核心概念定义、核心论断、关键数据、重要时间/人物/事件），逐维验证：

1. **来源资质核查**：核查发布机构背景、作者资质、发表渠道、引用情况、利益冲突
2. **交叉验证**：为每个关键事实寻找≥2个独立来源确认，优先使用不同类型/不同立场来源三角验证
3. **时效性评估**：经典理论关注最新修正，实证研究要求近10年，前沿领域要求近3-5年
4. **逻辑一致性审查**：检查内部逻辑自洽、论据支持、数据合理性、论证结构，主动寻找反证
5. **偏差识别**：识别确认偏差、幸存者偏差、权威崇拜、事后归因等10种高频偏差

#### 步骤 4：异常标记

对审查中发现的问题使用标准化标记：

- ⚠️ **待验证**：单一来源、尚未交叉验证
- ❓ **存疑**：与其他权威来源矛盾
- ⚖️ **争议观点**：学术界存在明确学派争议
- 🔍 **利益冲突提示**：来源与结论存在商业/职业/立场关联
- 📚 **歧义术语**：术语存在跨领域语义漂移

#### 步骤 5：验证日志记录

所有验证过程必须可审计、可追溯。每条记录包含：资料ID、来源类型、可信度评分、五维验证过程、交叉验证来源列表、偏差识别结果、审查结论。

> **可审计性原则**：任何第三方根据验证日志应能复现审查过程和结论。

#### 步骤 6：可信度动态调整

可信度不是静态标签，应根据新证据动态调整：

- 🟢 升级：找到≥2个独立一级来源支持且无矛盾反证
- 🔴 降级：核心结论被后续研究反驳、发现数据造假、来源被撤回
- ⏸️ 保持观察：新证据增加不确定性但不足以调整等级，加⚠️标记

### 场景二：AI协作/代码审查场景（多Agent对抗模式）

适用于代码审查、安全审计、上线前质量门禁、架构评审。

#### 步骤 1：明确审查目标和范围

- 明确审查目标：特定功能/模块/整个项目
- 明确审查维度：安全/性能/边界/时序，或全部
- 提供代码上下文：仓库路径、相关文件、架构说明、依赖关系

> 范围越明确，审查越聚焦。全面审查建议分模块进行。

#### 步骤 2：并发启动四大攻击者Agent

每个角色由独立Agent扮演，要求独立思考、互不干扰。明确告知每个Agent："你的目标是找出问题，不是证明代码正确。"

| 角色 | 攻击目标 | 典型攻击思路 |
|------|---------|------------|
| 🔴 **安全攻击者** | 数据安全、权限控制、系统完整性 | 构造注入攻击、尝试越权访问、检查敏感数据泄露、检查认证授权 |
| 🟡 **性能攻击者** | 系统稳定性、资源消耗、可用性 | 构造大输入/并发请求、触发死循环/递归/正则回溯、检查N+1查询/全表扫描 |
| 🟢 **边界攻击者** | 输入校验、异常处理、类型安全 | 构造空值/极值/特殊值/非法格式/异常状态/依赖异常 |
| 🔵 **时序攻击者** | 并发安全、重试机制、分布式一致性 | 并发操作同一资源、不按顺序调用接口、重复提交、部分失败、缓存不一致 |

> **关键**：不能让一个Agent同时扮演多个角色——角色混同会降低攻击性，回到"证明正确"的思维定式。

#### 步骤 3：攻击者输出问题清单

每个攻击者按严重程度输出问题，每个问题必须附带：攻击路径、复现条件、影响范围、修复建议。

| 优先级 | 定义 | 处理方式 |
|--------|------|---------|
| **P0** | 崩溃/安全漏洞/数据丢失/资损 | 必须立即修复，阻塞上线 |
| **P1** | 严重问题/性能瓶颈 | 上线前修复 |
| **P2** | 一般问题/边界case | 评估后决定是否阻塞上线 |
| **P3** | 优化建议/代码质量 | 可选修复，不阻塞上线 |

#### 步骤 4：汇总交叉验证

- 合并四个攻击者的问题清单，去重
- 重点关注不同攻击者发现的关联问题（如边界攻击者发现的空值可能被性能攻击者触发OOM）
- 按优先级排序，制定修复计划

#### 步骤 5：修复后回归对抗

- 修复发现的P0/P1问题
- **必须对修复后的代码重新执行对抗式审查**
- 验证两点：(1)原问题是否真正修复；(2)修复是否引入新问题
- 直到所有P0/P1问题修复，P2问题评估后决定是否阻塞上线

> 修复往往会引入新BUG——这是常态，不是例外。回归对抗是必须的。

#### 定期全局审查

每2-3周对整个项目做一次"从第一性原理出发的对抗式审查"，覆盖架构、依赖关系、代码质量、文档对应。

## 质量验收

### 知识研究场景验收标准

- 所有关键事实完成≥2个独立来源交叉验证
- 一级来源占比≥70%
- 🟢A级资料占比≥60%
- 🔴D级资料为0%
- 验证日志完整可追溯，任何第三方可复现审查过程
- 所有异常标记（⚠️/❓/⚖️/🔍）已记录并附说明

### 代码审查场景验收标准

- 四大攻击者角色全部独立执行，无角色混同
- 每个攻击者输出了具体问题清单（非空泛评论）
- 每个P0/P1问题附带了攻击路径和复现步骤
- 不同攻击者的问题清单已完成交叉验证和去重
- 修复后已完成回归对抗审查
- 不存在"表演式审查"——问题清单有实质性内容而非敷衍

### 自举验证（元审查）

对抗性审查指令集自身必须接受对抗性审查——这是方法论自洽性的要求。在完成每次审查后，执行以下元审查：

- **审查完整性检查**：是否覆盖了所有应审查的维度？是否有遗漏的攻击面？
- **偏差自检**：审查者自身是否存在确认偏差（倾向于找支持性证据）、权威偏差（对权威来源降低审查标准）？
- **方法论合规检查**：是否严格按照五维验证/四大攻击者流程执行？是否存在跳过步骤的情况？
- **反模式检测**：是否存在表演式审查、角色混同、过度攻击、一次性审查等7种反模式？

## 约束条件

- 不适用于快速原型/PoC验证、草稿/个人笔记、一次性脚本、创意发散阶段——这些场景投入产出比低
- 时间极度紧迫的应急处理时，先止血再复盘，应急阶段首要目标是快速恢复
- 不得替代传统代码审查——对抗性审查是补充而非替代，二者互补
- 不得替代第一性原理——对抗性审查找问题但不保证方向对，必须与第一性原理形成"生成-验证"闭环
- 成本收益阈值：当审查成本超过预期收益的 50% 时，降级为快速检查而非深度审查
- 与第一性原理指令集协同：第一性原理分析完成后，可触发对抗性审查验证方案稳健性
- 与复盘指令集协同：复盘识别"审查不充分"问题时，可触发对抗性审查深入排查
- 与洞察指令集协同：洞察的根因分析可复用对抗性审查的偏差识别方法
