# 第一性原理指令集

## 触发条件

- 面对复杂问题，常规经验方案失效或效果有限
- 需要从根本上理解事物本质，而非套用现有模式
- 创新性方案设计，需突破既有路径依赖
- 技术选型或架构决策中存在多重权衡，需归零思考
- 复盘或洞察过程中识别到"知其然而不知其所以然"的问题
- 需要验证既有方案的根本假设是否成立

## 输入规范

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| problem | string | 是 | 待分析的问题陈述 |
| domain | string | 否 | 问题领域：`technical`/`business`/`product`/`process` |
| constraints | list | 否 | 已知约束条件列表 |
| assumptions | list | 否 | 需要验证的既有假设列表 |
| depth | string | 否 | 拆解深度：`surface`/`deep`/`fundamental`，默认 `deep` |
| time_budget | string | 否 | 时间预算 |

## RACI责任分配矩阵

**RACI模型说明**：
- **R** = 负责执行（Responsible）：实际完成工作的角色
- **A** = 最终审批（Accountable）：对结果负最终责任，拥有最终决策权，每项活动有且仅有一个A
- **C** = 需咨询（Consulted）：决策前需征求意见、提供专业输入的角色，双向沟通
- **I** = 需知会（Informed）：决策后需告知进展与结果的角色，单向沟通

| 第一性原理核心活动 | orchestrator | architect | developer | reviewer | tester | co-founder |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|
| 触发分析与问题定义 | **R/A** | C | C | C | C | I |
| 假设剥离与归零（步骤2） | C | **R/A** | C | C | I | I |
| 基础要素识别（步骤3） | I | **R/A** | C | C | I | I |
| 公理提炼与自洽性验证（步骤4） | I | **R/A** | I | C | I | I |
| 自下而上重构方案（步骤5） | I | **A** | R | C | I | I |
| 方案验证与决策（步骤6） | I | C | I | **R/A** | I | I |
| 分析报告归档与通知 | **R/A** | I | I | C | I | I |
| 第一性原理分析质量验收 | C | C | I | **R/A** | I | I |
| 重大架构决策审批 | R | C | I | C | I | **A** |

### 审批权限边界

- **常规问题分析**：architect审批分析过程与公理提炼，reviewer负责质量验收
- **重构方案审批**：architect审批方案合理性，developer负责执行重构
- **重大架构决策**：涉及系统架构根本性调整时，co-founder最终审批
- **公理自洽性争议**：reviewer作为质量门禁，architect参与技术相关争议仲裁
- **分析结论执行**：由orchestrator分配至对应执行模块，不归属第一性原理指令本身
- **跨领域问题边界**：涉及多领域的问题，由orchestrator协调，architect主导技术领域分析

## 核心原则

### 第一性原理的定义

第一性原理是从最基础、不可再分的基础公理出发，自下而上推导问题解决方案的思维方式。其核心在于剥离类比与经验假设，回归事物本质，从公理层面重构认知与方案。

### 核心特征

- **剥离类比与经验假设**：拒绝从"过去如何做"推导"现在该怎么做"，质疑所有未经证实的经验性假设
- **识别不可再分的基础要素**：将问题拆解至最底层、无法再细分的基本组成单元
- **自下而上重构认知**：基于基础公理，通过严密推导链构建从公理到结论的完整逻辑链

### 与类比思维的区别

类比思维从相似性出发，通过"过去如何解决类似问题"推导当前方案；第一性原理从本质出发，通过"问题的本质是什么"推导方案。类比思维受限于已知解空间，第一性原理能发现类比无法触及的创新解。

### 与归纳思维的区别

归纳思维从个案总结规律，结论依赖于样本的代表性且可能被反例推翻；第一性原理从公理推导结论，只要公理成立且推导正确，结论必然成立。归纳思维适用于经验积累，第一性原理适用于本质洞察。

### 价值

- **突破路径依赖**：摆脱既有方案与思维定势的束缚
- **识别隐含假设**：揭示被默认接受但未经验证的前提
- **发现创新解空间**：在类比思维无法触及的领域发现新方案

## 执行步骤

### 步骤 1：问题识别与边界界定

- 明确问题的本质与边界
- 区分"症状"与"根因"
- 记录问题的上下文与已知约束

### 步骤 2：假设剥离与归零

- 列出所有隐含假设（假设清单）
- 逐条验证假设是否成立
- 剥离未经证实的经验性假设，归零至已知事实

### 步骤 3：基础要素识别

- 将问题拆解为不可再分的基础要素
- 验证每个要素是否真正"不可再分"
- 产出要素清单

### 步骤 4：公理提炼

- 从基础要素中提炼自洽的公理
- 验证公理的独立性与完备性
- 产出公理列表

### 步骤 5：自下而上重构

- 基于公理推导解决方案
- 构建从公理到结论的推导链
- 产出重构方案

### 步骤 6：方案验证与决策

- 验证重构方案是否覆盖原问题的全部边界
- 评估方案的成本收益与可行性
- 基于事实而非假设做出决策
- 记录验证过程

## 质量验收

- 基础要素真正不可再分，无经验性假设混入
- 公理之间自洽，无矛盾
- 重构方案覆盖原问题的全部边界
- 验证基于事实数据而非主观判断
- 推导链完整，每一步可追溯
- 方案成本收益评估客观

### 元审查（自检）

> 方法论类指令集必须包含自检机制，确保执行质量而非表演式走过场。参照 [meta-review-in-command 模式](../../docs/retrospective/patterns/methodology-patterns/governance-strategy/meta-review-in-command.md)。

- **完整性检查**：6步流程是否全部执行？是否有跳过的步骤？假设剥离是否遗漏了关键假设？
- **偏差自检**：分析者自身是否存在确认偏差（倾向于支持某个预设结论）？是否主动搜索了反方观点？
- **方法论合规**：是否严格按照"假设剥离→公理提炼→自下而上重构"顺序执行？是否在假设剥离前就跳到了重构？
- **反模式检测**：是否存在"为创新而创新"（方案不服务于原问题）？是否存在"过度拆解"（简单问题滥用复杂方法）？

## 约束条件

- 不适用于简单问题或已有成熟方案的场景，避免过度拆解导致成本失控
- 不得忽视经验价值的合理部分，第一性原理不等于否定一切经验
- 禁止为创新而创新，方案必须服务于原问题
- 拆解深度需匹配问题复杂度，简单问题不应滥用复杂方法
- 成本收益阈值：当拆解成本超过预期收益的 30% 时，应停止深入拆解
- 问题复杂度阈值：当问题可在 1-2 步内解决时，不应启用第一性原理
- 与复盘指令集协同：复盘识别"知其然而不知其所以然"时，可触发第一性原理深入分析
- 与洞察指令集协同：洞察的根因分析可复用第一性原理的假设剥离方法
- 与原子化指令集边界：原子化关注文件拆分，第一性原理关注思维拆解，二者不重叠
