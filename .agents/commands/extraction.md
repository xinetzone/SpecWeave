# 萃取指令集

## 触发条件

- 复盘或洞察过程中识别出可复用的经验/模式/规律
- 同类问题出现≥2次，需要总结为可复用方法
- 模式库中缺少对应场景的方法论文档
- 用户明确请求萃取/沉淀/模式化
- 复盘报告中提炼出具体改进建议需要标准化

## 输入规范

| 参数 | 类型 | 必选 | 说明 |
|------|------|------|------|
| source | string\|list | 是 | 萃取来源：单个洞察ID/复盘报告路径/多个案例的路径列表 |
| pattern_type | string | 是 | 模式类型：`methodology`/`architecture`/`code`/`process`/`creative`/`governance` |
| abstraction_level | string | 否 | 抽象层级：`project-specific`（仅本项目）/`domain-general`（领域通用）/`universal`（跨领域通用），默认`domain-general` |
| existing_patterns | list | 否 | 需参考的已有模式ID列表（防重复萃取） |
| maturity_target | string | 否 | 目标成熟度：`L1-draft`/`L2-validated`，默认`L1-draft` |

## RACI责任分配矩阵

**RACI模型说明**：
- **R** = 负责执行（Responsible）：实际完成工作的角色
- **A** = 最终审批（Accountable）：对结果负最终责任，拥有最终决策权，每项活动有且仅有一个A
- **C** = 需咨询（Consulted）：决策前需征求意见、提供专业输入的角色，双向沟通
- **I** = 需知会（Informed）：决策后需告知进展与结果的角色，单向沟通

| 萃取核心活动 | orchestrator | architect | developer | reviewer | co-founder |
|:---|:---:|:---:|:---:|:---:|:---:|
| 触发萃取与来源确认 | **R/A** | C | C | C | I |
| 案例收集与跨案例分析（步骤1） | R | **A** | C | C | I |
| 本质抽象与去场景化（步骤2） | C | **R/A** | C | C | I |
| 模式结构化成模板（步骤3） | R | C | C | **A** | I |
| 反模式与边界条件提炼（步骤4） | C | **R/A** | C | C | I |
| 可迁移性验证（步骤5） | C | R | C | **A** | I |
| 模式文档生成与入库（步骤6） | **R/A** | C | C | C | I |
| 索引更新与交叉引用修复 | **R/A** | I | I | C | I |
| 萃取质量验收 | C | C | I | **R/A** | I |
| 跨领域通用模式入库审批 | R | C | I | C | **A** |

### 审批权限边界

- **项目级模式（L1）**：reviewer审批模式质量，orchestrator负责入库
- **领域通用模式（L2）**：architect审批抽象合理性，reviewer负责格式与质量验收
- **跨领域通用模式（L3/L4）**：co-founder最终审批，确认模式不与现有体系冲突
- **模式重复/合并争议**：reviewer负责比对，architect参与技术判定
- **反模式提炼争议**：reviewer作为质量门禁，架构相关争议由architect仲裁

## 核心原则

### 可迁移性优先原则

萃取不是"总结这次怎么做的"，而是提炼"下次遇到类似情况应该怎么做"。判断萃取质量的黄金标准：**把你萃取的模式讲给一个不在这个项目里的人听，他能否直接用在自己的工作中**。

### 反模式对等原则

每个模式必须对等提炼反模式——告诉别人"不要怎么做"和"要怎么做"同等重要。只有正确做法没有反模式的萃取是不完整的，因为人们往往从错误中学到更多。

### 多案例支撑原则

成熟模式必须有≥2个独立案例支撑。只有1个案例的萃取只能标记为L1-draft（假设性模式），不能标记为已验证。如果只有1个案例，必须明确标注"待验证"并给出验证场景建议。

### 抽象层次适配原则

模式的抽象层级要适配使用场景：
- 太具体（"在智能文档项目中我们这样做"）→ 无法复用
- 太抽象（"要注意风险"）→ 正确的废话，没有指导价值
- 合适的抽象："当你做XX类型的项目时，在XX阶段应该用XX方法做XX事"

## 执行步骤

### 步骤 1：案例收集与跨案例分析

- 收集萃取来源中的所有相关案例（复盘报告、问题记录、成功经验等）
- 如果只有1个案例，标记为"单案例待验证"
- 提取每个案例的共同要素：
  - 触发场景是什么？（什么情况下会遇到这个问题）
  - 成功案例做对了什么？
  - 失败案例踩了什么坑？
- 识别案例间的共同结构，排除偶然性因素

### 步骤 2：本质抽象与去场景化

- 剥离具体项目/技术/人员细节，提炼本质结构
- 连续追问"如果换一个项目/技术栈/团队，这个经验还成立吗？"
- 识别模式的核心公理：这个模式基于什么假设成立？
- 提炼模式名称：4-8个字，好记、有传播性，避免学术化长名称

### 步骤 3：模式结构化成标准模板

将模式结构化为以下标准格式：

```markdown
---
id: [模式唯一ID]
title: [模式名称]
type: [pattern-type]
date: [创建日期]
maturity: [L1-draft/L2-validated/L3-mature/L4-optimized]
source: [来源复盘/洞察ID]
related_patterns: [相关模式ID列表]
tags: [标签列表]
---

# [模式名称]

## 触发场景
- 当______时，使用这个模式
- 适用于：______类型的场景
- 不适用于：______（边界条件）

## 核心做法
1. 第一步：
2. 第二步：
3. 第三步：
（3-7步，不超过7步）

## 反模式（不要这么做）
- ❌ 反模式1：
- ❌ 反模式2：
- ❌ 反模式3：

## 检验标准
做完之后怎么知道做对了？
- 标准1：
- 标准2：

## 迁移示例
这个模式还能用在什么其他场景？
- 场景1（非当前领域）：
- 场景2（跨领域）：
```

### 步骤 4：反模式与边界条件提炼

- 至少提炼3个反模式，反模式必须来自实际案例教训
- 明确模式的边界条件：什么情况下这个模式失效/不适用
- 识别"看起来像但其实不是"的易混淆场景
- 标注模式使用的前置条件（需要什么前提才能用）

### 步骤 5：可迁移性验证

- **跨场景验证**：想出至少1个非软件项目/非当前领域的应用场景
  - 例："红灯上报机制"可以用在家庭旅行规划、个人任务管理中
- **极端场景验证**：在极端条件下（时间极紧/资源极少/团队全是新人）模式是否成立
- **反例验证**：有没有不适用这个模式的反例？反例的共同特征是什么？
- 如果无法找到跨领域迁移场景，说明抽象程度不够，返回步骤2重新抽象

### 步骤 6：模式文档生成与入库

- 按照标准模板生成完整的模式文档
- 选择正确的存储目录：
  - 架构类模式 → `docs/retrospective/patterns/architecture-patterns/`
  - 代码类模式 → `docs/retrospective/patterns/code-patterns/`
  - 流程类模式 → `docs/retrospective/patterns/process-patterns/`
  - 方法论模式 → `docs/retrospective/patterns/methodology-patterns/`
  - 创意设计模式 → `docs/retrospective/patterns/creative-design/`
  - 治理策略模式 → `docs/retrospective/patterns/governance-strategy/`
- 标注成熟度等级：
  - L1-draft：单案例，待验证
  - L2-validated：≥2案例，已在本项目验证
  - L3-mature：跨项目验证，有明确边界条件
  - L4-optimized：经过对抗审查，工具化/自动化支持

### 交叉引用与索引更新（强制）

- 更新对应目录的 `README.md` 索引，加入新模式
- Grep搜索现有文档中是否有相关内容需要添加交叉引用
- 如果新模式替代/合并了旧模式，在旧模式中添加迁移说明，禁止直接删除旧模式
- 使用 `python .agents/scripts/check-links.py --path <pattern-dir>` 验证所有链接有效

| 产出物 | 格式 | 存储位置 |
|--------|------|---------|
| 模式文档 | Markdown + YAML frontmatter | 对应pattern类型子目录 |
| TOML元数据 | TOML | `.meta/toml/`对应目录 |

## 质量验收

- [ ] 模式名称好记（4-8字），能准确反映核心思想
- [ ] 触发场景清晰，包含"适用于"和"不适用于"边界
- [ ] 核心做法有3-7个具体步骤，可直接执行
- [ ] 至少3个反模式，且均来自实际案例教训
- [ ] 有明确的检验标准（做完怎么知道对了）
- [ ] 有≥1个跨领域迁移示例
- [ ] 多案例支撑（L2+）或明确标注"单案例待验证"（L1）
- [ ] 存储目录正确，索引已更新
- [ ] 所有链接验证通过，无断链
- [ ] YAML frontmatter字段完整，id唯一

## 约束条件

- 禁止为了凑模式而过度抽象——如果没有真正可复用的规律，不要强行萃取
- 禁止只萃取"正确做法"不提炼"反模式"
- 禁止将单一项目的偶发成功因素提炼为通用模式
- 萃取过程中发现模式重复/与现有模式冲突时，必须与现有模式比对，选择合并/补充/新建，禁止创建重复模式
- 模式入库不代表结束——模式需要在后续使用中持续演进成熟度
