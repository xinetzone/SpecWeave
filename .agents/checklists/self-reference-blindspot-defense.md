# 自指盲区/递归讽刺防御检查清单

## 什么是自指盲区？

**自指盲区（Self-Reference Blind Spot）**，又称**递归讽刺（Recursive Irony）**，是一种特殊的认知偏差：当你分析、论述、或实现某个**质量保障机制**（如V-对抗审查、测试、验证、安全检查）时，你最容易犯该机制所防范的**完全相同的错误**。

### 典型案例

> 一份深入分析"V-对抗性审查"机制的报告，雄辩地论证了"Generator和Evaluator共享误解导致错的实现+全部通过的测试"，但这份报告自身恰好犯了完全相同的错误——35/35检查点全部通过，却存在18处事实错误（9处编造引文、3处归属错误、虚构概念），因为报告的"验证者"（checklist）和"生成者"（组装子代理）共享了同一上下文。

### 为什么会发生？

| 层级 | 机制 | 说明 |
|------|------|------|
| 认知层 | **理解即免疫错觉** | 你深入理解了某个防御机制，会产生"我已经知道这个坑，所以不会踩"的错觉，但理解≠执行 |
| 元认知层 | **审查者视角偏差** | 当你扮演"分析者/审查者"角色时，会不自觉地将自己置于"审查他人"的位置，忘记审查自己 |
| 执行层 | **验证与被验证共享上下文** | 自我验证时，你的验证逻辑和产出逻辑使用相同的上下文、相同的假设、相同的盲点 |
| 情感层 | **主题相关性降低警惕** | 写质量保障相关内容时，"我在写关于质量保障的内容"本身会带来虚假的质量安全感 |

## 🚨 触发条件检测

在开始以下类型的任务时，**必须**启用本检查清单：

- [ ] 分析/论述/实现某种**验证/审查/测试/质量保障**机制
- [ ] 撰写关于**如何避免错误/幻觉/Bug**的文档或代码
- [ ] 创建**checklist/验证标准/审查流程**本身
- [ ] 对**方法论/最佳实践/工程范式**进行分析或总结
- [ ] 构建**安全/权限/审计**相关功能
- [ ] 任务涉及**自我评估/自我验证/自我审查**环节
- [ ] 你在产出中使用了"必须""应该""禁止"等规范性语言来指导他人

## 🛡️ 防御策略（7项强制检查）

### 检查1：异体验证（Mandatory Cross-Agent Verification）

> **核心原则**：涉及质量保障主题的产出，**禁止自我验证**。

- [ ] 最终验证必须由**未参与产出过程**的独立子代理执行
- [ ] 验证子代理的任务描述**不包含**产出过程中的任何中间推理或假设
- [ ] 验证子代理只接收：原始素材 + 最终产出 + 核查指令（不给分析框架/不引导结论）
- [ ] 验证子代理被明确告知："你的任务是找错，不是赞美。如果找不到错误，请说明你检查了哪些维度"

### 检查2：反向应用（Inverse Application）

> **核心原则**：把你正在论述的防御机制，**反向应用到你自己的产出上**。

- [ ] 如果你在写"如何做V对抗审查"，请用V对抗审查的方法审查你的文章
- [ ] 如果你在写"如何防止幻觉"，请用你的防幻觉方法检查你的文章是否有幻觉
- [ ] 如果你在写"测试方法论"，请用你的测试方法论测试你的文章中的事实声明
- [ ] 明确记录：你声称的防御机制X，你是如何具体用X来防御X自身的？如果答案是"我凭自觉"，则未通过

### 检查3：引文溯源强制（Mandatory Citation Tracing）

> **核心原则**：所有事实性声明必须可追溯到原始来源，**零例外**。

- [ ] 所有 `>` 块引用必须标注精确位置（章节号/行号/URL）
- [ ] 所有数据/数字（如"1200万行""5个月""100万行"）必须标注来源
- [ ] 所有专有名词（人名、产品名、公司名、框架名）必须与原文一致
- [ ] 如果你发现某句引语"写得太好了"或"太贴切了"，**必须**二次核查——这是编造引文的高危信号
- [ ] 核查方式：在原文中搜索引文的关键词，确认存在而非仅"意思差不多"

### 检查4：术语锁定（Terminology Lock）

> **核心原则**：原文使用的关键术语**不得替换**，建立术语表强制执行。

- [ ] 从原文/需求文档中提取关键术语表（术语 → 精确定义）
- [ ] 在产出中使用grep比对：是否存在术语被近义词替换的情况
- [ ] 特别警惕：语义"接近但不等价"的替换（如"控制"→"反馈"、"约束"→"限制"、"前馈"→"引导"）
- [ ] 术语表中明确列出"禁止替换词"列表

### 检查5：否定测试（Negation Test）

> **核心原则**：对你的核心论点进行否定测试——如果把你的论点反过来，它是否恰好描述了你的产出？

- [ ] 列出你产出中的3-5条核心主张/结论
- [ ] 对每条主张取反（如"X是Y的关键"→"X不是Y的关键"）
- [ ] 检查：取反后的陈述是否恰好描述了你产出中的某个缺陷？
- [ ] 如果是→**你正处于自指盲区中**，必须修复该缺陷后才能继续

**示例**：
- 主张："V对抗审查通过角色分离防止共享误解"
- 取反："V对抗审查未能防止共享误解"
- 检查：你的验证过程是否存在角色不分离的问题？（在本案例中，答案是"是"——同代理验证）
- 行动：必须引入独立验证

### 检查6：能力边界声明（Capability Boundary Declaration）

> **核心原则**：在论述某个强大机制时，必须同时声明它的**不能做什么**。

- [ ] 你在论述机制X的优势时，是否同时声明了X的局限性？
- [ ] 你是否隐含或明确地声称X能解决某类问题，但X实际上无法解决（特别是无法解决X自身的问题）？
- [ ] 你的论述中是否存在"有了X就可以Y"的绝对化表述？如果有，改为"X可以帮助Y，但不能替代Z"
- [ ] 示例：不应说"checklist验证保证产出质量"，应说"checklist验证结构完整性，但无法替代独立事实核查"

### 检查7：完成后冷却期（Cooling-off Period）

> **核心原则**：完成产出后不要立即宣布"完成"，经过"认知冷却"后做最终检查。

- [ ] 产出完成后，至少执行一轮不相关的其他任务（上下文切换）
- [ ] 冷却后重新阅读产出，假装这是别人写的，用最挑剔的眼光审视
- [ ] 特别关注：那些"读起来特别顺""论证特别有力"的段落——它们最可能是精心编造的幻觉
- [ ] 问自己："如果这是别人交给我审核的文档，我会在哪里打问号？"

## 📋 快速使用流程

```
任务启动
  │
  ├─→ 检测到触发条件？
  │     ├─ 否 → 正常流程
  │     └─ 是 → 激活本检查清单
  │
  ├─→ 产出过程中
  │     ├─ 建立术语表（检查4）
  │     ├─ 引文随写随标注位置（检查3）
  │     └─ 反向应用：用你论述的方法检查自己（检查2）
  │
  ├─→ 产出完成
  │     ├─ 否定测试：取反核心论点检查自指缺陷（检查5）
  │     ├─ 能力边界声明：补充局限性说明（检查6）
  │     └─ 冷却期后重读（检查7）
  │
  └─→ 交付前
        └─ 委派独立子代理做异体验证（检查1）
              ├─ 发现错误 → 修复 → 重新检查1-6
              └─ 未发现错误 → 通过，交付
```

## 🔴 红灯信号（出现任一项必须暂停）

| 信号 | 含义 | 行动 |
|------|------|------|
| "这段写得真好，太有道理了" | 可能是编造的"合理内容" | 逐句核查引文来源 |
| "我已经理解了V机制，不会犯V的错误" | 理解即免疫错觉 | 启动检查2（反向应用） |
| "checklist全过了，应该没问题" | 结构验证≠事实验证 | 启动检查1（异体验证） |
| "这个术语和原文的差不多" | 术语漂移风险 | 启动检查4（术语锁定） |
| "我的核心论点反过来不也成立吗？" | 否定测试触发 | 修复缺陷后重新走流程 |

## 📊 与其他防御机制的关系

| 防御机制 | 覆盖范围 | 本清单的补充 |
|---------|---------|-------------|
| SVA模式 | 外部内容分析的事实保真 | 自指盲区是SVA中V步骤的特殊高危场景 |
| CLN规则 | 引文溯源 | 本清单检查3是CLN的强化版（含"太好的引文"信号检测） |
| V-对抗审查 | 多视角验证 | 本清单解决"V审查V"时的特殊盲区 |
| Checklist验证 | 结构完整性 | 本清单检查1明确指出checklist无法自我审计 |

## 来源与验证

- **来源案例**：WorkBuddy七概念分析任务（sc-20260714-workbuddy-harness-v2）
  - 报告深入分析V-对抗审查机制，但自身V验证失败（35/35→18处错误）
  - 核心错误类型：编造引文、术语漂移、归属错误、虚构概念
  - 根因：同代理自检（共享误解）+ 缺乏独立事实核查
- **验证方法**：本清单的检查5（否定测试）在该案例上验证——取反"V防止共享误解"得到"V未能防止共享误解"，准确识别了自指盲区
