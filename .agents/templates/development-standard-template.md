# 开发规范文档模板

> **使用说明**：本模板从5个可复用模式（编译型Wheel镜像构建/隐式依赖检测/Docker网络容错/静态注册编译配置/配置持久化全链路覆盖）的共性结构中萃取而成。当你从项目实践中提炼出新的可复用规范时，复制本模板，替换 `{占位符}` 为实际内容，删除填写指引注释后即可入库。
>
> **适用场景**：将单次项目经验转化为团队可复用的开发规范文档。
>
> **与模式文档的关系**：模式文档侧重"怎么做"（步骤导向），开发规范文档侧重"必须怎么做"（条款导向）。本模板兼顾两者——核心步骤提供操作指引，反模式和检验标准提供规范约束。

---

## 〇、模板填写顺序

> **填写指引**：按以下顺序填写，避免跳跃导致的结构不一致：
>
> 1. **frontmatter**：填写id/title/type/date/maturity/source/related_patterns/tags
> 2. **触发场景**：明确"什么时候需要这个规范"——这是读者判断是否继续阅读的决策点
> 3. **问题背景**：解释"为什么需要这个规范"——没有问题背景的规范是教条
> 4. **核心步骤**：描述"怎么做"——每步必须是可执行的操作，而非抽象原则
> 5. **反模式**：列出"不要怎么做"——从实际踩坑中提炼，而非理论推测
> 6. **检验标准**：定义"做完怎么知道做对了"——可勾选的检查清单，而非主观判断
> 7. **迁移示例**：提供"还能用在哪里"——至少1个跨领域迁移场景，证明可迁移性
> 8. **与相关模式的关系**：建立交叉引用——孤立规范容易被遗忘
> 9. **Changelog**：记录版本历史——规范是演进的

## {规范标题}

> **标题命名规范**：使用"领域+方法+模式"格式，如"编译型Python Wheel运行时镜像构建模式"、"配置持久化全链路覆盖模式"。避免过于抽象的标题（如"配置管理规范"）或过于具体的标题（如"XMNN项目的config.cmake修复方法"）。

### frontmatter 模板

```yaml
---
id: "{kebab-case-id}"
title: "{规范标题}"
type: code-pattern | process-pattern | methodology-pattern | architecture-pattern
date: {YYYY-MM-DD}
maturity: L1 实验性 | L2 已验证 | L3 成熟 | L4 通用
maturity_note: "{成熟度说明：验证案例数量+升级条件}"
source: "{来源报告或复盘文档的相对路径#章节锚点}"
related_patterns:
  - "{相关模式文件相对路径}"
tags: ["{标签1}", "{标签2}", "{标签3}"]
validation_count: {验证案例数}
reuse_count: {复用次数}
---
```

> **填写指引**：
> - `id`：kebab-case，全局唯一，与文件名一致（不含扩展名）
> - `type`：根据规范性质选择——代码级用 `code-pattern`，流程级用 `process-pattern`，方法论用 `methodology-pattern`，架构级用 `architecture-pattern`
> - `maturity`：新萃取规范默认 `L1 实验性`，待第二个独立案例验证后升级 `L2 已验证`
> - `source`：必须标注来源（哪次复盘/哪个报告/哪个事件），几个月后回看时需要追溯产生背景
> - `validation_count`/`reuse_count`：量化数据，非主观标签。`validation_count` 是独立验证的案例数，`reuse_count` 是实际复用的次数

### 一、触发场景

> **填写指引**：本章节帮助读者快速判断"这个规范是否适用于我的场景"。必须包含三部分：适用场景列表、识别信号（具体症状）、不适用场景列表。

#### 适用场景

- {场景1：描述需要应用此规范的具体情况}
- {场景2：另一个适用情况}
- {场景3：第三个适用情况}

#### 识别信号

> **填写指引**：列出具体的、可观察的症状（如错误信息、日志特征、指标异常），而非抽象描述。读者可以通过搜索这些信号快速判断是否需要应用此规范。

- {信号1：具体的错误信息或症状}
- {信号2：另一个可观察的信号}
- {信号3：日志特征或指标异常}

**示例**（来自编译型Wheel运行时镜像构建模式）：
- `ImportError: libxxx.so: cannot open shared object file`
- `readelf -d libxxx.so | grep RPATH` 显示硬编码的 conda/构建环境路径
- 运行时报 `cannot restore segment after relocation` 或 `undefined symbol`

#### 不适用场景

> **填写指引**：明确列出"不需要应用此规范"的场景，避免过度适用。不适用场景与适用场景同等重要——它定义了规范的边界。

- {不适用场景1：描述不需要此规范的情况及原因}
- {不适用场景2：另一个不适用情况}
- {不适用场景3：第三个不适用情况}

### 二、问题背景

> **填写指引**：解释"为什么需要这个规范"——如果读者不理解问题背景，就会把规范当成教条而非智慧。问题背景应包含：(1) 问题的技术原理（为什么会出现这个问题）；(2) 常见做法的陷阱（为什么常规方案不工作）；(3) 修复的成本分析（为什么不采用看似简单的方案）。

#### {问题原理标题}

{解释问题背后的技术原理。必须让读者理解"为什么会出现这个问题"，而非只说"这是一个问题"。}

**示例**（来自配置持久化全链路覆盖模式）：

```
构建配置（如 CMake 的 config.cmake）通常存在三层修改路径，
任何一层遗漏都会导致修复失效：

L1: 模板层（config.cmake）—— 默认值
L2: 动态替换层（tasks.py）—— 运行时替换模板值
L3: 独立脚本层（rebuild.sh）—— 直接通过命令行参数指定

只修改 L1 而不检查 L2/L3，修复会静默失效。
```

#### {常见做法的陷阱标题}

{解释为什么常规方案不工作。必须基于实际踩坑经验，而非理论推测。}

#### {修复成本分析标题}（可选）

{如果存在看似简单但实际成本很高的方案，分析其成本。这帮助读者理解"为什么不采用简单方案"。}

### 三、核心步骤（{N}步法）

> **填写指引**：描述"怎么做"。每个步骤必须是可执行的操作（有具体命令或代码），而非抽象原则。步骤之间有明确的顺序关系（步骤N依赖步骤N-1的输出）。推荐3-7步，少于3步不需要"步骤法"，多于7步应拆分为多个子流程。
>
> **步骤命名规范**：使用"动词+对象"格式（如"搜索所有包含该配置项的文件"），避免使用抽象名词（如"准备工作"）。

#### 步骤1：{操作描述}

```bash
# {具体命令或代码}
{命令或代码}
```

**为什么**：{解释这一步的目的和原理。读者需要理解"为什么这么做"而非只记住"做什么"。}

**关键原则**：{这一步的关键注意事项。如果存在常见错误，在此指出。}

#### 步骤2：{操作描述}

```bash
# {具体命令或代码}
{命令或代码}
```

**为什么**：{解释原理}

> **填写指引**：根据需要继续添加步骤3-N。每个步骤保持相同的结构：操作描述→代码示例→为什么→关键原则。

### 四、反模式（不要这么做）

> **填写指引**：列出"不要怎么做"。反模式不是"反面教材"（做得差的例子），而是"模式失效的边界条件"——即"在X条件下，使用本规范会导致Y负面结果"。每个反模式必须包含三部分：错误描述、后果、正确做法。至少3个反模式。
>
> **反模式来源**：必须来自实际踩坑经验，而非理论推测。如果是推断的反模式，标注"[推断]"。

#### ❌ 反模式1：{错误做法名称}

- **错误**：{具体描述错误的操作}
- **后果**：{这个错误会导致什么问题}
- **正确做法**：{应该怎么做}

#### ❌ 反模式2：{错误做法名称}

- **错误**：{具体描述错误的操作}
- **后果**：{这个错误会导致什么问题}
- **正确做法**：{应该怎么做}

#### ❌ 反模式3：{错误做法名称}

- **错误**：{具体描述错误的操作}
- **后果**：{这个错误会导致什么问题}
- **正确做法**：{应该怎么做}

> **填写指引**：根据需要继续添加反模式4-N。来源自实际踩坑的反模式价值最高，推断的反模式标注"[推断]"。

### 五、检验标准

> **填写指引**：定义"做完怎么知道做对了"。每项标准必须是可验证的（有具体命令或检查方法），而非主观判断（如"代码质量好"）。检验标准用于提交前自检和Code Review。

做完之后怎么知道做对了？

- [ ] 标准1：{可验证的检查项，含具体命令或检查方法}
- [ ] 标准2：{可验证的检查项}
- [ ] 标准3：{可验证的检查项}
- [ ] 标准4：{可验证的检查项}
- [ ] 标准5：{可验证的检查项}

**示例**（来自编译型Wheel运行时镜像构建模式）：
- [ ] 标准1：`readelf -d` 确认 `.so` 的 RPATH 路径在运行时镜像中存在
- [ ] 标准2：`python -c "import <包>; <包>.<核心API>()"` 成功执行（不只是 import）
- [ ] 标准3：`ldd <wheel中的.so>` 无 "not found" 输出
- [ ] 标准4：从 tar.gz 加载全新镜像后验证通过（非在构建环境中验证）

### 七、待验证问题（升级条件）

> **填写指引**：列出规范升级到下一成熟度级别需要确认的问题。这帮助后续使用者知道"这个规范还有哪些不确定性"，也为升级提供明确的验证清单。

1. **{问题1}**：{具体需要验证的问题}
2. **{问题2}**：{具体需要验证的问题}
3. **{问题3}**：{具体需要验证的问题}

**升级路径**：
- L1→L2：在≥2个独立案例中验证有效
- L2→L3：在≥3个不同项目中复用，且有完整的反模式验证
- L3→L4：形成通用原则，可跨技术栈迁移

### 八、与相关模式的关系

> **填写指引**：建立交叉引用。孤立规范容易被遗忘，交叉引用让规范库形成网络。每个引用必须说明关系类型（配合使用/互补/具体展开/方法论基础等）。

- **[{模式名称}]({相对路径})**：{关系类型}——{具体关系描述}
- **[{模式名称}]({相对路径})**：{关系类型}——{具体关系描述}
- **[{检查清单名称}]({相对路径})**：{关系类型}——{具体关系描述}

**关系类型参考**：
- **配合使用**：两个规范经常一起使用
- **互补**：一个关注A维度，另一个关注B维度
- **具体展开**：本规范是另一个规范的细节展开
- **方法论基础**：另一个规范是本规范的方法论依据
- **L4模板化产物**：本规范是另一个规范的检查清单化产物

## 附录：模板使用检查清单

使用本模板编写规范文档后，提交前自检以下项目：

- [ ] **frontmatter完整**：id/title/type/date/maturity/source/related_patterns/tags/validation_count/reuse_count 全部填写
- [ ] **触发场景三要素**：适用场景 + 识别信号 + 不适用场景 均已填写
- [ ] **问题背景有技术原理**：解释了"为什么会出现这个问题"，而非只说"这是一个问题"
- [ ] **核心步骤可执行**：每个步骤有具体命令或代码，而非抽象原则
- [ ] **反模式≥3个**：每个反模式含错误描述+后果+正确做法，来自实际经验
- [ ] **检验标准可验证**：每项标准有具体命令或检查方法，而非主观判断
- [ ] **迁移示例≥1个跨领域**：证明规范的可迁移性，标注"[推断]"的待验证
- [ ] **交叉引用已建立**：与相关模式的关系已说明，无孤立规范
- [ ] **文件已入库**：存入 `docs/retrospective/patterns/` 对应分类子目录
- [ ] **索引已更新**：对应目录的 README 索引已添加新规范条目
- [ ] **链接已验证**：运行 `check-links.py` 验证所有链接有效

> **提交前必做**：运行 `python .agents/scripts/check-links.py --path <规范文件所在目录>` 验证链接有效性。
