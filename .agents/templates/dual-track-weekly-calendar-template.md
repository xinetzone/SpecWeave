# 双轨复盘周执行日历模板

> **来源**：[双轨复盘节奏模式](../docs/retrospective/patterns/methodology-patterns/retrospective-knowledge/dual-track-retrospective-cadence.md)。
> **用途**：每周日用此模板规划下周的事件驱动+时间驱动双轨复盘节奏，确保节奏性活动不被任务洪流挤压。
> **使用流程**：周日晚完成周复盘后 → 复制本模板到下周目录 → 填入下周已知事件驱动节点 → 填入时间驱动检查点 → 执行中随时补充。

## 二、时间驱动检查点（固定节奏，不可跳过）

> 这些是**必须执行**的固定节奏，即使当天"没什么特别的事"也要做。CI会自动提醒周复盘，日检靠自律+模板。

### 每日检查（工作日晚，5分钟，5个问题）

每日收尾时快速回答：

- [ ] **今日完成了什么？**（1-2句话，不是流水账）
- [ ] **有没有慢变问题被忽略？**（测试/覆盖率/链接/统计数字是否异常？）
- [ ] **有没有应该记下来的洞察？**（哪怕一句话，记到当日笔记中）
- [ ] **明天最重要的1件事是什么？**（不超过1件）
- [ ] **有没有需要升级到周层面的问题？**（标记 ⬆️ 到周复盘）

### 周复盘（周日，30分钟）

- [ ] 使用 [weekly-retrospective-template.md](weekly-retrospective-template.md) 完成结构化复盘
- [ ] CI 自动创建Issue（weekly-iteration-reminder.yml）
- [ ] 运行 `python .agents/scripts/docgen.py weekly` 获取数据快照
- [ ] 基于日检中标记的 ⬆️ 问题做根因分析
- [ ] 制定下周日历（即本模板）

### CI质量门禁（工作日午间，自动化）

- [ ] 每日午间推送后 CI 自动运行16步质量门，如红灯立即修复
- [ ] 关注 stats 异常告警（ACT-05 三防线：异常时自动创建Issue）
- [ ] 每周一查看上周覆盖率报告（pytest-cov 输出）

| 5 | | | | | ⬜ |

### 事件驱动复盘触发条件速查

| 事件发生时 | 复盘类型 | 最小产出 | 时间盒 |
|-----------|---------|---------|--------|
| 完成一个功能/修复一个bug | task-reports | 做了什么+遇到的问题+改进点 | 10min |
| 发现并修复一个有根因价值的bug | bugfix + incident | 根因分析（5-Whys）+预防措施 | 15min |
| 完成一个Spec | spec-system + 模式萃取 | 产出物+可复用模式 | 20min |
| 完成一个里程碑/大版本 | project-reports | 完整S1-S4四步复盘 | 30-60min |
| CI质量门红灯 | 即时修复 | 修复+根因（是否需要模式/规则升级） | 即时 |
| 发现值得沉淀的实践/反模式 | insight-extraction | 模式草稿或insight记录 | 5min |

## 六、风险预判与预案

| 潜在风险 | 概率 | 影响 | 预案 |
|---------|------|------|------|
| 紧急bug插入打乱计划 | 中 | 高 | 挪用Buffer时间，不挤压深度块；当天修复当天复盘 |
| 任务比预估复杂耗时超预期 | 中 | 中 | 拆分任务，先交付核心再迭代；不跳过事件复盘 |
| 连续几天跳过日检 | 低 | 中 | 日检简化为"一句话+明日1件事"，保持习惯不断 |
| CI红灯需要大量修复时间 | 低 | 高 | 优先修复CI再继续开发；CI修复本身就是事件复盘节点 |

## 使用提示

1. **不要完美主义**：日检5分钟内完成，写一句话比不写好100倍
2. **事件驱动随时加**：本周出现新的未预估任务/事件时，随时加到第三部分
3. **Buffer是必须的**：每周至少留半天不安排具体任务，应对意外
4. **深度块保护**：上午深度块不安排会议/消息回复/琐碎事务
5. **⬆️标记是核心**：日检中发现"这不是一天能解决的问题"立即标记⬆️，带到周复盘
6. **模板可以裁剪**：根据本周实际情况调整深度块和事件节点，但**时间驱动检查点不可删除**
