# {项目名} 项目启动 Spec

> **使用说明**：将 `{项目名}`、`{领域}`、`{一句话定位}` 等占位符替换为实际内容，其余结构均按竹简悟道方法论设计，开箱即用。

## Why
{一句话定位} 是一个以 {核心哲学/领域根基} 为根基的 {产品类型}。为从第一天起建立系统的知识管理与质量保障体系，本项目采用六条来自竹简悟道复盘的方法论作为启动基础设施：洞察驱动开发（定义"为什么"先于"做什么"）、约束驱动设计（边界内最大化自主空间）、滚动复盘（高频审计防止熵增）、Skill 双层架构（按工具和约束划分AI能力）、概念操作化三层递进（将核心概念转化为渐进式实践）、反效率设计（以差异化定位对抗主流惯性）。

## What Changes
- 创建 `.agents/` 项目规范骨架（project.md / conventions.md / constraints.md / workflows.md / git.md）
- 建立洞察库初始结构（insights/ 目录 + 编号规则 + 首条锚点洞察）
- 设立滚动复盘机制（首次复盘模板 + P0-P3 优先级体系）
- 根据领域需要设计 Skill 架构（1-2 个 Skill + references/ 五层递减结构）
- 定义核心概念词典（≥3个核心概念 + 三层递进展开计划）

## Impact
- Affected specs: 无（新项目启动）
- Affected code: 新建项目根目录 + `.agents/` 骨架
- 后续迭代：首次复盘 → 洞察驱动设计 → 原型 → 方法论萃取

## ADDED Requirements

### Requirement 1: 洞察驱动开发基础设施
系统 SHALL 在项目启动时建立洞察库基础设施，确保所有设计决策以洞察条目形式显式记录，先定义"为什么"再动手"做什么"。

#### Scenario: 洞察库初始化
- **WHEN** 项目启动
- **THEN** 创建 `{项目根}/.agents/docs/insights/` 目录，定义洞察编号规则（全局递增、分层分段）、标准洞察结构模板（来源 + 核心内容 + 分节展开），写入至少 1 条锚点洞察（定义项目的核心差异化定位）

#### Scenario: 洞察库结构分层
- **WHEN** 洞察数量超过 15 条
- **THEN** 按三层结构分层归档：产品层（直接指导产品设计）、架构层（支撑设计的底层原理）、元层（审视洞察过程本身）

---

### Requirement 2: 约束驱动设计体系
系统 SHALL 采用"负面清单"模式定义项目约束，优先定义"绝对不能做什么"而非"应该做什么"，在约束边界内最大化智能体的创造自主性。

#### Scenario: 约束清单初始化
- **WHEN** 项目启动
- **THEN** 创建 `constraints.md`，定义至少 5 条编号约束（C-01 至 C-0N），每条标注出处和违反后果。约束按类别分组（产品层/文档编写/代码技术/领域哲学），建立角色专用的精简约束速查手册

#### Scenario: 约束与洞察联动
- **WHEN** 新增洞察时
- **THEN** 评估是否需要新增约束条目。每条约束标注对应的洞察来源，形成可追溯链

---

### Requirement 3: 滚动复盘机制
系统 SHALL 建立滚动复盘机制，每 1-2 次变更触发一轮全项目审计，使用 P0-P3 四级优先级追踪问题，防止质量熵增。

#### Scenario: 首次复盘
- **WHEN** 项目完成第一轮设计决策（洞察≥5条）
- **THEN** 执行首次复盘：文件总览 → 条目列表 → 对照审计（洞察↔实现） → 优先级清单 → 总结。使用复盘报告标准模板（十节结构）

#### Scenario: 优先级体系
- **WHEN** 发现问题
- **THEN** 按 P0（架构阻塞，立即修复）→ P1（支撑缺失，本迭代修复）→ P2（完备性提升，择机修复）→ P3（规范性修正，批量修复）四级分级，使用 ⚠️ 标记追踪文件中需后续处理的事项

#### Scenario: 不一致追踪
- **WHEN** 发现跨文件不一致
- **THEN** 以 IN-* 编号追踪，标注严重度和状态，纳入同步矩阵管理

---

### Requirement 4: 核心概念词典与三层递进
系统 SHALL 在 project.md 中建立集中式的核心概念词典，作为所有智能体的统一语义锚点，并对每个核心概念设计三层递进（表层→中层→深层）的实践路径。

#### Scenario: 概念词典初始化
- **WHEN** 首条锚点洞察完成后
- **THEN** 在 `project.md` 中定义至少 1 个核心概念（从锚点洞察提取），标注为「生长型文档」。每个概念包含：名称、定义、在体系中的位置、与其他概念的关系。随洞察积累逐步扩充至 ≥3 个概念

#### Scenario: 三层递进设计
- **WHEN** 核心概念定义完成后
- **THEN** 为每个概念设计三层递进：表层（入门：识别问题）→ 中层（深入：掌握方法）→ 深层（本然：内化为直觉）。每层包含：概念描述、识别信号、转换逻辑、实践指引

---

### Requirement 5: Skill 双层架构设计
系统 SHALL 根据项目的 AI 能力需求，按底层工具和约束体系划分 Skill，而非按功能划分。每个 Skill 遵循 references/ 五层递减组织（核心概念→结构规范→约束→模板→检查清单）。

#### Scenario: Skill 需求识别
- **WHEN** 同一类 AI 协作需求重复出现 ≥3 次
- **THEN** 识别该需求是否可被现有通用 Skill 覆盖，按"底层工具是否相同、约束体系是否可合并"两个维度判断是否需要独立 Skill。每个 Skill 必须拥有明确的"非目标"声明

#### Scenario: Skill 创建
- **WHEN** 确定 Skill 领域
- **THEN** 创建 `{项目根}/.agents/skills/{skill-name}/` 目录，包含 SKILL.md（五步工作流）和 references/（五层递减组织）。统一工作流范式：消化上下文 → 出具策略 → 执行生成 → 检查迭代 → 保存交付

---

### Requirement 6: 文件同步矩阵
系统 SHALL 在 workflows.md 中定义"文件更新后同步矩阵"，将一致性维护从模糊期望转化为精确的传播规则。

#### Scenario: 同步矩阵定义
- **WHEN** 项目文件体系初步建立
- **THEN** 在 `workflows.md` 中以表格形式定义"修改了什么 → 需要同步更新什么"的传播规则，覆盖洞察新增/修改/删除 → contributive.md / project.md / AGENTS.md 的同步链路

---

### Requirement 7: 知识萃取预留
系统 SHALL 在项目启动时预埋知识萃取机制，确保项目经验可定期提炼为可迁移方法论。

#### Scenario: 可迁移知识目录
- **WHEN** 首次复盘完成且可识别 ≥3 条可迁移模式
- **THEN** 创建 `{项目根}/.agents/docs/knowledge-transfer/` 目录，初始化 `transferable-patterns.md`（面向 Agent 的可执行模板）和 `transferable-methods.md`（面向人类的通用方法论）的骨架

## Implementation Notes

### 启动顺序（按方法论依赖关系）

> **设计原则**：「洞察驱动开发」要求洞察先于规范——先定义"为什么"，再从洞察中提取约束与概念。约束必须能标注洞察来源，概念词典随洞察积累自然生长，复盘按条件触发而非绑定天数。

```
第1天：内容锚定（洞察 → 概念）
  ├── .agents/docs/insights/{日期}-insights-01.md  ← 锚点洞察（≥1条）
  │    定义项目核心差异化定位，"为什么存在"
  └── .agents/project.md           ← 核心概念词典 v0.1
       ≥1个核心概念（从锚点洞察提取），标注为「生长型文档，随洞察积累扩充」

第2天：框架契约（概念 → 约束 → 规范）
  ├── .agents/constraints.md       ← 边界护栏（≥5条编号约束，标注洞察来源）
  ├── .agents/conventions.md       ← 工程契约（命名格式/文档规范/引用格式）
  ├── .agents/git.md               ← 版本契约（Conventional Commits + 粒度要求）
  └── .agents/docs/insights/{日期}-insights-02.md  ← 第2条洞察（基于核心概念展开）

第3天及以后：持续产出 + 首次复盘
  ├── 持续产出洞察至 ≥5 条        ← 按设计决策节奏自然增长，不强制绑定天数
  ├── .agents/workflows.md         ← 洞察≥3条 + 文件体系初具规模后创建
  ├── .agents/docs/reviews/{日期}-project-review-01.md  ← 洞察≥5条后触发首次复盘
  └── .agents/project.md           ← 随洞察增长持续扩充概念词典至 ≥3 个概念
```

### 按需触发项（不绑定天数）

| 产出物 | 触发条件 | 理由 |
|--------|---------|------|
| `.agents/roles/` | 洞察≥8条 + 发现需专人负责某领域质量 | 角色过早定义会预设不存在的需求 |
| `.agents/skills/` | 同一类 AI 协作需求重复≥3次 | Skill 应来自实践归纳而非提前规划 |
| `.agents/docs/knowledge-transfer/` | 首次复盘完成 + 可识别≥3条可迁移模式 | 没有足够实践经验时萃取的是空壳 |

### 最小可行骨架

如果时间极其紧迫，至少创建以下 3 个文件即可起步：

| 文件 | 最小内容 | 提供价值 |
|------|---------|---------|
| 首条锚点洞察 | 项目核心差异化定位 + 为什么存在 | 一切约束和概念的源头 |
| project.md | ≥1 个核心概念（从锚点洞察提取），标注为生长型文档 | 统一语义锚点 |
| constraints.md | ≥5 条编号约束（分类 + 标注洞察和概念来源） | 行为边界 |
