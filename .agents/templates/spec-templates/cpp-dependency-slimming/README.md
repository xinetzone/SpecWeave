# C++ 依赖瘦身 Spec 模板

> **模板版本**: v1.0 | 萃取自 Caffe→tvm-ffi 瘦身实战经验（2026-07-23）

## 适用场景

当你需要对一个 C++ 项目进行**依赖替换/瘦身**时使用此模板，典型场景：

- 用现代 C++ 标准库（C++17/20）替换 boost 等老旧依赖
- 用轻量级 FFI 库（如 tvm-ffi）替换 boost::python/pybind11 等重绑定
- 移除 glog/gflags 等日志/命令行库依赖
- 将大型 C++ 项目精简为仅保留核心推理/计算功能的轻量版本

## 使用方法

### 1. 复制模板

```bash
# 在项目 .trae/specs/ 下创建新spec目录
cp -r .agents/templates/spec-templates/cpp-dependency-slimming/ [项目]/.trae/specs/[你的spec名称]/
cd [项目]/.trae/specs/[你的spec名称]/
```

### 2. 填写 spec.md

按照 spec.md 中的顺序填写，**必须**完成前置预检章节（Step 0/0.5/1）：

| 步骤 | 章节 | 强制程度 | 目的 |
|------|------|---------|------|
| Step 0 | 环境预检 | 🔴 强制 | 确认构建平台和工具链可用，避免跨平台浪费时间 |
| Step 0.5 | API映射前置验证 | 🔴 强制 | 每个API映射写最小验证程序，禁止"边猜边写" |
| Step 1 | 依赖闭包分析 | 🔴 强制 | 用工具获取include闭包，禁止"我觉得需要哪些文件" |
| Overview~Open Questions | 常规PRD内容 | 🟡 必要 | 填充目标/背景/需求/验收标准 |

### 3. 填写 tasks.md

tasks.md 已内置 **11个标准任务**，按依赖顺序排列。使用时：

1. 替换 `[占位符]` 为实际内容
2. 调整任务的具体文件列表（基于Step 1的闭包分析结果）
3. 不需要的任务标记为不适用或删除
4. **严格遵守增量构建规则**（tasks.md 开头列出的4条强制规则）

### 4. 填写 checklist.md

checklist.md 包含 **13个验证维度**：

- 一~三：前置预检验证
- 四：旧依赖残留审计
- 五：兼容层验证
- 六：核心功能验证
- 七：FFI/绑定验证
- 八：上层语言适配验证
- 九：功能模块覆盖验证
- 十：不迁移组件隔离
- 十一：代码质量
- 十二：端到端验证
- 十三：反模式检查（避免已知陷阱）

使用时替换 `[占位符]`，在执行过程中逐项勾选。

## 核心原则（来自实战教训）

### 🔴 6个强制前置步骤（违反必返工）

1. **环境预检先于代码**：先确认目标平台编译器可用，再写代码。Linux原生项目优先WSL。
2. **API映射先于迁移**：每个旧API→新API映射必须写10-20行验证程序确认用法正确，不能凭文档猜测。
3. **闭包分析先于文件复制**：用 `g++ -H` 等工具获取真实include依赖树，一次性列出所有必需迁移文件，禁止"打地鼠式补文件"。
4. **兼容层先于业务迁移**：compat/头文件必须先写且先有独立测试，验证语义等价后才能迁移业务代码。
5. **增量构建而非大爆炸**：CMake禁止GLOB_RECURSE；先5-10个文件编译通过，再逐层添加，每步ctest。
6. **每步验证而非最后集成**：每个Task完成后立即运行对应测试，不带着错误进入下一步。

### 🟡 5个高风险反模式（检查清单第十三节）

| 反模式 | 后果 | 防范措施 |
|--------|------|---------|
| 未做环境预检就写代码 | Windows上写Linux原生代码浪费30%时间 | Step 0 强制检查 |
| "边猜边写"API | 5-6轮返工 | Step 0.5 最小验证程序 |
| 正向猜测头文件 | 打地鼠式补文件 | Step 1 g++ -H 闭包分析 |
| 兼容层不验证就迁移 | bug扩散到所有调用点 | Task 2 先测兼容层 |
| GLOB_RECURSE + 一次性加所有文件 | 编译错误叠加无法定位 | tasks.md 强制规则 |
