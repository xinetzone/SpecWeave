# 跨领域映射标准模板

> 本文档是产品学习/竞品分析Wiki中"对AI Agent/其他系统设计启示"章节的标准写作模板。
> 遵循**四段式结构**：相似性论证 → 原则映射 → 具体设计 → 边界声明。
>
> 使用方法：复制本模板到产品学习Wiki的洞察/启示章节，替换所有 `{{占位符}}`。
> 适用于：产品学习Wiki末尾的"AI Agent启示"章节、竞品分析的跨领域迁移章节。

---

## 写作原则

1. **先论证相似性，再做映射**：不要一上来就列启示，先说明"为什么原产品的经验适用于目标领域"——两个领域面对的核心问题是否同构？
2. **具体而非类比**：映射必须落到可执行的设计原则上，不能停留在"这就像XX"的比喻层面。
3. **双向思考**：不仅说"原产品怎么做→目标领域该怎么做"，也要说"原产品的什么做法不适用于目标领域"（边界声明）。
4. **结合本项目**：每条映射后应有"对本项目的启示"段落，指出当前项目中具体如何落地。
5. **诚实标注推演性质**：如果映射是理论推演而非已验证经验，必须明确标注，不能把推演当作事实。

### 边界声明与差异注意

> 📝 **必写章节**：跨领域映射不是生搬硬套。必须诚实指出哪些经验不能直接迁移，目标领域有哪些原产品不存在的特殊约束。这是区分"有洞察力的映射"和"牵强附会的类比"的关键。

- **{{差异点1}}**：{{原产品的XX前提假设在目标领域中不成立，因此YY做法不能直接照搬，需要做ZZ调整。}}
- **{{差异点2}}**：{{目标领域存在原产品所没有的XX约束（如延迟要求、数据规模、用户角色差异），因此...}}
- **{{待验证声明}}**：以上映射中，{{哪些}}是经过实践验证的经验迁移，{{哪些}}是基于结构相似性的理论推演，需要在实际项目中验证后方可确认有效性。

> 📝 **小结写法**：一段话提炼跨领域映射的核心洞见，不超过100字。点出最根本的相通原则，而不是重复前面的细节。

## 质量检查清单

完成跨领域映射章节后，对照以下标准自检：

- [ ] **相似性论证**：首段是否说明了两个领域的核心同构关系？还是直接跳到列启示？
- [ ] **结构完整性**：每个映射点是否包含"产品做法→设计原则→本项目启示"三层？
- [ ] **具体性**：设计原则是否可执行、可验证？还是停留在"应该重视XX"的空话层面？
- [ ] **数量合理**：映射点是否在3-6个之间？过多则发散，过少则浅薄。
- [ ] **边界声明**：是否诚实指出了不适用的情况和待验证的推演？还是只挑选支持自己观点的相似点？
- [ ] **本项目结合**：是否每条映射都指出了当前项目的具体落地方式？还是写了一堆与本项目无关的"通用真理"？
- [ ] **推演标注**：理论推演部分是否明确标注了"待验证"？是否暗示了所有映射都是已验证经验？

## 反模式警告

| 反模式 | 表现 | 问题 |
|--------|------|------|
| **空洞类比** | "Agent就像一个员工，所以也需要管理" | 比喻≠映射，没有具体设计原则，读者不知道怎么做 |
| **万能映射** | 把原产品的每个功能都映射到目标领域，牵强附会 | 稀释真正有价值的映射，降低整体可信度 |
| **单向崇拜** | 只说原产品做得好的地方，不提差异和不适用之处 | 跨领域迁移必然有边界，选择性忽略差异会导致错误决策 |
| **泛本项目** | "我们在构建AI Agent时应该借鉴"——没有具体到哪个模块/功能 | 启示无法落地，看完不知道该做什么 |
| **推演即事实** | 把理论推演的映射表述为已验证的最佳实践 | 误导后续决策，应该明确标注"待验证" |
