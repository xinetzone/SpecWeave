# Tasks

> 主题：retrospectives-insights（复盘与洞察萃取）
> 适用场景：任务复盘、问题诊断、经验萃取、方法论分析、模式提取

- [ ] Task 0: 复盘准备
  - [ ] SubTask 0.1: 收集复盘对象的完整上下文（spec.md、tasks.md、checklist.md、执行日志、commit 历史）
  - [ ] SubTask 0.2: 确认复盘类型（单项任务复盘/专题问题复盘/里程碑复盘/方法论总结）
  - [ ] SubTask 0.3: 明确复盘目标（问题根因分析/经验萃取/方法论沉淀/改进建议）
  - [ ] SubTask 0.4: 如为元文档（即复盘 spec 体系本身的文档），添加 `<!-- meta_type: retrospective -->` 标记避免一致性检查误报
  - [ ] SubTask 0.5: 确定复盘报告的存放位置（docs/retrospective/ 下对应主题目录）
  - [ ] SubTask 0.6: 确定复盘报告结构框架（参照已有复盘文档格式）

- [ ] Task 1: 信息收集与分析
  - [ ] SubTask 1.1: 回顾目标与计划（预期是什么？原始计划是什么？）
  - [ ] SubTask 1.2: 对照实际结果（实际发生了什么？哪些完成了？哪些未完成？）
  - [ ] SubTask 1.3: 识别差异（哪些符合预期？哪些超出预期？哪些出现问题？）
  - [ ] SubTask 1.4: 对问题进行根因分析（使用 5 Why 或鱼骨图方法，追溯到根本原因而非表面现象）
  - [ ] SubTask 1.5: 萃取成功经验（哪些做法有效？为什么有效？如何复制？）
  - [ ] SubTask 1.6: 总结失败教训（哪些做法无效？根本原因是什么？如何避免？）
  - [ ] SubTask 1.7: 识别可复用模式（模式名称、适用场景、使用方法、注意事项）

- [ ] Task 2: 报告撰写
  - [ ] SubTask 2.1: 编写复盘报告（放在 docs/retrospective/ 对应主题目录下）
  - [ ] SubTask 2.2: 报告结构包含：背景与目标、结果对比、差异分析、根因分析、经验教训、可复用模式、改进建议
  - [ ] SubTask 2.3: 标记可复用模式/资产（使用统一格式，便于后续提取到模式库）
  - [ ] SubTask 2.4: 提出改进建议（建议需具体、可执行、有明确负责人或触发条件，避免空泛的"加强沟通"类建议）
  - [ ] SubTask 2.5: 如有需要，创建后续跟进 spec（如发现需要新增治理规则、检查工具等）
  - [ ] SubTask 2.6: 引用源文档时使用正确的相对路径，区分自引用和外部引用
  - [ ] SubTask 2.7: 对敏感信息进行脱敏（内部路径、个人信息等）

- [ ] Task 3: 归档与联动
  - [ ] SubTask 3.1: 在 docs/retrospective/README.md 中登记新报告
  - [ ] SubTask 3.2: 如复盘结论需要变更规范/工具，创建对应 spec 并建立双向关联
  - [ ] SubTask 3.3: 如萃取了可复用模式，更新 docs/retrospective/patterns/ 目录或资产清单
  - [ ] SubTask 3.4: 验证所有引用路径正确（元文档的自引用不会被一致性检查误报为错误）
  - [ ] SubTask 3.5: 在对应主题 README.md 的执行看板中登记完成状态

# Task Dependencies

- Task 0 必须最先执行（充分收集上下文是高质量复盘的前提）
- Task 1 依赖 Task 0 完成
- Task 2 依赖 Task 1 完成（分析完成后才能写报告）
- Task 3 依赖 Task 2 完成
- **重要**：复盘不是批判，是学习。聚焦于过程和系统，而非个人；关注"我们能学到什么"而非"谁的错"
