# 通用子代理输出质量校验清单

> 基于向日葵开机盒子分析任务中发现的"子代理误将TodoWrite工具标签写入文档"问题，以及Claude Code上下文注入学习复盘中发现的"子代理输出截断/偷懒返回摘要"问题萃取，适用于所有使用`general_purpose_task`委托子代理的场景（文档编写、代码修改、分析报告、对比研究等）。
>
> 与 [subagent-wiki-delivery-checklist.md](subagent-wiki-delivery-checklist.md) 的区别：本清单是通用型检查，适用于所有子代理任务；后者是wiki创作专用检查，包含更多wiki特有的frontmatter/编号/TOML检查。

---

## 🔒 子代理委托Query强制约束（P0级）

**每次调用`general_purpose_task`时，必须在query末尾附加以下内容（根据任务类型裁剪）：**

```
【输出格式强制约束 - 必须遵守】

1. 内容纯净性：你输出的所有内容都将直接写入文档/作为分析结果，绝对禁止包含任何工具调用格式，包括但不限于：
   - 禁止<seed:tool_call>、<function>、<parameter>等XML工具调用标签
   - 禁止出现TodoWrite、Read、Write、Edit、Grep、Shell等工具名称
   - 禁止出现toolcall_result、toolcall_status等系统标签
   - 禁止出现<system-reminder>、<available_skills>、<toolcall_result>等上下文标签
   - 输出内容应该是纯粹的文档正文/代码/分析结果，不包含任何工具调用或系统提示格式

2. 输出完整性要求（极其重要）：
   - 必须返回完整、详细的内容，绝对禁止返回摘要性、概括性的总结
   - 不要在内容末尾写"以上是...的总结"、"综上所述..."等收尾性语句来提前结束
   - 如果任务包含多个部分（如"分析A、对比B、总结C"），必须每个部分都给出完整详细内容，不能只完成一部分就宣布完成
   - 我需要的是可直接使用的完整产出，不是大纲或摘要

3. 任务粒度原则：
   - 一次general_purpose_task调用只委派一个原子任务（单个Task）
   - 禁止将多个不相关的子任务合并委派给同一个子代理（如"完成任务3、4、5"）
   - 如果有多个独立任务，使用多个general_purpose_task并行调用

4. 单一职责：本次任务只完成<描述具体任务>，不要执行额外操作
```

---

## ✅ 子代理交付前自检（子代理应执行）

子代理在标记任务完成前，应快速确认：
- [ ] 输出中没有`<seed:tool_call>`标签
- [ ] 输出中没有`<function `或`<parameter `标签
- [ ] 输出中没有`TodoWrite`、`toolcall_result`、`toolcall_status`等工具相关字符串出现在正文上下文中
- [ ] 输出内容直接是最终交付物，不需要主代理再做清理
- [ ] 输出是完整详细内容，不是摘要或大纲（禁止"以上是总结"类提前收尾）
- [ ] 任务要求的每个部分都已完整覆盖，没有遗漏
- [ ] 内容长度和详细程度符合任务预期，没有过度精简

## 🔍 主代理验收检查（接收子代理产出后必须执行）

**检查方法**：使用Grep工具扫描子代理修改/创建的文件，搜索以下关键词；同时人工检查输出完整性：

| # | 检查项 | 检查方法 | 发现问题后的处理 |
|---|--------|---------|----------------|
| 1 | `<seed:tool_call>`标签 | Grep搜索 | 立即删除整个标签块及被污染的段落 |
| 2 | `<function `标签 | Grep搜索 | 检查是否为误插入的工具调用，若是则删除 |
| 3 | `<parameter `标签 | Grep搜索 | 同上 |
| 4 | `TodoWrite`等工具名 | Grep搜索 | 正文/Markdown文档中出现则删除（代码注释中合法） |
| 5 | `toolcall_result/toolcall_status` | Grep搜索 | 立即删除 |
| 6 | `<system-reminder>`标签 | Grep搜索 | 立即删除 |
| 7 | **输出完整性检查（P0）** | 人工检查 | 若返回摘要/截断/内容不全：第一次→重新委派并明确强调完整性要求；第二次→切换兜底策略 |
| 8 | **任务覆盖度检查** | 对照任务描述 | 若遗漏部分要求→返回补充或主会话补全 |

**P1级 - 长文档多轮委托专项检查**：

当一个文档经过多轮子代理分批编写后，不仅要检查新增部分，还要对整个文档做一次全量扫描（之前轮次的子代理可能在之前的批次中也插入了标签）。Grep命令示例：

```
Grep pattern: <seed:tool_call>|TodoWrite|<function |toolcall_result
Path: 目标文件
output_mode: content
-n: true
```

**验收耗时**：单个文件10秒；长文档全量扫描30秒。
**重要性**：如果不做此检查，被污染的标签会进入版本控制，后续阅读文档时会出现奇怪的工具调用代码块，严重影响文档质量。

## 📋 通用质量检查项（根据任务类型选用）

除了内容纯净性外，根据子代理任务类型，还应检查：

### 文档编写类任务
- [ ] 标题层级正确（从h1开始，不跳级）
- [ ] frontmatter格式正确（如果有）
- [ ] 列表和表格格式渲染正常
- [ ] 段落衔接自然，无重复内容
- [ ] 章节编号连续（如果使用编号）

### 代码修改类任务
- [ ] 修改后的代码语法正确（无明显语法错误）
- [ ] 没有遗留的调试代码（print语句、console.log、注释掉的代码块）
- [ ] 变量命名与文件现有风格一致
- [ ] 没有硬编码的敏感信息（密钥、密码、token）

### 分析报告类任务
- [ ] 结论有事实/数据支撑
- [ ] 引用的信息准确
- [ ] 逻辑结构清晰
- [ ] 改进建议可行动
- [ ] 输出是完整详细内容，不是摘要或提纲
- [ ] 要求覆盖的每个维度/每个部分都有充分展开

## 🛡️ 失败重试与兜底策略（P0级原则）

基于实证：子代理连续两次返回不合格结果（摘要/截断/内容不全）后，第三次重试成功的概率<30%，继续重试的ROI为负。必须遵循以下策略：

```
子代理返回结果
    │
    ▼
┌─────────────────────┐
│ 验收检查（8项）      │
└─────────────────────┘
    │
    ├── 通过 ──→ 继续
    │
    └── 失败（第一次）
         │
         ▼
    ┌─────────────────────┐
    │ 重新委派，在query中  │
    │ 明确强调问题点和    │
    │ 完整性要求          │
    └─────────────────────┘
         │
         ▼
    ┌─────────────────────┐
    │ 再次验收检查        │
    └─────────────────────┘
         │
         ├── 通过 ──→ 继续
         │
         └── 失败（第二次）
              │
              ▼
         ┌─────────────────────┐
         │ ⚠️ 立即切换兜底策略  │
         │ 不要重试第三次！     │
         │                     │
         │ 选项1（推荐）：      │
         │ 主会话基于已有内容   │
         │ 直接整合完成剩余工作 │
         │                     │
         │ 选项2：              │
         │ 将任务拆分为更小粒度 │
         │ 重新委派（每个子任务 │
         │ 一个独立子代理）     │
         └─────────────────────┘
```

**关键原则**：
1. **事不过二**：子代理最多重试1次（即总共2次尝试），第二次失败后必须切换策略
2. **主会话兜底**：主会话已经有第一次返回的部分内容作为基础，直接整合通常比反复重试更高效
3. **任务拆分重试**：如果确实需要子代理执行，将大任务拆分为更小的原子任务，每个任务一个子代理
4. **记录失败原因**：在复盘中记录子代理失败的场景和原因，持续优化委派策略

**根因理解**：子代理输出截断/偷懒返回摘要不是"子代理不够聪明"，而是**Agentic laziness**——所有单上下文窗口Agent系统的固有行为模式，在任务完成边界模糊（分析类任务）或任务粒度过大时更容易触发。这是架构层面的特性，不是靠"写更好的prompt"就能完全解决的，需要通过流程设计（任务拆分、兜底策略、对抗验证）来应对。
