## Overview
- **Summary**: 对 `.agents/scripts/` 目录下的 28 个脚本文件进行系统性合并可行性分析，从功能职责、代码结构、依赖关系、执行频率、维护便利性等多维度评估，输出具体的合并建议或不合并理由，并形成可执行的决策报告。
- **Purpose**: 当前 `.agents/scripts/` 目录包含 26 个可执行脚本（Python/PowerShell/Bash）+ 1 个常量模块 + 1 个说明文档。经过前一轮共享库提取（lib/ 子包），脚本间重复代码已大幅消除，但仍存在脚本数量多、功能分组不明确、部分工具职责重叠的问题。本次分析旨在回答"哪些脚本应该合并、哪些不该合并、如何合并"，为后续脚本架构优化提供决策依据。
- **Target Users**: 项目架构师、开发者、维护者（AI 智能体与人类开发者）。

## Goals
- 全面评估 28 个脚本文件的功能职责、代码规模、依赖关系、CI 调用频率
- 按功能相关性识别脚本分组，分析组内脚本合并的可行性
- 对每个候选合并组给出明确建议：合并（含合并方案）、不合并（含理由）、保持现状但增加统一入口
- 评估合并的潜在收益（代码量减少、维护简化、用户体验提升）与风险（命令兼容性、调用方影响、回归风险）
- 输出结构化的合并决策报告，作为后续重构工作的依据

## Non-Goals (Out of Scope)
- 本次分析**不执行**实际的代码合并或重构操作，仅输出分析报告和建议
- 不对 `lib/` 共享库内部结构做调整（前一轮重构已完成）
- 不修改 CI 脚本的调用逻辑（仅在建议中提及需要同步更新的地方）
- 不删除任何现有脚本文件（即使建议废弃，也只在报告中标记）
- 不新增脚本功能，仅评估现有功能的组织方式

## Background & Context
- `.agents/scripts/` 目录现有结构：
  - **检查类脚本（16个）**：`check-*.py` 系列，覆盖链接、Mermaid、Spec 格式/一致性、原子化、文件名、gitignore、vendor、角色权限、溯源、报告归类、复盘索引等
  - **生成类脚本（4个）**：`generate-*.py` 系列，生成导航表、看板、应用索引、测试骨架
  - **工作流脚本（3个）**：`build-ref-index.py`、`finalize-atomization.py`、`verify-atomization.py`
  - **成熟度管理（2个）**：`pattern-maturity-stats.py`、`scan-maturity-upgrades.py`
  - **专项工具（1个）**：`check-action-items.py`
  - **CI 入口（2个）**：`ci-check.ps1`、`ci-check.sh`
  - **常量模块（1个）**：`constants.py`
  - **文档（1个）**：`README.md`
  - **共享库**：`lib/` 子包（前一轮重构已建立）
- CI 流程通过 `ci-check.ps1`/`ci-check.sh` 调用 9 个脚本（7 个检查 + 2 个生成）
- 前一轮重构已提取共享模块（lib/project.py、lib/cli.py、lib/frontmatter.py、lib/markdown.py、lib/link_fixer.py、lib/spec/），消除了约 280 行重复代码
- 初步扫描已识别 7 个功能分组：原子化工作流（4）、链接管理（4）、Spec 检查（3）、模式成熟度（3）、索引导航生成（4）、仓库合规检查（5）、其他专项（3）

## Functional Requirements
- **FR-1**: 分析报告必须覆盖用户列出的全部 28 个文件，不得遗漏
- **FR-2**: 对每个文件提供结构化档案：LOC、功能职责、依赖关系、CLI 接口、CI 调用情况、核心函数清单
- **FR-3**: 基于功能相关性将脚本划分为若干组，每组给出合并不合并的明确建议
- **FR-4**: 对建议合并的分组，提供具体合并方案：目标文件名、子命令设计、参数映射、向后兼容策略
- **FR-5**: 对建议不合并的文件，给出不合并的核心理由（单一职责、独立演进、调用频率差异等）
- **FR-6**: 分析合并的收益评估：预估可消除的重复代码行数、维护成本降低、用户体验提升
- **FR-7**: 分析合并的风险评估：命令行兼容性、CI 脚本影响、调用方（文档/其他脚本）引用更新、回归风险
- **FR-8**: 提供分阶段实施建议：优先级排序、前置依赖、推荐实施顺序
- **FR-9**: 识别不适合合并但适合增加"统一入口"的分组（如 `sw-lint.py` 聚合多个独立检查）
- **FR-10**: 分析报告以 Markdown 格式输出，包含决策表格、Mermaid 架构图、实施路线图

## Non-Functional Requirements
- **NFR-1**: 分析必须基于代码事实（实际读取文件内容），禁止主观臆测
- **NFR-2**: 合并建议需遵循 Unix 哲学与单一职责原则的平衡：避免过度合并导致上帝对象，也避免过度拆分导致工具碎片化
- **NFR-3**: 建议方案需考虑向后兼容性：现有命令行调用方式应尽可能保留（可通过包装脚本或子命令别名实现）
- **NFR-4**: 分析报告应结构清晰、结论明确，每个建议附带理由和证据（文件/行号引用）
- **NFR-5**: 收益与风险评估需量化或分级（高/中/低），避免模糊表述

## Constraints
- **Technical**: 脚本均为 Python 3（除两个 CI shell 脚本）；使用 argparse；依赖已建立的 `lib/` 共享库
- **Business**: 分析报告是后续重构的决策依据，必须在实际重构前完成
- **Dependencies**: 前一轮共享库重构（refactor-scripts-shared-lib）已完成，lib/ 模块结构稳定；CI 脚本 ci-check.ps1/ci-check.sh 引用了部分脚本路径，合并时需同步更新

## Assumptions
- 合并后的脚本使用子命令模式（argparse subparsers），类似 `git commit`、`pytest -k` 的风格
- 合并不会改变脚本的核心功能逻辑，仅改变入口组织方式
- 废弃的一次性脚本（如 verify-atomization.py）可以在分析中标记为建议移除
- CI 脚本在实施合并时会同步更新，不是本次分析的阻塞项
- 用户期望得到一份"可执行的决策报告"，而非泛泛而谈的理论分析

## Acceptance Criteria

### AC-1: 全量文件覆盖
- **Given**: 用户列出的 28 个文件路径
- **When**: 分析报告完成
- **Then**: 报告中包含全部 28 个文件的结构化档案，无遗漏
- **Verification**: `programmatic`
- **Notes**: 通过文件清单对照检查

### AC-2: 功能分组合理性
- **Given**: 28 个脚本的功能职责已梳理
- **When**: 进行功能分组
- **Then**: 分组基于功能内聚性（高内聚、低耦合），组内脚本有明确的协作关系或数据依赖，组间边界清晰
- **Verification**: `human-judgment`
- **Notes**: 审查分组逻辑是否合理，是否存在跨组脚本被遗漏或错分

### AC-3: 合并建议明确可执行
- **Given**: 每个功能分组
- **When**: 输出合并建议
- **Then**: 对每组给出"合并/不合并/统一入口"三选一的明确结论；若合并，给出目标文件名、子命令设计、参数映射表；若不合并，给出核心理由
- **Verification**: `human-judgment`
- **Notes**: 建议需具体到可以直接创建重构任务的程度

### AC-4: 收益风险量化评估
- **Given**: 合并方案已确定
- **When**: 进行收益风险分析
- **Then**: 收益按"代码去重/维护简化/体验提升"三个维度评估（高/中/低）；风险按"兼容性/回归/调用方影响"三个维度评估（高/中/低），并给出缓解措施
- **Verification**: `human-judgment`

### AC-5: 向后兼容策略
- **Given**: 建议合并的分组
- **When**: 设计合并方案
- **Then**: 每个合并方案包含向后兼容策略：现有 CLI 命令如何保留（包装脚本/子命令别名/兼容层）
- **Verification**: `human-judgment`

### AC-6: 实施路线图
- **Given**: 所有合并建议已确定
- **When**: 输出实施建议
- **Then**: 提供分阶段实施路线图（P0 高优先级/P1 中优先级/P2 低优先级），标注前置依赖和推荐顺序
- **Verification**: `human-judgment`

### AC-7: Mermaid 可视化
- **Given**: 分析报告
- **When**: 呈现分组与合并建议
- **Then**: 报告包含至少 2 个 Mermaid 图：合并前脚本依赖关系图（或功能分组图）、合并后架构图；Mermaid 语法符合项目安全编码五规则
- **Verification**: `programmatic`
- **Notes**: 使用 check-mermaid.py 验证

### AC-8: 报告格式规范
- **Given**: 最终分析报告
- **When**: 交付
- **Then**: 报告为 Markdown 格式，包含决策总表、详细分析、合并方案、风险评估、实施路线图等章节；文件路径引用使用相对路径；无 file:/// 绝对路径
- **Verification**: `programmatic`
