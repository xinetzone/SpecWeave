# 提示词萃取系统 Spec

## Why
当前 AI 开发中提示词（Prompt）质量直接影响模型输出效果，但缺乏系统化的手段对提示词进行收集、清洗、特征提取、质量评估和自动优化。需要一个全流程自动化系统，将原始提示词输入转化为高质量、结构化的优化提示词，提升 AI 开发效率和输出质量。

## What Changes
- 新增提示词输入模块，支持批量文件导入（CSV/JSON/TXT/Markdown）和单条手动输入
- 新增文本预处理模块，实现清洗、标准化、去冗余
- 新增特征提取模块，识别关键指令、约束条件、预期输出，支持 Markdown 结构解析
- 新增质量评估模块，基于清晰度、完整性、可执行性等维度量化评分
- 新增优化生成模块，对低质量提示词自动增强
- 新增流水线编排器，串联各模块实现全流程自动化
- 新增 Streamlit 可视化界面，展示萃取过程和结果，支持导出
- 新增单元测试和集成测试，覆盖各模块核心逻辑与端到端流程

## Impact
- Affected specs: 无（全新系统）
- Affected code: 新增 `prompt_extraction/` 目录，包含所有模块代码、测试、UI 和配置文件

## ADDED Requirements

### Requirement: 提示词输入机制
系统 SHALL 提供两种提示词输入方式：批量导入和单条输入。

#### Scenario: 批量导入 CSV 文件
- **WHEN** 用户通过 UI 上传包含提示词的 CSV 文件
- **THEN** 系统解析 CSV 文件，提取所有提示词记录，展示导入摘要（总数、字段映射）

#### Scenario: 批量导入 JSON 文件
- **WHEN** 用户通过 UI 上传 JSON 文件（对象数组格式）
- **THEN** 系统解析 JSON 并提取提示词列表，展示导入结果

#### Scenario: 批量导入 TXT 文件
- **WHEN** 用户上传 TXT 文件（每行一条提示词）
- **THEN** 系统按行解析，每行作为一条独立提示词处理

#### Scenario: 批量导入 Markdown 文件
- **WHEN** 用户上传 Markdown 文件（.md）
- **THEN** 系统按 Markdown 标题层级拆分，每个一级/二级标题区块作为一条独立提示词处理

#### Scenario: 单条手动输入
- **WHEN** 用户在 UI 文本框中输入提示词并提交
- **THEN** 系统接收该条提示词并进入处理流程

#### Scenario: 空输入处理
- **WHEN** 输入为空或文件无有效内容
- **THEN** 系统返回明确错误提示，不触发后续处理

### Requirement: 文本清洗与标准化
系统 SHALL 对原始提示词文本执行清洗和标准化处理，去除冗余信息与格式干扰。

#### Scenario: 去除多余空白
- **WHEN** 输入文本包含多余空格、制表符、换行
- **THEN** 系统将连续空白规范化为单个空格，去除首尾空白

#### Scenario: 去除格式标记
- **WHEN** 文本包含 Markdown、HTML 等格式标记
- **THEN** 系统去除或转换为纯文本，保留核心语义内容

#### Scenario: Markdown 结构保留
- **WHEN** 输入为 Markdown 格式文本
- **THEN** 系统在清洗冗余格式的同时保留标题层级、列表结构、代码块标记等语义结构信息，供特征提取模块使用

#### Scenario: 统一编码与字符集
- **WHEN** 输入文本包含全角/半角混用、特殊 Unicode 字符
- **THEN** 系统统一转换为半角字符，规范化标点符号

#### Scenario: 去除无关内容
- **WHEN** 文本包含 URL、email、代码块等非核心指令内容
- **THEN** 系统识别并标记为附属信息，核心指令文本单独保留

### Requirement: 特征提取
系统 SHALL 从预处理后的提示词中提取关键特征，包括指令、约束条件和预期输出。

#### Scenario: 识别核心指令
- **WHEN** 提示词包含"请"、"要求"、"需要"等动作指令
- **THEN** 系统提取并分类为指令列表，标注指令类型

#### Scenario: 识别约束条件
- **WHEN** 提示词包含"必须"、"不能"、"限制"、"字数"等限制性描述
- **THEN** 系统提取约束条件列表，标注约束类型（格式约束、内容约束、风格约束）

#### Scenario: 识别预期输出格式
- **WHEN** 提示词包含"返回"、"输出"、"格式"等输出描述
- **THEN** 系统提取预期输出格式描述，标注输出类型（JSON/文本/列表/代码）

#### Scenario: 无明确特征
- **WHEN** 提示词缺少明确的指令或约束
- **THEN** 系统返回空特征集，并在质量评估中标记为低完整性

#### Scenario: Markdown 结构解析
- **WHEN** 提示词使用 Markdown 格式组织（含标题、列表、代码块等）
- **THEN** 系统利用 Markdown 结构信息辅助特征提取：标题识别为指令分区，列表项识别为约束或步骤，代码块识别为预期输出示例

### Requirement: 质量评估指标体系
系统 SHALL 基于清晰度、完整性、可执行性三个维度对提示词进行量化评分。

#### Scenario: 清晰度评估
- **WHEN** 系统评估提示词清晰度
- **THEN** 基于文本长度、结构层次、歧义度计算 0-100 分，并生成改进建议

#### Scenario: 完整性评估
- **WHEN** 系统评估提示词完整性
- **THEN** 基于是否包含指令、约束、上下文、示例、输出格式等要素计算 0-100 分

#### Scenario: 可执行性评估
- **WHEN** 系统评估提示词可执行性
- **THEN** 基于指令是否可操作、约束是否可验证、输出是否可判定计算 0-100 分

#### Scenario: 综合评分
- **WHEN** 三个维度评估完成
- **THEN** 系统按加权计算生成综合评分（清晰度 30%、完整性 40%、可执行性 30%），并标注等级（优/良/中/差）

### Requirement: 优化算法
系统 SHALL 对低质量提示词（综合评分低于某阈值）自动进行优化增强。

#### Scenario: 触发优化
- **WHEN** 提示词综合评分低于 60 分
- **THEN** 系统自动触发优化流程，生成优化后的提示词

#### Scenario: 补充缺失要素
- **WHEN** 提示词缺少输出格式说明
- **THEN** 优化算法根据上下文推断并补充合理的输出格式

#### Scenario: 消歧增强
- **WHEN** 提示词存在歧义表述
- **THEN** 优化算法将其改写为更精确、可操作的表述

#### Scenario: 结构调整
- **WHEN** 提示词结构混乱、信息杂乱
- **THEN** 优化算法重组为"指令-约束-上下文-输出格式"的标准结构

#### Scenario: 优化前后对比
- **WHEN** 优化完成后
- **THEN** 系统展示优化前后的对比 diff，并标注改进点

### Requirement: 模块间数据流转
系统 SHALL 确保各模块之间数据流转顺畅，实现从原始输入到优化后输出的全流程自动化。

#### Scenario: 完整流水线执行
- **WHEN** 用户提交提示词后
- **THEN** 系统按 输入→预处理→特征提取→质量评估→（条件触发）优化生成 的顺序自动执行

#### Scenario: 模块间数据传递
- **WHEN** 上游模块完成处理
- **THEN** 下游模块接收标准化的数据结构（PromptRecord），无需手动转换

#### Scenario: 流水线错误处理
- **WHEN** 任一模块处理失败
- **THEN** 系统记录错误日志，跳过该条提示词，继续处理剩余条目，最后汇总失败记录

### Requirement: 可视化界面
系统 SHALL 提供 Web 可视化界面，展示萃取过程和结果，并支持导出优化后的提示词。

#### Scenario: 首页仪表盘
- **WHEN** 用户打开系统界面
- **THEN** 展示输入入口（批量上传/单条输入）、处理流程概览和统计摘要

#### Scenario: 单条提示词详情页
- **WHEN** 用户点击某条提示词
- **THEN** 展示原始文本、预处理结果、提取特征、质量评分（含雷达图）、优化对比（如有）

#### Scenario: 批量处理结果列表
- **WHEN** 批量导入处理完成
- **THEN** 以表格形式展示所有提示词的处理结果，支持按评分排序和筛选

#### Scenario: 导出功能
- **WHEN** 用户点击导出按钮
- **THEN** 系统支持导出为 CSV 格式（含原始提示词、优化后提示词、评分、特征），也支持仅导出优化后提示词

### Requirement: 测试体系
系统 SHALL 包含单元测试和集成测试，验证各模块处理效果和全流程稳定性。

#### Scenario: 单元测试覆盖
- **WHEN** 运行单元测试
- **THEN** 每个模块的核心函数均有对应的测试用例，覆盖正常输入、边界条件和异常输入

#### Scenario: 集成测试
- **WHEN** 运行集成测试
- **THEN** 验证端到端流水线：输入→预处理→特征提取→质量评估→优化生成，确认数据流转正确

#### Scenario: 不同类型提示词测试
- **WHEN** 使用多种类型的提示词（简短/冗长、清晰/模糊、完整/残缺）进行测试
- **THEN** 系统能正确处理各类输入，验证各模块鲁棒性
