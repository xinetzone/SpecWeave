# docs → .agents/docs 迁移 Spec

## Why
当前仓库的 `docs/` 承担人类阅读文档主目录，而 `.agents/docs/` 承担智能体侧文档容器，两者在物理位置上分离，但在治理上缺少统一的上层收纳关系。需要基于“物理聚合不等于语义混淆”的第一性原理，将原 `docs/` 整体迁入 `.agents/docs/` 容器内，同时保留“人类可读”与“智能体专属”两类文档的独立定位、可访问性与可验证性。

## What Changes
- 将 `d:\spaces\SpecWeave\docs` 整体迁移到 `d:\spaces\SpecWeave\.agents\docs`
- 迁移方式明确为“将原 `docs/` 目录内容并入 `.agents/docs/`”，而不是生成新的嵌套路径 `.agents/docs/docs/`
- 在迁移前对原 `docs/` 全量备份，并记录文件清单、数量、哈希与引用基线
- 全量梳理仓库内对原 `docs/` 路径的配置、链接与引用，迁移后统一更新到新路径
- 对 `.agents/docs/` 下“人类文档”与“智能体专属文档”建立清晰边界，避免职责混淆与无规则混放
- 为 `.agents/docs/README.md` 或等效索引文档定义最小边界声明模板，降低实现时的自由发挥空间
- 为迁移后的目录与文件补充“归类决策表”，统一判定哪些内容默认归为人类文档、哪些默认归为智能体专属文档
- 运行文件迁移完整性校验、链接有效性校验与人工抽查，确保迁移无遗漏、无损坏、无断链

## Impact
- Affected specs: `docs-restructure`
- Affected code: `docs/`、`.agents/docs/`、`AGENTS.md`、`.agents/` 下引用 `docs/` 的规范文档、`.trae/specs/` 中引用主仓库文档路径的规格文档、链接/迁移校验脚本的输入路径

## ADDED Requirements
### Requirement: 迁移后保持物理聚合与语义分层并存
系统 SHALL 将原根目录 `docs/` 迁移到 `.agents/docs/`，并以 `.agents/docs/` 作为迁移后的文档承载位置；迁移后需通过目录组织、索引或说明性文档明确区分“面向人类阅读”的文档与“面向智能体读取”的文档，避免语义混淆。

#### Scenario: 人类文档与智能体文档边界可辨识
- **WHEN** 用户或智能体读取 `.agents/docs/` 目录结构
- **THEN** 可以明确识别迁移后的文档承载位置是 `.agents/docs/`
- **AND** 目录结构或索引说明仍能区分人类文档与智能体专属文档的定位

### Requirement: 迁移实现不得额外生成嵌套 docs 根
系统 SHALL 以 `.agents/docs/` 作为唯一目标根目录执行迁移，不得在迁移结果中额外生成 `.agents/docs/docs/` 这一嵌套根路径。

#### Scenario: 目标根目录唯一且无二次包裹
- **WHEN** 执行目录移动方案
- **THEN** 原 `docs/` 的内容直接落位到 `.agents/docs/` 下
- **AND** 迁移结果中不存在新的嵌套根目录 `.agents/docs/docs/`

### Requirement: 独立定位必须通过显式边界规则保留
系统 SHALL 在迁移后通过 `README`、索引说明、AGENTS 路由或等效边界文档显式声明：哪些路径属于原人类文档体系，哪些路径属于智能体专属文档，且两者不得在无说明的情况下混放或互相覆盖。

#### Scenario: 目录语义边界有显式声明
- **WHEN** 迁移完成并更新导航文件
- **THEN** `.agents/docs/` 下存在明确的边界说明
- **AND** 读者可判断某个目录或文件属于“人类文档”还是“智能体专属文档”
- **AND** 原有智能体文档路径若存在，不会被迁移内容覆盖

### Requirement: 边界声明采用最小模板落地
系统 SHALL 在 `.agents/docs/README.md` 或等效索引文档中提供一段最小边界声明模板，至少覆盖“目录定位、适用对象、维护原则、禁止事项、查找入口”五类信息，确保实现人员无需自行发明说明结构。

#### Scenario: README 使用统一边界声明模板
- **WHEN** 维护者创建或更新 `.agents/docs/README.md`
- **THEN** 文档包含“本目录定位”“人类文档范围”“智能体专属文档范围”“维护原则”“快速查找入口”五个最小章节
- **AND** 文案明确说明：原 `docs/` 迁移内容现统一位于 `.agents/docs/`，但其读者定位仍以人类为主
- **AND** 文案明确说明：智能体专属文档不得被人类文档内容无说明覆盖

### Requirement: 目录归类必须遵循统一决策表
系统 SHALL 为 `.agents/docs/` 下的迁移内容提供统一的目录归类决策表，基于“主要读者、主要用途、发现方式、维护方式”四个维度判定目录或文件属于“人类文档”还是“智能体专属文档”。

#### Scenario: 维护者依据决策表判断目录归属
- **WHEN** 维护者新增、迁移或审查 `.agents/docs/` 下的目录或文件
- **THEN** 可以依据决策表快速判断其默认归属类别
- **AND** 不需要依赖个人经验或历史记忆做归类
- **AND** 遇到边界模糊内容时，有明确的升级判断规则

### Requirement: 边界模糊内容必须按升级规则处理
系统 SHALL 定义边界模糊内容的升级判断规则：当某目录或文件同时满足人类文档与智能体专属文档特征时，必须先判定“主要读者”和“主要触发方式”，必要时在 README 或 AGENTS 路由中补充显式说明后方可落位。

#### Scenario: 混合型文档按升级规则归类
- **WHEN** 某文件既被人阅读，也被 agent 或脚本读取
- **THEN** 先以主要读者与主要触发方式决定默认归属
- **AND** 若仍存在歧义，则在边界说明中补充归类理由与使用方式
- **AND** 未补充说明前不得直接混放到模糊位置

### Requirement: 迁移前必须完成可恢复备份与基线记录
系统 SHALL 在任何移动操作前完成对原 `docs/` 全量备份，并生成包含原始路径、文件数量、文件大小、内容哈希、关键引用方清单的迁移基线。

#### Scenario: 迁移前生成完整备份
- **WHEN** 迁移任务进入执行阶段
- **THEN** 先生成可恢复的 `docs/` 备份副本与基线清单
- **AND** 在备份和基线未完成前不得执行目录移动

### Requirement: 整目录迁移不得造成内容缺失或损坏
系统 SHALL 以保持目录结构语义的方式将 `docs/` 内容迁移到 `.agents/docs/` 下，并在迁移后通过文件数量、相对路径映射、内容哈希三重校验确认所有文件完整无损。

#### Scenario: 迁移后验证文件完整性
- **WHEN** 原 `docs/` 完成迁移至 `.agents/docs/`
- **THEN** 新路径下的文件总数与迁移前一致
- **AND** 所有文件内容哈希与基线一致
- **AND** 不存在遗漏、重复或损坏文件
- **AND** 原 `docs/` 的目录层级与分类语义保持可识别

### Requirement: 所有路径引用必须同步到新文档根
系统 SHALL 全量更新仓库内引用原 `docs/` 根路径的 Markdown 链接、导航索引、配置说明、规则文档、规格文档与脚本输入路径，使其指向 `.agents/docs/` 或新的相对路径。

#### Scenario: 仓库内不再残留旧文档根引用
- **WHEN** 迁移完成并执行路径修复
- **THEN** 仓库内不再残留指向旧根目录 `docs/` 的失效引用
- **AND** 所有跨文件相对路径在新目录层级下仍然有效

### Requirement: 迁移后仍可分别支持人类访问与智能体读取
系统 SHALL 在迁移完成后验证两类使用方式均正常工作：人类可以通过新路径浏览原文档体系，智能体程序可以继续基于 `.agents/docs/` 容器识别与读取所需文档。

#### Scenario: 双重访问方式通过验证
- **WHEN** 执行迁移后验证
- **THEN** 人类可读文档在 `.agents/docs/` 下可正常导航和访问
- **AND** 智能体专属文档的读取路径、目录结构与发现逻辑保持可用

### Requirement: 迁移闭环必须包含程序化校验
系统 SHALL 在迁移完成后运行至少一轮文件迁移校验与一轮链接校验，并保留结果，作为本次迁移完成的验收依据。

#### Scenario: 校验脚本全部通过
- **WHEN** 迁移与引用修复完成
- **THEN** `check-move` 类校验确认文件迁移完整
- **AND** `check-links` 类校验确认内部链接有效
- **AND** 若校验失败，则迁移任务不得视为完成

## MODIFIED Requirements
### Requirement: 人类可读文档的规范存储根路径
系统 SHALL 将人类可读文档的规范存储根路径从仓库根目录 `docs/` 调整为 `.agents/docs/`，并通过结构、索引与边界说明维持其独立语义。

### Requirement: 文档交叉引用的路径计算基准
系统 SHALL 以 `.agents/docs/` 为新的人类文档根路径重新计算跨文件相对路径，确保深度变化后的链接仍可正确解析。

## 边界声明模板（建议落地到 `.agents/docs/README.md`）

```md
# .agents/docs 文档边界说明

## 本目录定位
`.agents/docs/` 是统一文档容器，同时承载两类内容：
- 原 `docs/` 迁移而来的人类可读文档
- 原生面向智能体读取的专属文档

## 人类文档范围
- 来源：仓库原根目录 `docs/`
- 主要读者：项目维护者、开发者、贡献者
- 使用方式：阅读说明、知识检索、导航查阅、人工维护

## 智能体专属文档范围
- 来源：`.agents/` 体系内原生 agent 文档
- 主要读者：agent、自动化脚本、路由/治理流程
- 使用方式：上下文装载、规则读取、程序发现

## 维护原则
- 迁移后的物理位置统一为 `.agents/docs/`
- 人类文档与智能体文档必须通过目录结构或索引声明区分
- 新增文档时必须先判断其主要读者，再决定归类位置

## 禁止事项
- 禁止生成嵌套根目录 `.agents/docs/docs/`
- 禁止在无说明的情况下混放人类文档与智能体专属文档
- 禁止覆盖既有智能体专属文档路径

## 快速查找入口
- 人类文档：按原 `docs/` 迁移后的主题结构导航
- 智能体文档：按 agent 路由、规则、协议或索引导航
```

## 目录归类决策表

| 判定维度 | 默认归为人类文档 | 默认归为智能体专属文档 |
|---|---|---|
| 主要读者 | 项目维护者、开发者、贡献者、普通读者 | agent、自动化脚本、路由流程、治理流程 |
| 主要用途 | 阅读说明、知识检索、学习导航、人工维护 | 上下文装载、规则读取、程序发现、自动执行 |
| 发现方式 | 通过 README、主题索引、人工浏览进入 | 通过 AGENTS 路由、规则入口、脚本逻辑发现 |
| 维护方式 | 以人工编写、人工审阅、人工导航为主 | 以规则约束、程序读取、流程调用为主 |

## 归类规则

1. 若“主要读者”和“主要用途”都偏向人工阅读，则默认归为人类文档。
2. 若“主要读者”和“主要触发方式”都偏向 agent 或脚本调用，则默认归为智能体专属文档。
3. 若目录或文件来自原根目录 `docs/`，迁移后默认继承“人类文档”属性，除非另有显式说明。
4. 若目录或文件来自 `.agents/` 原生体系，默认继承“智能体专属文档”属性，除非另有显式说明。
5. 若出现混合型内容，先判断“主要读者”，再判断“主要触发方式”；仍模糊时，必须在 `.agents/docs/README.md` 或 AGENTS 路由中追加归类说明。

## 推荐默认映射

| 来源/类型 | 默认归属 | 备注 |
|---|---|---|
| 原 `docs/` 迁移内容 | 人类文档 | 迁移后物理位置改变，语义归属不变 |
| `.agents/` 下规则、协议、路由、角色、系统提示词 | 智能体专属文档 | 以 agent 装载和流程执行为主 |
| 供人阅读的知识库、教程、复盘、说明文档 | 人类文档 | 即使位于 `.agents/docs/` 下，也保持人类读者优先 |
| 供脚本读取的配置说明、执行协议、加载入口 | 智能体专属文档 | 若同时面向人类，需补充边界说明 |

## 抽查样本预设

迁移完成后的归类验证，优先从以下 5 类样本中选取至少 1 个实例进行抽查；若某类在当前仓库不存在，可记录“本仓库无该类样本”并以下一类补足总样本数：

| 样本类型 | 预期默认归属 | 抽查目的 |
|---|---|---|
| 原 `docs/` 迁移后的主题入口文档（如 README、主题索引） | 人类文档 | 验证迁移后入口导航仍以人类阅读为主 |
| 原 `docs/` 迁移后的知识类文档（如 knowledge、tutorial、guide） | 人类文档 | 验证知识资产不会因进入 `.agents/docs/` 而被误判为智能体专属 |
| 原 `docs/` 迁移后的复盘/报告类文档 | 人类文档 | 验证高频被人工查阅的报告仍保留人类读者优先属性 |
| `.agents/` 原生规则/协议/路由类文档 | 智能体专属文档 | 验证既有 agent 运行依赖文档未被迁移内容覆盖或稀释定位 |
| 混合型样本（既被人阅读又可能被流程引用的说明文档） | 按升级规则判定 | 验证边界模糊样本会触发归类说明，而不是被直接混放 |

## 抽查规则

1. 抽查样本总数不少于 5 个。
2. 优先覆盖“人类文档”和“智能体专属文档”两类样本，不能只验证单一侧。
3. 至少包含 1 个混合型样本，验证升级判断规则是否真正可用。
4. 每个样本都需记录：样本路径、默认归属、判定依据、是否需要补充边界说明。
5. 若某样本的实际用途与决策表冲突，必须回到 README 或 AGENTS 路由补充显式说明，并重新验证。

## 抽查记录模板

迁移后的归类验证记录建议以独立小节或附表形式保留，至少采用以下字段：

```md
## 归类抽查记录

| 样本路径 | 样本类型 | 默认归属 | 实际用途 | 判定依据 | 是否混合型 | 是否需要补充边界说明 | 处理结果 |
|---|---|---|---|---|---|---|---|
| `示例路径` | 主题入口文档/知识类文档/规则协议文档等 | 人类文档/智能体专属文档 | 简述实际用途 | 主要读者/用途/发现方式/维护方式 | 是/否 | 是/否 | 通过/补充说明后通过/待修复 |
```

## 抽查记录填写规则

1. `样本路径` 使用仓库相对路径，便于复查。
2. `样本类型` 优先使用抽查样本预设中的 5 类名称。
3. `默认归属` 只能填写“人类文档”或“智能体专属文档”。
4. `判定依据` 必须至少引用两个维度，例如“主要读者 + 主要用途”。
5. `是否混合型` 为“是”时，`处理结果` 不能只写“通过”，必须体现是否补充了边界说明。
6. 若 `是否需要补充边界说明` 为“是”，则必须在 README 或 AGENTS 路由中补充后再回填结果。

## REMOVED Requirements
### Requirement: 仓库根目录 `docs/` 作为人类文档的物理承载目录
**Reason**: 根目录 `docs/` 与 `.agents/docs/` 的物理分离不利于在统一文档容器下执行备份、治理、索引与程序化校验。
**Migration**: 原 `docs/` 子树整体迁移到 `.agents/docs/`；所有旧路径引用统一修复为新路径，后续以新路径作为唯一维护入口。
