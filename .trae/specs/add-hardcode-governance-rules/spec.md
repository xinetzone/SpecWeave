# 硬编码治理规则体系 Spec

## Why

项目中 `.agents/` 目录已具备角色定义、协作协议、工作流、工具规范等完整的智能体治理框架，但缺少对代码中硬编码问题的系统性治理规则。此前已完成 `hardcode-retrospective-system` 复盘，识别了大量硬编码问题并制定了分类型重构方案，但缺少一套持续有效的规则体系来预防新硬编码的引入，确保治理效果的长期可持续性。

## What Changes

在 `.agents/` 下新增 `rules/` 目录，建立完整的硬编码治理规则体系，包含以下核心模块：

- **识别标准**：明确定义什么是硬编码，覆盖固定字符串、数值、路径、URL、正则模式、编码常量等全部类别
- **允许场景**：规定允许使用硬编码的特殊场景清单，以及申请例外的审批流程
- **替代方案**：制定每种硬编码类型的推荐替代方案与实施指南（配置文件、环境变量、常量定义、枚举、资源文件等）
- **检测机制**：建立硬编码问题的自动化检测、人工审查与定期报告机制
- **执行验证**：确保所有规则具备可执行性和可验证性，提供量化衡量标准
- **索引文档**：提供 README 总览与导航，指导开发团队正确实施硬编码管理策略

同时更新 `AGENTS.md` 的路由表，将 `rules/` 纳入上下文路由体系。

## Impact

- Affected specs: `hardcode-retrospective-system`（从一次性复盘演进为持续治理）
- Affected code: 无直接代码修改，仅新增 `.agents/rules/` 下的规范文档
- Affected docs: `.agents/rules/` 全部新增文件、`AGENTS.md` 路由表更新

## ADDED Requirements

### Requirement: 硬编码识别标准定义

系统 SHALL 提供明确的硬编码识别标准，覆盖全部常见硬编码类别，使审查者和开发者能够准确判断一段代码是否属于硬编码。

#### Scenario: 覆盖全部硬编码类别
- **WHEN** 查阅识别标准文档
- **THEN** 必须包含以下类别的定义与示例：
  - 固定字符串（错误消息、日志文本、UI 标签、提示信息）
  - 固定数值（业务阈值、超时时间、分页大小、权重系数）
  - 固定路径（文件路径、目录路径、资源路径）
  - 固定 URL/端点（API 地址、第三方服务地址）
  - 固定编码值（字符编码、格式标识、协议标识）
  - 固定正则表达式模式
  - 固定颜色/样式值（CSS 颜色、字体大小）
  - 固定配置参数（连接池大小、线程数、缓存过期时间）

#### Scenario: 区分硬编码与合理常量
- **WHEN** 判断某段代码是否为硬编码
- **THEN** 必须提供明确的判断标准，区分"属于该模块固有逻辑的常量"与"应外部化管理的硬编码"
- **AND** 通过正例与反例对比说明边界情况

### Requirement: 允许硬编码场景与审批流程

系统 SHALL 规定允许使用硬编码的有限场景清单，并制定例外申请的审批流程。

#### Scenario: 允许场景清单
- **WHEN** 查阅允许场景文档
- **THEN** 必须明确列出允许硬编码的场景，至少包含：
  - 数学/物理常量（如 π、e、光速等不可变自然常量）
  - 协议/标准定义的固定值（如 HTTP 状态码、MIME 类型）
  - 单次使用且无复用可能的临时值（需附加条件限制）
  - 性能关键路径中经评审确认的硬编码优化
- **AND** 每个允许场景必须附带边界条件与使用限制

#### Scenario: 例外审批流程
- **WHEN** 开发者需要在非允许场景使用硬编码
- **THEN** 必须经过以下流程：
  1. 开发者在代码注释中使用 `HARDCODE-EXCEPTION:` 标记声明例外原因
  2. 提交至 reviewer 进行专项审查
  3. reviewer 评估替代方案的可行性
  4. 若确无合理替代方案，由 architect 或 orchestrator 最终批准
  5. 例外记录归档至项目硬编码例外清单
- **AND** 每项例外必须设定有效期，到期后强制重新评估

### Requirement: 替代方案指南

系统 SHALL 制定每种硬编码类别的推荐替代方案与实施指南。

#### Scenario: 替代方案匹配表
- **WHEN** 查阅替代方案指南
- **THEN** 必须提供硬编码类型与替代方案的映射表：
  | 硬编码类型 | 推荐替代方案 |
  |---|---|
  | 配置参数 | 配置文件（YAML/JSON/TOML）+ 环境变量覆盖 |
  | 业务常量 | 常量类/枚举定义 |
  | 路径 | 路径常量集中管理 |
  | URL/端点 | 配置文件 + 环境变量 |
  | 错误/提示信息 | 资源文件/消息字典（支持国际化） |
  | UI 文本 | 国际化资源文件（i18n） |
  | 正则模式 | 模式常量库 |
  | 颜色/样式 | 主题变量/设计令牌（Design Tokens） |
- **AND** 每种替代方案必须包含具体的代码示例

#### Scenario: 模板与脚手架
- **WHEN** 开发者需要实施某种替代方案
- **THEN** 指南必须提供可复制的模板/脚手架代码
- **AND** 模板必须遵循项目现有代码风格与目录约定

### Requirement: 检测与报告机制

系统 SHALL 建立硬编码问题的多层级检测与报告机制。

#### Scenario: 自动化检测
- **WHEN** 代码提交或 CI 流程执行
- **THEN** 必须集成硬编码检测工具/脚本，至少包括：
  - 正则模式扫描（检测疑似硬编码的字符串、数值、路径模式）
  - 自定义规则集（针对项目特定模式）
  - 与现有 CI 检查脚本（`ci-check.ps1`/`ci-check.sh`）集成
- **AND** 检测结果必须分类为 ERROR（必须修复）、WARNING（建议修复）、INFO（提示关注）

#### Scenario: 人工审查
- **WHEN** 代码审查流程执行
- **THEN** reviewer 的审查清单必须包含硬编码检查项
- **AND** 硬编码检查项的评分标准必须量化

#### Scenario: 定期报告
- **WHEN** 每个迭代周期结束
- **THEN** 必须生成硬编码问题趋势报告，包含：
  - 本期新增硬编码数量与类型分布
  - 本期已修复硬编码数量
  - 硬编码例外清单的状态更新
  - 趋势对比（与上一周期的变化）

### Requirement: 执行与验证规则

系统 SHALL 确保所有治理规则具备可执行性和可验证性。

#### Scenario: 规则可执行性
- **WHEN** 查阅任意治理规则
- **THEN** 每条规则必须包含：
  - 明确的触发条件（什么情况下应用此规则）
  - 具体的执行步骤（如何操作）
  - 可量化的衡量标准（怎样判定合格）

#### Scenario: 规则验证机制
- **WHEN** 需要验证规则执行效果
- **THEN** 必须提供验证手段：
  - 自动化脚本验证（可集成至 CI）
  - 检查点清单（供人工审查使用）
  - 基准数据对比（与上一周期对比变化量）

### Requirement: 文档索引与导航

系统 SHALL 提供完整的 README 索引文档，指导开发团队正确实施硬编码管理策略。

#### Scenario: 文档完整性
- **WHEN** 进入 `.agents/rules/` 目录
- **THEN** README.md 必须包含：
  - 规则体系架构图（Mermaid）
  - 各规则文档的链接与简介
  - 快速导航表（按场景和角色分类）
  - 使用指南（不同角色应如何查阅本规则体系）

### Requirement: AGENTS.md 路由表更新

系统 SHALL 将 `rules/` 规则体系纳入 AGENTS.md 的上下文路由表。

#### Scenario: 路由表更新
- **WHEN** AGENTS.md 被智能体加载
- **THEN** 必须包含 `.agents/rules/` 的入口，使智能体在执行相关任务时能够定位到硬编码治理规则
