# xinet 目录系统性内容萃取与归档方案 Spec

## Why

`d:\AI\.temp\.chaos\tests\xinet` 是一个未经治理的多项目聚合沙箱，包含 37 个嵌套 Git 仓库、9 个一级子项目、双份矛盾的 AI 指引文档以及明文密钥泄露风险。该目录呈现四维熵增（项目熵、版本熵、文档熵、配置熵），导致认知成本高、安全风险大、内容价值难以识别。需要制定系统性的萃取与归档方案，从混沌中提取精华、消除风险、建立有序管理体系。

## What Changes

- 对 `xinet` 目录进行全量文件扫描，建立内容分类体系（代码、文档、配置、备份、凭证、测试等）
- 建立内容价值评估标准，筛选具有保留价值的精华内容
- 设计三层归档体系：核心归档（高价值）、参考归档（中等价值）、临时归档（待清理）
- 制定归档命名规范，确保文件可追溯、可检索
- 建立定期回顾与更新机制，确保归档内容的时效性与准确性
- 创建归档索引清单与迁移记录，确保可追溯性
- 清理敏感凭证与冗余备份副本

## Impact

- Affected specs: 无现有 spec 受影响
- Affected code: `docs/retrospective/reports/insight-extraction/meta-methodology/retrospective-xinet-chaos-multiproject-analysis-20260625/`（需补充归档执行记录）
- Affected files: `d:\AI\.temp\.chaos\tests\xinet\` 下的全部文件（涉及扫描、分类、迁移、清理）

## ADDED Requirements

### Requirement: 全量文件扫描与分类

系统应扫描 `xinet` 目录下所有文件，建立内容分类体系。

#### Scenario: 文件分类完整
- **WHEN** 执行扫描操作
- **THEN** 生成包含所有文件的分类清单，按以下类别划分：
  - 代码文件（`.py`、`.ts`、`.js`、`.vue`、`.html`、`.css` 等）
  - 文档文件（`.md`、`.txt`、`.json` 配置说明等）
  - 配置文件（`package.json`、`pyproject.toml`、`config.yaml`、`tsconfig.json` 等）
  - 凭证文件（含密钥的配置文件）
  - 备份文件（`*-bak*`、`*-backup*`、`.git/` 嵌套仓库等）
  - 测试文件（`.test.*`、`test_*.py`、`.ipynb` 等）
  - 构建产物（`node_modules/`、`dist/`、`_build/` 等）
  - 其他（二进制文件、图片、工作区配置等）

### Requirement: 内容价值评估标准

系统应建立内容价值评估标准，用于筛选具有保留价值的精华内容。

#### Scenario: 价值评估可执行
- **WHEN** 对文件进行价值评估
- **THEN** 采用以下评估维度：
  - **完整性**：功能是否完整、文档是否齐全
  - **复用性**：是否包含可复用的模式、组件、工具函数
  - **创新性**：是否包含独特的设计理念、架构模式或技术方案
  - **关联性**：与项目其他部分的关联程度
  - **安全性**：是否包含敏感信息、是否存在安全风险
  - **时效性**：内容是否仍具有参考价值、是否已过时

#### Scenario: 价值分级明确
- **WHEN** 执行价值评估后
- **THEN** 文件被分为三个等级：
  - **高价值**：完整功能、可复用模式、核心架构、无安全风险
  - **中等价值**：部分功能可用、有参考价值但非核心
  - **低价值**：备份副本、重复内容、已过时、安全风险高

### Requirement: 分层归档体系

系统应设计三层归档体系，按价值等级进行有序存储。

#### Scenario: 归档结构完整
- **WHEN** 查看归档目录
- **THEN** 归档目录包含以下三层结构：
  - **核心归档（core/）**：存放高价值内容，包括核心代码、架构文档、可复用模式
  - **参考归档（reference/）**：存放中等价值内容，包括参考文档、辅助工具、配置示例
  - **临时归档（temporary/）**：存放待清理内容，包括备份副本、临时文件、待审查项

#### Scenario: 归档位置规范
- **WHEN** 执行归档操作
- **THEN** 归档文件存储至 `docs/retrospective/archives/xinet/` 目录下对应的层级

### Requirement: 归档命名规范

系统应制定统一的归档命名规范，确保文件可追溯、可检索。

#### Scenario: 文件名符合规范
- **WHEN** 文件被归档
- **THEN** 文件名遵循以下规则：
  - 采用 `kebab-case`（小写字母 + 连字符）命名
  - 包含原始路径标识，格式为 `<category>-<original-path>-<timestamp>`
  - 保留文件原始扩展名
  - 避免空格、中文、下划线或驼峰命名

### Requirement: 定期回顾与更新机制

系统应建立定期回顾与更新机制，确保归档内容的时效性与准确性。

#### Scenario: 回顾周期明确
- **WHEN** 查看归档机制文档
- **THEN** 包含以下回顾周期：
  - **月度回顾**：检查临时归档内容，决定保留或删除
  - **季度回顾**：评估核心归档内容的时效性，更新过时内容
  - **年度回顾**：全面审查归档体系，优化分类标准与存储结构

#### Scenario: 更新机制可执行
- **WHEN** 发现归档内容需要更新
- **THEN** 执行以下流程：
  - 创建更新记录，说明更新原因与内容变化
  - 更新归档文件内容
  - 更新归档索引清单
  - 通知相关角色

### Requirement: 归档索引与追溯

系统应建立归档索引清单，记录每个归档文件的来源、价值等级、归档时间与更新记录。

#### Scenario: 索引清单完整
- **WHEN** 查看归档索引
- **THEN** 索引清单包含以下字段：
  - 原始路径
  - 归档路径
  - 价值等级
  - 分类标签
  - 归档时间
  - 更新记录（时间、原因、变更内容）

### Requirement: 敏感信息清理

系统应识别并清理归档过程中的敏感信息。

#### Scenario: 敏感信息已移除
- **WHEN** 审查归档文件
- **THEN** 不包含明文 API Key、密码、令牌等敏感信息
- **AND** 敏感配置文件已加入 `.gitignore` 或使用环境变量替代

### Requirement: 质量检查标准

系统应建立归档质量检查标准，确保归档过程的规范性与准确性。

#### Scenario: 质量检查可执行
- **WHEN** 执行质量检查
- **THEN** 检查以下项目：
  - 文件分类是否准确
  - 价值评估是否合理
  - 归档路径是否符合规范
  - 文件命名是否符合规范
  - 敏感信息是否已清理
  - 索引清单是否完整
  - 归档文件是否可正常打开
