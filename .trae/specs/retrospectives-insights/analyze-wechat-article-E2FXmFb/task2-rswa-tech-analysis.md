# R-SWA核心技术深度解析：从"逐页失忆"到"超长程连贯解析"

## 一、"逐页失忆"问题的根源

### 1.1 现象：长文档处理的"切割-缝合"怪圈

当前主流OCR模型在处理长文档时，普遍采用一种笨拙的策略：将几十页的文档切割成一个个单页任务，逐页处理后再用外部调度器把结果勉强拼接起来。就像在运行一个`for`循环——处理完一页就清空记忆，再从头开始下一页。

这种方式虽然能勉强工作，但本质上只是工程上的权宜之计，离真正的连续智能理解还差着一大截。模型无法建立跨页的上下文关联，前后内容连贯性差，容易出现信息遗漏或重复。

### 1.2 技术根源：KV缓存的滚雪球式增长

"逐页失忆"的真正元凶，是**标准多头注意力（MHA）机制下KV缓存随输出长度线性增长**的固有缺陷。

在Transformer架构中，每生成一个新token，都需要将当前token的Key（K）和Value（V）向量添加到KV缓存中。后续每个token生成都要 attend 到缓存中**所有**历史token。这意味着：

- **内存占用随输出长度线性增长**：输出1000个token时KV缓存是1000份，输出10万个token时就是10万份——像滚雪球一样越滚越大
- **计算复杂度平方级上升**：每步注意力计算需要与所有历史token交互，解码步数越多，每步耗时越长
- **最终触达硬件极限**：内存吃不消、速度越来越慢，模型无法一次性处理完整长文档

**类比理解**：这就像一个人抄书时，每写一个字都要把之前写过的所有字重新读一遍——写第一页时还很快，写到第十页时速度慢得无法忍受，最后只能写一页撕一页，重新开始。

正是这个技术瓶颈，逼着所有模型不得不采用逐页处理策略，导致频频"失忆"。

## 二、"软遗忘"（Soft Forgetting）：人类抄书的认知智慧

### 2.1 人类长程任务的注意力模式

人类在进行超长连续任务（如抄书、翻译、长时转录）时，采用的是一种完全不同的认知策略，这正是R-SWA设计灵感的来源。

想象一下你正在手抄一本书：

- **眼睛盯着原书**：原文始终全局可见，确保抄的内容准确无误
- **余光瞄着刚写下的几行**：保持书写的连贯性和上下文衔接
- **聚焦即将写的下一个字**：当前注意力集中在即将输出的内容上
- **早些写过的内容自然淡出**：不需要一直记着半小时前写的具体文字，认知负荷维持在低位

### 2.2 "软遗忘"的核心机制

这种"该忘就忘"的能力被称为**"软遗忘"（soft forgetting）**，它有三个关键特征：

1. **参考信息全局保留**：原文（参考材料）始终完整可见，不随时间遗忘
2. **工作记忆滑动窗口**：只保留最近一段时间的输出上下文，更早的内容自然淡出
3. **认知负荷恒定**：无论任务多长，工作记忆容量保持不变，因此能持续稳定输出

正是靠这种机制，人类才能在极低认知负荷下完成超长程连续任务——抄完一整本书、翻译几百页文档、连续转录数小时音频，都不会因为"内存不足"而崩溃。

## 三、R-SWA：参考滑动窗口注意力机制详解

R-SWA（Reference Sliding Window Attention，参考滑动窗口注意力）是百度Unlimited OCR提出的核心技术，它精确复刻了人类抄书时的注意力模式，从根本上解决了KV缓存滚雪球问题。

### 3.1 非对称注意力设计

R-SWA最精妙的地方在于其**非对称的注意力范围设计**，对"参考token"和"输出token"采用完全不同的注意力策略：

#### 参考token侧：全局可见（Full Attention）

每生成一个token，R-SWA都会 attend 到**全部参考token**，包括：
- 整张图像的所有视觉token（文档页面的视觉编码）
- 提示词（prompt）token

**设计意图**：保证模型在生成过程中始终"看得见"完整原文，就像抄书时眼睛始终盯着原书。这从机制上确保了信息源的完整性，不会因为生成过程而丢失原文信息。

#### 输出侧：滑动窗口（Sliding Window）

在输出token这一侧，模型只回看前面**最近的128个token**，就像抄书时只瞄一眼刚写的那几行。

**设计意图**：维持输出的连贯性和局部上下文，避免语法断裂、格式混乱，但不需要记住所有历史输出。

### 3.2 实现方式：固定容量KV队列

在工程实现上，Unlimited OCR将模型的**所有注意力层全部替换为R-SWA**，从而把传统的无限增长KV缓存改造成一个**固定容量的队列**：

```
传统MHA的KV缓存：[K1, V1], [K2, V2], [K3, V3], ..., [Kn, Vn]  →  随n线性增长
R-SWA的KV缓存：  [K_{n-127}, V_{n-127}], ..., [Kn, Vn]        →  始终只有128个
```

队列工作机制：
1. 每生成一个新token，将其K/V向量加入队尾
2. 如果队列长度超过128，将队首（最老的token）挤出队列
3. 队列大小始终保持128个token的固定容量

### 3.3 效果：内存恒定，延迟平线

这种设计带来了革命性的效果：

- **内存占用恒定**：无论输出1万token还是10万token，KV缓存大小始终不变（128个输出token + 全部参考token）。参考token在prefill阶段计算一次后就固定不变，不随解码步数增长
- **延迟稳定**：每步注意力计算量恒定，不会因为输出变长而变慢
- **超长程支持**：理论上支持无限长输出，只要参考token在上下文窗口内

**类比理解**：R-SWA就像给工作台装了一个固定大小的"最近草稿区"——你始终能看到全部参考资料（原文），但工作台上只保留最近写的几行草稿。写满了就把最早的草稿移走，工作台永远有空间，速度永远不变。

## 四、标准MHA vs R-SWA：核心差异对比

| 对比维度 | 标准MHA（多头注意力） | R-SWA（参考滑动窗口注意力） |
|---|---|---|
| **注意力范围** | 对称注意力：所有token attend到全部历史token | 非对称注意力：参考token全局可见，输出token仅看最近128个 |
| **KV缓存大小** | 随输出长度线性增长：O(n)，n为已输出token数 | 固定容量：始终为128个输出token + 参考token数量 |
| **内存增长模式** | 滚雪球式增长：输出越长内存占用越大，最终OOM | 恒定内存：输出1万token和10万token内存占用相同 |
| **长文档表现** | 被迫逐页切割处理，跨页上下文断裂，"失忆"严重 | 一次推理读完几十页，上下文连贯，无信息丢失 |
| **单步延迟表现** | 随解码步数稳步攀升，输出越长越慢 | 从头到尾延迟平线，速度恒定 |
| **计算复杂度** | O(n²)，n为序列长度 | 近似O(1)（解码阶段每步计算量固定） |
| **信息完整性** | 历史输出完整保留，但内存代价极高 | 参考信息完整保留，早期输出"软遗忘"，符合认知规律 |
| **适用场景** | 短文本生成、对话等不需要超长输出的场景 | 长文档OCR、长文本转录、连续翻译等超长程解析任务 |

## 五、DeepEncoder：极致视觉压缩技术

R-SWA解决了输出侧的内存增长问题，但要真正实现几十页文档的一次推理，还需要视觉侧的高效压缩——这就是**DeepEncoder**发挥作用的地方。

### 5.1 技术来源

DeepEncoder最初源自DeepSeek OCR项目，是专门为文档视觉编码设计的高效视觉编码器。

### 5.2 压缩能力

DeepEncoder的核心能力是实现极高倍率的视觉token压缩：

- **输入**：一张1024×1024分辨率的PDF页面图像
- **输出**：仅仅256个视觉token
- **压缩率**：高达16倍（相比传统视觉编码器）

### 5.3 关键特性：视觉token不参与状态转移

DeepEncoder与R-SWA配合时还有一个至关重要的设计：**视觉token在R-SWA下不参与状态转移**。

这意味着：
- 视觉token在prefill阶段一次性编码完成后，就作为固定的参考信息存在
- 它们不会进入输出侧的滑动窗口队列，不被"挤出"
- 无论解码过程多长、输出多少token，图像信息永远清清楚楚，不会随解码过程逐渐退化

### 5.4 协同效果

DeepEncoder的极致压缩 + R-SWA的恒定缓存，两者配合产生了"1+1>2"的效果：
- 单页仅需256个视觉token，20页文档仅需5120个视觉token
- 在标准的32K上下文窗口里，一次前向推理就能转录数十页文档
- 从第一页到最后一页，视觉信息始终完整清晰

## 六、Flash Attention v3延迟测试：平线 vs 爬坡

报告中的Flash Attention v3延迟测试结果直观地展示了R-SWA的威力。

### 6.1 测试现象

- **DeepSeek OCR（标准MHA）**：随着解码步数增加，每步耗时**稳步攀升**——曲线从左下到右上持续增长
- **Unlimited OCR（R-SWA）**：从头到尾**一条平线**，纹丝不动——无论解码到第几步，每步延迟都保持恒定

### 6.2 结果解读

这个测试结果完美验证了R-SWA的设计预期：

1. **MHA的延迟爬坡**正好对应KV缓存的滚雪球增长——每多生成一个token，下一步就要多处理一个历史token，计算量线性增加，延迟自然稳步上升
2. **R-SWA的延迟平线**则证明了固定容量队列的有效性——每步都只处理128个输出token，计算量不随解码步数变化，因此延迟保持恒定

### 6.3 量化性能差距

在输出达到6144个token时：
- Unlimited OCR的TPS（每秒生成token数）：7847
- DeepSeek OCR的TPS：5822
- **性能差距高达35%**——而且输出越长，差距会越大

这意味着在长文档场景下，R-SWA不仅解决了"能不能跑完"的问题，还带来了显著的速度优势。

## 七、总结：R-SWA的范式意义

R-SWA不仅仅是一个OCR领域的技术优化，它代表了一种对长程连续任务的全新思考方式：

1. **从"记住一切"到"该忘就忘"**：借鉴人类认知的"软遗忘"机制，不再追求保留所有历史信息，而是区分"参考信息"和"工作记忆"
2. **非对称注意力的智慧**：打破传统注意力的对称设计，对不同类型的token采用不同的注意力策略
3. **恒定资源换无限长度**：用固定的内存和计算资源，支持理论上无限长的输出序列

正如论文末尾所言，R-SWA是一种**通用解析机制**，OCR只是第一站。未来它可能被推广到ASR（语音识别）、机器翻译、长文本摘要等所有需要超长程连续输出的场景——如果真能做到，那改变的将不只是OCR，而是整个序列生成范式。
