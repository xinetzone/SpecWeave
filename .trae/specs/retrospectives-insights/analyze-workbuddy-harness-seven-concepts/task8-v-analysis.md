## 1. V四要素在Harness Engineering中的体现

V（对抗性审查）的公理是"主动寻找证伪证据的认知防御机制，系统性构造反例暴露确认偏误盲区"。在文章中，V不是一个独立模块，而是贯穿Harness Engineering整个体系的横切概念——从约束层的权限检查，到反馈层的测试验证，再到编排层的多角色分工，V无处不在。

### 1.1 证伪导向：Evaluator独立验收+GAN对抗评估思路

V的"证伪导向"要求"主动构造反例而非证明正确"。文章中Anthropic的P-G-E三角色设计是证伪导向的工业级典范。

**借鉴GAN对抗评估思路：**

> 方案借鉴 GAN 的对抗评估思路，用 Claude Agent SDK 构建三个角色：Planner（把一句话需求展开成完整规格，定范围但不指定实现细节）、Generator（按 sprint 逐功能实现、用 git 版本控制、提交前先自检）、Evaluator（独立验收 Agent，用 Playwright 像真实用户一样操作运行中的应用，逐条核查、把 bug 定位到行号和原因后打回）。
>
> ——第6.2节（article-content.md:481）

GAN（生成对抗网络）的核心思想就是"生成器"和"判别器"的对抗训练：生成器努力生成以假乱真的样本，判别器努力区分真假。Anthropic把这个思路直接迁移到Agent工程：
- **Generator** = 生成器：努力产出"看起来正确"的实现
- **Evaluator** = 判别器：努力找出实现中的bug（证伪）
- 两者对抗迭代，最终产出更可靠的结果

这不是"Generator做完后Evaluator来证明正确"，而是"Evaluator主动攻击Generator的产出，努力证明它有bug"——这是纯粹的证伪思维。

**"自我评估不可靠"的认知是证伪思维的前提：**

> Anthropic（第二篇：Harness design for long-running application development）：在前一篇基础上发现两个更深的问题——接近上下文上限时模型会降低完成标准、**自我评估不可靠（Agent 评价自己产出时倾向给正面结论，前端设计这类缺少确定性测试的任务尤其明显）**。
>
> ——第6.2节（article-content.md:479）

这句话深刻揭示了为什么需要独立的V（对抗性审查）：
- 认知偏误是内置的——Agent评价自己的产出时，天生倾向于给正面结论（确认偏误）
- 越是缺少确定性测试的任务（如前端设计、业务逻辑），自我评估越不可靠
- 因此必须有独立于执行者的"攻击者"角色，专门负责寻找问题

证伪导向的核心逻辑是：**我们不应该试图"证明自己是对的"（这几乎总能找到支持证据），而应该努力"证明自己是错的"——如果经过系统性攻击仍找不到错，才暂时认为是对的。**

### 1.2 多角攻击：三权分立+计算型/推断型反馈分层

V的"多角攻击"要求"多角色独立审查避免单一视角盲区"。文章中两个机制实现了多角攻击：

**（1）Planner/Generator/Evaluator三权分立**

三个角色权责清晰、视角独立，从不同角度"攻击"产出：

| 角色 | 职责 | 攻击视角 | 不做什么 |
| --- | --- | --- | --- |
| **Planner** | 把需求展开成完整规格，定义"什么是正确" | 从需求角度攻击——规格是否完整？是否模糊？是否有歧义？ | 不指定实现细节，不写代码 |
| **Generator** | 按sprint逐功能实现，提交前自检 | 从实现角度攻击——代码是否符合规格？是否有明显错误？ | 不自己验收，不降低标准 |
| **Evaluator** | 独立验收，像真实用户一样操作，定位bug到行号后打回 | 从用户角度攻击——功能是否真的可用？是否符合规格？边界情况是否处理？ | 不修改代码，不告诉Generator怎么改 |

三权分立如何避免单一视角盲区？
- Planner如果同时做实现，会倾向于"怎么方便实现就怎么定义规格"
- Generator如果同时做验收，会倾向于"我写的代码肯定没问题"（自我评估不可靠）
- Evaluator如果同时参与实现，会倾向于"我知道这里怎么实现的，应该没问题"
- 三者独立，才能从需求→实现→用户三个视角形成交叉火力

**（2）计算型vs推断型反馈分层——不同类型的"攻击角度"**

> 这两类控制还可以按执行方式分成计算型和推断型。**计算型**控制由确定性程序执行，例如 LSP、类型检查、linter、单元测试、结构测试、依赖扫描、脚本和 codemod，优点是快、便宜、可重复，适合在 Agent 每次修改后反复运行。**推断型**控制依赖模型做语义判断，例如 Review Agent、架构审查 Agent、AI judge 和设计评估，优点是能覆盖"是否过度设计""是否误解需求""是否符合团队约定"这类难以写成规则的问题，缺点是更慢、更贵，也更不确定。
>
> ——第6.3节（article-content.md:499）

计算型和推断型提供了两种完全不同的攻击角度：
- **计算型攻击（确定性武器）**：语法错误、类型错误、风格问题、测试失败、架构分层违规——这些是"非黑即白"的问题，有明确的判定标准，程序可以自动攻击
- **推断型攻击（语义武器）**：需求误解、过度设计、不符合团队约定、用户体验问题——这些是"灰色地带"的问题，需要语义理解，必须用模型来攻击

两类武器互补：
- 只用计算型：能抓住机械错误，但抓不住语义错误（代码能跑但功能不对）
- 只用推断型：能抓住语义错误，但成本高、速度慢、不确定，不适合高频检查
- 两者结合：计算型做高频第一道防线，推断型做低频高价值第二道防线

### 1.3 偏差防御：系统性对抗三类认知偏误

V的"偏差防御"要求"系统性对抗确认偏误/权威崇拜等认知缺陷"。文章中至少识别并防御了三类关键偏误：

**偏误1：确认偏误——"实现和测试可能共享同一个误解"**

> **实现和测试可能共享同一个误解。** 同一个 Agent 既写实现又写测试，对需求的理解偏差会同时进两边。测试全过，仍不能证明实现符合原始业务意图。
>
> ——第08章（article-content.md:607）

确认偏误的表现：人（或Agent）一旦形成了对需求的某种理解，就会倾向于寻找支持这个理解的证据，而忽略反对证据。如果Generator自己写测试，测试会不自觉地围绕"我理解的需求"来写，而不是围绕"真实需求"来写——这样测试全过也不代表正确。

文章的防御机制：
- **角色分离**：Evaluator独立于Generator，不共享Generator的理解语境
- **Planner先定规格**：在实现前先由Planner把需求展开成完整规格，作为独立的"正确标准"
- **承认局限**：文章诚实指出即使P-G-E分离，如果Planner一开始就理解错了，Generator和Evaluator可能共享同一个误解——这是V的已知局限（详见第4节）

**偏误2：权威崇拜——"System Prompt只能引导不能强制"**

> **System Prompt 只能引导，不能强制。** 权限校验、Sandbox、Approval Gate、审计仍由模型外部的系统执行。
>
> ——第2.2节（article-content.md:123）

"提示词即权威"是Agent工程中常见的偏误：误以为把规则写在System Prompt里，模型就一定会遵守。但模型是概率生成系统，Prompt只能提高遵守规则的概率，不能100%保证。如果把安全约束只放在Prompt里，就相当于"把规则写在墙上指望所有人都遵守"——没有强制力。

文章的防御机制（多层防御，不依赖单一权威）：
- **第1层（引导）**：System Prompt/规则文件/Skills——引导模型往正确方向走
- **第2层（约束）**：Sandbox/Allowlist/Denylist——模型即使想违规也做不到（环境限制）
- **第3层（审批）**：Approval Gate——高风险操作必须经过人工确认
- **第4层（审计）**：Audit log——所有操作留痕，事后可追溯可问责

这是典型的"纵深防御"（defense in depth）思想：不依赖任何单一权威来源（即使是System Prompt），而是多层机制重叠防御。

**偏误3：覆盖偏误——"编辑前时间戳校验防止覆盖用户修改"**

> 编辑前的时间戳校验：Agent 改文件前比较"上次读取时间"和"文件最后修改时间"，若读取后被用户改过就拒绝写入、要求重读，避免覆盖最新修改。
>
> ——第6.3节（article-content.md:521）

"覆盖用户修改"是Agent协作中常见的偏误：Agent基于自己读取的文件版本进行修改，如果用户（或另一个Agent）在这期间修改了文件，Agent直接写入就会覆盖别人的改动。这本质上是"以我读到的版本为准"的认知偏误——假设世界在我读取后没有变化。

文章的防御机制：
- **乐观锁模式**：读取时记录时间戳，写入前比较时间戳
- **失败安全**：如果时间戳不匹配，拒绝写入并要求重读——宁可让Agent重新读一遍，也不冒险覆盖用户修改
- **把错误反馈给Agent**：不只是静默拒绝，而是告诉Agent"文件已被修改，请重新读取后再改"——这同时是反馈机制

### 1.4 审计可溯：Audit log/Git历史/进度文件交接

V的"审计可溯"要求"所有验证过程可复现可追溯可审计"。文章中三个机制保证了可追溯性：

**（1）Audit log留痕**

> 再加上 Audit log 让所有动作留痕、可追溯回放。
>
> ——第6.3节（article-content.md:521）

Audit log记录什么？
- 谁（哪个Agent/哪个用户）在什么时间做了什么操作
- 调用了什么工具，参数是什么，结果是什么
- 哪一步出了错，错误信息是什么
- 审批流程是怎样的，谁批准的

为什么这对V重要？
- 验证过程需要可复现：出了问题能回放当时的决策过程
- 责任需要可追溯：知道哪个环节出了问题，谁（或哪个机制）应该负责
- 攻击样本需要可积累：Audit log本身就是最好的"攻击案例库"，可以用来改进Harness

**（2）Git历史版本**

> Generator（按 sprint 逐功能实现、用 git 版本控制、提交前先自检）
>
> ——第6.2节（article-content.md:481）
>
> 用进度文件 + Git 历史做交接、恢复和回滚。
>
> ——第6.2节（article-content.md:471）

Git提供的可追溯性：
- 每一次变更都有提交记录，知道改了什么、什么时候改的、为什么改（提交信息）
- 可以回滚到任意历史版本——V发现问题后可以快速恢复
- 可以对比不同版本的差异，理解bug是在哪次提交引入的
- 跨会话交接时，新Agent可以通过Git历史理解之前做了什么

**（3）进度文件交接**

> 一开始就把要做的事拆成 200+ 条具体行为描述的功能清单（JSON）、每条标 pass/fail 并禁止删条目或降标准；一次只处理一项任务；统一的启动脚本 init.sh；除单测外用浏览器自动化做端到端验证；用进度文件 + Git 历史做交接、恢复和回滚。
>
> ——第6.2节（article-content.md:471）

进度文件的可追溯性：
- 任务拆分成可检查的清单，每条标pass/fail——清晰知道什么通过了、什么没通过
- 禁止删条目或降标准——防止"遇到困难就降低验收标准"的偏误
- 跨Agent/跨会话交接时，进度文件是"状态契约"——新接手的Agent不需要从头理解，看进度文件就知道做到哪了、什么通过了、什么没通过

## 2. Anthropic P-G-E三角色深度分析：V最精彩的工业级实现

Planner/Generator/Evaluator三角色分工是文章中V理念最集中、最成熟的工业级实现，值得深度分析。

### 2.1 三角色的权责与V四要素对应

| 角色 | 在V体系中的定位 | 对应V要素 | 核心职责 |
| --- | --- | --- | --- |
| **Planner** | 证伪标准的定义者 | 证伪导向（先定义什么是"错"） | 把模糊需求展开成可验证的规格，定义验收标准——相当于先说好"满足什么条件算bug" |
| **Generator** | 被攻击的执行者 | （执行层，不是V层） | 按规格实现+自检——产出被攻击的对象 |
| **Evaluator** | 独立的证伪执行者 | 证伪导向+多角攻击+偏差防御 | 独立验收、主动攻击、定位bug、打回——V的核心角色 |

**Planner的关键作用：先定义"什么是正确"，攻击才有标准**

V（对抗性审查）的一个前提是：必须先有明确的"正确标准"，否则攻击就没有方向。Planner的价值正在于此：
- 把一句话需求（如"做一个待办事项应用"）展开成200+条具体的、可验证的行为描述
- 每条标pass/fail，禁止删条目或降标准——防止"做不到就降低要求"
- 定范围但不指定实现细节——让Evaluator从"行为是否符合规格"攻击，而不是"代码是否按我喜欢的方式写"攻击

如果没有Planner先定义清晰的证伪标准，Evaluator的攻击就会变成主观评判——"我觉得这里不好"，而不是"这不符合规格第X条"。

**Evaluator的关键设计：独立+真实用户视角+定位到行号**

Evaluator的三个设计细节体现了V的精髓：

1. **独立验收**：Evaluator是独立的Agent（甚至可以用不同模型），不共享Generator的思维链和上下文——避免"自己改自己查"的确认偏误
2. **Playwright像真实用户一样操作**：不是读代码找问题（白盒测试），而是像用户一样点击、输入、观察结果（黑盒端到端验证）——这是最接近真实使用场景的攻击角度
3. **把bug定位到行号和原因后打回**：不只是说"这里有bug"，而是定位到具体哪一行、为什么是bug——给出可行动的证伪证据，而不是模糊的负面评价

### 2.2 三角色如何完整体现V的四要素

| V四要素 | P-G-E三角色中的体现 |
| --- | --- |
| **证伪导向** | Evaluator不是来"证明Generator做对了"，而是来"努力找出bug"——借鉴GAN的对抗思路；目标是证伪而非证实 |
| **多角攻击** | Planner从需求角度审查规格完整性，Generator从实现角度自检，Evaluator从用户角度端到端验证；三个独立视角形成交叉火力 |
| **偏差防御** | 三角色分离防御"自我评估不可靠"偏误；Planner先定规格禁止降标准防御"降低完成标准"偏误；Evaluator独立于实现防御确认偏误 |
| **审计可溯** | Git版本控制记录每一次变更；进度文件记录每条任务的pass/fail状态；Evaluator打回时定位到行号和原因——所有验证过程可追溯 |

### 2.3 V的已知局限："共享误解"问题的诚实承认

文章最有价值的地方之一，是诚实承认了V机制的已知局限：

> 但 Loop 不会自动解决以下问题：它不会自动产生正确目标（目标错误时，循环只会更快地朝错误方向执行）；**不会自动产生可信的验收标准（Generator 和 Evaluator 若共享同一个误解，仍可能出现"错的实现 + 全部通过的测试"）**；不会承担责任（发布、用户数据、支付、风控必须有明确的人类责任人和审批边界）；也不会替代工程师形成判断。
>
> ——第07章（article-content.md:589）

以及第08章更详细的阐述：
> **实现和测试可能共享同一个误解。** 同一个 Agent 既写实现又写测试，对需求的理解偏差会同时进两边。测试全过，仍不能证明实现符合原始业务意图。
>
> ——第08章（article-content.md:607）

这是一个极其深刻的认知：即使是独立的Generator和Evaluator，如果它们从Planner那里继承了同一个错误理解（或者Planner一开始就理解错了需求），那么整个对抗体系仍然会失效——Generator产出"符合错误规格"的实现，Evaluator按"错误规格"验收全部通过，但结果不符合真实业务需求。

这暴露了V的根本局限：**V能防御执行层面的错误，但不能防御认知层面的错误——如果"什么是正确"本身定义错了，对抗再激烈也只是在错误的方向上做得更"完美"。**

文章如何应对这个局限？不是声称能解决，而是诚实承认并给出缓解方案：
1. **人保留最终判断权**——核心结论"人负责选择方向、定义标准并承担责任"
2. **AI自治度随风险提高而降低**——核心业务逻辑（支付、风控、订单）AI自治度低，必须有人把关
3. **承认局限不夸大**——明确说"这还没解决"，而不是包装成万能方案

这种诚实本身就是V精神的体现——不自我证实，主动暴露局限。

## 3. 计算型vs推断型反馈分层与V的成本考量

V（对抗性审查）不是"越多越好"——审查是有成本的，需要在"发现更多bug"和"审查成本/速度"之间做权衡。文章提出的计算型vs推断型分层，本质上是V的成本效益优化策略。

### 3.1 两类反馈的特性对比

| 维度 | 计算型反馈 | 推断型反馈 |
| --- | --- | --- |
| **执行者** | 确定性程序（linter、类型检查、测试、脚本） | 模型（Review Agent、AI Judge、审查Agent） |
| **速度** | 快（毫秒到秒级） | 慢（秒到分钟级） |
| **成本** | 便宜（几乎零边际成本） | 贵（消耗token和计算资源） |
| **可重复性** | 100%可重复——同样输入必然同样结果 | 不确定——同样输入可能不同结果，有概率性 |
| **覆盖问题类型** | 机械性错误：语法、类型、风格、测试失败、架构规则 | 语义性错误：需求误解、过度设计、用户体验、业务逻辑 |
| **适合频率** | 高频——每次修改后都可以跑 | 低频——关键节点才跑（提交前、集成前、发布前） |
| **误报率** | 低（规则明确） | 较高（语义判断有主观性） |

原文证据：
> **计算型**控制由确定性程序执行，例如 LSP、类型检查、linter、单元测试、结构测试、依赖扫描、脚本和 codemod，优点是快、便宜、可重复，适合在 Agent 每次修改后反复运行。**推断型**控制依赖模型做语义判断，例如 Review Agent、架构审查 Agent、AI judge 和设计评估，优点是能覆盖"是否过度设计""是否误解需求""是否符合团队约定"这类难以写成规则的问题，缺点是更慢、更贵，也更不确定。
>
> ——第6.3节（article-content.md:499）

### 3.2 WorkBuddy的原则：计算型优先，推断型后置

> WorkBuddy 的原则是：**能用计算型信号解决的问题，优先交给确定性程序；需要语义判断的问题，再交给审查 Agent。**
>
> ——第6.3节（article-content.md:501）

这个原则体现了V的成本考量：

1. **左移快速检查**：
   > 反馈也要按时机分层：快速检查尽量左移到编辑后、提交前或 Agent 自我纠正循环中；更昂贵的架构审查、详细代码审查、端到端验证放到集成前后。
   >
   > ——第6.3节（article-content.md:501）

   计算型检查速度快成本低，应该放在最前面（编辑后立即跑），让Agent在第一时间纠正机械错误，不要等到昂贵的推断型检查才发现低级错误。

2. **成本与风险匹配**：
   - 高频率、低影响的检查（如语法、类型）→ 计算型，每次修改都跑
   - 低频率、高影响的检查（如架构审查、业务逻辑验证）→ 推断型，关键节点才跑
   - 持续运行的监控（如死代码扫描、依赖风险、SLO监控）→ 周期性传感器，后台持续运行

3. **形成V的火力梯度**：
   - 第一层（最密集）：编辑器内实时检查（LSP、类型检查）——毫秒级响应
   - 第二层（每次修改）：本地lint、单元测试——秒级响应
   - 第三层（提交前）：集成测试、代码风格检查——十秒级
   - 第四层（关键节点）：Review Agent审查、端到端测试——分钟级
   - 第五层（持续运行）：周期性扫描、运行时监控——后台持续

这不是"用不用V"的问题，而是"用什么成本的V在什么时机攻击"的问题。

### 3.3 对七概念V的启示：需要引入成本维度

七概念V目前强调"主动构造反例""多角攻击"，但没有明确讨论V的成本问题。文章的分层策略给七概念V一个重要启示：**V不是免费的，需要引入成本效益分析**。

七概念V可以补充三个成本相关的原则：
1. **攻击成本与问题价值匹配**：高风险问题（资金损失、数据安全）值得投入高成本的推断型V；低风险问题（代码风格）用低成本计算型V即可
2. **攻击频率与问题概率匹配**：高频发生的错误（语法错误）用高频V；低频但高影响的错误（架构腐化）用周期性V
3. **攻击速度与反馈时机匹配**：需要即时反馈的（编辑时）用快速V；可以等待的（发布前）用慢速但更全面的V

没有成本考量的V是不可持续的——如果每写一行代码都要等10分钟的AI审查，开发效率会不可接受。好的V体系应该像免疫系统：常驻的快速反应部队（计算型，如皮肤、吞噬细胞）应对日常病原体，特种部队（推断型，如抗体、T细胞）应对特定严重威胁，而不是每次都动用全军。

## 4. 业务正确性验证缺口：V的已知局限深度分析

文章第08章专门讨论了"功能和业务正确性的验证缺口"，这是V（对抗性审查）在当前技术水平下面临的系统性局限。七概念的V方法论同样面临这些局限。

### 4.1 四个原因的逐一分析

文章指出业务正确性验证缺口有四个原因，每个原因都击中了V的某个薄弱环节：

**原因1：需求本身很难完整说明→V的"证伪标准"不明确**

> **需求本身很难完整说明。** PRD 通常分别描述单项功能，不一定覆盖功能组合后的行为。例如一个会话同时支持"归档"和"置顶"，两个功能分别测试通过，组合后却出现新问题：已经置顶的会话能不能归档？如果 PRD 没有定义，Agent 可能自行决定，再把同一个理解同时写进实现和测试——工程检查全绿，但已加入一项未经确认的业务决策。
>
> ——第08章（article-content.md:605）

对V的挑战：
- V的证伪导向需要有明确的"什么是错"的标准
- 如果需求本身不完整（组合场景、边界情况没有定义），Evaluator就没有攻击依据
- 更糟的是，如果Planner/Generator/Evaluator自行"补全"了需求的模糊地带，它们可能补全成同一个错误理解，然后V体系证明这个错误理解"正确"

本质问题：**V能验证"实现是否符合规格"，但不能验证"规格是否符合真实业务需求"。**

**原因2：实现和测试可能共享同一个误解→V的"独立攻击者"被污染**

> **实现和测试可能共享同一个误解。** 同一个 Agent 既写实现又写测试，对需求的理解偏差会同时进两边。测试全过，仍不能证明实现符合原始业务意图。
>
> ——第08章（article-content.md:607）

对V的挑战：
- V的多角攻击依赖"攻击者"和"执行者"的独立性
- 如果两者共享同一个认知偏误（对需求的同一误解），对抗就失效了
- 即使Generator和Evaluator是独立Agent，如果它们从同一个Planner那里继承了错误的规格，独立性也救不了——这是第2.3节分析的"共享误解"问题

本质问题：**V的独立性是相对的——如果认知源头被污染，下游再独立也没用。**

**原因3：部分业务正确性缺少可计算判定标准→V缺少自动化武器**

> **部分业务正确性缺少可计算的判定标准。** 编译器能发现语法和类型问题，监控能度量性能和错误率，但业务意图通常需要业务人员确认。
>
> ——第08章（article-content.md:609）

对V的挑战：
- 计算型V（快速、便宜、可重复）依赖明确的可计算规则
- 但很多业务正确性问题是"用户体验好不好""是否符合业务意图""定价逻辑是否合理"这类语义问题
- 这些问题只能用推断型V（慢、贵、不确定），而且最终往往需要人来判断
- 缺少自动化武器意味着V无法高频覆盖这些问题，只能靠低频人工审查

本质问题：**V的火力受限于"什么可以被自动化检查"——越靠近业务语义，越难自动化。**

**原因4：业务错误成本很高→V失败代价不可承受**

> **业务错误的成本可能很高。** 核心业务错误可能造成资金损失、合规问题和用户流失，对验证可信度的要求远高于普通代码质量问题。
>
> ——第08章（article-content.md:611）

对V的挑战：
- V的目的是"发现尽可能多的bug"，但V本身不是100%准确的——会有漏网之鱼
- 对于代码风格问题，漏掉一两个问题代价不大
- 对于支付、风控、订单等核心业务，漏掉一个bug可能造成百万级损失
- 这意味着在高风险场景，即使V做得很好，也不能完全信任——必须有人来兜底

本质问题：**V的可信度与问题的重要性成反比——越重要的问题，越不能只依赖V。**

### 4.2 对七概念V方法论的启示

七概念的V（对抗性审查）作为通用方法论，同样面临这四个局限。文章给七概念V的启示是：

**启示1：V需要区分"规格内验证"和"规格外验证"**

七概念V目前主要讨论"如何证伪"，但应该明确区分两个层次：
- **规格内V**：验证"实现是否符合明确规格"——这部分V可以做得很好，P-G-E+计算型/推断型分层足够
- **规格外V**：验证"规格本身是否正确""未明确说明的组合场景是否正确""业务意图是否被正确理解"——这部分V能力有限，必须依赖人

七概念V不应该声称能解决所有验证问题，而应该明确自己的能力边界。

**启示2：V需要配套"认知源头净化"机制**

针对"共享误解"问题，V不能只在执行后检查，还需要在执行前（认知层）介入：
- Planner定规格时需要有独立审查——规格评审本身就是V的一部分
- 需求模糊地带不能让Agent自行补全，必须升级给人确认
- 关键业务决策需要显式记录并由人审批，不能隐含在实现中

这对应文章中"AI自治度随风险提高而降低"的分级原则：
| 场景 | AI 自治度上限 |
| --- | --- |
| 一次性脚本、内部工具 | 高 |
| 公开 API、跨系统改动 | 中 |
| 核心业务逻辑（支付、风控、订单） | 低 |

**启示3：V不是责任转移机制**

文章核心结论之一是："人负责选择方向、定义标准并承担责任；Agent负责执行、验证和加速迭代。"

V（对抗性审查）是帮助人发现问题的工具，不是替代人承担责任的机制。即使有最完善的V体系，最终责任仍然在人——这一点七概念V需要明确强调，避免"有了审查就可以放心"的错觉。

## 5. V作为横切概念：在Harness五层中的作用

V在Agent工程中不是独立的"验证阶段"（做完之后统一检查），而是横切所有层次的防御机制——执行前有V（预防）、执行中有V（拦截）、执行后有V（检测纠正）、跨周期有V（持续监控）。

### 5.1 V在Harness五层中的横切作用表

| Harness层次 | V的作用类型 | V的具体机制 | V的时机 |
| --- | --- | --- | --- |
| **运行环境层** | 预防型V（前馈） | Sandbox隔离环境；Allowlist/Denylist限定可操作命令和路径；权限边界；Approval Gate危险操作人工审批 | 执行前——阻止危险操作发生 |
| **引导层（Feedforward）** | 预防型V（前馈） | System Prompt/规则文件注入安全规则；Skills规定步骤和约束；上下文组织避免错误信息注入；Prompt Cache稳定前缀避免链接断裂 | 执行前——引导Agent往正确方向走，减少犯错概率 |
| **反馈层（Feedback）** | 检测型V（反馈） | 计算型反馈：LSP/类型检查/linter/单测/结构测试；推断型反馈：Review Agent/架构审查/端到端验证；工具结果错误信息（包含可修正参数）；编辑前时间戳校验（防覆盖）；Audit log留痕 | 执行后——发现错误并反馈给Agent纠正 |
| **编排层** | 过程型V（事中） | Planner/Generator/Evaluator三角色分离；多模型路由（验证用更可靠模型）；Teams多Agent协作交叉检查；并行工具调用结果交叉验证；审批门（高风险操作中断等待确认） | 执行中——在过程中拦截问题，不等到最后 |
| **迭代层** | 周期型V（长期） | 熵管理（Garbage Collection）周期性扫描代码漂移/死代码/失效文档/过期依赖；重复出现的问题反哺Harness改进；运行时健康传感器（错误率/延迟/SLO/日志异常）；Harness随模型能力演进持续调整 | 跨周期——持续发现和解决积累的问题，防止系统腐化 |

### 5.2 四类V形态的详细说明

V不仅是"做完后检查"（检测型），而是至少有四种形态：

**（1）前馈V（Feedforward V）——执行前的预防**
- 位置：引导层+运行环境层
- 思想："上医治未病"——最好的验证是不让错误有机会发生
- 机制：规则注入、权限约束、沙箱隔离、能力按需加载（避免给Agent犯错的工具）
- 例子：Approval Gate让高风险操作根本无法自动执行；Sandbox让Agent即使执行rm -rf也删不了真实文件

**（2）过程V（In-process V）——执行中的拦截**
- 位置：编排层
- 思想：在错误扩散前就拦截，不要等做完了才发现
- 机制：多角色分离、审批中断点、乐观锁（时间戳校验）、实时检查
- 例子：Agent改文件前发现时间戳不匹配立即停止，不覆盖用户修改；高风险操作弹出审批中断流程

**（3）反馈V（Feedback V）——执行后的检测**
- 位置：反馈层
- 思想：通过快速反馈让Agent自我纠正
- 机制：计算型测试/推断型审查/Audit log
- 例子：Agent改完代码后立即跑lint和测试，发现错误立即修正，不等到提交后

**（4）周期V（Periodic V）——跨周期的净化**
- 位置：迭代层
- 思想：有些问题是积累出来的，需要周期性扫描清理
- 机制：熵管理（GC）、漂移扫描、运行时监控、Harness自身迭代
- 例子：OpenAI后台运行周期性任务扫描代码漂移、自动创建重构PR；WorkBuddy周期性扫描死代码/失效文档/过期依赖

这四类V共同构成了纵深防御体系，比单一的"做完后测试"要可靠得多。

## 7. 差异与补充

### 7.1 七概念V强调"主动构造反例"，文章中Evaluator是按Planner规格核查（验证导向），是否缺少主动攻击？

**七概念V的理想状态**：主动构造反例——站在对手角度思考"什么情况会让这个系统崩溃"，刻意设计极端测试案例。

**文章中的Evaluator实际做的**：按Planner写好的规格逐条核查——验证"是否符合规格"，而不是"我能不能想出办法搞垮它"。

**分析：存在主动攻击不足，但这是工程务实选择**

文章中Evaluator的行为更接近"验收测试"（验证符合规格），而不是"渗透测试"（主动攻击找漏洞）。这体现在：
- Evaluator"逐条核查"Planner写好的pass/fail条目——是检查是否满足预定标准
- 用Playwright"像真实用户一样操作"——是模拟正常使用，不是刻意进行异常操作
- "把bug定位到行号后打回"——是发现不符合规格的地方，不是发现规格没覆盖的盲区

为什么不做更激进的主动攻击？可能有三个工程原因：
1. **成本问题**：主动构造反例需要更多创造性推理，推断成本更高
2. **误报问题**：主动攻击可能找出大量"不是bug的bug"（如极端边界情况在实际中不会发生），浪费Generator时间
3. **规格优先**：在业务场景中，"符合规格"是第一优先级——先保证写清楚的功能都做对，再考虑没写清楚的情况

但文章中确实存在部分"主动攻击"元素：
- 计算型反馈中的结构测试、依赖扫描——主动寻找架构违规和风险依赖
- 运行时健康传感器（错误率、延迟、SLO、日志异常）——主动监控异常
- 熵管理中的代码漂移扫描——主动发现腐化迹象

**对七概念V的启示：需要区分"验证型V"和"攻击型V"**

七概念V目前强调的"主动构造反例"更接近"攻击型V"（渗透测试思维），但工程实践中大部分V是"验证型V"（验收测试思维）。两者互补：
- **验证型V**：按预定标准检查，成本低、误报少、覆盖率可控——适合高频日常检查
- **攻击型V**：主动构造反例、探索边界、寻找盲区——成本高、可能误报、但能发现未知问题——适合低频深度审查

好的V体系应该两者结合，而不是只强调一种。

### 7.2 文章中Harness的约束层（Sandbox/Approval Gate）是V的预防形态，七概念V是否需要区分"预防型V"和"检测型V"？

七概念V目前更多强调"审查"（事后检测），但文章展示了V有更丰富的形态——包括事前预防。

**文章中的四类V形态（详见第5节）**：
1. 前馈V（预防）：Sandbox、权限、Approval Gate——阻止错误发生
2. 过程V（事中）：时间戳校验、审批中断——执行中拦截
3. 反馈V（检测）：测试、审查——执行后发现
4. 周期V（长期）：熵管理、监控——持续净化

**七概念V目前主要覆盖"检测型V"（反馈V+部分周期V），对"预防型V"和"过程V"讨论不足。**

**预防型V的核心逻辑**：最好的bug是从未发生的bug。与其事后检测再纠正，不如事前就不让Agent有犯错的能力。例如：
- 不提供删除生产数据库的工具→Agent不可能删库
- Sandbox隔离→Agent即使执行危险命令也不影响真实系统
- Approval Gate→高风险操作必须经过人，Agent无法私自执行

这比"让Agent做，做完再检查有没有删库"要可靠得多——检测总有漏网之鱼，预防从根源消除风险。

**对七概念V的启示：V不只是"找错"，更是"防错"**

七概念V应该扩展自己的定义，从"对抗性审查（事后检测）"扩展为"对抗性防御（事前预防+事中拦截+事后检测+持续净化）"：
- **预防型V**：通过环境设计、权限约束、能力限制，让某些类别的错误根本不可能发生
- **过程型V**：在执行过程中设置检查点和中断机制，错误一出现就拦截
- **检测型V**：执行后通过测试、审查发现错误
- **周期型V**：长期监控和扫描，发现积累性问题

这才是完整的防御纵深。

### 7.3 熵管理（Garbage Collection）作为周期性V，对七概念有何启示？

文章第6.4节"使用者视角的四类组件"中，熵管理（Garbage Collection）是V的一个容易被忽略但极其重要的形态：

> **4. 熵管理（Garbage Collection）：持续处理规则、代码和运行状态的漂移。** 现有检查主要防新增问题，对历史问题还要周期性扫描：WORKBUDDY.md/ OpenSpec 与代码是否一致、历史代码是否违反新规则、是否有重复实现 / 失效文档 / 过期依赖。还可以接入运行时健康传感器，例如延迟、错误率、可用性 SLO、日志异常。
>
> ——第6.4节（article-content.md:553）

以及OpenAI的实践：
> 后台运行周期性任务扫描代码漂移、自动创建重构PR。
>
> ——第6.2节（article-content.md:467）

**熵管理作为V的独特价值：**

传统V（测试、审查）主要防御"新增问题"——每次修改后检查有没有引入新bug。但复杂系统还有另一类问题：**腐化问题**——代码、规则、文档随着时间推移逐渐偏离正确状态：
- 新规则加了，但历史代码没更新，逐渐积累技术债
- 代码改了，但文档/规格没同步，文档逐渐失效
- 依赖过期了、有漏洞了，但没人注意
- 重复实现逐渐增多，没人清理
- 运行时指标逐渐恶化，但没触发告警阈值

这些问题不是某一次修改引入的，而是时间积累的结果——熵增是自然趋势。一次性的V（提交前测试、发布前审查）发现不了这类问题，必须有**周期性的V**持续扫描和清理。

**对七概念V的启示：V需要对抗时间维度的熵增**

七概念V目前主要关注"单次变更的验证"（空间维度），但忽略了"长期系统的腐化"（时间维度）。熵管理给七概念V的启示是：

1. **V不仅要防"坏变更"，还要防"系统腐化"**：即使每次变更都通过V检查，系统整体仍然可能随着时间推移逐渐变坏——需要周期性V对抗熵增

2. **V需要有"修复闭环"，不只是"发现问题"**：
   > "Agent 卡住时，我们把它当作信号——找出缺了什么（工具、护栏、文档），反哺回仓库——而且总是让 Codex 自己写这个修复。"
   > ——第6.4节（article-content.md:551）
   
   发现问题不是V的终点，把发现的问题转化为Harness改进（新增规则、补充文档、增加工具），防止同类问题重复发生——这才是完整的V闭环。

3. **V体系本身也需要迭代**：
   > 这类迭代需要证据支撑：一次失败可能是偶发，同类失败多次出现或风险很高时再调整 Harness。新增机制也要评估副作用。
   >
   > ——第6.3节（article-content.md:535）
   
   V机制不是一成不变的——需要根据发现的问题持续调整，这是"V的V"（元V）：审视V机制本身是否有效，是否需要调整。
