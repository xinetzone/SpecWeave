## 二、Agent工程 → 七概念 反向映射：文章独有概念输出

以下是WorkBuddy文章中存在但在七概念框架中没有直接对应或尚未充分发展的核心概念，这些是Agent工程实践对七概念方法论的"概念输出"：

| Agent工程概念 | 核心内容 | 原文证据 | 对七概念的启示/补充方向 |
| --- | --- | --- | --- |
| **MCP（模型上下文协议）** | 外部能力标准化接入协议；三类原语（Resources/Tools/Prompts）按驱动方（Agent/模型/用户）区分；按用户意图组织工具而非照搬底层API | > "MCP试图解决的是外部能力接入的标准化问题……MCP Server向Agent提供三种原语：Resources（Agent驱动）、Tools（模型驱动）、Prompts（用户驱动）。"（第2.3节）<br/><br/>> "按用户意图组织工具，不照搬底层API。"（第2.3节） | 七概念缺少"能力接入标准化"维度——方法论如何与外部系统/工具/知识源标准化对接？MCP的"三类原语按驱动方区分"思想可以迁移到知识管理：不同类型知识由不同角色主动检索/注入。"按用户意图组织"而非底层结构暴露，可以指导E萃取的模式设计——模式应该面向使用场景而非原始结构。 |
| **Plugin（能力打包分发）** | 将MCP+Skills+Rules+Hooks+Templates组合为可安装、可分发、按作用域（团队/项目/个人）安装的能力包 | > "Plugin解决的是'能力组合如何安装和分发'的问题。它把多种相关能力组合成可安装、可分发的单位。"（第2.5节）<br/><br/>```Plugin: team-dev-workflow ├─MCP ├─Skills ├─Rules ├─Hooks └─Templates``` | 七概念E萃取目前关注单个模式（Skill级），缺少"模式包/方法论包"的组合与分发概念。Plugin提示我们：模式不是孤立使用的，相关模式+规则+钩子+模板需要打包分发，并支持不同作用域安装。这对应组织级方法论落地——如何将整套治理机制打包为团队可一键启用的"治理Plugin"。 |
| **Prompt Cache（前缀缓存）** | 模型厂商缓存前缀计算结果，上下文组织需遵循前缀稳定、动态追加、按需加载规则以提高缓存命中率、降低成本 | > "它按前缀匹配，所以WorkBuddy在上下文组织上遵循几条规则：System Prompt/工具定义放前面保持稳定；对话历史只追加不修改；动态内容追加后面；工具/Skill按需加载；只在压缩/纠错时接受前缀变化。"（第4.1节） | 七概念A原子化强调链接完整，但没有"缓存/计算成本"维度。Prompt Cache揭示了：信息组织不仅影响认知负荷和导航成本，还直接影响计算成本（缓存命中率是关键工程指标）。稳定前缀+动态追加不仅是链接完整的要求，也是成本优化的要求——这为A的粒度评估增加了第三个维度：计算成本。 |
| **Approval Gate（审批门）** | 高风险操作（删除/发送/支付/发布）执行前暂停等待人工确认；"最好的回滚是不让错误发生" | > "删除、发送、支付、发布等高风险动作需要审批。"（第2.2节）<br/><br/>> "发送、发布、删除或其他不可轻易撤销的操作，执行前确认用户授权。"（第2.2节System Prompt示例） | 七概念C原子提交强调独立回滚，但主要是事后回滚。Approval Gate代表了一种更根本的原子性保障：事前阻止。"最好的回滚是阻止"这一理念可以补充到C的独立回滚要素中——回滚不应只有事后撤销一种方式，事前审批拦截是成本更低的回滚。Approval Gate也体现了人机责任分界——高风险节点人保留最终决策权。 |
| **计算型反馈 vs 推断型反馈分层** | 计算型：确定性程序执行（linter/类型检查/单测），快、便宜、可重复、适合高频；推断型：模型语义判断（Review Agent/AI judge），慢、贵、不确定、适合低频高价值 | > "计算型控制由确定性程序执行……优点是快、便宜、可重复，适合在Agent每次修改后反复运行。推断型控制依赖模型做语义判断……优点是能覆盖难以写成规则的问题，缺点是更慢、更贵，也更不确定。"（第6.3节）<br/><br/>> "WorkBuddy的原则是：能用计算型信号解决的问题，优先交给确定性程序；需要语义判断的问题，再交给审查Agent。"（第6.3节） | 这是对七概念R/V的重大工程化补充：复盘/审查不是"越多越好"或"越深越好"，而是需要按成本、速度、可重复性分层。计算型对应R中的"事实采集"和V中的"确定性检查"，推断型对应R中的"因果推演"和V中的"语义审查"。七概念可以引入这个分层：高频机械问题用低成本自动化V/R，低频语义问题用高成本深度V/R。这一原则不仅适用于Agent，也适用于人类团队——代码风格用lint自动化，架构决策用资深工程师审查。 |
| **渐进式加载（多级粒度阶梯）** | 不是一次性加载所有细节，而是建立"意图识别→名称简介→完整内容→参考脚本"多级粒度阶梯，按需逐级展开 | > "工具定义过多时……先让模型看到工具名称和简要描述，再根据任务需要加载更具体的工具说明。Skill也用同样机制，先看名称和描述，确认适用后再读完整SKILL.md。"（第4.2节）<br/><br/>"意图识别负责'先选对方向'，渐进式加载负责'再按需展开'。"（第4.2节） | 七概念A原子化目前主要讲U型曲线寻优——找到最优粒度点。但渐进式加载揭示了：最优粒度不是一个静态点，而是一个动态阶梯——系统应该能在不同粒度级别之间平滑移动，先粗后细、按需深入。这比"选一个粒度"更灵活：先看地图（粗粒度导航），需要时再放大看细节（细粒度执行）。七概念A应该从"点寻优"扩展为"阶梯设计+升降机制"。 |
| **P-G-E三权分立（Planner/Generator/Evaluator）** | 借鉴GAN对抗思路，将需求定义、实现执行、独立验收分离为三个独立角色，从需求/实现/用户三个视角交叉验证，解决"自我评估不可靠"和"实现测试共享误解"问题 | > "方案借鉴GAN的对抗评估思路……Planner（把一句话需求展开成完整规格，定范围但不指定实现细节）、Generator（按sprint逐功能实现，提交前先自检）、Evaluator（独立验收Agent，用Playwright像真实用户一样操作，逐条核查、把bug定位到行号和原因后打回）。"（第6.2节） | 七概念V强调"多角攻击"，但P-G-E给出了工业级的角色分工范式：三权分立不是"多几个人审查"，而是从根本上分离"定义正确""执行实现""独立验证"三个权责。Planner先定义什么是正确（证伪标准），Generator执行，Evaluator从用户视角攻击——这比单纯的"独立审查"更系统。七概念V的"多角攻击"可以具体化为角色分离模式。 |
| **Sandbox（沙箱隔离）+ Worktree独立工作区** | 物理隔离执行环境：Agent在沙箱中运行命令，即使执行rm -rf也不影响宿主机；每个任务在独立Worktree中执行，失败直接丢弃整个任务环境 | > "Sandbox（隔离环境，执行出错也不影响本机）。"（第6.1节）<br/><br/>> "一个可用的Loop至少需要这些组件：……独立执行环境（Isolated Workspace/Worktree）。"（第07章）<br/><br/>> "创建独立worktree和任务记录……如果失败……仍失败则停止并留下可诊断报告。"（第07章） | 七概念C原子提交主要讲逻辑隔离（分支/commit），Sandbox/Worktree代表了物理隔离的理念——最好的原子性保障不是事后回滚，也不是事前审批，而是物理隔离让错误无法扩散。"失败直接丢弃Worktree"比git revert更简单更可靠——因为物理上独立，失败不会留下任何痕迹。七概念可以补充"物理隔离"作为C独立回滚的第一道防线——环境设计层面保证原子性，比流程层面更可靠。 |
| **熵管理（Garbage Collection / 周期性净化）** | 周期性扫描代码漂移、死代码、失效文档、过期依赖、规则与代码不一致等跨任务积累的"熵增"问题；持续运行传感器监控运行时健康（错误率/延迟/SLO/日志异常） | > "熵管理（Garbage Collection）：持续处理规则、代码和运行状态的漂移。现有检查主要防新增问题，对历史问题还要周期性扫描。"（第6.4节）<br/><br/>> "后台运行周期性任务扫描代码漂移、自动创建重构PR。"（第6.2节引OpenAI） | 七概念R复盘和V审查主要关注单次变更/单个里程碑，但缺少对抗"时间维度熵增"的机制。即使每次变更都通过检查，系统仍会随时间腐化（新规则加了旧代码没更新、文档过期、依赖过时）。熵管理是周期性V/R——不是等出问题才复盘，而是主动定期扫描清理。七概念R/V应该补充"周期型复盘/审查"：日常反馈防新增问题，熵管理处理积累问题。 |
| **前馈+反馈控制论模型** | Harness是控制系统：前馈（Feedforward）在行动前提供目标/规则/环境提高首次正确率；反馈（Feedback）在行动后观察结果返回纠正信息；两者结合形成闭环 | > "WorkBuddy构建良好的Harness有两个目标：提高Agent首次执行的正确率，并提供反馈循环让系统能自动发现和纠正常见问题。"（第6.3节）<br/><br/>> "前馈提高第一次就做对的概率，反馈让Agent在问题进入人工审查前先自我纠正。"（第6.3节） | 七概念各概念之间有逻辑依赖，但没有用控制论模型统一框架。前馈+反馈模型提供了一个简洁的统一视角：F/R/I/E/A的规则/知识/粒度指导属于前馈（行动前提供正确引导），V审查/C验证/R反馈属于反馈（行动后纠正偏差）。这个模型可以帮助七概念从"概念集合"升级为"控制系统"——每个概念在前馈或反馈环中扮演什么角色。 |
| **AI自治度分级（人保留最终责任）** | 核心结论：人负责选择方向、定义标准并承担责任；Agent负责执行、验证和加速迭代；AI自治度随风险提高而降低（一次性脚本高、公开API中、核心业务逻辑低） | > "人负责选择方向、定义标准并承担责任；Agent负责执行、验证和加速迭代。"（第09章）<br/><br/>> "核心业务错误可能造成资金损失、合规问题和用户流失，对验证可信度的要求远高于普通代码质量问题。"（第08章）<br/><br/>| 场景 | AI自治度上限 |<br/>| 一次性脚本、内部工具 | 高 |<br/>| 公开API、跨系统改动 | 中 |<br/>| 核心业务逻辑 | 低 | | 七概念作为方法论主要应用于人类团队，默认人是执行主体和责任主体。但AI时代方法论需要明确人机分工边界——哪些环节可以交给AI自主执行，哪些环节人必须保留判断权。WorkBuddy的"责任在人、执行在Agent"原则和自治度分级表，为七概念在AI协作场景的应用提供了框架。七概念可以补充AI协作场景下的概念适配——哪些概念可以由AI自动化执行，哪些必须由人把关。 |
| **驾驭/约束/整合三类能力（Steer/Constrain/Integrate）** | Harness从词源（马具）拆解为三类能力：驾驭（控制方向/速度/停止，对应System Prompt/Skills/Task清单/错误提示）、约束（防止越界，对应Sandbox/Approval Gate/权限/测试/审计）、整合（配齐协同，对应工具/状态/协作/自动化） | > "Harness一词原指套在马身上的整套装备。从词源出发可以拆出三类能力：1.驾驭（Steer）：控制执行方向、速度和停止时机；2.约束（Constrain）：防止执行超出安全范围；3.整合（Integrate）：把各项能力配齐并协同。"（第6.1节） | 七概念七概念各自独立，但缺少一个"能力分类框架"来回答"一个可靠的治理系统/Agent系统需要哪几类能力"。Steer/Constrain/Integrate三分法提供了一个简洁的分类：引导正确方向（F/I/A）、防止越界错误（V/C）、配齐协同能力（R/E）。这个分类可以帮助七概念从"七个概念"升级为"三类能力"的更高阶抽象。 |
| **Agent = Model + Harness（共同进化视角）** | Agent不只是模型，而是模型加Harness的组合；模型和Harness共同进化；评估时必须评估"模型+Harness"组合，同一模型换了Harness表现可能差异很大 | > "LangChain提出一个视角：模型和Harness在共同进化。实际评估的对象通常是'模型+Harness'的组合——同一个模型换了工具名、Patch格式、压缩策略或错误回传方式后，表现可能明显变化。因此比较或升级模型时，要把Harness一起纳入Eval。"（第6.2节）<br/><br/>> "模型决定能力上限；上下文和Harness决定这个上限能否稳定落地。"（第09章） | 七概念方法论本身也存在类似关系：方法论（F/R/I/E/C/A/V）相当于Harness，使用方法论的人/团队相当于Model。同样的人/团队，用了好的方法论（Harness）表现差异很大；方法论也需要随团队能力（Model）演进而调整。"共同进化"视角提示我们：七概念不是静态规则手册，而是需要和使用它的团队共同进化的活体系——模型（人/AI）变强了，有些Harness约束可以精简；模型变弱了或场景变了，需要增加约束。 |

**统计说明**：
- A是两套体系中最直接的同构概念——Context Engineering的五动作、渐进式加载、Prompt Cache规则几乎是A四要素的逐字翻译
