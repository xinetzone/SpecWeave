## 1. A四要素在Context Engineering中的体现

A（原子化）与Context Engineering是两套体系中映射最直接的概念对。Context Engineering的核心命题正是"模型这一刻该看到什么"，本质上就是在寻找最优信息粒度——这与A公理"复杂系统向最优信息粒度的收敛过程，在认知负荷与导航成本间找到平衡点"完全一致。

### 1.1 粒度寻优：Context Engineering五动作都是U型曲线寻优

Context Engineering定义的五类动作（Write/Select/Retrieve/Compress/Isolate），本质上都是在认知负荷（cognitive_load）与导航成本（navigation_cost）之间寻找U型曲线最低点：

**为什么不是"全部放进去"？文章明确指出了堆token的反效果：**

> 一个常见误区是"上下文窗口很大，全部放进去"。无关信息既占成本，也降低模型对当前重点的判断准确度。**Context Engineering 追求相关、准确、及时，不是单纯堆 token。**
>
> ——第04章（article-content.md:305）

这句话直接体现了粒度寻优的核心逻辑：
- **粒度过粗（全部放进去）**：认知负荷过高——模型被无关信息干扰，判断准确度下降，token成本上升
- **粒度过细（什么都不放）**：导航成本过高——模型需要反复检索才能获得必要信息，效率低下
- **最优粒度**："相关、准确、及时"——恰好满足当前决策需要的最小信息集

**五动作与U型曲线寻优的对应关系：**

| Context Engineering动作 | 粒度调节方向 | 解决的问题 |
| --- | --- | --- |
| **Write（写入）** | 向当前上下文增加粒度单元 | 解决"信息缺失导致认知不完整"问题——别让模型靠猜 |
| **Select（选择）** | 从候选中筛选粒度单元 | 解决"粒度过粗导致认知负荷过高"问题——这是filter |
| **Retrieve（检索）** | 从外部拉取需要的粒度单元 | 解决"粒度过细导致导航成本过高"问题——这是pull |
| **Compress（压缩）** | 合并/外置粒度单元，降低粒度密度 | 解决"上下文过长导致认知负荷爆炸"问题——长内容外置只留结论与位置 |
| **Isolate（隔离）** | 将旁支粒度单元移到独立上下文 | 解决"旁支信息污染主上下文导致认知负荷升高"问题——Sub-agent只带回结果 |

**渐进式加载是U型曲线寻优的工程化实现：**

文章第4.2节详细描述的渐进式加载机制，正是粒度寻优的典型工程实践：

> 几乎所有外接能力都可以按这个思路组织：默认只暴露名称和简介，真正进入某个任务时再加载完整内容。这样上下文里始终只保留当前需要的能力，而不是把所有能力一次性铺开。
>
> ——第4.2节（article-content.md:345）

渐进式加载的U型曲线逻辑：
- **初始状态（默认只暴露名称简介）**：导航成本极低（模型一眼看到所有能力的概览），认知负荷也低（只看摘要不看细节），但信息不完整——这是"粗粒度快速导航"阶段
- **按需加载（确认适用后读完整内容）**：当确定某个工具/Skill/资源是当前需要的，再加载完整细节——这是"细粒度深度认知"阶段
- **U型平衡**：既避免了"一次性加载所有细节"导致的认知负荷爆炸（左端点），又避免了"什么细节都不加载"导致的反复检索导航（右端点）

工具结果截断策略也是同样逻辑：
> 工具结果过长时，WorkBuddy 给每个 Tool Result 设截断策略，超出时分页、截断或写入文件。截断时明确告诉模型"结果未完整"，附上总量、截断位置和继续读取方法，否则模型会把前 100 条误当成全部。
>
> ——第4.2节（article-content.md:337）

这避免了两种极端：要么返回全部结果（可能几十万token导致认知负荷爆炸），要么只返回截断结果不说明（导致模型误以为信息完整产生错误判断）。

### 1.2 单元独立：Sub-agent/MCP/Skill的自包含设计

A的"单元独立"要素要求"各模块单一职责独立可读语义完整"。文章中有三处机制直接保证了单元独立性：

**（1）Sub-agent隔离处理旁支任务**

> 第五，拆分调研任务并分配给 Sub-agent。OpenAI、Anthropic、LangChain 的资料可以分别交给不同 Sub-agent 处理。主 Agent 为每个 Sub-agent 提供统一输出格式，但只提供各自任务所需的上下文，减少信息干扰。
>
> ——第03章（article-content.md:263）

Sub-agent是典型的独立单元设计：
- **单一职责**：每个Sub-agent只负责一个调研对象（OpenAI/Anthropic/LangChain），不承担其他任务
- **独立上下文**：只获得各自任务所需的上下文，主上下文的其他信息不注入，避免干扰
- **语义完整输出**：每个Sub-agent返回"结论、证据、来源和适用范围"，是完整可独立理解的结果单元
- **隔离性**：Sub-agent的中间推理过程不污染主上下文，只带回最终结果

Isolate动作的定义也明确了这一点：
> **5. 隔离（Isolate）**：用独立会话或 Sub-agent 处理旁支任务，只把结果带回主线，避免污染主上下文。
>
> ——第04章（article-content.md:301）

**（2）MCP三类原语的独立封装**

MCP的Resources/Tools/Prompts三类原语，每一类都是独立封装的单元：

> **• Resources（资源）**：有 URI 标识、可读取的只读内容（一个文件、一条数据库记录、一段实时数据）。它是应用 / Agent 驱动的……本职用法是把读出的内容直接拼进 messages，不经过 tool。
>
> **• Tools（工具）**：模型能调用的动作 / 函数。它是模型驱动的……结果通常以 text 回流进上下文供模型继续推理，但也可以把 structuredContent 分流给 UI 而不进上下文。
>
> **• Prompts（提示模板）**：Server 预先组织好、可复用的一组消息。它是用户驱动的……模板内部还能引用 resource，把指令和文件内容打包带入。
>
> ——第2.3节（article-content.md:139-143）

三类原语的独立性体现在：
- **职责单一**：Resources负责"读内容"，Tools负责"执行动作"，Prompts负责"提供模板"，互不重叠
- **驱动方独立**：分别由Agent/模型/用户驱动，调用边界清晰
- **可独立使用**：不需要同时加载三类原语，按需使用
- **URI标识**：Resources有独立URI，可被唯一引用

**（3）Skill的自包含设计**

> Skill 把这些经过验证的工作方法保存下来，通常包含说明、步骤、脚本、命令和判断标准。模型处理匹配任务时先读 Skill，再按流程执行。
>
> ——第2.4节（article-content.md:165）

以"创建PR"Skill为例：
```markdown
适用场景：用户要求为当前仓库的代码变更创建 PR。流程：1. 读取 AGENTS.md / WORKBUDDY.md 和仓库贡献规范。2. 检查 git status，不覆盖用户的未提交修改。3. 阅读 diff，识别变更范围与风险。4. 运行与改动相关的格式化、类型检查和测试。5. 仅在验证通过后生成标题和描述，如实列出未验证项。6. 确认用户已授权发布，再 push 并创建 PR。完成标准：- PR 只包含本任务的变更。- 已记录所运行的测试与结果。- 标题、正文、Issue 关联符合团队规范。
```
——第2.4节（article-content.md:167-169）

Skill的自包含性体现在：
- **适用场景明确**：什么时候用这个Skill写得很清楚
- **流程完整**：从第一步到最后一步，不需要依赖其他Skill的步骤
- **失败分支包含**：Skill里还写清失败分支（测试失败如何判断、没有权限就停在草稿）
- **完成标准清晰**：什么叫"做好了"有明确判定标准
- **可版本化/可评审/可测试/可回滚**：文章第5.2节明确指出Skill具备这些独立单元的特性

### 1.3 链接完整：Prompt Cache前缀稳定与截断续读机制

A的"链接完整"要素要求"单元间引用关系完整可追溯"。文章中两个关键机制保证了链接完整性：

**（1）Prompt Cache前缀稳定策略保证信息引用不被破坏**

Prompt Cache的五条规则本质上是在维护上下文的链接完整性：

> 它按前缀匹配，所以 WorkBuddy 在上下文组织上遵循几条规则：
>
> ·System Prompt、基础工具定义、长期规则放前面，保持内容与顺序稳定；
>
> ·对话历史采用追加方式保存，不修改已发送过的消息；
>
> ·当前文件、任务进度、时间、工具结果、新加载的 Skill 等动态内容追加到后面；
>
> ·工具和 Skill 按需加载，避免每轮重新生成并排列完整能力列表；
>
> ·只在上下文过长需要压缩、或纠正错误信息时，才接受前缀变化和缓存重算。
>
> ——第4.1节（article-content.md:315-325）

这些规则如何保证链接完整？
- **稳定前缀不修改**：System Prompt和工具定义放前面保持稳定→模型对这些基础信息的"引用位置"不变，不会因为前缀变化导致之前的推理中"前面第X条说的是Y"这种引用失效
- **对话历史只追加不修改**：已发送的消息不修改→历史对话中的引用关系保持稳定，不会出现"之前说的A现在变成了B"导致的链接断裂
- **动态内容追加在后面**：新内容追加而不是插入中间→之前内容的相对位置不变，基于位置的引用仍然有效
- **按需加载而非重排列**：不每轮重新排列工具列表→工具在工具列表中的位置保持稳定，模型对"第N个工具是做什么的"的记忆不会混乱

链接完整性的核心是：**一旦信息进入上下文，其标识（位置、顺序、引用关系）就不应该随意变化，否则基于此建立的推理链就会断裂。** Prompt Cache规则虽然是为了缓存命中率设计，但客观上完美维护了上下文的链接完整性。

**（2）工具结果截断时的"续读指引"保证链接完整**

> 截断时明确告诉模型"结果未完整"，附上总量、截断位置和继续读取方法，否则模型会把前 100 条误当成全部。错误也不只返回 error 或一段堆栈，还要返回失败原因、可修正参数、是否可重试和建议下一步。
>
> ——第4.2节（article-content.md:337）

截断策略如何保证链接完整？
- **明确告知"不完整"**：模型知道当前看到的是部分结果，不是全部，不会误把截断位置当终点
- **附上总量**：模型知道总共有多少，能判断还需要读多少
- **附上截断位置**：模型知道从哪里继续读，不需要从头开始
- **提供继续读取方法**：有明确的"下一页"或"继续读取"的工具调用方式，链接到剩余内容

错误返回也是同理：不只返回error，而是返回失败原因、可修正参数、是否可重试、建议下一步——这保证了"错误→修正→重试"这条链接的完整性，不会因为信息不足导致链路断裂。

### 1.4 双向收敛：Compress（合并）与Isolate/Retrieve（拆分）的动态调整

A的"双向收敛"要素要求"支持拆分也支持合并的动态调整"。Context Engineering的Compress和Isolate/Retrieve动作构成了完美的双向收敛机制：

**一个方向：Compress（压缩/合并）——从细粒度到粗粒度**

> **4. 压缩（Compress）**：长内容外置到文件、只留结论与证据位置，同时清理窗口里过期或重复的内容；
>
> ——第04章（article-content.md:299）

什么时候需要压缩（合并）？
- 上下文过长，接近窗口上限
- 历史对话中的细节已不再需要，但结论需要保留
- 有重复或过期内容需要清理
- 长内容的细节已外置到文件，当前只需要摘要和指针

压缩的本质是**合并细粒度单元为粗粒度单元**：把多轮对话的细节合并成摘要，把长文件内容合并成"结论+文件位置指针"，把多个相关观察合并成总结。这是粒度变粗的方向。

**另一个方向：Isolate/Retrieve（隔离/检索）——从粗粒度到细粒度**

> **3. 检索（Retrieve）**：当前不在手的信息，从历史会话、资料库、工具目录里按需捞进来（这是 pull）；
>
> **5. 隔离（Isolate）**：用独立会话或 Sub-agent 处理旁支任务，只把结果带回主线，避免污染主上下文。
>
> ——第04章（article-content.md:297,301）

什么时候需要检索/隔离（拆分）？
- 当前粒度太粗，缺少细节需要深挖
- 出现旁支任务，需要独立上下文处理
- 需要从历史/外部获取之前没有的细粒度信息
- 主上下文已经压缩过，现在需要找回之前的细节

检索/隔离的本质是**拆分粗粒度单元为细粒度单元**：从摘要回到原始文件内容，从主任务拆分出子任务交给Sub-agent深入处理，从历史中捞取之前被压缩掉的细节。这是粒度变细的方向。

**双向收敛的动态循环：**

这不是单向过程，而是根据任务阶段动态调整的循环：
1. **任务开始**：粒度较粗——加载Skill摘要、项目概况，Retrieve必要背景
2. **深入执行**：粒度变细——加载完整Skill文档、读取相关文件、Isolate旁支任务
3. **阶段性总结**：粒度变粗——Compress已完成部分，只留结论和进度
4. **遇到新问题**：粒度再次变细——Retrieve更多信息、Isolate新的子任务
5. **任务收尾**：粒度最粗——Compress所有过程，只留最终结果和关键证据位置

文章中Memory的分阶段注入也是双向收敛的体现：
> 记忆的注入分阶段进行：冷启动时只注入少量高置信、高相关的人与项目摘要；请求理解时根据 query 激活候选 memory cards（仍保留来源和置信，不当成确定前提）；执行中需要证据时再回查原始会话或文件；任务收尾时从结果和用户纠正中提取候选记忆，做去重、冲突检查和作用域判定。
>
> ——第5.3节（article-content.md:417）

从冷启动（粗粒度摘要）→执行中（细粒度原始证据）→收尾（去重合并后的新记忆），这也是一个完整的双向收敛循环。

## 2. Prompt Cache与A的链接完整/双向收敛深度分析

Prompt Cache前缀复用规则虽然最初是为了成本优化（缓存命中率）设计，但它与A原子化的理念高度一致，是链接完整和双向收敛的工程化典范。

### 2.1 稳定前缀→导航成本低（缓存命中）

> ·System Prompt、基础工具定义、长期规则放前面，保持内容与顺序稳定；
>
> ——第4.1节（article-content.md:317）

稳定前缀对应A理念中的"导航成本低"：
- **认知地图稳定**：模型每轮看到的开头部分都是一样的，形成稳定的"认知地图"——知道在哪里找角色定义、在哪里找工具列表、在哪里找规则
- **引用位置固定**：模型可以稳定地引用"系统提示中说的XX原则"，不需要每轮重新定位
- **缓存复用降低计算成本**：前缀不需要重复计算，这是工程上的"导航成本低"——模型不需要每轮重新"理解"这些稳定信息

这对应U型曲线的右侧：如果前缀每轮都变，模型需要反复"重新导航"到基础信息，导航成本极高。

### 2.2 动态追加→认知负荷可控（只看新内容）

> ·对话历史采用追加方式保存，不修改已发送过的消息；
>
> ·当前文件、任务进度、时间、工具结果、新加载的 Skill 等动态内容追加到后面；
>
> ——第4.1节（article-content.md:319-321）

动态追加对应A理念中的"认知负荷可控"：
- **新信息都在末尾**：模型每轮只需要重点关注末尾新增的内容，不需要重新阅读整个上下文
- **已处理过的信息不变化**：之前阅读和理解过的内容保持稳定，不需要重新处理
- **认知增量式更新**：模型的认知是增量更新的，而不是每轮从头构建

这对应U型曲线的左侧：如果所有内容都混在一起且不断变化，认知负荷会极高——模型不知道哪些是新的、哪些是旧的、哪些变了。

### 2.3 前缀变化→缓存重算的代价→"只在必要时接受前缀变化"

> ·只在上下文过长需要压缩、或纠正错误信息时，才接受前缀变化和缓存重算。
>
> ——第4.1节（article-content.md:325）

为什么前缀变化是"昂贵"的？从A角度看，前缀变化同时破坏两个要素：
1. **链接完整性被破坏**：之前基于前缀位置建立的所有引用关系都失效了——"前面第X条说的Y"现在可能在不同位置
2. **导航成本急剧上升**：缓存失效需要重新计算整个前缀，模型需要重新"理解"整个上下文，而不是只处理增量

因此，文章明确划定了前缀变化的**仅有的两个合理场景**：
- **场景1：上下文过长需要压缩**——这是双向收敛中"合并"方向的必要操作，虽然破坏链接但为了避免认知负荷爆炸必须做
- **场景2：纠正错误信息**——错误信息如果继续留在前缀会持续污染后续推理，这是为了认知正确性必须付出的代价

这两个场景本质上都是A的"双向收敛"在起作用：当当前粒度导致系统偏离最优平衡点（要么认知负荷过高，要么信息有错误），就需要付出代价进行调整，重新收敛到新的平衡点。

### 2.4 Prompt Cache五条规则的A视角解读

| Prompt Cache规则 | A要素对应 | A视角解读 |
| --- | --- | --- |
| System Prompt/工具定义/长期规则放前面保持稳定 | 链接完整+导航成本低 | 基础认知单元位置固定，引用关系不破坏，导航成本最低 |
| 对话历史只追加不修改 | 链接完整 | 历史信息的标识和关系稳定，不出现"时间线改写"导致的链接断裂 |
| 动态内容追加到后面 | 认知负荷可控 | 新信息有固定位置（末尾），模型增量处理，认知负荷最小 |
| 工具/Skill按需加载 | 粒度寻优+单元独立 | 不一次性加载所有单元（粒度过粗），按需加载保持最优粒度；各单元独立可按需加载 |
| 只在压缩/纠错时接受前缀变化 | 双向收敛 | 粒度动态调整是允许的，但只在偏离平衡点时才付出代价调整 |

## 3. 意图识别前置路由与A粒度判断

A的层级归属是"认知层（粒度判断）→执行层（拆分/合并操作）"。文章中"意图识别是渐进式加载的前置环节"这句话，完美对应了A的认知层粒度判断先于执行层操作。

### 3.1 先选对方向（意图识别）→再按需展开（渐进式加载）

> 意图识别是这一思路的前置环节。用户的一句话背后可能对应问答、改代码、检索资料或调用外部系统等不同任务。系统先识别意图，再据此决定加载哪一类工具、Skill 和 MCP，让后续上下文只包含与当前任务相关的能力。意图识别负责"先选对方向"，渐进式加载负责"再按需展开"，两者共同控制上下文规模。
>
> ——第4.2节（article-content.md:348）

这与A的"认知层→执行层"层级完全对应：

**认知层（粒度判断）：意图识别**
- 判断用户当前意图属于哪个大类（问答/改代码/检索/调用外部系统）
- 这是"粗粒度分类"——先确定大方向，不急于加载细节
- 判断"应该加载哪一类能力"，而不是"加载所有能力"
- 对应A的"粒度寻优"中的U型曲线判断：如果方向选错了，后续加载再多细粒度信息也是错的——导航成本再低也没用，因为导航到了错误的地方

**执行层（拆分/合并操作）：渐进式加载**
- 根据意图识别结果，加载对应类别的工具/Skill/MCP
- 默认先加载名称和简介（粗粒度），确认需要时再加载完整内容（细粒度）
- 对应A的"单元独立"（每个工具/Skill独立加载）和"双向收敛"（按需调整加载粒度）

### 3.2 两者共同控制上下文规模——粒度寻优的两级调节

意图识别+渐进式加载构成了粒度寻优的两级调节机制：

**第一级：意图识别（大类粒度控制）**
- 解决"要不要加载X类能力"的问题
- 如果用户只是问一个简单问题，就不加载改代码相关的重量级工具
- 这是"大闸门"——避免无关类别的信息进入上下文
- 如果这一步做错了（意图识别错误），后续渐进式加载再精细也没用——会加载错误类别的能力

**第二级：渐进式加载（细节粒度控制）**
- 解决"加载X类能力后，加载多少细节"的问题
- 默认只加载名称简介，需要时再加载完整内容
- 这是"细调节"——在正确类别内保持最优粒度
- 如果这一步做错了（要么全加载要么全不加载），要么认知负荷过高要么导航成本过高

两级调节共同作用，才能让上下文规模保持在U型曲线最低点附近：
- 没有意图识别（只有渐进式加载）：会渐进式加载错误类别的能力，粒度过粗但方向错误
- 没有渐进式加载（只有意图识别）：一次加载正确类别的所有细节，粒度过粗导致认知负荷爆炸

## 5. 差异与补充

### 5.1 七概念A强调"U型曲线评分公式"，文章中是否有定量评估粒度的机制？

**七概念A的定量机制：**
> 粒度评分公式：score = 100 - |cognitive_load - navigation_cost|×10 - max(0, cognitive_load+navigation_cost-10)×5，合格≥70分

**文章中的对应情况：**

文章中**没有给出显式的定量评分公式**，但有多个隐含的定量/半定量评估指标：

1. **缓存命中率**：
   > "随着用户对积分（成本）越来越敏感，缓存命中率正在成为被普遍关注的工程指标。"
   > ——第4.1节（article-content.md:327）
   
   缓存命中率可以间接反映粒度组织的合理性：前缀越稳定、动态内容追加越规范，缓存命中率越高。但缓存命中率只反映了"导航成本"（计算成本）一侧，没有直接反映认知负荷。

2. **上下文窗口使用率**：
   文章多次提到"上下文过长需要压缩"、"接近上下文上限时模型会降低完成标准"，隐含了"上下文长度/窗口上限"这个比例指标。但这是一个粗略指标，不是cognitive_load和navigation_cost的平衡度量。

3. **工具选择准确率**：
   文章提到"工具越多、语义越重叠，模型选择越困难"，渐进式加载的目标之一是降低选择干扰。工具选择准确率可以作为粒度合理性的间接指标——如果粒度合适（相关工具都加载了，不相关的没加载），工具选择准确率应该更高。

**对七概念A的启示：**
七概念A的评分公式是理论框架，但在工程实践中，cognitive_load和navigation_cost很难直接度量。文章的工程实践给出了几个可度量的代理指标（proxies）：
- 导航成本侧：缓存命中率、工具检索时间、需要反复检索的次数
- 认知负荷侧：工具选择错误率、模型"忘记"之前信息的频率、无关信息导致的错误率
- 综合侧：任务完成率、首次执行正确率、平均纠正轮数

未来可以考虑把这些工程可度量的指标映射回评分公式，让A的粒度评估从理论走向可操作。

### 5.2 七概念A强调"双向收敛"（支持合并也支持拆分），文章中Compress和Isolate/Retrieve是否对称？

**表面上看是对称的：**
- Compress = 合并（细→粗）
- Isolate/Retrieve = 拆分（粗→细）

**但仔细分析存在不对称性：**

1. **触发时机不对称**：
   - Compress通常是**被动触发**：上下文过长、接近窗口上限、内容过期时才触发
   - Isolate/Retrieve通常是**主动触发**：遇到新任务、需要更多细节、出现旁支时主动触发
   
   文章中说"只在上下文过长需要压缩、或纠正错误信息时，才接受前缀变化"——压缩是"不得不做"的时候才做；而检索是"需要时就做"。

2. **信息保留不对称**：
   - Compress是**有损**的：长内容外置后，文件里可能保留了完整内容，但上下文里只留结论和指针——部分细节信息丢失了（至少在当前上下文中丢失）
   - Isolate/Retrieve是**无损**的：检索回来的信息是完整的，Sub-agent返回的结果也是完整的——信息是增加而不是减少

3. **代价不对称**：
   - Compress需要付出**前缀变化+缓存重算**的代价
   - Isolate/Retrieve主要付出**额外工具调用+上下文增长**的代价

4. **可逆性不对称**：
   - Compress后如果需要找回细节，可以通过Retrieve从文件中捞回来——可逆
   - Isolate/Retrieve后如果觉得信息太多，可以再次Compress——也可逆
   - 这一点上是对称的

**为什么会有这种不对称？这反映了什么本质？**

这种不对称不是设计缺陷，而是反映了认知过程的本质：
- **获取信息（拆分/检索）是常态**：执行任务过程中不断需要新信息，这是正向的认知构建过程
- **压缩信息（合并/总结）是非常态**：只有当信息多到影响认知时才需要压缩，这是认知过载时的防御机制

这对七概念A的启示是：双向收敛不是"一半时间合并一半时间拆分"的对称过程，而是"以拆分/检索为主、合并/压缩为辅"的非对称动态平衡。就像人的呼吸：吸气是获取氧气（常态），呼气是排出二氧化碳（必要但不是目的）。

### 5.3 文章中"渐进式加载"比七概念A的描述更工程化，对七概念有何启示？

七概念A的描述目前还停留在方法论层面（U型曲线、四要素、评分公式），文章中的渐进式加载给出了工程化落地的具体范式，对七概念有三点重要启示：

**启示1：粒度寻优应该是"多级渐进"而非"一次选对"**

七概念A说"找到U型曲线最低点"，容易让人误以为是一次性找到最优粒度就不动了。但渐进式加载展示的是多级渐进过程：
- 级别1：只看到类别（意图识别后加载对应类别）
- 级别2：只看到名称和简介（默认暴露）
- 级别3：看到完整内容（按需加载）
- 级别4：看到引用的参考资料和脚本（深入需要时）

这不是一次选一个粒度点，而是建立一个"粒度阶梯"，让系统可以在阶梯上平滑上下移动。对七概念A的启示：应该把"粒度寻优"从"找最优点"扩展为"设计粒度阶梯+升降机制"。

**启示2：单元独立需要配套"发现机制"**

七概念A强调"单元独立"（各模块单一职责独立可读），但容易忽略一个问题：单元独立了之后，怎么找到需要的单元？

文章给出的答案是"能力发现机制"：
> WorkBuddy 采用分阶段的能力发现机制：先让模型看到工具名称和简要描述，再根据任务需要加载更具体的工具说明，同时也提供了工具检索能力。
>
> ——第4.2节（article-content.md:339）

单元独立只是基础，还需要：
- 统一的"名称+简介"元数据格式，让单元可被快速扫描
- 检索能力，让需要的单元可以被找到
- 按需加载机制，不需要的时候不加载

对七概念A的启示：应该在"单元独立"要素之外，补充"单元可发现"——独立但找不到的单元是没有用的。

**启示3：链接完整需要"不完整提示+续读指引"**

七概念A强调"链接完整"（单元间引用关系完整可追溯），但容易假设"信息是完整的"。文章的截断策略告诉我们：在工程实践中，信息经常是不完整的（被截断、被分页、被外置）。

真正的链接完整不是"所有信息都在眼前"，而是：
- 明确告知"这里不完整"（避免误解）
- 告诉"总共有多少"（建立预期）
- 告诉"从哪里继续"（提供链接）
- 告诉"怎么继续"（提供方法）

这是一种"不完整情况下的链接完整"——比理想状态的"所有链接都完整"更符合工程实际。对七概念A的启示：链接完整不仅包括"完整信息的引用完整"，也应该包括"不完整信息的续读链路完整"。
