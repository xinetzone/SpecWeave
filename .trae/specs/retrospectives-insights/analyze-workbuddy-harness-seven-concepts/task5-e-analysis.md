# Task 5：E（萃取）维度映射分析

---

## 1. Skill对程序性知识的萃取

分析文章中Skill如何完整体现E萃取四要素：显化转换、抽象提升、漏斗过滤、形式化编码。

### 1.1 显化转换：隐性经验→显性知识编码

**E要素定义**：将隐性经验（"我知道怎么做但说不清楚"）转换为可传递、可审查、可执行的显性知识编码。

**文章中的体现：**

Skill的核心作用就是把"高手知道怎么做PR"这种隐性经验，显化为模型能读懂、能执行的结构化文档。

**原文证据：**
> "以在某个仓库提交 PR 为例，它不止调一次 API——还要读仓库规则、执行测试、生成变更说明、处理失败，最后才创建。Skill 把这些经过验证的工作方法保存下来，通常包含说明、步骤、脚本、命令和判断标准。"
> ——第2.4节（article-content.md:165）

**显化转换的四个层次：**

| 隐性经验 | 显性编码到Skill中 |
| --- | --- |
| "做PR前应该先看看仓库规则" | 流程第1步：读取 AGENTS.md / WORKBUDDY.md 和仓库贡献规范 |
| "别覆盖用户的未提交修改" | 流程第2步：检查 git status，不覆盖用户的未提交修改 |
| "改完代码要跑测试" | 流程第4步：运行与改动相关的格式化、类型检查和测试 |
| "要诚实说明哪些没验证" | 完成标准：已记录所运行的测试与结果；如实列出未验证项 |

**关键洞察**：文章中Skill不仅编码"正常流程怎么做"，还编码了"失败分支怎么处理"——
> "Skill 里还要写清失败分支：测试失败如何判断、没有权限就停在草稿。"
> ——第2.4节（article-content.md:177）

失败分支的显化是萃取成熟度的重要标志：新手只知道"正常路径"，高手知道"哪里会出错以及怎么处理"。Skill把这两种知识都显化了。

**显化转换的证据链：**

从第5.2节程序性记忆的论述可以反推：如果不显化，这些经验会以什么形式存在？
- 作为开发者的个人隐性知识，存在于脑子里，无法直接传给Agent
- 作为零散的历史对话，Memory可能错误地将局部经验升为通用策略
- 作为System Prompt中的零散提示，没有结构、没有验证标准、没有失败处理

Skill通过显化转换，把这些隐性知识变成了Agent能直接读取和执行的结构化资产。

### 1.2 抽象提升：具体事件→通用模式

**E要素定义**：从"这次做PR成功了"的具体事件经验，提升为"这类任务应该怎么做"的通用流程模式；抽象层级从具体实例跃迁到类别模式。

**文章中的体现：**

Tool和Skill的核心区分，本质就是"单次动作"vs"一类任务的模式"：
> **Tool 负责"一个动作"，Skill 负责"一类任务的做法"。**
> ——第2.4节（article-content.md:171）

**抽象提升的过程分析：**

以"创建PR"Skill为例，抽象提升经历了三个层级：

1. **L0具体事件层**："上次我改了auth.py，我是这样做PR的：先跑lint，再跑test_auth.py，然后写了一个标题……"
   - 这是具体经验，包含大量偶然细节（改的是哪个文件、跑了哪个具体测试）
   - 不能直接复用：下次改的可能不是auth.py，需要跑的测试也不同

2. **抽象中间层**："做PR大致需要：读规则→检查状态→看diff→跑测试→写描述→确认后创建"
   - 去掉了偶然细节（具体哪个文件），保留了任务本质结构
   - 但还不够精确：什么叫"相关测试"？失败了怎么办？什么情况要停下来？

3. **L2模式层（Skill）**：成为文章中展示的结构化Skill：
```markdown
适用场景：用户要求为当前仓库的代码变更创建 PR。
流程：
1. 读取 AGENTS.md / WORKBUDDY.md 和仓库贡献规范。
2. 检查 git status，不覆盖用户的未提交修改。
3. 阅读 diff，识别变更范围与风险。
4. 运行与改动相关的格式化、类型检查和测试。
5. 仅在验证通过后生成标题和描述，如实列出未验证项。
6. 确认用户已授权发布，再 push 并创建 PR。
完成标准：
- PR 只包含本任务的变更。
- 已记录所运行的测试与结果。
- 标题、正文、Issue 关联符合团队规范。
```
   - 有明确的**适用场景**（什么时候用这个Skill）
   - 有**可执行的步骤**（每一步做什么）
   - 有**完成标准**（怎么算做好了）
   - 有**失败分支**（隐含在步骤中：验证不通过就不继续，没授权就不push）

**抽象提升的关键标志：**
- ✅ 从"这次怎么做"变成"这类任务怎么做"
- ✅ 去除了情境化的偶然细节，保留了跨情境稳定的流程结构
- ✅ 增加了"适用场景"声明，明确模式的适用边界
- ✅ 增加了"完成标准"，明确模式的终点判断

原文中另一个抽象提升的例子是Plugin——比Skill更高一层的抽象：Plugin不是"某一类任务"的模式，而是"某一整套能力组合"的模式（研发工作流包含MCP+Skills+Rules+Hooks+Templates）。

### 1.3 漏斗过滤：经验→模式的质量门槛

**E要素定义**：不是所有经验都能升为模式；萃取过程是四层漏斗（事件→洞察→模式→原则）逐级精炼，每一层都有质量门槛，不合格的经验被过滤掉不进入下一层。

**文章中的体现：**

"为什么不把Procedural Memory放入长期记忆"这一决策，本质就是E漏斗的过滤机制——不是所有做事方法都有资格升为可复用的Skill，必须经过质量门槛。

**原文证据（关键引用）：**
> "所以 WorkBuddy 的选择是：**陈述性记忆放进Memory，经过验证的工作方法保存为Skill** ——Skill 可版本化、可评审、可测试、可回滚、按需加载。"
> ——第5.2节（article-content.md:395）

这"五个可"（版本化、可评审、可测试、可回滚、按需加载）就是Skill作为L2模式的质量准入门槛！

**Skill的漏斗过滤机制分析：**

四层漏斗对应到Skill的准入过程：

| 漏斗层级 | 对应文章内容 | 过滤/准入标准 |
| --- | --- | --- |
| **L0 事件** | 单次任务中Agent或人的成功/失败操作 | 没有过滤：所有发生过的事都是事件，记录在日志/历史中 |
| **L1 洞察** | 从事件中总结的初步规律（如"做PR前要先读规则"） | 过滤1：必须从多次重复事件中总结，单次偶然不算 |
| **L2 模式（Skill）** | 经过验证的工作方法 | 过滤2（"五个可"门槛）：<br>• 可版本化：有版本历史，知道改了什么<br>• 可评审：人可以审查Skill内容是否正确<br>• 可测试：Skill的步骤可以被验证<br>• 可回滚：Skill出问题可以回退到旧版本<br>• 按需加载：不是所有任务都加载，只在匹配时加载 |
| **L3 原则** | 跨Skill的通用设计原则（如前馈+反馈、计算型信号优先） | 过滤3：必须在多个不同Skill/场景中都成立，才能升为原则 |

**四个风险点如何对应"过滤"的必要性：**

文章第5.2节列出的程序性记忆注入长期记忆的四个风险，恰恰是因为这些经验**没有经过漏斗过滤**就被升为通用策略：

1. **局部经验被误升为通用策略** → 缺少"抽象提升"过滤：把L0具体事件直接当成L2模式
   - 过滤机制：Skill要求显式声明"适用场景"，不声明就是全局适用就不能入库
2. **干扰模型推理** → 缺少"按需加载"过滤：把经验默认注入所有任务，而不是匹配时才加载
   - 过滤机制：Skill按需加载，不是全局默认注入
3. **隐性改写Agent行为** → 缺少"可评审/可版本化/可回滚"过滤：经验悄悄改变行为，没有人审查、没有版本、出问题回不去
   - 过滤机制：Skill是显式文件，可评审、有版本、可回滚
4. **降低泛化与可控性** → 缺少"可测试"过滤：经验没有经过验证就被复用，不知道什么时候会失效
   - 过滤机制：Skill有完成标准和失败分支，可测试是否按预期工作

**关键洞察**：漏斗过滤不是"可选优化"，而是萃取的核心质量保障。没有过滤的萃取就是"污染"——把局部经验、错误经验、过时经验都当成模式复用，反而降低系统可靠性。

### 1.4 形式化编码：结构化表示支持复用检索

**E要素定义**：经过漏斗过滤的模式，需要用结构化格式编码，支持被Agent/人检索、理解、执行、验证。

**文章中的体现：**

Skill有明确的结构化格式，从文章中的PR Skill示例可以提取出Skill的形式化编码结构：

**Skill的结构化格式要素：**

| 编码要素 | PR Skill示例中的体现 | 作用 |
| --- | --- | --- |
| **适用场景** | "适用场景：用户要求为当前仓库的代码变更创建 PR。" | 检索触发：Agent看到任务时判断"该不该加载这个Skill" |
| **流程步骤** | 1-6步明确编号 | 执行指导：每一步做什么，顺序是什么 |
| **约束条件** | "不覆盖用户的未提交修改""仅在验证通过后生成""确认用户已授权发布" | 边界保护：什么不能做、什么条件下才能继续 |
| **完成标准** | 三条明确标准（只包含本任务变更、记录测试结果、符合规范） | 验收判断：怎么算"做好了"，避免过早判定完成 |
| **失败分支** | 隐含：验证不通过则不继续、没授权则停在草稿 | 异常处理：出问题怎么办，不是只有happy path |
| **脚本/命令** | Skill通常包含"脚本、命令"（文章第165行提到） | 执行载体：可执行的脚本而不仅是文字描述 |

**形式化编码对复用的支持：**

1. **可检索**：有"适用场景"字段，Agent可以通过语义匹配找到该用哪个Skill，而不是把所有Skill都加载进上下文
   - 这对应文章第4.2节的"渐进式加载"："先看名称和描述，确认适用后再读完整 SKILL.md"

2. **可理解**：结构化格式（步骤+约束+标准）比自然语言散文更容易被模型准确理解，减少歧义
   - 对比：如果PR流程只是一段自然语言描述，模型可能漏掉某一步或理解错顺序

3. **可执行**：步骤是编号的祈使句（"读取""检查""运行""生成"），可以直接转化为Agent的行动序列
   - 文章第165行：Skill包含"脚本、命令"——可直接执行，不只是指导

4. **可验证**：有明确的完成标准，Agent执行完可以对照检查"我是不是真的做完了"
   - 这对应Harness反馈层：Skill自带验收标准，不需要外部猜测"做好了吗"

5. **可组合**：因为每个Skill有明确边界（适用场景+完成标准），所以可以组合使用
   - 原文证据："一个'发周报'Skill 可能同时调用腾讯文档 MCP、知识库 MCP 和本地脚本"（article-content.md:177）

**MCP三类原语的形式化层级对比：**

MCP的Resources/Tools/Prompts三类原语也体现了不同层级的形式化编码：
- Resources：L0/L1层——原始数据，结构松散，需要Agent自己解读
- Tools：L1/L2层——有Schema的动作接口，参数明确，但不包含流程
- Prompts：L2层——预组织的消息模板，是更高阶的提示词模式

## 2. Plugin对能力组合的打包分发

分析Plugin作为更高层级萃取的角色：Plugin = MCP + Skills + Rules + Hooks + Templates的组合，如何体现从单一模式到能力包的萃取。

### 2.1 Plugin是L2到L2.5的萃取跃迁

Skill萃取的是"一类任务怎么做"（如create-pr、debug-ci），而Plugin萃取的是"一整个工作场景需要哪些能力组合"（如team-dev-workflow）。

**原文证据：**
> "MCP 负责连接外部系统，Skill 负责保存任务流程。但真实场景里，一个完整能力往往不只包含单个工具或单个流程，而是一组连接、流程、规则、Hooks 和模板。"
>
> "Plugin 解决的是'能力组合如何安装和分发'的问题。它把多种相关能力组合成可安装、可分发的单位。"
> ——第2.5节（article-content.md:181-183）

**Plugin的组成结构：**
```bash
Plugin: team-dev-workflow
├─ MCP：读写 Issue、MR、构建结果和内部文档        # 连接能力
├─ Skills：/issue-start、/create-pr、/debug-ci、/release  # 任务流程
├─ Rules：分支、Commit、安全、架构规范              # 约束规则
├─ Hooks：编辑后 lint，提交前测试，发布前审批       # 自动化触发
└─ Assets / Templates：PR 模板、发布说明、架构图    # 可复用资产
```
——第2.5节（article-content.md:191-193）

### 2.2 Plugin如何体现E四要素

| E要素 | Plugin中的体现 |
| --- | --- |
| **显化转换** | 把"一个研发团队需要搭建的整套AI协作环境"这种隐性需求，显化为可安装的能力包；以前团队要自己一个个配MCP、写Skill、加Rules，现在Plugin把这些隐性依赖都显化打包了 |
| **抽象提升** | 从"我们团队是这么配置的"具体经验，提升为"研发团队通用工作流"模式；不是为某个团队定制，而是抽象出研发团队通用的能力组合模式 |
| **漏斗过滤** | Plugin内的每个组件都经过验证：MCP是稳定的连接器、Skills是经过验证的流程、Rules是团队共识规范、Hooks是经过测试的自动化检查；不是随便把东西拼起来就叫Plugin，而是"能工作的组合" |
| **形式化编码** | Plugin有明确的结构定义（MCP+Skills+Rules+Hooks+Templates），支持按作用域安装（团队/项目/个人），是可分发、可安装的标准化单位 |

### 2.3 MCP vs Skill vs Plugin的萃取层级对比

| 概念 | 解决的问题 | 萃取层级 | 组合性 | 例子 |
| --- | --- | --- | --- | --- |
| **Tool** | 一个动作怎么执行 | L1（原子动作） | 最小组合单元 | read_file、run_command |
| **MCP** | 外部系统怎么接入 | L1.5（连接模式） | 提供一组Tools/Resources/Prompts | GitHub MCP、腾讯文档MCP |
| **Skill** | 一类任务怎么做 | L2（流程模式） | 组合多个Tools/MCP调用 | create-pr、debug-ci |
| **Plugin** | 一整套能力怎么打包分发 | L2.5（能力包模式） | 组合MCP+Skills+Rules+Hooks+Templates | team-dev-workflow |

**层级关系**：Plugin包含Skill，Skill调用MCP/Tool；每一层都是更高阶的组合萃取。

### 2.4 Plugin的"按需加载"同样是漏斗过滤

文章在Skill部分强调"按需加载"，Plugin同样遵循这个原则：
- Plugin不是全局默认安装所有能力，而是"按团队、项目或个人作用域安装"
- 一个Python后端团队不需要安装前端开发Plugin
- 这本质也是E漏斗的过滤：只有当前场景需要的能力组合才被加载，避免无关能力干扰

## 3. E四层漏斗在文章中的对应

映射四层知识漏斗（L0事件→L1洞察→L2模式→L3原则）到Agent知识管理体系。

| 漏斗层级 | 七概念定义 | Agent工程对应 | 原文证据 | 具体例子 |
| --- | --- | --- | --- | --- |
| **L0 事件** | 原始事实、具体发生的事 | 工具执行结果、日志、交互历史、Tool Result、用户输入输出 | "Agent 校验参数、检查权限，执行 API、脚本或本地函数；Agent 把执行结果作为 tool result 放回上下文"（article-content.md:77-79） | 一次git status的输出、一次测试失败的错误信息、用户说"帮我创建PR"这句话 |
| **L1 洞察** | C→M→A→B四元组可证伪规律，从事件中总结 | 从错误中总结的规则、单次任务后总结的注意事项、"先读再写"这类经验法则 | "工具结果过长时……截断时明确告诉模型'结果未完整'，附上总量、截断位置和继续读取方法，否则模型会把前100条误当成全部"（article-content.md:337） | "改文件前必须先读最新版本，避免覆盖用户修改""错误返回不能只有堆栈，要说明下一步怎么做" |
| **L2 模式** | ≥2个案例验证+有反模式配套的通用模式 | Skill（经过验证的工作方法）、Rules、Hooks | "Skill 把这些经过验证的工作方法保存下来，通常包含说明、步骤、脚本、命令和判断标准"（article-content.md:165）；"陈述性记忆放进Memory，经过验证的工作方法保存为Skill"（article-content.md:395） | create-pr Skill、提交前lint Hook、代码规范规则、PR模板 |
| **L3 原则** | 从第一性原理推导的跨领域公理，适用范围最广 | Harness设计原则、Context Engineering原则、Agent架构原则 | "能用计算型信号解决的问题，优先交给确定性程序"（article-content.md:501）；"System Prompt 只能引导，不能强制"（article-content.md:123）；"前馈提高第一次就做对的概率，反馈让Agent自我纠正"（article-content.md:497） | 前馈+反馈控制原则、计算型信号优先原则、能力分层接入原则（Tool→MCP→Skill→Plugin） |

### 3.1 四层漏斗的逐级精炼过程示例

以"文件覆盖问题"为例，展示一个知识如何从L0事件逐层萃取到L3原则：

**L0事件**：某次任务中，Agent读取了文件A，然后用户在外部修改了文件A，Agent写回时覆盖了用户的修改，导致用户工作丢失。
- 这是单次事件，记录在日志里，只是一个事实。

**L1洞察**：从这个事件+其他几个类似事件，总结出：
> C（条件）：Agent先读文件、然后过了一段时间才写回
> M（机制）：文件可能在读取后被其他进程/用户修改，直接写回会覆盖
> A（行动）：写回前检查文件最后修改时间，如果和读取时不一致就要求重读
> B（结果）：避免覆盖外部修改
- 这是"时间戳校验"洞察，有条件、机制、行动、结果。

**L2模式**：这个洞察被整合进Skill和Hooks：
- 所有涉及文件修改的Skill都包含"读取→修改→写入前校验"流程
- 有一个通用的edit-file Hook自动做时间戳校验
- 成为所有文件操作的标准模式
- 文章中提到："编辑前的时间戳校验：Agent 改文件前比较'上次读取时间'和'文件最后修改时间'，若读取后被用户改过就拒绝写入、要求重读，避免覆盖最新修改。"（article-content.md:521）

**L3原则**：进一步抽象为跨领域原则：
> "前馈+反馈"原则：执行前提供规则（前馈），执行后验证结果（反馈）
> "不信任假设"原则：不要假设环境在你操作期间保持不变，操作前要验证
- 这个原则不仅适用于文件修改，还适用于API调用、外部系统操作、并发场景等。

### 3.2 每层的过滤机制

| 层级 | 准入标准 | 什么会被过滤掉 |
| --- | --- | --- |
| L0→L1 | 事件重复发生≥N次，或一次事件风险极高；可以总结出因果机制；有明确的应对行动 | 单次偶然事件、无法总结规律的事件、纯噪声 |
| L1→L2 | 洞察在≥2个不同场景/任务中验证有效；有明确的适用边界；可以形式化为流程/规则/Hook；有反模式（什么情况下不适用） | 只在单一情境下有效的洞察、无法形式化的"感觉"、没有验证过的想法 |
| L2→L3 | 模式在≥3个不同领域/问题域都成立；可以从第一性原理推导出来，不只是经验归纳；跨团队/跨产品验证有效 | 只适用于特定技术栈/产品/团队的模式、经验性总结无法解释"为什么"、反例太多 |

## 4. Procedural Memory → Skill的关键决策分析

深度分析"为什么不把Procedural Memory放入长期Memory而是保存为Skill"这一决策：这一决策本质上是E漏斗中的"过滤"环节——不是所有经验都能升为模式。

### 4.1 四个风险点对应E萃取的质量门槛

文章第5.2节列出四个风险，逐一对应E萃取必须具备的质量属性：

| 风险点（如果程序性记忆进Memory） | 对应的E萃取质量门槛 | Skill如何满足这个门槛 |
| --- | --- | --- |
| **1. 局部经验被误升为通用策略**：一次任务里有效的步骤，换个项目未必成立 | **抽象层级正确性**：必须明确区分"局部经验"和"通用模式"，不能把L0事件直接当成L2模式 | Skill要求显式声明"适用场景"，不标注适用场景就是全局适用就不能入库；不是自动从单次经验学习，而是人工提炼明确适用范围 |
| **2. 干扰模型推理**：模型可能不再根据当前证据选路径，而是复用历史步骤，陷入局部最优 | **按需加载而非全局注入**：模式只在匹配时加载，不影响不相关任务的推理 | Skill是"按需加载"：先看名称和描述，确认适用再加载完整内容；不相关的Skill不会进入上下文干扰推理 |
| **3. 隐性改写Agent行为**：程序性内容实际上接近动态System Prompt，却可能缺少版本、评测、审批和回滚 | **可治理性**：模式必须是显式的、可审查的、可版本控制的、出问题能回滚的 | Skill"可版本化、可评审、可回滚"：Skill是显式文件，不是隐藏在Memory中的隐性影响；人可以审查Skill内容，可以版本控制，出问题可以回退到旧版本 |
| **4. 降低泛化与可控性**：如果Agent从每次任务自动学工作流，又没有稳定的奖励信号和Eval，系统很难判断它是在改进还是在积累偏见 | **可验证性**：模式必须经过验证，有验收标准，能判断它是在变好还是变差 | Skill"可测试"：有明确的完成标准和失败分支；不是自动学习不加验证，而是人工提炼并经过测试才入库；效果可观测可评估 |

### 4.2 决策本质：E漏斗的质量过滤

这个决策的本质不是"Memory和Skill哪个好"，而是：

> **从经验到模式的萃取必须经过质量漏斗，没有经过验证的经验不能自动升级为影响所有未来任务的模式。**

如果把Procedural Memory直接放入长期Memory自动注入，相当于跳过了E漏斗的L1→L2过滤环节：
- 没有人工评审（局部经验→通用策略的抽象提升没有把关）
- 没有适用范围声明（不知道什么时候该用什么时候不该用）
- 没有版本和回滚（错了改不回去）
- 没有按需加载（所有任务都被影响）
- 没有完成标准（不知道执行得对不对）

文章用一个流程图清晰展示了正确的萃取路径：
```
成功经验 / 稳定流程
        │
        ▼ 人工提炼、明确适用范围和完成标准
Skill（步骤 + 约束 + 脚本 + 验证方法）
        │
        ▼ 版本化、可评审、可测试、可回滚、按需加载
在匹配任务中指导 Agent 行动
```
——第5.2节（article-content.md:397-399）

这个流程就是E萃取漏斗的形象化：每一个"▼"都是一层过滤/精炼。

### 4.3 对七概念E萃取的启示

从WorkBuddy的这个决策中，七概念E萃取框架可以得到三个关键启示：

**启示1：萃取必须有准入门槛，不是所有经验都能成为模式**
- 很多萃取方法论强调"要萃取、要沉淀"，但很少强调"不要什么都沉淀"
- WorkBuddy的实践告诉我们：错误的萃取（把局部经验当通用模式）比不萃取更危险
- 准入门槛就是那五个"可"：可版本化、可评审、可测试、可回滚、按需加载
- 不满足门槛的经验，就让它留在日志/历史里作为RAG检索源，不要升为模式

**启示2：自动萃取风险极高，人机协同是当前可行路径**
- 文章明确选择"人工提炼"而非"Agent自动学习工作流"
- 原因是：没有稳定的奖励信号和Eval，自动学习可能积累偏见而非改进
- 七概念E萃取应该明确：在当前技术条件下，L1→L2（洞察→模式）的跃迁需要人工评审把关；L0→L1可以辅助自动化，但L1以上必须有人参与质量控制

**启示3：模式必须显式化，反对"隐性影响"**
- Procedural Memory进Memory的最大问题是"隐性改写Agent行为"——Agent的行为被改变了，但没有人明确知道是什么改变了它、为什么改变、改了什么
- E萃取的核心价值之一就是"显式化"：模式是可见的、可审查的、可讨论的、可改进的
- 隐性经验如果不能显化，就不应该成为影响系统行为的模式

## 6. 差异与补充

### 6.1 MCP的Resources/Tools/Prompts三类原语是否体现不同层级的萃取？

**答案：是的，三类原语恰好对应L0-L2不同萃取层级。**

| MCP原语 | 驱动方 | 萃取层级 | 形式化程度 | 分析 |
| --- | --- | --- | --- | --- |
| **Resources（资源）** | 应用/Agent驱动 | L0-L1 | 低：有URI标识的只读内容，但结构由资源本身决定，Agent需要自己解读 | 资源是原始数据层，相当于"给你原料自己加工"；对应L0事件和部分L1信息（如文档内容、数据库记录）；萃取程度最低，最接近原始事实 |
| **Tools（工具）** | 模型驱动 | L1-L2 | 中：有明确的参数Schema和描述，是结构化的动作接口，但不包含多步流程 | 工具是动作接口层，相当于"给你工具但你自己决定怎么组合使用"；对应L1洞察层的原子动作模式；有形式化（Schema）但没有流程 |
| **Prompts（提示模板）** | 用户驱动 | L2 | 高：Server预先组织好的可复用消息组，内部还能引用resource | 提示模板是预组织的高阶模式，相当于"给你一个现成的提问/交互模板"；对应L2模式层；萃取程度最高，已经预组织好了指令和内容的组合 |

**关键洞察**：三类原语的"驱动方"差异本质上是"谁来做萃取决策"：
- Resources：Agent决定什么时候读、怎么用 → Agent自己做萃取加工
- Tools：模型决定什么时候调用、怎么组合 → 模型做流程编排
- Prompts：用户（通过斜杠命令）决定什么时候用 → 人提前做了萃取预组织

萃取程度越高，需要使用者做的决策越少，但灵活性也越低。这符合"灵活性vs易用性"的通用权衡。

### 6.2 Plugin是L2还是L3层级？

**结论：Plugin是L2.5层——组合模式层，还没到L3原则层。**

分析：

| 判定维度 | 为什么是L2（模式） | 为什么不是L3（原则） |
| --- | --- | --- |
| **抽象程度** | Plugin是"特定能力组合"模式（如"研发工作流"），仍然是针对某类场景的解决方案 | L3原则是跨领域公理（如"前馈+反馈"），不绑定到特定场景或解决方案 |
| **复用范围** | Plugin在同类团队/同类场景中复用（如所有研发团队可以用team-dev-workflow），但不能跨领域复用（设计师团队不能直接用研发Plugin） | L3原则跨领域复用："前馈+反馈"不仅适用于Agent，还适用于管理、工程、控制系统等 |
| **形式化程度** | Plugin有具体的组件（MCP/Skills/Rules/Hooks），是"可安装的东西" | L3原则是抽象理念，不是可直接安装的组件；原则通过指导L2模式的设计来发挥作用 |
| **稳定性** | Plugin会随工具和流程变化而更新（如GitHub API变了MCP要更新） | L3原则相对稳定，不会因为具体工具变化而改变 |
| **对应文章证据** | "Plugin 是产品层的打包概念"（article-content.md:195）——是产品层的具体打包，不是抽象原则 | 文章中的L3原则是那些不加粗但贯穿全文的理念（如系统思维、分层思维、控制论思维） |

**Plugin的准确位置**：L2单任务模式（Skill）→ L2.5能力组合模式（Plugin）→ L3跨领域原则

### 6.3 七概念E强调"反模式配套"，文章中是否有对应的反模式/失败案例萃取？

**答案：有反模式意识，但还没有像Skill那样系统化、形式化的反模式库；这是文章E体系可以加强的地方。**

**文章中已有的反模式/失败案例萃取：**

| 反模式/失败案例 | 文章位置 | 萃取程度 |
| --- | --- | --- |
| "上下文窗口很大全部放进去" | 第04章（article-content.md:305） | L1：明确指出是误区，说明为什么错、应该怎么做，但没有形式化为独立的反模式条目 |
| "System Prompt能强制模型遵守规则" | 第2.2节（article-content.md:123） | L1-L2：明确否定，给出正确做法（外部约束），但没有作为反模式结构化 |

| "照搬底层API暴露给Agent" | 第2.3节（article-content.md:155） | L1：指出是错误做法，给出正确原则（按用户意图组织） |
| "Agent自我评估/自己写实现自己写测试" | 第6.2节、第08章 | L2：有机制解释（共享误解）、有解决方案（三角色分离） |

**现有反模式萃取的不足（对比七概念E的要求）：**

1. **没有结构化的反模式格式**
   - Skill有明确的SKILL.md格式（适用场景/流程/完成标准）
   - 但反模式没有类似的结构化格式（应该包含：反模式名称、问题描述、机制解释、错误信号、正确做法、例外情况）
   - 反模式目前散落在文章各章节，没有集中管理

2. **反模式没有像Skill那样"版本化、可评审、可测试"**
   - 文章没有提到"反模式库"或"失败案例库"的概念
   - 失败案例是用来说明问题的，但没有作为可复用资产被系统化管理

3. **反模式的"检测信号"萃取不足**
   - 好的反模式不仅告诉你"这是错的"，还要告诉你"怎么检测你正在犯这个错"
   - 文章中部分反模式有信号（如"看到Agent一次同时做很多事情"就是一次承担过多的信号），但没有系统化提取

**对文章E体系的建议（如果进一步完善）：**
- 建立Anti-Skill（反模式库），与Skill配套
- 每个反模式包含：名称、症状（怎么发现）、后果（为什么不好）、机制（为什么会这样）、纠正方法（怎么改）、预防措施（怎么避免）
- 反模式和Skill一样入库、版本化、可评审、按需加载
- 这样就形成了"正确怎么做（Skill）+ 不要怎么做（Anti-Skill）"的完整模式库

### 6.4 文章E体系相比七概念E框架的特色贡献

文章虽然不是专门讲萃取方法论的，但在实践层面为七概念E框架提供了几个非常有价值的具体贡献：

1. **"五个可"作为L2模式准入门槛是重大贡献**
   - 七概念E提到了"漏斗过滤"和"质量门槛"，但没有给出具体可操作的门槛标准
   - 文章的"可版本化、可评审、可测试、可回滚、按需加载"五个可，是非常具体、可操作、可验证的准入标准
   - 建议七概念E将这"五个可"（或适配后的版本）作为L2模式的标准准入门槛

2. **能力分层体系（Tool→MCP→Skill→Plugin）是萃取层级的优秀实例**
   - 七概念E给出了L0-L3四层漏斗，但比较抽象
   - 文章的能力分层是四层漏斗在Agent工程领域的具象化，每层边界清晰、责任明确、有实际例子
   - 这个分层模型可以迁移到其他领域作为萃取参考

3. **"显式化vs隐性影响"的权衡是重要补充**
   - 文章关于Procedural Memory不进Memory的论证，深刻揭示了"隐性模式"的危险性
   - 七概念E可以更强调"显式化原则"：模式必须是可见的、可审查的，反对隐性的、不可追踪的知识影响系统行为

4. **渐进式加载是"形式化编码"维度的重要补充**
   - 七概念E讲形式化编码主要关注"格式结构"
   - 文章补充了"加载策略"也是形式化编码的一部分：好的编码不仅结构好，还要支持按需检索加载，避免全量注入
   - 这对应"高内聚低耦合"原则在知识组织上的体现
