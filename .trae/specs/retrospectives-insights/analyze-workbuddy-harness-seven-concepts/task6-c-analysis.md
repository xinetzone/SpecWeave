# Task 6：C（原子提交）维度映射分析

---

## 1. C四要素在Agent工程中的体现

C（原子提交）的公理是"变更集的不可分割单一职责单元，满足同因同果、可独立回滚、review无认知跳跃"。在Agent工程中，这一公理不再局限于Git版本控制的commit粒度，而是延伸到**AI执行单元的原子性边界控制**——每一个Agent行动、每一次任务推进，都需要满足原子性四要素。

### 1.1 职责内聚：执行单元的单一职责保证

职责内聚要求变更集逻辑单一不可再分。在Agent工程中，这体现为多重机制保证Agent一次只做一件事，避免任务混杂导致的不可控。

**Anthropic"一次只处理一项任务"原则**是职责内聚的核心体现。文章在描述Anthropic第一篇长任务Harness实践时明确指出：

> 解决方案用"初始化 Agent + Coding Agent"两个角色完成跨会话交接：一开始就把要做的事拆成 200+ 条具体行为描述的功能清单（JSON）、每条标 pass/fail 并禁止删条目或降标准；**一次只处理一项任务**；统一的启动脚本 init.sh；除单测外用浏览器自动化做端到端验证；用进度文件 + Git 历史做交接、恢复和回滚。
>
> ——第6.2节（article-content.md:471）

"一次只处理一项任务"不是效率优化，而是**原子性的强制约束**：如果Agent同时处理多项任务，就会出现职责混杂——某个工具调用可能同时服务于A和B两个目标，一旦失败无法判断影响范围，回滚也无法精确进行。这对应Git commit中"一个commit只做一件事"的黄金法则。

**200+条JSON任务清单的原子化拆分**是职责内聚的载体：

> 一开始就把要做的事拆成 200+ 条具体行为描述的功能清单（JSON）、每条标 pass/fail 并禁止删条目或降标准
>
> ——第6.2节（article-content.md:471）

每条JSON条目都是一个不可再分的原子任务，具备明确的完成标准（pass/fail），不允许多个目标混在同一条目里。清单的"具体行为描述"要求尤其关键——不是模糊的"实现登录功能"，而是可验证的具体行为："用户输入正确邮箱密码后点击登录按钮，应跳转到首页并显示用户名"。

**禁止删条目或降标准**是职责内聚的强制性保障：Agent不能因为某条任务难、做不完，就偷偷从清单里删掉它；也不能因为接近上下文上限、时间不够，就降低验收标准（比如"测试只跑核心用例就行"）。这保证了原子单元的完整性和一致性。

**独立Worktree隔离执行环境**从物理层面保证职责内聚：

> 一个可用的 Loop 至少需要这些组件：触发器（Trigger / Automation）、**独立执行环境（Isolated Workspace / Worktree）**、Skills、Tools / Connectors / MCP、Sub-agents、Memory / Durable Artifacts、Sensors / Evals、Stop Conditions / Budget。
>
> ——第07章（article-content.md:574）

> 创建独立 worktree 和任务记录。读取仓库规则与 dependency-update Skill。查询可用更新与已知漏洞。选择一个可独立验证的更新，修改 lockfile。运行安装、类型检查、单测、构建和漏洞扫描。
>
> ——第07章Loop示例（article-content.md:584-586）

每个Loop任务在独立Worktree中执行，与主工作区和其他任务物理隔离，不会出现"一个修改同时影响多个功能"的职责混杂问题。这就像Git中每个功能在独立feature分支开发，不会互相干扰。

**Sub-agent处理旁支任务**从上下文层面保证职责内聚：

> 第五，拆分调研任务并分配给 Sub-agent。OpenAI、Anthropic、LangChain 的资料可以分别交给不同 Sub-agent 处理。主 Agent 为每个 Sub-agent 提供统一输出格式，但只提供各自任务所需的上下文，减少信息干扰。
>
> ——第03章（article-content.md:263）

Context Engineering的"隔离（Isolate）"动作对此有明确定义：

> **5. 隔离（Isolate）**：用独立会话或 Sub-agent 处理旁支任务，只把结果带回主线，避免污染主上下文。
>
> ——第04章（article-content.md:301）

旁支任务（如单独调研某家厂商、查某个API文档）交给Sub-agent在独立上下文中处理，主Agent主线任务保持单一职责，不会被旁支信息干扰。主Agent只接收Sub-agent返回的最终结果，不关心Sub-agent中间调用了什么工具、走了什么弯路，这对应Git中"不要在feature分支里混进无关修复"的原则。

### 1.2 因果闭合：执行完整性与同因同果的保证

因果闭合要求同因同果的完整变更单元，无遗漏无多余——同样的初始条件、同样的执行过程，应该产生同样的结果；完成一个原子单元所需要的所有变更都应该包含在内，不能遗漏必要部分，也不能夹带多余内容。

**每条任务标pass/fail、禁止删条目或降标准**是因果闭合的核心机制：

> 一开始就把要做的事拆成 200+ 条具体行为描述的功能清单（JSON）、每条标 pass/fail 并**禁止删条目或降标准**
>
> ——第6.2节（article-content.md:471）

这一机制从三个维度保证因果闭合：

1. **无遗漏（不删条目）**："禁止删条目"保证因果链不中断——不会因为某条任务难、Agent不想做，就偷偷删掉，导致最终产物缺少必要组成部分。就像Git commit不能因为某个文件难改就不add它——要完成一个功能，相关的代码、测试、文档都必须包含在同一个变更集中。

2. **无多余（一次一项）**："一次只处理一项任务"保证不会夹带无关变更——完成任务A时不会顺便改任务B的代码，避免出现"这个commit明明说改登录，怎么连支付逻辑也动了"的情况。

3. **同因同果（不降标准）**："禁止降标准"保证执行结果的一致性——不会因为接近上下文上限、模型累了、时间不够，就降低验收标准。Anthropic第二篇论文专门指出这是Agent的典型失败模式：

> 在前一篇基础上发现两个更深的问题——接近上下文上限时模型会降低完成标准、自我评估不可靠（Agent 评价自己产出时倾向给正面结论，前端设计这类缺少确定性测试的任务尤其明显）。
>
> ——第6.2节（article-content.md:479）

如果允许降标准，同样的初始条件（同一个任务清单）就会产生不同质量的结果——上下文充足时做得好，上下文快满时做得差——这就破坏了"同因同果"的因果闭合性。

**工具结果的完整回流**保证执行链条闭合：

> 错误也不只返回 error 或一段堆栈，还要返回失败原因、可修正参数、是否可重试和建议下一步。
>
> ——第4.2节（article-content.md:337）

这是因果闭合在工具调用层面的体现：工具不能只返回"成功/失败"二元结果，必须返回完整的因果信息——为什么失败？哪些参数可以修正？下一步该做什么？只有这样，Agent的ReAct循环（判断→行动→观察→再判断）才能闭合，不会出现"行动了但不知道结果如何，无法决定下一步"的因果断裂。

反馈层的设计也强调完整信息回流：

> 工具结果包含可纠正信息（文件未找到提示搜索路径、编辑失败提示重新读取、权限不足提示请求确认、命令报错返回完整 stderr）
>
> ——第6.3节（article-content.md:521）

每个工具结果都必须包含"原因+修正路径"，而不是只给一个错误码，这保证了每一步行动的因果链条都是完整闭合的。

**Planner/Generator/Evaluator三角色分离**从验收层面保证因果闭合：

> 方案借鉴 GAN 的对抗评估思路，用 Claude Agent SDK 构建三个角色：Planner（把一句话需求展开成完整规格，定范围但不指定实现细节）、Generator（按 sprint 逐功能实现、用 git 版本控制、提交前先自检）、Evaluator（独立验收 Agent，用 Playwright 像真实用户一样操作运行中的应用，逐条核查、把 bug 定位到行号和原因后打回）。
>
> ——第6.2节（article-content.md:481）

三角色分离解决了"实现和测试共享同一个误解"的问题——如果Generator自己写实现自己测，对需求的理解偏差会同时进两边，看起来"测试全过"但实际上因果链是断裂的（错误的因导致了错误的果，但错误的测试认为它是对的）。独立Evaluator从外部视角验收，保证"因"（需求规格）和"果"（实现结果）之间的因果关系真实成立。

### 1.3 独立回滚：多层安全撤销机制

独立回滚要求不依赖其他变更可安全撤销。Agent工程中通过多层隔离机制实现比Git更细粒度、更可靠的回滚能力。文章特别强调这些安全机制必须在模型外部强制执行：

> **System Prompt 只能引导，不能强制。** 权限校验、Sandbox、Approval Gate、审计仍由模型外部的系统执行。
>
> ——第2.2节（article-content.md:123）

这一原则至关重要——回滚和安全机制不能靠模型"自觉遵守"，必须由外部系统强制执行，就像数据库的约束不能靠应用程序"自觉"遵守一样。

**四层回滚防护体系**从时间维度层层设防：

| 防护层级 | 机制 | 回滚时机 | 原文证据 |
| --- | --- | --- | --- |
| **第一层：事前阻止** | Approval Gate审批 | 高风险操作执行前 | "删除、发送、支付、发布等高风险动作需要审批"（article-content.md:113） |
| **第二层：事中隔离** | Sandbox沙箱 | 执行过程中 | "Sandbox（隔离环境，执行出错也不影响本机）"（article-content.md:453） |
| **第三层：任务级丢弃** | 独立Worktree | 任务完成前 | "如果失败，将错误反馈给 Agent，在限定轮数内修正；仍失败则停止并留下可诊断报告"（article-content.md:586） |
| **第四层：事后回滚** | Git版本控制+进度文件 | 任务完成后 | "用进度文件 + Git 历史做交接、恢复和回滚"（article-content.md:471） |

**第一层：Approval Gate事前审批**是成本最低的回滚——最好的回滚是不让错误操作发生。

> 发送、发布、删除或其他不可轻易撤销的操作，执行前确认用户授权。
>
> ——第2.2节System Prompt示例（article-content.md:120）

Approval Gate在高风险操作执行前暂停，等待人工确认，相当于在原子单元边界设置了"提交确认点"——未通过审批的操作不会真正执行，自然不需要回滚。这比Git的"提交后回滚"粒度更前：Git只能回滚已提交的变更，Approval Gate可以在变更发生前就阻止它。

**第二层：Sandbox事中隔离**是执行时的安全屏障。

> **2. 约束（Constrain）。** 防止执行超出安全范围。对应到 Agent：权限边界（误删文件需要被拦截）、**Sandbox（隔离环境，执行出错也不影响本机）**、Approval Gate（危险操作需要人工确认）、Allowlist/ denylist（限定可操作的命令和路径），以及测试验证、rollback、audit log。
>
> ——第6.1节（article-content.md:453）

Sandbox在执行层就提供了隔离——Agent在沙箱中运行命令，即使执行了rm -rf也不会影响宿主机，相当于"执行前就预设了回滚点"。未通过验证的沙箱变更直接丢弃即可，不会留下任何痕迹。

**第三层：独立Worktree任务级回滚**提供了比Git分支更细的回滚粒度。

> 创建独立 worktree 和任务记录。……如果失败，将错误反馈给 Agent，在限定轮数内修正；仍失败则停止并留下可诊断报告。如果通过，生成 PR 草稿和风险摘要，交由人审批发布。
>
> ——第07章Loop示例（article-content.md:584-586）

每个任务在独立Worktree中执行，失败了直接丢弃Worktree即可，不会影响主分支和其他任务。这对应Git中"feature分支合并失败直接删除分支"的原子回滚模式，但粒度更细——每个独立任务（而不是每个大功能）都有自己的Worktree，回滚的影响范围精确到单个任务。

**第四层：Git+进度文件事后回滚**是最后一道防线。

> 用进度文件 + Git 历史做交接、恢复和回滚
>
> ——第6.2节（article-content.md:471）

> Generator（按 sprint 逐功能实现、用 git 版本控制、提交前先自检）
>
> ——第6.2节（article-content.md:481）

Git提供了版本级别的回滚能力，可以回到任意历史提交点；进度文件则记录了任务状态，可以在跨会话恢复时回滚到上一个稳定检查点。两者结合，即使前面三层都没拦住错误，也能事后恢复到正确状态。

**Allowlist/Denylist**是独立回滚的辅助机制——限定Agent可操作的命令和路径范围，从一开始就不让Agent有机会执行不可逆的危险操作，相当于把回滚边界提前到了能力暴露层面。

### 1.4 认知平滑：可审查性与无认知跳跃的保证

认知平滑要求review无跨主题认知跳跃——审查者（人或Evaluator Agent）在审查原子单元时，不需要在多个不相关主题之间来回切换，能够流畅地理解这个单元做了什么、为什么这么做、结果是什么。

**Task/Todo清单拆分**让review可以逐任务进行，不会出现大段混杂变更：

> Skills 规定某类任务的步骤，**Task / Todo 把大目标拆成清单**，错误消息中的自我纠正提示在卡住时提供方向（而非只给错误码）
>
> ——第6.1节驾驭能力（article-content.md:451）

> WorkBuddy 的 Agent 在这一点上做了类似设计：执行较大任务时，先把目标拆解成结构化任务清单，并在推进过程中持续更新状态。它同时处理"一次承担过多 / 过早判定完成"和"上下文遗忘"两类问题——显式任务状态可以让 Agent 在长对话里恢复进度，也让用户更容易判断任务是否真的完成。
>
> ——第6.2节（article-content.md:473）

结构化任务清单让用户（或Evaluator Agent）可以逐条审查，每条任务的目标、结果、状态都清晰，不会出现"看完500行diff才发现改了三件不相关的事"这种认知跳跃。每条任务都是一个独立的review单元，认知负荷可控。

**渐进式加载**避免信息过载导致的认知跳跃：

> 工具结果过长时，WorkBuddy 给每个 Tool Result 设截断策略，超出时分页、截断或写入文件。截断时明确告诉模型"结果未完整"，附上总量、截断位置和继续读取方法，否则模型会把前 100 条误当成全部。
>
> ——第4.2节（article-content.md:337）

> 工具定义过多时，每个 Tool Schema 都占上下文，工具越多、语义越重叠，模型选择越困难。WorkBuddy 采用分阶段的能力发现机制：先让模型看到工具名称和简要描述，再根据任务需要加载更具体的工具说明，同时也提供了工具检索能力。
>
> Skill 也用同样机制，先看名称和描述，确认适用后再读完整 SKILL.md，需要时再打开参考资料与脚本。
>
> ——第4.2节（article-content.md:339-341）

渐进式加载对认知平滑的贡献在于：它保证**任何时刻上下文中只包含当前任务相关的信息**，不会一次性把所有工具定义、所有Skill、所有长结果都塞进去。审查者（或模型本身）不需要在大量无关信息中筛选当前需要的内容，自然不会出现认知跳跃。这就像code review时应该一次只看一个commit，而不是一次性看整个分支的所有变更——信息太多会导致认知负荷过载，漏掉重要问题。

**Sub-agent结果隔离返回**保持主线上下文的认知连贯：

> 第五，拆分调研任务并分配给 Sub-agent。OpenAI、Anthropic、LangChain 的资料可以分别交给不同 Sub-agent 处理。主 Agent 为每个 Sub-agent 提供统一输出格式，但只提供各自任务所需的上下文，减少信息干扰。
>
> 第六，Sub-agent 阅读资料并返回结果。每个 Sub-agent 使用适合的工具获取信息，并返回结论、证据、来源和适用范围。
>
> ——第03章（article-content.md:263-265）

Sub-agent在独立上下文中处理旁支任务，中间过程（调用了什么工具、走了什么弯路、遇到了什么错误）都留在Sub-agent自己的上下文里，主Agent只接收结构化的最终结果（结论、证据、来源、适用范围）。这保证主上下文始终围绕主线任务，不会被旁支任务的中间细节打断认知流。这对应Git中"PR应该只包含最终代码，不包含WIP提交和调试日志"的原则——reviewer不需要看你中间怎么试错的，只需要看最终结果。

**进度文件交接**保证跨会话/跨Agent的认知连续性：

> 用进度文件 + Git 历史做交接、恢复和回滚
>
> ——第6.2节（article-content.md:471）

> 写入本轮结果、未解问题和下次运行需要的交接信息。
>
> ——第07章Loop示例（article-content.md:586）

进度文件记录了"做到哪了、为什么这么做、哪些做完了、哪些没做完、下一步该做什么"，让下一个Agent（或同一个Agent下次启动）不需要从头推理，不会出现认知断层。这对应Git commit message的作用——好的commit message让reviewer不需要看diff就能理解变更目的和上下文，避免"这个变更到底想解决什么问题"的认知困惑。

**audit log全链路追溯**让每一步行动都可审计：

> 再加上 Audit log 让所有动作留痕、可追溯回放。
>
> ——第6.3节反馈层（article-content.md:521）

> 这里有一个容易被忽略的要点：持有 API Key、发起请求、修改数据的是 Agent，不是模型。模型负责生成调用请求，Agent 负责执行外部操作。因此，权限、审批、参数校验和审计日志都必须由模型外部的工程机制执行。
>
> ——第2.1节（article-content.md:95）

Audit log记录了每一次工具调用的时间、参数、结果、审批状态，让任何问题都可以事后追溯，不会出现"不知道Agent什么时候改了什么"的黑箱。这对应Git reflog的作用，但粒度细到每一次工具调用——review时如果发现某个结果不对，可以顺着audit log一步步回溯，看是哪一步出了问题，不需要在大量变更中盲猜。

## 2. C语义在两套体系中的微妙差异

C（原子提交）在七概念框架和Agent工程中共享"不可分割的单一职责单元"这一核心公理，但在语义指向、作用对象、粒度控制上存在微妙而深刻的差异。理解这些差异，有助于我们将版本控制领域积累了几十年的原子性智慧迁移到Agent工程领域，同时避免生搬硬套。

### 2.1 七概念C：版本控制语义

七概念框架中的C（原子提交）源于Git等版本控制系统的实践，其核心语义是**版本控制语义**：

- **作用对象**：代码/文档等静态产物的变更历史
- **关注焦点**：变更之后如何记录、归档、回溯
- **时间点**：变更完成后——代码改完了，测试通过了，才commit
- **隔离机制**：逻辑隔离——分支和commit历史，物理上仍在同一个工作区
- **回滚方式**：事后回滚——发现问题后用git revert/reset回到之前的commit
- **典型单元**：Git commit——一组相关的代码/文档变更

在版本控制语义下，C的核心价值是：保持变更历史的清晰、可追溯、可回滚，让团队协作中每个人的工作不互相干扰，出了问题能够定位和恢复。

### 2.2 Agent工程C：执行控制语义

Agent工程中的C将原子性思想从"版本控制"扩展到了"执行控制"，其核心语义是**执行控制语义**：

- **作用对象**：AI Agent的动态行动过程
- **关注焦点**：执行过程中如何不跑偏、出错如何即时止损
- **时间点**：执行全过程——事前预防、事中控制、事后归档
- **隔离机制**：物理隔离——Sandbox、Worktree、Sub-agent独立上下文
- **回滚方式**：多层回滚——事前Approval Gate阻止、事中Sandbox隔离、事后Git/进度文件恢复
- **典型单元**：单次工具调用、单条Task、单个Loop任务

在执行控制语义下，C的核心价值是：保持Agent执行过程的可控、可预测、错误影响范围有限，让AI在自主执行时不会造成不可逆的损害，出了问题能够快速恢复。

文章中"独立Worktree和任务记录"的机制，典型地体现了Agent工程如何把"原子性"从版本控制扩展到执行隔离：

> 创建独立 worktree 和任务记录。读取仓库规则与 dependency-update Skill。查询可用更新与已知漏洞。选择一个可独立验证的更新，修改 lockfile。运行安装、类型检查、单测、构建和漏洞扫描。如果失败，将错误反馈给 Agent，在限定轮数内修正；仍失败则停止并留下可诊断报告。如果通过，生成 PR 草稿和风险摘要，交由人审批发布。写入本轮结果、未解问题和下次运行需要的交接信息。
>
> ——第07章Loop示例（article-content.md:584-586）

在这个依赖更新Loop例子中：
- **独立worktree**提供执行环境的原子隔离——这个更新任务的所有修改都在独立worktree中，不会污染主工作区
- **任务记录**提供执行状态的原子追踪——每一步做了什么、结果如何都被记录，可中断可恢复
- **"选择一个可独立验证的更新"**体现了职责内聚——一次只处理一个依赖更新，不混在一起
- **失败则停止并留下可诊断报告/通过则生成PR**体现了因果闭合——每个任务有明确的完成或失败状态，没有中间态
- **交由人审批发布**体现了独立回滚——未审批的变更不会真正合入主干，随时可以丢弃

这已经远远超出了Git commit的范畴——Git commit只在变更完成后记录版本，而Worktree+任务记录在整个执行过程中维持原子性边界。

### 2.3 共通本质：通过粒度控制实现失败可控

尽管存在上述语义差异，两套体系中C的底层哲学是完全一致的：

> **原子性的本质是粒度控制——通过找到"单一职责的最小可独立单元"这个平衡点，实现"失败可控、影响可限、回滚可行"。**

| 共性维度 | 七概念C（版本控制） | Agent工程C（执行控制） | 共通本质 |
| --- | --- | --- | --- |
| 核心目标 | 变更集不可分割 | 执行单元不可分割 | 单元边界清晰 |
| 设计原则 | 一个commit只做一件事 | 一次只处理一项任务 | 单一职责 |
| 失败应对 | 可独立回滚到任意commit | 错误可隔离撤销 | 失败影响范围有限 |
| 可审查性 | commit message + diff | task清单 + audit log | 可理解、可追溯 |
| 隔离机制 | 分支隔离 | Worktree/Sandbox/Sub-agent隔离 | 并发工作不互相干扰 |

两者都反对两种极端：
- **粒度过大**：一个commit包含10个不相关功能/Agent一次同时做10件事 → 难以review、难以回滚、出错了不知道影响什么
- **粒度过小**：每行代码一个commit/每个工具调用都当成原子单元 → 管理 overhead 太大、上下文碎片化、只见树木不见森林

原子性的艺术就是在这两个极端之间找到合适的平衡点——足够小到单一职责、可独立回滚；又足够大到认知连贯、管理成本可控。

### 2.4 关键差异对比表

| 差异维度 | 七概念C（Git commit级） | Agent工程C（执行单元级） |
| --- | --- | --- |
| **语义层** | 版本控制语义——记录变更历史 | 执行控制语义——控制行动过程 |
| **时间维度** | 事后归档——变更完成后才commit | 全时段控制——事前预防+事中隔离+事后归档 |
| **隔离强度** | 逻辑隔离——分支/commit历史，物理上同盘 | 物理隔离——Sandbox进程隔离+Worktree目录隔离+Sub-agent上下文隔离 |
| **回滚粒度** | commit级——回滚到某个版本点 | 多层粒度——操作级阻止→执行级丢弃→任务级删除→版本级恢复 |
| **验收方式** | 静态验收——看diff、跑测试 | 动态验收——pass/fail逐条核查+Playwright端到端验证+独立Evaluator验收 |
| **强制执行** | 靠开发者自觉+code review流程 | 靠Harness系统机制强制——模型无法绕过 |
| **描述载体** | Conventional Commits格式（type(scope): subject） | Task清单+进度文件+audit log多维度描述 |
| **确定性保障** | Git本身是确定性的——同样的commit内容一定产生同样的树 | Agent本身有概率性——需要通过Harness机制减少随机性（详见第5.3节） |

## 4. 关键洞察

### 4.1 Approval Gate与C独立回滚：为什么高风险操作必须人工确认？

Approval Gate是C"独立回滚"要素在Agent工程中的关键创新，它揭示了一个深刻的洞见：**最好的回滚不是执行后撤销，而是执行前阻止**。

传统Git工作流中，即使代码review发现问题，代码也已经commit到本地分支了——虽然可以reset，但"已经写过"的事实会产生认知锚定，开发者可能不愿意完全放弃自己写的代码。而Approval Gate把确认点放在了"执行"之前：Agent可以在Sandbox里自由尝试、修改文件、运行测试，但当执行到需要审批的操作（删除、发送、支付、发布）时，整个流程暂停，等待用户确认。未确认的操作根本不会发生，回滚成本为零。

为什么必须是人工确认而不是另一个Agent来检查？文章第08章给出了答案：

> 部分业务正确性缺少可计算的判定标准。编译器能发现语法和类型问题，监控能度量性能和错误率，但业务意图通常需要业务人员确认。
>
> ——第08章（article-content.md:609）

> 核心业务错误可能造成资金损失、合规问题和用户流失，对验证可信度的要求远高于普通代码质量问题。
>
> ——第08章（article-content.md:611）

这对应AI自治度分级原则：

> 所以当业务正确性缺少可靠验证时，AI 的自治程度需要随风险提高而降低：
>
> | 场景 | AI 自治度上限 |
> | --- | --- |
> | 一次性脚本、内部工具 | 高 |
> | 公开 API、跨系统改动 | 中 |
> | 核心业务逻辑（支付、风控、订单） | 低 |
>
> ——第08章（article-content.md:613-619）

Approval Gate本质上是在原子单元的边界上设置了"人类责任点"——Agent可以执行准备工作，但最终的"执行"决策必须由人来做，因为人承担最终责任。这也呼应了文章的核心结论：

> 人负责选择方向、定义标准并承担责任；Agent 负责执行、验证和加速迭代。
>
> ——第09章（article-content.md:681）

从C四要素角度看，Approval Gate同时服务于三个要素：
- **独立回滚**：事前阻止不可逆操作，回滚成本为零
- **因果闭合**：人工介入因果链，确认"因"（用户意图）和"果"（操作执行）一致
- **认知平滑**：审批过程让用户知道Agent要做什么，避免"Agent偷偷做了我不知道的事"

### 4.2 Worktree隔离：如何同时保证C的职责内聚和独立回滚？

独立Worktree是Agent工程中一个极其优雅的设计——它用一个机制同时满足了C的两个核心要素：职责内聚和独立回滚。

**Worktree如何保证职责内聚？**

职责内聚要求"一个执行单元只做一件事，不混杂无关内容"。Worktree从物理层面保证了这一点：

1. **目录隔离**：每个任务有自己独立的工作目录，修改文件A的任务不会看到修改文件B的任务产生的中间文件
2. **环境隔离**：每个Worktree可以有自己的依赖版本、环境变量、临时文件，不会出现"我明明改的是依赖X，怎么依赖Y也变了"的环境污染
3. **上下文隔离**（配合Sub-agent）：每个Worktree中的Agent只看到自己任务相关的上下文，不会被其他任务的信息干扰

文章中依赖更新Loop的例子非常典型："选择一个可独立验证的更新"——每个依赖更新在自己的Worktree中进行，不会出现"我本来只想更新react，结果顺手把lodash也改了"的职责混杂。

**Worktree如何保证独立回滚？**

独立回滚要求"不依赖其他变更可安全撤销"。Worktree让回滚变得异常简单：

1. **失败了直接丢弃**：如果任务在Worktree中失败（测试没过、验证不通过），直接删除这个Worktree就行，主工作区和其他任务完全不受影响
2. **成功了再合并**：只有验证通过的Worktree才会生成PR，经过审批后合入主干
3. **并行不干扰**：多个任务可以同时在各自的Worktree中执行，一个任务回滚不会影响其他正在执行的任务

这比Git分支的隔离性更强——Git分支共享同一个工作目录，切换分支时工作区文件会变化；而Worktree是多个独立的工作目录同时存在，物理上完全隔离。

**Worktree的双重职责是原子性思想的深化：**

在传统Git中，职责内聚靠commit message和code review（人工检查），独立回滚靠git revert/reset（事后操作）；而Worktree把这两者统一到了执行环境的设计中——只要你在独立Worktree中执行任务，天然就是职责内聚的（因为你在隔离环境里做不了别的任务的修改），天然就是可独立回滚的（因为失败了直接删目录就行）。

这是一个重要的方法论启示：**最好的原子性保障不是事后检查，而是事前的环境设计——让正确的事情容易做，让错误的事情（混杂多个任务、错误无法回滚）从物理上就难以发生。**

### 4.3 任务清单（200+条pass/fail）如何解决两个经典问题？对应哪些C要素？

Anthropic在长任务Agent实践中发现了两个典型失败模式，而200+条pass/fail任务清单正是为了解决这两个问题设计的。让我们分析这两个问题，以及它们对应C四要素中的哪些：

**问题1：一次承担过多工作**

> 处理长任务时发现两种典型失败：Agent 一次承担过多工作（中途耗尽上下文、给下一个 Agent 留下缺少说明的半成品）
>
> ——第6.2节（article-content.md:471）

"一次承担过多"的具体表现是什么？
- Agent试图同时实现多个功能，上下文被多个主题的信息占满，很快耗尽窗口
- 做到一半上下文满了，留下半拉子工程——功能A做了80%，功能B做了30%，功能C刚开头
- 下一个Agent接手时，不知道哪些做完了、哪些没做、做到什么程度了，只能从头再来或在半成品上继续犯错

**任务清单如何解决这个问题？**
- **原子化拆分**：把大目标拆成200+条具体行为的小任务，每条都是"一次只处理一项"的原子单元
- **逐条推进**：Agent一次只拿一条任务来做，做完一条标pass再做下一条
- **状态显式化**：每条任务都有明确的pass/fail状态，随时可以知道进度

**对应C四要素：**
- **主要对应职责内聚**：200+条拆分+一次一项，强制Agent保持单一职责，不能同时做多个任务
- **同时对应认知平滑**：显式状态让交接时不会有认知断层——下一个Agent不需要猜"到底做完了什么"

**问题2：过早判定完成**

> ……或看到部分成果就过早判定完成。
>
> ——第6.2节（article-content.md:471）

"过早判定完成"的具体表现是什么？
- 功能做了个happy path（正常流程），边缘情况没处理，Agent就说"做完了"
- 核心功能做完了，但相关的错误处理、测试、文档没做，Agent认为"主要功能完成了"
- 接近上下文上限时，模型会主动降低完成标准，觉得"差不多就行了"

**任务清单如何解决这个问题？**
- **禁止删条目**：清单上的200+条一条都不能少，不能因为某条是边缘情况就跳过
- **禁止降标准**：每条都要达到pass标准，不能"差不多就行"
- **独立验收**：Evaluator Agent逐条核查pass/fail，不是Generator自己说做完就算做完

这两个问题在第二篇论文中被进一步深化：

> 在前一篇基础上发现两个更深的问题——接近上下文上限时模型会降低完成标准、自我评估不可靠（Agent 评价自己产出时倾向给正面结论，前端设计这类缺少确定性测试的任务尤其明显）。
>
> ——第6.2节（article-content.md:479）

**对应C四要素：**
- **主要对应因果闭合**：禁止删条目保证"无遗漏"（所有必要部分都包含），禁止降标准保证"同因同果"（不会因为上下文压力降低质量），独立Evaluator保证"因果真实成立"（不是自己骗自己）
- **同时对应独立回滚**：每条任务独立标pass/fail，某条没通过可以单独重做这一条，不需要全部回滚

**总结：两个问题→两个要素**

| 经典失败模式 | 任务清单的解决方案 | 主要对应C要素 | 次要对应C要素 |
| --- | --- | --- | --- |
| **一次承担过多** | 拆成200+条小任务+一次只处理一项 | 职责内聚 | 认知平滑 |
| **过早判定完成** | 禁止删条目+禁止降标准+独立验收 | 因果闭合 | 独立回滚 |

任务清单之所以有效，是因为它同时从C四要素的多个维度发力，系统性地解决Agent长任务执行的可靠性问题。这不是简单的"列个todo list"，而是基于原子性公理设计的一整套执行控制机制。

## 5. 差异与补充

### 5.1 认知平滑与可审查性：文章中哪些机制保证Agent输出的可审查性？

七概念C强调"认知平滑"——review无跨主题认知跳跃。在Agent工程中，这不仅是"review起来舒服"的问题，更是**安全和可靠性的基础**——如果人类无法审查Agent做了什么，就无法信任它的输出。文章中有多层机制保证可审查性：

**第一层：执行前可审查——知道Agent要做什么**

1. **结构化任务清单**：执行前就把要做的事拆成明确的条目，用户可以在Agent开始前审查清单，知道它计划做什么
2. **System Prompt公开**：System Prompt定义了Agent的角色、工作原则、安全边界，用户可以查看Agent被什么样的规则约束
3. **Skill/Plugin可审查**：Skill是显式的Markdown文件，Plugin是可查看的能力包，用户可以审查Agent"知道怎么做"的流程是不是正确的

**第二层：执行中可审查——知道Agent在做什么**

1. **实时进度更新**：Agent执行过程中持续更新任务状态，用户可以看到"现在做到第几条了"
2. **Approval Gate暂停点**：高风险操作执行前暂停，用户可以看到"Agent现在要执行什么操作、参数是什么"，确认后才继续
3. **工具调用可见**：Audit log记录每一次工具调用，用户可以实时看到Agent调用了什么工具、传入了什么参数
4. **Sub-agent任务分配可见**：主Agent分配什么任务给Sub-agent是显式的，用户知道哪些工作被并行处理了

**第三层：执行后可审查——知道Agent做了什么、结果如何**

1. **pass/fail状态**：每条任务有明确的完成状态，哪些过了哪些没过一目了然
2. **进度文件交接信息**：任务完成后写入"本轮结果、未解问题、下次交接信息"，完整记录执行结果
3. **Audit log全链路追溯**：所有动作留痕，出了问题可以一步步回放，定位到具体哪一步出错
4. **独立Evaluator验收报告**：Evaluator Agent逐条核查后给出验收结果，bug定位到行号和原因
5. **Git历史**：所有代码变更都有Git提交记录，可以用传统code review方式审查diff

**第四层：认知平滑的结构性保障——避免信息过载**

1. **渐进式加载**：不是所有信息一次性展示，而是按需加载——看任务清单时不需要看所有工具定义，审查某条任务时只加载相关的上下文
2. **Sub-agent结果隔离**：不需要看Sub-agent中间怎么试错的，只看结构化的最终结果
3. **结果结构化**：工具结果包含失败原因、修正建议、下一步，不是一堆杂乱的堆栈

**与传统Git code review的对比：**

传统Git的可审查性主要靠"commit message + diff"，是静态的、事后的；而Agent工程的可审查性是多层的、贯穿执行全过程的——执行前知道要做什么、执行中知道在做什么、执行后知道做了什么。这是因为Agent是自主行动的实体，不像代码变更那样写完了才给人看，它必须在行动过程中保持可审查、可干预。

### 5.2 变更归档：文章中是否有对应"沉淀层C2"的机制？

七概念框架中C有两层：执行层C1（原子提交本身——保证单次变更的原子性）和沉淀层C2（变更归档——原子变更如何沉淀为可追溯的历史资产）。文章中是否有对应沉淀层C2的机制？

**答案：有，但相比执行层C1，沉淀层C2的机制相对分散，没有像Task清单、Worktree那样形成一个统一的概念。**

文章中对应沉淀层C2的机制分散在多处：

**1. Git历史——代码变更的归档**

> 用进度文件 + Git 历史做交接、恢复和回滚
>
> ——第6.2节（article-content.md:471）

> Generator（按 sprint 逐功能实现、用 git 版本控制、提交前先自检）
>
> ——第6.2节（article-content.md:481）

Git历史是最传统的变更归档方式——每个原子提交都被永久记录，形成完整的变更历史，可以blame、可以bisect、可以回到任意历史点。这对应沉淀层C2的核心要求：原子变更被归档保存，可追溯、可检索。

**2. 进度文件+任务记录——执行状态的归档**

> 创建独立 worktree 和任务记录。
>
> ——第07章Loop示例（article-content.md:584）

> 写入本轮结果、未解问题和下次运行需要的交接信息。
>
> ——第07章Loop示例（article-content.md:586）

任务记录和进度文件不仅是跨会话交接的载体，也是执行过程的归档——它记录了"任务是怎么一步步完成的、遇到了什么问题、怎么解决的、还有什么没解决"，比Git历史包含更多过程性信息。

**3. Audit log——行动轨迹的归档**

> Audit log 让所有动作留痕、可追溯回放。
>
> ——第6.3节反馈层（article-content.md:521）

Audit log是最细粒度的归档——每一次工具调用、每一次审批、每一次错误都被记录，形成完整的行动轨迹。这相当于Git的reflog，但粒度细到每一次操作。

**4. Memory——经验的归档**

> Memory 系统要做一次准入判断——哪些历史信息可以继续影响未来要做的任务。
>
> ——第05章（article-content.md:357）

Memory是更高层级的归档——不是记录"这次具体做了什么"，而是从多次执行中提取"稳定事实、用户偏好、行为模式"，作为未来任务的推理前提。

**5. Skill——成功经验的归档**

> 陈述性记忆放进Memory，经过验证的工作方法保存为Skill——Skill 可版本化、可评审、可测试、可回滚、按需加载。
>
> ——第5.2节（article-content.md:395）

Skill是最高层级的归档——把多次任务中验证有效的工作方法，沉淀为可复用的流程模式，这是从"变更历史"到"组织能力"的跃迁。

**沉淀层机制的成熟度分析：**

| 沉淀层机制 | 成熟度（1-10） | 对应七概念C2要求 | 不足 |
| --- | --- | --- | --- |
| Git历史 | 9 | 代码变更可追溯、可回滚 | 传统机制，非Agent特有 |
| Audit log | 8 | 行动轨迹全链路留痕 | 文章提到但未详细讲如何利用audit log做归档检索 |
| 进度文件/任务记录 | 7.5 | 执行状态交接和归档 | 格式还是各任务自定义，没有标准化的归档格式 |
| Memory | 7 | 经验事实的沉淀 | 有准入判断但主要是陈述性记忆，程序性记忆不进Memory |
| Skill | 9 | 成功流程模式的沉淀 | 有版本化、可评审、可测试、可回滚，非常成熟 |

**对七概念C2的启示：Agent工程的沉淀层是多层级的**

传统Git的C2沉淀层主要是"代码变更历史"这一层；而Agent工程的沉淀层是多层级的，从细到粗：
1. **操作级**：Audit log——每次工具调用
2. **任务级**：进度文件+任务记录——每个原子任务的执行过程和结果
3. **代码级**：Git历史——代码变更
4. **经验级**：Memory——跨任务的稳定事实和偏好
5. **模式级**：Skill——可复用的工作流程

这是一个从"具体行动"到"抽象模式"的沉淀漏斗，每一层都归档不同粒度的信息，服务于不同的目的。七概念C2可以借鉴这个多层级归档模型，而不是只关注commit历史这一个层面。

**沉淀层目前的缺口：**

文章没有明确提到一个统一的"归档检索"机制——这些沉淀下来的历史（Git、audit log、进度文件、Memory、Skill）如何被检索和利用？目前看：
- Git历史靠git log/blame/bisect等传统命令
- Memory靠自动注入
- Skill靠按需加载
- Audit log和进度文件似乎主要用于出问题时排查，没有作为主动检索的知识库

这是Agent工程C2层可以进一步完善的方向——建立统一的变更归档检索机制，让历史经验不仅被"存储"，还能被"主动召回"指导未来任务。

### 5.3 同因同果vs概率性：Harness如何减少Agent执行的随机性？

七概念C强调"同因同果"——同样的原因应该产生同样的结果，这是确定性系统的基本要求。但LLM本质上是概率性模型：同样的输入，每次输出可能都不一样；温度参数越高，随机性越大。这似乎与"同因同果"的因果闭合公理矛盾。

文章如何通过Harness机制减少Agent执行的随机性，尽可能逼近"同因同果"的确定性？

**第一层：用确定性程序替代模型判断——计算型信号优先**

这是文章最核心的抗随机性原则：

> **能用计算型信号解决的问题，优先交给确定性程序；需要语义判断的问题，再交给审查 Agent。**
>
> ——第6.3节（article-content.md:501）

计算型控制是什么？文章明确给出了定义：

> **计算型** 控制由确定性程序执行，例如 LSP、类型检查、linter、单元测试、结构测试、依赖扫描、脚本和 codemod，优点是快、便宜、可重复，适合在 Agent 每次修改后反复运行。
>
> ——第6.3节（article-content.md:499）

确定性程序的特点就是"同因同果"——同样的输入代码，linter给出的警告一定是一样的；同样的测试用例，通过还是失败一定是确定的。把这些判断交给确定性程序，而不是让模型自己判断"代码有没有问题"，就从根源上消除了这部分判断的随机性。

**反例：为什么自我评估不可靠？**

> 接近上下文上限时模型会降低完成标准、自我评估不可靠（Agent 评价自己产出时倾向给正面结论，前端设计这类缺少确定性测试的任务尤其明显）。
>
> ——第6.2节（article-content.md:479）

让Agent自己评估自己的产出是最不靠谱的——模型有"正面偏差"（倾向说自己做对了），而且判断标准会随上下文状态波动，这就是典型的"同因不同果"。解决方案是：
- 能用程序检查的（语法、类型、测试），让程序检查
- 必须语义判断的（设计好不好、符合不符合业务意图），让**独立的Evaluator Agent**检查，而不是Generator自己检查

**第二层：把标准写死——任务清单+规则文件+Skill**

随机性的另一个来源是"标准模糊"——如果"完成"的标准是模糊的，模型就可能每次理解不一样。文章通过显式化标准减少随机性：

1. **200+条pass/fail任务清单**：每条任务的完成标准是具体行为描述，不是模糊的"实现登录功能"
2. **禁止降标准**：明确告诉模型不能因为上下文压力降低验收标准
3. **规则文件（AGENTS.md/WORKBUDDY.md）**：把代码规范、架构约束、安全边界写成显式规则，不让模型靠猜
4. **Skill完成标准**：每个Skill有明确的完成标准（如PR Skill要求"已记录所运行的测试与结果"）

> 把标准写进规则文件、先确认需求再执行、随模型升级精简约束。
>
> ——第6.2节（article-content.md:481）

标准越明确、越显式，模型判断的自由度越小，结果的一致性就越高。

**第三层：前馈+反馈闭环——用反馈纠正随机偏差**

即使有明确标准，模型还是可能犯错（随机性导致偏离标准）。反馈机制负责把偏离拉回来：

前馈（执行前提供信息）提高第一次就做对的概率：
> 前馈提高第一次就做对的概率，反馈让 Agent 在问题进入人工审查前先自我纠正。
>
> ——第6.3节（article-content.md:497）

反馈（执行后验证结果）发现并纠正错误：
> 工具结果包含可纠正信息（文件未找到提示搜索路径、编辑失败提示重新读取、权限不足提示请求确认、命令报错返回完整 stderr）
>
> ——第6.3节（article-content.md:521）

错误信息不只是说"你错了"，还要告诉模型"怎么改"——这大大提高了纠正的确定性，不会出现"知道错了但不知道怎么改，每次尝试都不一样"的随机试错。

**第四层：环境约束——从物理上限制随机性的影响范围**

即使模型随机产生了错误操作，Sandbox、Worktree、Approval Gate这些环境约束也能保证错误不会造成不可控的影响：
- Sandbox：随机的rm -rf命令只删沙箱里的文件，不影响宿主
- Worktree：随机产生的错误修改只在独立Worktree里，不污染主分支
- Approval Gate：随机产生的高风险操作请求会被审批拦住，不会真正执行

这相当于给随机性加了一层"安全壳"——即使内部有随机波动，外部影响也是可控的。

**第五层：三角色分离——用对抗性评估抵消个体偏差**

> 方案借鉴 GAN 的对抗评估思路，用 Claude Agent SDK 构建三个角色：Planner、Generator、Evaluator。
>
> ——第6.2节（article-content.md:481）

单个Agent的判断有随机性和偏差，但三个角色分离可以互相抵消：
- Planner定规格，不写实现，避免"实现者自己定标准"
- Generator写实现，不做最终验收，避免"自己说自己做对了"
- Evaluator独立验收，像真实用户一样操作，逐条核查

Evaluator和Generator是**不同的Agent实例**（甚至可以是不同模型），Generator的随机偏差不会传递给Evaluator，这大大降低了"错的实现+错的测试一起通过"的概率。

**同因同果的边界：Agent追求的是"结果确定性"而非"过程确定性"**

需要承认的是：无论Harness做得多好，LLM内部的token生成过程永远是概率性的——同一个任务，Agent这次可能先调用A工具再调用B工具，下次可能先调用B工具再调用A工具，中间可能走不同的弯路。

但Harness追求的不是"过程确定性"（每次走完全一样的步骤），而是**"结果确定性"**：
- 无论中间过程怎么走，最终都要满足pass/fail标准
- 无论犯了多少错，反馈机制都会把它拉回正确轨道
- 无论模型怎么随机尝试，Sandbox/Approval Gate保证不会造成不可逆损害
- 无论Generator怎么自我感觉良好，独立Evaluator都会客观验收

这就像人写代码——两个不同的开发者实现同一个功能，写出来的代码可能不一样（过程不同），但都要通过同样的测试、满足同样的验收标准（结果相同）。"同因同果"在Agent工程中被重新诠释为：**同样的需求规格+同样的验收标准，最终产出应该满足同样的质量要求，即使中间路径可能不同。**

这是C因果闭合要素在概率性AI时代的新发展——不是追求机械的过程确定性，而是通过多层Harness机制，在概率性的基础上构建结果的可靠性和一致性。
