# AI Agent多代理协作中的幻觉防御

> **从一次"35/35检查点全过但有18处事实错误"的报告说起**
>
> 一份深入分析"对抗性审查如何防止共享误解"的报告，自身恰好犯了"共享误解"的经典错误——验证者和被验证者共享同一上下文，导致35项检查全部通过，却存在18处事实错误。这不是段子，这是我们本周踩的坑。

---

## 📖 一、故事：那份"完美通过"的报告是怎么翻车的

### 背景

我们用7个子代理协作，花了一个完整会话周期，以"七概念方法论"为透镜，分析了一篇关于AI Agent工程的万字长文。流程包括：Spec Mode规划（PRD→12个任务→35项检查清单）→ 7批次子代理并行分析 → 最终子代理组装报告 → checklist验证。

### 结果

- ✅ **35/35检查点全部通过**
- ✅ **810行结构化分析报告**
- ✅ **7个概念映射、Mermaid图、映射表、附录齐全**
- ❌ **18处事实错误**（13处严重 + 5处轻微）

### 用户一句话击穿

> "WorkBuddy好像不是字节跳动内部研发助手吧？"

### 错误全景

| 错误类型 | 数量 | 典型案例 |
|---------|------|---------|
| 🔴 编造引文 | 9处 | "Harness是Agent的方向盘"（原文没说过）、"既当运动员又当裁判"（原文没说过）、"Skill就是代码"（原文没说过） |
| 🔴 案例归属错误 | 3处 | OpenAI部分混入了WorkBuddy和Anthropic的实践；Anthropic两篇论文的内容张冠李戴；LangChain部分虚构了三层架构 |
| 🔴 虚构概念 | 2处 | "Golden Set回归测试""A/B实验"——原文根本没提 |
| 🟡 术语漂移 | 14处 | 原文"计算型**控制**"→ 报告"计算型**反馈**"，丢失了前馈维度 |
| 🟡 章节标注错误 | 6处 | "第08章 总结"实际是"第08章 还没解决的问题" |

### 讽刺的核心

报告中写得最好、论证最有力的章节是**V-对抗性审查**——雄辩地分析了"Generator和Evaluator共享同一个误解，会导致错的实现+全部通过的测试"。

**而这份报告自身，恰好被这个机制击败了。**

## 🔍 二、四个核心发现

### 发现1：最终组装是多代理协作的"幻觉高发区"

100%的严重错误集中在Task 12（最终组装阶段）。中间产物task1-output.md中的元数据和18条关键引用**全部准确**。

**为什么？** 组装子代理面对3664行多源中间产物+681行原文，在超长上下文压缩融合中：
1. **记忆混入**：把其他文章的信息（字节跳动、苏孟晋）混了进来
2. **创作冲动**：为了让论述更生动，"合理创作"了比喻性引语
3. **完整性补全**：为了让论述更完整，虚构了原文不存在的概念

**关键认知**：中间产物各自正确 ≠ 组装后正确。1+1可能小于2，信息整合本身就是风险点。

### 发现2："有引用"≠"引用正确"——结构化验证的盲区

我们的checklist验证的是**结构完整性**（"有没有引文？"→有，打✅），而不是**事实准确性**（"引文对不对？"→没查）。

这就像一个代码审查只检查"有没有测试"，不检查"测试测的对不对"。

| 验证类型 | 检查问题 | 能否防幻觉 |
|---------|---------|-----------|
| 结构验证 | "有没有？" | ❌ 不能 |
| 事实验证 | "对不对？" | ✅ 能 |

### 发现3：术语漂移是最隐蔽的幻觉

编造引文容易发现——引号里的话原文没有，一搜就知道。术语漂移极难发现——用近义词替换原文术语，读起来"很合理"，但悄悄改变了概念边界。

**本次案例**：原文"控制"（覆盖前馈+反馈两个方向）→ 报告"反馈"（只剩反馈，丢失前馈）。14处替换，没有一处在结构检查中被发现。

### 发现4：分析防御机制时最容易被防御机制击败

这就是我们所说的**"自指盲区"或"递归讽刺"**：
- 写测试方法论的人，自己的代码可能没测
- 写安全规范的人，自己的系统可能有安全漏洞
- 写"如何防止幻觉"的人，自己的文章可能有幻觉
- 写"V对抗审查防止共享误解"的人，自己的V验证恰好犯了共享误解

原因很简单：**理解一个防御机制≠自动免疫于该机制的攻击目标**。理解了"共享误解"问题，不代表你就不会共享误解。

## 🛠️ 三、两个可复用模式

### 模式1：SVA（Source-Verify-Assemble）——外部内容分析的事实保真工作流

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  S 源锚定    │────→│  V 独立验证  │────→│  A 约束组装  │
│             │     │             │     │             │
│ • 提取原文   │     │ • 新子代理   │     │ • 引文标位置 │
│ • 建立术语表 │     │ • 逐条比对   │     │ • 禁意译引用 │
│ • 提取元数据 │     │ • 只查不改   │     │ • SSOT约束  │
│ • SSOT确立  │     │ • 输出错单   │     │ • 禁虚构概念 │
└─────────────┘     └─────────────┘     └─────────────┘
```

**核心改变**：在"组装"和"交付"之间，强制插入一个**未参与组装**的独立子代理做事实核查。

**V（独立验证）核查5维度**：
1. 引文真实性——>块引用每句话在原文中可定位？
2. 术语一致性——关键术语与术语表一致？
3. 案例归属——数据/引述张冠李戴？
4. 章节标注——标注的章节号准确？
5. 虚构概念——提到的概念/框架/数字在原文中存在？

### 模式2：CLN（Citation-Line-Number）——引文行号强制规则

委派子代理分析外部材料时，任务描述中加入：

> **强制规则**：所有引用原文的`>`块引用，必须标注精确位置（如"——第6.3节"或"——原文L245"）。无法定位的说法标记为[推断]而非引用。禁止伪造行号或章节号。

这个简单规则解决两个问题：
- 子代理写引语时会更谨慎（因为要找位置，找不到就不能当引文写）
- 后续核查效率极高（直接goto位置验证）

## ✅ 四、自指盲区防御——7项检查清单

当你在写关于"如何避免错误""如何做好验证""质量保障方法论"这类内容时，**必须**执行以下检查：

### 🔴 必做项（不可跳过）

| # | 检查项 | 做法 |
|---|--------|------|
| 1 | **异体验证** | 最终验证由未参与产出的新子代理执行，不给中间推理，只给原文+产出+核查指令 |
| 2 | **反向应用** | 把你正在论述的防御方法，用来防御你自己的产出。你写V审查，就用V审查V；你写测试，就用测试方法测你的事实声明 |
| 3 | **引文溯源** | 所有>块引用标注位置；所有数字标注来源；"写得太好的引文"必须二次核查 |

### 🟡 强烈建议

| # | 检查项 | 做法 |
|---|--------|------|
| 4 | **术语锁定** | 先提取术语表（术语→定义→禁换词），产出后用grep比对是否有近义词替换 |
| 5 | **否定测试** | 把你的核心论点取反，如果取反后恰好描述了你产出中的缺陷→你正处于盲区中，必须修复 |
| 6 | **边界声明** | 论述X的优势时，必须同时声明X的局限性（如"checklist验证结构完整性，但无法替代独立事实核查"） |
| 7 | **冷却重读** | 完成后执行一轮不相关任务，再假装是审核者重读，重点审视"读起来特别顺"的段落 |

### 🚦 红灯信号（出现任一项立即暂停）

- ❗ "这段写得真好，太有道理了" → 可能是编造的"合理内容"
- ❗ "我理解了V机制，不会犯V的错误" → 理解即免疫错觉
- ❗ "checklist全过了，应该没问题" → 结构验证≠事实验证
- ❗ "这个术语和原文差不多" → 术语漂移预警
- ❗ "我的核心论点反过来好像也成立" → 否定测试触发

## 📋 五、快速上手：下次分析外部内容时

```
□ 1. 提取原文，保存为source.md
□ 2. 从原文提取关键术语表glossary（术语+定义+禁换词）
□ 3. 子代理任务描述中加入CLN引文行号规则
□ 4. 并行委派分析任务
□ 5. 最终组装时：以source.md为唯一事实源(SSOT)
□ 6. 【新增】组装完成后，委派新子代理做SVA独立核查（5维度）
□ 7. 【新增】如果分析主题涉及验证/审查/质量保障本身，额外执行自指盲区7项检查
□ 8. 修复核查发现的错误
□ 9. 交付
```
