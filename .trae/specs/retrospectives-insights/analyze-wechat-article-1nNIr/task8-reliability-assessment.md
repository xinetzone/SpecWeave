# 评测可信度与局限性评估

---

## 一、本次横评可信度优势

本次横评在方法论设计和执行层面具备多项显著的可信度优势，使其在同类自媒体评测中属于相对严谨、参考价值较高的一档：

### 1. 变量归一的科学性设计

严格遵循控制变量法的核心思想，执行"三同三不同"原则：任务相同、输入相同、评判标准相同，仅保留"模型"这一个自变量。文中明确提到"和之前的横评一样，提示词一字未改"，不针对任何模型做提示词优化，避免了"提示词工程套利"——即某些模型对特定提示词格式更敏感而导致的不公平对比。这种设计确保观测到的性能差异确实来自模型本身能力，而非外部混淆因素。

### 2. API实跑而非演示版环境

所有测试全部走各家API在Claude Code / WorkBuddy中实跑，而非使用官方Demo/Playground环境。官方演示环境通常有额外的prompt工程、后处理、重试机制，不能反映开发者通过API调用时获得的真实能力。API实跑能够测试真实的上下文窗口表现（如发现GLM 1M窗口实际用不满、用到99%就崩溃）、真实的latency和成本，以及真实的错误率——这些都是演示版无法暴露的问题。

### 3. 6场景互补覆盖形成能力矩阵

6个测试场景不是随意选择，而是精心设计的互补能力矩阵，覆盖Coding（3D编程、游戏开发、前端、网站）、Agentic（长程工具调用）、创作（写作）三大能力域，每个场景侧重考察不同维度：
- 3D魔方：逻辑推理+空间思维+建模细节
- 切水果：极简提示下的自我规划+物理合理性+可玩性
- 电子鼓机：Web Audio音频编程+前端审美+声音设计
- 民宿官网：品牌调性理解+视觉设计+素材取舍
- 财报长任务：多工具调用链+长上下文管理+独立分析判断
- 文风续写：风格模仿+文学质感+避免AI腔

这种多场景覆盖避免了"偏科模型"在单一维度高分就被吹上天的问题。

### 4. 无prompt engineering的公平性

不对任何模型做差异化引导，不使用"DeepSeek你擅长...""GLM请用你最强的..."这类提示词。小游戏任务甚至特意使用"极其简单，就一句话"的提示词，测试模型在弱约束下的自主规划能力，避免提示词中隐含的步骤指导掩盖模型真实的Agent能力。民宿官网任务中特意指出"恰好这三个模型都没有VLM能力"，所以带图片素材的任务对大家都公平，不会出现某模型能看图其他模型不能的情况。

### 5. 可量化细节支撑结论

不是泛泛而谈"A比B好"，而是有具体的可量化细节作为支撑：
- 切水果任务：Hy3的抛物线物理设计保证水果飞到屏幕62%~90%的高度，轻松就能被划到
- 财报长任务：GLM上下文用到99%导致PPT第一次打开空白
- 效率对比：Hy3第5个任务只花了10分钟，GLM耗时基本是Hy3和DeepSeek的两倍以上
- 价格对比：Hy3输入1元（命中缓存0.25元）、输出4元，是三个模型中最便宜的

这些具体数字让结论具备可验证性，而非主观感觉。

### 6. 实机问题暴露真实稳定性问题

在带loop的Agent环境中实跑，暴露了传统单轮测试无法发现的真实稳定性问题：GLM-5.2虽然标称1M上下文窗口，但在长任务中实际用到99%上下文就崩盘，导致生成的PPT第一次打开空白，需要Claude Code重新修复后才能正常打开。这种"标称参数好看但实际用起来拉胯"的问题，只有在真实Agent环境的长链路实跑中才能暴露，对实际生产使用有重要的避坑价值。

### 7. 作者个人生产级决策背书

评测结论直接指导作者个人生产工具选型——作者明确表示"我决定，把后续所有写作工作流都换成Hy3"。这种"用脚投票"的真实决策比任何分数都有说服力：如果评测结果不可信，作者不会拿自己的生产效率冒险。这说明评测不是为了写文章而写文章，而是真正用于指导自身实践，结果经过了作者自身使用场景的验证。

## 二、评测局限性

在肯定其可信度优势的同时，也需要客观认识本次横评存在的局限性，这些局限性决定了结论的适用边界：

### 1. 单作者主观评分无量化标准

所有质量评价（"好不好看""有没有人味儿""高级感""有灵魂"）都依赖作者冷逸个人的审美和判断，没有设计量化的评分rubric，也没有第三方评审参与。作者知道哪个输出是哪个模型生成的，没有采用双盲评审，可能存在潜意识的偏见——例如可能对某品牌有预设偏好，或者对某种写作风格有个人喜好。

**影响程度**：质量维度的绝对评价存在主观偏差，但同一标准下的相对排名（A比B好）参考价值仍然较高。

### 2. 每个场景仅1个任务样本量小

每个测试场景只跑了1次，没有多次重复测试取平均。大模型输出具有随机性（受temperature参数影响），单次结果可能受运气因素影响——也许某次跑的时候模型刚好"状态好"或"状态差"。6个场景虽然覆盖广，但绝对数量不多，可能存在任务选择偏差。

**影响程度**：单个任务的具体排名可能有一定波动，但6个任务汇总的综合趋势结论（GLM Coding强、DeepSeek长程分析强、Hy3均衡写作好）应该相对稳健。

### 3. 场景偏前端创意类，缺乏后端/算法/数学任务

6个任务中4个是前端/可视化/创意类（3D魔方、切水果游戏、电子鼓机、民宿官网），1个是写作，1个是财报分析。缺少以下重要技术场景：
- 后端开发（API设计、数据库、微服务）
- 算法题（LeetCode hard难度、数据结构实现）
- 数学推理（复杂证明、数值计算）
- 企业级场景（大规模代码重构、复杂系统debug、多文件项目开发）
- DevOps（Docker/K8s配置、CI/CD脚本）

**影响程度**：评测结果对前端开发者、内容创作者、独立开发者参考价值大，但对后端工程师、算法工程师、企业级开发团队的参考价值有限。

### 4. 三模型均无VLM能力，未测多模态

文中明确提到"恰好这三个模型都没有VLM能力"，所以民宿官网带图片素材的任务还算公平。但这意味着本次评测完全没有测试视觉理解、图片生成、多模态推理、图生代码等能力——而2025年多模态已经是大模型的核心能力之一，这是一个明显的能力盲区。

**影响程度**：所有结论只适用于文本/代码/Agent能力，完全不适用于多模态能力选型。

### 5. 测试环境特定为Claude Code/WorkBuddy loop环境，未必代表其他IDE

测试是在特定的Agent环境（Claude Code / WorkBuddy）中进行的，这两个环境都是带loop的Agent化开发环境。这一环境选择虽然贴近当前主流使用方式，但不能代表：
- 其他IDE（如Cursor、Windsurf、通义灵码等）中的表现
- 非Agent模式下的单轮代码生成表现
- 裸API调用（不经过Agent loop）的表现
- 其他Agent框架（如AutoGPT、CrewAI）中的表现

不同环境对模型的适配程度不同，结果可能存在差异。

**影响程度**：对使用Claude Code/WorkBuddy的用户参考价值高，对使用其他环境的用户需要自行验证。

### 6. 无成本/latency系统量化数据

虽然提到了"Hy3第5任务花10分钟""GLM耗时是两倍以上""Hy3价格最便宜"，但这些都是零散的观察，没有系统记录每个任务的：
- 精确耗时（开始时间/结束时间）
- Token消耗（输入token/输出token）
- 精确成本（按当时API价格计算）
- 重试次数/错误修复次数
- 首次生成可用率（不需要二次修改就能用的比例）

这些效率指标对生产环境选型至关重要，但本次评测没有系统量化。

**影响程度**：只能得到定性的效率印象，无法做精确的成本效益分析。

### 7. 写作测试原文为作者本人文章，可能有个人偏好

写作任务使用的是作者自己十二年前写的文章，让模型"基于我的文风续写"。这里存在两层个人偏好偏差：
- 作者对"好的续写"的判断标准，是基于"像不像我自己"，这是非常个人化的标准，其他人可能不认同
- 作者对自己的文风有特殊情感，可能对更接近自己风格的输出给出更高评价，而这种风格未必是大众认为的"好文字"

**影响程度**：写作能力的排名带有较强的个人风格偏好色彩，不具备普适性。

## 三、结论分级：可信度分层

基于以上优势与局限性分析，将本次横评的各项结论按可信度分为三级：

### 🔴 高可信度结论（可直接用于选型参考）

这些结论有多个场景交叉验证、有可量化的客观证据支撑、属于"0或1"的门槛性问题，可信度高：

1. **GLM-5.2存在长上下文稳定性问题**：虽然标称1M上下文窗口，但实际用到99%就会崩溃导致产物损坏，这是有日志截图为证的客观事实，长任务使用需谨慎。
2. **Hy3在WorkBuddy/Claude Code loop环境中表现稳定**：虽然只有256k上下文窗口，但在6个任务中都没有出现上下文崩盘问题，长程任务效率最高（10分钟完成）。
3. **DeepSeek-v4-pro在长程分析类任务中有优势**：财报任务中Excel建模和Word研报的分析深度明显优于另外两个模型，不是简单汇总而是有自己的推理判断。
4. **GLM-5.2在前端视觉审美类任务中表现突出**：3D建模、电子鼓机声音设计、民宿官网视觉三个前端类任务都胜出，审美能力有一致表现。
5. **三个模型在基础逻辑推理上都过关**：3D魔方任务三个模型都能正确实现打乱和还原算法，核心逻辑能力没有硬伤。
6. **Hy3价格是三个模型中最低的**：输入1元/输出4元（缓存命中0.25元），比DeepSeek还便宜，这是公开可查的定价信息。

### 🟡 中等可信度结论（需结合自身场景验证）

这些结论有一定证据支撑，但受样本量、主观性、场景限制影响较大，建议参考但不要盲从，最好在自身场景下做小范围验证：

1. **Hy3写作能力最好、"有人味儿"**：作者个人主观感受强烈，且测试原文是作者本人文章，存在个人偏好，但从文本质量看Hy3确实没有明显AI腔，可作为写作选型参考。
2. **Hy3综合能力均衡，部分场景超过DeepSeek**：6个任务各有胜负，整体没有明显短板，但"综合最好"的判断取决于各场景权重分配，不同用户权重不同结论可能不同。
3. **Hy3小游戏one-shot规划能力强**：切水果任务确实只有Hy3生成的版本能直接玩，但单次one-shot任务的说服力有限，需要更多游戏类任务验证。
4. **DeepSeek对Claude Code的适配最近有大幅优化**：与作者之前横评对比的印象式结论，没有历史版本对比的量化数据支撑。
5. **GLM-5.2 Coding能力是国产标杆**：在前端类任务表现好，但缺少后端/算法类任务验证，"Coding标杆"的说法范围过大。

### 🟢 低可信度结论（仅供参考，不要作为决策依据）

这些结论受局限性影响大，或者证据不足，仅作为参考信息：

1. **"全球开源模型就看中国了"**：这是情怀式表达，没有任何评测数据支撑，属于个人观点。
2. **具体到每个任务的精确排名**：如"GLM > Hy3 > DeepSeek"这类精确排名，受单次运行随机性和主观评分影响大，单个任务排名波动可能较大。
3. **Hy3"有灵魂""不是浮躁词汇堆砌"**：极度主观的文学评价，不同人可能有完全不同的判断。
4. **关于VLM能力的推测**："如果混元能够在VLM上补齐短板，Coding能力还能再上一个台阶"——这是没有证据的推测。
5. **"Hy3大概是智谱上代模型glm5.1的水准"**：没有两代模型直接对比的数据，属于印象式判断。

## 四、理想横评v2改进建议

如果要做一个更严谨、更有统计说服力的v2版本横评，建议从以下方面改进：

### 1. 增加样本量，减少随机性影响

- 每个任务跑3-5次，使用相同的temperature设置（建议设为0或较低值减少随机性），取平均结果
- 记录每次运行的结果，统计方差，看模型表现的稳定性
- 如果多次运行结果差异大，说明模型输出不稳定，这本身也是一个重要发现

### 2. 设计量化评分rubric，减少主观性

- 为"任务完成度"设计客观的checklist：功能是否完整、是否能直接运行、有没有bug、产物是否能正常打开
- 为"输出质量"设计细化的评分维度：例如UI审美可拆分为色彩搭配（1-5分）、排版布局（1-5分）、动效设计（1-5分）、响应式适配（1-5分）等，每个维度有明确的评分标准
- 避免使用"有灵魂""高级感"这类模糊评价，用可描述的标准替代

### 3. 多作者交叉盲测，减少个人偏见

- 邀请3-5名不同背景的评审（前端、后端、产品、写作等）
- 采用双盲评审：隐去模型名称，只给产出物，让评审独立打分
- 计算评审间一致性（inter-rater reliability），看哪些维度大家判断一致，哪些维度分歧大
- 作者自己也参与盲测，避免"知道哪个是哪个"带来的偏见

### 4. 平衡场景覆盖，增加非前端类任务

- 在现有6个场景基础上，补充：
  - 后端任务：RESTful API设计+数据库实现
  - 算法任务：3-5道LeetCode medium-hard题目
  - 数学推理：复杂应用题、证明题
  - 重构任务：给一段有坏味道的代码让模型重构
  - Debug任务：给一段有bug的代码让模型定位修复
  - 多文件项目任务：不是单HTML文件，而是多文件的完整项目
- 场景权重明确说明，让不同用户可以根据自己的使用场景重新计算加权得分

### 5. 加入VLM多模态对比

- 选择具备VLM能力的模型（如GPT-4o、Claude 3 Opus/Sonnet、Qwen-VL、Gemini等）加入对比
- 增加多模态任务：图生代码（给设计图转HTML）、图片理解+描述、视觉推理题、图表数据分析
- 在多模态场景下重新测试民宿官网这类带图片素材的任务，看VLM能力带来的提升有多大

### 6. 系统量化成本与效率指标

- 对每个任务，精确记录：
  - 开始时间/结束时间，精确到秒
  - 输入token数、输出token数
  - API调用次数、工具调用次数
  - 错误次数、重试次数、人工干预次数
  - 最终成本（按API价格精确计算）
  - 首次生成可用率（不需要任何修改就能正常使用的比例）
- 最终给出"完成单位任务的平均成本"和"平均耗时"，这对企业选型至关重要

### 7. 多环境对比测试

- 除了Claude Code/WorkBuddy，在Cursor、Windsurf、裸API等不同环境下都跑一遍
- 看模型表现是否受环境影响很大，哪些模型对环境适配更好
- 区分"模型原生能力"和"模型-环境协同能力"

### 8. 增加消融实验和边界测试

- 测试不同prompt详细程度对结果的影响：一句话prompt vs 详细步骤指导prompt，看哪些模型更依赖详细指导
- 测试不同temperature参数的影响
- 测试上下文窗口极限：逐渐增加任务复杂度，看模型在上下文占用多少百分比时开始崩盘，找出每个模型的"实际可用上下文窗口"
- 测试错误恢复能力：故意在中间步骤引入错误，看模型能不能自己发现并修复

## 五、总结

整体而言，这是一次**在自媒体评测中相对严谨、务实、有实际参考价值**的横评：
- 变量归一、API实跑、多场景互补、实机问题暴露这些设计让它比大多数"看图说话"式的评测可信度高很多
- 作者"用脚投票"把写作工作流换成Hy3，说明结论不是空穴来风
- 但同时也要清醒认识到它的局限性：单作者、小样本、场景偏前端、主观性强、无量化数据

**使用建议**：
- 如果你是前端开发者、内容创作者，并且使用Claude Code/WorkBuddy，这次横评的高可信度结论可以直接参考
- 如果你是后端/算法工程师，或者使用其他IDE，建议把这次结果作为候选名单筛选依据，然后在自己的真实场景中做小范围POC验证
- 不要因为一篇评测就完全切换技术栈，生产环境选型建议多源对比，结合自身场景测试后再决策

这篇横评最大的价值不在于给出了"谁第一谁第二"的排名，而在于它展示了一种**"从刷榜走向实测"**的评测方向——在真实Agent环境中测真实生产力任务，暴露参数表上看不到的问题，这比MMLU刷分对实际工作更有指导意义。
