# 七核心概念方法论体系整合 - Product Requirement Document

## Overview
- **Summary**: 基于SpecWeave项目从初创到当前（2026-07，800+次提交、237+个可复用模式、82% Spec完成率）的完整发展历程，运用第一性原理进行全面系统性复盘，制定「复盘、洞察、萃取、原子提交、原子化、第一性原理、对抗性审查」七个核心概念之间的具体组合规则与应用流程，形成一套逻辑严密、可操作、可验证的元方法论体系，指导未来项目的高效执行与持续优化。
- **Purpose**: 当前七个核心概念各自有独立的指令集定义，但缺乏统一的组合规则、触发时序、交互机制和质量标准，导致执行中存在概念使用随意、组合方式凭经验、闭环断裂等问题。通过第一性原理复盘项目历史，提炼七概念的本质定位与相互作用规律，建立标准化的组合应用方法论。
- **Target Users**: SpecWeave项目所有AI智能体角色（orchestrator/architect/developer/reviewer/tester/co-founder）、采用SpecWeave规范体系的外部项目开发者、方法论研究与实践者。

## Goals
- 基于项目800+次提交的真实历史数据，完成从初创到当前的全面系统性复盘
- 明确七个核心概念在项目管理全周期中的本质定位与层级关系
- 制定七概念之间的触发条件、组合规则、交互流程与质量门禁
- 建立覆盖「感知→认知→验证→执行→沉淀」全链路的闭环方法论体系
- 产出可落地的操作手册、检查清单、决策树与Mermaid流程图
- 完成方法论自身的对抗性审查与自举验证，确保自洽性

## Non-Goals (Out of Scope)
- 不修改七个核心概念各自既有的独立指令集定义
- 不开发新的自动化脚本或工具（本任务为方法论制定，非工具实现）
- 不对历史提交进行批量修复或重构
- 不替代现有各指令集的RACI责任矩阵
- 不涉及vendor/flexloop子模块的专项方法论
- 不创建新的角色或权限体系

## Background & Context
- **项目发展阶段**（基于.trae/specs/README.md里程碑）：
  1. 基础奠基期：Git仓库初始化 → AGENTS.md全局契约 → 角色体系 → 工具协议
  2. 规范建设期：一致性检查 → 命名规范 → Spec框架 → IDE适配
  3. 品牌文档期：品牌定位（SpecWeave）→ 蓝图规划 → 系统规划 → 协作场景
  4. 知识沉淀期：文档重组 → 复盘体系 → 经验萃取 → 洞察重组 → 治理细分
  5. 体系完善期：角色扩展 → 治理规则 → 协作环境 → 同步机制
  6. 迁移归档期：内容萃取 → 迁移落地 → 分类整理 → 任务看板
- **现有资产**：
  - 6个核心概念已有独立指令集（[retrospective.md](../../../../.agents/commands/retrospective.md)、[insight.md](../../../../.agents/commands/insight.md)、[first-principles.md](../../../../.agents/commands/first-principles.md)、[atomic-commit.md](../../../../.agents/commands/atomic-commit.md)、[atomization.md](../../../../.agents/commands/atomization.md)、[adversarial-review.md](../../../../.agents/commands/adversarial-review.md)）
  - 「萃取」对应[self-extraction.md](../../../../.agents/modules/self-extraction.md)模块定义
  - 已有[methodology-analysis-report.md](../../../../.agents/docs/methodology-analysis-report.md)覆盖6个概念（缺第一性原理、对抗性审查），且已原子化为15个模式
  - 四层自我演进体系：感知层（洞察/复盘）→ 认知层（萃取/进化）→ 执行层（迭代/验证）→ 治理层（管理/发展）
  - 237+个可复用模式库，其中方法论模式177+个
- **关键问题识别**（基于项目实践观察）：
  - 七概念的使用边界不清晰，存在过度使用或应触发未触发的情况
  - 概念组合顺序凭经验，缺乏标准化触发时序
  - 闭环验证机制不健全，存在"只复盘不萃取"、"只原子化不审查"等断裂
  - 缺乏统一的决策树来判断何时使用哪个/哪些概念组合
  - 质量验收标准分散在各指令集中，缺乏组合场景下的综合门禁

## Functional Requirements
- **FR-1**: 项目全面系统性复盘报告
  - 覆盖从初创（Git初始化）到当前（2026-07-10）的完整时间线
  - 按六大发展阶段梳理关键事件、决策点、产出物、问题与经验
  - 提取七概念在各阶段的实际应用案例与效果数据
  - 识别七概念应用中的成功模式、反模式与教训
- **FR-2**: 七概念本质定位与层级模型
  - 运用第一性原理，剥离经验类比，提炼每个概念的不可再分基础要素
  - 建立七概念在项目管理周期中的定位图谱（感知层/认知层/验证层/执行层/沉淀层）
  - 明确概念间的依赖关系、前置条件与后置输出
  - 绘制概念关系全景Mermaid图
- **FR-3**: 组合触发条件与决策树
  - 针对项目管理典型场景（里程碑完成、问题发生、新功能开发、重构、文档整理、知识沉淀等），定义标准化的概念触发规则
  - 建立触发决策树：从问题特征判断应启动哪些概念、以什么顺序启动
  - 定义单一概念使用、双概念组合、多概念协同的场景边界
  - 明确成本收益阈值：何时不应使用某概念（避免过度方法论）
- **FR-4**: 标准化组合应用流程
  - 定义5种核心组合流程：
    1. 里程碑闭环流程（复盘→洞察→萃取→原子提交）
    2. 问题解决流程（第一性原理→对抗性审查→原子提交）
    3. 重构优化流程（洞察→原子化→对抗性审查→原子提交）
    4. 知识沉淀流程（复盘→萃取→原子化→原子提交）
    5. 创新突破流程（第一性原理→对抗性审查→洞察→萃取）
  - 每个流程包含：触发条件、输入输出、执行步骤、角色分工、质量门禁
  - 流程间的跳转与嵌套规则
- **FR-5**: 交互机制与接口规范
  - 定义概念间的数据传递格式（上一概念的输出如何成为下一概念的输入）
  - 建立跨概念的追溯机制：从最终产出可回溯到每个环节的决策依据
  - 定义异常处理流程：某一环节失败时的回退、重试或升级路径
  - 明确RACI在组合场景下的责任聚合规则
- **FR-6**: 质量标准与验收门禁
  - 七概念组合应用的质量检查清单
  - 每个组合流程的通过/不通过判定标准
  - 反模式识别清单（表演式复盘、过度原子化、形式化审查等）
  - 成熟度评估标准（L1-L4）
- **FR-7**: 方法论自举验证（对抗性审查）
  - 使用对抗性审查方法对本方法论自身进行七模块协议验证
  - 来源交叉验证、逻辑一致性审查、偏差识别
  - 验证方法论的自洽性、完备性、可操作性
  - 记录验证日志与异常处理

## Non-Functional Requirements
- **NFR-1**: 可操作性：所有流程必须包含可直接执行的检查清单，每个步骤有明确的"做什么/怎么做/验收标准"
- **NFR-2**: 可追溯性：方法论中的每条规则必须能追溯到项目历史中的具体案例或第一性原理推导
- **NFR-3**: 简洁性：核心规则文档≤5000字符，提供Quick Reference速查表（≤1页）
- **NFR-4**: 一致性：新规则不与现有7个独立指令集矛盾，对现有指令集形成补充而非替代
- **NFR-5**: 可验证性：所有质量标准必须是可检查的（programmatic或human-judgment）
- **NFR-6**: 自洽性：方法论自身必须通过对抗性审查，不能存在内部矛盾或循环论证

## Constraints
- **Technical**:
  - 必须遵循项目现有文档规范（YAML frontmatter、相对路径引用、Mermaid图表、原子化原则）
  - 产出物格式为Markdown，存储在`docs/retrospective/patterns/methodology-patterns/governance-strategy/`目录下
  - 必须通过项目CI检查（链接检查、重复代码检测等）
- **Business**:
  - 方法论需兼容现有800+次提交形成的实践习惯，不要求历史追溯修改
  - 需考虑智能体执行的可理解性，规则表述无歧义
- **Dependencies**:
  - 依赖现有7个概念的独立指令集文档
  - 依赖项目git历史数据（800+提交）
  - 依赖现有237+个可复用模式库
  - 依赖[methodology-analysis-report.md](../../../../.agents/docs/methodology-analysis-report.md)原子化产出的15个基础模式

## Assumptions
- 七个核心概念的独立指令集定义是稳定且正确的，本任务不修改它们
- 项目git历史完整可查，能够提取关键事件与决策点
- 现有可复用模式库（237+）覆盖了大部分实践场景，可作为规则提炼的依据
- 第一性原理分析适用于方法论层面的抽象（而非仅用于具体技术问题）
- 对抗性审查的知识研究场景（七模块协议）可直接应用于方法论自身验证
- 项目使用中文作为主要沟通语言，所有产出物为中文

## Acceptance Criteria

### AC-1: 全面系统性复盘报告完成
- **Given**: 项目git历史、现有文档、237+模式库
- **When**: 完成第一阶段复盘工作
- **Then**: 产出完整的项目发展复盘报告，包含：
  - 六大阶段时间线，每个阶段≥3个关键事件
  - 七概念在各阶段的应用案例统计（使用次数、成功率、典型问题）
  - ≥10个成功模式与≥5个反模式的提炼
  - 数据可验证（提交数、文件数、模式数等均通过实际统计）
- **Verification**: `programmatic`
- **Notes**: 数据统计使用git命令实际运行获取，禁止估算

### AC-2: 七概念本质定位与层级模型清晰
- **Given**: 七个概念的现有指令集
- **When**: 完成第一性原理分析
- **Then**: 产出七概念定位模型：
  - 每个概念提炼出≤5个不可再分的基础要素
  - 五层定位图谱（感知/认知/验证/执行/沉淀）清晰标注每个概念的位置
  - 概念间依赖关系无循环、无矛盾
  - Mermaid关系图完整可读
- **Verification**: `human-judgment`
- **Notes**: 需通过architect角色的技术评审

### AC-3: 触发决策树完整可用
- **Given**: 项目管理典型场景清单（≥15种）
- **When**: 完成触发规则制定
- **Then**: 产出触发决策树：
  - 覆盖≥15种典型场景，每个场景明确应使用的概念组合
  - 决策树逻辑清晰，从问题特征到概念选择路径无歧义
  - 包含成本收益判断节点（何时不用/少用）
  - Mermaid决策图可渲染
- **Verification**: `human-judgment`

### AC-4: 五种核心组合流程可操作
- **Given**: 触发决策树
- **When**: 完成流程设计
- **Then**: 产出5种组合流程文档：
  - 每个流程包含触发条件、输入、输出、步骤、角色、门禁
  - 流程步骤数≤10步，每步有明确验收标准
  - 流程间跳转与嵌套规则清晰
  - 每个流程配有Mermaid流程图
- **Verification**: `programmatic` + `human-judgment`
- **Notes**: 流程步骤可通过检查清单逐项验证

### AC-5: 交互机制与接口规范明确
- **Given**: 五个组合流程
- **When**: 完成交互设计
- **Then**: 产出接口规范：
  - 每个概念的输入/输出格式有明确定义
  - 跨概念追溯机制说明清晰（从产出物可回溯到每个环节）
  - 异常处理流程覆盖≥3种失败场景（某环节不通过、数据缺失、角色冲突）
  - RACI聚合规则说明清楚
- **Verification**: `human-judgment`

### AC-6: 质量标准与检查清单完备
- **Given**: 组合流程与交互规范
- **When**: 完成质量标准制定
- **Then**: 产出质量门禁体系：
  - 综合检查清单≥20项，覆盖所有组合场景
  - 每个流程有明确的通过/不通过判定标准
  - 反模式清单≥8种，每种有识别特征与避免方法
  - L1-L4成熟度评估标准可操作
- **Verification**: `programmatic`
- **Notes**: 检查清单可直接用于实际执行时的逐项打勾

### AC-7: 方法论自举验证通过
- **Given**: 完整方法论体系
- **When**: 执行对抗性审查
- **Then**: 自举验证完成：
  - 七模块协议完整执行（来源三级分类、可信度评分、五维验证、异常标记等）
  - 🟢A级结论占比≥60%，🔴D级结论为0
  - 所有⚠️/❓/⚖️标记均有说明和处理方案
  - 验证日志完整可追溯，第三方可复现审查过程
  - 方法论自洽性得到验证（无内部矛盾、无循环论证）
- **Verification**: `programmatic` + `human-judgment`
- **Notes**: 由reviewer角色执行最终验收

### AC-8: 产出物归档与索引更新
- **Given**: 所有方法论文档通过验证
- **When**: 完成归档
- **Then**: 产出物正确归档：
  - 核心方法论文档原子化存储在`docs/retrospective/patterns/methodology-patterns/governance-strategy/`
  - 每个文件有正确的YAML frontmatter（id、title、source、x-toml-ref）
  - 更新相关索引（README、导航表、模式索引）
  - 所有链接有效，无断链
  - 通过CI检查（链接检查、重复检测）
- **Verification**: `programmatic`
