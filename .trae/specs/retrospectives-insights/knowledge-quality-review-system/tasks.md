## [x] Task 3: 产出"高效阅读技术文档"七概念实战案例
- **Priority**: high
- **Depends On**: Task 1
- **Description**: 
  - 选一个真实的开源技术文档作为示范对象（推荐用FastAPI官方文档或React官方文档，大家都熟悉）
  - 严格按照七概念七个部分对应：
    1. R（事实层·文档类型分析）：先判断你手里拿的是什么文档——API参考/入门教程/架构设计/规范标准/源码注释？不同类型读法完全不一样
    2. F（第一性原理·目标设定）：打开文档之前先问自己"我到底要解决什么问题"，别一上来就从第一页开始读
    3. I（洞察层·信息提取）：怎么快速扫结构、找核心概念、定位关键API、识别"哪些是必看哪些是备查"
    4. V（对抗层·难点攻克）：遇到看不懂的地方怎么自己杠自己——"这句话换个说法是什么意思？""如果不这么做会怎样？""有没有反例？"
    5. E（萃取层·知识转化）：看完了怎么沉淀——写代码片段/做对比表格/提炼模式/写踩坑笔记
    6. C（提交层·效果验证）：怎么证明你真的读懂了——费曼讲法/写个最小demo/给别人讲一遍/做课后题
    7. A（原子化·经验沉淀）：阅读技术文档的5个常见反模式（比如"从头读到尾"、"收藏等于学会"等）
  - 每个部分必须有具体可操作的方法，配上这个真实文档的例子（比如"我读FastAPI文档的时候，第一步是先看目录..."）
- **Acceptance Criteria Addressed**: AC-2, AC-5
- **Test Requirements**:
  - `human-judgement` TR-3.1: 七个部分完整，每个部分都对应到七概念的一个字母
  - `human-judgement` TR-3.2: 有真实文档作为示范案例，不是空讲方法
  - `human-judgement` TR-3.3: 每个方法都是读者能立刻照着用的
  - `human-judgement` TR-3.4: 读完这个案例，读者下次拿到新文档知道从哪下手
- **Notes**: 这是七概念的实战案例，目标是让读者通过这个例子真正理解七概念不是空架子，是能用的

## [x] Task 4: 产出"找三个人杠一遍"标准化操作SOP
- **Priority**: high
- **Depends On**: Task 1
- **Description**: 
  - 准备阶段（会前）：
    - 资料准备清单：作者需要提前给评审人发什么（文档链接、背景介绍、你希望重点看什么）
    - 评审人选择标准：什么人适合当评审？（不能都是跟你一样背景的）三个角色怎么分配（新手视角/专家视角/黑盒视角）
    - 时间安排：不同版本（快速30分钟/标准90分钟/深度3小时）的时间分配表
    - 会前物料：评审表模板、会议议程、文档打印/共享方式
  - 实施阶段（会中）：
    - 三种杠法详解：
      - 事实杠："你这里说XXX，有什么证据/数据/案例支撑吗？"
      - 逻辑杠："你说因为A所以B，那有没有可能A发生了但B没发生？反例是什么？"
      - 场景杠："你这个方法在XX极端场景下还成立吗？如果是小白用呢？如果赶时间呢？"
    - 沟通话术模板：怎么提意见作者不反感？怎么避免人身攻击？怎么区分"我不喜欢"和"这有问题"
    - 杠点识别checklist：拿到一份文档，从哪些角度找问题（20个常见杠点）
    - 分歧解决机制：杠起来了谁拍板？什么情况需要主持人仲裁？什么问题可以保留意见？
  - 收尾阶段（会后）：
    - 意见整理模板：每条意见怎么记（原文位置→问题描述→建议→严重程度）
    - 采纳判定标准：什么意见必须改、什么意见可以讨论、什么意见可以不采纳（说明理由）
    - 闭环跟踪：改完怎么验证？谁复查？怎么避免下次犯同样问题？
  - 裁剪指南：快速版（1个人15分钟自查+1个人快速过）/标准版（3个人90分钟完整流程）/深度版（3-5人3小时工作坊）分别怎么操作
- **Acceptance Criteria Addressed**: AC-3, AC-5
- **Test Requirements**:
  - `human-judgement` TR-4.1: SOP步骤具体，照着走就能开评审会
  - `human-judgement` TR-4.2: 有时间分配表，每个环节多长时间明确
  - `human-judgement` TR-4.3: 有沟通话术和分歧处理，解决"评审会变成吵架会/互相恭维会"的问题
  - `human-judgement` TR-4.4: 三种强度版本都有，适应不同场景
- **Notes**: SOP的目标是解决"想评审但不知道怎么组织"的问题，让评审会有效率不走过场

## [x] Task 5: 产出2份知识评审清单模板（含填写示例）
- **Priority**: high
- **Depends On**: Task 1
- **Description**: 
  - 模板1：通用技术文档评审清单（适合API文档、教程、技术方案、设计文档）
    - 评审维度：准确性/完整性/可读性/逻辑性/实用性/规范性
    - 每个维度3-5个具体评审项
    - 评分标准：1-5分，每个分值有明确描述（1分是什么样、3分是什么样、5分是什么样）
    - 完整表格：评审项、评分、问题描述、修改建议、严重程度（必须改/建议改/可选）
    - 评审人信息、评审日期、总体结论（通过/修改后通过/不通过）
    - 附上一份填写好的示例（用一段示例文档填完一整份清单，展示怎么用）
  - 模板2：方法论/模式库评审清单（适合SOP、最佳实践、可复用模式、流程规范）
    - 评审维度：可迁移性/反模式完整性/边界清晰度/可操作性/成熟度
    - 每个维度3-5个具体评审项
    - 特别增加"跨场景验证"相关评审项（这个模式换个地方能用吗？）
    - 同样有评分标准、意见栏、结论区
    - 同样附上填写示例
  - 两个模板都做成可以直接复制使用的Markdown表格格式
- **Acceptance Criteria Addressed**: AC-4, AC-5
- **Test Requirements**:
  - `human-judgement` TR-5.1: 两个模板结构完整，要素齐全（维度/评审项/评分/意见/建议/结论）
  - `human-judgement` TR-5.2: 每个评审项具体可判断，不是"内容是否优质"这种模糊项
  - `human-judgement` TR-5.3: 有填写示例，用户看了就知道怎么填
  - `programmatic` TR-5.4: Markdown表格语法正确，能正常渲染
- **Notes**: 模板必须"开箱即用"，用户不需要自己再设计，复制过去就能填
