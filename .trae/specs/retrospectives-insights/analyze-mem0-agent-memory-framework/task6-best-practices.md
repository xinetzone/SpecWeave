# Agent 接入最佳实践与工程经验萃取

## 一、接入时序：先检索、后回复、再异步写入

将 Mem0 接入 Agent 的正确时序为三个阶段：

```
用户消息 → 1. memory.search() 检索记忆，注入 system/context
         → 2. Agent 基于上下文推理并回复用户
         → 3. 异步 memory.add() 抽取本轮对话的长期记忆
```

**时序关键原则**：
- **检索必须在推理之前**：不先检索就回复，相当于模型"失忆"状态下回答，无法利用历史偏好和决策。
- **写入必须在回复之后**：先响应用户，再异步执行记忆抽取。LLM 抽取记忆是耗时操作，如果同步等待 add() 完成再回复，会严重阻塞用户响应速度。
- **写入建议异步化**：将 add() 放入后台任务/消息队列执行，不阻塞主请求链路。

## 二、五大接入原则

### 原则 1：作用域设计清晰

| 作用域字段 | 适用对象 | 设计理由 |
|---|---|---|
| `user_id` | 用户个人偏好、历史决策、长期目标 | 区分不同用户的记忆，保证 A 用户的偏好不会泄露给 B 用户 |
| `run_id` | 单次任务/会话的上下文 | 同一用户的不同任务之间相互隔离，避免任务间记忆串扰 |
| `agent_id` | Agent 自身的行为记录或能力沉淀 | Agent 可以积累跨用户的通用知识和操作经验 |

**设计理由**：三层作用域实现了用户维度、任务维度、Agent 维度的记忆隔离，检索时通过作用域字段精确过滤，保证记忆不会乱串。

### 原则 2：检索在推理前

先调用 `search()`，再将 Top-K 相关记忆作为上下文注入给业务 LLM 的 system message 或 context 中。

**设计理由**：大模型本身无状态，不先检索注入记忆，模型无法知道用户的偏好、历史决策和任务状态，相当于每次对话都是"第一次见面"。检索注入是让 Agent 具备长期记忆能力的前提条件。

### 原则 3：写入在回复后

先完成用户响应，再异步调用 `add()` 抽取和保存记忆。

**设计理由**：
1. **性能考虑**：`add()` 内部涉及 LLM 调用（记忆抽取）、向量化、向量库写入、SQLite 写入、实体抽取等多个步骤，同步执行会显著增加用户等待时间。
2. **用户体验**：用户最关心的是得到回复，记忆保存是后台行为，不应阻塞主交互流程。
3. **容错考虑**：即使 add() 失败（如 LLM API 临时不可用），用户已经收到了回复，不影响核心交互。

### 原则 4：metadata 做业务隔离

在 add() 时通过 metadata 传入业务维度标识，如 `project_id`、`workspace_id`、`category` 等。

**设计理由**：user_id/agent_id/run_id 是 Mem0 的内置作用域，但实际业务往往需要更多维度的过滤能力。例如同一用户在不同项目下的记忆需要隔离检索，metadata 提供了灵活的业务层过滤手段。

### 原则 5：高精度场景启用 reranker

在客服、医疗、法务、企业知识库等对记忆召回精度要求极高的场景，配置并启用 reranker 对候选结果做二次排序。

**设计理由**：多路召回（语义+BM25+实体）虽然提升了召回率，但融合排序后的结果不一定完全符合人工判断的相关性标准。Reranker 通常使用交叉编码器（Cross-Encoder）对 query-memory 对做更精细的相关性打分，比单纯的向量相似度+分数融合更准确，适合对错误容忍度低的场景。

## 三、生产级工程经验汇总

| 经验点 | 设计价值 |
|---|---|
| 批量优先、失败降级 | 生产吞吐保障：优先 `embed_batch` 批量向量化 + 一次性 insert，最大化吞吐；provider 不支持或调用失败时自动降级逐条处理，API 失败不崩溃，兼顾性能与韧性 |
| 候选池 4 倍扩容 | 多召回后排序：内部将 top k 扩大 4 倍形成候选池，避免多路召回阶段过早丢弃语义相关但初始分数不高的记忆，为后续融合打分保留更多候选 |
| 动态归一化 max_possible | 多信号分数不虚高：根据实际启用的信号数量（1.0/1.5/2.0/2.5）动态选择归一化分母，确保不同信号组合下的分数可比，不会因为开了更多检索方式导致分数膨胀 |
| ADD-only 保留演化轨迹 | 支持时间推理、多跳检索：新事实作为新记忆加入而非覆盖旧记忆，保留事实变更历史，可回答"什么时候改变的"类问题；同时支持冲突记忆并存 |
| 多层去重（LLM + md5） | 防止记忆膨胀：LLM 层在 Prompt 中参考旧记忆主动避免重复抽取；SDK 层对记忆文本计算 md5 hash，跨批次精确去重，两层配合控制 ADD-only 策略下的记忆数量 |
| SQLite 双表分离 | 事件追溯与上下文窗口各司其职：`history` 表记录 ADD/UPDATE/DELETE 完整事件链（含旧值/新值/时间/角色/actor），支持记忆审计与回溯；`messages` 表只保留每个 scope 最近 10 条消息，供下次 add 构建轻量上下文 |
| 实体抽取用轻量模型（spaCy） | 成本/速度考虑：不每次调 LLM 做实体抽取，而是使用本地 spaCy NLP 模型 + 规则，大幅降低每次 add/search 的成本和延迟，适合高频交互场景 |
| BM25 + 语义 + 实体三路互补 | 关键词精确 + 语义理解 + 实体关联：BM25 解决精确词/日期/术语硬匹配，语义解决同义表达软匹配，实体解决围绕实体的关联召回，三者覆盖不同类型的信息需求 |
| 异步写入 | 不阻塞用户响应：add() 包含 LLM 调用、向量化、多存储写入等耗时操作，异步执行确保用户先收到回复，记忆保存作为后台任务不影响交互体验 |

## 四、不适用场景与替代建议

### 不适用场景：写作 Agent 长文本生成

文章明确指出，不是所有 Agent 都适合 Mem0 这种全量长期记忆架构。典型不适用场景是**写作类 Agent**：

- **问题**：写作 Agent 可能输出长草稿甚至完整正文，内容量极大。如果全量调用 `memory.add()` 去抽取记忆和实体，会抽取出大量零散的、非长期价值的信息，导致记忆库膨胀、信噪比极低。
- **本质原因**：长文本创作的输出内容大多是过程性草稿、措辞选择、段落组织，这些不属于"值得长期记住的用户偏好/计划/事实"，不应进入长期记忆。

### 替代建议

1. **内容筛选后写入**：在调用 `memory.add()` 前，先由业务逻辑或 Agent 自身判断哪些内容值得写入长期记忆，只将关键决策、用户偏好、最终结论等信息传入 add()，丢弃草稿和过程性内容。
2. **封装为工具按需调用**：将 `memory.add()` 封装为 Agent 可调用的工具（function/tool），让 Agent 自主判断什么时候需要记住什么信息，而不是每轮对话自动写入。Agent 在认为有必要时主动调用工具保存记忆，这样既保留了记忆能力，又避免了无差别全量写入带来的膨胀问题。
