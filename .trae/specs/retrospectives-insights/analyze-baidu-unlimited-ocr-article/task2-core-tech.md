# 百度 Unlimited-OCR 核心技术突破提炼

本文档基于 article-content.md 内容，提炼百度 Unlimited-OCR 的三大核心技术突破。

---

## 一、R-SWA（Reference Sliding Window Attention）机制

### 核心思想
R-SWA 是 Unlimited-OCR 解决长文档解析问题的核心创新，其设计灵感来源于人类抄书时的注意力模式：**眼睛始终盯着全部原文（参考 token 全可见），但落笔时只回头看刚写下的几行（输出侧只回溯最近的 token）**。这种非对称注意力设计，既保证了模型始终能"看到"完整的原始文档，又避免了输出序列无限增长带来的内存负担。

### 原文依据
> 它提出了一种叫做Reference Sliding Window Attention(R-SWA)的方法来模仿人抄书时的注意方式。
> 
> 眼睛盯着原文，只回头看刚写下的几行。
> 
> 每生成一个 token，R-SWA 都会去看全部"参考 token"，以确保模型一直能看到完整的原文。
> 
> 但是在输出的那边，它只回溯到前面的 128 个 token，就跟你抄书的时候只看了一眼刚刚写下的那几行一样。

### 实现方式
落实到工程实现上，R-SWA 做了两项关键改动：
1. **全注意力层替换**：将模型中所有的标准注意力层全部替换为 R-SWA 注意力层
2. **KV cache 改造**：将原本无限增长的 KV cache 改为**固定大小的队列结构**，新 token 进入时，最旧的一个 token 会被自动挤出队列

### "软遗忘"机制解释
R-SWA 本质上引入了一种"软遗忘"机制：
- **不遗忘参考信息**：原文（输入侧）的所有 token 始终保持完全可见，模型永远能查阅完整的原始文档
- **有选择地遗忘输出历史**：对于已经生成的输出内容，模型只保留最近 128 个 token 的上下文，更早的输出内容被"软遗忘"——不是真的删除信息，而是不再参与注意力计算
- **为什么这有效**：OCR/文档解析任务本质上是"逐字转录"任务，类似于抄书，只需要记得"刚刚写到哪里"和"原文是什么"，不需要记住很早之前写过的具体字句

### 解决的问题
R-SWA 从根本上解决了长文档解析中的**记忆丢失问题**和**速度下降问题**。在标准注意力机制下，模型要处理越来越长的输出序列，KV cache 持续膨胀，导致内存溢出和推理速度急剧下降——不是模型"不想记住"，而是内存根本"装不下"。R-SWA 通过非对称注意力设计，用一种符合任务特性的"软遗忘"策略，实现了真正的"无限长度"文档解析能力。

## 二、DeepEncoder 编码器

### 来源与定位
DeepEncoder 是配合 R-SWA 使用的视觉编码器，**直接继承自 DeepSeek-OCR 项目**，是 Unlimited-OCR 能够高效处理 PDF 文档的关键组件。

### 原文依据
> 配合 R-SWA 的，是继承自 DeepSeek-OCR 的 DeepEncoder。
> 
> 该编码器可以将一个 1024×1024 的 PDF 页面压缩成只有 256 个视觉 token，压缩比达到 16 倍。
> 
> 对于长文档而言，视觉token编码只进行一次就停止了，整个解码过程不参与状态转移。
> 
> 图像信息一直都很清楚，并不会变得越来越模糊。
> 
> 这个设计有个很关键的地方：视觉 token 不参与状态转移。
> 
> R-SWA 把视觉token挡在了状态转移之外，这就是它与普通的线性注意拉开距离的地方。

### 核心设计
DeepEncoder 的关键设计可以概括为三点：

1. **16倍高效压缩**
   - 输入：一张 1024×1024 分辨率的 PDF 页面图像
   - 输出：仅 256 个视觉 token
   - 压缩比：16倍（相比普通视觉编码器，token 数量大幅减少）

2. **一次性编码，不参与状态转移**
   - 视觉 token 在整个解码过程开始前**只编码一次**
   - 编码完成后，这些视觉 token 作为参考侧的一部分存在，**不参与解码过程的状态转移**
   - 也就是说，无论输出序列多长，视觉信息始终保持原始编码状态，不会被反复"刷新"或"稀释"

3. **与 R-SWA 的协同设计**
   - R-SWA 的非对称注意力机制将视觉 token 完全挡在输出侧状态转移之外
   - 视觉 token 属于"参考 token"，始终可见但不会像输出 token 那样被滑窗挤出
   - 这是与普通线性注意力的本质区别：普通线性注意力中视觉信息会参与每一步的状态计算，随着输出变长逐渐被"稀释"导致图像模糊

### 解决的问题
DeepEncoder 解决了长文档 OCR 中的**视觉信息衰减问题**。在传统的多模态模型中，视觉 token 会参与每一步解码的状态转移，随着输出序列变长，图像信息在注意力计算中被不断稀释，导致后面生成的内容对图像细节的感知越来越模糊——这就是为什么很多模型解析长文档时"越往后越不准"。DeepEncoder 的一次性编码+不参与状态转移设计，确保了**图像信息从头到尾都保持清晰**，配合 R-SWA 的固定窗口机制，真正实现了几十页文档"一气呵成"。

## 三、固定大小 KV cache 队列

### 问题根源
在标准 Transformer 自回归解码过程中，KV cache（键值缓存）是加速推理的关键技术，但也是长序列处理的瓶颈所在：
- **线性增长**：每生成一个新 token，就要将其 key 和 value 存入 KV cache，导致 KV cache 大小随输出长度**线性增长**
- **内存爆炸**：输出 1 万 token 和 10 万 token，KV cache 内存占用相差 10 倍
- **速度下降**：KV cache 越大，每一步注意力计算的开销越大，推理速度越来越慢
- **记忆丢失**：当 KV cache 超过显存容量时，要么截断上下文（导致早期信息丢失），要么 OOM 崩溃

### 原文依据
> 究其原因，在标准的注意力机制下KV cache会随着输出长度线性增加。
> 
> 内存承受不了了，速度越来越慢，并不是模型不想记住东西，而是记不住。
> 
> Unlimited-OCR 的主要突破是将"软遗忘"机制加入到模型中。
> 
> 把KV cache变成一个固定大小的队列，在产生新的token的时候，最旧的一个会被挤出去。
> 
> 输出 1 万个 token 和 10 万个 token，内存占用是完全一样的。

### 解决方案
Unlimited-OCR 将 KV cache 改造为**固定大小的队列（FIFO）结构**：
- 队列大小固定为 128（对应输出侧滑动窗口大小）
- 每生成一个新 token，将其加入队尾
- 如果队列已满，**队首（最旧的 token）自动出队被"挤出"**
- 整个队列始终只保留最近 128 个输出 token 的 KV 值

### 与 R-SWA 的协同工作机制
固定大小 KV cache 是 R-SWA 机制在工程上的直接体现，二者是同一设计的两个层面：

| 维度 | 参考侧（原文/图像） | 输出侧（生成内容） |
|---|---|---|
| 注意力可见性 | 全部可见（始终完整） | 仅可见最近 128 token |
| KV cache 行为 | 静态存储，不增不减 | 固定大小队列，FIFO 淘汰 |
| 内存占用 | 恒定（输入长度决定） | 恒定（128 token 固定） |
| 信息特性 | 原始参考信息，永不遗忘 | 近期输出上下文，"软遗忘"早期内容 |

这种设计的精妙之处在于：
1. **内存恒定**：无论输出 1 万 token 还是 10 万 token，KV cache 内存占用完全相同
2. **速度恒定**：每一步注意力计算的范围固定（全参考+128输出），TPS 不会随输出长度下降
3. **符合任务特性**：文档解析是"转录"任务而非"创作"任务，只需要参考原文和近期输出，不需要回溯很早之前的生成内容

### 解决的问题
固定大小 KV cache 队列从工程层面彻底解决了长序列推理的**两大顽疾**：
1. **内存爆炸问题**：显存占用从 O(n) 降为 O(1)（输出侧），让解析 40+ 页文档成为可能
2. **速度衰减问题**：推理速度不随输出长度下降，Unlimited-OCR 在输出到 6144 token 时 TPS 仍高达 7847，而 DeepSeek-OCR 已跌到 5822，差距达 35%

文章中用一句话总结了这个设计的价值：**"R-SWA 对于解析任务来说就是一份真正的'免费午餐'"**——用极小的改动（替换注意力层+改造 KV cache），获得了"无限长度"解析能力和持续高速推理的双重收益。
