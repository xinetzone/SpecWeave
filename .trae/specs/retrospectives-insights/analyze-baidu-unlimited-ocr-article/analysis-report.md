## 执行摘要

百度开源的 Unlimited-OCR 是一个在文档 OCR 领域具有突破性意义的项目。该模型总参数 3B、实际激活仅 500M，却在 OmniDocBench v1.5 上取得 93.23%、v1.6 上取得 93.92% 的端到端 SOTA 成绩，以 1/470 的参数量反超 235B 的 Qwen3-VL。其核心创新在于 R-SWA（Reference Sliding Window Attention）非对称注意力机制，通过模仿人类"抄书"的注意力模式——参考侧全可见、输出侧仅保留最近 128 token 的滑动窗口——配合 DeepEncoder 视觉编码器的一次性编码设计，从根本上解决了长文档 OCR 的记忆丢失和速度衰减问题，实现了 40+ 页文档"一气呵成"解析且推理速度恒定（输出 6144 token 时 TPS 达 7847，领先 DeepSeek-OCR 35%）。本报告系统梳理其技术原理、性能数据、使用方式与局限性，并提炼对 AI 架构创新、小模型路线及 SpecWeave 系统的深度启示。

## 第一部分：结构化学习笔记

### 一、项目概述

Unlimited-OCR 是百度最新开源的端到端文档 OCR 模型，其核心定位是解决长文档解析中的两大顽疾：**记忆丢失**和**速度下降**。

**核心指标**：
- 总参数量：3B（MoE 稀疏激活架构）
- 实际激活参数：500M（约 5 亿）
- 基准测试成绩：OmniDocBench v1.5 93.23%、v1.6 93.92%（端到端 SOTA）
- 核心能力：支持 40+ 页 PDF 文档一次性解析，不丢信息、不减速
- 训练背景：基于 DeepSeek-OCR 继续训练约 4000 步
- GitHub 地址：https://github.com/baidu/Unlimited-OCR

项目的核心突破在于引入了"软遗忘"机制，跳出了传统长上下文方案"追求全记住"的思维定式，通过符合任务本质的注意力机制设计，实现了真正的"无限长度"文档解析能力。

### 二、三大核心技术突破

#### 1. R-SWA（Reference Sliding Window Attention）机制

**核心原理**：R-SWA 是解决长文档解析问题的核心创新，设计灵感来源于人类抄书时的注意力模式——眼睛始终盯着全部原文（参考 token 全可见），但落笔时只回头看刚写下的几行（输出侧只回溯最近的 token）。

**关键设计**：
- **非对称注意力**：参考侧（输入图像/原文）始终全部可见，输出侧仅回溯最近 128 个 token
- **全注意力层替换**：将模型中所有标准注意力层替换为 R-SWA 注意力层
- **"软遗忘"机制**：不遗忘参考信息，但有选择地遗忘早期输出历史——OCR 是"抄录"任务而非"创作"任务，只需要记得"刚刚写到哪里"和"原文是什么"

**解决的问题**：从根本上解决了长文档解析中标准注意力机制导致的 KV cache 线性增长、内存爆炸、速度衰减问题。

#### 2. DeepEncoder 视觉编码器

**核心原理**：继承自 DeepSeek-OCR 的视觉编码器，与 R-SWA 协同工作，高效处理 PDF 页面图像。

**关键设计**：
- **16 倍高效压缩**：将一张 1024×1024 分辨率的 PDF 页面图像压缩为仅 256 个视觉 token
- **一次性编码**：视觉 token 在解码开始前只编码一次，整个解码过程中不参与状态转移
- **与 R-SWA 协同**：R-SWA 将视觉 token 挡在输出侧状态转移之外，视觉信息始终保持原始编码状态，不会被稀释

**解决的问题**：解决了传统多模态模型中视觉 token 随输出变长逐渐被"稀释"导致图像模糊、长文档识别"越往后越不准"的问题。

#### 3. 固定大小 KV cache 队列

**核心原理**：将标准 Transformer 中无限增长的 KV cache 改造为固定大小的 FIFO 队列结构。

**关键设计**：
- 队列大小固定为 128（对应输出侧滑动窗口）
- 新 token 进入队尾，队首（最旧 token）自动出队
- 参考侧 KV 静态存储、不增不减；输出侧 KV 固定大小、FIFO 淘汰

| 维度 | 参考侧（原文/图像） | 输出侧（生成内容） |
|---|---|---|
| 注意力可见性 | 全部可见（始终完整） | 仅可见最近 128 token |
| KV cache 行为 | 静态存储，不增不减 | 固定大小队列，FIFO 淘汰 |
| 内存占用 | 恒定（输入长度决定） | 恒定（128 token 固定） |
| 信息特性 | 原始参考信息，永不遗忘 | 近期输出上下文，"软遗忘"早期内容 |

**解决的问题**：
- **内存爆炸问题**：输出 1 万 token 和 10 万 token 内存占用完全相同
- **速度衰减问题**：每步注意力计算范围固定，TPS 不随输出长度下降

正如文章总结："R-SWA 对于解析任务来说就是一份真正的'免费午餐'"——用极小的改动获得了"无限长度"解析能力和持续高速推理的双重收益。

### 四、快速上手指南

Unlimited-OCR 提供两种推理方式，分别适用于不同场景。

#### 通用前置处理（两种方式都需要）

PDF 文件不能直接输入模型，需先转换为图片：
```
PDF 文件 → PyMuPDF 转图片（DPI=300） → 图片列表 → 模型解析 → 输出结果
```

#### 1. Transformers 方式（快速上手）

**适用场景**：快速体验、小批量处理、开发调试、个人学习研究。

**依赖安装**：
```bash
pip install torch transformers pymupdf
```

**使用方式**：从 Hugging Face 加载 `baidu/Unlimited-OCR` 模型，直接通过 Python 脚本调用，支持单页图片解析和多页 PDF 批量处理（需先用 PyMuPDF 将 PDF 转图片）。README 中提供了完整示例代码。

#### 2. SGLang 方式（高性能推理）

**适用场景**：大批量文档处理、生产环境部署、高并发服务场景。

**服务器启动**：
```bash
python -m sglang.launch_server --model-path baidu/Unlimited-OCR --port 30000
```

**特性**：提供 OpenAI-compatible HTTP API，支持流式输出，服务端支持高并发。

#### 3. 两种方式对比

| 对比维度 | Transformers 方式 | SGLang 方式 |
|---|---|---|
| **定位** | 快速上手 | 高性能推理 |
| **部署复杂度** | 低，直接 Python 脚本运行 | 中，需启动服务端 |
| **依赖安装** | torch, transformers, pymupdf | sglang（需额外安装） |
| **API 接口** | 原生 Python 函数调用 | OpenAI-compatible HTTP API |
| **流式输出** | 需自行实现 | 原生支持 |
| **并发处理** | 单线程/单批次 | 服务端支持高并发 |
| **吞吐量** | 较低，适合小批量 | 高，适合大批量 |
| **适用阶段** | 开发调试、快速体验 | 生产部署、大规模应用 |

**方式选择**：第一次尝试或处理少量文档选 Transformers；部署服务或处理大量文档选 SGLang。

## 第二部分：深度洞察报告

### 一、技术创新本质：机制创新 > 参数堆砌

Unlimited-OCR 最深刻的启示不在于它刷新了 SOTA，而在于它以 500M 激活参数反超 235B 大模型这一 1:470 的参数效率反差，迫使我们重新审视 AI 技术演进的底层逻辑。

#### 1. "归纳偏置"的力量

500M 打败 235B 证明：对于特定任务，任务适配的机制设计比通用参数堆砌高效至少两个数量级。

通用大模型走"一条架构打天下"路线，用标准全注意力处理所有任务，导致：垂直任务存在大量冗余计算、长序列复杂度 O(n²)、为通用性稀释了垂直任务性能密度。

Unlimited-OCR 走"任务定制架构"路线，它深刻理解 OCR 本质——这是"有参考、逐字输出"的抄录任务而非"无中生有"的创作任务——然后围绕这个本质注入强归纳偏置：
- 参考信息应该始终可见，永不遗忘
- 生成历史只需要保留近期上下文
- 视觉信息是静态参考，不应参与动态状态转移

当架构设计与任务认知模式匹配时，模型不需要用海量参数去"学习"这些本该内置的逻辑，参数效率自然有数量级提升。

#### 2. "软遗忘"的哲学启示

标准注意力追求"全记住"是一种认知上的"蛮力"。人类认知从来不是这样工作的——抄书者不需要记住 10 分钟前抄的具体字句，只需要：眼睛盯着原文（外部记忆）、记得笔停在哪里（工作记忆）、知道当前语义位置。

R-SWA 的"软遗忘"是范式转变：不是"如何记住更多"，而是"哪些该记、哪些该忘"。
- **该记住的永不遗忘**：原始文档、参考资料作为"参考侧"静态存储
- **该遗忘的主动遗忘**：生成历史、中间步骤过了窗口期就主动丢弃

这可以推广为长上下文处理的通用框架：信息分区→差异化记忆策略→外部记忆优先→遗忘即设计。人类能处理远超工作记忆容量的任务，靠的不是"大脑记住一切"，而是"外部存储+选择性注意+工作记忆聚焦当前"的组合策略。

#### 3. 视觉 token 不参与状态转移的关键洞见

"视觉 token 不参与状态转移"是 R-SWA 区别于普通线性注意力的核心。普通线性注意力虽然也把复杂度降到 O(n)，但所有 token 都参与状态转移，视觉信息在循环传递中不断被"稀释"——这就是很多多模态模型长文档识别"越往后越模糊"的原因。

Unlimited-OCR 的解决方案：视觉 token 根本不进状态转移，编码一次后作为参考侧"静静待在那里"，每步注意力都能直接访问原始编码的视觉信息。

这背后是一个具有普遍意义的架构思想：**根据信息的生命周期和变化特性，将其放入不同的处理通路**。

| 信息类型 | 生命周期 | 变化特性 | 处理方式 |
|---|---|---|---|
| **静态参考信息** | 整个任务期间恒定 | 编码后不再变化 | 一次性编码，静态存储，全注意力可见 |
| **动态生成状态** | 随生成过程不断更新 | 每步都有新信息加入 | 滑窗保留，FIFO 淘汰，参与状态转移 |

普通 Transformer 和线性注意力都不做这个区分，导致静态信息被动态信息稀释、长期信息被短期信息淹没。R-SWA 第一次在架构层面明确做了这个区分。

### 二、小模型路线的行业启示

Unlimited-OCR 不是孤例，它代表了一条正在浮现的 AI 发展路线：与其盲目追求更大的通用模型，不如在特定领域用小模型配合专用机制实现 SOTA。

#### 1. MoE+稀疏激活的效率哲学

Unlimited-OCR 采用 MoE 架构（3B 总参数/500M 激活）解决了稠密模型的根本矛盾：**知识容量与推理成本的矛盾**。

- **总参数大（3B）**：模型整体有足够容量存储领域知识，不同"专家"子网络可以专精不同类型文档（表格、公式、普通文本、手写体等）
- **激活参数小（500M）**：每步推理只激活相关专家，实际计算量小，推理速度快

这是"鱼与熊掌兼得"：知识容量与推理效率解耦。对端侧部署意义重大——500M 激活参数可以在消费级 GPU 甚至高端手机 NPU 上流畅运行，同时 3B 总参数保证能力不会太弱。

#### 2. 专用模型与通用大模型的分化共存

Unlimited-OCR 的成功预示着**分化共存**的产业格局：

| 维度 | 通用大模型 | 专用小模型（+机制创新） |
|---|---|---|
| **目标** | 通用性、跨领域能力 | 垂直领域 SOTA、极致效率 |
| **参数规模** | 几十 B 到几百 B 起步 | 几 B 总参数，几百 M 激活 |
| **训练成本** | 数千万到数亿美元 | 基于基座继续训练几千步 |
| **推理成本** | 高，长序列下成本急剧上升 | 低，速度恒定，长序列不减速 |
| **部署方式** | 云端 API 为主 | 端侧部署、边缘计算、本地服务 |

两者是互补关系而非替代关系：通用大模型作为"中央大脑"处理开放性、创造性任务；专用小模型作为"外围器官"处理高频、标准化、对延迟和成本敏感的垂直任务。这类似于计算机系统的存储分层——CPU 缓存、内存、硬盘各有分工。

#### 3. "继续训练 4000 步"的创新路径

Unlimited-OCR 基于 DeepSeek-OCR "稍微继续训练了 4000 步左右"就达到 SOTA，这揭示了高效创新路径：
1. **不要重复造轮子**：继承成熟基座的视觉编码器、基础能力、训练框架
2. **创新聚焦在关键差异点**：4000 步训练核心是让模型适应 R-SWA 机制，而非重新学习 OCR
3. **小步快跑验证假设**：不需要万亿 token 预训练、不需要万卡集群，几张卡几天就能验证架构创新

重大创新不一定需要巨大资源。当你找到正确的机制改进方向时，基于现有成熟基座做"架构层面的外科手术"，可能比从零开始预训练大模型更高效。

### 三、R-SWA 可迁移性分析

R-SWA 的核心思想（参考侧全可见+输出侧滑窗+静态/动态信息分区）具有很强普适性，适用于所有"有明确静态参考、逐 token 输出、以转录/复述/基于参考回答为主"的任务。

| 场景 | 适配度 | 核心改造点 | 预期收益 |
|---|---|---|---|
| OCR/文档转录 | ⭐⭐⭐⭐⭐ | 直接使用 | 无限长度、恒定速度、SOTA 精度 |
| 代码生成 | ⭐⭐⭐⭐ | 多粒度参考、动态窗口、锚点保留 | 无限项目上下文、快速代码补全 |
| 客服对话 | ⭐⭐⭐⭐ | 状态槽位硬保留、知识库全可见 | 无限对话轮次、稳定响应速度 |
| 长文档摘要 | ⭐⭐⭐ | 分层参考、全局语义向量 | 支持超长文档、速度恒定 |
| 开放域创作 | ⭐⭐ | 需要大幅改造 | 不适用 |

**代码生成场景**：代码生成与 OCR 高度相似——有明确参考（已有代码、API 文档）、逐 token 输出、局部连贯要求。改造方向：参考侧扩展为多粒度（代码+符号表+类型信息）、输出侧窗口动态调整（函数定义时扩大窗口）、增加"回溯锚点"机制（函数名、类名等关键锚点不参与滑窗淘汰）。预期收益：支持"无限长度"代码库上下文，打开几十个文件的项目速度不下降。

**客服对话场景**：客服本质是"基于知识库的问答+多轮对话管理"。改造方向：知识库作为参考侧全可见、对话历史滑窗保留、关键状态（用户ID、订单号、核心问题）作为"特殊 token"硬保留。预期收益：支持"无限长度"对话历史，不会因为轮次太多"忘记"关键信息或变慢。

### 四、对 SpecWeave 的可行动启示

R-SWA 和 Unlimited-OCR 的技术思想可以直接应用于 SpecWeave 多智能体协作系统，解决长文档处理、上下文管理、智能体效率等实际问题。以下提炼两条可立即落地的具体建议：

#### 启示 1：实现"规范前置+对话滑窗"的上下文管理中间件

**问题背景**：多轮对话中，随着任务推进对话历史越来越长，导致：上下文窗口溢出（早期关键信息被截断）、注意力分散（无关历史干扰）、推理变慢（上下文越长延迟越高）。

**具体做法**：
1. 在多智能体系统中增加上下文管理层，对每次发给 LLM 的 prompt 进行结构化组装
2. Prompt 结构固定分为三个区：
   - **System 区（静态参考）**：全局核心规则、当前任务相关规范、角色定义——任务启动时确定，期间不变化，始终完整放在 prompt 开头
   - **Anchor 区（关键状态）**：已确认需求、已做出决策、当前任务进度、关键路径和文件——动态更新但每轮完整保留
   - **History 区（滑窗对话）**：最近 20-30 轮对话历史，更早对话不放入主上下文
3. 被滑窗移出的对话历史存入日志，需要时（如用户提到"之前我们说过..."）通过检索重新调入
4. 每轮对话后，用规则或小模型判断是否有需要提取为 Anchor 的关键信息，有则更新 Anchor 区

**预期收益**：
- 核心规范永不遗忘——彻底解决"智能体跳过启动协议"、"不遵守规范"的问题
- 上下文长度恒定——不会因为对话轮次太多导致溢出或速度下降
- 注意力聚焦——智能体始终聚焦当前任务和关键状态，不被无关历史干扰
- 实现成本低——不需要训练新模型，只是 prompt 工程层面的改造

#### 启示 2：建立"文档编码器+按需检索"的长文档处理机制

**问题背景**：SpecWeave 需要处理大量规范文档（.agents/ 下的规则、协议、模板）、大型代码库，当前将完整文件塞入上下文导致：上下文装不下、早期规范被遗忘、处理速度随文档变大变慢。

**具体做法**：
1. 对于规范文档、知识库、大型代码库，不要每次都把完整文件塞进上下文
2. 预处理阶段（类似 DeepEncoder 的一次性编码）：
   - 将长文档按章节/主题拆分为有意义片段
   - 对每个片段生成摘要、提取关键词、建立向量索引
   - 对代码库建立符号表（类、函数、变量定义位置）和调用关系图
3. 执行阶段：
   - 初始上下文只加载最核心的入口文档（如 AGENTS.md、任务 spec）
   - 智能体需要具体规范/代码时，通过"查阅文档"工具检索对应片段
   - 检索到的片段作为临时参考加入当前轮上下文，使用后下一轮可丢弃（除非被提升为 Anchor）
4. 建立"规范路由表"：智能体启动时根据任务类型，从路由表中知道应该加载哪些核心规范，其他规范按需检索

**预期收益**：
- 支持任意大小的代码库和文档集——不再受上下文窗口限制
- 核心信息加载快——初始 prompt 短小精悍，首 token 延迟低
- 信息相关性高——每轮只引入和当前步骤相关的文档片段，避免信息过载
- 可扩展性好——新增规范/代码不需要改 prompt 模板，只需要更新索引
