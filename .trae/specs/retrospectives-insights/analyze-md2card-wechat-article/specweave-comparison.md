# md2card 与 SpecWeave 体系对照分析

## 分析概述

本报告从4个维度对照分析 md2card 独立开发案例中的方法论与 SpecWeave 现有规范体系，提炼可借鉴点与需警惕点，为 SpecWeave 的演进提供外部参考。分析严格结合 SpecWeave 现有具体文件与规范，避免空谈。

---

## 维度一：AI辅助开发工作流对照

### md2card 做法

md2card 采用 **Codex Image2Code 三步工作流**：
1. 通过 Codex Image2 生成设计稿，建立视觉基准
2. 根据设计稿生成 HTML（设计稿先行比直接写 HTML 效果好很多）
3. 基于 HTML 生成可交互组件，逐步集成到产品中

核心特征：**可视化设计先行、渐进式生成、对话式纠偏**。开发者通过与 AI 多轮对话调整设计，每一步都有明确的输入输出参照，最终一个周末完成 MVP。

### SpecWeave 现状

SpecWeave 采用 **Skill 体系与规范驱动开发**，与 md2card 的自由式 AI 开发形成鲜明对比：

| 对比项 | md2card | SpecWeave |
|---|---|---|
| AI 工作模式 | 对话式自由探索，Image→HTML→组件渐进生成 | Skill 封装标准化能力，按规范调用 |
| 知识传递 | 通过多轮对话纠偏，经验存在于对话历史中 | 通过 `.agents/` 规范体系显式传递知识（启动协议、上下文路由、角色定义等） |
| 质量保障 | 依赖开发者个人判断和人工审查 | 阶段守卫、CI 检查、代码审查等多层机制 |
| 复用方式 | 前作代码复用（MDXNotes → md2card） | Skill 复用 + `.agents/scripts/lib/` 共享工具库 + 可复用模式库 |

SpecWeave 的关键规范依据：
- **启动协议**（[AGENTS.md:3-21](AGENTS.md#L3-L21)）：强制要求读取规范后再加载 Skill 或生成产出物，禁止跳过
- **上下文路由表**（[.agents/context-routing.md](.agents/context-routing.md)）：任务类型→必读规范映射，防范"就近直觉"偏差
- **Skill 创建规范**：需读取 vendor 方法论资产 + SpecWeave 补充规范（五要素模型、双方案模式等）
- **全局核心规则**（[.agents/global-core-rules.md:13](.agents/global-core-rules.md#L13)）：启动协议优先，违反导致非线性返工成本

### 可借鉴点

1. **原型快速迭代的"设计稿先行"理念**：md2card 的"设计稿→HTML→组件"渐进式生成，比直接让 AI 从零写代码质量显著提升。SpecWeave 在前端类 Skill 开发（如 `frontend-design`、`web-dev`）中可以引入类似的"视觉基准先行"原则——先生成设计稿/Mockup 作为参照，再基于参照生成代码，而非直接生成最终代码。

2. **对话式纠偏与显式规范的互补**：md2card 证明在快速探索阶段，多轮对话纠偏是高效的；SpecWeave 的显式规范在稳定期和团队协作中不可替代。两者不是对立关系——**Skill 内部可以封装"渐进式生成"流程，而 Skill 的触发条件、输入输出格式由显式规范定义**。

3. **每步有明确输入输出的渐进式生成优于一次性生成**：md2card 三步法的核心洞察是"分三步生成比一步到位质量高、错误少、易调试"。这与 SpecWeave 的三阶段递进原则（[.agents/global-core-rules.md:22](.agents/global-core-rules.md#L22)）——治理、知识库、抽象均遵循三阶段递进，顺序不可颠倒——在哲学层面高度一致。可以将这一原则进一步应用到 Skill 的执行流程设计中。

### 需警惕点

1. **完全依赖对话式纠偏无法规模化**：md2card 是单人周末项目，对话式纠偏依赖开发者个人记忆和判断力。SpecWeave 作为多智能体协作体系，如果只靠对话纠偏而无显式规范，会导致：
   - 不同会话/不同智能体产出质量不一致
   - 上下文压缩后经验丢失（AGENTS.md 步骤 2.2 已明确指出此风险）
   - 无法进行自动化质量门禁

2. **"设计稿先行"不等于"无规范先行"**：md2card 的快速迭代建立在开发者已有 MDXNotes 技术积累和明确的产品定位之上，并非无方向的探索。SpecWeave 的启动协议要求"先读规范再动手"，看似慢实则快——AGENTS.md 明确警告："跳过5分钟的规范读取可能导致30分钟以上的重构返工"（[AGENTS.md:21](AGENTS.md#L21)）。

3. **AI 生成代码必须有人工审查机制**：md2card 方法论也明确指出"不要完全依赖AI生成，生成的代码必须人工审查"。SpecWeave 现有 CI 检查（[.agents/context-routing.md:58](.agents/context-routing.md#L58)）、代码审查、阶段守卫等机制正是对这一风险的制度化应对。

## 维度二：快速原型与阶段守卫平衡对照

### md2card 做法

md2card 采用 **周末 MVP 验证法**：
- 核心功能裁剪：只做"Markdown→知识卡片"一件事，砍掉一切非必要功能
- 速度优先于完美：MVP 不需要好看，只需要能用、能解决核心问题
- 一个周末完成开发，发布到 V2EX 获取真实用户反馈
- 但核心体验不能崩：提前解决了 API 存图和服务器动态扩缩容问题

核心哲学：**快速验证需求，但核心功能必须可用**。

### SpecWeave 现状

SpecWeave 建立了严格的 **阶段守卫体系**，同时通过 `.temp/` 目录机制提供快速原型空间：

| 对比项 | md2card | SpecWeave |
|---|---|---|
| 开发速度 | 周末 MVP，速度优先 | 8阶段标准流程，质量优先 |
| 原型区域 | 无明确区分，直接开发上线 | `.temp/` 暂存开发 → `apps/` 稳定迁移（[.agents/protocols/app-development-workflow.md](.agents/protocols/app-development-workflow.md)） |
| 质量门禁 | 核心功能可用即可，无正式审查 | 全量迁移需满足4项条件：测试100%通过、代码审查通过、无P0/P1缺陷、文档完善 |
| 阶段拦截 | 无显式拦截机制，靠开发者自律 | 跨阶段拦截机制 + 阶段跳转审批流程（[.agents/rules/stage-guardrails.md](.agents/rules/stage-guardrails.md)） |
| 快速通道 | 默认快速，质量后续迭代 | L0 探针豁免规则（适用于探索性任务） |

SpecWeave 的关键规范依据：
- **阶段守卫规则**（[.agents/rules/stage-guardrails.md:7-19](.agents/rules/stage-guardrails.md#L7-L19)）：定义标准8阶段序列、各阶段操作边界、跨阶段拦截与跳转审批
- **应用开发生命周期**（[.agents/protocols/app-development-workflow.md:58-98](.agents/protocols/app-development-workflow.md#L58-L98)）：`.temp/` 暂存阶段允许频繁修改、不要求代码审查，聚焦快速迭代与功能验证
- **L0 探针豁免规则**（[.agents/rules/stage-guardrails.md:27](.agents/rules/stage-guardrails.md#L27)）：为探索性任务提供规范化的快速通道
- **迁移条件**（[.agents/protocols/app-development-workflow.md:112-119](.agents/protocols/app-development-workflow.md#L112-L119)）：全量迁移必须满足测试通过、审查通过、无阻塞缺陷、文档完善4项条件

### 可借鉴点

1. **`.temp/` 快速原型 → 验证通过再迁移到 `apps/` 的工作流正是 md2card 模式的制度化体现**：SpecWeave 已经有了这个机制，但需要强化其"快速验证"定位。md2card 的经验表明，`.temp/` 阶段应该：
   - 聚焦单一核心痛点验证，不追求功能完整
   - 允许"不完美但能用"的代码快速上线获取反馈
   - 但核心体验（如 md2card 的 API 存图和服务器扩容）必须提前预案

2. **提炼"快速验证但保留质量框架"的平衡点**：md2card 的"速度优先"不等于"无质量底线"——它提前解决了服务器扩容问题，避免首发即崩溃。对应到 SpecWeave：
   - `.temp/` 阶段可以跳过代码审查、测试覆盖率等"增量质量"要求
   - 但不能跳过"核心功能可用"、"无致命缺陷"等"底线质量"要求
   - 这正是 L0 探针豁免规则的设计哲学，但可以更明确地定义"底线质量"清单

3. **功能裁剪的艺术**：md2card 只做"Markdown→知识卡片"一件事，其他功能（编辑、模板、协作等）全部砍掉。SpecWeave 在新功能开发中可以借鉴：
   - `.temp/` 阶段的新 Skill/新工具只实现最小核心功能
   - 不提前设计"未来可能需要"的抽象层（方法论6也警告"不要过度设计复用层"）
   - 验证通过后再考虑扩展和规范化

### 需警惕点

1. **过度规范化会扼杀探索速度**：如果 `.temp/` 阶段也强制要求完整的规范读取、文档编写、测试覆盖，就失去了"快速原型"的意义。md2card 一个周末完成 MVP 的关键是"只做核心的事"。SpecWeave 需要确保：
   - `.temp/` 阶段的门禁确实比 `apps/` 宽松
   - L0 探针豁免机制在实践中不被过度限制
   - 启动协议的"按需读取"原则（[.agents/global-core-rules.md:15](.agents/global-core-rules.md#L15)）得到贯彻——只读取与当前任务直接相关的规范，避免一次性加载全部上下文

2. **"周末MVP"不适合所有场景**：md2card 方法论明确指出"复杂度极高、需要强合规、重资产的产品不适用"。对应到 SpecWeave：
   - 核心规则体系（如 AGENTS.md、阶段守卫、数据安全）的修改不适用快速原型模式
   - 新 Skill 探索、新工具原型、UI 组件开发适合 `.temp/` 快速验证
   - 需要根据任务类型分级处理，而非一刀切

3. **快速原型不等于无架构设计**：md2card 复用了 MDXNotes 的 Markdown 编译能力，说明即使在快速开发中，核心能力的模块化和复用意识仍然重要。SpecWeave 的 `.temp/` 开发虽然可以快速迭代，但应鼓励：
   - 有意识地识别可复用的核心能力
   - 避免硬编码路径和紧耦合（硬编码治理规则已有明确要求）
   - 为后续可能的迁移保留清晰的边界

## 维度三：用户验证与规范驱动对照

### md2card 做法

md2card 采用 **真实用户反馈优先** 的方法论：
- "需要有真正的、真实的一批用户来体验产品"
- "不要完全独立一个人去判断方向，一旦方向错了，可能白白浪费几个月甚至更久的时间"
- 快速做出 MVP → 发布到 V2EX → 通过用户实际痛点迭代产品
- 月活 1.5 万中大部分为用户自主访问，验证了用户驱动的有效性

核心哲学：**用户反馈是方向判断的唯一标准，避免闭门造车**。

### SpecWeave 现状

SpecWeave 采用 **规范前置** 的方法论，两者形成"探索期vs稳定期"的典型对照：

| 对比项 | md2card | SpecWeave |
|---|---|---|
| 方向判断依据 | 真实用户行为和反馈 | 规范体系定义的最佳实践和流程 |
| 知识获取方式 | 发布产品→收集反馈→迭代 | 启动协议→读取规范→按规范执行 |
| 风险应对 | 快速试错，错了就改 | 前置预防，通过规范避免犯错 |
| 适用阶段 | 从0到1探索期，方向不确定 | 稳定协作期，方向已明确需保证一致性 |

SpecWeave 的关键规范依据：
- **启动协议强制读取规范**（[AGENTS.md:3-21](AGENTS.md#L3-L21)）：在执行任务前必须读取对应规范，"凭经验做对不等于按方法论做对"
- **前置文档强制读取协议**（[.agents/context-routing.md:82](.agents/context-routing.md#L82)）：必读清单/确认机制/PDR-LOG结构化日志
- **查阅知识库规则**（[.agents/global-core-rules.md:25](.agents/global-core-rules.md#L25)）：执行任务前主动查阅技术知识库与复盘文档，了解已有经验，避免重复踩坑
- **歧义主动澄清**（[.agents/global-core-rules.md:19](.agents/global-core-rules.md#L19)）：遇到需求不明确必须先提问澄清，禁止自行猜测意图
- **复盘体系**（[AGENTS.md:57-65](AGENTS.md#L57-L65)）：通过复盘沉淀可复用模式，形成组织记忆

### 可借鉴点

1. **新工具/新功能可采用 MVP→用户反馈→再规范化的路径**：SpecWeave 现有体系在"从零创建新东西"时偏重规范前置，可以引入 md2card 的验证逻辑：
   - **新 Skill 开发**：先在 `.temp/` 或实验环境快速实现原型→在实际任务中试用→收集使用反馈→再规范化（编写 SKILL.md、添加验证脚本、纳入上下文路由）
   - **新规范制定**：先通过几个实际案例验证方法论有效性→再沉淀为正式规范→而非一开始就试图定义完美规范
   - 这与三阶段递进原则（[.agents/global-core-rules.md:22](.agents/global-core-rules.md#L22)）中"治理（修复→预防→闭环）"的路径一致

2. **建立"内部用户反馈"机制**：md2card 通过 V2EX 获取外部用户反馈，SpecWeave 作为内部工具体系，可以建立：
   - Skill 使用效果追踪（哪些 Skill 经常被调用、哪些经常出错）
   - 规范违反案例收集（阶段守卫日志 SG-LOG 已经在做这件事）
   - 智能体间的"用户反馈"——某个 Skill 不好用，使用方应能提出改进建议
   - 定期复盘会议/报告，使用实际数据而非主观判断来优化规范

3. **区分"探索期规范"和"稳定期规范"**：md2card 的经验表明，不同阶段需要不同的方法论：
   - **探索期**（新工具/新方向）：轻规范、重验证、允许快速试错
   - **稳定期**（核心流程/高频使用）：重规范、强门禁、保证一致性
   - SpecWeave 的 L0-L3 流程分级模板（[.agents/rules/stage-guardrails.md:27](.agents/rules/stage-guardrails.md#L27)）已经有这个思路，可以进一步强化并明确各分级的适用场景

### 需警惕点

1. **"用户反馈优先"不能成为不读规范的借口**：md2card 是面向外部用户的产品，用户反馈直接对应市场需求；SpecWeave 的"用户"是 AI 智能体和开发者，规范本身就是大量"用户反馈"（复盘、踩坑经验）的沉淀。跳过规范直接探索，等于无视之前积累的所有经验教训——AGENTS.md 的警告"跳过5分钟规范读取导致30分钟返工"正是此意。

2. **内部工具的"用户反馈"有其特殊性**：md2card 的用户反馈是"用脚投票"（访问量、月活），而 SpecWeave 的"用户"（智能体）可能无法像真实用户那样给出高质量反馈——它们会按照规范执行，即使规范有问题也不会主动抱怨。因此：
   - 不能完全依赖"智能体反馈"来判断规范好坏
   - 需要通过日志分析（SG-LOG/PDR-LOG/CMD-LOG）、任务成功率、返工率等客观指标来评估
   - 人类开发者的复盘和洞察仍然是规范优化的关键输入

3. **规范前置与用户验证不是非此即彼**：md2card 虽然快速迭代，但并非无方向的乱试——它建立在对 AI 时代痛点的深刻理解之上。SpecWeave 的规范前置也是如此：
   - 规范不是拍脑袋制定的，而是来自复盘和实践验证
   - 规范本身也需要接受实践检验并持续迭代
   - 闭环路径：实践→复盘→规范→再实践→再复盘→优化规范
   - 这正是"修复即闭环"原则（[.agents/global-core-rules.md:24](.agents/global-core-rules.md#L24)）的体现

## 维度四：内容优先与文档媒介选择对照

### md2card 做法

md2card 采用 **内容优先于模板** 的产品设计哲学：
- 与 Canva"先选模板再填内容"相反，md2card 是"用户已有 Markdown 内容→一键生成卡片"
- 复杂的排版、适配、拆分等工作全部自动化，用户只需"粘贴→点击"
- 内容是起点而非终点，产品自动处理形式
- 多平台适配自动完成（小红书3:4、公众号横图等）

核心哲学：**结构化但不束缚创造力，形式服务于内容**。

### SpecWeave 现状

SpecWeave 采用 **spec/tasks/checklist 结构化文档体系**，与 md2card 的"内容优先"形成有趣的对照：

| 对比项 | md2card | SpecWeave |
|---|---|---|
| 文档哲学 | 内容先于形式，用户提供内容，工具自动处理形式 | 形式与内容并重，通过结构化模板保证文档质量和一致性 |
| 模板使用 | 反模板——不先选模板，直接处理内容 | 有明确模板——任务模板、交接模板、Mermaid模板、复盘模板等 |
| 自动化程度 | 一键转换，零配置 | 部分自动化（docgen 生成导航/看板、链接检查自动修复），但大量内容需人工按规范编写 |
| 文档边界 | 无明确边界，输出即最终产物 | 明确区分 AI 面向文档（`.agents/`）和人类面向文档（`docs/`、`README.md`） |

SpecWeave 的关键规范依据：
- **派生产物溯源**（[AGENTS.md:54](AGENTS.md#L54)）：派生产物须在 TOML frontmatter 携带 `source` 字段标注来源
- **文档边界分离**（[AGENTS.md:53](AGENTS.md#L53)）：`AGENTS.md`/`.agents/` 面向 AI 智能体，`README.md`/`docs/` 面向人类读者，职责分离
- **原子化操作**（[.agents/context-routing.md:53](.agents/context-routing.md#L53)）：原子化收尾一键处理断链修复、导航更新、看板刷新
- **文档索引自动化**（[.agents/context-routing.md:52](.agents/context-routing.md#L52)）：docgen 自动生成导航/看板/应用清单，标记区域幂等覆盖
- **元文档优先原则**（[.agents/global-core-rules.md:23](.agents/global-core-rules.md#L23)）：优先优化入口文档、索引等元文档，ROI最高

### 可借鉴点

1. **工具输出优先保证内容质量，格式/排版自动化处理**：md2card 的核心洞察是"用户关心的是内容能否快速分享出去，而不是调格式"。对应到 SpecWeave：
   - 智能体产出文档时，应先保证内容（分析、决策、代码）质量，格式合规性尽量自动化
   - 现有 docgen 生成导航/看板、link-check 自动修复路径已经是这个方向
   - 可以进一步：自动生成/校验 frontmatter、自动检查文档结构是否符合模板、自动格式化 Mermaid 图表
   - 让规范检查工具（如 `check-stage-guardrails.py`、`check-spec-consistency.py`）承担更多"形式自动化"工作，减少人工对格式的关注

2. **提炼"结构化但不束缚创造力"的文档哲学**：md2card 的"内容优先"不是"无结构"——Markdown 本身就是结构化的，只是不强制用户选模板。SpecWeave 的结构化文档体系可以借鉴：
   - 模板提供"骨架"但不强制"血肉"——必填项保证最低结构质量，可选项给予灵活空间
   - L0/L1/L2 三层架构（[.agents/context-routing.md:26](.agents/context-routing.md#L26)）正是这一理念的体现：L0 ONBOARDING <100行快速上手，L1 SKILL+REGISTRY <500行标准文档，L2 深度文档不限
   - 可以进一步明确：哪些文档部分是"必须结构化"的（如 frontmatter、章节标题、元数据），哪些是"自由发挥"的（如分析内容、经验描述、代码实现）

3. **"内容即营销"理念在文档体系中的应用**：md2card 通过"用户用产品制作的内容本身就是活广告"实现零成本获客。对应到 SpecWeave：
   - 好的文档本身就是最好的推广——一份结构清晰、内容扎实的 spec 或复盘报告，比任何"如何使用"的指南都更有说服力
   - `.trae/specs/` 下的实际案例（如本次 md2card 分析系列）就是最好的"使用示范"
   - 可以在文档索引中加入"优秀案例"或"推荐阅读"，通过高质量内容引导规范采纳

### 需警惕点

1. **过度结构化确实可能束缚创造力**：md2card 反对"先选模板再填内容"有其道理——如果模板过于僵化，创作者会花大量时间"填模板"而非"思考内容"。SpecWeave 需要警惕：
   - 模板过于复杂导致智能体花更多精力"符合格式"而非"解决问题"
   - 检查清单过长导致执行成本过高，反而诱发绕过规范的行为
   - 元文档优先原则（[.agents/global-core-rules.md:23](.agents/global-core-rules.md#L23)）已经意识到入口文档要精简（>100行优先精简），可以将这一原则扩展到所有模板设计

2. **"一键自动化"有其适用边界**：md2card 能一键转换是因为 Markdown→卡片的转换规则相对明确。SpecWeave 的很多文档（如复盘报告、技术方案、角色定义）涉及深度思考和创造性分析，无法完全自动化：
   - 可以自动化的：格式校验、链接检查、索引生成、frontmatter 生成、目录结构检查
   - 不能自动化的：核心洞察提炼、架构决策、经验总结、创意设计
   - 工具应该自动化"机械性工作"，将人的精力释放到"创造性工作"上

3. **"内容优先"不等于"可以忽略形式"**：md2card 虽然内容优先，但排版质量必须达标（"自动排版质量要达标，否则用户宁愿手动调整"）。SpecWeave 的结构化文档同样如此：
   - 格式规范的存在是为了提升可读性、可维护性、可检索性，而非为了规范而规范
   - 当格式要求与内容表达冲突时，应该有机制允许例外（如豁免审批流程）
   - 但例外必须被记录和追踪，而非默默绕过——这正是阶段守卫的拦截与审批机制的价值

## 综合启示

### 两种方法论的本质：控制论光谱上的不同位置

md2card 与 SpecWeave 的差异，本质上是**探索-利用频谱**（exploration-exploitation spectrum）上的不同位置：

| 位置 | 方法论 | 核心特征 | 适用场景 | SpecWeave对应机制 |
|---|---|---|---|---|
| 探索端 | md2card模式 | 快速试错、用户反馈、最小成本验证 | 新方向、新工具、不确定领域 | `.temp/`暂存开发、L0探针豁免 |
| 中间 | 平衡模式 | 快速验证但保留质量框架、MVP→反馈→规范化 | 新功能开发、现有体系扩展 | 三阶段递进、修复即闭环 |
| 利用端 | SpecWeave规范模式 | 规范前置、质量门禁、自动化检查 | 核心流程、高频协作、稳定模块 | 阶段守卫、CI检查、Skill体系 |

### 给 SpecWeave 的5条具体建议

1. **强化 `.temp/` 的"快速验证"定位**：明确 `.temp/` 阶段的"底线质量"清单（核心功能可用+无致命缺陷）和"增量质量"豁免项（代码审查/测试覆盖率/文档完善），让快速原型真正快起来。

2. **新 Skill/新工具采用"原型→试用→规范化"路径**：不要一开始就要求完美的 SKILL.md 和完整的验证脚本，先做核心能力原型→在实际任务中试用3-5次→根据使用反馈再规范化。这与 md2card"先上线再迭代"的哲学一致。

3. **扩大自动化格式处理的范围**：将更多"机械性工作"（frontmatter校验、结构检查、格式修复）自动化，让智能体和开发者将精力集中在内容质量上。借鉴 md2card"粘贴→点击→完成"的体验。

4. **建立基于使用数据的规范优化闭环**：通过 SG-LOG/PDR-LOG/CMD-LOG 等结构化日志，分析哪些规范经常被违反、哪些检查经常失败、哪些 Skill 经常出错——用数据而非主观判断来优化规范，借鉴 md2card"用户行为说话"的理念。

5. **明确文档模板的"必填结构"和"自由区域"**：模板定义最低结构要求（frontmatter、核心章节标题），但给予内容表达充分的自由度。借鉴 md2card"结构（Markdown）+自由（内容）"的平衡，以及 L0/L1/L2 三层架构的渐进式披露理念。

### 最终结论

md2card 的成功不是"无规范"的成功，而是**"在正确的时机用正确的方法论"的成功**——一个周末 MVP、快速用户验证、内容优先设计，这些方法论在从0到1探索期极为有效；但当产品进入稳定维护期、团队规模扩大、协作复杂度提升时，SpecWeave 式的规范驱动、阶段守卫、质量门禁就成为必要。

**对 SpecWeave 最大的启示不是"放松规范"，而是"让规范更聪明"**——在需要探索的地方提供足够的自由度（`.temp/`、L0豁免、渐进式披露），在需要稳定的地方坚守质量底线（阶段守卫、CI检查、测试要求），并通过自动化和数据闭环让规范本身持续进化。这正是 md2card"内容优先、形式自动化"哲学在方法论层面的体现：**好的方法论应该像 md2card 一样——你专注于核心内容（解决问题），它自动处理形式（流程合规）。**
