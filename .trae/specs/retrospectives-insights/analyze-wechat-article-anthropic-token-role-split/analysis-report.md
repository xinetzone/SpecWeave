## 四、病毒式说法与四角色框架

### 4.1 @0xCodila 推文传播数据

- **发布时间**：当地时间 7 月 2 日晚上 10 点 09 分
- **署名来源**：声称来自"Anthropic 产品团队"
- **传播数据**：232 赞、458 收藏、3.8 万次查看
- **核心主张**：同样数量的 token，换一种分配方式，准确率从 15% 跳到 90%，"同样成本，效果翻 5 倍"

推文原文为："instead of one AI doing everything - split it into four: one works, one checks, one scores, one learns - same cost, but 5x better"（把一个 AI 要干的所有事拆成四份：一个执行、一个检查、一个打分、一个学习，成本没变，效果翻了 5 倍）。

### 4.2 四角色框架详解

@0xCodila 将这套说法总结成一张清单，四个角色，四个动作：

| 角色 | 英文原义 | 中文释义 | 职责定位 |
|---|---|---|---|
| execute（执行者） | do the work | 负责把活干完 | 专注产出，不被审查与复盘分散注意力 |
| advise（顾问） | check the direction | 负责把关方向对不对 | 在执行者偏航时提供方向修正 |
| grade（评分官） | pass or fail against a rubric | 按一份评分标准判断通过还是失败 | 用明确标准拦截幻觉与低质量输出 |
| dream（复盘者） | inspect, learn, write to memory, sharpen next round | 检查、学习、写进记忆、给下一轮做准备 | 沉淀经验，为下一轮迭代提供改进方向 |

文章用通俗语言解释了这一框架的逻辑：一个模型如果既要写方案、又要自己审、还要自己打分、自己总结教训，四件事全挤在一个脑子里，难免顾此失彼。拆开之后，干活的专心干活，把关的专心挑毛病，打分的照标准来，复盘的负责把经验存下来，就像一个正常的团队分工，谁都不用一心多用。

### 4.3 衍生版本与"可套用模板"特征

文章指出，同一个创作者圈子里还流传着另一个版本：

- **主语变化**：从"Anthropic 产品团队"变为"Anthropic 平台工程负责人"
- **起点变化**：从 15% 变为 42%
- **预算新增**：增加了 600000 token 的具体预算
- **角色框架不变**：execute/advise/grade/dream 四角色框架完全保留

文章判断："这不太像一次转述出了偏差，更像一套可以反复套用的模板——换个头衔，换一组数字，同一个故事可以讲很多遍。" 这一识别对判断病毒式传播话术的特征具有重要价值。

### 5.1 关键词逐字核对方法

3. **工程博客**：Anthropic 官方工程博客，包括《How we built our multi-agent research system》

### 5.2 "查无实据"判定依据

1. Anthropic 从未在任何公开材料中把 token 工作拆成 execute、advise、grade、dream 这套说法

## 六、官方证据链

### 6.1 Claude Code 官方定位与真实案例

Claude Code 的官方定位是"an agentic coding system that reads your codebase, makes changes across files, runs tests, and delivers committed code"（一套能读懂代码库、跨文件修改代码、跑测试、交付已提交代码的智能编码系统）。

官方给出的三个真实案例：

| 公司 | 应用场景 | 效果 |
|---|---|---|
| Stripe | 1 万行 Scala 代码迁移到 Java | 压缩到 4 天完成 |
| Ramp | 故障排查 | 时间减少 80% |
| Rakuten | 新功能交付周期 | 从 24 个工作日缩短至 5 天 |

这三个案例虽未直接对应"四角色框架"，但证明了 Claude Code 作为多智能体编码系统的实际工程价值。

### 6.2 Agent Teams（智能体团队）

**设计机制**：一个 lead session（主线程）负责协调，底下带着几个 teammate（队友），每个队友都有自己独立的 context window（上下文窗口），彼此之间还能互相发消息、共享任务清单。

**推荐用法**：一个盯用户体验，一个啃技术架构，还专门留一个唱反调的，当 devil's advocate（魔鬼代言人）。

**对应病毒式说法中的角色**：Agent Teams 直接对应"分工"角色——将不同职责分配给不同的智能体，每个智能体在独立上下文中专注自己的任务。

> 文档出处：Claude Code 文档"Orchestrate teams of Claude Code sessions"，强调"each in its own context window"。

### 6.3 Subagents（子智能体）

**设计机制**：当一个任务会把主对话灌满搜索结果、日志、大段文件内容，而这些内容之后又用不上时，就该交给子智能体单独处理，只把结论带回来。用户可以自己写一份带 YAML frontmatter 的配置文件，定义名字、描述、允许用的工具、模型、权限。

**内置示例**：code-reviewer（代码评审员）、security-reviewer（安全评审员）、debugger（调试员）。

**对应病毒式说法中的角色**：Subagents 本质上正是"grade"和"advise"两个角色的落地方式——code-reviewer 和 security-reviewer 对应"grade"（按标准判断通过或失败），debugger 对应"advise"（把关方向与质量）。

> 文档出处：Create custom subagents 文档，独立上下文加可复用角色配置。

### 6.4 /goal 循环

**设计机制**：设定一个完成条件，Claude 会持续干活，直到条件满足为止。每一轮结束，都有一个更小更快的模型检查条件是否成立，不成立就继续下一轮，不用人一直盯着。

**官方强调**："少了这道检查，循环就没有意义。一个模型自己给自己打勾，几乎永远都是通过。"

**对应病毒式说法中的角色**：/goal 循环对应"dream"角色——持续检查、学习、为下一轮做准备。小快模型的检查机制正是"grade"功能的体现，而循环本身带有"dream"的迭代进化特征。

> 文档出处：Keep Claude working toward a goal 文档，核心在于"a small fast model checks whether the condition holds"。

### 6.5 工程博客《How we built our multi-agent research system》

**发布时间**：2025 年 6 月 13 日

**核心引述**："multi-agent systems work mainly because they help spend enough tokens to solve the problem."（多智能体系统之所以有效，主要原因是它们能帮你把足够多的 token，花在解决问题的关键处。）

**内部评测数据**：用 Claude Opus 4 当 Lead（主控），Sonnet 4 当 Subagents（子智能体）并行工作，这套多智能体系统在研究类评测上，比单独一个 Opus 4 模型，效果高出 90.2%。

**token 消耗**：多智能体系统消耗的 token 大约是普通对话的 15 倍。

**关键判断**：15% 到 90% 这组数字查无实据，但 90.2% 这个数字白纸黑字写在 Anthropic 自己的博客里。两者未必是同一件事，方向却完全一致——分工比堆料更管用。

### 6.6 官方实现与病毒式说法的对应关系总结

| 病毒式说法角色 | 官方真实实现 | 对应程度 |
|---|---|---|
| execute（执行者） | Agent Teams 中的 teammate、Subagents 中的执行型子智能体 | 直接对应 |
| advise（顾问） | Subagents 中的 debugger、Agent Teams 中的 devil's advocate | 直接对应 |
| grade（评分官） | Subagents 中的 code-reviewer/security-reviewer、/goal 中的小快模型检查 | 直接对应 |
| dream（复盘者） | /goal 循环的持续迭代、子智能体的独立记忆目录 | 直接对应 |
| 15%→90% 数据 | 90.2% 数据（工程博客） | 方向一致，数值不同 |

## 七、拆分角色有效的五个原因

文章指出，普通人的直觉是"模型越强、token 越多、上下文越长，效果就该越好"，但这套直觉漏掉了几个容易被忽视的坑。拆分角色之所以有效，有五个底层原因：

### 7.1 上下文会被自己的错误弄脏（Context Pollution）

**机理**：一个模型如果同时要想点子、写代码、自我审查、还要复盘，它会把自己的草稿、假设、甚至错误全部留在同一个上下文里，后面的输出很容易被前面踩过的坑锚住。

**工程价值**：拆开角色之后，每个模型只看见跟自己任务相关的那部分，上下文干净得多。这直接对应 Agent Teams 中"each in its own context window"的设计——独立上下文窗口从根本上避免了跨任务的污染。

### 7.2 自己给自己打分，几乎永远合格（Self-Grading 失效）

**机理**：官方反复提到，没有独立的 verifier（验证器），循环就只是自我重复。一个模型自己给自己打勾，几乎永远都是通过。

**工程价值**：grade 这个角色逼着系统按一份明确的标准判断通过或失败，幻觉和低质量输出因此被大幅拦下来。这正是 /goal 循环中"小快模型检查条件"机制存在的根本原因——官方特意强调"少了这道检查，循环就没有意义"。

### 7.3 专业化是人类团队早就验证过的常识

**机理**：前端、后端、测试、产品，分工本身就是效率来源。

**工程价值**：放到模型身上，execute 专注把活干完，advise 专注方向对不对，grade 专注质量够不够格，dream 专注沉淀经验——每个角色的任务窄了，反而做得更精。这一逻辑与 Subagents 的设计理念完全一致：将复杂任务分解为专精的子智能体，每个子智能体只处理特定类型的工作。

### 7.4 并行加独立记忆，才是真正的杠杆

**机理**：官方多智能体系统里，子智能体各自带着独立上下文和记忆目录，可以同时探索不同的假设，再把提炼后的结果交回给主控。

**工程价值**：dream 角色做的，正是"检查、学习、写入记忆"这件事。并行探索能力意味着系统可以同时尝试多条路径，而独立记忆目录确保了经验可以被沉淀和复用，而非在单次会话结束后丢失。

### 7.5 token 花在哪，比花了多少更能决定效果

**机理**：工程博客里提到，很多时候性能差异其实来自有没有把足够的 token 用在正确的地方。

**工程价值**：多智能体表面上烧更多 token（约 15 倍），但换算到每一次成功产出的成本上，反而更低。这一判断颠覆了"省 token 就是好"的朴素认知——关键不在于总消耗量，而在于 token 是否被分配到了最能产生价值的环节。

## 八、历史脉络

文章指出，拆分角色这件事并非 Anthropic 首创，在 Anthropic 之前就已经有人摸索过。

### 8.1 2023 年前后的学术探索

| 方法 | 核心思想 | 对应角色 |
|---|---|---|
| ReAct | 让模型"思考"和"行动"交替进行 | execute 与 advise 交替的雏形 |
| Reflexion | 让模型自己写检讨、自己重写答案 | dream 角色最早的雏形 |
| Self-Refine | 让模型自我改进答案 | dream 角色最早的雏形 |

### 8.2 Karpathy Loop（卡帕西自动研究框架）

@0xCodila 本人几天前刚发过一篇关于"Karpathy Loop"的长文，提到卡帕西（Andrej Karpathy）式的自动研究框架：

| 组件 | 职责 | 对应角色 |
|---|---|---|
| train.py | 负责干活 | execute（执行者） |
| prepare.py | 负责评分（不能被 agent 改动） | grade（评分官） |
| program.md | 负责给方向 | advise（顾问） |

agent 自己做实验、自己回滚、自己记录状态，这一框架已经具备了四角色分工的雏形。

### 8.3 工业界的分工实践

不少公司内部早就把 coding agent 拆成几个固定角色，分头干活：

| 角色 | 职责 | 对应四角色 |
|---|---|---|
| Planner（规划者） | 任务规划与分解 | advise |
| Coder（写代码的） | 代码实现 | execute |
| Tester（测试者） | 质量验证 | grade |

## 十二、范式转变意义

### 12.1 从"堆料式扩规模"到"分工式提效率"的范式转变

文章揭示的 2026 年 AI 工程界真实转向，本质上是一场从"堆料式扩规模"到"分工式提效率"的范式转变。

**堆料式范式的逻辑**：模型越强、token 越多、上下文越长，效果就该越好。这一逻辑在 2023-2025 年大模型发展的早期阶段确实有效——更大的参数量、更长的上下文窗口、更多的训练数据，确实带来了显著的性能提升。

**分工式范式的逻辑**：token 该往哪个角色身上砸，比 token 喂了多少更重要。这一逻辑在 2026 年开始凸显，其背景是大模型能力趋于饱和，单纯堆料带来的边际收益递减，而通过角色分工优化 token 分配，能够在不增加（甚至减少）总预算的前提下获得显著效果提升。

### 12.2 三重意义

这一范式转变具有三重意义：

**技术层意义（多智能体架构创新）**：从单一模型承担所有任务，转向多智能体协作架构。Agent Teams、Subagents、/goal 循环等技术实现，标志着 AI 系统设计从"单体智能"向"群体智能"的演进。

**应用层意义（token 经济学优化）**：从"省 token 就是好"的朴素认知，转向"token 花在哪比花了多少更能决定效果"的精细化经济学。多智能体虽然消耗约 15 倍 token，但每次成功产出的成本反而更低，这一计算方式颠覆了传统的成本评估模型。

**认知层意义（角色分工是放大智能的杠杆）**：从"更大模型等于更强能力"的线性思维，转向"角色分工是放大智能真正的杠杆"的系统思维。这一认知转变意味着，智能的提升不再仅仅依赖模型规模的扩张，而可以通过组织方式的优化来实现。

## 十五、可复用方法论

基于文章分析，提炼以下四个可复用的方法论启示：

### 15.1 角色分工设计原则

**原则**：将一个 AI 要干的所有事拆分为 execute（执行）、advise（顾问）、grade（评分）、dream（复盘）四个角色。

**映射关系**：
- execute → Agent Teams 中的 teammate、Subagents 中的执行型子智能体
- advise → Subagents 中的 debugger、Agent Teams 中的 devil's advocate
- grade → Subagents 中的 code-reviewer/security-reviewer、/goal 中的小快模型检查
- dream → /goal 循环的持续迭代、子智能体的独立记忆目录

**实践要点**：每个角色的任务要窄，任务窄了反而做得更精。角色之间通过明确的输入输出关系协作，而非让一个模型承担所有职责。

### 15.2 verifier 依赖原则

**原则**：独立的 verifier（验证器）是多智能体系统的灵魂，少了 verifier，整个系统就退化成一场更贵的自言自语。

**实践要点**：
- verifier 必须独立于执行者，不能让模型自己给自己打勾
- verifier 需要有明确的评分标准（rubric），而非主观判断
- 可靠的 verifier 形式包括：测试用例、评分标准、人工审核
- /goal 循环中"小快模型检查条件"就是 verifier 机制的官方实现

**边界判断**：如果没有可靠的 verifier，不应盲目采用多智能体架构——此时单一会话可能反而更高效。

### 15.3 独立上下文与记忆原则

**原则**：每个角色应拥有独立的 context window（上下文窗口）和记忆目录，避免上下文污染。

**机理**：一个模型如果同时要想点子、写代码、自我审查、还要复盘，它会把自己的草稿、假设、甚至错误全部留在同一个上下文里，后面的输出很容易被前面踩过的坑锚住。

**实践要点**：
- Agent Teams 中每个 teammate 拥有独立的 context window
- Subagents 只把结论带回来，不污染主对话
- 子智能体各自带着独立上下文和记忆目录，可以同时探索不同的假设
- dream 角色负责"检查、学习、写入记忆"，确保经验可被沉淀和复用

### 15.4 token 分配优化原则

**原则**：token 花在哪，比花了多少更能决定效果。

**依据**：工程博客指出"multi-agent systems work mainly because they help spend enough tokens to solve the problem"——多智能体系统之所以有效，主要原因是它们能帮你把足够多的 token，花在解决问题的关键处。

**实践要点**：
- 多智能体表面烧更多 token（约 15 倍），但每次成功产出的成本反而更低
- 关键不在于总消耗量，而在于 token 是否被分配到了最能产生价值的环节
- 评估多智能体系统的成本时，应以"每次成功产出的成本"为单位，而非总 token 消耗
