# Harness Engineering 七概念分析 — 对抗审查

> **质量门V**: 审查意见≥5条且具体，至少采纳2条修正

---

## 审查意见

### 审查意见1："模型只是引擎，Harness才是整辆车"的类比可能误导对模型重要性的认知

**问题**：文章和洞察将模型比作引擎，Harness比作整辆车，这个类比强调了Harness的重要性，但可能导致过度否定模型能力的价值。在真实场景中，一辆好车需要好引擎和好调校，两者缺一不可。

**证据**：F055提到"模型之间的纯能力差距还在，但正在变得没那么决定性"——这个表述已经承认模型差距存在，只是"没那么决定性"，但类比可能让人忽略这一点。

**建议**：在模式文档中补充"模型与Harness协同进化"的观点，强调两者是互补关系而非替代关系。

**采纳决策**：✅ 采纳——将在模式文档中补充协同进化观点。

---

### 审查意见2：文件系统记忆模式缺乏具体的实现策略和技术细节

**问题**：洞察和模式都强调了文件系统作为持久记忆的重要性，但缺乏具体的实现策略：如何决定什么放进文件？文件格式是什么？如何组织文件目录结构？如何实现跨轮次检索？

**证据**：模式文档中提到了文件系统记忆的原则，但没有给出具体的目录结构模板、文件命名规范、检索策略等实现细节。

**建议**：参考文章中提到的.mini-harness/memory.md模式，在模式文档中补充具体的目录结构模板和实现策略。

**采纳决策**：✅ 采纳——将在模式文档中补充目录结构模板和实现策略。

---

### 审查意见3：子代理职责分层模式的边界定义不够清晰

**问题**：模式中定义了主代理和子代理的职责边界，但没有明确回答"什么时候应该使用子代理？"和"如何决定哪些任务下放给子代理？"这两个关键问题。

**证据**：模式文档列出了六种子代理类型，但没有提供决策框架来判断何时应该引入子代理，以及如何选择合适的子代理类型。

**建议**：增加一个决策矩阵或流程图，帮助用户判断何时使用子代理以及如何选择子代理类型。

**采纳决策**：❌ 暂不采纳——当前模式文档聚焦于职责分层的核心思想，决策框架可以作为后续扩展内容，不影响核心模式的有效性。

---

### 审查意见4：Harness架构分层模式的五层架构与行业标准术语存在差异

**问题**：模式中定义的五层架构（Workflow/Tools/Permissions/Memory/Evaluation）与行业中常见的Agent架构（如Plan/Act/Observe/Reflect）存在术语差异，可能导致理解障碍。

**证据**：文章引用的Lilian Weng原文使用的术语与模式中定义的不完全一致，可能影响模式的可复用性。

**建议**：在模式文档中增加术语映射表，将五层架构与行业标准术语对应起来，降低理解成本。

**采纳决策**：❌ 暂不采纳——当前术语体系基于文章内容，保持一致性更重要，术语映射可以作为后续扩展。

---

### 审查意见5：模式的迁移验证场景选择过于局限

**问题**：模式的迁移验证场景主要集中在智能体领域（测试平台、文档工具、数据分析），缺乏非智能体领域的迁移验证，可能影响模式的通用性。

**证据**：两个模式的迁移验证场景都在智能体/AI工具领域，没有扩展到传统软件工程、项目管理等领域。

**建议**：增加非智能体领域的迁移验证场景，如传统软件开发流程、运维管理、团队协作等，验证模式的跨领域适用性。

**采纳决策**：✅ 采纳——将在模式文档中补充非智能体领域的迁移验证场景。

---

### 审查意见6：洞察中的"下次行动"缺乏可衡量的成功标准

**问题**：三条洞察的"下次行动"部分都提出了具体的行动项，但缺乏可衡量的成功标准（SMART原则），难以判断行动是否成功。

**证据**：例如洞察1的行动项"制定Harness层能力提升路线图"没有明确的成功标准。

**建议**：为每个行动项补充可衡量的成功标准，使行动项更加具体可执行。

**采纳决策**：❌ 暂不采纳——当前洞察的行动项作为方向性指导即可，具体的成功标准需要根据实际项目情况制定。

## 采纳的修正

### 修正1：补充模型与Harness协同进化观点

在[harness-architecture-layered-model.md](file:///d:/spaces/SpecWeave/docs/retrospective/patterns/methodology-patterns/governance-strategy/harness-architecture-layered-model.md)中新增"协同进化"章节：

```
## 八、模型与Harness协同进化

模型能力和Harness架构是互补关系，而非替代关系：

| 维度 | 模型能力 | Harness架构 |
|------|---------|------------|
| 核心价值 | 提供智能推理能力 | 组织和放大智能能力 |
| 进化方向 | 更大参数、更强推理、更长上下文 | 更好的workflow、更完善的memory、更精细的permission |
| 协同效应 | 模型越强，Harness能做的事情越多 | Harness越好，模型能力越能发挥 |

**关键原则**：不要只追求其中一方的极致，而要追求两者的协同优化。一个强大的模型配上简陋的Harness，或者一个强大的Harness配上弱小的模型，都无法发挥最大价值。
```

### 修正2：补充文件系统记忆的具体实现策略

在[harness-architecture-layered-model.md](file:///d:/spaces/SpecWeave/docs/retrospective/patterns/methodology-patterns/governance-strategy/harness-architecture-layered-model.md)中补充目录结构模板和实现策略：

```
### 步骤2.1：目录结构模板

```
.harness/
├── memory/           # 持久记忆
│   ├── logs/         # 执行日志
│   ├── diffs/        # 代码diff
│   ├── failures/     # 失败记录
│   ├── summaries/    # 任务摘要
│   └── decisions/    # 决策过程
├── artifacts/        # 结果产物
│   ├── reports/      # 报告
│   ├── outputs/      # 输出文件
│   └── snapshots/    # 状态快照
├── config/           # 配置
│   ├── permissions/  # 权限配置
│   └── workflows/    # 工作流配置
└── context/          # 当前上下文（临时）
    └── current.md    # 当前任务上下文
```

### 步骤2.2：文件命名规范

- 执行日志：`run-{timestamp}-{task-id}.log`
- 代码diff：`diff-{timestamp}-{file-path}.diff`
- 失败记录：`failure-{timestamp}-{task-id}.md`
- 任务摘要：`summary-{timestamp}-{task-id}.md`
- 决策过程：`decision-{timestamp}-{decision-id}.md`

### 步骤2.3：检索策略

- 热数据：最近3轮的结果直接放入上下文窗口
- 温数据：最近24小时的结果建立索引，支持关键词搜索
- 冷数据：更早的结果归档，按需检索
```
